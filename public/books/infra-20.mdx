# 07. 인프라 20문항

> CI(Continuous Integration)/CD(Continuous Deployment), Docker, AWS, 배포 운영 감각을 확인하는 20문항입니다.

## 총평

프론트엔드 면접에서 인프라는 "알면 좋다" 수준이 아니라, 3년차쯤 되면 운영 감각이 있는지를 보는 질문으로 자주 나온다. 특히 좋은 답변은 아래를 보여줘야 한다.

- 자동화의 목적을 이해하는가
- 배포를 "코드 올리기"가 아니라 "운영 가능한 상태 만들기"로 보는가
- 작은 규모와 큰 규모의 선택 기준을 말할 수 있는가
- 도구 이름만 아는 게 아니라 왜 그 조합을 골랐는지 설명할 수 있는가

즉 이 섹션은 "Docker 써봤습니다", "AWS 써봤습니다"보다 **운영 문제를 어떻게 줄일지 생각하는 사람인지**를 본다.

### 전체 흐름 맵

```
CI/CD 파이프라인 (자동화의 뼈대)
  → GitHub Actions (파이프라인 구현)
    → 테스트 in CI (품질 게이트)
      → 배포 자동화 (운영 가능 상태)
        → PM2 (프로세스 관리)
          → EC2 배포 (서버 운영)
            → Nginx (리버스 프록시)
              → Docker (환경 격리)
                → Dockerfile 최적화 (빌드 효율)
                  → 멀티 스테이지 (이미지 경량화)
                    → 환경변수 보안 (비밀 관리)
                      → 환경 분리 (dev/staging/prod)
                        → AWS 서비스 (클라우드 인프라)
                          → 정적/동적 분리 (아키텍처)
                            → 영상 파이프라인 (무거운 워크로드)
                              → 롤백 전략 (장애 대응)
                                → 관측성 (로그/메트릭/알림)
                                  → 모노레포 CI (대규모 구조)
                                    → 1인 개발 인프라 (현실적 선택)
                                      → 확장 전략 (성장 경로)
```

## 1. CI와 CD의 차이를 설명해보세요.

### 합격 답변 예시

CI는 Continuous Integration으로, 코드 변경을 자주 통합하면서 자동 빌드와 테스트로 문제를 빠르게 검증하는 과정입니다. CD는 보통 Continuous Delivery 또는 Continuous Deployment로 이야기하는데, 검증된 변경을 실제 배포 가능한 상태까지 자동화하거나, 경우에 따라 운영 배포까지 자동화하는 흐름입니다. 즉 CI는 통합과 검증에 더 가깝고, CD는 전달과 배포 자동화에 더 가깝습니다.

### 가산점 포인트

Delivery와 Deployment의 차이를 구분해서 말하면 좋다. Delivery는 "언제든 배포할 수 있는 상태를 유지"하는 것이고, Deployment는 "검증 통과 시 자동으로 운영에 반영"하는 것이다. 대부분의 팀은 Delivery 수준에서 운영하며, 완전한 Deployment까지 가려면 테스트 커버리지와 모니터링이 충분히 성숙해야 한다.

### 상세 해설

#### 왜 CI가 필요했나

CI 이전에는 개발자들이 각자 브랜치에서 며칠~몇 주간 작업한 뒤 한꺼번에 합치는 "빅뱅 통합"을 했다. 이때 발생하는 문제:

- 충돌이 거대해져서 합치는 데만 며칠 소요
- "내 브랜치에서는 됐는데 합치니까 안 됨" — 통합 버그
- 누구의 변경이 문제인지 추적이 어려움

CI의 핵심 아이디어는 **"작게, 자주 합쳐서 문제를 빨리 발견한다"**는 것이다. 매일, 혹은 매 PR마다 자동으로 빌드하고 테스트해서 깨지면 바로 알 수 있게 만든다.

```yaml
# GitHub Actions CI 예시 — PR마다 자동 검증
name: CI
on: [pull_request]
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint          # 코드 스타일 검증
      - run: pnpm typecheck     # 타입 안전성 검증
      - run: pnpm test          # 단위 테스트
      - run: pnpm build         # 빌드 가능 여부 검증
```

#### CI → CD로 이어지는 이유

CI로 "코드가 합쳐져도 안전하다"를 확인했으면, 다음 질문은 자연스럽다: **"그러면 이걸 어떻게 사용자에게 전달할 것인가?"** 여기서 CD가 등장한다.

```
코드 변경 → CI(빌드+테스트) → CD(배포 자동화)
              ↑ 여기서 깨지면         ↑ 여기서 자동으로
              즉시 알림                운영에 전달
```

#### 자주 하는 오해

- "CI/CD는 GitHub Actions 같은 도구를 말한다" → **틀림.** CI/CD는 프로세스이고, GitHub Actions는 그 프로세스를 구현하는 도구 중 하나다.
- "CD만 있으면 CI는 필요 없다" → **틀림.** CD는 CI 위에 쌓이는 것이다. 검증 없이 배포 자동화만 하면 버그를 자동으로 운영에 뿌리는 셈.
- "CI/CD를 도입하면 버그가 사라진다" → **틀림.** CI/CD는 버그를 **빨리 발견하고 빨리 수정**하게 해줄 뿐, 버그 자체를 막지는 않는다.

#### 실무 에피소드

> CI 없이 수동 테스트만 하던 팀에서, 금요일 오후에 세 명이 동시에 main에 머지했다가 월요일에 프로덕션이 터졌다. 원인 추적에만 하루가 걸렸다. CI를 도입한 뒤에는 PR 단계에서 깨지는 코드가 머지 전에 걸러져서, "합쳐놓고 터지는" 사고가 크게 줄었다.

> CI/CD의 목적을 이해했으니, 다음에는 **실제로 파이프라인을 어떻게 설계하는지** 구체적으로 봐야 한다. 도구의 대표 격인 GitHub Actions를 기준으로 살펴보자.

## 2. GitHub Actions를 이용한 배포 파이프라인을 설계한다면 어떤 단계로 구성하겠습니까?

### 합격 답변 예시

저는 보통 트리거, 의존성 설치, 정적 검사, 테스트, 빌드, 아티팩트 생성, 배포, 배포 후 헬스체크, 실패 시 알림 또는 롤백 순서로 구성합니다. PR 단계에서는 lint, typecheck, unit test 중심으로 빠르게 검증하고, main 브랜치 머지 이후에는 build와 deploy를 수행하는 식으로 나누는 편이 좋습니다. 핵심은 배포 전에 최대한 많은 문제를 앞단에서 잡고, 배포 후에는 성공 여부를 확인할 수 있는 검증 단계까지 두는 것입니다.

### 가산점 포인트

PR 파이프라인과 배포 파이프라인을 **분리해서 설계**한다고 말하면 좋다. PR은 빠른 피드백(lint, typecheck, test), 배포는 안정적 전달(build, deploy, health check)에 집중해야 한다. 또 캐시 전략(`actions/cache`)을 언급하면 CI 경험의 깊이가 드러난다.

### 상세 해설

#### 왜 단계 순서가 중요한가 — Fail-Fast 원칙

파이프라인 설계의 핵심은 **비용이 적은 검증을 먼저, 비용이 큰 작업은 나중에** 두는 것이다. 이유는 단순하다:

```
lint (10초) → typecheck (20초) → test (60초) → build (90초) → deploy (120초)
   ↑ 여기서 실패하면             ↑ 여기까지 올 필요 없음
   10초만 낭비                    전체 5분 낭비
```

lint에서 세미콜론 하나 빠진 걸 잡을 수 있는데, build까지 가서 터지면 시간 낭비다.

```yaml
# 실무용 배포 파이프라인 예시
name: Deploy
on:
  push:
    branches: [main]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'           # 의존성 캐시로 설치 시간 단축
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck
      - run: pnpm test

  deploy:
    needs: verify               # verify 통과 후에만 실행
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - name: Deploy to EC2
        run: |
          ssh ${{ secrets.EC2_HOST }} "cd /app && git pull && pnpm install && pnpm build && pm2 reload all"
      - name: Health check
        run: |
          sleep 10
          curl -f https://myapp.com/api/health || exit 1
```

#### 자주 하는 오해

- "GitHub Actions는 배포 도구다" → **틀림.** Actions는 워크플로 자동화 도구이고, 배포는 그 안에서 실행하는 작업 중 하나일 뿐이다.
- "모든 단계를 하나의 job에 넣어야 한다" → **비효율적.** 검증과 배포를 분리하면 검증 실패 시 배포 job이 아예 실행되지 않아서 리소스를 아낄 수 있다.

#### 실무 에피소드

> deploy 단계에 health check를 안 넣었더니, 배포는 "성공"으로 표시됐는데 실제로는 서버가 죽어 있었다. 배포 = 코드 전달이 아니라 **배포 = 서비스가 정상 동작하는 상태까지 확인**이라는 걸 그때 깨달았다.

> 파이프라인의 뼈대를 잡았으니, 다음에는 **테스트를 CI에 넣는 구체적인 이유**를 봐야 한다. "테스트가 좋다"는 건 알지만, 왜 CI에 넣어야 하는지가 핵심이다.

## 3. 테스트를 CI 파이프라인에 넣는 이유를 설명해보세요.

### 합격 답변 예시

테스트를 CI에 넣는 이유는 개발자 로컬 환경 차이나 사람의 실수를 줄이고, 공통 품질 기준을 자동으로 강제하기 위해서입니다. 특히 회귀 버그를 빠르게 막고, 여러 사람이 동시에 작업하는 환경에서 '내 컴퓨터에서는 됐다' 문제를 줄이는 데 효과가 큽니다. 즉 테스트를 CI에 넣는 목적은 자동화 자체보다, 변경이 들어올 때마다 팀 전체가 같은 기준으로 검증받게 만드는 데 있습니다.

### 가산점 포인트

CI의 테스트는 "개인의 습관"이 아니라 **"팀 프로세스의 품질 게이트"**라고 말하면 좋다. 로컬에서 테스트를 까먹고 push해도, CI가 자동으로 잡아주기 때문이다. 이것이 **merge quality gate**의 핵심이다.

### 상세 해설

#### 왜 로컬 테스트만으로는 부족한가

개발자 A의 맥북과 개발자 B의 우분투에서 같은 테스트가 다른 결과를 낼 수 있다. 이유:

- Node 버전 차이
- OS별 파일 시스템 차이 (대소문자 구분)
- 환경변수 차이
- 설치된 전역 패키지 차이

CI는 **매번 깨끗한 환경에서 처음부터 실행**하기 때문에, 이런 "내 컴퓨터에서는 됐는데" 문제를 원천 차단한다.

```yaml
# CI에서 테스트를 quality gate로 사용하는 예시
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: pnpm test -- --coverage
      - name: Check coverage threshold
        run: |
          # 커버리지 80% 미만이면 실패
          pnpm test -- --coverage --coverageThreshold='{"global":{"branches":80}}'
```

#### 자주 하는 오해

- "CI에 테스트를 넣으면 개발이 느려진다" → **단기적으로는 맞지만 장기적으로는 틀림.** 테스트 없이 배포한 버그를 프로덕션에서 수정하는 비용이 CI에서 잡는 비용보다 훨씬 크다.
- "테스트가 통과하면 배포해도 안전하다" → **과신.** CI 테스트는 **알려진 시나리오**만 검증한다. 테스트에 없는 엣지 케이스는 여전히 놓칠 수 있다.

#### 실무 에피소드

> 프론트엔드 팀에서 테스트 없이 "수동 확인" 후 머지하다가, 날짜 포맷 유틸 함수를 수정한 PR이 결제 페이지의 날짜 표시를 깨뜨렸다. 해당 페이지를 수동으로 확인하지 않았기 때문. CI에 단위 테스트를 넣은 뒤에는 유틸 함수 변경 시 모든 사용처가 자동 검증되어 이런 회귀 버그가 줄었다.

> 테스트를 CI에 넣는 이유를 알았으니, 다음에는 **배포 자동화가 왜 필요한지** — 자동화가 없는 조직과 있는 조직의 실제 차이를 봐야 한다.

## 4. 배포 자동화가 없는 조직과 있는 조직의 차이를 운영 관점에서 설명해보세요.

### 합격 답변 예시

배포 자동화가 없는 조직은 배포가 사람의 기억과 수동 절차에 의존하기 때문에 재현성이 낮고, 실수 가능성이 높습니다. 반면 자동화가 있는 조직은 빌드, 테스트, 배포 절차가 코드화되어 있어서 일관성과 속도가 높고, 장애가 났을 때도 어떤 버전이 어떻게 나갔는지 추적하기 쉽습니다. 즉 자동화의 핵심 가치는 단순히 빠른 배포가 아니라, 예측 가능성과 복구 가능성을 높이는 데 있습니다.

### 가산점 포인트

자동화의 본질은 **속도가 아니라 "재현 가능한 안정적 반복"**이라고 말하면 좋다. 배포 절차가 코드화되어 있으면, 사람이 바뀌어도 같은 결과를 낼 수 있고, 장애 시 "누가 뭘 바꿨는지" 추적이 가능하다. 이것이 속도보다 중요하다.

### 상세 해설

#### 수동 배포의 실제 문제

| | 수동 배포 | 자동 배포 |
|---|---|---|
| **재현성** | 사람마다 절차가 다름 | 같은 스크립트가 매번 실행 |
| **추적성** | "누가 언제 뭘 올렸지?" | Git 커밋 + 워크플로 로그 |
| **롤백 속도** | "이전 버전이 뭐였지?" | 이전 빌드 아티팩트 즉시 재배포 |
| **야간 배포** | 담당자가 깨어 있어야 함 | 자동 실행 + 실패 시 알림 |
| **인수인계** | 문서화가 안 되면 지식 유실 | 파이프라인 코드가 곧 문서 |

#### 왜 "코드화"가 핵심인가

수동 배포의 근본 문제는 **절차가 사람의 머릿속에만 있다**는 것이다:

```bash
# 이런 절차가 사람의 기억에만 있으면?
ssh production-server
cd /var/www/myapp
git pull origin main
npm install
npm run build
pm2 restart all
# ...아, 환경변수 바꿔야 했는데 까먹었다
```

자동화하면 이 모든 절차가 코드로 남는다. **파이프라인 코드 = 배포 절차서**다.

#### 자주 하는 오해

- "자동화하면 무조건 빨라진다" → **핵심이 아님.** 빠르지만 매번 다르게 동작하면 의미 없다. 핵심은 **매번 같은 결과를 내는 것**.
- "소규모 팀에는 자동화가 과하다" → **오히려 소규모일수록 자동화가 중요.** 사람이 적을수록 한 명이 실수하면 복구할 사람이 없다.

#### 실무 에피소드

> 3명짜리 팀에서 수동 배포를 하다가, 담당자가 퇴사한 뒤 배포 절차를 아무도 모르는 상황이 생겼다. 서버 접속 정보, 배포 순서, 환경변수 설정까지 모두 노션 문서에 있었는데, 문서가 3개월 전 버전이라 맞지 않았다. GitHub Actions로 배포를 코드화한 뒤에는 파이프라인 자체가 "살아있는 문서"가 됐다.

> 배포 자동화의 가치를 알았으니, 다음에는 **배포된 서버의 프로세스를 어떻게 관리하는지** — PM2의 역할을 봐야 한다.

---

## 5. PM2는 왜 쓰는지 설명해보세요.

### 합격 답변 예시

PM2는 Node.js 프로세스를 안정적으로 운영하기 위해 씁니다. 프로세스가 죽었을 때 자동 재시작, 무중단 reload, 로그 관리, 클러스터 실행 같은 기능을 제공해서 작은 규모의 서버 운영에서 매우 실용적입니다. 즉 단순히 node server.js로 띄우는 것보다 운영 편의성과 복구력을 크게 높여줍니다.

### 가산점 포인트

`node server.js`와 PM2의 결정적 차이는 **"프로세스가 죽었을 때 무슨 일이 일어나는가"**다. node로 직접 실행하면 프로세스가 죽으면 끝이지만, PM2는 자동으로 다시 살린다. 또 `pm2 reload`는 무중단 재시작을 지원해서, 배포 중 사용자에게 downtime이 노출되지 않는다.

### 상세 해설

#### 왜 `node server.js`만으로는 부족한가

```bash
# 이렇게 실행하면?
node server.js

# 문제 1: SSH 세션 끊으면 프로세스도 죽음
# 문제 2: 에러로 프로세스가 crash하면 아무도 다시 안 살려줌
# 문제 3: 로그가 터미널에만 찍히고 파일로 안 남음
# 문제 4: CPU 코어를 하나만 사용
```

PM2는 이 모든 문제를 해결한다:

```js
// ecosystem.config.js — PM2 설정 파일
module.exports = {
  apps: [{
    name: 'my-next-app',
    script: 'node_modules/.bin/next',
    args: 'start',
    instances: 'max',        // CPU 코어 수만큼 클러스터
    exec_mode: 'cluster',    // 클러스터 모드
    watch: false,
    max_memory_restart: '500M',  // 메모리 500MB 넘으면 재시작
    env: {
      NODE_ENV: 'production',
      PORT: 3000,
    },
  }],
};
```

```bash
pm2 start ecosystem.config.js  # 설정대로 실행
pm2 reload all                 # 무중단 재시작 (배포 시)
pm2 logs                       # 실시간 로그 확인
pm2 monit                      # CPU/메모리 모니터링
```

#### 자주 하는 오해

- "PM2는 개발용 도구다" → **틀림.** PM2는 운영 환경에서 프로세스를 관리하기 위한 도구다. 개발에서는 `next dev`를 쓴다.
- "Docker를 쓰면 PM2가 필요 없다" → **상황에 따라 다름.** Docker + 오케스트레이션(k8s)이 있으면 컨테이너 수준에서 재시작을 관리하지만, EC2 + Docker 단독이면 PM2가 여전히 유용할 수 있다.

#### 실무 에피소드

> 새벽에 메모리 누수로 Node 프로세스가 죽었는데, PM2 없이 `node server.js`로 실행해놔서 아침까지 서비스가 다운이었다. PM2의 `max_memory_restart` 옵션을 걸어둔 뒤에는 메모리가 임계값을 넘으면 자동 재시작되어 다운타임이 사라졌다.

> PM2로 프로세스를 관리할 수 있게 됐으니, 다음에는 **EC2에 애플리케이션을 실제로 배포하는 전체 절차**를 봐야 한다.

---

## 6. EC2에 Node/Next 애플리케이션을 배포할 때 기본 절차를 설명해보세요.

### 합격 답변 예시

먼저 EC2 인스턴스를 준비하고 보안 그룹과 SSH 접근을 설정합니다. 그 다음 런타임과 패키지 매니저를 설치하고, 코드 배포나 artifact 전달 방식을 정한 뒤 환경변수를 주입하고 build를 수행합니다. 애플리케이션은 PM2 같은 프로세스 매니저로 실행하고, 외부에는 Nginx 같은 리버스 프록시를 앞단에 두어 HTTPS, 도메인 연결, 포트 은닉을 처리합니다. 마지막으로 로그 확인, 헬스체크, 재시작 설정, 롤백 절차까지 마련해야 운영 가능한 상태가 됩니다.

### 가산점 포인트

"코드를 서버에 올리고 npm start" 수준에서 끝내면 약하다. **"운영 가능한 상태"**에는 HTTPS, 프로세스 자동 재시작, 로그 수집, 헬스체크, 롤백이 모두 포함된다는 것을 보여줘야 한다.

### 상세 해설

#### 전체 절차 시각화

```
1. EC2 인스턴스 생성
   └─ 보안 그룹: 22(SSH), 80(HTTP), 443(HTTPS) 열기

2. 서버 초기 설정
   └─ Node.js, pnpm, PM2, Nginx, certbot 설치

3. 코드 배포
   └─ git clone / GitHub Actions에서 scp / Docker pull

4. 환경변수 주입
   └─ .env 파일 또는 PM2 ecosystem.config.js

5. 빌드 & 실행
   └─ pnpm build → pm2 start ecosystem.config.js

6. Nginx 리버스 프록시
   └─ 80/443 → localhost:3000 프록시

7. HTTPS 인증서
   └─ certbot으로 Let's Encrypt 인증서 발급

8. 운영 확인
   └─ 헬스체크, 로그 확인, 재시작 테스트
```

```bash
# 실제 서버 초기 설정 예시
# 1. Node.js 설치
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

# 2. pnpm, PM2 설치
npm install -g pnpm pm2

# 3. 코드 가져오기
git clone https://github.com/user/my-app.git /var/www/myapp
cd /var/www/myapp

# 4. 의존성 설치 + 빌드
pnpm install --frozen-lockfile
pnpm build

# 5. PM2로 실행
pm2 start ecosystem.config.js
pm2 save       # 재부팅 시 자동 실행 등록
pm2 startup    # 시스템 시작 시 PM2 자동 실행
```

#### 자주 하는 오해

- "EC2에 코드 올리면 배포 끝이다" → **시작일 뿐.** HTTPS, 프로세스 관리, 로그, 헬스체크, 롤백까지 갖춰야 "운영 가능한 상태"다.
- "보안 그룹에서 모든 포트를 열어야 한다" → **위험.** 필요한 포트(22, 80, 443)만 열고, Node 앱 포트(3000)는 외부에 직접 노출하지 않는다.

> EC2 배포 절차를 알았으니, 다음에는 **왜 Nginx를 앞단에 두는지** — 리버스 프록시의 역할을 자세히 봐야 한다.

---

## 7. Nginx를 앞단에 두는 이유를 설명해보세요.

### 합격 답변 예시

Nginx를 앞단에 두는 이유는 리버스 프록시 역할, SSL 종료, 도메인 연결, 정적 파일 전달, 캐시, 압축, 포트 은닉, 로드밸런싱 같은 운영 기능을 맡기기 위해서입니다. Node나 Next 앱을 인터넷에 직접 노출하는 것보다, Nginx가 앞에서 요청을 정리해주면 보안과 운영 제어가 훨씬 좋아집니다. 즉 애플리케이션은 비즈니스 로직에 집중하고, 웹 서버는 전달과 제어를 담당하게 역할을 나누는 것입니다.

### 가산점 포인트

**"역할 분리"**가 핵심이다. Node 앱이 HTTPS 처리, 압축, 정적 파일 서빙까지 직접 하면 코드가 복잡해지고, 보안 설정 실수의 위험도 커진다. Nginx에 이 역할을 위임하면 앱은 순수한 비즈니스 로직에만 집중할 수 있다.

### 상세 해설

#### Nginx가 처리하는 것들

```nginx
# /etc/nginx/sites-available/myapp
server {
    listen 80;
    server_name myapp.com;
    return 301 https://$host$request_uri;  # HTTP → HTTPS 리다이렉트
}

server {
    listen 443 ssl;
    server_name myapp.com;

    ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;

    # gzip 압축 — 네트워크 전송량 감소
    gzip on;
    gzip_types text/plain application/json application/javascript text/css;

    # 정적 파일 직접 서빙 — Node 앱에 부담을 주지 않음
    location /_next/static/ {
        alias /var/www/myapp/.next/static/;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # 나머지는 Node 앱으로 프록시
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

#### 왜 Node 앱을 직접 노출하면 안 되나

- **포트 노출**: Node 앱이 3000번 포트를 직접 인터넷에 여는 건 보안상 좋지 않다
- **HTTPS 복잡성**: Node에서 TLS를 직접 처리하면 인증서 갱신, 설정 관리가 복잡해짐
- **정적 파일 효율**: Nginx는 정적 파일 서빙에 최적화되어 있어 Node보다 훨씬 빠르다
- **DDoS 방어**: Nginx의 rate limiting으로 기본적인 보호가 가능

#### 자주 하는 오해

- "Next.js는 자체 서버가 있으니 Nginx가 필요 없다" → **개발에서는 맞지만 운영에서는 다름.** HTTPS, 압축, 캐시 헤더, rate limiting 같은 운영 기능은 Nginx가 훨씬 잘한다.

> Nginx로 앱 앞단을 정리했으니, 다음에는 **환경 자체를 격리하는 방법** — Docker의 이미지와 컨테이너 개념을 봐야 한다.

---

## 8. Docker 이미지와 컨테이너의 차이를 설명해보세요.

### 합격 답변 예시

Docker 이미지는 실행에 필요한 파일 시스템과 설정이 담긴 불변 템플릿이고, 컨테이너는 그 이미지를 실제로 실행한 인스턴스입니다. 즉 이미지는 실행 설계도에 가깝고, 컨테이너는 그 설계도를 바탕으로 떠 있는 실제 프로세스라고 볼 수 있습니다. 같은 이미지로 여러 컨테이너를 띄울 수 있다는 점이 핵심입니다.

### 가산점 포인트

이미지는 **불변(immutable)**이라는 점이 중요하다. 한 번 빌드한 이미지는 바뀌지 않기 때문에, "내 컴퓨터에서는 됐는데" 문제가 사라진다. 어디서 실행하든 같은 이미지 = 같은 환경 = 같은 결과다.

### 상세 해설

#### 비유

이미지는 **금형**, 컨테이너는 그 금형으로 찍어낸 **제품**이라고 보면 된다. 금형 하나로 제품을 여러 개 만들 수 있듯, 이미지 하나로 컨테이너를 여러 개 띄울 수 있다.

#### 왜 이 구분이 중요한가

```bash
# 이미지 빌드 — 한 번 만들면 불변
docker build -t myapp:1.0 .

# 컨테이너 실행 — 이미지에서 프로세스를 띄움
docker run -d --name myapp-1 -p 3000:3000 myapp:1.0
docker run -d --name myapp-2 -p 3001:3000 myapp:1.0  # 같은 이미지, 다른 포트

# 이미지 = 불변 템플릿 (레이어 구조, read-only)
# 컨테이너 = 실행 인스턴스 (이미지 위에 writable 레이어 추가)
```

#### 레이어 구조의 의미

Docker 이미지는 여러 레이어가 쌓인 구조다. 각 Dockerfile 명령어가 하나의 레이어를 만들고, 이 레이어는 read-only다. 컨테이너가 실행되면 맨 위에 writable 레이어가 추가된다.

```
┌──────────────────────┐
│  Writable Layer      │ ← 컨테이너 실행 시 추가 (로그, 임시 파일 등)
├──────────────────────┤
│  RUN pnpm build      │ ← 이미지 레이어 (read-only)
│  COPY . .            │
│  RUN pnpm install    │
│  COPY package.json   │
│  FROM node:20-alpine │ ← 베이스 이미지 레이어
└──────────────────────┘
```

#### 자주 하는 오해

- "컨테이너에서 파일을 수정하면 이미지도 바뀐다" → **틀림.** 이미지는 불변이고, 컨테이너의 writable 레이어에서만 변경이 일어난다. 컨테이너를 삭제하면 변경도 사라진다.
- "Docker를 쓰면 VM처럼 OS 전체가 올라간다" → **틀림.** 컨테이너는 호스트 OS 커널을 공유하므로 VM보다 훨씬 가볍다.

> Docker의 이미지와 컨테이너 개념을 알았으니, 다음에는 **이미지를 효율적으로 빌드하는 방법** — Dockerfile 레이어 캐시를 봐야 한다.

---

## 9. Dockerfile 작성 시 레이어 캐시를 어떻게 활용하나요?

### 합격 답변 예시

변경이 적은 단계는 위쪽에, 자주 바뀌는 단계는 아래쪽에 두는 식으로 레이어 캐시를 활용합니다. 예를 들어 package.json과 lockfile만 먼저 복사해서 의존성을 설치하고, 그 다음 소스 코드를 복사하면 소스만 바뀌었을 때 의존성 설치 레이어를 재사용할 수 있습니다. 이렇게 하면 빌드 시간이 크게 줄어듭니다.

### 가산점 포인트

"Docker build 속도"가 단순 편의가 아니라 **CI 시간, 배포 시간, 개발 반복 속도**에 직접 영향을 준다는 점까지 말하면 좋다. CI에서 매 PR마다 Docker 빌드를 하는데, 캐시를 잘 활용하면 5분짜리 빌드가 30초로 줄 수 있다.

### 상세 해설

#### 왜 순서가 중요한가 — 캐시 무효화 전파

Docker는 레이어를 위에서 아래로 순서대로 빌드한다. **한 레이어가 바뀌면 그 아래 모든 레이어의 캐시도 무효화**된다.

```dockerfile
# ❌ 나쁜 예 — 소스 코드가 바뀔 때마다 의존성도 다시 설치
COPY . .                    # 소스가 바뀌면 이 레이어부터 무효화
RUN pnpm install            # ← 매번 다시 실행됨 (느림)
RUN pnpm build

# ✅ 좋은 예 — 의존성 파일만 먼저 복사
COPY package.json pnpm-lock.yaml ./   # 의존성 파일이 안 바뀌면 캐시 유지
RUN pnpm install --frozen-lockfile     # ← 캐시 재사용 (빠름)
COPY . .                               # 소스만 바뀌면 여기서부터만 재빌드
RUN pnpm build
```

#### 시간 차이 체감

```
나쁜 순서: 소스 한 줄 수정 → 의존성 재설치 (3분) + 빌드 (1분) = 4분
좋은 순서: 소스 한 줄 수정 → 의존성 캐시 히트 (0초) + 빌드 (1분) = 1분
```

CI에서 하루에 30번 빌드한다면, 하루 90분 차이가 난다.

#### 자주 하는 오해

- "COPY . . 을 맨 위에 넣는 게 편하다" → **빌드 시간 폭발.** 소스 코드 한 줄만 바뀌어도 모든 레이어가 다시 빌드된다.
- ".dockerignore가 없어도 된다" → **위험.** node_modules, .git, .env 같은 불필요한 파일이 빌드 컨텍스트에 포함되면 빌드가 느려지고 이미지가 커진다.

> 레이어 캐시로 빌드 속도를 높였으니, 다음에는 **이미지 크기 자체를 줄이는 방법** — 멀티 스테이지 빌드를 봐야 한다.

## 10. 멀티 스테이지 빌드가 왜 필요한지 설명해보세요.

### 합격 답변 예시

멀티 스테이지 빌드는 빌드에 필요한 도구와 실제 실행에 필요한 산출물을 분리하기 위해 사용합니다. 예를 들어 첫 번째 단계에서는 빌드 도구와 dev dependency를 포함해 애플리케이션을 빌드하고, 마지막 단계에서는 빌드 결과물만 복사해 훨씬 작은 런타임 이미지를 만들 수 있습니다. 이렇게 하면 이미지 크기, 보안, 배포 속도 측면에서 유리합니다.

### 가산점 포인트

멀티 스테이지의 핵심은 **"빌드 도구는 런타임에 필요 없다"**는 원칙이다. TypeScript 컴파일러, webpack, devDependencies는 빌드 시에만 필요하고, 운영 이미지에는 빌드 결과물만 있으면 된다. 이 분리가 보안 surface도 줄이고 이미지 크기도 줄인다.

### 상세 해설

#### 왜 하나의 스테이지로는 부족한가

```dockerfile
# ❌ 단일 스테이지 — 빌드 도구까지 최종 이미지에 포함
FROM node:20
WORKDIR /app
COPY . .
RUN pnpm install           # devDependencies 포함
RUN pnpm build
CMD ["pnpm", "start"]
# 결과: 이미지 크기 ~1.2GB (node:20 + 모든 의존성 + 소스코드)
```

```dockerfile
# ✅ 멀티 스테이지 — 빌드 결과물만 최종 이미지에 포함
# Stage 1: 빌드
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile
COPY . .
RUN pnpm build

# Stage 2: 실행 (빌드 도구 없음)
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public
CMD ["node", "server.js"]
# 결과: 이미지 크기 ~150MB (alpine + 빌드 결과물만)
```

#### 크기 차이의 의미

| | 단일 스테이지 | 멀티 스테이지 |
|---|---|---|
| **이미지 크기** | ~1.2GB | ~150MB |
| **pull 시간** | 느림 | 빠름 |
| **보안 surface** | 넓음 (빌드 도구 포함) | 좁음 (실행에 필요한 것만) |
| **취약점 스캔** | 발견 많음 | 발견 적음 |

#### 자주 하는 오해

- "멀티 스테이지는 Docker 고급 기능이다" → **기본 중의 기본.** 프로덕션 이미지에 devDependencies가 들어가면 안 되므로, 운영에서는 사실상 필수다.
- "alpine 이미지를 쓰면 멀티 스테이지가 필요 없다" → **별개의 문제.** alpine은 OS 크기를 줄이는 것이고, 멀티 스테이지는 빌드 도구를 제거하는 것이다. 둘 다 써야 최적이다.

> 이미지를 경량화했으니, 다음에는 **애플리케이션의 비밀값을 어떻게 안전하게 관리하는지** — 환경변수 보안을 봐야 한다.

## 11. 환경변수 관리에서 가장 중요한 보안 포인트는 무엇인가요?

### 합격 답변 예시

가장 중요한 포인트는 비밀값을 저장소에 직접 커밋하지 않는 것과, 클라이언트에 노출되면 안 되는 값과 공개 가능한 값을 명확히 구분하는 것입니다. 또 개발, 스테이징, 운영 환경별로 값을 분리하고, 누가 어떤 비밀값에 접근 가능한지도 최소화해야 합니다. 즉 환경변수 관리는 편의성보다 비밀 분리와 노출 범위 통제가 핵심입니다.

### 가산점 포인트

Next.js에서 `NEXT_PUBLIC_` 접두사가 붙은 환경변수는 **클라이언트 번들에 포함**된다는 점을 반드시 알아야 한다. 즉 `NEXT_PUBLIC_API_KEY`는 브라우저 소스에서 볼 수 있다. 비밀값은 절대 `NEXT_PUBLIC_` 접두사를 붙이면 안 되고, 서버 사이드에서만 접근해야 한다.

### 상세 해설

#### 왜 "환경변수니까 안전하다"는 착각인가

```ts
// ❌ 위험 — NEXT_PUBLIC_은 클라이언트 번들에 포함됨
// .env
NEXT_PUBLIC_STRIPE_SECRET_KEY=sk_live_xxxxx  // 브라우저에서 보임!

// ✅ 안전 — 서버 전용 환경변수
// .env
STRIPE_SECRET_KEY=sk_live_xxxxx  // 서버에서만 접근 가능
NEXT_PUBLIC_STRIPE_PUBLIC_KEY=pk_live_xxxxx  // 공개키는 괜찮음
```

```bash
# .env.example — 저장소에 커밋하는 건 이 파일
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
STRIPE_SECRET_KEY=sk_test_your_key_here

# .env — 실제 값이 든 파일은 .gitignore에 추가
# .gitignore에 반드시 포함:
# .env
# .env.local
# .env.production
```

#### 환경변수 관리 체크리스트

1. **Git에 비밀값 커밋 금지** — `.env`는 `.gitignore`에 추가
2. **NEXT_PUBLIC_ 구분** — 공개 가능한 값만 접두사 부여
3. **환경별 분리** — dev/staging/prod 각각 다른 값
4. **접근 권한 최소화** — 필요한 사람만 운영 비밀값에 접근
5. **rotation 가능** — 유출 시 빠르게 교체할 수 있는 구조

#### 자주 하는 오해

- "환경변수는 코드에 안 들어가니 안전하다" → **틀림.** `NEXT_PUBLIC_`이 붙으면 빌드 시 코드에 인라인된다. 브라우저 DevTools에서 다 보인다.
- ".env 파일을 Git에 올려도 private 레포니까 괜찮다" → **위험.** Git 히스토리에 남으면 레포가 공개될 때, 또는 접근 권한이 바뀔 때 유출된다.

#### 실무 에피소드

> 신입 개발자가 `NEXT_PUBLIC_DATABASE_URL`이라고 설정해서, 데이터베이스 접속 정보가 프론트엔드 번들에 포함된 적이 있다. 다행히 스테이징 환경이었지만, `NEXT_PUBLIC_` 접두사의 의미를 팀 전체가 다시 교육받게 된 계기였다.

> 환경변수 보안을 잡았으니, 다음에는 **환경 자체를 왜 나눠야 하는지** — dev/staging/prod 분리의 이유를 봐야 한다.

## 12. 개발/스테이징/운영 환경을 나눠야 하는 이유를 설명해보세요.

### 합격 답변 예시

환경을 나누는 이유는 실험과 검증, 실제 운영을 분리하기 위해서입니다. 개발 환경은 빠른 반복이 중요하고, 스테이징은 운영과 유사한 환경에서 배포 전 검증을 하기 좋으며, 운영은 안정성이 최우선입니다. 환경을 분리하지 않으면 테스트 중 데이터 오염, 설정 혼선, 운영 장애 위험이 커집니다.

### 가산점 포인트

환경 분리는 단순히 서버를 여러 개 두는 것이 아니라, **"검증 단계 자체를 분리"**하는 것이라고 말하면 좋다. 스테이징에서 터진 문제는 사용자에게 영향이 없지만, 운영에서 터지면 매출에 직접 영향을 준다.

### 상세 해설

#### 각 환경의 역할과 차이

| | 개발 (dev) | 스테이징 (staging) | 운영 (production) |
|---|---|---|---|
| **목적** | 빠른 실험 | 운영 유사 검증 | 실제 서비스 |
| **데이터** | 목 데이터 / 시드 | 운영 유사 데이터 | 실제 사용자 데이터 |
| **안정성** | 깨져도 됨 | 안정적이어야 함 | 최우선 |
| **배포 주기** | 수시 | PR 머지마다 | 검증 후 |
| **비밀값** | 테스트용 키 | 스테이징 전용 키 | 운영 전용 키 |

#### 왜 분리하지 않으면 문제인가

```
환경이 하나뿐일 때:
  개발자가 테스트하다 DB를 날림 → 운영 데이터 손실
  테스트용 결제 → 실제 결제 발생
  디버그 로그 ON → 운영에서 성능 저하
```

#### 실무 에피소드

> 스테이징 없이 개발에서 바로 운영으로 배포했는데, 이미지 리사이즈 라이브러리 버전이 개발 환경에서는 됐지만 운영 서버의 OS에서는 바이너리 호환이 안 됐다. 스테이징에서 한 번만 테스트했으면 잡을 수 있었던 문제.

> 환경 분리의 필요성을 알았으니, 다음에는 **클라우드에서 환경을 어떻게 구성하는지** — AWS의 핵심 서비스 역할을 봐야 한다.

## 13. AWS S3, EC2, CloudFront 각각의 역할을 설명해보세요.

### 합격 답변 예시

S3는 객체 스토리지로 이미지, 영상, 정적 파일, 백업 등을 저장하는 데 적합합니다. EC2는 애플리케이션 서버를 직접 실행할 수 있는 가상 머신이고, CloudFront는 CDN으로서 정적 리소스를 사용자 가까운 엣지에서 빠르게 전달합니다. 실무에서는 정적 자산은 S3+CloudFront 조합으로 두고, 동적 처리나 SSR 서버는 EC2 같은 컴퓨팅 인스턴스에 두는 식으로 역할을 나누는 경우가 많습니다.

### 가산점 포인트

단순히 서비스 이름과 정의를 나열하는 게 아니라, **"왜 이 조합을 쓰는가"**를 말하면 좋다. S3만으로도 정적 파일을 서빙할 수 있지만, CloudFront를 앞에 두면 전 세계 엣지 캐시 + HTTPS + 커스텀 도메인을 쉽게 붙일 수 있다.

### 상세 해설

#### 역할 분리 아키텍처

```
사용자 요청
    ↓
┌─────────────────────────────────────────┐
│ CloudFront (CDN)                         │
│  ├─ 정적 자산: S3에서 캐시해서 전달       │
│  └─ 동적 요청: EC2로 프록시               │
└─────────────────────────────────────────┘
    ↓                    ↓
┌──────────┐     ┌──────────────┐
│   S3     │     │    EC2       │
│ 이미지   │     │ Next.js SSR  │
│ JS/CSS   │     │ API Routes   │
│ 영상     │     │ 동적 처리     │
└──────────┘     └──────────────┘
```

#### 각 서비스를 왜 쓰는가

- **S3**: 파일을 저장하는 데 서버가 필요 없다. 무한에 가까운 용량, 높은 내구성(99.999999999%), 저렴한 비용
- **EC2**: 서버에서 코드를 실행해야 할 때. SSH 접근, 커스텀 런타임, 직접 제어 가능
- **CloudFront**: 사용자와 가까운 엣지 서버에서 콘텐츠를 전달. S3 직접 접근보다 빠르고 HTTPS도 쉬움

#### 자주 하는 오해

- "S3에서 직접 정적 사이트를 호스팅하면 CloudFront가 필요 없다" → **가능하지만 제한적.** S3 직접 호스팅은 HTTPS가 안 되고, 리전 하나에서만 서빙된다. CloudFront를 붙여야 글로벌 캐시 + HTTPS가 된다.

> AWS 서비스의 역할을 알았으니, 다음에는 **정적 리소스와 동적 서버를 분리하면 구체적으로 어떤 장점이 있는지** 봐야 한다.

---

## 14. 정적 리소스와 동적 서버를 분리하면 어떤 장점이 있나요?

### 합격 답변 예시

정적 자산은 CDN 캐시를 통해 빠르게 전달하고, 동적 서버는 API나 SSR처럼 계산이 필요한 요청에 집중할 수 있어서 역할 분리가 명확해집니다. 이렇게 하면 서버 부하가 줄고, 확장 전략도 달라져 비용 효율이 좋아집니다. 또 장애가 났을 때 정적 자산 전달 문제와 동적 처리 문제를 나눠서 볼 수 있어 운영에도 유리합니다.

### 가산점 포인트

분리의 핵심 이유는 **"변경 빈도와 캐시 전략이 다르다"**는 것이다. JS/CSS/이미지는 빌드 시 생성되어 거의 안 바뀌므로 오래 캐시할 수 있지만, API 응답은 매 요청마다 달라질 수 있다. 이 둘을 같은 서버에서 처리하면 캐시 전략이 꼬인다.

### 상세 해설

#### 왜 같은 서버에서 둘 다 처리하면 비효율적인가

```
분리 전:
  EC2 하나에서 정적 파일 + SSR + API 전부 처리
  → 이미지 요청이 많으면 SSR 응답이 느려짐
  → 서버 스케일업밖에 방법이 없음

분리 후:
  정적 파일 → S3 + CloudFront (CDN이 알아서 처리)
  SSR + API → EC2 (비즈니스 로직에 집중)
  → 이미지 요청이 아무리 많아도 EC2에 부담 없음
  → 각각 독립적으로 스케일 가능
```

#### 장애 격리 관점

정적 자산과 서버가 분리되어 있으면, 서버가 죽어도 이미 캐시된 정적 파일은 계속 서빙된다. 사용자 입장에서 "페이지는 뜨는데 API가 안 됨" vs "아예 안 뜸"은 큰 차이다.

> 정적/동적 분리를 이해했으니, 다음에는 **무거운 워크로드를 서버에서 어떻게 격리하는지** — 영상 파이프라인 같은 CPU 집약 작업을 봐야 한다.

## 15. ffmpeg 기반 영상 파이프라인을 운영 환경에 넣는다면 어떤 리소스 문제를 고려해야 하나요?

### 합격 답변 예시

영상 인코딩은 CPU 사용량, 메모리, 디스크 I/O, 임시 파일 저장 공간, 처리 시간, 동시 작업 수를 먼저 고려해야 합니다. 특히 웹 서버와 같은 인스턴스에서 무거운 ffmpeg 작업을 같이 돌리면 다른 요청 응답 성능이 급격히 나빠질 수 있습니다. 그래서 작업 큐, 비동기 처리, 별도 워커 분리 같은 구조를 고려하는 것이 좋습니다.

### 가산점 포인트

핵심은 **"서빙과 처리를 분리"**하는 것이다. 웹 서버는 빠른 응답이 생명이고, 영상 인코딩은 오래 걸리는 CPU 집약 작업이다. 이 둘을 같은 프로세스/서버에서 돌리면 서로 간섭한다.

### 상세 해설

#### 왜 웹 서버와 같이 돌리면 위험한가

```
웹 서버 + ffmpeg가 같은 EC2에서 실행될 때:

CPU 사용률: ████████████████████ 100%
                  ↑ ffmpeg가 CPU를 다 가져감
웹 서버 응답 시간: 200ms → 5000ms → timeout
                  ↑ 사용자가 느끼는 건 "사이트가 죽었다"
```

#### 올바른 아키텍처

```
사용자 → 웹 서버 (요청 접수, 즉시 응답)
              ↓
         작업 큐 (Redis / SQS)
              ↓
         워커 서버 (ffmpeg 실행, 별도 인스턴스)
              ↓
         완료 시 → S3에 결과 저장 + 웹훅 알림
```

```ts
// 웹 서버: 업로드 요청을 받고 즉시 응답
app.post('/upload-video', async (req, res) => {
  const jobId = await uploadToS3(req.file);
  await jobQueue.add('encode', { jobId, format: 'mp4' });
  res.json({ jobId, status: 'processing' });  // 즉시 응답
});

// 워커: 별도 프로세스에서 인코딩 처리
jobQueue.process('encode', async (job) => {
  await ffmpeg(job.data.jobId, job.data.format);
  await notifyCompletion(job.data.jobId);
});
```

#### 자주 하는 오해

- "서버 사양을 높이면 된다" → **근본 해결이 아님.** CPU를 아무리 높여도 ffmpeg가 돌아가는 동안은 웹 서버가 영향을 받는다. 분리가 답이다.

> 무거운 워크로드를 분리하는 방법을 알았으니, 다음에는 **배포가 실패했을 때 어떻게 되돌리는지** — 롤백 전략을 봐야 한다.

---

## 16. 배포 후 롤백 전략은 어떻게 가져가야 하나요?

### 합격 답변 예시

롤백은 배포 실패 후 수동으로 급히 생각하는 게 아니라, 배포 전략 안에 미리 포함되어 있어야 합니다. 이전 안정 버전의 artifact나 이미지가 항상 남아 있어야 하고, health check 실패 시 빠르게 이전 버전으로 되돌릴 수 있어야 합니다. 핵심은 롤백도 배포만큼 자동화되고 예측 가능해야 한다는 점입니다.

### 가산점 포인트

**"롤백의 성공 기준은 실행이 아니라 health check 통과"**라고 말하면 좋다. 이전 버전이 실행됐다고 끝이 아니라, health check가 통과해야 "정상 복구"다. 또 DB 마이그레이션이 포함된 배포에서는 롤백이 단순하지 않다는 점도 언급하면 깊이가 있다.

### 상세 해설

#### 롤백이 가능하려면 필요한 것

```yaml
# GitHub Actions에서 이전 버전 롤백 예시
name: Rollback
on:
  workflow_dispatch:
    inputs:
      version:
        description: '롤백할 버전 태그'
        required: true

jobs:
  rollback:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy previous version
        run: |
          ssh ${{ secrets.EC2_HOST }} "
            cd /app
            git checkout ${{ inputs.version }}
            pnpm install --frozen-lockfile
            pnpm build
            pm2 reload all
          "
      - name: Health check
        run: curl -f https://myapp.com/api/health || exit 1
```

#### 롤백이 복잡해지는 경우

- **DB 마이그레이션 포함 배포**: 코드를 롤백해도 DB 스키마가 이미 바뀌었으면 충돌 발생
- **해결**: forward-only migration (이전 버전 코드와 호환되는 마이그레이션만 작성)

#### 자주 하는 오해

- "문제가 생기면 이전 커밋을 다시 배포하면 된다" → **너무 단순.** 빌드 시간, 환경변수 변경, DB 마이그레이션 등 고려할 것이 많다. 이전 빌드 아티팩트가 저장되어 있어야 빠른 롤백이 가능하다.

#### 실무 에피소드

> 금요일 저녁에 배포했다가 500 에러가 터졌는데, 이전 빌드 아티팩트를 안 남겨둬서 롤백하려면 git checkout + install + build를 다시 해야 했다. 15분이 걸렸다. 이후 Docker 이미지 태그로 관리하기 시작해서 롤백이 `docker pull + restart` 10초로 줄었다.

> 롤백 전략을 잡았으니, 다음에는 **장애를 빠르게 감지하고 대응하는 체계** — 로그, 메트릭, 알림을 봐야 한다.

---

## 17. 장애 발생 시 로그, 메트릭, 알림 체계를 어떻게 구성하면 좋을까요?

### 합격 답변 예시

로그는 원인 분석용, 메트릭은 추세 감지용, 알림은 즉시 대응용으로 나눠서 구성하는 게 좋습니다. 애플리케이션 로그와 시스템 메트릭(CPU, 메모리, 디스크), HTTP 에러율과 응답 시간, 배포 이벤트를 함께 봐야 실제 원인 분리가 가능합니다. 또 알림은 너무 많이 오면 무뎌지기 때문에, 실제 대응이 필요한 임계값 위주로 설계하는 것이 중요합니다.

### 가산점 포인트

**"모든 에러에 알림을 보내면 좋다"가 아니다.** 그건 경보 피로(alert fatigue)를 만든다. 알림은 **"지금 당장 사람이 행동해야 하는 것"**만 보내야 한다. 나머지는 대시보드에서 확인하면 된다.

### 상세 해설

#### 세 가지의 역할 차이

```
장애 발생!
    ↓
메트릭 (Metrics): "에러율이 5%에서 30%로 뛰었다" → 추세 감지
    ↓
알림 (Alert): "에러율 임계값 초과" → Slack/PagerDuty 알림
    ↓
로그 (Logs): "TypeError: Cannot read property 'id' of null at /api/users" → 원인 분석
```

#### 구체적 구성 예시

```ts
// 구조화된 로그 — JSON 형식으로 남기면 검색/필터가 쉬움
const logger = {
  error: (message: string, context: Record<string, unknown>) => {
    console.error(JSON.stringify({
      level: 'error',
      message,
      timestamp: new Date().toISOString(),
      ...context,
    }));
  },
};

// 사용 예시
logger.error('Payment failed', {
  userId: user.id,
  orderId: order.id,
  errorCode: 'STRIPE_DECLINED',
  requestId: req.headers['x-request-id'],
});
```

#### 자주 하는 오해

- "`console.log`로 충분하다" → **개발에서는 맞지만 운영에서는 부족.** 구조화된 JSON 로그여야 검색, 필터, 집계가 가능하다.
- "로그를 많이 남길수록 좋다" → **오히려 노이즈.** 중요한 이벤트만 적절한 레벨(error, warn, info)로 남겨야 장애 시 빠르게 찾을 수 있다.

> 관측성 체계를 잡았으니, 다음에는 **대규모 프로젝트에서 CI를 어떻게 효율적으로 관리하는지** — 모노레포 환경의 파이프라인 설계를 봐야 한다.

## 18. 모노레포 환경에서 앱별로 CI 파이프라인을 분리할지 통합할지 기준을 말해보세요.

### 합격 답변 예시

기준은 변경 영향 범위, 빌드 시간, 앱 간 결합도, 배포 주기입니다. 앱 간 독립성이 높고 배포 타이밍이 다르면 분리하는 게 좋고, 공통 패키지 영향이 크고 함께 검증해야 하는 게 많으면 일부 통합이 유리합니다. 실무에서는 전체 공통 검증과 변경 영향 기반 선택 실행을 적절히 섞는 방식이 현실적입니다.

### 가산점 포인트

**"한쪽만 옳다"가 아니라, 결합도와 배포 단위가 기준**이라고 말하는 것이 강한 답변이다. 또 `turbo run build --filter=...affected` 같은 도구를 언급하면 실무 경험이 드러난다.

### 상세 해설

#### 왜 이 질문이 어려운가

모노레포는 장점(코드 공유, 일관된 도구)과 단점(빌드 시간, 영향 범위)이 동시에 있다. CI 설계에서 핵심은 **"변경하지 않은 앱은 빌드하지 않는다"**는 원칙이다.

```yaml
# 변경된 패키지만 빌드하는 GitHub Actions 예시
jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      web: ${{ steps.filter.outputs.web }}
      api: ${{ steps.filter.outputs.api }}
    steps:
      - uses: dorny/paths-filter@v2
        id: filter
        with:
          filters: |
            web:
              - 'apps/web/**'
              - 'packages/shared/**'    # 공통 패키지도 포함
            api:
              - 'apps/api/**'
              - 'packages/shared/**'

  build-web:
    needs: detect-changes
    if: needs.detect-changes.outputs.web == 'true'
    runs-on: ubuntu-latest
    steps:
      - run: pnpm --filter web build
```

#### 분리 vs 통합 판단 기준

| 기준 | 분리가 유리 | 통합이 유리 |
|------|-----------|-----------|
| **앱 간 결합도** | 낮음 (독립 배포 가능) | 높음 (공통 패키지 의존) |
| **배포 주기** | 앱마다 다름 | 함께 배포 |
| **빌드 시간** | 각각 빌드해야 빠름 | 전체 빌드해도 부담 없음 |
| **팀 구조** | 앱별 팀 분리 | 하나의 팀 |

> 모노레포 CI를 이해했으니, 다음에는 **현실적인 제약 안에서 인프라를 설계하는 감각** — 1인 개발 서비스의 인프라를 봐야 한다.

---

## 19. 1인 개발 서비스에서 비용과 안정성을 동시에 고려해 인프라를 설계한 경험이 있다면 설명해보세요.

### 합격 답변 예시

1인 개발에서는 이상적인 분산 구조보다 내가 실제로 운영 가능한 단순한 구조를 고르는 게 더 중요하다고 생각합니다. 예를 들어 EC2 + PM2 + Nginx + GitHub Actions 같은 조합은 완전한 대규모 구조는 아니지만, 배포 자동화, 재시작, HTTPS, 로그, 롤백 같은 기본 운영 요소를 낮은 비용으로 갖출 수 있습니다. 즉 비용과 안정성의 균형은 "가장 최신 기술"보다 "내가 장애를 복구할 수 있는 구조"를 선택하는 데 있다고 봅니다.

### 가산점 포인트

**"관리 가능한 단순함"**이 핵심 키워드다. 1인 개발에서 k8s를 도입하면 인프라 관리에 시간이 다 빠진다. "지금 규모에 맞는 최소한의 운영 체계"를 갖추고, 규모가 커지면 그때 확장하는 게 현실적이다.

### 상세 해설

#### 1인 개발 인프라의 현실적 선택 기준

```
질문: "이 기술이 최신인가?" ← 잘못된 기준
질문: "이 구조에서 장애가 나면 내가 혼자 복구할 수 있는가?" ← 올바른 기준
```

#### 비용 비교 (월 기준 예상)

| 구성 | 월 비용 | 운영 복잡도 |
|------|--------|-----------|
| EC2 t3.micro + PM2 + Nginx | ~$10 | 낮음 |
| Vercel (무료 플랜) | $0~$20 | 매우 낮음 |
| ECS Fargate + ALB | ~$50+ | 중간 |
| k8s (EKS) | ~$73+ (클러스터만) | 높음 |

#### 자주 하는 오해

- "1인 개발이라도 Docker + k8s를 써야 제대로 한 거다" → **과잉 엔지니어링.** 사용자가 100명인 서비스에 k8s를 도입하면 인프라 관리가 본업이 된다.
- "PM2 + EC2는 레거시 구조다" → **규모에 맞는 합리적 선택.** Netflix가 쓰는 구조가 아니라, **내 서비스에 맞는 구조**가 좋은 구조다.

> 1인 개발의 현실적 선택을 알았으니, 마지막으로 **이 구조에서 규모가 커지면 어떻게 확장하는지** — 성장 경로를 봐야 한다.

---

## 20. GitHub Actions + AWS EC2 + PM2 조합을 선택한 이유와, 더 큰 규모가 되면 어떻게 확장할지 말해보세요.

### 합격 답변 예시

초기에는 이 조합이 단순하고, 비용이 낮고, 배포 자동화와 프로세스 관리까지 빠르게 갖출 수 있어서 적합합니다. GitHub Actions로 배포 절차를 자동화하고, EC2로 서버를 직접 제어하며, PM2로 프로세스 재시작과 운영 편의를 챙길 수 있기 때문입니다. 규모가 커지면 무중단 배포, 중앙 로그, 오토스케일링, 컨테이너 기반 배포, 워커 분리, 로드밸런서 도입 같은 방향으로 점진적으로 확장하는 게 좋습니다.

### 가산점 포인트

**"왜 지금 이 선택이 맞았고, 언제 바꿔야 하는지"**를 함께 말하면 강하다. "트래픽이 EC2 한 대로 감당이 안 되면 ALB + 다중 인스턴스, 배포 중 다운타임이 문제가 되면 Blue-Green, 로그 추적이 어려워지면 중앙 로깅 도입" 같은 **전환 트리거**를 말할 수 있어야 한다.

### 상세 해설

#### 확장 경로 시각화

```
현재: EC2 1대 + PM2 + Nginx + GitHub Actions
  ↓ 트래픽 증가
단계 1: ALB + EC2 2대 (로드밸런싱)
  ↓ 배포 시 다운타임 이슈
단계 2: Docker + ECR + Blue-Green 배포
  ↓ 서비스 종류 증가 (API, 워커, 크론)
단계 3: ECS Fargate (컨테이너 오케스트레이션)
  ↓ 팀/서비스 복잡도 급증
단계 4: EKS (Kubernetes) — 정말 필요할 때만
```

#### 각 단계의 전환 트리거

| 현재 문제 | 다음 단계 | 전환 이유 |
|----------|----------|---------|
| 트래픽이 한 대로 부족 | ALB + 다중 EC2 | 수평 확장 필요 |
| 배포 중 다운타임 | Blue-Green 배포 | 무중단 배포 필요 |
| 환경 재현성 문제 | Docker + ECR | "내 서버에서는 됐는데" 제거 |
| 서비스 종류 증가 | ECS Fargate | 컨테이너 관리 자동화 |
| 로그 추적 어려움 | 중앙 로깅 (CloudWatch / Datadog) | 여러 서버의 로그를 한 곳에서 |

#### 자주 하는 오해

- "처음부터 확장 가능한 구조로 만들어야 한다" → **과잉 설계.** 사용자가 없는데 오토스케일링을 고민하면 시간 낭비다. **현재 규모에 맞는 구조를 잡고, 문제가 생기면 그때 확장**하는 게 맞다.
- "EC2 + PM2는 확장이 안 된다" → **단계적 확장이 가능.** PM2 클러스터 → 다중 EC2 → Docker → ECS 순서로 자연스럽게 진화할 수 있다.

#### 실무 에피소드

> EC2 1대로 운영하다가 트래픽이 늘어서 응답 시간이 2초를 넘겼다. 처음에는 EC2 사양을 올렸지만(수직 확장), 금방 한계에 닿았다. ALB + EC2 2대로 바꾸니 응답 시간이 다시 200ms로 돌아왔다. 이때 "수직 확장의 한계"와 "수평 확장의 필요성"을 체감했다.

## 보완내용

### 왜 보완이 필요한가

기존 인프라 20문항은 CI/CD, PM2, EC2, Nginx, Docker, AWS, 롤백, 관측성까지 다루고 있어서 소규모 운영 경험을 보기엔 충분히 좋다. 다만 운영 성숙도를 더 분명하게 보려면 배포 전략, 비밀 관리, 관측성 심화, 이미지 배포 체계 같은 질문이 추가되면 더 강해진다.

특히 "배포는 해봤다"와 "운영 체계를 이해한다"는 다른 수준이기 때문에, 아래 질문을 추가하면 운영 감각의 깊이를 더 잘 확인할 수 있다.

### 추가하면 좋은 질문

#### 1. Blue-Green 배포와 Canary 배포의 차이를 설명해보세요.

배포 전략을 실제로 이해하는지 보기 좋은 질문이다. 롤백 용이성, 위험 분산, 트래픽 전환 방식까지 설명할 수 있으면 좋다.

##### 합격 답변 예시

Blue-Green 배포는 동일한 환경 두 세트(Blue와 Green)를 준비해 두고, 새 버전을 Green에 배포한 뒤 트래픽을 한 번에 전환하는 방식입니다. 문제가 생기면 다시 Blue로 돌리면 되기 때문에 롤백이 매우 빠릅니다. 반면 Canary 배포는 새 버전에 전체 트래픽의 일부(예: 5~10%)만 먼저 보내서 문제가 없는지 확인한 뒤 점진적으로 비율을 높이는 방식입니다. Blue-Green은 전환이 빠르고 롤백이 단순하지만 인프라 비용이 두 배 가까이 들고, Canary는 위험을 분산할 수 있지만 트래픽 분배와 모니터링 체계가 잘 갖춰져 있어야 합니다.

##### 상세 해설

이 질문의 핵심은 단순히 두 전략의 정의가 아니라 각각의 트레이드오프를 이해하는지다.

- Blue-Green
  - 환경 두 벌 유지
  - 트래픽 전환은 한 번에 (DNS 전환 또는 로드밸런서 전환)
  - 롤백 = 이전 환경으로 다시 전환
  - 단점: 인프라 비용, 데이터베이스 마이그레이션 동기화 주의
- Canary
  - 소량 트래픽으로 검증 후 점진 확대
  - 실사용자 기반 검증 가능
  - 단점: 트래픽 라우팅 설정 복잡, 모니터링 필수
- 좋은 답변 포인트
  - "둘 다 무중단 배포 전략이지만, Blue-Green은 빠른 전환과 롤백에, Canary는 위험 분산에 더 적합합니다"라고 정리하면 좋다

#### 2. readiness probe와 liveness probe 같은 헬스체크 개념을 설명해보세요.

"프로세스가 떠 있다"와 "실제로 요청을 받을 준비가 됐다"를 구분하는지 확인하는 질문이다. 배포 후 안정성을 어떻게 판단하는지 드러난다.

##### 합격 답변 예시

liveness probe는 컨테이너가 살아 있는지를 확인하는 검사이고, readiness probe는 실제로 트래픽을 받을 준비가 됐는지를 확인하는 검사입니다. 예를 들어 애플리케이션이 기동은 됐지만 아직 DB 연결이나 캐시 워밍업이 끝나지 않았다면 liveness는 통과하지만 readiness는 실패해야 합니다. liveness 실패 시에는 컨테이너를 재시작하고, readiness 실패 시에는 해당 인스턴스에 트래픽을 보내지 않는 식으로 역할이 다릅니다. 이 구분이 중요한 이유는 배포 직후 준비가 안 된 인스턴스에 요청이 들어가면 사용자에게 에러가 노출되기 때문입니다.

##### 상세 해설

이 질문은 "서버가 떠 있다 = 정상"이라는 단순한 사고를 넘어서는지 본다.

- liveness probe
  - 프로세스가 정상 동작 중인지 확인
  - 실패하면 컨테이너 재시작
  - 예: `/healthz` 엔드포인트가 200 응답
- readiness probe
  - 트래픽 수신 준비가 됐는지 확인
  - 실패하면 로드밸런서에서 제외
  - 예: DB 커넥션 확인, 캐시 로딩 완료 확인
- 핵심 차이
  - liveness: "죽었나 살았나"
  - readiness: "일할 준비가 됐나"
- 좋은 답변 포인트
  - 배포 시 rolling update에서 readiness probe가 없으면 준비 안 된 Pod에 트래픽이 가서 장애가 발생할 수 있다는 실무 연결까지 하면 좋다

#### 3. Secrets Manager / Parameter Store 같은 비밀 관리 서비스를 왜 쓰나요?

환경변수 질문을 한 단계 더 운영적으로 확장하는 문항이다. 비밀값 회전, 접근 통제, 감사 가능성까지 같이 설명할 수 있으면 좋다.

##### 합격 답변 예시

비밀 관리 서비스를 쓰는 이유는 크게 세 가지입니다. 첫째, 비밀값을 코드나 설정 파일에 직접 넣지 않고 중앙에서 안전하게 관리할 수 있습니다. 둘째, 누가 언제 어떤 비밀값에 접근했는지 감사(audit) 로그를 남길 수 있습니다. 셋째, 비밀값 회전(rotation)을 자동화하거나 최소한 쉽게 할 수 있어서 유출 시 대응이 빨라집니다. 즉 `.env` 파일이나 CI 환경변수에 직접 넣는 것보다 접근 통제, 감사 가능성, 회전 편의성 면에서 훨씬 운영에 적합합니다.

##### 상세 해설

이 질문은 환경변수 관리(11번)의 심화 버전이다.

- `.env` 파일이나 CI secret의 한계
  - 누가 읽었는지 추적 어려움
  - 회전 시 여러 곳 수동 변경 필요
  - 권한 세분화 어려움
- Secrets Manager / Parameter Store 장점
  - 중앙 관리
  - IAM 기반 접근 통제
  - 자동 회전 지원
  - 감사 로그
  - VPC 내부에서만 접근 가능하도록 네트워크 제한 가능
- 좋은 답변 포인트
  - "작은 규모에서는 CI secret으로 충분하지만, 팀이 커지거나 비밀값이 많아지면 중앙 관리 체계가 필요하고, 그때 Secrets Manager 같은 서비스가 필요해집니다"라고 규모별 판단까지 말하면 좋다

#### 4. ECR 같은 컨테이너 레지스트리를 왜 쓰고, 이미지 태깅 전략은 어떻게 가져가나요?

Docker를 쓴 경험이 있어도 이미지 배포 체계까지 이해하는지는 또 다르다. `latest`만 쓰는 전략이 왜 위험한지 설명할 수 있으면 운영 성숙도가 높다.

##### 합격 답변 예시

컨테이너 레지스트리는 빌드된 Docker 이미지를 안전하게 저장하고 배포 환경에서 빠르게 가져다 쓰기 위해 사용합니다. ECR 같은 서비스를 쓰면 AWS IAM과 연동해 접근 통제가 되고, 같은 네트워크 안에서 pull 속도도 빠릅니다. 태깅 전략은 `latest`만 쓰는 것은 피하고, Git 커밋 해시나 빌드 번호를 태그로 쓰는 것이 좋습니다. 그래야 지금 운영에 어떤 코드가 올라가 있는지 추적할 수 있고, 롤백할 때도 정확한 버전을 지정할 수 있습니다.

##### 상세 해설

이 질문은 Docker를 "개발용"이 아니라 "배포 체계의 일부"로 이해하는지 본다.

- 컨테이너 레지스트리를 쓰는 이유
  - 이미지를 중앙에서 관리
  - 배포 환경에서 일관된 이미지 사용 보장
  - 접근 통제 (누가 push/pull 가능한지)
  - 취약점 스캔 연동 가능
- `latest` 태그가 위험한 이유
  - 어떤 버전인지 알 수 없음
  - 캐시 문제로 이전 이미지가 재사용될 수 있음
  - 롤백 시 "이전 latest"를 찾을 수 없음
- 권장 태깅 전략
  - `git sha`: `myapp:abc1234`
  - `빌드 번호`: `myapp:build-42`
  - `시맨틱 버전`: `myapp:1.2.3`
- 좋은 답변 포인트
  - "이미지 태그는 배포 추적성과 롤백 가능성의 핵심이고, latest만 쓰면 둘 다 잃게 됩니다"라고 정리하면 좋다

#### 5. IaC(Infrastructure as Code)를 왜 쓰나요?

Terraform을 직접 쓰지 않았더라도, 인프라를 수동 클릭 대신 코드로 관리하는 이유를 설명할 수 있어야 한다. 재현성, 리뷰 가능성, 환경 일관성과 연결하기 좋다.

##### 합격 답변 예시

IaC를 쓰는 이유는 인프라 설정을 사람의 기억이나 수동 클릭이 아니라, 코드로 관리해서 재현 가능하고 리뷰 가능하게 만들기 위해서입니다. 예를 들어 Terraform이나 AWS CDK로 인프라를 정의하면, 동일한 환경을 언제든 다시 만들 수 있고, 변경 이력을 Git으로 추적할 수 있으며, PR 리뷰를 통해 인프라 변경도 코드 변경처럼 검증할 수 있습니다. 즉 IaC의 핵심 가치는 자동화보다 재현성, 일관성, 리뷰 가능성에 있습니다.

##### 상세 해설

이 질문은 도구(Terraform 등)를 써봤는지보다, 인프라를 코드로 관리하는 사고방식을 이해하는지 본다.

- 수동 관리의 문제
  - 콘솔 클릭은 기록이 안 남음
  - 누가 뭘 바꿨는지 추적 어려움
  - 동일 환경 재현 불가능
  - 실수 시 복구 어려움
- IaC의 장점
  - 재현성: 같은 코드로 같은 환경 생성
  - 리뷰 가능성: PR로 인프라 변경 검토
  - 버전 관리: Git 이력으로 변경 추적
  - 환경 일관성: 개발/스테이징/운영을 같은 코드로 생성
  - 자동화: CI/CD와 연결해 인프라 변경도 파이프라인으로 관리
- 좋은 답변 포인트
  - "배포 자동화가 애플리케이션 코드의 재현성을 높이듯, IaC는 인프라의 재현성을 높이는 같은 맥락의 사고방식입니다"라고 연결하면 좋다

#### 6. 로그, 메트릭 외에 tracing이 왜 필요한가요?

관측성을 한 단계 더 깊게 보는 질문이다. 분산된 요청 흐름을 어떻게 추적하는지, 장애 분석에서 tracing이 왜 필요한지 말할 수 있으면 좋다.

##### 합격 답변 예시

로그는 개별 이벤트의 상세 기록이고, 메트릭은 시스템 상태의 수치적 요약인데, 이 둘만으로는 하나의 요청이 여러 서비스를 거치면서 어디서 느려졌는지 또는 어디서 실패했는지 파악하기 어렵습니다. tracing은 하나의 요청이 시스템을 통과하는 전체 경로를 추적해서, 각 단계별 소요 시간과 성공 여부를 시각적으로 보여줍니다. 예를 들어 사용자 요청이 프론트 서버, 백엔드 서버, DB를 거치는 과정에서 병목이 어디인지를 trace ID 하나로 연결해서 볼 수 있습니다. 즉 로그는 "무슨 일이 있었는지", 메트릭은 "얼마나 이상한지", tracing은 "어디서 문제가 생겼는지"를 보여주는 관측성의 세 번째 축입니다.

##### 상세 해설

이 질문은 관측성(observability)의 세 가지 축을 이해하는지 본다.

- 관측성의 세 축
  - 로그(Logs): 개별 이벤트의 상세 기록, 원인 분석에 강함
  - 메트릭(Metrics): 수치 요약과 추세 감지, 대시보드와 알림에 강함
  - 트레이싱(Traces): 요청 흐름 추적, 병목과 장애 지점 파악에 강함
- tracing이 특히 필요한 상황
  - 마이크로서비스 또는 여러 서버를 거치는 요청
  - "느린데 어디서 느린지 모르겠다"는 상황
  - 간헐적 타임아웃의 원인 추적
- 주요 도구
  - Jaeger, Zipkin, AWS X-Ray, Datadog APM
  - OpenTelemetry로 표준화 추세
- 좋은 답변 포인트
  - "로그와 메트릭만으로는 '어디가 문제인지'를 보기 어렵고, tracing이 있어야 요청 단위로 전체 흐름을 볼 수 있습니다"라고 정리하면 좋다
  - 프론트엔드에서도 Performance API나 웹 바이탈 추적이 tracing과 비슷한 맥락이라고 연결하면 더 좋다

### 이 섹션 보완 핵심

인프라 섹션은 지금도 충분히 좋지만, 배포 전략, 헬스체크, 비밀 관리, 레지스트리 운영, IaC, tracing까지 더하면 "배포 경험자"와 "운영 체계를 이해하는 사람"을 더 분명하게 구분할 수 있다.
