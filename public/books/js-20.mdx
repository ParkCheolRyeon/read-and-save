# 01. js20문항

> 자바스크립트 실행 모델, 비동기, 브라우저 동작, 메모리까지 한 번에 훑는 20문항입니다.

## 1. var, let, const의 차이를 스코프, 호이스팅, 재할당 가능성 관점에서 설명해보세요.

| 선언 | 스코프 | TDZ | 재할당 | 실무 메모 |
| --- | --- | --- | --- | --- |
| `var` | 함수 스코프 | 없음 | 가능 | 재선언 가능, 전역 스크립트에서는 전역 객체 프로퍼티가 될 수 있음 |
| `let` | 블록 스코프 | 있음 | 가능 | 재선언 불가, 기본적인 가변 바인딩 |
| `const` | 블록 스코프 | 있음 | 불가 | 바인딩은 고정되지만 객체 내부 변경은 가능 |

### 합격 답변 예시

var는 함수 스코프를 가지고, let과 const는 블록 스코프를 가집니다. 세 선언 모두 호이스팅은 되지만, var는 선언과 동시에 undefined로 초기화되고, let과 const는 초기화 전까지 TDZ에 있어서 접근하면 에러가 납니다. 재할당은 var, let은 가능하고 const는 불가능합니다. 다만 const로 선언한 객체의 내부 프로퍼티 변경까지 막는 건 아닙니다. 실무에서는 재선언과 스코프 문제를 줄이기 위해 기본적으로 const, 필요할 때만 let을 사용하고 var는 거의 사용하지 않습니다.

### 가산점 포인트

var는 전역에서 선언 시 전역 객체 프로퍼티가 될 수 있다는 점, let/const는 그렇지 않다는 점까지 말하면 좋다.

### 상세 해설

가장 많이 나오는 질문이다. 하지만 여기서 중요한 건 “단순 비교표 암기”가 아니라 왜 위험했는가를 이해하는 것이다.

#### 스코프

- var: 함수 스코프
- let, const: 블록 스코프

```ts
if (true) {
  var a = 1;
  let b = 2;
  const c = 3;
}
console.log(a); // 1
console.log(b); // ReferenceError
console.log(c); // ReferenceError
```

var는 if 블록을 무시하고 함수 단위로 묶인다. 이게 옛날 코드에서 버그를 정말 많이 만들었다.

#### 호이스팅

많은 지원자가 틀린다.

- var: 호이스팅되고 undefined로 초기화
- let, const: 호이스팅되지만 초기화 전 접근 불가(TDZ)

```ts
console.log(x); // undefined
var x = 1;
console.log(y); // ReferenceError
let y = 2;
```

즉, let/const도 호이스팅은 된다. 다만 “사용 가능 상태”가 아니어서 TDZ에 걸리는 것이다.

#### 재할당

- var, let: 재할당 가능
- const: 재할당 불가

```ts
const obj = { name: "a" };
obj.name = "b"; // 가능
obj = {}; // 불가
```

const는 “값이 완전히 불변”이 아니라, 바인딩 자체가 고정되는 것이다.

#### 비유

var는 회사에서 아무 칸막이 없이 다 같이 쓰는 넓은 책상 느낌이다. 어디서 누가 덮어썼는지 추적이 힘들다. let/const는 각자 칸막이 있는 자리다. 범위가 명확해서 실수가 줄어든다.

## 2. 실행 컨텍스트가 무엇인지, 함수 호출 시 어떤 순서로 생성되고 종료되는지 설명해보세요.

### 합격 답변 예시

실행 컨텍스트는 자바스크립트 코드가 실행되기 위해 필요한 환경 정보 묶음입니다. 여기에는 변수/함수 선언 정보, 스코프 체인용 렉시컬 환경, this 바인딩 등이 포함됩니다. 코드가 실행되면 전역 실행 컨텍스트가 먼저 생성되고, 함수가 호출될 때마다 새로운 함수 실행 컨텍스트가 생성되어 콜 스택에 쌓입니다. 함수 실행이 끝나면 해당 컨텍스트는 스택에서 제거되고, 다시 이전 컨텍스트로 돌아갑니다.

### 가산점 포인트

생성 단계와 실행 단계를 나눠 말하면 훨씬 좋다.

### 상세 해설

이 질문은 JS를 “표면 문법”이 아니라 “실행 모델”로 이해하는지 보는 질문이다.

#### 실행 컨텍스트 안에 들어있는 것

쉽게 말하면, 코드를 실행하기 위한 작업 공간이다. 보통 아래가 핵심이다.

- 현재 스코프에서 선언된 식별자 정보
- 외부 스코프 참조

#### this

함수 호출 시 인자 정보 등

#### 생성 순서

- 전역 실행 컨텍스트 생성
- 전역 코드 실행

#### 함수 호출 시 함수 실행 컨텍스트 생성

- 콜 스택에 push
- 함수 종료 시 pop

```ts
function a() {
  console.log("a start");
  b();
  console.log("a end");
}
function b() {
  console.log("b");
}
a();
```

실행 흐름은 대략 이렇다.

- 전역 컨텍스트 생성
- a() 호출
- a 컨텍스트 생성
- b() 호출
- b 컨텍스트 생성
- b 종료
- a 종료
- 전역으로 복귀

#### 생성 단계 / 실행 단계

여기까지 말하면 꽤 좋다.

- 생성 단계: 변수/함수 선언 등록, 스코프 구성, this 결정
- 실행 단계: 실제 코드 한 줄씩 수행

이게 호이스팅과도 연결된다. 호이스팅은 “실행 전에 선언 정보가 먼저 등록되는 현상”이기 때문이다.

#### 비유

실행 컨텍스트는 요리할 때 각 요리사가 받은 작업대라고 보면 된다. 요리 하나 시작할 때 작업대가 열리고, 끝나면 정리된다. 작업대 위에는 재료(변수), 레시피 참조(외부 스코프), 담당자 정보(this)가 놓인다.

## 3. 렉시컬 스코프와 클로저를 설명하고, 실무에서 클로저가 유용했던 예를 말해보세요.

### 합격 답변 예시

렉시컬 스코프는 함수가 어디서 호출됐는지가 아니라 어디에 선언됐는지에 따라 상위 스코프가 결정되는 규칙입니다. 클로저는 함수가 선언될 당시의 렉시컬 환경을 계속 참조할 수 있는 구조를 말합니다. 그래서 외부 함수 실행이 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있습니다. 실무에서는 이벤트 핸들러 팩토리, 디바운스 구현, private state 은닉 같은 곳에서 클로저가 유용했습니다.

### 가산점 포인트

“클로저는 함수와 그 함수가 참조하는 렉시컬 환경의 조합”처럼 정의를 깔끔하게 말하면 좋다.

### 상세 해설

#### 렉시컬 스코프

JS는 호출 위치가 아니라 선언 위치로 스코프가 정해진다.

```ts
const x = 10;
function outer() {
  const x = 20;
  function inner() {
    console.log(x);
  }
  return inner;
}
const fn = outer();
fn(); // 20
```

fn()이 어디서 호출됐든, inner는 자신이 선언된 위치의 x를 본다.

#### 클로저

클로저는 내부 함수가 외부 함수의 변수에 접근할 수 있는 이유다.

```ts
function createCounter() {
  let count = 0;
  return function () {
    count += 1;
    return count;
  };
}
const counter = createCounter();
console.log(counter()); // 1
console.log(counter()); // 2
```

createCounter()는 끝났지만 count는 살아 있다. 왜냐하면 반환된 내부 함수가 그 환경을 참조하고 있기 때문이다.

#### 실무 예시

#### 디바운스

```ts
function debounce(fn, delay) {
  let timerId;
  return function (...args) {
    clearTimeout(timerId);
    timerId = setTimeout(() => fn.apply(this, args), delay);
  };
}
```

#### 핸들러 생성기

```ts
function createClickHandler(id) {
  return function () {
    console.log("clicked:", id);
  };
}
```

#### private state

```ts
function createStore() {
  let state = 0;
  return {
    get() {
      return state;
    },
    set(value) {
      state = value;
    },
  };
}
```

외부에서 state를 직접 만지지 못하게 할 수 있다.

#### 비유

클로저는 함수가 들고 다니는 작은 가방이라고 보면 된다. 함수가 만들어질 때 필요한 바깥 변수들을 그 가방에 담아두고, 나중에 다른 곳에서 실행돼도 그걸 꺼내 쓴다.

## 4. 클로저가 잘못 사용됐을 때 생길 수 있는 문제는 무엇인가요?

### 합격 답변 예시

클로저 자체는 유용하지만, 불필요하게 큰 객체나 DOM 참조를 계속 잡고 있으면 메모리 사용량이 늘 수 있습니다. 또 반복문에서 var와 함께 쓰면 의도와 다르게 같은 변수를 공유해서 버그가 날 수 있습니다. 그리고 비동기 처리나 콜백에서 오래된 값을 캡처하면 stale closure처럼 최신 상태가 아니라 과거 값을 참조하는 문제도 생길 수 있습니다. 그래서 필요한 범위만 캡처하고, 오래 유지될 필요가 없는 참조는 정리하는 습관이 중요합니다.

### 가산점 포인트

루프 + var 예시를 바로 보여주면 좋다.

### 상세 해설

#### 문제 1) 메모리 유지

클로저가 참조하는 값은 쉽게 GC 대상이 안 된다.

```ts
function attachHandler() {
  const bigData = new Array(1000000).fill("data");
  return function () {
    console.log(bigData.length);
  };
}
```

이 핸들러가 오래 살아 있으면 bigData도 같이 살아 있을 수 있다.

#### 문제 2) 반복문 + var

```ts
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 3, 3, 3
```

var는 블록 스코프가 아니라 하나의 i를 공유해서 그렇다. 해결:

```ts
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 0, 1, 2
```

#### 문제 3) 오래된 값 참조

비동기 로직에서 클로저가 “그 시점의 값”을 잡고 있으면, 나중에 실행될 때 최신 상태가 아닐 수 있다. 이건 특히 UI 상태 관리, 타이머, 이벤트 핸들러에서 많이 나온다.

#### 핵심 한 줄

- 클로저 문제의 본질은
- “필요한 것까지 오래 들고 있느냐”
- 혹은

“같은 변수 하나를 여러 타이밍이 공유하느냐”다.

## 5. 원시값과 참조값의 차이를 설명하고, 객체를 복사할 때 주의할 점을 말해보세요.

### 합격 답변 예시

원시값은 숫자, 문자열, 불리언, null, undefined, symbol, bigint처럼 값 자체가 저장되는 타입이고, 복사할 때도 값이 그대로 복사됩니다. 반면 객체, 배열, 함수 같은 참조형은 변수에 객체 자체가 들어간다기보다 그 객체를 가리키는 참조가 저장됩니다. 그래서 객체를 다른 변수에 대입하면 독립 복사가 아니라 같은 객체를 함께 바라보게 됩니다. 객체 복사 시에는 얕은 복사인지 깊은 복사인지 구분해야 하고, 특히 중첩 객체가 있으면 의도치 않게 원본이 같이 바뀔 수 있어서 주의해야 합니다.

### 가산점 포인트

“참조형도 엄밀히 말하면 참조값이 복사된다”라고 표현하면 좋다.

### 상세 해설

#### 원시값

```ts
let a = 10;
let b = a;
b = 20;
console.log(a); // 10
console.log(b); // 20
```

서로 독립적이다.

#### 참조형

```ts
const obj1 = { name: "kim" };
const obj2 = obj1;
obj2.name = "lee";
console.log(obj1.name); // "lee"
obj2 = obj1은 “복제”가 아니라 “같은 대상 공유”다.
```

#### 면접에서 중요한 포인트

많은 사람이 “객체는 참조로 전달된다”고만 말하는데, 더 정확히는 참조값이 복사된다고 말하는 편이 좋다. 즉 함수 인자 전달도 pass-by-reference라고 단정하기보다, 객체를 가리키는 값이 복사되어 전달된다고 이해하는 게 더 정확하다.

## 6. 얕은 복사와 깊은 복사의 차이는 무엇이며, 각각 어떤 상황에서 문제가 되나요?

### 합격 답변 예시

얕은 복사는 1단계 프로퍼티만 복사하고, 내부 중첩 객체는 기존 참조를 그대로 공유하는 복사입니다. 깊은 복사는 중첩된 구조까지 재귀적으로 복사해서 원본과 완전히 분리된 새 객체를 만드는 방식입니다. 얕은 복사는 간단한 상태 업데이트에는 충분하지만, 중첩 객체가 있을 때 의도치 않게 원본이 같이 바뀌는 문제가 생길 수 있습니다. 깊은 복사는 안전하지만 비용이 더 크고, 방법에 따라 함수나 Date, Map, Set 같은 타입이 깨질 수 있어서 상황에 맞게 선택해야 합니다.

### 가산점 포인트

structuredClone과 JSON.stringify/parse 차이를 언급하면 좋다.

### 상세 해설

#### 얕은 복사

```ts
const user = {
```

name: "kim",

```ts
  address: {
```

city: "Seoul",

```ts
  },
};
const copied = { ...user };
copied.address.city = "Busan";
console.log(user.address.city); // "Busan"
```

겉 껍데기만 새로 만들고 내부 객체는 공유한다.

#### 깊은 복사

중첩된 곳까지 전부 새로 만들어야 한다.

```ts
const copied = structuredClone(user);
```

#### JSON 복사의 한계

```ts
const copied = JSON.parse(JSON.stringify(user));
```

- 이건 간단한 plain object에는 쓸 수 있지만,
- 함수
- undefined
- Date
- Map
- Set
- 순환 참조

등에서 문제가 생길 수 있다.

- 실무에서 문제 되는 상황
- 상태 관리에서 원본이 같이 오염됨
- 캐시 데이터가 의도치 않게 변함
- 디버깅할 때 “왜 여기서 바뀌었지?” 추적이 어려움

#### 면접 포인트

단순히 “얕은 복사는 한 단계, 깊은 복사는 전체”에서 끝내지 말고 언제 문제가 되는지를 말해야 한다.

## 7. this는 어떻게 결정되나요? 일반 함수, 메서드, 생성자 함수, 화살표 함수에서 각각 설명해보세요.

### 합격 답변 예시

this는 함수가 선언된 위치보다 호출 방식에 의해 결정되는 경우가 대부분입니다. 일반 함수는 단순 호출이면 strict mode에서는 undefined, 비엄격 모드에서는 전역 객체를 가리킬 수 있습니다. 메서드 호출에서는 점 앞의 객체가 this가 되고, 생성자 함수는 new로 호출할 때 새로 생성된 인스턴스를 this로 바인딩합니다. 화살표 함수는 자체적인 this를 가지지 않고, 선언된 상위 스코프의 this를 그대로 캡처합니다.

### 가산점 포인트

new > bind/call/apply > method call > default binding 같은 우선순위를 말하면 아주 좋다.

### 상세 해설

this는 JS에서 가장 헷갈리는 주제 중 하나다. 핵심은 “누가 호출했는가”다.

## 1. 일반 함수 호출

```ts
function fn() {
  console.log(this);
}
fn();
```

- strict mode: undefined
- sloppy mode: 전역 객체

## 2. 메서드 호출

```ts
const obj = {
```

name: "kim",

```ts
  say() {
    console.log(this.name);
  },
};
obj.say(); // "kim"
```

obj.say()에서 say를 호출한 주체는 obj다.

## 3. 생성자 함수 호출

```ts
function Person(name) {
  this.name = name;
}
const p = new Person("kim");
```

- new를 쓰면
- 새 객체 생성
- 그 객체를 this에 바인딩
- 프로토타입 연결
- 생성자 실행
- 특별한 반환이 없으면 새 객체 반환

## 4. 화살표 함수

```ts
const obj = {
```

name: "kim",

```ts
  say: () => {
    console.log(this.name);
  },
};
```

obj.say(); 여기서 화살표 함수의 this는 obj가 아니다. 화살표 함수는 자기 this가 없고 바깥 스코프에서 가져온다.

#### 핵심

- 일반 함수: 호출 방식에 따라 달라짐
- 화살표 함수: 선언 위치의 상위 this 사용

## 8. 화살표 함수와 일반 함수의 차이를 this, arguments, 생성자 사용 가능 여부 중심으로 설명해보세요.

### 합격 답변 예시

화살표 함수는 자체적인 this와 arguments를 가지지 않고 상위 스코프의 값을 사용합니다. 반면 일반 함수는 호출 방식에 따라 자신의 this가 결정되고, 자체 arguments 객체도 가집니다. 또 일반 함수는 생성자 함수로 사용할 수 있지만 화살표 함수는 new로 호출할 수 없습니다. 그래서 콜백처럼 상위 컨텍스트를 유지하고 싶을 때는 화살표 함수가 편하고, 동적인 this가 필요하거나 생성자로 쓸 함수는 일반 함수를 사용해야 합니다.

### 상세 해설

#### this

```ts
const obj = {
```

value: 1,

```ts
  regular: function () {
    console.log(this.value);
  },
  arrow: () => {
    console.log(this.value);
  },
};
obj.regular(); // 1
obj.arrow();   // 보통 undefined
```

#### arguments

```ts
function regular() {
  console.log(arguments);
}

function outer() {
  const arrow = () => {
    console.log(arguments); // outer의 arguments를 참조
  };

  arrow("ignored");
}
```

화살표 함수는 자신만의 `arguments` 객체를 만들지 않는다. 필요하면 rest parameter를 쓰는 편이 일반적이다.

```ts
const arrow = (...args) => {
  console.log(args);
};
```

#### 생성자 가능 여부

```ts
const A = () => {};
new A(); // TypeError
```

화살표 함수는 prototype도 생성자 용도로 쓰이지 않는다.

#### 면접 포인트

“화살표 함수가 더 최신이라서 더 좋다” 식으로 말하면 약하다. 중요한 건 용도에 맞는 선택이다.

## 9. 프로토타입 체인이 무엇인지 설명하고, 클래스 문법이 내부적으로 어떻게 동작하는지 말해보세요.

### 합격 답변 예시

자바스크립트 객체는 프로퍼티를 찾을 때 현재 객체에 없으면 자신의 프로토타입을 따라 올라가며 탐색하는데, 이 연결 구조를 프로토타입 체인이라고 합니다. 클래스 문법은 이런 프로토타입 기반 상속을 더 읽기 쉽게 감싼 문법적 설탕에 가깝습니다. 클래스 메서드는 실제로 인스턴스마다 복제되는 게 아니라 보통 클래스의 프로토타입에 올라가고, extends는 프로토타입 체인을 연결해 상속을 구현합니다. 즉 자바스크립트는 본질적으로 프로토타입 기반 언어이고, 클래스는 그 위에 얹힌 표현 방식입니다.

### 가산점 포인트

“인스턴스가 메서드를 직접 갖는 게 아니라 프로토타입을 통해 공유한다”는 말을 꼭 넣자.

### 상세 해설

#### 프로토타입 체인

```ts
const arr = [1, 2, 3];
arr.map(...)
```

arr 객체 자체에 map이 없더라도 Array.prototype에 있으니 사용할 수 있다.

- 탐색 흐름은 대략:
- arr에서 찾음
- 없으면 Array.prototype
- 없으면 Object.prototype
- 없으면 null

이 연결이 프로토타입 체인이다.

#### 클래스 내부 동작

```ts
class Person {
  constructor(name) {
    this.name = name;
  }
  sayHi() {
    console.log(this.name);
  }
}
```

이건 개념적으로 비슷하게 보면 된다.

```ts
function Person(name) {
  this.name = name;
}
Person.prototype.sayHi = function () {
  console.log(this.name);
};
```

즉 클래스는 JS를 Java처럼 바꾸는 게 아니라, 프로토타입 기반 구조를 더 읽기 좋게 표현한 것이다.

#### 상속

```ts
class Animal {
  speak() {
    console.log("...");
  }
}
class Dog extends Animal {
  speak() {
    console.log("woof");
  }
}
```

Dog.prototype이 Animal.prototype을 참조하도록 연결된다.

## 11. 이벤트 루프를 설명해보세요. 콜 스택, 태스크 큐, 마이크로태스크 큐를 함께 설명해보세요.

### 합격 답변 예시

자바스크립트 엔진은 기본적으로 단일 스레드에서 콜 스택을 기준으로 동작하고, 비동기 작업은 브라우저나 런타임의 API에 위임됩니다. 비동기 작업이 끝나면 콜백은 큐로 들어오고, 이벤트 루프는 콜 스택이 비었을 때 큐의 작업을 스택으로 옮겨 실행합니다. 이때 마이크로태스크 큐는 일반 태스크 큐보다 우선순위가 높아서, 현재 실행 중인 작업이 끝난 뒤 다음 태스크로 넘어가기 전에 먼저 비워집니다. 대표적으로 Promise.then, catch, finally는 마이크로태스크이고, setTimeout은 태스크 큐에 들어갑니다.

### 가산점 포인트

렌더링은 보통 마이크로태스크가 비워진 뒤 사이사이에 일어난다고 말하면 좋다.

### 상세 해설

이 질문은 아주 중요하다. 왜냐하면 “비동기 코드가 왜 저 순서로 실행됐는지”를 설명할 수 있어야 하기 때문이다.

#### 기본 구조

- 콜 스택: 지금 실행 중인 함수들
- 태스크 큐(매크로태스크): setTimeout, setInterval, I/O 등
- 마이크로태스크 큐: Promise 후속 처리 등

#### 예시

```ts
console.log(1);
setTimeout(() => console.log(2), 0);
Promise.resolve().then(() => console.log(3));
console.log(4);
```

- 출력:
- 1
- 4
- 3
- 2

왜?

- 동기 코드 먼저 실행: 1, 4
- 현재 태스크 종료
- 마이크로태스크 비움: 3
- 그다음 태스크 실행: 2
- 실무에서 중요한 이유
- 로딩 상태가 왜 늦게 바뀌는지
- 렌더 직전/직후 순서가 왜 그런지

```ts
setTimeout(..., 0)이 왜 즉시가 아닌지
```

이런 걸 설명하는 기반이 된다.

### 합격 답변 예시

  });

## 13. async/await가 내부적으로 어떻게 동작하는지 Promise와 연결해서 설명해보세요.

### 합격 답변 예시

async/await는 Promise 기반 비동기 코드를 더 동기식처럼 읽히게 해주는 문법이고, 내부적으로는 Promise 체인을 더 읽기 좋은 형태로 풀어쓴 것에 가깝습니다. async 함수는 항상 Promise를 반환하고, await는 Promise가 resolve 또는 reject될 때까지 함수 내부 실행을 일시 중단한 뒤 결과를 받아 다음 코드를 이어갑니다. 다만 이 중단은 스레드를 멈추는 게 아니라, 해당 함수의 나머지 부분을 나중에 마이크로태스크로 이어서 실행하는 방식입니다. 그래서 try/catch로 비동기 에러를 동기 코드처럼 다루기 쉬워집니다.

### 가산점 포인트

await는 Promise가 아니어도 Promise.resolve처럼 감싸 처리된다는 점을 말하면 좋다.

### 상세 해설

async 함수는 항상 Promise 반환

```ts
async function foo() {
  return 1;
}
foo(); // Promise<1>
```

await

```ts
async function foo() {
  const result = await fetchData();
  console.log(result);
}
```

이건 개념적으로 비슷하다.

```ts
function foo() {
  return fetchData().then((result) => {
    console.log(result);
  });
}
```

중요한 오해 교정 await는 “JS 전체를 멈추는 것”이 아니다. 그 함수 내부 흐름만 잠시 멈추고, 런타임은 다른 작업을 계속 처리한다.

## 14. Promise.all, Promise.allSettled, Promise.race, Promise.any의 차이를 설명해보세요.

### 합격 답변 예시

Promise.all은 모든 Promise가 성공해야 전체가 성공하고, 하나라도 실패하면 즉시 reject됩니다. Promise.allSettled는 성공/실패와 상관없이 전부 끝날 때까지 기다리고, 각각의 결과 상태를 배열로 반환합니다. Promise.race는 가장 먼저 settle되는 Promise 결과를 그대로 따르고, 성공이든 실패든 먼저 끝난 쪽이 이깁니다. Promise.any는 가장 먼저 성공한 Promise를 반환하고, 모두 실패하면 AggregateError로 reject됩니다.

### 가산점 포인트

각각의 실무 용도를 같이 말하면 좋다.

### 상세 해설

```ts
Promise.all
await Promise.all([a(), b(), c()]);
```

- 전부 성공해야 성공
- 하나 실패하면 전체 실패
- 사용 예:
- 페이지 진입 전에 필요한 필수 데이터 병렬 조회

```ts
Promise.allSettled
await Promise.allSettled([a(), b(), c()]);
```

- 다 끝날 때까지 기다림
- 각 결과를 개별적으로 확인 가능
- 사용 예:
- 여러 요청 결과를 모아 실패한 것만 따로 표시

```ts
Promise.race
await Promise.race([request(), timeout()]);
```

- 가장 먼저 끝난 것
- 성공/실패 상관없음
- 사용 예:
- 타임아웃 구현

```ts
Promise.any
await Promise.any([cdn1(), cdn2(), cdn3()]);
```

- 가장 먼저 성공한 것
- 전부 실패하면 reject
- 사용 예:
- 여러 대체 자원 중 하나만 성공하면 되는 경우

## 15. 디바운스와 스로틀의 차이를 설명하고, 각각 어떤 UI 상황에서 쓰는지 말해보세요.

### 합격 답변 예시

디바운스는 이벤트가 연속해서 발생할 때 마지막 호출만 일정 시간 뒤에 실행되도록 하는 방식이고, 스로틀은 일정 시간 동안 최대 한 번만 실행되도록 제한하는 방식입니다. 디바운스는 입력이 멈춘 뒤 검색 요청을 보내는 자동완성 같은 UI에 적합하고, 스로틀은 스크롤, 리사이즈, 드래그처럼 연속 이벤트를 너무 자주 처리하지 않도록 제어할 때 적합합니다. 즉 디바운스는 “끝나고 한 번”, 스로틀은 “중간중간 제한적으로 실행”에 가깝습니다.

### 상세 해설

#### 디바운스

계속 입력되면 타이머를 계속 초기화한다. searchInput.addEventListener("input", debounce(handleSearch, 300));

- 사용 예:
- 검색창 자동완성
- 폼 validation
- 입력 끝난 뒤 서버 호출
- 스로틀

정해진 간격마다 한 번만 허용한다.

```ts
window.addEventListener("scroll", throttle(handleScroll, 100));
```

- 사용 예:
- 스크롤 위치 계산
- 무한 스크롤 감시
- resize 대응

#### 핵심 구분

- 디바운스: 잠잠해진 뒤 1번
- 스로틀: 진행 중에도 간헐적으로 실행

## 16. 이벤트 버블링과 캡처링을 설명하고, 이벤트 위임이 왜 유용한지 말해보세요.

### 합격 답변 예시

이벤트는 기본적으로 캡처 단계에서 상위에서 하위로 내려오고, 타깃에 도달한 뒤 버블링 단계에서 다시 하위에서 상위로 올라갑니다. 브라우저에서 일반적으로 등록한 이벤트 리스너는 기본적으로 버블링 단계에서 동작합니다. 이벤트 위임은 부모 요소 하나에 리스너를 두고 이벤트가 전파되는 특성을 이용해 자식 요소들의 이벤트를 처리하는 방식입니다. 동적으로 추가되는 요소에도 대응하기 쉽고, 리스너 수를 줄여 관리와 성능 측면에서 유리합니다.

### 가산점 포인트

event.target, currentTarget, closest를 언급하면 실무감이 좋다.

### 상세 해설

#### 흐름

- 캡처링: 위 → 아래
- 타깃
- 버블링: 아래 → 위

#### 위임 예시

```ts
list.addEventListener("click", (event) => {
  const button = event.target.closest("button[data-id]");
  if (!button) return;
  console.log(button.dataset.id);
});
```

#### 왜 좋은가?

- 자식마다 리스너 안 달아도 됨
- 동적 요소 추가에 강함
- 코드 집중 관리 가능
- 주의

모든 이벤트가 버블링하는 건 아니다. 예를 들어 일부 이벤트는 특성이 다를 수 있다. 이 정도까지 언급하면 꽤 좋다.

## 17. 브라우저 렌더링 과정에서 JavaScript가 어떤 식으로 영향을 주는지 설명해보세요.

## 19. Map과 Object, Set과 Array의 차이를 설명하고, 어떤 기준으로 선택하는지 말해보세요.

### 합격 답변 예시

Object는 기본적으로 문자열이나 심볼 키를 가진 일반 객체 표현에 적합하고, Map은 키 타입에 제한이 없고 삽입 순서가 보장되며 크기 확인과 추가/삭제가 명확해서 키-값 저장소 용도에 더 적합합니다. Array는 순서가 있는 목록을 다룰 때 적합하고 중복을 허용하지만, Set은 중복 없는 값 집합을 다룰 때 적합합니다. 그래서 데이터가 “일반적인 엔티티 객체”인지, “사전/캐시 같은 컬렉션”인지, “중복 없는 집합”인지에 따라 선택하는 게 좋습니다. 실무에서는 키-값 저장소는 Map, 고유값 관리나 중복 제거는 Set, 순차 데이터 렌더링은 Array, 도메인 모델 표현은 Object를 많이 사용합니다.

### 가산점 포인트

Object.create(null)이나 hasOwn를 언급하면 더 좋지만 필수는 아니다.

### 상세 해설

- Object
- 일반 객체 모델링에 자연스러움
- JSON 직렬화 친화적
- 키는 문자열/심볼 중심
- Map
- 모든 타입을 키로 사용 가능
- size, clear, has, get, set API 명확
- 삽입 순서 유지

```ts
const map = new Map();
map.set({}, "value");
```

- Array
- 순서 있음
- 중복 허용
- map, filter, reduce 등 풍부한 순회 메서드
- Set
- 중복 제거에 강함

```ts
const unique = [...new Set([1, 1, 2, 3])];
```

#### 선택 기준

- 키-값 저장소인가? → Map
- 일반 도메인 객체인가? → Object
- 순서 있는 목록인가? → Array
- 중복 없는 집합인가? → Set

### 왜 보완이 필요한가

기존 20문항은 실행 컨텍스트, 클로저, 비동기, 브라우저 이벤트 흐름을 폭넓게 다루고 있어서 기본 체력 점검용으로는 충분히 강하다. 다만 면접에서 실제 변별력이 더 크게 나는 지점은 `this`, 호이스팅, 컬렉션 순회, 모듈 시스템처럼 “자주 틀리지만 대충 외우면 설명이 안 되는 언어 디테일”이다.

특히 3년차 이상을 가르려면 “문법을 써봤다”보다 “언어가 왜 그렇게 동작하는지 안다”를 드러내는 질문이 더 필요하다. 아래 질문을 보강하면 단순 암기형 답변과 런타임 모델을 이해한 답변을 더 선명하게 나눌 수 있다.

### 추가하면 좋은 질문

#### 1. TDZ(Temporal Dead Zone)가 무엇인지 설명해보세요. `let`/`const` 호이스팅과 어떻게 연결되나요?

이 질문은 `let`, `const`가 “호이스팅되지 않는다”라고 잘못 외운 답변을 바로 걸러낸다. 정확한 답변은 “호이스팅은 되지만, 초기화 전 접근이 막혀 있어서 TDZ 구간에서 에러가 난다”여야 한다.

#### 2. `call`, `apply`, `bind`의 차이를 설명하고 언제 쓰는지 말해보세요.

기존 `this` 질문만으로는 “화살표 함수는 상위 `this`를 쓴다” 수준에서 끝날 수 있다. 이 질문을 추가하면 호출 시점 `this` 바인딩을 정말 이해하는지, 그리고 함수 재사용이나 핸들러 고정 같은 실무 사례까지 설명할 수 있는지 확인하기 좋다.

#### 3. `null`, `undefined`, `NaN`의 차이를 설명하고, `typeof`의 함정을 말해보세요.

실무에서 자주 사고 나는 디테일이다. `typeof null === "object"`, `NaN !== NaN`, “프로퍼티가 없음”과 `undefined` 값의 차이를 설명할 수 있으면 자바스크립트의 예외 케이스를 정확히 다뤄본 사람이라는 인상을 준다.

#### 4. `for...in`과 `for...of`의 차이, 그리고 iterable이 무엇인지 설명해보세요.

이 질문은 배열, 객체, `Map`, `Set`을 얼마나 구분해서 이해하는지 보기 좋다. 순회 대상이 “키인지 값인지”, 그리고 어떤 자료구조가 iterable 프로토콜을 따르는지 설명할 수 있으면 컬렉션 감각이 드러난다.

#### 5. ES Modules와 CommonJS의 차이를 설명해보세요.

프론트엔드 3년차 이상이라면 번들러, 트리 쉐이킹, 정적 분석과 연결되는 모듈 시스템 감각이 필요하다. 단순 문법 차이보다 `import`/`export`의 정적 구조와 `require`의 런타임 특성을 구분해서 설명할 수 있으면 좋다.

#### 6. `WeakMap` / `WeakSet`은 언제 쓰고, `Map` / `Set`과 어떤 차이가 있나요?

필수 기본기까지는 아니지만 상단권 변별력이 높은 질문이다. 키 참조 방식과 GC, 메모리 누수 방지 같은 주제를 함께 묶어 설명할 수 있어서 “자료구조 + 메모리 모델” 이해도를 같이 볼 수 있다.

### 이 섹션 보완 핵심

JS 섹션은 이미 강하다. 여기에 TDZ, `this` 심화, 값 비교 예외, iterable, 모듈 시스템, 약한 참조 컬렉션까지 더하면 문법형 답변자와 언어 동작을 이해한 사람을 훨씬 더 선명하게 가를 수 있다.
