# 10. 프로젝트 맞춤 20문항

> 프로젝트 선택 이유, 운영 경험, 문제 정의 능력을 드러내는 맞춤형 20문항입니다.

### 전체 흐름 맵

```
자기소개 (정체성 선언)
  → 회사 역할과 기여 (경험 범위)
    → 프로젝트 선택 이유 (기술 판단)
      → 아키텍처 설계 경험 (구조적 사고)
        → 상태 관리 전략 (상태 설계)
          → 성능 최적화 경험 (측정→개선)
            → 네트워크/장애 대응 (운영 감각)
              → 테스트 전략 (품질 기준)
                → 배포/인프라 경험 (운영 체계)
                  → 팀 협업 (소통 방식)
                    → 성장 방향 (학습 태도)
```

## 10. 프로젝트 관련 질문 20문항

- (이력서/포트폴리오 기반 맞춤형)
- 여기서는 네가 앞서 준 정보와 대화 맥락을 바탕으로, 면접에서 설득력 있게 말할 수 있는 방향으로 정리하겠다. 핵심은 이거다. 네 포트폴리오의 강점은 “기술이 많다”가 아니라 기술 선택이 문제 정의와 운영 가능성으로 연결된다는 점이다. 즉 프로젝트 답변은 항상 아래 구조로 가면 강하다.

- 문제 상황
- 왜 그게 문제였는지
- 어떤 기준으로 설계/선택했는지
- 결과와 배운 점

## 1. 본인을 1분 자기소개로 말한다면 어떻게 소개하겠습니까?

### 합격 답변 예시

저는 프론트엔드 개발자이지만, 단순히 화면을 만드는 것보다 사용자 흐름이 실제로 끊기지 않게 만드는 데 더 관심이 많습니다. 실무에서는 React, Next.js, TypeScript 기반으로 다양한 서비스와 어드민을 개발했고, 상태 구조 설계, 복잡한 UI 상호작용, 네트워크 불안정 대응, 배포와 운영 자동화까지 함께 다뤄왔습니다. 특히 1인 개발 프로젝트에서는 프론트엔드뿐 아니라 백엔드, 배포, 운영까지 맡으면서 기술 선택을 항상 사용자 경험과 운영 가능성 기준으로 판단해왔고, 이런 점이 제 강점이라고 생각합니다.

### 상세 해설

자기소개는 기술 나열이 아니라 정체성 선언이다.

- 좋은 구조는:
- 나는 어떤 문제를 잘 푸는 사람인가
- 어떤 기술 환경을 다뤄왔는가
- 어떤 강점이 있는가

## 2. 아이스크림아트에서 맡았던 역할과 기여 범위를 설명해보세요.

### 합격 답변 예시

아이스크림아트에서는 단순 퍼블리싱 역할에 머물지 않고, 실제 서비스 기능 개발과 구조 개선, 운영 이슈 대응까지 점점 기여 범위를 넓혀갔습니다. 사용자 인터랙션이 많은 서비스, 내부 운영 툴, 관리자 기능, 상태 구조 재설계, 네트워크와 세션 문제 대응 등 서로 다른 유형의 문제를 경험했고, 그 과정에서 단순 구현보다 ‘왜 이 구조가 유지보수에 유리한가’를 많이 고민했습니다. 즉 한 프로젝트만 깊게 한 게 아니라, 여러 도메인에서 문제 유형을 다뤄본 경험이 강점이라고 설명하면 좋습니다.

### 상세 해설

이 질문은 “무슨 일을 했냐”보다 직무 범위가 얼마나 확장됐는지를 보여주는 게 중요하다.

## 3. 여러 프로젝트 중 면접에서 가장 먼저 강조하고 싶은 프로젝트는 무엇이고, 왜인가요?

### 합격 답변 예시

저는 양치킹을 가장 먼저 강조할 것 같습니다. 이유는 이 프로젝트가 1인 개발로 프론트엔드, 백엔드, 배포, 운영까지 전 과정을 맡았고, 실제 교육 현장에서 네트워크 불안정과 사용성 문제를 직접 다뤘기 때문입니다. 즉 구현력뿐 아니라 문제 정의, 안정성 설계, 운영 판단까지 한 번에 보여줄 수 있는 프로젝트라서 제 강점을 가장 잘 설명할 수 있습니다.

### 상세 해설

이 질문은 정답이 없다. 하지만 **“왜 이 프로젝트가 내 강점을 가장 잘 보여주는가”**가 중요하다.

- 양치킹은 특히
- 1인 개발
- 실제 운영
- 불안정 네트워크 대응
- CI/CD
- 사용자 흐름 보호

를 함께 말할 수 있어서 강하다.

## 4. 양치킹에서 1인 개발·배포·운영을 했다고 했는데, 가장 어려웠던 의사결정은 무엇이었나요?

### 합격 답변 예시

가장 어려웠던 의사결정은 기술적으로 더 멋진 구조를 고를지, 실제 학교 현장에서 안정적으로 버티는 구조를 고를지였습니다. 교육 현장에서는 네트워크가 불안정하고 수업 시간 안에 입력이 끊기면 바로 사용자 불만으로 이어지기 때문에, 저는 이상적인 구조보다 fallback UI, 재시도, 임시 저장, 배포 자동화처럼 운영 가능한 구조를 우선했습니다. 즉 이 프로젝트에서는 최신 기술보다도 ‘사용자가 실제로 문제 없이 쓸 수 있느냐’가 더 중요한 기준이었다고 답하면 좋습니다.

### 상세 해설

이 답변은 너의 철학과 아주 잘 맞는다. 면접관 입장에서는 이런 답변을 들으면 “기술 과시보다 제품과 운영을 보는 사람”이라는 인상을 받는다.

## 5. 양치킹에서 학교 Wi-Fi 환경을 고려한 fallback UI와 재시도 로직은 구체적으로 어떻게 설계했나요?

### 합격 답변 예시

핵심은 네트워크 상태를 단순히 online/offline으로 보지 않고, 실제 요청 실패와 응답 지연을 기준으로 사용자 경험을 설계한 점입니다. 즉 요청이 실패했다고 바로 작업을 잃게 두지 않고, 재시도 흐름과 fallback UI를 두고, 중요한 입력은 임시 저장이나 낙관적 반영을 통해 사용자가 흐름을 이어갈 수 있게 했습니다. 사용자 입장에서는 ‘왜 안 되는지’보다 ‘지금 뭘 하면 되는지’가 중요해서, 에러 메시지보다는 복구 경로를 먼저 보여주는 식으로 설계했다고 답하면 좋습니다.

### 상세 해설

이 질문에서 중요한 건 기술보다 UX(User Experience)다.

- 좋은 구조:
  - 네트워크 불안정 감지
  - 실패 유형 구분
  - 재시도
  - 임시 저장
  - 복구 가능한 메시지
- 즉 “에러를 잘 보여줬다”보다 에러가 있어도 사용자가 일을 끝낼 수 있게 했다가 핵심이다.

## 6. 양치킹의 일 평균 1,000건 이상 기록 생성 규모에서 장애 대응은 어떤 방식으로 했나요?

### 합격 답변 예시

이 정도 규모에서는 장애를 완전히 없애는 것보다, 장애가 났을 때 기록 손실을 줄이고 빠르게 복구하는 구조가 중요하다고 봤습니다. 운영 측면에서는 배포 자동화와 프로세스 관리로 재시작과 복구를 빠르게 만들고, 사용자 측면에서는 네트워크 실패나 세션 문제로 기록이 증발하지 않도록 재시도와 임시 저장을 포함한 UX를 설계했습니다. 즉 장애 대응을 서버 운영과 사용자 경험 두 층으로 나눠서 접근했다고 답하면 좋습니다.

### 상세 해설

이 질문은 운영 관점의 균형감을 본다.

- 좋은 답변은
- 인프라 대응
- UX 대응

둘 다 들어가야 한다.

## 7. Tool-Manager에서 왜 기존 Vue 구조를 React로 전면 재설계해야 한다고 판단했나요?

### 합격 답변 예시

기존 구조가 단순 기능 추가로는 더 이상 유지보수가 어렵다고 판단했기 때문입니다. 특히 설정 데이터 구조가 복잡하고, 순서 변경과 편집 UX가 점점 중요해지면서 기존 구조에 기능을 덧붙이는 것보다 상태 흐름과 컴포넌트 책임을 다시 설계하는 편이 장기적으로 낫다고 봤습니다. 그래서 단순 마이그레이션이 아니라, 데이터 구조와 UI 상호작용 모델을 React 기준으로 다시 정의하는 방향을 택했다고 설명하면 좋습니다.

### 상세 해설

이 질문의 핵심은 “React가 좋아서 바꿨다”가 아니다.

- 좋은 답변은
- 기존 구조의 한계
- 유지보수 비용 증가
- 상태 동기화 문제
- 구조 재설계 필요성

을 말해야 한다.

## 8. Tool-Manager의 Config 구조를 Zustand와 Zod로 풀었다고 했는데, 그 구조를 구체적으로 설명해보세요.

### 합격 답변 예시

편집 중인 Config 상태는 사용자가 실시간으로 여러 부분을 바꾸기 때문에 전역적으로 관리할 필요가 있었고, 그 역할을 Zustand가 맡았습니다. 반면 저장되거나 서버와 주고받는 데이터는 구조가 틀리면 바로 장애로 이어질 수 있어서, Zod를 경계 검증용으로 두어 데이터 정합성을 확인했습니다. 즉 Zustand는 편집 경험을 위한 상태 저장소, Zod는 저장/전송 시점의 계약 검증기로 역할을 분리했다고 설명하면 좋습니다.

### 상세 해설

이 답변은 굉장히 좋다. 왜냐하면 TS, 상태관리, 런타임 검증을 한 번에 연결하기 때문이다.

- 핵심 구조
- 편집 중 상태 → Zustand
- 저장/전송 경계 → Zod
- 실시간 UX와 데이터 정합성 분리
- 이렇게 말하면

“툴만 썼다”가 아니라 역할을 분리해서 설계했다가 된다.

## 9. Tool-Manager의 드래그 기반 편집 UX에서 가장 까다로웠던 상태 동기화 문제는 무엇이었나요?

### 합격 답변 예시

가장 까다로웠던 건 드래그 중 임시 위치 변화와 실제 저장되는 순서를 같은 상태로 다루지 않는 것이었습니다. 드래그 중에는 hover와 preview 같은 일시적 상태가 계속 바뀌지만, 저장 시점에는 확정된 순서만 의미가 있기 때문에 이 둘을 섞으면 상태가 쉽게 꼬입니다. 그래서 상호작용 중 상태와 영속 상태를 분리하고, stable key와 공통 DnD 컴포넌트 구조를 통해 흐름을 정리했다고 답하면 좋습니다.

### 상세 해설

이 질문은 DnD(Drag and Drop) UI를 제대로 해본 사람인지 잘 드러난다.

#### 핵심 포인트

- transient state vs persistent state
- drag preview vs final order
- key 안정성
- store 업데이트 타이밍

이걸 말할 수 있으면 강하다.

## 10. 아트봉봉 스쿨에서 VOC(Voice of Customer)를 유형화했다고 했는데, 실제로 어떤 기준으로 묶고 우선순위를 정했나요?

### 합격 답변 예시

단순히 문의 개수만 세기보다, 사용자가 어떤 흐름에서 반복적으로 막히는지를 기준으로 묶는 방식이 더 효과적이었습니다. 예를 들어 특정 기능 하나의 버그보다도, 이용권 관리나 수업 흐름처럼 반복적으로 혼란이 생기는 구간은 운영 비용과 사용자 피로도를 함께 높이기 때문에 우선순위를 높게 봤습니다. 즉 개별 증상보다 사용자 흐름 단위로 문제를 묶고, 반복성과 영향도를 기준으로 우선순위를 정했다고 설명하면 좋습니다.

### 상세 해설

- 이 답변의 포인트는
- “VOC를 처리했다”가 아니라

운영 데이터를 구조적으로 읽었다는 점이다.

## 11. 아트봉봉 스쿨에서 9월 30건 수준이던 VOC가 12월 6건까지 줄었다고 했는데, 가장 효과가 컸던 개선 하나를 설명해보세요.

### 합격 답변 예시

가장 효과가 컸던 개선은 특정 버튼이나 문구 하나를 바꾼 게 아니라, 사용자가 가장 자주 길을 잃는 흐름의 정보 구조를 다시 잡은 것이었습니다. 즉 화면 단위 수정이 아니라 탐색 구조를 바꾸면서, 사용자가 ‘어디서 뭘 해야 하는지’ 이해하기 쉽게 만든 것이 VOC 감소에 더 크게 기여했습니다. 이 경험을 통해 개별 버그 수정도 중요하지만, 반복되는 문의는 구조 문제일 가능성이 크다는 걸 배웠다고 답하면 좋습니다.

### 상세 해설

면접관은 이 질문에서 “문제 해결의 수준”을 본다.

- 증상 해결
- 구조 해결

둘 중 후자를 말하면 훨씬 강하다.

## 12. 세션 만료, iframe 로딩 지연, 네트워크 불안정이 동시에 생기는 상황을 어떻게 분리해서 대응했나요?

### 합격 답변 예시

## 20. 다음 회사에 간다면 지금까지의 경험 중 어떤 역량을 가장 먼저 가져가서 성과로 연결할 수 있다고 생각하나요?

### 합격 답변 예시

제가 가장 먼저 성과로 연결할 수 있는 역량은 문제를 단순 구현 이슈가 아니라 사용자 흐름과 운영 비용 문제로 재정의하는 능력이라고 생각합니다. 지금까지의 경험을 보면 복잡한 UI 상태 설계, 관리자 자동화, 네트워크 불안정 대응, 배포와 운영 자동화처럼 서로 다른 문제를 결국 ‘운영 가능한 서비스’로 연결해왔습니다. 그래서 새로운 회사에서도 단순히 기능을 만드는 것보다, 어디서 사용자 흐름이 끊기고 팀 유지보수 비용이 커지는지 빠르게 파악해서 구조적으로 개선하는 쪽에서 가장 먼저 기여할 수 있다고 답하면 좋습니다.

### 상세 해설

마지막 질문은 결국 **“당신을 왜 뽑아야 하죠?”**의 다른 표현이다.

- 좋은 답변은
- 기술 스택 나열
- 보다
- 내가 가장 빨리 만들 수 있는 가치

를 말해야 한다. 네 경우엔 그게 문제 정의 + 구조 설계 + 운영 가능성이다.

## 보완내용

### 왜 보완이 필요한가

기존 프로젝트 질문은 양치킹, Tool-Manager, Art-BonBon, Ecole-Admin, 모노레포, VS Code Extension까지 잘 묶여 있어서 강점이 분명하다. 다만 프로젝트 면접은 결국 “무엇을 만들었는가”보다 “어떻게 판단했고, 무엇을 배웠고, 다음엔 무엇을 바꾸겠는가”를 보는 자리이기도 하다.

아래 질문을 추가하면 성과 자랑형 답변을 넘어서 회고, 실패 대응, 협업, 새 조직 적응력까지 더 설득력 있게 보여줄 수 있다.

### 추가하면 좋은 질문

#### 1. 프로젝트에서 했던 기술 선택 중, 지금 다시 한다면 바꾸고 싶은 결정 하나는 무엇인가요?

이 질문은 회고 능력을 아주 잘 보여준다. 성공 사례만 말하는 사람보다 판단을 다시 점검할 줄 아는 사람이 더 신뢰를 준다.

##### 합격 답변 예시

하나를 꼽는다면 초기 프로젝트에서 상태 관리 라이브러리를 도입할 때 팀 규모나 프로젝트 복잡도를 충분히 고려하지 않고 선택했던 경험입니다. 당시에는 커뮤니티에서 많이 쓰는 도구라는 이유만으로 도입했는데, 실제로 프로젝트 규모에 비해 보일러플레이트가 과했고 오히려 유지보수 비용을 높였습니다. 지금 다시 한다면 프로젝트의 상태 복잡도와 팀 숙련도를 먼저 판단하고, 정말 필요한 수준의 도구를 선택할 것입니다. 이 경험 이후로 기술 선택 시 “왜 이게 이 맥락에서 최선인가”를 항상 먼저 따지게 됐습니다.

##### 상세 해설

이 질문의 핵심은 “후회”가 아니라 판단 기준의 성장을 보여주는 것이다.

- 좋은 답변 구조:
  - 당시 어떤 기준으로 선택했는지
  - 실제로 어떤 문제가 생겼는지
  - 지금이라면 어떤 기준으로 바꿀지
  - 그 경험이 이후 판단에 어떻게 영향을 줬는지
- 주의할 점:
  - “그때는 몰랐다”로 끝내면 안 된다
  - 반드시 “지금은 이런 기준으로 본다”로 마무리해야 한다
  - 기술 자체의 좋고 나쁨이 아니라, 맥락과 기준의 변화를 보여줘야 한다

#### 2. 가장 크게 실패했거나 되돌렸던 결정 하나를 설명해보세요.

기술 면접에서 매우 좋은 질문이다. 실패를 숨기지 않고, 원인과 복구 과정을 구조적으로 설명할 수 있는지 볼 수 있다.

##### 합격 답변 예시

운영 중인 서비스에서 기능 개선을 배포한 뒤, 예상하지 못한 엣지 케이스로 일부 사용자의 데이터 흐름이 끊긴 적이 있습니다. 원인은 기존 사용자의 데이터 상태를 충분히 고려하지 않고 새로운 구조를 적용한 것이었습니다. 발견 즉시 롤백하고, 기존 데이터와 호환되는 마이그레이션 경로를 설계한 뒤 다시 배포했습니다. 이 경험 이후로는 기능을 바꿀 때 “기존 사용자의 상태”를 반드시 체크리스트에 넣게 됐고, 배포 전 롤백 시나리오를 미리 준비하는 습관이 생겼습니다.

##### 상세 해설

실패 질문에서 면접관이 보는 건 세 가지다.

- 실패를 인정하는 태도: 남 탓이나 환경 탓이 아니라 본인의 판단 한계를 솔직하게 말할 수 있는지
- 복구 과정의 구조: 당황해서 임시 방편을 쓴 게 아니라, 원인 파악 → 롤백 → 재설계 → 재배포 흐름을 밟았는지
- 이후 변화: 같은 실수를 반복하지 않기 위해 프로세스나 습관이 어떻게 바뀌었는지
- 피해야 할 답변:
  - “크게 실패한 적은 없다” → 경험 부족이나 회고 부재로 보인다
  - 실패를 말하면서 결국 자랑으로 끝나는 답변 → 진정성이 떨어진다

#### 3. 본인의 성과를 어떻게 측정했나요? 숫자로 설명할 수 있나요?

기존 챕터에도 결과 수치가 일부 들어가 있으니, “내가 한 일”과 “실제 영향”을 더 분명하게 연결하기 좋다.

##### 합격 답변 예시

정량적으로 가장 명확했던 건 아트봉봉 스쿨에서 VOC를 월 30건에서 6건으로 줄인 사례입니다. 이 수치는 운영팀에서 집계하던 문의 데이터를 기반으로 확인했고, 단순히 기능을 추가해서가 아니라 사용자 흐름의 정보 구조를 재설계하면서 반복 문의 자체를 줄인 결과였습니다. 양치킹에서는 일 평균 1,000건 이상의 기록이 안정적으로 생성되는 것 자체가 시스템 안정성의 지표였습니다. 모든 프로젝트에서 공식적인 KPI(Key Performance Indicator)를 부여받진 않았지만, 운영 문의 감소, 장애 빈도, 배포 안정성 같은 간접 지표를 기준으로 성과를 확인했습니다.

##### 상세 해설

이 질문의 핵심은 “큰 숫자”가 아니라 측정하려는 태도와 연결 논리다.

- 좋은 답변 구조:
  - 어떤 지표를 봤는지
  - 그 지표가 왜 의미 있었는지
  - before/after를 어떻게 확인했는지
- 숫자가 없는 경우에도:
  - “공식 지표는 없었지만, 이런 간접 지표로 확인했다”고 말하면 충분하다
  - 측정 불가능했던 것을 솔직히 말하되, 측정하려 노력한 태도를 보여주면 된다
- 피해야 할 답변:
  - 막연한 “성능이 좋아졌다”, “사용자가 편해졌다” → 근거 없는 주장으로 보인다

#### 4. 협업 과정에서 의견 충돌이 있었던 사례와, 어떻게 정렬했는지 설명해보세요.

현재 포트폴리오는 기술과 운영 강점이 잘 드러나지만, 팀 협업 역량을 직접적으로 보여주려면 이런 질문이 유용하다.

##### 합격 답변 예시

기존 서비스의 구조를 개선하는 과정에서, 기획 쪽에서는 기존 흐름을 최대한 유지하면서 기능만 추가하자는 입장이었고, 저는 현재 구조로는 기능을 덧붙일수록 유지보수 비용이 급격히 올라간다고 판단한 적이 있습니다. 이때 단순히 “구조가 나쁘다”고 주장하기보다, 현재 구조에서 기능을 추가했을 때 예상되는 구체적인 문제점과 소요 시간을 정리해서 공유했습니다. 동시에 구조를 바꿨을 때의 비용과 장기적 이점도 함께 비교해서, 결국 핵심 흐름만 먼저 재설계하고 나머지는 점진적으로 개선하는 방향으로 합의했습니다. 이 경험에서 배운 건 “내가 맞다”를 증명하는 게 아니라, 서로의 우려를 구체적 근거로 정리하면 합의점이 보인다는 것이었습니다.

##### 상세 해설

협업 충돌 질문에서 면접관이 보는 건 결론이 아니라 과정이다.

- 좋은 답변 구조:
  - 어떤 상황에서 어떤 입장 차이가 있었는지
  - 상대의 관점을 이해하려 했는지
  - 어떤 근거와 방식으로 조율했는지
  - 결과적으로 어떤 합의에 이르렀는지
- 핵심 포인트:
  - “내가 이겨서 결과가 좋았다”가 아니라 “서로 다른 관점을 조율해서 더 나은 방향을 찾았다”가 되어야 한다
  - 혼자 결정한 게 아니라 팀이 납득할 수 있는 근거를 만들었다는 점이 중요하다
- 피해야 할 답변:
  - “충돌이 없었다” → 협업 경험 부족으로 보인다
  - 상대를 비난하는 톤 → 팀워크 역량에 의문이 생긴다

#### 5. 입사 후 90일 안에 가장 먼저 파악할 것과, 빠르게 기여할 수 있는 영역은 무엇이라고 생각하나요?

회사 적합성 질문으로 아주 좋다. 현재 경험을 새 조직의 맥락에 어떻게 옮길지 보여줄 수 있다.

##### 합격 답변 예시

처음 30일은 기존 코드베이스의 구조와 배포 흐름, 팀의 작업 방식을 파악하는 데 집중할 것입니다. 어떤 기술을 쓰는지보다 왜 그런 구조를 선택했는지, 현재 가장 자주 발생하는 운영 이슈는 무엇인지를 이해하는 게 우선입니다. 그다음 30일은 작은 개선을 통해 팀에 기여하면서 신뢰를 쌓겠습니다. MVP(Minimum Viable Product) 수준이라도 버그 수정, 코드 리뷰 참여, 기존 흐름의 작은 개선처럼 임팩트는 작더라도 팀이 체감할 수 있는 기여를 하겠습니다. 마지막 30일에는 앞서 파악한 내용을 바탕으로 사용자 흐름이 끊기는 구간이나 운영 비용이 큰 영역에 구조적 개선을 제안하겠습니다. 지금까지의 경험에서 가장 빠르게 기여할 수 있는 영역은 복잡한 UI 상태 정리, 운영 이슈 구조화, 사용자 흐름 개선이라고 생각합니다.

##### 상세 해설

이 질문은 겸손함과 자신감의 균형이 핵심이다.

- 좋은 답변 구조:
  - 1단계: 파악과 학습 (기존 맥락 존중)
  - 2단계: 작은 기여로 신뢰 확보
  - 3단계: 본격적 제안과 개선
- 핵심 포인트:
  - “들어가자마자 바꾸겠다”는 오만해 보인다
  - “시키는 대로 하겠다”는 주도성이 없어 보인다
  - 기존 팀의 맥락을 존중하면서도, 본인의 강점이 어디서 빛날 수 있는지를 구체적으로 말해야 한다
- 피해야 할 답변:
  - 지원 회사를 전혀 모르는 상태에서 일반론만 말하는 것 → 사전 조사를 했다는 인상을 주면 더 좋다

#### 6. 대표 프로젝트 하나를 STAR(Situation, Task, Action, Result) 구조로 2분 안에 설명해보세요.

질문이라기보다 연습 포맷에 가깝지만 매우 중요하다. 프로젝트 답변이 길어지거나 산만해지는 걸 막아준다.

##### 합격 답변 예시

양치킹을 예로 들겠습니다. **Situation**: 초등학교 양치 교육용 앱을 1인 개발로 맡았는데, 학교 Wi-Fi 환경이 불안정해서 기존 방식대로 만들면 수업 중 기록이 유실되는 문제가 예상됐습니다. **Task**: 네트워크가 불안정한 환경에서도 하루 1,000건 이상의 기록이 안전하게 저장되고, 수업 흐름이 끊기지 않는 서비스를 만드는 것이 목표였습니다. **Action**: fallback UI와 재시도 로직을 설계해서 요청 실패 시에도 사용자가 작업을 이어갈 수 있게 했고, 임시 저장으로 데이터 유실을 방지했습니다. 배포는 CI/CD와 프로세스 관리 자동화로 장애 복구 시간을 최소화했습니다. **Result**: 실제 학교 현장에서 안정적으로 운영됐고, 네트워크 문제로 인한 기록 유실 없이 일 평균 1,000건 이상의 데이터가 꾸준히 생성됐습니다.

##### 상세 해설

STAR 구조는 면접 답변의 기본 프레임이다. 모든 프로젝트 질문에 적용할 수 있다.

- 각 단계별 핵심:
  - **S (Situation)**: 배경과 제약 조건을 간결하게. 30초 이내.
  - **T (Task)**: 내가 해결해야 했던 구체적 목표. 한 문장이면 충분.
  - **A (Action)**: 실제로 한 일. 기술과 판단의 근거를 함께. 가장 길어도 된다.
  - **R (Result)**: 정량적이든 정성적이든 결과와 배운 점. 숫자가 있으면 더 좋다.
- 연습 팁:
  - 타이머를 켜고 2분 안에 끝내는 연습을 하면 좋다
  - 녹음해서 들어보면 불필요한 반복이나 산만한 부분이 바로 보인다
  - 프로젝트별로 STAR를 미리 정리해두면 어떤 질문이 와도 변형해서 쓸 수 있다

### 이 섹션 보완 핵심

프로젝트 섹션은 이미 탄탄하다. 여기에 실패, 회고, 정량 성과, 협업, 90일 기여 계획, STAR 구조까지 더하면 기술 소개를 넘어서 판단력과 성장 가능성을 더 분명하게 보여줄 수 있다.
