# 챕터 7. 브라우저 요청 흐름에서 인증과 캐시, CDN까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **브라우저 요청은 단순히 fetch 한 번이 아니라, DNS 조회, 연결 수립, 보안 계층, HTTP 메시지, 인증, 캐시, CDN까지 이어지는 하나의 흐름으로 이해해야 한다.**

### 전체 흐름 맵

```
URL 입력
  → **DNS(Domain Name System, 도메인 이름 시스템 — 도메인을 IP 주소로 변환)** 조회 (도메인 → IP)
    → **TCP(Transmission Control Protocol, 전송 제어 프로토콜 — 신뢰성 있는 연결 기반 통신)** 연결 (3-way handshake)
      → **TLS(Transport Layer Security, 전송 계층 보안 — 통신 암호화 프로토콜)** handshake (HTTPS 보안)
        → HTTP 요청/응답 (메서드, 헤더, 상태코드)
          → 인증 (쿠키, 세션, 토큰)
            → CORS / Preflight (브라우저 보안 정책)
              → HTTP 캐시 (강한 캐시 + 조건부 요청)
                → **CDN(Content Delivery Network, 콘텐츠 전송 네트워크 — 전 세계 엣지 서버에 캐시하여 빠르게 전달)** (edge 캐시, 지리적 분산)
                  → 실시간 통신 (WebSocket, SSE, Polling)
```

## 60. 브라우저 요청의 전체 흐름 — DNS에서 렌더링까지

> **핵심 요약** — URL 입력부터 화면 표시까지: DNS 조회 → TCP 연결(1 **RTT(Round-Trip Time, 왕복 지연 시간)**) → TLS handshake(1-2 RTT) → HTTP 요청/응답 → 하위 리소스 로딩 → 렌더링. 첫 방문이 느린 이유는 이 단계들이 누적되기 때문.

### 큰 흐름

```
1. URL 해석 (프로토콜, 호스트, 포트, 경로 분리)
2. DNS 조회 (도메인 → IP 주소)
     브라우저 캐시 → OS 캐시 → DNS Resolver → Root → **TLD(Top-Level Domain, 최상위 도메인 — .com, .kr 같은 도메인 최상위 부분)** → Authoritative
3. TCP 연결 수립 (SYN → SYN-ACK → ACK, 최소 1 RTT)
4. TLS handshake (HTTPS일 때, TLS 1.3은 1 RTT)
5. HTTP 요청 전송
6. 서버 응답 수신
7. HTML 파싱 + 추가 리소스 요청 (CSS, JS, 이미지, 폰트)
8. 렌더 트리 구성 → 레이아웃 → 페인트 → 컴포지트
```

### 브라우저 요청 전체 라이프사이클

```
Client                                          Server
  │                                               │
  │─── DNS 조회 (도메인 → IP) ──────────────────→│
  │                                               │
  │─── TCP 3-way handshake (1 RTT) ─────────────→│
  │     SYN →                                     │
  │          ← SYN-ACK                            │
  │     ACK →                                     │
  │                                               │
  │─── TLS handshake (1 RTT, TLS 1.3) ─────────→│
  │     ClientHello →                             │
  │          ← ServerHello + 인증서               │
  │     검증 + 키 교환 →                          │
  │                                               │
  │─── HTTP 요청 ───────────────────────────────→│
  │     GET /page HTTP/1.1                        │
  │          ← HTTP 응답 (HTML)                   │
  │                                               │
  │     추가 리소스 요청 (CSS, JS, 이미지)        │
  │          ← 리소스 응답                        │
  │                                               │
  [HTML 파싱 → DOM 생성 → CSSOM → 렌더 트리 → 화면 표시]

첫 방문이 느린 이유: 위 단계들이 직렬로 누적되기 때문
```

### DevTools Network 탭에서 보이는 타이밍

```
Timing breakdown (DevTools → Network → 요청 선택 → Timing):

┌─────────────┬──────────────────────────────────────────┐
│ Queueing    │ 브라우저 내부 큐 대기                     │
│ DNS Lookup  │ DNS 조회 (캐시 히트면 0ms)               │
│ TCP         │ TCP 3-way handshake                      │
│ **SSL(Secure Sockets Layer, 보안 소켓 계층 — TLS의 이전 이름)**/TLS     │ TLS handshake (HTTPS)                    │
│ Request     │ 요청 전송                                │
│ Waiting     │ 서버 처리 시간 (TTFB에 해당)             │
│ Download    │ 응답 다운로드                             │
└─────────────┴──────────────────────────────────────────┘
```

"API가 느리다"고 할 때, 실제로는 DNS/TCP/TLS/서버처리 중 어디가 느린지를 분해해서 봐야 한다.

### DNS 프리페치 — 초기 로딩 개선

```html
<link rel="dns-prefetch" href="https://fonts.googleapis.com" />
<link rel="dns-prefetch" href="https://cdn.example.com" />
<link rel="preconnect" href="https://api.example.com" />
<!-- preconnect는 DNS + TCP + TLS까지 미리 수행 -->
```

### HTTP 버전별 차이

| 버전 | 핵심 특징 |
|---|---|
| HTTP/1.1 | keep-alive 기본, 한 연결당 한 요청씩 순서대로 |
| HTTP/2 | 멀티플렉싱(한 연결에서 동시 여러 요청), 헤더 압축(**HPACK(HTTP/2 헤더 압축 알고리즘)**) |
| HTTP/3 | QUIC(**UDP(User Datagram Protocol, 비연결형 전송 프로토콜)** 기반), 연결 수립 더 빠름, Head-of-Line blocking 감소 |

### HTTP/1.1 vs HTTP/2 vs HTTP/3 — 핵심 차이

| 특성 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| 전송 프로토콜 | TCP | TCP | QUIC (UDP 기반) |
| 멀티플렉싱 | ❌ (파이프라이닝은 HOL 블로킹) | ✅ 하나의 TCP 연결에서 여러 스트림 | ✅ 스트림별 독립 (HOL 블로킹 해소) |
| 헤더 압축 | ❌ | HPACK | QPACK |
| 연결 수립 | TCP 1 RTT + TLS 2 RTT | TCP 1 RTT + TLS 1 RTT | 0-1 RTT (QUIC이 전송+암호화 통합) |
| 서버 푸시 | ❌ | ✅ (실무에서 잘 안 씀) | ✅ |

**프론트엔드에서 체감하는 차이:**
- HTTP/1.1: 도메인당 6개 연결 제한 → 도메인 샤딩, 스프라이트 시트 필요
- HTTP/2: 멀티플렉싱으로 연결 1개면 충분 → 번들 스플리팅이 자유로워짐
- HTTP/3: 네트워크 전환(Wi-Fi→LTE)에도 연결 유지(Connection Migration)

> HTTP/3은 2024년 기준 주요 CDN(Cloudflare, AWS CloudFront)과 브라우저(Chrome, Firefox, Safari)에서 지원한다.

### TLS — HTTPS의 보안 계층

| | HTTP | HTTPS |
|---|---|---|
| **보안** | 평문 전송 | TLS로 암호화 |
| **포트** | 80 | 443 |
| **인증서** | 없음 | 서버 인증서로 신원 확인 |
| **필수 여부** | - | Service Worker, Geolocation, Clipboard 등 민감 API는 HTTPS 필수 |

TLS 1.3은 1-RTT handshake로 성능이 개선됐고, 재연결 시 0-RTT도 가능.

### 실무 에피소드 — 이런 상황에서 만난다

> 해외 사용자가 "사이트가 너무 느려요"라고 신고. DevTools Timing을 보니 DNS 80ms + TCP 120ms + TLS 150ms + 서버 응답 200ms = 첫 바이트까지 550ms. CDN을 도입하고 `preconnect`를 추가하니 DNS/TCP/TLS 비용이 대폭 줄어 TTFB가 200ms로 개선.

### 자주 하는 오해

- 브라우저 요청은 서버에 바로 간다 → **DNS, TCP, TLS 단계가 있다.**
- HTTPS면 무조건 안전하다 → **XSS(Cross-Site Scripting, 교차 사이트 스크립팅 — 악성 스크립트 주입 공격), CSRF(Cross-Site Request Forgery, 교차 사이트 요청 위조 — 사용자 모르게 인증된 요청을 보내는 공격) 등 다른 취약점은 별도.**
- HTTP/2면 모든 게 빠르다 → **서버 응답 자체가 느리면 여전히 느리다.**

### 면접 압축 답변

- **10초**: 브라우저 요청은 DNS → TCP 연결 → TLS → HTTP 요청/응답 → 렌더링까지 이어지며, 첫 방문은 이 단계들이 누적되어 느립니다.
- **30초**: URL 입력 시 DNS로 IP를 찾고, TCP 3-way handshake로 연결을 수립하고, HTTPS면 TLS handshake를 거쳐 HTTP 요청을 보냅니다. DevTools Timing 탭에서 각 단계의 소요 시간을 분해해 볼 수 있으며, `dns-prefetch`/`preconnect`로 초기 지연을 줄이고 HTTP/2 멀티플렉싱으로 동시 요청 성능을 개선합니다.

## 61. HTTP 메서드, 상태코드, 멱등성

> **핵심 요약** — 메서드는 요청 의도를, 멱등성은 재시도 안전성을, 상태코드는 UI 복구 전략을 결정한다. `fetch`는 4xx/5xx에서 throw하지 않으므로 `response.ok`를 직접 확인해야 한다.

### HTTP 메서드와 멱등성

| 메서드 | 의도 | 멱등 | 안전 | 재시도 |
|---|---|---|---|---|
| `GET` | 조회 | O | O | 안전 |
| `POST` | 생성/처리 | X | X | 위험 (멱등성 키 필요) |
| `PUT` | 전체 교체 | O | X | 안전 |
| `PATCH` | 부분 수정 | 구현에 따라 | X | 주의 |
| `DELETE` | 삭제 | O | X | 안전 |
| `OPTIONS` | 사전 확인 | O | O | - |

**멱등성** = 같은 요청을 여러 번 보내도 서버 **상태**가 한 번 보냈을 때와 동일. 응답이 항상 같다는 뜻이 아님 (`DELETE /items/1` → 첫 번째 200, 두 번째 404이지만 서버 상태는 동일).

### 상태코드 — UI 복구 전략의 입력값

| 코드 | 의미 | 프론트 대응 |
|---|---|---|
| `200` | 성공 | 정상 렌더링 |
| `201` | 생성 성공 | 완료 피드백, 리스트 갱신 |
| `204` | 성공, body 없음 | body 파싱하지 않기 |
| `301/308` | 영구 리다이렉트 | 브라우저 자동 처리, SEO 영향 |
| `302/307` | 임시 리다이렉트 | 브라우저 자동 처리 |
| `304` | 변경 없음 | 캐시 사용 |
| `400` | 잘못된 요청 | 입력값 검증 오류 메시지 |
| `401` | 인증 필요 | 로그인 화면 리다이렉트 |
| `403` | 권한 없음 | "접근 권한이 없습니다" |
| `404` | 리소스 없음 | "찾을 수 없습니다" |
| `429` | 요청 제한 | Retry-After 참고 후 재시도 |
| `500` | 서버 오류 | 일반 오류 fallback |
| `503` | 서비스 중단 | 유지보수 안내 |

### fetch의 에러 처리 — 흔한 함정

```js
// 주의: fetch는 4xx/5xx에서 reject하지 않는다!
const res = await fetch(url);

if (res.ok) {
  return res.json();
}

switch (res.status) {
  case 401:
    redirectToLogin();
    break;
  case 429:
    const retryAfter = res.headers.get("Retry-After");
    throw new RateLimitError(retryAfter);
  default:
    throw new Error(`요청 실패: ${res.status}`);
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 결제 API를 `POST`로 호출하는데 네트워크 에러가 나서 프론트에서 자동 retry를 했더니 결제가 2번 처리됨. 이후 결제 API에는 `Idempotency-Key` 헤더를 포함해서 서버가 중복 요청을 식별하도록 변경. GET 조회는 React Query의 `retry: 3`으로 자동 재시도하되, POST mutation은 retry를 끔.

### 면접 압축 답변

- **10초**: 메서드는 요청 의도, 멱등성은 재시도 안전성, 상태코드는 UI 복구 전략을 결정합니다. `fetch`는 4xx/5xx에서 throw하지 않습니다.
- **30초**: GET/PUT/DELETE는 멱등하므로 재시도가 안전하지만, POST는 멱등하지 않아 멱등성 키가 필요합니다. 상태코드는 401(재로그인), 429(재시도 간격), 5xx(서버 오류 안내)처럼 UI 복구 전략을 다르게 가져가야 합니다. fetch는 네트워크 에러만 reject하고 HTTP 에러는 정상 resolve되므로 `response.ok`를 확인해야 합니다.

## 62. 인증 — 쿠키, 세션, 토큰, 보안 속성

> **핵심 요약** — 쿠키는 저장/전송 수단, 세션은 서버 상태 모델, 토큰은 인증 정보 형식. `Secure`/`HttpOnly`/`SameSite`로 쿠키 보안을 확보하고, Access/Refresh Token을 분리해 보안과 UX를 균형잡는다.

### HTTP는 stateless — 상태를 어떻게 유지하나

HTTP 자체는 요청 간 맥락을 기억하지 않는다. 쿠키가 상태를 얹는 핵심 장치.

```http
# 서버 응답 — 쿠키 설정
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600

# 이후 브라우저 요청 — 자동으로 쿠키 포함
Cookie: sessionId=abc123
```

### 쿠키 vs localStorage vs sessionStorage

| | 쿠키 | localStorage | sessionStorage |
|---|---|---|---|
| **서버 전송** | 매 요청 자동 | 수동만 | 수동만 |
| **용량** | ~4KB | ~5-10MB | ~5-10MB |
| **만료** | Max-Age/Expires | 영구 | 탭 닫으면 삭제 |
| **JS 접근** | HttpOnly면 불가 | 가능 | 가능 |
| **용도** | 인증, 서버 필요 데이터 | 클라이언트 캐시 | 임시 상태 |

### 쿠키, 세션, 토큰 — 층위가 다른 개념

| | 쿠키 | 세션 | 토큰 (**JWT(JSON Web Token, 자기 포함형 인증 토큰)**) |
|---|---|---|---|
| **정체** | 저장/전송 수단 | 서버 상태 모델 | 인증 정보 형식 |
| **저장 위치** | 브라우저 | 서버 (메모리/DB) | 클라이언트 (메모리/쿠키) |
| **서버 조회** | - | 필요 (세션 저장소) | 불필요 (자체 검증) |
| **스케일링** | - | 세션 공유 필요 | stateless, 스케일링 유리 |

```
[세션 기반] 서버가 상태를 기억
클라이언트 → 로그인 → 서버 (세션 생성, DB에 저장)
          ← Set-Cookie: sessionId=abc
이후 요청: Cookie: sessionId=abc → 서버 (DB에서 조회)

[토큰 기반] 클라이언트가 증명을 들고 다님
클라이언트 → 로그인 → 서버 (JWT 생성, 저장 안 함)
          ← { accessToken: "eyJ..." }
이후 요청: Authorization: Bearer eyJ... → 서버 (토큰 자체를 검증)
```

### Cookie / Session / Token — 관계 정리

```
┌─────────────────────────────────────────────────────┐
│                   인증 방식                          │
│                                                     │
│  세션 기반                    토큰 기반              │
│  ┌───────────┐               ┌───────────┐         │
│  │  Session   │               │   JWT     │         │
│  │ (서버 메모리│               │(자기 포함형│         │
│  │  에 저장)  │               │ 서버 조회  │         │
│  └─────┬─────┘               │ 불필요)   │         │
│        │                     └─────┬─────┘         │
│        ▼                           ▼               │
│  ┌─────────────────────────────────────────┐       │
│  │           전송 수단                      │       │
│  │  Cookie          │  Authorization 헤더  │       │
│  │  (자동 전송,     │  (수동 첨부,         │       │
│  │   CSRF 위험)     │   XSS 위험)         │       │
│  └─────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘

Cookie = 전송 수단 (택배 상자)
Session = 서버에 저장된 상태 (서버 금고)
Token = 자기 포함형 인증 정보 (신분증)
```

### 쿠키 보안 속성 — 인증 설계의 핵심

| 속성 | 역할 | 미설정 시 위험 |
|---|---|---|
| `Secure` | HTTPS에서만 전송 | HTTP에서 평문 전송 → 도청 |
| `HttpOnly` | JS에서 접근 불가 | XSS로 `document.cookie` 탈취 |
| `SameSite=Lax` | 같은 사이트 + top-level navigation만 | CSRF 취약 |
| `SameSite=Strict` | 같은 사이트만 | 외부 링크 진입 시 쿠키 미전송 |
| `SameSite=None` | 크로스 사이트도 전송 (Secure 필수) | 넓은 CSRF 노출 |

### Access Token / Refresh Token — 보안과 UX의 균형

| | Access Token | Refresh Token |
|---|---|---|
| **수명** | 짧음 (15분~1시간) | 길음 (7일~30일) |
| **용도** | API 접근 권한 증명 | 새 Access Token 발급 |
| **저장** | 메모리 또는 쿠키 | HttpOnly 쿠키 (권장) |
| **탈취 시** | 피해 기간 제한적 | 서버에서 revoke 가능 |

```js
// 401 감지 시 토큰 갱신 패턴 (간략)
async function fetchWithAuth(url, options = {}) {
  let res = await fetch(url, {
    ...options,
    headers: { ...options.headers, Authorization: `Bearer ${getAccessToken()}` },
  });

  if (res.status === 401) {
    const refreshRes = await fetch("/api/auth/refresh", {
      method: "POST",
      credentials: "include", // HttpOnly 쿠키로 refresh token 전송
    });

    if (!refreshRes.ok) { redirectToLogin(); return; }

    const { accessToken } = await refreshRes.json();
    setAccessToken(accessToken);
    return fetchWithAuth(url, options); // 원래 요청 재시도
  }

  return res;
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> JWT를 localStorage에 저장했는데, XSS 취약점이 발견되면서 토큰이 탈취될 수 있는 상황. Access Token은 메모리(변수)에, Refresh Token은 HttpOnly 쿠키에 저장하도록 변경. XSS로 메모리 변수에 접근하기는 훨씬 어렵고, HttpOnly 쿠키는 JS에서 아예 읽지 못함.

### 자주 하는 오해

- 쿠키 = 세션이다 → **층위가 다르다.** 쿠키는 운반 수단, 세션은 서버 상태 모델.
- JWT는 안전하다 → **payload는 Base64 인코딩일 뿐 암호화가 아니다.** 누구나 디코딩 가능.
- 토큰 기반이면 CSRF 고민 없다 → **Authorization 헤더 사용 시 CSRF는 덜 위험하지만, 쿠키에 토큰을 담으면 CSRF 고려 필요.**

### 면접 압축 답변

- **10초**: 쿠키는 저장/전송 수단, 세션은 서버 상태, 토큰은 인증 정보 형식입니다. `Secure`/`HttpOnly`/`SameSite`로 쿠키 보안을 확보합니다.
- **30초**: 세션 기반은 서버가 상태를 관리하고 쿠키로 세션 ID를 전달하며, 토큰 기반은 클라이언트가 JWT를 들고 서버가 stateless하게 검증합니다. Access Token을 짧게, Refresh Token을 길게 분리해 탈취 피해를 제한하고, Refresh Token은 HttpOnly 쿠키에 보관합니다. 인증 쿠키에는 Secure + HttpOnly + SameSite를 설정해야 XSS/CSRF를 방어합니다.

### Cookie vs Bearer Token — 위협 모델 비교

| 관점 | Cookie 기반 | Bearer Token (Authorization 헤더) |
|---|---|---|
| 전송 방식 | 브라우저가 자동 첨부 | 개발자가 수동 첨부 |
| CSRF 취약 | ✅ (자동 전송이므로) | ❌ (수동 첨부라 공격자가 토큰을 모름) |
| XSS 취약 | HttpOnly면 JS 접근 불가 | localStorage에 저장하면 XSS로 탈취 가능 |
| 서버 상태 | 세션 ID면 서버 저장 필요 | JWT면 stateless |
| 모바일 지원 | 쿠키 다루기 번거로움 | 헤더에 넣기만 하면 됨 |

**실무 선택 기준:**
- 웹 전용 + SSR → Cookie + HttpOnly + SameSite=Lax (CSRF 방어 + XSS 방어)
- 웹 + 모바일 겸용 → Bearer Token + 메모리 저장 (refresh token은 HttpOnly cookie)
- 가장 안전한 조합: Access Token은 메모리에, Refresh Token은 HttpOnly Cookie에

## 63. CORS와 Preflight

> **핵심 요약** — **CORS(Cross-Origin Resource Sharing, 교차 출처 리소스 공유)**는 브라우저의 Same-Origin Policy 위에서 서버가 다른 origin의 요청을 허용할지 HTTP 헤더로 알려주는 메커니즘. 서버-서버 통신에는 없다. Preflight는 비표준 메서드/헤더 사용 시 브라우저가 자동으로 보내는 사전 확인 OPTIONS 요청.

### Same-Origin Policy — CORS의 배경

Origin = **프로토콜 + 호스트 + 포트** 조합.

```
https://example.com:443       ← 하나의 origin
https://example.com:443/page  ← 같은 origin (경로는 무관)
http://example.com:443        ← 다른 origin (프로토콜 다름)
https://api.example.com:443   ← 다른 origin (호스트 다름)
```

브라우저는 다른 origin으로의 스크립트 기반 요청을 기본 차단. **Postman에서 되지만 브라우저에서 안 되는 이유**가 이것.

### CORS 주요 헤더

```http
# 서버 응답
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true      # 쿠키 포함 요청 허용
Access-Control-Max-Age: 86400               # preflight 캐시 시간
```

### Preflight — 언제 발생하나

| Simple Request (preflight 없음) | Preflight 발생 |
|---|---|
| GET, HEAD, POST만 | PUT, DELETE, PATCH 등 |
| 기본 헤더만 (Accept, Content-Language 등) | Authorization, Content-Type: application/json 등 |
| Content-Type이 form/text만 | application/json 등 |

```http
# Preflight 요청 (브라우저가 자동 전송)
OPTIONS /api/data HTTP/1.1
Origin: https://example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization

# Preflight 응답
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
```

### credentials 포함 요청의 제약

```js
fetch("https://api.example.com/data", {
  credentials: "include",  // 쿠키를 함께 보냄
});
// → 서버 필수 설정:
//   Access-Control-Allow-Credentials: true
//   Access-Control-Allow-Origin: https://example.com (와일드카드 * 불가!)
```

### 실무 에피소드 — 이런 상황에서 만난다

> 로컬 개발(`localhost:3000`)에서 API 서버(`localhost:8080`)로 요청하면 CORS 에러. 개발 환경에서는 Next.js의 `rewrites`나 Vite의 `proxy` 설정으로 같은 origin으로 프록시해서 해결. 프로덕션에서는 서버에 `Access-Control-Allow-Origin` 설정.

### 자주 하는 오해

- CORS는 서버 간에도 적용된다 → **아니다.** 브라우저 **SOP(Same-Origin Policy, 동일 출처 정책)** 위에서만 동작.
- CORS 에러는 네트워크 실패다 → **서버는 정상 응답했지만 브라우저가 정책 위반으로 차단한 것.**
- Preflight는 fetch 버그다 → **브라우저 보안 메커니즘.**
- `Access-Control-Allow-Origin: *`이면 다 된다 → **credentials 포함 시 와일드카드 불가.**

### 면접 압축 답변

- **10초**: CORS는 브라우저 SOP 위에서 서버가 다른 origin 요청을 허용할지 헤더로 알려주는 메커니즘이고, Preflight는 비표준 메서드/헤더 사용 시 자동으로 보내는 OPTIONS 사전 확인입니다.
- **30초**: Origin은 프로토콜+호스트+포트 조합이며, 브라우저는 다른 origin 요청을 기본 차단합니다. 서버가 `Access-Control-Allow-*` 헤더로 허용 여부를 알려주고, PUT/DELETE이나 Authorization 헤더 사용 시 Preflight OPTIONS가 먼저 발생합니다. 쿠키 포함 요청은 와일드카드 origin을 쓸 수 없어 서버 설정이 더 엄격해집니다.

## 64. HTTP 캐시와 CDN

> **핵심 요약** — HTTP 캐시는 불필요한 요청을 줄이는 기본 성능 최적화. 브라우저 캐시(private)와 CDN 캐시(public)는 층이 다르다. `Cache-Control`로 정책을 제어하고, `ETag`/`Last-Modified`로 재검증한다. `no-cache`는 저장 금지가 아니라 매번 재검증.

### 캐시 동작 흐름

```
[첫 요청]
브라우저 → 서버
         ← 200 OK + Cache-Control: max-age=3600 + ETag: "abc123"

[3600초 이내 — 강한 캐시]
브라우저 → (서버에 요청 안 함, 캐시에서 바로 사용)
         ← 200 OK (from disk cache)

[3600초 이후 — 조건부 요청 + 재검증]
브라우저 → If-None-Match: "abc123" → 서버
         ← 304 Not Modified (body 없이 응답, 기존 캐시 재사용)
```

### Cache-Control 주요 지시자

| 지시자 | 의미 |
|---|---|
| `no-store` | 어디에도 저장하지 않음 (민감 데이터) |
| `no-cache` | 저장하되, 사용 전 반드시 재검증 (**저장 금지가 아님!**) |
| `private` | 브라우저만 캐시 (CDN 저장 금지) |
| `public` | CDN 등 공유 캐시도 저장 가능 |
| `max-age=N` | N초간 fresh — 서버에 묻지 않고 사용 |
| `s-maxage=N` | CDN용 max-age (max-age보다 우선) |
| `immutable` | max-age 내 새로고침(F5)에도 재검증 안 함 |

### 캐시 층 — 브라우저 vs CDN

```
사용자 요청
  ↓
[1] 브라우저 캐시 (private — 이 사용자만)
  ↓ miss
[2] CDN Edge 캐시 (public — 같은 지역 사용자 공유)
  ↓ miss
[3] Origin 서버
```

```http
# 브라우저만 캐시 (개인화 응답)
Cache-Control: private, max-age=60

# 브라우저 + CDN 모두 (공개 정적 자산)
Cache-Control: public, max-age=300, s-maxage=3600

# 아무 곳에도 저장 안 함 (결제 정보 등)
Cache-Control: no-store
```

**위험**: 개인화 응답(마이페이지)이 `public`으로 CDN에 캐시되면 다른 사용자에게 노출될 수 있다.

### 실무 캐시 전략

| 리소스 | Cache-Control | 이유 |
|---|---|---|
| 해시 파일명 정적 자산 (`main.a1b2c3.js`) | `public, max-age=31536000, immutable` | 내용 바뀌면 파일명이 바뀜 |
| HTML (`index.html`) | `no-cache` + ETag | 항상 최신 버전 확인 |
| 인증된 사용자 API | `private, max-age=60` | CDN 저장 방지 |
| 민감 응답 (결제 등) | `no-store` | 어디에도 저장 금지 |
| 공개 API (상품 목록) | `public, s-maxage=60` | CDN에서 1분 캐시 |

### CDN — 지리적 분산 캐시

```
[Cache Miss]  사용자(한국) → CDN Edge(한국) → Origin(미국) → 응답 (edge에 캐시 저장)
[Cache Hit]   사용자(한국) → CDN Edge(한국) → 캐시된 응답 (origin 요청 없음)
```

Vercel, Netlify, CloudFront 같은 현대 배포 플랫폼은 기본으로 CDN을 사용. `public` + `s-maxage`로 CDN 캐시를 제어.

### 실무 에피소드 — 이런 상황에서 만난다

> 배포 후 사용자에게 "사이트가 안 바뀌었어요" 불만. 원인은 HTML에 `Cache-Control`이 없어서 브라우저 휴리스틱 캐시가 오래된 HTML을 보여줌. HTML에 `no-cache` + ETag를 설정하고, JS/CSS는 해시 파일명 + 긴 max-age로 바꾸니 배포 즉시 반영.

### 자주 하는 오해

- `no-cache`는 저장 금지다 → **아니다.** 매번 재검증. 저장 금지는 `no-store`.
- 캐시는 브라우저만 있다 → **CDN edge, 프록시 등 여러 층이 있다.**
- CDN은 대형 서비스만 쓴다 → **Vercel, Netlify 등 현대 플랫폼은 기본 CDN.**

### 면접 압축 답변

- **10초**: HTTP 캐시는 `Cache-Control`로 정책을 정하고, 브라우저(private)와 CDN(public) 두 층이 있습니다. `no-cache`는 저장 금지가 아니라 매번 재검증입니다.
- **30초**: max-age 내에는 서버에 묻지 않고 캐시를 사용하고(강한 캐시), 만료 후 ETag로 재검증합니다. 해시 파일명 정적 자산은 `immutable`로 오래 캐시하고, HTML은 `no-cache`로 매번 확인합니다. CDN은 edge에서 콘텐츠를 제공해 지연 시간과 origin 부담을 줄이며, 개인화 응답은 `private`으로 CDN 캐시를 방지해야 합니다.

## 65. 실시간 통신 — WebSocket, SSE, Polling

> **핵심 요약** — Polling은 주기적 HTTP 요청(단순하지만 낭비), SSE는 서버→클라이언트 단방향 푸시(자동 재연결), WebSocket은 양방향 full-duplex(채팅/게이밍). 용도에 따라 선택.

### 비교 테이블

| | Polling | SSE | WebSocket |
|---|---|---|---|
| **방향** | 클라이언트 → 서버 | 서버 → 클라이언트 | 양방향 |
| **프로토콜** | HTTP | HTTP | `ws://` / `wss://` |
| **자동 재연결** | 직접 구현 | 내장 | 직접 구현 |
| **바이너리** | X | X | O |
| **오버헤드** | 높음 | 낮음 | 매우 낮음 |
| **적합한 사례** | 단순 대시보드 | 알림, 피드, 주가 | 채팅, 게이밍, 협업 |

```tsx
// SSE — EventSource API
useEffect(() => {
  const es = new EventSource("/api/notifications");
  es.onmessage = (e) => setNotifications((prev) => [JSON.parse(e.data), ...prev]);
  return () => es.close();
}, []);

// WebSocket
useEffect(() => {
  const ws = new WebSocket("wss://api.example.com/chat");
  ws.onmessage = (e) => setMessages((prev) => [...prev, JSON.parse(e.data)]);
  ws.onclose = () => { /* 수동 재연결 로직 */ };
  return () => ws.close();
}, []);
```

### 실무 에피소드 — 이런 상황에서 만난다

> 알림 기능을 WebSocket으로 구현했는데, 프록시/로드밸런서 설정이 복잡하고 재연결 로직도 직접 만들어야 했다. 알림은 서버→클라이언트 단방향이라 SSE로 변경하니 기존 HTTP 인프라 그대로 쓸 수 있고 자동 재연결도 내장되어 운영이 훨씬 간단해짐.

### 면접 압축 답변

- **10초**: Polling은 주기적 HTTP, SSE는 서버→클라이언트 단방향 푸시, WebSocket은 양방향 full-duplex입니다.
- **30초**: SSE는 `EventSource` API로 자동 재연결이 내장되어 알림/피드에 적합합니다. WebSocket은 양방향 + 바이너리 지원으로 채팅/게이밍에 적합하지만 재연결은 직접 구현해야 합니다. 업데이트 빈도가 낮으면 Polling이 가장 단순합니다.

## 66. ESM vs CJS — 모듈 시스템과 번들 최적화

> **핵심 요약** — ESM(`import`/`export`)은 정적 분석과 tree-shaking이 가능해서 번들 크기를 줄일 수 있다. CJS(`require`/`module.exports`)는 동적이라 tree-shaking이 어렵다. 현대 프론트엔드는 ESM 기반.

### 비교 테이블

| | ESM | CJS |
|---|---|---|
| **문법** | `import` / `export` | `require` / `module.exports` |
| **분석** | 정적 (빌드 타임) | 동적 (런타임) |
| **Tree-shaking** | 가능 | 불가 |
| **코드 스플리팅** | `import()` 네이티브 | 별도 처리 |
| **브라우저** | 네이티브 (`<script type="module">`) | 번들러 필수 |
| **순환 참조** | 라이브 바인딩 | 값 복사 (빈 객체 위험) |

```js
// ESM — lodash-es: get만 번들에 포함
import { get } from "lodash-es";

// CJS — lodash 전체가 포함될 수 있음
const { get } = require("lodash");
```

### 실무 에피소드 — 이런 상황에서 만난다

> 번들 크기를 줄이려고 분석했더니 lodash가 전체 포함되어 70KB를 차지. `import { get } from "lodash"`를 `import { get } from "lodash-es"`로 바꾸니 tree-shaking이 적용되어 사용하는 함수만 번들에 포함, 70KB → 4KB로 감소.

### 면접 압축 답변

- **10초**: ESM은 정적 분석으로 tree-shaking이 가능해 번들을 줄이고, CJS는 동적이라 tree-shaking이 어렵습니다. 현대 프론트엔드는 ESM 기반입니다.
- **30초**: ESM은 `import`/`export`로 빌드 타임에 의존 관계가 확정되어 사용하지 않는 코드를 제거할 수 있습니다. CJS는 `require()`가 런타임에 결정되므로 전체 모듈이 번들에 포함될 수 있습니다. 라이브러리 선택 시 ESM 지원 여부(lodash-es, date-fns)가 번들 최적화에 중요합니다.

# 챕터 7 핵심 요약 — 한 번의 요청이 해석되고, 보호되고, 재사용되고, 더 빠르게 전달되기까지

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **브라우저의 요청 한 번이 사용자 화면에 도달하기까지, 어떤 단계를 거치고 각 단계에서 성능과 보안은 어떻게 결정되는가?**

---

### 1단계: 요청이 서버에 닿기까지 — DNS, TCP, TLS

**DNS로 도메인을 IP로 변환**하고, **TCP 3-way handshake로 연결을 수립**하고, HTTPS면 **TLS handshake로 암호화 채널**을 연다. 첫 방문이 느린 이유는 이 단계들이 누적되기 때문이다. `dns-prefetch`와 `preconnect`로 초기 지연을 줄인다.

### 2단계: 요청의 의도와 재시도 안전성 — HTTP 메서드, 상태코드, 멱등성

메서드는 **요청 의도**를, 멱등성은 **재시도 안전성**을, 상태코드는 **UI 복구 전략**을 결정한다. POST는 멱등하지 않아 **멱등성 키**가 필요하다. `fetch`는 4xx/5xx에서 reject하지 않으므로 **`response.ok`를 직접 확인**해야 한다.

### 3단계: 요청 간 맥락을 유지하는 방법 — 쿠키, 세션, 토큰

쿠키는 **저장/전송 수단**, 세션은 **서버 상태 모델**, 토큰(JWT)은 **인증 정보 형식** — 층위가 다른 개념이다. **Access Token을 짧게, Refresh Token을 길게 분리**하고, 인증 쿠키에는 `Secure` + `HttpOnly` + `SameSite`를 설정한다.

### 4단계: 다른 출처의 요청을 허용하는 메커니즘 — CORS와 Preflight

CORS는 **서버가 다른 origin의 요청을 허용할지 HTTP 헤더로 알려주는** 메커니즘이다. 서버-서버 통신에는 적용되지 않는다. 쿠키 포함 요청에서는 **와일드카드 origin을 쓸 수 없다.**

### 5단계: 불필요한 요청을 줄이는 기본 최적화 — HTTP 캐시와 CDN

`no-cache`는 저장 금지가 아니라 **매번 재검증**이고, 저장 금지는 `no-store`다. 해시 파일명 정적 자산은 `immutable`로 오래 캐시하고, HTML은 `no-cache`로 매번 확인하는 것이 실무 표준이다. 개인화 응답이 `public`으로 CDN에 캐시되면 다른 사용자에게 노출될 위험이 있다.

### 6단계: 용도에 맞는 실시간 통신 — WebSocket, SSE, Polling

단방향이면 SSE(자동 재연결 내장), 양방향이면 WebSocket(채팅/게이밍), 빈도가 낮으면 Polling이 가장 단순하다.

### 7단계: 번들 크기와 직결되는 모듈 시스템 — ESM vs CJS

ESM은 **정적 분석으로 tree-shaking이 가능**해서 사용하지 않는 코드를 제거할 수 있다. `lodash`를 `lodash-es`로 바꾸는 것만으로 70KB에서 4KB로 줄어든다.

---

### 한 줄로 이어 읽기

**DNS/TCP/TLS**가 연결을 수립하고 → **HTTP 메서드와 상태코드**가 요청 의도와 복구 전략을 정하고 → **쿠키/세션/토큰**이 stateless한 HTTP 위에 인증을 얹고 → **CORS**가 출처 정책 안에서 교차 요청을 제어하고 → **HTTP 캐시**와 **CDN**이 불필요한 요청을 줄이고 → **SSE/WebSocket**이 실시간 통신을 가능하게 하고 → **ESM**이 정적 분석으로 번들 크기를 줄인다.
