# 챕터 7. 브라우저 요청 흐름에서 인증과 캐시, CDN까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **브라우저 요청은 단순히 fetch 한 번이 아니라, DNS 조회, 연결 수립, 보안 계층, HTTP 메시지, 인증, 캐시, CDN까지 이어지는 하나의 흐름으로 이해해야 한다.**

## 81. 브라우저에서 URL을 입력하면 어떤 큰 흐름이 시작되나

프론트엔드 면접에서 네트워크 질문이 나오면 많은 사람이 곧바로 HTTP 메서드나 상태코드부터 말한다.
그런데 더 좋은 출발점은 이거다.

> **사용자가 URL을 입력한 뒤 페이지가 보이기까지, 브라우저는 어떤 단계를 거치는가?**

### 큰 흐름

아주 크게 보면 보통 이런 순서다.

1. URL 해석 (프로토콜, 호스트, 포트, 경로 등을 분리)
2. DNS 조회 (도메인 이름을 IP 주소로 변환)
3. TCP 연결 수립 (3-way handshake)
4. HTTPS라면 TLS handshake
5. HTTP 요청 전송
6. 서버 응답 수신
7. 캐시 여부 판단
8. HTML 파싱과 추가 리소스 요청
9. CSS/JS/이미지 등 하위 자원 로딩
10. 렌더 트리 구성, 레이아웃, 페인트, 컴포지트를 거쳐 최종 렌더링

### 왜 이게 중요한가

이 큰 그림이 있어야 아래 질문들이 전부 연결된다.

- DNS는 왜 필요한가
- HTTP와 HTTPS 차이는 무엇인가
- TLS는 왜 필요한가
- 상태코드는 어디서 오나
- 캐시는 언제 이득이 되나
- CDN은 어디에 끼는가

### 자주 하는 오해

- 브라우저 요청은 그냥 서버에 바로 간다 → 실제로는 여러 단계가 있다
- HTTP만 알면 네트워크는 충분하다 → DNS, 연결, 캐시, 보안도 함께 봐야 한다
- 프론트엔드는 요청만 보내면 된다 → 사용자 체감 성능은 요청 전후 전체 흐름의 영향을 받는다

### 면접 압축 답변

**10초 답변:** 브라우저에서 URL을 입력하면 DNS 조회, TCP/TLS 연결, HTTP 요청/응답, 하위 리소스 로딩, 렌더링까지 이어지는 전체 흐름이 시작됩니다.

**30초 답변:** 브라우저에서 URL을 입력하면 단순히 서버로 바로 요청을 보내는 것이 아니라, DNS 조회로 IP를 찾고, TCP 3-way handshake로 연결을 수립하고, HTTPS라면 TLS handshake로 보안 계층을 준비한 뒤 HTTP 요청/응답을 주고받습니다. 이후 HTML을 파싱하면서 CSS, JS, 이미지 등 하위 리소스를 추가로 요청하고, 렌더 트리 구성과 레이아웃, 페인트를 거쳐 최종 렌더링까지 이어집니다.

> 이제 큰 흐름을 봤다. 그다음엔 "도메인 이름을 실제 서버 주소로 어떻게 바꾸는가"라는 질문으로 넘어가야 한다. 즉 **DNS**다.

## 82. DNS는 왜 웹 요청의 첫 관문인가

사용자는 `example.com` 같은 도메인을 입력하지만,
브라우저가 실제로 통신하려면 IP 주소가 필요하다.
DNS(Domain Name System)는 바로 이 변환을 담당한다. 사람이 읽기 쉬운 도메인 이름을 기계가 통신에 사용하는 IP 주소로 바꿔주는 분산형 시스템이다.

### 핵심 개념

DNS는 인터넷의 전화번호부 같은 역할을 한다.

- 사람은 도메인 이름을 기억한다
- 기계는 IP 주소로 통신한다
- DNS가 둘 사이를 연결한다

### 왜 이게 중요한가

이걸 이해해야 아래가 자연스럽다.

- 왜 도메인만 알아서는 실제 요청을 못 보내는가
- 왜 DNS 지연도 성능 문제인가
- 왜 CDN edge나 origin 서버도 결국 DNS 뒤에서 선택될 수 있는가

### DNS 조회를 아주 단순하게 보면

일반적인 흐름은 아래처럼 이해하면 된다.

1. 브라우저 캐시 → OS 캐시 → hosts 파일 순으로 먼저 확인한다
2. 없으면 DNS resolver(보통 ISP나 설정된 DNS 서버)에 묻는다
3. resolver가 Root → TLD(.com) → Authoritative DNS 순으로 따라가며 IP를 찾는다
4. 찾은 IP를 브라우저에 돌려준다 (TTL만큼 캐시)
5. 브라우저는 그 IP로 연결을 시도한다

### 실무에서 쓸 수 있는 DNS 프리페치

외부 도메인의 리소스를 미리 조회하면 초기 로딩 체감이 빨라질 수 있다.

```html
<!-- 브라우저가 미리 DNS 조회를 수행한다 -->
<link rel="dns-prefetch" href="https://fonts.googleapis.com" />
<link rel="dns-prefetch" href="https://cdn.example.com" />
```

### 자주 하는 오해

- DNS는 한 서버가 전부 해결한다 → 여러 단계(Root, TLD, Authoritative)와 캐시가 얽힌다
- DNS는 한 번만 일어나고 끝이다 → TTL(Time To Live)에 따라 캐시가 만료되면 재조회된다
- DNS는 백엔드 인프라 이야기라 프론트엔드와 무관하다 → 초기 응답 지연과 직결된다

### 실무 연결

웹 성능에서 DNS lookup 시간은 첫 요청 체감에 영향을 준다.
특히 외부 스크립트, 써드파티 폰트, 이미지 CDN, 분석 스크립트가 많아질수록 DNS 의존 지점도 늘어난다.

### 면접 압축 답변

**10초 답변:** DNS는 도메인 이름을 IP 주소로 바꾸는 시스템으로, 웹 요청의 첫 단계이며 DNS 지연이 초기 로딩 성능에 영향을 줍니다.

**30초 답변:** DNS는 사람이 사용하는 도메인 이름을 브라우저가 통신에 필요한 IP 주소로 바꾸는 분산형 시스템입니다. 브라우저 캐시, OS 캐시를 거쳐 없으면 DNS resolver가 Root, TLD, Authoritative 서버를 순서대로 조회합니다. TTL 기반 캐시 전략이 있어 반복 조회를 줄이지만, 외부 도메인이 많아지면 DNS 의존 지점도 늘어나므로 `dns-prefetch` 같은 힌트로 초기 로딩을 개선할 수 있습니다.

> 이제 서버 주소를 찾았다. 그다음엔 실제로 그 서버와 **연결을 어떻게 맺는가**를 봐야 한다.

## 83. DNS 다음에는 왜 연결 수립이 필요하나

IP를 알았다고 해서 바로 HTTP를 보낼 수 있는 건 아니다.
브라우저와 서버 사이에는 실제 통신 경로를 준비하는 단계가 필요하다.

### 핵심 개념

HTTP는 애플리케이션 계층 프로토콜이고,
그 아래에서 TCP(전송 계층)가 실제 데이터 전송을 위한 연결을 받쳐줘야 한다.

실무 감각에서는 보통 이렇게 이해하면 충분하다.

- 주소를 찾는 단계: DNS
- 통신 길을 여는 단계: TCP 연결 수립 (3-way handshake)
- 그 위에서 요청/응답을 주고받는 단계: HTTP

### TCP 3-way handshake

TCP 연결은 세 단계를 거친다.

1. **SYN** — 클라이언트가 서버에 연결 요청을 보낸다
2. **SYN-ACK** — 서버가 요청을 수락하고 응답한다
3. **ACK** — 클라이언트가 확인 응답을 보내고 연결이 수립된다

이 과정에 최소 1 RTT(Round Trip Time)가 소요된다.

### 왜 이게 중요한가

이 감각이 있어야 아래가 이해된다.

- 왜 첫 요청은 느리고 후속 요청은 상대적으로 빠를 수 있는가 (keep-alive, connection reuse)
- 왜 HTTP/2의 멀티플렉싱이 성능에 도움이 되는가
- 왜 TLS handshake가 추가 RTT 비용과 연결되는가

### 자주 하는 오해

- DNS가 끝나면 바로 응답이 온다 → TCP 연결 수립이 더 필요하다
- 연결 비용은 무시해도 된다 → 첫 방문 체감 성능에 영향이 크다
- HTTP가 네트워크 전부다 → 실제론 TCP/IP 계층 위에서 동작한다

### 실무 연결

처음 방문한 사이트가 느리게 느껴지는 이유에는:

- DNS 조회
- TCP 연결 수립 (1 RTT)
- TLS handshake (추가 1-2 RTT)
- 서버 응답 시간

이 모두 섞여 있다.

그래서 "API가 느리다"라고 말할 때도 실제로는 어느 단계가 느린지 DevTools의 Timing 탭에서 분해해서 봐야 한다.

### 면접 압축 답변

**10초 답변:** DNS로 IP를 찾은 뒤에는 TCP 3-way handshake로 실제 통신 경로를 수립해야 HTTP 메시지를 주고받을 수 있습니다.

**30초 답변:** DNS로 IP를 찾은 뒤에는 실제 통신을 위한 TCP 연결 수립 단계가 필요합니다. SYN, SYN-ACK, ACK 세 단계의 3-way handshake에 최소 1 RTT가 걸리고, HTTPS라면 TLS handshake에 추가 RTT가 필요합니다. 그래서 첫 방문이 느린 이유는 이 단계들이 누적되기 때문이며, keep-alive나 HTTP/2 멀티플렉싱이 후속 요청 성능을 개선하는 이유이기도 합니다.

> 이제 연결 수립이 필요하다는 걸 봤다. HTTPS라면 여기에 추가되는 **TLS**를 봐야 한다.

## 84. HTTPS와 TLS는 왜 함께 설명해야 하나

HTTP와 HTTPS를 단순히 "s가 붙었냐 안 붙었냐"로 설명하면 약하다.
핵심은 HTTPS가 **TLS 위에서 보호되는 HTTP**라는 점이다. TLS(Transport Layer Security)는 인터넷 통신의 기밀성(Confidentiality)과 데이터 무결성(Integrity)을 제공하는 보안 프로토콜이다.

### 핵심 개념

- HTTP는 웹 리소스를 요청/응답하는 프로토콜이다
- HTTPS는 그 HTTP 통신을 TLS로 보호한 형태다
- TLS는 세 가지 역할을 한다:
  - **암호화** — 통신 내용을 제3자가 엿볼 수 없게 한다
  - **인증** — 서버 인증서로 서버 신원을 확인한다
  - **무결성** — 전송 중 데이터가 변조되지 않았음을 보장한다

### 왜 이게 중요한가

이걸 이해해야 아래 질문을 제대로 설명할 수 있다.

- 왜 로그인 페이지는 HTTPS가 필수인가
- 왜 쿠키에 `Secure` 속성이 중요한가
- 왜 중간에서 내용을 엿보거나 바꾸기 어려워지는가

### TLS handshake를 아주 거칠게 보면

HTTPS 연결에서는 TCP 연결 이후, HTTP 요청 전에 대략 이런 일이 일어난다.

1. 클라이언트가 지원하는 TLS 버전과 암호 스위트(cipher suite) 목록을 보낸다
2. 서버가 사용할 버전과 스위트를 선택하고 인증서를 보낸다
3. 클라이언트가 인증서를 검증한다 (CA 체인 확인)
4. 키 교환을 통해 세션 키를 합의한다
5. 이후 모든 HTTP 메시지는 이 세션 키로 암호화해 전송한다

TLS 1.2에서는 이 과정에 2 RTT가 걸린다. TLS 1.3에서는 1 RTT로 줄었고, 재연결 시 0-RTT도 가능하다.

### 자주 하는 오해

- HTTPS면 무조건 완벽하게 안전하다 → XSS, CSRF 등 다른 취약점은 여전히 존재할 수 있다
- TLS는 느려서 피해야 한다 → TLS 1.3은 1-RTT handshake로 빨라졌고, 현대 웹에선 기본 전제다
- HTTPS는 백엔드만 신경 쓰면 된다 → 프론트 쿠키, mixed content, 보안 전제와 연결된다

### 실무 연결

프론트엔드 관점에서도 HTTPS/TLS 감각은 중요하다.

- `Secure` 쿠키는 HTTPS에서만 전송된다
- Service Worker는 localhost를 제외하면 HTTPS에서만 등록 가능하다
- Geolocation, Camera, Clipboard 등 민감한 브라우저 API는 Secure Context(HTTPS)를 요구한다
- HTTP 페이지에서 HTTPS 리소스를 로드하면 mixed content로 차단될 수 있다
- OAuth/인증 콜백은 HTTPS가 아니면 redirect 과정에서 토큰이 노출될 수 있다

### 면접 압축 답변

**10초 답변:** HTTPS는 TLS로 보호되는 HTTP이며, TLS는 암호화, 서버 인증, 데이터 무결성을 제공합니다.

**30초 답변:** HTTPS는 단순히 HTTP에 s가 붙은 것이 아니라, TLS 프로토콜 위에서 동작하는 HTTP 통신입니다. TLS는 서버 인증서로 신원을 확인하고, 키 교환으로 세션 키를 합의해 통신을 암호화하며, 전송 중 변조를 방지합니다. TLS 1.3에서는 1-RTT handshake로 성능도 개선됐습니다. 프론트엔드에서도 Secure 쿠키, Service Worker, 민감 API 등이 HTTPS를 전제로 하므로 중요합니다.

> 이제 안전한 연결 위에서 통신할 수 있게 됐다. 그다음엔 그 위를 흐르는 **HTTP 메시지** 자체를 봐야 한다.

## 85. HTTP는 정확히 어떤 프로토콜인가

HTTP(HyperText Transfer Protocol)는 웹에서 리소스를 가져오기 위한 애플리케이션 계층 프로토콜이다. 요청과 응답 메시지를 교환하는 클라이언트-서버 모델이며, 본질적으로 stateless하지만 쿠키를 통해 상태를 얹을 수 있다.

### 핵심 개념

HTTP를 이해할 때는 아래를 같이 봐야 한다.

- 클라이언트가 요청을 보낸다
- 서버가 응답을 돌려준다
- 메시지는 method, path, headers, body 같은 구조를 가진다
- 응답은 status code, headers, body를 가진다

### HTTP 버전별 차이

| 버전 | 핵심 특징 |
|------|-----------|
| HTTP/1.1 | keep-alive 기본, 파이프라이닝(제한적), 한 연결당 한 요청씩 순서대로 |
| HTTP/2 | 멀티플렉싱(하나의 연결에서 동시에 여러 요청/응답), 헤더 압축(HPACK), 서버 푸시 |
| HTTP/3 | QUIC(UDP 기반) 위에서 동작, 연결 수립 더 빠름, HoL blocking 감소 |

### 왜 이게 중요한가

이걸 알아야:

- REST API를 읽고
- 브라우저 DevTools 네트워크 탭을 해석하고
- 캐시 헤더를 이해하고
- 인증 흐름을 설명할 수 있다

### 요청/응답 예시

```http
GET /products?page=2 HTTP/1.1
Host: example.com
Accept: application/json
Authorization: Bearer some-token
```

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: private, max-age=60

{"items":[...]}
```

### fetch로 보는 실제 코드

```js
const response = await fetch('https://example.com/products?page=2', {
  method: 'GET',
  headers: {
    'Accept': 'application/json',
    'Authorization': `Bearer ${token}`,
  },
});

// 상태코드 확인
console.log(response.status);      // 200
console.log(response.headers.get('Content-Type')); // application/json

const data = await response.json();
```

### 자주 하는 오해

- HTTP는 JSON 전송 프로토콜이다 → JSON은 흔한 payload 형식일 뿐, HTML/이미지/XML 등 무엇이든 전송 가능하다
- body는 항상 있다 → GET 요청이나 204 응답에는 보통 body가 없다
- 상태코드만 보면 충분하다 → 헤더(캐시 정책, 인증 정보 등)와 body 의미도 중요하다

### 면접 압축 답변

**10초 답변:** HTTP는 웹 리소스를 주고받는 클라이언트-서버 프로토콜이며, 요청은 method/path/headers/body, 응답은 status code/headers/body 구조입니다.

**30초 답변:** HTTP는 웹 리소스를 주고받기 위한 클라이언트-서버 프로토콜입니다. 요청은 method, path, headers, body 같은 구조를 가지고, 응답은 status code, headers, body를 가집니다. HTTP/1.1은 keep-alive, HTTP/2는 멀티플렉싱과 헤더 압축, HTTP/3는 QUIC 기반으로 각각 성능이 개선됐습니다. 프론트엔드에서는 이 구조를 알아야 네트워크 탭과 API 동작을 제대로 해석할 수 있습니다.

> 이제 HTTP 메시지 구조를 봤다. 그다음엔 요청 의도를 가장 먼저 드러내는 **HTTP 메서드**로 넘어가야 한다.

## 86. HTTP 메서드와 멱등성은 왜 retry 설계와 연결되나

메서드는 단순 문법이 아니다.
클라이언트가 서버에 어떤 의도를 가지고 요청하는지를 드러낸다.

### 핵심 개념

대표 메서드는 보통 이렇게 이해하면 된다.

| 메서드 | 의도 | 멱등성 | 안전(Safe) |
|--------|------|--------|------------|
| `GET` | 조회 | O | O |
| `POST` | 생성/처리 요청 | X | X |
| `PUT` | 전체 갱신 (대상 리소스를 요청 payload로 대체) | O | X |
| `PATCH` | 부분 수정 | X (구현에 따라 다름) | X |
| `DELETE` | 삭제 | O | X |
| `OPTIONS` | 가능 메서드/사전 확인 (preflight) | O | O |

여기에 중요한 개념이 **멱등성(Idempotency)**이다.

멱등성은 같은 요청을 여러 번 보내도 서버의 상태가 한 번 보냈을 때와 동일하게 유지되는 성질을 말한다. 응답이 항상 동일하다는 뜻이 아니다. 예를 들어 `DELETE /items/1`을 두 번 보내면 첫 번째는 200, 두 번째는 404를 받을 수 있지만, 서버 상태는 동일하다(아이템이 삭제된 상태).

### 왜 이게 중요한가

이걸 이해해야:

- 어떤 요청은 재시도해도 상대적으로 안전하고
- 어떤 요청은 중복 수행 위험이 큰지
판단할 수 있다.

```js
// GET은 멱등하므로 재시도가 비교적 안전하다
async function fetchWithRetry(url, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      const res = await fetch(url);
      if (res.ok) return res.json();
    } catch (e) {
      if (i === retries - 1) throw e;
    }
  }
}

// POST(결제 등)는 멱등하지 않으므로 중복 방지가 필요하다
async function submitPayment(data) {
  const idempotencyKey = crypto.randomUUID();

  const res = await fetch('/api/payments', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Idempotency-Key': idempotencyKey, // 서버에서 중복 요청을 식별
    },
    body: JSON.stringify(data),
  });

  return res.json();
}
```

### 자주 하는 오해

- `POST`는 무조건 위험하고 `GET`은 무조건 안전하다 → 서버 구현까지 같이 봐야 한다. GET으로 부작용을 일으키는 API도 존재할 수 있다
- 메서드는 이름만 다를 뿐 의미 차이는 별로 없다 → API 설계와 재시도 정책에 중요하다
- 멱등성은 백엔드만 신경 쓰는 문제다 → 프론트의 retry/backoff와 바로 연결된다
- 멱등성은 "응답이 항상 같다"는 뜻이다 → 서버 상태가 동일하게 유지되는 것이지, 응답 코드까지 같다는 뜻이 아니다

### 실무 연결

프론트엔드에서:

- 실패한 조회 재시도 (React Query의 `retry` 옵션 등)
- 저장 버튼 중복 클릭 방지 (debounce 또는 로딩 상태)
- mutation 중복 요청 방지 (idempotency key)
- optimistic update rollback

을 설계할 때 멱등성 감각이 필요하다.

### 면접 압축 답변

**10초 답변:** HTTP 메서드는 요청 의도를, 멱등성은 같은 요청 반복 시 서버 상태가 동일한지를 나타내며, 이에 따라 재시도 안전성이 달라집니다.

**30초 답변:** HTTP 메서드는 단순 이름이 아니라 요청 의도를 나타내고, 멱등성은 같은 요청을 여러 번 보내도 서버 상태가 한 번 보냈을 때와 동일한지를 뜻합니다. GET, PUT, DELETE는 멱등하므로 재시도가 비교적 안전하지만, POST는 멱등하지 않아 결제 같은 중요 요청에서는 idempotency key 같은 중복 방지 설계가 필요합니다.

> 이제 요청 의도는 봤다. 그다음엔 서버가 결과를 어떻게 표현하는지, 즉 **상태코드**로 넘어가야 한다.

## 87. 상태코드는 왜 단순 숫자가 아니라 UI 설계 신호인가

많은 학습자가 상태코드를 "200 성공, 404 없음, 500 서버 에러" 정도로만 외운다.
그런데 프론트엔드 실무에서는 상태코드가 **UI와 복구 전략의 입력값**이 된다.

### 핵심 개념

대표 범위는 이렇게 보면 된다.

- `2xx`: 성공
- `3xx`: 리다이렉션
- `4xx`: 클라이언트 측 요청 문제
- `5xx`: 서버 측 오류

### 실무에서 자주 만나는 상태코드

| 코드 | 의미 | 프론트 대응 |
|------|------|-------------|
| `200` | 성공 | 정상 렌더링 |
| `201` | 생성 성공 | 생성 완료 피드백, 리스트 갱신 |
| `204` | 성공, body 없음 | 삭제 완료 등에서 body 파싱하지 않기 |
| `301` | 영구 리다이렉트 (메서드 변경 가능) | 브라우저가 자동 처리, SEO에 영향 |
| `302` | 임시 리다이렉트 (메서드 변경 가능) | 브라우저가 자동 처리 |
| `307` | 임시 리다이렉트 (메서드 유지) | POST → POST 유지 필요 시 사용 |
| `308` | 영구 리다이렉트 (메서드 유지) | POST → POST 유지 + SEO 영구 이전 |
| `304` | 변경 없음(캐시 사용) | 브라우저가 캐시된 응답 사용 |
| `400` | 잘못된 요청 | 입력값 검증 오류 메시지 |
| `401` | 인증 필요 | 로그인 화면으로 리다이렉트 |
| `403` | 권한 없음 | "접근 권한이 없습니다" UI |
| `404` | 리소스 없음 | "찾을 수 없습니다" UI |
| `409` | 충돌 | 동시 편집 충돌 처리 |
| `429` | 너무 많은 요청 | 재시도 간격 고려 (Rate Limit) |
| `500` | 서버 오류 | 일반 오류 fallback UI |
| `503` | 서비스 일시 중단 | 유지보수 안내 UI |

### fetch에서 상태코드 분기 처리

주의: `fetch`는 4xx/5xx에서 reject하지 않는다. 네트워크 에러(서버에 아예 도달 불가)만 reject한다.

```js
async function apiCall(url) {
  const res = await fetch(url);

  // fetch는 4xx/5xx에서 throw하지 않는다!
  // 네트워크 에러(서버 미도달)만 catch로 간다
  if (res.ok) {
    return res.json();
  }

  switch (res.status) {
    case 401:
      // 토큰 만료 → 로그인 화면으로
      redirectToLogin();
      break;
    case 403:
      throw new Error('접근 권한이 없습니다');
    case 404:
      throw new Error('요청한 리소스를 찾을 수 없습니다');
    case 429:
      // Retry-After 헤더 확인
      const retryAfter = res.headers.get('Retry-After');
      throw new RateLimitError(retryAfter);
    default:
      throw new Error(`요청 실패: ${res.status}`);
  }
}
```

### 자주 하는 오해

- 에러는 전부 하나의 토스트로 처리하면 된다 → 사용자 경험이 거칠어진다
- 4xx, 5xx는 둘 다 그냥 실패다 → 4xx는 클라이언트 문제(입력 수정 등), 5xx는 서버 문제(재시도 또는 안내)로 복구 전략이 다르다
- `fetch`는 상태코드가 4xx/5xx면 자동으로 throw한다 → **아니다**. `fetch`는 네트워크 에러만 reject하고, HTTP 에러 응답은 정상적으로 resolve된다. `response.ok`를 확인해야 한다

### 면접 압축 답변

**10초 답변:** 상태코드는 UI와 복구 전략의 신호이며, 401은 재로그인, 403은 권한 부족, 429는 재시도 간격 고려처럼 다르게 반응해야 합니다.

**30초 답변:** 상태코드는 단순 숫자가 아니라 UI와 복구 전략의 신호입니다. 4xx는 클라이언트 문제이므로 입력 수정이나 로그인 유도가 적절하고, 5xx는 서버 문제이므로 재시도나 오류 안내가 맞습니다. 또한 fetch는 4xx/5xx에서 자동으로 throw하지 않고 정상 resolve되므로, `response.ok`나 `response.status`를 직접 확인해 분기 처리해야 합니다.

> 이제 요청/응답의 기본 구조를 봤다. 그다음엔 HTTP가 본질적으로 stateless라는 점, 그리고 그 위에 상태를 얹는 **쿠키**로 넘어가야 한다.

## 88. HTTP가 stateless라면 로그인 상태는 어떻게 유지하나

HTTP 자체는 요청과 요청 사이의 문맥을 기본적으로 기억하지 않는 stateless 프로토콜이다. 다만 쿠키를 통해 상태를 얹어 세션을 유지할 수 있다.

### 핵심 개념

stateless라는 건:

- 서버가 이전 요청의 맥락을 자동으로 기억하지 않는다는 뜻이다
- 매 요청은 독립적이다

그래서 로그인 상태, 장바구니, 사용자 선호 같은 문맥을 유지하려면
별도의 상태 유지 장치가 필요하다.

대표적인 것이 쿠키다.

### 쿠키 기본 감각

쿠키는 서버가 `Set-Cookie` 응답 헤더로 브라우저에 저장을 요청하고,
브라우저가 이후 같은 서버 요청에 `Cookie` 헤더로 함께 보내는 작은 데이터다.

```http
# 서버 응답 — 쿠키 설정
HTTP/1.1 200 OK
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=3600

# 이후 브라우저 요청 — 자동으로 쿠키 포함
GET /api/profile HTTP/1.1
Cookie: sessionId=abc123
```

### 왜 이게 중요한가

이걸 이해해야:

- 세션 기반 인증이 어떻게 동작하는지
- 쿠키 속성 보안이 왜 중요한지
- 브라우저 자동 전송의 의미 (CSRF 위험의 근원)
- `localStorage`와의 차이를 설명할 수 있다

### 쿠키 vs localStorage vs sessionStorage

| 구분 | 쿠키 | localStorage | sessionStorage |
|------|------|-------------|----------------|
| 서버 전송 | 자동 (매 요청) | 수동만 가능 | 수동만 가능 |
| 용량 | 약 4KB | 약 5-10MB | 약 5-10MB |
| 만료 | `Max-Age`/`Expires` 설정 | 영구 (직접 삭제) | 탭/창 닫으면 삭제 |
| JS 접근 | `HttpOnly`면 불가 | 가능 | 가능 |
| 주 용도 | 인증, 서버 필요 데이터 | 클라이언트 캐시 | 임시 상태 |

### 자주 하는 오해

- 쿠키는 그냥 브라우저 저장소다 → 요청마다 자동 전송된다는 점이 핵심이다 (이것이 CSRF 공격이 가능한 이유이기도 하다)
- `localStorage`와 본질적으로 같다 → 서버 전송 여부, 보안 속성, 용량, 만료 정책이 다르다
- 쿠키엔 아무거나 담아도 된다 → 약 4KB 제한이 있고, 매 요청에 포함되므로 꼭 필요한 것만 담아야 한다

### 실무 연결

쿠키는 자주 아래 역할에 쓰인다.

- 세션 식별자 저장
- 로그인 상태 유지
- CSRF 관련 토큰 운용
- 언어/테마 같은 서버도 알아야 하는 값 저장

### 면접 압축 답변

**10초 답변:** HTTP는 stateless이므로 쿠키로 상태를 얹어 세션을 유지합니다. 쿠키는 매 요청에 자동 전송되는 점이 localStorage와 다릅니다.

**30초 답변:** HTTP는 본질적으로 stateless이기 때문에 로그인 상태 같은 문맥을 자동으로 기억하지 않습니다. 서버가 `Set-Cookie` 헤더로 브라우저에 쿠키를 저장하게 하고, 이후 요청마다 `Cookie` 헤더로 자동 전송되는 방식으로 세션을 유지합니다. localStorage와 달리 매 요청에 포함되므로 인증에 적합하지만, 자동 전송 특성이 CSRF 공격의 원인이 되기도 합니다.

> 이제 상태를 얹는 기본 도구인 쿠키를 봤다. 그다음엔 쿠키/세션/토큰이 각각 어떻게 다른지로 넘어가야 한다.

## 89. 쿠키, 세션, 토큰은 어떻게 구분해야 하나

이 세 개는 자주 한 덩어리로 섞여 말해진다.
하지만 역할을 분리해 보면 훨씬 선명해진다.

### 핵심 개념

#### 쿠키

브라우저 저장과 전송 메커니즘 (운반 수단)

#### 세션

서버 쪽에서 사용자 상태를 관리하는 개념 (서버 메모리/DB에 저장)

#### 토큰

클라이언트가 들고 다니는 인증/인가 정보 표현 방식 (JWT 등)

즉:

- 쿠키는 **저장/전송 수단** (무엇을 담느냐에 따라 역할이 달라짐)
- 세션은 **서버 상태 모델** (서버가 사용자 상태를 기억하고, 쿠키로 세션 ID만 전달)
- 토큰은 **인증 정보의 형식** (JWT 같은 자기 완결적 정보, 서버 조회 없이도 검증 가능)

에 더 가깝다.

### 세션 기반 vs 토큰 기반 인증 비교

```
[세션 기반]
클라이언트 → 로그인 요청 → 서버 (세션 생성, 세션 저장소에 보관)
          ← Set-Cookie: sessionId=abc123
클라이언트 → API 요청 (Cookie: sessionId=abc123) → 서버 (세션 저장소에서 abc123 조회)

[토큰 기반]
클라이언트 → 로그인 요청 → 서버 (JWT 생성, 서버에 저장하지 않음)
          ← { accessToken: "eyJ..." }
클라이언트 → API 요청 (Authorization: Bearer eyJ...) → 서버 (토큰 자체를 검증)
```

### 왜 이게 중요한가

이걸 구분해야 아래 오해를 줄일 수 있다.

- "쿠키 인증"과 "세션 인증"을 같은 말로 뭉개는 문제
- 토큰을 localStorage에만 둔다고 단정하는 문제 (쿠키에도 담을 수 있다)
- 브라우저 자동 전송(쿠키)과 수동 헤더 부착(Authorization) 차이를 놓치는 문제

### 자주 하는 오해

- 쿠키 = 세션이다 → 쿠키는 운반 수단, 세션은 서버 상태 모델이다
- 토큰 = localStorage다 → 토큰을 쿠키에 담는 것도 흔한 패턴이다
- 세션은 서버만, 토큰은 클라이언트만 관련 있다 → 세션 ID도 클라이언트(쿠키)에 저장되고, 토큰도 서버에서 생성/검증한다
- JWT는 안전하다 → JWT payload는 Base64 인코딩일 뿐 암호화가 아니다. 누구나 디코딩해서 읽을 수 있으므로 민감 정보를 넣으면 안 된다

### 실무 연결

예를 들어:

- 세션 기반 인증: 브라우저는 쿠키를 보내고, 서버는 세션 저장소를 조회 → 서버 스케일링 시 세션 공유가 과제
- 토큰 기반 인증: 클라이언트가 `Authorization` 헤더에 토큰을 실어 보냄 → 서버가 stateless하게 검증 가능, 스케일링에 유리
- 토큰도 쿠키에 담을 수 있음 → `HttpOnly` 쿠키에 JWT를 담으면 XSS 방어에 유리
- 쿠키도 인증 말고 언어/테마 같은 값 저장에 쓸 수 있음

### 면접 압축 답변

**10초 답변:** 쿠키는 저장/전송 수단, 세션은 서버 상태 모델, 토큰은 인증 정보 형식으로 서로 다른 층위의 개념입니다.

**30초 답변:** 쿠키는 브라우저 저장과 자동 전송 메커니즘이고, 세션은 서버가 사용자 상태를 기억하는 방식이며, 토큰은 JWT 같은 인증 정보 표현 형식입니다. 세션 기반은 서버가 상태를 관리하므로 스케일링 시 세션 공유가 과제이고, 토큰 기반은 서버가 stateless하게 검증할 수 있어 스케일링에 유리합니다. 셋을 구분해야 인증 설계에서 운반 수단, 상태 관리, 정보 형식을 각각 적절히 선택할 수 있습니다.

> 이제 저장 수단과 상태 모델을 구분했다. 그다음엔 보안적으로 자주 나오는 **쿠키 속성**과 인증 설계로 넘어가야 한다.

## 90. 인증 쿠키에서는 왜 Secure, HttpOnly, SameSite가 중요하나

인증 쿠키는 민감한 데이터를 운반하므로 보안 속성 설정이 매우 중요하다.

### 핵심 개념

#### `Secure`

HTTPS 요청에서만 전송되게 한다. HTTP에서는 쿠키가 전송되지 않으므로 네트워크 도청으로부터 보호된다.

#### `HttpOnly`

JavaScript(`document.cookie`)에서 쿠키를 읽지 못하게 한다. XSS 공격으로 쿠키가 탈취되는 것을 방지한다.

#### `SameSite`

크로스 사이트 요청과 함께 쿠키를 보낼지에 대한 정책을 제어한다. CSRF 방어의 핵심이다.

- `Strict`: 같은 사이트 요청에서만 전송. 가장 엄격
- `Lax`: 같은 사이트 + top-level navigation(링크 클릭 등)에서 전송. 대부분의 브라우저 기본값
- `None`: 크로스 사이트에서도 전송. 반드시 `Secure`와 함께 사용해야 함

### 실제 Set-Cookie 예시

```http
# 안전한 인증 쿠키 설정
Set-Cookie: sessionId=abc123; Secure; HttpOnly; SameSite=Lax; Max-Age=3600; Path=/

# 크로스 사이트에서도 필요한 경우 (결제 콜백 등)
Set-Cookie: token=xyz789; Secure; HttpOnly; SameSite=None; Max-Age=3600; Path=/
```

```js
// HttpOnly 쿠키는 JS에서 접근할 수 없다
console.log(document.cookie); // sessionId는 보이지 않음

// HttpOnly가 아닌 쿠키만 JS에서 접근 가능
// → 그래서 인증 쿠키는 반드시 HttpOnly를 설정해야 한다
```

### 왜 이게 중요한가

인증 쿠키는 민감하다.
보안 속성을 제대로 주지 않으면:

- `Secure` 미설정 → HTTP에서 쿠키가 평문 전송되어 도청 위험
- `HttpOnly` 미설정 → XSS 공격 시 `document.cookie`로 쿠키 탈취 가능
- `SameSite` 미설정 또는 `None` → CSRF 공격에 취약해질 수 있음

### 자주 하는 오해

- 쿠키는 저장만 잘되면 된다 → 전송 조건과 접근 제한이 더 중요할 수 있다
- `HttpOnly`면 모든 공격이 막힌다 → XSS를 통한 직접 쿠키 탈취는 막지만, XSS 자체나 CSRF는 별도 방어가 필요하다
- `Secure`만 있으면 끝이다 → `SameSite`, 서버 검증, CSRF 토큰 등 전체 인증 설계가 함께 가야 한다
- `SameSite=Lax`면 CSRF가 완전히 해결된다 → POST 요청은 차단되지만, GET으로 상태를 변경하는 API가 있다면 여전히 취약하다

### 면접 압축 답변

**10초 답변:** `Secure`는 HTTPS 전송만, `HttpOnly`는 JS 접근 차단, `SameSite`는 크로스 사이트 전송 정책을 제어해 쿠키 보안을 높입니다.

**30초 답변:** 인증 쿠키에서는 `Secure`, `HttpOnly`, `SameSite` 속성이 핵심입니다. `Secure`는 HTTPS 요청에서만 전송하게 해 도청을 방지하고, `HttpOnly`는 JavaScript 접근을 막아 XSS 쿠키 탈취를 방지하며, `SameSite`는 크로스 사이트 요청에서의 전송 정책을 제어해 CSRF 방어를 보완합니다. 이 세 속성을 조합하되, 서버 검증과 CSRF 토큰 등 전체 인증 설계와 함께 가야 합니다.

> 이제 쿠키 보안 속성을 봤다. 그다음엔 토큰 기반 인증에서 자주 나오는 **Access Token / Refresh Token** 구조로 넘어가야 한다.

## 91. Access Token과 Refresh Token은 왜 나눠 쓰나

이건 HTTP 표준 그 자체라기보다,
실무 인증 설계에서 매우 자주 나오는 패턴이다.

### 핵심 개념

#### Access Token

상대적으로 짧은 수명(보통 15분~1시간)으로 API 접근 권한을 증명

#### Refresh Token

새 Access Token을 발급받기 위한 더 장기적인 자격(보통 7일~30일)

### 왜 이게 중요한가

이렇게 나누는 이유는:

- Access Token이 짧은 수명이므로 탈취되더라도 피해 기간이 제한된다
- Refresh Token은 더 안전한 곳(HttpOnly 쿠키 등)에 보관하고, 서버에서 폐기(revoke)할 수 있다
- 사용자는 매번 로그인하지 않아도 세션이 유지된다

### 토큰 갱신 흐름 코드

```js
// fetch wrapper에서 401 감지 시 토큰 갱신
let isRefreshing = false;
let refreshSubscribers = [];

function onTokenRefreshed(newToken) {
  refreshSubscribers.forEach((callback) => callback(newToken));
  refreshSubscribers = [];
}

async function fetchWithAuth(url, options = {}) {
  let accessToken = getAccessToken();

  const res = await fetch(url, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${accessToken}`,
    },
  });

  if (res.status === 401) {
    // 이미 갱신 중이면 대기
    if (isRefreshing) {
      return new Promise((resolve) => {
        refreshSubscribers.push((newToken) => {
          resolve(fetchWithAuth(url, options));
        });
      });
    }

    isRefreshing = true;

    try {
      const refreshRes = await fetch('/api/auth/refresh', {
        method: 'POST',
        credentials: 'include', // HttpOnly 쿠키로 refresh token 전송
      });

      if (!refreshRes.ok) {
        // refresh도 실패하면 로그아웃
        redirectToLogin();
        return;
      }

      const { accessToken: newToken } = await refreshRes.json();
      setAccessToken(newToken);
      onTokenRefreshed(newToken);

      // 원래 요청 재시도
      return fetchWithAuth(url, options);
    } finally {
      isRefreshing = false;
    }
  }

  return res;
}
```

### 자주 하는 오해

- Refresh Token이 있으면 무조건 안전하다 → 보관 위치(HttpOnly 쿠키 권장)와 서버 검증(revocation)이 중요하다
- Access Token은 길게 잡을수록 UX가 좋다 → 탈취 시 피해 기간이 그만큼 길어진다
- 토큰 기반 인증이면 CSRF/XSS 고민이 사라진다 → `Authorization` 헤더를 사용하면 CSRF는 덜 위험하지만, 토큰을 쿠키에 담으면 CSRF 고려가 필요하고, XSS는 저장 방식과 무관하게 항상 방어해야 한다

### 실무 연결

프론트엔드에서는 주로 아래를 신경 쓴다.

- 토큰 만료 시 재발급 흐름 (interceptor 패턴)
- 동시에 여러 요청이 401을 받을 때 갱신 중복 방지 (위 코드의 `refreshSubscribers` 패턴)
- 만료 후 로그인 화면으로 보내는 UX
- Access Token 저장 위치: 메모리(가장 안전) > HttpOnly 쿠키 > localStorage(XSS 취약)

### 면접 압축 답변

**10초 답변:** 짧은 수명의 Access Token으로 API 접근을 처리하고, Refresh Token으로 재발급해 세션을 유지합니다. 핵심은 저장 위치와 갱신 흐름 설계입니다.

**30초 답변:** Access Token과 Refresh Token을 나누는 이유는 짧은 수명의 Access Token으로 탈취 시 피해를 제한하고, Refresh Token으로 세션 지속성을 보완하기 위해서입니다. 프론트에서는 401 감지 시 Refresh Token으로 재발급하는 interceptor를 구현하고, 동시 요청의 갱신 중복을 방지하며, Refresh Token은 HttpOnly 쿠키에 보관하는 것이 권장됩니다.

> 이제 인증 정보와 브라우저 전송 이야기를 봤다. 그다음엔 브라우저 보안 정책과 자주 충돌하는 **CORS**로 넘어가야 한다.

## 92. CORS는 왜 브라우저에서만 유독 문제처럼 느껴지나

CORS(Cross-Origin Resource Sharing)는 서버가 어떤 다른 origin에서의 접근을 허용할지 HTTP 헤더로 나타내는 메커니즘이다. 브라우저의 Same-Origin Policy 때문에 스크립트 기반 cross-origin 요청은 기본적으로 제한되며, 서버가 명시적으로 허용해야 한다.

### 핵심 개념

CORS는 서버가
"이 origin에서 오는 브라우저 요청을 허용할지"를 알려주는 규칙이다.

핵심 배경은 **Same-Origin Policy**다.

Origin은 **프로토콜 + 호스트 + 포트** 세 가지의 조합이다.

```
https://example.com:443  ← 이것이 하나의 origin
https://example.com:443/page  ← 같은 origin (경로는 무관)
http://example.com:443   ← 다른 origin (프로토콜 다름)
https://api.example.com:443  ← 다른 origin (호스트 다름)
```

브라우저는 보안 때문에
다른 origin으로의 스크립트 기반 요청(fetch, XMLHttpRequest)을 기본적으로 제한한다.
**서버(curl, Postman 등)에서는 Same-Origin Policy가 없으므로 CORS 제한이 없다.**

### 왜 이게 중요한가

프론트엔드에서 흔히 만나는 문제:

- 로컬 개발 서버(`localhost:3000`)에서 API(`localhost:8080`)로 요청이 막힘
- 인증 쿠키 포함 요청이 실패 (`credentials: 'include'` + 서버 설정 필요)
- 브라우저에서는 막히는데 Postman에서는 됨 (Postman은 브라우저가 아니므로 SOP 미적용)
- preflight 때문에 예상보다 요청이 더 나감

### CORS 관련 주요 헤더

```http
# 서버 응답에 포함하는 CORS 헤더
Access-Control-Allow-Origin: https://example.com   # 허용할 origin (* 가능하지만 credentials와 함께 쓸 수 없음)
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true              # 쿠키 포함 요청 허용
Access-Control-Max-Age: 86400                       # preflight 캐시 시간(초)
```

```js
// 프론트에서 credentials(쿠키) 포함 요청
fetch('https://api.example.com/data', {
  credentials: 'include',  // 쿠키를 함께 보내려면 필수
  headers: {
    'Content-Type': 'application/json',
  },
});
// → 서버에서 Access-Control-Allow-Credentials: true 와
//   Access-Control-Allow-Origin: https://example.com (와일드카드 * 불가)
//   를 반드시 설정해야 한다
```

### 자주 하는 오해

- CORS는 백엔드 서버 간 통신에도 동일하게 적용된다 → **아니다.** CORS는 브라우저의 Same-Origin Policy 위에서 동작하므로, 서버-서버 통신에는 CORS가 없다
- CORS 에러는 네트워크 실패다 → 서버는 정상 응답했지만 브라우저가 정책 위반으로 응답을 차단한 것이다
- 서버가 응답만 주면 프론트는 읽을 수 있다 → 적절한 `Access-Control-Allow-*` 헤더가 없으면 브라우저가 응답을 차단한다
- `Access-Control-Allow-Origin: *`이면 다 된다 → credentials(쿠키) 포함 요청에서는 와일드카드를 쓸 수 없다

### 면접 압축 답변

**10초 답변:** CORS는 브라우저의 Same-Origin Policy 위에서 서버가 다른 origin의 요청을 허용할지 HTTP 헤더로 알려주는 메커니즘입니다.

**30초 답변:** CORS는 브라우저의 Same-Origin Policy(프로토콜+호스트+포트 일치 정책) 위에서, 서버가 `Access-Control-Allow-Origin` 등 헤더로 다른 origin의 요청을 허용할지 알려주는 메커니즘입니다. 서버-서버 통신에는 SOP가 없으므로 CORS 제한이 없고, 그래서 Postman에선 되지만 브라우저에선 막히는 상황이 생깁니다. 쿠키 포함 요청은 와일드카드 origin을 쓸 수 없어 서버 설정이 더 엄격해집니다.

> 이제 CORS가 왜 필요한지는 봤다. 그다음엔 자주 같이 등장하는 **preflight**를 봐야 한다.

## 93. Preflight는 왜 생기고, 언제 발생하나

CORS를 이해했다면 다음 질문은 자연스럽다.

> **왜 실제 요청 전에 OPTIONS가 한 번 더 가죠?**

### 핵심 개념

Preflight는 브라우저가 실제 요청 전에
"이 메서드와 헤더 조합을 정말 허용하는지"를 먼저 묻는 사전 확인 요청이다.

`OPTIONS` 메서드로 전송된다.

### Simple Request vs Preflight Request

**Preflight가 발생하지 않는 조건 (Simple Request):**
- 메서드가 `GET`, `HEAD`, `POST` 중 하나
- 헤더가 `Accept`, `Accept-Language`, `Content-Language`, `Content-Type` 정도만 포함
- `Content-Type`이 `application/x-www-form-urlencoded`, `multipart/form-data`, `text/plain` 중 하나

**Preflight가 발생하는 조건:**
- `PUT`, `DELETE`, `PATCH` 등 Simple Request 조건 외 메서드
- `Authorization`, `Content-Type: application/json` 등 CORS-safelisted 범위 밖의 헤더

```http
# Preflight 요청 (브라우저가 자동으로 보냄)
OPTIONS /api/data HTTP/1.1
Origin: https://example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization

# Preflight 응답 (서버가 허용 여부를 알려줌)
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
```

### 왜 이게 중요한가

이걸 알아야:

- 왜 네트워크 탭에 요청이 두 번 보이는지 (OPTIONS + 실제 요청)
- 왜 커스텀 헤더(`Authorization`, `Content-Type: application/json`)를 추가하자마자 CORS 문제가 생겼는지
- 왜 인증 포함 요청에서 서버 설정이 더 까다로워지는지

설명할 수 있다.

### 자주 하는 오해

- Preflight는 fetch 버그다 → 아니다, 브라우저 보안 메커니즘이다
- 요청이 두 번 간 건 React StrictMode 때문이다 → StrictMode의 이중 실행은 개발 모드에서 useEffect 등에 적용되고, 네트워크 탭의 OPTIONS는 preflight다. 구분해야 한다
- Preflight는 모든 cross-origin 요청에서 항상 발생한다 → Simple Request 조건을 충족하면 발생하지 않는다

### 실무 연결

Preflight 부담을 줄이려면:

- `Access-Control-Max-Age`로 preflight 응답을 캐시한다 (브라우저가 캐시 기간 동안 OPTIONS를 재전송하지 않음)
- 가능하다면 Simple Request 조건을 충족하도록 설계한다 (하지만 `Authorization` 헤더 사용 시 피하기 어렵다)
- 서버에서 OPTIONS 핸들러를 빠르게 응답하도록 구성한다

### 면접 압축 답변

**10초 답변:** Preflight는 브라우저가 cross-origin 요청 전에 OPTIONS로 허용 여부를 확인하는 과정이며, 메서드/헤더 조건에 따라 발생합니다.

**30초 답변:** Preflight는 브라우저가 실제 cross-origin 요청 전에 `OPTIONS` 요청으로 서버의 허용 여부를 먼저 확인하는 과정입니다. GET/HEAD/POST + 기본 헤더의 Simple Request에서는 발생하지 않지만, `Authorization` 헤더나 `Content-Type: application/json`, PUT/DELETE 같은 메서드를 사용하면 발생합니다. `Access-Control-Max-Age`로 preflight 캐시를 설정해 성능 부담을 줄일 수 있습니다.

> 이제 cross-origin 보안 정책을 봤다. 그다음엔 요청을 아예 덜 보내거나 더 싸게 만드는 **캐시**로 넘어가야 한다.

## 94. HTTP 캐시는 왜 가장 기본적인 성능 최적화인가

HTTP 캐시는 불필요한 네트워크 요청을 줄여 로드 성능을 개선하는 가장 효과적인 방법 중 하나다. 핵심 헤더로 `Cache-Control`, `ETag`, `Last-Modified`가 있다.

### 핵심 개념

HTTP 캐시는
이미 받은 응답을 다시 활용해 불필요한 네트워크 요청을 줄이는 방식이다.

캐시의 두 가지 큰 흐름:

1. **강한 캐시 (Fresh)** — `Cache-Control: max-age` 기간 내라면 서버에 묻지도 않고 캐시 사용
2. **조건부 요청 (Stale + Revalidation)** — 만료 후 `ETag`/`Last-Modified`로 서버에 "바뀌었나?" 확인 → 304면 캐시 재사용

### 왜 이게 중요한가

프론트엔드 성능 최적화에서 캐시는 생각보다 큰 영향을 준다.

왜냐하면:

- 같은 JS/CSS/이미지를 반복 다운로드하지 않게 하고
- 재방문 체감을 크게 개선하고
- origin 요청 수를 줄이며
- CDN과 함께 쓰면 효과가 더 커지기 때문이다

### 캐시 동작 흐름

```
[첫 요청]
브라우저 → 서버
         ← 200 OK + Cache-Control: max-age=3600 + ETag: "abc123"
         (브라우저가 응답을 캐시에 저장)

[3600초 이내 재요청 — 강한 캐시]
브라우저 → (서버에 요청하지 않음, 캐시에서 바로 사용)
         ← 200 OK (from disk cache)

[3600초 이후 재요청 — 조건부 요청]
브라우저 → If-None-Match: "abc123" → 서버
         ← 304 Not Modified (변경 없으므로 body 없이 응답)
         (브라우저가 기존 캐시를 재사용)
```

### 자주 하는 오해

- 캐시는 브라우저가 알아서 잘한다 → 서버 헤더가 매우 중요하다. 헤더가 없으면 브라우저의 휴리스틱 캐시에 의존하게 되어 예측 불가능하다
- 캐시는 정적 파일에만 의미 있다 → HTML/API 응답에도 정책이 중요할 수 있다
- `no-cache`는 "절대 캐시하지 말라"는 뜻이다 → **아니다.** `no-cache`는 "캐시에 저장하되, 사용 전에 반드시 서버에 재검증하라"는 뜻이다. 절대 저장하지 말라는 것은 `no-store`다

### 면접 압축 답변

**10초 답변:** HTTP 캐시는 이미 받은 응답을 재사용해 네트워크 요청을 줄이는 기본 성능 최적화이며, 서버의 `Cache-Control` 헤더가 핵심입니다.

**30초 답변:** HTTP 캐시는 이미 받은 응답을 재사용해 불필요한 네트워크 요청을 줄이는 기본 성능 최적화입니다. `max-age` 기간 내에는 서버에 묻지 않고 캐시를 사용하고(강한 캐시), 만료 후에는 `ETag`나 `Last-Modified`로 재검증해 304 응답이면 기존 캐시를 재사용합니다. `no-cache`는 저장 금지가 아니라 매번 재검증하라는 뜻이고, 저장 금지는 `no-store`입니다.

> 이제 캐시의 존재 이유를 봤다. 그다음엔 캐시 정책을 실제로 표현하는 **Cache-Control** 지시자를 자세히 봐야 한다.

## 95. Cache-Control, ETag, Last-Modified는 어떻게 같이 이해해야 하나

캐시는 "저장할까 말까"만의 문제가 아니다.
더 정확히는:

- 얼마나 오래 쓸까
- 만료 뒤 어떻게 재검증할까
- 누구까지 저장해도 될까

의 문제다.

### 핵심 개념

#### `Cache-Control`

캐시 정책의 중심 헤더. 주요 지시자:

| 지시자 | 의미 |
|--------|------|
| `no-store` | 아예 저장하지 않는다. 민감한 응답에 사용 |
| `no-cache` | 저장하되, 사용 전 반드시 서버에 재검증. 이름이 오해를 부른다 |
| `private` | 브라우저 캐시에만 저장. CDN 등 공유 캐시에 저장 금지 |
| `public` | 공유 캐시(CDN 등)에도 저장 가능 |
| `max-age=N` | N초 동안 신선(fresh)한 것으로 간주. 서버에 묻지 않고 사용 |
| `s-maxage=N` | 공유 캐시(CDN)에 대한 max-age. `max-age`보다 우선 |
| `must-revalidate` | 만료 후 반드시 서버에 재검증. 오프라인에서 stale 캐시 사용 금지 |
| `immutable` | max-age 내에서 새로고침(F5)에도 재검증하지 않는다 |

#### `ETag`

리소스 버전을 식별하는 토큰. 만료 후 재검증 시 `If-None-Match` 헤더로 비교에 사용된다.

#### `Last-Modified`

마지막 변경 시각 기반 재검증 힌트. `If-Modified-Since` 헤더로 비교에 사용된다. `ETag`보다 정밀도가 낮다(초 단위).

### 실무에서 흔한 캐시 전략

```http
# 1. 해시 파일명의 정적 자산 (main.a1b2c3.js)
# → 내용이 바뀌면 파일명이 바뀌므로 오래 캐시해도 안전
Cache-Control: public, max-age=31536000, immutable

# 2. HTML (index.html)
# → 항상 최신 버전을 확인해야 하므로 매번 재검증
Cache-Control: no-cache
ETag: "v1.2.3"

# 3. 인증된 사용자 API 응답
# → CDN에 저장하면 안 되고, 브라우저만 짧게 캐시
Cache-Control: private, max-age=60

# 4. 민감한 응답 (결제 정보 등)
# → 어디에도 저장하면 안 됨
Cache-Control: no-store
```

### 자주 하는 오해

- `no-cache`는 저장 금지다 → 실제로는 "저장하되 사용 전 반드시 재검증"이라는 뜻이다. 저장 금지는 `no-store`
- `private`면 보안이 완벽하다 → 캐시 저장 범위 제한이지 데이터 암호화가 아니다
- ETag가 있으면 Cache-Control은 필요 없다 → Cache-Control이 freshness 정책을 정하고, ETag는 재검증에 쓰인다. 역할이 다르다
- `max-age=0`과 `no-cache`는 같다 → 비슷하지만 `no-cache`가 더 명시적이고, 일부 엣지 케이스에서 동작이 다를 수 있다

### 실무 연결

정적 자산은 보통:

- 해시 파일명 (`main.a1b2c3.js`)
- 긴 `max-age` + `immutable`
전략과 잘 맞는다. (Next.js, Vite 등 빌드 도구가 자동으로 해시 파일명을 생성한다)

반면 인증 페이지나 사용자 맞춤 응답은:

- `private` (CDN 캐시 방지)
- `no-store` (아예 저장 금지)
- `Vary: Cookie` (쿠키에 따라 다른 캐시)
같은 정책을 더 신중히 봐야 한다.

### 면접 압축 답변

**10초 답변:** `Cache-Control`은 저장/재사용 정책, `ETag`/`Last-Modified`는 만료 후 재검증 힌트를 담당합니다. `no-cache`는 저장 금지가 아니라 매번 재검증입니다.

**30초 답변:** `Cache-Control`은 캐시 저장과 재사용 정책의 중심이고, `ETag`와 `Last-Modified`는 만료 후 재검증에 사용됩니다. 해시 파일명의 정적 자산에는 `public, max-age=31536000, immutable`이 적합하고, HTML에는 `no-cache`로 매번 재검증하며, 사용자 맞춤 응답에는 `private`이나 `no-store`를 써야 합니다. `no-cache`는 이름과 달리 저장 금지가 아니라 매번 재검증하라는 뜻이므로 주의가 필요합니다.

> 이제 브라우저 캐시 정책을 봤다. 그다음엔 이 캐시를 지리적으로 더 가까운 곳에서 제공하는 **CDN**으로 넘어가야 한다.

## 96. CDN은 왜 단순 파일 저장소가 아니라 네트워크 성능 계층인가

CDN(Content Delivery Network)은 사용자 가까이에 캐시된 콘텐츠를 제공하는 지리적으로 분산된 서버 네트워크다. 정적 자산뿐 아니라 HTML, JS, CSS, 이미지, 비디오 전달 성능과 원본 서버 부담 감소에 기여한다.

### 핵심 개념

CDN은 사용자와 origin 서버 사이에 있는
지리적으로 분산된 전달 계층으로 이해하면 좋다.

핵심 역할은:

- 사용자 가까운 edge에서 응답 제공 (지연 시간 감소)
- origin 요청 수 감소 (origin 보호)
- 대규모 트래픽 분산 (DDoS 완화에도 도움)
- TLS termination을 edge에서 처리 (연결 수립 지연 감소)

### CDN 동작 흐름

```
[첫 사용자 요청 — Cache Miss]
사용자(한국) → CDN Edge(한국) → Origin(미국)
                              ← 응답 + Cache-Control: public, max-age=86400
             ← 응답 (edge가 캐시에 저장)

[같은 리소스에 대한 후속 요청 — Cache Hit]
사용자(한국) → CDN Edge(한국)
             ← 캐시된 응답 (origin에 요청하지 않음)
```

### 왜 이게 중요한가

CDN을 이해하면 아래가 연결된다.

- 왜 해외 사용자도 더 빠르게 받을 수 있는가 (가까운 edge에서 응답)
- 왜 origin 부하가 줄어드는가 (edge 캐시 히트)
- 왜 캐시와 CDN을 같이 설명해야 하는가 (`Cache-Control`의 `public`, `s-maxage`가 CDN 동작을 제어)
- 왜 정적 자산 전달 비용과 성능이 함께 좋아질 수 있는가

### 자주 하는 오해

- CDN은 대형 글로벌 서비스만 쓴다 → Vercel, Netlify, CloudFront 등 현대 배포 플랫폼은 기본으로 CDN을 사용한다
- CDN은 이미지 저장소다 → HTML, JS, CSS, 폰트, 심지어 API 응답 일부까지도 전략에 따라 캐시할 수 있다
- CDN이 있으면 origin 성능은 신경 안 써도 된다 → cache miss, 동적 콘텐츠, purge 시점에는 origin 성능이 여전히 중요하다

### 실무 연결

프론트엔드에서 CDN 감각은 아래와 연결된다.

- 이미지 최적화 (CDN의 이미지 리사이즈/포맷 변환 기능)
- 정적 빌드 산출물 배포 (Next.js, Vite 등의 빌드 결과물)
- 웹폰트 배포 (Google Fonts 등이 CDN을 통해 제공됨)
- 지역 사용자 체감 속도 개선
- 트래픽 급증 시 origin 보호

### 면접 압축 답변

**10초 답변:** CDN은 사용자 가까운 edge 서버에서 콘텐츠를 제공해 지연 시간과 origin 부담을 줄이는 네트워크 성능 계층입니다.

**30초 답변:** CDN은 사용자 가까운 edge 서버에서 콘텐츠를 제공해 지연 시간을 줄이고 origin 부담을 줄이는 전달 계층입니다. `Cache-Control: public, s-maxage` 같은 헤더로 CDN 캐시 정책을 제어하며, 첫 요청 시 origin에서 가져와 edge에 캐시한 뒤 후속 요청은 edge에서 직접 응답합니다. Vercel, Netlify 같은 현대 배포 플랫폼은 기본으로 CDN을 사용하므로 프론트엔드에서도 자연스럽게 접하게 됩니다.

> 이제 브라우저 캐시와 CDN까지 봤다. 그다음엔 이 둘을 어떻게 나눠 생각해야 하는지 정리해야 한다.

## 97. 브라우저 캐시와 CDN 캐시는 왜 함께 설명해야 하나

많은 사람이 캐시를 "브라우저에 저장되나?" 정도로만 생각한다.
하지만 실무에서는 캐시 층이 여러 개일 수 있다.

### 핵심 개념

아주 단순하게 보면:

```
사용자 요청
  ↓
[1] 브라우저 캐시 (private, 사용자 기기)
  ↓ cache miss
[2] CDN 캐시 (public/shared, edge 서버)
  ↓ cache miss
[3] Origin 서버
```

- 브라우저 캐시: 사용자 기기 가까이 (해당 사용자만 사용)
- CDN 캐시: 네트워크 edge 가까이 (같은 지역의 여러 사용자가 공유)
- origin: 원본 서버

즉 같은 리소스라도
어디에서 재사용되는지 층이 다를 수 있다.

### Cache-Control과 캐시 층의 관계

```http
# 브라우저만 캐시 (CDN은 저장하지 않음)
Cache-Control: private, max-age=60

# 브라우저 + CDN 모두 캐시
Cache-Control: public, max-age=300, s-maxage=3600
# → 브라우저는 5분, CDN은 1시간 캐시

# 아무 곳에도 캐시하지 않음
Cache-Control: no-store
```

### 왜 이게 중요한가

이걸 이해해야:

- 왜 어떤 응답은 내 브라우저는 다시 안 받지만 다른 사용자는 CDN에서 받을 수 있는지
- 왜 `private`와 `public`이 중요한지 (개인화 응답이 CDN에 캐시되면 다른 사용자에게 노출될 수 있다)
- 왜 사용자 맞춤 응답은 CDN 캐시 정책을 조심해야 하는지

설명할 수 있다.

### 자주 하는 오해

- 캐시는 다 똑같다 → 캐시 위치(브라우저/CDN/프록시)와 범위(private/public)가 다르다
- 브라우저 캐시만 잡으면 충분하다 → 글로벌 사용자에겐 CDN edge 캐시도 중요하다
- CDN 캐시를 많이 쓰면 사용자별 응답도 다 빨라진다 → **위험하다.** 개인화 응답(마이페이지 등)이 `public`으로 CDN에 캐시되면 다른 사용자에게 해당 정보가 노출될 수 있다

### 실무 연결

예:

- 버전된 정적 JS/CSS/이미지 → `public, max-age=31536000, immutable` (브라우저 + CDN 모두 캐시)
- 로그인 후 개인 대시보드 HTML → `private, no-cache` 또는 `no-store` (CDN 캐시 금지)
- 공개 API 응답 (상품 목록 등) → `public, s-maxage=60` (CDN에서 1분 캐시)
- 인증된 사용자 API 응답 → `private, max-age=0` (CDN 저장 금지, 매번 origin 확인)

### 면접 압축 답변

**10초 답변:** 브라우저 캐시는 사용자 기기, CDN 캐시는 edge 서버에서 동작하며, `private`/`public`으로 캐시 층을 구분해야 개인화 응답 노출을 방지할 수 있습니다.

**30초 답변:** 브라우저 캐시는 사용자 기기 근처에서 해당 사용자만 재사용하고, CDN 캐시는 edge 서버에서 여러 사용자 요청을 처리합니다. `private`은 브라우저만, `public`은 CDN에도 캐시를 허용하며, `s-maxage`로 CDN 캐시 시간을 별도 지정할 수 있습니다. 개인화 응답이 `public`으로 CDN에 캐시되면 다른 사용자에게 노출될 수 있으므로, 공개 자산과 개인화 응답을 구분해 정책을 다르게 잡아야 합니다.

## 97-1. 실시간 통신 — WebSocket, SSE, Polling

웹 애플리케이션에서 서버와 실시간으로 데이터를 주고받아야 할 때가 있다.
알림, 채팅, 실시간 대시보드 같은 기능이 대표적이다.
이때 선택지는 크게 세 가지다.

### 핵심 개념

**Polling (폴링)**

가장 단순한 방식이다. 클라이언트가 일정 간격으로 서버에 HTTP 요청을 보내서 새 데이터가 있는지 확인한다.

- 구현이 단순하고 기존 HTTP 인프라 그대로 사용 가능
- 새 데이터가 없어도 요청을 보내므로 **낭비가 크다**
- 간격이 짧으면 서버 부담, 길면 실시간성 저하

```tsx
// Polling 예시
function usePollData(url: string, interval = 5000) {
  const [data, setData] = useState(null);

  useEffect(() => {
    const fetchData = async () => {
      const res = await fetch(url);
      const json = await res.json();
      setData(json);
    };

    fetchData(); // 초기 요청
    const id = setInterval(fetchData, interval);
    return () => clearInterval(id);
  }, [url, interval]);

  return data;
}
```

**SSE (Server-Sent Events)**

서버에서 클라이언트로의 **단방향 푸시**를 HTTP 위에서 제공한다.

- `EventSource` API를 사용하며, `text/event-stream` 형식으로 응답
- 자동 재연결(auto-reconnect) 내장
- 텍스트 데이터만 전송 가능 (바이너리 불가)
- HTTP/2와 잘 맞고, 기존 인프라(프록시, 로드밸런서)와 호환이 좋다

```tsx
// SSE 예시: EventSource API
useEffect(() => {
  const eventSource = new EventSource("/api/notifications");

  eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    setNotifications((prev) => [data, ...prev]);
  };

  eventSource.onerror = () => {
    // 브라우저가 자동으로 재연결을 시도한다
    console.log("SSE 연결 끊김 — 자동 재연결 중...");
  };

  return () => eventSource.close();
}, []);
```

**WebSocket**

클라이언트와 서버 사이의 **양방향(full-duplex) 통신** 채널을 제공한다.

- `ws://` 또는 `wss://` 프로토콜 사용
- 한 번 연결(handshake)하면 이후 오버헤드가 매우 적다
- 텍스트와 바이너리 데이터 모두 전송 가능
- 자동 재연결이 없으므로 직접 구현해야 한다

```tsx
// WebSocket 기본 사용 예시
useEffect(() => {
  const ws = new WebSocket("wss://api.example.com/chat");

  ws.onopen = () => {
    console.log("WebSocket 연결됨");
    ws.send(JSON.stringify({ type: "join", room: "general" }));
  };

  ws.onmessage = (event) => {
    const message = JSON.parse(event.data);
    setMessages((prev) => [...prev, message]);
  };

  ws.onclose = () => {
    console.log("WebSocket 연결 종료 — 재연결 필요");
    // 수동 재연결 로직 필요
  };

  return () => ws.close();
}, []);
```

### 비교 표

| 항목 | Polling | SSE | WebSocket |
|------|---------|-----|-----------|
| 방향 | 클라이언트 → 서버 (요청/응답) | 서버 → 클라이언트 (단방향) | 양방향 (full-duplex) |
| 프로토콜 | HTTP | HTTP | `ws://` / `wss://` |
| 자동 재연결 | 직접 구현 | 내장 | 직접 구현 |
| 바이너리 지원 | X | X (텍스트만) | O |
| 오버헤드 | 높음 (매번 HTTP 요청) | 낮음 (연결 유지) | 매우 낮음 (handshake 이후) |
| 인프라 호환성 | 최고 | 좋음 | 별도 설정 필요할 수 있음 |
| 적합한 사례 | 단순 대시보드, 낮은 빈도 업데이트 | 알림, 피드, 주가 표시 | 채팅, 게이밍, 실시간 협업 |

### 언제 무엇을 쓸 것인가

- **알림, 뉴스피드, 실시간 주가** → SSE: 서버에서 클라이언트로 보내기만 하면 되고, 자동 재연결이 편리하다
- **채팅, 게이밍, 실시간 협업 편집** → WebSocket: 양방향 통신이 필수이고, 바이너리 데이터(음성, 파일)도 보내야 할 수 있다
- **단순 대시보드, 업데이트 빈도가 낮은 경우** → Polling: 구현이 가장 단순하고, 기존 REST API 그대로 활용할 수 있다

### 자주 하는 오해

- 실시간이면 무조건 WebSocket이다 → 서버→클라이언트 단방향이면 SSE가 더 간단하고 충분하다
- Polling은 쓸모없다 → 업데이트 빈도가 낮고 구현 단순성이 중요하면 여전히 유효하다
- WebSocket은 HTTP보다 항상 빠르다 → handshake 후에는 오버헤드가 적지만, 짧은 요청/응답에는 HTTP가 더 적합할 수 있다

### 면접 압축 답변

**10초**: Polling은 주기적 HTTP 요청, SSE는 서버→클라이언트 단방향 푸시, WebSocket은 양방향 통신이며, 용도에 따라 선택합니다.

**30초**: Polling은 일정 간격으로 HTTP 요청을 보내는 가장 단순한 방식이지만 낭비가 큽니다. SSE는 `EventSource` API를 통해 서버에서 클라이언트로 단방향 푸시를 제공하며 자동 재연결이 내장되어 있습니다. WebSocket은 `ws://` 프로토콜로 양방향 통신을 지원하고 바이너리 데이터도 보낼 수 있지만 재연결은 직접 구현해야 합니다. 알림/피드에는 SSE, 채팅/게이밍에는 WebSocket, 단순 대시보드에는 Polling이 적합합니다.

## 97-2. ESM vs CJS — 모듈 시스템은 왜 중요한가

자바스크립트의 모듈 시스템은 코드를 어떻게 나누고 불러오는지를 정하는 근본 구조다.
현재 두 가지 주요 모듈 시스템이 공존하고 있다.

### 핵심 개념

**ESM (ECMAScript Modules)**

- `import` / `export` 문법 사용
- **정적 분석** 가능: import 경로가 빌드 타임에 확정된다
- **Tree-shakable**: 사용하지 않는 export를 제거할 수 있다
- 비동기 로딩: `import()` 동적 import 지원
- strict mode가 기본 적용
- 브라우저에서 네이티브 지원 (`<script type="module">`)

```tsx
// ESM 문법
import { useState, useEffect } from "react";
import utils from "./utils.js";

export function greet(name) {
  return `Hello, ${name}`;
}

export default function App() {
  return <div>App</div>;
}

// 동적 import (코드 스플리팅에 사용)
const module = await import("./heavyModule.js");
```

**CJS (CommonJS)**

- `require()` / `module.exports` 문법 사용
- **동적**: require 경로가 런타임에 결정될 수 있다
- **Tree-shaking 불가**: 어떤 export가 사용되는지 빌드 타임에 알 수 없다
- 동기 로딩: `require()`는 파일을 동기적으로 읽는다
- Node.js의 기본 모듈 시스템 (Node.js가 ESM도 지원하지만, 레거시 코드 대부분이 CJS)

```js
// CJS 문법
const { useState, useEffect } = require("react");
const utils = require("./utils");

function greet(name) {
  return `Hello, ${name}`;
}

module.exports = { greet };
module.exports.default = function App() {
  return "App";
};
```

### 왜 이게 중요한가 — 번들링과 성능

이 구분이 중요한 핵심 이유는 **번들 크기와 성능**에 직접적 영향을 주기 때문이다.

| 항목 | ESM | CJS |
|------|-----|-----|
| Tree-shaking | O (정적 분석 가능) | X (동적이라 분석 불가) |
| 코드 스플리팅 | `import()` 네이티브 지원 | 별도 처리 필요 |
| 번들 크기 | 사용하지 않는 코드 제거 가능 | 전체 모듈 포함될 가능성 높음 |
| 브라우저 지원 | 네이티브 (`<script type="module">`) | 번들러 필수 |
| 순환 참조 처리 | 라이브 바인딩 (참조를 공유) | 값 복사 (순환 시 빈 객체 위험) |

**Tree-shaking 예시**:

```tsx
// ESM — lodash-es 사용 시 get만 번들에 포함됨
import { get } from "lodash-es";
const value = get(obj, "a.b.c");

// CJS — lodash 전체가 번들에 포함될 수 있음
const { get } = require("lodash");
const value = get(obj, "a.b.c");
```

Webpack, Vite, Rollup 같은 번들러는 ESM 기반 코드에서 사용하지 않는 export를 제거(tree-shaking)할 수 있다. CJS는 `require()`가 런타임에 동적으로 결정되므로 빌드 타임에 어떤 코드가 실제로 사용되는지 판단하기 어렵다.

### 자주 하는 오해

- `import`와 `require`는 단순 문법 차이다 → 번들 크기와 최적화 가능 여부에 직접적 차이가 있다
- Node.js에서는 CJS만 쓸 수 있다 → `"type": "module"` 설정이나 `.mjs` 확장자로 ESM 사용 가능
- ESM이면 무조건 tree-shaking 된다 → 사이드 이펙트가 있는 모듈은 번들러가 제거하지 못할 수 있다 (`sideEffects` 필드 확인 필요)
- 둘을 섞어 쓰면 문제 없다 → CJS에서 ESM을 require하거나, ESM에서 CJS를 import할 때 호환 문제가 생길 수 있다

### 실무 연결

- 새 프로젝트는 ESM 기반으로 시작하는 것이 좋다
- 라이브러리를 선택할 때 ESM 지원 여부를 확인한다 (lodash → lodash-es, moment → date-fns 등)
- `package.json`의 `"type": "module"`과 `"exports"` 필드를 이해해야 한다
- Next.js, Vite 등 현대 프레임워크/번들러는 ESM을 기본으로 사용한다

### 면접 압축 답변

**10초**: ESM은 `import`/`export` 기반으로 정적 분석과 tree-shaking이 가능하고, CJS는 `require`/`module.exports` 기반으로 동적이지만 tree-shaking이 어렵습니다.

**30초**: ESM은 `import`/`export` 문법을 사용하며 정적 분석이 가능해서 번들러가 사용하지 않는 코드를 제거(tree-shaking)할 수 있습니다. CJS는 `require()`가 런타임에 동적으로 결정되므로 tree-shaking이 어렵고 번들 크기가 커질 수 있습니다. 현대 프론트엔드 도구는 ESM을 기본으로 사용하며, 라이브러리 선택 시에도 ESM 지원 여부가 번들 최적화에 중요합니다.

## 98. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 브라우저 요청은 DNS부터 시작한다
- 주소를 찾은 뒤 TCP 연결과 TLS 보안 계층을 준비한다
- HTTPS는 TLS 위의 HTTP다
- HTTP는 요청/응답 메시지 기반 프로토콜이다
- 메서드와 멱등성은 retry 전략과 연결된다
- 상태코드는 UI 복구 전략의 입력값이다
- HTTP는 stateless이므로 쿠키 같은 장치로 상태를 얹는다
- 쿠키, 세션, 토큰은 서로 다른 층위 개념이다
- 인증 쿠키 속성(Secure, HttpOnly, SameSite)은 보안 설계의 핵심이다
- Access Token / Refresh Token 분리는 보안과 UX의 균형이다
- CORS와 preflight는 브라우저 Same-Origin Policy 위의 보안 메커니즘이다
- HTTP 캐시는 요청을 줄이는 기본 성능 전략이다
- CDN은 사용자 가까운 곳에서 전달과 캐시를 최적화한다
- 브라우저 캐시와 CDN 캐시는 private/public으로 구분해 함께 설계해야 한다

즉,
이번 챕터는 단순 네트워크 용어 암기가 아니라,
**"사용자의 한 번의 요청이 어떻게 해석되고, 보호되고, 인증되고, 재사용되고, 더 빠르게 전달되는가"**를 배우는 챕터다.
