# 챕터 7. 브라우저 요청 흐름에서 인증과 캐시, CDN까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **브라우저 요청은 단순히 fetch 한 번이 아니라, DNS 조회, 연결 수립, 보안 계층, HTTP 메시지, 인증, 캐시, CDN까지 이어지는 하나의 흐름으로 이해해야 한다.**

---

## 81. 브라우저에서 URL을 입력하면 어떤 큰 흐름이 시작되나

프론트엔드 면접에서 네트워크 질문이 나오면 많은 사람이 곧바로 HTTP 메서드나 상태코드부터 말한다.  
그런데 더 좋은 출발점은 이거다.

> **사용자가 URL을 입력한 뒤 페이지가 보이기까지, 브라우저는 어떤 단계를 거치는가?**

### 큰 흐름

아주 크게 보면 보통 이런 순서다.

1. URL 해석
2. DNS 조회
3. 서버와 연결 수립
4. HTTPS라면 TLS handshake
5. HTTP 요청 전송
6. 서버 응답 수신
7. 캐시 여부 판단
8. HTML 파싱과 추가 리소스 요청
9. CSS/JS/이미지 등 하위 자원 로딩
10. 렌더링

### 왜 이게 중요한가

이 큰 그림이 있어야 아래 질문들이 전부 연결된다.

- DNS는 왜 필요한가
- HTTP와 HTTPS 차이는 무엇인가
- TLS는 왜 필요한가
- 상태코드는 어디서 오나
- 캐시는 언제 이득이 되나
- CDN은 어디에 끼는가

### 자주 하는 오해

- 브라우저 요청은 그냥 서버에 바로 간다 → 실제로는 여러 단계가 있다
- HTTP만 알면 네트워크는 충분하다 → DNS, 연결, 캐시, 보안도 함께 봐야 한다
- 프론트엔드는 요청만 보내면 된다 → 사용자 체감 성능은 요청 전후 전체 흐름의 영향을 받는다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 큰 흐름을 봤다.  
그다음엔 “도메인 이름을 실제 서버 주소로 어떻게 바꾸는가”라는 질문으로 넘어가야 한다.

즉 **DNS**다.

### 면접 압축 답변

브라우저에서 URL을 입력하면 단순히 서버로 바로 요청을 보내는 것이 아니라, DNS 조회로 IP를 찾고, 연결과 보안 계층을 준비한 뒤 HTTP 요청/응답을 주고받고, 이후 하위 리소스를 추가로 요청해 최종 렌더링까지 이어지는 흐름이 시작됩니다.

## 82. DNS는 왜 웹 요청의 첫 관문인가

사용자는 `example.com` 같은 도메인을 입력하지만,  
브라우저가 실제로 통신하려면 IP 주소가 필요하다.  
DNS는 바로 이 변환을 담당한다. DNS는 사람이 읽기 쉬운 도메인 이름을 IP 주소로 바꾸는 시스템으로 설명된다. :contentReference[oaicite:1]{index=1}

### 핵심 개념

DNS는 인터넷의 전화번호부 같은 역할을 한다.

- 사람은 도메인 이름을 기억한다
- 기계는 IP 주소로 통신한다
- DNS가 둘 사이를 연결한다

### 왜 이게 중요한가

이걸 이해해야 아래가 자연스럽다.

- 왜 도메인만 알아서는 실제 요청을 못 보내는가
- 왜 DNS 지연도 성능 문제인가
- 왜 CDN edge나 origin 서버도 결국 DNS 뒤에서 선택될 수 있는가

### DNS 조회를 아주 단순하게 보면

일반적인 흐름은 아래처럼 이해하면 된다.

1. 브라우저/OS가 먼저 캐시를 본다
2. 없으면 DNS resolver에 묻는다
3. resolver가 필요한 정보를 따라가며 IP를 찾는다
4. 찾은 IP를 브라우저에 돌려준다
5. 브라우저는 그 IP로 연결을 시도한다

### 자주 하는 오해

- DNS는 한 서버가 전부 해결한다 → 여러 단계와 캐시가 얽힌다
- DNS는 한 번만 일어나고 끝이다 → TTL과 캐시에 따라 재조회될 수 있다
- DNS는 백엔드 인프라 이야기라 프론트엔드와 무관하다 → 초기 응답 지연과 직결된다

### 실무 연결

웹 성능에서 DNS lookup 시간은 첫 요청 체감에 영향을 준다.  
특히 외부 스크립트, 써드파티 폰트, 이미지 CDN, 분석 스크립트가 많아질수록 DNS 의존 지점도 늘어난다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 서버 주소를 찾았다.  
그다음엔 실제로 그 서버와 **연결을 어떻게 맺는가**를 봐야 한다.

### 면접 압축 답변

DNS는 사람이 사용하는 도메인 이름을 브라우저가 통신에 필요한 IP 주소로 바꾸는 시스템입니다. 그래서 웹 요청의 첫 단계에 가깝고, DNS 지연이나 캐시 전략도 초기 로딩 성능에 영향을 줍니다.

## 83. DNS 다음에는 왜 연결 수립이 필요하나

IP를 알았다고 해서 바로 HTTP를 보낼 수 있는 건 아니다.  
브라우저와 서버 사이에는 실제 통신 경로를 준비하는 단계가 필요하다.

### 핵심 개념

HTTP는 애플리케이션 계층 프로토콜이고,  
그 아래에서 실제 데이터 전송을 위한 연결 계층이 받쳐줘야 한다.

실무 감각에서는 보통 이렇게 이해하면 충분하다.

- 주소를 찾는 단계: DNS
- 통신 길을 여는 단계: 연결 수립
- 그 위에서 요청/응답을 주고받는 단계: HTTP

### 왜 이게 중요한가

이 감각이 있어야 아래가 이해된다.

- 왜 첫 요청은 느리고 후속 요청은 상대적으로 빠를 수 있는가
- 왜 keep-alive나 connection reuse가 성능에 영향을 주는가
- 왜 TLS handshake가 RTT 비용과 연결되는가

### 자주 하는 오해

- DNS가 끝나면 바로 응답이 온다 → 연결 수립이 더 필요하다
- 연결 비용은 무시해도 된다 → 첫 방문 체감 성능에 영향이 크다
- HTTP가 네트워크 전부다 → 실제론 아래 계층 위에서 동작한다

### 실무 연결

처음 방문한 사이트가 느리게 느껴지는 이유에는:

- DNS
- 연결 수립
- TLS handshake
- 서버 응답 시간
  이 모두 섞여 있다.

그래서 “API가 느리다”라고 말할 때도 실제로는 어느 단계가 느린지 분해해서 봐야 한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 연결 수립이 필요하다는 걸 봤다.  
그다음엔 HTTPS라면 여기에 추가되는 **TLS**를 봐야 한다.

### 면접 압축 답변

DNS로 IP를 찾은 뒤에는 실제 통신을 위한 연결 수립 단계가 필요합니다. 즉 주소를 안다고 곧바로 HTTP 메시지를 주고받는 게 아니라, 먼저 브라우저와 서버 사이의 통신 경로를 준비한 뒤 그 위에서 요청과 응답이 오갑니다.

## 84. HTTPS와 TLS는 왜 함께 설명해야 하나

HTTP와 HTTPS를 단순히 “s가 붙었냐 안 붙었냐”로 설명하면 약하다.  
핵심은 HTTPS가 **TLS 위에서 보호되는 HTTP**라는 점이다. TLS는 인터넷 통신의 프라이버시와 데이터 무결성을 제공하는 널리 쓰이는 보안 프로토콜로 설명된다. :contentReference[oaicite:2]{index=2}

### 핵심 개념

- HTTP는 웹 리소스를 요청/응답하는 프로토콜이다
- HTTPS는 그 HTTP 통신을 TLS로 보호한 형태다
- TLS는 통신 내용을 암호화하고, 서버 신원을 확인하며, 전송 중 변조 위험을 줄이는 역할을 한다

### 왜 이게 중요한가

이걸 이해해야 아래 질문을 제대로 설명할 수 있다.

- 왜 로그인 페이지는 HTTPS가 필수인가
- 왜 쿠키에 `Secure` 속성이 중요한가
- 왜 중간에서 내용을 엿보거나 바꾸기 어려워지는가

### TLS handshake를 아주 거칠게 보면

HTTPS 연결에서는 요청 전에 대략 이런 일이 일어난다.

- 사용할 TLS 버전과 암호 스위트 합의
- 서버 인증서 확인
- 세션 키 합의
- 이후 HTTP를 암호화해 전송

### 자주 하는 오해

- HTTPS면 무조건 완벽하게 안전하다 → 다른 취약점은 여전히 존재할 수 있다
- TLS는 느려서 피해야 한다 → 현대 웹에선 기본 전제다
- HTTPS는 백엔드만 신경 쓰면 된다 → 프론트 쿠키, mixed content, 보안 전제와 연결된다

### 실무 연결

프론트엔드 관점에서도 HTTPS/TLS 감각은 중요하다.

- `Secure` 쿠키
- 서비스 워커
- 브라우저 API 일부 제약
- mixed content 차단
- OAuth/인증 콜백 안정성

같은 것들이 전부 HTTPS 전제와 연결되기 때문이다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 안전한 연결 위에서 통신할 수 있게 됐다.  
그다음엔 그 위를 흐르는 **HTTP 메시지** 자체를 봐야 한다.

### 면접 압축 답변

HTTPS는 단순히 HTTP에 s가 붙은 것이 아니라, TLS로 보호되는 HTTP 통신입니다. TLS는 서버 신원 확인과 통신 암호화, 데이터 무결성 보호를 담당하므로 로그인, 세션, 쿠키 보안까지 프론트엔드 관점에서도 매우 중요합니다.

## 85. HTTP는 정확히 어떤 프로토콜인가

HTTP는 웹에서 리소스를 가져오기 위한 애플리케이션 계층 프로토콜이며, 요청과 응답 메시지를 교환하는 클라이언트-서버 프로토콜로 설명된다. 또한 본질적으로 stateless지만 cookies를 통해 상태를 얹을 수 있다고 MDN은 정리한다. :contentReference[oaicite:3]{index=3}

### 핵심 개념

HTTP를 이해할 때는 아래를 같이 봐야 한다.

- 클라이언트가 요청을 보낸다
- 서버가 응답을 돌려준다
- 메시지는 method, path, headers, body 같은 구조를 가진다
- 응답은 status code, headers, body를 가진다

### 왜 이게 중요한가

이걸 알아야:

- REST API를 읽고
- 브라우저 devtools 네트워크 탭을 해석하고
- 캐시 헤더를 이해하고
- 인증 흐름을 설명할 수 있다

### 요청 메시지 감각

요청은 보통 이런 요소를 가진다.

- 메서드 (`GET`, `POST`, `PUT`, `DELETE`, `OPTIONS` 등)
- 경로
- 헤더
- 경우에 따라 body

### 응답 메시지 감각

응답은 보통 이런 요소를 가진다.

- 상태코드 (`200`, `404`, `500` 등)
- 헤더
- body

### 자주 하는 오해

- HTTP는 JSON 전송 프로토콜이다 → JSON은 흔한 payload 형식일 뿐이다
- body는 항상 있다 → 메서드와 상황에 따라 다르다
- 상태코드만 보면 충분하다 → 헤더와 body 의미도 중요하다

### 예시

```http
GET /products?page=2 HTTP/1.1
Host: example.com
Accept: application/json
Authorization: Bearer some-token
```

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: private, max-age=60

{"items":[...]}
```

### 여기서 다음 개념으로 어떻게 넘어가나

이제 HTTP 메시지 구조를 봤다.  
그다음엔 요청 의도를 가장 먼저 드러내는 **HTTP 메서드**로 넘어가야 한다.

### 면접 압축 답변

HTTP는 웹 리소스를 주고받기 위한 클라이언트-서버 프로토콜입니다. 요청은 method, path, headers, body 같은 구조를 가지고, 응답은 status code, headers, body를 가집니다. 프론트엔드에서는 이 구조를 알아야 네트워크 탭과 API 동작을 제대로 해석할 수 있습니다.

## 86. HTTP 메서드와 멱등성은 왜 retry 설계와 연결되나

메서드는 단순 문법이 아니다.  
클라이언트가 서버에 어떤 의도를 가지고 요청하는지를 드러낸다.

### 핵심 개념

대표 메서드는 보통 이렇게 이해하면 된다.

- `GET`: 조회
- `POST`: 생성/처리 요청
- `PUT`: 전체 갱신 느낌
- `PATCH`: 부분 수정 느낌
- `DELETE`: 삭제
- `OPTIONS`: 가능 메서드/사전 확인

여기에 중요한 개념이 **멱등성**이다.

멱등성은 같은 요청을 여러 번 보내도 결과가 본질적으로 같아야 하는 성질을 말한다.

### 왜 이게 중요한가

이걸 이해해야:

- 어떤 요청은 재시도해도 상대적으로 안전하고
- 어떤 요청은 중복 수행 위험이 큰지
  판단할 수 있다.

예를 들어:

- `GET`은 보통 재시도에 더 안전한 편이다
- 결제 같은 `POST`는 중복 실행 위험이 있을 수 있다

### 자주 하는 오해

- `POST`는 무조건 위험하고 `GET`은 무조건 안전하다 → 서버 구현까지 같이 봐야 한다
- 메서드는 이름만 다를 뿐 의미 차이는 별로 없다 → API 설계와 재시도 정책에 중요하다
- 멱등성은 백엔드만 신경 쓰는 문제다 → 프론트의 retry/backoff와 바로 연결된다

### 실무 연결

프론트엔드에서:

- 실패한 조회 재시도
- 저장 버튼 중복 클릭 방지
- mutation 중복 요청 방지
- optimistic update rollback
  을 설계할 때 멱등성 감각이 필요하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 요청 의도는 봤다.  
그다음엔 서버가 결과를 어떻게 표현하는지, 즉 **상태코드**로 넘어가야 한다.

### 면접 압축 답변

HTTP 메서드는 단순 이름이 아니라 요청 의도를 나타내고, 멱등성은 같은 요청을 여러 번 보내도 결과가 본질적으로 같은지를 뜻합니다. 이 개념을 알아야 어떤 요청을 재시도해도 되는지, 어떤 요청은 중복 실행을 더 조심해야 하는지 판단할 수 있습니다.

---

## 87. 상태코드는 왜 단순 숫자가 아니라 UI 설계 신호인가

많은 학습자가 상태코드를 “200 성공, 404 없음, 500 서버 에러” 정도로만 외운다.  
그런데 프론트엔드 실무에서는 상태코드가 **UI와 복구 전략의 입력값**이 된다.

### 핵심 개념

대표 범위는 이렇게 보면 된다.

- `2xx`: 성공
- `3xx`: 리다이렉션
- `4xx`: 클라이언트 측 요청 문제
- `5xx`: 서버 측 오류

### 왜 이게 중요한가

상태코드에 따라 UI 반응이 달라져야 한다.

예:

- `401` → 로그인 필요
- `403` → 권한 없음
- `404` → 리소스 없음
- `409` → 충돌
- `429` → 너무 많은 요청, 재시도 간격 고려
- `500` → 서버 오류 fallback

### 자주 하는 오해

- 에러는 전부 하나의 토스트로 처리하면 된다 → 사용자 경험이 거칠어진다
- 4xx, 5xx는 둘 다 그냥 실패다 → 복구 전략이 다르다
- fetch는 상태코드가 4xx/5xx면 자동 throw 한다 → 기본적으로는 네트워크 오류와 다르다

### 실무 연결

좋은 프론트는:

- 인증 만료인지
- 권한 부족인지
- 존재하지 않는 데이터인지
- 잠시 후 다시 시도해야 하는지
  를 구분해 다른 UI를 보여줄 수 있어야 한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 요청/응답의 기본 구조를 봤다.  
그다음엔 HTTP가 본질적으로 stateless라는 점, 그리고 그 위에 상태를 얹는 **쿠키**로 넘어가야 한다.

### 면접 압축 답변

상태코드는 단순 숫자가 아니라 UI와 복구 전략의 신호입니다. 예를 들어 401은 재로그인, 403은 권한 부족, 404는 없음, 429는 재시도 간격 고려처럼 다르게 반응해야 하므로 프론트엔드에서도 의미를 나눠 해석하는 것이 중요합니다.

---

## 88. HTTP가 stateless라면 로그인 상태는 어떻게 유지하나

HTTP 자체는 요청과 요청 사이의 문맥을 기본적으로 기억하지 않는 stateless 프로토콜로 설명된다. 다만 쿠키를 통해 상태를 얹어 세션을 유지할 수 있다. :contentReference[oaicite:4]{index=4}

### 핵심 개념

stateless라는 건:

- 이전 요청을 자동으로 기억하지 않는다는 뜻이다

그래서 로그인 상태, 장바구니, 사용자 선호 같은 문맥을 유지하려면  
별도의 상태 유지 장치가 필요하다.

대표적인 것이 쿠키다.

### 쿠키 기본 감각

쿠키는 서버가 브라우저에 저장을 요청하고,  
브라우저가 이후 같은 서버 요청에 함께 보내는 작은 데이터다. MDN은 쿠키가 서버가 브라우저에 보내는 작은 데이터이며, 이후 동일 서버 요청과 함께 다시 전송될 수 있다고 설명한다. :contentReference[oaicite:5]{index=5}

### 왜 이게 중요한가

이걸 이해해야:

- 세션 기반 인증
- 쿠키 속성 보안
- 브라우저 자동 전송
- localStorage와의 차이
  를 설명할 수 있다.

### 자주 하는 오해

- 쿠키는 그냥 브라우저 저장소다 → 요청마다 자동 전송된다는 점이 중요하다
- localStorage와 본질적으로 같다 → 서버 전송 여부가 다르다
- 쿠키엔 아무거나 담아도 된다 → 크기와 보안 제약이 있다

### 실무 연결

쿠키는 자주 아래 역할에 쓰인다.

- 세션 식별자 저장
- 로그인 상태 유지
- CSRF 관련 토큰 운용
- 언어/테마 같은 서버도 알아야 하는 값 저장

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태를 얹는 기본 도구인 쿠키를 봤다.  
그다음엔 쿠키/세션/토큰이 각각 어떻게 다른지로 넘어가야 한다.

### 면접 압축 답변

HTTP는 본질적으로 stateless이기 때문에 로그인 상태 같은 문맥을 자동으로 기억하지 않습니다. 그래서 서버가 브라우저에 쿠키를 저장하게 하고, 이후 요청마다 그 쿠키를 함께 보내는 방식으로 세션 같은 상태를 유지할 수 있습니다.

---

## 89. 쿠키, 세션, 토큰은 어떻게 구분해야 하나

이 세 개는 자주 한 덩어리로 섞여 말해진다.  
하지만 역할을 분리해 보면 훨씬 선명해진다.

### 핵심 개념

#### 쿠키

브라우저 저장과 전송 메커니즘

#### 세션

서버 쪽에서 사용자 상태를 관리하는 개념

#### 토큰

클라이언트가 들고 다니는 인증/인가 정보 표현 방식

즉:

- 쿠키는 저장/전송 수단
- 세션은 서버 상태 모델
- 토큰은 인증 정보의 형식
  에 더 가깝다

### 왜 이게 중요한가

이걸 구분해야 아래 오해를 줄일 수 있다.

- “쿠키 인증”과 “세션 인증”을 같은 말로 뭉개는 문제
- 토큰을 localStorage에만 둔다고 단정하는 문제
- 브라우저 자동 전송과 수동 헤더 부착 차이를 놓치는 문제

### 자주 하는 오해

- 쿠키 = 세션이다 → 틀림
- 토큰 = localStorage다 → 틀림
- 세션은 서버만, 토큰은 클라이언트만 관련 있다 → 틀림

### 실무 연결

예를 들어:

- 세션 기반 인증: 브라우저는 쿠키를 보내고, 서버는 세션 저장소를 조회
- 토큰 기반 인증: 클라이언트가 `Authorization` 헤더 등에 토큰을 실어 보냄
- 토큰도 쿠키에 담을 수 있음
- 쿠키도 인증 말고 언어/테마 같은 값 저장에 쓸 수 있음

### 여기서 다음 개념으로 어떻게 넘어가나

이제 저장 수단과 상태 모델을 구분했다.  
그다음엔 보안적으로 자주 나오는 **쿠키 속성**과 인증 설계로 넘어가야 한다.

### 면접 압축 답변

쿠키는 브라우저 저장과 전송 메커니즘이고, 세션은 서버가 사용자 상태를 기억하는 방식이며, 토큰은 인증 정보를 표현하는 형식입니다. 그래서 셋은 같은 개념이 아니라 서로 다른 층위의 개념으로 구분해서 이해하는 게 중요합니다.

---

## 90. 인증 쿠키에서는 왜 Secure, HttpOnly, SameSite가 중요하나

MDN은 `Secure` 쿠키는 HTTPS 요청에만 전송되고, `HttpOnly` 쿠키는 JavaScript에서 접근할 수 없으며, 이런 속성들이 쿠키 보안을 높이는 핵심이라고 설명한다. :contentReference[oaicite:6]{index=6}

### 핵심 개념

#### `Secure`

HTTPS 요청에서만 전송되게 한다

#### `HttpOnly`

JavaScript에서 쿠키를 읽지 못하게 한다

#### `SameSite`

크로스 사이트 요청과 함께 쿠키를 보낼지에 대한 정책을 제어한다

### 왜 이게 중요한가

인증 쿠키는 민감하다.  
보안 속성을 제대로 주지 않으면:

- 노출 위험이 커지고
- 스크립트 접근 위험이 생기고
- CSRF 방어가 약해질 수 있다

### 자주 하는 오해

- 쿠키는 저장만 잘되면 된다 → 전송 조건과 접근 제한이 더 중요할 수 있다
- `HttpOnly`면 모든 공격이 막힌다 → 완전한 해결책은 아니다
- `Secure`만 있으면 끝이다 → `SameSite`, 서버 검증, 전체 인증 설계가 함께 가야 한다

### 실무 연결

보통 인증용 쿠키는:

- HTTPS 전용
- JS 접근 차단
- 크로스 사이트 전송 정책 명확화
  를 함께 고려한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 쿠키 보안 속성을 봤다.  
그다음엔 토큰 기반 인증에서 자주 나오는 **Access Token / Refresh Token** 구조로 넘어갈 수 있다.

### 면접 압축 답변

인증 쿠키에서는 `Secure`, `HttpOnly`, `SameSite` 같은 속성이 중요합니다. `Secure`는 HTTPS 전송만 허용하고, `HttpOnly`는 JavaScript 접근을 막으며, `SameSite`는 크로스 사이트 요청에서의 전송 정책을 제어해 보안 설계를 보완합니다.

---

## 91. Access Token과 Refresh Token은 왜 나눠 쓰나

이건 HTTP 표준 그 자체라기보다,  
실무 인증 설계에서 매우 자주 나오는 패턴이다.

### 핵심 개념

#### Access Token

상대적으로 짧은 수명으로 API 접근 권한을 증명

#### Refresh Token

새 Access Token을 발급받기 위한 더 장기적인 자격

### 왜 이게 중요한가

이렇게 나누는 이유는:

- 짧은 수명의 access token으로 노출 시 피해 범위를 줄이고
- refresh token으로 사용자 경험을 유지하기 위해서다

### 자주 하는 오해

- refresh token이 있으면 무조건 안전하다 → 보관 위치와 서버 검증이 중요하다
- access token은 길게 잡을수록 UX가 좋다 → 노출 시 위험 범위가 커질 수 있다
- 토큰 기반 인증이면 CSRF/XSS 고민이 사라진다 → 여전히 저장 위치와 전송 방식이 중요하다

### 실무 연결

프론트엔드에서는 주로 아래를 신경 쓴다.

- 토큰 만료 시 재발급 흐름
- 동시에 여러 요청이 401을 받을 때 갱신 중복 방지
- 만료 후 로그인 화면으로 보내는 UX
- interceptor / fetch wrapper 설계

### 여기서 다음 개념으로 어떻게 넘어가나

이제 인증 정보와 브라우저 전송 이야기를 봤다.  
그다음엔 브라우저 보안 정책과 자주 충돌하는 **CORS**로 넘어가야 한다.

### 면접 압축 답변

Access Token과 Refresh Token을 나누는 이유는 짧은 수명의 access token으로 API 접근을 처리하고, refresh token으로 세션 지속성을 보완하기 위해서입니다. 다만 핵심은 단순 분리가 아니라 저장 위치, 만료 처리, 재발급 흐름까지 함께 설계하는 것입니다.

---

## 92. CORS는 왜 브라우저에서만 유독 문제처럼 느껴지나

MDN은 CORS를 서버가 어떤 다른 origin에서의 접근을 허용할지 HTTP 헤더로 나타내는 메커니즘으로 설명하며, 브라우저의 same-origin policy 때문에 스크립트 기반 cross-origin 요청은 기본적으로 제한된다고 설명한다. 또한 특정 메서드/헤더 조합에서는 preflight 요청이 먼저 간다고 정리한다. :contentReference[oaicite:7]{index=7}

### 핵심 개념

CORS는 서버가  
“이 origin에서 오는 브라우저 요청을 허용할지”를 알려주는 규칙이다.

핵심 배경은 **same-origin policy**다.

브라우저는 보안 때문에  
다른 origin으로의 스크립트 기반 요청을 기본적으로 제한한다.

### 왜 이게 중요한가

프론트엔드에서 흔히 만나는 문제:

- 로컬 개발 서버에서 API 요청이 막힘
- 인증 쿠키 포함 요청이 실패
- 브라우저에서는 막히는데 Postman에서는 됨
- preflight 때문에 예상보다 요청이 더 나감

이 모든 걸 설명할 수 있어야 한다.

### 자주 하는 오해

- CORS는 백엔드 서버 간 통신에도 동일하게 적용된다 → 브라우저 문맥이 핵심이다
- CORS 에러는 네트워크 실패다 → 브라우저 보안 정책 차단일 수 있다
- 서버가 응답만 주면 프론트는 읽을 수 있다 → 적절한 헤더가 필요하다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 CORS가 왜 필요한지는 봤다.  
그다음엔 자주 같이 등장하는 **preflight**를 봐야 한다.

### 면접 압축 답변

CORS는 브라우저의 same-origin policy 위에서 서버가 어떤 다른 origin의 요청을 허용할지 HTTP 헤더로 알려주는 메커니즘입니다. 그래서 브라우저에서는 막히지만 Postman에선 되는 상황이 생길 수 있고, 프론트엔드 개발에서 자주 문제처럼 느껴집니다.

---

## 93. Preflight는 왜 생기고, 언제 발생하나

CORS를 이해했다면 다음 질문은 자연스럽다.

> **왜 실제 요청 전에 OPTIONS가 한 번 더 가죠?**

### 핵심 개념

preflight는 브라우저가 실제 요청 전에  
“이 메서드와 헤더 조합을 정말 허용하는지”를 먼저 묻는 사전 확인 요청이다.

보통 `OPTIONS` 메서드로 간다.

### 왜 이게 중요한가

이걸 알아야:

- 왜 네트워크 탭에 요청이 두 번 보이는지
- 왜 커스텀 헤더를 추가하자마자 CORS 문제가 생겼는지
- 왜 인증 포함 요청에서 서버 설정이 더 까다로워지는지
  설명할 수 있다.

### 자주 하는 오해

- preflight는 fetch 버그다 → 아니다
- 요청이 두 번 간 건 React StrictMode 때문이다 → 경우에 따라 구분해야 한다
- preflight는 항상 발생한다 → 요청 조건에 따라 다르다

### 실무 연결

preflight가 자주 생기면:

- 지연이 늘 수 있고
- 서버 설정이 복잡해질 수 있으며
- 개발 환경에서 오해를 부르기 쉽다

그래서 어떤 헤더와 메서드를 쓰는지, credentials가 있는지까지 같이 본다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 cross-origin 보안 정책을 봤다.  
그다음엔 요청을 아예 덜 보내거나 더 싸게 만드는 **캐시**로 넘어가야 한다.

### 면접 압축 답변

Preflight는 브라우저가 실제 cross-origin 요청 전에 `OPTIONS` 요청으로 허용 여부를 먼저 확인하는 과정입니다. 메서드나 헤더 조합에 따라 발생하며, 네트워크 지연과 서버 설정 복잡도에도 영향을 줄 수 있습니다.

## 94. HTTP 캐시는 왜 가장 기본적인 성능 최적화인가

web.dev는 HTTP 캐시가 불필요한 네트워크 요청을 줄여 로드 성능을 개선하는 효과적인 방법이며, 핵심 헤더로 `Cache-Control`, `ETag`, `Last-Modified`를 꼽는다. 또한 `private`, `public`, `no-store`, `no-cache` 같은 지시자를 어떻게 쓸지 설명한다. :contentReference[oaicite:8]{index=8}

### 핵심 개념

HTTP 캐시는  
이미 받은 응답을 다시 활용해 불필요한 네트워크 요청을 줄이는 방식이다.

핵심 헤더는 보통 아래를 같이 본다.

- `Cache-Control`
- `ETag`
- `Last-Modified`

### 왜 이게 중요한가

프론트엔드 성능 최적화에서 캐시는 생각보다 큰 영향을 준다.

왜냐하면:

- 같은 JS/CSS/이미지를 반복 다운로드하지 않게 하고
- 재방문 체감을 크게 개선하고
- origin 요청 수를 줄이며
- CDN과 함께 쓰면 효과가 더 커지기 때문이다

### 자주 하는 오해

- 캐시는 브라우저가 알아서 잘한다 → 서버 헤더가 매우 중요하다
- 캐시는 정적 파일에만 의미 있다 → HTML/API 응답에도 정책이 중요할 수 있다
- `no-cache`는 절대 캐시하지 말라는 뜻이다 → 이름 때문에 자주 오해한다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 캐시의 존재 이유를 봤다.  
그다음엔 캐시 정책을 실제로 표현하는 **Cache-Control**로 넘어가야 한다.

### 면접 압축 답변

HTTP 캐시는 이미 받은 응답을 재사용해 불필요한 네트워크 요청을 줄이는 기본 성능 최적화입니다. 핵심은 브라우저가 알아서 잘하는 게 아니라, 서버가 `Cache-Control`, `ETag`, `Last-Modified` 같은 헤더로 재사용 정책을 어떻게 주느냐에 달려 있다는 점입니다.

---

## 95. Cache-Control, ETag, Last-Modified는 어떻게 같이 이해해야 하나

캐시는 “저장할까 말까”만의 문제가 아니다.  
더 정확히는:

- 얼마나 오래 쓸까
- 만료 뒤 어떻게 재검증할까
- 누구까지 저장해도 될까

의 문제다.

### 핵심 개념

#### `Cache-Control`

캐시 정책의 중심 헤더

자주 보는 값:

- `no-store`
- `no-cache`
- `private`
- `public`
- `max-age=...`

#### `ETag`

리소스 버전을 식별하는 토큰 같은 역할  
만료 후 재검증 시 비교에 사용될 수 있다

#### `Last-Modified`

마지막 변경 시각 기반 재검증 힌트

### 왜 이게 중요한가

이걸 알면 아래 구분이 쉬워진다.

- 버전이 붙은 정적 JS 파일 → 길게 캐시 가능
- 로그인한 사용자 HTML → `private` 고려
- 민감한 응답 → `no-store` 고려
- 자주 안 바뀌는 이미지 → 긴 캐시 + 파일명 버전 관리

### 자주 하는 오해

- `no-cache`는 저장 금지다 → 실제론 재검증 후 사용 가능이라는 뜻에 더 가깝다
- `private`면 보안이 완벽하다 → 캐시 범위 제한이지 전체 보안 해결책은 아니다
- ETag가 있으면 Cache-Control은 필요 없다 → 역할이 다르다

### 실무 연결

정적 자산은 보통:

- 해시 파일명
- 긴 `max-age`
  전략과 잘 맞는다.

반면 인증 페이지나 사용자 맞춤 응답은:

- `private`
- `no-store`
- `Vary`
  같은 정책을 더 신중히 봐야 한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 브라우저 캐시 정책을 봤다.  
그다음엔 이 캐시를 지리적으로 더 가까운 곳에서 제공하는 **CDN**으로 넘어가야 한다.

### 면접 압축 답변

`Cache-Control`은 캐시 저장과 재사용 정책을, `ETag`와 `Last-Modified`는 만료 후 재검증 힌트를 담당합니다. 그래서 정적 자산엔 긴 캐시와 해시 파일명이 잘 맞고, 사용자 맞춤 응답엔 `private`나 `no-store` 같은 정책을 더 신중하게 써야 합니다.

---

## 96. CDN은 왜 단순 파일 저장소가 아니라 네트워크 성능 계층인가

Cloudflare는 CDN을 사용자 가까이에 캐시된 콘텐츠를 제공하는 지리적으로 분산된 서버 집합으로 설명하며, 정적 자산뿐 아니라 HTML, JS, CSS, 이미지, 비디오 전달 성능과 원본 서버 부담 감소에 기여한다고 설명한다. :contentReference[oaicite:9]{index=9}

### 핵심 개념

CDN은 사용자와 origin 서버 사이에 있는  
지리적으로 분산된 전달 계층으로 이해하면 좋다.

핵심 역할은:

- 사용자 가까운 edge에서 응답 제공
- origin 요청 수 감소
- 지연 시간 감소
- 대규모 트래픽 분산

### 왜 이게 중요한가

CDN을 이해하면 아래가 연결된다.

- 왜 해외 사용자도 더 빠르게 받을 수 있는가
- 왜 origin 부하가 줄어드는가
- 왜 캐시와 CDN을 같이 설명해야 하는가
- 왜 정적 자산 전달 비용과 성능이 함께 좋아질 수 있는가

### 자주 하는 오해

- CDN은 대형 글로벌 서비스만 쓴다 → 규모와 상관없이 유용할 수 있다
- CDN은 이미지 저장소다 → HTML, JS, CSS, API 응답 일부까지도 전략에 따라 다를 수 있다
- CDN이 있으면 origin 성능은 신경 안 써도 된다 → 여전히 중요하다

### 실무 연결

프론트엔드에서 CDN 감각은 아래와 연결된다.

- 이미지 최적화
- 정적 빌드 산출물 배포
- 폰트 배포
- 지역 사용자 체감 속도 개선
- 트래픽 급증 시 origin 보호

### 여기서 다음 개념으로 어떻게 넘어가나

이제 브라우저 캐시와 CDN까지 봤다.  
그다음엔 이 둘을 어떻게 나눠 생각해야 하는지 정리해야 한다.

### 면접 압축 답변

CDN은 사용자 가까운 edge 서버에서 콘텐츠를 제공해 지연 시간을 줄이고 origin 부담을 줄이는 전달 계층입니다. 따라서 단순 저장소가 아니라 캐시와 전송 경로를 최적화하는 네트워크 성능 계층으로 이해하는 게 더 정확합니다.

## 97. 브라우저 캐시와 CDN 캐시는 왜 함께 설명해야 하나

많은 사람이 캐시를 “브라우저에 저장되나?” 정도로만 생각한다.  
하지만 실무에서는 캐시 층이 여러 개일 수 있다.

### 핵심 개념

아주 단순하게 보면:

- 브라우저 캐시: 사용자 기기 가까이
- CDN 캐시: 네트워크 edge 가까이
- origin: 원본 서버

즉 같은 리소스라도  
어디에서 재사용되는지 층이 다를 수 있다.

### 왜 이게 중요한가

이걸 이해해야:

- 왜 어떤 응답은 내 브라우저는 다시 안 받지만 다른 사용자는 CDN에서 받을 수 있는지
- 왜 `private`와 `public`이 중요한지
- 왜 사용자 맞춤 응답은 CDN 캐시 정책을 조심해야 하는지
  설명할 수 있다.

### 자주 하는 오해

- 캐시는 다 똑같다 → 캐시 위치와 범위가 다르다
- 브라우저 캐시만 잡으면 충분하다 → 글로벌 사용자에겐 CDN도 중요하다
- CDN 캐시를 많이 쓰면 사용자별 응답도 다 빨라진다 → 개인정보/개인화 응답은 신중해야 한다

### 실무 연결

예:

- 버전된 정적 JS/CSS/이미지 → 브라우저 + CDN 캐시 모두 활용하기 좋음
- 로그인 후 개인 대시보드 HTML → 훨씬 더 신중한 캐시 정책 필요
- API 응답 → 공개 응답인지 개인 응답인지 구분이 중요

### 여기서 다음 개념으로 어떻게 넘어가나

이제 요청 흐름, 보안, 인증, 캐시, CDN이 하나의 구조로 연결됐다.  
그다음엔 전체를 다시 요약해 면접형 사고로 정리해야 한다.

### 면접 압축 답변

브라우저 캐시는 사용자 기기 근처에서 재사용되고, CDN 캐시는 edge 서버에서 여러 사용자 요청을 더 가깝게 처리할 수 있습니다. 따라서 캐시는 한 층이 아니라 여러 층으로 존재할 수 있고, 공개 자산과 개인화 응답을 구분해 정책을 다르게 잡아야 합니다.

## 98. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 브라우저 요청은 DNS부터 시작한다
- 주소를 찾은 뒤 연결과 보안 계층을 준비한다
- HTTPS는 TLS 위의 HTTP다
- HTTP는 요청/응답 메시지 기반 프로토콜이다
- 메서드와 멱등성은 retry 전략과 연결된다
- 상태코드는 UI 복구 전략의 입력값이다
- HTTP는 stateless이므로 쿠키 같은 장치로 상태를 얹는다
- 쿠키, 세션, 토큰은 서로 다른 층위 개념이다
- 인증 쿠키 속성은 보안 설계의 핵심이다
- CORS와 preflight는 브라우저 보안 정책이다
- HTTP 캐시는 요청을 줄이는 기본 성능 전략이다
- CDN은 사용자 가까운 곳에서 전달과 캐시를 최적화한다
- 브라우저 캐시와 CDN 캐시는 함께 설계해야 한다

즉,  
이번 챕터는 단순 네트워크 용어 암기가 아니라,  
**“사용자의 한 번의 요청이 어떻게 해석되고, 보호되고, 인증되고, 재사용되고, 더 빠르게 전달되는가”**를 배우는 챕터다.
