# React Closure Learning Flow

이 문서는 단순 면접 답변 암기용 문답집이 아니라, **한 개념이 왜 다음 개념을 필요하게 만드는지**를 따라가며 학습하고, 마지막에 면접 답변으로 압축할 수 있게 만든 학습형 플로우 문서다.

### 핵심 원칙

1. 질문을 독립적으로 외우지 않는다.
2. **선행 개념 → 원리 → 오해 교정 → 실무 연결 → 다음 개념** 순서로 읽는다.
3. 마지막에만 10초 / 30초 / 1분 답변으로 압축한다.
4. 한 문항을 이해했으면 반드시 다음 문항으로 이어지는 다리를 만든다.

### 이 문서를 읽는 순서

1. 각 섹션의 **핵심 요약**을 먼저 읽는다.
2. 본문에서 **왜 이 개념이 필요한가**를 이해한다.
3. **자주 하는 오해**와 **실무 에피소드**를 확인한다.
4. 마지막에 **면접 압축 답변**을 말로 꺼내 본다.
5. 막히면 다시 위 설명으로 돌아가 원리부터 복구한다.

### 전체 흐름 맵

```
var/let/const (스코프)
  → 실행 컨텍스트 (변수가 사는 작업 공간)
    → 렉시컬 스코프 (외부 변수를 찾는 기준)
      → 클로저 (환경을 기억하는 함수)
        → for 루프 클로저 (스코프 + 클로저의 합류점)
          → 클로저 오용 (메모리 누수, 오래된 참조)
            → React state snapshot (렌더마다 새 값)
              → setState batching (왜 즉시 안 바뀌나)
                → stale closure (오래된 렌더의 값을 봄)
                  → useEffect dependency (정리와 재실행 계약)
                    → ref (렌더 무관 최신값 보관함)
                      → useMemo / useCallback (참조 안정성)
```

# 챕터 1. 실행 컨텍스트에서 stale closure까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **자바스크립트 함수가 값을 어떻게 기억하는지 이해하면, React에서 값이 왜 오래된 채로 남는지도 설명할 수 있다.**

---

## 1. var / let / const를 먼저 보는 이유

> **핵심 요약** — 클로저를 이해하려면 변수의 생존 범위(스코프)부터 잡아야 한다. `var`는 함수 스코프, `let`/`const`는 블록 스코프. 세 키워드 모두 호이스팅되지만 동작 방식이 다르다.

클로저나 stale closure를 제대로 이해하려면, 가장 먼저 **변수가 어느 범위에서 살아 있는지**부터 알아야 한다. 많은 사람이 클로저를 어렵게 느끼는 이유는 사실 클로저가 어려워서가 아니라, 변수의 생존 범위와 스코프를 먼저 잡지 않았기 때문이다.

### 핵심 비교

|  | `var` | `let` | `const` |
|---|---|---|---|
| **스코프** | 함수 스코프 | 블록 스코프 | 블록 스코프 |
| **호이스팅** | O (undefined로 초기화) | O (TDZ — 초기화 전 접근 불가) | O (TDZ — 초기화 전 접근 불가) |
| **재선언** | 가능 | 불가 | 불가 |
| **재할당** | 가능 | 가능 | **불가** |
| **전역 선언 시** | `window` 프로퍼티로 등록 | `window`에 등록 안 됨 | `window`에 등록 안 됨 |

```js
function example() {
  console.log(a); // undefined — var는 호이스팅 + undefined 초기화
  // console.log(b); // ReferenceError — let은 호이스팅되지만 TDZ
  var a = 1;
  let b = 2;
}

if (true) {
  var x = 10; // 함수 스코프 → if 밖에서도 접근 가능
  let y = 20; // 블록 스코프 → if 밖에서 접근 불가
}
console.log(x); // 10
// console.log(y); // ReferenceError
```

### `const`의 정확한 의미

`const`는 **값 전체를 불변으로 만드는 게 아니라 바인딩 재할당을 막는다.** 이 구분은 나중에 React의 `const [count, setCount]`를 이해하는 데 직접 연결된다.

```js
const obj = { a: 1 };
obj.a = 2;      // 가능 — 프로퍼티 변경은 재할당이 아님
// obj = {};    // TypeError — 바인딩 자체를 바꾸는 건 불가
```

### 자주 하는 오해

- `let`/`const`는 호이스팅되지 않는다 → **틀림.** 호이스팅은 되지만 TDZ 때문에 초기화 전 접근이 막힐 뿐이다.
- `const`는 객체도 완전히 불변이다 → **틀림.** 재할당만 막는다. 객체 내부는 변경 가능.
- `var`는 그냥 옛날 문법일 뿐이라 의미 없다 → **틀림.** 함수 스코프와 전역 오염 문제, 그리고 for 루프 클로저 문제를 이해하는 데 중요하다.

### 면접 압축 답변

- **10초**: `var`는 함수 스코프, `let`/`const`는 블록 스코프입니다. 세 키워드 모두 호이스팅되지만 `let`/`const`는 TDZ가 있고, `const`는 재할당만 막습니다.
- **30초**: 세 선언 모두 호이스팅되지만 `var`는 `undefined`로 초기화되고, `let`/`const`는 TDZ 구간에서 접근이 막힙니다. `const`는 바인딩 재할당을 막을 뿐 객체 내부 변경은 가능합니다. 실무에서는 재선언과 스코프 버그를 줄이기 위해 기본적으로 `const`, 필요할 때만 `let`을 씁니다.

> 이제 "변수가 어디까지 보이느냐"를 알았으니, 다음에는 **코드가 실행될 때 이 변수들을 담는 작업 공간이 어떻게 만들어지는지**를 봐야 한다. 그게 실행 컨텍스트다.

## 2. 실행 컨텍스트는 왜 필요한가

> **핵심 요약** — 실행 컨텍스트는 "이 코드가 실행되는 순간의 환경 정보 묶음"이다. 콜 스택에 쌓이고 함수가 끝나면 빠지지만, 내부의 렉시컬 환경은 참조가 있으면 살아남는다. 이게 클로저의 핵심.

스코프 규칙만으로는 아직 부족하다. 실제 실행 순간에는 "현재 어떤 함수가 돌고 있고, 어떤 변수들이 사용 가능하며, 외부 스코프는 어디인지"를 관리해야 한다. 이 실행 단위의 작업 공간이 **실행 컨텍스트**다.

### 핵심 개념

- 전역 코드가 시작되면 **전역 실행 컨텍스트**가 만들어진다.
- 함수가 호출될 때마다 **새로운 함수 실행 컨텍스트**가 생성된다.
- 실행 컨텍스트는 **콜 스택**에 쌓였다가 함수 종료 시 제거된다.
- 각 실행 컨텍스트 안에는 **렉시컬 환경(Lexical Environment)**이 있다.

### 콜 스택 시각화

```js
function outer() {
  let count = 0;
  return function inner() {
    return ++count;
  };
}
const fn = outer();
fn();
```

```
실행 순서:

1. [Global EC]              ← 전역 컨텍스트 생성
2. [Global EC] [outer EC]   ← outer() 호출 → outer EC 생성
3. [Global EC]              ← outer 종료 → outer EC 제거
                               (하지만 inner가 outer의 렉시컬 환경을 참조 중!)
4. [Global EC] [inner EC]   ← fn() 호출 → inner EC 생성
5. [Global EC]              ← inner 종료 → inner EC 제거
```

여기서 핵심적인 구분이 있다.

**실행 컨텍스트**는 함수가 끝나면 콜 스택에서 제거된다. 하지만 그 안의 **렉시컬 환경**은, 다른 함수가 참조하고 있으면 GC되지 않고 살아남는다. 이게 바로 클로저가 가능한 이유다.

```js
const fn = outer();
// 이 시점에서 outer의 실행 컨텍스트는 콜 스택에서 사라졌다.
// 그러나 inner가 outer의 렉시컬 환경을 참조하고 있으므로,
// count가 담긴 렉시컬 환경은 살아있다.
fn(); // 1
fn(); // 2
```

### 렉시컬 환경의 구조

각 렉시컬 환경은 두 가지를 가진다:

1. **환경 레코드(Environment Record)** — 현재 스코프의 식별자(변수, 함수 선언)를 저장
2. **외부 렉시컬 환경 참조(Outer Lexical Environment Reference)** — 상위 스코프를 가리키는 포인터

이 "외부 참조"가 체인처럼 연결되어 **스코프 체인**을 형성한다. 변수를 찾을 때 현재 환경 → 외부 환경 → ... → 전역 환경 순서로 탐색한다.

### 자주 하는 오해

- 실행 컨텍스트 = 콜 스택 → **같지 않다.** 콜 스택은 컨텍스트가 쌓이는 자료구조이고, 실행 컨텍스트는 그 안의 개별 실행 환경이다.
- 함수가 끝나면 그 안의 변수가 전부 사라진다 → **틀림.** 렉시컬 환경이 다른 함수에 의해 참조되면 살아남는다.
- 호이스팅은 코드를 물리적으로 위로 옮기는 것이다 → **틀림.** 실행 컨텍스트 **생성 단계**에서 선언 정보가 환경 레코드에 먼저 등록되는 것이다. 코드 위치는 안 바뀐다.

### 면접 압축 답변

- **10초**: 실행 컨텍스트는 JS 코드 실행에 필요한 환경 정보 묶음입니다. 함수가 끝나면 콜 스택에서 제거되지만, 내부의 렉시컬 환경은 클로저에 의해 유지될 수 있습니다.
- **30초**: 전역 실행 컨텍스트가 먼저 만들어지고, 함수 호출마다 새로운 컨텍스트가 콜 스택에 쌓입니다. 각 컨텍스트 안에는 변수/함수 선언 정보를 담은 환경 레코드와, 상위 스코프를 가리키는 외부 참조가 있습니다. 이 외부 참조가 체인으로 연결되어 스코프 체인을 형성하고, 실행 컨텍스트가 스택에서 제거된 뒤에도 렉시컬 환경이 참조되면 살아남아 클로저가 가능합니다.

> 이제 "실행될 때 작업 공간이 생긴다"는 걸 알았다. 다음엔 그 작업 공간이 **외부 스코프를 어떤 기준으로 참조하는지** 봐야 한다. 그게 렉시컬 스코프다.

## 3. 렉시컬 스코프는 왜 클로저의 전제인가

> **핵심 요약** — JS 함수는 "어디서 호출됐는지"가 아니라 **"어디서 선언됐는지"**를 기준으로 외부 변수를 찾는다. 이 규칙 덕분에 함수가 다른 곳에서 실행되어도 원래 환경을 안정적으로 참조할 수 있다.

### 핵심 개념

- 함수의 상위 스코프는 **호출 위치가 아니라 선언 위치**로 결정된다.
- 따라서 반환된 함수가 다른 곳에서 실행되어도 원래 선언 당시의 외부 환경을 본다.

```js
let value = "global";

function outer() {
  let value = "outer";
  return function inner() {
    return value; // 선언 위치 기준 → "outer"
  };
}

function other() {
  let value = "other";
  const fn = outer();
  return fn(); // 호출은 other 안이지만, inner는 outer 안에서 선언됨
}

other(); // "outer" — 호출 위치(other)가 아닌 선언 위치(outer) 기준
```

### 렉시컬 스코프 vs 동적 스코프

| | 렉시컬 스코프 (JS) | 동적 스코프 (Bash, 일부 Lisp) |
|---|---|---|
| **기준** | 함수가 **선언된** 위치 | 함수가 **호출된** 위치 |
| **위 예시 결과** | `"outer"` | `"other"` |
| **예측 가능성** | 코드만 보면 알 수 있다 | 실행 흐름을 따라가야 안다 |

클로저가 가능한 근거가 바로 여기 있다. 만약 JS가 동적 스코프였다면, 내부 함수가 외부 상태를 안정적으로 기억하기 어렵다. 호출 위치에 따라 매번 다른 값을 볼 테니까.

### `this`와의 차이 — 혼동하기 쉬운 포인트

| | 렉시컬 스코프 (변수) | `this` (일반 함수) | `this` (화살표 함수) |
|---|---|---|---|
| **결정 시점** | 선언 시 | 호출 시 | 선언 시 (렉시컬) |
| **결정 기준** | 코드 위치 | 호출 방식 (`obj.fn()`, `call`, `new`) | 상위 스코프의 `this` |

```js
const obj = {
  value: 42,
  // 일반 함수: this는 호출 방식에 따라 결정
  regular() { return this.value; },
  // 화살표 함수: this는 선언 위치의 this (여기서는 obj를 감싸는 스코프)
  arrow: () => this.value,
};

obj.regular(); // 42 — obj.regular()로 호출했으므로 this === obj
obj.arrow();   // undefined — 화살표 함수의 this는 선언 시점의 상위 this (전역)
```

### 면접 압축 답변

- **10초**: 렉시컬 스코프는 함수의 상위 스코프가 호출 위치가 아닌 선언 위치로 결정되는 규칙이며, 클로저의 전제가 됩니다.
- **30초**: 렉시컬 스코프 덕분에 반환된 함수도 선언 당시의 외부 변수를 계속 참조할 수 있고, 이것이 클로저가 가능한 근거입니다. 참고로 `this`는 일반 함수에서는 호출 방식으로 결정되지만, 화살표 함수에서는 렉시컬하게 결정됩니다.

> 이제 내부 함수가 외부 환경을 바라볼 수 있는 기준을 알았다. 다음 질문은 자연스럽다. **그 외부 환경 참조가 함수 종료 후에도 유지될 수 있나?** 이게 클로저다.

## 4. 클로저는 왜 상태 보존으로 이어지나

> **핵심 요약** — 클로저는 "함수 + 그 함수가 참조하는 렉시컬 환경"의 조합이다. 값을 복사하는 게 아니라 **환경을 참조**하므로, 외부 함수가 끝나도 내부 함수가 외부 변수에 접근할 수 있다.

### 핵심 개념

클로저는 **함수와 그 함수가 참조하는 렉시컬 환경의 조합**이다. 즉 외부 함수 실행이 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있는 상태다.

이 구조 덕분에 자바스크립트 함수는 단순한 계산 단위가 아니라 **상태를 품은 행위자**처럼 동작할 수 있다.

```js
function makeCounter() {
  let count = 0; // 이 변수는 makeCounter가 끝나도 사라지지 않는다
  return {
    increment: () => ++count,
    getCount: () => count,
  };
}

const counter = makeCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.getCount();  // 2 — count는 살아있다

// 핵심: increment와 getCount는 같은 렉시컬 환경의 count를 참조한다.
// makeCounter() 실행 컨텍스트는 사라졌지만, 렉시컬 환경은 유지된다.
```

여기서 중요한 점: 클로저는 **값을 복사해서 저장하는 게 아니라, 환경을 참조**한다. 그래서 `increment`로 `count`를 바꾸면 `getCount`에서도 바뀐 값이 보인다. 두 함수가 같은 렉시컬 환경을 공유하기 때문이다.

이 "환경 참조" 방식은 `let`/`var`처럼 변경 가능한(mutable) 바인딩에서 특히 중요하다. 나중에 React로 넘어가면, `const`로 선언된 불변 바인딩 위에서 클로저가 동작하는 경우를 보게 되는데, 이때는 조금 다른 양상이 된다. 이건 섹션 7에서 다시 다룬다.

### 실무에서 클로저가 쓰이는 대표 패턴

```js
// 1. 디바운스 — 타이머 ID를 클로저로 기억
function debounce(fn, delay) {
  let timerId;
  return function (...args) {
    clearTimeout(timerId);
    timerId = setTimeout(() => fn(...args), delay);
  };
}

// 2. 이벤트 핸들러 팩토리 — 특정 인자를 바인딩
function createClickHandler(itemId) {
  return function (event) {
    console.log(`Item ${itemId} clicked`);
  };
}

// 3. 모듈 패턴 (IIFE + 클로저) — 상태를 외부에서 직접 접근 불가
const authModule = (function () {
  let token = null;
  return {
    setToken(t) { token = t; },
    getToken() { return token; },
    isLoggedIn() { return token !== null; },
  };
})();
// token에 직접 접근 불가 — authModule.setToken()으로만 변경 가능
```

### 자주 하는 오해

- 클로저는 값을 복사해서 저장한다 → **틀림.** 환경을 참조하므로, 같은 환경을 공유하는 함수들은 같은 변수의 최신 값을 본다.
- 클로저는 특별한 문법이다 → **틀림.** 함수가 만들어지는 기본 동작에서 자연스럽게 생긴다. 자바스크립트에서는 모든 함수가 클로저다.
- 클로저와 IIFE는 같은 개념이다 → **틀림.** IIFE는 즉시 실행 함수 표현식일 뿐이고, 클로저는 그 안에서 반환된 함수가 외부 환경을 참조하는 현상이다.

### 면접 압축 답변

- **10초**: 클로저는 함수가 선언 당시의 렉시컬 환경을 계속 참조할 수 있는 구조로, 값을 복사하는 게 아니라 환경 자체를 참조합니다.
- **30초**: 클로저는 환경을 참조하기 때문에 외부 함수가 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있습니다. 이를 이용해 상태 보존, 디바운스, 모듈 패턴 같은 실무 패턴을 구현할 수 있습니다. 같은 환경을 공유하는 함수들은 같은 변수의 최신 값을 본다는 점이 핵심입니다.

> 클로저가 환경을 참조한다는 걸 알았다. 그런데 이 원리가 **루프와 만나면** 예상 밖의 결과가 나온다. 이게 면접 단골 질문이다.

## 5. for 루프와 클로저 — 스코프와 클로저의 합류점

> **핵심 요약** — `var`를 쓴 for 루프에서 비동기 콜백은 루프가 끝난 뒤의 값만 본다. `let`은 반복마다 새 블록 스코프를 만들어 각 반복의 값을 캡처한다.

이 문제는 var/let/const(스코프)와 클로저가 만나는 지점이다. 면접에서 매우 자주 나오고, 원리를 모르면 "그냥 let 쓰세요"밖에 답을 못 한다.

### 문제 코드

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}
// 예상: 0, 1, 2
// 실제: 3, 3, 3
```

### 왜 이렇게 되나?

1. `var i`는 **함수 스코프**다 → for 블록 안에 갇히지 않고, 하나의 `i`가 전체 루프에서 공유된다.
2. `setTimeout` 콜백은 즉시 실행되지 않는다 → 1초 뒤에 실행된다.
3. 콜백이 실행될 때 루프는 이미 끝났다 → `i`는 이미 3이다.
4. 세 콜백 모두 **같은 렉시컬 환경의 같은 `i`**를 참조한다 → 전부 3.

### 해결 방법 비교

```js
// 방법 1: let을 쓴다 (가장 간단)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 1000);
}
// 0, 1, 2 — let은 반복마다 새 블록 스코프 + 새 바인딩을 만든다

// 방법 2: IIFE로 각 반복의 값을 캡처 (ES6 이전 방식)
for (var i = 0; i < 3; i++) {
  (function (captured) {
    setTimeout(() => console.log(captured), 1000);
  })(i);
}
// 0, 1, 2 — IIFE가 호출될 때마다 새 실행 컨텍스트 + 새 렉시컬 환경
```

### 핵심 정리

| | `var` + for | `let` + for |
|---|---|---|
| **스코프** | 함수 스코프 (하나의 i 공유) | 블록 스코프 (반복마다 새 i) |
| **클로저가 캡처하는 것** | 루프 종료 후의 최종 값 | 각 반복 시점의 값 |
| **setTimeout 결과** | 전부 같은 값 | 각각 다른 값 |

### 실무 에피소드 — 이런 상황에서 만난다

> 리스트를 렌더링하면서 각 아이템에 클릭 핸들러를 동적으로 붙일 때, 인덱스를 잘못 캡처하면 모든 아이템이 마지막 인덱스의 동작을 한다. 예를 들어 탭 UI에서 탭 3개를 루프로 생성했는데, 어떤 탭을 클릭해도 마지막 탭의 내용만 보이는 버그. 원인은 `var`로 선언한 인덱스를 콜백이 공유하고 있었기 때문이다.

### 면접 압축 답변

- **10초**: `var`는 함수 스코프라 for 루프에서 하나의 변수를 공유해 비동기 콜백이 최종 값만 봅니다. `let`은 반복마다 새 스코프를 만들어 해결합니다.
- **30초**: `var`로 선언한 for 루프에서는 콜백들이 같은 변수를 공유하므로, 루프가 끝난 뒤 실행되면 최종 값만 보게 됩니다. `let`은 반복마다 새 블록 스코프와 새 바인딩을 만들어 각 반복의 값을 캡처합니다. ES6 이전에는 IIFE로 같은 효과를 냈습니다.

> 클로저는 유용하지만, **잘못 유지되면 오래된 값이나 필요 없는 참조도 계속 남길 수 있다.** 다음은 이 문제를 본다.

## 6. 클로저가 잘못 사용되면 왜 문제가 되나

> **핵심 요약** — 클로저 자체가 문제가 아니다. 불필요한 참조를 정리하지 않아서 GC가 회수 못하는 게 문제다.

### 핵심 개념

클로저는 필요 이상으로 큰 객체나 DOM 참조, 오래된 콜백을 붙잡을 수 있다. 그러면 메모리 회수가 지연되거나, 의도와 다른 오래된 값을 사용하는 문제가 생긴다.

```js
function setupHandler() {
  const hugeData = new Array(1000000).fill("x"); // 거대한 데이터

  document.getElementById("btn").addEventListener("click", () => {
    // 이 핸들러가 살아있는 한, hugeData도 GC되지 않는다.
    // 실제로는 hugeData를 안 써도, 같은 렉시컬 환경에 있으면 유지될 수 있다.
    console.log("clicked");
  });
}
```

### 실무에서 흔한 클로저 누수 패턴

| 패턴 | 문제 | 해결 |
|---|---|---|
| 이벤트 리스너 미해제 | 컴포넌트가 사라져도 핸들러가 남아있음 | `removeEventListener` 또는 React `useEffect` cleanup |
| setInterval 미정리 | 페이지 이동 후에도 계속 실행 | `clearInterval` + cleanup |
| 큰 객체를 클로저 안에 유지 | 필요 없는 데이터가 GC 안 됨 | 필요한 값만 추출 후 참조 끊기 |
| WebSocket/EventSource 미해제 | 연결이 계속 살아있음 | `close()` + cleanup |

### 디버깅 방법 — 크롬 DevTools

메모리 누수가 의심될 때:
1. **DevTools → Memory 탭 → Heap Snapshot** 찍기
2. 특정 액션(페이지 이동, 모달 닫기) 전후로 스냅샷 비교
3. **Retained Size**가 큰 객체를 찾아서 어떤 클로저가 잡고 있는지 확인
4. Retainers 트리에서 참조 체인 역추적

### 실무 에피소드 — 이런 상황에서 만난다

> SPA에서 페이지 전환 후에도 이전 페이지의 WebSocket 연결이 계속 살아있어서 서버 리소스가 누적되는 버그. 원인은 `useEffect`에서 WebSocket을 열었는데 cleanup에서 `close()`를 안 한 것. 한 사용자가 페이지를 왔다 갔다 할 때마다 연결이 하나씩 쌓이고, 서버 로그에 커넥션 수가 계속 올라가는 걸로 발견했다.

### 자주 하는 오해

- 클로저가 있으면 무조건 메모리 누수다 → **틀림.** 참조가 필요 없어졌을 때 정리하지 않은 게 문제다.
- 메모리 누수는 브라우저가 GC를 못 해서 생긴다 → **절반만 맞다.** 더 정확히는 아직 참조가 남아 있어서 GC 회수 조건이 만족되지 않는 경우가 많다.

### 면접 압축 답변

- **10초**: 클로저 자체가 문제가 아니라, 불필요한 참조를 정리하지 않아 GC가 회수하지 못하는 게 문제입니다.
- **30초**: 이벤트 리스너를 해제하지 않거나, 필요 이상으로 큰 객체를 클로저 안에 두면 메모리 누수가 됩니다. React에서는 `useEffect`의 cleanup 함수로 구독 해제, 타이머 정리, 연결 종료를 처리합니다. 의심되면 DevTools Memory 탭에서 Heap Snapshot을 찍어 참조 체인을 역추적합니다.

> 여기서부터 이야기가 React로 넘어간다. 순수 JS에서는 "오래된 참조를 붙잡는 문제" 정도로 보이지만, React에서는 렌더가 반복되면서 **각 렌더 시점의 값**이 존재한다. 그래서 "오래된 참조"가 훨씬 자주, 그리고 더 미묘하게 문제를 만든다.

## 7. React state는 왜 snapshot처럼 동작하나

> **핵심 요약** — React 컴포넌트는 렌더마다 다시 실행되는 함수다. `useState`는 매 렌더마다 그 시점의 값을 `const`로 반환하므로, 각 렌더는 자기만의 값 세트를 가진다.

React를 그냥 "함수 모음"으로 보면 잘 안 잡힌다. React 컴포넌트는 함수이긴 하지만, **렌더 단위로 다시 실행되는 함수**다.

즉 컴포넌트가 한 번 실행될 때마다, 그 렌더 시점의 `props`, `state`, `context`를 기준으로 JSX를 계산한다.

```
렌더 1: count = 0, handleClick은 count=0을 캡처
렌더 2: count = 1, handleClick은 count=1을 캡처
렌더 3: count = 2, handleClick은 count=2을 캡처
→ 각 렌더는 독립된 "스냅샷"이다.
```

### 섹션 4와의 연결 — 클로저 캡처 방식의 차이

섹션 4에서 "클로저는 값을 복사하는 게 아니라 환경을 참조한다"고 했다. `let count = 0`처럼 mutable 바인딩이면, 같은 환경을 공유하는 함수들이 변경된 값을 함께 본다.

그런데 React에서는 상황이 다르다.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  // count는 const — 이 렌더 안에서는 절대 바뀌지 않는다

  const handleClick = () => {
    setTimeout(() => {
      console.log(count); // 이 렌더의 count 값이 고정되어 있다
    }, 3000);
  };

  return <button onClick={handleClick}>{count}</button>;
}
```

`useState`가 반환하는 값은 `const`다. 매 렌더마다 새로운 실행이 일어나고, 새로운 `const count`가 만들어진다. 클로저는 여전히 환경을 참조하지만, 그 환경 안의 바인딩이 불변(`const`)이기 때문에 **결과적으로 특정 렌더의 값이 고정된 것처럼** 동작한다.

### React가 state를 저장하는 곳

"state가 외부 값"이라고 했는데, 구체적으로 어디에 있을까?

React는 각 컴포넌트 인스턴스에 대응하는 **Fiber 노드**를 가지고 있다. Hooks의 state는 이 Fiber 노드의 `memoizedState` 링크드 리스트에 저장된다. `useState`를 호출하면 React가 현재 Fiber에서 해당 순서의 hook state를 꺼내서 반환하는 것이다.

```
Fiber 노드
├── memoizedState: hook1 → hook2 → hook3 (링크드 리스트)
├── stateNode: 실제 DOM 또는 컴포넌트 인스턴스
└── ...
```

이게 **Hook을 조건문이나 반복문 안에서 쓰면 안 되는 이유**다. 호출 순서로 hook을 식별하기 때문에, 순서가 바뀌면 엉뚱한 state를 반환한다.

### 자주 하는 오해

- state는 그냥 함수 안의 변수다 → **틀림.** React의 Fiber 노드가 관리하는 외부 값이다. `useState`는 매 렌더마다 현재 시점의 값을 반환한다.
- `setState`를 부르면 지금 실행 중인 함수 안의 state 값도 바로 바뀐다 → **틀림.** 현재 렌더의 `const` 값은 바뀔 수 없다. 다음 렌더에서 새 값이 반환된다.

### 면접 압축 답변

React의 state는 현재 렌더 기준 snapshot처럼 동작합니다. 컴포넌트가 렌더될 때마다 `useState`는 Fiber 노드에 저장된 값을 `const`로 반환하고, 이벤트 핸들러나 effect는 자신이 생성된 렌더의 값을 클로저로 캡처합니다. `setState`는 현재 값을 바꾸는 게 아니라 다음 렌더를 예약합니다. Hook이 호출 순서로 식별되기 때문에 조건문 안에서 Hook을 쓰면 안 됩니다.

## 8. setState가 즉시 바뀌지 않는 이유

> **핵심 요약** — `setState`는 변수 재할당이 아니라 다음 렌더 예약이다. React 18부터는 어디서든 automatic batching이 적용되어 여러 `setState`가 하나의 렌더로 합쳐진다.

"내가 `setCount(1)` 했는데 왜 바로 `count`가 1이 안 되지?"

이건 React가 게을러서가 아니라, 렌더 모델 자체가 그렇게 생겼기 때문이다. 현재 실행 중인 이벤트 핸들러는 이미 어떤 렌더의 snapshot 위에서 돌아가고 있다. 그 안에서 `setState`를 호출해도, React는 지금 돌고 있는 그 함수 내부 변수 값을 덮어쓰지 않는다. 대신 **다음 렌더를 예약**한다.

### 같은 값 기반 vs 함수형 업데이트

```tsx
// 문제: 같은 snapshot 기준으로 계산
const handleClick = () => {
  setCount(count + 1); // 0 + 1 = 1
  setCount(count + 1); // 0 + 1 = 1 (같은 count=0 기준)
  // 결과: count는 1이 된다. 2가 아니다.
  console.log(count);  // 여전히 0 — const는 바뀌지 않는다
};

// 해결: 함수형 업데이트 — 이전 결과를 이어받아 계산
const handleClick = () => {
  setCount((prev) => prev + 1); // prev = 0 → 1
  setCount((prev) => prev + 1); // prev = 1 → 2
  // 결과: count는 2가 된다.
};
```

| | 값 기반 업데이트 | 함수형 업데이트 |
|---|---|---|
| **기준** | 현재 렌더의 snapshot 값 | 업데이트 큐의 이전 결과 |
| **동일 핸들러에서 여러 번** | 마지막 값만 적용 | 누적 적용 |
| **언제 쓰나** | 이전 값과 무관한 세팅 | 이전 값 기반 계산 |

### Automatic Batching — React 17 vs 18

```tsx
// React 17: 이벤트 핸들러 안에서만 batching
onClick={() => {
  setA(1); // batched
  setB(2); // batched → 렌더 1회
}}

setTimeout(() => {
  setA(1); // React 17에서는 즉시 렌더
  setB(2); // 또 렌더 → 총 2회
}, 100);

// React 18: 어디서든 batching
setTimeout(() => {
  setA(1); // batched
  setB(2); // batched → 렌더 1회
}, 100);
```

### flushSync — batching을 깨야 할 때

극히 드물지만, DOM 측정처럼 즉시 렌더가 필요한 경우:

```tsx
import { flushSync } from "react-dom";

const handleClick = () => {
  flushSync(() => {
    setCount((prev) => prev + 1);
  });
  // 이 시점에서 DOM이 이미 업데이트되어 있다
  console.log(ref.current.textContent); // 업데이트된 값
};
```

### 실무 에피소드 — 이런 상황에서 만난다

> 장바구니에서 "수량 +1" 버튼을 빠르게 여러 번 클릭했는데 수량이 1만 올라가는 버그. 원인은 `setQuantity(quantity + 1)`로 값 기반 업데이트를 쓰고 있었기 때문. 클릭 이벤트가 같은 렌더의 `quantity`를 기준으로 계산해서, 아무리 빨리 눌러도 같은 값 + 1이 반복됐다. `setQuantity(prev => prev + 1)`로 바꾸면 해결.

### 자주 하는 오해

- React state가 비동기다 → **표현이 부정확하다.** `setState` 호출 자체는 동기적으로 큐에 들어간다. 다만 현재 렌더의 값을 즉시 바꾸지 않고 다음 렌더를 스케줄링하는 것이다.

### 면접 압축 답변

`setState`는 현재 렌더의 state 변수를 즉시 바꾸는 것이 아니라 다음 렌더를 예약합니다. React 18부터는 이벤트 핸들러, setTimeout, Promise 등 어디서든 automatic batching이 적용되어 여러 `setState`가 하나의 렌더로 합쳐집니다. 같은 snapshot 기준으로 계산되므로, 누적 업데이트가 필요하면 함수형 업데이트(`prev => prev + 1`)를 써야 합니다. DOM 측정 등 즉시 반영이 필요하면 `flushSync`를 쓸 수 있습니다.

> 이걸 이해하면, stale closure가 왜 생기고 함수형 업데이트가 왜 필요한지 한 번에 묶인다. 이제 다시 클로저로 돌아가 보자.

## 9. stale closure는 정확히 어디서 터지나

> **핵심 요약** — 예전 렌더에서 만들어진 함수가 오래 살아남아 최신 state 대신 캡처 당시의 오래된 값을 계속 참조하는 현상. 클로저 자체의 결함이 아니라, 렌더 단위와 결합되어 생기는 시점 불일치다.

클로저는 함수가 외부 환경을 기억하는 구조다. React 컴포넌트도 함수이기 때문에, 이벤트 핸들러나 effect 내부 함수는 **그 렌더의 값**을 캡처한다.

문제는 컴포넌트는 계속 다시 렌더되는데, 예전 렌더에서 만들어진 함수가 **오래 살아남을 수 있다**는 점이다.

### 전형적인 예시

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count);
      // 렌더 0에서 만들어진 함수 → count는 항상 0
    }, 1000);
    return () => clearInterval(id);
  }, []);
  // dependency가 [] → effect는 최초 렌더에서만 실행
  // → setInterval 콜백은 렌더 0의 count(= 0)를 캡처한 채 영원히 남는다

  return (
    <button onClick={() => setCount((p) => p + 1)}>
      {count}
    </button>
  );
}
// 버튼을 아무리 눌러도 콘솔에는 항상 0이 찍힌다.
```

### stale closure가 발생하기 쉬운 곳

| 위치 | 왜 위험한가 |
|---|---|
| `setInterval` / `setTimeout` | 한번 등록되면 콜백이 오래 살아남음 |
| 이벤트 리스너 (`addEventListener`) | 해제하지 않으면 영원히 유지 |
| effect 내부 비동기 콜백 | API 응답이 오래 걸리면 그 사이에 렌더가 진행됨 |
| Promise `.then()` / `.catch()` | 체이닝된 콜백이 이전 렌더의 값을 봄 |
| 외부 라이브러리에 넘긴 callback | 라이브러리가 내부적으로 캐싱할 수 있음 |

### 해결 전략 요약 (상세는 뒤 섹션에서)

```
1순위: dependency 배열을 정확히 채운다 → 섹션 10
2순위: 함수형 업데이트를 쓴다 (prev => ...)
3순위: ref로 최신 값을 참조한다 → 섹션 11
```

### 디버깅 사고 과정

stale closure가 의심될 때:
1. **증상 확인** — "특정 값이 업데이트 안 된다" 또는 "예전 값이 찍힌다"
2. **의심 지점 찾기** — 해당 값을 사용하는 콜백/effect가 어떤 렌더에서 만들어졌는지 확인
3. **dependency 검사** — 그 콜백을 만드는 effect/훅의 dependency 배열에 해당 값이 있는지
4. **캡처 시점 확인** — `console.log("render:", count)` 같은 로그를 컴포넌트 본문에 넣어서 렌더 시점과 콜백 안의 값을 비교

### 실무 에피소드 — 이런 상황에서 만난다

> 실시간 채팅에서 "읽지 않은 메시지 수"를 뱃지로 보여주는 기능. WebSocket `onmessage` 핸들러를 `useEffect([], [])`로 한 번만 등록했는데, 새 메시지가 와도 뱃지 카운트가 항상 0에서 1로만 바뀌고 누적이 안 됐다. 원인은 핸들러가 초기 렌더의 `unreadCount = 0`을 캡처한 채 `setUnreadCount(unreadCount + 1)` → 항상 `0 + 1 = 1`. 함수형 업데이트로 바꾸면 해결.

### 면접 압축 답변

- **10초**: Stale closure는 예전 렌더에서 생성된 함수가 오래 살아남아 캡처 당시의 오래된 값을 계속 참조하는 현상입니다.
- **30초**: `setInterval`, 이벤트 리스너, 비동기 콜백 등 장기 생존하는 함수에서 주로 발생합니다. 해결 순서는 dependency 배열 정확히 채우기, 함수형 업데이트(`prev => ...`), ref 사용 순이며, 디버깅할 때는 콜백이 어떤 렌더에서 만들어졌는지를 추적합니다.

## 10. useEffect dependency는 왜 중요한가

> **핵심 요약** — dependency 배열은 "이 effect가 어떤 값에 의존하는지 선언하는 계약"이다. 빈 배열은 "아무것에도 의존 안 함"이라는 선언이므로, 안에서 state/props를 읽으면 거짓말이 된다.

`useEffect`는 본질적으로 **"렌더 결과를 바탕으로 외부 시스템과 동기화하는 훅"**이다. API 요청, 이벤트 구독, 타이머 등록, 외부 라이브러리 초기화 같은 것들이 여기에 해당한다.

### cleanup → 재실행 사이클

dependency를 이해하려면 cleanup 개념도 함께 알아야 한다. effect가 재실행될 때, React는 **이전 effect의 cleanup을 먼저 실행**한 다음 새 effect를 실행한다.

```tsx
useEffect(() => {
  const id = setInterval(() => {
    console.log(count);
  }, 1000);

  // cleanup: 다음 렌더에서 effect가 다시 실행되기 전에 호출된다
  return () => clearInterval(id);
}, [count]);
// count가 바뀔 때마다:
// 1. 이전 interval을 clearInterval로 정리
// 2. 새 count를 캡처한 새 interval을 등록
```

```
렌더 1 (count=0): effect 실행 → interval 등록 (count=0 캡처)
렌더 2 (count=1): cleanup 실행 → 이전 interval 해제
                   effect 실행 → 새 interval 등록 (count=1 캡처)
렌더 3 (count=2): cleanup 실행 → 이전 interval 해제
                   effect 실행 → 새 interval 등록 (count=2 캡처)
언마운트:         cleanup 실행 → 마지막 interval 해제
```

이 cleanup → 재실행 사이클이 stale closure를 방지하는 핵심 메커니즘이다.

### dependency 배열의 의미

| dependency | 의미 | 실행 타이밍 |
|---|---|---|
| `[count]` | "count에 의존한다" | count가 바뀔 때마다 |
| `[count, userId]` | "count와 userId에 의존한다" | 둘 중 하나가 바뀔 때마다 |
| `[]` | "아무것에도 의존하지 않는다" | 마운트 시 1회만 |
| 생략 | "매 렌더마다 실행" | 모든 렌더 후 |

### 실무에서 흔히 겪는 문제

| 증상 | 원인 | 해결 |
|---|---|---|
| 검색어가 바뀌었는데 예전 결과가 표시됨 | dependency에 `query`를 안 넣음 | `[query]` 추가 |
| interval이 최신 count를 못 봄 | `[]`에서 count를 읽음 | `[count]` 추가 또는 함수형 업데이트 |
| API가 무한 호출됨 | dependency에 매 렌더마다 새로 만들어지는 객체/배열을 넣음 | `useMemo`로 참조 안정화 |
| 이전 props 기준 로직이 계속 동작 | dependency 누락 | ESLint `exhaustive-deps` 규칙 활용 |

### 실무 에피소드 — 이런 상황에서 만난다

> 검색 페이지에서 검색어를 빠르게 바꿨을 때, "사과" 검색 결과가 "바나나" 검색 결과보다 늦게 도착해서 "바나나"를 검색했는데 "사과" 결과가 화면에 뜨는 버그. 이건 stale closure + cleanup 미처리가 결합된 문제다. effect cleanup에서 이전 요청을 취소(AbortController)하거나, 응답 도착 시 현재 검색어와 요청 시 검색어를 비교하는 방식으로 해결한다.

```tsx
useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then((res) => res.json())
    .then(setResults)
    .catch((err) => {
      if (err.name !== "AbortError") throw err;
    });

  // cleanup: 다음 검색어로 넘어가면 이전 요청을 취소
  return () => controller.abort();
}, [query]);
```

### 자주 하는 오해

- dependency 배열은 rerender 최적화 옵션이다 → **본질은 의존 관계 명시다.**
- 빈 배열을 쓰면 안전하다 → **오히려 위험할 수 있다.** 안에서 state나 props를 읽으면 stale closure.
- cleanup은 컴포넌트가 언마운트될 때만 실행된다 → **틀림.** dependency가 바뀌어 effect가 재실행될 때도 이전 cleanup이 먼저 실행된다.

### 면접 압축 답변

- **10초**: `useEffect` dependency 배열은 effect가 어떤 값에 의존하는지 선언하는 계약이며, dependency가 바뀌면 cleanup 후 재실행됩니다.
- **30초**: dependency가 바뀌면 이전 effect의 cleanup이 먼저 실행되고 새 값을 캡처한 effect가 다시 실행됩니다. 이 cleanup → 재실행 사이클이 stale closure를 방지합니다. 빈 배열은 "아무것에도 의존 안 함"이라는 선언이므로 안에서 state를 읽으면 거짓말이 됩니다. 실무에서는 ESLint `exhaustive-deps` 규칙과 AbortController 같은 cleanup 패턴을 함께 씁니다.

> 그런데 항상 dependency를 늘리는 게 답은 아니다. 어떤 경우에는 effect를 재실행시키고 싶지 않으면서도 **항상 최신 값**은 읽고 싶을 수 있다. 이때 `ref`가 등장한다.

## 11. ref는 왜 stale closure 회피에 쓰이나

> **핵심 요약** — `ref`는 렌더를 트리거하지 않으면서 렌더 간에 같은 객체를 유지하는 컨테이너다. 오래 살아 있는 콜백 안에서 `ref.current`를 읽으면 항상 최신 값을 볼 수 있다.

### state vs ref 비교

| | `state` | `ref` |
|---|---|---|
| **값 변경 시** | 리렌더 발생 | 리렌더 안 함 |
| **렌더 간 동일성** | 매 렌더마다 새 `const` 값 | 같은 객체 유지 (`{current: ...}`) |
| **용도** | UI에 반영되는 값 | 렌더와 무관한 값 (타이머 ID, 이전 값, DOM 노드) |
| **읽는 시점** | 렌더 시점의 snapshot | 항상 최신 (`.current`) |

### stale closure를 ref로 해결하는 패턴

```tsx
function useInterval(callback: () => void, delay: number) {
  const savedCallback = useRef(callback);

  // 매 렌더마다 ref에 최신 callback을 넣어둔다
  useEffect(() => {
    savedCallback.current = callback;
  });

  // interval은 한 번만 등록한다
  useEffect(() => {
    const id = setInterval(() => {
      savedCallback.current(); // 항상 최신 callback을 실행
    }, delay);
    return () => clearInterval(id);
  }, [delay]);
}
```

이 패턴에서 `setInterval` 콜백은 최초 렌더에서 만들어진 함수지만, 실행할 때마다 `savedCallback.current`를 통해 **최신 callback**을 읽는다. ref 객체 자체는 렌더 간에 동일한 객체이므로, 오래된 클로저 안에서도 `.current`로 최신 값에 접근할 수 있다.

### ref의 또 다른 용도 — DOM 접근

```tsx
function TextInput() {
  const inputRef = useRef<HTMLInputElement>(null);

  const focusInput = () => {
    inputRef.current?.focus(); // DOM 노드에 직접 접근
  };

  return <input ref={inputRef} />;
}
```

ref는 "값 보관함"과 "DOM 접근" 두 가지 용도로 쓰인다. 같은 `useRef`지만 맥락이 다르다는 걸 구분해야 한다.

### stale closure 해결 우선순위

```
1순위: dependency 배열을 정확히 채운다
       → effect가 재실행되면서 최신 값을 캡처

2순위: 함수형 업데이트를 쓴다
       → setState(prev => prev + 1)로 현재 snapshot에 의존 안 함

3순위: ref를 쓴다
       → 위 두 방법으로 안 되고, 재렌더 없이 최신값 참조가 꼭 필요할 때
```

### 자주 하는 오해

- `ref`는 state 대체재다 → **틀림.** UI를 갱신해야 하는 값은 state가 맞다. ref 값이 바뀌어도 화면은 안 바뀐다.
- stale closure가 생기면 무조건 `ref`로 해결하면 된다 → **틀림.** dependency 정리와 함수형 업데이트를 먼저 고려해야 한다.
- `ref.current`는 렌더 중에 읽어도 안전하다 → **주의.** 렌더 중에 ref를 읽으면 어떤 렌더에서 어떤 값이 읽히는지 예측하기 어려워진다. 이벤트 핸들러나 effect 안에서 읽는 게 안전하다.

### 면접 압축 답변

- **10초**: `ref`는 렌더를 트리거하지 않으면서 렌더 간에 같은 객체를 유지하는 컨테이너로, `ref.current`로 항상 최신 값을 읽을 수 있습니다.
- **30초**: 렌더 간에 같은 객체가 유지되므로, 오래 살아 있는 콜백 안에서 `ref.current`를 읽으면 항상 최신 값을 볼 수 있습니다. 단, stale closure의 만능 해결책이 아니라 dependency 정리 → 함수형 업데이트 → ref 순서로 고려해야 합니다. DOM 접근 용도로도 쓰이며, 렌더 중에는 ref를 읽지 않는 것이 안전합니다.

---

## 12. useMemo / useCallback은 왜 결국 참조 안정성 문제인가

> **핵심 요약** — 이 훅들은 "성능 최적화"로만 외우면 절반만 아는 것이다. 핵심은 **참조 안정성** — 같은 값이면 같은 참조를 유지해서 불필요한 하위 작업을 방지하는 것.

### 왜 참조가 문제인가

매 렌더마다 함수 본문이 다시 실행되므로, 객체/배열/함수는 **매번 새로 만들어진다**. 값이 같아도 참조가 다르다.

```js
// 매 렌더마다
const style = { color: "red" }; // 새 객체 → 새 참조
const handler = () => {};       // 새 함수 → 새 참조

// 비교
{ color: "red" } === { color: "red" } // false — 참조가 다름
```

이게 왜 문제인가:
1. `React.memo`로 감싼 자식에 이 값을 props로 넘기면 → 얕은 비교에서 "다르다"고 판단 → 불필요한 리렌더
2. `useEffect`의 dependency에 넣으면 → 매번 "바뀌었다"고 판단 → 무한 실행

### useMemo — 값의 참조 안정화

```tsx
// 용도 1: 비싼 계산 캐싱
const sortedList = useMemo(() => {
  return [...items].sort((a, b) => a.score - b.score);
}, [items]);

// 용도 2: 객체 참조 안정화
const style = useMemo(
  () => ({ color: darkMode ? "white" : "black" }),
  [darkMode],
);
// style은 darkMode가 바뀔 때만 새 객체가 된다
```

### useCallback — 함수의 참조 안정화

`useCallback(fn, deps)`는 `useMemo(() => fn, deps)`와 동일하다.

```tsx
const handleClick = useCallback(() => {
  setCount((prev) => prev + 1);
}, []);
// handleClick의 참조가 유지되므로,
// React.memo로 감싼 자식에게 넘겨도 불필요한 리렌더가 없다
```

### React.memo와의 관계

```
부모 렌더 → props 생성 → 자식에 전달
                ↓
         React.memo가 이전 props와 얕은 비교
                ↓
         같으면 → 자식 렌더 건너뜀
         다르면 → 자식 리렌더
                ↓
         매 렌더마다 새 객체/함수를 만들면 → 항상 "다르다" → memo 무효화
                ↓
         useMemo/useCallback으로 참조 유지 → memo 효과 발휘
```

`React.memo` 없이 `useCallback`만 쓰면 의미가 거의 없다. 참조를 안정화해봐야 비교하는 곳이 없으면 아무도 그 안정성을 활용하지 않는다.

### React Compiler (React 19+)

React 19부터 도입된 **React Compiler**는 빌드 타임에 자동으로 메모이제이션을 적용한다. 장기적으로 `useMemo`/`useCallback`을 직접 쓸 필요가 줄어들 전망이다. 하지만 원리를 이해하는 건 여전히 중요하다 — 컴파일러가 하는 일이 결국 이 훅들과 같은 것이기 때문이다.

### 실무 에피소드 — 이런 상황에서 만난다

> 대시보드에 차트 10개가 있는데, 필터 하나를 바꿀 때마다 모든 차트가 다 리렌더되면서 버벅거렸다. 프로파일러로 확인하니 각 차트 컴포넌트에 넘기는 `data` props가 매 렌더마다 새 배열로 만들어지고 있었다. `useMemo`로 각 차트의 데이터를 해당 필터 조건이 바뀔 때만 재계산하게 바꾸고, 차트 컴포넌트를 `React.memo`로 감쌌더니 해결.

### 자주 하는 오해

- `useCallback`을 쓰면 무조건 성능이 좋아진다 → **틀림.** `React.memo`로 감싼 자식이 없으면 참조 안정화의 이점이 없다.
- dependency를 일부러 빼면 더 최적화된다 → **틀림.** 오래된 값을 붙잡아서 버그가 된다.
- 원시값에도 참조 안정화가 필요하다 → **틀림.** 원시값은 값 자체로 비교되므로 참조 안정화 개념이 없다.
- 모든 곳에 memo를 써야 한다 → **틀림.** memo 자체도 비용이다. 프로파일러로 병목을 확인한 뒤 적용하는 게 맞다.

### 면접 압축 답변

- **10초**: `useMemo`는 값의 참조 안정화, `useCallback`은 함수의 참조 안정화에 사용되며, `React.memo`와 함께 써야 불필요한 리렌더를 방지할 수 있습니다.
- **30초**: 매 렌더마다 객체/함수는 새로 만들어져 참조가 달라지므로, `React.memo`로 감싼 자식이나 `useEffect` dependency에서 불필요한 갱신이 발생합니다. `useMemo`/`useCallback`으로 참조를 안정화하면 이를 방지할 수 있습니다. 단, 이 훅들도 dependency 배열에 의존하므로 잘못 다루면 stale closure로 이어집니다. React 19의 React Compiler가 자동 메모이제이션을 지원하지만, 원리를 이해하는 것은 여전히 중요합니다.

# 챕터 1 핵심 요약 — 함수가 값을 기억하는 구조, 그리고 React에서 그것이 깨지는 순간

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **자바스크립트 함수는 값을 어떻게 기억하고, React에서는 왜 그 기억이 "오래된 채"로 남을 수 있는가?**

이 질문에 답하려면, 가장 밑바닥부터 쌓아 올려야 한다.

---

### 1단계: 변수는 어디까지 살아있는가 — 스코프

모든 것의 출발은 **변수의 생존 범위**다. `var`는 함수 단위로 살고, `let`/`const`는 블록(`{}`) 단위로 산다. 이 차이를 모르면 for 루프에서 클로저가 왜 이상하게 동작하는지 설명할 수 없다. 그리고 `const`는 "값이 안 변한다"가 아니라 **"바인딩을 다시 연결할 수 없다"**는 뜻이다. 이 구분이 나중에 React의 `const [count, setCount]`를 이해하는 열쇠가 된다.

### 2단계: 코드가 실행될 때 무슨 일이 일어나는가 — 실행 컨텍스트

변수가 살아있으려면, 그 변수를 담고 있는 **작업 공간**이 필요하다. 함수가 호출될 때마다 실행 컨텍스트가 만들어지고 콜 스택에 쌓인다. 함수가 끝나면 컨텍스트는 스택에서 빠진다. **하지만** 그 안의 렉시컬 환경(변수가 실제로 저장된 곳)은, 누군가 참조하고 있으면 사라지지 않는다. 이 한 문장이 클로저의 전부다.

### 3단계: 외부 변수를 어떤 기준으로 찾는가 — 렉시컬 스코프

함수가 외부 변수를 찾을 때, **호출된 위치가 아니라 선언된 위치**를 기준으로 찾는다. 덕분에 함수를 어디서 실행하든 원래 환경의 변수를 안정적으로 볼 수 있다. 만약 이 규칙이 없었다면, 클로저는 존재할 수 없다.

### 4단계: 환경을 기억하는 함수 — 클로저

위 세 가지가 합쳐지면 클로저가 된다. 클로저는 특별한 문법이 아니라, **함수가 선언 당시의 렉시컬 환경을 계속 참조할 수 있는 구조**다. 핵심은 "값을 복사한다"가 아니라 **"환경을 참조한다"**는 점. 같은 환경을 공유하는 함수들은 같은 변수의 변화를 함께 본다.

### 5단계: 스코프와 클로저가 부딪히는 지점 — for 루프

`var`는 함수 스코프라서 루프 전체가 하나의 `i`를 공유한다. 비동기 콜백이 나중에 실행될 때, 이미 루프가 끝난 뒤의 최종값만 본다. `let`은 반복마다 새 블록 스코프를 만들어 각 반복의 값을 별도로 캡처한다. **스코프의 단위가 클로저의 캡처 단위를 결정한다.**

### 6단계: 클로저의 그림자 — 불필요한 참조와 메모리 누수

클로저가 환경을 참조한다는 건, 필요 없는 것까지 붙잡을 수 있다는 뜻이다. 이벤트 리스너를 해제 안 하거나, 거대한 데이터를 클로저 안에 방치하면 GC가 회수하지 못한다. 클로저 자체가 문제가 아니라, **정리하지 않은 참조가 문제다.**

---

여기서부터 무대가 React로 바뀐다. 순수 JS에서는 하나의 실행 안에서 값이 바뀌는 게 자연스럽지만, React는 **렌더 단위로 함수가 반복 실행**된다. 이 차이가 "오래된 값" 문제를 훨씬 미묘하게 만든다.

---

### 7단계: 매 렌더가 독립된 스냅샷이다 — React state

React 컴포넌트는 렌더될 때마다 다시 실행되는 함수다. `useState`는 매 렌더마다 그 시점의 값을 `const`로 돌려준다. 클로저는 여전히 환경을 참조하지만, 그 환경 안의 바인딩이 `const`이므로 **각 렌더는 자기만의 고정된 값 세트**를 갖는다. state의 실체는 Fiber 노드의 `memoizedState`에 있고, Hook은 호출 순서로 식별된다.

### 8단계: setState는 "다음 렌더 예약"이다 — batching

`setState`를 호출해도 현재 렌더의 `const` 값은 바뀌지 않는다. React는 다음 렌더를 스케줄링할 뿐이다. 같은 핸들러에서 여러 번 호출하면 같은 스냅샷 기준으로 계산되므로, 누적이 필요하면 `prev => prev + 1` 같은 함수형 업데이트를 써야 한다. React 18부터는 어디서든 automatic batching이 적용된다.

### 9단계: 오래된 렌더의 값을 보는 함수 — stale closure

여기서 모든 이야기가 합류한다. 예전 렌더에서 만들어진 콜백이 `setInterval`, 이벤트 리스너, 비동기 응답 등으로 오래 살아남으면, 그 함수는 **생성 당시 렌더의 값만 본다**. state는 이미 바뀌었는데, 함수가 아직 옛날 스냅샷을 붙잡고 있는 것. 클로저의 원리 자체는 정상인데, 렌더가 반복되는 React 모델과 결합되어 **시점 불일치**가 생기는 것이다.

### 10단계: 계약을 지키는 방법 — useEffect dependency

dependency 배열은 "이 effect가 어떤 값에 의존하는가"라는 **선언적 계약**이다. 값이 바뀌면 이전 effect를 cleanup하고 새 값을 캡처한 effect를 다시 실행한다. 이 **cleanup → 재실행 사이클**이 stale closure를 자연스럽게 방지한다. 빈 배열 `[]`은 "아무것에도 의존 안 한다"는 선언이므로, 안에서 state를 읽으면 거짓말이 되고, 거짓말의 대가가 stale closure다.

### 11단계: 렌더를 건드리지 않고 최신 값을 읽는 방법 — ref

`ref`는 렌더 간에 **동일한 객체**가 유지되는 컨테이너다. `.current`에 값을 넣어두면 어떤 렌더에서 만들어진 함수든 항상 최신 값을 읽을 수 있다. 단, ref는 최후의 수단이다. **dependency 정리 → 함수형 업데이트 → ref** 순서로 고려해야 한다.

### 12단계: 같은 값이면 같은 참조를 유지하라 — useMemo / useCallback

매 렌더마다 객체와 함수는 새로 만들어져 참조가 달라진다. `React.memo`로 감싼 자식에 넘기면 "다르다"고 판단해 불필요한 리렌더가 발생한다. `useMemo`는 값의 참조를, `useCallback`은 함수의 참조를 안정화한다. **`React.memo` 없이 `useCallback`만 쓰면 의미가 없다** — 비교하는 곳이 없으면 안정화할 이유도 없다.

---

### 한 줄로 이어 읽기

**스코프**가 변수의 생존 범위를 정하고 → **실행 컨텍스트**가 그 변수를 담는 작업 공간을 만들고 → **렉시컬 스코프**가 외부 변수를 찾는 기준을 고정하고 → 이 셋이 합쳐져 **클로저**가 되고 → React에서는 렌더마다 함수가 다시 실행되므로 각 렌더가 **스냅샷**이 되고 → 오래 살아남은 함수가 옛 스냅샷을 붙잡으면 **stale closure**가 되고 → 이를 방지하기 위해 **dependency**, **함수형 업데이트**, **ref**, **참조 안정화**를 쓴다.

> 다음 챕터에서는 "함수가 값을 어떻게 기억하는가"에서 **"React가 변경을 어떻게 감지하는가"**로 축이 이동한다. 원시값과 참조값, 불변성, 참조 동일성 — 이것들이 리렌더링 조건의 뿌리다.
