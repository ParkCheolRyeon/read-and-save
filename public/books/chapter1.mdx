# React Closure Learning Flow

이 문서는 단순 면접 답변 암기용 문답집이 아니라, **한 개념이 왜 다음 개념을 필요하게 만드는지**를 따라가며 학습하고, 마지막에 면접 답변으로 압축할 수 있게 만든 학습형 플로우 문서다.

핵심 원칙은 아래와 같다.

1. 질문을 독립적으로 외우지 않는다.
2. **선행 개념 → 원리 → 오해 교정 → 실무 연결 → 다음 개념** 순서로 읽는다.
3. 마지막에만 10초 / 30초 / 1분 답변으로 압축한다.
4. 한 문항을 이해했으면 반드시 다음 문항으로 이어지는 다리를 만든다.

### 이 문서를 읽는 순서

1. 각 섹션의 **왜 이 개념이 필요한가**를 먼저 읽는다.
2. 그다음 **자주 하는 오해**를 확인한다.
3. 마지막에 **면접 압축 답변**을 말로 꺼내 본다.
4. 막히면 다시 위 설명으로 돌아가 원리부터 복구한다.

# 챕터 1. 실행 컨텍스트에서 stale closure까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **자바스크립트 함수가 값을 어떻게 기억하는지 이해하면, React에서 값이 왜 오래된 채로 남는지도 설명할 수 있다.**

---

## 1. var / let / const를 먼저 보는 이유

클로저나 stale closure를 제대로 이해하려면, 가장 먼저 **변수가 어느 범위에서 살아 있는지**부터 알아야 한다. 많은 사람이 클로저를 어렵게 느끼는 이유는 사실 클로저가 어려워서가 아니라, 변수의 생존 범위와 스코프를 먼저 잡지 않았기 때문이다.

### 핵심 개념

- `var`는 함수 스코프다.
- `let`, `const`는 블록 스코프다.
- 세 선언 모두 호이스팅되지만, `let`/`const`는 TDZ 때문에 초기화 전 접근이 막힌다.
- `const`는 값 전체를 불변으로 만드는 게 아니라 바인딩 재할당을 막는다.

```js
function example() {
  console.log(a); // undefined — var는 호이스팅 + undefined 초기화
  // console.log(b); // ReferenceError — let은 호이스팅되지만 TDZ
  var a = 1;
  let b = 2;
}

if (true) {
  var x = 10; // 함수 스코프 → if 밖에서도 접근 가능
  let y = 20; // 블록 스코프 → if 밖에서 접근 불가
}
console.log(x); // 10
// console.log(y); // ReferenceError
```

스코프를 이해해야 "내부 함수가 외부 변수에 접근한다"는 말이 정확히 어떤 범위의 값을 가리키는지 보인다. 이게 곧 렉시컬 스코프와 클로저의 출발점이다.

### 자주 하는 오해

- `let`/`const`는 호이스팅되지 않는다 → 틀림. 호이스팅은 되지만 TDZ 때문에 초기화 전 접근이 막힐 뿐이다.
- `const`는 객체도 완전히 불변이다 → 틀림. `const obj = { a: 1 }`에서 `obj.a = 2`는 가능하다. 재할당(`obj = ...`)만 막는다.
- `var`는 그냥 옛날 문법일 뿐이라 의미 없다 → 틀림. 함수 스코프와 전역 오염 문제를 이해하는 데 중요하다.

### 면접 압축 답변

- **10초**: `var`는 함수 스코프, `let`/`const`는 블록 스코프입니다. `let`/`const`는 TDZ가 있고, `const`는 재할당만 막습니다.
- **30초**: 세 선언 모두 호이스팅되지만 `var`는 `undefined`로 초기화되고, `let`/`const`는 TDZ 구간에서 접근이 막힙니다. 실무에서는 재선언과 스코프 버그를 줄이기 위해 기본적으로 `const`, 필요할 때만 `let`을 씁니다.

> 이제 "변수가 어디까지 보이느냐"를 알았으니, 다음에는 **코드가 실행될 때 이 변수들을 담는 작업 공간이 어떻게 만들어지는지**를 봐야 한다. 그게 실행 컨텍스트다.

## 2. 실행 컨텍스트는 왜 필요한가

스코프 규칙만으로는 아직 부족하다. 실제 실행 순간에는 "현재 어떤 함수가 돌고 있고, 어떤 변수들이 사용 가능하며, 외부 스코프는 어디인지"를 관리해야 한다. 이 실행 단위의 작업 공간이 실행 컨텍스트다.

### 핵심 개념

- 전역 코드가 시작되면 전역 실행 컨텍스트가 만들어진다.
- 함수가 호출될 때마다 새로운 함수 실행 컨텍스트가 생성된다.
- 실행 컨텍스트는 콜 스택에 쌓였다가 함수 종료 시 제거된다.
- 각 실행 컨텍스트 안에는 **렉시컬 환경(Lexical Environment)**이 있다. 여기에 식별자 정보, 외부 스코프 참조, `this` 바인딩 같은 정보가 담긴다.

여기서 핵심적인 구분이 하나 있다.

**실행 컨텍스트**는 함수가 끝나면 콜 스택에서 제거된다. 하지만 그 안의 **렉시컬 환경**은, 다른 함수가 참조하고 있으면 GC되지 않고 살아남는다. 이게 바로 클로저가 가능한 이유다.

```js
function outer() {
  let count = 0; // outer의 렉시컬 환경에 저장됨
  return function inner() {
    return ++count;
  };
}

const fn = outer();
// 이 시점에서 outer의 실행 컨텍스트는 콜 스택에서 사라졌다.
// 그러나 inner가 outer의 렉시컬 환경을 참조하고 있으므로,
// count가 담긴 렉시컬 환경은 살아있다.
fn(); // 1
fn(); // 2
```

클로저를 단순히 "함수가 외부 변수에 접근하는 현상" 정도로 이해하면 얕다. 실제로는 **실행 컨텍스트는 사라지지만 렉시컬 환경은 참조가 있는 한 유지된다**는 것이 핵심이다.

### 자주 하는 오해

- 실행 컨텍스트 = 콜 스택 → 완전히 같지는 않다. 콜 스택은 컨텍스트가 쌓이는 구조이고, 실행 컨텍스트는 그 안의 하나의 실행 환경이다.
- 함수가 끝나면 그 안의 변수가 전부 사라진다 → 틀림. 렉시컬 환경이 다른 함수에 의해 참조되면 살아남는다. 이게 클로저다.
- 호이스팅은 파서가 위로 끌어올린다 → 실제로는 실행 컨텍스트 생성 단계에서 선언 정보가 먼저 등록된다고 이해하는 편이 정확하다.

### 면접 압축 답변

- **10초**: 실행 컨텍스트는 JS 코드 실행에 필요한 환경 정보 묶음입니다. 함수가 끝나면 콜 스택에서 제거되지만, 내부의 렉시컬 환경은 클로저에 의해 유지될 수 있습니다.
- **30초**: 전역 실행 컨텍스트가 먼저 만들어지고, 함수 호출마다 새로운 컨텍스트가 콜 스택에 쌓입니다. 각 컨텍스트 안에는 변수/함수 선언 정보, 외부 스코프 참조, `this` 바인딩 등을 담은 렉시컬 환경이 있습니다. 실행 컨텍스트는 함수 종료 시 스택에서 제거되지만, 렉시컬 환경은 다른 함수가 참조하면 살아남아서 클로저가 가능해집니다.

> 이제 "실행될 때 작업 공간이 생긴다"는 걸 알았다. 다음엔 그 작업 공간이 **외부 스코프를 어떤 기준으로 참조하는지** 봐야 한다. 그게 렉시컬 스코프다.

## 3. 렉시컬 스코프는 왜 클로저의 전제인가

자바스크립트에서 함수는 어디서 호출됐는지가 아니라 **어디서 선언됐는지**를 기준으로 외부 변수를 찾는다. 이 선언 위치 기반 규칙이 렉시컬 스코프다.

### 핵심 개념

- 함수의 상위 스코프는 호출 위치가 아니라 선언 위치로 결정된다.
- 따라서 반환된 함수가 다른 곳에서 실행되어도 원래 선언 당시의 외부 환경을 본다.

```js
let value = "global";

function outer() {
  let value = "outer";
  return function inner() {
    return value; // 선언 위치 기준 → "outer"
  };
}

function other() {
  let value = "other";
  const fn = outer();
  return fn(); // 호출은 other 안이지만, inner는 outer 안에서 선언됨
}

other(); // "outer" — 호출 위치(other)가 아닌 선언 위치(outer) 기준
```

클로저가 가능한 근거가 바로 여기 있다. 만약 자바스크립트가 선언 위치가 아니라 호출 위치 기준으로 외부 변수를 찾았다면(동적 스코프), 내부 함수가 안정적으로 외부 상태를 기억하기 어렵다.

### 자주 하는 오해

- 함수가 어디서 호출됐느냐가 상위 스코프를 정한다 → 틀림. 위 예시에서 `inner()`는 `other` 안에서 호출됐지만 `outer`의 변수를 본다.
- 렉시컬 스코프와 `this`는 같은 규칙으로 결정된다 → 틀림. 일반 함수의 `this`는 호출 방식으로 결정되고, 렉시컬 스코프는 선언 위치로 결정된다. 단, **화살표 함수의 `this`는 예외적으로 렉시컬**이다. 화살표 함수는 자체 `this`를 갖지 않고 선언된 위치의 `this`를 그대로 사용한다.

### 면접 압축 답변

렉시컬 스코프는 함수가 어디에서 호출됐는지가 아니라 어디에 선언됐는지에 따라 상위 스코프가 결정되는 규칙입니다. 그래서 반환된 함수도 선언 당시의 외부 변수를 계속 참조할 수 있고, 이게 클로저의 전제가 됩니다. 참고로 `this`는 일반 함수에서는 호출 방식으로 결정되지만, 화살표 함수에서는 렉시컬하게 결정됩니다.

> 이제 내부 함수가 외부 환경을 바라볼 수 있는 기준을 알았다. 다음 질문은 자연스럽다. **그 외부 환경 참조가 함수 종료 후에도 유지될 수 있나?** 이게 클로저다.

## 4. 클로저는 왜 상태 보존으로 이어지나

### 핵심 개념

클로저는 **함수와 그 함수가 참조하는 렉시컬 환경의 조합**이다. 즉 외부 함수 실행이 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있는 상태다.

이 구조 덕분에 자바스크립트 함수는 단순한 계산 단위가 아니라 **상태를 품은 행위자**처럼 동작할 수 있다.

```js
function makeCounter() {
  let count = 0; // 이 변수는 makeCounter가 끝나도 사라지지 않는다
  return {
    increment: () => ++count,
    getCount: () => count,
  };
}

const counter = makeCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.getCount(); // 2 — count는 살아있다

// 핵심: increment와 getCount는 같은 렉시컬 환경의 count를 참조한다.
// makeCounter() 실행 컨텍스트는 사라졌지만, 렉시컬 환경은 유지된다.
```

여기서 중요한 점이 하나 있다. 클로저는 **값을 복사해서 저장하는 게 아니라, 환경을 참조**한다. 그래서 `increment`로 `count`를 바꾸면 `getCount`에서도 바뀐 값이 보인다. 두 함수가 같은 렉시컬 환경을 공유하기 때문이다.

이 "환경 참조" 방식은 `let`/`var`처럼 변경 가능한(mutable) 바인딩에서 특히 중요하다. 나중에 React로 넘어가면, `const`로 선언된 불변 바인딩 위에서 클로저가 동작하는 경우를 보게 되는데, 이때는 조금 다른 양상이 된다. 이건 섹션 6에서 다시 다룬다.

### 자주 하는 오해

- 클로저는 값을 복사해서 저장한다 → 틀림. 환경을 참조하므로, 같은 환경을 공유하는 함수들은 같은 변수의 최신 값을 본다.
- 클로저는 특별한 문법이다 → 틀림. 함수가 만들어지는 기본 동작에서 자연스럽게 생긴다. 자바스크립트에서는 모든 함수가 클로저다.

### 실무 연결

- 디바운스에서 타이머 ID를 기억할 수 있다.
- 이벤트 핸들러 팩토리에서 특정 인자를 바인딩할 수 있다.
- 외부에서 직접 만지지 못하는 상태를 함수 안에 숨길 수 있다(모듈 패턴).

### 면접 압축 답변

클로저는 함수가 선언될 당시의 렉시컬 환경을 계속 참조할 수 있는 구조입니다. 값을 복사하는 게 아니라 환경 자체를 참조하기 때문에, 외부 함수가 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있고, 이를 이용해 상태 보존, 디바운스, 이벤트 핸들러 팩토리 같은 패턴을 구현할 수 있습니다.

> 클로저는 유용하지만, **잘못 유지되면 오래된 값이나 필요 없는 참조도 계속 남길 수 있다.** 그게 클로저 오용 문제다.

## 5. 클로저가 잘못 사용되면 왜 문제가 되나

### 핵심 개념

클로저는 필요 이상으로 큰 객체나 DOM 참조, 오래된 콜백을 붙잡을 수 있다. 그러면 메모리 회수가 지연되거나, 의도와 다른 오래된 값을 사용하는 문제가 생긴다.

```js
function setupHandler() {
  const hugeData = new Array(1000000).fill("x"); // 거대한 데이터

  document.getElementById("btn").addEventListener("click", () => {
    // 이 핸들러가 살아있는 한, hugeData도 GC되지 않는다.
    // 실제로는 hugeData를 안 써도, 같은 렉시컬 환경에 있으면 유지될 수 있다.
    console.log("clicked");
  });
}
```

많은 사람이 "클로저 = 좋은 것"으로만 외운다. 하지만 실제 서비스에서는 클로저가 잘못 설계되면 메모리 누수, 오래된 참조 사용, 이벤트 핸들러 누적 같은 문제가 생긴다.

### 자주 하는 오해

- 클로저가 있으면 무조건 메모리 누수다 → 틀림. 참조가 필요 없어졌을 때 정리하지 않은 게 문제다.
- 메모리 누수는 브라우저가 GC를 못 해서 생긴다 → 절반만 맞다. 더 정확히는 아직 참조가 남아 있어서 GC 회수 조건이 만족되지 않는 경우가 많다.

### 면접 압축 답변

클로저 자체가 문제가 아니라, 불필요한 참조를 정리하지 않는 게 문제입니다. 이벤트 리스너를 해제하지 않거나, 필요 이상으로 큰 객체를 클로저 안에 두면 GC가 회수하지 못해 메모리 누수가 됩니다. 클로저를 쓸 때는 생명주기와 정리 시점을 함께 설계해야 합니다.

> 여기서부터 이야기가 React로 넘어간다. 순수 JS에서는 "오래된 참조를 붙잡는 문제" 정도로 보이지만, React에서는 렌더가 반복되면서 **각 렌더 시점의 값**이 존재한다. 그래서 "오래된 참조"가 훨씬 자주, 그리고 더 미묘하게 문제를 만든다.

---

## 6. React state는 왜 snapshot처럼 동작하나

React를 그냥 "함수 모음"으로 보면 잘 안 잡힌다. React 컴포넌트는 함수이긴 하지만, **렌더 단위로 다시 실행되는 함수**다.

즉 컴포넌트가 한 번 실행될 때마다, 그 렌더 시점의 `props`, `state`, `context`를 기준으로 JSX를 계산한다.

이 말은 곧:

- 첫 번째 렌더에는 첫 번째 값 세트가 있고
- 두 번째 렌더에는 두 번째 값 세트가 있고
- 세 번째 렌더에는 세 번째 값 세트가 있다

는 뜻이다.

### 섹션 4와의 연결 — 클로저 캡처 방식의 차이

섹션 4에서 "클로저는 값을 복사하는 게 아니라 환경을 참조한다"고 했다. `let count = 0`처럼 mutable 바인딩이면, 같은 환경을 공유하는 함수들이 변경된 값을 함께 본다.

그런데 React에서는 상황이 다르다.

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  // count는 const — 이 렌더 안에서는 절대 바뀌지 않는다

  const handleClick = () => {
    setTimeout(() => {
      console.log(count); // 이 렌더의 count 값이 고정되어 있다
    }, 3000);
  };

  return <button onClick={handleClick}>{count}</button>;
}
```

`useState`가 반환하는 값은 `const`다. 매 렌더마다 새로운 실행이 일어나고, 새로운 `const count`가 만들어진다. 클로저는 여전히 환경을 참조하지만, 그 환경 안의 바인딩이 불변(`const`)이기 때문에 **결과적으로 특정 렌더의 값이 고정된 것처럼** 동작한다.

이게 React state가 "snapshot처럼 동작한다"는 말의 정확한 의미다. 렌더 1의 핸들러는 렌더 1의 `count`를 보고, 렌더 2의 핸들러는 렌더 2의 `count`를 본다. 각자의 렉시컬 환경이 다르기 때문이다.

### 자주 하는 오해

- state는 그냥 함수 안의 변수다 → 틀림. React가 렌더 간에 관리하는 외부 값이다. `useState`는 매 렌더마다 현재 시점의 값을 반환한다.
- `setState`를 부르면 지금 실행 중인 함수 안의 state 값도 바로 바뀐다 → 틀림. 현재 렌더의 `const` 값은 바뀔 수 없다. 다음 렌더에서 새 값이 반환된다.

### 면접 압축 답변

React의 state는 현재 렌더 기준 snapshot처럼 동작합니다. 컴포넌트가 렌더될 때마다 `useState`는 그 시점의 값을 `const`로 반환하고, 이벤트 핸들러나 effect는 자신이 생성된 렌더의 값을 클로저로 캡처합니다. `setState`는 현재 값을 바꾸는 게 아니라 다음 렌더를 예약합니다.

## 7. setState가 즉시 바뀌지 않는 이유

"내가 `setCount(1)` 했는데 왜 바로 `count`가 1이 안 되지?"

이건 React가 게을러서가 아니라, 렌더 모델 자체가 그렇게 생겼기 때문이다. 현재 실행 중인 이벤트 핸들러는 이미 어떤 렌더의 snapshot 위에서 돌아가고 있다. 그 안에서 `setState`를 호출해도, React는 지금 돌고 있는 그 함수 내부 변수 값을 덮어쓰지 않는다. 대신 **다음 렌더를 예약**한다.

### 예시

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    setCount(count + 1);
    // 두 호출 모두 같은 렌더의 count(= 0)를 기준으로 계산한다.
    // 결과: count는 1이 된다. 2가 아니다.
    console.log(count); // 여전히 0 — const는 바뀌지 않는다
  };

  return <button onClick={handleClick}>{count}</button>;
}
```

React 18부터는 **automatic batching**이 기본이다. 이벤트 핸들러뿐 아니라 `setTimeout`, Promise 콜백 안에서도 여러 `setState` 호출이 하나의 렌더로 합쳐진다. 위 코드에서 두 `setCount` 호출은 같은 `count` snapshot(= 0)을 기준으로 계산하므로, 둘 다 `0 + 1 = 1`이 되어 결과는 1이다.

함수형 업데이트는 이 문제를 해결한다.

```tsx
const handleClick = () => {
  setCount((prev) => prev + 1); // prev = 0 → 1
  setCount((prev) => prev + 1); // prev = 1 → 2
  // 결과: count는 2가 된다.
};
```

이건 "현재 렌더에서 보이는 count"를 다시 쓰는 게 아니라, 업데이트 큐에서 **이전 결과를 이어받아 계산**하는 방식이다.

여기서 배워야 하는 핵심은 두 가지다.

1. `setState`는 변수 재할당이 아니다. 다음 렌더를 예약하는 것이다.
2. 현재 렌더의 `const` 값은 바뀌지 않는다. 누적 업데이트가 필요하면 함수형 업데이트를 쓴다.

### 자주 하는 오해

- React state가 비동기다 → 표현이 너무 뭉뚱그려져 있다. 더 정확히는 현재 렌더의 값을 즉시 바꾸지 않고 다음 렌더를 스케줄링한다고 보는 편이 좋다. `setState` 호출 자체는 동기적으로 큐에 들어간다.

### 면접 압축 답변

`setState`는 현재 렌더의 state 변수를 즉시 바꾸는 것이 아니라 다음 렌더를 예약합니다. React 18부터는 어디서든 automatic batching이 적용되어, 같은 함수 안의 여러 `setState`가 하나의 렌더로 합쳐집니다. 같은 snapshot 기준으로 계산되므로, 누적 업데이트가 필요하면 함수형 업데이트(`prev => prev + 1`)를 써야 합니다.

> 이걸 이해하면, stale closure가 왜 생기고 함수형 업데이트가 왜 필요한지 한 번에 묶인다. 이제 다시 클로저로 돌아가 보자.

## 8. stale closure는 정확히 어디서 터지나

클로저는 함수가 외부 환경을 기억하는 구조다. React 컴포넌트도 함수이기 때문에, 이벤트 핸들러나 effect 내부 함수는 **그 렌더의 값**을 캡처한다.

문제는 컴포넌트는 계속 다시 렌더되는데, 예전 렌더에서 만들어진 함수가 **오래 살아남을 수 있다**는 점이다.

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count);
      // 렌더 0에서 만들어진 함수 → count는 항상 0
    }, 1000);
    return () => clearInterval(id);
  }, []);
  // dependency가 [] → effect는 최초 렌더에서만 실행
  // → setInterval 콜백은 렌더 0의 count(= 0)를 캡처한 채 영원히 남는다

  return (
    <button onClick={() => setCount((p) => p + 1)}>
      {count}
    </button>
  );
}
// 버튼을 아무리 눌러도 콘솔에는 항상 0이 찍힌다.
```

이게 stale closure의 전형적인 예시다. 발생하기 쉬운 경우:

- `setInterval` / `setTimeout`
- 이벤트 리스너
- effect 내부 비동기 콜백
- Promise `then` / `catch`
- 외부 라이브러리에 넘긴 callback

이런 함수들은 한번 등록되면 나중에도 계속 실행될 수 있다. 그런데 그 함수가 예전 렌더에서 만들어졌다면, 최신 state가 아니라 **예전 state를 계속 본다**. 이게 stale closure다.

즉 stale closure는 "클로저가 잘못된 기술"이 아니라, **클로저가 React의 렌더 단위와 만났을 때 생기는 시점 문제**다.

### 면접 압축 답변

Stale closure는 예전 렌더에서 생성된 함수가 오래 살아남아 최신 state 대신 캡처 당시의 오래된 값을 계속 참조하는 현상입니다. `setInterval`, 이벤트 리스너, 비동기 콜백 등 장기 생존하는 함수에서 주로 발생하며, 클로저 자체의 문제가 아니라 React의 렌더별 값 캡처 모델과 결합되어 생기는 시점 불일치입니다.

## 9. useEffect dependency는 왜 중요한가

`useEffect`는 본질적으로 "렌더 결과를 바탕으로 외부 시스템과 동기화하는 훅"이다. API 요청, 이벤트 구독, 타이머 등록, 외부 라이브러리 초기화 같은 것들이 여기에 해당한다.

effect 안에서 어떤 값들을 읽는다면, 그 effect는 그 값들에 의존하고 있는 것이다. 바로 그래서 dependency 배열이 필요하다.

dependency 배열은 단순히 "이 effect를 언제 다시 돌릴까요?" 옵션이 아니다. 더 정확히는:

> **이 effect가 어떤 값에 의존하는지 선언하는 계약**

이다.

### cleanup 함수 — effect의 생명주기

dependency를 이해하려면 cleanup 개념도 함께 알아야 한다. effect가 재실행될 때, React는 **이전 effect의 cleanup을 먼저 실행**한 다음 새 effect를 실행한다.

```tsx
useEffect(() => {
  const id = setInterval(() => {
    console.log(count);
  }, 1000);

  // cleanup: 다음 렌더에서 effect가 다시 실행되기 전에 호출된다
  return () => clearInterval(id);
}, [count]);
// count가 바뀔 때마다:
// 1. 이전 interval을 clearInterval로 정리
// 2. 새 count를 캡처한 새 interval을 등록
```

이 cleanup → 재실행 사이클이 stale closure를 방지하는 핵심 메커니즘이다. dependency에 `count`를 넣으면, count가 바뀔 때마다 오래된 interval이 정리되고 최신 값을 캡처한 새 interval이 등록된다.

dependency를 빼먹으면 effect 내부 함수는 예전 렌더에서 만들어진 채 남고, cleanup도 호출되지 않아 그 안에서는 예전 값만 본다.

### 실무에서 생기는 문제

- 검색어가 바뀌었는데 예전 검색어로 요청을 계속 보냄
- interval이 최신 count를 못 봄
- 구독 해제 타이밍이 꼬임
- 이전 props 기준 로직이 계속 동작함

### 자주 하는 오해

- dependency 배열은 rerender 최적화 옵션이다 → 본질은 최적화보다 의존 관계 명시다.
- 빈 배열을 쓰면 안전하다 → 오히려 오래된 값을 영원히 잡아둘 수 있다. 빈 배열은 "이 effect는 아무 값에도 의존하지 않는다"는 선언이므로, 안에서 state나 props를 읽으면 거짓말이 된다.
- cleanup은 컴포넌트가 언마운트될 때만 실행된다 → 틀림. dependency가 바뀌어 effect가 재실행될 때도 이전 cleanup이 먼저 실행된다.

### 면접 압축 답변

`useEffect` dependency 배열은 effect가 어떤 값에 의존하는지 선언하는 계약입니다. dependency가 바뀌면 이전 effect의 cleanup이 먼저 실행되고, 새 값을 캡처한 effect가 다시 실행됩니다. 필요한 값을 빼면 cleanup/재실행 사이클이 동작하지 않아 stale closure 문제가 생깁니다.

> 그런데 항상 dependency를 늘리는 게 답은 아니다. 어떤 경우에는 effect를 재실행시키고 싶지 않으면서도 **항상 최신 값**은 읽고 싶을 수 있다. 이때 `ref`가 등장한다.

---

## 10. ref는 왜 stale closure 회피에 쓰이나

`ref`는 React에서 아주 독특한 존재다.

- state처럼 값을 담을 수 있는데
- 바뀌어도 렌더를 일으키지 않고
- 같은 객체를 계속 유지한다

즉 `ref.current`는 **렌더를 트리거하지 않는 최신 값 보관함**처럼 쓸 수 있다.

```tsx
function useInterval(callback: () => void, delay: number) {
  const savedCallback = useRef(callback);

  // 매 렌더마다 ref에 최신 callback을 넣어둔다
  useEffect(() => {
    savedCallback.current = callback;
  });

  // interval은 한 번만 등록한다
  useEffect(() => {
    const id = setInterval(() => {
      savedCallback.current(); // 항상 최신 callback을 실행
    }, delay);
    return () => clearInterval(id);
  }, [delay]);
}
```

이 패턴에서 `setInterval` 콜백은 최초 렌더에서 만들어진 함수지만, 실행할 때마다 `savedCallback.current`를 통해 **최신 callback**을 읽는다. ref 객체 자체는 렌더 간에 동일한 객체이므로, 오래된 클로저 안에서도 `.current`로 최신 값에 접근할 수 있다.

핵심적인 역할 구분:

- **state**는 UI 반영을 위한 값 — 바뀌면 렌더가 필요하다
- **ref**는 렌더와 무관한 최신 참조 유지용 값 — 바뀌어도 렌더할 필요 없다

### 자주 하는 오해

- `ref`는 state 대체재다 → 틀림. UI를 갱신해야 하는 값은 state가 맞다. ref 값이 바뀌어도 화면은 안 바뀐다.
- stale closure가 생기면 무조건 `ref`로 해결하면 된다 → 틀림. 먼저 dependency를 정확히 쓰고, 함수형 업데이트를 쓰고, effect를 정말 필요한 곳에만 쓴 뒤에도 "재렌더 없이 최신값 참조"가 꼭 필요할 때 `ref`를 쓰는 게 맞다.

### 면접 압축 답변

`ref`는 렌더를 발생시키지 않으면서 값을 유지하는 안정적인 컨테이너입니다. 렌더 간에 같은 객체가 유지되므로, 오래 살아 있는 콜백 안에서 `ref.current`를 읽으면 항상 최신 값을 볼 수 있습니다. 단, ref는 stale closure의 만능 해결책이 아니라 dependency 정리와 함수형 업데이트를 먼저 고려한 뒤 사용해야 합니다.

---

## 11. useMemo / useCallback은 왜 결국 참조 안정성 문제인가

지금까지 본 걸 한 줄로 요약하면 이렇다.

- 함수는 클로저로 값을 기억한다
- React는 렌더마다 새로운 const 바인딩을 만든다
- callback / effect는 특정 렌더의 값을 캡처한다
- dependency를 잘못 다루면 stale closure가 생긴다
- 최신 값 참조가 필요하면 `ref`를 고려한다

그다음 자연스럽게 이어지는 질문이 있다. **함수나 값의 참조를 안정적으로 유지하는 문제는 어떻게 보나?**

많은 사람이 이 훅들을 "성능 최적화 훅"으로만 외운다. 그런데 실제로는 두 가지 용도가 있다.

### useMemo의 두 가지 용도

1. **비싼 계산 결과 캐싱** — 원시값이든 객체든, 매 렌더마다 다시 계산하기 비싼 값을 메모이제이션한다.
2. **객체/배열의 참조 안정화** — 자식 컴포넌트에 props로 넘길 때, 매 렌더마다 새 객체가 만들어지면 자식이 불필요하게 리렌더된다. `useMemo`로 참조를 유지하면 이를 방지한다.

```tsx
// 용도 1: 비싼 계산 캐싱
const sortedList = useMemo(() => {
  return items.sort((a, b) => a.score - b.score);
}, [items]);

// 용도 2: 참조 안정화
const style = useMemo(() => ({
  color: darkMode ? "white" : "black",
}), [darkMode]);
// style은 darkMode가 바뀔 때만 새 객체가 된다
```

### useCallback

`useCallback`은 함수의 참조를 안정화한다. `useCallback(fn, deps)`는 `useMemo(() => fn, deps)`와 동일하다.

```tsx
const handleClick = useCallback(() => {
  setCount((prev) => prev + 1);
}, []);
// handleClick의 참조가 유지되므로,
// React.memo로 감싼 자식에게 넘겨도 불필요한 리렌더가 없다
```

### React.memo와의 관계

`React.memo`는 props가 같으면 자식 렌더를 건너뛸 수 있게 해준다. 그런데 props 비교는 **얕은 비교(shallow comparison)**다. 매 렌더마다 새로 만들어진 객체나 함수는 참조가 다르므로 "다른 props"로 판단된다. 그래서 `useMemo`/`useCallback`으로 참조를 안정화해야 `React.memo`가 효과를 발휘한다.

이것도 결국 dependency를 제대로 이해해야만 쓸 수 있다. dependency를 틀리면 stale closure가 생기고, 오래된 값이 memoized 되고, "최적화한 줄 알았는데 버그 만든 상태"가 된다.

### 자주 하는 오해

- `useCallback`을 쓰면 무조건 성능이 좋아진다 → 틀림. `React.memo`로 감싼 자식이 없으면 참조 안정화의 이점이 없다.
- dependency를 일부러 빼면 더 최적화된다 → 틀림. 오래된 값을 붙잡아서 버그가 된다.
- 원시값에도 참조 안정화가 필요하다 → 틀림. 원시값은 값 자체로 비교되므로 참조 안정화 개념이 없다.

### 면접 압축 답변

`useMemo`는 비싼 계산 캐싱과 객체 참조 안정화, `useCallback`은 함수 참조 안정화에 사용됩니다. `React.memo`와 함께 쓸 때 불필요한 자식 리렌더를 방지할 수 있습니다. 단, 이 훅들도 dependency 배열에 의존하므로 dependency를 잘못 다루면 stale closure로 이어집니다.
