# React Closure Learning Flow

이 문서는 단순 면접 답변 암기용 문답집이 아니라, **한 개념이 왜 다음 개념을 필요하게 만드는지**를 따라가며 학습하고, 마지막에 면접 답변으로 압축할 수 있게 만든 학습형 플로우 문서다.

핵심 원칙은 아래와 같다.

1. 질문을 독립적으로 외우지 않는다.
2. **선행 개념 → 원리 → 오해 교정 → 실무 연결 → 다음 개념** 순서로 읽는다.
3. 마지막에만 10초 / 30초 / 1분 답변으로 압축한다.
4. 한 문항을 이해했으면 반드시 다음 문항으로 이어지는 다리를 만든다.

---

# 챕터 1. 실행 컨텍스트에서 stale closure까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **자바스크립트 함수가 값을 어떻게 기억하는지 이해하면, React에서 값이 왜 오래된 채로 남는지도 설명할 수 있다.**

---

## 1. var / let / const를 먼저 보는 이유

클로저나 stale closure를 제대로 이해하려면, 가장 먼저 **변수가 어느 범위에서 살아 있는지**부터 알아야 한다. 많은 사람이 클로저를 어렵게 느끼는 이유는 사실 클로저가 어려워서가 아니라, 변수의 생존 범위와 스코프를 먼저 잡지 않았기 때문이다.

### 핵심 개념

- `var`는 함수 스코프다.
- `let`, `const`는 블록 스코프다.
- 세 선언 모두 호이스팅되지만, `let`/`const`는 TDZ 때문에 초기화 전 접근이 막힌다.
- `const`는 값 전체를 불변으로 만드는 게 아니라 바인딩 재할당을 막는다.

### 왜 이게 중요한가

스코프를 이해해야 “내부 함수가 외부 변수에 접근한다”는 말이 정확히 어떤 범위의 값을 가리키는지 보인다. 이게 곧 렉시컬 스코프와 클로저의 출발점이다.

### 자주 하는 오해

- `let`/`const`는 호이스팅되지 않는다 → 틀림
- `const`는 객체도 완전히 불변이다 → 틀림
- `var`는 그냥 옛날 문법일 뿐이라 의미 없다 → 틀림. 함수 스코프와 전역 오염 문제를 이해하는 데 중요하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 “변수가 어디까지 보이느냐”를 알았으니, 다음에는 **코드가 실제로 실행될 때 이 변수들을 담는 작업 공간이 어떻게 만들어지는지**를 봐야 한다. 그게 실행 컨텍스트다.

### 면접 압축 답변

- 10초: `var`는 함수 스코프, `let`/`const`는 블록 스코프입니다. `let`/`const`는 TDZ가 있고, `const`는 재할당만 막습니다.
- 30초: 세 선언 모두 호이스팅되지만 `var`는 `undefined`로 초기화되고, `let`/`const`는 TDZ 구간에서 접근이 막힙니다. 실무에서는 재선언과 스코프 버그를 줄이기 위해 기본적으로 `const`, 필요할 때만 `let`을 씁니다.

## 2. 실행 컨텍스트는 왜 필요한가

스코프 규칙만으로는 아직 부족하다. 실제 실행 순간에는 “현재 어떤 함수가 돌고 있고, 어떤 변수들이 사용 가능하며, 외부 스코프는 어디인지”를 관리해야 한다. 이 실행 단위의 작업 공간이 실행 컨텍스트다.

### 핵심 개념

- 전역 코드가 시작되면 전역 실행 컨텍스트가 만들어진다.
- 함수가 호출될 때마다 새로운 함수 실행 컨텍스트가 생성된다.
- 실행 컨텍스트는 콜 스택에 쌓였다가 함수 종료 시 제거된다.
- 여기에는 식별자 정보, 외부 스코프 참조, `this` 바인딩 같은 정보가 포함된다.

### 왜 이게 중요한가

클로저를 단순히 “함수가 외부 변수에 접근하는 현상” 정도로 이해하면 얕다. 실제로는 함수 실행이 끝난 뒤에도 어떤 환경 정보가 유지될 수 있는지가 핵심인데, 그걸 보려면 실행 컨텍스트와 렉시컬 환경 개념이 먼저 있어야 한다.

### 자주 하는 오해

- 실행 컨텍스트 = 콜 스택 → 완전히 같지는 않다. 콜 스택은 컨텍스트가 쌓이는 구조이고, 실행 컨텍스트는 그 안의 하나의 실행 환경이다.
- 호이스팅은 파서가 위로 끌어올린다 → 실제로는 생성 단계에서 선언 정보가 먼저 등록된다고 이해하는 편이 정확하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 “실행될 때 작업 공간이 생긴다”는 걸 알았다. 다음엔 그 작업 공간이 **외부 스코프를 어떤 기준으로 참조하는지** 봐야 한다. 그게 렉시컬 스코프다.

### 면접 압축 답변

- 10초: 실행 컨텍스트는 JS 코드 실행에 필요한 환경 정보 묶음입니다.
- 30초: 전역 실행 컨텍스트가 먼저 만들어지고, 함수 호출마다 새로운 컨텍스트가 콜 스택에 쌓입니다. 여기에는 변수/함수 선언 정보, 외부 스코프 참조, `this` 바인딩 등이 포함됩니다.

---

## 3. 렉시컬 스코프는 왜 클로저의 전제인가

자바스크립트에서 함수는 어디서 호출됐는지가 아니라 **어디서 선언됐는지**를 기준으로 외부 변수를 찾는다. 이 선언 위치 기반 규칙이 렉시컬 스코프다.

### 핵심 개념

- 함수의 상위 스코프는 호출 위치가 아니라 선언 위치로 결정된다.
- 따라서 반환된 함수가 다른 곳에서 실행되어도 원래 선언 당시의 외부 환경을 본다.

### 왜 이게 중요한가

클로저가 가능한 근거가 바로 여기 있다. 만약 자바스크립트가 선언 위치가 아니라 호출 위치 기준으로 외부 변수를 찾았다면, 내부 함수가 안정적으로 외부 상태를 기억하기 어렵다.

### 자주 하는 오해

- 함수가 어디서 호출됐느냐가 상위 스코프를 정한다 → 틀림
- 렉시컬 스코프와 `this`는 같은 규칙으로 결정된다 → 틀림. `this`는 호출 방식, 렉시컬 스코프는 선언 위치다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 내부 함수가 외부 환경을 바라볼 수 있는 기준을 알았다. 다음 질문은 자연스럽다. **그 외부 환경 참조가 함수 종료 후에도 유지될 수 있나?** 이게 클로저다.

### 면접 압축 답변

렉시컬 스코프는 함수가 어디에서 호출됐는지가 아니라 어디에 선언됐는지에 따라 상위 스코프가 결정되는 규칙입니다. 그래서 반환된 함수도 선언 당시의 외부 변수를 계속 참조할 수 있고, 이게 클로저의 전제가 됩니다.

---

## 4. 클로저는 왜 상태 보존으로 이어지나

### 핵심 개념

클로저는 **함수와 그 함수가 참조하는 렉시컬 환경의 조합**이다. 즉 외부 함수 실행이 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있는 상태다.

### 왜 이게 중요한가

이 구조 덕분에 자바스크립트 함수는 단순한 계산 단위가 아니라 **상태를 품은 행위자**처럼 동작할 수 있다. 카운터, 디바운스, 팩토리 함수, private state 비슷한 패턴이 모두 여기서 나온다.

### 자주 하는 오해

- 클로저는 값을 복사해서 저장한다 → 틀림. 값을 통째로 복제해 들고 있는 게 아니라, 그 값을 찾을 수 있는 환경을 계속 참조한다.
- 클로저는 특별한 문법이다 → 틀림. 함수가 만들어지는 기본 동작에서 자연스럽게 생긴다.

### 실무 연결

- 디바운스에서 타이머 ID를 기억할 수 있다.
- 이벤트 핸들러 팩토리에서 특정 인자를 바인딩할 수 있다.
- 외부에서 직접 만지지 못하는 상태를 함수 안에 숨길 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

여기서 한 번 더 나아가야 한다. 클로저는 유용하지만, **잘못 유지되면 오래된 값이나 필요 없는 참조도 계속 남길 수 있다.** 그게 클로저 오용 문제다.

### 면접 압축 답변

클로저는 함수가 선언될 당시의 렉시컬 환경을 계속 참조할 수 있는 구조입니다. 그래서 외부 함수가 끝난 뒤에도 내부 함수가 외부 변수에 접근할 수 있고, 이를 이용해 상태 보존, 디바운스, 이벤트 핸들러 팩토리 같은 패턴을 구현할 수 있습니다.

---

## 5. 클로저가 잘못 사용되면 왜 문제가 되나

### 핵심 개념

클로저는 필요 이상으로 큰 객체나 DOM 참조, 오래된 콜백을 붙잡을 수 있다. 그러면 메모리 회수가 지연되거나, 의도와 다른 오래된 값을 사용하는 문제가 생긴다.

### 왜 이게 중요한가

많은 사람이 “클로저 = 좋은 것”으로만 외운다. 하지만 실제 서비스에서는 클로저가 잘못 설계되면 메모리 누수, 오래된 참조 사용, 이벤트 핸들러 누적 같은 문제가 생긴다. 여기서부터 React stale closure 이야기가 훨씬 쉬워진다.

### 자주 하는 오해

- 클로저가 있으면 무조건 메모리 누수다 → 틀림
- 메모리 누수는 브라우저가 GC를 못 해서 생긴다 → 절반만 맞다. 더 정확히는 아직 참조가 남아 있어서 회수 조건이 만족되지 않는 경우가 많다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 중요한 분기점이다. 순수 JS에서는 “오래된 참조를 붙잡는 문제” 정도로 보일 수 있지만, React에서는 렌더가 반복되면서 **각 렌더 시점의 값**이 존재한다. 그래서 “오래된 참조”가 더 자주 문제를 만든다. 다음은 React state snapshot이다.

## 6. React state는 왜 snapshot처럼 동작하나

### 먼저 연결하기

앞에서 봤듯이, 함수는 자신이 만들어질 때의 환경을 기억한다. React 컴포넌트도 함수이기 때문에, 렌더될 때마다 그 렌더 시점의 `props`와 `state`를 바탕으로 실행된다.

### 핵심 개념

- React는 state를 mutable 변수처럼 즉석에서 바꾸지 않는다.
- 각 렌더는 그 시점의 state 값을 가진다.
- 이벤트 핸들러나 effect는 특정 렌더에서 만들어진 함수이므로, 그 렌더의 값을 캡처할 수 있다.

### 왜 이게 중요한가

이걸 이해하지 못하면 “왜 `setState`를 했는데 바로 값이 안 바뀌지?” “왜 effect 안에서 오래된 값이 보이지?”가 전부 이상하게 느껴진다.

### 자주 하는 오해

- state는 그냥 함수 안의 변수다 → 틀림
- `setState`를 부르면 지금 실행 중인 함수 안의 state 값도 바로 바뀐다 → 틀림

### 여기서 다음 개념으로 어떻게 넘어가나

이제 자연스럽게 다음 질문이 생긴다. **그렇다면 `setState`는 정확히 무엇을 하는가? 왜 즉시 값을 바꾸지 않는가?**

### 면접 압축 답변

React의 state는 현재 렌더 기준 snapshot처럼 동작합니다. `setState`는 현재 핸들러 안의 값을 즉시 바꾸는 게 아니라 다음 렌더를 예약하기 때문에, 특정 렌더에서 생성된 함수는 그 시점의 state를 캡처할 수 있습니다.

---

## 7. setState가 즉시 바뀌지 않는 이유

### 핵심 개념

- `setState`는 “현재 변수 값을 바꾸는 명령”이라기보다 “다음 렌더를 예약하는 요청”에 가깝다.
- React는 업데이트를 batching할 수 있다.
- 같은 렌더 안에서 여러 번 `setCount(count + 1)`를 호출하면 같은 snapshot을 기준으로 계산될 수 있다.
- 함수형 업데이트는 이 문제를 피한다.

### 예시

```ts
setCount(count + 1);
setCount(count + 1);
```

이 코드는 사람 입장에서는 2 증가처럼 보이지만, React는 현재 렌더의 같은 `count` snapshot을 기준으로 계산할 수 있다.

```ts
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);
```

이 방식은 업데이트 큐에서 이전 결과를 이어받아 계산하므로 누적 업데이트가 가능하다.

### 왜 이게 중요한가

이걸 이해하면 stale closure, batching, 함수형 업데이트, dependency 관리가 한 덩어리로 보이기 시작한다.

### 자주 하는 오해

- React state가 비동기다 → 표현이 너무 뭉뚱그려져 있다. 더 정확히는 현재 렌더의 값을 즉시 바꾸지 않고 다음 렌더를 스케줄링한다고 보는 편이 좋다.

### 여기서 다음 개념으로 어떻게 넘어가나

그렇다면 이제 이런 질문이 나온다. **현재 렌더의 값이 캡처되고 다음 렌더는 나중에 오는데, effect나 callback은 어떤 값을 기준으로 동작할까?** 여기서 `useEffect` dependency가 중요해진다.

### 면접 압축 답변

`setState`는 현재 렌더의 state 변수를 즉시 바꾸는 것이 아니라 다음 렌더를 예약합니다. 그래서 같은 핸들러 안에서는 기존 snapshot 기준으로 동작하고, 누적 업데이트가 필요하면 함수형 업데이트를 써야 합니다.

---

## 8. useEffect dependency는 왜 중요한가

### 핵심 개념

- Effect는 렌더 결과를 기반으로 외부 시스템과 동기화하는 훅이다.
- dependency 배열은 “이 effect가 어떤 값에 의존하는가”를 선언한다.
- 배열에 필요한 값을 빼면, effect 내부 함수가 예전 렌더 값을 잡은 채 남을 수 있다.

### 왜 이게 중요한가

여기서 stale closure가 실제 버그가 된다. 예전 state로 interval이 돌거나, 오래된 props로 API 요청을 보내거나, 최신 값을 반영하지 못하는 현상이 생긴다.

### 자주 하는 오해

- dependency 배열은 rerender 최적화 옵션이다 → 본질은 최적화보다 의존 관계 명시다.
- 빈 배열을 쓰면 안전하다 → 오히려 오래된 값을 영원히 잡아둘 수 있다.

### 실무 연결

- 검색어가 바뀌어도 이전 검색어 기준 요청이 계속 나감
- interval 콜백이 최신 state를 못 봄
- 구독 해제가 늦거나 cleanup이 꼬임

### 여기서 다음 개념으로 어떻게 넘어가나

그런데 어떤 경우에는 effect를 재실행시키고 싶지 않으면서도 **항상 최신 값**은 보고 싶을 수 있다. 이때 `ref`가 등장한다.

### 면접 압축 답변

`useEffect` dependency 배열은 effect가 어떤 값에 의존하는지 선언하는 계약입니다. 필요한 값을 빼면 effect 내부 함수가 예전 렌더 값을 캡처한 채 남아서 stale closure 문제가 생길 수 있습니다.

---

## 9. ref는 왜 stale closure 회피에 쓰이나

### 핵심 개념

- `ref`는 렌더를 일으키지 않으면서 값을 담아둘 수 있는 안정적인 컨테이너다.
- `.current`는 같은 객체를 유지하므로, 콜백이 오래 살아 있어도 최신 값을 읽도록 설계할 수 있다.

### 왜 이게 중요한가

effect를 매번 다시 만들면 비용이나 부작용이 커질 수 있는 경우가 있다. 예를 들어 이벤트 리스너, interval, 외부 라이브러리 콜백처럼 오래 살아 있는 로직에서 최신 값을 읽어야 할 때 `ref`가 유용하다.

### 자주 하는 오해

- `ref`는 state 대체재다 → 틀림. UI를 갱신해야 하는 값은 state가 맞다.
- stale closure가 생기면 무조건 `ref`로 해결하면 된다 → 틀림. 먼저 dependency를 정확히 쓰고, `ref`는 정말 “재렌더 없이 최신값 참조”가 필요한 경우에 써야 한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 dependency와 `ref`를 이해했으니, 다음으로는 **참조 안정성** 자체가 중요한 이유가 보인다. 그게 `useMemo`, `useCallback`, `React.memo`와 연결된다.

### 면접 압축 답변

`ref`는 렌더를 발생시키지 않으면서 값을 유지하는 안정적인 컨테이너입니다. 오래 살아 있는 콜백이 최신 값을 읽어야 할 때 `.current`를 통해 stale closure를 완화하는 데 사용할 수 있습니다.

---

## 10. useMemo / useCallback은 왜 결국 참조 안정성 문제인가

### 핵심 개념

- `useMemo`는 계산 결과나 값의 참조를 안정화한다.
- `useCallback`은 함수 참조를 안정화한다.
- 둘 다 dependency 배열을 잘못 다루면 stale closure를 만들 수 있다.

### 왜 이게 중요한가

많은 사람이 이 훅들을 “성능 마법”처럼 외운다. 하지만 본질은 참조 안정성과 dependency 관리다. 결국 앞에서 배운 렌더 시점 값, 클로저, dependency 배열을 이해해야 이 훅도 정확히 쓸 수 있다.

### 자주 하는 오해

- `useCallback`을 쓰면 무조건 성능이 좋아진다 → 틀림
- dependency를 일부러 빼면 더 최적화된다 → 틀림. 오히려 오래된 값을 붙잡을 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

여기까지 오면 다음 흐름은 자연스럽게 **값/참조 모델 → 불변성 → React 리렌더링**으로 이어진다. 즉 이제는 “함수가 값을 어떻게 기억하는가”에서 “React가 변경을 어떻게 감지하는가”로 넘어갈 수 있다.

---

# stale closure 다음 흐름을 더 길게 연결해서 이해하기

여기서부터는 위 내용을 조금 더 부드럽게 이어서 읽을 수 있도록, 중간 다리 문장을 하나의 흐름으로 다시 적는다.

## React state는 왜 snapshot처럼 동작한다고 말하나

React를 그냥 “함수 모음”으로 보면 잘 안 잡힌다. React 컴포넌트는 함수이긴 하지만, **렌더 단위로 다시 실행되는 함수**다.

즉 컴포넌트가 한 번 실행될 때마다, 그 렌더 시점의 `props`, `state`, `context`를 기준으로 JSX를 계산한다.

이 말은 곧:

- 첫 번째 렌더에는 첫 번째 값 세트가 있고
- 두 번째 렌더에는 두 번째 값 세트가 있고
- 세 번째 렌더에는 세 번째 값 세트가 있다

는 뜻이다.

그래서 특정 이벤트 핸들러나 effect 내부 함수는 “컴포넌트 전체의 영원한 최신 state”를 바라보는 게 아니라, **그 함수가 생성된 렌더 시점의 값**을 기준으로 동작하게 된다.

이걸 snapshot처럼 이해하면 좋다.

즉 React state는 “하나의 mutable 변수”라기보다 **각 렌더가 가진 값의 스냅샷**에 가깝다.

많은 사람이 이렇게 생각한다.

> state는 그냥 변수인데 React가 좀 늦게 반영하는 거 아닌가?

이렇게 보면 계속 꼬인다. 더 정확하게는:

> React는 현재 렌더의 값을 바꾸는 게 아니라, 다음 렌더에서 사용할 새 값을 예약한다.

이 관점으로 봐야 한다.

---

## 그럼 setState는 왜 즉시 바뀌지 않나

이제 자연스럽게 다음 질문이 나온다.

“내가 `setCount(1)` 했는데 왜 바로 `count`가 1이 안 되지?”

이건 React가 게을러서가 아니라, 렌더 모델 자체가 그렇게 생겼기 때문이다.

현재 실행 중인 이벤트 핸들러는 이미 어떤 렌더의 snapshot 위에서 돌아가고 있다.

그 안에서 `setState`를 호출해도, React는 지금 돌고 있는 그 함수 내부 변수 값을 덮어쓰지 않는다. 대신 **다음 렌더를 예약**한다.

그래서 이 코드를 보면:

```ts
setCount(count + 1);
setCount(count + 1);
```

사람은 “2 증가”를 기대하기 쉽지만, React는 두 호출 모두 **현재 렌더의 같은 `count` 값**을 기준으로 계산할 수 있다.

그래서 함수형 업데이트가 필요해진다.

```ts
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);
```

이건 “현재 렌더에서 보이는 count”를 다시 쓰는 게 아니라, 업데이트 큐에서 **이전 결과를 이어받아 계산**하는 방식이다.

즉 여기서 배워야 하는 핵심은 두 가지다.

첫째, `setState`는 변수 재할당이 아니다.  
둘째, React는 현재 렌더를 바꾸는 게 아니라 다음 렌더를 준비한다.

이걸 이해하면, 왜 stale closure가 생기고 왜 함수형 업데이트가 필요한지 한 번에 묶인다.

---

## 그럼 stale closure는 정확히 어디서 터지나

이제 다시 클로저로 돌아가 보자.

클로저는 함수가 외부 환경을 기억하는 구조다. React 컴포넌트도 함수이기 때문에, 이벤트 핸들러나 effect 내부 함수는 **그 렌더의 값**을 캡처한다.

문제는 컴포넌트는 계속 다시 렌더되는데, 예전 렌더에서 만들어진 함수가 **오래 살아남을 수 있다**는 점이다.

대표적인 경우가 이렇다.

- `setInterval`
- `setTimeout`
- 이벤트 리스너
- effect 내부 비동기 콜백
- Promise `then` / `catch`
- 외부 라이브러리에 넘긴 callback

이런 함수들은 한번 등록되면, 나중에도 계속 실행될 수 있다.

그런데 그 함수가 예전 렌더에서 만들어졌다면, 최신 state가 아니라 **예전 state를 계속 본다**.

이게 stale closure다.

즉 stale closure는 “클로저가 잘못된 기술”이 아니라, **클로저가 React의 렌더 단위와 만났을 때 생기는 시점 문제**다.

---

## 그래서 useEffect dependency가 왜 중요해지나

이제 effect를 보자.

`useEffect`는 본질적으로 “렌더 결과를 바탕으로 외부 시스템과 동기화하는 훅”이다.

예를 들면:

- API 요청
- 이벤트 구독
- 타이머 등록
- 외부 라이브러리 초기화
- DOM 바깥 상태와 연결

그런데 effect 안에서 어떤 값들을 읽는다면, 그 effect는 그 값들에 의존하고 있는 것이다.

바로 그래서 dependency 배열이 필요하다.

dependency 배열은 단순히 “이 effect를 언제 다시 돌릴까요?” 옵션이 아니다.

더 정확히는:

> 이 effect가 어떤 값에 의존하는지 선언하는 계약

이다.

여기서 dependency를 빼먹으면 무슨 일이 생기냐.

effect 내부 함수는 예전 렌더에서 만들어진 채 남고, 그 안에서는 예전 값만 본다.

예를 들어:

- 검색어가 바뀌었는데 예전 검색어로 요청을 계속 보냄
- interval이 최신 count를 못 봄
- 구독 해제 타이밍이 꼬임
- 이전 props 기준 로직이 계속 동작함

즉 dependency 누락은 “조금 덜 정확한 최적화”가 아니라, **오래된 렌더 값을 영구히 붙잡는 버그**가 될 수 있다.

많이 하는 오해도 있다.

> dependency 배열은 rerender 최적화용이다

이건 절반만 맞다. 핵심은 최적화가 아니라 **정확한 의존성 명시**다.

---

## 그런데 항상 dependency를 늘리는 게 답은 아닌 이유

여기서 한 단계 더 깊어져야 한다.

많은 초중급 학습자는 여기서 이렇게 생각한다.

> 오케이, 그러면 dependency를 다 넣으면 끝이네?

항상 그렇지는 않다.

왜냐하면 어떤 경우에는 effect를 재실행시키고 싶지 않은데, 그 안에서 **최신 값은 계속 읽고 싶을 수 있기 때문**이다.

예를 들어:

- 한 번만 등록한 이벤트 리스너
- 장시간 살아있는 interval
- 외부 라이브러리에 넘긴 callback
- 재등록 비용이 큰 구독

이런 경우에는 effect를 값 변화마다 다시 만들면 오히려 성능이나 구조가 더 나빠질 수 있다.

그래서 여기서 `ref`가 등장한다.

---

## ref는 왜 stale closure 회피에 자주 등장하나

`ref`는 React에서 아주 독특한 존재다.

- state처럼 값을 담을 수 있는데
- 바뀌어도 렌더를 일으키지 않고
- 같은 객체를 계속 유지한다

즉 `ref.current`는 “렌더를 트리거하지 않는 최신 값 보관함”처럼 쓸 수 있다.

이게 왜 stale closure와 연결되냐면, 오래 살아 있는 callback은 예전 렌더에서 생성되었더라도, 그 안에서 `ref.current`를 읽으면 **최신 값**을 볼 수 있기 때문이다.

여기서 중요한 건:

- state는 UI 반영을 위한 값
- ref는 렌더와 무관한 최신 참조 유지용 값

이라는 역할 차이다.

오해하면 안 되는 것도 있다.

`ref`는 stale closure의 만능 해결책이 아니다.

먼저 해야 할 건:

- dependency를 정확히 쓰기
- 함수형 업데이트 쓰기
- effect를 정말 필요한 곳에만 쓰기

그다음에도 “재렌더 없이 최신 값 참조”가 꼭 필요할 때 `ref`를 쓰는 게 맞다.

즉 `ref`는 편법이 아니라, **렌더링과 무관한 mutable reference가 필요할 때 쓰는 도구**다.

---

## 여기서 useMemo, useCallback으로 왜 이어지나

지금까지 본 걸 한 줄로 요약하면 이렇다.

- 함수는 클로저로 값을 기억한다
- React는 렌더마다 다른 값을 가진다
- callback / effect는 특정 렌더의 값을 캡처한다
- dependency를 잘못 다루면 stale closure가 생긴다
- 최신 값 참조가 필요하면 `ref`를 고려한다

그다음 자연스럽게 이어지는 질문이:

> 그렇다면 함수나 값의 참조를 안정적으로 유지하는 문제는 어떻게 보나?

이다.

여기서 `useMemo`, `useCallback`, `React.memo`가 들어온다.

많은 사람이 이 훅들을 “성능 최적화 훅”으로만 외운다. 그런데 실제 핵심은 **참조 안정성**이다.

- `useMemo`는 값의 참조를 안정화하고
- `useCallback`은 함수의 참조를 안정화하고
- `React.memo`는 props가 같으면 자식 렌더를 건너뛸 수 있게 해준다

그런데 이것도 결국 dependency를 제대로 이해해야만 쓸 수 있다.

왜냐하면 dependency를 틀리면:

- stale closure가 생기고
- 오래된 값이 memoized 되고
- callback이 최신 상태를 못 보고
- “최적화한 줄 알았는데 버그 만든 상태”가 되기 때문이다

즉 이 훅들도 사실은 **클로저와 렌더 시점 값 이해가 전제**다.

---

## 그럼 여기서 다음 개념은 어디로 이어지나

여기까지 오면 다음 축은 거의 자동이다.

이제 함수가 값을 어떻게 기억하는지는 봤다. 그러면 다음엔 React가 변경을 어떻게 감지하는지를 봐야 한다.

그래서 자연스럽게 이어진다.

### 다음 흐름

**원시값 / 참조값 → 얕은 복사 / 깊은 복사 → 불변성 → 참조 동일성 → React 리렌더링 → React.memo / useMemo / useCallback → 상태 구조 설계**

이 흐름이 중요한 이유는, 앞에서는 **함수와 렌더 시점 값**을 다뤘다면 이제는 **객체와 참조, 리렌더링 기준**을 다루게 되기 때문이다.

즉, 지금까지는 “함수가 무엇을 기억하느냐”였고, 다음부터는 “React가 무엇을 변경으로 판단하느냐”로 넘어가는 거다.

---

# 이 챕터를 읽은 뒤 자연스럽게 이어질 다음 질문들

1. 원시값과 참조값은 왜 React 리렌더링 이해의 출발점인가?
2. 얕은 복사와 깊은 복사는 왜 불변성과 연결되는가?
3. 부모가 리렌더되면 자식은 왜 기본적으로 다시 렌더되는가?
4. `React.memo`는 어떤 조건에서만 의미가 생기는가?
5. DnD나 차트 같은 복잡한 화면에서 상태 구조를 어떻게 설계해야 하는가?

# 사용법

이 문서는 아래 순서로 보는 게 좋다.

1. 각 섹션의 **왜 이 개념이 필요한가**를 먼저 읽는다.
2. 그다음 **자주 하는 오해**를 확인한다.
3. 마지막에 **면접 압축 답변**을 말로 꺼내 본다.
4. 막히면 다시 위 설명으로 돌아가 원리부터 복구한다.

이 방식이면 단순 답변 암기가 아니라, 내부 개념 구조까지 같이 자란다.
