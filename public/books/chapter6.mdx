# 챕터 6. Next.js 렌더링 전략에서 서버 경계 설계까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **Next.js는 React 위에 서버 렌더링, 라우팅, 데이터 페칭, 캐싱, 서버 경계를 얹어서 "언제 무엇을 어디서 렌더할 것인가"를 설계하게 만드는 프레임워크라는 관점으로 이해해야 한다.**

### 전체 흐름 맵

```
Next.js가 React 위에서 푸는 문제
  → App Router (파일 시스템 라우팅 + 중첩 레이아웃)
    → 렌더링 전략 (SSR / SSG / ISR / CSR / Streaming)
      → Server Component / Client Component (실행 환경 분리)
        → 'use client' (번들 경계 선언)
          → Server Actions (서버 mutation)
            → 데이터 페칭 위치 (서버 vs 클라이언트)
              → 캐싱과 재검증 (신선도 설계)
                → Route Handlers / BFF (서버 엔드포인트)
                  → URL 상태 설계 (탐색 가능한 상태)
                    → Middleware (요청 초반 분기)
```

## 66. Next.js는 React 위에서 어떤 문제를 더 풀어주나

> **핵심 요약** — Next.js는 React를 대체하는 게 아니라, 라우팅, 서버 렌더링, 데이터 페칭, 캐싱, 서버 경계까지 포함한 애플리케이션 구조로 확장한다.

### React만으로는 답하기 어려운 질문들

- 첫 화면을 더 빨리 보여주려면?
- SEO가 필요한 페이지는?
- 서버에서 비밀값을 써야 하는 로직은?
- 어떤 페이지는 미리 만들고, 어떤 페이지는 요청마다 만들까?
- 클라이언트 번들을 어떻게 줄일까?

Next.js는 이런 질문에 대한 선택지를 제공한다.

### App Router — 현재 공식 중심 구조

```
app/
├── layout.tsx          ← 루트 레이아웃
├── page.tsx            ← / 경로
├── loading.tsx         ← Suspense 기반 로딩 상태
├── error.tsx           ← Error Boundary 기반 에러 처리
├── dashboard/
│   ├── layout.tsx      ← 대시보드 중첩 레이아웃
│   ├── page.tsx        ← /dashboard 경로
│   └── settings/
│       └── page.tsx    ← /dashboard/settings 경로
└── api/
    └── users/
        └── route.ts    ← /api/users 엔드포인트
```

| 파일 | 역할 |
|---|---|
| `layout.tsx` | 중첩 레이아웃 — 하위 세그먼트와 공유 |
| `page.tsx` | 해당 경로의 UI |
| `loading.tsx` | Suspense fallback — 로딩 상태 |
| `error.tsx` | Error Boundary — 에러 격리 |
| `route.ts` | Route Handler — 서버 엔드포인트 |

App Router의 핵심은 단순 페이지 라우팅이 아니라 **중첩 레이아웃 + 서버/클라이언트 경계 + Suspense 통합**이다.

### 실무 에피소드 — 이런 상황에서 만난다

> CRA(Create React App)로 만든 SPA를 운영하다가 SEO가 필요한 마케팅 페이지를 추가해야 했는데, CSR만으로는 검색엔진 노출이 안 됐다. Next.js로 마이그레이션하면서 마케팅 페이지는 SSG, 대시보드는 CSR, 상품 상세는 ISR로 페이지별 렌더링 전략을 다르게 가져갈 수 있게 됨.

### 자주 하는 오해

- Next.js는 React보다 상위라서 React를 몰라도 된다 → **틀림.** React 위에 얹어진 것이다.
- Next.js는 SSR 프레임워크다 → **너무 좁은 설명.** 서버 경계, 캐싱, 라우팅까지 포함.
- Next.js를 쓰면 성능이 자동으로 좋아진다 → **틀림.** 설계에 따라 다르다.

### 면접 압축 답변

- **10초**: Next.js는 React 위에 라우팅, 서버 렌더링, 데이터 페칭, 캐싱을 얹어 "언제 무엇을 어디서 렌더할지" 설계하게 만드는 프레임워크입니다.
- **30초**: App Router는 파일 시스템 라우팅에 중첩 레이아웃, Server/Client Components, Suspense 기반 로딩/에러 처리를 통합합니다. 단순 페이지 라우팅을 넘어 서버 경계와 렌더링 전략을 함께 설계하게 됩니다.

## 67. SSR, SSG, ISR, CSR, Streaming은 각각 어떤 문제를 푸는가

> **핵심 요약** — 정의 암기가 아니라 "데이터 신선도, SEO, 초기 로드, 인터랙션 요구"에 따라 어떤 조합이 적절한지 판단하는 게 핵심. 한 앱 안에서도 페이지별로 전략이 다르다.

### 렌더링 전략 비교

| 전략 | 렌더 시점 | 데이터 신선도 | TTFB | SEO | 대표 용례 |
|---|---|---|---|---|---|
| **SSG** | 빌드 시 | 빌드 시점 고정 | 매우 빠름 (CDN) | 좋음 | 마케팅, 문서, 블로그 |
| **ISR** | 빌드 시 + 주기적 재생성 | 설정 주기만큼 | 빠름 (캐시) | 좋음 | 상품 상세, 뉴스 |
| **SSR** | 요청 시 | 항상 최신 | 서버 응답 시간 | 좋음 | 대시보드, 검색 결과 |
| **CSR** | 브라우저에서 | 클라이언트 요청 시 | HTML 빠름, 콘텐츠 느림 | 약함 | 로그인 후 개인화 UI |
| **Streaming** | 요청 시 (점진적) | 부분별 최신 | 점진적 | 좋음 | 복합 페이지 (일부 빠르게, 일부 나중에) |

### Streaming SSR — App Router의 강력한 기능

기존 SSR은 모든 데이터가 준비될 때까지 HTML 전체를 보류했다. Streaming은 **준비된 부분부터 점진적으로 전송**한다.

```tsx
// app/dashboard/page.tsx
import { Suspense } from "react";

export default function Dashboard() {
  return (
    <div>
      <h1>대시보드</h1>
      {/* 헤더는 즉시 전송 */}

      <Suspense fallback={<p>매출 데이터 로딩...</p>}>
        <SalesChart />  {/* 느린 API — 나중에 스트리밍 */}
      </Suspense>

      <Suspense fallback={<p>알림 로딩...</p>}>
        <NotificationList />  {/* 또 다른 느린 API — 독립적으로 스트리밍 */}
      </Suspense>
    </div>
  );
}
```

```
기존 SSR:  [전체 데이터 준비] ──────────────── [HTML 전송]
                                                 ↑ 사용자가 여기서야 뭔가 봄

Streaming: [헤더 준비] → [헤더 전송] → [차트 준비] → [차트 스트리밍] → [알림 스트리밍]
            ↑ 사용자가 여기서 이미 헤더를 봄
```

### 선택 가이드

```
이 페이지의 데이터는...
│
├── 빌드 시 확정? → SSG (마케팅, 문서)
│     └── 가끔 바뀜? → ISR (revalidate 설정)
│
├── 요청마다 최신이어야 함? → SSR
│     └── 일부는 빠르고 일부는 느림? → Streaming + Suspense
│
└── 서버 렌더링 불필요, 인터랙션 중심? → CSR (use client)
```

### 실무 에피소드 — 이런 상황에서 만난다

> 이커머스에서 상품 상세 페이지를 SSR로 했더니 매 요청마다 서버 응답이 500ms 걸려서 체감 느림. ISR로 바꾸고 `revalidate: 300`(5분)으로 설정하니 대부분 요청이 캐시에서 즉시 응답. 재고 같은 실시간 데이터는 클라이언트에서 별도 fetch.

### 자주 하는 오해

- SSR이 항상 더 빠르다 → **서버 응답이 느리면 SSG보다 훨씬 느릴 수 있다.**
- 한 앱은 한 방식만 써야 한다 → **페이지별로 다른 전략을 적용하는 게 일반적이다.**
- CSR은 SEO가 안 된다 → **검색 엔진에 따라 다르지만, 안정적 SEO를 원하면 서버 렌더링이 유리.**

### 면접 압축 답변

- **10초**: SSG(빌드 시), ISR(빌드 + 주기적 재생성), SSR(요청 시), CSR(브라우저), Streaming(점진적 전송)은 각각 렌더링 시점과 캐시 전략이 다릅니다.
- **30초**: 데이터 신선도, SEO, 초기 로드 속도, 인터랙션 요구에 따라 전략을 다르게 가져갑니다. Streaming은 Suspense와 결합해서 준비된 부분부터 점진적으로 보여줘서, 전체 데이터가 준비될 때까지 기다리는 기존 SSR의 한계를 보완합니다.

## 68. Server Component와 Client Component

> **핵심 요약** — Server Component는 서버에서 실행되어 번들에 포함 안 됨. Client Component는 상태/이벤트/브라우저 API가 필요할 때. Client Component도 서버에서 프리렌더된 뒤 hydration됨.

### 비교 테이블

| | Server Component (기본값) | Client Component (`'use client'`) |
|---|---|---|
| **실행 위치** | 서버 | 서버(프리렌더) + 클라이언트(hydration) |
| **번들 포함** | 안 됨 | 포함됨 |
| **async/await** | 직접 사용 가능 | 불가 (hooks로 대체) |
| **useState/useEffect** | 사용 불가 | 사용 가능 |
| **이벤트 핸들러** | 사용 불가 | 사용 가능 |
| **브라우저 API** | 접근 불가 | 접근 가능 |
| **DB/ORM 직접 접근** | 가능 | 불가 |
| **비밀값(API key 등)** | 안전 | 노출 위험 |

### 핵심 설계 질문

> **이 로직은 서버에 두는 게 자연스러운가, 클라이언트에 두는 게 자연스러운가?**

```tsx
// Server Component — 데이터 조회 + 정적 UI
// app/posts/[id]/page.tsx
export default async function PostPage({ params }) {
  const post = await getPost(params.id); // 서버에서 직접 DB 접근

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
      <CommentSection postId={params.id} /> {/* 인터랙션만 Client */}
    </article>
  );
}

// Client Component — 상태 + 이벤트
// app/posts/[id]/comment-section.tsx
"use client";
import { useState } from "react";

export default function CommentSection({ postId }) {
  const [text, setText] = useState("");
  return (
    <div>
      <textarea value={text} onChange={(e) => setText(e.target.value)} />
      <button onClick={() => submitComment(postId, text)}>작성</button>
    </div>
  );
}
```

### Serialization 제약 — SC에서 CC로 넘길 수 있는 것

Server Component에서 Client Component로 props를 넘길 때, **직렬화 가능한 값만** 전달할 수 있다.

| 전달 가능 | 전달 불가 |
|---|---|
| string, number, boolean, null | 함수 (onClick 등) |
| 배열, 일반 객체 | 클래스 인스턴스 |
| Date (문자열로 변환) | Map, Set |
| JSX (children으로) | Symbol |

```tsx
// OK: 직렬화 가능한 값
<ClientChild data={post} count={42} />

// NG: 함수는 전달 불가
<ClientChild onClick={() => console.log("hi")} />
// → 함수는 Client Component 안에서 정의해야 함
```

### 'use client'는 번들 경계

`'use client'`는 단순 "이 파일은 클라이언트"가 아니라 **모듈 그래프 상의 번들 경계**다.

```
Server Component 트리
    │
    ├── layout.tsx (SC)
    ├── page.tsx (SC)
    │     └── getPost() — 서버에서만 실행, 번들에 안 들어감
    │
    └── 'use client' ← 여기가 경계
          │
          ├── comment-section.tsx (CC) — 번들에 포함
          └── 이 파일이 import하는 모든 모듈도 번들에 포함!
```

**실무 원칙**: `'use client'`는 가능한 작은 leaf 컴포넌트에만 붙이고, 상위 layout/page는 서버에 남긴다.

### 실무 에피소드 — 이런 상황에서 만난다

> 페이지 전체에 `'use client'`를 붙여서 서버 전용 DB 접근이 안 되고, 모든 코드가 클라이언트 번들에 포함되면서 번들 크기가 2배가 된 사례. `'use client'`를 좋아요 버튼, 댓글 입력 같은 인터랙션 leaf에만 붙이고, 페이지와 레이아웃은 서버에 남기니 번들이 절반으로 줄어듦.

### 자주 하는 오해

- Client Component는 전부 CSR이다 → **틀림.** 서버에서 HTML로 프리렌더된 뒤 hydration.
- Server Component 안에 Client Component를 children으로 넘기면 서버가 된다 → **틀림.** `'use client'` 선언으로 결정.
- `'use client'`를 붙이면 그 파일만 영향 → **하위 import 전체가 클라이언트 번들에 포함.**

### 면접 압축 답변

- **10초**: Server Component는 서버에서 실행되어 번들에 포함 안 되고, Client Component는 상태/이벤트가 필요할 때 씁니다. `'use client'`는 모듈 그래프의 번들 경계입니다.
- **30초**: Server Component에서는 async/await로 직접 데이터를 가져오고 비밀값을 안전하게 쓸 수 있습니다. Client Component도 서버에서 프리렌더된 뒤 hydration됩니다. SC에서 CC로는 직렬화 가능한 값만 전달 가능하고, `'use client'`는 가능한 leaf 쪽에만 붙여야 번들 비용을 줄일 수 있습니다.

## 69. Server Actions — 서버 mutation의 새로운 방식

> **핵심 요약** — Server Actions는 클라이언트에서 호출하면 서버에서 실행되는 함수다. Route Handler 없이 폼 제출, 데이터 mutation을 처리할 수 있고, `revalidatePath`/`revalidateTag`로 캐시 무효화까지 한 흐름으로 가능하다.

### 핵심 개념

```tsx
// app/actions.ts
"use server";

import { revalidatePath } from "next/cache";

export async function createPost(formData: FormData) {
  const title = formData.get("title") as string;
  const body = formData.get("body") as string;

  await db.posts.create({ title, body });

  revalidatePath("/posts"); // 캐시 무효화 → 목록 갱신
}
```

```tsx
// app/posts/new/page.tsx (Server Component에서 직접 사용)
import { createPost } from "../actions";

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" placeholder="제목" />
      <textarea name="body" placeholder="본문" />
      <button type="submit">작성</button>
    </form>
  );
}
```

```tsx
// Client Component에서도 사용 가능
"use client";

import { createPost } from "../actions";
import { useTransition } from "react";

export default function NewPostForm() {
  const [isPending, startTransition] = useTransition();

  const handleSubmit = (formData: FormData) => {
    startTransition(async () => {
      await createPost(formData);
    });
  };

  return (
    <form action={handleSubmit}>
      <input name="title" />
      <button type="submit" disabled={isPending}>
        {isPending ? "저장 중..." : "작성"}
      </button>
    </form>
  );
}
```

### Server Actions vs Route Handlers

| | Server Actions | Route Handlers |
|---|---|---|
| **선언** | `'use server'` 함수 | `route.ts` 파일 |
| **호출 방식** | 함수 호출 / form action | HTTP 요청 (fetch) |
| **용도** | 폼 제출, 데이터 mutation | 외부 API, webhook, BFF |
| **타입 안전성** | 함수 시그니처로 자연스러움 | 요청/응답 수동 타이핑 |
| **캐시 무효화** | `revalidatePath/Tag` 직접 호출 | 동일 |

### 실무 에피소드 — 이런 상황에서 만난다

> 게시글 작성을 Route Handler POST로 구현했는데, 작성 후 목록 갱신을 위해 클라이언트에서 다시 fetch → invalidate 하는 코드가 번거로웠다. Server Action으로 바꾸고 action 안에서 `revalidatePath("/posts")`를 호출하니, 작성 → 캐시 무효화 → 목록 갱신이 한 흐름으로 깔끔해짐.

### 자주 하는 오해

- Server Actions는 Route Handler를 완전히 대체한다 → **용도가 다르다.** 외부 webhook이나 BFF는 Route Handler가 더 적합.
- `'use server'`는 `'use client'`의 반대다 → **아니다.** `'use server'`는 함수를 서버 실행으로 표시하는 것이고, `'use client'`는 컴포넌트를 클라이언트 번들에 포함시키는 경계다.

### 면접 압축 답변

- **10초**: Server Actions는 `'use server'`로 표시된 함수로, 클라이언트에서 호출하면 서버에서 실행됩니다. 폼 제출과 mutation에 Route Handler 없이 직접 사용할 수 있습니다.
- **30초**: Server Actions는 form action이나 함수 호출로 서버 mutation을 처리하고, `revalidatePath`/`revalidateTag`로 캐시 무효화까지 한 흐름으로 가능합니다. Route Handler는 외부 API나 webhook 같은 HTTP 엔드포인트에 여전히 필요합니다.

## 70. 데이터 페칭과 캐싱 설계

> **핵심 요약** — Server Component에서 직접 fetch하는 것이 기본. 캐싱은 성능 옵션이 아니라 데이터 신선도 설계다. Next.js 15부터 fetch 기본값이 `no-store`이므로 캐시가 필요하면 명시적으로 설정해야 한다.

### 데이터 페칭 위치

| 데이터 성격 | 페칭 위치 | 이유 |
|---|---|---|
| 공개 콘텐츠 (게시글, 상품) | Server Component | 서버에서 직접 fetch, SEO, 번들 절감 |
| 비밀값 필요 (API key, DB) | Server Component / Route Handler | 클라이언트에 노출 안 됨 |
| 사용자 인터랙션 기반 (검색, 필터) | Client Component (SWR/React Query) | 실시간 반응 필요 |
| 초기 데이터 + 후속 갱신 | 서버에서 초기 로드 + 클라이언트에서 갱신 | 하이브리드 패턴 |

```tsx
// Server Component — async/await로 직접 fetch
export default async function UsersPage() {
  const users = await fetch("https://api.example.com/users").then((r) => r.json());
  return <ul>{users.map((u) => <li key={u.id}>{u.name}</li>)}</ul>;
}

// Client Component — 인터랙션 기반 fetch
"use client";
import useSWR from "swr";

export default function SearchResults({ query }) {
  const { data, isLoading } = useSWR(query ? `/api/search?q=${query}` : null, fetcher);
  if (isLoading) return <p>검색 중...</p>;
  return <ul>{data?.map((item) => <li key={item.id}>{item.title}</li>)}</ul>;
}
```

### 캐싱 — 신선도와 성능의 트레이드오프

Next.js 15부터 `fetch`의 기본값이 `no-store`로 변경됨. 캐시가 필요하면 명시적으로 설정.

```tsx
// 시간 기반 재검증
const data = await fetch("https://api.example.com/posts", {
  next: { revalidate: 60 }, // 60초마다 재생성
});

// 캐시 안 함 (기본값)
const freshData = await fetch("https://api.example.com/dashboard", {
  cache: "no-store",
});

// 태그 기반 재검증
const taggedData = await fetch("https://api.example.com/products", {
  next: { tags: ["products"] },
});
```

```tsx
// Server Action에서 수동 재검증
"use server";
import { revalidateTag, revalidatePath } from "next/cache";

export async function updateProduct(id: string, data: FormData) {
  await db.products.update(id, data);
  revalidateTag("products");    // 태그로 무효화
  revalidatePath("/products");  // 경로로 무효화
}
```

### 캐시 판단 기준

| 데이터 | 캐시 전략 | 이유 |
|---|---|---|
| 마케팅 페이지, 문서 | `revalidate: 3600` (1시간) | 거의 안 바뀜 |
| 블로그, 상품 상세 | `revalidate: 300` (5분) | 가끔 바뀜 |
| 대시보드, 검색 결과 | `cache: 'no-store'` | 항상 최신 필요 |
| mutation 후 | `revalidateTag/Path` | 즉시 반영 필요 |

### 실무 에피소드 — 이런 상황에서 만난다

> 상품을 수정했는데 상품 목록에 반영이 안 되는 버그. 원인은 목록 페이지의 fetch에 `revalidate: 3600`이 걸려있어서 1시간 동안 이전 캐시를 보여줌. Server Action에서 수정 후 `revalidateTag("products")`를 호출하도록 바꿔서 mutation 후 즉시 반영.

### 자주 하는 오해

- 캐시는 무조건 많이 걸수록 좋다 → **stale 이슈가 생길 수 있다.**
- Next.js는 기본적으로 모든 fetch를 캐시한다 → **Next.js 15부터 기본값이 `no-store`.**
- 서버에서만 fetch해야 한다 → **인터랙션 기반 데이터는 클라이언트 fetch가 자연스럽다.**

### 면접 압축 답변

- **10초**: Server Component에서 async/await로 직접 fetch하는 것이 기본이고, 캐싱은 데이터 신선도에 따라 시간 기반 또는 태그 기반으로 설정합니다.
- **30초**: Next.js 15부터 fetch 기본값이 no-store이므로 캐시가 필요하면 명시적으로 revalidate를 설정합니다. mutation 후에는 revalidateTag/Path로 관련 캐시를 무효화합니다. 인터랙션 기반 데이터는 Client Component에서 SWR/React Query로 처리하는 하이브리드 패턴이 일반적입니다.

## 71. Route Handlers와 BFF

> **핵심 요약** — Route Handlers는 Next.js 앱 안에서 서버 엔드포인트를 만드는 구조. BFF(Backend for Frontend) 패턴으로 여러 백엔드 API를 합치고, 비밀값을 숨기고, 프론트 친화적 응답을 만들 수 있다.

### Route Handler 기본

```tsx
// app/api/users/route.ts
import { NextResponse } from "next/server";

export async function GET() {
  const users = await db.users.findMany();
  return NextResponse.json(users);
}

export async function POST(request: Request) {
  const body = await request.json();
  const user = await db.users.create(body);
  return NextResponse.json(user, { status: 201 });
}
```

### BFF 패턴 — 여러 API를 합쳐 프론트 친화적 응답

```tsx
// app/api/product-detail/[id]/route.ts
export async function GET(request: Request, { params }) {
  const [product, reviews, inventory] = await Promise.all([
    fetch(`${PRODUCT_API}/products/${params.id}`).then((r) => r.json()),
    fetch(`${REVIEW_API}/reviews?productId=${params.id}`).then((r) => r.json()),
    fetch(`${INVENTORY_API}/stock/${params.id}`).then((r) => r.json()),
  ]);

  return NextResponse.json({
    id: product.id,
    name: product.name,
    price: product.price,
    reviewCount: reviews.length,
    averageRating: reviews.reduce((sum, r) => sum + r.rating, 0) / reviews.length,
    inStock: inventory.quantity > 0,
  });
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 상품 상세 페이지에서 상품 정보, 리뷰, 재고를 3개 API에서 가져오는데 클라이언트에서 3번 fetch하니 로딩이 길었다. Route Handler에서 `Promise.all`로 3개를 병렬 요청하고 프론트 친화적 shape로 합쳐서 내려주니 클라이언트는 1번만 호출. 로딩 시간이 3초에서 1초로 줄어듦.

### Route Handler vs Server Action vs Server Component

| | Server Component | Server Action | Route Handler |
|---|---|---|---|
| **용도** | 데이터 조회 + UI | 폼 제출, mutation | HTTP 엔드포인트, webhook |
| **호출** | 렌더링 시 자동 | 함수 호출 / form action | fetch로 HTTP 요청 |
| **외부 접근** | 불가 | 불가 | 가능 (webhook 등) |

Server Component에서 직접 DB 접근이 가능하면 굳이 Route Handler를 거칠 필요가 없다. Route Handler는 **외부에서 접근해야 하는 엔드포인트**(webhook, 외부 서비스 연동)나 **BFF 역할**에 적합.

### 면접 압축 답변

- **10초**: Route Handlers는 앱 안에서 서버 엔드포인트를 만들고, BFF로 여러 API를 합쳐 프론트 친화적 응답을 제공합니다.
- **30초**: Server Component에서 직접 데이터를 가져올 수 있으면 Route Handler는 불필요합니다. Route Handler는 외부 webhook, BFF, 비밀값 사용 같은 HTTP 엔드포인트가 필요한 경우에 적합하고, mutation은 Server Actions가 더 자연스럽습니다.

## 72. URL 상태와 Middleware

> **핵심 요약** — 공유/복원/뒤로가기가 필요한 상태는 URL에 둔다. Middleware는 요청 초반에 인증 리다이렉트, locale 분기 같은 공통 정책을 적용하는 Edge 계층이다.

### URL 상태 — 언제 URL에 두나

> **"이 URL을 다른 사람에게 공유했을 때 의미가 있는가?"**가 기준이다.

| URL에 적합 | URL에 부적합 |
|---|---|
| 검색어 `?q=nextjs` | 모달 열림/닫힘 (대부분) |
| 필터 `?category=frontend` | 드래그 좌표 |
| 정렬 `?sort=latest` | 호버 상태 |
| 페이지네이션 `?page=3` | 임시 입력값 |
| 선택된 탭 `?tab=reviews` | 토스트 알림 |

```tsx
// Server Component에서 URL 상태로 데이터 페칭
export default async function ProductsPage({ searchParams }) {
  const sort = searchParams.sort || "latest";
  const page = searchParams.page || "1";

  const products = await fetch(
    `https://api.example.com/products?sort=${sort}&page=${page}`
  ).then((r) => r.json());

  return (
    <div>
      <FilterBar currentSort={sort} />
      <ProductList products={products} />
      <Pagination currentPage={Number(page)} />
    </div>
  );
}
```

```tsx
// Client Component에서 URL 상태 변경
"use client";
import { useRouter, useSearchParams } from "next/navigation";

export default function FilterBar({ currentSort }) {
  const router = useRouter();
  const searchParams = useSearchParams();

  function updateFilter(key, value) {
    const params = new URLSearchParams(searchParams.toString());
    params.set(key, value);
    router.push(`/products?${params.toString()}`);
  }

  return (
    <select value={currentSort} onChange={(e) => updateFilter("sort", e.target.value)}>
      <option value="latest">최신순</option>
      <option value="price">가격순</option>
    </select>
  );
}
```

### Middleware — 요청 초반 공통 분기

```tsx
// middleware.ts (프로젝트 루트)
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const token = request.cookies.get("auth-token");

  if (!token && request.nextUrl.pathname.startsWith("/dashboard")) {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/dashboard/:path*"],
};
```

| Middleware에 적합 | Middleware에 부적합 |
|---|---|
| 인증 토큰 존재 확인 → 리다이렉트 | 실제 권한 검증 (DB 조회) |
| locale/국가 분기 | 무거운 비즈니스 로직 |
| A/B 테스트 rewrite | DB 접근 (Edge Runtime 제약) |
| 봇 트래픽 감지 | 복잡한 데이터 처리 |

### 실무 에피소드 — 이런 상황에서 만난다

> 검색 결과 페이지에서 필터를 useState로 관리했는데, 결과를 공유하면 상대방에게 필터 없는 기본 화면이 보이는 문제. 필터값을 URL search params로 옮기니 공유 링크에 필터가 포함되어 같은 화면이 복원됨. 서버에서도 searchParams로 접근해서 초기 데이터를 미리 준비할 수 있게 됨.

### 면접 압축 답변

- **10초**: 공유/복원/뒤로가기가 필요한 상태는 URL에, 요청 초반 공통 정책(인증 리다이렉트, locale)은 Middleware에 둡니다.
- **30초**: URL 상태는 Server Component의 searchParams로 서버 데이터 페칭의 입력값이 될 수 있어서, 공유 링크 복원뿐 아니라 SEO와 초기 데이터 준비에도 유리합니다. Middleware는 Edge Runtime에서 실행되므로 가벼운 확인(토큰 존재, locale 분기)에 적합하고, DB 접근 같은 무거운 작업은 Server Component나 Route Handler에서 처리합니다.
