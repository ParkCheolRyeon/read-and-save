# 챕터 6. Next.js 렌더링 전략에서 서버 경계 설계까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **Next.js는 React 위에 서버 렌더링, 라우팅, 데이터 페칭, 캐싱, 서버 경계를 얹어서 “언제 무엇을 어디서 렌더할 것인가”를 설계하게 만드는 프레임워크라는 관점으로 이해해야 한다.**

---

## 66. Next.js는 React 위에서 어떤 문제를 더 풀어주나

앞 챕터에서 React의 선언형 렌더링 모델과 상태 구조를 봤다면,  
이제 자연스럽게 이런 질문이 나온다.

> **그런데 React만으로 앱을 만들면 서버 렌더링, 라우팅, 데이터 페칭, 배포 친화적 구조는 어떻게 다뤄야 할까?**

여기서 Next.js가 등장한다.

### 핵심 개념

Next.js는 React 자체를 대체하는 게 아니라,  
React 앱을 **라우팅, 렌더링 전략, 서버 실행, 데이터 페칭, 캐싱**까지 포함한 애플리케이션 구조로 확장해준다.

공식 App Router 문맥에서 특히 중요한 축은 아래와 같다.

- 파일 시스템 기반 라우팅
- 레이아웃과 중첩 UI 구조
- Server Components / Client Components
- 서버 중심 데이터 페칭
- 캐싱과 재검증
- Route Handlers 같은 서버 엔드포인트 구조

### 왜 이게 중요한가

React만으로도 화면은 만들 수 있다.  
하지만 실제 서비스에선 아래 질문이 계속 생긴다.

- 첫 화면을 더 빨리 보여주려면?
- SEO가 필요한 페이지는?
- 서버에서 비밀값을 써야 하는 로직은?
- 데이터를 어디서 가져오는 게 자연스러운가?
- 어떤 페이지는 미리 만들고, 어떤 페이지는 요청마다 만들까?
- 클라이언트 번들을 어떻게 줄일까?

Next.js는 이런 질문에 대한 선택지를 제공한다.

### 자주 하는 오해

- Next.js는 React보다 상위라서 React를 몰라도 된다 → 틀림
- Next.js를 쓰면 성능 문제가 자동으로 해결된다 → 틀림
- Next.js는 SSR 프레임워크다 → 너무 좁은 설명이다
- Next.js는 프론트 전용이다 → 서버 경계와 서버 실행 문맥도 포함된다

### 실무 연결

프론트엔드 면접에서 Next.js를 잘 설명한다는 건  
“라우팅 라이브러리 이름을 안다”가 아니라,  
**어떤 UI를 서버에 두고 어떤 인터랙션을 클라이언트로 내릴지 판단할 수 있다**는 뜻에 가깝다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 Next.js가 React 위에서 푸는 문제를 봤다.  
그다음엔 현재 공식 문서 기준의 중심 구조인 **App Router**로 넘어가야 한다.

### 면접 압축 답변

Next.js는 React 위에 라우팅, 서버 렌더링, 데이터 페칭, 캐싱, 서버 경계 같은 애플리케이션 구조를 얹어주는 프레임워크입니다. 그래서 단순 화면 구성보다 “언제 무엇을 어디서 렌더하고, 어디서 데이터를 가져올지”를 설계하게 만든다는 점이 핵심입니다.

## 67. App Router는 왜 Next.js의 현재 중심 구조인가

현재 공식 문서 기준으로 Next.js는 App Router를 중심으로 학습 흐름을 잡고 있다.  
이 구조는 React의 최신 기능들을 더 직접적으로 활용할 수 있게 설계되어 있다. :contentReference[oaicite:1]{index=1}

### 핵심 개념

App Router는 파일 시스템 기반 라우터이면서,  
아래 기능들과 강하게 연결된다.

- layout
- page
- loading
- error
- route
- Server Components
- Client Components
- Suspense 기반 분할 렌더링

### 왜 이게 중요한가

이전에는 페이지 단위 중심 사고가 강했다면,  
App Router에선 **중첩 레이아웃과 서버/클라이언트 경계**가 훨씬 중요해진다.

즉 단순히 “URL이 무엇을 가리키는가”를 넘어서,

- 이 세그먼트는 어떤 layout을 공유하는가
- 어느 부분은 서버에서 준비되는가
- 어느 부분은 클라이언트 인터랙션이 필요한가
- 어떤 경계에서 로딩과 에러를 나눌 것인가

까지 같이 보게 된다.

### 자주 하는 오해

- App Router는 Pages Router를 완전히 모르면 이해할 수 없다 → 꼭 그렇진 않다
- App Router는 폴더 이름만 바뀐 구조다 → 틀림
- App Router에선 서버와 클라이언트를 별로 고민 안 해도 된다 → 오히려 더 중요하다

### 실무 연결

App Router 감각이 있으면  
“이 페이지는 SSR인가?” 수준을 넘어서

- layout 분리
- server/client boundary
- segment 기반 로딩
- 데이터 페칭 위치
- 캐싱 전략

까지 하나의 구조로 볼 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 App Router 구조를 알았다.  
그 다음엔 가장 자주 나오는 렌더링 전략 질문, 즉 **SSR / SSG / ISR / CSR**로 넘어가야 한다.

### 면접 압축 답변

App Router는 현재 공식 문서 기준의 중심 구조로, 파일 시스템 라우팅에 레이아웃, Server/Client Components, Suspense 기반 분할 렌더링 같은 개념을 강하게 결합합니다. 그래서 단순 페이지 라우팅보다 서버 경계와 중첩 UI 구조를 함께 설계하게 만든다는 점이 중요합니다.

---

## 68. SSR, SSG, ISR, CSR은 각각 어떤 문제를 푸는가

Next.js 면접에서 가장 자주 나오는 주제 중 하나다.  
그런데 이걸 “정의 암기”로만 외우면 실제 설계 판단으로 이어지지 않는다.

### 핵심 개념

#### SSR (Server-Side Rendering)

요청 시점에 서버에서 HTML을 생성해 응답하는 방식

#### SSG (Static Site Generation)

빌드 시점에 HTML을 미리 생성해 두는 방식

#### ISR (Incremental Static Regeneration)

정적 생성 기반이지만, 일정 조건이나 시간 후 재생성/재검증이 가능한 방식

#### CSR (Client-Side Rendering)

초기 HTML보다 클라이언트 자바스크립트가 중심이 되어 화면을 구성하는 방식

### 왜 이게 중요한가

이 질문의 핵심은 정의가 아니라 **선택 기준**이다.

예를 들어:

- 항상 최신 데이터가 중요하면 SSR 쪽 감각이 필요할 수 있고
- 자주 안 바뀌는 공개 콘텐츠는 SSG/캐시 기반이 유리할 수 있고
- 정적 기반이지만 주기적 갱신이 필요하면 ISR/재검증 감각이 필요할 수 있고
- 로그인 이후 개인화 UI는 CSR/클라이언트 상호작용 비중이 커질 수 있다

### 자주 하는 오해

- SSR이 항상 더 빠르다 → 상황에 따라 다르다
- CSR은 SEO가 안 된다 → 너무 단순화된 설명이다
- ISR은 그냥 SSG의 옵션이다 → 캐시와 재검증 설계 감각이 함께 필요하다
- 한 앱은 한 방식만 써야 한다 → 보통은 섞인다

### 실무 연결

좋은 답변은 이런 식이다.

- 공개 마케팅 페이지 → 정적/캐시 친화적
- 자주 바뀌는 대시보드 데이터 → 런타임 데이터 중심
- 블로그/상품 상세 → 정적 + 재검증 전략 가능
- 로그인 이후 인터랙션 영역 → 클라이언트 컴포넌트와 클라이언트 상태 활용

### 여기서 다음 개념으로 어떻게 넘어가나

이제 렌더링 전략 이름은 봤다.  
그 다음엔 App Router 기준에서 더 본질적인 질문이 나온다.

**그럼 이 렌더링 전략은 결국 Server Component와 Client Component 구분과 어떻게 연결되는가?**

### 면접 압축 답변

SSR, SSG, ISR, CSR은 각각 렌더링 시점과 캐시 전략이 다른 방식입니다. 중요한 건 정의 암기보다, 데이터 신선도, SEO, 초기 로드 속도, 인터랙션 요구사항에 따라 어떤 조합이 더 적절한지 판단하는 것입니다.

## 69. Server Component와 Client Component는 왜 나뉘는가

App Router에서 가장 중요한 개념 축 중 하나다.  
공식 문서 기준으로 `page`와 `layout`은 기본적으로 Server Component이며,  
상태, 이벤트 핸들러, 브라우저 API가 필요할 때 Client Component를 사용한다. :contentReference[oaicite:2]{index=2}

### 핵심 개념

#### Server Component가 잘 맞는 일

- 데이터 소스 가까이서 fetch
- 비밀값, 토큰, 서버 전용 자원 사용
- 클라이언트 자바스크립트 번들 줄이기
- 초기 표시를 더 빠르게 준비하기

#### Client Component가 잘 맞는 일

- state
- event handler
- `useEffect`
- 브라우저 API 사용
- 사용자 인터랙션

### 왜 이게 중요한가

이 구분이 없으면  
모든 걸 클라이언트로 보내서 번들이 커지거나,  
반대로 인터랙션이 필요한데 서버 쪽에 두려는 이상한 구조가 된다.

즉 Next.js 설계 핵심은 보통 이 질문이다.

> **이 로직은 서버에 두는 게 자연스러운가, 클라이언트에 두는 게 자연스러운가?**

### 자주 하는 오해

- Server Component는 서버에서만 보이고 클라이언트엔 아예 전달되지 않는다 → HTML과 결과는 사용자에게 전달된다
- Client Component는 전부 CSR이다 → App Router 안에서 Server Component와 함께 조합될 수 있다
- Client Component를 쓰면 무조건 나쁘다 → 인터랙션엔 필요하다
- Server Component가 항상 정답이다 → 브라우저 이벤트와 상태는 클라이언트가 필요하다

### 실무 연결

좋은 설계는 보통:

- 데이터 조회와 정적 UI는 서버 쪽
- 버튼, 입력, 모달, 드롭다운 같은 인터랙션은 클라이언트 쪽
  으로 나눈다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 서버/클라이언트 구분을 알았다.  
그다음엔 이를 실제 파일에서 선언하는 대표 경계인 **`'use client'`** 로 넘어가야 한다.

### 면접 압축 답변

App Router에서 Server Component와 Client Component를 나누는 이유는, 데이터 페칭과 비밀값 처리, 번들 절감은 서버에 두고, state와 이벤트 핸들러, 브라우저 API가 필요한 부분만 클라이언트로 보내기 위해서입니다. 핵심은 기술 선택이 아니라 실행 환경 분리입니다.

## 70. 'use client'는 정확히 무엇을 의미하나

많은 사람이 `'use client'`를 “이 파일은 클라이언트에서 돈다” 정도로만 이해한다.  
그런데 공식 문서 기준으로 더 정확한 표현은 **클라이언트 진입 경계(entry point)** 를 선언하는 것이다. :contentReference[oaicite:3]{index=3}

### 핵심 개념

`'use client'`는 파일 최상단에서  
해당 컴포넌트를 **Server Components 트리 안에서 클라이언트 경계로 끌어올리는 선언**이다.

그리고 중요한 점은:

- 그 파일만 클라이언트가 되는 게 아니라
- 그 아래로 연결되는 클라이언트 모듈 그래프가 함께 영향을 받는다는 점이다

### 왜 이게 중요한가

이걸 모르고 `'use client'`를 상위 큰 파일에 붙이면,
원래 서버에 둘 수 있었던 것들까지 클라이언트 번들에 포함될 수 있다.

즉 `'use client'`는 단순 문법이 아니라,  
**번들 경계와 실행 환경 경계**를 자르는 선언이다.

### 자주 하는 오해

- 인터랙션 있는 컴포넌트마다 전부 붙여야 한다 → 꼭 그렇진 않다
- `'use client'`를 붙이면 그 파일만 영향 받는다 → 하위 import 구조도 중요하다
- `'use client'`는 성능 최적화와 상관없다 → 클라이언트 번들 크기와 직접 연결된다

### 실무 연결

그래서 실무에서는 보통:

- 최대한 작은 인터랙션 leaf 쪽에만 `'use client'`를 두고
- 상위 layout/page는 가능한 서버에 남기는 구조를 선호한다

### 예시

```tsx
// app/posts/[id]/page.tsx
import LikeButton from "./like-button";

export default async function Page() {
  const post = await getPost();

  return (
    <article>
      <h1>{post.title}</h1>
      <LikeButton likes={post.likes} />
    </article>
  );
}
```

```tsx
// app/posts/[id]/like-button.tsx
"use client";

import { useState } from "react";

export default function LikeButton({ likes }: { likes: number }) {
  const [count, setCount] = useState(likes);

  return <button onClick={() => setCount((prev) => prev + 1)}>{count}</button>;
}
```

### 여기서 다음 개념으로 어떻게 넘어가나

이제 `'use client'`가 경계라는 걸 봤다.  
그다음엔 자연스럽게 **클라이언트로 내리면 어떤 비용이 생기나**를 봐야 한다.

### 면접 압축 답변

`'use client'`는 단순히 “이 파일은 브라우저에서 실행된다”는 표시가 아니라, Server Components 트리 안에서 클라이언트 진입 경계를 선언하는 역할을 합니다. 그래서 상위 큰 파일에 무심코 붙이면 하위 모듈까지 클라이언트 번들에 포함될 수 있어 경계를 작게 잡는 감각이 중요합니다.

## 71. 'use client'의 비용은 무엇인가

앞 질문에서 `'use client'`는 단순 문법이 아니라 경계라고 했다.  
그렇다면 당연히 비용도 따라온다.

### 핵심 개념

Client Component로 내릴수록 보통 아래 비용을 더 고려해야 한다.

- 더 많은 클라이언트 자바스크립트 전송
- 더 많은 파싱/실행 비용
- hydration 대상 증가
- 브라우저 환경 의존 증가

### 왜 이게 중요한가

이걸 모르고 모든 걸 클라이언트로 두면  
Next.js를 쓰면서도 사실상 서버 컴포넌트의 이점을 많이 잃게 된다.

즉 좋은 설계는:

- 가능한 것은 서버에 두고
- 꼭 인터랙션이 필요한 부분만 클라이언트로 내리는 것
  에 가깝다.

### 자주 하는 오해

- Client Component는 어차피 현대 브라우저가 다 감당한다 → 저사양 기기/느린 네트워크에선 체감 차이가 크다
- 인터랙션 하나 있으면 페이지 전체를 클라이언트로 바꿔야 한다 → 더 작은 단위로 자를 수 있다
- 클라이언트에서 fetch하는 게 더 단순하다 → 번들, 로딩, SEO, 데이터 계층을 같이 봐야 한다

### 실무 연결

실무에서는 자주 이런 판단이 나온다.

- 상단 summary, 본문, 메타 정보 → 서버
- 좋아요 버튼, 필터 토글, 정렬 드롭다운 → 클라이언트
- 무거운 차트 상호작용 → 클라이언트
- 데이터 자체는 서버에서 준비 후 props로 전달 가능

### 여기서 다음 개념으로 어떻게 넘어가나

이제 클라이언트 경계 비용을 봤다.  
그다음엔 자연스럽게 **데이터를 어디서 fetch하는가**로 넘어가야 한다.

### 면접 압축 답변

`'use client'`의 비용은 클라이언트 번들 증가, 파싱과 실행 비용 증가, hydration 대상 증가로 이어질 수 있다는 점입니다. 그래서 인터랙션이 필요한 부분만 작게 클라이언트 경계로 두고, 나머지는 서버에 남기는 설계가 보통 더 유리합니다.

## 72. App Router에서는 데이터를 어디서 가져오는 게 자연스러운가

현재 공식 문서 흐름에선 Server Component에서 `fetch`나 데이터베이스/ORM 접근을 사용하는 패턴이 중심이고, Client Component에선 React의 `use` API나 커뮤니티 라이브러리(SWR, React Query 등)를 활용할 수 있다. :contentReference[oaicite:4]{index=4}

### 핵심 개념

App Router에선 기본적으로 아래 감각이 자연스럽다.

- 서버에서 준비해도 되는 데이터는 Server Component에서 fetch
- 인터랙션에 반응해 클라이언트에서 갱신되는 데이터는 Client Component 또는 서버 상태 도구 활용
- 서버와 클라이언트를 억지로 한쪽으로 몰지 말고 역할에 맞게 나눔

### 왜 이게 중요한가

이걸 이해하면:

- 첫 화면 로딩 구조
- SEO 대응
- 비밀값 사용 위치
- 캐시 전략
- 클라이언트 번들 크기
  를 함께 판단할 수 있다.

### 자주 하는 오해

- App Router에선 무조건 서버 fetch만 해야 한다 → 틀림
- 클라이언트 fetch는 구식이다 → 틀림
- 클라이언트에서 가져오면 Next.js 장점을 잃는다 → 상황에 따라 필요한 패턴이다

### 실무 연결

예:

- 게시글 본문, 상품 기본 정보, 공개 페이지 데이터 → 서버 fetch가 자연스러울 수 있음
- 검색 자동완성, 실시간 필터, 사용자 조작에 따른 즉시 재조회 → 클라이언트 fetch가 자연스러울 수 있음

### 여기서 다음 개념으로 어떻게 넘어가나

이제 fetch 위치를 봤다.  
그다음엔 자연스럽게 **캐싱과 재검증을 어떻게 설계하는가**로 넘어가야 한다.

### 면접 압축 답변

App Router에서는 서버에서 준비해도 되는 데이터는 Server Component에서 가져오고, 사용자 상호작용에 따라 자주 바뀌는 데이터는 Client Component나 서버 상태 도구로 다루는 구분이 자연스럽습니다. 핵심은 한쪽만 고집하는 게 아니라 데이터 성격에 맞게 fetch 위치를 정하는 것입니다.

## 73. Next.js의 캐싱은 왜 “성능 옵션”이 아니라 설계 문제인가

현재 공식 문서 기준으로 Next.js App Router는 `fetch`, `revalidatePath`, `revalidateTag`, `use cache`, `cacheComponents` 같은 캐싱·재검증 도구를 제공한다. 또한 문서에는 `fetch` 응답 자체는 기본적으로 캐시되지 않지만, 라우트 출력은 prerender 및 캐시될 수 있다고 설명한다. :contentReference[oaicite:5]{index=5}

### 핵심 개념

Next.js에서 캐싱을 생각할 때 중요한 질문은 이거다.

- 이 데이터는 얼마나 자주 바뀌는가
- 사용자마다 달라지는가
- stale 허용이 가능한가
- 빌드 시점에 만들어도 되는가
- 재검증은 시간 기반인가, 이벤트 기반인가

### 왜 이게 중요한가

캐시는 단순히 “빠르게 해주는 옵션”이 아니라,  
**데이터 신선도와 성능 사이의 트레이드오프**를 정하는 설계다.

예를 들어:

- 문서/블로그/상품 안내 페이지 → stale 허용 가능
- 재고/알림/개인 대시보드 → 더 신선한 데이터 요구
- 관리 페이지 수정 후 즉시 반영 → on-demand 재검증 감각 필요

### 자주 하는 오해

- 캐시는 무조건 많이 걸수록 좋다 → stale 이슈가 생길 수 있다
- fetch만 캐시하면 끝이다 → 경로/세그먼트 출력, 태그 재검증도 같이 볼 수 있다
- 재검증은 시간만 정하면 된다 → mutation 이후 태그/경로 무효화가 더 잘 맞을 수 있다

### 실무 연결

좋은 답변은 보통 이렇게 간다.

- 정적인 콘텐츠는 더 aggressively 캐시
- 변경 후 즉시 반영이 필요한 데이터는 태그/경로 재검증 고려
- 사용자별 개인화 데이터는 요청 시 처리 or 짧은 캐시 전략 고려

### 여기서 다음 개념으로 어떻게 넘어가나

이제 캐싱을 큰 그림에서 봤다.  
그다음엔 Next.js 안에서 서버 엔드포인트를 만드는 **Route Handlers**로 넘어가야 한다.

### 면접 압축 답변

Next.js의 캐싱은 단순 성능 옵션이 아니라 데이터 신선도와 응답 속도 사이의 트레이드오프를 설계하는 문제입니다. 그래서 어떤 데이터는 미리 생성하고 캐시하며, 어떤 데이터는 요청 시점에 가져오고, 어떤 데이터는 태그나 경로 단위로 재검증할지 함께 판단해야 합니다.

## 74. Route Handlers는 왜 필요하고, 어디에 쓰나

공식 문서 기준으로 Route Handlers는 App Router의 `app` 디렉터리 안에서 `route.ts|js` 파일로 정의하며, Web Request/Response API 기반의 커스텀 요청 처리를 제공한다. 이는 Pages Router의 API Routes에 대응되는 역할로 설명된다. :contentReference[oaicite:6]{index=6}

### 핵심 개념

Route Handlers는 Next.js 앱 안에서  
서버 엔드포인트를 직접 만들 수 있는 구조다.

예를 들어:

- `/app/api/users/route.ts`
- `/app/posts/[id]/route.ts`

처럼 만들 수 있다.

### 왜 이게 중요한가

이게 필요한 이유는:

- 외부 API를 감싸는 BFF 역할
- 서버 전용 비밀값 사용
- 요청 가공/검증/정규화
- 웹훅 처리
- 파일 업로드 처리
  같은 서버 문맥이 프론트 앱 옆에 자연스럽게 붙기 때문이다.

### 자주 하는 오해

- Next.js는 화면만 만들고 API는 별도 서버가 필수다 → 틀림
- Route Handlers가 있으면 백엔드 설계는 불필요하다 → 틀림
- 모든 서버 로직을 Route Handlers에 몰아도 된다 → 역할 분리가 여전히 중요하다

### 실무 연결

대표적인 사용처:

- 외부 CMS/API 응답 정리
- 인증 토큰 부착 후 대리 요청
- 클라이언트에 숨기고 싶은 API key 사용
- webhook endpoint
- 내부 BFF 계층

### 여기서 다음 개념으로 어떻게 넘어가나

이제 서버 엔드포인트 구조를 봤다.  
그다음엔 실제 제품 설계 감각으로 자주 나오는 **BFF**로 넘어갈 수 있다.

### 면접 압축 답변

Route Handlers는 App Router 안에서 서버 엔드포인트를 만들 수 있게 해주는 구조로, BFF 역할이나 외부 API 래핑, 웹훅 처리, 서버 비밀값 사용 같은 문맥에 잘 맞습니다. 즉 Next.js 앱 안에 필요한 서버 경계를 자연스럽게 둘 수 있게 해줍니다.

## 75. BFF는 왜 프론트엔드 관점에서도 중요한가

BFF(Backend for Frontend)는 프론트엔드 면접에서 자주 나오는 개념이다.  
특히 Next.js는 서버 경계를 앱 안에 두기 쉬워서 이 이야기를 하기가 좋다.

### 핵심 개념

BFF는 프론트엔드가 직접 여러 백엔드 서비스와 통신하는 대신,  
프론트엔드 요구에 맞게 가공된 서버 계층을 하나 두는 구조다.

### 왜 이게 중요한가

BFF를 두면:

- 여러 API 응답을 합쳐서 프론트 친화적 shape로 만들 수 있고
- 인증/인가 로직을 중앙화할 수 있고
- 클라이언트에 비밀값을 숨길 수 있고
- 요청 수를 줄이거나 응답을 더 프론트 친화적으로 정리할 수 있다

### 자주 하는 오해

- BFF는 거창한 대규모 시스템에만 필요하다 → 작은 규모에서도 유용할 수 있다
- Route Handler를 만들면 곧 BFF다 → 역할과 목적을 더 분명히 가져가야 한다
- BFF가 있으면 프론트가 단순해진다 → 때론 백엔드/프론트 경계 설계가 더 중요해진다

### 실무 연결

예:

- 상품 상세를 위해 리뷰, 재고, 추천 API를 합쳐 하나로 제공
- 사용자별 권한 확인 후 필요한 필드만 프론트에 제공
- 외부 API 스키마를 내부 UI 친화적 모델로 바꿔 전달

### 여기서 다음 개념으로 어떻게 넘어가나

이제 서버 계층을 앱 가까이에 두는 이유를 봤다.  
그다음엔 라우팅과 상태가 만나는 대표 주제인 **URL 상태 설계**로 넘어가야 한다.

### 면접 압축 답변

BFF는 프론트엔드 요구에 맞는 서버 계층을 두어 여러 백엔드 응답을 가공하고 인증, 비밀값, 응답 정규화를 처리하는 구조입니다. Next.js에서는 Route Handlers 같은 서버 경계를 통해 이런 역할을 앱 가까이에 둘 수 있다는 점이 중요합니다.

---

## 76. URL 상태는 왜 전역 상태와 다른 축으로 봐야 하나

필터, 정렬, 탭, 페이지네이션 같은 값을 어디에 둘지는 프론트엔드 설계에서 자주 헷갈리는 지점이다.  
이때 중요한 질문은 이거다.

> **이 상태는 새로고침, 공유 링크, 뒤로가기/앞으로가기에 살아 있어야 하는가?**

그렇다면 URL 상태 후보일 가능성이 크다.

### 핵심 개념

URL 상태는 단순 UI 메모리가 아니라,
**라우팅과 탐색의 일부가 되는 상태**다.

예:

- 검색어
- 필터
- 정렬
- 현재 페이지
- 선택된 탭
- 상세 모달 대상 id

### 왜 이게 중요한가

URL 상태로 두면 얻는 이점이 있다.

- 공유 가능한 링크
- 새로고침 후 복원
- 브라우저 히스토리와 정합성
- 검색/필터 화면의 사용성 향상

### 자주 하는 오해

- 상태는 전부 store에 두는 게 관리가 쉽다 → URL이 더 자연스러운 상태도 많다
- URL 상태는 SEO용이다 → 탐색성과 복원성도 중요하다
- 사소한 토글도 URL에 넣어야 한다 → 노이즈가 될 수 있다

### 실무 연결

좋은 기준:

- 공유할 가치가 있는가
- 뒤로가기에서 기대되는가
- 외부에서 직접 진입할 때 복원돼야 하는가
- 앱 내부 일시 인터랙션에 가깝나

### 여기서 다음 개념으로 어떻게 넘어가나

이제 URL 상태를 봤다.  
그다음엔 Next.js에서 **서버와 클라이언트가 URL 상태를 어떻게 함께 다루는가**로 이어진다.

### 면접 압축 답변

URL 상태는 단순 전역 상태가 아니라 라우팅과 탐색의 일부가 되는 상태입니다. 검색어, 필터, 정렬, 페이지네이션처럼 공유 링크와 새로고침 복원, 뒤로가기 정합성이 중요한 값은 store보다 URL에 두는 편이 더 자연스러운 경우가 많습니다.

---

## 77. Next.js에서 URL 상태와 데이터 페칭은 어떻게 함께 설계하나

URL 상태를 이해했다면 다음 질문은 자연스럽다.

> **query string이나 route param이 바뀌면, 데이터는 어디서 다시 가져와야 할까?**

### 핵심 개념

App Router에선 URL이 서버 렌더링과 데이터 페칭의 입력값이 될 수 있다.  
즉 URL 상태는 단순 클라이언트 메모리가 아니라,  
**서버 fetch의 key 역할**도 할 수 있다.

### 왜 이게 중요한가

이 설계를 잘하면:

- 새로고침 시 같은 결과 복원
- 공유 링크로 같은 화면 복원
- 서버에서 초기 데이터 준비
- 클라이언트에서 후속 인터랙션 확장
  이 자연스러워진다.

### 자주 하는 오해

- 필터는 클라이언트 state로만 둬야 한다 → URL/서버 입력으로 함께 갈 수 있다
- URL이 바뀌면 무조건 전체 페이지를 클라이언트에서 다시 그린다 → 서버/클라이언트 구성이 다를 수 있다
- 모든 검색 입력을 즉시 URL에 반영해야 한다 → debounce나 commit 시점 분리가 필요할 수 있다

### 실무 연결

예:

- `/products?sort=price&page=2`
- `/dashboard?tab=analytics`
- `/search?q=nextjs`

이런 값들은  
단순 UI 상태이면서 동시에 데이터 조회 기준이 된다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 URL과 데이터가 연결되는 구조를 봤다.  
그다음엔 요청을 가로채거나 초기 단계에서 분기하는 **middleware** 로 넘어갈 수 있다.

### 면접 압축 답변

Next.js에서는 URL 상태가 단순 클라이언트 메모리가 아니라 서버 데이터 페칭의 입력값이 될 수 있습니다. 그래서 검색어, 필터, 페이지 같은 값을 URL에 두면 공유와 복원성뿐 아니라 초기 서버 데이터 준비와도 자연스럽게 연결됩니다.

---

## 78. Middleware는 언제 유용하고, 언제 과용되기 쉬운가

Next.js에선 요청 초기에 특정 로직을 적용하고 싶을 때 middleware를 고려할 수 있다.  
다만 이건 만능 훅처럼 쓰는 게 아니라, 정말 요청 초반 공통 분기가 필요할 때 생각하는 게 좋다.

### 핵심 개념

middleware는 라우트 진입 전에 실행되는 공통 요청 처리 계층으로 볼 수 있다.

주로:

- 인증/인가 리다이렉트
- 지역화 분기
- 특정 경로 리라이트
- 공통 정책 적용

같은 곳에서 쓰기 좋다.

### 왜 이게 중요한가

middleware를 이해하면  
페이지 컴포넌트 안에서 하기엔 너무 늦은 작업과,  
요청 초기에 걸러야 하는 작업을 구분하게 된다.

### 자주 하는 오해

- 로직은 다 middleware에 두면 깔끔하다 → 과하면 복잡성과 디버깅 난이도가 커진다
- middleware는 클라이언트 라우팅 훅 같은 것이다 → 서버/엣지 쪽 요청 경계에 더 가깝다
- 인증은 무조건 middleware에서 해야 한다 → 페이지/서버 컴포넌트/Route Handler와 역할 분리가 필요할 수 있다

### 실무 연결

예:

- 로그인 안 된 사용자는 `/dashboard` 접근 시 `/login`으로
- 국가/언어별 경로 분기
- 특정 실험군에 따른 rewrite

### 여기서 다음 개념으로 어떻게 넘어가나

이제 Next.js의 요청 초반 경계까지 봤다.  
그다음엔 이 모든 걸 합쳐 **서버 중심 기본값 + 필요한 곳만 클라이언트**라는 설계 원칙으로 정리할 수 있다.

### 면접 압축 답변

middleware는 요청 초기에 공통 정책을 적용해야 할 때 유용합니다. 예를 들어 인증 리다이렉트, locale 분기, rewrite 같은 작업에 잘 맞지만, 모든 로직을 몰아넣는 만능 계층으로 쓰면 오히려 구조가 복잡해질 수 있습니다.

---

## 79. Next.js App Router에서 좋은 기본 설계 원칙은 무엇인가

여기까지 오면 Next.js를 단순 기능 모음이 아니라 설계 원칙으로 볼 수 있어야 한다.

### 핵심 원칙

1. **기본은 서버 중심으로 생각한다**
2. **인터랙션이 필요한 부분만 클라이언트로 내린다**
3. **URL은 탐색 가능한 상태만 담는다**
4. **서버 데이터와 클라이언트 UI 상태를 구분한다**
5. **캐시는 성능 옵션이 아니라 데이터 신선도 설계다**
6. **서버 경계(Route Handlers/BFF)는 외부 API와 비밀값, 응답 정규화에 활용한다**
7. **`'use client'`는 경계 비용이 있다는 걸 항상 의식한다**

### 왜 이게 중요한가

이 원칙이 있으면  
Next.js 면접 질문이 기능별 암기가 아니라  
**구조적 판단 근거**로 바뀐다.

### 자주 하는 오해

- Next.js 정답은 항상 서버 우선이다 → 인터랙션 요구사항과 데이터 성격을 같이 봐야 한다
- 페이지 단위로만 생각하면 된다 → layout/segment/boundary 감각이 중요하다
- fetch 위치만 정하면 설계가 끝난다 → 캐시, 재검증, URL 상태, 경계 설계가 같이 간다

### 면접 압축 답변

Next.js App Router에서 좋은 기본 원칙은 서버 중심으로 시작하되, 인터랙션이 필요한 부분만 클라이언트로 내리고, URL 상태와 서버 상태, 로컬 UI 상태를 분리해서 설계하는 것입니다. 여기에 캐시와 서버 경계까지 함께 보면 기능 암기보다 구조적 판단이 가능해집니다.

## 80. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- Next.js는 React 위에 앱 구조를 얹어주는 프레임워크다
- App Router는 현재 공식 문서 기준의 중심 구조다
- 렌더링 전략은 SSR/SSG/ISR/CSR 같은 시점과 캐시 문제를 포함한다
- Server Component와 Client Component는 실행 환경 분리다
- `'use client'`는 진입 경계 선언이자 번들 경계다
- 데이터 fetch 위치는 서버/클라이언트 역할에 따라 달라진다
- 캐시는 성능과 신선도의 트레이드오프 설계다
- Route Handlers와 BFF는 서버 경계를 앱 가까이에 둘 수 있게 한다
- URL 상태는 탐색 가능한 상태를 담는 좋은 자리다
- middleware는 요청 초반 공통 분기에 유용하다
- 좋은 기본값은 “서버 중심 + 필요한 곳만 클라이언트”다

즉,  
이번 챕터는 단순히 Next.js API를 배우는 챕터가 아니라,  
**“React 앱을 서버·클라이언트·캐시·라우팅·데이터 경계까지 포함해 어떻게 설계할 것인가”**를 배우는 챕터다.
