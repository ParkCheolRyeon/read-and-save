# 챕터 11. 프로젝트 면접 스토리에서 문제 정의와 운영 판단까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **프로젝트 면접 답변은 무엇을 만들었는가보다, 어떤 문제를 어떻게 정의했고 왜 그런 구조를 선택했으며 운영 가능한 결과로 어떻게 연결했는가를 설명하는 구조로 가져가야 한다.**

### 전체 흐름 맵

```
프로젝트 면접 핵심 (기술 나열이 아니라 문제 정의와 판단 기준)
  → 대표 프로젝트 선택 + 자기소개 연결 (강점 압축 사례 → 정체성 선언)
    → 프로젝트 설명 골격 + 문제 정의 (답변 뼈대 → "누가 왜 막혔는가")
      → 기술 선택 ("좋은 기술"이 아니라 "맞는 기술")
        → 운영 제약 + 장애 대응 (현장 조건 → 복구 가능 UX)
          → 성과 (숫자보다 연결 논리)
            → 실패 + 회고 (판단 기준의 성장)
              → 협업 충돌 + 90일 계획 (정렬 방식 → 적응과 기여)
                → 다음 챕터로의 연결
```

## 147. 프로젝트 면접에서 가장 먼저 보여줘야 하는 건 무엇인가

> **핵심 요약** — 프로젝트 면접의 핵심은 기술 스택이 아니라, 문제를 어떻게 정의하고 어떤 기준으로 판단했는지다. **문제 정의 → 판단 기준 → 구현 → 결과 → 배운 점** 흐름을 기본으로 잡아야 한다.

### 핵심 개념

많은 지원자가 프로젝트 면접에서
기술 스택과 구현 기능부터 길게 말한다.

하지만 면접관이 더 궁금해하는 건 보통 이거다.

> **이 사람은 문제를 어떤 방식으로 보고, 어떤 기준으로 판단하는가?**

프로젝트 답변의 핵심은 기술 나열이 아니라
**문제 정의 → 판단 기준 → 구현 → 결과 → 배운 점**의 흐름이다.

즉 좋은 프로젝트 답변은:

- 무엇을 만들었는지보다
- 왜 그 문제가 중요했는지
- 어떤 선택지가 있었는지
- 왜 이 선택을 했는지를 더 잘 드러내야 한다.

### 왜 이게 중요한가

기술 스택은 프로젝트마다 다를 수 있다.
하지만 문제를 정의하고 구조를 설계하는 방식은
지원자의 일관된 역량으로 보인다.

즉 면접관은 프로젝트를 통해 이런 걸 본다.

- 이 사람이 기능 요청을 어떻게 해석하는가
- 단기 구현과 장기 유지보수 사이에서 무엇을 우선하는가
- 사용자 경험과 운영 비용을 같이 볼 수 있는가
- 문제를 증상 수준이 아니라 구조 수준으로 보는가

### 자주 하는 오해

- 프로젝트 면접은 기술 자랑 시간이다 → 틀림
- 많이 만든 프로젝트일수록 유리하다 → 핵심은 깊이와 판단 기준이다
- 구현 디테일을 많이 말하면 전문적으로 보인다 → 문제 정의가 빠지면 오히려 흐려질 수 있다

### 실무 에피소드 — 이런 상황에서 만난다

> 면접에서 프로젝트 설명을 시작하자마자 "React, TypeScript, Zustand, React Query를 사용했고 컴포넌트를 30개 만들었습니다"로 시작한 지원자가 있었다. 면접관이 "그래서 왜 그 프로젝트를 만든 건가요?"라고 물으니 말문이 막혔다. 반면 다른 지원자는 "관리자가 매일 같은 데이터를 수작업으로 옮기고 있었는데, 이게 하루 40분씩 반복되는 구조적 문제였습니다"로 시작했더니 이후 모든 기술 선택이 자연스럽게 따라왔다.

### 면접 압축 답변

- **10초**: 프로젝트 면접의 핵심은 기술 스택이 아니라, 문제를 어떻게 정의하고 어떤 기준으로 판단했는지입니다.
- **30초**: 프로젝트 면접에서 가장 먼저 보여줘야 하는 건 기술 스택 자체보다, 문제를 어떻게 정의하고 어떤 기준으로 판단했는지입니다. 무엇을 만들었는가보다 왜 그 문제를 중요하게 봤고, 어떤 선택지 중에서 왜 이 선택을 했으며, 운영 가능한 결과로 어떻게 연결했는지가 더 핵심입니다.

## 148. 대표 프로젝트 선택과 자기소개 연결

> **핵심 요약** — 대표 프로젝트는 "가장 최신"이 아니라 **내 강점을 가장 압축적으로 보여주는 사례**여야 한다. 그리고 1분 자기소개는 그 프로젝트가 나를 증명하게 만드는 구조로 연결해야 한다.

### 대표 프로젝트를 고르는 기준

대표 프로젝트는 단순히 "가장 최신 것"이나 "가장 화려한 것"이 아니라,
**내 강점을 가장 압축적으로 보여주는 프로젝트**여야 한다.

좋은 대표 프로젝트는 보통 아래 조건을 가진다.

- 문제 정의가 선명하다
- 내가 맡은 범위가 넓다
- 기술 선택 이유를 설명하기 쉽다
- 운영/장애/성능/구조 판단까지 말할 수 있다
- 결과와 배운 점이 뚜렷하다

### 왜 이게 중요한가

프로젝트 면접은 시간이 짧다.
그래서 모든 프로젝트를 비슷한 무게로 말하면 오히려 강점이 흐려진다.

즉 대표 프로젝트 하나는
**나를 설명하는 압축 샘플**처럼 써야 한다.

좋은 전략은 이렇다.

- 1순위 대표 프로젝트 1개
- 보조 프로젝트 2~3개
- 대표 프로젝트에서 못 보여준 역량을 보조 프로젝트로 보완

예:

- 대표 프로젝트 → 운영/안정성/1인 개발 역량
- 보조 프로젝트 A → 복잡한 상태 관리와 UI 설계
- 보조 프로젝트 B → 사용자 문의 감소, 관리자 UX, 운영 자동화

### 자기소개를 프로젝트와 연결하는 법

자기소개는 프로젝트 설명과 따로 노는 문장이 되기 쉽다.
하지만 가장 좋은 자기소개는 곧바로 프로젝트 이야기로 이어져야 한다.

좋은 1분 자기소개는 아래 흐름을 가진다.

1. 나는 어떤 문제를 잘 푸는 사람인가
2. 어떤 기술 환경을 다뤄왔는가
3. 내 강점이 어떤 프로젝트에서 드러났는가

즉 자기소개와 프로젝트를 분리하지 말고,
프로젝트가 자기소개를 증명하게 만들어야 한다.

### 예시 흐름

- 저는 프론트엔드 개발자지만 단순히 화면 구현보다 사용자 흐름이 실제로 끊기지 않게 만드는 데 관심이 많습니다.
- React, Next.js, TypeScript 기반으로 복잡한 상태 구조, 네트워크 불안정 대응, 관리자 도구, 배포와 운영 자동화까지 경험했습니다.
- 특히 대표 프로젝트에서는 프론트엔드뿐 아니라 운영 가능한 구조를 직접 설계하면서, 기술 선택을 항상 사용자 경험과 운영 비용 기준으로 판단해왔습니다.

이 구조가 좋은 이유는:

- 기술 스택 나열로 끝나지 않고
- 내가 무엇을 중요하게 보는 개발자인지 드러나고
- 이후 프로젝트 질문이 들어왔을 때 자연스럽게 이어지기 때문이다

### 자주 하는 오해

- 가장 큰 서비스면 무조건 대표 프로젝트다 → 꼭 그렇진 않다
- 가장 구현량이 많으면 유리하다 → 판단 기준 설명이 더 중요하다
- 자기소개는 이력 요약이다 → 정체성 선언이 더 중요하다
- 프로젝트는 뒤에서 따로 설명하면 된다 → 시작부터 연결되는 편이 강하다

### 실무 에피소드 — 이런 상황에서 만난다

> 프로젝트가 4개 있는 지원자가 각각을 2분씩 동일한 무게로 설명했다. 8분이 지나고 나서도 면접관은 "이 사람의 핵심 강점이 뭐지?" 감이 안 잡혔다. 반면 대표 프로젝트 하나를 3분 깊게 말하고 나머지를 30초씩 보완 역할로 붙인 지원자는 강점이 뚜렷하게 전달됐다. 자기소개에서 "저는 사용자 흐름이 끊기지 않는 구조를 만드는 데 집중합니다"라고 선언한 뒤 대표 프로젝트가 바로 그 증거로 이어지니, 면접 전체 흐름이 한 줄로 꿰어졌다.

### 면접 압축 답변

- **10초**: 대표 프로젝트는 가장 최신이 아니라 내 강점을 가장 압축적으로 보여주는 사례여야 하고, 자기소개가 그 프로젝트로 자연스럽게 이어져야 합니다.
- **30초**: 대표 프로젝트는 문제 정의가 선명하고, 기술 선택 이유를 설명하기 쉽고, 운영이나 장애 대응까지 말할 수 있는 프로젝트를 고릅니다. 자기소개에서는 내가 어떤 문제를 잘 푸는 사람인지 먼저 선언하고, 대표 프로젝트가 바로 그 증거가 되게 연결하면 이후 질문도 자연스럽게 흐릅니다.

## 149. 프로젝트 설명 골격과 문제 정의

> **핵심 요약** — 프로젝트 답변의 기본 골격은 **문제 상황 → 왜 실제 문제인가 → 판단 기준 → 선택 → 결과 → 배운 점**이다. 특히 문제 정의는 "기능 설명"이 아니라 "누가 어떤 상황에서 왜 막히고 있었는가"를 먼저 깔아야 이후 모든 설명에 맥락이 생긴다.

### 기본 답변 골격

프로젝트 면접에서 흔들리지 않으려면
프로젝트마다 같은 골격으로 답할 수 있어야 한다.

가장 안정적인 프로젝트 답변 골격은 아래다.

1. **문제 상황** — 어떤 맥락이었는가
2. **왜 그게 실제 문제였는지** — 사용자나 운영에 어떤 영향이 있었는가
3. **어떤 기준으로 판단했는지** — 속도? 안정성? 유지보수? 사용자 경험?
4. **어떤 구조나 기술을 선택했는지** — 대안과 비교해서 왜 이걸 골랐는가
5. **결과와 성과** — 정량/정성 지표
6. **배운 점 / 다시 한다면 바꿀 점** — 회고

이 골격은 거의 모든 프로젝트에 적용 가능하다.

이 구조가 있으면:

- 질문이 바뀌어도 중심을 잃지 않고
- 구현 디테일에만 빠지지 않으며
- 꼬리 질문에도 답변 연결이 쉽다

### 예시 흐름

| 단계 | 내용 |
|---|---|
| 문제 상황 | 현장에서 데이터 기록 흐름이 네트워크 환경에 취약했다 |
| 왜 실제 문제였나 | 업무 시간 안에 입력이 끊기면 바로 작업 중단과 불만으로 이어졌다 |
| 판단 기준 | 최신 기술보다 운영 가능한 안정성이 우선이었다 |
| 선택 | fallback UI, retry, 임시 저장, CI/CD |
| 결과 | 일 평균 1,000건 이상 기록 생성 구조 안정 운영 |
| 배운 점 | 기술 선택은 이상적 구조보다 현장 제약을 먼저 봐야 했다 |

### 문제 정의가 먼저 와야 하는 이유

많은 프로젝트 답변이 약해지는 가장 큰 이유는
문제를 말하지 않고 곧바로 구현 이야기로 들어가기 때문이다.

문제 정의란 단순히
"이런 기능이 필요했다"가 아니다.

더 정확히는:

- **누가** — 어떤 사용자 또는 이해관계자가
- **어떤 상황에서** — 어떤 환경이나 조건 하에서
- **무엇 때문에 막히고 있었는지** — 구체적인 병목이나 불편
- **그 문제가 왜 반복적이고 중요한지** — 비즈니스나 운영에 미치는 영향

를 설명하는 것이다.

문제 정의가 선명해야:

- 기술 선택 이유도 자연스럽고
- 결과도 의미가 생기고
- 면접관이 "아, 그래서 그 판단을 했구나"라고 따라올 수 있다

문제 정의가 빠진 답변은 면접관 입장에서 "그래서 왜 그걸 만든 건데?"라는 질문이 계속 남게 된다. 반면 문제가 먼저 깔리면 이후의 모든 설명이 그 문제를 해결하기 위한 맥락 안에서 이해된다.

### 좋은 문제 정의 vs 약한 문제 정의

```
// 약한 문제 정의
"데이터 기록 기능을 만들었습니다."

// 좋은 문제 정의
"현장에서 업무 중에 데이터를 실시간으로 기록해야 하는데,
네트워크가 불안정한 환경이라 입력 도중 요청이 실패하면
담당자가 업무 흐름을 멈추고 처음부터 다시 입력해야 하는 상황이었습니다.
이게 반복되면서 현장 불만이 쌓였고,
단순 기능 버그가 아니라 업무 중단 문제로 봐야 했습니다."
```

### 자주 하는 오해

- 프로젝트 설명은 기능 순서대로 해야 한다 → 문제 정의 중심이 더 강하다
- 기술 선택부터 말하는 게 전문적으로 보인다 → 맥락이 먼저 있어야 한다
- 기능 요구사항을 말하면 문제 정의다 → 불충분하다. 요구사항은 "무엇을 만들라"이고, 문제 정의는 "왜 만들어야 하는가"이다
- 문제를 길게 말하면 지루하다 → 오히려 이후 답변이 선명해진다

### 실무 에피소드 — 이런 상황에서 만난다

> 같은 프로젝트를 설명하는데 두 지원자의 인상이 완전히 달랐다. 한 명은 "게시판 CRUD를 만들었습니다. React Query로 서버 상태를 관리하고 Zustand로 클라이언트 상태를 분리했습니다"로 시작했고, 다른 한 명은 "관리자가 매일 엑셀에서 데이터를 복사해서 시스템에 붙여넣고 있었는데, 이 과정에서 입력 오류가 월 평균 15건씩 발생하고 있었습니다"로 시작했다. 후자가 말하는 기술 선택은 전부 그 문제를 해결하기 위한 맥락 안에서 이해됐다.

### 면접 압축 답변

- **10초**: 프로젝트 설명은 문제 상황 → 판단 기준 → 선택 → 결과 → 회고 순서가 가장 안정적이고, 문제 정의는 "누가 왜 막혔는가"가 핵심입니다.
- **30초**: 프로젝트 하나를 설명할 때 가장 안정적인 구조는 문제 상황, 실제 영향, 판단 기준, 선택한 구조, 결과, 배운 점 순서입니다. 특히 문제 정의는 기능 설명이 아니라 누가 어떤 상황에서 왜 반복적으로 막히고 있었는지를 설명하는 것이고, 이 부분이 선명해야 이후 기술 선택과 결과도 맥락을 갖게 됩니다.

## 150. 기술 선택은 왜 "좋은 기술"이 아니라 "맞는 기술"의 문제인가

> **핵심 요약** — 기술 선택 답변에서 "유명해서", "최신이라서"는 약하다. **문제와 가장 잘 맞는가, 팀 규모에 적절한가, 운영 비용을 감당 가능한가, 사용자 경험에 실제로 도움이 되는가**를 기준으로 설명해야 한다.

### 핵심 개념

프로젝트 면접에서 기술 선택 질문은 정말 자주 나온다.
여기서 가장 약한 답변은 이런 식이다.

- 많이 써서 익숙해서
- 커뮤니티에서 많이 써서
- 최신이라서

이런 답변은 이유가 너무 약하다.

기술 선택은 보통 아래를 기준으로 설명하는 게 강하다.

- **문제와 가장 잘 맞는가** — 이 기술이 해결하려는 문제를 직접적으로 다루는가
- **팀/프로젝트 규모에 적절한가** — 과하지도, 부족하지도 않은가
- **운영과 유지보수 비용을 감당 가능한가** — 장기적으로 관리할 수 있는가
- **사용자 경험에 실제로 도움이 되는가** — 기술적 우아함이 아닌 실질적 개선인가

즉 "좋은 기술"보다
**현재 맥락에 맞는 기술**이 핵심이다.

### 역할 분리 기준이 드러나는 답변

예를 들어 상태 관리 도구와 데이터 검증 도구를 같이 쓴 프로젝트에서
"유명해서 썼다"가 아니라 이렇게 가야 한다.

- 편집 중 실시간 상태는 전역적으로 관리할 필요가 있었다 → 전역 상태 관리 도구
- 저장/전송 경계에서 데이터 정합성 검증이 필요했다 → 스키마 검증 도구
- 즉 편집 경험과 데이터 계약 검증의 역할을 분리했다

이렇게 역할 분리 기준이 드러나야 한다.

```typescript
// 전역 상태: 편집 중 실시간 상태 관리 (전역적이고 빈번하게 변경)
const useItemStore = create<ItemState>((set) => ({
  items: [],
  selectedId: null,
  updateItem: (id, patch) =>
    set((state) => ({
      items: state.items.map((t) =>
        t.id === id ? { ...t, ...patch } : t
      ),
    })),
}));

// 스키마 검증: 저장/전송 경계에서 데이터 정합성 검증
const itemSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1),
  category: z.enum(["typeA", "typeB", "typeC"]),
  order: z.number().int().nonnegative(),
});

// 저장 시점에 검증 — 편집 중에는 자유롭게, 저장할 때만 엄격하게
function saveItem(item: unknown) {
  const result = itemSchema.safeParse(item);
  if (!result.success) {
    return { ok: false, errors: result.error.flatten() };
  }
  return api.save(result.data);
}
```

핵심은 **편집 중 자유도**와 **저장 시점 엄격성**의 역할 분리다. 면접에서 이런 식으로 설명하면 기술 선택이 역할 기반 의사결정이었음이 드러난다.

### 자주 하는 오해

- 최신 기술을 썼다고 하면 좋아 보인다 → 맥락이 없으면 약하다
- 기술 비교를 많이 하면 전문적으로 보인다 → 판단 기준이 더 중요하다
- 하나의 기술 장점만 말하면 충분하다 → 왜 이 프로젝트에 맞았는지까지 필요하다

### 실무 에피소드 — 이런 상황에서 만난다

> 면접에서 "왜 Zustand를 선택했나요?"라는 질문에 "Redux보다 보일러플레이트가 적어서요"라고 답한 지원자가 있었다. 면접관이 "그럼 Jotai나 Recoil도 보일러플레이트가 적은데, 왜 Zustand이었나요?"라고 되물었더니 답이 막혔다. 반면 "편집 화면에서 여러 컴포넌트가 동일한 아이템 상태를 실시간으로 읽고 써야 했고, 이 패턴이 구독 기반 전역 스토어와 가장 잘 맞아서 선택했습니다"라고 답한 지원자는 꼬리 질문에도 흔들리지 않았다.

### 면접 압축 답변

- **10초**: 기술 선택은 "좋은 기술"이 아니라, 현재 문제와 운영 조건에 가장 맞는 기술을 고르는 문제입니다.
- **30초**: 기술 선택은 유명하거나 최신이라서가 아니라, 현재 문제와 팀 규모, 운영 조건에 가장 맞는 기술을 고르는 문제입니다. 예를 들어 전역 상태 관리 도구는 편집 중 실시간 상태를, 스키마 검증 도구는 저장 경계의 데이터 검증을 담당하게 했는데, 이건 편집 자유도와 저장 엄격성이라는 역할을 분리하기 위한 선택이었습니다.

## 151. 운영 제약과 장애 대응

> **핵심 요약** — 운영 제약(네트워크 불안정, 세션 만료, 사용자 실수 가능성 등)을 인식하고 기술 판단에 반영했다는 것 자체가 실무 감각의 증거다. 장애 대응은 시스템 복구뿐 아니라 **사용자 흐름 보호**(retry → 임시 저장 → fallback UI)까지 포함해야 깊이가 드러난다.

### 운영 제약이 프로젝트 답변을 강하게 만드는 이유

운영 제약이란 보통 이런 것들이다.

- 네트워크 불안정
- 세션 만료
- 오래된 기기 환경
- 관리자 업무의 반복성
- 사용자 실수 가능성
- 배포 빈도와 장애 비용
- 데이터 손실 위험

운영 제약은 "이상적인 기술 설계"와 "현실에서 돌아가는 설계" 사이의 간격을 만드는 요소다. 이 간격을 인식하고 의사결정에 반영했다는 것 자체가 실무 감각의 증거가 된다.

이걸 말할 수 있어야
기술 선택이 단순 취향이 아니라 **운영 판단**으로 보인다.

### 장애 대응이 프로젝트 경험을 깊게 보이게 하는 이유

운영 경험이 없는 답변은 보통 happy path 위주로 끝난다.
반면 실제 서비스를 다뤄본 사람은
"안 되는 상황에서 어떻게 복구할지"를 말할 수 있다.

좋은 장애 대응 답변은 보통 두 층을 함께 가진다.

- **시스템 대응** — retry, timeout, circuit breaker, 롤백 등 기술적 복구 경로
- **사용자 경험 대응** — 사용자가 현재 무슨 일이 일어나고 있는지 알 수 있게 하고, 데이터를 잃지 않게 보호하는 것

즉 서버나 프로세스만 복구하는 게 아니라,
사용자가 흐름을 잃지 않게 만드는 것도 포함된다.

### 코드로 보는 장애 대응 패턴

```typescript
// 1단계: retry 로직 — 일시적 네트워크 실패에 대응
async function fetchWithRetry(
  fn: () => Promise<Response>,
  maxRetries = 3,
  delay = 1000
) {
  let lastError: unknown;
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const res = await fn();
      if (res.ok) return res;
      if (attempt === maxRetries - 1) return res;
    } catch (e) {
      lastError = e;
      if (attempt === maxRetries - 1) throw e;
    }
    await new Promise((r) => setTimeout(r, delay * (attempt + 1)));
  }
  throw lastError;
}

// 2단계: 임시 저장 — retry로도 안 되면 데이터 보호
function saveToLocalDraft(key: string, data: unknown) {
  try {
    localStorage.setItem(
      `draft:${key}`,
      JSON.stringify({ data, savedAt: Date.now() })
    );
  } catch {
    // localStorage 용량 초과 시 가장 오래된 draft 제거 후 재시도
  }
}

// 3단계: fallback UI — 사용자에게 복구 경로 안내
function SubmitFallback({ draftKey }: { draftKey: string }) {
  return (
    <div role="alert">
      <p>저장 중 문제가 발생했습니다.</p>
      <p>입력 내용은 임시 저장되었습니다.</p>
      <button onClick={() => retryFromDraft(draftKey)}>
        다시 시도
      </button>
    </div>
  );
}
```

핵심은 retry → 임시 저장 → 사용자 안내라는 **3단계 방어**다. 에러 메시지 하나 띄우는 것과, 데이터를 보호하고 복구 경로를 제공하는 것은 완전히 다른 수준의 대응이다.

### 자주 하는 오해

- 운영 얘기는 인프라 담당자만 중요하게 본다 → 제품 개발자에게도 핵심이다
- 운영 제약은 부정적인 얘기라 약점처럼 보인다 → 오히려 실전 감각으로 보일 수 있다
- 장애 대응은 백엔드 문제다 → 프론트도 흐름 보호를 설계해야 한다
- 에러 메시지를 띄우면 충분하다 → 복구 가능성이 더 중요하다
- 장애는 드문 예외라 면접에서 굳이 말 안 해도 된다 → 오히려 깊이를 보여준다

### 실무 에피소드 — 이런 상황에서 만난다

> 현장에서 쓰이는 데이터 입력 앱이 있었는데, 와이파이가 불안정한 환경에서 사용자가 폼을 작성하다가 제출 시점에 네트워크가 끊기면 입력 내용이 통째로 날아갔다. 처음에는 "네트워크 오류입니다" 토스트 하나만 띄웠는데, 현장에서 매일 같은 불만이 올라왔다. retry 로직을 넣고, 실패 시 localStorage에 임시 저장하고, 복구 버튼을 제공하는 3단계 방어를 적용하니 데이터 유실 관련 문의가 사라졌다. 기술적으로 대단한 건 아니었지만, 사용자 입장에서는 "데이터가 안 날아간다"는 신뢰가 핵심이었다.

### 면접 압축 답변

- **10초**: 운영 제약을 설명할 수 있으면 기술 선택이 취향이 아니라 운영 판단으로 보이고, 장애 대응은 시스템 복구와 사용자 흐름 보호를 함께 말해야 깊어집니다.
- **30초**: 운영 제약과 현장 맥락을 설명할 수 있으면 기술 선택이 단순 취향이 아니라 운영 판단으로 보입니다. 장애 대응도 정상 동작뿐 아니라 실패 상황에서의 복구까지 설명할 수 있어야 합니다. retry, 임시 저장, fallback UI처럼 시스템 복구와 사용자 흐름 보호를 함께 설계한 점을 보여주면 에러 메시지 하나 띄우는 것과는 완전히 다른 수준의 대응이 됩니다.

## 152. 성과는 왜 숫자 자체보다 연결 논리가 중요하나

> **핵심 요약** — 좋은 성과 답변은 숫자 나열이 아니라 **어떤 지표를 왜 봤고, 내 작업이 그 변화와 어떻게 연결되는가**를 설명하는 것이다. 정량 지표가 없어도 간접 지표(운영 문의 감소, 반복 오류 감소 등)로 충분히 설득력을 만들 수 있다.

### 핵심 개념

프로젝트 면접에서 숫자를 말하면 분명 도움이 된다.
하지만 숫자만 던지면 의미가 약할 수 있다.

좋은 성과 답변은 보통 아래 구조를 가진다.

1. **어떤 지표를 봤는가** — 왜 이 지표를 선택했는가
2. **그 지표가 왜 중요한가** — 비즈니스나 사용자 경험과 어떻게 연결되는가
3. **무엇이 바뀌었는가** — before/after
4. **그 변화가 내 작업과 어떻게 연결되는가** — 어떤 구조 변경이 이 결과를 만들었는가

즉 숫자는 **맥락 안에서 해석**돼야 한다.

### 좋은 성과 연결 vs 약한 성과 연결

```
// 약한 성과 답변
"운영 문의가 30건에서 6건으로 줄었습니다."

// 좋은 성과 답변
"관리자 도구에서 가장 반복되는 문의를 분석해보니,
정보 탐색 구조가 불명확해서 같은 질문이 반복되고 있었습니다.
정보 구조를 재설계하고 자주 묻는 흐름에 대한 안내를 추가했더니,
월 문의가 30건에서 6건으로 줄었습니다.
즉 기능 추가가 아니라 정보 구조 개선이 핵심이었습니다."
```

### 자주 하는 오해

- 큰 숫자만 있으면 된다 → 숫자가 왜 중요한지 설명해야 한다
- 공식 KPI가 없으면 성과를 말할 수 없다 → 간접 지표도 충분히 의미 있다. 운영 문의 감소, 반복 오류 감소, 배포 실패율 감소 등도 입증 가능한 성과다
- 정량이 없으면 약하다 → 정성 + 정량 연결이 중요하다

### 실무 에피소드 — 이런 상황에서 만난다

> "페이지 로딩 속도를 3초에서 1.2초로 줄였습니다"라고 말한 지원자에게 면접관이 "그래서 사용자에게 어떤 차이가 있었나요?"라고 물었다. 답이 없었다. 반면 "검색 결과 페이지에서 이탈률이 40%였는데, 로딩 체감 시간을 줄이기 위해 스켈레톤 UI와 점진적 로딩을 적용했더니 이탈률이 25%로 떨어졌습니다. 속도 자체보다 사용자가 '기다릴 만하다'고 느끼는 구간을 만든 게 핵심이었습니다"라고 답한 지원자는 숫자와 의미가 함께 전달됐다.

### 면접 압축 답변

- **10초**: 성과는 숫자 자체보다, 그 숫자가 내 작업과 어떻게 연결되는지를 설명하는 것이 핵심입니다.
- **30초**: 성과는 숫자 자체보다, 어떤 지표를 왜 중요하게 봤고 그 변화가 내 작업과 어떻게 연결되는지를 설명하는 것이 중요합니다. 예를 들어 "문의가 30건에서 6건으로 줄었다"만 말하면 약하지만, "정보 구조를 재설계해서 반복 문의가 구조적으로 줄었다"까지 연결하면 훨씬 설득력이 생깁니다.

## 153. 실패 경험과 회고

> **핵심 요약** — 실패 답변은 숨기기보다 **판단 → 문제 발생 → 복구 → 기준 변화** 구조로 말해야 하고, "지금 다시 만든다면?" 질문은 **당시 맥락과 지금 기준의 차이**를 설명해서 성장성을 보여줘야 한다.

### 실패를 구조적으로 말하는 법

실패 질문이 나오면 많은 사람이 방어적으로 변한다.
하지만 실제로는 이 질문이 오히려 신뢰를 만들 기회가 되기도 한다.

좋은 실패 답변은 아래 구조를 가진다.

1. **어떤 판단을 했는가** — 당시의 맥락과 선택
2. **왜 그 판단이 문제였는가** — 어떤 가정이 틀렸거나 무엇을 놓쳤는가
3. **어떤 문제가 발생했는가** — 구체적 영향
4. **어떻게 복구했는가** — 즉각 대응
5. **이후 무엇을 바꿨는가** — 프로세스나 기준의 변화

즉 실패를 자랑처럼 포장하지 말고,
**판단 기준의 성장**으로 연결해야 한다.

주의할 점은 실패 답변에서 "치명적이지만 복구 불가능했던 실패"를 꺼내는 것보다, **복구 과정과 학습이 명확한 실패**를 선택하는 것이 전략적으로 더 좋다는 것이다.

### "지금 다시 만든다면 무엇을 바꾸겠나"

이 질문은 단순 후회 질문이 아니다.
현재의 시각으로 예전 판단을 다시 평가할 수 있는지 보는 질문이다.

좋은 답변은 아래를 포함한다.

- 당시 어떤 기준으로 선택했는가
- 지금 보면 어떤 한계가 있었는가
- 지금이라면 무엇을 다르게 선택할 것인가
- 그 이유는 무엇인가

즉 "옛날의 나"를 무시하는 게 아니라
**그때의 맥락과 지금의 기준 차이**를 설명해야 한다.

이 질문에 잘 답하려면 두 가지를 동시에 보여줘야 한다: 당시 판단이 무근거가 아니었다는 점, 그리고 지금은 더 나은 기준을 가지고 있다는 점.

### 회고 감각이 드러나는 코드 예시

```typescript
// 당시 선택: 모든 상태를 전역 store에 넣음
const useStore = create((set) => ({
  items: [],
  selectedItem: null,
  filterQuery: "",      // <- URL에 있어야 할 상태
  sortOrder: "recent",  // <- URL에 있어야 할 상태
  modalOpen: false,     // <- 로컬 상태로 충분
}));

// 지금이라면: URL 상태와 로컬 상태를 분리
// URL 상태 — 공유 가능하고 브라우저 뒤로가기에 반응해야 하는 것
const searchParams = useSearchParams();
const filterQuery = searchParams.get("q") ?? "";
const sortOrder = searchParams.get("sort") ?? "recent";

// 전역 상태 — 여러 컴포넌트가 동시에 읽고 써야 하는 것
const useItemStore = create((set) => ({
  items: [],
  selectedItem: null,
}));

// 로컬 상태 — 해당 컴포넌트 안에서만 의미 있는 것
const [modalOpen, setModalOpen] = useState(false);
```

이런 식으로 말하면
회고 능력과 구조 감각이 함께 드러난다.

### 자주 하는 오해

- 실패한 적이 없다고 말하는 게 안전하다 → 오히려 얕아 보일 수 있다. 실무 경험이 있는 사람은 반드시 실패를 겪는다
- 실패를 자랑처럼 미화해야 한다 → 진정성이 떨어진다. 사실 기반으로 담담하게 말하는 게 낫다
- "다시 해도 똑같이 한다"가 자신감 있어 보인다 → 오히려 성장성이 약해 보일 수 있다
- 예전 결정을 부정하면 안 된다 → 당시 맥락을 설명하면 괜찮다

### 실무 에피소드 — 이런 상황에서 만난다

> 프로젝트 초기에 모든 상태를 전역 스토어에 넣었다. 당시에는 "어디서든 접근 가능하니까 편하겠지"라는 판단이었는데, 프로젝트가 커지면서 필터 값이 URL에 반영되지 않아 "링크를 공유하면 필터가 초기화된다"는 문제가 반복됐다. 나중에 URL 상태, 전역 상태, 로컬 상태의 경계를 분리하는 리팩토링을 했는데, 처음부터 이 기준을 잡았으면 리팩토링 비용이 절반 이하였을 것이다. 이후 프로젝트에서는 상태를 만들 때 "이 값은 누구와 공유돼야 하는가?"를 먼저 묻는 습관이 생겼다.

### 면접 압축 답변

- **10초**: 실패는 복구 과정과 기준 변화를 구조적으로 말하고, 회고 질문은 당시 맥락과 지금 기준의 차이를 설명하면 됩니다.
- **30초**: 실패 경험은 숨기기보다, 어떤 판단이 왜 문제였고 어떻게 복구했으며 이후 기준이 어떻게 바뀌었는지를 구조적으로 설명하는 편이 강합니다. "지금 다시 만든다면" 질문도 후회가 아니라, 당시 판단이 무근거는 아니었지만 지금은 더 나은 기준을 가지고 있다는 점을 동시에 보여주면 회고 능력과 성장성이 함께 드러납니다.

## 154. 협업 충돌과 90일 계획

> **핵심 요약** — 협업 질문은 "누가 맞았나"가 아니라 **서로의 우려를 어떤 근거로 정렬했는가**가 핵심이고, 90일 계획은 **파악 → 빠른 기여 → 구조적 개선** 순서로 겸손함과 자신감을 함께 보여줘야 한다.

### 협업에서 의견 충돌을 말하는 법

협업 질문에서 가장 피해야 할 건
상대를 틀린 사람처럼 묘사하는 것이다.

좋은 협업 답변은 보통 아래 구조를 가진다.

1. **어떤 맥락에서 입장 차이가 있었는가** — 프로젝트 상황
2. **상대는 무엇을 우려했는가** — 상대 관점을 공정하게 서술
3. **나는 무엇을 우려했는가** — 내 관점과 근거
4. **어떤 근거로 정렬했는가** — 데이터, 사용자 피드백, 기술적 분석 등
5. **어떤 합의안으로 갔는가** — 최종 결정과 그 결과

즉 핵심은 승부가 아니라 **정렬 방식**이다.

### 입사 후 90일 계획

이 질문은 "새 조직에서 얼마나 빨리 적응하고 기여할 수 있나"를 보는 질문이다.
여기서 너무 공격적으로 말하면 기존 맥락을 무시하는 사람처럼 보이고,
너무 조심스럽게 말하면 기여감이 약해 보일 수 있다.

좋은 답변은 보통 3단계로 나뉜다.

#### 1단계: 파악 (1~30일)

- 코드베이스 구조와 컨벤션
- 배포 흐름과 CI/CD 파이프라인
- 팀 협업 방식 (코드 리뷰, 스프린트 등)
- 현재 자주 발생하는 운영 이슈

#### 2단계: 빠른 기여 (30~60일)

- 작은 버그 수정
- 리뷰 참여
- 작은 개선 (DX 개선, 테스트 추가 등)
- 팀이 체감하는 즉시 가치 제공

#### 3단계: 구조적 개선 제안 (60~90일)

- 사용자 흐름 단절 지점 개선
- 운영 비용 큰 영역 최적화
- 상태 구조나 성능, 배포 개선 포인트 제안

이 구조는:

- 기존 맥락을 존중하면서
- 빠른 기여 의지도 보여주고
- 장기적으로 구조 개선까지 생각한다는 점을 드러낸다

### 자주 하는 오해

- 충돌이 없었다고 말하는 게 좋다 → 오히려 협업 깊이가 약해 보일 수 있다
- 내가 맞았고 설득했다로 끝내면 된다 → 팀워크 역량이 약해 보일 수 있다
- 입사하자마자 큰 구조를 바꾸겠다고 말한다 → 위험해 보일 수 있다
- 배우기만 하겠다고 말한다 → 기여 의지가 약해 보인다
- 90일 계획은 형식적 질문이다 → 조직 적응력과 문제 파악 능력을 본다

### 실무 에피소드 — 이런 상황에서 만난다

> 기획팀은 기존 UI 흐름을 유지해야 일정이 안전하다고 봤고, 개발팀은 현재 구조의 유지보수 비용이 이미 한계에 왔다고 봤다. 양쪽 다 근거가 있었다. 결국 "전체를 한 번에 바꾸지 말고, 가장 문제가 큰 핵심 흐름 하나만 먼저 재설계하고 나머지는 점진 개선한다"는 합의안으로 갔다. 결과적으로 일정도 크게 밀리지 않았고, 핵심 흐름의 유지보수 비용이 눈에 띄게 줄었다. 면접에서 이런 이야기를 할 때 핵심은 "내가 이겼다"가 아니라 "어떤 기준으로 합의점을 찾았는가"다.

### 면접 압축 답변

- **10초**: 협업 충돌은 누가 맞았는지보다 어떻게 정렬했는지가 중요하고, 90일 계획은 파악 → 빠른 기여 → 구조 개선 순서가 안정적입니다.
- **30초**: 협업에서 의견 충돌은 상대 관점을 공정하게 서술하고 데이터나 사용자 피드백 같은 근거로 정렬한 과정이 드러나야 합니다. 90일 계획은 기존 코드와 팀 맥락을 먼저 이해하고, 작은 개선으로 신뢰를 쌓은 뒤, 이후에 구조적인 개선 포인트를 제안하는 흐름이 기존 팀을 존중하면서도 적극적으로 기여하겠다는 의지를 함께 보여줍니다.

이 챕터의 전체 흐름을 활용하는 순서는 이렇다.

1. 프로젝트 면접의 핵심을 기술 나열이 아니라 문제 정의와 판단 기준으로 다시 잡는다.
2. 대표 프로젝트와 자기소개 연결 구조를 먼저 만든다.
3. 프로젝트별 문제 정의, 기술 선택, 운영 제약, 장애 대응, 성과를 같은 골격으로 정리한다.
4. 실패, 회고, 협업 충돌, 90일 계획까지 붙여서 사람 자체의 판단 기준을 드러낸다.
5. 마지막에 전체 프로젝트 답변을 하나의 스토리로 묶는다.

이렇게 보면 프로젝트 면접이 단순 경험 나열이 아니라, 내가 어떤 기준으로 문제를 보고 구조를 설계하고 운영 가능한 결과로 연결하는 사람인지를 보여주는 자리라는 감각이 더 선명해진다.

# 챕터 11 핵심 요약 — 경험을 문제 해결 철학으로 묶어 전달하는 프로젝트 면접 구조

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **프로젝트 면접에서 기술 나열을 넘어, 내가 문제를 어떻게 정의하고 판단하는 사람인지를 어떤 구조로 보여줄 것인가?**

---

### 1단계: 면접관이 진짜 보는 것을 이해한다 — 기술이 아니라 판단 기준

프로젝트 면접의 핵심은 기술 스택이 아니라, **문제를 어떻게 정의하고 어떤 기준으로 판단했는가**다.

### 2단계: 나를 압축적으로 보여줄 사례를 고른다 — 대표 프로젝트와 자기소개

대표 프로젝트는 "가장 최신"이 아니라 **내 강점을 가장 압축적으로 보여주는 사례**여야 한다.

### 3단계: 흔들리지 않는 답변 뼈대를 세운다 — 프로젝트 설명 골격과 문제 정의

가장 안정적인 답변 구조는 **문제 상황 → 왜 실제 문제인가 → 판단 기준 → 선택 → 결과 → 배운 점**이다. 문제 정의는 기능 설명이 아니라 **"누가 어떤 상황에서 왜 반복적으로 막히고 있었는가"**를 깔아야 한다.

### 4단계: 기술 선택을 역할 기반 의사결정으로 설명한다 — "맞는 기술"

**문제와 가장 잘 맞는가, 팀 규모에 적절한가, 운영 비용을 감당 가능한가, 사용자 경험에 실제로 도움이 되는가**가 기준이다.

---

여기서부터 "무엇을 만들었는가"에서 "현실에서 어떻게 운영하고 복구했는가"로 전환된다.

---

### 5단계: 현장 조건을 기술 판단에 반영한다 — 운영 제약과 장애 대응

**운영 제약을 인식하고 기술 판단에 반영했다는 것 자체가 실무 감각의 증거**다. 장애 대응은 시스템 복구뿐 아니라 **사용자 흐름 보호(임시 저장, fallback UI)**까지 포함해야 한다.

### 6단계: 숫자에 맥락을 입힌다 — 성과의 연결 논리

좋은 성과 답변은 숫자 나열이 아니라, **어떤 지표를 왜 봤고, 내 작업이 그 변화와 어떻게 연결되는가**를 설명하는 것이다.

### 7단계: 실패를 판단 기준의 성장으로 연결한다 — 실패와 회고

실패 답변은 **판단 → 문제 발생 → 복구 → 기준 변화** 구조로 말해야 한다. 예전 결정을 부정하지 않고, 당시 맥락과 지금 기준의 차이를 설명한다.

### 8단계: 정렬 방식과 적응 전략을 보여준다 — 협업 충돌과 90일 계획

협업 질문의 핵심은 **서로의 우려를 어떤 근거로 정렬했는가**다. 90일 계획은 **파악(1~30일) → 빠른 기여(30~60일) → 구조적 개선(60~90일)** 순서다.

---

### 한 줄로 이어 읽기

**면접관이 보는 것**이 기술이 아니라 판단 기준이고 → **대표 프로젝트와 자기소개**가 나를 압축적으로 증명하고 → **답변 골격과 문제 정의**가 뼈대를 세우고 → **기술 선택**이 역할 기반 의사결정으로 설명되고 → **운영 제약과 장애 대응**이 현장 감각을 증명하고 → **성과의 연결 논리**가 숫자에 의미를 입히고 → **실패와 회고**가 판단 기준의 성장을 드러내고 → **협업과 90일 계획**이 정렬 방식과 적응 전략을 보여준다.
