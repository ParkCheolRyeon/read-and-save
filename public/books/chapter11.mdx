# 챕터 11. 프로젝트 면접 스토리에서 문제 정의와 운영 판단까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **프로젝트 면접 답변은 무엇을 만들었는가보다, 어떤 문제를 어떻게 정의했고 왜 그런 구조를 선택했으며 운영 가능한 결과로 어떻게 연결했는가를 설명하는 구조로 가져가야 한다.**

---

## 147. 프로젝트 면접에서 가장 먼저 보여줘야 하는 건 무엇인가

많은 지원자가 프로젝트 면접에서
기술 스택과 구현 기능부터 길게 말한다.

하지만 면접관이 더 궁금해하는 건 보통 이거다.

> **이 사람은 문제를 어떤 방식으로 보고, 어떤 기준으로 판단하는가?**

### 핵심 개념

프로젝트 답변의 핵심은 기술 나열이 아니라
**문제 정의 → 판단 기준 → 구현 → 결과 → 배운 점**의 흐름이다.

즉 좋은 프로젝트 답변은:

- 무엇을 만들었는지보다
- 왜 그 문제가 중요했는지
- 어떤 선택지가 있었는지
- 왜 이 선택을 했는지를 더 잘 드러내야 한다.

### 왜 이게 중요한가

기술 스택은 프로젝트마다 다를 수 있다.
하지만 문제를 정의하고 구조를 설계하는 방식은
지원자의 일관된 역량으로 보인다.

즉 면접관은 프로젝트를 통해 이런 걸 본다.

- 이 사람이 기능 요청을 어떻게 해석하는가
- 단기 구현과 장기 유지보수 사이에서 무엇을 우선하는가
- 사용자 경험과 운영 비용을 같이 볼 수 있는가
- 문제를 증상 수준이 아니라 구조 수준으로 보는가

### 자주 하는 오해

- 프로젝트 면접은 기술 자랑 시간이다 → 틀림
- 많이 만든 프로젝트일수록 유리하다 → 핵심은 깊이와 판단 기준이다
- 구현 디테일을 많이 말하면 전문적으로 보인다 → 문제 정의가 빠지면 오히려 흐려질 수 있다

### 실무 연결

네 포트폴리오의 강점은
"기술을 많이 썼다"보다
**기술 선택이 사용자 흐름과 운영 가능성으로 연결된다**는 점이다.

양치킹, Tool-Manager, 아트봉봉, Ecole-Admin 모두
이 관점으로 묶으면 훨씬 강해진다.

> 이제 프로젝트 면접의 핵심 관점을 잡았다. 그다음엔 실제로 **어떤 프로젝트를 대표 사례로 먼저 꺼낼지** 정해야 한다.

### 면접 압축 답변

**10초 답변**
프로젝트 면접의 핵심은 기술 스택이 아니라, 문제를 어떻게 정의하고 어떤 기준으로 판단했는지입니다.

**30초 답변**
프로젝트 면접에서 가장 먼저 보여줘야 하는 건 기술 스택 자체보다, 문제를 어떻게 정의하고 어떤 기준으로 판단했는지입니다. 무엇을 만들었는가보다 왜 그 문제를 중요하게 봤고, 어떤 선택지 중에서 왜 이 선택을 했으며, 운영 가능한 결과로 어떻게 연결했는지가 더 핵심입니다.

## 148. 대표 프로젝트는 어떻게 고르는 게 좋은가

프로젝트가 여러 개 있으면
면접에서 어떤 걸 앞에 세울지부터 전략이 필요하다.

### 핵심 개념

대표 프로젝트는 단순히 "가장 최신 것"이나 "가장 화려한 것"이 아니라,
**내 강점을 가장 압축적으로 보여주는 프로젝트**여야 한다.

좋은 대표 프로젝트는 보통 아래 조건을 가진다.

- 문제 정의가 선명하다
- 내가 맡은 범위가 넓다
- 기술 선택 이유를 설명하기 쉽다
- 운영/장애/성능/구조 판단까지 말할 수 있다
- 결과와 배운 점이 뚜렷하다

### 왜 이게 중요한가

프로젝트 면접은 시간이 짧다.
그래서 모든 프로젝트를 비슷한 무게로 말하면 오히려 강점이 흐려진다.

즉 대표 프로젝트 하나는
**나를 설명하는 압축 샘플**처럼 써야 한다.

### 네 흐름에 맞는 선택

현재 네 포트폴리오 기준으로
가장 먼저 강조하기 좋은 프로젝트는 보통 **양치킹** 쪽이다.

이유는:

- 1인 개발
- 프론트엔드 + 백엔드 + 배포 + 운영
- 실제 현장 제약
- 네트워크 불안정 대응
- fallback / retry / 임시 저장
- 운영 가능한 구조 판단

까지 한 번에 보여주기 때문이다.

### 자주 하는 오해

- 가장 큰 서비스면 무조건 대표 프로젝트다 → 꼭 그렇진 않다
- 가장 구현량이 많으면 유리하다 → 판단 기준 설명이 더 중요하다
- 여러 프로젝트를 비슷하게 다 말해야 공정하다 → 오히려 강점이 분산된다

### 실무 연결

좋은 전략은 이렇다.

- 1순위 대표 프로젝트 1개
- 보조 프로젝트 2~3개
- 대표 프로젝트에서 못 보여준 역량을 보조 프로젝트로 보완

예:

- 양치킹 → 운영/안정성/1인 개발
- Tool-Manager → 상태 구조와 DnD 복잡 UI
- 아트봉봉 / Ecole-Admin → VOC 감소, 관리자 UX, 운영 자동화

> 이제 대표 프로젝트를 정했다. 그다음엔 실제 면접 도입부인 **1분 자기소개와 프로젝트 연결**로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
대표 프로젝트는 가장 최신이 아니라, 내 강점을 가장 압축적으로 보여주는 사례여야 합니다.

**30초 답변**
대표 프로젝트는 가장 최신이거나 화려한 프로젝트가 아니라, 내 강점을 가장 압축적으로 보여주는 사례여야 합니다. 문제 정의가 선명하고, 기술 선택 이유를 설명하기 쉽고, 운영이나 장애 대응까지 말할 수 있는 프로젝트를 대표로 세우는 것이 좋습니다. 저는 양치킹이 1인 개발 범위, 운영 경험, 안정성 설계까지 한 번에 설명할 수 있어서 대표 사례로 가장 강합니다.

## 149. 1분 자기소개는 프로젝트와 어떻게 연결해야 하나

자기소개는 프로젝트 설명과 따로 노는 문장이 되기 쉽다.
하지만 가장 좋은 자기소개는 곧바로 프로젝트 이야기로 이어져야 한다.

### 핵심 개념

좋은 1분 자기소개는 아래 흐름을 가진다.

1. 나는 어떤 문제를 잘 푸는 사람인가
2. 어떤 기술 환경을 다뤄왔는가
3. 내 강점이 어떤 프로젝트에서 드러났는가

즉 자기소개와 프로젝트를 분리하지 말고,
프로젝트가 자기소개를 증명하게 만들어야 한다.

### 예시 흐름

- 저는 프론트엔드 개발자지만 단순히 화면 구현보다 사용자 흐름이 실제로 끊기지 않게 만드는 데 관심이 많습니다.
- React, Next.js, TypeScript 기반으로 복잡한 상태 구조, 네트워크 불안정 대응, 관리자 도구, 배포와 운영 자동화까지 경험했습니다.
- 특히 양치킹 같은 1인 개발 프로젝트에선 프론트엔드뿐 아니라 운영 가능한 구조를 직접 설계하면서, 기술 선택을 항상 사용자 경험과 운영 비용 기준으로 판단해왔습니다.

### 왜 이게 중요한가

이 구조가 좋은 이유는:

- 기술 스택 나열로 끝나지 않고
- 내가 무엇을 중요하게 보는 개발자인지 드러나고
- 이후 프로젝트 질문이 들어왔을 때 자연스럽게 이어지기 때문이다

### 자주 하는 오해

- 자기소개는 이력 요약이다 → 정체성 선언이 더 중요하다
- 기술을 많이 말할수록 좋아 보인다 → 판단 기준이 더 중요하다
- 프로젝트는 뒤에서 따로 설명하면 된다 → 시작부터 연결되는 편이 강하다

> 이제 자기소개와 프로젝트 연결 방식은 잡혔다. 그다음엔 프로젝트 하나를 설명할 때 쓸 **기본 답변 골격**이 필요하다.

### 면접 압축 답변

**10초 답변**
좋은 자기소개는 기술 나열이 아니라, 내가 어떤 문제를 잘 푸는 사람인지 선언하고 프로젝트로 증명하는 구조입니다.

**30초 답변**
좋은 1분 자기소개는 기술 나열보다, 내가 어떤 문제를 잘 푸는 사람인지 먼저 드러내고 그걸 대표 프로젝트가 증명하게 만드는 구조가 좋습니다. 자기소개와 프로젝트를 분리하지 말고, 프로젝트 경험이 내 정체성을 설명하도록 연결하면 이후 질문도 자연스럽게 이어집니다.

## 150. 프로젝트 하나를 설명할 때 가장 안정적인 기본 골격은 무엇인가

프로젝트 면접에서 흔들리지 않으려면
프로젝트마다 같은 골격으로 답할 수 있어야 한다.

### 핵심 개념

가장 안정적인 프로젝트 답변 골격은 아래다.

1. **문제 상황** — 어떤 맥락이었는가
2. **왜 그게 실제 문제였는지** — 사용자나 운영에 어떤 영향이 있었는가
3. **어떤 기준으로 판단했는지** — 속도? 안정성? 유지보수? 사용자 경험?
4. **어떤 구조나 기술을 선택했는지** — 대안과 비교해서 왜 이걸 골랐는가
5. **결과와 성과** — 정량/정성 지표
6. **배운 점 / 다시 한다면 바꿀 점** — 회고

이 골격은 거의 모든 프로젝트에 적용 가능하다.

### 왜 이게 중요한가

이 구조가 있으면:

- 질문이 바뀌어도 중심을 잃지 않고
- 구현 디테일에만 빠지지 않으며
- 꼬리 질문에도 답변 연결이 쉽다

### 예시: 양치킹 흐름

| 단계 | 내용 |
|---|---|
| 문제 상황 | 학교 현장 기록 흐름이 네트워크에 취약했다 |
| 왜 실제 문제였나 | 수업 시간 안에 입력이 끊기면 바로 불만으로 이어졌다 |
| 판단 기준 | 최신 기술보다 운영 가능한 안정성이 우선이었다 |
| 선택 | fallback UI, retry, 임시 저장, CI/CD |
| 결과 | 일 평균 1,000건 이상 기록 생성 구조 운영 |
| 배운 점 | 기술 선택은 이상적 구조보다 현장 제약을 먼저 봐야 했다 |

### 자주 하는 오해

- 프로젝트 설명은 기능 순서대로 해야 한다 → 문제 정의 중심이 더 강하다
- 기술 선택부터 말하는 게 전문적으로 보인다 → 맥락이 먼저 있어야 한다
- 결과는 숫자만 말하면 된다 → 숫자가 왜 의미 있는지도 같이 말해야 한다

> 이제 프로젝트 설명 골격을 잡았다. 그다음엔 이 골격의 첫 부분인 **문제 정의를 어떻게 날카롭게 만들지**로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
프로젝트 설명은 문제 상황 → 판단 기준 → 선택 → 결과 → 회고 순서가 가장 안정적입니다.

**30초 답변**
프로젝트 하나를 설명할 때 가장 안정적인 구조는 문제 상황, 문제의 실제 영향, 판단 기준, 선택한 구조나 기술, 결과, 배운 점 순서입니다. 이 골격을 쓰면 질문이 바뀌어도 중심을 잃지 않고, 꼬리 질문에도 자연스럽게 연결할 수 있어서 구현 나열형 답변보다 훨씬 설득력 있습니다.

## 151. 문제 정의는 왜 "기능 설명"보다 먼저 와야 하나

많은 프로젝트 답변이 약해지는 가장 큰 이유는
문제를 말하지 않고 곧바로 구현 이야기로 들어가기 때문이다.

### 핵심 개념

문제 정의란 단순히
"이런 기능이 필요했다"가 아니다.

더 정확히는:

- **누가** — 어떤 사용자 또는 이해관계자가
- **어떤 상황에서** — 어떤 환경이나 조건 하에서
- **무엇 때문에 막히고 있었는지** — 구체적인 병목이나 불편
- **그 문제가 왜 반복적이고 중요한지** — 비즈니스나 운영에 미치는 영향

를 설명하는 것이다.

### 왜 이게 중요한가

문제 정의가 선명해야:

- 기술 선택 이유도 자연스럽고
- 결과도 의미가 생기고
- 면접관이 "아, 그래서 그 판단을 했구나"라고 따라올 수 있다

문제 정의가 빠진 답변은 면접관 입장에서 "그래서 왜 그걸 만든 건데?"라는 질문이 계속 남게 된다. 반면 문제가 먼저 깔리면 이후의 모든 설명이 그 문제를 해결하기 위한 맥락 안에서 이해된다.

### 실무 연결

예를 들어 양치킹에서 핵심 문제는
"양치 기록 기능이 있었다"가 아니다.

더 정확히는:

- 학교 현장에서는 네트워크가 불안정하고
- 기록 흐름이 수업 맥락 안에서 끊기면
- 단순 버그가 아니라 사용자 업무 중단으로 이어진다

는 점이다.

이렇게 말해야 운영과 UX가 함께 드러난다.

### 좋은 문제 정의 vs 약한 문제 정의

```
// 약한 문제 정의
"양치 기록 기능을 만들었습니다."

// 좋은 문제 정의
"학교 현장에서 수업 중에 양치 기록을 남겨야 하는데,
네트워크가 불안정한 환경이라 입력 도중 요청이 실패하면
교사가 수업 흐름을 멈추고 다시 기록해야 하는 상황이었습니다.
이게 반복되면서 현장 불만이 쌓였고,
단순 기능 버그가 아니라 업무 중단 문제로 봐야 했습니다."
```

### 자주 하는 오해

- 기능 요구사항을 말하면 문제 정의다 → 불충분하다. 요구사항은 "무엇을 만들라"이고, 문제 정의는 "왜 만들어야 하는가"이다
- 기술적 어려움을 곧 문제라고 말하면 된다 → 사용자 맥락이 빠질 수 있다
- 문제를 길게 말하면 지루하다 → 오히려 이후 답변이 선명해진다

> 이제 문제 정의의 중요성을 봤다. 그다음엔 면접관이 가장 듣고 싶어하는 부분 중 하나인 **기술 선택 이유**로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
문제 정의는 기능 설명이 아니라, 누가 왜 막히고 있었는지를 설명하는 것이고 이게 먼저 와야 기술 선택에 맥락이 생깁니다.

**30초 답변**
문제 정의는 단순 기능 설명이 아니라, 누가 어떤 상황에서 무엇 때문에 반복적으로 막히고 있었는지를 설명하는 것입니다. 이 부분이 선명해야 이후 기술 선택과 결과도 맥락을 갖게 됩니다. 예를 들어 양치킹에서는 "양치 기록 기능을 만들었다"가 아니라 "네트워크 불안정한 학교 현장에서 수업 중 기록이 끊기면 업무 중단으로 이어진다"가 진짜 문제 정의입니다.

## 152. 기술 선택은 왜 "좋은 기술"이 아니라 "맞는 기술"의 문제인가

프로젝트 면접에서 기술 선택 질문은 정말 자주 나온다.
여기서 가장 약한 답변은 이런 식이다.

- 많이 써서 익숙해서
- 커뮤니티에서 많이 써서
- 최신이라서

이런 답변은 이유가 너무 약하다.

### 핵심 개념

기술 선택은 보통 아래를 기준으로 설명하는 게 강하다.

- **문제와 가장 잘 맞는가** — 이 기술이 해결하려는 문제를 직접적으로 다루는가
- **팀/프로젝트 규모에 적절한가** — 과하지도, 부족하지도 않은가
- **운영과 유지보수 비용을 감당 가능한가** — 장기적으로 관리할 수 있는가
- **사용자 경험에 실제로 도움이 되는가** — 기술적 우아함이 아닌 실질적 개선인가

즉 "좋은 기술"보다
**현재 맥락에 맞는 기술**이 핵심이다.

### 실무 연결: Tool-Manager의 기술 선택

예를 들어 Tool-Manager에서 Zustand와 Zod를 말할 때도
"유명해서 썼다"가 아니라 이렇게 가야 한다.

- 편집 중 실시간 상태는 전역적으로 관리할 필요가 있었다 → Zustand
- 저장/전송 경계에서 데이터 정합성 검증이 필요했다 → Zod
- 즉 편집 경험과 데이터 계약 검증의 역할을 분리했다

이렇게 역할 분리 기준이 드러나야 한다.

```typescript
// Zustand: 편집 중 실시간 상태 관리 (전역적이고 빈번하게 변경)
const useToolStore = create<ToolState>((set) => ({
  tools: [],
  selectedId: null,
  updateTool: (id, patch) =>
    set((state) => ({
      tools: state.tools.map((t) =>
        t.id === id ? { ...t, ...patch } : t
      ),
    })),
}));

// Zod: 저장/전송 경계에서 데이터 정합성 검증
const toolSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1),
  category: z.enum(["brush", "pen", "eraser"]),
  order: z.number().int().nonnegative(),
});

// 저장 시점에 검증 — 편집 중에는 자유롭게, 저장할 때만 엄격하게
function saveTool(tool: unknown) {
  const result = toolSchema.safeParse(tool);
  if (!result.success) {
    // 검증 실패 시 사용자에게 어떤 필드가 문제인지 안내
    return { ok: false, errors: result.error.flatten() };
  }
  return api.save(result.data);
}
```

이 코드에서 핵심은 **편집 중 자유도**(Zustand)와 **저장 시점 엄격성**(Zod)의 역할 분리다. 면접에서 이런 식으로 설명하면 기술 선택이 역할 기반 의사결정이었음이 드러난다.

### 자주 하는 오해

- 최신 기술을 썼다고 하면 좋아 보인다 → 맥락이 없으면 약하다
- 기술 비교를 많이 하면 전문적으로 보인다 → 판단 기준이 더 중요하다
- 하나의 기술 장점만 말하면 충분하다 → 왜 이 프로젝트에 맞았는지까지 필요하다

> 이제 기술 선택의 기준을 봤다. 그다음엔 네 포트폴리오의 핵심 강점인 **운영 제약과 현장 맥락을 어떻게 말할지**로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
기술 선택은 "좋은 기술"이 아니라, 현재 문제와 운영 조건에 가장 맞는 기술을 고르는 문제입니다.

**30초 답변**
기술 선택은 "좋은 기술"을 고르는 문제가 아니라, 현재 문제와 팀, 운영 조건에 가장 맞는 기술을 고르는 문제입니다. 예를 들어 Tool-Manager에서 Zustand는 편집 중 실시간 상태를, Zod는 저장 경계의 데이터 검증을 담당하게 했는데, 이건 유명해서가 아니라 편집 자유도와 저장 엄격성이라는 역할을 분리하기 위한 선택이었습니다.

## 153. 운영 제약과 현장 맥락은 왜 프로젝트 답변을 강하게 만드나

이건 네 포트폴리오에서 가장 강한 부분 중 하나다.
기술만 보는 사람이 아니라, 기술이 놓이는 현실 조건을 같이 보는 사람처럼 보이게 만든다.

### 핵심 개념

운영 제약이란 보통 이런 것들이다.

- 네트워크 불안정
- 세션 만료
- 오래된 기기 환경
- 관리자 업무의 반복성
- 사용자 실수 가능성
- 배포 빈도와 장애 비용
- 데이터 손실 위험

운영 제약은 "이상적인 기술 설계"와 "현실에서 돌아가는 설계" 사이의 간격을 만드는 요소다. 이 간격을 인식하고 의사결정에 반영했다는 것 자체가 실무 감각의 증거가 된다.

### 왜 이게 중요한가

이걸 말할 수 있어야
기술 선택이 단순 취향이 아니라 **운영 판단**으로 보인다.

예:

- 양치킹의 fallback / retry / 임시 저장
- CSV Bulk Upload의 검증-오류 처리-재시도
- Art-bonbon의 MPEG-DASH 기반 재생 경험 설계
- 아트봉봉 스쿨의 VOC 구조화

이런 건 전부 "기능 구현"보다
현장 제약을 먼저 읽은 결과라고 설명할 수 있다.

### 자주 하는 오해

- 운영 얘기는 인프라 담당자만 중요하게 본다 → 제품 개발자에게도 핵심이다
- 사용자는 기술 구조를 모르니 설명할 필요 없다 → 사용자 흐름 단절은 기술 문제와 직결된다
- 운영 제약은 부정적인 얘기라 약점처럼 보인다 → 오히려 실전 감각으로 보일 수 있다

> 이제 운영 맥락을 읽는 감각을 봤다. 그다음엔 그 판단이 실제로 어떤 구조로 이어졌는지, 즉 **복구 가능 UX와 장애 대응**으로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
운영 제약을 설명할 수 있으면 기술 선택이 취향이 아니라 운영 판단으로 보입니다.

**30초 답변**
운영 제약과 현장 맥락을 설명할 수 있으면 기술 선택이 단순 취향이 아니라 운영 판단으로 보입니다. 네트워크 불안정, 세션 만료, 관리자 반복 업무 같은 현실 조건을 먼저 읽고 구조를 설계했다는 점이 프로젝트 답변을 훨씬 강하게 만듭니다. 이건 "이상적 설계"와 "현실에서 돌아가는 설계" 사이의 간격을 인식한 결과입니다.

## 154. 장애 대응과 fallback UX는 왜 프로젝트 경험을 깊게 보이게 하나

운영 경험이 없는 답변은 보통 happy path 위주로 끝난다.
반면 실제 서비스를 다뤄본 사람은
"안 되는 상황에서 어떻게 복구할지"를 말할 수 있다.

### 핵심 개념

좋은 장애 대응 답변은 보통 두 층을 함께 가진다.

- **시스템 대응** — retry, timeout, circuit breaker, 롤백 등 기술적 복구 경로
- **사용자 경험 대응** — 사용자가 현재 무슨 일이 일어나고 있는지 알 수 있게 하고, 데이터를 잃지 않게 보호하는 것

즉 서버나 프로세스만 복구하는 게 아니라,
사용자가 흐름을 잃지 않게 만드는 것도 포함된다.

### 실무 연결

양치킹에서 이걸 설명할 때 좋은 구조는 이렇다.

- 네트워크 실패를 단순 online/offline으로 보지 않았다
- 실제 요청 실패와 지연을 기준으로 봤다
- retry와 fallback UI를 뒀다
- 중요한 입력은 임시 저장이나 낙관적 흐름을 고려했다
- 에러 메시지보다 복구 경로를 먼저 보여주었다

이렇게 말하면
운영과 UX를 함께 본 사람처럼 보인다.

### 코드로 보는 장애 대응 패턴

```typescript
// 1단계: retry 로직 — 일시적 네트워크 실패에 대응
async function fetchWithRetry(
  fn: () => Promise<Response>,
  maxRetries = 3,
  delay = 1000
) {
  let lastError: unknown;
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const res = await fn();
      if (res.ok) return res;
      if (attempt === maxRetries - 1) return res; // 마지막 시도면 non-ok 응답이라도 반환
    } catch (e) {
      lastError = e;
      if (attempt === maxRetries - 1) throw e;
    }
    await new Promise((r) => setTimeout(r, delay * (attempt + 1)));
  }
  throw lastError; // 타입 안전을 위한 fallback
}

// 2단계: 임시 저장 — retry로도 안 되면 데이터 보호
function saveToLocalDraft(key: string, data: unknown) {
  try {
    localStorage.setItem(
      `draft:${key}`,
      JSON.stringify({ data, savedAt: Date.now() })
    );
  } catch {
    // localStorage 용량 초과 시 가장 오래된 draft 제거 후 재시도
  }
}

// 3단계: fallback UI — 사용자에게 복구 경로 안내
function SubmitFallback({ draftKey }: { draftKey: string }) {
  return (
    <div role="alert">
      <p>저장 중 문제가 발생했습니다.</p>
      <p>입력 내용은 임시 저장되었습니다.</p>
      <button onClick={() => retryFromDraft(draftKey)}>
        다시 시도
      </button>
    </div>
  );
}
```

핵심은 retry → 임시 저장 → 사용자 안내라는 **3단계 방어**다. 에러 메시지 하나 띄우는 것과, 데이터를 보호하고 복구 경로를 제공하는 것은 완전히 다른 수준의 대응이다.

### 자주 하는 오해

- 장애 대응은 백엔드 문제다 → 프론트도 흐름 보호를 설계해야 한다
- 에러 메시지를 띄우면 충분하다 → 복구 가능성이 더 중요하다
- 장애는 드문 예외라 면접에서 굳이 말 안 해도 된다 → 오히려 깊이를 보여준다

> 이제 장애 대응 경험의 강점을 봤다. 그다음엔 프로젝트 결과를 단단하게 만드는 **성과와 지표**로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
장애 대응은 서버 복구만이 아니라, 사용자 흐름 보호까지 포함해야 깊은 경험으로 보입니다.

**30초 답변**
장애 대응 경험이 강한 이유는 정상 동작뿐 아니라 실패 상황에서의 복구까지 설명할 수 있기 때문입니다. 좋은 답변은 retry, 임시 저장, fallback UI처럼 시스템 복구와 사용자 흐름 보호를 함께 설계한 점을 보여줍니다. 에러 메시지 하나 띄우는 것과 데이터를 보호하고 복구 경로를 안내하는 것은 완전히 다른 수준의 대응입니다.

## 155. 성과는 왜 숫자 자체보다 연결 논리가 중요하나

프로젝트 면접에서 숫자를 말하면 분명 도움이 된다.
하지만 숫자만 던지면 의미가 약할 수 있다.

### 핵심 개념

좋은 성과 답변은 보통 아래 구조를 가진다.

1. **어떤 지표를 봤는가** — 왜 이 지표를 선택했는가
2. **그 지표가 왜 중요한가** — 비즈니스나 사용자 경험과 어떻게 연결되는가
3. **무엇이 바뀌었는가** — before/after
4. **그 변화가 내 작업과 어떻게 연결되는가** — 어떤 구조 변경이 이 결과를 만들었는가

즉 숫자는 **맥락 안에서 해석**돼야 한다.

### 실무 연결

예:

- 아트봉봉 스쿨에서 VOC 30건 → 6건
- 양치킹 일 평균 1,000건 이상 기록 생성
- 운영 문의 감소
- 반복 오류 감소
- 배포 안정성 증가

이런 수치는 그냥 나열하는 게 아니라,
"어떤 흐름의 구조를 바꿨더니 왜 이런 결과가 나왔는가"까지 같이 말해야 한다.

### 좋은 성과 연결 vs 약한 성과 연결

```
// 약한 성과 답변
"VOC가 30건에서 6건으로 줄었습니다."

// 좋은 성과 답변
"관리자 도구에서 가장 반복되는 VOC를 분석해보니,
정보 탐색 구조가 불명확해서 같은 질문이 반복되고 있었습니다.
정보 구조를 재설계하고 자주 묻는 흐름에 대한 안내를 추가했더니,
월 VOC가 30건에서 6건으로 줄었습니다.
즉 기능 추가가 아니라 정보 구조 개선이 핵심이었습니다."
```

### 자주 하는 오해

- 큰 숫자만 있으면 된다 → 숫자가 왜 중요한지 설명해야 한다
- 공식 KPI가 없으면 성과를 말할 수 없다 → 간접 지표도 충분히 의미 있다. 운영 문의 감소, 반복 오류 감소, 배포 실패율 감소 등도 입증 가능한 성과다
- 정량이 없으면 약하다 → 정성 + 정량 연결이 중요하다

> 이제 성과를 말하는 법을 봤다. 그다음엔 면접에서 신뢰를 크게 높이는 질문인 **실패와 되돌림**으로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
성과는 숫자 자체보다, 그 숫자가 내 작업과 어떻게 연결되는지를 설명하는 것이 핵심입니다.

**30초 답변**
성과는 숫자 자체보다, 어떤 지표를 왜 중요하게 봤고 그 변화가 내 작업과 어떻게 연결되는지를 설명하는 것이 중요합니다. 예를 들어 "VOC가 30건에서 6건으로 줄었다"만 말하면 약하지만, "정보 구조를 재설계해서 반복 문의가 구조적으로 줄었다"까지 연결하면 훨씬 설득력이 생깁니다.

## 156. 실패 경험은 왜 숨기기보다 구조적으로 말하는 편이 강한가

실패 질문이 나오면 많은 사람이 방어적으로 변한다.
하지만 실제로는 이 질문이 오히려 신뢰를 만들 기회가 되기도 한다.

### 핵심 개념

좋은 실패 답변은 아래 구조를 가진다.

1. **어떤 판단을 했는가** — 당시의 맥락과 선택
2. **왜 그 판단이 문제였는가** — 어떤 가정이 틀렸거나 무엇을 놓쳤는가
3. **어떤 문제가 발생했는가** — 구체적 영향
4. **어떻게 복구했는가** — 즉각 대응
5. **이후 무엇을 바꿨는가** — 프로세스나 기준의 변화

즉 실패를 자랑처럼 포장하지 말고,
**판단 기준의 성장**으로 연결해야 한다.

주의할 점은 실패 답변에서 "치명적이지만 복구 불가능했던 실패"를 꺼내는 것보다, **복구 과정과 학습이 명확한 실패**를 선택하는 것이 전략적으로 더 좋다는 것이다.

### 실무 연결

예를 들어:

- 기존 사용자 상태를 충분히 고려하지 않고 구조를 바꿨다
- 일부 데이터 흐름이 끊겼다
- 바로 롤백했다
- 호환 경로와 체크리스트를 다시 설계했다
- 이후 배포 전 확인 기준이 달라졌다

이렇게 가면
실패를 숨기지 않으면서도 회복 능력을 보여줄 수 있다.

### 자주 하는 오해

- 실패한 적이 없다고 말하는 게 안전하다 → 오히려 얕아 보일 수 있다. 실무 경험이 있는 사람은 반드시 실패를 겪는다
- 실패를 말하면 감점될 수 있다 → 복구와 학습이 더 중요하다
- 실패를 자랑처럼 미화해야 한다 → 진정성이 떨어진다. 사실 기반으로 담담하게 말하는 게 낫다

> 이제 실패를 어떻게 말해야 하는지 봤다. 그다음엔 실패와 닿아 있는 주제인 **다시 만든다면 무엇을 바꿀지**로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
실패 경험은 숨기기보다, 복구 과정과 이후 기준 변화를 구조적으로 설명하는 편이 강합니다.

**30초 답변**
실패 경험은 숨기기보다, 어떤 판단이 왜 문제였고 어떻게 복구했으며 이후 기준이 어떻게 바뀌었는지를 구조적으로 설명하는 편이 더 강합니다. 면접관은 무실수보다도 실패를 다루는 방식과 회복력을 보고 싶어하는 경우가 많습니다. 복구 과정과 학습이 명확한 사례를 골라 판단 기준의 성장으로 연결하는 것이 좋습니다.

## 157. "지금 다시 만든다면 무엇을 바꾸겠나"는 왜 회고 능력을 보여주나

이 질문은 단순 후회 질문이 아니다.
현재의 시각으로 예전 판단을 다시 평가할 수 있는지 보는 질문이다.

### 핵심 개념

좋은 답변은 아래를 포함한다.

- 당시 어떤 기준으로 선택했는가
- 지금 보면 어떤 한계가 있었는가
- 지금이라면 무엇을 다르게 선택할 것인가
- 그 이유는 무엇인가

즉 "옛날의 나"를 무시하는 게 아니라
**그때의 맥락과 지금의 기준 차이**를 설명해야 한다.

이 질문에 잘 답하려면 두 가지를 동시에 보여줘야 한다: 당시 판단이 무근거가 아니었다는 점, 그리고 지금은 더 나은 기준을 가지고 있다는 점.

### 실무 연결

예를 들어:

- 프로젝트 규모에 비해 상태관리 도구가 과했다
- 공용 패키지 공유 범위를 너무 넓게 잡았다
- 초기 구조에서 운영 로그를 더 일찍 심었어야 했다
- URL 상태와 로컬 상태의 경계를 더 빨리 분리했어야 했다

```typescript
// 당시 선택: 모든 상태를 전역 store에 넣음
const useStore = create((set) => ({
  tools: [],
  selectedTool: null,
  filterQuery: "",      // ← URL에 있어야 할 상태
  sortOrder: "recent",  // ← URL에 있어야 할 상태
  modalOpen: false,     // ← 로컬 상태로 충분
}));

// 지금이라면: URL 상태와 로컬 상태를 분리
// URL 상태 — 공유 가능하고 브라우저 뒤로가기에 반응해야 하는 것
const searchParams = useSearchParams();
const filterQuery = searchParams.get("q") ?? "";
const sortOrder = searchParams.get("sort") ?? "recent";

// 전역 상태 — 여러 컴포넌트가 동시에 읽고 써야 하는 것
const useToolStore = create((set) => ({
  tools: [],
  selectedTool: null,
}));

// 로컬 상태 — 해당 컴포넌트 안에서만 의미 있는 것
const [modalOpen, setModalOpen] = useState(false);
```

이런 식으로 말하면
회고 능력과 구조 감각이 함께 드러난다.

### 자주 하는 오해

- 예전 결정을 부정하면 안 된다 → 당시 맥락을 설명하면 괜찮다. "당시에는 이 기준이 합리적이었는데, 운영하면서 이런 한계가 보였다"가 좋은 표현이다
- "다시 해도 똑같이 한다"가 자신감 있어 보인다 → 오히려 성장성이 약해 보일 수 있다
- 후회 포인트는 작을수록 안전하다 → 너무 무난하면 약하다

> 이제 회고 능력까지 봤다. 그다음엔 협업 프로젝트에서 자주 나오는 **의견 충돌과 정렬 방식**으로 넘어갈 수 있다.

### 면접 압축 답변

**10초 답변**
이 질문은 후회가 아니라, 당시 판단 기준과 지금 기준의 차이를 설명할 수 있는지 보는 질문입니다.

**30초 답변**
"지금 다시 만든다면 무엇을 바꾸겠나"는 후회보다 회고 능력을 보는 질문입니다. 당시 판단이 무근거는 아니었지만 지금은 더 나은 기준을 가지고 있다는 점을 동시에 보여줘야 합니다. 예를 들어 "URL 상태와 로컬 상태를 처음부터 분리했을 것"처럼 구체적으로 말하면, 기술 선택을 맥락 기반 의사결정으로 본다는 점이 드러납니다.

## 158. 협업에서 의견 충돌은 왜 "누가 맞았나"보다 "어떻게 정렬했나"가 중요하나

협업 질문에서 가장 피해야 할 건
상대를 틀린 사람처럼 묘사하는 것이다.

### 핵심 개념

좋은 협업 답변은 보통 아래 구조를 가진다.

1. **어떤 맥락에서 입장 차이가 있었는가** — 프로젝트 상황
2. **상대는 무엇을 우려했는가** — 상대 관점을 공정하게 서술
3. **나는 무엇을 우려했는가** — 내 관점과 근거
4. **어떤 근거로 정렬했는가** — 데이터, 사용자 피드백, 기술적 분석 등
5. **어떤 합의안으로 갔는가** — 최종 결정과 그 결과

즉 핵심은 승부가 아니라 **정렬 방식**이다.

### 실무 연결

예를 들어:

- 기획은 기존 흐름 유지와 일정 안정성을 우려
- 개발은 현재 구조의 유지보수 비용 증가를 우려
- 그래서 구조를 다 뒤엎는 대신
- 핵심 흐름부터 먼저 재설계하고 나머지는 점진 개선

이렇게 말하면
실무적인 협업 감각이 드러난다.

### 자주 하는 오해

- 충돌이 없었다고 말하는 게 좋다 → 오히려 협업 깊이가 약해 보일 수 있다
- 내가 맞았고 설득했다로 끝내면 된다 → 팀워크 역량이 약해 보일 수 있다
- 협업은 성향 문제다 → 근거와 우선순위 정렬이 더 중요하다

> 이제 협업 정렬 감각을 봤다. 그다음엔 새 회사 적응 질문으로 자주 연결되는 **입사 후 90일 기여 계획**으로 넘어가야 한다.

### 면접 압축 답변

**10초 답변**
협업 충돌에서 중요한 건 누가 맞았는지가 아니라, 서로의 우려를 어떻게 정렬했는지입니다.

**30초 답변**
협업에서 의견 충돌은 누가 맞았는지보다, 서로의 우려를 어떻게 구체적인 근거로 정리하고 어떤 합의점으로 정렬했는지가 더 중요합니다. 좋은 답변은 승리 서사가 아니라, 상대 관점을 공정하게 서술하고 데이터나 사용자 피드백 같은 근거로 정렬한 과정이 드러나야 합니다.

## 159. 입사 후 90일 계획은 왜 겸손함과 자신감을 같이 보여줘야 하나

이 질문은 "새 조직에서 얼마나 빨리 적응하고 기여할 수 있나"를 보는 질문이다.
여기서 너무 공격적으로 말하면 기존 맥락을 무시하는 사람처럼 보이고,
너무 조심스럽게 말하면 기여감이 약해 보일 수 있다.

### 핵심 개념

좋은 답변은 보통 3단계로 나뉜다.

#### 1단계: 파악 (1~30일)

- 코드베이스 구조와 컨벤션
- 배포 흐름과 CI/CD 파이프라인
- 팀 협업 방식 (코드 리뷰, 스프린트 등)
- 현재 자주 발생하는 운영 이슈

#### 2단계: 빠른 기여 (30~60일)

- 작은 버그 수정
- 리뷰 참여
- 작은 개선 (DX 개선, 테스트 추가 등)
- 팀이 체감하는 즉시 가치 제공

#### 3단계: 구조적 개선 제안 (60~90일)

- 사용자 흐름 단절 지점 개선
- 운영 비용 큰 영역 최적화
- 상태 구조나 성능, 배포 개선 포인트 제안

### 왜 이게 중요한가

이 구조는:

- 기존 맥락을 존중하면서
- 빠른 기여 의지도 보여주고
- 장기적으로 구조 개선까지 생각한다는 점을 드러낸다

### 실무 연결

네 포트폴리오 흐름에 맞게 말하면 좋다.

- 처음엔 현재 구조와 운영 이슈를 이해하겠다
- 그다음엔 작은 개선으로 신뢰를 쌓겠다
- 이후엔 사용자 흐름과 운영 비용이 큰 영역에서 구조적으로 기여하겠다

### 자주 하는 오해

- 입사하자마자 큰 구조를 바꾸겠다고 말한다 → 위험해 보일 수 있다
- 배우기만 하겠다고 말한다 → 기여 의지가 약해 보인다
- 90일 계획은 형식적 질문이다 → 조직 적응력과 문제 파악 능력을 본다

> 이제 프로젝트 경험을 새 회사 가치로 연결하는 방식까지 봤다. 그다음엔 이 챕터 전체를 한 번 정리해야 한다.

### 면접 압축 답변

**10초 답변**
90일 계획은 파악 → 빠른 기여 → 구조적 개선 제안 순서로, 겸손함과 기여 의지를 함께 보여줘야 합니다.

**30초 답변**
입사 후 90일 계획은 기존 코드와 팀 맥락을 먼저 이해하고, 작은 개선으로 신뢰를 쌓은 뒤, 이후에 구조적인 개선 포인트를 제안하는 흐름이 가장 안정적입니다. 이렇게 하면 기존 팀을 존중하면서도 적극적으로 기여하겠다는 의지를 함께 보여줄 수 있습니다.

## 160. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

| 문제 정의 | 기능 설명이 아니라 "누가 왜 막혔는가" |
| 기술 선택 | "좋은 기술"이 아니라 "맞는 기술" |
| 운영 제약 | 현장 조건이 기술 판단의 근거가 됨 |
| 장애 대응 | 시스템 복구 + 사용자 흐름 보호 |
| 성과 | 숫자보다 연결 논리 |
| 실패 | 판단 기준의 성장으로 연결 |
| 회고 | 당시 맥락과 지금 기준의 차이 |
| 협업 충돌 | 승패가 아니라 정렬 방식 |
| 90일 계획 | 파악 → 기여 → 구조 개선 |

즉,
이번 챕터는 단순 프로젝트 소개법이 아니라,
**"내 경험을 어떤 문제 해결 철학과 운영 감각으로 묶어 면접에서 설득력 있게 전달할 것인가"**를 배우는 챕터다.

---

# 이 챕터를 읽은 뒤 자연스럽게 이어질 다음 질문들

1. 지금까지 만든 전체 플로우 문서에서 빠진 문항은 무엇인가?
2. 각 챕터를 하루 단위 면접 준비 루틴으로 어떻게 배치해야 하나?
3. 10초 / 30초 / 1분 답변 버전은 어떻게 따로 압축해야 하나?
4. 꼬리질문 대응용 서브 노트를 어떻게 만들면 좋나?
5. 실제 면접 리허설용 랜덤 질문 세트는 어떻게 구성하면 좋나?

---

# 사용법

이 챕터는 아래 순서로 보면 좋다.

1. 프로젝트 면접의 핵심을 기술 나열이 아니라 문제 정의와 판단 기준으로 다시 잡는다.
2. 대표 프로젝트와 자기소개 연결 구조를 먼저 만든다.
3. 프로젝트별 문제 정의, 기술 선택, 운영 제약, 장애 대응, 성과를 같은 골격으로 정리한다.
4. 실패, 회고, 협업 충돌, 90일 계획까지 붙여서 사람 자체의 판단 기준을 드러낸다.
5. 마지막에 전체 프로젝트 답변을 하나의 스토리로 묶는다.

이렇게 보면 프로젝트 면접이 단순 경험 나열이 아니라, 내가 어떤 기준으로 문제를 보고 구조를 설계하고 운영 가능한 결과로 연결하는 사람인지를 보여주는 자리라는 감각이 더 선명해진다.
