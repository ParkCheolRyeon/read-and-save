# 챕터 9. 테스트 전략에서 CI 신뢰성까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **테스트는 단순히 버그를 찾는 활동이 아니라, 변경해도 되는 자신감을 만들고 그 자신감을 CI 파이프라인까지 연결하는 구조라는 관점으로 이해해야 한다.**

---

## 115. 테스트는 왜 단순 버그 탐지보다 “변경 자신감”과 더 강하게 연결되나

테스트를 처음 배우면 흔히 이렇게 생각한다.

> **테스트는 버그를 잡기 위한 것**

물론 맞는 말이다.  
하지만 실무에서는 이 설명만으로는 부족하다.

### 핵심 개념

테스트의 더 본질적인 역할은  
**코드를 바꾸어도 괜찮다는 자신감을 팀에 제공하는 것**이다.

즉 테스트는:

- 현재 동작이 맞는지 확인하고
- 이후 리팩토링이나 기능 추가 때
- 기존 동작이 깨졌는지 빠르게 알게 해주는 장치다

### 왜 이게 중요한가

실무에서 진짜 무서운 건  
“지금 당장 버그가 있나?”보다  
“이걸 수정하면 어디가 같이 깨질지 모른다”는 상태다.

테스트가 없으면:

- 코드 수정이 무서워지고
- 안전한 리팩토링이 어려워지고
- 배포 직전 수동 확인이 과도해지고
- 팀 전체 속도가 느려질 수 있다

### 자주 하는 오해

- 테스트는 QA가 하는 일이다 → 개발 단계의 안전장치이기도 하다
- 테스트는 버그가 많을 때만 필요하다 → 오히려 변경이 많은 코드일수록 중요하다
- 테스트를 많이 쓰면 개발이 무조건 느려진다 → 장기적으로는 변경 비용을 낮추는 경우가 많다

### 실무 연결

테스트가 특히 강한 순간은 이런 때다.

- 큰 리팩토링
- 공용 컴포넌트 교체
- 상태 구조 변경
- API 응답 처리 로직 수정
- 디자인 시스템 마이그레이션

즉 테스트는 지금보다  
**다음 변경을 위한 안전망**에 더 가깝다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트의 존재 이유를 봤다.  
그다음엔 자연스럽게 이런 질문이 나온다.

**그렇다면 테스트는 어떤 층으로 나뉘고, 각 층은 무엇을 확인하는가?**

그래서 다음은 unit / integration / E2E로 넘어간다.

### 면접 압축 답변

테스트의 본질적인 가치는 단순 버그 탐지보다, 코드를 바꿔도 괜찮다는 변경 자신감을 팀에 제공하는 데 있습니다. 특히 리팩토링과 공용 구조 변경이 잦은 프론트엔드에서는 테스트가 미래 변경 비용을 낮추는 안전망 역할을 합니다.

## 116. unit, integration, E2E는 무엇을 각각 검증하나

테스트를 배울 때 가장 먼저 구조적으로 잡아야 하는 게 테스트 계층이다.  
많은 사람이 세 가지 이름은 외우지만, 실제로 **어떤 실패를 어디서 잡아야 하는지**가 흐릿하다.

### 핵심 개념

#### Unit Test

가장 작은 단위의 로직이나 함수, 컴포넌트 조각이 예상대로 동작하는지 본다.

#### Integration Test

여러 조각이 함께 연결되었을 때 기대한 흐름으로 동작하는지 본다.

#### E2E Test

사용자 시나리오 전체가 실제 브라우저 환경에서 끝까지 동작하는지 본다.

### 왜 이게 중요한가

이 구분이 중요해야:

- 너무 많은 걸 unit에 몰아넣지 않고
- 너무 사소한 걸 E2E로 밀어넣지 않으며
- 실패 원인을 더 빠르게 좁힐 수 있다

### 자주 하는 오해

- unit test가 많으면 E2E는 필요 없다 → 틀림
- E2E만 있으면 다 커버된다 → 느리고 디버깅이 어렵다
- integration은 unit과 E2E 사이의 애매한 단계다 → 실제론 매우 중요한 계층이다

### 실무 연결

좋은 기준은 이렇다.

- 순수 로직 검증 → unit
- 컴포넌트와 상태, 이벤트 흐름 검증 → integration
- 로그인, 결제, 업로드, 검색 같은 사용자 핵심 흐름 → E2E

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트 계층은 봤다.  
그다음엔 각각을 조금 더 구체적으로 이해하기 위해 **unit test가 잘 맞는 대상**으로 들어가야 한다.

### 면접 압축 답변

unit, integration, E2E는 테스트 범위와 목적이 다릅니다. unit은 작은 로직 단위의 정확성을, integration은 여러 조각의 연결 흐름을, E2E는 실제 사용자 시나리오 전체를 검증합니다. 그래서 하나가 다른 하나를 완전히 대체하는 구조가 아니라 서로 다른 실패를 잡는 계층으로 보는 게 맞습니다.

## 117. unit test는 어디에 가장 잘 맞고, 어디엔 잘 안 맞나

unit test는 가장 빠르고 가볍게 돌릴 수 있는 경우가 많아서  
테스트의 첫 층으로 매우 유용하다.  
하지만 모든 걸 unit test로 해결하려고 하면 오히려 이상해진다.

### 핵심 개념

unit test는  
**작고 독립적인 단위가 정해진 입력에서 정해진 출력을 내는가**를 보기에 좋다.

대표적으로:

- 유틸 함수
- 파서
- 포맷터
- 상태 전환 함수
- 순수 계산 로직
- 간단한 커스텀 훅의 순수 분기

### 왜 이게 중요한가

이런 로직은:

- 빠르게 실행되고
- 실패 원인이 명확하고
- 리팩토링 시 회귀를 빠르게 잡아낼 수 있다

### 자주 하는 오해

- React 앱이면 unit test는 의미 없다 → 순수 로직은 여전히 많다
- 단위가 작을수록 항상 좋다 → 지나치게 쪼개면 의도가 흐려질 수 있다
- UI도 다 unit으로 충분히 테스트 가능하다 → 상호작용 흐름은 integration이 더 자연스러울 수 있다

### 실무 연결

프론트엔드에서 unit test가 특히 강한 부분:

- 날짜/숫자 포맷 함수
- 검색어 정규화
- 테이블 정렬 로직
- permission 계산
- 폼 검증 함수
- API 응답 normalize 함수

### 여기서 다음 개념으로 어떻게 넘어가나

이제 작은 단위 테스트가 잘 맞는 대상을 봤다.  
그다음엔 실제 컴포넌트와 사용자 이벤트가 엮일 때 중요한 **integration test**로 넘어가야 한다.

### 면접 압축 답변

unit test는 유틸 함수나 상태 전환 함수처럼 작고 독립적인 로직을 검증하는 데 가장 잘 맞습니다. 빠르고 실패 원인이 명확하다는 장점이 있지만, 실제 UI 상호작용 흐름까지 모두 unit test로 해결하려 하면 오히려 테스트가 인위적이고 읽기 어려워질 수 있습니다.

## 118. integration test는 왜 프론트엔드에서 특히 중요하나

프론트엔드에서 버그는 단일 함수 자체보다  
**여러 조각이 연결되는 순간** 더 자주 생긴다.

예를 들어:

- 버튼 클릭
- 상태 업데이트
- 조건부 렌더링
- API 상태 반영
- 에러 메시지 표시
  같은 흐름은 하나의 함수만 보면 잘 안 드러난다.

### 핵심 개념

integration test는  
여러 단위가 함께 연결되었을 때 실제 사용자 관점의 흐름이 맞는지 보는 테스트다.

즉:

- 컴포넌트
- state
- 이벤트
- 비동기 처리
- DOM 반영
  이 연결된 결과를 본다.

### 왜 이게 중요한가

프론트엔드에선 아래 같은 버그가 많다.

- 클릭은 되는데 로딩 UI가 안 뜸
- 에러 메시지가 조건에 맞게 안 나옴
- 입력값은 바뀌는데 제출 버튼 활성화가 안 됨
- 비동기 완료 후 화면이 기대대로 안 갱신됨

이런 건 순수 함수 테스트보다 integration test가 더 잘 잡는다.

### 자주 하는 오해

- integration test는 느리고 무거워서 최소화해야 한다 → 프론트엔드에선 오히려 핵심 계층인 경우가 많다
- DOM이 나오면 다 E2E다 → 아니다
- integration test는 단위 테스트보다 덜 정확하다 → 실제 사용자 흐름엔 더 적합할 수 있다

### 실무 연결

대표적인 대상:

- 로그인 폼 제출 흐름
- 필터 변경 후 리스트 갱신
- 모달 열기/닫기
- 토스트 표시
- optimistic update UI
- 에러 fallback 렌더링

### 여기서 다음 개념으로 어떻게 넘어가나

이제 integration test의 역할을 봤다.  
그다음엔 브라우저 전체에서 실제 사용자 흐름을 검증하는 **E2E test**로 넘어가야 한다.

### 면접 압축 답변

integration test는 여러 단위가 함께 연결된 실제 UI 흐름을 검증하는 테스트입니다. 프론트엔드에선 버튼 클릭, 상태 갱신, 비동기 처리, DOM 반영이 함께 엮이는 버그가 많기 때문에, integration test가 실질적인 안정성을 높이는 핵심 계층이 되는 경우가 많습니다.

## 119. E2E test는 왜 가장 비싸지만 가장 현실적인가

E2E는 보통 가장 무겁고 느린 테스트 계층이다.  
그런데도 중요한 이유는 분명하다.

### 핵심 개념

E2E test는  
실제 사용자처럼 브라우저를 통해 애플리케이션 전체 흐름을 끝까지 검증한다.

즉:

- 라우팅
- 실제 렌더링
- 네트워크
- 폼 입력
- 화면 전환
- 브라우저 동작
  을 더 현실에 가깝게 본다.

### 왜 이게 중요한가

아래 문제는 E2E에서 더 잘 드러난다.

- 라우팅 연결 실패
- 실제 브라우저 이벤트 차이
- hydration 이후 이상 동작
- 인증 흐름 문제
- 중요한 사용자 시나리오 붕괴

### 자주 하는 오해

- E2E 하나면 나머지 테스트는 필요 없다 → 느리고 실패 원인 추적이 어렵다
- E2E는 QA 자동화랑 같은 말이다 → 개발자가 직접 구조적으로 설계해야 효과가 크다
- 모든 페이지를 다 E2E 해야 한다 → 핵심 시나리오 우선이다

### 실무 연결

E2E가 특히 잘 맞는 대표 시나리오:

- 로그인
- 회원가입
- 검색
- 장바구니/결제
- 파일 업로드
- 핵심 관리자 작업
- 세션 만료 후 재로그인 흐름

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트 계층 전체를 봤다.  
그다음엔 이런 질문이 나온다.

**그렇다면 어떤 걸 어떤 층에 배치해야 ROI가 가장 좋은가?**

### 면접 압축 답변

E2E test는 실제 브라우저에서 사용자 시나리오 전체를 검증하기 때문에 가장 현실적인 테스트 계층입니다. 다만 실행 비용이 크고 실패 원인 파악이 어렵기 때문에, 모든 걸 맡기기보다 로그인, 결제, 업로드 같은 핵심 흐름에 집중하는 것이 보통 더 효율적입니다.

## 120. 테스트 ROI는 어떻게 판단해야 하나

실무에서는 테스트를 많이 쓰는 것 자체가 목적이 아니다.  
더 중요한 건:

> **어디에 테스트를 투자해야 가장 큰 안정성을 얻는가**

이다.

### 핵심 개념

테스트 ROI를 볼 때는 보통 아래를 같이 본다.

- 깨졌을 때 피해가 큰가
- 변경이 잦은가
- 수동 확인 비용이 큰가
- 자동화로 반복 이득이 큰가
- 테스트 유지보수 비용이 감당 가능한가

### 왜 이게 중요한가

이 감각이 없으면:

- 사소한 UI 문구 테스트에 시간 과투자
- 거의 안 바뀌는 부분을 과하게 자동화
- 정말 중요한 인증/결제 흐름은 비워두는
  상태가 될 수 있다

### 자주 하는 오해

- 커버리지 높으면 테스트 전략도 좋다 → 틀림
- 많은 테스트가 무조건 좋은 테스트다 → 유지비용도 봐야 한다
- 핵심 로직만 테스트하면 UI는 안 해도 된다 → 실제 UX 흐름은 별도 문제다

### 실무 연결

ROI가 큰 대표 후보:

- 인증
- 업로드
- 결제/저장
- 공용 컴포넌트
- 상태 전환이 복잡한 화면
- 장애가 나면 영향 범위 큰 관리자 기능

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트를 어디에 투자해야 하는지 봤다.  
그다음엔 프론트엔드 테스트에서 항상 나오는 질문인 **mock**으로 넘어가야 한다.

### 면접 압축 답변

테스트 ROI는 단순 개수보다, 깨졌을 때 영향이 큰지, 변경이 잦은지, 수동 검증 비용이 큰지를 기준으로 판단하는 게 좋습니다. 그래서 인증, 업로드, 저장처럼 실패 비용이 큰 흐름에 우선 투자하는 편이 더 실용적입니다.

## 121. mock은 왜 유용하면서도 동시에 위험할 수 있나

mock은 테스트에서 매우 자주 쓴다.  
하지만 mock을 많이 쓴다고 무조건 좋은 테스트가 되는 건 아니다.

### 핵심 개념

mock은 외부 의존성을 실제 구현 대신 제어 가능한 가짜로 대체하는 방식이다.

예:

- API 응답
- 브라우저 API
- 타이머
- 외부 모듈
- 무거운 라이브러리

### 왜 이게 중요한가

mock은 아래 장점을 준다.

- 테스트를 더 빠르고 안정적으로 만든다
- 외부 실패 요인을 줄인다
- 특정 시나리오를 재현하기 쉽다
- 네트워크나 시간 흐름을 통제할 수 있다

하지만 동시에 위험도 있다.

- 실제 구현과 너무 멀어진다
- 내부 구현에 과하게 결합된다
- “테스트는 통과하는데 실제론 깨지는” 상태가 될 수 있다

### 자주 하는 오해

- mock을 많이 쓸수록 테스트가 좋아진다 → 틀림
- mock이 없으면 테스트를 못 한다 → 수준을 조절하면 된다
- 모든 외부 의존성을 다 mock 해야 한다 → 필요에 따라 다르다

### 실무 연결

좋은 기준은 이렇다.

- 진짜로 느리거나 불안정한 외부 의존성만 mock
- 내부 구현보다 사용자 관찰 가능한 결과를 우선 검증
- API 계약은 너무 과도하게 가짜로 왜곡하지 않기

### 여기서 다음 개념으로 어떻게 넘어가나

이제 mock의 장단점을 봤다.  
그다음엔 프론트엔드에서 자주 나오는 **API mocking과 실제 서버 사용의 균형**으로 넘어갈 수 있다.

### 면접 압축 답변

mock은 외부 의존성을 제어 가능한 가짜로 바꿔 테스트를 더 빠르고 안정적으로 만드는 데 유용합니다. 하지만 과도하게 사용하면 실제 구현과 멀어져서 테스트는 통과하지만 실제 앱은 깨지는 상황이 생길 수 있으므로, 필요한 경계에서만 신중하게 써야 합니다.

## 122. API mocking은 언제 좋고, 언제 실제 서버가 더 낫나

프론트엔드 테스트에선 API를 어떻게 다룰지가 매우 중요하다.  
여기서도 무조건 한쪽이 정답은 아니다.

### 핵심 개념

API mocking은 테스트를:

- 더 빠르게
- 더 안정적으로
- 더 재현 가능하게
  만드는 데 유리하다.

반면 실제 서버나 실제 계약 기반 테스트는:

- 실제 응답 형태와 더 가깝고
- 계약 불일치를 더 잘 드러낼 수 있다.

### 왜 이게 중요한가

이 균형이 중요하지 않으면:

- 전부 mock으로 해서 현실과 멀어지거나
- 전부 실제 서버로 해서 테스트가 느리고 불안정해질 수 있다

### 자주 하는 오해

- API는 무조건 mock해야 테스트가 빠르다 → 핵심 계약 검증은 실제에 가까울수록 좋을 수 있다
- 실제 서버만 써야 믿을 수 있다 → 불안정성과 속도 문제가 생길 수 있다
- mocking은 임시 대응이다 → 전략적으로 잘 쓰면 강력하다

### 실무 연결

보통 이렇게 나누면 좋다.

- unit / integration → mocking 적극 활용 가능
- E2E 핵심 시나리오 → 실제에 가까운 환경 검토
- 계약 검증 → 응답 shape와 오류 케이스를 현실적으로 반영

### 여기서 다음 개념으로 어떻게 넘어가나

이제 mock과 실제 환경의 균형을 봤다.  
그다음엔 테스트를 무너뜨리는 대표 문제인 **flaky test**로 넘어가야 한다.

### 면접 압축 답변

API mocking은 테스트를 빠르고 안정적으로 만들지만, 실제 응답 계약과 너무 멀어질 수 있습니다. 그래서 unit과 integration에선 적극 활용하되, 핵심 E2E 시나리오나 계약 검증에선 실제에 가까운 환경을 일부 유지하는 식으로 균형을 잡는 것이 중요합니다.

## 123. flaky test는 왜 팀 신뢰를 무너뜨리나

테스트가 실패하는 것보다 더 나쁜 경우가 있다.  
가끔은 실패하고, 가끔은 통과하는 테스트다.  
이게 flaky test다.

### 핵심 개념

flaky test는 코드가 실제로 바뀌지 않았는데도  
환경, 타이밍, 비결정성 때문에 결과가 들쭉날쭉한 테스트다.

### 왜 이게 중요한가

flaky test가 위험한 이유는  
테스트 시스템 전체에 대한 신뢰를 깎기 때문이다.

사람들이 이렇게 생각하기 시작한다.

- “또 테스트 시스템 문제겠지”
- “재실행하면 통과할 거야”
- “빨간불이어도 무시하자”

이 상태가 되면 자동화 가치가 급격히 떨어진다.

### 자주 하는 원인

- 비동기 타이밍 불안정
- 네트워크 의존
- 환경 초기화 불충분
- 전역 상태 오염
- 랜덤 값
- 시간 의존
- 과도한 `setTimeout` 기반 대기

### 자주 하는 오해

- flaky는 어쩔 수 없다 → 줄일 수 있다
- E2E라서 원래 flaky하다 → 더 신중한 설계가 필요하다
- 한두 개쯤은 괜찮다 → 누적되면 신뢰가 무너진다

### 실무 연결

flaky를 줄이려면:

- 기다림 조건을 명시적으로 바꾸고
- 테스트 독립성을 지키고
- 랜덤/시간/네트워크를 통제하고
- 셀렉터와 상태 초기화를 더 안정적으로 설계해야 한다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트가 왜 흔들리는지 봤다.  
그다음엔 비동기 UI 테스트에서 특히 자주 하는 실수로 넘어가야 한다.

### 면접 압축 답변

flaky test는 실제 코드 변화와 무관하게 통과와 실패가 들쭉날쭉한 테스트입니다. 이런 테스트가 늘어나면 팀이 테스트 결과를 믿지 않게 되어 자동화 전체 가치가 떨어지므로, 비동기 제어와 환경 초기화, 독립성 보장을 더 엄격하게 관리해야 합니다.

## 124. 비동기 UI 테스트는 왜 특히 조심해야 하나

프론트엔드 테스트에서 비동기 UI는 가장 흔한 함정 중 하나다.  
사용자 이벤트 이후에 바로 DOM이 바뀌지 않는 경우가 많기 때문이다.

### 핵심 개념

비동기 UI 테스트에선 아래가 엮인다.

- 이벤트 발생
- state 업데이트
- 비동기 요청
- 렌더 재반영
- loading / success / error 상태 전환

즉 테스트도 단순 즉시 검증이 아니라  
**상태가 바뀌는 과정을 기다리며 검증**해야 한다.

### 왜 이게 중요한가

이걸 모르고 바로 assert 하면:

- 로컬에선 통과
- CI에선 실패
- 재실행하면 또 통과
  같은 flaky가 생기기 쉽다.

### 자주 하는 오해

- 클릭 후 바로 요소가 나와야 한다 → 비동기라면 기다려야 할 수 있다
- `setTimeout`으로 대충 기다리면 된다 → 불안정하다
- 비동기 테스트는 원래 flaky하다 → 기다림 조건을 더 정확히 잡아야 한다

### 실무 연결

좋은 비동기 테스트 감각은 이렇다.

- 시간 자체를 기다리지 말고 **상태 변화 조건을 기다린다**
- “1초 뒤”보다 “이 요소가 나타날 때까지”
- “setTimeout”보다 “이 로딩이 사라질 때까지”

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트를 안정적으로 작성하는 기본 감각을 봤다.  
그다음엔 실제 프론트엔드 특수 UI인 **DnD, 업로드, 세션 만료** 같은 시나리오 테스트로 넘어갈 수 있다.

### 면접 압축 답변

비동기 UI 테스트는 이벤트 이후 DOM이 바로 바뀌지 않을 수 있기 때문에, 시간 자체를 대충 기다리는 대신 로딩 해제나 특정 요소 등장처럼 상태 변화 조건을 기준으로 검증해야 안정적입니다. 그렇지 않으면 flaky test가 되기 쉽습니다.

## 125. 복잡한 프론트엔드 시나리오는 무엇을 특별히 테스트해야 하나

프론트엔드엔 단순 폼 외에도  
테스트하기 까다로운 시나리오가 많다.

### 대표 시나리오

- DnD
- 파일 업로드
- 차트 상호작용
- 세션 만료
- 네트워크 재시도
- optimistic update
- 모달/포커스 트랩
- 무한 스크롤

### 왜 이게 중요한가

이런 시나리오는 단순 함수 정확성보다  
**사용자 흐름과 상태 전이, 예외 상황**이 중요하다.

예를 들어:

- 업로드 성공만이 아니라 실패와 재시도
- 세션 만료 후 재로그인 유도
- DnD 결과 저장 여부
- optimistic update 후 롤백

같은 걸 봐야 한다.

### 자주 하는 오해

- 복잡하면 수동 테스트만 하면 된다 → 반복 회귀를 놓치기 쉽다
- 복잡한 UI는 테스트하기 어렵다 → 그래서 더 중요한 경우가 많다
- happy path만 자동화하면 충분하다 → 예외 흐름이 더 치명적일 수 있다

### 실무 연결

좋은 테스트는 보통:

- 정상 경로
- 실패 경로
- 취소/되돌리기
- 중복 클릭/중복 요청
  까지 같이 본다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트 대상 시나리오를 봤다.  
그다음엔 이런 테스트를 로컬에서만 돌릴 게 아니라 **CI에 어떻게 연결할지**로 넘어가야 한다.

### 면접 압축 답변

복잡한 프론트엔드 시나리오는 happy path보다 상태 전이와 예외 흐름을 함께 검증해야 합니다. 예를 들어 업로드는 성공뿐 아니라 실패와 재시도, 세션 만료는 만료 자체보다 이후 복구 UX까지 테스트해야 실제 안정성이 높아집니다.

## 126. 테스트는 왜 CI에 연결되어야 의미가 커지나

테스트를 로컬에서만 돌리면 개인 습관에 머물기 쉽다.  
진짜 힘이 생기는 순간은 CI에 연결될 때다.

### 핵심 개념

CI에서 테스트를 돌린다는 건  
**모든 변경이 공통 품질 게이트를 통과해야 한다**는 뜻이다.

즉:

- 누가 올렸든
- 어느 브랜치든
- 같은 기준으로 검증한다

### 왜 이게 중요한가

이걸 통해:

- “내 컴퓨터에선 되는데요”를 줄이고
- 배포 전에 회귀를 더 일찍 막고
- 코드 리뷰와 배포 신뢰도를 높일 수 있다

### 자주 하는 오해

- 테스트는 개발자 로컬에서만 꼼꼼히 하면 된다 → 사람마다 편차가 생긴다
- CI에 넣으면 너무 느려진다 → 계층별 전략으로 조절할 수 있다
- E2E까지 매 커밋마다 다 돌려야 한다 → 리스크와 시간에 따라 나눌 수 있다

### 실무 연결

보통 이런 구조가 자연스럽다.

- PR마다: 빠른 unit / integration
- main merge 전후: 더 넓은 테스트
- 배포 직전/야간: 무거운 E2E 세트

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트와 CI가 연결되는 이유를 봤다.  
그다음엔 어떤 테스트를 **어느 단계에서 돌릴지** 전략으로 넘어가야 한다.

### 면접 압축 답변

테스트가 CI에 연결되어야 의미가 커지는 이유는, 개인 습관이 아니라 팀 공통 품질 게이트가 되기 때문입니다. 이렇게 해야 누가 변경하든 같은 기준으로 검증되고, 회귀를 더 이른 단계에서 막을 수 있습니다.

## 127. 어떤 테스트를 어느 CI 단계에서 돌려야 하나

모든 테스트를 모든 시점에 다 돌리는 건 이상적일 수 있지만,  
실무에선 속도와 신뢰성의 균형이 중요하다.

### 핵심 개념

CI 테스트 전략은 보통 **빠른 것부터 자주**,  
**무거운 것은 더 선택적으로** 배치하는 식으로 간다.

### 예시 전략

#### PR 단위

- lint
- type check
- 빠른 unit test
- 핵심 integration test

#### merge / main 단위

- 더 넓은 integration
- 핵심 E2E smoke test

#### 배포 직전 / 스케줄 실행

- 전체 E2E
- 브라우저 매트릭스 확대
- 느리지만 중요한 회귀 시나리오

### 왜 이게 중요한가

이 전략이 없으면:

- PR이 너무 느려지거나
- 반대로 중요한 시나리오가 빠지거나
- 모두가 CI를 귀찮아하게 될 수 있다

### 자주 하는 오해

- 빠른 테스트만 돌리면 충분하다 → 핵심 사용자 흐름이 빠질 수 있다
- 느리더라도 전부 PR마다 돌려야 안전하다 → 개발 속도를 과도하게 떨어뜨릴 수 있다
- 테스트 배치는 한 번 정하면 끝이다 → 팀과 서비스 변화에 따라 조정해야 한다

### 실무 연결

좋은 CI 전략은 보통:

- 빠른 피드백
- 핵심 시나리오 보호
- flaky 억제
- 배포 신뢰
  를 동시에 노린다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 CI 전략까지 봤다.  
그다음엔 테스트 시스템 전체를 무너뜨리는 문제, 즉 **느린 테스트와 flaky test를 어떻게 관리할지**로 넘어가야 한다.

### 면접 압축 답변

CI에서는 모든 테스트를 동일하게 배치하기보다, 빠른 unit과 integration은 자주, 무거운 E2E는 핵심 흐름 위주로 더 선택적으로 돌리는 전략이 보통 효율적입니다. 핵심은 피드백 속도와 배포 신뢰를 함께 확보하는 것입니다.

---

## 128. 느린 테스트와 flaky test는 왜 별도로 관리해야 하나

느린 테스트와 flaky test는 둘 다 테스트 경험을 나쁘게 만든다.  
하지만 성격이 조금 다르다.

### 핵심 개념

- 느린 테스트는 피드백 속도를 떨어뜨린다
- flaky test는 신뢰 자체를 떨어뜨린다

즉:

- 느린 테스트는 “귀찮음”을 만들고
- flaky test는 “불신”을 만든다

둘 다 CI 품질에 큰 악영향을 준다.

### 왜 이게 중요한가

테스트 시스템이 망가지는 전형적인 흐름은 이렇다.

1. 테스트가 느려진다
2. 모두가 기다리기 싫어한다
3. flaky가 조금씩 쌓인다
4. 빨간불을 무시하기 시작한다
5. 품질 게이트가 형식화된다

### 자주 하는 오해

- 느린 테스트는 참고 기다리면 된다 → 팀 전체 개발 속도를 갉아먹는다
- flaky는 다시 돌리면 되니 큰 문제 아니다 → 신뢰를 무너뜨린다
- 테스트는 많을수록 좋다 → 유지 가능한 속도와 신뢰가 중요하다

### 실무 연결

그래서 좋은 팀은:

- 느린 테스트 목록을 따로 보고
- flaky를 따로 추적하고
- 기준 넘는 테스트는 분리하거나 개선하고
- 중요한 테스트가 계속 믿을 수 있게 관리한다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 테스트 시스템 운영 관점까지 봤다.  
그다음엔 전체 챕터를 한 번 구조적으로 정리해야 한다.

### 면접 압축 답변

느린 테스트는 피드백 속도를 떨어뜨리고, flaky test는 테스트 시스템 자체에 대한 신뢰를 무너뜨립니다. 그래서 둘은 단순 불편함이 아니라 CI 품질을 훼손하는 별도 운영 문제로 보고 지속적으로 관리해야 합니다.

---

## 129. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 테스트는 단순 버그 탐지보다 변경 자신감을 만든다
- unit / integration / E2E는 서로 다른 실패를 잡는다
- 프론트엔드에선 integration test가 특히 중요하다
- E2E는 가장 비싸지만 가장 현실적인 사용자 흐름을 검증한다
- 테스트 ROI는 실패 비용과 변경 빈도로 판단해야 한다
- mock은 유용하지만 과도하면 현실과 멀어진다
- API mocking과 실제 서버 사용은 균형이 필요하다
- flaky test는 팀 신뢰를 무너뜨린다
- 비동기 UI 테스트는 기다림 조건을 더 명확히 잡아야 한다
- 복잡한 프론트엔드 시나리오는 happy path 외 예외 흐름도 테스트해야 한다
- 테스트는 CI에 연결될 때 팀 공통 품질 게이트가 된다
- 테스트는 단계별로 적절히 배치해야 피드백 속도와 신뢰를 함께 얻을 수 있다

즉,  
이번 챕터는 단순 테스트 도구 사용법이 아니라,  
**“어떤 실패를 어디서 잡고, 어떤 자신감을 어떤 비용으로 확보할 것인가”**를 배우는 챕터다.

1. CI/CD는 테스트와 어떻게 이어져야 하나?
2. GitHub Actions 파이프라인은 어떤 흐름으로 설계하면 좋나?
3. Docker와 Nginx, PM2, EC2는 각각 어떤 역할을 하나?
4. 환경변수와 배포 환경 분리는 왜 중요한가?
5. 롤백, 로그, 메트릭, 알림은 운영 안정성과 어떻게 연결되나?

# 사용법

이 챕터는 아래 순서로 보면 좋다.

1. 테스트의 목적을 버그 탐지보다 변경 자신감으로 잡는다.
2. unit / integration / E2E를 실패 유형 기준으로 구분한다.
3. mock, API mocking, flaky를 통해 테스트 품질 감각을 세운다.
4. 복잡한 프론트엔드 시나리오와 비동기 테스트 주의점을 익힌다.
5. 마지막에 CI 연결과 단계별 테스트 배치 전략으로 운영 관점까지 묶는다.

이렇게 보면 테스트가 단순 확인 작업이 아니라, 팀이 안전하게 바꾸고 배포할 수 있게 만드는 구조라는 감각이 더 선명해진다.
