# 챕터 9. 테스트 전략에서 CI 신뢰성까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **테스트는 단순히 버그를 찾는 활동이 아니라, 변경해도 되는 자신감을 만들고 그 자신감을 CI(Continuous Integration, 지속적 통합 — 코드 변경을 자동으로 빌드하고 테스트하는 프로세스) 파이프라인까지 연결하는 구조다.**

### 전체 흐름 맵

```
테스트의 본질 (변경 자신감)
  → 테스트 계층 (unit / integration / E2E)
    → 테스트 ROI (어디에 투자할 것인가)
      → mock과 API mocking (외부 의존성 통제)
        → flaky test와 비동기 UI (안정성 확보)
          → 복잡한 시나리오 (예외 흐름까지)
            → CI 연결과 단계별 배치 (팀 품질 게이트)
```

## 77. 테스트는 왜 "변경 자신감"과 연결되나

> **핵심 요약** — 테스트의 본질적 가치는 버그 탐지보다, 코드를 바꿔도 괜찮다는 자신감을 팀에 제공하는 것이다. 테스트가 없으면 수정 자체가 무서워져 팀 속도가 느려진다.

테스트가 특히 강한 순간: 큰 리팩토링, 공용 컴포넌트 교체, 상태 구조 변경, API 응답 처리 로직 수정, 디자인 시스템 마이그레이션.

### 실무 에피소드 — 이런 상황에서 만난다

> 공용 Button 컴포넌트를 디자인 시스템 v2로 교체하는데, 앱 전체 200곳에서 사용 중. 테스트가 없었다면 수작업으로 200곳을 확인해야 했지만, integration test로 주요 페이지 흐름이 커버되어 있어서 교체 후 테스트 돌려서 깨진 3곳을 즉시 발견.

### 면접 압축 답변

테스트의 본질적 가치는 단순한 버그 탐지가 아니라 **변경 자신감**입니다. 코드를 바꿔도 기존 동작이 보장된다는 안전망이 있으면, 리팩토링이나 기능 추가를 과감하게 진행할 수 있습니다. 반대로 테스트가 없으면 수정 자체가 무서워져 팀 속도가 느려지고, 배포 전 수동 확인에 과도한 시간을 쓰게 됩니다.

## 78. unit, integration, E2E — 테스트 계층

> **핵심 요약** — unit은 작은 로직, integration은 컴포넌트 연결 흐름, **E2E(End-to-End, 종단 간 — 시작부터 끝까지 사용자 시나리오 전체를 검증하는 테스트)**는 실제 브라우저에서 사용자 시나리오 전체를 검증한다. 프론트엔드에서는 연결 지점 버그가 많아 integration 비중이 특히 중요하다.

### 계층 비교

| | Unit | Integration | E2E |
|---|---|---|---|
| **검증 대상** | 함수, 순수 로직 | 컴포넌트 + 상태 + 이벤트 흐름 | 실제 브라우저 시나리오 전체 |
| **도구** | Jest, Vitest | React Testing Library | Playwright, Cypress |
| **속도** | 매우 빠름 | 빠름 | 느림 |
| **실패 원인 파악** | 쉬움 | 중간 | 어려움 |
| **현실성** | 낮음 | 중간 | 높음 |
| **비용** | 낮음 | 중간 | 높음 |

### 테스트 피라미드

```
          ╱╲
         ╱  ╲          E2E (End-to-End)
        ╱ E2E╲         느림 · 비쌈 · 높은 신뢰도
       ╱──────╲        실제 브라우저에서 사용자 시나리오
      ╱        ╲
     ╱Integration╲     Integration (통합)
    ╱──────────────╲    중간 속도 · 컴포넌트 간 상호작용
   ╱                ╲
  ╱    Unit Tests    ╲  Unit (단위)
 ╱────────────────────╲  빠름 · 저렴 · 격리된 로직 검증
╱                      ╲

← 많이                     적게 →  (테스트 수)
← 빠름                     느림 →  (실행 속도)
← 낮음                     높음 →  (신뢰도)
```

### 각 계층의 실전 코드

```ts
// Unit: 순수 함수 테스트
describe("formatPrice", () => {
  it("숫자를 원화 형식으로 변환", () => {
    expect(formatPrice(1000)).toBe("1,000원");
  });
});
```

```tsx
// Integration: 컴포넌트 흐름 테스트 (React Testing Library)
it("입력 후 제출하면 onSearch가 호출된다", async () => {
  const handleSearch = jest.fn();
  render(<SearchForm onSearch={handleSearch} />);

  await userEvent.type(screen.getByPlaceholderText("검색어"), "리액트");
  await userEvent.click(screen.getByText("검색"));

  expect(handleSearch).toHaveBeenCalledWith("리액트");
});
```

```ts
// E2E: 실제 브라우저 시나리오 (Playwright)
test("로그인 후 대시보드로 이동", async ({ page }) => {
  await page.goto("/login");
  await page.getByLabel("이메일").fill("user@example.com");
  await page.getByLabel("비밀번호").fill("password123");
  await page.getByRole("button", { name: "로그인" }).click();

  await expect(page).toHaveURL("/dashboard");
  await expect(page.getByText("환영합니다")).toBeVisible();
});
```

### 어디에 무엇을 배치하나

| 대상 | 계층 |
|---|---|
| 유틸 함수, 포맷터, 검증 로직 | Unit |
| 폼 제출, 필터 변경 후 리스트, 모달, 토스트 | Integration |
| 로그인, 결제, 업로드, 핵심 사용자 흐름 | E2E |

### 실무 에피소드 — 이런 상황에서 만난다

> unit test만 잔뜩 있었는데, 검색 필터를 바꿨을 때 리스트가 갱신 안 되는 버그를 못 잡음. 필터 → API 호출 → 리스트 갱신 흐름이 연결되는 지점의 integration test가 없었기 때문. "각 함수는 맞는데 연결하면 안 되는" 전형적인 케이스.

### 면접 압축 답변

**Unit** 테스트는 순수 함수 같은 작은 로직의 정확성을, **integration** 테스트는 컴포넌트와 상태와 이벤트가 연결되는 흐름을, **E2E** 테스트는 실제 브라우저에서 사용자 시나리오 전체를 검증합니다. 프론트엔드에서는 "각 함수는 맞는데 연결하면 안 되는" 버그가 많기 때문에 **integration 비중이 특히 중요**하고, E2E는 비용이 크므로 로그인이나 결제 같은 핵심 흐름에 집중하는 것이 효율적입니다.

## 79. 테스트 ROI — 어디에 투자할 것인가

> **핵심 요약** — 테스트 **ROI(Return on Investment, 투자 대비 효과 — 여기서는 테스트 투자 대비 얻는 변경 자신감)**는 커버리지 수치가 아니라 "깨졌을 때 영향이 큰가, 변경이 잦은가, 수동 검증 비용이 큰가"로 판단한다. 인증·결제·공용 컴포넌트가 높은 ROI.

### ROI 판단 기준

| 기준 | 높은 ROI | 낮은 ROI |
|---|---|---|
| 실패 영향 | 결제, 인증, 데이터 저장 | 문구 오타, 마진 1px |
| 변경 빈도 | 자주 바뀌는 공용 로직 | 거의 안 바뀌는 설정 |
| 수동 검증 비용 | 매번 5분 걸리는 흐름 | 눈으로 바로 확인 가능 |
| 유지보수 비용 | 안정적 셀렉터, 명확한 조건 | 내부 구현에 강결합 |

### 실무 에피소드 — 이런 상황에서 만난다

> 커버리지 90%를 달성했는데, 결제 흐름에 테스트가 하나도 없어서 결제 로직 변경 후 장애 발생. 커버리지 숫자보다 "이게 깨지면 얼마나 아픈가"가 기준이어야 했다.

### 면접 압축 답변

테스트 **ROI**는 커버리지 숫자가 아니라 "깨졌을 때 얼마나 아픈가, 변경이 얼마나 잦은가, 수동으로 확인하는 데 비용이 얼마나 큰가"로 판단합니다. 커버리지가 100%여도 결제나 인증 같은 중요한 시나리오가 빠져 있으면 의미가 없고, 반대로 거의 변하지 않는 설정 코드에 테스트를 잔뜩 두는 것은 ROI가 낮습니다. **인증, 결제, 공용 컴포넌트**처럼 실패 비용이 크고 변경이 잦은 곳에 우선 투자해야 합니다.

## 80. mock과 API mocking

> **핵심 요약** — mock은 외부 의존성을 제어 가능한 가짜로 대체해 테스트를 빠르고 안정적으로 만든다. 하지만 과도하면 실제와 멀어진다. **MSW(Mock Service Worker, 네트워크 레벨에서 API 요청을 가로채는 테스트 라이브러리)**는 네트워크 레벨에서 요청을 가로채므로 가장 현실적인 API mocking 방법.

### mock, stub, spy 구분

| | mock | stub | spy |
|---|---|---|---|
| **역할** | 호출 여부·인자 검증 가능 | 정해진 응답만 반환 | 실제 구현 유지 + 관찰 |
| **검증** | 호출됐는지 확인 | 안 함 | 호출됐는지 확인 |

### MSW — 네트워크 레벨 API mocking

```ts
// mocks/handlers.ts
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("/api/users/:id", ({ params }) => {
    return HttpResponse.json({ id: params.id, name: "홍길동" });
  }),
];
```

```tsx
// 테스트에서 사용
beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

it("유저 정보를 표시한다", async () => {
  render(<UserProfile userId="1" />);
  await waitFor(() => expect(screen.getByText("홍길동")).toBeInTheDocument());
});
```

### mock의 균형

| 상황 | 접근 |
|---|---|
| unit / integration | MSW로 API mocking 적극 활용 |
| 핵심 E2E | 실제에 가까운 환경(스테이징) |
| 계약 검증 | 응답 shape와 에러 케이스를 현실적으로 |

### 실무 에피소드 — 이런 상황에서 만난다

> 모든 API를 `jest.mock`으로 가짜 응답을 넣었는데, 실제 서버 응답 형태가 바뀌어도 테스트는 통과 → 배포 후 깨짐. MSW로 전환하고 실제 API 응답과 가까운 mock 데이터를 유지하니 계약 불일치를 테스트에서 잡을 수 있게 됨.

### 면접 압축 답변

**mock**은 외부 의존성을 제어 가능한 가짜로 대체해서 테스트를 빠르고 안정적으로 만들어주지만, 과도하게 쓰면 실제 구현과 너무 멀어져서 "테스트는 통과하는데 실제론 깨지는" 위험한 상태가 됩니다. API mocking에서는 **MSW**가 네트워크 레벨에서 요청을 가로채기 때문에 fetch나 axios를 직접 mock하는 것보다 훨씬 현실적입니다. unit/integration에서는 MSW를 적극 활용하되, 핵심 E2E는 스테이징 같은 실제에 가까운 환경에서 돌리는 균형이 중요합니다.

## 81. flaky test와 비동기 UI 테스트

> **핵심 요약** — flaky test는 코드 변화 없이 결과가 들쭉날쭉한 테스트로, 누적되면 팀이 테스트를 무시하게 된다. 가장 흔한 원인은 `setTimeout` 기반 대기 — `waitFor`/`findBy`로 상태 변화 조건을 기다려야 안정적.

### flaky를 만드는 안티패턴

```tsx
// BAD: setTimeout으로 대충 기다리기
await new Promise((r) => setTimeout(r, 500));
expect(screen.getByText("홍길동")).toBeInTheDocument();

// GOOD: 상태 변화를 명시적으로 기다리기
await waitFor(() => expect(screen.getByText("홍길동")).toBeInTheDocument());

// GOOD: findBy 사용 (waitFor + getBy 축약)
expect(await screen.findByText("홍길동")).toBeInTheDocument();
```

### React Testing Library 대기 패턴

| 패턴 | 용도 |
|---|---|
| `getBy` | 즉시 존재해야 하는 요소 (동기) |
| `queryBy` | 존재하지 않는 것을 확인 (null 반환) |
| `findBy` | 비동기로 나타나는 요소 대기 |
| `waitFor` | 임의의 비동기 조건 대기 |
| `waitForElementToBeRemoved` | 로딩 스피너 등 사라짐 대기 |

### 비동기 UI 테스트 — 흐름 전체 검증

```tsx
it("클릭 → 로딩 → 완료 순서로 전환", async () => {
  render(<AsyncButton />);

  expect(screen.getByText("저장")).toBeInTheDocument();       // 초기
  await userEvent.click(screen.getByText("저장"));
  expect(screen.getByText("저장 중...")).toBeInTheDocument(); // 로딩
  await waitFor(() =>
    expect(screen.getByText("저장 완료!")).toBeInTheDocument() // 완료
  );
});
```

### flaky 대응 전략

| 원인 | 대응 |
|---|---|
| 비동기 타이밍 | `waitFor`/`findBy`로 교체 |
| 이전 테스트 상태 잔존 | 각 테스트 자체 초기화 |
| 네트워크 의존 | MSW로 통제 |
| 랜덤/시간 의존 | `jest.useFakeTimers()` |
| 불안정한 셀렉터 | `data-testid` 사용 |

### 실무 에피소드 — 이런 상황에서 만난다

> CI에서 10번 중 2번 실패하는 테스트가 3개 있었는데 "재실행하면 돼"로 방치. 2달 후 flaky가 10개로 늘어나면서 팀이 빨간불을 무시하기 시작 → 진짜 버그도 놓침. flaky를 격리(quarantine) 후 기한 내 수정/삭제 정책을 도입하니 CI 신뢰 회복.

### 면접 압축 답변

**Flaky test**는 코드 변화 없이 결과가 들쭉날쭉한 테스트로, 누적되면 팀이 빨간불을 무시하게 되어 자동화 전체의 가치가 무너집니다. 가장 흔한 원인은 `setTimeout`으로 대충 기다리는 패턴인데, 대신 **waitFor**나 **findBy**로 상태 변화 조건을 명시적으로 기다려야 안정적입니다. 테스트 간 독립성을 보장하고, 네트워크는 MSW로 통제하며, CI에서 flaky를 별도로 추적해서 격리하고 기한 내에 수정하거나 삭제하는 운영 정책을 두는 것이 중요합니다.

## 82. 복잡한 프론트엔드 시나리오 테스트

> **핵심 요약** — DnD, 업로드, 세션 만료, optimistic update 같은 시나리오는 happy path보다 상태 전이와 예외 흐름(실패, 롤백, 중복 요청)을 함께 검증해야 실제 안정성이 높아진다.

### 정상 + 예외를 함께 보는 테스트

```tsx
// optimistic update: 성공 + 실패 롤백
it("좋아요 클릭 후 서버 실패하면 롤백", async () => {
  server.use(http.post("/api/like", () => new HttpResponse(null, { status: 500 })));

  render(<LikeButton initialCount={10} />);
  await userEvent.click(screen.getByRole("button", { name: "좋아요" }));

  expect(screen.getByText("11")).toBeInTheDocument();  // optimistic
  await waitFor(() => expect(screen.getByText("10")).toBeInTheDocument()); // 롤백
});
```

### 실무 에피소드 — 이런 상황에서 만난다

> 파일 업로드 기능에서 happy path(성공)만 테스트했는데, 대용량 파일 업로드 중 네트워크 끊김 시 "업로드 완료"로 표시되는 버그. 업로드 실패 + 재시도 + 취소 흐름까지 테스트를 확장하니 해당 버그를 사전에 잡을 수 있게 됨.

### 면접 압축 답변

업로드, **optimistic update**, 세션 만료 같은 복잡한 시나리오는 성공하는 happy path만 테스트하면 부족합니다. 실패 시 롤백이 제대로 되는지, 중복 요청은 어떻게 처리되는지, 취소 흐름은 정상 동작하는지까지 **상태 전이와 예외 흐름을 함께 검증**해야 실제 안정성이 높아집니다. 이런 시나리오는 수동 테스트로는 반복 회귀를 놓치기 쉬우므로 자동화가 더욱 중요합니다.

### Visual Regression Test — 화면이 의도대로 바뀌었는지 검증

코드 변경 후 "기능은 동작하는데 화면이 깨졌다"를 잡는 테스트다. 스크린샷을 비교하여 픽셀 단위의 변경을 감지한다.

```ts
// Playwright visual regression 예시
test("상품 카드 스냅샷", async ({ page }) => {
  await page.goto("/products/1");
  await expect(page.locator(".product-card")).toHaveScreenshot("product-card.png", {
    maxDiffPixelRatio: 0.01, // 1% 이내 차이 허용
  });
});
```

**주의**: OS/폰트/브라우저 버전에 따라 렌더링이 달라지므로, CI 환경을 Docker로 고정하는 것이 핵심이다.

### Snapshot Test 안티패턴 — 무분별한 toMatchSnapshot

```tsx
// ❌ 안티패턴: 컴포넌트 전체를 스냅샷
test("renders", () => {
  const { container } = render(<ComplexForm />);
  expect(container).toMatchSnapshot(); // 작은 변경에도 깨짐 → 무의미하게 업데이트
});

// ✅ 대안: 구체적 검증
test("에러 메시지를 올바르게 표시", () => {
  render(<ComplexForm />);
  fireEvent.click(screen.getByText("제출"));
  expect(screen.getByRole("alert")).toHaveTextContent("이메일을 입력해주세요");
});
```

스냅샷 테스트는 "깨져도 그냥 업데이트" 습관이 생기면 테스트가 아니라 의식(ritual)이 된다.

### 비결정적 요소 제어 — fake timer, date, random

테스트에서 `Date.now()`, `setTimeout`, `Math.random()`은 실행마다 결과가 달라져 flaky test의 원인이 된다.

```ts
// Vitest fake timer 예시
import { vi, beforeEach, afterEach } from "vitest";

beforeEach(() => {
  vi.useFakeTimers();
  vi.setSystemTime(new Date("2025-01-15T09:00:00Z")); // 날짜 고정
});

afterEach(() => {
  vi.useRealTimers();
});

test("3초 후 토스트가 사라진다", async () => {
  render(<Toast message="저장됨" />);
  expect(screen.getByText("저장됨")).toBeInTheDocument();

  vi.advanceTimersByTime(3000); // 3초 빠르게 감기

  expect(screen.queryByText("저장됨")).not.toBeInTheDocument();
});
```

| 비결정적 요소 | 제어 방법 |
|---|---|
| `setTimeout` / `setInterval` | `vi.useFakeTimers()` + `vi.advanceTimersByTime()` |
| `Date.now()` / `new Date()` | `vi.setSystemTime()` |
| `Math.random()` | `vi.spyOn(Math, 'random').mockReturnValue(0.5)` |
| `crypto.randomUUID()` | mock 또는 seed 기반 구현 주입 |

## 83. 테스트와 CI — 팀 품질 게이트

> **핵심 요약** — 테스트가 CI에 연결되면 개인 습관이 아니라 팀 공통 품질 게이트가 된다. PR에는 빠른 unit/integration, main 병합이나 야간에는 무거운 E2E를 배치하는 단계별 전략.

### CI 단계별 테스트 배치

| 시점 | 돌릴 테스트 | 이유 |
|---|---|---|
| **PR마다** | lint, type check, unit, 핵심 integration | 빠른 피드백 |
| **main 병합** | 넓은 integration, 핵심 E2E smoke | 품질 확인 |
| **배포 전/야간** | 전체 E2E, 브라우저 매트릭스 | 안정성 확보 |

```yaml
# .github/workflows/test.yml (PR 단위)
name: Test
on:
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: "npm" }
      - run: npm ci
      - run: npm run lint && npm run type-check
      - run: npm test -- --coverage
```

```yaml
# .github/workflows/e2e.yml (main + 야간)
name: E2E
on:
  push: { branches: [main] }
  schedule: [{ cron: "0 3 * * *" }]
jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci && npx playwright install --with-deps
      - run: npx playwright test
      - if: failure()
        uses: actions/upload-artifact@v4
        with: { name: playwright-report, path: playwright-report/ }
```

### CI 파이프라인 — 단계별 테스트 배치

```
PR 생성 시 (빠른 피드백):
┌────────┐  ┌──────────┐  ┌────────────┐  ┌───────┐
│  lint  │→│ typecheck │→│ unit test  │→│ build │
└────────┘  └──────────┘  └────────────┘  └───────┘
                                            (~3분)

main 머지 시 (중간 검증):
┌────────────────────┐  ┌───────────────────┐
│  integration test  │→│  preview deploy   │
└────────────────────┘  └───────────────────┘
                                    (~10분)

매일 밤 (전체 검증):
┌──────────────────┐  ┌────────────┐
│  E2E (Playwright)│→│  보고서    │
└──────────────────┘  └────────────┘
                            (~30분)
```

### 느린 테스트 vs flaky test — 별도 관리

| | 느린 테스트 | flaky test |
|---|---|---|
| **문제** | 피드백 속도 저하 ("귀찮음") | 신뢰 자체 파괴 ("불신") |
| **대응** | 병렬화, 별도 job 분리 | 추적, 격리, 기한 내 수정/삭제 |

### 실무 에피소드 — 이런 상황에서 만난다

> 테스트 없이 배포하다가 공용 컴포넌트 리팩토링 후 5개 페이지가 동시에 깨짐. 이후 PR마다 unit/integration을 CI로 돌리고, main 병합 시 핵심 E2E를 추가. "내 컴퓨터에선 되는데요"가 사라지고 배포 전 회귀를 미리 잡을 수 있게 됨.

### 면접 압축 답변

테스트가 **CI**에 연결되면 개인 습관이 아니라 **팀 공통 품질 게이트**가 됩니다. PR마다 빠른 lint, type check, unit, 핵심 integration을 돌려서 즉각 피드백하고, main 병합 시에는 넓은 integration과 핵심 E2E smoke 테스트를, 야간에는 전체 E2E와 브라우저 매트릭스를 돌려 안정성을 확보합니다. 느린 테스트는 병렬화로, flaky test는 격리 후 기한 내 수정이나 삭제로 별도 관리해야 CI 신뢰가 유지됩니다.

# 챕터 9 핵심 요약 — 변경 자신감을 만들고 CI까지 연결하는 테스트 구조

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **테스트는 단순히 버그를 찾는 활동인가, 아니면 코드를 바꿔도 된다는 자신감을 팀에 제공하는 구조인가?**

---

### 1단계: 테스트의 진짜 가치를 이해한다 — 변경 자신감

테스트의 본질적 가치는 버그 탐지보다, **코드를 바꿔도 기존 동작이 보장된다는 안전망을 팀에 제공하는 것**이다.

### 2단계: 어떤 실패를 어디서 잡을지 나눈다 — 테스트 계층

unit은 작은 순수 로직, integration은 컴포넌트·상태·이벤트의 연결 흐름, E2E는 실제 브라우저에서 사용자 시나리오 전체를 검증한다. **프론트엔드에서는 integration 비중이 특히 중요하다.**

### 3단계: 한정된 자원을 어디에 쓸지 판단한다 — 테스트 ROI

**ROI는 "깨졌을 때 영향이 큰가, 변경이 잦은가, 수동 검증 비용이 큰가"로 판단**한다. 인증·결제·공용 컴포넌트가 높은 ROI다.

### 4단계: 외부 의존성을 통제한다 — mock과 API mocking

**과도한 mock은 "테스트는 통과하는데 실제론 깨지는" 상태를 만든다.** MSW는 네트워크 레벨에서 요청을 가로채므로 더 현실적이다.

---

여기서부터 "무엇을 테스트할 것인가"에서 "테스트를 어떻게 신뢰할 수 있게 만들 것인가"로 전환된다.

---

### 5단계: 불안정한 테스트를 제거한다 — flaky test와 비동기 UI

flaky test는 **누적되면 팀이 테스트 결과를 무시하게 되어 자동화 전체의 가치가 무너진다.** `waitFor`/`findBy`로 상태 변화 조건을 기다려야 안정적이다.

### 6단계: 예외 흐름까지 검증한다 — 복잡한 시나리오 테스트

**happy path보다 상태 전이와 예외 흐름(실패, 롤백, 중복 요청)을 함께 검증**해야 실제 안정성이 높아진다.

### 7단계: 개인 습관을 팀 품질 게이트로 올린다 — CI 연결

테스트가 CI에 연결되면 **팀 공통 품질 게이트**가 된다. PR에는 빠른 unit/integration, main 병합이나 야간에 무거운 E2E를 배치한다.

---

### 한 줄로 이어 읽기

**변경 자신감**이 테스트의 본질이고 → **테스트 계층(unit/integration/E2E)**이 어떤 실패를 어디서 잡을지 나누고 → **ROI 판단**이 투자 우선순위를 정하고 → **mock과 MSW**가 외부 의존성을 통제하고 → **flaky test 제거**가 테스트 신뢰를 지키고 → **복잡한 시나리오 테스트**가 예외 흐름까지 검증하고 → **CI 단계별 배치**가 개인 습관을 팀 품질 게이트로 끌어올린다.

> 다음 챕터에서는 "변경 자신감을 어떻게 만드는가"에서 **"검증된 코드를 어떻게 안전하게 배포하고 운영하는가"**로 축이 이동한다. CI/CD, Docker, 환경 분리, 롤백, 로그/메트릭/알림이 운영 안정성의 뼈대다.
