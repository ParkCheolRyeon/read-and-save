# 챕터 9. 테스트 전략에서 CI 신뢰성까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **테스트는 단순히 버그를 찾는 활동이 아니라, 변경해도 되는 자신감을 만들고 그 자신감을 CI 파이프라인까지 연결하는 구조라는 관점으로 이해해야 한다.**

## 115. 테스트는 왜 단순 버그 탐지보다 "변경 자신감"과 더 강하게 연결되나

테스트를 처음 배우면 흔히 이렇게 생각한다.

> **테스트는 버그를 잡기 위한 것**

물론 맞는 말이다.
하지만 실무에서는 이 설명만으로는 부족하다.

### 핵심 개념

테스트의 더 본질적인 역할은
**코드를 바꾸어도 괜찮다는 자신감을 팀에 제공하는 것**이다.

즉 테스트는:

- 현재 동작이 맞는지 확인하고
- 이후 리팩토링이나 기능 추가 때
- 기존 동작이 깨졌는지 빠르게 알게 해주는 장치다

### 왜 이게 중요한가

실무에서 진짜 무서운 건
"지금 당장 버그가 있나?"보다
"이걸 수정하면 어디가 같이 깨질지 모른다"는 상태다.

테스트가 없으면:

- 코드 수정이 무서워지고
- 안전한 리팩토링이 어려워지고
- 배포 직전 수동 확인이 과도해지고
- 팀 전체 속도가 느려질 수 있다

### 자주 하는 오해

- 테스트는 QA가 하는 일이다 → 개발 단계의 안전장치이기도 하다
- 테스트는 버그가 많을 때만 필요하다 → 오히려 변경이 많은 코드일수록 중요하다
- 테스트를 많이 쓰면 개발이 무조건 느려진다 → 장기적으로는 변경 비용을 낮추는 경우가 많다

### 실무 연결

테스트가 특히 강한 순간은 이런 때다.

- 큰 리팩토링
- 공용 컴포넌트 교체
- 상태 구조 변경
- API 응답 처리 로직 수정
- 디자인 시스템 마이그레이션

즉 테스트는 지금보다
**다음 변경을 위한 안전망**에 더 가깝다.

> 이제 테스트의 존재 이유를 봤다. 그렇다면 테스트는 어떤 층으로 나뉘고, 각 층은 무엇을 확인하는가?

### 면접 압축 답변

- **10초**: 테스트의 본질적 가치는 버그 탐지보다, 코드를 바꿔도 괜찮다는 변경 자신감을 팀에 제공하는 데 있습니다.
- **30초**: 테스트의 본질적 가치는 단순 버그 탐지보다, 코드를 바꿔도 괜찮다는 변경 자신감을 팀에 제공하는 데 있습니다. 특히 리팩토링과 공용 구조 변경이 잦은 프론트엔드에서는 테스트가 미래 변경 비용을 낮추는 안전망 역할을 합니다. 테스트가 없으면 수정 자체가 무서워져 팀 전체 속도가 느려질 수 있습니다.

## 116. unit, integration, E2E는 무엇을 각각 검증하나

테스트를 배울 때 가장 먼저 구조적으로 잡아야 하는 게 테스트 계층이다.
많은 사람이 세 가지 이름은 외우지만, 실제로 **어떤 실패를 어디서 잡아야 하는지**가 흐릿하다.

### 핵심 개념

#### Unit Test

가장 작은 단위의 로직이나 함수, 컴포넌트 조각이 예상대로 동작하는지 본다.

#### Integration Test

여러 조각이 함께 연결되었을 때 기대한 흐름으로 동작하는지 본다.

#### E2E Test

사용자 시나리오 전체가 실제 브라우저 환경에서 끝까지 동작하는지 본다.

### 테스트 피라미드 구조

일반적으로 테스트는 피라미드 구조로 설계한다.

- **하단 (unit)**: 가장 많고, 가장 빠르고, 가장 저렴하다
- **중간 (integration)**: 적당한 수, 적당한 속도
- **상단 (E2E)**: 가장 적고, 가장 느리고, 가장 비싸다

다만 프론트엔드에서는 Kent C. Dodds가 제안한 **테스트 트로피** 모델처럼 integration 비중을 가장 높이는 접근도 널리 쓰인다. 이유는 프론트엔드의 버그가 대부분 컴포넌트 간 연결 지점에서 발생하기 때문이다.

### 왜 이게 중요한가

이 구분이 중요해야:

- 너무 많은 걸 unit에 몰아넣지 않고
- 너무 사소한 걸 E2E로 밀어넣지 않으며
- 실패 원인을 더 빠르게 좁힐 수 있다

### 자주 하는 오해

- unit test가 많으면 E2E는 필요 없다 → 틀림. unit은 연결 지점의 실패를 못 잡는다
- E2E만 있으면 다 커버된다 → 느리고 디버깅이 어렵다
- integration은 unit과 E2E 사이의 애매한 단계다 → 실제론 매우 중요한 계층이다

### 실무 연결

좋은 기준은 이렇다.

- 순수 로직 검증 → unit
- 컴포넌트와 상태, 이벤트 흐름 검증 → integration
- 로그인, 결제, 업로드, 검색 같은 사용자 핵심 흐름 → E2E

> 이제 테스트 계층은 봤다. 각각을 조금 더 구체적으로 이해하기 위해 unit test가 잘 맞는 대상부터 들어간다.

### 면접 압축 답변

- **10초**: unit은 작은 로직, integration은 연결 흐름, E2E는 사용자 시나리오 전체를 검증하며, 서로 다른 실패를 잡는 계층입니다.
- **30초**: unit, integration, E2E는 테스트 범위와 목적이 다릅니다. unit은 작은 로직 단위의 정확성을, integration은 여러 조각의 연결 흐름을, E2E는 실제 사용자 시나리오 전체를 검증합니다. 하나가 다른 하나를 완전히 대체하는 구조가 아니라 서로 다른 실패를 잡는 계층으로 보는 게 맞습니다. 프론트엔드에서는 연결 지점 버그가 많아서 integration 비중이 특히 중요합니다.

## 117. unit test는 어디에 가장 잘 맞고, 어디엔 잘 안 맞나

unit test는 가장 빠르고 가볍게 돌릴 수 있는 경우가 많아서
테스트의 첫 층으로 매우 유용하다.
하지만 모든 걸 unit test로 해결하려고 하면 오히려 이상해진다.

### 핵심 개념

unit test는
**작고 독립적인 단위가 정해진 입력에서 정해진 출력을 내는가**를 보기에 좋다.

대표적으로:

- 유틸 함수
- 파서
- 포맷터
- 상태 전환 함수
- 순수 계산 로직
- 간단한 커스텀 훅의 순수 분기

### 코드 예시

```ts
// utils/formatPrice.ts
export function formatPrice(value: number): string {
  return value.toLocaleString("ko-KR") + "원";
}
```

```ts
// utils/formatPrice.test.ts
import { formatPrice } from "./formatPrice";

describe("formatPrice", () => {
  it("숫자를 한국 원화 형식으로 변환한다", () => {
    expect(formatPrice(1000)).toBe("1,000원");
    expect(formatPrice(0)).toBe("0원");
  });

  it("큰 숫자도 올바르게 포맷한다", () => {
    expect(formatPrice(1234567)).toBe("1,234,567원");
  });
});
```

```ts
// utils/validateEmail.ts
export function validateEmail(email: string): boolean {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}
```

```ts
// utils/validateEmail.test.ts
import { validateEmail } from "./validateEmail";

describe("validateEmail", () => {
  it("유효한 이메일은 true를 반환한다", () => {
    expect(validateEmail("user@example.com")).toBe(true);
  });

  it("@ 없는 문자열은 false를 반환한다", () => {
    expect(validateEmail("invalid-email")).toBe(false);
  });

  it("빈 문자열은 false를 반환한다", () => {
    expect(validateEmail("")).toBe(false);
  });
});
```

이처럼 입력과 출력이 명확하고, 외부 의존성이 없는 함수가 unit test에 가장 잘 맞는다.

### 왜 이게 중요한가

이런 로직은:

- 빠르게 실행되고
- 실패 원인이 명확하고
- 리팩토링 시 회귀를 빠르게 잡아낼 수 있다

### 자주 하는 오해

- React 앱이면 unit test는 의미 없다 → 순수 로직은 여전히 많다
- 단위가 작을수록 항상 좋다 → 지나치게 쪼개면 의도가 흐려질 수 있다
- UI도 다 unit으로 충분히 테스트 가능하다 → 상호작용 흐름은 integration이 더 자연스러울 수 있다

### 실무 연결

프론트엔드에서 unit test가 특히 강한 부분:

- 날짜/숫자 포맷 함수
- 검색어 정규화
- 테이블 정렬 로직
- permission 계산
- 폼 검증 함수
- API 응답 normalize 함수

> 이제 작은 단위 테스트가 잘 맞는 대상을 봤다. 그다음은 실제 컴포넌트와 사용자 이벤트가 엮일 때 중요한 integration test다.

### 면접 압축 답변

- **10초**: unit test는 유틸 함수나 상태 전환 함수처럼 작고 독립적인 로직을 검증하는 데 가장 잘 맞습니다.
- **30초**: unit test는 유틸 함수나 상태 전환 함수처럼 작고 독립적인 로직을 검증하는 데 가장 잘 맞습니다. 빠르고 실패 원인이 명확하다는 장점이 있지만, 실제 UI 상호작용 흐름까지 모두 unit test로 해결하려 하면 오히려 테스트가 인위적이고 읽기 어려워질 수 있습니다. 그래서 순수 로직은 unit, 흐름은 integration으로 나누는 게 좋습니다.

## 118. integration test는 왜 프론트엔드에서 특히 중요하나

프론트엔드에서 버그는 단일 함수 자체보다
**여러 조각이 연결되는 순간** 더 자주 생긴다.

예를 들어:

- 버튼 클릭
- 상태 업데이트
- 조건부 렌더링
- API 상태 반영
- 에러 메시지 표시

같은 흐름은 하나의 함수만 보면 잘 안 드러난다.

### 핵심 개념

integration test는
여러 단위가 함께 연결되었을 때 실제 사용자 관점의 흐름이 맞는지 보는 테스트다.

즉:

- 컴포넌트
- state
- 이벤트
- 비동기 처리
- DOM 반영

이 연결된 결과를 본다.

### 코드 예시

React Testing Library를 사용한 전형적인 integration test:

```tsx
// components/SearchForm.tsx
export function SearchForm({ onSearch }: { onSearch: (query: string) => void }) {
  const [query, setQuery] = useState("");

  return (
    <form onSubmit={(e) => { e.preventDefault(); onSearch(query); }}>
      <input
        placeholder="검색어를 입력하세요"
        value={query}
        onChange={(e) => setQuery(e.target.value)}
      />
      <button type="submit">검색</button>
    </form>
  );
}
```

```tsx
// components/SearchForm.test.tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { SearchForm } from "./SearchForm";

describe("SearchForm", () => {
  it("입력 후 제출하면 onSearch가 검색어와 함께 호출된다", async () => {
    const handleSearch = jest.fn();
    render(<SearchForm onSearch={handleSearch} />);

    const input = screen.getByPlaceholderText("검색어를 입력하세요");
    await userEvent.type(input, "리액트");
    await userEvent.click(screen.getByText("검색"));

    expect(handleSearch).toHaveBeenCalledWith("리액트");
  });

  it("빈 입력으로 제출하면 빈 문자열이 전달된다", async () => {
    const handleSearch = jest.fn();
    render(<SearchForm onSearch={handleSearch} />);

    await userEvent.click(screen.getByText("검색"));

    expect(handleSearch).toHaveBeenCalledWith("");
  });
});
```

이 테스트는 단일 함수가 아니라 **입력 → 상태 변경 → 이벤트 → 콜백** 흐름 전체를 검증한다. 사용자 관점에서 "검색어를 치고 버튼을 누르면 검색이 실행되는가"를 확인하는 것이다.

### 왜 이게 중요한가

프론트엔드에선 아래 같은 버그가 많다.

- 클릭은 되는데 로딩 UI가 안 뜸
- 에러 메시지가 조건에 맞게 안 나옴
- 입력값은 바뀌는데 제출 버튼 활성화가 안 됨
- 비동기 완료 후 화면이 기대대로 안 갱신됨

이런 건 순수 함수 테스트보다 integration test가 더 잘 잡는다.

### 자주 하는 오해

- integration test는 느리고 무거워서 최소화해야 한다 → 프론트엔드에선 오히려 핵심 계층인 경우가 많다
- DOM이 나오면 다 E2E다 → 아니다. jsdom 위에서 컴포넌트를 렌더하는 것은 E2E가 아니다
- integration test는 단위 테스트보다 덜 정확하다 → 실제 사용자 흐름엔 더 적합할 수 있다

### 실무 연결

대표적인 대상:

- 로그인 폼 제출 흐름
- 필터 변경 후 리스트 갱신
- 모달 열기/닫기
- 토스트 표시
- optimistic update UI
- 에러 fallback 렌더링

> 이제 integration test의 역할을 봤다. 그다음은 브라우저 전체에서 실제 사용자 흐름을 검증하는 E2E test다.

### 면접 압축 답변

- **10초**: 프론트엔드에서는 여러 조각이 연결되는 지점에서 버그가 자주 발생하므로, integration test가 실질적인 안정성을 높이는 핵심 계층입니다.
- **30초**: integration test는 여러 단위가 함께 연결된 실제 UI 흐름을 검증하는 테스트입니다. 프론트엔드에선 버튼 클릭, 상태 갱신, 비동기 처리, DOM 반영이 함께 엮이는 버그가 많기 때문에, integration test가 실질적인 안정성을 높이는 핵심 계층이 되는 경우가 많습니다. React Testing Library처럼 사용자 관점에서 테스트하는 도구가 이 계층에 잘 맞습니다.

## 119. E2E test는 왜 가장 비싸지만 가장 현실적인가

E2E는 보통 가장 무겁고 느린 테스트 계층이다.
그런데도 중요한 이유는 분명하다.

### 핵심 개념

E2E test는
실제 사용자처럼 브라우저를 통해 애플리케이션 전체 흐름을 끝까지 검증한다.

즉:

- 라우팅
- 실제 렌더링
- 네트워크
- 폼 입력
- 화면 전환
- 브라우저 동작

을 더 현실에 가깝게 본다.

### 코드 예시

Playwright를 사용한 E2E test:

```ts
// e2e/login.spec.ts
import { test, expect } from "@playwright/test";

test("로그인 후 대시보드로 이동한다", async ({ page }) => {
  await page.goto("/login");

  await page.getByLabel("이메일").fill("user@example.com");
  await page.getByLabel("비밀번호").fill("password123");
  await page.getByRole("button", { name: "로그인" }).click();

  // 로그인 후 대시보드로 리다이렉트
  await expect(page).toHaveURL("/dashboard");
  await expect(page.getByText("환영합니다")).toBeVisible();
});

test("잘못된 비밀번호로 로그인하면 에러 메시지가 표시된다", async ({ page }) => {
  await page.goto("/login");

  await page.getByLabel("이메일").fill("user@example.com");
  await page.getByLabel("비밀번호").fill("wrong-password");
  await page.getByRole("button", { name: "로그인" }).click();

  await expect(page.getByText("이메일 또는 비밀번호가 올바르지 않습니다")).toBeVisible();
  await expect(page).toHaveURL("/login");
});
```

Integration test와의 차이: E2E는 **실제 브라우저** 위에서 실제 네트워크 요청, 실제 라우팅, 실제 쿠키/세션까지 포함해서 검증한다. 그만큼 현실적이지만 느리고, 실패 원인을 좁히기 어렵다.

### 왜 이게 중요한가

아래 문제는 E2E에서 더 잘 드러난다.

- 라우팅 연결 실패
- 실제 브라우저 이벤트 차이
- hydration 이후 이상 동작
- 인증 흐름 문제 (쿠키, 세션, 리다이렉트)
- 중요한 사용자 시나리오 붕괴

### 자주 하는 오해

- E2E 하나면 나머지 테스트는 필요 없다 → 느리고 실패 원인 추적이 어렵다
- E2E는 QA 자동화랑 같은 말이다 → 개발자가 직접 구조적으로 설계해야 효과가 크다
- 모든 페이지를 다 E2E 해야 한다 → 핵심 시나리오 우선이다

### 실무 연결

E2E가 특히 잘 맞는 대표 시나리오:

- 로그인
- 회원가입
- 검색
- 장바구니/결제
- 파일 업로드
- 핵심 관리자 작업
- 세션 만료 후 재로그인 흐름

> 이제 테스트 계층 전체를 봤다. 그다음 질문은 자연스럽다: 어떤 걸 어떤 층에 배치해야 ROI가 가장 좋은가?

### 면접 압축 답변

- **10초**: E2E는 실제 브라우저에서 사용자 시나리오 전체를 검증하므로 가장 현실적이지만, 비용이 커서 핵심 흐름에 집중해야 합니다.
- **30초**: E2E test는 실제 브라우저에서 사용자 시나리오 전체를 검증하기 때문에 가장 현실적인 테스트 계층입니다. 다만 실행 비용이 크고 실패 원인 파악이 어렵기 때문에, 모든 걸 맡기기보다 로그인, 결제, 업로드 같은 핵심 흐름에 집중하는 것이 보통 더 효율적입니다. Playwright나 Cypress 같은 도구를 사용합니다.

## 120. 테스트 ROI는 어떻게 판단해야 하나

실무에서는 테스트를 많이 쓰는 것 자체가 목적이 아니다.
더 중요한 건:

> **어디에 테스트를 투자해야 가장 큰 안정성을 얻는가**

이다.

### 핵심 개념

테스트 ROI를 볼 때는 보통 아래를 같이 본다.

- 깨졌을 때 피해가 큰가
- 변경이 잦은가
- 수동 확인 비용이 큰가
- 자동화로 반복 이득이 큰가
- 테스트 유지보수 비용이 감당 가능한가

### ROI 판단 매트릭스

| 기준 | 높은 ROI | 낮은 ROI |
|------|----------|----------|
| 실패 영향 | 결제, 인증, 데이터 저장 | 문구 오타, 마진 1px |
| 변경 빈도 | 자주 변경되는 공용 로직 | 거의 안 바뀌는 설정 파일 |
| 수동 검증 비용 | 매번 수동으로 5분 걸리는 흐름 | 눈으로 바로 확인 가능한 UI |
| 유지보수 비용 | 안정적인 셀렉터, 명확한 조건 | 내부 구현에 강결합된 테스트 |

### 왜 이게 중요한가

이 감각이 없으면:

- 사소한 UI 문구 테스트에 시간 과투자
- 거의 안 바뀌는 부분을 과하게 자동화
- 정말 중요한 인증/결제 흐름은 비워두는

상태가 될 수 있다.

### 자주 하는 오해

- 커버리지 높으면 테스트 전략도 좋다 → 틀림. 커버리지 100%여도 중요한 시나리오가 빠질 수 있다
- 많은 테스트가 무조건 좋은 테스트다 → 유지비용도 봐야 한다
- 핵심 로직만 테스트하면 UI는 안 해도 된다 → 실제 UX 흐름은 별도 문제다

### 실무 연결

ROI가 큰 대표 후보:

- 인증
- 업로드
- 결제/저장
- 공용 컴포넌트
- 상태 전환이 복잡한 화면
- 장애가 나면 영향 범위 큰 관리자 기능

> 이제 테스트를 어디에 투자해야 하는지 봤다. 그다음은 프론트엔드 테스트에서 항상 나오는 mock이다.

### 면접 압축 답변

- **10초**: 테스트 ROI는 단순 개수보다, 깨졌을 때 영향이 큰지, 변경이 잦은지, 수동 검증 비용이 큰지를 기준으로 판단합니다.
- **30초**: 테스트 ROI는 단순 개수나 커버리지보다, 깨졌을 때 영향이 큰지, 변경이 잦은지, 수동 검증 비용이 큰지를 기준으로 판단하는 게 좋습니다. 그래서 인증, 업로드, 저장처럼 실패 비용이 큰 흐름에 우선 투자하는 편이 더 실용적입니다. 커버리지 100%여도 중요한 시나리오가 빠지면 의미가 없습니다.

## 121. mock은 왜 유용하면서도 동시에 위험할 수 있나

mock은 테스트에서 매우 자주 쓴다.
하지만 mock을 많이 쓴다고 무조건 좋은 테스트가 되는 건 아니다.

### 핵심 개념

mock은 외부 의존성을 실제 구현 대신 제어 가능한 가짜로 대체하는 방식이다.

비슷한 용어가 많아서 구분하면:

- **mock**: 호출 여부와 인자를 검증할 수 있는 가짜 객체
- **stub**: 정해진 응답만 돌려주는 가짜 (검증 없음)
- **spy**: 실제 구현은 유지하면서 호출을 관찰

실무에서는 이 셋을 통칭해서 "mock"이라고 부르는 경우가 많다.

### 코드 예시

Jest에서 모듈을 mock하는 패턴:

```ts
// api/fetchUser.ts
export async function fetchUser(id: string) {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}
```

```ts
// api/fetchUser.test.ts
import { fetchUser } from "./fetchUser";

// global fetch를 mock (afterEach에서 반드시 복원)
beforeEach(() => {
  jest.spyOn(global, "fetch");
});
afterEach(() => {
  jest.restoreAllMocks();
});

describe("fetchUser", () => {
  it("유저 데이터를 반환한다", async () => {
    (fetch as jest.Mock).mockResolvedValueOnce({
      json: async () => ({ id: "1", name: "홍길동" }),
    });

    const user = await fetchUser("1");

    expect(fetch).toHaveBeenCalledWith("/api/users/1");
    expect(user).toEqual({ id: "1", name: "홍길동" });
  });

  it("네트워크 에러를 전파한다", async () => {
    (fetch as jest.Mock).mockRejectedValueOnce(new Error("Network Error"));

    await expect(fetchUser("1")).rejects.toThrow("Network Error");
  });
});
```

### 왜 이게 중요한가

mock은 아래 장점을 준다.

- 테스트를 더 빠르고 안정적으로 만든다
- 외부 실패 요인을 줄인다
- 특정 시나리오(에러, 타임아웃 등)를 재현하기 쉽다
- 네트워크나 시간 흐름을 통제할 수 있다

하지만 동시에 위험도 있다.

- 실제 구현과 너무 멀어진다
- 내부 구현에 과하게 결합된다
- "테스트는 통과하는데 실제론 깨지는" 상태가 될 수 있다

### 자주 하는 오해

- mock을 많이 쓸수록 테스트가 좋아진다 → 틀림
- mock이 없으면 테스트를 못 한다 → 수준을 조절하면 된다
- 모든 외부 의존성을 다 mock 해야 한다 → 필요에 따라 다르다

### 실무 연결

좋은 기준은 이렇다.

- 진짜로 느리거나 불안정한 외부 의존성만 mock
- 내부 구현보다 사용자 관찰 가능한 결과를 우선 검증
- API 계약은 너무 과도하게 가짜로 왜곡하지 않기

> 이제 mock의 장단점을 봤다. 그다음은 프론트엔드에서 자주 나오는 API mocking과 실제 서버 사용의 균형이다.

### 면접 압축 답변

- **10초**: mock은 외부 의존성을 가짜로 바꿔 테스트를 빠르고 안정적으로 만들지만, 과도하면 실제와 멀어질 수 있습니다.
- **30초**: mock은 외부 의존성을 제어 가능한 가짜로 바꿔 테스트를 더 빠르고 안정적으로 만드는 데 유용합니다. 하지만 과도하게 사용하면 실제 구현과 멀어져서 테스트는 통과하지만 실제 앱은 깨지는 상황이 생길 수 있으므로, 필요한 경계에서만 신중하게 써야 합니다. mock, stub, spy의 차이도 알아두면 좋습니다.

## 122. API mocking은 언제 좋고, 언제 실제 서버가 더 낫나

프론트엔드 테스트에선 API를 어떻게 다룰지가 매우 중요하다.
여기서도 무조건 한쪽이 정답은 아니다.

### 핵심 개념

API mocking은 테스트를:

- 더 빠르게
- 더 안정적으로
- 더 재현 가능하게

만드는 데 유리하다.

반면 실제 서버나 실제 계약 기반 테스트는:

- 실제 응답 형태와 더 가깝고
- 계약 불일치를 더 잘 드러낼 수 있다.

### 코드 예시

MSW(Mock Service Worker)를 사용한 API mocking:

```ts
// mocks/handlers.ts
import { http, HttpResponse } from "msw";

export const handlers = [
  // 주의: 구체적인 경로를 와일드카드보다 먼저 선언해야 한다
  http.get("/api/users/not-found", () => {
    return new HttpResponse(null, { status: 404 });
  }),

  http.get("/api/users/:id", ({ params }) => {
    return HttpResponse.json({
      id: params.id,
      name: "홍길동",
      email: "hong@example.com",
    });
  }),
];
```

```ts
// mocks/server.ts
import { setupServer } from "msw/node";
import { handlers } from "./handlers";

export const server = setupServer(...handlers);
```

```tsx
// components/UserProfile.test.tsx
import { render, screen, waitFor } from "@testing-library/react";
import { server } from "../mocks/server";
import { UserProfile } from "./UserProfile";

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

it("유저 정보를 불러와 표시한다", async () => {
  render(<UserProfile userId="1" />);

  await waitFor(() => {
    expect(screen.getByText("홍길동")).toBeInTheDocument();
  });
});
```

MSW는 네트워크 레벨에서 요청을 가로채기 때문에, fetch나 axios 같은 HTTP 클라이언트를 직접 mock하는 것보다 실제 환경에 더 가깝다. 브라우저와 Node.js 모두에서 쓸 수 있어 integration test와 E2E test 모두에 활용 가능하다.

### 왜 이게 중요한가

이 균형이 중요하지 않으면:

- 전부 mock으로 해서 현실과 멀어지거나
- 전부 실제 서버로 해서 테스트가 느리고 불안정해질 수 있다

### 자주 하는 오해

- API는 무조건 mock해야 테스트가 빠르다 → 핵심 계약 검증은 실제에 가까울수록 좋을 수 있다
- 실제 서버만 써야 믿을 수 있다 → 불안정성과 속도 문제가 생길 수 있다
- mocking은 임시 대응이다 → MSW처럼 전략적으로 잘 쓰면 강력하다

### 실무 연결

보통 이렇게 나누면 좋다.

- unit / integration → MSW 등으로 mocking 적극 활용
- E2E 핵심 시나리오 → 실제에 가까운 환경(스테이징 서버 등) 검토
- 계약 검증 → 응답 shape와 오류 케이스를 현실적으로 반영

> 이제 mock과 실제 환경의 균형을 봤다. 그다음은 테스트를 무너뜨리는 대표 문제인 flaky test다.

### 면접 압축 답변

- **10초**: API mocking은 unit과 integration에서 적극 활용하되, 핵심 E2E나 계약 검증에선 실제에 가까운 환경을 유지하는 균형이 중요합니다.
- **30초**: API mocking은 테스트를 빠르고 안정적으로 만들지만, 실제 응답 계약과 너무 멀어질 수 있습니다. 그래서 unit과 integration에선 MSW 같은 도구로 적극 활용하되, 핵심 E2E 시나리오나 계약 검증에선 실제에 가까운 환경을 일부 유지하는 식으로 균형을 잡는 것이 중요합니다.

## 123. flaky test는 왜 팀 신뢰를 무너뜨리나

테스트가 실패하는 것보다 더 나쁜 경우가 있다.
가끔은 실패하고, 가끔은 통과하는 테스트다.
이게 flaky test다.

### 핵심 개념

flaky test는 코드가 실제로 바뀌지 않았는데도
환경, 타이밍, 비결정성 때문에 결과가 들쭉날쭉한 테스트다.

### 왜 이게 중요한가

flaky test가 위험한 이유는
테스트 시스템 전체에 대한 신뢰를 깎기 때문이다.

사람들이 이렇게 생각하기 시작한다.

- "또 테스트 시스템 문제겠지"
- "재실행하면 통과할 거야"
- "빨간불이어도 무시하자"

이 상태가 되면 자동화 가치가 급격히 떨어진다.

### 자주 하는 원인

- 비동기 타이밍 불안정 (가장 흔함)
- 네트워크 의존
- 환경 초기화 불충분 (이전 테스트의 상태가 남아 있음)
- 전역 상태 오염
- 랜덤 값
- 시간 의존 (`Date.now()`, 타임존 차이 등)
- 과도한 `setTimeout` 기반 대기

### 코드 예시: flaky를 만드는 안티패턴

```tsx
// BAD: setTimeout으로 대충 기다리기
it("로딩 후 데이터가 표시된다", async () => {
  render(<UserList />);

  // 이 시간이 환경마다 다르다 - flaky의 원인
  await new Promise((r) => setTimeout(r, 500));

  expect(screen.getByText("홍길동")).toBeInTheDocument();
});
```

```tsx
// GOOD: 상태 변화를 명시적으로 기다리기
it("로딩 후 데이터가 표시된다", async () => {
  render(<UserList />);

  // 요소가 나타날 때까지 기다린다
  await waitFor(() => {
    expect(screen.getByText("홍길동")).toBeInTheDocument();
  });
});
```

```tsx
// GOOD: findBy 사용 (waitFor + getBy 축약)
it("로딩 후 데이터가 표시된다", async () => {
  render(<UserList />);

  expect(await screen.findByText("홍길동")).toBeInTheDocument();
});
```

### 자주 하는 오해

- flaky는 어쩔 수 없다 → 원인을 추적하면 대부분 줄일 수 있다
- E2E라서 원래 flaky하다 → 더 신중한 설계가 필요하다
- 한두 개쯤은 괜찮다 → 누적되면 팀 전체가 테스트를 무시하기 시작한다

### 실무 연결

flaky를 줄이려면:

- 시간 기반 대기(`setTimeout`)를 상태 기반 대기(`waitFor`, `findBy`)로 교체
- 테스트 간 독립성 보장 (각 테스트가 자체 초기화)
- 랜덤/시간/네트워크를 `jest.useFakeTimers()`, MSW 등으로 통제
- data-testid 같은 안정적인 셀렉터 사용
- CI에서 flaky test를 별도 추적하고 격리

> 이제 테스트가 왜 흔들리는지 봤다. 그다음은 비동기 UI 테스트에서 특히 자주 하는 실수다.

### 면접 압축 답변

- **10초**: flaky test는 코드 변화 없이 결과가 들쭉날쭉한 테스트로, 누적되면 팀이 테스트 결과를 믿지 않게 됩니다.
- **30초**: flaky test는 실제 코드 변화와 무관하게 통과와 실패가 들쭉날쭉한 테스트입니다. 이런 테스트가 늘어나면 팀이 테스트 결과를 믿지 않게 되어 자동화 전체 가치가 떨어지므로, 비동기 타이밍 제어와 환경 초기화, 테스트 독립성 보장을 더 엄격하게 관리해야 합니다. 가장 흔한 원인은 setTimeout 기반 대기이며, waitFor나 findBy로 교체해야 합니다.

## 124. 비동기 UI 테스트는 왜 특히 조심해야 하나

프론트엔드 테스트에서 비동기 UI는 가장 흔한 함정 중 하나다.
사용자 이벤트 이후에 바로 DOM이 바뀌지 않는 경우가 많기 때문이다.

### 핵심 개념

비동기 UI 테스트에선 아래가 엮인다.

- 이벤트 발생
- state 업데이트
- 비동기 요청
- 렌더 재반영
- loading / success / error 상태 전환

즉 테스트도 단순 즉시 검증이 아니라
**상태가 바뀌는 과정을 기다리며 검증**해야 한다.

### 코드 예시: 비동기 흐름 전체를 테스트하기

```tsx
// components/AsyncButton.tsx
// saveData는 외부에서 주입하거나 별도 모듈에서 import한다고 가정
async function saveData() {
  await fetch("/api/save", { method: "POST" });
}

export function AsyncButton() {
  const [status, setStatus] = useState<"idle" | "loading" | "done">("idle");

  const handleClick = async () => {
    setStatus("loading");
    await saveData();
    setStatus("done");
  };

  return (
    <div>
      <button onClick={handleClick} disabled={status === "loading"}>
        {status === "loading" ? "저장 중..." : "저장"}
      </button>
      {status === "done" && <p>저장 완료!</p>}
    </div>
  );
}
```

```tsx
// components/AsyncButton.test.tsx
import { render, screen, waitFor } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { AsyncButton } from "./AsyncButton";

it("클릭하면 로딩 → 완료 순서로 전환된다", async () => {
  render(<AsyncButton />);

  // 1. 초기 상태 확인
  expect(screen.getByText("저장")).toBeInTheDocument();

  // 2. 클릭
  await userEvent.click(screen.getByText("저장"));

  // 3. 로딩 상태 확인
  expect(screen.getByText("저장 중...")).toBeInTheDocument();

  // 4. 완료 상태까지 기다린다
  await waitFor(() => {
    expect(screen.getByText("저장 완료!")).toBeInTheDocument();
  });
});
```

핵심은 각 단계를 순서대로, **적절한 기다림과 함께** 검증하는 것이다.

### 왜 이게 중요한가

이걸 모르고 바로 assert 하면:

- 로컬에선 통과
- CI에선 실패
- 재실행하면 또 통과

같은 flaky가 생기기 쉽다.

### 자주 하는 오해

- 클릭 후 바로 요소가 나와야 한다 → 비동기라면 기다려야 할 수 있다
- `setTimeout`으로 대충 기다리면 된다 → 불안정하다
- 비동기 테스트는 원래 flaky하다 → 기다림 조건을 더 정확히 잡으면 안정적으로 만들 수 있다

### React Testing Library의 대기 패턴 정리

| 패턴 | 용도 |
|------|------|
| `getBy` | 즉시 존재해야 하는 요소 (동기) |
| `queryBy` | 존재하지 않는 것을 확인할 때 (null 반환) |
| `findBy` | 비동기로 나타나는 요소를 기다릴 때 |
| `waitFor` | 임의의 비동기 조건을 기다릴 때 |
| `waitForElementToBeRemoved` | 로딩 스피너 등이 사라지길 기다릴 때 |

### 실무 연결

좋은 비동기 테스트 감각은 이렇다.

- 시간 자체를 기다리지 말고 **상태 변화 조건을 기다린다**
- "1초 뒤"보다 "이 요소가 나타날 때까지"
- "setTimeout"보다 "이 로딩이 사라질 때까지"
- `act()` 경고가 나면 비동기 업데이트를 제대로 기다리지 않은 신호다

> 이제 테스트를 안정적으로 작성하는 기본 감각을 봤다. 그다음은 실제 프론트엔드 특수 UI인 DnD, 업로드, 세션 만료 같은 시나리오 테스트다.

### 면접 압축 답변

- **10초**: 비동기 UI 테스트는 시간 기반 대기 대신 상태 변화 조건 기반으로 검증해야 안정적입니다.
- **30초**: 비동기 UI 테스트는 이벤트 이후 DOM이 바로 바뀌지 않을 수 있기 때문에, setTimeout으로 대충 기다리는 대신 waitFor나 findBy처럼 상태 변화 조건을 기준으로 검증해야 안정적입니다. 그렇지 않으면 로컬에선 통과하지만 CI에선 실패하는 flaky test가 되기 쉽습니다.

## 125. 복잡한 프론트엔드 시나리오는 무엇을 특별히 테스트해야 하나

프론트엔드엔 단순 폼 외에도
테스트하기 까다로운 시나리오가 많다.

### 대표 시나리오

- DnD (Drag and Drop)
- 파일 업로드
- 차트 상호작용
- 세션 만료
- 네트워크 재시도
- optimistic update
- 모달/포커스 트랩
- 무한 스크롤

### 왜 이게 중요한가

이런 시나리오는 단순 함수 정확성보다
**사용자 흐름과 상태 전이, 예외 상황**이 중요하다.

예를 들어:

- 업로드 성공만이 아니라 실패와 재시도
- 세션 만료 후 재로그인 유도
- DnD 결과 저장 여부
- optimistic update 후 서버 실패 시 롤백

같은 걸 봐야 한다.

### 코드 예시: optimistic update 테스트

```tsx
// components/LikeButton.test.tsx
it("좋아요 클릭 시 즉시 반영 후, 서버 실패하면 롤백된다", async () => {
  // 서버 실패를 시뮬레이션
  server.use(
    http.post("/api/like", () => {
      return new HttpResponse(null, { status: 500 });
    })
  );

  render(<LikeButton initialCount={10} />);

  await userEvent.click(screen.getByRole("button", { name: "좋아요" }));

  // optimistic update: 즉시 11로 표시
  expect(screen.getByText("11")).toBeInTheDocument();

  // 서버 실패 후 롤백: 다시 10으로 돌아감
  await waitFor(() => {
    expect(screen.getByText("10")).toBeInTheDocument();
  });
});
```

### 자주 하는 오해

- 복잡하면 수동 테스트만 하면 된다 → 반복 회귀를 놓치기 쉽다
- 복잡한 UI는 테스트하기 어렵다 → 그래서 더 중요한 경우가 많다
- happy path만 자동화하면 충분하다 → 예외 흐름이 더 치명적일 수 있다

### 실무 연결

좋은 테스트는 보통:

- 정상 경로
- 실패 경로
- 취소/되돌리기
- 중복 클릭/중복 요청

까지 같이 본다.

> 이제 테스트 대상 시나리오를 봤다. 그다음은 이런 테스트를 로컬에서만 돌릴 게 아니라 CI에 어떻게 연결할지다.

### 면접 압축 답변

- **10초**: 복잡한 프론트엔드 시나리오는 happy path보다 상태 전이와 예외 흐름을 함께 검증해야 합니다.
- **30초**: 복잡한 프론트엔드 시나리오는 happy path보다 상태 전이와 예외 흐름을 함께 검증해야 합니다. 예를 들어 업로드는 성공뿐 아니라 실패와 재시도, optimistic update는 서버 실패 시 롤백, 세션 만료는 이후 복구 UX까지 테스트해야 실제 안정성이 높아집니다.

## 126. 테스트는 왜 CI에 연결되어야 의미가 커지나

테스트를 로컬에서만 돌리면 개인 습관에 머물기 쉽다.
진짜 힘이 생기는 순간은 CI에 연결될 때다.

### 핵심 개념

CI에서 테스트를 돌린다는 건
**모든 변경이 공통 품질 게이트를 통과해야 한다**는 뜻이다.

즉:

- 누가 올렸든
- 어느 브랜치든
- 같은 기준으로 검증한다

### 왜 이게 중요한가

이걸 통해:

- "내 컴퓨터에선 되는데요"를 줄이고
- 배포 전에 회귀를 더 일찍 막고
- 코드 리뷰와 배포 신뢰도를 높일 수 있다

### 코드 예시: GitHub Actions 기본 테스트 워크플로

```yaml
# .github/workflows/test.yml
name: Test

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - run: npm ci

      - name: Lint & Type Check
        run: |
          npm run lint
          npm run type-check

      - name: Unit & Integration Tests
        run: npm test -- --coverage

      - name: Upload Coverage
        uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/
```

### 자주 하는 오해

- 테스트는 개발자 로컬에서만 꼼꼼히 하면 된다 → 사람마다 편차가 생긴다
- CI에 넣으면 너무 느려진다 → 계층별 전략으로 조절할 수 있다
- E2E까지 매 커밋마다 다 돌려야 한다 → 리스크와 시간에 따라 나눌 수 있다

### 실무 연결

보통 이런 구조가 자연스럽다.

- PR마다: 빠른 unit / integration
- main merge 전후: 더 넓은 테스트
- 배포 직전/야간: 무거운 E2E 세트

> 이제 테스트와 CI가 연결되는 이유를 봤다. 그다음은 어떤 테스트를 어느 단계에서 돌릴지 전략이다.

### 면접 압축 답변

- **10초**: 테스트가 CI에 연결되면 개인 습관이 아니라 팀 공통 품질 게이트가 됩니다.
- **30초**: 테스트가 CI에 연결되어야 의미가 커지는 이유는, 개인 습관이 아니라 팀 공통 품질 게이트가 되기 때문입니다. 이렇게 해야 누가 변경하든 같은 기준으로 검증되고, 회귀를 더 이른 단계에서 막을 수 있습니다. GitHub Actions 같은 도구로 PR마다 자동 실행하는 것이 일반적입니다.

## 127. 어떤 테스트를 어느 CI 단계에서 돌려야 하나

모든 테스트를 모든 시점에 다 돌리는 건 이상적일 수 있지만,
실무에선 속도와 신뢰성의 균형이 중요하다.

### 핵심 개념

CI 테스트 전략은 보통 **빠른 것부터 자주**,
**무거운 것은 더 선택적으로** 배치하는 식으로 간다.

### 예시 전략

#### PR 단위

- lint
- type check
- 빠른 unit test
- 핵심 integration test

#### merge / main 단위

- 더 넓은 integration
- 핵심 E2E smoke test

#### 배포 직전 / 스케줄 실행

- 전체 E2E
- 브라우저 매트릭스 확대
- 느리지만 중요한 회귀 시나리오

### 코드 예시: 단계별 CI 구성

```yaml
# .github/workflows/e2e.yml
name: E2E Tests

on:
  push:
    branches: [main]
  schedule:
    - cron: "0 3 * * *"  # 매일 새벽 3시

jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci
      - run: npx playwright install --with-deps

      - name: Run E2E tests
        run: npx playwright test

      - name: Upload test results
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/
```

이렇게 하면 PR에서는 빠른 테스트만 돌려 피드백 속도를 유지하고, main 병합이나 야간 스케줄에서 무거운 E2E를 돌려 안정성을 확보할 수 있다.

### 왜 이게 중요한가

이 전략이 없으면:

- PR이 너무 느려지거나
- 반대로 중요한 시나리오가 빠지거나
- 모두가 CI를 귀찮아하게 될 수 있다

### 자주 하는 오해

- 빠른 테스트만 돌리면 충분하다 → 핵심 사용자 흐름이 빠질 수 있다
- 느리더라도 전부 PR마다 돌려야 안전하다 → 개발 속도를 과도하게 떨어뜨릴 수 있다
- 테스트 배치는 한 번 정하면 끝이다 → 팀과 서비스 변화에 따라 조정해야 한다

### 실무 연결

좋은 CI 전략은 보통:

- 빠른 피드백
- 핵심 시나리오 보호
- flaky 억제
- 배포 신뢰

를 동시에 노린다.

> 이제 CI 전략까지 봤다. 마지막으로 느린 테스트와 flaky test를 운영 관점에서 어떻게 관리할지 정리한다.

### 면접 압축 답변

- **10초**: 빠른 unit과 integration은 PR마다, 무거운 E2E는 핵심 흐름 위주로 선택적으로 돌리는 전략이 효율적입니다.
- **30초**: CI에서는 모든 테스트를 동일하게 배치하기보다, 빠른 unit과 integration은 PR마다, 무거운 E2E는 핵심 흐름 위주로 main 병합이나 야간 스케줄에 배치하는 전략이 효율적입니다. 핵심은 피드백 속도와 배포 신뢰를 함께 확보하는 것이고, 팀 상황에 따라 지속적으로 조정해야 합니다.

## 128. 느린 테스트와 flaky test는 왜 별도로 관리해야 하나

느린 테스트와 flaky test는 둘 다 테스트 경험을 나쁘게 만든다.
하지만 성격이 다르고, 대응 방식도 다르다.

### 핵심 개념

- **느린 테스트**는 피드백 속도를 떨어뜨린다 → "귀찮음"을 만든다
- **flaky test**는 신뢰 자체를 떨어뜨린다 → "불신"을 만든다

둘 다 CI 품질에 큰 악영향을 주지만, 해결 방향이 다르다.

### 테스트 시스템이 망가지는 전형적인 흐름

1. 테스트가 느려진다
2. 모두가 기다리기 싫어한다
3. flaky가 조금씩 쌓인다
4. 빨간불을 무시하기 시작한다
5. 품질 게이트가 형식화된다

이 단계까지 가면 테스트 시스템을 복구하는 비용이 매우 커진다.

### 느린 테스트 대응

- 실행 시간이 긴 테스트 목록을 CI에서 추출해 모니터링
- 병렬 실행 (`jest --maxWorkers`, Playwright sharding)
- 불필요하게 무거운 셋업 최소화
- 느린 테스트를 별도 job으로 분리

### flaky test 대응

- CI에서 flaky 발생률 추적 (재실행 시 결과가 달라지는 테스트)
- flaky로 확인된 테스트는 즉시 수정하거나 격리(quarantine)
- 격리된 테스트는 기한을 정해 수정 또는 삭제
- 절대 "재실행하면 되니까 괜찮다"로 넘기지 않기

### 자주 하는 오해

- 느린 테스트는 참고 기다리면 된다 → 팀 전체 개발 속도를 갉아먹는다
- flaky는 다시 돌리면 되니 큰 문제 아니다 → 신뢰를 무너뜨린다
- 테스트는 많을수록 좋다 → 유지 가능한 속도와 신뢰가 중요하다

### 면접 압축 답변

- **10초**: 느린 테스트는 피드백 속도를 떨어뜨리고, flaky test는 테스트 자체에 대한 신뢰를 무너뜨리므로, 별도 운영 문제로 관리해야 합니다.
- **30초**: 느린 테스트는 피드백 속도를 떨어뜨리고, flaky test는 테스트 시스템 자체에 대한 신뢰를 무너뜨립니다. 둘은 단순 불편함이 아니라 CI 품질을 훼손하는 별도 운영 문제입니다. 느린 테스트는 병렬화와 분리로, flaky test는 추적과 격리로 지속적으로 관리해야 합니다.

## 129. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 테스트는 단순 버그 탐지보다 변경 자신감을 만든다
- unit / integration / E2E는 서로 다른 실패를 잡는다
- 프론트엔드에선 integration test가 특히 중요하다
- E2E는 가장 비싸지만 가장 현실적인 사용자 흐름을 검증한다
- 테스트 ROI는 실패 비용과 변경 빈도로 판단해야 한다
- mock은 유용하지만 과도하면 현실과 멀어진다
- API mocking과 실제 서버 사용은 균형이 필요하다
- flaky test는 팀 신뢰를 무너뜨린다
- 비동기 UI 테스트는 기다림 조건을 더 명확히 잡아야 한다
- 복잡한 프론트엔드 시나리오는 happy path 외 예외 흐름도 테스트해야 한다
- 테스트는 CI에 연결될 때 팀 공통 품질 게이트가 된다
- 테스트는 단계별로 적절히 배치해야 피드백 속도와 신뢰를 함께 얻을 수 있다

즉,
이번 챕터는 단순 테스트 도구 사용법이 아니라,
**"어떤 실패를 어디서 잡고, 어떤 자신감을 어떤 비용으로 확보할 것인가"**를 배우는 챕터다.

# 이 챕터를 읽은 뒤 자연스럽게 이어질 다음 질문들

1. CI/CD는 테스트와 어떻게 이어져야 하나?
2. GitHub Actions 파이프라인은 어떤 흐름으로 설계하면 좋나?
3. Docker와 Nginx, PM2, EC2는 각각 어떤 역할을 하나?
4. 환경변수와 배포 환경 분리는 왜 중요한가?
5. 롤백, 로그, 메트릭, 알림은 운영 안정성과 어떻게 연결되나?

# 사용법

이 챕터는 아래 순서로 보면 좋다.

1. 테스트의 목적을 버그 탐지보다 변경 자신감으로 잡는다.
2. unit / integration / E2E를 실패 유형 기준으로 구분한다.
3. mock, API mocking, flaky를 통해 테스트 품질 감각을 세운다.
4. 복잡한 프론트엔드 시나리오와 비동기 테스트 주의점을 익힌다.
5. 마지막에 CI 연결과 단계별 테스트 배치 전략으로 운영 관점까지 묶는다.

이렇게 보면 테스트가 단순 확인 작업이 아니라, 팀이 안전하게 바꾸고 배포할 수 있게 만드는 구조라는 감각이 더 선명해진다.
