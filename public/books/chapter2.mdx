# 챕터 2. 값 모델에서 불변성과 React 리렌더링까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **자바스크립트에서 값이 어떻게 저장되고 비교되는지 이해하면, React가 왜 어떤 변경에는 반응하고 어떤 변경에는 둔감한지도 설명할 수 있다.**

## 11. 원시값과 참조값은 왜 React 이해의 출발점인가

앞 챕터에서는 함수가 값을 어떻게 기억하는지를 봤다.
이제는 방향을 조금 바꿔야 한다.

React에서 상태 업데이트와 리렌더링을 제대로 이해하려면, 먼저 자바스크립트가 값을 어떤 방식으로 다루는지부터 봐야 한다.
특히 **원시값**과 **참조값**의 차이를 모르면, 불변성도 애매하고, 리렌더링 최적화도 감으로만 하게 된다.

### 핵심 개념

- 원시값은 숫자, 문자열, 불리언, `null`, `undefined`, `symbol`, `bigint`처럼 **값 자체**로 비교되는 타입이다.
- 객체, 배열, 함수는 참조값이다.
- 원시값은 변수에 대입하면 값이 복사된다.
- 참조값은 변수에 대입하면 내부 데이터 전체가 아니라 **같은 대상을 가리키는 참조**가 복사된다.

```ts
// 원시값: 값 자체가 복사
let a = 10;
let b = a;
b = 20;
console.log(a); // 10 — a는 영향 없음

// 참조값: 참조가 복사
const obj1 = { name: "park" };
const obj2 = obj1;
obj2.name = "kim";
console.log(obj1.name); // "kim" — 같은 객체를 가리키므로 원본도 바뀜
```

### 왜 이게 중요한가

React는 상태를 바꿨을 때 "무언가 달라졌는지"를 판단해야 한다.
이때 React는 `Object.is()`를 사용해서 이전 값과 다음 값을 비교한다.

- 원시값은 `Object.is(10, 10) === true`처럼 값으로 비교되므로 직관적이다.
- 참조값은 `Object.is(prevObj, nextObj)`로 비교하므로, 내부가 바뀌었어도 **같은 참조면 변화 없음**으로 판단한다.

```ts
const prev = { count: 1 };
const next = prev;
next.count = 2;

Object.is(prev, next); // true — 같은 참조이므로 React는 "변화 없음"으로 판단
```

즉 원시값/참조값 이해는 단순 JS 기초가 아니라,
곧바로 아래 질문들로 이어진다.

- 왜 객체 상태를 직접 수정하면 안 되는가
- 왜 `React.memo`가 참조 안정성에 민감한가
- 왜 `useMemo`, `useCallback`의 dependency가 중요한가

### 자주 하는 오해

- 객체를 변수에 대입하면 객체 전체가 복사된다 → 틀림. 참조만 복사된다.
- 배열을 새 변수에 넣었으니 독립적인 배열이다 → 틀림. 같은 배열을 가리킨다.
- React는 객체 내부까지 자동으로 깊게 비교한다 → 틀림. 기본적으로 `Object.is()`로 참조만 비교한다.

### 면접 압축 답변

- **10초**: 원시값은 값 자체로 비교되고, 참조값은 같은 대상을 가리키는지로 비교됩니다. React는 `Object.is`로 상태 변경을 판단하기 때문에 이 차이가 중요합니다.
- **30초**: 원시값은 복사하면 독립적인 값이 되지만, 객체나 배열 같은 참조값은 같은 대상을 가리키는 참조가 복사됩니다. React의 `useState`는 `Object.is()`로 이전 값과 다음 값을 비교해서 리렌더 여부를 결정하기 때문에, 참조가 유지되면 내부가 바뀌어도 변화를 감지하지 못하고, 반대로 새 참조를 만들면 변경을 명확하게 표현할 수 있습니다.

> 이제 참조값이 복사될 때 실제로 무슨 일이 생기는지 봤으니, 다음에는 얕은 복사와 깊은 복사가 왜 다른지를 봐야 한다.

## 12. 얕은 복사와 깊은 복사는 왜 구분해야 하나

원시값과 참조값을 알았다면, 이제 "복사"라는 말이 실제로 무엇을 뜻하는지 더 정확히 봐야 한다.

많은 사람이 `...spread`나 `Object.assign`을 쓰면 객체가 완전히 새로 복사됐다고 생각한다.
하지만 그건 대부분 얕은 복사다.

### 핵심 개념

- **얕은 복사**는 바로 한 단계까지만 새로 복사한다. 중첩된 객체나 배열 안쪽은 여전히 기존 참조를 공유한다.
- **깊은 복사**는 내부 중첩 구조까지 전부 독립적으로 복사하는 방식이다.
- 대표적인 얕은 복사 방법: `{ ...obj }`, `[...arr]`, `Object.assign({}, obj)`
- 대표적인 깊은 복사 방법: `structuredClone(obj)`, `JSON.parse(JSON.stringify(obj))`

### 예시

```ts
const user = {
  name: "park",
  profile: {
    age: 30,
  },
};

// 얕은 복사
const copied = { ...user };
copied.profile.age = 40;

console.log(user.profile.age); // 40 — 중첩 객체는 같은 참조를 공유

// 깊은 복사 — structuredClone (모던 브라우저/Node 17.0+)
const deepCopied = structuredClone(user);
deepCopied.profile.age = 50;

console.log(user.profile.age); // 40 — 완전히 독립적인 복사본
```

겉 객체는 새로 만들어졌지만, profile은 여전히 같은 참조를 보고 있기 때문에 원본도 같이 바뀐다.

### 왜 이게 중요한가

이걸 모르고 상태를 업데이트하면,
겉보기에는 새 객체를 만든 것 같아도 내부 중첩 데이터는 원본을 같이 건드릴 수 있다.

```tsx
// 흔한 실수: 얕은 복사로 중첩 상태 업데이트
const [user, setUser] = useState({
  name: "park",
  settings: { theme: "dark", lang: "ko" },
});

// 잘못된 방법 — settings 내부를 직접 변경
const updated = { ...user };
updated.settings.theme = "light"; // 원본 user.settings도 바뀜!
setUser(updated);

// 올바른 방법 — 변경 경로를 따라 새 참조 생성
setUser((prev) => ({
  ...prev,
  settings: { ...prev.settings, theme: "light" },
}));
```

그 결과:

- 예측하지 못한 값 변경이 생기고
- 디버깅이 어려워지고
- React 최적화도 꼬일 수 있다

### 자주 하는 오해

- spread 문법을 쓰면 깊은 복사다 → 틀림. 1단계만 복사된다.
- `JSON.stringify`/`parse`면 모든 깊은 복사 문제가 해결된다 → 틀림. 함수, `Date`, `Map`, `Set`, `undefined`, 순환 참조 등은 깨지거나 사라진다.
- `structuredClone`이면 모든 게 된다 → 대부분 되지만 함수, DOM 노드, `Symbol` 등은 복사할 수 없다.
- 깊은 복사가 항상 정답이다 → 틀림. 비용이 크고 구조를 과하게 복제할 수 있다.

### 실무 연결

실무에서는 무조건 깊은 복사를 하는 게 아니라,
**정말 바뀌는 경로에만 필요한 수준으로 새 참조를 만들고 나머지는 유지하는 구조적 공유(structural sharing)**가 더 중요하다. Immer 같은 라이브러리가 이 패턴을 쉽게 만들어준다.

```ts
import { produce } from "immer";

const nextUser = produce(user, (draft) => {
  draft.settings.theme = "light"; // 직접 수정처럼 보이지만 내부적으로 구조적 공유
});
```

### 면접 압축 답변

- **10초**: 얕은 복사는 1단계까지만 새로 만들고, 중첩 객체는 참조를 공유합니다. React 상태 업데이트에서는 변경 경로를 따라 참조를 새로 만들어야 합니다.
- **30초**: 얕은 복사는 `spread`나 `Object.assign`으로 1단계만 새로 복사하고 내부 중첩 참조는 공유합니다. 깊은 복사는 `structuredClone` 등으로 내부까지 독립 복사합니다. React 상태 업데이트에서는 무조건 깊은 복사보다, 변경이 일어난 경로만 새 참조로 만들고 나머지는 유지하는 구조적 공유가 더 효율적입니다.

> 이제 "복사"가 생각보다 단순하지 않다는 걸 봤다. 다음엔 왜 React가 이런 복사 전략을 좋아하는지, 즉 불변성으로 넘어간다.

## 13. 불변성은 왜 React에서 그렇게 중요할까

이제 자주 듣는 말이 나온다.

"React에서는 불변성이 중요하다."

그런데 이 말을 단순 암기로 들고 가면 약하다.
중요한 건 불변성이 **왜** 필요한지다.

### 핵심 개념

불변성은 값을 절대 바꾸지 않는다는 종교적인 규칙이 아니라,
**기존 값을 직접 바꾸지 않고 새 값으로 표현하는 방식**이다.

즉:

- 기존 객체를 mutate하지 않고
- 바뀐 부분을 반영한 새 객체를 만들어
- "변경이 일어났다"는 사실을 참조 변화로 표현하는 것

이다.

### 왜 이게 중요한가

React의 `useState`와 `useReducer`는 상태가 바뀌었는지를 `Object.is()`로 판단한다. 이것은 깊은 비교가 아니라 참조 비교다.

```tsx
function Counter() {
  const [user, setUser] = useState({ count: 0 });

  const handleBadClick = () => {
    // mutate: Object.is(prev, next)가 true이므로 리렌더 안 됨
    user.count += 1;
    setUser(user); // 같은 참조를 넘기므로 React는 변화 없다고 판단
  };

  const handleGoodClick = () => {
    // immutable: 새 참조를 만들므로 Object.is(prev, next)가 false
    setUser((prev) => ({ ...prev, count: prev.count + 1 }));
  };

  return <div>{user.count}</div>;
}
```

불변성을 지키면:

- **변경 감지가 확실해진다**: 참조가 바뀌었으면 변했고, 안 바뀌었으면 안 변한 것
- **디버깅이 쉬워진다**: 이전 상태가 보존되므로 시간 여행 디버깅이 가능
- **메모이제이션이 자연스러워진다**: `React.memo`, `useMemo` 등이 참조 비교로 동작하므로
- **동시성 모드에서 안전해진다**: React 18의 concurrent features는 렌더 도중 중단/재시작이 가능한데, mutate된 값은 이 과정에서 예측 불가능한 상태가 될 수 있다

### 자주 하는 오해

- 불변성은 무조건 성능을 느리게 만든다 → 항상 그런 건 아니다. 구조적 공유를 쓰면 실제 복사량은 적다.
- mutate만 안 하면 끝이다 → 구조가 너무 깊으면 업데이트 코드가 복잡해질 수 있으므로 Immer 같은 도구를 고려해야 한다.
- React는 mutate해도 결국 다시 렌더되니 상관없다 → `Object.is()`로 같은 참조면 리렌더 자체가 일어나지 않는다. 겉으로 동작하는 것처럼 보이는 경우는 다른 이유로 리렌더가 트리거된 것이다.

### 면접 압축 답변

- **10초**: React는 `Object.is()`로 상태 변경을 감지하므로, 기존 객체를 mutate하면 변화를 인식하지 못합니다. 새 참조로 변경을 표현하는 것이 불변성의 핵심입니다.
- **30초**: React의 `useState`는 `Object.is()`로 이전 값과 다음 값을 비교해서 리렌더 여부를 결정합니다. 기존 객체를 직접 수정하면 참조가 그대로이므로 변화를 감지하지 못합니다. 불변성을 지키면 변경 감지가 확실해지고, 메모이제이션과 시간 여행 디버깅, React 18의 concurrent features에서도 안전합니다.

> 불변성이 왜 필요한지는 보였다. 다음엔 React가 실제로 "같다/다르다"를 어떻게 판단하는지, 즉 참조 동일성으로 넘어간다.

## 14. 참조 동일성은 왜 리렌더링과 최적화의 핵심이 되나

불변성이 중요한 이유를 한 단계 더 깊게 들어가면 결국 참조 동일성 문제다.

### 핵심 개념

참조 동일성은 두 값이 내용이 같아 보이느냐가 아니라,
**정확히 같은 대상을 가리키느냐**를 보는 개념이다.

```ts
const a = { x: 1 };
const b = { x: 1 };
console.log(Object.is(a, b)); // false — 구조가 같아도 다른 객체

const c = a;
console.log(Object.is(a, c)); // true — 같은 참조
```

a와 b는 구조가 같아 보여도 서로 다른 객체다.
반면 c는 같은 참조다.

참고로 `Object.is()`는 `===`와 거의 같지만 두 가지 차이가 있다:

```ts
Object.is(NaN, NaN);   // true  (===는 false)
Object.is(+0, -0);     // false (===는 true)
```

React는 내부적으로 `===`가 아니라 `Object.is()`를 사용한다.

### 왜 이게 중요한가

React에서 다음 상황들이 전부 참조 동일성과 연결된다.

- `useState`의 bailout: `Object.is(prevState, nextState)`가 true면 리렌더를 건너뜀
- `useEffect`/`useMemo`/`useCallback`의 dependency 비교: 배열의 각 항목을 `Object.is()`로 비교
- `React.memo`의 props 비교: 각 prop을 `Object.is()`로 얕게 비교
- Context의 value 비교: `Object.is(prevValue, nextValue)`로 비교

```tsx
function Parent() {
  const [count, setCount] = useState(0);

  // 매 렌더마다 새 객체 생성 — 참조가 항상 다름
  const style = { color: "red" };

  // 매 렌더마다 새 함수 생성 — 참조가 항상 다름
  const handleClick = () => console.log("click");

  return <Child style={style} onClick={handleClick} />;
}

// React.memo를 써도 style과 onClick이 매번 새 참조이므로 memo 효과 없음
const Child = React.memo(({ style, onClick }) => {
  return <button style={style} onClick={onClick}>Click</button>;
});
```

즉 React 최적화에서 "같다"는 말은 종종
깊은 의미 비교가 아니라
**참조가 유지되었는가**에 더 가깝다.

### 자주 하는 오해

- 내용이 같으면 React도 같은 값으로 본다 → 틀림. `Object.is()`는 참조를 비교한다.
- 새 객체를 만들었지만 필드가 같으니 최적화에 영향이 없다 → 틀림. 참조가 바뀌면 memo, useEffect 등이 다시 실행될 수 있다.
- 참조 안정성이 중요하니 무조건 모든 걸 memo 해야 한다 → 틀림. memo 자체도 비교 비용이 있다.

### 면접 압축 답변

- **10초**: React는 `Object.is()`로 참조 동일성을 비교합니다. 내용이 같아도 새 객체면 "변경됨"으로 판단하므로, 참조 안정성이 최적화의 핵심입니다.
- **30초**: React의 state bailout, dependency 비교, memo props 비교는 모두 `Object.is()`로 참조 동일성을 확인합니다. 그래서 매 렌더마다 새 객체나 새 함수를 만들면 최적화가 깨질 수 있고, `useMemo`나 `useCallback`으로 참조를 안정시키는 것이 의미를 가집니다.

> React가 내부 값을 세밀하게 다 보는 게 아니라 참조를 중요한 신호로 쓴다는 감이 왔다면, 다음엔 자연스럽게 리렌더링이 언제 발생하는가로 넘어가야 한다.

## 15. React 리렌더링은 정확히 언제 발생하나

앞에서 참조와 변경을 봤다면, 이제 React가 "다시 계산"을 언제 시작하는지 봐야 한다.

### 핵심 개념

React 컴포넌트는 아래 상황에서 다시 렌더된다.

1. **자신의 state가 바뀔 때** — `Object.is()`로 비교해서 다르면 리렌더
2. **부모가 다시 렌더될 때** — 기본적으로 자식도 함께 리렌더
3. **자신이 사용하는 context 값이 바뀔 때** — Provider의 value가 새 참조면 구독자 전부 리렌더
4. **외부 스토어 구독 값이 바뀔 때** — `useSyncExternalStore` 등

중요한 건, **리렌더링은 "다시 계산"이지 곧바로 DOM 변경을 뜻하는 건 아니라는 점**이다.

### 렌더 단계와 커밋 단계

```
setState 호출
    ↓
[렌더 단계] 컴포넌트 함수 재실행 → 새 JSX 생성 → 이전 결과와 비교(재조정)
    ↓
[커밋 단계] 실제로 달라진 DOM 노드만 업데이트 → useLayoutEffect → 브라우저 페인트 → useEffect
```

- **렌더 단계**: 순수 계산. 부수효과 없어야 한다. React 18에서는 이 단계가 중단/재시작될 수 있다.
- **커밋 단계**: 실제 DOM 변경. `useLayoutEffect`는 DOM 변경 직후 동기적으로, `useEffect`는 페인트 이후 비동기적으로 실행된다.

### React 18 자동 배칭 (Automatic Batching)

React 18부터 **모든 상태 업데이트는 자동으로 배칭**된다. 이것은 선택사항이 아니라 기본 동작이다.

```tsx
function handleClick() {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React 18: 두 업데이트가 하나로 배칭 → 리렌더 1회
}

// React 17까지는 이벤트 핸들러 안에서만 배칭됐지만,
// React 18부터는 setTimeout, Promise, 네이티브 이벤트 안에서도 자동 배칭
setTimeout(() => {
  setCount((c) => c + 1);
  setFlag((f) => !f);
  // React 18: 여전히 배칭됨 → 리렌더 1회
  // React 17: 배칭 안 됨 → 리렌더 2회
}, 1000);
```

배칭을 명시적으로 풀고 싶을 때만 `flushSync`를 쓴다.

### 왜 이게 중요한가

많은 사람이 렌더와 커밋을 섞어 생각한다.

- 리렌더는 자주 일어나도 될 수 있다 — 실제 DOM 반영이 적으면 큰 문제가 아니다
- 성능 문제는 "렌더 횟수"보다 **"무엇이 비싼가"**까지 같이 봐야 한다
- 자동 배칭 덕분에 여러 setState를 연속으로 호출해도 리렌더는 한 번만 일어난다

### 자주 하는 오해

- 리렌더가 일어나면 무조건 화면도 많이 바뀐다 → 틀림. 커밋 단계에서 diff가 적으면 DOM 변경은 최소화된다.
- 부모가 렌더돼도 자식은 영향이 없다 → 기본적으로 틀림. 자식도 함께 리렌더된다.
- 렌더 자체가 항상 나쁘다 → 틀림. React의 설계 자체가 "렌더는 저렴하게, 커밋은 필요한 만큼만"이다.
- `setState`를 여러 번 호출하면 그만큼 리렌더된다 → React 18에서는 자동 배칭으로 한 번만 리렌더된다.

### 면접 압축 답변

- **10초**: React는 state 변경, 부모 리렌더, context 변경 시 다시 렌더됩니다. 렌더는 UI 계산이고 실제 DOM 변경은 커밋 단계에서 일어납니다.
- **30초**: React 컴포넌트는 state 변경, 부모 리렌더, context 변경, 외부 스토어 변경으로 다시 렌더됩니다. 렌더 단계는 순수 계산이고, 실제 DOM 변경은 커밋 단계에서 diff 결과를 기준으로 최소한만 일어납니다. React 18부터는 모든 상태 업데이트가 자동 배칭되므로, 여러 setState를 연속 호출해도 리렌더는 한 번만 일어납니다.

> 그렇다면 바로 다음 질문이 나온다. 부모가 리렌더되면 자식은 항상 다시 렌더될까?

## 16. 부모가 리렌더되면 자식도 항상 다시 렌더되나

이 질문은 React 성능 면접에서 정말 자주 나온다.

### 핵심 개념

**기본적으로는 그렇다.**
부모가 다시 렌더되면 자식도 재귀적으로 다시 렌더되는 것이 기본값이다.

다만 렌더를 건너뛸 수 있는 조건이 있다.

1. **React.memo**: props가 이전과 같으면 렌더 건너뜀
2. **children 패턴**: 부모의 state 변경이 children을 새로 만들지 않는 구조

```tsx
// children 패턴으로 불필요한 리렌더 방지
function Layout({ children }) {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount((c) => c + 1)}>{count}</button>
      {children} {/* children은 부모(Layout의 부모)에서 만들어졌으므로 참조 유지 */}
    </div>
  );
}

// 사용
<Layout>
  <ExpensiveComponent /> {/* Layout의 count가 바뀌어도 리렌더 안 됨 */}
</Layout>
```

이 패턴이 동작하는 이유: `children`은 `Layout`의 부모가 만든 JSX이므로, `Layout`의 state가 바뀌어도 `children`의 참조는 변하지 않는다. React는 같은 참조의 엘리먼트는 다시 렌더하지 않는다.

### 왜 이게 중요한가

이걸 모르면 성능 문제가 생겼을 때
"왜 이 자식이 자꾸 렌더되지?"를 설명하기 어렵다.

그리고 반대로,
이걸 너무 단순하게 이해하면 또 이런 실수를 한다.

> 그러면 다 React.memo 씌우면 되겠네?

그건 또 아니다.

### 자주 하는 오해

- 부모가 렌더돼도 자식은 독립적이라 자동으로 안 렌더된다 → 틀림. 기본값은 함께 리렌더다.
- 자식 리렌더는 모두 나쁘다 → 틀림. 렌더 비용이 작으면 문제 없다.
- React.memo를 쓰면 무조건 해결된다 → 틀림. props가 매번 새 참조면 memo 효과가 없다.

### 실무 연결

진짜 중요한 건:

- 자식 렌더 자체가 비싼가
- props가 안정적인가
- memo 비교 비용보다 이득이 큰가
- children 패턴이나 컴포넌트 분리로 구조적으로 해결 가능한가

를 같이 보는 것이다.

### 면접 압축 답변

- **10초**: 기본적으로 부모가 리렌더되면 자식도 함께 다시 렌더됩니다. `React.memo`나 children 패턴으로 건너뛸 수 있습니다.
- **30초**: React의 기본 동작은 부모 리렌더 시 자식도 재귀적으로 리렌더하는 것입니다. `React.memo`로 props가 같으면 건너뛸 수 있고, children을 외부에서 주입하는 패턴으로 구조적으로 리렌더 범위를 줄일 수도 있습니다. 무조건 memo를 붙이는 것보다 렌더 비용과 props 안정성을 같이 고려하는 게 중요합니다.

> 이제 "건너뛸 수 있는 조건"이 궁금해진다. 그게 바로 React.memo다.

## 17. React.memo는 참조 불안정이 왜 문제인지를 보여주는 도구다

### 핵심 개념

React.memo는 **이전 props와 다음 props를 얕게 비교(`Object.is()`)해서 같다면 렌더 결과를 재사용**하려는 도구다.

이 챕터에서 계속 봐온 참조 동일성 이야기가 여기에서 실전으로 연결된다.
매 렌더마다 새 객체, 새 배열, 새 함수를 props로 넘기면 memo의 비교는 항상 실패한다.

```tsx
function Parent() {
  const [count, setCount] = useState(0);

  // memo 효과 없는 패턴: 매번 새 참조
  return (
    <MemoChild
      style={{ color: "red" }}       // 매 렌더마다 새 객체
      items={[1, 2, 3]}               // 매 렌더마다 새 배열
      onClick={() => console.log(count)} // 매 렌더마다 새 함수
    />
  );
}

const MemoChild = React.memo(Child);
```

```tsx
function Parent() {
  const [count, setCount] = useState(0);

  // memo 효과 있는 패턴: 참조 안정화
  const style = useMemo(() => ({ color: "red" }), []);
  const items = useMemo(() => [1, 2, 3], []);
  const handleClick = useCallback(() => {
    console.log(count);
  }, [count]);

  return <MemoChild style={style} items={items} onClick={handleClick} />;
}

const MemoChild = React.memo(Child);
```

즉 memo는 혼자 쓰는 도구가 아니라, **참조 안정성**(useMemo, useCallback)과 함께 봐야 한다.

### 자주 하는 오해

- React.memo는 항상 성능을 올린다 → 틀림. 비교 비용이 렌더 비용보다 클 수 있다.
- props 구조를 안 바꾸고 memo만 감싸면 된다 → 틀림. props 참조가 불안정하면 memo가 매번 실패한다.
- 렌더 자체가 가벼운 컴포넌트도 memo를 씌워야 한다 → 대부분 불필요하다.

### 면접 압축 답변

- **10초**: `React.memo`는 props가 같으면 렌더를 건너뛰는 도구인데, props 참조가 매번 바뀌면 효과가 없으므로 `useMemo`/`useCallback`과 함께 써야 합니다.
- **30초**: `React.memo`는 이전/다음 props를 `Object.is()`로 얕게 비교해서 같으면 렌더를 재사용합니다. 하지만 매 렌더마다 새 객체나 새 함수를 props로 넘기면 비교가 항상 실패합니다. 그래서 `useMemo`와 `useCallback`으로 참조를 안정시키는 것과 함께 사용해야 의미가 있고, 렌더 비용이 낮은 컴포넌트에는 굳이 쓸 필요가 없습니다.

> memo가 참조 안정성에 기대고 있다는 것을 봤다. 이번에는 비슷한 맥락에서 key가 왜 identity 문제인지를 짧게 본다.

## 18. key는 참조 모델 위의 identity 문제다

### 핵심 개념

리스트 렌더링에서 key는 단순 경고 제거용 값이 아니다.
React가 각 요소의 **정체성(identity)**을 구분하기 위한 기준이다.

React는 리스트 변경 시 어떤 항목이 유지되고, 어떤 항목이 새로 생기고, 어떤 항목이 사라졌는지를 판단해야 한다. 이때 key가 불안정하면 상태가 엉뚱한 항목에 붙고, input 값이 섞이고, 불필요한 재생성이 생긴다.

### 예시: index를 key로 쓸 때 생기는 문제

```tsx
function TodoList() {
  const [todos, setTodos] = useState([
    { id: 1, text: "할 일 A" },
    { id: 2, text: "할 일 B" },
    { id: 3, text: "할 일 C" },
  ]);

  const addToFront = () => {
    setTodos([{ id: Date.now(), text: "새 할 일" }, ...todos]);
  };

  return (
    <div>
      <button onClick={addToFront}>맨 앞에 추가</button>
      {todos.map((todo, index) => (
        // index를 key로 쓰면: 맨 앞에 추가할 때 모든 항목의 key가 밀림
        // key=0이었던 "할 일 A"가 key=1로 바뀌면서 상태가 꼬임
        <TodoItem key={index} todo={todo} />

        // 안정적인 id를 key로 쓰면: 각 항목의 identity가 유지됨
        // <TodoItem key={todo.id} todo={todo} />
      ))}
    </div>
  );
}

function TodoItem({ todo }) {
  const [checked, setChecked] = useState(false);
  return (
    <label>
      <input
        type="checkbox"
        checked={checked}
        onChange={() => setChecked(!checked)}
      />
      {todo.text}
    </label>
  );
}
```

`index`를 key로 쓰면 맨 앞에 아이템을 추가했을 때, 기존 아이템들의 index가 모두 1씩 밀린다. React는 key가 같으면 같은 컴포넌트로 인식하므로, 체크박스 상태가 엉뚱한 항목에 붙게 된다.

### 자주 하는 오해

- key는 경고만 없애면 된다 → 틀림. identity를 올바르게 전달해야 한다.
- index를 항상 써도 된다 → 리스트 순서가 바뀌지 않고, 항목에 상태가 없을 때만 괜찮다.
- key는 성능만의 문제다 → 틀림. 상태 정체성 문제이기도 하다.

### 면접 압축 답변

- **10초**: key는 리스트에서 각 요소의 identity를 React에 알려주는 값입니다. 불안정한 key는 상태 혼동과 성능 저하를 유발합니다.
- **30초**: key는 React가 리스트의 각 요소를 식별하는 기준입니다. index를 key로 쓰면 순서가 바뀔 때 상태가 엉뚱한 항목에 붙는 문제가 생기므로, 고유하고 안정적인 id를 쓰는 것이 원칙입니다.

## 19. 값 모델 이해가 상태 설계의 기반이 되는 이유

여기까지 오면 자연스럽게 "상태를 어디에 두고 어떻게 나누는가"라는 질문이 떠오를 것이다. 상태 소유 범위(state ownership), 끌어올리기(lifting state up), Context와 외부 상태관리 도구의 선택 같은 주제가 그것이다. 이 내용은 **챕터 5**에서 본격적으로 다룬다.

다만 챕터 5의 내용을 제대로 이해하려면, 이 챕터에서 다진 값 모델 기초가 전제가 된다. 참조 동일성이 리렌더를 결정하고, 불변성이 변경 감지의 기반이 된다는 것을 모르면, "Context의 value가 바뀌면 왜 구독자 전체가 리렌더되는가"나 "Zustand의 selector가 왜 부분 구독을 가능하게 하는가" 같은 질문에 정확히 답할 수 없다. 값이 어떻게 비교되는지를 먼저 알아야, 상태 설계의 트레이드오프도 구체적으로 판단할 수 있다.

## 다음 챕터로의 연결

이 챕터에서는 자바스크립트의 값 모델 — 원시값과 참조값, 얕은/깊은 복사, 불변성, 참조 동일성 — 이 React의 리렌더링과 최적화에 어떻게 연결되는지를 봤다.

핵심은 하나다: **React는 값의 내용이 아니라 참조가 바뀌었는지를 본다.** 이 원칙 위에서 `Object.is()` 비교, 불변 업데이트, `React.memo`, key identity가 모두 설명된다.

다음 챕터(챕터 3)에서는 방향을 바꿔서 **실행 순서와 비동기 제어**로 넘어간다.

- 동기와 비동기는 정확히 무엇이 다른가?
- 이벤트 루프와 마이크로태스크는 Promise와 어떻게 연결되나?
- 여러 비동기 요청이 겹치면 왜 race condition이 생기나?
- AbortController와 retry/backoff는 어떤 상황에서 같이 고려해야 하나?

값이 어떻게 저장되고 비교되는지를 알았으니, 이제 그 값들이 **언제, 어떤 순서로** 도착하는지를 볼 차례다.
