# 챕터 2. 값 모델에서 불변성과 React 리렌더링까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **자바스크립트에서 값이 어떻게 저장되고 비교되는지 이해하면, React가 왜 어떤 변경에는 반응하고 어떤 변경에는 둔감한지도 설명할 수 있다.**

### 전체 흐름 맵

```
원시값 / 참조값 (값이 어떻게 저장되나)
  → 얕은 복사 / 깊은 복사 (복사하면 실제로 무슨 일이 생기나)
    → 불변성 (왜 직접 바꾸면 안 되나)
      → 참조 동일성 (React는 "같다"를 어떻게 판단하나)
        → 리렌더링 조건 (언제 다시 그리나)
          → 부모-자식 리렌더 (자식은 왜 같이 그려지나)
            → React.memo (건너뛸 수 있는 조건)
              → key (리스트에서 identity를 어떻게 잡나)
                → 상태 설계의 기반으로 연결
```

## 13. 원시값과 참조값은 왜 React 이해의 출발점인가

> **TL;DR** — 원시값은 값 자체로 비교되고, 참조값은 같은 대상을 가리키는지로 비교된다. React는 `Object.is()`로 상태 변경을 판단하기 때문에 이 차이를 모르면 리렌더링과 최적화를 감으로 하게 된다.

앞 챕터에서는 함수가 값을 어떻게 기억하는지를 봤다. 이제는 방향을 바꿔야 한다. React에서 상태 업데이트와 리렌더링을 제대로 이해하려면, 먼저 자바스크립트가 값을 어떤 방식으로 다루는지부터 봐야 한다.

### 핵심 비교

| | 원시값 (Primitive) | 참조값 (Reference) |
|---|---|---|
| **종류** | `number`, `string`, `boolean`, `null`, `undefined`, `symbol`, `bigint` | `object`, `array`, `function`, `Map`, `Set` 등 |
| **저장 위치** | 스택(Stack)에 값 자체 저장 | 힙(Heap)에 실제 데이터, 스택에 참조(주소) 저장 |
| **대입 시** | 값이 복사됨 | 참조(주소)가 복사됨 |
| **비교 시** | 값 자체 비교 | 참조(같은 대상인지) 비교 |
| **변경 시** | 불변 — 새 값을 만들어야 함 | 내부 수정 가능 (mutable) |

### 스택과 힙 — 값이 실제로 어디에 사는가

```
스택 (Stack)                    힙 (Heap)
┌──────────────────┐
│ a = 10           │           ┌─────────────────────┐
│ b = 10           │           │ { name: "park" }    │
│ obj1 = 0x001  ───┼──────────▶│   (주소: 0x001)     │
│ obj2 = 0x001  ───┼──────────▶│                     │
└──────────────────┘           └─────────────────────┘
```

- 원시값 `a`, `b`는 스택에 값 자체가 들어간다. `b = a`를 하면 10이라는 값이 복사된다.
- 참조값 `obj1`은 스택에 **주소(참조)**만 들어가고, 실제 데이터는 힙에 있다.
- `obj2 = obj1`을 하면 **주소가 복사**되므로 같은 힙 데이터를 가리킨다.

```js
// 원시값: 값 자체가 복사
let a = 10;
let b = a;
b = 20;
console.log(a); // 10 — a는 영향 없음

// 참조값: 참조가 복사
const obj1 = { name: "park" };
const obj2 = obj1;
obj2.name = "kim";
console.log(obj1.name); // "kim" — 같은 객체를 가리키므로 원본도 바뀜
```

### 왜 이게 React에서 중요한가

React는 상태를 바꿨을 때 "무언가 달라졌는지"를 판단해야 한다. 이때 `Object.is()`를 사용해서 이전 값과 다음 값을 비교한다.

```js
// 원시값: 직관적
Object.is(10, 10);     // true — 값이 같으면 같다
Object.is("a", "b");   // false

// 참조값: 내용이 아니라 참조를 본다
const prev = { count: 1 };
const next = prev;
next.count = 2;
Object.is(prev, next); // true — 같은 참조이므로 React는 "변화 없음"으로 판단!
```

즉 원시값/참조값 이해는 단순 JS 기초가 아니라, 곧바로 이 질문들로 이어진다:

- 왜 객체 상태를 직접 수정하면 안 되는가
- 왜 `React.memo`가 참조 안정성에 민감한가
- 왜 `useMemo`, `useCallback`의 dependency가 중요한가

### 실무 에피소드 — 이런 상황에서 만난다

> 폼(form)에서 유저 정보 객체를 state로 관리하는데, `user.name = "kim"`으로 직접 수정한 뒤 `setUser(user)`를 호출했더니 화면이 안 바뀌는 버그. React DevTools로 보면 state는 바뀌어 있는데 렌더가 안 된다. 원인은 `Object.is(prev, next)`가 true — 같은 참조를 넘겼기 때문이다. `setUser({ ...user, name: "kim" })`으로 새 객체를 만들면 해결.

### 자주 하는 오해

- 객체를 변수에 대입하면 객체 전체가 복사된다 → **틀림.** 참조(주소)만 복사된다.
- 배열을 새 변수에 넣었으니 독립적인 배열이다 → **틀림.** 같은 배열을 가리킨다.
- React는 객체 내부까지 자동으로 깊게 비교한다 → **틀림.** `Object.is()`로 참조만 비교한다.

### 면접 압축 답변

- **10초**: 원시값은 스택에 값 자체가 저장되어 값으로 비교되고, 참조값은 힙에 데이터가 있고 스택에 참조가 저장되어 참조로 비교됩니다. React는 `Object.is()`로 상태 변경을 판단하므로 이 차이가 중요합니다.
- **30초**: 원시값은 복사하면 독립적인 값이 되지만, 객체나 배열 같은 참조값은 같은 대상을 가리키는 참조가 복사됩니다. React의 `useState`는 `Object.is()`로 이전 값과 다음 값을 비교해서 리렌더 여부를 결정하기 때문에, 참조가 유지되면 내부가 바뀌어도 변화를 감지하지 못하고, 새 참조를 만들면 변경을 명확하게 표현할 수 있습니다.

> 이제 참조값이 복사될 때 실제로 무슨 일이 생기는지 봤으니, 다음에는 얕은 복사와 깊은 복사가 왜 다른지를 봐야 한다.

## 14. 얕은 복사와 깊은 복사는 왜 구분해야 하나

> **TL;DR** — spread(`...`)는 1단계만 새로 복사하는 얕은 복사다. 중첩 객체는 여전히 원본과 참조를 공유한다. React 상태 업데이트에서는 변경 경로를 따라 필요한 만큼만 새 참조를 만드는 게 핵심이다.

많은 사람이 `...spread`나 `Object.assign`을 쓰면 객체가 완전히 새로 복사됐다고 생각한다. 하지만 그건 대부분 얕은 복사다.

### 핵심 개념

```js
const user = {
  name: "park",
  profile: { age: 30 },
};

// 얕은 복사
const copied = { ...user };
copied.profile.age = 40;
console.log(user.profile.age); // 40 — 중첩 객체는 같은 참조를 공유!

// 깊은 복사
const deepCopied = structuredClone(user);
deepCopied.profile.age = 50;
console.log(user.profile.age); // 40 — 완전히 독립적인 복사본
```

```
얕은 복사 후:                        깊은 복사 후:

user ──┬── name: "park"              user ──┬── name: "park"
       └── profile ─── { age: 40 }          └── profile ─── { age: 40 }
                           ▲
copied ─┬── name: "park"              deepCopied ─┬── name: "park"
        └── profile ───────┘                      └── profile ─── { age: 50 }
                                                       (독립된 새 객체)
   참조 공유!
```

### 복사 방법 비교

| 방법 | 깊이 | 함수 | Date/Map/Set | 순환 참조 | undefined | 비고 |
|---|---|---|---|---|---|---|
| `{ ...obj }` / `[...arr]` | 1단계 | - | - | - | - | 가장 흔한 방법 |
| `Object.assign({}, obj)` | 1단계 | - | - | - | - | spread와 동일 |
| `JSON.parse(JSON.stringify())` | 전체 | 사라짐 | 깨짐 | 에러 | 사라짐 | 한계 많음 |
| `structuredClone()` | 전체 | 사라짐 | 복사됨 | 지원 | 유지 | 모던 환경 권장 |
| Immer `produce()` | 변경 경로만 | 유지 | 유지 | - | 유지 | 구조적 공유 |

### Object.freeze — 실수 방지용 얕은 불변

`Object.freeze()`는 객체의 프로퍼티 추가/삭제/변경을 막는다. 하지만 **얕은 수준에서만** 동작한다.

```js
const config = Object.freeze({
  api: "https://api.example.com",
  settings: { timeout: 3000 },
});

// config.api = "new"; // TypeError (strict mode) — 막힘
config.settings.timeout = 5000; // 통과! — 중첩 객체는 freeze 안 됨
```

개발 중 mutation 실수를 잡는 데 유용하지만, 깊은 freeze가 필요하면 재귀적으로 적용하거나 라이브러리를 써야 한다.

### React 상태 업데이트에서의 올바른 복사

```tsx
const [user, setUser] = useState({
  name: "park",
  settings: { theme: "dark", lang: "ko" },
});

// 잘못된 방법 — 얕은 복사 후 중첩 객체 직접 변경
const updated = { ...user };
updated.settings.theme = "light"; // 원본 user.settings도 바뀜!
setUser(updated);

// 올바른 방법 — 변경 경로를 따라 새 참조 생성
setUser((prev) => ({
  ...prev,
  settings: { ...prev.settings, theme: "light" },
}));

// Immer로 간결하게 — 직접 수정처럼 보이지만 내부적으로 구조적 공유
import { produce } from "immer";
setUser(produce((draft) => {
  draft.settings.theme = "light";
}));
```

핵심은 **무조건 깊은 복사가 아니라, 변경이 일어난 경로만 새 참조로 만들고 나머지는 유지하는 구조적 공유(structural sharing)**다. 변경되지 않은 `user.name`은 이전 값을 그대로 참조하므로, 그 부분에 의존하는 컴포넌트는 리렌더를 건너뛸 수 있다.

### 실무 에피소드 — 이런 상황에서 만난다

> 설정 페이지에서 테마를 바꿨는데 언어 설정까지 초기화되는 버그. 원인은 `setSettings({ theme: "light" })`로 덮어써서 기존 `lang` 필드가 사라진 것. spread로 나머지를 유지해야 하는데 빠뜨린 실수. 중첩이 깊은 설정 객체에서는 Immer를 도입해서 이런 실수를 원천 차단.

### 자주 하는 오해

- spread 문법을 쓰면 깊은 복사다 → **틀림.** 1단계만 복사된다.
- `JSON.stringify`/`parse`면 모든 깊은 복사 문제가 해결된다 → **틀림.** 함수, `Date`, `Map`, `Set`, `undefined`, 순환 참조 등은 깨지거나 사라진다.
- `structuredClone`이면 모든 게 된다 → 대부분 되지만 함수, DOM 노드, `Symbol` 등은 복사할 수 없다.
- 깊은 복사가 항상 정답이다 → **틀림.** 비용이 크고 구조적 공유의 이점을 잃는다.

### 면접 압축 답변

- **10초**: 얕은 복사는 1단계까지만 새로 만들고, 중첩 객체는 참조를 공유합니다. React 상태 업데이트에서는 변경 경로를 따라 참조를 새로 만들어야 합니다.
- **30초**: spread나 `Object.assign`은 1단계만 새로 복사하고 내부 중첩 참조는 공유합니다. 깊은 복사는 `structuredClone` 등으로 가능하지만, React 상태 업데이트에서는 무조건 깊은 복사보다 변경이 일어난 경로만 새 참조로 만들고 나머지는 유지하는 구조적 공유가 더 효율적입니다. 중첩이 깊으면 Immer 같은 라이브러리로 안전하게 처리합니다.

> 이제 "복사"가 생각보다 단순하지 않다는 걸 봤다. 다음엔 왜 React가 이런 복사 전략을 필요로 하는지, 즉 불변성으로 넘어간다.

## 15. 불변성은 왜 React에서 그렇게 중요할까

> **TL;DR** — 불변성은 종교가 아니라 전략이다. 기존 값을 직접 바꾸지 않고 새 참조로 변경을 표현하면, React가 `Object.is()`로 변경을 확실하게 감지할 수 있고, 메모이제이션과 동시성 모드까지 안전해진다.

"React에서는 불변성이 중요하다." 이 말을 단순 암기로 들고 가면 약하다. 중요한 건 **왜** 필요한지다.

### 핵심 개념

불변성은 값을 절대 바꾸지 않는다는 종교적인 규칙이 아니라, **기존 값을 직접 바꾸지 않고 새 값으로 표현하는 방식**이다.

- 기존 객체를 mutate하지 않고
- 바뀐 부분을 반영한 새 객체를 만들어
- "변경이 일어났다"는 사실을 **참조 변화로 표현**하는 것

### mutation vs immutable update

```tsx
function Counter() {
  const [user, setUser] = useState({ count: 0 });

  // mutation: Object.is(prev, next)가 true → 리렌더 안 됨
  const handleBadClick = () => {
    user.count += 1;
    setUser(user); // 같은 참조 → React는 변화 없다고 판단
  };

  // immutable: 새 참조 → Object.is(prev, next)가 false → 리렌더 됨
  const handleGoodClick = () => {
    setUser((prev) => ({ ...prev, count: prev.count + 1 }));
  };

  return <div>{user.count}</div>;
}
```

### 불변성이 가져다 주는 것

| 이점 | 설명 |
|---|---|
| **변경 감지가 확실해진다** | 참조가 바뀌었으면 변했고, 안 바뀌었으면 안 변한 것 — 깊은 비교 불필요 |
| **메모이제이션이 자연스러워진다** | `React.memo`, `useMemo` 등이 참조 비교로 동작하므로 불변 업데이트와 궁합이 좋다 |
| **디버깅이 쉬워진다** | 이전 상태가 보존되므로 시간 여행 디버깅(Redux DevTools 등)이 가능 |
| **동시성 모드에서 안전하다** | React 18의 concurrent features는 렌더 중단/재시작이 가능한데, mutate된 값은 이 과정에서 예측 불가능한 상태가 될 수 있다 |

### useReducer + Immer — 복잡한 상태에서의 불변 업데이트

중첩이 깊은 상태에서 spread를 여러 번 쓰면 가독성이 떨어진다. 이때 `useReducer` + Immer 조합이 효과적이다.

```tsx
import { useReducer } from "react";
import { produce } from "immer";

const initialState = {
  user: {
    name: "park",
    address: {
      city: "seoul",
      detail: { street: "강남대로", zip: "06000" },
    },
  },
};

// Immer의 produce를 reducer로 사용
const reducer = produce((draft, action) => {
  switch (action.type) {
    case "UPDATE_STREET":
      // 직접 수정처럼 보이지만, Immer가 내부적으로 불변 업데이트를 수행
      draft.user.address.detail.street = action.payload;
      break;
    case "UPDATE_CITY":
      draft.user.address.city = action.payload;
      break;
  }
});

function AddressForm() {
  const [state, dispatch] = useReducer(reducer, initialState);
  // ...
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 할 일 목록에서 완료 체크를 눌렀는데 화면이 안 바뀌는 버그. 원인은 `todo.completed = true`로 직접 mutation한 뒤 `setTodos(todos)`를 호출한 것. 같은 배열 참조이므로 React가 변화를 감지 못함. `setTodos(todos.map(t => t.id === id ? { ...t, completed: true } : t))`로 불변 업데이트하니 해결.

### 자주 하는 오해

- 불변성은 무조건 성능을 느리게 만든다 → **항상 그런 건 아니다.** 구조적 공유를 쓰면 실제 복사량은 적다.
- mutate만 안 하면 끝이다 → 구조가 너무 깊으면 업데이트 코드가 복잡해질 수 있으므로 Immer 같은 도구를 고려해야 한다.
- React는 mutate해도 결국 다시 렌더되니 상관없다 → **틀림.** `Object.is()`로 같은 참조면 리렌더 자체가 일어나지 않는다. 겉으로 동작하는 것처럼 보이는 경우는 다른 이유(부모 리렌더 등)로 트리거된 것이다.

### 면접 압축 답변

- **10초**: React는 `Object.is()`로 상태 변경을 감지하므로, 기존 객체를 mutate하면 변화를 인식하지 못합니다. 새 참조로 변경을 표현하는 것이 불변성의 핵심입니다.
- **30초**: React의 `useState`는 `Object.is()`로 이전 값과 다음 값을 비교해서 리렌더 여부를 결정합니다. 기존 객체를 직접 수정하면 참조가 그대로이므로 변화를 감지하지 못합니다. 불변성을 지키면 변경 감지가 확실해지고, 메모이제이션과 시간 여행 디버깅, React 18의 concurrent features에서도 안전합니다. 중첩이 깊으면 Immer + useReducer 조합으로 가독성과 안전성을 함께 확보합니다.

> 불변성이 왜 필요한지는 보였다. 다음엔 React가 실제로 "같다/다르다"를 어떻게 판단하는지, 그 메커니즘을 더 구체적으로 본다.

## 16. 참조 동일성은 왜 리렌더링과 최적화의 핵심이 되나

> **TL;DR** — React에서 "같다"는 거의 항상 `Object.is()` — 내용이 같아 보여도 참조가 다르면 "변경됨"으로 판단한다. state bailout, dependency 비교, memo props 비교, context value 비교가 전부 이 규칙을 따른다.

### 핵심 개념

참조 동일성은 두 값이 내용이 같아 보이느냐가 아니라, **정확히 같은 대상을 가리키느냐**를 보는 개념이다.

```js
const a = { x: 1 };
const b = { x: 1 };
Object.is(a, b); // false — 구조가 같아도 다른 객체

const c = a;
Object.is(a, c); // true — 같은 참조
```

참고로 `Object.is()`는 `===`와 거의 같지만 두 가지 예외가 있다:

| 비교 | `===` | `Object.is()` |
|---|---|---|
| `NaN === NaN` | `false` | `true` |
| `+0 === -0` | `true` | `false` |

React는 내부적으로 `===`가 아니라 `Object.is()`를 사용한다.

### React에서 참조 동일성이 쓰이는 곳

| 위치 | 비교 방식 | 결과 |
|---|---|---|
| `useState` bailout | `Object.is(prevState, nextState)` | true면 리렌더 건너뜀 |
| `useEffect` dependency | 배열 각 항목을 `Object.is()`로 비교 | 하나라도 다르면 effect 재실행 |
| `useMemo`/`useCallback` dependency | 배열 각 항목을 `Object.is()`로 비교 | 하나라도 다르면 재계산 |
| `React.memo` props | 각 prop을 `Object.is()`로 얕게 비교 | 하나라도 다르면 리렌더 |
| Context value | `Object.is(prevValue, nextValue)` | 다르면 구독자 전부 리렌더 |

```tsx
function Parent() {
  const [count, setCount] = useState(0);

  // 매 렌더마다 새 객체/함수 생성 — 참조가 항상 다름
  const style = { color: "red" };
  const handleClick = () => console.log("click");

  return <MemoChild style={style} onClick={handleClick} />;
}

// React.memo를 써도 style과 onClick이 매번 새 참조이므로 memo 효과 없음
const MemoChild = React.memo(({ style, onClick }) => {
  return <button style={style} onClick={onClick}>Click</button>;
});
```

### 디버깅 — "왜 리렌더가 이렇게 많지?"

React DevTools Profiler로 불필요한 리렌더를 추적하는 방법:

1. **React DevTools → Profiler 탭 → 녹화 시작**
2. 문제 상황 재현 후 녹화 중지
3. **"Why did this render?"** 패널 확인 — 어떤 props/state가 바뀌었는지 보여줌
4. 바뀐 prop이 내용은 같은데 참조만 다른 경우 → `useMemo`/`useCallback`으로 안정화 필요

또는 개발 중 빠르게 확인하려면:
- **React DevTools → Settings → "Highlight updates when components render"** 체크
- 렌더되는 컴포넌트가 초록색 테두리로 깜빡임 → 불필요한 리렌더를 시각적으로 확인

### 실무 에피소드 — 이런 상황에서 만난다

> 모달을 열 때마다 모달 뒤의 리스트 전체가 깜빡이면서 느려지는 현상. Profiler로 확인하니 모달의 state가 바뀔 때마다 리스트에 넘기는 `filter` 객체가 매 렌더마다 새로 생성되고 있었다. `const filter = { status: "active" }`가 렌더 함수 안에 있어서 매번 새 참조. `useMemo`로 감싸서 해결.

### 자주 하는 오해

- 내용이 같으면 React도 같은 값으로 본다 → **틀림.** `Object.is()`는 참조를 비교한다.
- 새 객체를 만들었지만 필드가 같으니 최적화에 영향이 없다 → **틀림.** 참조가 바뀌면 memo, useEffect 등이 다시 실행될 수 있다.
- 참조 안정성이 중요하니 무조건 모든 걸 memo 해야 한다 → **틀림.** memo 자체도 비교 비용이 있다. 프로파일러로 병목을 확인한 뒤 적용.

### 면접 압축 답변

- **10초**: React는 `Object.is()`로 참조 동일성을 비교합니다. 내용이 같아도 새 객체면 "변경됨"으로 판단하므로, 참조 안정성이 최적화의 핵심입니다.
- **30초**: React의 state bailout, dependency 비교, memo props 비교, context value 비교는 모두 `Object.is()`로 참조 동일성을 확인합니다. 매 렌더마다 새 객체나 새 함수를 만들면 최적화가 깨질 수 있고, `useMemo`나 `useCallback`으로 참조를 안정시키는 것이 의미를 가집니다. 불필요한 리렌더가 의심되면 React DevTools Profiler로 어떤 prop이 바뀌었는지 확인합니다.

> React가 내부 값을 세밀하게 다 보는 게 아니라 참조를 중요한 신호로 쓴다는 감이 왔다면, 다음엔 자연스럽게 리렌더링이 언제 발생하는가로 넘어가야 한다.

## 17. React 리렌더링은 정확히 언제 발생하나

> **TL;DR** — 리렌더링 트리거는 4가지: state 변경, 부모 리렌더, context 변경, 외부 스토어 변경. 리렌더는 "다시 계산"이지 곧바로 DOM 변경이 아니다. 렌더 단계(순수 계산)와 커밋 단계(실제 DOM 반영)는 분리되어 있다.

### 리렌더링 트리거 조건

| 조건 | 설명 |
|---|---|
| **자신의 state가 바뀔 때** | `Object.is()`로 비교해서 다르면 리렌더 |
| **부모가 다시 렌더될 때** | 기본적으로 자식도 함께 리렌더 (섹션 18에서 상세) |
| **사용하는 context 값이 바뀔 때** | Provider의 value가 새 참조면 구독자 전부 리렌더 |
| **외부 스토어 구독 값이 바뀔 때** | `useSyncExternalStore` 등 |

### 렌더 단계 vs 커밋 단계

```
setState 호출
    ↓
┌─────────────────────────────────────────────────────┐
│ [렌더 단계] — 순수 계산, 부수효과 없어야 함           │
│  컴포넌트 함수 재실행 → 새 JSX 생성 → 재조정(diffing) │
│  ※ React 18에서는 이 단계가 중단/재시작될 수 있다     │
└─────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────┐
│ [커밋 단계] — 실제 DOM 변경                           │
│  달라진 DOM 노드만 업데이트                            │
│    → useLayoutEffect (DOM 변경 직후, 동기)            │
│      → 브라우저 페인트                                │
│        → useEffect (페인트 이후, 비동기)              │
└─────────────────────────────────────────────────────┘
```

### useLayoutEffect vs useEffect — 타이밍 차이

| | `useEffect` | `useLayoutEffect` |
|---|---|---|
| **실행 시점** | 브라우저 페인트 **이후** | DOM 변경 직후, 페인트 **이전** |
| **실행 방식** | 비동기 | 동기 (페인트를 블로킹) |
| **용도** | API 호출, 구독, 타이머 등 대부분의 side effect | DOM 측정, 스크롤 위치 조정, 깜빡임 방지 |
| **주의** | 기본 선택지 | 페인트를 블로킹하므로 무거운 작업 금지 |

```tsx
// useEffect: 페인트 후 실행 → 깜빡일 수 있음
useEffect(() => {
  // 툴팁 위치를 DOM 측정 후 조정
  const rect = ref.current.getBoundingClientRect();
  setPosition({ top: rect.bottom, left: rect.left });
}, []);
// 문제: 초기 위치(0,0) → 페인트 → 올바른 위치 — 깜빡임!

// useLayoutEffect: 페인트 전 실행 → 깜빡임 없음
useLayoutEffect(() => {
  const rect = ref.current.getBoundingClientRect();
  setPosition({ top: rect.bottom, left: rect.left });
}, []);
// 초기 위치 → DOM 변경 → 위치 보정 → 페인트 — 깜빡임 없음
```

### 실무 에피소드 — 이런 상황에서 만난다

> 드롭다운 메뉴가 열릴 때 위치가 한 프레임 동안 왼쪽 상단(0,0)에 나타났다가 제자리로 이동하는 깜빡임. `useEffect`로 DOM 위치를 측정하고 있었는데, 페인트 후에 실행되니까 첫 프레임에서 잘못된 위치가 보인 것. `useLayoutEffect`로 바꾸면 페인트 전에 위치가 확정되어 깜빡임 해결.

### 자주 하는 오해

- 리렌더가 일어나면 무조건 화면도 많이 바뀐다 → **틀림.** 커밋 단계에서 diff가 적으면 DOM 변경은 최소화된다.
- 부모가 렌더돼도 자식은 영향이 없다 → **기본적으로 틀림.** 자식도 함께 리렌더된다.
- 렌더 자체가 항상 나쁘다 → **틀림.** React의 설계 자체가 "렌더는 저렴하게, 커밋은 필요한 만큼만"이다.
- `setState`를 여러 번 호출하면 그만큼 리렌더된다 → **React 18에서는 자동 배칭**으로 한 번만 리렌더된다.
- `useLayoutEffect`와 `useEffect`는 같다 → **틀림.** 실행 타이밍이 다르고, `useLayoutEffect`는 페인트를 블로킹한다.

### 면접 압축 답변

- **10초**: React는 state 변경, 부모 리렌더, context 변경 시 다시 렌더됩니다. 렌더는 UI 계산이고 실제 DOM 변경은 커밋 단계에서 일어납니다.
- **30초**: React 컴포넌트는 state 변경, 부모 리렌더, context 변경, 외부 스토어 변경으로 다시 렌더됩니다. 렌더 단계는 순수 계산이고, 실제 DOM 변경은 커밋 단계에서 diff 결과를 기준으로 최소한만 일어납니다. `useEffect`는 페인트 이후 비동기로, `useLayoutEffect`는 페인트 이전 동기로 실행됩니다. DOM 측정이나 깜빡임 방지가 필요하면 `useLayoutEffect`를 씁니다.

> 그렇다면 바로 다음 질문이 나온다. 부모가 리렌더되면 자식은 항상 다시 렌더될까?

## 18. 부모가 리렌더되면 자식도 항상 다시 렌더되나

> **TL;DR** — 기본적으로 그렇다. 하지만 `React.memo`, children 패턴, 컴포넌트 분리로 구조적으로 방지할 수 있다. 무조건 memo를 붙이는 것보다 구조적 해결이 우선이다.

### 핵심 개념

부모가 다시 렌더되면 자식도 재귀적으로 다시 렌더되는 것이 **기본값**이다. 이건 자식의 props가 바뀌었는지와 관계없다.

렌더를 건너뛸 수 있는 방법은 크게 세 가지다.

### 방법 1: React.memo — props 비교로 건너뛰기

```tsx
const MemoChild = React.memo(ExpensiveComponent);
// props가 이전과 Object.is() 기준으로 같으면 렌더 건너뜀
```

### 방법 2: children 패턴 — 구조적으로 방지

```tsx
function Layout({ children }) {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount((c) => c + 1)}>{count}</button>
      {children} {/* Layout의 부모에서 만들어졌으므로 참조 유지 */}
    </div>
  );
}

// 사용
<Layout>
  <ExpensiveComponent /> {/* Layout의 count가 바뀌어도 리렌더 안 됨 */}
</Layout>
```

이 패턴이 동작하는 이유: `children`은 `Layout`의 **부모**가 만든 JSX이므로, `Layout`의 state가 바뀌어도 `children`의 참조는 변하지 않는다.

### 방법 3: state를 사용하는 부분만 분리

```tsx
// Before: Header 전체가 count 때문에 리렌더
function Header() {
  const [count, setCount] = useState(0);
  return (
    <header>
      <Logo />              {/* count와 무관한데 리렌더됨 */}
      <Navigation />         {/* count와 무관한데 리렌더됨 */}
      <Counter count={count} setCount={setCount} />
    </header>
  );
}

// After: count를 사용하는 부분만 분리
function Header() {
  return (
    <header>
      <Logo />
      <Navigation />
      <CounterSection /> {/* state가 여기 안에만 있음 */}
    </header>
  );
}

function CounterSection() {
  const [count, setCount] = useState(0);
  return <Counter count={count} setCount={setCount} />;
}
// Logo와 Navigation은 CounterSection의 state가 바뀌어도 리렌더 안 됨
```

### 판단 기준 — 언제 뭘 써야 하나

```
리렌더가 문제인가?
  │
  ├── 프로파일러에서 병목이 확인됨
  │     │
  │     ├── state를 사용하는 부분을 분리할 수 있나? → 컴포넌트 분리 (가장 먼저)
  │     ├── children 패턴으로 해결되나? → children 주입
  │     └── 위 방법으로 안 되면 → React.memo + useMemo/useCallback
  │
  └── 프로파일러에서 문제 없음 → 그냥 두기
```

### 실무 에피소드 — 이런 상황에서 만난다

> 대시보드 상단에 실시간 시계가 있는데, 1초마다 시간이 바뀔 때마다 대시보드 전체(차트, 테이블, 필터)가 리렌더되면서 버벅거림. 시계 컴포넌트가 대시보드 최상위에 state를 두고 있어서 1초마다 전체 트리가 리렌더된 것. 시계를 별도 컴포넌트로 분리하고 state를 그 안에 격리시켜서 해결. memo 한 줄 안 쓰고 구조만 바꿔서 해결한 케이스.

### 자주 하는 오해

- 부모가 렌더돼도 자식은 독립적이라 자동으로 안 렌더된다 → **틀림.** 기본값은 함께 리렌더.
- 자식 리렌더는 모두 나쁘다 → **틀림.** 렌더 비용이 작으면 문제 없다.
- React.memo를 쓰면 무조건 해결된다 → **틀림.** props가 매번 새 참조면 memo 효과가 없다.
- memo가 최적화의 첫 번째 수단이다 → **틀림.** 컴포넌트 분리와 children 패턴이 구조적으로 더 근본적인 해결.

### 면접 압축 답변

- **10초**: 기본적으로 부모가 리렌더되면 자식도 함께 다시 렌더됩니다. `React.memo`, children 패턴, 컴포넌트 분리로 건너뛸 수 있습니다.
- **30초**: React의 기본 동작은 부모 리렌더 시 자식도 재귀적으로 리렌더하는 것입니다. 해결 우선순위는 state를 사용하는 컴포넌트 분리 → children 패턴 → `React.memo`입니다. 무조건 memo를 붙이는 것보다 구조적으로 리렌더 범위를 줄이는 게 더 근본적이고, 프로파일러로 실제 병목을 확인한 뒤 최적화하는 것이 중요합니다.

> 이제 "건너뛸 수 있는 조건"이 궁금해진다. 그게 바로 React.memo다.

## 19. React.memo는 참조 불안정이 왜 문제인지를 보여주는 도구다

> **TL;DR** — React.memo는 이전/다음 props를 얕게 비교해서 같으면 렌더를 재사용한다. 하지만 props 참조가 매번 바뀌면 비교가 항상 실패한다. memo 혼자가 아니라 `useMemo`/`useCallback`과 함께 써야 의미가 있다.

### memo 효과 없는 패턴 vs 있는 패턴

```tsx
// memo 효과 없음: 매번 새 참조
function Parent() {
  const [count, setCount] = useState(0);
  return (
    <MemoChild
      style={{ color: "red" }}          // 매 렌더마다 새 객체
      items={[1, 2, 3]}                  // 매 렌더마다 새 배열
      onClick={() => console.log(count)} // 매 렌더마다 새 함수
    />
  );
}

// memo 효과 있음: 참조 안정화
function Parent() {
  const [count, setCount] = useState(0);
  const style = useMemo(() => ({ color: "red" }), []);
  const items = useMemo(() => [1, 2, 3], []);
  const handleClick = useCallback(() => console.log(count), [count]);

  return <MemoChild style={style} items={items} onClick={handleClick} />;
}

const MemoChild = React.memo(Child);
```

### 커스텀 비교 함수

기본 얕은 비교로 부족할 때, 직접 비교 함수를 넘길 수 있다.

```tsx
const MemoChild = React.memo(Child, (prevProps, nextProps) => {
  // true를 반환하면 렌더 건너뜀, false면 리렌더
  return prevProps.id === nextProps.id && prevProps.name === nextProps.name;
  // data 같은 무거운 prop은 비교에서 제외하는 전략
});
```

주의: 커스텀 비교 함수를 쓰면 비교에서 빠진 prop이 바뀌어도 렌더되지 않으므로, **의도적으로 빼는 것인지 실수인지** 명확히 해야 한다.

### React Compiler (React 19+)

React 19부터 도입된 **React Compiler**는 빌드 타임에 자동으로 메모이제이션을 적용한다. 장기적으로 `React.memo`, `useMemo`, `useCallback`을 직접 쓸 필요가 줄어들 전망이다.

```tsx
// React Compiler가 적용되면, 이 코드가 자동으로 메모이제이션됨
function Parent() {
  const [count, setCount] = useState(0);
  const style = { color: "red" }; // 컴파일러가 자동으로 캐싱
  return <Child style={style} />;
}
```

하지만 원리를 이해하는 건 여전히 중요하다 — 컴파일러가 하는 일이 결국 이 훅들과 같은 것이기 때문이고, 컴파일러가 최적화하지 못하는 edge case도 존재한다.

### 실무 에피소드 — 이런 상황에서 만난다

> 댓글 목록 컴포넌트에 `React.memo`를 적용했는데 여전히 매번 리렌더됨. 원인은 부모에서 `comments.filter(...)`로 매 렌더마다 새 배열을 만들어 props로 넘기고 있었기 때문. `useMemo`로 필터 결과를 캐싱하니 memo가 효과를 발휘.

### 자주 하는 오해

- React.memo는 항상 성능을 올린다 → **틀림.** 비교 비용이 렌더 비용보다 클 수 있다.
- props 구조를 안 바꾸고 memo만 감싸면 된다 → **틀림.** props 참조가 불안정하면 memo가 매번 실패.
- 렌더 자체가 가벼운 컴포넌트도 memo를 씌워야 한다 → 대부분 불필요.
- 커스텀 비교 함수를 쓰면 더 정확하다 → **주의 필요.** 빠뜨린 prop이 있으면 버그가 된다.

### 면접 압축 답변

- **10초**: `React.memo`는 props가 같으면 렌더를 건너뛰는 도구인데, props 참조가 매번 바뀌면 효과가 없으므로 `useMemo`/`useCallback`과 함께 써야 합니다.
- **30초**: `React.memo`는 이전/다음 props를 `Object.is()`로 얕게 비교해서 같으면 렌더를 재사용합니다. 매 렌더마다 새 객체나 새 함수를 props로 넘기면 비교가 항상 실패합니다. 커스텀 비교 함수로 제어할 수도 있고, React 19의 React Compiler가 자동 메모이제이션을 지원하지만, 참조 동일성의 원리를 이해하는 것은 여전히 중요합니다.

> memo가 참조 안정성에 기대고 있다는 것을 봤다. 이번에는 비슷한 맥락에서 key가 왜 identity 문제인지를 짧게 본다.

## 20. key는 참조 모델 위의 identity 문제다

> **TL;DR** — key는 경고 제거용이 아니라, React가 각 요소의 **정체성(identity)**을 구분하는 기준이다. 불안정한 key는 상태 혼동을 일으키고, key 변경은 강제 리마운트 수단으로 쓸 수 있다.

### 핵심 개념

React는 리스트 변경 시 어떤 항목이 유지되고, 어떤 항목이 새로 생기고, 어떤 항목이 사라졌는지를 판단해야 한다. 이때 **key**가 identity 기준이 된다.

### index를 key로 쓸 때 생기는 문제

```tsx
function TodoList() {
  const [todos, setTodos] = useState([
    { id: 1, text: "할 일 A" },
    { id: 2, text: "할 일 B" },
    { id: 3, text: "할 일 C" },
  ]);

  const addToFront = () => {
    setTodos([{ id: Date.now(), text: "새 할 일" }, ...todos]);
  };

  return (
    <div>
      <button onClick={addToFront}>맨 앞에 추가</button>
      {todos.map((todo, index) => (
        // index를 key로 쓰면: 맨 앞에 추가할 때 모든 항목의 key가 밀림
        <TodoItem key={index} todo={todo} />
      ))}
    </div>
  );
}

function TodoItem({ todo }) {
  const [checked, setChecked] = useState(false);
  return (
    <label>
      <input type="checkbox" checked={checked} onChange={() => setChecked(!checked)} />
      {todo.text}
    </label>
  );
}
```

```
Before:  key=0:"할 일 A"(✓)  key=1:"할 일 B"  key=2:"할 일 C"
After:   key=0:"새 할 일"     key=1:"할 일 A"  key=2:"할 일 B"  key=3:"할 일 C"

React가 보는 것: key=0은 이전에도 있었고 지금도 있다 → 같은 컴포넌트!
결과: "새 할 일"에 체크박스가 ✓ 상태로 나타남 (A의 상태가 옮겨 붙음)
```

### key를 리셋 메커니즘으로 활용하기

key가 바뀌면 React는 해당 컴포넌트를 **아예 언마운트하고 새로 마운트**한다. 이걸 이용하면 컴포넌트의 state를 강제로 초기화할 수 있다.

```tsx
// 유저가 바뀌면 프로필 폼의 입력 상태를 완전히 초기화하고 싶을 때
function App() {
  const [userId, setUserId] = useState(1);

  return (
    <div>
      <UserSelector onSelect={setUserId} />
      {/* key가 바뀌면 ProfileForm이 완전히 새로 마운트됨 */}
      <ProfileForm key={userId} userId={userId} />
    </div>
  );
}

function ProfileForm({ userId }) {
  const [name, setName] = useState(""); // key가 바뀌면 이 state도 초기화
  const [email, setEmail] = useState("");
  // ...
}
```

이 패턴은 실무에서 자주 쓰인다:
- 탭 전환 시 폼 초기화
- 선택된 아이템이 바뀔 때 에디터 리셋
- 채팅방 전환 시 입력 상태 초기화

### 실무 에피소드 — 이런 상황에서 만난다

> 관리자 페이지에서 유저를 선택하면 오른쪽에 프로필 편집 폼이 나오는데, 유저 A를 편집하다가 유저 B를 선택하면 A의 입력값이 그대로 남아있는 버그. `useEffect`로 userId가 바뀔 때 폼을 리셋하는 코드를 넣었지만 edge case가 많았다. `key={userId}`를 ProfileForm에 붙이니 깔끔하게 해결 — 유저가 바뀌면 폼이 아예 새로 마운트되면서 모든 내부 state가 초기화.

### 자주 하는 오해

- key는 경고만 없애면 된다 → **틀림.** identity를 올바르게 전달해야 한다.
- index를 항상 써도 된다 → 리스트 순서가 바뀌지 않고, 항목에 state가 없을 때만 괜찮다.
- key는 성능만의 문제다 → **틀림.** 상태 정체성 문제이기도 하다.
- key를 바꾸면 비효율적이다 → 의도적 리마운트가 필요한 상황에서는 가장 깔끔한 해결책.

### 면접 압축 답변

- **10초**: key는 리스트에서 각 요소의 identity를 React에 알려주는 값입니다. 불안정한 key는 상태 혼동을 유발하고, key 변경은 강제 리마운트 수단으로도 쓸 수 있습니다.
- **30초**: key는 React가 리스트의 각 요소를 식별하는 기준입니다. index를 key로 쓰면 순서가 바뀔 때 상태가 엉뚱한 항목에 붙는 문제가 생기므로, 고유하고 안정적인 id를 쓰는 것이 원칙입니다. 반대로 key를 의도적으로 바꾸면 컴포넌트를 완전히 리마운트시킬 수 있어서, 폼 초기화나 에디터 리셋 같은 상황에서 유용합니다.

## 21. 값 모델 이해가 상태 설계의 기반이 되는 이유

> **TL;DR** — 이 챕터의 모든 개념(참조 동일성, 불변성, 얕은 비교)은 결국 상태를 어디에 두고 어떻게 나눌 것인가의 기반이 된다. 챕터 5에서 다루는 상태 설계의 전제 조건.

여기까지 오면 자연스럽게 "상태를 어디에 두고 어떻게 나누는가"라는 질문이 떠오를 것이다. 상태 소유 범위(state ownership), 끌어올리기(lifting state up), Context와 외부 상태관리 도구의 선택 같은 주제가 그것이다. 이 내용은 **챕터 5**에서 본격적으로 다룬다.

### 이 챕터에서 배운 것이 상태 설계에 어떻게 연결되나

| 이 챕터의 개념 | 상태 설계에서의 연결 |
|---|---|
| 참조 동일성 | Context의 value가 바뀌면 구독자 전부 리렌더 → value 구조 설계가 중요 |
| 불변성 | 상태를 어떤 모양으로 정규화할지 → 업데이트 복잡도에 직결 |
| 얕은 비교 | Zustand의 selector가 부분 구독을 가능하게 하는 원리 |
| React.memo | props 구조를 어떻게 설계해야 memo가 효과적인지 |
| key | 리스트 상태에서 identity를 어떻게 관리할지 |

값이 어떻게 비교되는지를 먼저 알아야, 상태 설계의 트레이드오프도 구체적으로 판단할 수 있다.
