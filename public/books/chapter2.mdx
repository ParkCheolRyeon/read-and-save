# 챕터 2. 값 모델에서 불변성과 React 리렌더링까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **자바스크립트에서 값이 어떻게 저장되고 비교되는지 이해하면, React가 왜 어떤 변경에는 반응하고 어떤 변경에는 둔감한지도 설명할 수 있다.**

---

## 11. 원시값과 참조값은 왜 React 이해의 출발점인가

앞 챕터에서는 함수가 값을 어떻게 기억하는지를 봤다.  
이제는 방향을 조금 바꿔야 한다.

React에서 상태 업데이트와 리렌더링을 제대로 이해하려면, 먼저 자바스크립트가 값을 어떤 방식으로 다루는지부터 봐야 한다.  
특히 **원시값**과 **참조값**의 차이를 모르면, 불변성도 애매하고, 리렌더링 최적화도 감으로만 하게 된다.

### 핵심 개념

- 원시값은 숫자, 문자열, 불리언, `null`, `undefined`, `symbol`, `bigint`처럼 비교적 “값 자체”로 다뤄지는 값이다.
- 객체, 배열, 함수는 참조값이다.
- 원시값은 복사하면 값이 복사되는 느낌에 가깝고,
- 참조값은 복사하면 내부 데이터 전체가 아니라 **같은 대상을 가리키는 참조**가 복사된다.

### 왜 이게 중요한가

React는 상태를 바꿨을 때 “무언가 달라졌는지”를 판단해야 한다.  
그런데 참조값은 내부를 다 뜯어보지 않고도 **참조가 바뀌었는지**로 변화 여부를 판단하는 경우가 많다.

즉 원시값/참조값 이해는 단순 JS 기초가 아니라,
곧바로 아래 질문들로 이어진다.

- 왜 객체 상태를 직접 수정하면 안 되는가
- 왜 `React.memo`가 참조 안정성에 민감한가
- 왜 `useMemo`, `useCallback`의 dependency가 중요한가

### 자주 하는 오해

- 객체를 변수에 대입하면 객체 전체가 복사된다 → 틀림
- 배열을 새 변수에 넣었으니 독립적인 배열이다 → 틀림
- React는 객체 내부까지 자동으로 깊게 비교한다 → 보통 틀림

### 예시

```ts
const a = { name: "park" };
const b = a;

b.name = "kim";

console.log(a.name); // "kim"
```

여기서 b는 a를 복사한 게 아니라, 같은 객체를 가리키는 참조를 복사한 것이다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 참조값이 복사될 때 실제로 무슨 일이 생기는지 봤으니, 다음에는 얕은 복사와 깊은 복사가 왜 다른지로 넘어갈 수 있다.

### 면접 압축 답변

원시값은 값 자체가 복사되는 느낌에 가깝고, 객체나 배열 같은 참조값은 복사할 때 같은 대상을 가리키는 참조가 복사됩니다. React에서는 이 차이가 중요합니다. 상태를 객체로 다룰 때 참조가 유지되면 변화 감지가 어렵고, 반대로 새 참조를 만들면 변경을 더 명확하게 표현할 수 있기 때문입니다.

## 12. 얕은 복사와 깊은 복사는 왜 구분해야 하나

원시값과 참조값을 알았다면, 이제 “복사”라는 말이 실제로 무엇을 뜻하는지 더 정확히 봐야 한다.

많은 사람이 ...spread나 Object.assign을 쓰면 객체가 완전히 새로 복사됐다고 생각한다.
하지만 그건 대부분 얕은 복사다.

### 핵심 개념

- 얕은 복사는 바로 한 단계까지만 새로 복사한다.
- 중첩된 객체나 배열 안쪽은 여전히 기존 참조를 공유할 수 있다.
- 깊은 복사는 내부 중첩 구조까지 전부 독립적으로 복사하는 방식이다.

### 예시

```ts
const user = {
  name: "park",
  profile: {
    age: 30,
  },
};

const copied = { ...user };
copied.profile.age = 40;

console.log(user.profile.age); // 40
```

겉 객체는 새로 만들어졌지만, profile은 여전히 같은 참조를 보고 있기 때문에 원본도 같이 바뀐다.

### 왜 이게 중요한가

이걸 모르고 상태를 업데이트하면,
겉보기에는 새 객체를 만든 것 같아도 내부 중첩 데이터는 원본을 같이 건드릴 수 있다.

그 결과:

- 예측하지 못한 값 변경이 생기고
- 디버깅이 어려워지고
- React 최적화도 꼬일 수 있다

### 자주 하는 오해

- spread 문법을 쓰면 깊은 복사다 → 틀림
- JSON stringify/parse면 모든 깊은 복사 문제가 해결된다 → 틀림. 함수, 날짜, Map, Set 등은 깨질 수 있다
- 깊은 복사가 항상 정답이다 → 틀림. 비용이 크고 구조를 과하게 복제할 수 있다

### 실무 연결

실무에서는 무조건 깊은 복사를 하는 게 아니라,
정말 바뀌는 경로에만 필요한 수준으로 새 참조를 만들고 나머지는 유지하는 구조적 공유가 더 중요하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 “복사”가 생각보다 단순하지 않다는 걸 봤다.
다음엔 왜 React가 이런 복사 전략을 좋아하는지, 즉 불변성으로 넘어가야 한다.

### 면접 압축 답변

얕은 복사는 1단계까지만 새로 만들고 내부 중첩 참조는 공유할 수 있습니다. 깊은 복사는 내부까지 모두 독립 복사합니다. React 상태 업데이트에서는 중첩 구조를 잘못 얕은 복사하면 원본까지 함께 바뀌는 문제가 생길 수 있어서, 변경 경로를 기준으로 참조를 정확히 새로 만드는 게 중요합니다.

## 13. 불변성은 왜 React에서 그렇게 중요할까

이제 자주 듣는 말이 나온다.

“React에서는 불변성이 중요하다.”

그런데 이 말을 단순 암기로 들고 가면 약하다.
중요한 건 불변성이 왜 필요한지다.

### 핵심 개념

불변성은 값을 절대 바꾸지 않는다는 종교적인 규칙이 아니라,
기존 값을 직접 바꾸지 않고 새 값으로 표현하는 방식이다.

즉:

- 기존 객체를 mutate하지 않고
- 바뀐 부분을 반영한 새 객체를 만들어
- “변경이 일어났다”는 사실을 더 명확하게 표현하는 것

이다.

### 왜 이게 중요한가

React는 상태 변경을 추적할 때, 내부를 전부 깊게 뒤져서 비교하기보다 참조 변화를 중요한 신호로 삼는 경우가 많다.

불변성을 지키면:

- 변경 지점을 추적하기 쉬워지고
- 디버깅이 쉬워지고
- 이전 상태와 다음 상태를 비교하기 쉬워지고
- 메모이제이션과 리렌더링 최적화가 훨씬 자연스러워진다

### 자주 하는 오해

- 불변성은 무조건 성능을 느리게 만든다 → 항상 그런 건 아니다
- mutate만 안 하면 끝이다 → 구조가 너무 깊으면 오히려 유지보수가 어려워질 수 있다
- React는 mutate해도 결국 다시 렌더되니 상관없다 → 일부 경우엔 겉으로는 동작해 보여도 구조가 불안정해진다

### 예시

```ts
// 안 좋은 예
user.name = “kim”;
setUser(user);

// 더 나은 예
setUser((prev) => ({
  ...prev,
  name: “kim”,
}));
```

두 번째 방식은 “이전 상태를 직접 바꾸는 것”이 아니라,
“새 상태를 만들어 넘기는 것”이라서 변경 의도가 더 명확하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 불변성이 왜 필요한지는 보였다.
다음엔 React가 실제로 변경을 어떻게 구분하는지, 즉 참조 동일성으로 넘어가야 한다.

### 면접 압축 답변

React에서 불변성이 중요한 이유는 기존 값을 직접 수정하는 대신 새 참조로 변경을 표현하면, 변화 추적과 디버깅, 메모이제이션이 쉬워지기 때문입니다. 핵심은 절대 수정 금지가 아니라, 변경을 명확하게 표현하는 구조를 만드는 것입니다.

## 14. 참조 동일성은 왜 리렌더링과 최적화의 핵심이 되나

불변성이 중요한 이유를 한 단계 더 깊게 들어가면 결국 참조 동일성 문제다.

### 핵심 개념

참조 동일성은 두 값이 내용이 같아 보이느냐가 아니라,
정확히 같은 대상을 가리키느냐를 보는 개념이다.

```ts
const a = { x: 1 };
const b = { x: 1 };
console.log(a === b); // false

const c = a;
console.log(a === c); // true
```

a와 b는 구조가 같아 보여도 서로 다른 객체다.
반면 c는 같은 참조다.

### 왜 이게 중요한가

React에서 다음 상황들이 전부 참조 동일성과 연결된다.

- props가 같다고 판단할 수 있는가
- memoization이 의미가 있는가
- dependency가 바뀌었는가
- 상태가 실제로 달라졌는가

즉 React 최적화에서 “같다”는 말은 종종
깊은 의미 비교가 아니라
참조가 유지되었는가에 더 가깝다.

### 자주 하는 오해

- 내용이 같으면 React도 같은 값으로 본다 → 보통 틀림
- 새 객체를 만들었지만 필드가 같으니 최적화에 영향이 없다 → 틀림. 참조가 바뀌면 영향이 생길 수 있다
- 참조 안정성이 중요하니 무조건 모든 걸 memo 해야 한다 → 틀림

### 여기서 다음 개념으로 어떻게 넘어가나

이제 React가 내부 값을 세밀하게 다 보는 게 아니라, 참조를 중요한 신호로 쓴다는 감이 왔다.
그 다음엔 자연스럽게 리렌더링이 언제 발생하는가로 넘어가야 한다.

### 면접 압축 답변

참조 동일성은 두 객체가 내용상 비슷한지가 아니라 같은 참조인지 보는 개념입니다. React의 props 비교, dependency 체크, memoization은 이런 참조 안정성과 밀접하게 연결되기 때문에, 객체와 함수가 매 렌더마다 새로 만들어지면 최적화가 깨질 수 있습니다.

## 15. React 리렌더링은 정확히 언제 발생하나

앞에서 참조와 변경을 봤다면, 이제 React가 “다시 계산”을 언제 시작하는지 봐야 한다.

### 핵심 개념

React 컴포넌트는 대체로 아래 상황에서 다시 렌더된다.

- 자신의 state가 바뀔 때
- 부모가 다시 렌더될 때
- 자신이 사용하는 context 값이 바뀔 때
- 외부 스토어 구독 값이 바뀔 때

중요한 건, 리렌더링은 다시 계산이지 곧바로 DOM 변경을 뜻하는 건 아니라는 점이다.

### 왜 이게 중요한가

많은 사람이 렌더와 커밋을 섞어 생각한다.
하지만 렌더는 “다음 UI를 계산하는 단계”이고,
실제 DOM 변경은 커밋 단계에서 일어난다.

그래서:

- 리렌더는 자주 일어나도 될 수 있고
- 실제 DOM 반영이 적으면 큰 문제는 아닐 수 있다
- 성능 문제는 “렌더 횟수”보다 “무엇이 비싼가”까지 같이 봐야 한다

### 자주 하는 오해

- 리렌더가 일어나면 무조건 화면도 많이 바뀐다 → 틀림
- 부모가 렌더돼도 자식은 영향이 없다 → 기본적으로 틀림
- 렌더 자체가 항상 악이다 → 틀림

### 여기서 다음 개념으로 어떻게 넘어가나

그렇다면 바로 다음 질문이 나온다.

부모가 리렌더되면 자식은 항상 다시 렌더될까?

### 면접 압축 답변

React 컴포넌트는 주로 state 변경, 부모 리렌더, context 변경, 외부 구독 값 변경으로 다시 렌더됩니다. 다만 리렌더는 다음 UI를 계산하는 단계일 뿐이고, 실제 DOM 변경은 커밋 단계에서 필요한 경우에만 일어납니다.

## 16. 부모가 리렌더되면 자식도 항상 다시 렌더되나

이 질문은 React 성능 면접에서 정말 자주 나온다.

### 핵심 개념

기본적으로는 그렇다.
부모가 다시 렌더되면 자식도 재귀적으로 다시 렌더되는 것이 기본값이다.

다만 예외가 있다.

- React.memo
- 같은 props 유지
- 특정 최적화 구조

이런 조건이 붙으면 자식 렌더를 건너뛸 수 있다.

### 왜 이게 중요한가

이걸 모르면 성능 문제가 생겼을 때
“왜 이 자식이 자꾸 렌더되지?”를 설명하기 어렵다.

그리고 반대로,
이걸 너무 단순하게 이해하면 또 이런 실수를 한다.

> 그러면 다 React.memo 씌우면 되겠네?

그건 또 아니다.

### 자주 하는 오해

- 부모가 렌더돼도 자식은 독립적이라 자동으로 안 렌더된다 → 틀림
- 자식 리렌더는 모두 나쁘다 → 틀림
- React.memo를 쓰면 무조건 해결된다 → 틀림

### 실무 연결

진짜 중요한 건:

- 자식 렌더 자체가 비싼가
- props가 안정적인가
- memo 비용보다 이득이 큰가

를 같이 보는 것이다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 “건너뛸 수 있는 조건”이 궁금해진다.
그게 바로 React.memo다.

### 면접 압축 답변

기본적으로 부모가 리렌더되면 자식도 함께 다시 렌더됩니다. 다만 자식이 React.memo로 감싸져 있고 props가 이전과 동일하다고 판단되면 렌더를 건너뛸 수 있습니다. 그래서 기본값은 재렌더, 최적화가 있을 때만 예외라고 이해하는 게 정확합니다.

---

## 17. React.memo는 언제 의미가 있고, 언제 거의 의미가 없나

### 핵심 개념

React.memo는 이전 props와 다음 props가 같다면 렌더 결과를 재사용하려는 도구다.

즉 핵심은:

- 자식 컴포넌트 렌더 비용이 어느 정도 있고
- 부모 리렌더가 자주 일어나며
- props가 자주 같게 유지될 수 있을 때

효과가 있다.

### 왜 이게 중요한가

많은 사람이 memo를 “붙이면 무조건 좋은 최적화”로 생각한다.
하지만 props로 매번 새 객체, 새 배열, 새 함수를 넘기면 memo는 거의 의미가 없어질 수 있다.

즉 memo는 혼자 쓰는 훅이 아니라:

- 참조 안정성
- useMemo
- useCallback

과 같이 봐야 한다.

### 자주 하는 오해

- React.memo는 항상 성능을 올린다 → 틀림
- 리렌더 횟수만 줄면 무조건 좋은 최적화다 → 틀림
- props 구조를 안 바꾸고 memo만 감싸면 된다 → 틀림

### 실무 예시

아주 가벼운 컴포넌트라면 memo 비교 비용이 오히려 더 아까울 수 있다.
반대로 차트, 긴 리스트 아이템, 무거운 레이아웃 계산이 들어간 자식은 memo가 도움이 될 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

여기서 자연스럽게 이어지는 질문은:

- 그럼 props를 어떻게 안정적으로 유지하지?
- 함수 props는 왜 자꾸 바뀌지?
- 값 props는 왜 매번 새 참조가 되지?

즉 다시 useMemo, useCallback, 그리고 상태 구조 설계로 이어진다.

### 면접 압축 답변

React.memo는 부모가 자주 리렌더되더라도 자식의 props가 같게 유지될 때 의미가 있습니다. 반대로 매번 새 객체나 새 함수를 props로 넘기면 memo 효과가 거의 사라질 수 있어서, 참조 안정성과 함께 봐야 합니다.

---

## 18. key는 왜 identity 문제와 연결되나

이제 참조 안정성과 함께 꼭 붙여야 하는 개념이 key다.

### 핵심 개념

리스트 렌더링에서 key는 단순 경고 제거용 값이 아니다.
React가 각 요소의 정체성(identity) 을 구분하기 위한 기준이다.

### 왜 이게 중요한가

React는 리스트 변경 시,
어떤 항목이 유지되고 어떤 항목이 새로 생기고 어떤 항목이 사라졌는지를 판단해야 한다.

이때 key가 불안정하면:

- 상태가 엉뚱한 항목에 붙고
- input 값이 섞이고
- 애니메이션이 이상해지고
- 불필요한 재생성이 생길 수 있다

### 자주 하는 오해

- key는 경고만 없애면 된다 → 틀림
- index를 항상 써도 된다 → 틀림
- key는 성능만의 문제다 → 틀림. 상태 정체성 문제이기도 하다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 identity까지 이해했으니, 다음 질문은 자연스럽다.

그렇다면 상태는 어디에 두는 게 가장 안정적인가?

즉 다음은 상태 소유 범위(state ownership) 로 이어진다.

### 면접 압축 답변

key는 리스트에서 각 요소의 정체성을 React에 알려주는 값입니다. 불안정한 key를 쓰면 상태가 잘못 유지되거나 input 값이 엉뚱한 항목에 붙는 문제가 생길 수 있습니다. 그래서 key는 단순 경고 제거용이 아니라 identity 설계의 일부입니다.

---

## 19. 상태는 언제 끌어올리고, 언제 지역에 두고, 언제 전역으로 빼야 하나

지금까지는 값과 참조, 리렌더링을 봤다.
이제 한 단계 더 올라가서 상태를 어디에 둘 것인가를 봐야 한다.

### 핵심 개념

상태 위치를 정할 때 가장 중요한 질문은 이거다.

이 상태를 누가 필요로 하는가?

- 한 컴포넌트만 필요하면 지역 상태
- 형제 컴포넌트가 같이 써야 하면 공통 부모로 끌어올리기
- 앱 여러 곳에서 넓게 필요하면 context나 외부 상태 관리 고려
- 서버에서 가져온 데이터면 로컬 상태보다 서버 상태 관리 전략 고려

### 왜 이게 중요한가

상태 위치를 잘못 잡으면:

- props drilling이 심해지고
- 전역 상태가 불필요하게 커지고
- 리렌더링 범위가 넓어지고
- 유지보수가 어려워진다

### 자주 하는 오해

- 공유되면 무조건 전역 상태다 → 틀림
- props drilling이 싫으니 다 context로 빼면 된다 → 틀림
- 폼 상태도 다 전역으로 빼는 게 관리가 쉽다 → 상황에 따라 다름

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태 위치를 고민했다면, 다음에는 자연스럽게
Context API와 Zustand/Redux 같은 외부 상태관리 도구는 어떤 차이가 있나로 이어진다.

### 면접 압축 답변

상태 위치는 누가 그 값을 필요로 하는지 기준으로 정하는 게 핵심입니다. 한 컴포넌트만 쓰면 지역 상태, 형제가 함께 쓰면 공통 부모로 끌어올리고, 앱 전역에서 반복적으로 필요할 때만 전역 상태를 고려하는 게 보통 더 안정적입니다.

---

## 20. Context API와 Zustand/Redux 같은 외부 상태관리 도구는 어떻게 구분하나

### 핵심 개념

Context는 값을 깊은 트리에 전달하기 위한 도구에 가깝고,
Zustand나 Redux는 상태 업데이트 흐름과 구독 구조를 더 체계적으로 다루기 위한 도구에 가깝다.

### 왜 이게 중요한가

많은 사람이 Context를 “전역 상태관리”라고 부르지만,
정확히는 “전역적으로 접근 가능한 전달 메커니즘”에 더 가깝다.

상태 규모가 커지고,
업데이트 빈도가 많고,
디버깅이나 구조적 분리가 중요해지면
외부 상태관리 도구가 더 적합할 수 있다.

### 자주 하는 오해

- Context만 있으면 모든 전역 상태 문제가 해결된다 → 틀림
- Redux나 Zustand는 무조건 큰 서비스에서만 쓴다 → 틀림
- 외부 상태관리 도구를 쓰면 설계 고민이 사라진다 → 틀림

### 실무 연결

예를 들어:

- 테마, 언어, 현재 사용자 정도는 Context로 충분할 수 있다
- 복잡한 편집기 상태, 드래그 상태, 설정 패널, undo/redo 흐름은 Zustand 같은 도구가 더 잘 맞을 수 있다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태 구조를 큰 그림에서 봤다.
다음 챕터에서는 자연스럽게 아래 축으로 이어진다.

### 다음 흐름

**동기/비동기 → 이벤트 루프 → Promise → async/await → race condition → AbortController → 네트워크 재시도와 요청 취소**

즉 지금까지는 “렌더링과 상태”를 다뤘다면,
다음부터는 “실행 순서와 비동기 제어”로 넘어가는 것이다.

### 면접 압축 답변

Context는 값을 트리에 전달하는 데 유용하지만, 복잡한 상태 업데이트 흐름과 세밀한 구독 제어까지 해결해주지는 않습니다. 상태 규모와 변경 빈도, 디버깅 필요성이 커질수록 Zustand나 Redux 같은 외부 상태관리 도구가 더 적합할 수 있습니다.
