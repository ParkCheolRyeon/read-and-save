# 챕터 10. 인프라와 배포에서 운영 안정성까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **인프라는 단순히 서버를 띄우는 작업이 아니라, 코드를 더 안전하게 배포하고 문제를 더 빨리 감지하고 더 쉽게 되돌릴 수 있게 만드는 운영 구조라는 관점으로 이해해야 한다.**

### 전체 흐름 맵

```
CI/CD 파이프라인 (왜 중요한가 + CI vs CD + GitHub Actions + 캐시/artifact)
  → Docker — 환경 일관성과 이미지 빌드 (Docker + Dockerfile + 레이어 캐시/multi-stage)
    → Nginx, PM2, EC2 — 배포 실행 환경 (정적 서빙 + 프로세스 관리 + VM 배포)
      → 환경변수와 환경 분리 (운영 경계 + dev/staging/prod)
        → S3 + CloudFront — 정적 자산 배포 (origin + CDN + 캐시 전략)
          → 롤백 전략 (빠른 복구 능력)
            → 로그, 메트릭, 알림 (운영 가시성)
              → 모노레포와 CI/CD (영향 범위 기반 전략)
```

## 84. CI/CD 파이프라인

> **핵심 요약** — CI는 코드 품질을 자동 검증하는 흐름이고, **CD(Continuous Delivery/Deployment, 지속적 전달/배포)**는 검증된 결과물을 배포하는 흐름이다. GitHub Actions로 lint → type check → test → build 순서의 fail fast 파이프라인을 구성하고, 캐시는 속도 최적화, artifact는 결과물 전달로 구분한다.

### 프론트엔드에게 인프라는 왜 중요한가

많은 사람이 인프라를 "백엔드나 데브옵스가 보는 것"으로 생각한다.
그런데 프론트엔드도 실제 서비스에서는 배포와 운영의 영향을 직접 받는다.

코드가 로컬에서 돌아가는 것과, 실제 사용자 환경에서 안정적으로 제공되는 것은 전혀 다른 문제이기 때문이다.

- 정적 파일이 어디에 배포되는가
- 환경변수가 어떻게 주입되는가
- 빌드 결과물이 어떻게 전달되는가
- 장애가 나면 어떻게 감지하고 되돌리는가
- 서버와 CDN, 캐시가 어떻게 연결되는가

인프라 감각이 없으면 이런 상황에서 설명이 약해진다.

- "왜 로컬에서는 되는데 배포 후엔 안 되지?"
- "왜 환경변수 값이 비어 있지?"
- "왜 새 배포 후 이전 JS와 충돌하지?"
- "왜 특정 지역에서만 느리지?"
- "왜 장애가 났는데 어디부터 봐야 할지 모르지?"

### CI vs CD — 역할 분리

CI/CD는 자주 한 덩어리로 말하지만, 역할을 나눠 보면 훨씬 선명하다.

| | CI (Continuous Integration) | CD (Continuous Delivery/Deployment) |
|---|---|---|
| **목적** | 코드 품질 자동 검증 | 검증된 결과물을 배포 |
| **범위** | lint, type check, test, build | artifact 생성, 이미지 빌드, 배포, 헬스체크 |
| **실패 의미** | 품질 기준 미달 → 병합 차단 | 배포 실패 → 롤백 판단 |
| **주체** | 모든 PR에서 자동 실행 | main 병합 후 또는 수동 승인 후 |

### CI/CD 파이프라인 전체 흐름

```
                    CI (지속적 통합)                    CD (지속적 배포)
┌──────────────────────────────────────┐  ┌──────────────────────────┐
│                                      │  │                          │
│  Push → lint → typecheck → test      │  │  staging → 승인 → prod  │
│              → build → 아티팩트 저장 │──→│  deploy     배포        │
│                                      │  │                          │
└──────────────────────────────────────┘  └──────────────────────────┘
       자동                                    자동 또는 수동 승인
```

참고로 Continuous **Delivery**와 Continuous **Deployment**는 다르다.
Delivery는 배포 가능 상태까지 자동화하되 실제 배포는 수동 승인을 거치는 것이고,
Deployment는 승인 없이 자동으로 운영까지 반영하는 것이다.

### 좋은 GitHub Actions 파이프라인

좋은 파이프라인은 **빠른 실패(fail fast)** 원칙이 핵심이다.
가벼운 검사를 앞에, 무거운 작업은 뒤에 둔다.

```yaml
name: ci

on:
  pull_request:
  push:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'yarn'
      - run: corepack enable
      - run: yarn install --immutable
      - run: yarn lint
      - run: yarn typecheck
      - run: yarn test
      - run: yarn build
```

프론트엔드 파이프라인에서 자주 중요한 것:

- lockfile 기준 설치 보장 (`--immutable` / `--frozen-lockfile`)
- 캐시 전략 (`actions/setup-node`의 `cache` 옵션 또는 `actions/cache`)
- node 버전 고정
- monorepo면 affected 범위만 검사
- preview deploy와 production deploy 분리

### 캐시 vs artifact

둘 다 "다시 안 만들기"와 관련돼 보여서 헷갈리기 쉽지만, 목적이 다르다.

| | 캐시 | artifact |
|---|---|---|
| **목적** | 속도 최적화를 위한 재사용 | 결과물을 다음 단계로 전달 |
| **예시** | 의존성 캐시, 빌드 캐시 | 빌드 산출물, 테스트 리포트, coverage |
| **수명** | 같은 키면 재사용 | 특정 실행에 묶임 |

```yaml
# 캐시 예시: 의존성 설치 속도 개선
- uses: actions/setup-node@v4
  with:
    node-version: 20
    cache: 'yarn'

# artifact 예시: 빌드 결과를 다음 job에 전달
- uses: actions/upload-artifact@v4
  with:
    name: build-output
    path: dist/
    retention-days: 7

# 다른 job에서 artifact 받기
- uses: actions/download-artifact@v4
  with:
    name: build-output
    path: dist/
```

### 자주 하는 오해

- 프론트엔드는 화면만 만들면 끝이다 → 실제론 배포와 운영까지 이어진다
- CI/CD는 그냥 자동 배포다 → 품질 검증과 배포 흐름을 함께 포함한다
- 테스트만 돌리면 CI다 → 빌드, 타입체크, 린트 등도 포함될 수 있다
- 캐시와 artifact는 같은 개념이다 → 목적이 다르다
- Vercel 같은 플랫폼을 쓰면 인프라를 몰라도 된다 → 추상화될 뿐, 개념이 사라지는 건 아니다

### 실무 에피소드 — 이런 상황에서 만난다

> 테스트를 build 뒤에 배치해놨는데, lint에서 잡힐 수 있는 import 오류가 10분짜리 빌드 이후에야 발견됨. 파이프라인 순서를 lint → typecheck → test → build로 바꾸니 실패 피드백이 평균 8분에서 40초로 줄었다. 또한 `node_modules` 캐시 없이 매번 `yarn install`을 돌리던 걸 `actions/setup-node`의 cache 옵션으로 전환하니 설치 시간이 90초에서 5초로 단축.

### 면접 압축 답변

- **10초**: CI는 코드 품질 자동 검증, CD는 검증된 결과물 배포입니다. 파이프라인은 빠른 실패 원칙으로 가벼운 검사를 앞에 둡니다.
- **30초**: CI는 변경이 공통 품질 기준을 통과하는지 자동으로 검증하는 흐름이고, CD는 검증된 결과물을 실제 배포 가능한 상태로 만들고 환경에 반영하는 흐름입니다. GitHub Actions에서는 lint, type check, test를 빌드 앞에 배치해 실패를 빨리 드러내고, 캐시로 속도를 높이며, artifact로 결과물을 다음 단계에 전달하는 구조를 잡습니다.

## 85. Docker — 환경 일관성과 이미지 빌드

> **핵심 요약** — Docker는 애플리케이션과 실행 환경을 이미지로 묶어 로컬/CI/서버 간 환경 차이를 줄인다. Dockerfile은 "실행 방법"이 아니라 "배포 재현성 문서"에 가깝고, lockfile 먼저 복사 → 레이어 캐시 활용 → multi-stage build로 이미지 크기와 보안을 동시에 챙긴다.

### Docker가 해결하는 문제

Docker는 애플리케이션과 실행에 필요한 환경을
**이미지라는 형태로 묶어** 어디서든 더 일관되게 실행하게 해주는 도구다.

| | Docker 컨테이너 | VM (가상 머신) |
|---|---|---|
| **가상화 수준** | 호스트 OS 커널 공유, 프로세스 격리 | OS 전체 가상화 |
| **무게** | 가벼움 (MB 단위) | 무거움 (GB 단위) |
| **시작 속도** | 초 단위 | 분 단위 |
| **환경 일관성** | 이미지 기반 재현 | OS 이미지 기반 재현 |

Docker가 유용한 이유는 "로컬에선 되는데 서버에선 안 되는" 문제를 줄이는 데 있다.

- Node 버전 차이
- OS 패키지 차이
- 런타임 의존성 누락
- 빌드 환경 차이

### Dockerfile — 배포 재현성 문서

Dockerfile을 단순히 명령어 모음처럼 보면 약하다.
실제로는 **이 앱이 어떤 환경에서 어떻게 빌드되고 실행되는지**를 선언하는 문서에 가깝다.

```dockerfile
# 1단계: 빌드
FROM node:20-alpine AS builder
WORKDIR /app

COPY package.json yarn.lock ./
RUN corepack enable && yarn install --immutable

COPY . .
RUN yarn build

# 2단계: 실행 (빌드 결과만 가져옴)
FROM node:20-alpine AS runner
WORKDIR /app

COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

EXPOSE 3000
ENV PORT=3000
CMD ["node", "server.js"]
```

위 예시는 Next.js의 `output: 'standalone'` 설정을 전제로 한다.
standalone 모드를 쓰면 `node_modules`를 별도로 설치하지 않아도
필요한 의존성이 빌드 결과에 포함되므로 최종 이미지가 훨씬 가벼워진다.

### 레이어 캐시와 multi-stage build

Docker는 각 명령(`RUN`, `COPY` 등) 단계를 레이어처럼 쌓는다.
변하지 않은 단계는 재사용할 수 있다.

```dockerfile
# 좋은 예: lockfile 먼저 복사 → 캐시 효율 높음
COPY package.json yarn.lock ./
RUN yarn install --immutable
COPY . .

# 나쁜 예: 전체 복사 → 소스 변경마다 install도 재실행
COPY . .
RUN yarn install --immutable
```

multi-stage build는 빌드용 단계와 실행용 단계를 나눠
최종 이미지엔 필요한 결과물만 남기는 방식이다.

| 전략 | 효과 |
|---|---|
| lockfile 먼저 복사 | 의존성 설치 레이어 캐시 |
| multi-stage 분리 | 최종 이미지 크기 축소 |
| `.dockerignore` 활용 | 불필요 파일 제외 (`node_modules`, `.git`, `.env`) |
| dev dependency 제거 | runtime 이미지 보안/크기 개선 |

### Docker 멀티스테이지 빌드 + 레이어 캐싱

```
Builder Stage:                    Runner Stage:
┌──────────────────────┐         ┌──────────────────────┐
│ FROM node:20-alpine  │         │ FROM node:20-alpine  │
├──────────────────────┤         ├──────────────────────┤
│ COPY package*.json   │ 캐시✅  │ COPY --from=builder  │
│ RUN npm ci           │ 캐시✅  │   /app/.next         │
├──────────────────────┤         │   /app/public        │
│ COPY . .             │ 변경시  │   /app/node_modules  │
│ RUN npm run build    │ 재빌드  ├──────────────────────┤
└──────────────────────┘         │ CMD ["node",         │
  ↑ 빌드 후 버려짐                │  "server.js"]        │
                                 └──────────────────────┘
                                   ↑ 실행에 필요한 것만 포함
                                     → 이미지 크기 대폭 감소
```

### 자주 하는 오해

- Docker는 백엔드만 쓰는 기술이다 → 프론트 정적 배포/SSR 앱에도 유용할 수 있다
- Docker를 쓰면 운영 문제가 사라진다 → 재현성과 환경 통일을 도와줄 뿐이다
- Docker 이미지 하나면 어디서나 완전히 똑같다 → 환경변수와 외부 자원 차이는 여전히 있다
- Dockerfile은 되기만 하면 된다 → 크기, 보안, 재현성을 같이 봐야 한다
- 이미지가 커도 동작만 하면 괜찮다 → 배포 속도와 보안, 운영 비용에 영향이 있다

### 실무 에피소드 — 이런 상황에서 만난다

> Next.js 앱을 Docker로 배포했는데 이미지가 1.2GB. 원인은 single-stage로 빌드해서 `node_modules` 전체와 소스 코드가 그대로 포함된 것. multi-stage build로 전환하고 standalone 모드를 적용하니 이미지가 150MB로 줄었다. 추가로 `COPY . .`를 맨 위에 뒀더니 소스 한 줄 바꿀 때마다 `yarn install`부터 다시 도는 문제도 있었는데, lockfile 먼저 복사하는 순서로 바꾸니 빌드 시간이 5분에서 40초로 단축.

### 면접 압축 답변

- **10초**: Docker는 앱과 실행 환경을 이미지로 묶어 환경 차이를 줄이며, multi-stage build로 이미지 크기와 보안을 동시에 챙깁니다.
- **30초**: Docker는 애플리케이션과 실행 환경을 이미지로 묶어 로컬, CI, 서버 사이의 환경 차이를 줄여주는 도구입니다. Dockerfile은 배포 재현성 문서에 가깝고, lockfile을 먼저 복사해 레이어 캐시를 활용하고 multi-stage build로 빌드/실행 단계를 분리하면 더 작고 안전한 이미지를 만들 수 있습니다.

## 86. Nginx, PM2, EC2 — 배포 실행 환경

> **핵심 요약** — Nginx는 정적 파일 서빙과 reverse proxy를 맡고, PM2는 Node 프로세스의 자동 재시작과 클러스터 모드를 담당하며, **EC2(Elastic Compute Cloud, AWS 가상 서버 서비스)**는 이 모든 것을 직접 운영하는 VM 환경이다. SPA의 `try_files`, PM2 클러스터 모드, EC2 운영 책임 범위를 이해해야 한다.

### Nginx — 정적 서빙과 reverse proxy

Nginx는 보통 아래 역할 중 하나 이상을 맡는다.

| 역할 | 설명 |
|---|---|
| 정적 파일 서빙 | React/Vite 빌드 결과물 제공 |
| reverse proxy | Next.js 서버 앞단, `/api` 요청 전달 |
| SSL/TLS termination | HTTPS 처리 |
| 압축 및 캐시 헤더 | gzip/brotli, `Cache-Control` 설정 |
| 로드 밸런싱 | 여러 백엔드 인스턴스 분산 |

리버스 프록시(reverse proxy)란 클라이언트 앞에 서서 요청을 받아 뒤의 실제 서버로 전달하는 중간 서버다. 클라이언트는 실제 서버의 존재를 모른다.

SSL/TLS termination이란 Nginx가 HTTPS 암호화/복호화를 대신 처리해서, 뒤의 Node.js 서버는 평문 HTTP만 다루면 되는 구조를 말한다.

```nginx
server {
  listen 80;
  server_name example.com;

  # gzip 압축
  gzip on;
  gzip_types text/plain application/javascript text/css application/json;

  location / {
    root /usr/share/nginx/html;
    # SPA 라우팅 핵심: 파일이 없으면 index.html로 보냄
    try_files $uri $uri/ /index.html;
  }

  location /api/ {
    proxy_pass http://backend:8080/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  # 정적 자산에 대한 장기 캐시
  location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

SPA 배포에선 `try_files` 설정이 특히 중요하다.
사용자가 `/about` 같은 URL로 직접 진입하면, 서버에는 해당 경로의 파일이 없기 때문에
`index.html`로 보내야 클라이언트 라우터가 처리할 수 있다.

또한 reverse proxy에서 `proxy_set_header`로 원래 클라이언트 정보를 전달하지 않으면,
백엔드가 실제 사용자 IP를 알 수 없게 될 수 있다.

### 배포 아키텍처 — Nginx + PM2 + Node.js

```
사용자 요청
    │
    ▼
┌─────────────────────────────┐
│         Nginx               │
│  - 정적 파일 직접 제공      │
│  - SSL/TLS termination      │
│  - gzip 압축                │
│  - 리버스 프록시 → :3000    │
└────────────┬────────────────┘
             │
             ▼
┌─────────────────────────────┐
│         PM2 (프로세스 매니저) │
│  ┌───────┐ ┌───────┐       │
│  │Node #1│ │Node #2│ ...   │  ← CPU 코어 수만큼 클러스터
│  │:3000  │ │:3000  │       │
│  └───────┘ └───────┘       │
│  - 무중단 재시작 (reload)   │
│  - 크래시 시 자동 복구      │
└─────────────────────────────┘
```

### PM2 — Node 프로세스 관리

Next.js처럼 Node 프로세스를 직접 띄우는 앱은
정적 파일 서빙과 다르게 "프로세스" 자체를 관리해야 한다.

| 기능 | 설명 |
|---|---|
| 프로세스 데몬화 | 백그라운드 실행 |
| 자동 재시작 | 크래시 후 복구 |
| 클러스터 모드 | 멀티 코어 활용 |
| 로그 관리 | stdout/stderr 수집 |
| 무중단 재시작 | `pm2 reload`로 downtime 없이 갱신 |

```js
// ecosystem.config.js
module.exports = {
  apps: [
    {
      name: 'next-app',
      // standalone 모드면 script: 'server.js'로 변경
      script: 'node_modules/.bin/next',
      args: 'start',
      instances: 'max',        // CPU 코어 수만큼 프로세스 생성
      exec_mode: 'cluster',    // 클러스터 모드
      env: {
        NODE_ENV: 'production',
        PORT: 3000,
      },
    },
  ],
};
```

```bash
# 시작
pm2 start ecosystem.config.js

# 상태 확인
pm2 status

# 무중단 재시작
pm2 reload next-app

# 로그 확인
pm2 logs next-app
```

Docker 안에서 Node 앱을 띄울 때는 보통 PM2 대신 Docker의 `restart: always` 정책과
컨테이너 오케스트레이션(**ECS(Elastic Container Service, AWS 컨테이너 오케스트레이션 서비스)**, Kubernetes 등)이 프로세스 관리를 대신한다.

### EC2 — VM 기반 배포

EC2에 배포한다는 건
가상 서버 한 대 이상을 직접 운영하며 그 안에 애플리케이션을 띄운다는 뜻에 가깝다.

직접 챙겨야 할 수 있는 것:

- OS 환경, 런타임 설치
- 포트, 방화벽/보안 그룹
- 프로세스 실행, 재시작
- 로그, HTTPS/Nginx 설정

| | EC2 (VM 직접 운영) | Vercel / Netlify (플랫폼) |
|---|---|---|
| **자유도** | 높음 | 제한적 |
| **운영 책임** | OS, 런타임, 보안, 로그 전부 | 플랫폼이 추상화 |
| **적합한 경우** | SSR 커스텀, 사내 도구, 특수 의존성 | 정적 사이트, 표준 Next.js |
| **비용 구조** | 인스턴스 시간 기반 | 요청/빌드 기반 |

Vercel이나 Netlify 같은 플랫폼은 이 모든 인프라 관리를 추상화해주는 것이다.
EC2를 이해하면 그 추상화가 무엇을 감추고 있는지도 감이 온다.

### 자주 하는 오해

- Nginx는 백엔드 서버다 → 요청을 받아 전달/서빙하는 앞단 역할에 가깝다
- PM2는 배포 도구다 → 배포라기보다 프로세스 관리 도구다
- Docker를 쓰면 PM2는 항상 필요 없다 → Docker 없이 직접 서버를 운영하거나 클러스터 모드가 필요하면 PM2가 여전히 유용하다
- EC2에 배포 = 파일만 올리면 끝 → 실제론 운영까지 포함된다
- 정적 사이트에도 PM2가 필요하다 → Node 프로세스가 없으면 보통 불필요하다

### 실무 에피소드 — 이런 상황에서 만난다

> Next.js SSR 앱을 EC2에 배포했는데, 트래픽이 몰리면 서버가 죽고 아무도 모르는 상황. `node server.js`로 직접 실행하고 있어서 크래시 후 자동 복구가 없었다. PM2 클러스터 모드로 전환하니 4코어 활용 + 자동 재시작이 가능해졌고, Nginx를 앞에 두어 HTTPS 처리와 정적 자산 캐시를 분리하니 응답 시간도 개선. 추가로 SPA 부분의 `try_files` 설정이 빠져 있어서 직접 URL 접근 시 404가 뜨던 문제도 함께 해결.

### 면접 압축 답변

- **10초**: Nginx는 정적 서빙과 reverse proxy, PM2는 Node 프로세스 관리, EC2는 이 모든 것을 직접 운영하는 VM 환경입니다.
- **30초**: Nginx는 정적 파일 서빙, reverse proxy, HTTPS, 캐시 헤더 제어를 맡고, PM2는 Node 프로세스의 자동 재시작과 클러스터 모드를 담당합니다. EC2는 이 모든 것을 직접 운영하는 VM 환경으로 자유도는 높지만 책임도 큽니다. Vercel 같은 플랫폼은 이걸 추상화해주는 것이고, EC2를 이해하면 추상화 뒤에서 무슨 일이 일어나는지 감을 잡을 수 있습니다.

## 87. 환경변수와 환경 분리

> **핵심 요약** — 환경변수는 단순 설정값이 아니라, 환경을 나누고 비밀값을 분리하는 운영 경계다. 프론트엔드에서는 `NEXT_PUBLIC_`/`VITE_` 접두사로 클라이언트 노출 여부가 갈리며, 개발/스테이징/운영 환경을 분리해야 배포 사고를 줄일 수 있다.

### 환경변수 — 운영 경계

환경변수는 코드 안에 하드코딩하지 않고
환경별로 달라지는 설정을 주입하는 방법이다.

예:

- API base URL
- 인증 비밀값
- DB 연결 문자열
- 분석 도구 key
- feature flag 값

### Next.js 환경변수 구분

```bash
# .env.local

# 서버에서만 사용 (클라이언트 번들에 포함되지 않음)
DATABASE_URL=postgresql://localhost:5432/mydb
SECRET_KEY=super-secret-value

# 클라이언트에 노출됨 (NEXT_PUBLIC_ 접두사)
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_GA_ID=G-XXXXXXXXXX
```

```tsx
// 서버 컴포넌트 / API Route에서만 접근 가능
const db = process.env.DATABASE_URL;

// 클라이언트에서도 접근 가능
const apiUrl = process.env.NEXT_PUBLIC_API_URL;
```

Next.js에서 `NEXT_PUBLIC_` 접두사가 없는 환경변수는 서버에서만 접근 가능하다.
Vite에서는 `VITE_` 접두사가 같은 역할을 한다.
**이 구분을 모르면 비밀값이 클라이언트 번들에 포함되는 보안 사고가 날 수 있다.**

### 배포 환경에서 환경변수 주입

| 방법 | 용도 |
|---|---|
| CI/CD secrets | GitHub Actions 등에서 빌드 시 주입 |
| Docker `--env-file` / `-e` | 컨테이너 실행 시 주입 |
| AWS Parameter Store / Secrets Manager | 클라우드 환경 비밀값 관리 |
| Kubernetes ConfigMap / Secret | 클러스터 환경 설정/비밀값 |

### 개발 / 스테이징 / 운영 환경 분리

| | 개발 환경 | 스테이징 환경 | 운영 환경 |
|---|---|---|---|
| **목적** | 로컬 개발과 빠른 반복 | 운영과 최대한 비슷하게 검증 | 실제 사용자 트래픽 |
| **API** | localhost | staging-api.example.com | api.example.com |
| **데이터** | 로컬/mock | 샌드박스 | 실제 데이터 |
| **도메인** | localhost:3000 | staging.example.com | example.com |

```bash
# .env.development
NEXT_PUBLIC_API_URL=http://localhost:4000
NEXT_PUBLIC_ENV=development

# .env.staging
NEXT_PUBLIC_API_URL=https://staging-api.example.com
NEXT_PUBLIC_ENV=staging

# .env.production
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_ENV=production
```

```yaml
# GitHub Actions에서 환경별 배포 분기
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        if: github.ref == 'refs/heads/develop'
        run: ./deploy.sh staging

      - name: Deploy to production
        if: github.ref == 'refs/heads/main'
        run: ./deploy.sh production
```

환경을 나누지 않으면:

- 운영 데이터에 실수로 영향
- 검증되지 않은 코드 직접 반영
- 환경별 설정 차이 미발견
- 장애 위험 증가

### 자주 하는 오해

- `.env` 파일만 있으면 충분하다 → 배포 환경 주입 방식도 중요하다
- 프론트 환경변수는 다 숨겨진다 → 클라이언트 번들에 포함되면 사용자에게 노출될 수 있다
- 편하니까 비밀값도 프론트 env에 넣으면 된다 → 절대 안 되는 경우가 많다
- 작은 서비스는 환경 분리가 필요 없다 → 배포 사고를 막는 기본 장치가 될 수 있다
- 같은 코드면 환경 차이도 별 의미 없다 → 실제론 API, 도메인, 비밀값, 캐시가 다를 수 있다

### 실무 에피소드 — 이런 상황에서 만난다

> 스테이징 환경 없이 운영에 바로 배포하다가, API 응답 형태가 바뀐 걸 모르고 배포 → 운영에서 화이트스크린. 스테이징이 있었다면 사전에 발견했을 문제. 환경 분리를 도입하고 스테이징에서 검증 후 운영에 배포하는 프로세스를 만들었다. 또한 `NEXT_PUBLIC_` 접두사 없이 API 키를 넣어서 클라이언트 번들에 포함된 적도 있는데, 코드 리뷰에서 잡지 못하면 실제 보안 사고로 이어질 뻔했다.

### 면접 압축 답변

- **10초**: 환경변수는 환경별 설정과 비밀값을 코드에서 분리하는 운영 경계이며, 개발/스테이징/운영 분리는 배포 안전성의 기본입니다.
- **30초**: 환경변수는 단순 설정값이 아니라 환경별 차이를 주입하고 비밀값을 코드에서 분리하는 운영 경계입니다. 프론트엔드에선 `NEXT_PUBLIC_`이나 `VITE_` 접두사로 클라이언트 번들 포함 여부를 구분하는 감각이 중요하고, 개발/스테이징/운영을 나누면 검증 없이 운영에 반영되는 사고를 막을 수 있습니다.

## 88. S3 + CloudFront — 정적 자산 배포

> **핵심 요약** — **S3(Simple Storage Service, AWS 객체 스토리지 서비스)** 같은 저장소에 정적 파일을 보관하고, CloudFront 같은 CDN으로 사용자 가까이 전달하는 구조는 프론트엔드 정적 배포의 기본 패턴이다. 파일명 해시 기반 캐시, HTML과 정적 자산의 캐시 정책 분리, 배포 후 캐시 무효화까지 함께 설계해야 한다.

### 핵심 구조

아주 단순하게 보면:

- S3 같은 스토리지 → 정적 파일 보관 (origin)
- CloudFront 같은 CDN → 사용자 가까운 엣지 서버에서 전달

이 조합은 정적 웹 자산을 빠르고 안정적으로 제공하는 데 잘 맞는다.

| 계층 | 역할 | 예시 |
|---|---|---|
| Origin (저장소) | 정적 파일 보관 | S3, **GCS(Google Cloud Storage, 구글 클라우드 스토리지)** |
| CDN (엣지) | 사용자 가까이 전달 + 캐시 | CloudFront, Cloudflare |

### 캐시 전략

정적 배포에서 자주 중요한 것:

- **파일명 해시 기반 캐시 전략**: `main.a1b2c3.js` 형태로 빌드하면, 내용이 바뀔 때 파일명도 바뀌므로 캐시 갱신이 자연스럽다
- **HTML과 정적 자산의 캐시 정책 분리**: `index.html`은 `no-cache`로, JS/CSS는 `immutable`로 설정하는 것이 일반적이다
- **배포 후 캐시 무효화**: CloudFront invalidation 등으로 이전 캐시를 비워야 새 버전이 전달된다
- **SPA 라우팅 처리**: CloudFront의 커스텀 에러 응답 설정으로 404를 `index.html`로 보내야 한다

### S3 + CloudFront 정적 배포 구조

```
빌드 서버                   AWS
┌──────────┐    업로드     ┌──────────┐
│ npm run  │──────────────→│    S3    │ (origin)
│  build   │               │  버킷   │
└──────────┘               └────┬─────┘
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
              ┌──────────┐ ┌──────────┐ ┌──────────┐
              │ Edge 서울 │ │ Edge 도쿄│ │Edge 미국 │
              │CloudFront│ │CloudFront│ │CloudFront│
              └─────┬────┘ └─────┬────┘ └─────┬────┘
                    ▼           ▼           ▼
                 한국 사용자   일본 사용자   미국 사용자

캐시 정책:
  HTML → Cache-Control: no-cache (항상 최신 확인)
  JS/CSS → Cache-Control: public, max-age=31536000, immutable
```

### 자주 하는 오해

- 정적 파일은 그냥 아무 서버에 두면 된다 → 전달 성능과 운영 편의가 달라질 수 있다
- CDN은 대규모 서비스만 필요하다 → 지역 분산과 캐시는 작은 서비스에도 유리할 수 있다
- 정적 호스팅이면 배포 문제는 끝이다 → 캐시 무효화, 버전 관리가 중요하다

### 실무 에피소드 — 이런 상황에서 만난다

> S3 + CloudFront로 SPA를 배포했는데, 새 배포 후에도 사용자에게 이전 버전이 보이는 문제. 원인은 `index.html`에도 장기 캐시를 걸어둔 것. `index.html`은 `Cache-Control: no-cache`로, JS/CSS는 해시 파일명 + `immutable`로 분리하니 해결. 또한 CloudFront에 커스텀 에러 응답을 설정하지 않아서 `/about` 직접 접근 시 403이 뜨던 문제도 함께 잡았다.

### 면접 압축 답변

- **10초**: S3 + CloudFront는 정적 파일을 저장소에 두고 CDN으로 빠르게 전달하는 구조로, 프론트엔드 정적 배포에 자주 쓰입니다.
- **30초**: S3와 CloudFront 같은 조합은 정적 파일을 저장소에 두고 CDN을 통해 사용자 가까이 전달하는 구조로, 프론트엔드 정적 배포에서 자주 쓰입니다. 파일명 해시 기반 캐시와 HTML/정적자산 캐시 정책 분리, 배포 후 캐시 무효화까지 함께 설계해야 안정적인 배포가 가능합니다.

### 모던 배포 플랫폼 — Vercel, Cloud Run, Preview Deploy

전통적 EC2+PM2 외에 서버리스/컨테이너 기반 플랫폼이 프론트엔드 배포의 주류가 되고 있다.

| 플랫폼 | 특징 | 적합한 경우 |
|---|---|---|
| **Vercel** | Next.js 공식 호스팅, Edge Runtime, 자동 Preview Deploy | Next.js 프로젝트, 빠른 출시 |
| **Cloud Run** (GCP) | Docker 컨테이너를 서버리스로 실행, 요청 기반 과금 | 커스텀 서버, 비용 최적화 |
| **AWS Amplify** | Fullstack 배포, SSR 지원, GitHub 연동 | AWS 생태계 내 프로젝트 |
| **Netlify** | 정적 사이트 + Edge Functions | Jamstack, 정적 위주 |

### Preview Deploy — PR마다 독립 환경

```
main branch    → production.example.com  (프로덕션)
PR #42         → pr-42.preview.example.com  (자동 생성)
PR #43         → pr-43.preview.example.com  (자동 생성)
```

Preview Deploy의 장점:
- **디자이너/PM이 코드 리뷰 없이 화면을 직접 확인**할 수 있다
- PR 머지 전에 실제 환경에서 동작을 검증할 수 있다
- Vercel, Netlify는 기본 제공, 자체 인프라는 GitHub Actions + Docker로 구성 가능

### Feature Flag — 배포와 릴리스를 분리

```tsx
// Feature flag로 배포와 릴리스를 분리
function Dashboard() {
  const { isEnabled } = useFeatureFlag("new-chart-widget");

  return (
    <div>
      <ExistingWidget />
      {isEnabled && <NewChartWidget />}  {/* 플래그가 켜진 사용자만 */}
    </div>
  );
}
```

**왜 필요한가:**
- 코드는 main에 머지하되, 기능 노출은 별도로 제어
- 문제 발생 시 롤백 없이 플래그만 끄면 됨
- A/B 테스트, 점진적 출시(1% → 10% → 100%)에 활용
- 도구: LaunchDarkly, Unleash, PostHog, 또는 자체 구현 (DB + API)

## 89. 롤백 전략

> **핵심 요약** — 운영에선 완벽한 무오류 배포보다 빠른 복구 능력이 더 현실적이다. 이전 artifact/image를 항상 보관하고, 재배포 경로를 열어두며, 캐시 무효화와 API 호환성까지 고려해야 실제로 롤백이 가능하다.

### 핵심 개념

롤백은 새 배포에 문제가 있을 때
이전 안정 버전으로 되돌리는 전략이다.

실제 운영에선 완벽한 무오류 배포보다
**빠른 복구 능력**이 더 현실적이고 중요할 때가 많다.

롤백이 어려우면:

- 장애 시간이 길어지고
- 사용자 피해가 커지고
- 배포 자체를 두려워하게 된다

### 롤백 방식별 비교

```bash
# Docker 기반: 이전 이미지 태그로 재배포
docker pull myapp:v1.2.3
docker stop myapp && docker rm myapp && docker run -d --name myapp myapp:v1.2.3

# S3/CloudFront 기반: 이전 빌드 artifact를 다시 업로드
aws s3 sync s3://builds/v1.2.3/ s3://production-bucket/ --delete
aws cloudfront create-invalidation --distribution-id EXXXXXX --paths "/*"

# GitHub Actions 기반: 이전 커밋으로 재배포 트리거
gh workflow run deploy.yml --ref v1.2.3
```

핵심은 **이전 버전을 언제든 다시 배포할 수 있는 경로**가 항상 열려 있어야 한다는 것이다.

### 롤백 체크리스트

| 항목 | 설명 |
|---|---|
| 이전 artifact/image 보관 | 최소 N개 버전 유지 |
| 재배포 경로 확보 | 한 명령으로 이전 버전 배포 가능 |
| 캐시 무효화 | CDN/브라우저 캐시 갱신 |
| API 호환성 | 이전 프론트와 현재 API가 호환되는지 확인 |
| 데이터 정합성 | 롤백 후에도 데이터가 깨지지 않는지 |

### 자주 하는 오해

- 테스트를 잘하면 롤백은 거의 필요 없다 → 현실에선 여전히 필요하다
- 롤백은 수동으로 천천히 하면 된다 → 긴급 상황에선 빠른 자동화가 중요하다
- 프론트엔드는 정적 파일이라 롤백이 쉽다 → 캐시와 API 호환성 때문에 꼬일 수 있다

### 실무 에피소드 — 이런 상황에서 만난다

> 금요일 오후 배포 후 결제 페이지에서 에러 발생. 이전 Docker 이미지를 보관하고 있었지만, 롤백 스크립트가 없어서 수동으로 **SSH(Secure Shell, 암호화된 원격 접속 프로토콜)** 접속 → 이전 이미지 pull → 컨테이너 재시작에 15분 소요. 이후 GitHub Actions에 롤백 워크플로우를 만들어 `gh workflow run rollback.yml --ref v1.2.3` 한 줄로 롤백 가능하게 개선. 추가로 API 응답 스키마가 바뀐 상태에서 프론트만 롤백했더니 여전히 깨지는 경험도 있어서, API 호환성 체크를 롤백 프로세스에 포함시켰다.

### 면접 압축 답변

- **10초**: 롤백은 문제 발생 시 이전 안정 버전으로 빠르게 되돌리는 전략으로, 배포 설계의 핵심입니다.
- **30초**: 롤백은 새 배포에 문제가 생겼을 때 이전 안정 버전으로 빠르게 되돌리는 전략입니다. 운영에선 무오류 배포를 기대하기보다, 문제가 생겼을 때 얼마나 빨리 복구할 수 있는지가 더 중요합니다. 이전 이미지/artifact 보관, 캐시 무효화, API 호환성까지 고려해야 실제로 롤백이 가능합니다.

## 90. 로그, 메트릭, 알림

> **핵심 요약** — 로그는 사건 디테일, 메트릭은 추세, 알림은 감지 장치다. 셋을 세트로 설계해야 문제를 빨리 발견하고 대응할 수 있다. 프론트엔드에서는 Sentry로 에러를 추적하고, Web Vitals로 성능을 측정하며, 임계치 초과 시 슬랙 알림을 보내는 구조가 기본이다.

### 로그 / 메트릭 / 알림 비교

| | 로그 | 메트릭 | 알림 |
|---|---|---|---|
| **역할** | 사건 단위 기록 | 숫자로 집계되는 상태/성능 | 이상 상태를 사람에게 통보 |
| **강점** | 디테일 (어떤 에러, 어떤 요청, 어떤 스택) | 추세 (에러율 상승, 응답 시간 저하) | 놓치지 않게 (임계치 초과 → 즉시 알림) |
| **도구** | Sentry, DataDog Logs, CloudWatch | Web Vitals, DataDog, Grafana | Sentry 알림, Slack, PagerDuty |

### 프론트엔드에서 자주 쓰는 구조

```
로그:
- Sentry (클라이언트 에러 추적)
- DataDog Logs / CloudWatch Logs (서버 로그)

메트릭:
- Web Vitals (LCP, INP, CLS 등 성능 지표)
- 커스텀 메트릭 (페이지별 로딩 시간, API 실패율)
- DataDog / Grafana 대시보드

알림:
- Sentry 알림 (새로운 에러, 에러 급증)
- Slack / PagerDuty 연동
- CloudWatch Alarm
```

```ts
// Sentry 초기화 예시
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: 0.1, // 10% 샘플링
  environment: process.env.NEXT_PUBLIC_ENV,
});
```

### 운영에서 자주 보는 신호

프론트엔드/웹 운영에서 자주 감지해야 하는 것:

- 배포 직후 에러율 증가
- API 실패율 증가
- 특정 페이지 성능 저하
- 클라이언트 런타임 에러
- 이미지/자산 로드 실패
- 로그인/업로드 실패 증가

이런 신호를 메트릭으로 잡고, 임계치를 넘으면 알림이 가고, 로그를 보고 원인을 파악하는 흐름이 세트로 돌아야 한다.

### 자주 하는 오해

- 로그만 많으면 된다 → 너무 많으면 오히려 보기 어렵다
- 알림은 많을수록 안전하다 → 알림 피로(alert fatigue)를 만들어 정작 중요한 알림을 무시하게 된다
- 프론트엔드는 서버 로그만 보면 된다 → 클라이언트 에러 추적도 중요하다

### 실무 에피소드 — 이런 상황에서 만난다

> 배포 후 특정 페이지에서 "흰 화면"이 보인다는 CS가 들어왔는데, 서버 로그에는 아무 이상이 없었다. 원인은 클라이언트 런타임 에러. Sentry를 연동하고 나서야 `TypeError: Cannot read properties of undefined`가 특정 브라우저에서만 발생하는 걸 확인. 이후 Sentry 에러율 알림을 슬랙에 연결하고, 배포 직후 에러 급증 시 자동 알림이 오게 설정하니 CS보다 먼저 문제를 감지할 수 있게 됐다.

### 면접 압축 답변

- **10초**: 로그는 디테일, 메트릭은 추세, 알림은 감지 장치입니다. 셋을 세트로 설계해야 문제를 빨리 발견하고 대응할 수 있습니다.
- **30초**: 로그는 사건 단위 기록이고, 메트릭은 추세를 보는 숫자이며, 알림은 이상 상태를 사람에게 알려주는 장치입니다. 프론트엔드에서는 Sentry로 클라이언트 에러를 추적하고, Web Vitals로 성능을 측정하며, 임계치 초과 시 슬랙 알림을 보내는 식으로 세트를 구성합니다. 셋 중 하나만이 아니라 함께 설계해야 운영 안정성이 높아집니다.

## 91. 모노레포와 CI/CD

> **핵심 요약** — 모노레포에서는 변경마다 전체를 빌드하면 CI 비용이 급격히 커지므로, Turborepo나 Nx 같은 도구로 영향을 받은 범위만 선별 검사하는 전략이 필수다. 공용 패키지 변경 시 영향 범위 확대, 앱별 preview deploy, 캐시 활용이 핵심이다.

### 핵심 문제

모노레포에서는:

- 여러 앱/패키지가 한 저장소에 있고
- 공용 패키지 변경이 여러 앱에 영향을 줄 수 있으며
- 모든 걸 매번 전부 빌드/테스트하면 비용이 커질 수 있다

이 감각이 없으면:

- PR마다 전체 빌드가 너무 오래 걸리고
- 사소한 변경에도 모든 앱이 다시 검사되며
- 파이프라인이 팀 생산성을 심하게 떨어뜨릴 수 있다

### affected 범위 기반 CI

```yaml
# Turborepo 기반 affected 범위만 검사하는 예시
name: ci

on:
  pull_request:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 변경 범위 비교를 위해 전체 히스토리 필요
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      # --filter로 변경된 패키지만 lint/test/build
      - run: pnpm turbo lint test build --filter=...[HEAD~1]
```

Turborepo, Nx 같은 도구는 변경 범위를 자동으로 계산해서
영향을 받은 앱/패키지만 선별적으로 검사하고 빌드할 수 있다.

### 좋은 모노레포 CI 전략

| 전략 | 설명 |
|---|---|
| affected 계산 | 변경된 패키지와 의존하는 앱만 검사 |
| 공용 패키지 변경 시 확대 | 의존하는 모든 앱까지 검사 범위 확장 |
| 캐시 활용 | Turborepo remote cache 등으로 중복 빌드 방지 |
| 앱별 preview deploy | 변경된 앱만 preview 환경에 배포 |
| 공용 타입/유틸 회귀 검사 | 공용 인터페이스 변경 시 소비자 앱 테스트 |

### 자주 하는 오해

- 모노레포면 무조건 다 같이 빌드해야 안전하다 → 영향을 받은 범위 중심 전략이 필요할 수 있다
- 공용 패키지가 있으니 분리 배포는 의미 없다 → 앱별 배포 기준도 중요하다
- 캐시를 잘 쓰면 모든 문제가 해결된다 → 영향 범위 계산과 작업 그래프 설계가 같이 중요하다

### 실무 에피소드 — 이런 상황에서 만난다

> 모노레포에 앱 3개 + 공용 패키지 5개가 있는데, PR마다 전체 빌드+테스트에 25분이 걸려 개발 속도가 심각하게 저하. Turborepo의 `--filter` 옵션으로 변경된 패키지와 의존 앱만 검사하게 바꾸니 평균 CI 시간이 4분으로 줄었다. 추가로 공용 `ui` 패키지를 수정했을 때 의존하는 앱 3개가 모두 검사되어야 하는데, `--filter` 설정이 잘못돼서 1개만 검사되던 걸 발견하고 의존 그래프 기반 필터로 수정.

### 면접 압축 답변

- **10초**: 모노레포에서는 변경마다 전체를 빌드하면 비용이 크므로, 영향 범위만 선별 검사하는 전략이 중요합니다.
- **30초**: 모노레포에서는 여러 앱과 공용 패키지가 한 저장소에 묶여 있기 때문에, 모든 변경마다 전체를 빌드하고 테스트하면 CI 비용이 급격히 커질 수 있습니다. Turborepo나 Nx 같은 도구로 영향을 받은 범위만 선별적으로 검사하고 배포하는 전략이 특히 중요해집니다.

## 다음 챕터로의 연결

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 프론트엔드도 배포와 운영 구조를 이해해야 한다
- CI와 CD는 품질 게이트와 배포 흐름으로 나뉜다
- GitHub Actions 파이프라인은 빠른 실패와 재현성을 설계해야 한다
- 캐시와 artifact는 목적이 다르다
- Docker는 환경 일관성과 재현성을 높인다
- Dockerfile은 배포 환경 문서이자 재현성 선언이다
- 레이어 캐시와 multi-stage build는 속도와 크기, 보안에 중요하다
- Nginx는 정적 서빙과 프록시, HTTPS 앞단 역할을 맡는다
- PM2는 Node 프로세스 운영을 안정화한다
- EC2 배포는 자유도와 운영 책임을 함께 가져온다
- 환경변수는 환경 분리와 비밀값 관리의 경계다
- 개발/스테이징/운영 분리는 배포 안전성을 높인다
- S3/CloudFront 같은 구조는 정적 자산 배포와 전달 최적화에 강하다
- 롤백은 배포 전략의 핵심이다
- 로그, 메트릭, 알림은 운영 가시성의 기본이다
- 모노레포는 CI/CD 전략을 더 정교하게 만든다

즉,
이번 챕터는 단순 인프라 용어 암기가 아니라,
**"어떻게 더 안전하게 빌드하고, 배포하고, 감지하고, 복구할 것인가"**를 배우는 챕터다.

### 이 챕터를 읽은 뒤 자연스럽게 이어질 다음 질문들

1. 내 프로젝트를 면접에서 어떤 순서로 설명해야 하나?
2. 기술 스택 나열 말고 문제 정의 중심으로 말하려면 어떻게 해야 하나?
3. 성능 개선과 장애 대응 경험은 어떤 구조로 말하는 게 좋나?
4. 운영 환경 제약과 의사결정 기준은 어떻게 스토리화하나?
5. 아쉬운 점과 다시 만든다면 바꿀 점은 어떻게 답해야 하나?

### 이 챕터의 추천 학습 순서

1. CI/CD의 목적을 품질 게이트와 배포 자동화로 나눠 이해한다.
2. GitHub Actions, 캐시, artifact로 파이프라인 구조를 잡는다.
3. Docker, Nginx, PM2, EC2로 실행 환경과 배포 환경 감각을 세운다.
4. 환경변수와 환경 분리, 정적 자산 배포 구조를 이해한다.
5. 마지막에 롤백과 로그/메트릭/알림까지 묶어 운영 안정성 관점으로 정리한다.

이렇게 보면 인프라가 단순 서버 관리가 아니라, 프론트엔드 코드를 더 안전하게 배포하고 더 쉽게 운영하기 위한 구조라는 감각이 더 선명해진다.

# 챕터 10 핵심 요약 — 코드를 안전하게 배포하고 빠르게 복구하는 운영 구조

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **로컬에서 돌아가는 코드를 실제 사용자 환경에서 안정적으로 제공하려면, 빌드부터 배포, 감지, 복구까지 어떤 구조가 필요한가?**

---

### 1단계: 품질 검증과 배포 흐름을 자동화한다 — CI/CD 파이프라인

**좋은 파이프라인은 빠른 실패(fail fast) 원칙으로 lint → typecheck → test → build 순서를 지킨다.**

### 2단계: 실행 환경을 이미지로 묶는다 — Docker

Docker는 **"로컬에선 되는데 서버에선 안 되는" 문제를 줄인다.** lockfile 먼저 복사로 레이어 캐시를 활용하고, multi-stage build로 이미지 크기와 보안을 동시에 챙긴다.

### 3단계: 서빙, 프로세스 관리, 인프라 운영을 나눈다 — Nginx, PM2, EC2

**SPA 배포에서는 `try_files` 설정이 핵심이고, Vercel 같은 플랫폼은 이 모든 인프라 관리를 추상화해주는 것이다.**

### 4단계: 환경별 경계를 나눈다 — 환경변수와 환경 분리

환경변수는 **환경을 나누고 비밀값을 분리하는 운영 경계**다. `NEXT_PUBLIC_`/`VITE_` 접두사 구분을 모르면 비밀값이 클라이언트 번들에 포함되는 보안 사고가 날 수 있다.

---

여기서부터 "어떻게 배포하는가"에서 "배포 후 무엇이 필요한가"로 전환된다.

---

### 5단계: 정적 자산을 빠르고 안정적으로 전달한다 — S3 + CloudFront

**파일명 해시 기반 캐시, HTML과 정적 자산의 캐시 정책 분리, 배포 후 캐시 무효화까지 함께 설계**해야 한다.

### 6단계: 빠르게 되돌릴 수 있는 경로를 확보한다 — 롤백 전략

완벽한 무오류 배포보다 **빠른 복구 능력**이 더 현실적이다. 이전 artifact를 보관하고 한 명령으로 재배포할 수 있는 경로를 열어둔다.

### 7단계: 문제를 빨리 발견하고 대응한다 — 로그, 메트릭, 알림

로그는 사건 디테일, 메트릭은 추세, 알림은 감지 장치다. **셋을 세트로 설계해야 문제를 빨리 발견하고 대응할 수 있다.**

### 8단계: 모노레포 환경에서 CI/CD를 정교하게 만든다 — 영향 범위 기반 전략

**Turborepo나 Nx 같은 도구로 영향을 받은 범위만 선별 검사하는 전략이 필수**다.

---

### 한 줄로 이어 읽기

**CI/CD 파이프라인**이 품질 검증과 배포를 자동화하고 → **Docker**가 실행 환경을 이미지로 묶어 재현성을 확보하고 → **Nginx/PM2/EC2**가 서빙·프로세스·인프라를 나누고 → **환경변수**가 운영 경계를 정하고 → **S3+CloudFront**가 정적 자산을 전달하고 → **롤백 전략**이 빠른 복구 경로를 열어두고 → **로그/메트릭/알림**이 문제를 감지하고 → **모노레포 CI/CD**가 영향 범위 기반으로 효율을 높인다.

> 다음 챕터에서는 "코드를 어떻게 배포하고 운영하는가"에서 **"내 경험을 면접에서 어떻게 구조적으로 전달하는가"**로 축이 이동한다. 프로젝트 면접 답변의 뼈대를 세운다.
