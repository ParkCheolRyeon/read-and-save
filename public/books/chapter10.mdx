# 챕터 10. 인프라와 배포에서 운영 안정성까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **인프라는 단순히 서버를 띄우는 작업이 아니라, 코드를 더 안전하게 배포하고 문제를 더 빨리 감지하고 더 쉽게 되돌릴 수 있게 만드는 운영 구조라는 관점으로 이해해야 한다.**

---

## 130. 프론트엔드에게 인프라는 왜 중요한가

많은 사람이 인프라를 "백엔드나 데브옵스가 보는 것"으로 생각한다.
그런데 프론트엔드도 실제 서비스에서는 배포와 운영의 영향을 직접 받는다.

### 핵심 개념

프론트엔드에게 인프라가 중요한 이유는
코드가 로컬에서 돌아가는 것과, 실제 사용자 환경에서 안정적으로 제공되는 것은 전혀 다른 문제이기 때문이다.

즉 아래가 전부 연결된다.

- 정적 파일이 어디에 배포되는가
- 환경변수가 어떻게 주입되는가
- 빌드 결과물이 어떻게 전달되는가
- 장애가 나면 어떻게 감지하고 되돌리는가
- 서버와 CDN, 캐시가 어떻게 연결되는가

### 왜 이게 중요한가

인프라 감각이 없으면 이런 상황에서 설명이 약해진다.

- "왜 로컬에서는 되는데 배포 후엔 안 되지?"
- "왜 환경변수 값이 비어 있지?"
- "왜 새 배포 후 이전 JS와 충돌하지?"
- "왜 특정 지역에서만 느리지?"
- "왜 장애가 났는데 어디부터 봐야 할지 모르지?"

### 자주 하는 오해

- 프론트엔드는 화면만 만들면 끝이다 → 실제론 배포와 운영까지 이어진다
- 서버 지식은 백엔드만 알면 된다 → 프론트 배포 문제도 서버/캐시/환경 문제와 자주 엮인다
- Vercel 같은 플랫폼을 쓰면 인프라를 몰라도 된다 → 추상화될 뿐, 개념이 사라지는 건 아니다

### 실무 연결

프론트엔드가 인프라를 알면 특히 좋은 순간은:

- CI/CD 설계
- 배포 실패 원인 추적
- 환경별 API 연결 문제 해결
- CDN/캐시 이슈 분석
- 장애 대응과 롤백 판단

같은 곳이다.

### 면접 압축 답변

- **10초**: 프론트엔드도 배포, 환경변수, 캐시, 장애 대응을 직접 다루기 때문에 인프라 이해가 실무에서 중요합니다.
- **30초**: 프론트엔드에게 인프라는 단순 서버 지식이 아니라, 코드를 실제 사용자 환경에 어떻게 안정적으로 배포하고 운영할지에 대한 감각입니다. 배포 실패, 환경변수 문제, 캐시 충돌, 장애 대응은 프론트엔드도 직접 마주치기 때문에 인프라 이해가 실무에서 매우 중요합니다.

> 인프라가 왜 필요한지 봤으니, 이제 가장 먼저 실무에서 맞닥뜨리는 자동화 흐름인 **CI/CD**를 살펴보자.

## 131. CI와 CD는 왜 분리해서 이해해야 하나

CI/CD는 자주 한 덩어리로 말하지만, 역할을 나눠 보면 훨씬 선명하다.

### 핵심 개념

#### CI (Continuous Integration)

여러 사람이 만든 변경을 자주 합치고,
그 변경이 최소한의 품질 기준을 통과하는지 자동으로 확인하는 흐름

#### CD (Continuous Delivery / Deployment)

검증된 결과물을 실제 배포 가능한 상태로 만들고,
환경에 따라 자동 또는 반자동으로 배포하는 흐름

참고로 Continuous **Delivery**와 Continuous **Deployment**는 다르다.
Delivery는 배포 가능 상태까지 자동화하되 실제 배포는 수동 승인을 거치는 것이고,
Deployment는 승인 없이 자동으로 운영까지 반영하는 것이다.

### 왜 이게 중요한가

이 둘을 구분해야 아래 질문이 선명해진다.

- 어디까지가 품질 게이트인가
- 어디서부터 배포 파이프라인인가
- 어떤 실패는 CI에서 막고
- 어떤 실패는 배포 단계에서 관리해야 하는가

### 자주 하는 오해

- CI/CD는 그냥 자동 배포다 → 품질 검증과 배포 흐름을 함께 포함한다
- 테스트만 돌리면 CI다 → 빌드, 타입체크, 린트 등도 포함될 수 있다
- CD는 무조건 완전 자동 배포다 → 팀 정책에 따라 반자동(Delivery)일 수도 있다

### 실무 연결

좋은 구조는 보통 이렇게 나뉜다.

- CI: lint, type check, test, build
- CD: artifact 생성, 이미지 빌드, 배포, 헬스체크, 롤백 준비

### 면접 압축 답변

- **10초**: CI는 코드 품질을 자동 검증하는 흐름이고, CD는 검증된 결과물을 배포하는 흐름입니다. 분리해야 품질 게이트와 배포 제어를 명확히 설계할 수 있습니다.
- **30초**: CI는 변경이 공통 품질 기준을 통과하는지 자동으로 검증하는 흐름이고, CD는 검증된 결과물을 실제 배포 가능한 상태로 만들고 환경에 반영하는 흐름입니다. Continuous Delivery는 수동 승인 후 배포, Continuous Deployment는 자동 배포까지 포함합니다. 둘을 나눠 이해해야 어디서 품질을 막고 어디서 배포를 제어할지 설계가 선명해집니다.

> CI와 CD의 역할 분리를 알았으니, 이제 이걸 실제로 어떻게 파이프라인으로 묶는지 **GitHub Actions** 기반으로 살펴보자.

## 132. 좋은 GitHub Actions 파이프라인은 어떤 흐름을 가져야 하나

파이프라인은 단순히 "자동화가 있다"보다
**무엇을 어떤 순서로 검증하고 실패를 어디서 끊을지**가 중요하다.

### 핵심 개념

좋은 파이프라인은 보통 아래 흐름을 가진다.

1. 코드 체크아웃
2. 의존성 설치
3. 캐시 활용
4. lint
5. type check
6. test
7. build
8. artifact 또는 이미지 생성
9. 배포
10. 배포 후 헬스 체크

### 왜 이게 중요한가

순서가 중요한 이유는
실패를 더 빨리 드러내는 게 비용이 낮기 때문이다.

예를 들어:

- lint에서 바로 잡을 걸
- 무거운 빌드 뒤에야 발견하면 낭비가 크다

즉 **빠른 실패(fail fast)** 원칙이 핵심이다.
가벼운 검사를 앞에, 무거운 작업은 뒤에 둔다.

### 자주 하는 오해

- 파이프라인은 한 파일에 다 몰아두면 된다 → 단계와 책임이 너무 섞일 수 있다
- 무조건 병렬이 좋다 → 의존관계가 있는 작업도 있다
- 테스트만 통과하면 배포해도 된다 → build와 deploy 전후 검증도 필요하다

### 예시

```yaml
name: ci

on:
  pull_request:
  push:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'yarn'
      - run: corepack enable
      - run: yarn install --immutable
      - run: yarn lint
      - run: yarn typecheck
      - run: yarn test
      - run: yarn build
```

### 실무 연결

프론트엔드 파이프라인에서 자주 중요한 것:

- lockfile 기준 설치 보장 (`--immutable` / `--frozen-lockfile`)
- 캐시 전략 (`actions/setup-node`의 `cache` 옵션 또는 `actions/cache`)
- node 버전 고정
- monorepo면 affected 범위만 검사
- preview deploy와 production deploy 분리

### 면접 압축 답변

- **10초**: 좋은 파이프라인은 lint, type check 같은 빠른 검사를 먼저 두고, build와 deploy는 뒤에 배치해 실패를 빨리 드러냅니다.
- **30초**: 좋은 GitHub Actions 파이프라인은 lint, type check, test 같은 빠른 품질 게이트를 먼저 두고, build와 deploy는 그 뒤에 배치해 실패를 빨리 드러내는 구조를 가집니다. 핵심은 자동화가 있다는 사실보다, 어떤 검증을 어떤 순서로 통과시킬지에 있습니다.

> 파이프라인의 큰 흐름을 봤으니, 이제 파이프라인을 빠르게 만들기 위한 **캐시와 artifact 전략**을 살펴보자.

## 133. CI 캐시와 artifact는 왜 다르게 봐야 하나

둘 다 "다시 안 만들기"와 관련돼 보여서 헷갈리기 쉽다.
하지만 목적이 다르다.

### 핵심 개념

#### 캐시

같은 작업을 더 빠르게 하기 위해 재사용하는 것

예:

- 의존성 캐시
- 빌드 캐시
- 패키지 매니저 캐시

#### artifact

특정 실행 결과물을 이후 단계나 이후 시점에 전달하기 위해 저장하는 것

예:

- 빌드 산출물
- 테스트 리포트
- coverage 결과
- Docker image reference 정보

### 왜 이게 중요한가

이 차이를 알면:

- 무엇을 "속도 최적화" 대상으로 볼지
- 무엇을 "배포 전달물"로 볼지

구분이 선다.

### 자주 하는 오해

- 캐시와 artifact는 같은 개념이다 → 목적이 다르다
- 빌드 결과는 무조건 캐시하면 된다 → 재현성과 전달 경로를 함께 봐야 한다
- 캐시를 많이 걸수록 좋다 → 오래된 캐시로 꼬일 수 있다

### 예시

```yaml
# 캐시 예시: 의존성 설치 속도 개선
- uses: actions/setup-node@v4
  with:
    node-version: 20
    cache: 'yarn'

# artifact 예시: 빌드 결과를 다음 job에 전달
- uses: actions/upload-artifact@v4
  with:
    name: build-output
    path: dist/
    retention-days: 7

# 다른 job에서 artifact 받기
- uses: actions/download-artifact@v4
  with:
    name: build-output
    path: dist/
```

### 실무 연결

예를 들어:

- `node_modules`나 pnpm/yarn cache는 CI 속도 개선용
- `dist/`나 정적 빌드 결과는 배포 전달용 artifact
- coverage 리포트는 분석용 artifact

### 면접 압축 답변

- **10초**: 캐시는 속도 최적화를 위한 재사용이고, artifact는 결과물을 다음 단계로 전달하기 위한 저장입니다.
- **30초**: CI 캐시는 같은 작업을 더 빠르게 반복하기 위한 재사용이고, artifact는 특정 실행 결과물을 다음 단계나 배포 단계로 전달하기 위한 저장물입니다. 그래서 캐시는 속도 최적화, artifact는 결과 전달과 재현성 보장이라는 관점으로 구분하는 것이 좋습니다.

> 자동화 흐름과 결과물 관리를 봤으니, 이제 배포 단위를 더 일관되게 만드는 **Docker**로 넘어가자.

## 134. Docker는 왜 배포 환경 일관성 문제를 줄여주나

Docker를 처음 보면 "컨테이너 기술"이라는 말만 남고 실감이 안 올 수 있다.
프론트엔드 관점에선 더 단순하게 이해하는 게 좋다.

### 핵심 개념

Docker는 애플리케이션과 실행에 필요한 환경을
**이미지라는 형태로 묶어** 어디서든 더 일관되게 실행하게 해주는 도구다.

즉:

- 내 로컬
- CI
- 서버

가 최대한 비슷한 전제 위에서 실행되도록 만든다.

참고로 Docker 컨테이너는 VM(가상 머신)과 다르다.
VM은 OS 전체를 가상화하지만, 컨테이너는 호스트 OS의 커널을 공유하면서 프로세스를 격리한다.
그래서 훨씬 가볍고 빠르게 실행된다.

### 왜 이게 중요한가

Docker가 유용한 이유는
"로컬에선 되는데 서버에선 안 되는" 문제를 줄이는 데 있다.

예:

- Node 버전 차이
- OS 패키지 차이
- 런타임 의존성 누락
- 빌드 환경 차이

### 자주 하는 오해

- Docker는 백엔드만 쓰는 기술이다 → 프론트 정적 배포/SSR 앱에도 유용할 수 있다
- Docker를 쓰면 운영 문제가 사라진다 → 재현성과 환경 통일을 도와줄 뿐이다
- Docker 이미지 하나면 어디서나 완전히 똑같다 → 환경변수와 외부 자원 차이는 여전히 있다

### 실무 연결

프론트엔드에서 Docker가 특히 잘 맞는 경우:

- Next.js 서버 배포
- Nginx로 정적 파일 서빙
- FFmpeg 같은 추가 의존성 포함
- CI에서 같은 빌드 환경 보장
- preview / staging / production 일관성 확보

### 면접 압축 답변

- **10초**: Docker는 애플리케이션과 실행 환경을 이미지로 묶어 환경 차이를 줄여주는 도구입니다.
- **30초**: Docker는 애플리케이션과 실행 환경을 이미지로 묶어 로컬, CI, 서버 사이의 환경 차이를 줄여주는 도구입니다. VM과 달리 호스트 커널을 공유해 가볍고 빠르게 실행되며, 특히 배포 환경 일관성과 재현성을 높이는 데 강점이 있습니다.

> Docker의 존재 이유를 봤으니, 이제 실제로 이미지를 어떻게 만드는지 **Dockerfile**을 살펴보자.

## 135. Dockerfile은 왜 "실행 방법"이 아니라 "배포 재현성 문서"에 가깝나

Dockerfile을 단순히 명령어 모음처럼 보면 약하다.
실제로는 **이 앱이 어떤 환경에서 어떻게 빌드되고 실행되는지**를 선언하는 문서에 가깝다.

### 핵심 개념

Dockerfile은 보통 아래를 담는다.

- 어떤 베이스 이미지를 쓸지 (`FROM`)
- 어떤 파일을 복사할지 (`COPY`)
- 어떤 의존성을 설치할지 (`RUN`)
- 어떤 빌드 명령을 돌릴지 (`RUN`)
- 최종 컨테이너는 무엇을 실행할지 (`CMD` / `ENTRYPOINT`)

### 예시

```dockerfile
# 1단계: 빌드
FROM node:20-alpine AS builder
WORKDIR /app

COPY package.json yarn.lock ./
RUN corepack enable && yarn install --immutable

COPY . .
RUN yarn build

# 2단계: 실행 (빌드 결과만 가져옴)
FROM node:20-alpine AS runner
WORKDIR /app

COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/public ./public

EXPOSE 3000
ENV PORT=3000
CMD ["node", "server.js"]
```

위 예시는 Next.js의 `output: 'standalone'` 설정을 전제로 한다.
standalone 모드를 쓰면 `node_modules`를 별도로 설치하지 않아도
필요한 의존성이 빌드 결과에 포함되므로 최종 이미지가 훨씬 가벼워진다.

### 왜 이게 중요한가

이 구조가 중요한 이유는:

- 빌드와 실행 단계를 분리할 수 있고
- 더 작은 최종 이미지로 줄일 수 있으며
- CI와 서버에서 같은 빌드 결과를 재현하기 쉽기 때문이다

### 자주 하는 오해

- Dockerfile은 되기만 하면 된다 → 크기, 보안, 재현성을 같이 봐야 한다
- 한 단계면 충분하다 → multi-stage가 더 적절할 수 있다
- 모든 파일을 다 복사하면 편하다 → 캐시와 보안, 이미지 크기에 불리할 수 있다

### 면접 압축 답변

- **10초**: Dockerfile은 앱이 어떤 환경에서 빌드되고 실행되는지를 재현 가능하게 선언하는 문서입니다.
- **30초**: Dockerfile은 단순 실행 스크립트가 아니라, 애플리케이션이 어떤 환경에서 어떻게 빌드되고 실행되는지를 재현 가능하게 기록하는 문서입니다. multi-stage build로 빌드와 실행 단계를 분리하면 이미지 크기와 보안, 빌드 재현성까지 함께 챙길 수 있습니다.

> Dockerfile의 기본 목적을 봤으니, 이제 실무에서 중요한 **레이어 캐시와 multi-stage build**를 더 자세히 살펴보자.

## 136. 레이어 캐시와 multi-stage build는 왜 중요하나

Docker를 쓴다고 끝이 아니다.
실무에서는 이미지 빌드 속도와 최종 이미지 크기까지 같이 봐야 한다.

### 핵심 개념

#### 레이어 캐시

Docker는 각 명령(`RUN`, `COPY` 등) 단계를 레이어처럼 쌓는다.
변하지 않은 단계는 재사용할 수 있다.

예를 들어 `package.json`과 `yarn.lock`만 먼저 복사하고 `yarn install`을 실행하면,
소스 코드가 바뀌더라도 의존성 설치 단계는 캐시에서 재사용된다.
반대로 소스 코드를 먼저 복사하면 매번 의존성까지 다시 설치해야 한다.

```dockerfile
# 좋은 예: lockfile 먼저 복사 → 캐시 효율 높음
COPY package.json yarn.lock ./
RUN yarn install --immutable
COPY . .

# 나쁜 예: 전체 복사 → 소스 변경마다 install도 재실행
COPY . .
RUN yarn install --immutable
```

#### multi-stage build

빌드용 단계와 실행용 단계를 나눠
최종 이미지엔 필요한 결과물만 남기는 방식이다.

### 왜 이게 중요한가

이걸 잘 활용하면:

- CI 빌드 속도를 줄이고
- 최종 이미지 크기를 줄이고
- 불필요한 개발 의존성을 빼고
- 보안 노출 범위를 줄일 수 있다

### 자주 하는 오해

- 이미지가 커도 동작만 하면 괜찮다 → 배포 속도와 보안, 운영 비용에 영향이 있다
- 모든 파일을 먼저 복사해도 상관없다 → 캐시 효율이 떨어진다
- multi-stage는 너무 고급이라 작은 앱엔 과하다 → 생각보다 많은 경우에 유용하다

### 실무 연결

좋은 습관:

- lockfile / package manifest를 먼저 복사
- 의존성 설치를 앞에 두고 소스 복사는 뒤로
- 빌드 결과물만 runner 단계로 전달
- dev dependency는 runtime 이미지에서 제거
- `.dockerignore`로 불필요한 파일 제외 (`node_modules`, `.git`, `.env` 등)

### 면접 압축 답변

- **10초**: 레이어 캐시는 빌드 속도를, multi-stage build는 이미지 크기와 보안을 개선합니다.
- **30초**: 레이어 캐시는 변하지 않은 빌드 단계를 재사용해 Docker 빌드 속도를 높이고, multi-stage build는 빌드 단계와 실행 단계를 분리해 더 작고 안전한 최종 이미지를 만들게 해줍니다. lockfile을 먼저 복사해 의존성 설치를 캐시하고, 빌드 결과만 runner 단계에 넘기는 것이 기본 패턴입니다.

> 컨테이너 이미지를 효율적으로 만드는 방법을 봤으니, 이제 실제 배포에서 프론트 요청을 받아주는 **Nginx**를 살펴보자.

## 137. Nginx는 프론트엔드 배포에서 어떤 역할을 하나

Nginx를 보면 많은 사람이 "웹서버"라는 단어까진 외우지만,
프론트엔드 배포에서 정확히 무엇을 하는지는 흐릿할 수 있다.

### 핵심 개념

Nginx는 보통 아래 역할 중 하나 이상을 맡는다.

- 정적 파일 서빙
- reverse proxy
- SSL/TLS termination
- 압축 및 캐시 헤더 제어
- 로드 밸런싱 앞단

### 왜 이게 중요한가

프론트엔드 배포에선 아래 상황과 자주 연결된다.

- React/Vite 빌드 결과 정적 파일 제공
- Next.js 서버 앞단 프록시
- `/api` 요청을 다른 백엔드로 전달
- gzip/brotli, 캐시 헤더 설정
- 도메인/HTTPS 처리

### 자주 하는 오해

- Nginx는 백엔드 서버다 → 요청을 받아 전달/서빙하는 앞단 역할에 가깝다
- 프론트 정적 파일은 CDN만 있으면 충분하다 → origin과 앞단 제어가 여전히 필요할 수 있다
- Nginx 설정은 한 번만 하면 끝이다 → 라우팅, 캐시, 보안 헤더, 프록시 정책이 계속 연결된다

### 예시

```nginx
server {
  listen 80;
  server_name example.com;

  # gzip 압축
  gzip on;
  gzip_types text/plain application/javascript text/css application/json;

  location / {
    root /usr/share/nginx/html;
    # SPA 라우팅 핵심: 파일이 없으면 index.html로 보냄
    try_files $uri $uri/ /index.html;
  }

  location /api/ {
    proxy_pass http://backend:8080/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  # 정적 자산에 대한 장기 캐시
  location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

### 실무 연결

SPA 배포에선 `try_files` 설정이 특히 중요하다.
사용자가 `/about` 같은 URL로 직접 진입하면, 서버에는 해당 경로의 파일이 없기 때문에
`index.html`로 보내야 클라이언트 라우터가 처리할 수 있다.

또한 reverse proxy에서 `proxy_set_header`로 원래 클라이언트 정보를 전달하지 않으면,
백엔드가 실제 사용자 IP를 알 수 없게 될 수 있다.

### 면접 압축 답변

- **10초**: Nginx는 프론트엔드 배포에서 정적 파일 서빙, reverse proxy, HTTPS 처리, 캐시 헤더 제어를 맡습니다.
- **30초**: Nginx는 프론트엔드 배포에서 정적 파일 서빙, reverse proxy, HTTPS 처리, 캐시 헤더 제어 같은 역할을 맡는 경우가 많습니다. 특히 SPA에서 `try_files`로 클라이언트 라우팅을 지원하거나, Next.js 앞단에서 프록시로 요청을 전달하는 등 요청을 어떻게 받아 처리할지 설계하는 데 중요합니다.

> 정적 서빙과 프록시 역할을 봤으니, 이제 Node 프로세스를 관리하는 **PM2** 같은 프로세스 매니저를 살펴보자.

## 138. PM2 같은 프로세스 매니저는 왜 필요한가

Next.js처럼 Node 프로세스를 직접 띄우는 앱은
정적 파일 서빙과 다르게 "프로세스" 자체를 관리해야 한다.

### 핵심 개념

PM2 같은 프로세스 매니저는
서버에서 Node 앱 프로세스를 더 안정적으로 실행하고 관리하게 해준다.

대표 기능:

- 프로세스 실행 및 데몬화
- 크래시 후 자동 재시작
- 클러스터 모드로 멀티 코어 활용
- 로그 관리
- 프로세스 목록 및 상태 모니터링

### 왜 이게 중요한가

그냥 `node server.js`로 띄우면:

- 터졌을 때 자동 복구가 없고
- 로그 관리가 불편하고
- 배포/재시작 관리가 번거롭다
- 멀티 코어를 활용할 수 없다

그래서 운영 환경에선 별도 프로세스 관리 도구가 유용하다.

### 예시

```js
// ecosystem.config.js
module.exports = {
  apps: [
    {
      name: 'next-app',
      // standalone 모드면 script: 'server.js'로 변경
      script: 'node_modules/.bin/next',
      args: 'start',
      instances: 'max',        // CPU 코어 수만큼 프로세스 생성
      exec_mode: 'cluster',    // 클러스터 모드
      env: {
        NODE_ENV: 'production',
        PORT: 3000,
      },
    },
  ],
};
```

```bash
# 시작
pm2 start ecosystem.config.js

# 상태 확인
pm2 status

# 무중단 재시작
pm2 reload next-app

# 로그 확인
pm2 logs next-app
```

### 자주 하는 오해

- PM2는 배포 도구다 → 배포라기보다 프로세스 관리 도구다
- Docker를 쓰면 PM2는 항상 필요 없다 → Docker 컨테이너 런타임(restart policy 등)이 비슷한 역할을 해주지만, 클러스터 모드가 필요하거나 Docker 없이 직접 서버를 운영하는 경우엔 PM2가 여전히 유용하다
- 정적 사이트에도 PM2가 필요하다 → Node 프로세스가 없으면 보통 불필요하다

### 실무 연결

PM2가 의미 있는 환경:

- Next.js custom server
- SSR Node 앱
- API 서버
- background worker

Docker 안에서 Node 앱을 띄울 때는 보통 PM2 대신 Docker의 `restart: always` 정책과
컨테이너 오케스트레이션(ECS, Kubernetes 등)이 프로세스 관리를 대신한다.

### 면접 압축 답변

- **10초**: PM2는 Node 프로세스의 실행, 자동 재시작, 클러스터 모드, 로그 관리를 담당하는 프로세스 매니저입니다.
- **30초**: PM2 같은 프로세스 매니저는 Node 애플리케이션을 서버에서 더 안정적으로 실행하고 재시작, 로그 관리, 클러스터 모드를 통한 멀티 코어 활용을 돕는 도구입니다. Docker 환경에서는 컨테이너 런타임이 일부 역할을 대신하지만, 직접 서버를 운영할 때는 여전히 유용합니다.

> 앱 프로세스 관리 감각을 봤으니, 이제 실제로 서버를 호스팅하는 대표 환경인 **EC2**를 살펴보자.

## 139. EC2에 프론트엔드 앱을 배포한다는 건 어떤 뜻인가

EC2 같은 VM 기반 배포는
플랫폼형 배포보다 자유도가 크지만, 그만큼 직접 챙길 것도 많다.

### 핵심 개념

EC2에 배포한다는 건
가상 서버 한 대 이상을 직접 운영하며 그 안에 애플리케이션을 띄운다는 뜻에 가깝다.

즉 직접 챙겨야 할 수 있다.

- OS 환경
- 런타임 설치
- 포트
- 방화벽/보안 그룹
- 프로세스 실행
- 로그
- 재시작
- HTTPS/Nginx 설정

### 왜 이게 중요한가

이 감각이 있어야:

- "왜 Vercel보다 번거로운지"
- "왜 대신 유연한지"
- "어떤 문제가 내 책임 영역인지"

를 설명할 수 있다.

### 자주 하는 오해

- EC2에 배포 = 파일만 올리면 끝 → 실제론 운영까지 포함된다
- 서버 한 대면 충분하다 → 장애, 트래픽, 복구 전략이 더 필요할 수 있다
- 클라우드는 자동으로 안정적이다 → 구성하지 않으면 그렇지 않다

### 실무 연결

프론트엔드가 EC2에 올라가는 대표 경우:

- Next.js SSR 앱
- Nginx 정적 서빙
- Docker Compose 기반 단일 서버 운영
- 사내 도구/관리자 페이지 운영

Vercel이나 Netlify 같은 플랫폼은 이 모든 인프라 관리를 추상화해주는 것이다.
EC2를 이해하면 그 추상화가 무엇을 감추고 있는지도 감이 온다.

### 면접 압축 답변

- **10초**: EC2 배포는 가상 서버에서 런타임, 포트, 보안, 프로세스, 로그까지 직접 운영하는 것입니다. 자유도는 높지만 책임도 큽니다.
- **30초**: EC2에 프론트엔드 앱을 배포한다는 건 단순히 파일을 올리는 것이 아니라, 가상 서버에서 런타임과 포트, 프로세스, 보안 그룹, 로그, 재시작까지 직접 운영한다는 뜻에 가깝습니다. Vercel 같은 플랫폼은 이걸 추상화해주는 것이고, EC2를 이해하면 추상화 뒤에서 무슨 일이 일어나는지 감을 잡을 수 있습니다.

> 서버 운영의 기본 단위를 봤으니, 이제 배포 중 자주 터지는 문제인 **환경변수와 환경 분리**를 살펴보자.

## 140. 환경변수는 왜 단순 설정값이 아니라 운영 경계인가

환경변수는 자주 "API URL 넣는 곳" 정도로만 생각되지만,
실제로는 환경을 나누고 비밀값을 분리하는 중요한 경계다.

### 핵심 개념

환경변수는 코드 안에 하드코딩하지 않고
환경별로 달라지는 설정을 주입하는 방법이다.

예:

- API base URL
- 인증 비밀값
- DB 연결 문자열
- 분석 도구 key
- feature flag 값

### 왜 이게 중요한가

환경변수가 중요한 이유는:

- 개발/스테이징/운영을 분리하고
- 비밀값을 코드 저장소에서 떼어내고
- 같은 코드로 다른 환경을 운영할 수 있게 해주기 때문이다

### 자주 하는 오해

- `.env` 파일만 있으면 충분하다 → 배포 환경 주입 방식도 중요하다
- 프론트 환경변수는 다 숨겨진다 → 클라이언트 번들에 포함되면 사용자에게 노출될 수 있다
- 편하니까 비밀값도 프론트 env에 넣으면 된다 → 절대 안 되는 경우가 많다

### 예시: Next.js 환경변수 구분

```bash
# .env.local

# 서버에서만 사용 (클라이언트 번들에 포함되지 않음)
DATABASE_URL=postgresql://localhost:5432/mydb
SECRET_KEY=super-secret-value

# 클라이언트에 노출됨 (NEXT_PUBLIC_ 접두사)
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_GA_ID=G-XXXXXXXXXX
```

```tsx
// 서버 컴포넌트 / API Route에서만 접근 가능
const db = process.env.DATABASE_URL;

// 클라이언트에서도 접근 가능
const apiUrl = process.env.NEXT_PUBLIC_API_URL;
```

Next.js에서 `NEXT_PUBLIC_` 접두사가 없는 환경변수는 서버에서만 접근 가능하다.
Vite에서는 `VITE_` 접두사가 같은 역할을 한다.
**이 구분을 모르면 비밀값이 클라이언트 번들에 포함되는 보안 사고가 날 수 있다.**

### 실무 연결

특히 프론트엔드에선:

- 서버 전용 env (DB 접속 정보, API 키 등)
- 클라이언트 노출 env (공개 API URL, 분석 도구 ID 등)

를 구분하는 감각이 중요하다.

배포 환경에서 환경변수를 주입하는 방법도 다양하다:
- CI/CD 파이프라인의 secrets
- Docker의 `--env-file` 또는 `-e` 옵션
- AWS Parameter Store / Secrets Manager
- Kubernetes ConfigMap / Secret

### 면접 압축 답변

- **10초**: 환경변수는 환경별 설정과 비밀값을 코드에서 분리하는 운영 경계입니다. 프론트에서는 클라이언트 노출 여부 구분이 특히 중요합니다.
- **30초**: 환경변수는 단순 설정값이 아니라 환경별 차이를 주입하고 비밀값을 코드에서 분리하는 운영 경계입니다. 특히 프론트엔드에선 `NEXT_PUBLIC_`이나 `VITE_` 같은 접두사로 클라이언트 번들에 포함되는 값과 서버 전용 값을 구분하는 감각이 중요합니다. 이 구분을 놓치면 비밀값이 사용자에게 노출되는 보안 사고로 이어질 수 있습니다.

> 환경변수의 의미를 봤으니, 자연스럽게 **개발/스테이징/운영 환경 분리**를 살펴보자.

---

## 141. 개발, 스테이징, 운영 환경은 왜 나눠야 하나

환경 분리는 귀찮아 보여도,
실무에서 안정성을 크게 좌우한다.

### 핵심 개념

#### 개발 환경

로컬 개발과 빠른 반복이 중심

#### 스테이징 환경

운영과 최대한 비슷하게 검증하는 환경

#### 운영 환경

실제 사용자 트래픽을 받는 환경

### 왜 이게 중요한가

환경을 나누지 않으면:

- 운영 데이터에 실수로 영향
- 검증되지 않은 코드 직접 반영
- 환경별 설정 차이 미발견
- 장애 위험 증가

가 생긴다.

### 자주 하는 오해

- 작은 서비스는 환경 분리가 필요 없다 → 배포 사고를 막는 기본 장치가 될 수 있다
- 스테이징은 복잡해서 생략해도 된다 → 운영과 차이가 커질수록 더 필요해진다
- 같은 코드면 환경 차이도 별 의미 없다 → 실제론 API, 도메인, 비밀값, 캐시가 다를 수 있다

### 예시: 환경별 설정 분리

```bash
# .env.development
NEXT_PUBLIC_API_URL=http://localhost:4000
NEXT_PUBLIC_ENV=development

# .env.staging
NEXT_PUBLIC_API_URL=https://staging-api.example.com
NEXT_PUBLIC_ENV=staging

# .env.production
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_ENV=production
```

```yaml
# GitHub Actions에서 환경별 배포 분기
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        if: github.ref == 'refs/heads/develop'
        run: ./deploy.sh staging

      - name: Deploy to production
        if: github.ref == 'refs/heads/main'
        run: ./deploy.sh production
```

### 실무 연결

좋은 환경 분리는 보통:

- 별도 API endpoint
- 별도 env set
- 별도 도메인
- 별도 데이터 또는 샌드박스

를 갖는다.

### 면접 압축 답변

- **10초**: 개발/스테이징/운영을 나누면 검증 없이 운영에 반영되는 사고를 막고, 같은 코드로 환경별 차이를 안전하게 관리할 수 있습니다.
- **30초**: 개발, 스테이징, 운영 환경을 나누는 이유는 빠른 개발과 실제 운영 안정성을 동시에 확보하기 위해서입니다. 특히 스테이징은 운영과 최대한 비슷한 조건에서 검증할 수 있게 해주므로, 배포 사고를 줄이는 중요한 완충지대 역할을 합니다.

> 환경 분리까지 봤으니, 이제 프론트 자산 배포에서 자주 나오는 **S3 / CloudFront 같은 정적 호스팅 구조**를 살펴보자.

---

## 142. S3와 CloudFront 같은 구조는 프론트 배포에서 왜 자주 쓰이나

정적 프론트엔드 배포에서 자주 보이는 조합이 있다.
바로 "정적 저장소 + CDN" 구조다.

### 핵심 개념

아주 단순하게 보면:

- S3 같은 스토리지 → 정적 파일 보관 (origin)
- CloudFront 같은 CDN → 사용자 가까운 엣지 서버에서 전달

이 조합은 정적 웹 자산을 빠르고 안정적으로 제공하는 데 잘 맞는다.

### 왜 이게 중요한가

이 구조는:

- 정적 파일 배포가 단순하고
- CDN 캐시를 활용하기 쉽고
- 글로벌 사용자 전달 성능을 높이고
- origin 서버 부담을 줄이는 데 유리하다

### 자주 하는 오해

- 정적 파일은 그냥 아무 서버에 두면 된다 → 전달 성능과 운영 편의가 달라질 수 있다
- CDN은 대규모 서비스만 필요하다 → 지역 분산과 캐시는 작은 서비스에도 유리할 수 있다
- 정적 호스팅이면 배포 문제는 끝이다 → 캐시 무효화, 버전 관리가 중요하다

### 실무 연결

정적 배포에서 자주 중요한 것:

- **파일명 해시 기반 캐시 전략**: `main.a1b2c3.js` 형태로 빌드하면, 내용이 바뀔 때 파일명도 바뀌므로 캐시 갱신이 자연스럽다
- **HTML과 정적 자산의 캐시 정책 분리**: `index.html`은 `no-cache`로, JS/CSS는 `immutable`로 설정하는 것이 일반적이다
- **배포 후 캐시 무효화**: CloudFront invalidation 등으로 이전 캐시를 비워야 새 버전이 전달된다
- **SPA 라우팅 처리**: CloudFront의 커스텀 에러 응답 설정으로 404를 `index.html`로 보내야 한다

### 면접 압축 답변

- **10초**: S3 + CloudFront는 정적 파일을 저장소에 두고 CDN으로 빠르게 전달하는 구조로, 프론트엔드 정적 배포에 자주 쓰입니다.
- **30초**: S3와 CloudFront 같은 조합은 정적 파일을 저장소에 두고 CDN을 통해 사용자 가까이 전달하는 구조로, 프론트엔드 정적 배포에서 자주 쓰입니다. 파일명 해시 기반 캐시와 HTML/정적자산 캐시 정책 분리, 배포 후 캐시 무효화까지 함께 설계해야 안정적인 배포가 가능합니다.

> 정적 자산 배포 구조를 봤으니, 이제 새 배포가 문제를 만들었을 때 대응하는 **롤백**을 살펴보자.

---

## 143. 롤백은 왜 배포 전략의 핵심인가

배포 전략을 설명할 때 많은 사람이
"어떻게 배포하는가"까지만 말하고 끝낸다.
하지만 운영에선 그보다 중요한 질문이 있다.

> **문제가 생기면 얼마나 빨리 되돌릴 수 있는가?**

### 핵심 개념

롤백은 새 배포에 문제가 있을 때
이전 안정 버전으로 되돌리는 전략이다.

### 왜 이게 중요한가

실제 운영에선 완벽한 무오류 배포보다
**빠른 복구 능력**이 더 현실적이고 중요할 때가 많다.

롤백이 어려우면:

- 장애 시간이 길어지고
- 사용자 피해가 커지고
- 배포 자체를 두려워하게 된다

### 자주 하는 오해

- 테스트를 잘하면 롤백은 거의 필요 없다 → 현실에선 여전히 필요하다
- 롤백은 수동으로 천천히 하면 된다 → 긴급 상황에선 빠른 자동화가 중요하다
- 프론트엔드는 정적 파일이라 롤백이 쉽다 → 캐시와 API 호환성 때문에 꼬일 수 있다

### 예시: 롤백 방식별 비교

```bash
# Docker 기반: 이전 이미지 태그로 재배포
docker pull myapp:v1.2.3
docker stop myapp && docker rm myapp && docker run -d --name myapp myapp:v1.2.3

# S3/CloudFront 기반: 이전 빌드 artifact를 다시 업로드
aws s3 sync s3://builds/v1.2.3/ s3://production-bucket/ --delete
aws cloudfront create-invalidation --distribution-id EXXXXXX --paths "/*"

# GitHub Actions 기반: 이전 커밋으로 재배포 트리거
gh workflow run deploy.yml --ref v1.2.3
```

핵심은 **이전 버전을 언제든 다시 배포할 수 있는 경로**가 항상 열려 있어야 한다는 것이다.

### 실무 연결

좋은 롤백 감각은 이런 걸 포함한다.

- 이전 artifact/image 보관 (최소 N개 버전)
- 이전 버전 재배포 경로 확보
- 캐시 무효화 전략 이해
- DB/API 변경과 프론트 호환성 고려 (API 버전이 바뀌면 이전 프론트와 호환되지 않을 수 있다)
- 롤백 후에도 데이터 정합성이 깨지지 않는지 확인

### 면접 압축 답변

- **10초**: 롤백은 문제 발생 시 이전 안정 버전으로 빠르게 되돌리는 전략으로, 배포 설계의 핵심입니다.
- **30초**: 롤백은 새 배포에 문제가 생겼을 때 이전 안정 버전으로 빠르게 되돌리는 전략입니다. 운영에선 무오류 배포를 기대하기보다, 문제가 생겼을 때 얼마나 빨리 복구할 수 있는지가 더 중요합니다. 이전 이미지/artifact 보관, 캐시 무효화, API 호환성까지 고려해야 실제로 롤백이 가능합니다.

> 배포 후 복구 전략을 봤으니, 이제 복구보다 앞서 **문제를 빨리 감지하는 로그/메트릭/알림**을 살펴보자.

---

## 144. 로그, 메트릭, 알림은 왜 세트로 봐야 하나

운영에서 문제를 잘 해결하려면
문제를 "빨리 알고, 빨리 파악하고, 빨리 대응"할 수 있어야 한다.
이때 자주 같이 보는 세 가지가 로그, 메트릭, 알림이다.

### 핵심 개념

#### 로그

무슨 일이 일어났는지 사건 단위로 남기는 기록

#### 메트릭

숫자로 집계되는 상태 변화나 성능 지표

#### 알림

임계치를 넘거나 이상 상태일 때 사람에게 알려주는 장치

### 왜 이게 중요한가

이 셋은 역할이 다르다.

- 로그는 디테일을 보여준다 (무슨 에러가, 어떤 요청에서, 어떤 스택으로)
- 메트릭은 추세를 보여준다 (에러율이 올라가고 있다, 응답 시간이 느려지고 있다)
- 알림은 문제를 놓치지 않게 한다 (에러율이 5%를 넘으면 슬랙/PagerDuty로 알림)

### 자주 하는 오해

- 로그만 많으면 된다 → 너무 많으면 오히려 보기 어렵다
- 알림은 많을수록 안전하다 → 알림 피로(alert fatigue)를 만들어 정작 중요한 알림을 무시하게 된다
- 프론트엔드는 서버 로그만 보면 된다 → 클라이언트 에러 추적도 중요하다

### 예시: 프론트엔드에서 자주 쓰는 도구

```
로그:
- Sentry (클라이언트 에러 추적)
- DataDog Logs / CloudWatch Logs (서버 로그)

메트릭:
- Web Vitals (LCP, INP, CLS 등 성능 지표)
- 커스텀 메트릭 (페이지별 로딩 시간, API 실패율)
- DataDog / Grafana 대시보드

알림:
- Sentry 알림 (새로운 에러, 에러 급증)
- Slack / PagerDuty 연동
- CloudWatch Alarm
```

```ts
// Sentry 초기화 예시
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: 0.1, // 10% 샘플링
  environment: process.env.NEXT_PUBLIC_ENV,
});
```

### 실무 연결

프론트엔드/웹 운영에서 자주 보는 것:

- 배포 직후 에러율 증가
- API 실패율 증가
- 특정 페이지 성능 저하
- 클라이언트 런타임 에러
- 이미지/자산 로드 실패
- 로그인/업로드 실패 증가

이런 신호를 메트릭으로 잡고, 임계치를 넘으면 알림이 가고, 로그를 보고 원인을 파악하는 흐름이 세트로 돌아야 한다.

### 면접 압축 답변

- **10초**: 로그는 디테일, 메트릭은 추세, 알림은 감지 장치입니다. 셋을 세트로 설계해야 문제를 빨리 발견하고 대응할 수 있습니다.
- **30초**: 로그는 사건 단위 기록이고, 메트릭은 추세를 보는 숫자이며, 알림은 이상 상태를 사람에게 알려주는 장치입니다. 프론트엔드에서는 Sentry로 클라이언트 에러를 추적하고, Web Vitals로 성능을 측정하며, 임계치 초과 시 슬랙 알림을 보내는 식으로 세트를 구성합니다. 셋 중 하나만이 아니라 함께 설계해야 운영 안정성이 높아집니다.

> 운영 가시성의 기본을 봤으니, 마지막으로 **모노레포에서의 CI/CD 전략**을 살펴보자.

---

## 145. 모노레포에서는 CI/CD 전략이 왜 더 중요해지나

모노레포는 생산성을 높일 수 있지만,
파이프라인 설계가 단순 레포보다 더 중요해지기 쉽다.

### 핵심 개념

모노레포에서는:

- 여러 앱/패키지가 한 저장소에 있고
- 공용 패키지 변경이 여러 앱에 영향을 줄 수 있으며
- 모든 걸 매번 전부 빌드/테스트하면 비용이 커질 수 있다

### 왜 이게 중요한가

이 감각이 없으면:

- PR마다 전체 빌드가 너무 오래 걸리고
- 사소한 변경에도 모든 앱이 다시 검사되며
- 파이프라인이 팀 생산성을 심하게 떨어뜨릴 수 있다

### 자주 하는 오해

- 모노레포면 무조건 다 같이 빌드해야 안전하다 → 영향을 받은 범위 중심 전략이 필요할 수 있다
- 공용 패키지가 있으니 분리 배포는 의미 없다 → 앱별 배포 기준도 중요하다
- 캐시를 잘 쓰면 모든 문제가 해결된다 → 영향 범위 계산과 작업 그래프 설계가 같이 중요하다

### 예시

```yaml
# Turborepo 기반 affected 범위만 검사하는 예시
name: ci

on:
  pull_request:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 변경 범위 비교를 위해 전체 히스토리 필요
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      # --filter로 변경된 패키지만 lint/test/build
      - run: pnpm turbo lint test build --filter=...[HEAD~1]
```

Turborepo, Nx 같은 도구는 변경 범위를 자동으로 계산해서
영향을 받은 앱/패키지만 선별적으로 검사하고 빌드할 수 있다.

### 실무 연결

좋은 모노레포 CI 전략은 보통:

- affected apps/packages 계산
- 공용 패키지 변경 시 영향 범위 확대
- 캐시 활용
- 앱별 preview deploy
- 공용 타입/유틸의 회귀 검사

를 같이 본다.

### 면접 압축 답변

- **10초**: 모노레포에서는 변경마다 전체를 빌드하면 비용이 크므로, 영향 범위만 선별 검사하는 전략이 중요합니다.
- **30초**: 모노레포에서는 여러 앱과 공용 패키지가 한 저장소에 묶여 있기 때문에, 모든 변경마다 전체를 빌드하고 테스트하면 CI 비용이 급격히 커질 수 있습니다. Turborepo나 Nx 같은 도구로 영향을 받은 범위만 선별적으로 검사하고 배포하는 전략이 특히 중요해집니다.

## 146. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 프론트엔드도 배포와 운영 구조를 이해해야 한다
- CI와 CD는 품질 게이트와 배포 흐름으로 나뉜다
- GitHub Actions 파이프라인은 빠른 실패와 재현성을 설계해야 한다
- 캐시와 artifact는 목적이 다르다
- Docker는 환경 일관성과 재현성을 높인다
- Dockerfile은 배포 환경 문서이자 재현성 선언이다
- 레이어 캐시와 multi-stage build는 속도와 크기, 보안에 중요하다
- Nginx는 정적 서빙과 프록시, HTTPS 앞단 역할을 맡는다
- PM2는 Node 프로세스 운영을 안정화한다
- EC2 배포는 자유도와 운영 책임을 함께 가져온다
- 환경변수는 환경 분리와 비밀값 관리의 경계다
- 개발/스테이징/운영 분리는 배포 안전성을 높인다
- S3/CloudFront 같은 구조는 정적 자산 배포와 전달 최적화에 강하다
- 롤백은 배포 전략의 핵심이다
- 로그, 메트릭, 알림은 운영 가시성의 기본이다
- 모노레포는 CI/CD 전략을 더 정교하게 만든다

즉,
이번 챕터는 단순 인프라 용어 암기가 아니라,
**"어떻게 더 안전하게 빌드하고, 배포하고, 감지하고, 복구할 것인가"**를 배우는 챕터다.
