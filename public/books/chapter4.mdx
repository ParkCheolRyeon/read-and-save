# 챕터 4. TypeScript에서 런타임 검증까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **TypeScript는 내부 계약을 더 명확하게 만드는 도구이고, Zod 같은 런타임 검증은 외부 경계를 안전하게 지키는 도구라는 흐름으로 이해해야 한다.**

## 33. 왜 TypeScript를 쓰는가

TypeScript는 프론트엔드 면접에서 거의 빠지지 않는 주제다.  
그런데 여기서 “자동완성이 좋아서요” 정도로만 말하면 얕다.

TypeScript의 핵심 가치는 단순 편의성이 아니라,  
**코드베이스 안의 계약을 명시하고, 그 계약이 깨지는 지점을 더 빨리 발견하게 해주는 것**에 있다.

### 핵심 개념

- TypeScript는 자바스크립트 위에 얹힌 **정적 타입 시스템**이다.
- 런타임 전에 코드의 형태와 제약을 검사할 수 있게 해준다.
- 특히 함수 인자, 반환값, 객체 구조, 공용 API의 계약을 더 명확하게 표현할 수 있다.

### 왜 이게 중요한가

프론트엔드 앱이 커질수록 아래 문제가 많아진다.

- 함수가 어떤 데이터를 기대하는지 헷갈림
- API 응답 구조를 잘못 가정함
- 리팩토링 중에 곳곳이 깨짐
- 공용 컴포넌트 사용 규칙이 흐려짐

TypeScript는 이 문제를 전부 없애주지는 않지만,  
적어도 “틀린 연결”을 더 빨리 드러나게 해준다.

### 자주 하는 오해

- TypeScript를 쓰면 버그가 사라진다 → 틀림
- TypeScript는 개발 속도를 무조건 느리게 만든다 → 상황에 따라 다르다
- 자동완성이 좋아지는 게 핵심이다 → 핵심은 계약 명시와 리팩토링 안정성이다
- TypeScript를 쓰면 런타임 검증도 필요 없다 → 틀림

### 실무 연결

특히 프론트엔드에서는:

- 컴포넌트 props
- 상태 구조
- API 응답 타입
- 폼 데이터 구조
- 공용 훅 반환 타입

같은 지점에서 TypeScript 이점이 크다.

### 여기서 다음 개념으로 어떻게 넘어가나

TypeScript를 쓰는 이유를 알았다면, 이제 자연스럽게 이런 질문이 나온다.

**그 계약을 어떤 문법으로 표현할 것인가?**

그래서 다음은 `type`과 `interface`로 넘어간다.

### 면접 압축 답변

TypeScript의 핵심 가치는 자동완성보다 코드베이스 안의 계약을 명시하고, 잘못된 연결을 더 빨리 발견하게 해준다는 데 있습니다. 특히 프론트엔드에서는 props, API 응답, 공용 컴포넌트와 훅의 계약을 명확하게 표현하는 데 큰 도움이 됩니다.

## 34. type과 interface는 어떻게 구분해야 하나

TypeScript를 쓰기 시작하면 가장 먼저 자주 맞닥뜨리는 질문이 `type`과 `interface`다.  
많은 사람이 둘을 “거의 비슷한 것” 정도로만 외우는데, 실무에서는 의도를 나눠 쓰는 감각이 중요하다.

### 핵심 개념

둘 다 객체 구조를 표현할 수 있지만, 성격에는 차이가 있다.

#### `interface`

- 객체 형태의 계약을 표현하는 데 자연스럽다
- 선언 병합이 가능하다
- 클래스 구현 계약과도 잘 어울린다

#### `type`

- 객체뿐 아니라 유니온, 인터섹션, 튜플, 원시 타입 별칭 등 더 넓은 표현이 가능하다
- 더 “타입 연산” 친화적이다

### 왜 이게 중요한가

이 질문이 중요한 이유는 단순 문법 선호 문제가 아니라,  
**내가 지금 표현하려는 것이 객체 계약인지, 타입 조합인지**를 구분하게 만들기 때문이다.

### 자주 하는 오해

- `type`과 `interface`는 완전히 똑같다 → 틀림
- 무조건 `interface`가 더 좋다 → 틀림
- 무조건 `type`이 최신이라 더 좋다 → 틀림
- 둘 중 하나만 써야 일관성 있다 → 팀 기준에 따라 다르지만, 용도 분리가 더 중요할 수 있다

### 추천 감각

실무에선 보통 이렇게 가져가면 깔끔하다.

- 객체 shape 중심 계약 → `interface`
- 유니온/인터섹션/조합형 타입 → `type`

예시:

```ts
interface User {
  id: string;
  name: string;
}

type Status = "idle" | "loading" | "success" | "error";

type UserWithStatus = User & {
  status: Status;
};
```

### 여기서 다음 개념으로 어떻게 넘어가나

이제 “타입을 어떻게 표현하는가”를 봤다.  
그 다음엔 “안전하지 않은 타입과 안전한 타입은 무엇이 다른가”로 넘어가야 한다.

그래서 다음은 `any`, `unknown`, `never`다.

### 면접 압축 답변

`interface`는 객체 계약을 표현할 때 자연스럽고, `type`은 유니온이나 인터섹션 같은 타입 조합까지 더 폭넓게 표현할 수 있습니다. 그래서 객체 shape 중심이면 `interface`, 조합형 타입 설계가 많으면 `type`이 더 잘 맞는 경우가 많습니다.

## 35. any는 왜 위험하고, unknown은 왜 더 안전한가

TypeScript를 도입했는데도 코드가 불안한 경우가 있다.  
그럴 때 자주 보이는 원인이 `any`다.

### 핵심 개념

#### `any`

- 타입 검사를 사실상 꺼버린다
- 어떤 값이든 들어갈 수 있고, 어떤 연산도 허용된다

#### `unknown`

- 어떤 값이 들어올 수 있다는 점은 같지만
- 사용하기 전에 **타입을 좁히는 과정**이 필요하다

### 왜 이게 중요한가

`any`를 많이 쓰면 겉으로는 TypeScript 프로젝트인데,  
실제로는 중요한 경계에서 타입 안전성이 거의 없는 상태가 된다.

반면 `unknown`은  
“아직 뭔지 확실하지 않다”는 사실을 타입 시스템에 솔직하게 표현한다.

즉:

- `any`는 모르는 척하고 지나가는 느낌
- `unknown`은 모른다는 걸 인정하고 확인을 강제하는 느낌

이다.

### 예시

```ts
function logLength(value: any) {
  console.log(value.length);
}
```

이 코드는 타입 에러가 안 난다.  
하지만 런타임에 숫자가 들어오면 바로 깨질 수 있다.

```ts
function logLength(value: unknown) {
  if (typeof value === "string" || Array.isArray(value)) {
    console.log(value.length);
  }
}
```

이 방식은 사용 전에 타입을 확인하게 만든다.

### 자주 하는 오해

- `unknown`은 쓸모없는 너무 엄격한 타입이다 → 틀림
- `any`는 임시로만 쓰면 괜찮다 → 임시가 영구가 되기 쉽다
- 외부 API 응답은 어차피 모르는 데이터니까 `any`가 편하다 → 그래서 더 위험하다

### 실무 연결

특히 아래 경계에선 `unknown` 감각이 중요하다.

- 외부 API 응답
- 사용자 입력
- `postMessage`
- localStorage 파싱 결과
- JSON 파싱 결과

이런 값은 내부 코드가 만든 것이 아니므로,  
처음부터 믿지 않는 게 맞다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 “모르는 값”을 안전하게 다루는 필요성이 보였다.  
그 다음 질문은 자연스럽다.

**그럼 unknown 값을 실제로 쓸 수 있게 만들려면 어떻게 해야 하나?**

여기서 narrowing이 나온다.

### 면접 압축 답변

`any`는 타입 검사를 사실상 꺼버리기 때문에 TypeScript의 장점을 많이 잃게 만듭니다. 반면 `unknown`은 값이 무엇인지 아직 모른다는 사실을 표현하고, 사용 전에 타입을 확인하게 강제해서 더 안전합니다.

## 36. never는 왜 필요한가

`never`는 처음 보면 가장 낯설다.  
하지만 이 타입은 “절대 일어나지 않아야 하는 상태”를 드러내는 데 매우 강하다.

### 핵심 개념

`never`는 **도달할 수 없는 값**, 또는 **절대 반환되지 않는 경우**를 표현한다.

대표적인 경우:

- 항상 throw만 하는 함수
- 무한 루프
- exhaustive check에서 남아선 안 되는 분기

### 왜 이게 중요한가

실무에서 `never`가 진짜 빛나는 지점은  
유니온 타입 분기를 빠짐없이 처리했는지 확인할 때다.

예시:

```ts
type Status = "idle" | "loading" | "success" | "error";

function renderStatus(status: Status) {
  switch (status) {
    case "idle":
      return "대기 중";
    case "loading":
      return "로딩 중";
    case "success":
      return "성공";
    case "error":
      return "에러";
    default: {
      const _exhaustiveCheck: never = status;
      return _exhaustiveCheck;
    }
  }
}
```

나중에 `"cancelled"` 같은 상태가 추가되었는데 switch에서 빠뜨리면,  
`never` 체크가 깨져서 누락을 더 빨리 잡아낼 수 있다.

### 자주 하는 오해

- `never`는 거의 안 쓰는 마이너 문법이다 → 실무에서 분기 안정성에 꽤 유용하다
- `never`는 그냥 `void`와 비슷하다 → 전혀 다르다
- `never`는 아무 값이나 담을 수 있다 → 오히려 반대다

### 여기서 다음 개념으로 어떻게 넘어가나

`unknown`과 `never`를 봤으니,  
이제 “모르는 값을 어떻게 점점 더 구체적으로 바꿔 가는가”를 봐야 한다.

즉 narrowing으로 넘어간다.

### 면접 압축 답변

`never`는 절대 발생하면 안 되는 상태나 도달할 수 없는 분기를 표현하는 데 사용됩니다. 특히 유니온 타입을 switch로 분기할 때 exhaustive check에 활용하면, 새로운 케이스를 빠뜨렸을 때 컴파일 단계에서 더 빨리 잡아낼 수 있습니다.

## 37. Narrowing은 왜 TypeScript의 핵심 감각인가

TypeScript를 처음 쓸 때는 타입 선언 자체에만 집중하기 쉽다.  
하지만 실제로 중요한 건 선언보다도 **흐름 속에서 타입을 점점 더 구체화하는 능력**이다.  
그게 narrowing이다.

### 핵심 개념

Narrowing은 넓은 타입을 조건 검사나 제어 흐름을 통해 더 구체적인 타입으로 좁혀 가는 과정이다.

예:

```ts
function printValue(value: string | number) {
  if (typeof value === "string") {
    console.log(value.toUpperCase());
  } else {
    console.log(value.toFixed(2));
  }
}
```

여기서 `typeof` 검사 이후엔 TypeScript가 분기별 타입을 다르게 이해한다.

### 왜 이게 중요한가

실무에서 데이터는 처음부터 깔끔한 단일 타입으로만 들어오지 않는다.

- 유니온 타입
- nullable 값
- 외부 응답
- 선택적 필드
- 이벤트 타입 분기

이런 곳에서 narrowing이 없으면 타입 시스템이 너무 불편하게 느껴진다.

### 자주 하는 오해

- TypeScript는 선언만 잘하면 끝이다 → 틀림. 흐름 속 타입 좁히기가 더 중요할 때가 많다
- `as`로 강제 캐스팅하면 narrowing 대신 써도 된다 → 위험하다
- 런타임 조건과 타입 시스템은 별개다 → TypeScript는 제어 흐름을 읽어 타입을 좁힌다

### narrowing을 만드는 대표 수단

- `typeof`
- `instanceof`
- `in`
- truthy/falsy 검사
- equality check
- 사용자 정의 타입 가드

### 여기서 다음 개념으로 어떻게 넘어가나

이제 narrowing이 왜 중요한지는 보였다.  
그 다음엔 넓은 타입을 더 세밀하게 다룰 때 자주 붙는 **유니온과 인터섹션**으로 넘어가야 한다.

### 면접 압축 답변

Narrowing은 넓은 타입을 조건 검사와 제어 흐름을 통해 더 구체적인 타입으로 좁혀 가는 과정입니다. TypeScript를 실무에서 잘 쓰려면 선언 자체보다도 이런 흐름 속 타입 좁히기를 잘 다루는 게 더 중요합니다.

## 38. 유니온과 인터섹션은 어떻게 다르게 생각해야 하나

TypeScript에서 타입 설계를 시작하면 곧바로 만나는 게 유니온과 인터섹션이다.

### 핵심 개념

#### 유니온 `|`

“여러 가능성 중 하나”를 표현한다.

```ts
type InputValue = string | number;
```

#### 인터섹션 `&`

“여러 속성을 동시에 모두 만족하는 것”을 표현한다.

```ts
type BaseUser = { id: string; name: string };
type WithRole = { role: "admin" | "user" };

type User = BaseUser & WithRole;
```

### 왜 이게 중요한가

이 개념은 곧바로 아래 설계로 이어진다.

- 상태 머신 설계
- API 응답 성공/실패 모델링
- 공용 props 조합
- feature flag별 타입 조합

### 자주 하는 오해

- 유니온은 타입 여러 개를 다 합치는 거다 → “그중 하나”에 더 가깝다
- 인터섹션은 겹치는 속성만 남긴다 → 집합 개념을 기계적으로 대입하면 헷갈린다
- 유니온이면 모든 멤버의 속성을 바로 쓸 수 있다 → narrowing이 먼저 필요할 수 있다

### 실무 연결

예를 들어 네트워크 상태를 이렇게 모델링하면 설명력이 좋아진다.

```ts
type RequestState =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: string[] }
  | { status: "error"; message: string };
```

이런 형태는 그냥 `loading: boolean`, `error: string | null`, `data: string[] | null`처럼 흩어진 상태보다 훨씬 의미가 명확하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 타입을 조합하는 감각이 생겼다.  
그 다음엔 반복되는 구조를 재사용 가능하게 만드는 **제네릭**으로 넘어가야 한다.

### 면접 압축 답변

유니온은 여러 가능성 중 하나를 표현하고, 인터섹션은 여러 속성을 동시에 만족하는 타입을 표현합니다. 실무에서는 상태 모델링이나 공용 props 설계, API 성공/실패 구조를 더 명확하게 표현할 때 유용합니다.

## 39. 제네릭은 왜 재사용성과 안전성을 동시에 주나

제네릭은 TypeScript에서 가장 중요한 축 중 하나다.  
그런데 처음엔 어렵게 느껴지기 쉽다. 이유는 “타입 매개변수”라는 표현이 추상적으로 들리기 때문이다.

### 핵심 개념

제네릭은 **구체 타입을 나중에 주입할 수 있게 남겨둔 타입 매개변수**라고 생각하면 된다.

즉:

- 구조는 재사용하고
- 실제 타입은 사용하는 쪽에서 결정하게 하는 방식이다

### 예시

```ts
function identity<T>(value: T): T {
  return value;
}
```

이 함수는 문자열이 들어오면 문자열을, 숫자가 들어오면 숫자를 그대로 보존한다.

### 왜 이게 중요한가

제네릭이 없으면:

- 너무 구체적인 함수가 되거나
- `any`를 써서 안전성이 떨어지기 쉽다

제네릭을 쓰면:

- 재사용성
- 타입 추론
- 입력과 출력 간 관계 보존

을 동시에 얻을 수 있다.

### 실무 연결

프론트엔드에서는 제네릭이 아래에서 특히 많이 중요해진다.

- 공용 리스트 컴포넌트
- 테이블 컴포넌트
- 폼 유틸
- API 래퍼 함수
- 커스텀 훅

예시:

```ts
type ApiResponse<T> = {
  data: T;
  success: boolean;
};

async function fetchJson<T>(url: string): Promise<ApiResponse<T>> {
  const response = await fetch(url);
  return response.json();
}
```

### 자주 하는 오해

- 제네릭은 라이브러리 만드는 사람만 쓴다 → 틀림
- 제네릭은 복잡해서 피하는 게 낫다 → 너무 피하면 `any`와 중복 코드가 늘어난다
- `<T>`는 무조건 아무 타입이나 받는 마법이다 → 제약 조건을 줄 수도 있다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 제네릭이 왜 필요한지는 알았다.  
그 다음엔 “제네릭도 아무 타입이나 받게 두는 게 아니라, 제한을 걸 수 있어야 한다”는 점으로 넘어가야 한다.

즉 extends와 제약 조건으로 이어진다.

### 면접 압축 답변

제네릭은 구조는 재사용하되 실제 타입은 사용하는 쪽에서 나중에 결정하게 하는 방식입니다. 그래서 입력과 출력의 관계를 유지하면서도 `any` 없이 유연한 공용 함수와 컴포넌트를 만들 수 있습니다.

---

## 40. 제네릭 제약 extends는 왜 필요한가

제네릭이 자유롭기만 하면 오히려 내부에서 안전하게 다루기 어려울 수 있다.  
그래서 필요한 게 제약 조건이다.

### 핵심 개념

`extends`를 사용하면 “이 제네릭 타입은 최소한 이런 조건은 만족해야 한다”라고 제한할 수 있다.

예:

```ts
function getLength<T extends { length: number }>(value: T) {
  return value.length;
}
```

이 함수는 `length`가 있는 값만 받을 수 있다.

### 왜 이게 중요한가

이 제약이 없으면 함수 내부에서 `.length`를 안전하게 쓸 수 없다.  
즉 제네릭은 자유로움만이 아니라 **안전한 최소 조건**과 함께 가야 한다.

### 실무 연결

- `id`가 있는 데이터 목록
- `value`/`label` 구조를 가진 옵션
- key 기반 테이블 컬럼
- 특정 shape가 보장되어야 하는 훅 반환값

이런 곳에서 제네릭 제약은 매우 자주 쓰인다.

### 자주 하는 오해

- 제네릭은 항상 완전 자유 타입이어야 한다 → 틀림
- `extends`는 상속 개념이랑 완전히 동일하다 → 타입 제약으로 이해하는 게 더 실무적이다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 재사용 가능한 타입 구조를 더 잘 만들 수 있게 됐다.  
그 다음엔 이런 타입을 더 편하게 다루는 도구인 `keyof`, `typeof`, `in`, 그리고 유틸리티 타입으로 넘어가야 한다.

### 면접 압축 답변

제네릭 제약은 타입 매개변수에 최소 조건을 부여해서 함수 내부에서 더 안전하게 값을 다루게 해줍니다. 즉 유연성만 주는 것이 아니라, 안전한 재사용성을 만드는 장치라고 볼 수 있습니다.

---

## 41. keyof, typeof, in은 왜 타입 설계에서 자주 쓰이나

TypeScript가 강력한 이유는 단순히 타입을 적을 수 있어서가 아니라,  
기존 값과 구조에서 타입 정보를 끌어와 재사용할 수 있기 때문이다.

### 핵심 개념

#### `keyof`

객체 타입의 key들을 유니온으로 뽑아낸다.

```ts
type User = {
  id: string;
  name: string;
};

type UserKey = keyof User; // "id" | "name"
```

#### `typeof`

값으로부터 타입을 추론해 가져온다.

```ts
const defaultUser = {
  id: "1",
  name: "park",
};

type DefaultUser = typeof defaultUser;
```

#### `in`

매핑된 타입에서 key를 순회할 때 자주 사용한다.

```ts
type Flags<T> = {
  [K in keyof T]: boolean;
};
```

### 왜 이게 중요한가

이 도구들이 있으면:

- 중복 타입 선언을 줄이고
- 값과 타입을 더 가깝게 유지하고
- 공용 컴포넌트와 유틸 타입을 더 강력하게 만들 수 있다

### 자주 하는 오해

- 타입은 항상 수동으로 일일이 적어야 한다 → 틀림
- `typeof`는 런타임 자바스크립트 `typeof`와 완전히 같은 개념이다 → 문맥이 다르다
- `keyof`는 객체 값을 뽑는다 → key 이름을 뽑는다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 타입 조작의 기본 재료를 봤다.  
그 다음엔 실무에서 아주 자주 쓰는 **유틸리티 타입**으로 넘어갈 수 있다.

### 면접 압축 답변

`keyof`, `typeof`, `in`은 기존 타입이나 값에서 정보를 끌어와 재사용 가능한 타입을 만드는 데 자주 쓰입니다. 덕분에 타입 중복을 줄이고, 값과 타입의 일관성을 더 잘 유지할 수 있습니다.

---

## 42. 유틸리티 타입은 왜 실무에서 자주 보이나

TypeScript를 쓰다 보면 반복되는 타입 변형 작업이 많다.  
이때 유틸리티 타입은 일종의 표준 도구 상자 역할을 한다.

### 자주 쓰는 유틸리티 타입

- `Partial<T>`
- `Required<T>`
- `Pick<T, K>`
- `Omit<T, K>`
- `Record<K, T>`
- `Readonly<T>`

### 왜 이게 중요한가

이걸 잘 쓰면:

- 수정용 payload 타입
- 일부 필드만 필요한 뷰 모델
- 키-값 맵 구조
- 읽기 전용 설정 객체

같은 걸 매번 새로 작성하지 않아도 된다.

### 예시

```ts
interface User {
  id: string;
  name: string;
  age: number;
}

type UserPreview = Pick<User, "id" | "name">;
type UserPatch = Partial<User>;
type UserMap = Record<string, User>;
```

### 자주 하는 오해

- 유틸리티 타입을 많이 쓰면 무조건 고급스럽다 → 과하면 읽기 어려워질 수 있다
- `Partial`은 폼 수정에 항상 정답이다 → 상태 의미가 불명확해질 수 있다
- `Omit`은 마음대로 빼도 된다 → 도메인 의미를 흐릴 수 있다

### 실무 연결

가독성을 해치지 않는 선에서 쓰는 게 중요하다.  
즉 “짧아서 좋은 타입”이 아니라,  
**읽었을 때 의도가 선명한 타입**이 좋은 타입이다.

### 여기서 다음 개념으로 어떻게 넘어가나

유틸리티 타입까지 보면 타입 조작 감각이 꽤 생긴다.  
그 다음엔 literal type과 `as const`처럼 값을 더 좁게 고정하는 방식으로 이어질 수 있다.

### 면접 압축 답변

유틸리티 타입은 반복되는 타입 변형 작업을 표준화해주는 도구입니다. `Pick`, `Omit`, `Partial`, `Record` 등을 활용하면 중복 선언을 줄일 수 있지만, 과하게 쓰면 의도가 흐려질 수 있어서 가독성과 함께 봐야 합니다.

---

## 43. as const는 왜 단순 문법이 아니라 의도 표현인가

TypeScript는 기본적으로 값을 어느 정도 넓게 추론한다.  
그런데 어떤 경우엔 오히려 값을 더 좁게 고정해야 의미가 선명해진다.  
여기서 `as const`가 유용하다.

### 핵심 개념

`as const`를 쓰면:

- 리터럴 값을 더 좁게 유지하고
- 객체 속성을 읽기 전용으로 보고
- 배열을 readonly tuple처럼 다룰 수 있다

예:

```ts
const status = "success" as const;
```

이 경우 `status`는 단순 `string`이 아니라 `"success"`라는 더 좁은 타입이 된다.

### 왜 이게 중요한가

이건 특히 아래에서 강하다.

- 상태 상수 정의
- action type 정의
- 옵션 목록 정의
- switch 분기와 exhaustive check 강화

### 자주 하는 오해

- `as const`는 그냥 편하게 쓰는 캐스팅이다 → 단순 캐스팅보다 “값을 더 좁게 고정하는 의도”에 가깝다
- `as const`를 남발하면 좋다 → 과하면 타입이 너무 딱딱해질 수 있다

### 예시

```ts
const REQUEST_STATUS = {
  IDLE: "idle",
  LOADING: "loading",
  SUCCESS: "success",
  ERROR: "error",
} as const;

type RequestStatus = (typeof REQUEST_STATUS)[keyof typeof REQUEST_STATUS];
```

### 여기서 다음 개념으로 어떻게 넘어가나

이제 TypeScript 안에서 내부 계약을 꽤 잘 만들 수 있게 됐다.  
하지만 중요한 질문이 남는다.

**외부에서 들어오는 값도 TypeScript만으로 안전할까?**

여기서 런타임 검증 필요성으로 넘어간다.

### 면접 압축 답변

`as const`는 값을 더 좁은 리터럴 타입으로 고정해서 의도를 더 명확하게 표현하게 해줍니다. 특히 상태 상수, 액션 타입, 옵션 목록처럼 값 자체가 의미를 갖는 구조에서 유용합니다.

---

## 44. TypeScript가 있어도 왜 런타임 검증이 필요한가

여기서 TypeScript 학습의 중요한 전환점이 온다.  
많은 사람이 이 지점에서 처음으로 “아, TypeScript만으로는 충분하지 않구나”를 느낀다.

### 핵심 개념

TypeScript는 **컴파일 타임** 도구다.  
즉 개발 중에는 많은 걸 잡아주지만, 실제 런타임에 들어오는 데이터가 정말 그 타입인지까지는 보장하지 못한다.

### 왜 이게 중요한가

프론트엔드 앱에서 위험한 데이터는 보통 외부에서 들어온다.

- API 응답
- 사용자 폼 입력
- query string
- localStorage 데이터
- 업로드 파일 메타데이터
- iframe/postMessage 데이터

이 값들은 TypeScript 타입 선언을 해놨다고 해서 자동으로 안전해지는 게 아니다.

### 예시

```ts
type User = {
  id: string;
  name: string;
};

async function fetchUser(): Promise<User> {
  const response = await fetch("/api/user");
  return response.json();
}
```

겉으로는 `Promise<User>`처럼 보이지만,  
실제로 서버가 `{ id: 1, nickname: "park" }` 같은 엉뚱한 데이터를 보내도 TypeScript는 런타임에서 막아주지 못한다.

### 자주 하는 오해

- 타입을 선언했으니 응답도 그 타입일 것이다 → 틀림
- 백엔드가 있으니 프론트는 검증 안 해도 된다 → 경계는 항상 조심해야 한다
- `as User`로 캐스팅하면 안전하다 → 오히려 위험을 숨길 수 있다

### 실무 연결

즉 TypeScript는 **내부 코드 계약**엔 강하지만,  
외부 경계는 별도 검증이 필요하다.

이제 여기서 Zod 같은 런타임 검증 도구가 나온다.

### 면접 압축 답변

TypeScript는 컴파일 타임에 내부 코드 계약을 검사해주지만, 실제 런타임에 외부에서 들어오는 데이터가 정말 그 타입인지까지 보장하지는 못합니다. 그래서 API 응답이나 사용자 입력 같은 경계에서는 별도의 런타임 검증이 필요합니다.

---

## 45. Zod는 왜 TypeScript 다음 단계로 자주 언급되나

TypeScript만으로는 런타임 데이터를 믿을 수 없다는 걸 봤다.  
그 다음 단계에서 자주 나오는 도구가 Zod다.

### 핵심 개념

Zod는 런타임에서 데이터 구조를 검증할 수 있게 해주는 스키마 라이브러리다.

즉:

- “이 데이터가 정말 우리가 기대한 shape인가?”를 런타임에 검사하고
- 실패하면 에러를 내거나 fallback 처리하게 해준다

### 왜 이게 중요한가

Zod가 강한 이유는:

- 런타임 검증을 하고
- 그 스키마로부터 TypeScript 타입도 추론할 수 있어서
- 타입 선언과 검증 로직의 괴리를 줄여준다는 점이다

### 예시

```ts
import { z } from "zod";

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
});

type User = z.infer<typeof UserSchema>;
```

이제 이 스키마를 통해:

- 런타임 검증
- TypeScript 타입 추론

둘 다 가져갈 수 있다.

### 자주 하는 오해

- Zod가 있으면 TypeScript가 필요 없다 → 틀림. 역할이 다르다
- TypeScript가 있으면 Zod는 중복이다 → 틀림. 검증 시점이 다르다
- 모든 데이터에 무조건 Zod를 붙여야 한다 → 비용과 중요도를 같이 봐야 한다

### 실무 연결

Zod는 특히 아래 경계에서 강하다.

- API 응답 파싱
- 폼 입력 검증
- 설정 파일 검증
- URL search params 파싱
- localStorage 복원 데이터 검증

### 여기서 다음 개념으로 어떻게 넘어가나

이제 TypeScript와 Zod가 각자 뭘 담당하는지 구분할 수 있어야 한다.  
즉 다음은 **정적 계약과 런타임 검증의 역할 분리**로 이어진다.

### 면접 압축 답변

Zod는 런타임에서 실제 데이터 구조를 검증하는 도구입니다. TypeScript가 컴파일 타임 계약을 다룬다면, Zod는 외부에서 들어오는 값을 실제로 확인하는 역할을 하므로 둘은 대체 관계가 아니라 보완 관계입니다.

---

## 46. TypeScript와 Zod는 어떻게 역할을 나눠 생각해야 하나

여기서 많은 사람이 가장 헷갈린다.  
“둘 다 타입 관련인데 뭐가 다른 거지?”

가장 깔끔한 감각은 이렇다.

### 핵심 개념

- TypeScript는 **내부 코드 계약**
- Zod는 **외부 경계 검증**

즉:

- 팀 내부에서 작성한 함수, 컴포넌트, 상태 구조는 TypeScript로 명확히 하고
- 외부에서 들어오는 불신 데이터는 Zod로 확인한다

### 왜 이게 중요한가

이 구분이 서면 타입 설계가 훨씬 선명해진다.

예를 들어:

- 컴포넌트 props → TypeScript
- API 응답 원본 → Zod 검증
- 검증 통과한 결과를 내부 모델로 사용 → TypeScript
- 폼 입력 → Zod + TypeScript
- 공용 도메인 타입 → TypeScript
- 런타임 파싱 → Zod

### 자주 하는 오해

- 내부 객체도 전부 Zod로 감싸야 한다 → 필요 이상일 수 있다
- 외부 응답도 그냥 타입 선언만 하면 된다 → 위험하다
- `z.infer`만 쓰면 모든 타입 설계가 해결된다 → 도메인 의미를 생각한 별도 타입 설계가 더 나을 때도 있다

### 실무 연결

좋은 설계는 보통 이런 흐름이다.

1. 외부 입력 수신
2. Zod로 검증
3. 통과한 데이터를 내부 모델로 사용
4. 내부에서는 TypeScript 계약을 기반으로 안전하게 조작

### 여기서 다음 개념으로 어떻게 넘어가나

이제 타입과 검증의 큰 흐름을 봤다.  
그 다음엔 이 감각을 실제 프론트엔드 구조에 적용해야 한다.

예를 들면:

- 공용 훅 반환 타입 설계
- API wrapper 타입 설계
- 폼 입력 검증 흐름
- 모노레포에서 타입 공유

### 면접 압축 답변

TypeScript는 내부 코드의 계약을 명확하게 만드는 도구이고, Zod는 외부에서 들어오는 불신 데이터를 런타임에 검증하는 도구입니다. 그래서 둘은 중복이 아니라 역할이 다르며, 보통 외부 경계에선 Zod로 확인하고 내부에선 TypeScript 계약으로 운영하는 방식이 가장 깔끔합니다.

## 47. API 응답은 왜 처음부터 믿지 않는 게 좋은가

이건 사실 앞의 흐름을 실무 감각으로 다시 압축한 질문이다.  
프론트엔드에서 API 응답은 “우리 시스템에서 온 데이터”처럼 보여도, 결국 외부 경계다.

### 핵심 개념

API 응답은 아래 이유로 항상 기대와 다를 수 있다.

- 백엔드 배포 버전 차이
- 필드 누락
- 타입 변경
- 부분 실패
- nullable 값 증가
- 임시 대응 코드 유입

### 왜 이게 중요한가

응답을 처음부터 완전히 신뢰하면:

- UI가 특정 필드에 의존하다 깨지고
- 옵셔널 체인이 남발되고
- 에러 위치가 화면 렌더 시점까지 밀리고
- 디버깅이 어려워진다

반대로 경계에서 검증하고 normalize하면,  
내부에서는 더 선명한 타입과 도메인 모델로 작업할 수 있다.

### 실무 연결

예를 들어:

- 원본 응답 DTO와
- 내부 UI 모델을

분리하는 감각이 중요하다.

즉 서버가 주는 shape와,  
UI가 정말 필요로 하는 shape를 분리하는 것이다.

### 자주 하는 오해

- 서버 타입 생성만 받아오면 충분하다 → 실제 응답 신뢰 문제는 여전히 남을 수 있다
- optional을 많이 붙이면 안전하다 → 오히려 내부 코드가 흐려질 수 있다

### 여기서 다음 개념으로 어떻게 넘어가나

API 응답을 믿지 않는 감각은 자연스럽게  
**폼 입력 검증**과 **URL/query parameter 파싱**으로도 이어진다.

### 면접 압축 답변

API 응답은 타입 선언이 있다고 해서 자동으로 안전해지지 않기 때문에, 처음부터 완전히 신뢰하기보다 경계에서 검증하고 필요하면 내부 모델로 정규화하는 편이 더 안정적입니다.

---

## 48. 폼 입력과 URL 파라미터는 왜 타입 선언만으로 부족한가

프론트엔드에서 외부 입력의 대표는 API만이 아니다.  
사실 사용자 입력과 URL 파라미터도 매우 중요한 외부 경계다.

### 핵심 개념

아래 값들은 모두 “문자열 기반이거나 사용자 행동에 의해 깨질 수 있는 값”이다.

- form input
- search params
- route params
- localStorage
- 파일 메타데이터

즉 TypeScript 타입을 적어놨다고 해서 실제 값이 그 모양이라는 보장은 없다.

### 왜 이게 중요한가

예를 들어:

- 숫자여야 하는 값이 빈 문자열일 수 있고
- enum처럼 기대한 값이 아닌 임의 문자열이 들어올 수 있고
- 필수값이 누락될 수 있다

이런 상황은 폼/URL 단계에서 한번 정리해줘야  
이후 로직이 훨씬 단순해진다.

### 실무 연결

특히 아래 조합이 강하다.

- React Hook Form + Zod
- URLSearchParams 파싱 + Zod
- 설정 객체 로딩 + Zod

### 자주 하는 오해

- input type이 number면 값도 number다 → DOM input 값은 기본적으로 문자열이다
- query param은 우리가 만든 링크니까 안전하다 → 사용자가 직접 수정하거나 공유 링크가 깨질 수 있다
- 화면 단 validation만 해도 충분하다 → 경계 검증은 별도로 보는 게 좋다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 TypeScript 챕터는 거의 마무리다.  
다음에는 다시 React 렌더링 모델 쪽으로 올라가서,  
타입 계약이 실제 컴포넌트 구조 설계와 어떻게 연결되는지를 봐야 한다.

### 면접 압축 답변

폼 입력과 URL 파라미터는 TypeScript 타입을 적어놨다고 해서 실제 값이 그 형태라는 보장이 없습니다. 그래서 숫자 변환, enum 검증, 필수값 체크 같은 런타임 검증이 필요하고, Zod 같은 도구가 이런 경계 정리에 잘 맞습니다.

## 49. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- TypeScript는 내부 코드 계약을 명확히 한다
- `type`과 `interface`는 표현 목적이 조금 다르다
- `any`는 위험하고, `unknown`은 더 정직하다
- `never`는 도달하면 안 되는 상태를 드러낸다
- narrowing은 넓은 타입을 실제 흐름 속에서 좁혀 가는 핵심 감각이다
- 유니온과 인터섹션은 상태와 도메인 모델을 더 선명하게 만든다
- 제네릭은 재사용성과 타입 관계 보존을 동시에 준다
- 유틸리티 타입과 `as const`는 의도 표현을 더 정교하게 만든다
- TypeScript는 런타임을 보장하지 못하므로 외부 경계엔 검증이 필요하다
- Zod는 그 런타임 검증을 담당한다

즉,  
이번 챕터는 단순 문법 챕터가 아니라  
**“내부 계약은 어떻게 명확히 하고, 외부 데이터는 어떻게 안전하게 들여올 것인가”**를 배우는 챕터다.
