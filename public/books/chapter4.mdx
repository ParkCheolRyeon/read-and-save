# 챕터 4. TypeScript에서 런타임 검증까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **TypeScript는 내부 계약을 더 명확하게 만드는 도구이고, Zod 같은 런타임 검증은 외부 경계를 안전하게 지키는 도구라는 흐름으로 이해해야 한다.**

### 전체 흐름 맵

```
왜 TypeScript를 쓰는가 (계약 명시 도구)
  → type vs interface (어떻게 표현하나)
    → any / unknown / never (안전하지 않은 타입 vs 안전한 타입)
      → narrowing (넓은 타입을 좁혀 가는 감각)
        → 유니온 / 인터섹션 (타입 조합)
          → 제네릭 + 제약 (재사용성과 안전성)
            → keyof / typeof / template literal (타입 연산)
              → 유틸리티 타입 (표준 도구 상자)
                → as const / satisfies (의도 표현)
                  → 런타임 검증의 필요성 (TS의 한계)
                    → Zod (외부 경계 검증)
                      → 실무 적용 (API, 폼, URL)
```

## 32. 왜 TypeScript를 쓰는가

> **핵심 요약** — TypeScript의 핵심 가치는 자동완성이 아니라 **코드베이스 안의 계약을 명시하고, 잘못된 연결을 컴파일 타임에 발견하는 것**이다. 구조적 타이핑 기반이라 Java/C#과는 타입 비교 방식이 다르다.

### 핵심 개념

- TypeScript는 자바스크립트 위에 얹힌 **정적 타입 시스템**이다.
- 런타임 전에 코드의 형태와 제약을 검사할 수 있게 해준다.
- 컴파일 결과물은 순수 자바스크립트이며, **타입 정보는 런타임에 완전히 사라진다.**

### 구조적 타이핑 — TS의 핵심 특성

TypeScript는 Java/C#처럼 **이름(nominal)**이 아니라 **구조(structural)**로 타입을 비교한다. 이름이 달라도 구조가 같으면 호환된다.

```ts
interface Point {
  x: number;
  y: number;
}

interface Coordinate {
  x: number;
  y: number;
}

const p: Point = { x: 1, y: 2 };
const c: Coordinate = p; // OK — 이름이 달라도 구조가 같으면 호환

// 더 많은 속성을 가진 타입도 호환 (초과 속성은 허용)
const p3d = { x: 1, y: 2, z: 3 };
const p2d: Point = p3d; // OK — x, y가 있으므로 Point로 사용 가능
```

이걸 모르면 "왜 다른 이름의 타입인데 에러가 안 나지?"라는 혼란이 생긴다.

| | 구조적 타이핑 (TypeScript) | 명목적 타이핑 (Java/C#) |
|---|---|---|
| **비교 기준** | 구조(shape)가 같으면 호환 | 명시적으로 같은 타입/상속이어야 호환 |
| **장점** | 유연함, 덕 타이핑과 궁합 좋음 | 의도하지 않은 호환 방지 |
| **주의점** | 의도치 않은 타입 호환 가능 | 보일러플레이트 많음 |

> 덕 타이핑(duck typing)은 '오리처럼 걷고 오리처럼 꽥꽥 울면 오리다'라는 비유에서 온 개념으로, 타입 이름이 아니라 실제 가진 속성과 메서드로 호환성을 판단하는 방식이다.

### 실무 에피소드 — 이런 상황에서 만난다

> 리팩토링 중 API 응답 타입의 필드 이름을 바꿨는데, 그 필드를 사용하는 10개 컴포넌트에서 동시에 빨간 줄이 떴다. JS였으면 배포 후에야 "undefined가 보여요" 버그로 발견했을 것. TS 덕분에 컴파일 타임에 모든 영향 범위를 한 번에 파악.

### 자주 하는 오해

- TypeScript를 쓰면 버그가 사라진다 → **틀림.** 타입 수준의 실수를 줄여줄 뿐, 로직 버그는 여전히 발생한다.
- 자동완성이 좋아지는 게 핵심이다 → **핵심은 계약 명시와 리팩토링 안정성이다.**
- TypeScript를 쓰면 런타임 검증도 필요 없다 → **틀림.** 이 챕터 후반부에서 다룬다.
- TypeScript는 Java처럼 이름으로 타입을 비교한다 → **틀림.** 구조적 타이핑이다.

### 면접 압축 답변

- **10초**: TypeScript는 코드베이스 안의 계약을 명시하고 잘못된 연결을 컴파일 타임에 발견하게 해주는 정적 타입 시스템입니다. 구조적 타이핑 기반이라 이름이 아닌 구조로 호환성을 판단합니다.
- **30초**: TypeScript의 핵심 가치는 함수 인자, 반환값, props, API 응답 타입 같은 계약을 명시해서 리팩토링 안정성을 높이는 것입니다. 구조적 타이핑이라 같은 shape이면 이름이 달라도 호환됩니다. 다만 타입 정보는 빌드 시 제거되므로 런타임 검증은 별도로 필요합니다.

> TypeScript를 쓰는 이유를 알았다면, 다음은 **그 계약을 어떤 문법으로 표현할 것인가**다.

## 33. type과 interface는 어떻게 구분해야 하나

> **핵심 요약** — 객체 shape 중심 계약은 `interface`, 유니온/인터섹션/조합형 타입은 `type`. 둘 다 객체를 표현할 수 있지만 선언 병합, 확장 문법, 표현 범위에서 차이가 있다.

### 비교 테이블

| | `interface` | `type` |
|---|---|---|
| **객체 shape** | 자연스러움 | 가능 |
| **유니온/튜플/원시 별칭** | 불가 | 가능 |
| **선언 병합** | 가능 (같은 이름 여러 번 선언 → 합침) | 불가 (같은 이름 → 에러) |
| **확장 방식** | `extends` | `&` (인터섹션) |
| **클래스 implements** | 자연스러움 | 가능하지만 제한적 |
| **컴퓨티드 프로퍼티** | 불가 | 가능 (`in` 키워드) |

### extends vs & — 확장 방식의 차이

```ts
// interface 방식
interface Animal {
  name: string;
}
interface Dog extends Animal {
  breed: string;
}
```

```ts
// type 방식
type Animal = {
  name: string;
};
type Dog = Animal & {
  breed: string;
};
```

둘 다 결과는 같지만, `extends`는 **충돌 시 에러**를 내고 `&`는 **충돌 시 never로 합쳐진다**는 차이가 있다.

```ts
interface A { x: string }
interface B extends A { x: number } // 에러! — string과 number 충돌

type A = { x: string }
type B = A & { x: number } // B["x"]는 string & number → never (조용히 깨짐)
```

### 선언 병합 — 언제 유용한가

```ts
// 라이브러리 타입 확장에 유용
interface Window {
  gtag: (...args: any[]) => void;
}
// 기존 Window 인터페이스에 gtag가 추가됨

// type은 선언 병합이 안 된다
type Window = { gtag: (...args: any[]) => void }; // 에러! — 중복 식별자
```

### type만 가능한 표현들

```ts
type ID = string | number;                    // 유니온
type Pair = [string, number];                 // 튜플
type Callback = (value: string) => void;      // 함수 타입 별칭
type Status = "idle" | "loading" | "error";   // 문자열 리터럴 유니온
```

### 실무 추천 감각

```ts
// 객체 shape 계약 → interface
interface User {
  id: string;
  name: string;
}

// 조합형 타입 → type
type Status = "idle" | "loading" | "success" | "error";

// 인터섹션으로 합성 → type
type UserWithStatus = User & { status: Status };
```

### 실무 에피소드 — 이런 상황에서 만난다

> 팀에서 type과 interface를 혼용하다가, 외부 라이브러리의 Window 타입을 확장해야 하는 상황에서 type으로 선언 병합을 시도하다 에러. interface로 바꾸니 해결. 이후 "객체 shape 계약은 interface, 조합형 타입은 type"이라는 팀 컨벤션을 정함.

### 자주 하는 오해

- `type`과 `interface`는 완전히 똑같다 → **틀림.** 선언 병합, 유니온 표현, 확장 시 충돌 처리 등에서 차이가 있다.
- 무조건 `interface`가 더 좋다 / 무조건 `type`이 최신이라 더 좋다 → **둘 다 틀림.** 용도에 맞게 구분.

### 면접 압축 답변

- **10초**: `interface`는 객체 계약에 자연스럽고 선언 병합이 가능하며, `type`은 유니온이나 인터섹션 같은 조합형 타입까지 표현할 수 있습니다.
- **30초**: 객체 shape 중심이면 `interface`, 유니온/인터섹션/튜플 같은 조합이면 `type`이 더 적합합니다. `interface`는 `extends`로 확장하면 충돌 시 에러를 내고, `type`은 `&`로 합성하면 충돌 시 never가 됩니다. 선언 병합은 라이브러리 타입 확장에 유용하며 `interface`만 가능합니다.

> 이제 "타입을 어떻게 표현하는가"를 봤다. 다음은 **안전하지 않은 타입과 안전한 타입은 무엇이 다른가** — `any`, `unknown`, `never` 비교다.

## 34. any, unknown, never는 어떻게 다른가

> **핵심 요약** — `any`는 타입 검사를 끄는 탈출구, `unknown`은 "모르지만 확인하겠다"는 정직한 표현, `never`는 "절대 발생하면 안 된다"는 바닥 타입. 세 개를 한 축으로 이해해야 한다.

### 3종 비교 테이블

| | `any` | `unknown` | `never` |
|---|---|---|---|
| **의미** | 아무거나 OK (검사 포기) | 아직 뭔지 모름 (확인 필요) | 절대 발생 안 함 (바닥 타입) |
| **대입 받기** | 모든 타입 → any ✅ | 모든 타입 → unknown ✅ | 어떤 타입도 → never ❌ |
| **다른 타입에 대입** | any → 모든 타입 ✅ (위험!) | unknown → 다른 타입 ❌ (narrowing 필요) | never → 모든 타입 ✅ (비어있으니 가능) |
| **프로퍼티 접근** | 자유 (검사 안 함) | 불가 (narrowing 먼저) | 불가 (값 자체가 없음) |
| **용도** | 마이그레이션, 임시 탈출구 | 외부 데이터, catch error | exhaustive check, 불가능한 분기 |

### any의 오염 전파 — 가장 위험한 특성

`any`가 위험한 이유는 단순히 "검사를 안 한다"가 아니라, **다른 타입으로 전파(오염)된다**는 점이다.

```ts
function getConfig(): any {
  return { timeout: 3000 };
}

const config = getConfig();          // config: any
const timeout = config.timeout;      // timeout: any — 오염 전파!
const doubled = timeout * 2;         // doubled: number (any와 연산하면 추론됨)
const label = timeout.toUpperCase(); // 에러 없음! — 런타임에서 터짐

// any가 하나 들어오면 그것을 사용하는 모든 변수가 any로 물든다
```

### any 대신 쓸 수 있는 것들

| 상황 | any 대신 | 이유 |
|---|---|---|
| 외부 데이터 | `unknown` | narrowing 강제 |
| 함수 인자 아무거나 | 제네릭 `<T>` | 타입 관계 보존 |
| 임시로 타입 에러 무시 | `// @ts-expect-error` | 다음 줄만 무시, any 오염 없음 |
| 특정 줄 무시 | `// @ts-ignore` | 에러 유무와 관계없이 무시 (덜 안전) |
| JSON 파싱 결과 | `unknown` + Zod | 런타임 검증 |

```ts
// @ts-expect-error vs @ts-ignore
// @ts-expect-error — 다음 줄에 에러가 있어야 함. 에러가 없으면 오히려 경고
// @ts-ignore — 에러 유무 상관없이 무시. 나중에 에러가 사라져도 모름

// @ts-expect-error: 타입 에러가 예상되는 테스트 코드
expectType<string>(42); // 의도적 에러 — @ts-expect-error로 명시적 무시
```

### unknown — 정직한 "모름"

```ts
function logLength(value: unknown) {
  // value.length; // 에러! — unknown은 바로 사용 불가
  if (typeof value === "string" || Array.isArray(value)) {
    console.log(value.length); // OK — narrowing 이후 사용 가능
  }
}
```

### never — 도달하면 안 되는 곳

```ts
type Status = "idle" | "loading" | "success" | "error";

function renderStatus(status: Status): string {
  switch (status) {
    case "idle": return "대기 중";
    case "loading": return "로딩 중";
    case "success": return "성공";
    case "error": return "에러";
    default: {
      // Status에 새 값이 추가되면 여기서 컴파일 에러
      const _exhaustive: never = status;
      return _exhaustive;
    }
  }
}
// "cancelled"가 추가되면 default에서 never에 string을 대입하려 해서 에러!
```

### never와 조건부 타입

TypeScript 내장 유틸리티 `Exclude`, `Extract`는 never를 이용해 동작한다.

```ts
// Exclude의 동작 원리
type Exclude<T, U> = T extends U ? never : T;

type Result = Exclude<"a" | "b" | "c", "a">;
// "a" extends "a" ? never : "a" → never
// "b" extends "a" ? never : "b" → "b"
// "c" extends "a" ? never : "c" → "c"
// never | "b" | "c" → "b" | "c"
```

never는 유니온에서 **자동으로 사라진다**. 이 성질을 이용해 타입 필터링이 가능하다.

### 실무 에피소드 — 이런 상황에서 만난다

> 레거시 코드에서 API 래퍼 함수가 `any`를 반환하고 있었는데, 그 반환값을 쓰는 모든 곳이 `any`로 물들어서 TypeScript를 쓰는 의미가 거의 없었다. 반환 타입을 `unknown`으로 바꾸고, 각 사용처에서 Zod로 검증하도록 점진적으로 마이그레이션해서 타입 안전성을 회복.

### 자주 하는 오해

- `unknown`은 쓸모없는 너무 엄격한 타입이다 → **틀림.** "모른다"를 정직하게 표현하는 가장 안전한 방법.
- `any`는 임시로만 쓰면 괜찮다 → 임시가 영구가 되기 쉽고, 오염이 전파된다.
- `never`는 거의 안 쓴다 → exhaustive check와 조건부 타입 필터링에서 핵심 역할.

### 면접 압축 답변

- **10초**: `any`는 타입 검사를 끄고 오염이 전파됩니다. `unknown`은 확인 전에 사용할 수 없어서 안전합니다. `never`는 도달 불가능한 분기를 표현하고 exhaustive check에 쓰입니다.
- **30초**: `any`는 모든 타입에 대입 가능하고 모든 연산이 허용돼서 다른 변수까지 오염시킵니다. `unknown`은 받을 수는 있지만 narrowing 없이는 사용할 수 없어서 외부 데이터에 적합합니다. `never`는 바닥 타입으로 exhaustive check와 `Exclude` 같은 조건부 타입 필터링의 핵심입니다.

> `unknown`과 `never`를 봤으니, 이제 "모르는 값을 어떻게 점점 더 구체적으로 바꿔 가는가"를 봐야 한다. 즉 narrowing이다.

## 35. Narrowing은 왜 TypeScript의 핵심 감각인가

> **핵심 요약** — Narrowing은 넓은 타입을 조건 검사와 제어 흐름을 통해 구체적인 타입으로 좁혀 가는 과정이다. TypeScript를 실무에서 잘 쓰려면 선언보다도 이 흐름 속 타입 좁히기가 더 중요하다.

### 핵심 개념

```ts
function printValue(value: string | number) {
  if (typeof value === "string") {
    console.log(value.toUpperCase()); // string으로 좁혀짐
  } else {
    console.log(value.toFixed(2));    // number로 좁혀짐
  }
}
```

### narrowing을 만드는 수단 비교

| 수단 | 용도 | 예시 |
|---|---|---|
| `typeof` | 원시 타입 구분 | `typeof x === "string"` |
| `instanceof` | 클래스 인스턴스 구분 | `x instanceof Date` |
| `in` | 특정 속성 존재 여부 | `"name" in x` |
| truthy/falsy | null/undefined 걸러내기 | `if (x)` |
| equality | 특정 값 비교 | `x === "idle"` |
| **타입 가드 함수** | 복잡한 조건 재사용 | `function isCat(x): x is Cat` |
| **assertion 함수** | 검증 실패 시 throw | `function assertString(x): asserts x is string` |

### 사용자 정의 타입 가드

```ts
interface Cat { meow: () => void }
interface Dog { bark: () => void }

// 반환 타입 `animal is Cat`이 핵심
function isCat(animal: Cat | Dog): animal is Cat {
  return "meow" in animal;
}

function handleAnimal(animal: Cat | Dog) {
  if (isCat(animal)) {
    animal.meow(); // Cat으로 좁혀짐
  } else {
    animal.bark(); // Dog으로 좁혀짐
  }
}
```

### assertion 함수 — 타입 가드와의 차이

타입 가드는 `boolean`을 반환해서 if/else 분기에서 쓰지만, assertion 함수는 **실패 시 throw하고 성공 시 이후 코드에서 타입이 좁혀진다.**

```ts
// assertion 함수: 검증 실패 시 throw, 성공 시 이후 코드에서 타입 보장
function assertString(value: unknown): asserts value is string {
  if (typeof value !== "string") {
    throw new Error(`Expected string, got ${typeof value}`);
  }
}

function process(input: unknown) {
  assertString(input);
  // 이 아래부터 input은 string으로 좁혀짐
  console.log(input.toUpperCase());
}
```

| | 타입 가드 (`is`) | Assertion 함수 (`asserts`) |
|---|---|---|
| **반환** | `boolean` | `void` (실패 시 throw) |
| **사용 패턴** | `if (isX(value))` 분기 | 호출 후 이하 코드에서 타입 보장 |
| **실패 처리** | else 분기로 처리 | 예외 발생 |
| **용도** | 분기가 필요할 때 | "이 시점 이후로는 이 타입이다"를 보장할 때 |

### 실무 에피소드 — 이런 상황에서 만난다

> API 응답에서 `type` 필드에 따라 다른 구조의 데이터가 오는데, 매번 `as`로 캐스팅하니까 실제 데이터가 기대와 다를 때 런타임에서 터졌다. 타입 가드 함수를 만들어서 `type` 필드 기준으로 narrowing하니 각 분기에서 안전하게 접근 가능해졌고, 기대와 다른 데이터가 오면 타입 가드에서 걸려서 조기에 발견.

### 자주 하는 오해

- TypeScript는 선언만 잘하면 끝이다 → **틀림.** 흐름 속 타입 좁히기가 더 중요할 때가 많다.
- `as`로 강제 캐스팅하면 narrowing 대신 써도 된다 → **위험.** `as`는 "내가 맞으니 믿어라"고 강요하는 것이다.
- 런타임 조건과 타입 시스템은 별개다 → **TypeScript는 제어 흐름을 읽어 타입을 좁힌다.**

### 면접 압축 답변

- **10초**: Narrowing은 넓은 타입을 `typeof`, `instanceof`, `in`, 타입 가드, assertion 함수 등으로 더 구체적인 타입으로 좁히는 과정입니다.
- **30초**: TypeScript를 실무에서 잘 쓰려면 선언보다 흐름 속 타입 좁히기가 더 중요합니다. 타입 가드(`is`)는 분기에서, assertion 함수(`asserts`)는 호출 이후 코드에서 타입을 보장합니다. `as` 캐스팅은 검사를 우회하므로 narrowing을 먼저 시도해야 합니다.

> 이제 narrowing이 왜 중요한지는 보였다. 다음엔 넓은 타입을 더 세밀하게 다루는 **유니온과 인터섹션**으로 넘어간다.

## 36. 유니온과 인터섹션은 어떻게 다르게 생각해야 하나

> **핵심 요약** — 유니온(`|`)은 "여러 가능성 중 하나", 인터섹션(`&`)은 "모든 속성을 동시에 만족". 태그드 유니온으로 상태를 모델링하면 boolean flag 기반보다 훨씬 안전하다.

### 핵심 비교

| | 유니온 `A \| B` | 인터섹션 `A & B` |
|---|---|---|
| **의미** | A 또는 B 중 하나 | A와 B를 동시에 만족 |
| **바로 쓸 수 있는 속성** | A와 B의 **공통** 속성만 | A와 B의 **모든** 속성 |
| **특정 멤버 접근** | narrowing 필요 | 바로 가능 |
| **실무 용도** | 상태 분기, 다형성 | 타입 합성, mixin |

### 태그드 유니온 vs boolean flag — 상태 모델링 비교

```ts
// Bad: boolean flag 기반 — 불가능한 상태 조합이 가능
interface RequestState {
  isLoading: boolean;
  isError: boolean;
  data: string[] | null;
  error: string | null;
}
// isLoading: true이면서 isError: true인 상태가 타입 수준에서 가능!
// data와 error가 동시에 있는 상태도 가능!

// Good: 태그드 유니온 — 불가능한 상태 조합이 타입 수준에서 차단
type RequestState =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: string[] }
  | { status: "error"; message: string };
// loading이면서 error인 상태가 존재할 수 없다
// success일 때만 data가 있고, error일 때만 message가 있다
```

```ts
function renderState(state: RequestState) {
  switch (state.status) {
    case "idle":    return "대기 중";
    case "loading": return "로딩 중...";
    case "success": return state.data.join(", ");     // data 안전 접근
    case "error":   return `에러: ${state.message}`;  // message 안전 접근
  }
}
```

### boolean flags vs 판별 유니온 — 불가능한 상태

```
❌ boolean flags로 모델링:
  isLoading  isError  isSuccess  | 가능한가?
  ─────────  ───────  ─────────  ├─────────
  true       false    false      | ✅ 로딩 중
  false      true     false      | ✅ 에러 발생
  false      false    true       | ✅ 성공
  true       true     false      | ❌ 로딩 중인데 에러?
  true       false    true       | ❌ 로딩 중인데 성공?
  true       true     true       | ❌ 전부 true?
  → 8가지 조합 중 3가지만 유효, 나머지 5가지는 불가능한 상태!

✅ 판별 유니온으로 모델링:
  { status: "loading" }                    | ✅
  { status: "error", error: Error }        | ✅
  { status: "success", data: T }           | ✅
  → 불가능한 상태 자체가 타입 레벨에서 존재하지 않음
```

### 실무 에피소드 — 이런 상황에서 만난다

> 결제 상태를 `isPending`, `isCompleted`, `isFailed` boolean 3개로 관리했는데, 한 번은 `isPending: true, isCompleted: true`인 불가능한 상태가 되어서 결제 완료인데 로딩 스피너가 도는 버그가 발생. 태그드 유니온으로 `{ status: "pending" } | { status: "completed", receiptId: string } | ...`로 바꾸니 불가능한 조합이 타입에서 차단.

### 자주 하는 오해

- 유니온은 타입 여러 개를 다 합치는 거다 → **"그중 하나"에 더 가깝다.**
- 인터섹션은 겹치는 속성만 남긴다 → **오히려 반대.** 모든 속성을 동시에 가져야 한다.
- boolean flag로 상태 관리하면 충분하다 → **불가능한 상태 조합을 타입이 허용하게 된다.**

### 면접 압축 답변

- **10초**: 유니온은 여러 가능성 중 하나, 인터섹션은 모든 속성을 동시에 만족하는 타입입니다. 태그드 유니온으로 상태를 모델링하면 불가능한 조합을 타입에서 차단할 수 있습니다.
- **30초**: 유니온 타입은 특정 멤버 속성에 접근하려면 narrowing이 필요합니다. 실무에서는 `status` 같은 판별자로 태그드 유니온을 구성하면, boolean flag 기반보다 불가능한 상태 조합을 타입 수준에서 방지할 수 있습니다. 인터섹션은 여러 타입의 속성을 모두 합쳐야 하므로 타입 합성에 유용합니다.

> 이제 타입을 조합하는 감각이 생겼다. 다음엔 반복되는 구조를 재사용 가능하게 만드는 **제네릭**으로 넘어간다.

## 37. 제네릭은 왜 재사용성과 안전성을 동시에 주나

> **핵심 요약** — 제네릭은 구조는 재사용하되 실제 타입은 사용하는 쪽에서 결정하게 하는 방식이다. `extends`로 제약을 걸면 유연성과 안전성을 동시에 확보할 수 있다.

### 핵심 개념

제네릭은 **구체 타입을 나중에 주입할 수 있게 남겨둔 타입 매개변수**다.

```ts
function identity<T>(value: T): T {
  return value;
}

const str = identity("hello"); // T = string → 반환도 string
const num = identity(42);      // T = number → 반환도 number
```

### 제네릭이 없으면 생기는 문제

```ts
// any를 쓰면: 타입 관계가 끊어짐
function identity(value: any): any {
  return value;
}
const result = identity("hello"); // result: any — string 정보 사라짐

// 타입마다 함수를 만들면: 중복
function identityString(value: string): string { return value; }
function identityNumber(value: number): number { return value; }
```

### 제네릭 제약 — extends

```ts
// 제약 없이: 내부에서 .length를 안전하게 쓸 수 없음
function getLength<T>(value: T) {
  return value.length; // 에러! — T에 length가 있다는 보장이 없음
}

// 제약 있음: length가 있는 타입만 받음
function getLength<T extends { length: number }>(value: T) {
  return value.length; // OK
}

getLength("hello");   // OK — string에 length 있음
getLength([1, 2, 3]); // OK — 배열에 length 있음
getLength(42);        // 에러! — number에 length 없음
```

### 다중 제약과 기본값

```ts
// 다중 제약: id와 name을 모두 가진 타입만 허용
function displayItem<T extends { id: string } & { name: string }>(item: T) {
  return `${item.id}: ${item.name}`;
}

// 제네릭 기본값: 타입을 안 넣으면 기본값 사용
type ApiResponse<T = unknown> = {
  data: T;
  success: boolean;
  timestamp: number;
};

// 타입 명시
const userRes: ApiResponse<User> = { data: user, success: true, timestamp: Date.now() };

// 기본값 사용
const unknownRes: ApiResponse = { data: something, success: true, timestamp: Date.now() };
// data: unknown
```

### 실무 패턴

```ts
// API 래퍼
async function fetchJson<T>(url: string): Promise<ApiResponse<T>> {
  const response = await fetch(url);
  return response.json();
}
const result = await fetchJson<User[]>("/api/users");
// result.data는 User[]

// id가 있는 목록에서 찾기
function findById<T extends { id: string }>(items: T[], id: string): T | undefined {
  return items.find((item) => item.id === id);
}
// User, Product, Order 등 id만 있으면 어떤 타입이든 사용 가능

// React 컴포넌트에서 제네릭
interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
  keyExtractor: (item: T) => string;
}

function List<T>({ items, renderItem, keyExtractor }: ListProps<T>) {
  return <ul>{items.map((item) => <li key={keyExtractor(item)}>{renderItem(item)}</li>)}</ul>;
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 공용 테이블 컴포넌트를 만들었는데, 처음에 `data: any[]`로 했더니 컬럼 정의에서 필드명을 오타 내도 에러가 안 났다. 제네릭으로 `<T>`를 받고 컬럼의 `key`를 `keyof T`로 제약하니, 존재하지 않는 필드를 지정하면 컴파일에서 에러가 떴다.

### 자주 하는 오해

- 제네릭은 라이브러리 만드는 사람만 쓴다 → **틀림.** 공용 컴포넌트나 훅을 만들 때 자주 필요.
- `<T>`는 아무 타입이나 받는 마법이다 → `extends`로 제약을 걸 수 있다.
- 제네릭은 복잡해서 피하는 게 낫다 → 피하면 `any`와 중복 코드가 늘어난다.

### 면접 압축 답변

- **10초**: 제네릭은 구조를 재사용하되 실제 타입은 사용처에서 결정하게 합니다. `extends`로 제약을 걸면 안전한 재사용이 가능합니다.
- **30초**: 제네릭은 `any` 없이 유연한 공용 함수와 컴포넌트를 만들게 해줍니다. `extends`로 최소 조건을 부여하면 함수 내부에서 안전하게 값을 다룰 수 있고, 기본값으로 사용 편의성도 확보할 수 있습니다. React에서는 공용 리스트, 테이블, API 래퍼, 커스텀 훅에서 특히 자주 쓰입니다.

> 이제 재사용 가능한 타입 구조를 만들 수 있게 됐다. 다음엔 기존 타입에서 정보를 끌어오는 **타입 연산 도구**로 넘어간다.

## 38. keyof, typeof, template literal은 왜 타입 설계에서 자주 쓰이나

> **핵심 요약** — `keyof`는 객체의 키를 유니온으로 추출하고, `typeof`는 값에서 타입을 추론하고, template literal type은 문자열 기반 타입을 프로그래밍할 수 있게 한다. 이 도구들로 중복 없이 타입을 파생할 수 있다.

### keyof — 객체 키를 유니온으로

```ts
interface User {
  id: string;
  name: string;
  age: number;
}

type UserKey = keyof User; // "id" | "name" | "age"

// 실무 패턴: 타입 안전한 property 접근
function getValue<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user: User = { id: "1", name: "park", age: 30 };
const name = getValue(user, "name");   // string
const age = getValue(user, "age");     // number
// getValue(user, "email");            // 에러! — "email"은 keyof User가 아님
```

### typeof — 값에서 타입 추출

```ts
const defaultConfig = {
  apiUrl: "https://api.example.com",
  timeout: 5000,
  retries: 3,
};

type Config = typeof defaultConfig;
// { apiUrl: string; timeout: number; retries: number }

// 런타임 typeof와 구분
if (typeof value === "string") { ... }  // 런타임 — "string" 문자열 반환
type T = typeof defaultConfig;           // 타입 문맥 — 타입 자체를 추출
```

### Template Literal Type — 문자열 기반 타입 프로그래밍

```ts
// 기본: 문자열 리터럴 조합
type EventName = `on${Capitalize<"click" | "focus" | "blur">}`;
// "onClick" | "onFocus" | "onBlur"

// 실무: CSS 단위 타입
type CSSUnit = "px" | "rem" | "em" | "vh" | "vw" | "%";
type CSSValue = `${number}${CSSUnit}`;
// "16px", "1.5rem", "100vh" 등

const padding: CSSValue = "16px";   // OK
const margin: CSSValue = "auto";    // 에러!

// API 경로 타입
type APIPath = `/api/${string}`;
const path: APIPath = "/api/users";  // OK
const bad: APIPath = "/users";       // 에러!
```

### ReturnType, Parameters — 함수에서 타입 추출

```ts
function createUser(name: string, age: number) {
  return { id: crypto.randomUUID(), name, age };
}

type User = ReturnType<typeof createUser>;
// { id: string; name: string; age: number }

type CreateUserParams = Parameters<typeof createUser>;
// [string, number]
```

### Mapped Type — `in`으로 키 순회

```ts
type Flags<T> = {
  [K in keyof T]: boolean;
};

type UserFlags = Flags<User>;
// { id: boolean; name: boolean; age: boolean }

// 실무: 모든 필드를 nullable로
type Nullable<T> = {
  [K in keyof T]: T[K] | null;
};
```

### 실무 에피소드 — 이런 상황에서 만난다

> API 응답의 필드명이 바뀔 때마다 타입과 유틸 함수를 동시에 수정해야 했는데, `keyof typeof`로 API 응답 객체에서 타입을 파생하니 원본만 바꾸면 관련 타입이 자동으로 따라감. 수정 누락 버그가 사라짐.

### 자주 하는 오해

- 타입은 항상 수동으로 일일이 적어야 한다 → **틀림.** 기존 값이나 타입에서 끌어올 수 있다.
- `typeof`는 런타임과 완전히 같은 개념이다 → **이름은 같지만 타입 문맥에서 역할이 다르다.**
- template literal type은 실무에서 안 쓴다 → **이벤트 이름, CSS 값, API 경로 등에서 강력하다.**

### 면접 압축 답변

- **10초**: `keyof`는 객체 키를 유니온으로 추출하고, `typeof`는 값에서 타입을 추론하며, template literal type은 문자열 패턴 기반 타입을 만들 수 있습니다.
- **30초**: 이 도구들로 기존 타입이나 값에서 새 타입을 파생할 수 있어서 중복 선언을 줄입니다. `ReturnType`, `Parameters`는 함수의 반환/인자 타입을 추출하고, mapped type은 `in`으로 키를 순회해서 변형합니다. template literal type은 이벤트 이름, CSS 값, API 경로 같은 문자열 기반 패턴에 유용합니다.

> 이제 타입 조작의 기본 재료를 봤다. 다음엔 실무에서 자주 쓰는 **유틸리티 타입**으로 넘어간다.

## 39. 유틸리티 타입은 왜 실무에서 자주 보이나

> **핵심 요약** — 유틸리티 타입은 반복되는 타입 변형을 표준화한 도구 상자다. 과하게 쓰면 읽기 어려워지므로 "의도가 선명한 타입"을 기준으로 사용.

### 자주 쓰는 유틸리티 타입

| 유틸리티 | 역할 | 예시 |
|---|---|---|
| `Partial<T>` | 모든 속성을 선택적으로 | 수정 payload |
| `Required<T>` | 모든 속성을 필수로 | 완전한 데이터 보장 |
| `Pick<T, K>` | 특정 속성만 추출 | 리스트용 요약 타입 |
| `Omit<T, K>` | 특정 속성만 제외 | 생성 시 id 제외 |
| `Record<K, T>` | 키-값 맵 구조 | 딕셔너리 |
| `Readonly<T>` | 모든 속성을 읽기 전용으로 | 설정 객체 |
| `Extract<T, U>` | T에서 U에 해당하는 것만 추출 | 유니온 필터 |
| `Exclude<T, U>` | T에서 U에 해당하는 것 제외 | 유니온 필터 |
| `NonNullable<T>` | null/undefined 제거 | nullable 값 정리 |
| `ReturnType<T>` | 함수 반환 타입 추출 | 함수에서 타입 파생 |
| `Parameters<T>` | 함수 인자 타입 추출 | 함수에서 타입 파생 |

### 실무 패턴

```ts
interface User {
  id: string;
  name: string;
  email: string;
  age: number;
  createdAt: Date;
}

// 생성 시 — id와 createdAt은 서버가 만듦
type CreateUserInput = Omit<User, "id" | "createdAt">;

// 수정 시 — 일부만 보내도 됨
type UpdateUserInput = Partial<Pick<User, "name" | "email" | "age">>;

// 리스트 표시용 — 최소 정보만
type UserPreview = Pick<User, "id" | "name">;

// 상태별 카운트 맵
type StatusCount = Record<"idle" | "loading" | "success" | "error", number>;
```

### 커스텀 유틸리티 타입 만들기

```ts
// 특정 키만 required로 만들기
type RequireKeys<T, K extends keyof T> = T & Required<Pick<T, K>>;

type UserWithEmail = RequireKeys<Partial<User>, "email">;
// email만 필수, 나머지는 선택적

// 깊은 Partial
type DeepPartial<T> = {
  [K in keyof T]?: T[K] extends object ? DeepPartial<T[K]> : T[K];
};
```

### 실무 에피소드 — 이런 상황에서 만난다

> 유저 생성 API와 수정 API에서 같은 User 타입을 쓰는데, 생성 시에는 id가 없고 수정 시에는 전부 optional이어야 했다. `Omit<User, "id">`로 생성 타입, `Partial<User>`로 수정 타입을 만들어서 하나의 모델에서 용도별 타입을 파생.

### 자주 하는 오해

- 유틸리티 타입을 많이 쓰면 무조건 고급스럽다 → **과하면 읽기 어려워진다.**
- `Partial`은 폼 수정에 항상 정답이다 → **어떤 필드가 진짜 선택적이고 어떤 필드가 필수인지 구분이 사라질 수 있다.**
- `Omit`은 마음대로 빼도 된다 → **도메인 의미를 흐릴 수 있다.**

### 면접 압축 답변

- **10초**: `Pick`, `Omit`, `Partial`, `Record` 등은 반복되는 타입 변형을 표준화합니다. `Extract`, `Exclude`는 유니온 필터링에 쓰입니다.
- **30초**: 유틸리티 타입은 생성/수정/표시 등 용도별 타입을 기존 모델에서 파생할 때 유용합니다. 커스텀 유틸리티 타입도 만들 수 있어서 프로젝트 전용 패턴을 표준화할 수 있습니다. 단, "짧은 타입"이 아니라 "읽었을 때 의도가 선명한 타입"이 기준이어야 합니다.

> 유틸리티 타입까지 보면 타입 조작 감각이 꽤 생긴다. 다음엔 값을 더 좁게 고정하는 **as const**와 **satisfies**로 이어진다.

## 40. as const와 satisfies는 왜 의도 표현 도구인가

> **핵심 요약** — `as const`는 값을 좁은 리터럴 타입으로 고정하고, `satisfies`는 타입 호환을 검증하면서도 추론된 좁은 타입을 유지한다. 둘 다 "이 값의 의도를 더 정확히 표현"하는 도구다.

### as const — 값을 좁게 고정

```ts
// as const 없이: 넓은 추론
const status = "success";        // 타입: string
const colors = ["red", "green"]; // 타입: string[]

// as const: 좁은 리터럴 타입으로 고정
const status2 = "success" as const;          // 타입: "success"
const colors2 = ["red", "green"] as const;   // 타입: readonly ["red", "green"]
```

### as const 실무 패턴

```ts
// 상태 상수 정의
const REQUEST_STATUS = {
  IDLE: "idle",
  LOADING: "loading",
  SUCCESS: "success",
  ERROR: "error",
} as const;

type RequestStatus = (typeof REQUEST_STATUS)[keyof typeof REQUEST_STATUS];
// "idle" | "loading" | "success" | "error"

// as const 없었다면:
// (typeof REQUEST_STATUS)[keyof typeof REQUEST_STATUS] → string
```

### satisfies — 타입 검증 + 좁은 추론 유지 (TS 4.9+)

`satisfies`는 비교적 새로운 연산자로, **타입 호환성을 검증하면서도 추론된 좁은 타입을 유지**한다.

```ts
// 문제: 타입 선언하면 추론이 넓어짐
const palette1: Record<string, string | number[]> = {
  red: "#ff0000",
  green: "#00ff00",
  blue: [0, 0, 255],
};
palette1.red.toUpperCase(); // 에러! — string | number[]이므로
```

```ts
// 해결: satisfies로 검증하면서 좁은 추론 유지
const palette2 = {
  red: "#ff0000",
  green: "#00ff00",
  blue: [0, 0, 255],
} satisfies Record<string, string | number[]>;

palette2.red.toUpperCase();   // OK — red는 string으로 추론
palette2.blue.map((v) => v);  // OK — blue는 number[]로 추론
```

### as const + satisfies — 최강 조합

```ts
// 라우트 설정: 값은 좁게 고정 + 구조는 검증
const routes = {
  home: "/",
  users: "/users",
  userDetail: "/users/:id",
} as const satisfies Record<string, string>;

// routes.home의 타입: "/" (리터럴)
// Record<string, string> 구조를 만족하는지도 검증됨
// 오타나 잘못된 타입이 들어가면 satisfies에서 에러
```

| | `as const` | `satisfies` | `as const satisfies` |
|---|---|---|---|
| **값 타입** | 좁은 리터럴 | 추론된 그대로 | 좁은 리터럴 |
| **구조 검증** | 없음 | 있음 | 있음 |
| **readonly** | 강제 | 아님 | 강제 |
| **용도** | 상수 정의 | 타입 호환 + 추론 유지 | 상수 정의 + 구조 검증 |

### 실무 에피소드 — 이런 상황에서 만난다

> 라우트 상수 객체를 정의했는데, 오타로 `/uers` 같은 잘못된 경로가 들어가도 `string` 타입이라 에러가 안 났다. `as const satisfies Record<string, string>`을 적용하니 값은 리터럴로 유지되면서 구조 검증도 되어, 자동완성에서 정확한 경로 리터럴이 나옴.

### 자주 하는 오해

- `as const`는 그냥 편하게 쓰는 캐스팅이다 → **`as`와 이름이 비슷하지만 성격이 다르다.** 값을 더 좁게 고정하는 의도 표현.
- `satisfies`는 `as`와 같다 → **`as`는 타입을 강제하고, `satisfies`는 호환성만 검증한다.** 추론된 타입은 유지.

### 면접 압축 답변

- **10초**: `as const`는 값을 좁은 리터럴 타입으로 고정하고, `satisfies`는 타입 호환을 검증하면서 추론된 좁은 타입을 유지합니다.
- **30초**: `as const`는 상수 정의에서 넓은 추론을 방지하고, `satisfies`(TS 4.9+)는 타입 안전성 검증과 좁은 추론을 동시에 제공합니다. 둘을 함께 쓰면(`as const satisfies`) 상수의 리터럴 타입을 유지하면서도 구조적 제약을 검증할 수 있습니다.

> 이제 TypeScript 안에서 내부 계약을 꽤 잘 만들 수 있게 됐다. 하지만 중요한 질문이 남는다. **외부에서 들어오는 값도 TypeScript만으로 안전할까?**

## 41. TypeScript가 있어도 왜 런타임 검증이 필요한가

> **핵심 요약** — TypeScript 타입 정보는 빌드 시 완전히 사라진다. 외부에서 들어오는 데이터(API 응답, 폼 입력, URL 파라미터 등)가 정말 기대한 타입인지는 런타임에서 별도로 확인해야 한다.

### 핵심: 컴파일 타임 vs 런타임

```
┌─────────────────────────┐     ┌─────────────────────────┐
│    컴파일 타임            │     │    런타임                │
│                         │     │                         │
│  TypeScript 타입 검사    │ ──▶ │  순수 JavaScript만 남음  │
│  타입 에러 → 빌드 실패   │     │  타입 정보 완전 제거     │
│                         │     │  외부 데이터 = 보장 없음 │
└─────────────────────────┘     └─────────────────────────┘
```

```ts
type User = { id: string; name: string };

async function fetchUser(): Promise<User> {
  const response = await fetch("/api/user");
  return response.json(); // 실제로 User인지 확인하지 않음!
}
// 서버가 { id: 1, nickname: "park" }을 보내도 TS는 막지 못한다
```

### 외부 경계 — 검증이 필요한 곳

| 경계 | 왜 위험한가 |
|---|---|
| API 응답 | 서버 배포 버전 차이, 필드 누락, 타입 변경 |
| 사용자 폼 입력 | 빈 문자열, 잘못된 형식, 필수값 누락 |
| URL search params | 사용자가 직접 수정 가능, 문자열 기반 |
| localStorage | 스키마 변경 후 이전 데이터 잔존 |
| postMessage / iframe | 외부 시스템에서 오는 데이터 |
| 파일 메타데이터 | 확장자, 크기, 타입 보장 없음 |

### 수동 검증 vs Zod

```ts
// 수동 검증 — 번거롭고 빠뜨리기 쉬움
async function fetchUser(): Promise<User> {
  const data: unknown = await (await fetch("/api/user")).json();
  if (
    typeof data === "object" && data !== null &&
    "id" in data && typeof (data as any).id === "string" &&
    "name" in data && typeof (data as any).name === "string"
  ) {
    return data as User;
  }
  throw new Error("Invalid user data");
}

// Zod — 스키마 선언 + 검증 + 타입 추론을 한 번에
import { z } from "zod";
const UserSchema = z.object({ id: z.string(), name: z.string() });
type User = z.infer<typeof UserSchema>;

async function fetchUser(): Promise<User> {
  const data = await (await fetch("/api/user")).json();
  return UserSchema.parse(data); // 검증 실패 시 ZodError
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 백엔드가 API 응답에서 `createdAt` 필드를 `created_at`으로 바꿨는데, 프론트 TS 타입은 `createdAt`으로 선언되어 있어서 컴파일 에러가 없었다. 배포 후에야 "날짜가 안 보여요" 버그로 발견. Zod로 API 응답을 경계에서 검증했다면 즉시 에러가 떴을 것.

### 자주 하는 오해

- 타입을 선언했으니 응답도 그 타입일 것이다 → **틀림.** 타입은 빌드 시 사라진다.
- `as User`로 캐스팅하면 안전하다 → **오히려 위험을 숨긴다.**
- 백엔드가 있으니 프론트는 검증 안 해도 된다 → **경계는 항상 조심해야 한다.**

### 면접 압축 답변

- **10초**: TypeScript 타입 정보는 빌드 시 제거되므로, 외부에서 들어오는 데이터가 정말 기대한 타입인지는 런타임에서 별도로 검증해야 합니다.
- **30초**: TypeScript는 컴파일 타임에 내부 계약을 검사하지만, API 응답, 폼 입력, URL 파라미터 같은 외부 데이터는 빌드 후 타입 정보가 없으므로 보장할 수 없습니다. `as` 캐스팅은 위험을 숨기고, 수동 검증은 번거롭기 때문에 Zod 같은 스키마 기반 런타임 검증 도구를 쓰는 것이 실무 표준입니다.

> 이제 런타임 검증이 왜 필요한지 이해했다. 다음은 Zod의 구체적인 사용법과 역할이다.

---

## 42. Zod는 왜 TypeScript 다음 단계로 자주 언급되나

> **핵심 요약** — Zod는 런타임 검증 + TypeScript 타입 추론을 한 번에 해결하는 스키마 라이브러리다. 스키마 하나로 검증 로직과 타입 선언의 괴리를 없앤다.

### 핵심 개념

```ts
import { z } from "zod";

// 스키마 정의 = 검증 규칙 + 타입 소스
const UserSchema = z.object({
  id: z.string().uuid(),
  name: z.string().min(1),
  email: z.string().email(),
  age: z.number().int().min(0).optional(),
});

// 스키마에서 타입 추론 — 별도 type 선언 불필요
type User = z.infer<typeof UserSchema>;
// { id: string; name: string; email: string; age?: number }
```

### parse vs safeParse

```ts
// .parse() — 실패 시 예외를 던짐
try {
  const user = UserSchema.parse(data);
} catch (e) {
  if (e instanceof z.ZodError) {
    console.error(e.issues); // 어떤 필드가 왜 실패했는지 상세 정보
  }
}

// .safeParse() — 예외 없이 결과 객체를 반환
const result = UserSchema.safeParse(data);
if (result.success) {
  const user = result.data; // 타입: User
} else {
  console.error(result.error.issues);
}
```

| | `.parse()` | `.safeParse()` |
|---|---|---|
| **실패 시** | ZodError throw | `{ success: false, error }` |
| **성공 시** | 검증된 값 반환 | `{ success: true, data }` |
| **용도** | 실패 = 예외인 경우 | 에러 핸들링이 필요한 경우 |

### 실무 적용 패턴

```ts

  const response = await fetch("/api/user");
  const data = await response.json();

}

// 2. 폼 입력 검증 (React Hook Form + Zod)

function LoginForm() {
  const { register, handleSubmit, formState: { errors } } = useForm({
    resolver: zodResolver(formSchema),
  });
  // ...

const SearchParamsSchema = z.object({

function parseSearchParams(params: URLSearchParams) {
  return SearchParamsSchema.parse({
    page: params.get("page"),

}
```

### **DTO(Data Transfer Object, 데이터 전송 객체 — API 응답 형태를 내부 모델로 변환할 때 쓰는 객체)** → 내부 모델 변환

```ts
// 서버 응답 스키마 (외부 경계 — snake_case, 문자열 날짜)
const UserDTOSchema = z.object({
  user_id: z.string(),
  user_name: z.string(),
  created_at: z.string(),
});

// 내부 UI 모델 (내부 계약 — camelCase, Date 객체)
interface User {
  id: string;
  name: string;
  createdAt: Date;
}

// 변환 함수
function toUser(dto: z.infer<typeof UserDTOSchema>): User {
  return {
    id: dto.user_id,
    name: dto.user_name,
    createdAt: new Date(dto.created_at),
  };
}

// 사용
async function fetchUser(): Promise<User> {
  const data = await (await fetch("/api/user")).json();
  return toUser(UserDTOSchema.parse(data));
}
```

### End-to-End 타입 안전성 트렌드

최근에는 API 경계의 타입 불일치를 아예 없애려는 도구들이 등장하고 있다.

| 도구 | 접근 방식 |
|---|---|
| **tRPC** | 서버/클라이언트가 같은 TS 프로젝트에서 타입을 공유 — Zod 스키마가 서버에서 정의되면 클라이언트에서 자동 추론 |
| **OpenAPI + codegen** | API 스펙에서 타입과 클라이언트 코드를 자동 생성 |
| **GraphQL codegen** | GraphQL 스키마에서 TS 타입을 자동 생성 |

이런 도구를 쓰더라도 "외부 경계에서 런타임 검증이 필요하다"는 원칙은 동일하다.

### 자주 하는 오해

- Zod가 있으면 TypeScript가 필요 없다 → **역할이 다르다.** 내부 계약은 TS, 외부 경계는 Zod.
- TypeScript가 있으면 Zod는 중복이다 → **검증 시점이 다르다.** 컴파일 vs 런타임.
- 모든 데이터에 무조건 Zod를 붙여야 한다 → **외부 경계만.** 내부 코드 사이 데이터까지 전부 검증할 필요는 없다.

### 면접 압축 답변

- **10초**: Zod는 런타임 데이터 검증과 TypeScript 타입 추론을 스키마 하나로 동시에 해결합니다.
- **30초**: Zod는 `z.infer`로 스키마에서 타입을 추론하므로 타입 선언과 검증 로직이 분리되지 않습니다. `.parse()`로 실패 시 예외를, `.safeParse()`로 결과 객체를 받을 수 있습니다. API 응답, 폼 입력, URL 파라미터, localStorage 같은 외부 경계에서 쓰고, tRPC나 OpenAPI codegen 같은 end-to-end 타입 안전성 도구와 함께 사용하는 것이 최신 트렌드입니다.

### tsconfig 실전 설정 — strict만으로는 부족하다

`"strict": true`는 시작일 뿐이다. 실무에서 타입 안전성을 높이는 추가 설정:

| 설정 | 효과 | 예시 |
|---|---|---|
| `noUncheckedIndexedAccess` | 배열/객체 인덱스 접근 시 `undefined` 가능성 강제 | `arr[0]`의 타입이 `T \| undefined`가 됨 |
| `exactOptionalPropertyTypes` | `age?: number`에 `undefined` 명시 대입 차단 | `{ age: undefined }` 불가, 키 자체를 생략해야 함 |
| `noPropertyAccessFromIndexSignature` | 인덱스 시그니처 프로퍼티를 `.`으로 접근 차단 | `obj.key` 대신 `obj["key"]` 강제 |

```json
// tsconfig.json 추천 설정
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noPropertyAccessFromIndexSignature": true,
    "forceConsistentCasingInFileNames": true,
    "verbatimModuleSyntax": true
  }
}
```

#### noUncheckedIndexedAccess가 중요한 이유

```tsx
const items: string[] = ["a", "b", "c"];

// noUncheckedIndexedAccess: false (기본값)
const first = items[0]; // string ← 빈 배열이면 런타임 에러!

// noUncheckedIndexedAccess: true
const first = items[0]; // string | undefined ← 안전하게 체크 강제
if (first !== undefined) {
  console.log(first.toUpperCase()); // 여기서는 string 확정
}
```

> **면접 팁**: "TypeScript를 얼마나 엄격하게 쓰느냐"는 tsconfig 설정에서 드러난다. `strict: true` 외에 `noUncheckedIndexedAccess`를 켠다고 말하면 실전 경험을 보여줄 수 있다.

# 챕터 4 핵심 요약 — 내부 계약을 세우고, 외부 경계를 지키는 두 겹의 안전장치

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **TypeScript는 어디까지 지켜주고, 어디서부터는 우리가 직접 지켜야 하는가?**

---

### 1단계: 왜 TypeScript를 쓰는가 — 계약이라는 관점

출발은 TypeScript의 **핵심 가치**다. 자동완성이 좋아진다는 게 아니라, **코드베이스 안의 계약을 명시하고 잘못된 연결을 컴파일 타임에 발견하는 것**이 본질이다. TypeScript는 **구조(shape)**로 타입을 비교하는 구조적 타이핑 기반이다.

### 2단계: 계약을 어떤 문법으로 표현하는가 — type vs interface

`interface`는 **객체 shape 계약과 선언 병합**에 자연스럽고, `type`은 **유니온, 인터섹션, 튜플 같은 조합형 표현**까지 가능하다. `extends`는 충돌 시 에러를 내고, `&`는 충돌 시 조용히 `never`가 되는 차이도 실무에서 중요하다.

### 3단계: 안전한 타입과 위험한 타입 — any, unknown, never

`any`는 **검사를 끄고 오염을 전파**하는 탈출구이고, `unknown`은 **"모르지만 확인 후 쓰겠다"는 정직한 선언**이며, `never`는 **도달하면 안 되는 곳을 표시하는 바닥 타입**이다. 한 곳의 `any`가 그것을 사용하는 모든 변수를 `any`로 물들인다.

### 4단계: 넓은 타입을 좁혀 가는 감각 — narrowing

`typeof`, `instanceof`, `in`, 사용자 정의 타입 가드(`is`), assertion 함수(`asserts`)로 넓은 타입을 좁힌다. **선언보다 흐름 속 타입 좁히기가 실무에서 더 중요할 때가 많다.** `as` 캐스팅은 검사를 우회하므로 narrowing을 먼저 시도해야 한다.

### 5단계: 여러 가능성과 합성 — 유니온과 인터섹션

유니온(`|`)은 **"여러 가능성 중 하나"**, 인터섹션(`&`)은 **"모든 속성을 동시에 만족"**이다. 가장 강력한 패턴은 **태그드 유니온**으로, `status` 같은 판별자 필드로 상태를 모델링하면 불가능한 상태 조합을 타입 수준에서 차단할 수 있다.

### 6단계: 구조는 재사용하되 타입은 나중에 — 제네릭

반복되는 구조를 `any` 없이 재사용하려면 **제네릭**이 필요하다. `<T>`로 타입 매개변수를 남겨두고, `extends`로 **제약을 걸면 유연성과 안전성을 동시에 확보**한다.

### 7단계: 기존 타입에서 새 타입을 파생하는 도구 — keyof, typeof, 유틸리티 타입

`keyof`는 **객체 키를 유니온으로 추출**하고, `typeof`는 **값에서 타입을 추론**하고, `Pick`, `Omit`, `Partial`, `Record` 같은 유틸리티 타입은 **용도별 타입을 기존 모델에서 파생**한다.

### 8단계: 값의 의도를 더 정확하게 표현하는 법 — as const, satisfies

`as const`는 **값을 좁은 리터럴 타입으로 고정**하고, `satisfies`는 **타입 호환을 검증하면서 추론된 좁은 타입을 유지**한다.

---

여기서부터 축이 바뀐다. TypeScript는 내부 계약을 잘 만들어주지만, 타입 정보는 **빌드 시 완전히 사라진다**.

---

### 9단계: TypeScript의 한계 — 런타임에는 타입이 없다

API 응답, 폼 입력, URL 파라미터, localStorage — **외부 경계에서 들어오는 데이터가 정말 기대한 타입인지는 런타임에서 별도로 확인**해야 한다.

### 10단계: 스키마 하나로 검증과 타입을 동시에 — Zod

Zod는 **스키마 정의 하나로 런타임 검증 + TypeScript 타입 추론을 동시에 해결**한다. API 응답, 폼 입력, URL 파라미터, localStorage 복원 등 외부 경계마다 적용한다.

---

### 한 줄로 이어 읽기

**구조적 타이핑**이 TypeScript의 비교 방식을 정하고 → `type`/`interface`로 **계약을 표현**하고 → `any`/`unknown`/`never`로 **안전성 수준을 제어**하고 → **narrowing**으로 넓은 타입을 좁히고 → **유니온**과 **태그드 유니온**으로 상태를 모델링하고 → **제네릭**으로 구조를 재사용하고 → `keyof`/`typeof`/유틸리티 타입으로 **타입을 파생**하고 → `as const`/`satisfies`로 **의도를 고정**하고 → 그럼에도 빌드 시 타입이 사라지므로 → **Zod**로 외부 경계를 런타임에서 지킨다.
