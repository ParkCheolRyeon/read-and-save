# 01. 이력서/포트폴리오 기반 면접 30문항

> 실제 면접에서 이력서와 포트폴리오를 보고 물어볼 수 있는 질문들입니다. 기술 지식이 아닌, **본인의 경험과 판단**을 묻는 질문이 중심이며, 실제 코드 기반의 구체적 디테일을 포함합니다.

### 전체 흐름 맵

```
자기소개 / 지원 동기
  → 프로젝트 경험 (양치킹, 아트봉봉 스쿨, Tool-Manager 등)
    → 기술 선택 이유 (React 마이그레이션, Zustand, Zod 등)
      → 문제 해결 경험 (네트워크, VOC, 성능)
        → 아키텍처/설계 (모노레포, API 추상화, 상태 관리)
          → 테스트/배포/인프라
            → 협업/성장/가치관
```

---

## 1. 간단한 자기소개와 지원 동기를 말씀해주세요.

> 면접관의 의도: 지원자가 자신의 커리어를 구조적으로 정리할 수 있는지, 그리고 지원 동기에서 회사와의 접점을 찾았는지를 30초~1분 안에 파악하려는 질문이다.

### 답변 타이밍: 1분

### 합격 답변 예시

안녕하세요, 2년차 프론트엔드 개발자 박철련입니다. 현재 아이스크림아트 기술연구소에서 Next.js, React, TypeScript 기반의 B2G 교육 서비스를 개발하고 있습니다. 개인 프로젝트로는 전국 10개 학교, 6,000명 이상이 실사용하는 양치킹이라는 서비스를 기획부터 운영까지 1인으로 담당하고 있고요. 또 개발 생산성에 관심이 많아서 Gemini API를 활용한 커밋 메시지 자동 생성 VS Code Extension을 만들어 Marketplace에 배포했는데, 100건 이상 다운로드되고 있습니다. 저는 "사용자가 겪는 순간"을 중심으로 기술을 선택하는 걸 중요하게 생각합니다. 화려한 스택보다 쓰임새가 분명한 선택을 하고, 문제를 끝까지 추적해서 해결하는 개발자로 성장하고 싶어 지원하게 되었습니다.

### 이렇게 말하면 감점

- "우송대학교 컴퓨터정보공학과를 중퇴하고..." 처럼 학력/자격증부터 시작하는 패턴. 경력 중심으로 말해야 한다.
- "열심히 하겠습니다", "성장하고 싶습니다" 같은 추상적 각오로 끝내는 패턴. 구체적 경험이 빠지면 기억에 남지 않는다.
- 2분 이상 장황하게 이어가는 패턴. 자기소개가 길어지면 면접관의 집중도가 급격히 떨어진다.

### 가산점 포인트

단순 경력 나열이 아니라 "나는 어떤 기준으로 개발하는 사람인가"를 한 문장으로 전달하면 인상적이다. 양치킹의 실사용 규모(6,000명+), VS Code Extension의 다운로드 수(100건+) 같은 숫자를 자연스럽게 녹이면 신뢰도가 올라간다. 지원 회사의 도메인이나 기술 스택(Next.js, React 등)과 본인 경험의 접점을 한 마디라도 넣으면 "이 회사를 조사하고 왔구나"라는 인상을 줄 수 있다.

### 상세 해설

#### 자기소개 구조 공식

자기소개는 30초~1분이 적정이다. 다음 3단계로 구성한다.

| 단계 | 시간 | 내용 |
|------|------|------|
| 1. 이름 + 포지션 + 현직 | 10초 | "2년차 프론트엔드, 아이스크림아트 기술연구소" |
| 2. 대표 경험 1~2개 | 30초 | 양치킹(규모감), VS Code Extension(기술적 호기심) |
| 3. 개발 기준 + 지원 동기 | 20초 | "사용자가 겪는 순간 중심" → 지원 회사와 연결 |

#### 지원 동기 연결 전략

지원 동기는 "내가 뭘 하고 싶다"가 아니라 "내가 해온 것이 이 회사에서 어떻게 이어지는가"로 말해야 한다. 예를 들어 교육 도메인 회사라면 양치킹의 교육 현장 경험을, DX 중심 팀이라면 VS Code Extension 경험을 연결한다.

### 꼬리 질문 대비

#### "SQLD, 컴활1급이 있는데 이 자격증들이 프론트엔드에 어떻게 도움이 되나요?"
SQLD는 Prisma ORM으로 쿼리를 작성할 때 인덱스와 조인 최적화 판단에 직접적으로 도움이 됩니다. 컴활1급은 엑셀/문서 자동화 관점에서 교육청 보고용 데이터 다운로드 기능을 설계할 때 사용자 관점을 이해하는 데 도움이 되었습니다.

#### "중퇴를 하셨는데, 그 이후 어떻게 개발을 공부하셨나요?"
독학과 실전 프로젝트 중심으로 학습했습니다. 양치킹을 직접 만들면서 프론트엔드부터 백엔드, 인프라까지 필요한 기술을 하나씩 익혔고, 그 과정에서 "만들면서 배우는 것"이 저에게 가장 효과적인 학습 방식이라는 걸 알게 되었습니다.

#### "본인의 기술 스택 중 가장 자신 있는 것은?"
Next.js와 React입니다. 회사 프로젝트와 양치킹 모두 Next.js 기반이라 App Router, Server Components, API Routes 등을 실무에서 반복적으로 사용하고 있고, 상태 관리는 Zustand와 React-Query 조합을 선호합니다.

## 2. 양치킹 프로젝트를 소개해주세요. 왜 만들게 되었고, 어떤 서비스인가요?

> 면접관의 의도: 사이드 프로젝트의 실제 규모와 기술적 깊이를 파악하고, 기획부터 운영까지의 경험 범위를 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

양치킹은 초등학교부터 고등학교까지 학생들의 양치 습관을 기록하고 관리하는 교육용 서비스입니다. 학생용 앱은 Expo 기반 React Native로 만들어서 학교 내 태블릿에서 바로 접근할 수 있게 했고, 교사용 어드민 대시보드는 Next.js로 구현했습니다. 데이터베이스는 Prisma ORM을 사용하고, 배포는 AWS EC2에 GitHub Actions와 PM2로 CI/CD 파이프라인을 구성해서 운영하고 있습니다. 현재 전국 10개 학교에서 6,000명 이상의 학생이 실사용 중이고, 하루 평균 약 1,000건의 양치 기록이 생성됩니다. Chart.js를 활용한 통계 시각화와 교육청 보고용 이미지 다운로드 기능도 제공하고 있고요. 학교 Wi-Fi 환경이 불안정한 상황에 대응하기 위해 네트워크 상태 감지, 재시도 로직, Fallback UI 등도 직접 설계했습니다. 기획부터 개발, 배포, 운영, VOC 대응까지 전 과정을 1인으로 수행하고 있습니다.

### 이렇게 말하면 감점

- 기술 스택만 나열하고 서비스의 목적과 사용자를 설명하지 않는 패턴. "누가 왜 쓰는 서비스인지"가 빠지면 프로젝트의 가치가 전달되지 않는다.
- "사이드 프로젝트입니다"로 가볍게 넘기는 패턴. 10개 학교 6,000명은 사이드 프로젝트 수준이 아니라 실서비스 규모다.
- 1인 개발이라는 점을 자랑만 하고 한계를 인지하지 못하는 패턴. 코드 리뷰 부재, 설계 검증 부재 같은 한계도 함께 인식하고 있어야 한다.

### 가산점 포인트

"왜 만들었는지"에 대한 동기와 "어떤 문제를 해결하는지"를 연결하면 좋다. 단순 포트폴리오가 아니라 실제 교육 현장의 니즈를 해결하는 프로덕트임을 강조한다. 학생용(Expo RN) + 교사용(Next.js)이라는 멀티 플랫폼 구성, CI/CD 자동화, AWS 인프라 구축까지 1인으로 처리한 범위를 자연스럽게 드러내면 기술 폭이 넓다는 인상을 줄 수 있다.

### 상세 해설

#### 아키텍처 전체 구성

```
[학생 앱 - Expo RN]  →  [API Server]  ←  [교사 어드민 - Next.js]
                            ↓
                    [Prisma ORM + DB]
                            ↓
              [AWS EC2 + PM2 + GitHub Actions]
                            ↓
              [Crontab 배치 집계 + Chart.js 통계]
```

학생 앱은 학교 태블릿에 최적화된 Expo React Native 웹앱이고, 교사 어드민은 Next.js로 서버 사이드 렌더링과 API Routes를 활용한다. 데이터 계층은 Prisma ORM으로 통일해서 타입 안전한 쿼리를 작성하고, 배포는 GitHub Actions가 push 이벤트를 감지하면 EC2에 PM2로 무중단 배포하는 파이프라인이다.

#### 규모감을 전달하는 방법

면접에서 숫자는 신뢰도를 높이는 가장 강력한 도구다. 다음 숫자들을 자연스럽게 녹여 말한다.

| 지표 | 수치 | 의미 |
|------|------|------|
| 학교 수 | 10개 | 다양한 환경 대응 경험 |
| 실사용자 | 6,000명+ | 사이드 프로젝트 이상의 규모 |
| 일일 기록 | 1,000건+ | 운영 안정성 검증 |
| 개발 인원 | 1명 | 기획~운영 전 범위 |

### 꼬리 질문 대비

#### "6,000명이면 동시 접속이 몇 명 정도인가요?"
학교별로 양치 시간대가 다르기 때문에 전체가 동시에 접속하지는 않습니다. 피크 시점은 점심시간 직후인데, 학교당 평균 100~200명 정도가 동시에 접속하는 패턴입니다. EC2 단일 인스턴스로 충분히 처리되는 규모이고, PM2 클러스터 모드로 CPU 코어를 활용하고 있습니다.

#### "수익 모델은 어떻게 되나요?"
처음에 외주 프로젝트로 시작했고, 지금은 교육적 가치와 포트폴리오 목적으로 운영하고 있습니다. 다만 실사용자가 있는 서비스를 혼자 운영하면서 배포 자동화, 네트워크 대응, VOC 처리 같은 실전 경험을 쌓을 수 있었기 때문에 투자 대비 가치가 충분하다고 생각합니다.

#### "Expo RN과 Next.js를 동시에 선택한 이유는?"
학생용 앱은 학교 태블릿에서 앱처럼 동작해야 해서 Expo React Native를 선택했고, 교사용은 PC 웹 브라우저 접근이 주이고 서버 사이드 렌더링과 API 라우팅이 필요해서 Next.js를 선택했습니다. 두 플랫폼 모두 React 기반이라 컴포넌트 설계 패턴과 상태 관리 로직을 공유할 수 있다는 점도 고려했습니다.

## 3. 양치킹에서 학교 Wi-Fi 불안정 문제를 어떻게 해결했나요?

> 면접관의 의도: 기술적 문제를 사용자 관점에서 정의하고, 단계적으로 해결해나가는 과정을 STAR 구조로 설명할 수 있는지 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

학교에 설치된 태블릿이 Wi-Fi 음영 지역에 놓여 있는 경우가 많았는데, 문제는 사용자인 학생이나 교사가 "지금 네트워크가 안 되는 건지, 앱 자체가 고장 난 건지" 구분을 못 한다는 거였습니다. 그래서 문의가 들어와도 원인 파악에 시간이 걸리고, 운영 부담이 커지는 상황이었습니다. 이걸 해결하기 위해 네트워크 상태를 체크해서 비정상이면 "네트워크 연결을 확인해주세요"라는 Fallback UI를 보여주고 메인 화면으로 리디렉션하는 흐름을 만들었습니다. 이걸 통해 사용자가 원인을 스스로 인지할 수 있게 되었고, 문의 내용이 체계화되면서 운영 부담이 크게 줄었습니다. 이후 회사인 아이스크림아트에서 이 경험을 발전시켜 4계층 네트워크 감지 시스템을 구축했습니다. usePingCheck로 60초 간격 서버 핑, useNavigatorOnline으로 브라우저 온라인 상태, useNetworkEffectiveType으로 네트워크 품질, useContinuousNavigatorOffline으로 1.5초 디바운스 기반 연속 오프라인 감지까지 계층별로 분리해서 더 정교하게 대응할 수 있게 만들었습니다.

### 이렇게 말하면 감점

- "Wi-Fi가 불안정해서 Fallback UI를 만들었습니다"로 끝나는 패턴. 왜 필요했는지(사용자가 원인 구분 불가 → 문의 혼선)라는 문제 정의가 빠지면 깊이가 없어 보인다.
- 기술 구현만 나열하고 결과(운영 부담 감소)를 말하지 않는 패턴. STAR에서 Result가 빠지면 "그래서 어떻게 됐는데?"라는 후속 질문을 유발한다.
- 양치킹 경험에서 끝나고 회사에서의 발전을 연결하지 않는 패턴. 성장 스토리가 빠지면 "그때만 했네"라는 인상을 준다.

### 가산점 포인트

"사용자가 원인을 구분할 수 없어서 문의가 비정형적이었다"라는 문제 정의가 핵심이다. 기술적 해결보다 "운영 관점에서 왜 이게 필요했는지"를 먼저 말하면 좋다. 양치킹에서의 1차 해결 → 회사에서의 4계층 고도화라는 성장 스토리를 연결하면 "경험에서 배워 발전시키는 개발자"라는 인상을 줄 수 있다.

### 상세 해설

#### STAR 구조로 정리

| 단계 | 내용 |
|------|------|
| **Situation** | 학교 Wi-Fi 불안정, 태블릿이 음영 지역에 위치 |
| **Task** | 사용자가 네트워크 문제를 스스로 인지할 수 있게 해야 함 |
| **Action** | 네트워크 상태 체크 → Fallback UI → 메인 리디렉션 |
| **Result** | 문의 체계화, 운영 부담 감소 |

#### 4계층 네트워크 감지 시스템 (회사 고도화 버전)

```typescript
// 1계층: 서버 핑 체크 (60초 간격)
const { isServerReachable } = usePingCheck({ interval: 60_000 });

// 2계층: 브라우저 온라인 상태
const { isOnline } = useNavigatorOnline();

// 3계층: 네트워크 품질 (effectiveType: 4g/3g/2g/slow-2g)
const { effectiveType } = useNetworkEffectiveType();

// 4계층: 연속 오프라인 감지 (1.5초 디바운스)
const { isContinuouslyOffline } = useContinuousNavigatorOffline({
  debounceMs: 1500,
});
```

각 계층이 서로 다른 레벨의 네트워크 문제를 감지한다. 1계층은 서버 레벨, 2계층은 OS 레벨, 3계층은 네트워크 품질 레벨, 4계층은 순간 끊김 vs 지속 끊김을 구분한다. 이렇게 분리하면 "인터넷은 되는데 서버가 죽었다"와 "Wi-Fi 자체가 끊겼다"를 구분해서 사용자에게 정확한 메시지를 보여줄 수 있다.

#### 왜 디바운스가 필요한가

`navigator.onLine` 이벤트는 Wi-Fi가 잠깐 흔들리는 것만으로도 false → true를 반복한다. 디바운스 없이 바로 Fallback UI를 띄우면 화면이 깜빡이면서 오히려 사용자 혼란을 가중시킨다. 1.5초 디바운스를 적용하면 순간적인 끊김은 무시하고, 실제로 지속적인 오프라인 상태일 때만 UI를 전환한다.

### 꼬리 질문 대비

#### "navigator.onLine만으로는 부족한 이유가 뭔가요?"
`navigator.onLine`은 OS 레벨의 네트워크 연결만 확인합니다. Wi-Fi에 연결되어 있지만 인터넷이 안 되는 경우(captive portal 등)에는 true를 반환하기 때문에 실제 서버 도달 가능 여부는 ping 체크로 별도 확인해야 합니다.

#### "Fallback UI를 보여주는 것 외에 데이터 유실 방지는 어떻게 했나요?"
네트워크가 끊긴 상태에서 사용자가 양치 기록을 시도하면 로컬에 임시 저장한 뒤, 네트워크가 복구되었을 때 재시도하는 방식으로 유실을 방지했습니다. 재시도 로직은 지수 백오프를 적용해서 서버에 과부하를 주지 않도록 했습니다.

#### "이 네트워크 감지 시스템을 다른 프로젝트에도 적용할 수 있나요?"
네, 커스텀 훅으로 분리되어 있어서 어떤 React 프로젝트에서든 재사용할 수 있습니다. 실제로 양치킹에서 시작한 패턴을 회사 프로젝트에 적용한 것 자체가 재사용 사례입니다. 각 훅이 독립적이라 필요한 계층만 선택해서 사용할 수도 있습니다.

## 4. 양치킹의 통계 기능을 "반/학년 단위"에서 "전체 트렌드형 대시보드"로 고도화한 배경과 과정을 설명해주세요.

> 면접관의 의도: 사용자 요구(VOC)를 기술적 구현으로 전환하는 과정과, 요구사항 변경에 대응하는 방식을 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

처음에는 반별, 학년별 양치 통계만 제공했습니다. 그런데 운영하면서 교사분들로부터 "전체 흐름을 한눈에 보고 싶다", "교육청에 보고할 때 차트 이미지가 필요하다"는 요구가 반복적으로 들어왔습니다. 단순히 기능을 추가하는 게 아니라, 실제 사용 시나리오를 먼저 파악했습니다. 교사가 월별 양치율 추이를 확인하고, 차트를 이미지로 다운로드해서 교육청 보고서에 붙여넣는 흐름이었습니다. Chart.js를 선택한 이유는 Canvas 기반이라 이미지 export가 간단하고, 번들 사이즈 대비 차트 종류가 충분했기 때문입니다. 데이터가 누적되면서 쿼리 성능이 느려지는 문제도 발생했는데, Prisma 쿼리를 최적화하고 AWS Crontab 기반 배치 집계를 도입해서 대시보드 로딩 시간을 개선했습니다. 이 경험을 통해 "구현 난이도보다 사용 흐름과 운영 비용을 먼저 보고 우선순위를 정하는 기준"을 체득했습니다.

### 이렇게 말하면 감점

- "Chart.js로 차트를 만들었습니다"로 끝나는 패턴. 왜 Chart.js인지, 다른 선택지와 비교한 근거가 없으면 기술 선택 역량이 드러나지 않는다.
- VOC가 있었다는 사실만 언급하고 구체적인 사용 시나리오를 설명하지 않는 패턴. "교육청 보고용 이미지 다운로드"라는 구체적 니즈를 말해야 현실감이 생긴다.
- 성능 문제가 발생했다는 점을 숨기는 패턴. 문제를 인지하고 배치 집계로 해결했다는 과정이 오히려 가산점이다.

### 가산점 포인트

사용자 요구(VOC)를 받아서 기능을 확장한 과정을 보여주는 것이 핵심이다. "교육청 보고용 이미지 다운로드"라는 비기술적 요구를 기술적으로 해결한 경험은 "사용자 중심 사고"를 증명한다. 데이터 누적에 따른 성능 이슈를 Prisma 최적화 + Crontab 배치로 해결한 점까지 연결하면 운영 경험의 깊이가 드러난다.

### 상세 해설

#### 고도화 과정 타임라인

```
[초기] 반/학년별 단순 통계
  ↓ VOC: "전체 트렌드를 보고 싶다"
[1차] Chart.js 기반 트렌드 차트 추가
  ↓ VOC: "교육청 보고서에 넣을 이미지가 필요하다"
[2차] Canvas → toDataURL 이미지 다운로드
  ↓ 데이터 누적 → 쿼리 성능 저하
[3차] Prisma 최적화 + Crontab 배치 집계
```

#### Chart.js 선택 근거

| 라이브러리 | 장점 | 단점 | 판단 |
|-----------|------|------|------|
| D3.js | 자유도 높음 | 학습 곡선 높음, 오버스펙 | 대시보드 수준에 불필요 |
| Recharts | React 친화 | React 종속, RN 웹앱 호환 이슈 | 학생앱과 통일 어려움 |
| Chart.js | Canvas 기반, 이미지 export 쉬움 | 커스터마이징 한계 | 선택 |

Chart.js의 Canvas 기반 렌더링은 `toDataURL()`로 바로 PNG 이미지를 추출할 수 있어서 교육청 보고용 이미지 다운로드 요구사항과 정확히 맞았다.

#### 배치 집계 전략

```
[매일 새벽 3시 - AWS Crontab]
  → 전일 양치 기록 집계
  → 학교별/학년별/반별 통계 테이블에 저장
  → 대시보드는 집계 테이블에서 조회 (원본 테이블 직접 조회 X)
```

이 방식으로 대시보드 로딩 시 수만 건의 원본 데이터를 매번 집계하지 않고, 사전 계산된 통계만 조회하게 되어 응답 속도가 크게 개선되었다.

### 꼬리 질문 대비

#### "Chart.js 말고 다른 라이브러리는 고려 안 했나요?"
D3는 대시보드 수준의 차트에 오버스펙이었고, Recharts는 React 종속이라 Expo RN 웹앱과의 호환성을 고려하면 적합하지 않았습니다. Chart.js는 Canvas 기반이라 이미지 export가 간단하고, 회사 프로젝트에서도 Chart.js + react-chartjs-2를 사용 중이라 경험 재활용도 가능했습니다.

#### "이미지 다운로드는 구체적으로 어떻게 구현했나요?"
Chart.js가 렌더링하는 Canvas 엘리먼트에서 `toDataURL('image/png')`을 호출해서 base64 이미지를 추출하고, 이를 `<a>` 태그의 download 속성으로 트리거하는 방식입니다. 복합 차트가 필요한 경우에는 html2canvas로 특정 DOM 영역 전체를 캡처하는 방식도 병행했습니다.

#### "Crontab 배치가 실패하면 어떻게 대응하나요?"
배치 작업 실행 결과를 로그 파일에 기록하고, 실패 시 PM2 로그에서 확인할 수 있도록 해두었습니다. 배치가 실패하더라도 대시보드는 이전 집계 데이터를 보여주기 때문에 서비스 자체가 중단되지는 않습니다. 다만 이 부분은 알림 자동화가 부족한 점이 개선 과제입니다.

## 5. 1인 개발에서 가장 어려웠던 점은 무엇이고, 어떻게 극복했나요?

> 면접관의 의도: 혼자서도 체계적으로 일할 수 있는지, 동시에 1인 개발의 한계를 인지하고 협업 환경에서 성장할 준비가 되어 있는지를 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

가장 어려웠던 점은 기능 개발과 운영을 동시에 해야 한다는 거였습니다. 새 기능을 만들면서도 기존 서비스가 중단되면 안 되고, 교사분들의 문의에도 대응해야 했거든요. 이걸 극복하기 위해 우선 배포 과정을 자동화했습니다. GitHub Actions로 main 브랜치에 push하면 EC2에 PM2가 무중단 배포하는 파이프라인을 구성해서, 수동 배포에 드는 시간과 실수를 제거했습니다. 운영 측면에서는 VOC를 우선순위화하는 습관을 들였습니다. "긴급한 것"과 "중요한 것"을 구분해서, 서비스 장애는 즉시 대응하고 기능 요구는 주간 단위로 모아서 처리하는 방식으로 운영했습니다. 다만 솔직히 한계도 있었습니다. 코드 리뷰를 받을 사람이 없다 보니 설계 판단에 확신이 부족할 때가 있었고, 혼자만의 패턴에 갇힐 수 있다는 점을 느꼈습니다. 그래서 협업 환경에서 코드 리뷰와 설계 토론을 통해 더 성장하고 싶다는 생각이 강해졌습니다.

### 이렇게 말하면 감점

- "어려웠지만 다 혼자 해냈습니다"로 끝나는 패턴. 한계를 인지하지 못하면 "협업이 안 되는 사람"이라는 인상을 줄 수 있다.
- 구체적인 시스템(CI/CD, 우선순위 체계) 없이 "열심히 했습니다"로 마무리하는 패턴. 체계 없이 노력만으로 해결했다는 인상은 지속 가능성이 없어 보인다.
- 1인 개발의 어려움만 강조하고 극복 과정을 말하지 않는 패턴. 어려웠다는 말에서 끝나면 "그래서 뭘 배웠는데?"라는 후속 질문이 나온다.

### 가산점 포인트

"어려웠다 → 시스템을 만들어 극복했다 → 그래도 한계가 있었다 → 협업 환경에서 성장하고 싶다"라는 4단계 구조로 답변하면 완성도가 높다. CI/CD 자동화, VOC 우선순위화 같은 구체적 시스템을 언급하면 "체계적으로 일하는 사람"이라는 인상을 주고, 코드 리뷰 부재라는 한계를 솔직히 인정하면 자기 객관화 능력을 보여줄 수 있다.

### 상세 해설

#### 1인 개발에서 만든 자동화 체계

```
[코드 작성] → [git push to main]
  → [GitHub Actions: lint + build 체크]
    → [SSH로 EC2 접속]
      → [git pull + npm install + PM2 reload]
        → [무중단 배포 완료]
```

이 파이프라인 덕분에 배포에 드는 시간이 수동 기준 15분에서 자동 3분으로 줄었고, 배포 실수(빌드 안 하고 올리기, 환경변수 누락 등)가 사라졌다.

#### VOC 우선순위 분류 기준

| 우선순위 | 기준 | 대응 시점 | 예시 |
|---------|------|----------|------|
| P0 - 긴급 | 서비스 장애, 데이터 유실 | 즉시 | 양치 기록 저장 실패 |
| P1 - 높음 | 핵심 기능 오동작 | 당일 | 통계 수치 오류 |
| P2 - 보통 | UX 개선 요구 | 주간 | "전체 트렌드를 보고 싶다" |
| P3 - 낮음 | 편의 기능 요청 | 백로그 | "다크모드 지원해주세요" |

#### 1인 개발의 강점과 한계

| 구분 | 강점 | 한계 |
|------|------|------|
| 의사결정 | 빠른 결정, 실행까지 지연 없음 | 검증 없는 판단, 편향 가능 |
| 설계 | 전체 아키텍처 이해 | 코드 리뷰 부재로 맹점 발생 |
| 운영 | 문제 발생 시 즉시 파악 | 번아웃 위험, 백업 인력 없음 |
| 성장 | 풀스택 경험 축적 | 특정 영역 깊이 부족 가능 |

### 꼬리 질문 대비

#### "코드 리뷰 없이 코드 품질은 어떻게 유지했나요?"
ESLint와 TypeScript strict 모드를 적용해서 기본적인 코드 품질은 도구로 보장했고, 커밋 컨벤션을 지키려고 VS Code Extension까지 직접 만들었습니다. 다만 설계 수준의 리뷰는 도구로 대체하기 어려워서, 이 부분이 협업 환경을 원하는 가장 큰 이유입니다.

#### "협업 경험이 부족한데 팀에 들어오면 어떻게 적응할 것 같나요?"
회사에서 아트봉봉 스쿨 프로젝트를 팀으로 진행한 경험이 있습니다. Git 브랜치 전략, PR 기반 코드 리뷰, Jira 태스크 관리 등을 경험했고, 오히려 1인 개발에서 전체 흐름을 파악한 경험이 팀 내에서 맥락을 빠르게 이해하는 데 도움이 될 것 같습니다.

#### "VOC 우선순위를 잘못 판단한 적은 없나요?"
있습니다. 초기에 "다크모드 지원" 같은 편의 기능을 먼저 만들었는데, 정작 교사분들이 원한 건 통계 이미지 다운로드였습니다. 그 이후로는 VOC를 모아서 빈도와 영향도를 기준으로 분류하는 습관을 들였고, 이 경험이 이후 체계적인 우선순위 판단에 큰 도움이 되었습니다.

## 6. VS Code Extension(커밋 메시지 자동 생성)을 만든 이유와 기술적 구현을 설명해주세요.

> 면접관의 의도: DX(Developer Experience)에 대한 관심과, 작은 제품을 설계부터 배포까지 완성해 공개한 경험의 기술적 깊이를 확인하려는 질문이다.

### 답변 타이밍: 3분

### 합격 답변 예시

커밋 컨벤션을 지키는 건 협업과 히스토리 품질에 중요하지만, 매번 규칙에 맞게 작성하는 비용이 누적되면 일관성이 깨지기 쉽습니다. 이 불편을 해결하고자 Gemini 2.0 Flash 모델을 활용해 커밋 메시지를 자동 생성하는 VS Code Extension을 만들었습니다.

핵심 구현을 말씀드리면, 먼저 프롬프트 구성입니다. `buildPrompt()` 함수에서 staged diff를 최대 40KB로 캡핑하고, 최근 20개 커밋 메시지를 스타일 힌트로 포함시킵니다. 사용자가 `COMMIT_CONVENTION.md` 파일을 작성해두면 그 규칙을 우선 적용하고, 없으면 기본 규칙으로 폴백합니다.

API 호출에서는 `callGeminiGenerateContentWithRetry()`로 30초 타임아웃과 최대 2회 재시도를 구현했는데, 백오프 간격은 800ms에 2의 attempt 제곱을 곱하는 지수 백오프입니다. HTTP 429가 오면 Retry-After 헤더를 파싱해서 백오프와 비교해 더 긴 쪽을 적용합니다.

생성된 메시지는 `validate.ts`에서 정규식으로 컨벤션 형식을 검증하고, 실패하면 Gemini에 수리 프롬프트를 보내 자동 교정합니다. 최종 결과는 `repo.inputBox.value`에 직접 주입해서 복사/붙여넣기 단계를 제거했습니다. API Key는 VS Code SecretStorage에 안전하게 저장하고, 런타임 의존성은 0개로 VS Code API와 Node.js 내장 모듈만 사용합니다. 멀티 레포 환경에서는 QuickPick UI로 대상 레포를 선택할 수 있게 했고, 한/영 자동 번역도 지원합니다. Marketplace에 배포해서 100건 이상 다운로드되고 있습니다.

### 이렇게 말하면 감점

- "Gemini API를 호출해서 커밋 메시지를 생성합니다"로 끝나는 패턴. API 호출만으로는 누구나 할 수 있다. 재시도 전략, 검증 로직, 보안 처리 같은 엔지니어링 디테일이 차별점이다.
- 에러 처리와 엣지 케이스를 언급하지 않는 패턴. 네트워크 실패, rate limit, 잘못된 응답 등에 어떻게 대응하는지가 실력을 보여주는 부분이다.
- "temperature 0.2를 사용했습니다" 같은 설정값만 나열하는 패턴. 왜 0.2인지(일관성 확보), 왜 maxOutputTokens 400인지(커밋 메시지는 짧아야 하니까) 이유를 함께 말해야 한다.

### 가산점 포인트

"개인의 불편을 해결해 공용 가치로 확장했다"는 관점이 좋다. 구현 디테일에서 API Key 보안(SecretStorage), 요청 제어(AbortController + 지수 백오프), 응답 검증(정규식 + 자동 수리), 런타임 의존성 0개 같은 키워드를 자연스럽게 녹이면 기술적 깊이를 강하게 어필할 수 있다. "temperature 0.2로 일관성을 확보하고, maxOutputTokens 400으로 응답 길이를 제한한 이유"까지 설명하면 LLM 활용 경험도 보여줄 수 있다.

### 상세 해설

#### 전체 아키텍처

```
[사용자가 커밋 메시지 생성 명령 실행]
  ↓
[멀티 레포?] → Yes: QuickPick UI로 선택 / No: 바로 진행
  ↓
[buildPrompt()]
  - staged diff (최대 40KB)
  - 최근 20개 커밋 메시지
  - COMMIT_CONVENTION.md 또는 기본 규칙
  ↓
[callGeminiGenerateContentWithRetry()]
  - Gemini 2.0 Flash (temperature: 0.2, maxOutputTokens: 400)
  - AbortController 30초 타임아웃
  - 800ms × 2^attempt 지수 백오프, 최대 2회
  - HTTP 429: Retry-After 파싱
  ↓
[validate.ts - 정규식 검증]
  - 통과 → repo.inputBox.value에 주입
  - 실패 → Gemini 수리 프롬프트 → 재검증
  ↓
[SCM 입력창에 결과 표시]
```

#### 재시도 전략 상세

```typescript
// callGeminiGenerateContentWithRetry() 핵심 로직
async function callGeminiGenerateContentWithRetry(
  prompt: string,
  options: { maxRetries: 2, timeoutMs: 30_000 }
) {
  for (let attempt = 0; attempt <= options.maxRetries; attempt++) {
    try {
      const controller = new AbortController();
      const timer = setTimeout(() => controller.abort(), options.timeoutMs);

      const response = await fetch(GEMINI_URL, {
        signal: controller.signal,
        // ...
      });
      clearTimeout(timer);

      if (response.status === 429) {
        const retryAfter = parseRetryAfter(response.headers);
        const backoff = 800 * Math.pow(2, attempt);
        await sleep(Math.max(retryAfter, backoff));
        continue;
      }

      return await response.json();
    } catch (error) {
      if (isTransientError(error) && attempt < options.maxRetries) {
        await sleep(800 * Math.pow(2, attempt));
        continue;
      }
      throw error;
    }
  }
}
```

HTTP 429(rate limit)일 때는 서버가 보내주는 Retry-After 값과 자체 백오프 값을 비교해서 더 긴 쪽을 적용한다. 이는 서버의 의도를 존중하면서도 최소한의 대기 시간을 보장하는 전략이다.

#### 검증 + 자동 수리 흐름

```typescript
// validate.ts
function validateCommitMessage(message: string): boolean {
  // Conventional Commits 정규식
  const pattern = /^(feat|fix|docs|style|refactor|test|chore)(\(.+\))?: .{1,72}/;
  return pattern.test(message);
}

// 검증 실패 시 → 수리 프롬프트
// "다음 메시지가 컨벤션에 맞지 않습니다. 규칙에 맞게 수정해주세요: ..."
```

LLM이 생성한 결과를 무조건 신뢰하지 않고 정규식으로 형식을 검증한 뒤, 실패하면 수리 프롬프트를 통해 자동 교정을 시도한다. 이 "생성 → 검증 → 수리" 파이프라인은 LLM 출력의 품질을 안정적으로 유지하는 핵심 패턴이다.

#### 보안 설계

- **API Key**: VS Code의 `SecretStorage` API를 사용해 OS 키체인에 암호화 저장. 설정 파일이나 환경변수에 평문으로 노출되지 않는다.
- **런타임 의존성 0개**: VS Code API + Node.js 내장 `https` 모듈만 사용. 외부 npm 패키지를 사용하지 않아서 supply chain 공격 표면을 최소화했다.
- **diff 캡핑**: staged diff를 40KB로 제한해서 대용량 diff가 API로 전송되는 것을 방지한다.

### 꼬리 질문 대비

#### "Gemini API를 선택한 이유가 뭔가요?"
무료 티어로 개인 개발자가 충분히 사용할 수 있는 품질이었고, REST API 형태라 별도 SDK 없이 Node.js 내장 모듈만으로 연동할 수 있었습니다. temperature 0.2로 설정하면 커밋 메시지처럼 일관성이 중요한 텍스트에서 안정적인 품질을 보여줬습니다. maxOutputTokens를 400으로 제한한 것도 커밋 메시지가 불필요하게 길어지는 것을 방지하기 위해서입니다.

#### "SCM 입력창 주입은 어떻게 구현했나요?"
VS Code Git Extension이 제공하는 `Repository` 객체의 `inputBox.value` 속성에 직접 문자열을 할당하는 방식입니다. `vscode.extensions.getExtension('vscode.git')`으로 Git Extension에 접근한 뒤, 활성 레포의 inputBox에 값을 넣으면 사용자가 SCM 패널에서 바로 확인하고 커밋할 수 있습니다. 복사/붙여넣기 단계가 완전히 제거됩니다.

#### "한/영 번역은 어떤 방식인가요?"
`vscode.env.language` 값을 기반으로 auto 모드에서는 사용자 VS Code 언어 설정에 맞춰 자동으로 한국어 또는 영어 커밋 메시지를 생성합니다. 수동으로 ko/en을 지정할 수도 있고, 내부적으로는 프롬프트에 언어 지시를 포함시키는 방식이라 별도 번역 API 없이 Gemini가 직접 해당 언어로 생성합니다.

## 7. 아트봉봉 스쿨에서 VOC 기반 UX 개선으로 월 단위 고객 불편을 70% 이상 감소시켰다고 했는데, 구체적으로 어떻게 했나요?

> 면접관의 의도: 사용자 피드백을 체계적으로 분류하고, 데이터 기반으로 우선순위를 정해 실제 지표 개선까지 연결하는 능력을 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

Teacher 채널로 인입되는 VOC를 유형화(탐색/동선/표현/상태/세션)해서 반복 패턴을 분석했습니다. 가장 빈번한 유형부터 우선 개선하는 방식으로 진행했고, 리스트 탐색 UX(정렬/검색/필터) 정비로 선택 오류를 줄이고, 이용권 구매 및 선물하기 흐름을 개선해 운영 혼선을 최소화했습니다. 구체적으로는 URL 기반 필터 상태 관리(`useZodSearchParams`)를 도입해 새로고침·탭 이동 후에도 검색/필터 상태가 유지되도록 했고, 데이터 그리드에 스켈레톤 로딩을 적용해 체감 성능을 개선했습니다. 결과적으로 9월 약 30건이던 VOC가 10월 17건, 11월 7건, 12월 6건으로 지속 감소했습니다.

### 이렇게 말하면 감점

- "VOC가 많아서 고쳤습니다"처럼 유형화 과정 없이 나열만 하는 패턴
- 수치 없이 "많이 줄었습니다"로 끝내는 패턴. 30→17→7→6이라는 구체적 숫자가 신뢰를 만든다
- 기술 구현만 말하고 "왜 이 개선이 필요했는지" 사용자 맥락을 빠뜨리는 패턴

### 가산점 포인트

숫자(30→17→7→6)를 구체적으로 말하면 신뢰도가 매우 높아진다. "유형화"라는 키워드가 체계적 접근을 보여준다. 실제 코드 수준의 구현(`useZodSearchParams`, 스켈레톤)을 언급하면 기술적 깊이도 함께 보여줄 수 있다.

### 상세 해설

이 질문은 "문제를 어떻게 구조화하고 측정하는가"를 본다.

**실제 코드 기반 개선 사례 (`InquiryList.tsx`):**

```typescript
// URL 기반 필터 상태 관리 - 새로고침해도 필터 유지
const [filter, pushFilter] = useZodSearchParams(getInquirySearchParamSchema, {});

// 듀얼 검색 모드 (제목/내용)

// 스켈레톤 로딩으로 체감 성능 개선
cell: (props) => isLoading ? <Skeleton /> : <CategoryChip {...props} />
```

**커스텀 테이블 훅 (`useTable`):**
- `ColumnDef<RowData>` 타입으로 타입 안전한 테이블 정의
- 정렬/필터/페이지네이션이 URL과 동기화
- 스켈레톤 로딩 상태 통합

### 꼬리 질문 대비

#### "VOC를 어떻게 수집했나요?"
Teacher 채널(Slack/Jira)로 인입되는 문의를 스프레드시트에 날짜·유형·심각도로 기록했습니다. 주 1회 모아서 반복 패턴을 분석하고, 빈도가 높은 유형부터 개선 대상으로 선정했습니다.

#### "URL 기반 상태 관리를 선택한 이유가 있나요?"
세 가지 장점 때문입니다. 첫째 새로고침해도 필터가 유지됩니다. 둘째 URL을 공유하면 동일한 필터 상태로 접근할 수 있습니다. 셋째 브라우저 뒤로가기로 이전 검색 상태로 돌아갈 수 있습니다. Zod로 파라미터를 검증하니 잘못된 URL이 들어와도 안전하게 기본값으로 폴백됩니다.

#### "이 프로세스를 다른 프로젝트에도 적용했나요?"
네, bb-user-web에서도 동일한 `useZodSearchParams` 패턴을 사용해서 콘텐츠 목록의 탭/필터/페이지네이션을 URL로 관리했고, SessionStorage와 조합해서 결제 흐름 중 상태 유실을 방지했습니다.

## 8. 아트봉봉 스쿨(Student)에서 색채 능력검사(CCT) 기능을 설계·개발한 과정을 설명해주세요.

> 면접관의 의도: 신규 기능을 주도적으로 설계하고 복잡한 상태 관리와 오프라인 대응까지 고려한 경험이 있는지 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

색채 심리검사는 진입-중간-최종 결과로 이어지는 단계별 플로우로 설계했습니다. 불안정한 학교 네트워크에서도 이탈 없는 검사 흐름을 유지하는 것이 핵심 과제였습니다. Framer Motion 기반으로 단계 전환 애니메이션을 구현해 시각적 "성취감"을 주는 인터랙션을 만들었고, 이 인터랙션은 공통 컴포넌트로 구조화해 재사용 기반을 마련했습니다. 검사 응답 데이터는 IndexedDB(`idb-keyval`)에 저장해서 브라우저가 꺼지거나 네트워크가 끊겨도 검사를 이어서 진행할 수 있도록 했습니다.

### 이렇게 말하면 감점

- "Framer Motion으로 애니메이션 넣었습니다"처럼 기술 나열만 하고 UX적 목적(성취감, 몰입감)을 설명하지 않는 패턴
- IndexedDB를 선택한 이유 없이 "로컬에 저장했습니다"로 끝내는 패턴. localStorage 대비 용량 이점과 비동기 특성을 설명해야 한다
- 학교 네트워크라는 사용 환경 제약을 언급하지 않는 패턴. 왜 오프라인 대응이 필요했는지가 핵심이다

### 가산점 포인트

IndexedDB persistence로 검사 데이터 유실을 방지한 것은 "불안정한 환경에서의 UX"를 깊이 고민했다는 증거다. 단순 기능 구현이 아닌 사용 환경(학교 Wi-Fi)을 고려한 설계 판단을 강조하면 좋다.

### 상세 해설

**실제 코드 기반 아키텍처:**

1. **Zustand 멀티슬라이스 스토어** (`scenarioStore.ts`):
   CCT는 7개 심리검사(SWT, AFD, DAP, HTD, VDT, CCT, CCT-Drawing) 중 하나로, Immer 미들웨어와 함께 슬라이스 패턴으로 구성됨:
   ```typescript
   export const useScenarioStore = create<ScenarioStore>()(
     immer((...a) => ({
       ...createSWTSlice(...a),
       ...createCCTSlice(...a),
       ...createCCTDrawingSlice(...a),
       // ... 7개 슬라이스 합성
     }))
   );
   ```

2. **CCT 슬라이스 구조** (`cctSlice.ts`):
   - `setCCTCurrentStage(stage, subStage)`: 검사 진행 단계 관리
   - `addCCTSaveAnswerData(sectionKey, questionKey, answer[])`: 색상 선택 저장
   - `loadCCTData(key)`: IndexedDB에서 이전 데이터 복원
   - `clearCCT(key)`: 검사 완료 후 정리
   - 상태 변경 시마다 `idbSet()`으로 자동 영속화

3. **결과 화면 구성요소**:
   - `FlippingCard.tsx`: 3D 카드 플립 애니메이션 (rotateY 1800도, 커스텀 이징 `[0.1, 0.5, 0.2, 1]`)
   - `CountResult.ts`: 점수 계산 로직
   - `RadarGraph`: Chart.js 기반 레이더 차트
   - `ScatteredKeys.tsx`, `OutroTypewriter.tsx`: 결과 서사 연출

4. **Intro 연출** (`CCT/Intro/index.tsx`):
   - `typewriter-effect`로 스토리 텍스트 순차 출력
   - Framer Motion으로 배경 이미지 페이드, 버튼 등장 애니메이션
   - 스텝별 자동 진행 (storyStepDelay 기반)

### 꼬리 질문 대비

#### "IndexedDB를 선택한 이유는?"
localStorage는 5MB 제한이 있고 동기 API라 메인 스레드를 블로킹합니다. 심리검사 응답에는 이미지 데이터가 포함될 수 있어 용량이 부족하고, `idb-keyval` 라이브러리로 Promise 기반 KV 저장을 간단하게 구현할 수 있었습니다.

#### "검사 도중 네트워크가 끊기면 어떻게 되나요?"
상태 변경마다 `idbSet()`으로 IndexedDB에 자동 영속화되기 때문에 네트워크가 끊겨도 로컬 데이터는 안전합니다. 재접속 시 `loadCCTData(key)`로 이전 진행 상태를 복원하고, 검사 완료 시점에 서버로 결과를 전송합니다.

#### "7개 슬라이스를 하나의 스토어로 합친 이유는?"
`getData(scenario)` 패턴으로 시나리오 키만 넘기면 해당 검사의 데이터에 접근할 수 있는 통일된 인터페이스를 제공하기 위해서입니다. 슬라이스 간 독립성은 유지하면서도 하나의 스토어에서 접근 가능해 컴포넌트에서의 사용이 단순해집니다.

## 9. Tool-Manager에서 Vue 레거시를 React로 마이그레이션한 이유와 과정을 설명해주세요.

> 면접관의 의도: 레거시 코드를 분석하고 마이그레이션을 주도한 경험, 그리고 기술 선택의 근거를 논리적으로 설명할 수 있는지 확인하려는 질문이다.

### 답변 타이밍: 2분

### 합격 답변 예시

기존 Vue2 기반 Tool-Manager는 상태 불일치와 페이지 간 값 전달 복잡성이 누적되어, 기능 추가·개선 요청에 대응하기 어려운 상태였습니다. 구조를 분석한 후 React 기반 전면 재설계를 제안했고, 팀 합의를 거쳐 주도적으로 진행했습니다. Zustand + Immer로 Config 상태를 전역 스토어에 일원화하고 12개 도메인 슬라이스(brush, color, image, video, paper, overlay, layer, board, contents, origin, preset, fetching)로 분리했습니다. `lodash.isEqual`로 originConfig과 현재 config의 deep comparison을 수행해 미저장 변경 감지(`needSave()`)도 구현했습니다.

### 이렇게 말하면 감점

- "Vue가 나빠서 React로 바꿨습니다" — 기술 비교가 아닌 문제 해결 관점이어야 한다
- 점진적 마이그레이션을 고려했는지 묻는 꼬리 질문에 답하지 못하는 패턴
- 12개 슬라이스라는 구조적 결정의 근거를 설명하지 못하는 패턴

### 가산점 포인트

"기술 선택은 취향이 아닌 문제 해결을 위한 판단"이라는 기준을 강조하면 좋다. 12개 슬라이스라는 구체적 숫자와 실제 도메인명을 나열하면 설계에 대한 깊은 이해를 보여줄 수 있다.

### 상세 해설

**실제 코드 기반 아키텍처:**

1. **스토어 설계** (`store/index.ts`):
   ```typescript
   export const useStore = create<TStore>()(
     immer((...a) => ({
       loading: true,
       originConfig: undefined,
       ...fetchingActionSlice(...a),
       ...brushActionSlice(...a),
       ...colorActionSlice(...a),
       // ... 12개 슬라이스 합성
     }))
   );
   ```

2. **Immer 미들웨어 활용**:
   Immer 덕분에 draft 기반 직관적 mutation이 가능:
   ```typescript
   updateColor: (color: string, index: number) => {
     set((state) => {
       if (state.config.palette.colors.includes(color)) {
         throw new Error("중복된 색상이 있습니다.");
       }
       state.config.palette.colors[index] = color;
     });
   }
   ```

3. **Config 병합 전략** (`fetchingActionSlice`):
   API에서 받은 config를 기본값과 `lodash.mergeWith`로 병합. null 값은 기존 값 유지:
   ```typescript
   config = mergeWith({}, defaultConfig, config, (objValue, srcValue) => {
     if (srcValue == null) return objValue;
   });
   ```

4. **브러시 타입 시스템** (`store/type.ts`):
   20종 브러시를 Discriminated Union으로 설계:
   ```typescript
   type BrushUnion = TPencil | TCrayon | TPen | TMarker | TWaterBrush | ...;
   ```
   각 브러시 타입의 `name` 리터럴로 타입 내로잉 가능

5. **마이그레이션 호환 로직**: 데이터 로딩 시 구 타입(isScissors, type:3)을 신 타입(scissors2, type:5)으로 자동 변환

꼬리 질문 대비:
- "점진적 마이그레이션은 고려 안 했나요?" → 규모가 크지 않아 전면 재작성이 더 효율적. 구 데이터와의 호환은 로딩 시 마이그레이션 로직으로 해결
- "에러 핸들링은?" → 슬라이스에서 throw한 Error를 호출부에서 try-catch로 잡아 alert
- "미저장 변경 감지?" → `lodash.isEqual(config, originConfig)` + `window.onbeforeunload`

## 10. 모노레포 환경에서 개발한 경험을 설명해주세요. 어떤 장단점이 있었나요?

### 합격 답변 예시

아이스크림아트에서 PNPM 워크스페이스 기반 모노레포로 Student 웹앱, Teacher 어드민, Gallery Board 등을 함께 관리했습니다. 공통 패키지로 `@school/api-client`(Axios 래퍼), `@libs/dto`(타입 안전한 DTO), `@school/logger`(플러그인 기반 로깅), `@school/translation`(i18n) 등을 분리해 양쪽 앱에서 재사용했습니다. 카탈로그 기능으로 React, Next.js, Vitest 등 핵심 의존성 버전을 워크스페이스 레벨에서 중앙 관리했습니다.

### 이렇게 말하면 감점

- "모노레포를 사용했습니다"만 말하고 구체적 패키지 구성이나 장단점 사례가 없는 패턴
- 장점만 나열하고 실제 겪은 문제(버전 충돌 등)와 해결을 말하지 않는 패턴
- PNPM을 선택한 이유를 설명하지 못하고 "원래 쓰고 있었습니다"로 끝내는 패턴

### 가산점 포인트

실제 패키지 이름과 역할을 구체적으로 나열하면 좋다. PNPM 카탈로그를 활용한 버전 중앙 관리는 모노레포 운영의 깊은 이해를 보여준다.

- 공통 패키지 재사용 (api-client, dto, logger)
- 관련 변경의 원자적 커밋/리뷰

#### "PNPM을 선택한 이유는?"

#### "패키지 간 타입 공유는 어떻게 했나요?"
`@libs/dto` 패키지에 공통 DTO를 정의하고 양쪽 앱에서 import합니다. Prisma 모델에서 파생된 타입도 여기에 정리해서 API 요청/응답 타입이 프론트와 백엔드 간 일치하도록 했습니다. TypeScript의 project references로 빌드 순서도 관리합니다.

#### "CI 빌드 시간은 어떻게 관리했나요?"
변경된 패키지만 빌드하도록 `pnpm --filter` 옵션을 활용했습니다. 예를 들어 `@school/logger`만 수정했으면 해당 패키지와 이를 의존하는 앱만 재빌드합니다. Turbo 도입도 검토 중이었습니다.

## 11. Playwright E2E 테스트와 Vitest 단위 테스트를 도입한 경험을 설명해주세요.

> 면접관의 의도: 테스트 전략을 설계하고 CI에 통합한 경험이 있는지, E2E와 단위 테스트의 역할을 구분하는지 확인하려는 질문이다.

### 이렇게 말하면 감점

- "테스트를 작성했습니다"만 말하고 왜 이 시점에 도입했는지 맥락이 없는 패턴
- E2E와 단위 테스트의 역할 구분 없이 "다 테스트했습니다"로 뭉뚱그리는 패턴
- CI 통합을 언급하지 않아 "로컬에서만 돌렸다"는 인상을 주는 패턴

### 가산점 포인트

테스트 인프라의 구체적 설정(의존성, 비디오 녹화, Storybook 통합)을 언급하면 "단순히 테스트를 작성했다"를 넘어 "테스트 전략을 설계했다"는 인상을 준다.

### 상세 해설

**Playwright E2E 설정** (`playwright.config.ts`):
```typescript
{

  expect: { timeout: 15_000 },
  use: {
    screenshot: "only-on-failure",

    trace: "on-first-retry",
    locale: "ko-KR",
  },

  ],
```

- `@vitest/browser-playwright`로 실제 브라우저 환경에서 렌더링 테스트
- 모듈 mock은 `vi.hoisted()` + `vi.mock()` 패턴 사용

### 꼬리 질문 대비

#### "테스트 커버리지는 어느 정도인가요?"
핵심 사용자 흐름(로그인, 이용권 구매, 선물하기)을 E2E로, 유틸 함수와 API 클라이언트를 단위 테스트로 커버했습니다. 수치적 커버리지보다 "배포 시 깨지면 안 되는 흐름"을 우선 커버하는 전략이었고, 점진적으로 확대 중이었습니다.

#### "E2E 테스트가 CI에서 깨지면 어떻게 대응하나요?"
실패 시 비디오와 스크린샷이 자동 저장되므로 재현 없이 원인을 파악할 수 있습니다. 테스트 자체의 flaky 여부를 먼저 확인하고, 실제 버그라면 수정 후 PR을 다시 올립니다. `retries: 1` 설정으로 네트워크 등 일시적 이유의 실패는 한 번 재시도합니다.

#### "Storybook 통합 테스트의 장점이 뭔가요?"
컴포넌트를 실제 브라우저에서 렌더링하면서 테스트하므로 jsdom 기반보다 정확합니다. Storybook에서 이미 작성한 스토리를 Vitest에서 재사용하니 테스트 작성 비용도 줄어듭니다.

## 12. Framer Motion을 활용한 애니메이션 구현 경험을 구체적으로 설명해주세요.

#### "CSS 애니메이션 대신 Framer Motion을 쓴 이유는?"
React 상태와 자연스럽게 연동할 수 있고, `AnimatePresence`로 마운트/언마운트 애니메이션을 선언적으로 처리할 수 있기 때문입니다. gesture(drag, tap) 지원도 CSS로는 불가능합니다. `variants`로 상태별 애니메이션을 정의하면 코드가 훨씬 깔끔해집니다.

#### "성능 이슈는 없었나요?"
transform과 opacity만 사용해서 GPU 합성 레이어에서 처리되도록 했습니다. layout 트리거(width, height, top 등)를 피하면 reflow 없이 60fps를 유지할 수 있습니다. 모바일 저사양 기기에서도 테스트해서 프레임 드롭이 없는 것을 확인했습니다.

#### "공통 컴포넌트는 어떻게 설계했나요?"
duration, direction, delay 등을 props로 받는 wrapper 컴포넌트를 만들었습니다. `FadeIn`, `SlideUp` 같은 기본 모션과 `CardFlip` 같은 복합 모션으로 분리해서, 다른 플로우에서도 재사용할 수 있게 했습니다.

## 13. Art-bonbon B2C에서 고해상도 영상 최적화를 위해 HLS를 적용한 과정을 설명해주세요.

> 면접관의 의도: 미디어 스트리밍 프로토콜에 대한 이해도와 브라우저 호환성 처리, 성능 최적화 경험을 확인하고자 한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "HLS랑 DASH는 비슷한 거라서 아무거나 써도 됩니다" — 프로토콜 차이를 모르는 것으로 판단된다
- "Safari도 hls.js로 똑같이 처리했습니다" — Safari 네이티브 HLS 지원을 모르면 실무 경험이 의심된다
- "영상은 그냥 video 태그에 src 넣으면 되는 거 아닌가요?" — 스트리밍 프로토콜을 이해하지 못한 것이다

### 합격 답변 예시

커리큘럼 소개 영상이 고해상도 단일 mp4로 제공되다 보니 네트워크 환경에 따라 초기 로딩이 지연되어 이탈 요인이 되고 있었습니다. HLS(HTTP Live Streaming) 방식을 도입해 영상을 `.m3u8` 매니페스트 + 세그먼트로 변환했습니다. 클라이언트에서는 `hls.js` 라이브러리로 적응적 비트레이트 스트리밍을 구현했고, Safari는 네이티브 HLS를 지원하므로 분기 처리했습니다. 버퍼를 1초로 설정해 저지연 재생을 실현했습니다.

### 가산점 포인트

HLS의 구체적 동작 원리(매니페스트 + 세그먼트)와 Safari 분기 처리를 설명하면 깊이가 보인다. 프론트엔드 개발자가 영상 파이프라인까지 고려한다는 점이 차별화 포인트.

### 상세 해설

**실제 코드 기반 구현:**

1. **hls.js 커스텀 훅** (`useSliceVideo.tsx`):
   ```typescript
   const hls = new Hls({ maxBufferLength: 1 }); // 1초 버퍼 (저지연)
   hls.loadSource(videoSrc);  // .m3u8 파일
   hls.attachMedia(video);

   // Safari 네이티브 HLS fallback
   if (video.canPlayType("application/vnd.apple.mpegurl")) {
     video.src = videoSrc;
   }
   ```

2. **React-Player + HLS** (`VideoPlayer.tsx`):
   ```typescript
   <ReactPlayer
     url={`${S3_URL}/${videoUrl}`}
     config={{
       file: {
         hlsOptions: {
           startLevel: 2,        // 최고 품질로 시작
           maxBufferLength: 10,  // 10초 버퍼
         },
       },
     }}
   />
   ```

3. **뷰포트 기반 지연 재생**: Intersection Observer로 뷰포트에 들어올 때만 재생

**HLS vs 단일 mp4:**
| 항목 | 단일 mp4 | HLS |
|------|---------|-----|
| 초기 로딩 | 전체 파일 다운로드 시작 | 첫 세그먼트만 로드 |
| 품질 적응 | 고정 | 네트워크에 따라 자동 전환 |
| CDN 캐시 | 파일 단위 | 세그먼트 단위 (효율적) |

### 꼬리 질문 대비

#### "HLS와 MPEG-DASH의 차이점은?"
HLS는 Apple 프로토콜로 .m3u8+.ts, DASH는 국제 표준으로 .mpd+.m4s를 사용합니다. Safari는 HLS만 네이티브 지원하고 DASH는 지원하지 않습니다. hls.js로 크로스 브라우저 대응이 충분했기 때문에 HLS를 선택했습니다.

#### "maxBufferLength를 1로 설정하면 끊김이 발생하지 않나요?"
1초 버퍼는 프리뷰용 짧은 영상에 적합한 설정입니다. 본편 영상에는 React-Player에서 maxBufferLength를 10으로 올려 안정적 재생을 보장했습니다. 사용 맥락에 따라 버퍼 전략을 다르게 가져간 것이 핵심입니다.

#### "Adaptive Bitrate Streaming은 어떻게 동작하나요?"
hls.js가 내부적으로 ABR 알고리즘을 돌려서 대역폭과 버퍼 상태에 따라 품질을 자동 전환합니다. startLevel을 2로 지정한 이유는 최저 품질부터 시작하면 초반 화질이 낮아 보여서, 중간 품질부터 시작한 뒤 ABR이 판단하도록 한 것입니다.

## 14. Ecole-Admin에서 CSV Bulk Upload 기능을 구현한 경험을 설명해주세요.

> 면접관의 의도: 파일 파싱, 데이터 검증, 서버 액션, 트랜잭션 처리 등 풀스택에 가까운 구현 능력을 확인하고자 한다.

### 합격 답변 예시

기존에는 학원별 학생을 한 명씩 수동 등록하고 있어 대량 등록 시 비효율과 입력 오류가 반복되는 상황이었습니다. CSV 파일 업로드 → 파싱 → 한국어 헤더를 영문으로 매핑 → Zod 스키마로 행 단위 검증 → 에러 행 분리 → 유효 데이터만 Server Action으로 전송하는 파이프라인을 구현했습니다. Server Action에서는 Prisma `$transaction`으로 원자적 bulk insert를 수행하고, 등록 가능 인원 초과 여부도 사전 체크합니다.

### 가산점 포인트

프론트엔드 개발자가 데이터 흐름 전반(파싱 → 검증 → 트랜잭션)을 이해하고 구현했다는 점이 차별화 포인트. 한국어 헤더 매핑이라는 세부 구현까지 언급하면 실제로 만든 사람임을 증명할 수 있다.

#### "왜 클라이언트와 서버 양쪽에서 검증하나요?"
클라이언트 검증은 UX를 위한 것이고 서버 검증은 무결성을 위한 것입니다. 클라이언트만 하면 API 직접 호출로 우회 가능하고, 서버만 하면 제출 후에야 오류를 알 수 있습니다. 네트워크 전송 중 변조 가능성도 있으므로 safeParse로 이중 검증합니다.

#### "Discriminated Union 패턴의 장점은?"
TypeScript 타입 좁히기가 자동으로 동작합니다. `if (result.code === "SUCCESS")` 안에서 `result.count`에 타입 안전하게 접근 가능하고, `as const`로 리터럴 타입이니 switch exhaustive check도 됩니다.

#### "$transaction을 사용한 이유는?"
200명 등록 중 150번째에서 에러 나면 149명만 저장됩니다. 사용자가 다시 업로드하면 중복 등록 발생. $transaction으로 하나라도 실패 시 전체 롤백하여 부분 저장을 원천 차단합니다.

## 15. Zustand를 상태 관리로 선택한 이유는 무엇인가요? 프로젝트별로 어떻게 다르게 활용했나요?

> 면접관의 의도: 상태 관리 라이브러리를 상황에 맞게 설계하는 판단력과 슬라이스 패턴, 미들웨어 활용 등 심화 경험을 확인하고자 한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "Zustand는 간단해서 하나의 큰 store에 다 넣었습니다" — 슬라이스 패턴을 모르면 규모 있는 경험이 의심된다
- "상태 관리는 다 비슷해서 Redux든 Zustand든 상관없습니다" — 각 라이브러리의 트레이드오프를 이해하지 못한 것이다
- "서버 상태도 Zustand로 관리했습니다" — React Query와의 역할 분리를 모르는 것이다

### 합격 답변 예시

Zustand는 세 프로젝트에서 각각 다른 패턴으로 활용했습니다. Tool-Manager에서는 12개 도메인 슬라이스 + Immer 미들웨어로 Config CRUD를 관리했고, 아트봉봉 스쿨에서는 7개 심리검사 슬라이스 + IndexedDB persistence로 검사 데이터를 관리했습니다. Teacher 앱에서는 다이얼로그 스토어를 generic 타입의 `resolve` 콜백 패턴으로 설계해, Promise처럼 다이얼로그 결과를 받을 수 있게 했습니다. 서버 상태는 React Query로 분리하고, Zustand는 순수 클라이언트 상태만 담당하게 해서 역할을 명확히 구분했습니다.

### 가산점 포인트

"Zustand를 썼다"에서 끝나지 않고, 프로젝트별로 다른 패턴을 적용한 이유를 설명하면 상태 관리에 대한 깊은 이해를 보여줄 수 있다. React Query와의 역할 분담도 핵심 포인트.

### 상세 해설

**프로젝트별 Zustand 활용 비교:**

| 프로젝트 | 슬라이스 수 | 미들웨어 | Persistence | 특징 |
|----------|------------|---------|-------------|------|
| Tool-Manager | 12개 | Immer | 없음 (API 저장) | Config CRUD |
| 스쿨 Student | 7개 | Immer | IndexedDB (idb-keyval) | 심리검사 데이터 |
| 스쿨 Teacher | 1개 | 없음 | 없음 | 다이얼로그 제어 |
| 스쿨 Debug | 1개 | persist | localStorage | 디버그 패널 토글 |

**다이얼로그 스토어 패턴** (`dialogStore.ts`):
```typescript
type DialogRequestComponentProps<ReturnType> = {
  resolve: (value: ReturnType) => void;
};
// 다이얼로그를 열면 resolve 콜백을 받고, 닫을 때 값을 반환
```

**React Query 연동** (`useSchoolApi`):
- `queryKey`를 API pathname + request JSON으로 자동 생성
- 지수 백오프 재시도: `2000 * 2^attempt`, 최대 60초
- `addApiLog(api, request, response)`로 디버그 로깅

### 꼬리 질문 대비

#### "12개 슬라이스가 많지 않나요? 분리 기준이 뭔가요?"
도메인 기준입니다. 브러시, 색상, 배경지 등 기능 단위가 명확히 나뉘고, 슬라이스당 상태 2-3개 + 액션 3-4개로 파일이 50줄 이내입니다. 하나의 거대한 store는 500줄을 넘겨서 유지보수가 어려워집니다.

#### "IndexedDB 영속화는 왜 필요했나요?"
학생이 학습 중 브라우저를 닫아도 진행 상태가 유지되어야 했습니다. localStorage는 5MB 제한이 있어 이미지 포함 데이터에 부족하고, IndexedDB는 수백 MB까지 가능합니다. `idb-keyval`로 간단한 KV 저장 패턴을 적용했습니다.

#### "Recoil 대신 Zustand를 추천하는 이유는?"
Recoil은 Meta에서 공식 지원이 중단되었고, atom 기반이라 상태 흐름을 한눈에 파악하기 어렵습니다. Zustand는 번들 약 1KB, 슬라이스 패턴으로 확장성이 좋고, `getState()`로 React 외부 접근이 가능해 유연합니다.

## 16. Zod를 여러 프로젝트에서 활용했는데, 어떤 상황에서 왜 Zod를 사용하나요?

> 면접관의 의도: 런타임 검증 도구의 다양한 활용 패턴과 프로젝트 상황에 따른 기술 선택 판단력을 확인하고자 한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "Zod는 모든 프로젝트에서 반드시 써야 합니다" — 상황에 따라 수동 검증이 적합할 수 있다는 판단력 부족
- "TypeScript 타입이 있으니 런타임 검증은 불필요합니다" — 타입 소거와 런타임 안전성의 차이를 모르는 것
- "Zod 스키마를 z.any()로 다 처리했습니다" — 사용 의미가 없어진다

### 합격 답변 예시

TypeScript는 컴파일 타임에만 타입을 보장하기 때문에, 외부 입력(사용자 입력, API 응답, 파일 파싱)에서는 런타임 검증이 반드시 필요합니다. Zod는 스키마를 정의하면 TypeScript 타입도 자동 추론(`z.infer`)되어 이중 작성이 줄어듭니다. Ecole-Admin에서는 CSV 업로드 행 검증과 Server Action 입력 검증에, 아트봉봉 스쿨에서는 Formik과 조합(`zod-formik-adapter`)해 폼 단계별 검증에 사용했습니다. URL 검색 파라미터 검증에는 `zod-search-params`를 통해 URL과 타입 시스템을 동기화했습니다.

### 가산점 포인트

"시스템 경계에서의 런타임 검증"이라는 원칙을 말하면 기술적 깊이를 보여줄 수 있다. Tool-Manager는 Zod 대신 슬라이스 내 수동 검증을 사용했는데, 그 이유(Config 앱 특성상 UI 레벨에서 즉각 피드백이 더 적합)까지 설명하면 도구 선택의 유연성을 보여준다.

### 상세 해설

**프로젝트별 Zod 활용 패턴:**

1. **Ecole-Admin**: CSV 행 검증 + Server Action 입력 검증 + Prisma 자동 생성
   ```typescript
   // .refine()으로 교차 필드 검증
   .refine((data) => data.groupCount <= CURRICULUM_GROUP_COUNT[data.kind])
   // .nativeEnum()으로 Prisma enum 연동
   gender: z.nativeEnum(Gender).nullish()
   ```

2. **아트봉봉 스쿨**: Formik + Zod + i18n 에러 메시지
   ```typescript
   region: z.string({ required_error: tForScan("필수항목입니다") })
   ```

3. **bb-user-web**: URL 검색 파라미터 + MBTI 응답 검증
   ```typescript
   export const paymentContentsSchema = z.object({
     tab: z.enum(["palette", "all", "master", "fun"]).default("all"),
     cr: z.array(z.string()).default([]),
     lv: z.array(z.string()).default([]),
   });
   ```

4. **Tool-Manager는 Zod 미사용**: 슬라이스 내 수동 검증 (`validateColorListLength`) + 정규식 (`CONTENT_ID_REGEX`) 사용. Config 앱 특성상 저장 시점이 아닌 조작 시점에 즉각 에러를 throw하는 것이 UX에 적합했기 때문.

### 꼬리 질문 대비

#### "TypeScript 타입과 Zod 스키마의 차이가 뭔가요?"
TypeScript 타입은 컴파일 후 사라지고, API 응답이나 CSV 같은 외부 데이터는 런타임에 형태를 알 수 없습니다. Zod는 런타임에 실제 데이터를 검증하고, `z.infer`로 타입도 자동 추론되어 스키마와 타입이 한 곳에서 관리됩니다.

#### "prisma-zod-generator는 어떤 원리로 동작하나요?"
Prisma의 generator 시스템을 활용합니다. `prisma generate` 명령 시 각 모델에 대응하는 Zod 스키마가 자동 생성됩니다. String은 z.string(), Int는 z.number(), enum은 z.nativeEnum()으로 매핑되어 DB 마이그레이션과 동시에 검증 스키마가 갱신됩니다.

#### "refine과 superRefine의 차이는?"
refine은 단일 boolean으로 하나의 에러를 추가하고, superRefine은 `ctx.addIssue`로 여러 에러를 세밀하게 추가합니다. CSV에서 각 행별로 다른 에러를 표시해야 할 때 superRefine이 유용합니다.

## 17. GitHub Actions 기반 CI/CD 경험을 설명해주세요. 배포 파이프라인을 어떻게 구성했나요?

> 면접관의 의도: 빌드, 테스트, 배포 자동화에 대한 실무 경험과 코드 품질 관리에 대한 의식 수준을 확인하고자 한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "배포는 수동으로 서버에 접속해서 git pull 합니다" — 자동화에 대한 인식 부족
- "테스트는 시간이 없어서 작성하지 않았습니다" — 품질 관리 의지가 없다고 판단
- "린트는 에디터에서만 확인하면 됩니다" — 팀 전체의 코드 일관성을 보장하지 못한다

### 합격 답변 예시

양치킹에서는 GitHub Actions + AWS EC2 + PM2 조합으로 배포 파이프라인을 구성했습니다. main 브랜치에 push되면 Actions가 빌드·테스트를 실행하고, 성공 시 EC2에 SSH로 접속해 코드를 pull하고 PM2로 재시작하는 흐름입니다. 아트봉봉 스쿨에서는 Playwright E2E 테스트를 CI에 포함시켜 배포 전 안정성을 검증하는 구조로 확장했고, Git 훅은 Lefthook으로 pre-commit(lint staged) + pre-push(타입 체크)를 설정해 로컬에서도 품질을 관리합니다.

### 가산점 포인트

"자동 배포를 했다"보다 "왜 자동화가 필요했는가"(1인 운영에서 수동 배포의 리스크)를 먼저 설명하면 좋다. Lefthook 같은 로컬 훅까지 언급하면 개발 프로세스 전체를 설계한다는 인상.

### 상세 해설

**양치킹 배포 파이프라인:**
```
Push to main → GitHub Actions → npm install & build → SSH to EC2 → git pull → pm2 restart
```

**아트봉봉 스쿨 CI 파이프라인:**
```
PR 생성 → Lefthook pre-commit (lint) → CI: 빌드 + Vitest → Playwright E2E → 실패 시 블록
```

**Lefthook 설정** (`lefthook.yml`):
- `pre-commit`: lint staged files
- `pre-push`: 타입 체킹

### 꼬리 질문 대비

#### "Lefthook과 Husky의 차이점은?"
Husky는 .husky 디렉토리에 쉘 스크립트를 작성하고 lint-staged가 별도로 필요합니다. Lefthook은 YAML 하나로 모든 설정이 가능하고 `staged_files` glob이 내장되어 있습니다. Go로 작성되어 병렬 실행 성능이 좋습니다.

#### "E2E 테스트가 느려서 CI가 오래 걸리면?"
세 가지 전략입니다. 첫째 Playwright workers 옵션으로 병렬 실행, 둘째 크리티컬 패스만 E2E로 나머지는 유닛으로, 셋째 PR에서는 관련 테스트만 실행하고 main 머지 시 전체 실행으로 분리합니다.

#### "PM2를 사용한 이유는?"
크래시 시 자동 재시작, 클러스터 모드 CPU 활용, 무중단 배포(reload)를 지원합니다. 단일 EC2에서 운영하는 규모라 컨테이너 오케스트레이션은 불필요했고, PM2로 충분했습니다.

## 18. 프론트엔드 개발에서 가장 중요하게 생각하는 가치는 무엇인가요?

> 면접관의 의도: 기술적 역량을 넘어 개발 철학과 문제 해결 태도, 팀 내에서 어떤 방향성을 가진 개발자인지 파악하고자 한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "최신 기술을 빠르게 도입하는 것이 가장 중요합니다" — 기술 자체가 목적이 되면 불필요한 복잡성을 추가한다
- "코드 품질만 높으면 됩니다" — 사용자 경험과 비즈니스 가치를 고려하지 않는 개발자로 보인다
- "가치관이라기보다 주어진 일을 잘하면 되는 거 아닌가요?" — 주도성과 방향성이 부족하다

### 합격 답변 예시

"사용자가 겪는 순간"을 가장 중요하게 생각합니다. 양치킹에서 네트워크 불안정 문제를 해결할 때, 아트봉봉 스쿨에서 VOC를 데이터로 유형화할 때, 모두 "사용자의 경험"이 출발점이었습니다. 코드가 아무리 깔끔해도 사용자 흐름이 끊기면 가치가 없고, 반대로 작은 UX 개선이 서비스 신뢰도와 운영 효율에 직결되는 것을 체감했습니다.

### 가산점 포인트

추상적 가치관만 말하지 않고, 본인의 프로젝트 경험에서 구체적으로 이 가치관이 드러난 순간을 함께 언급하면 진정성이 높아진다.

### 상세 해설

포트폴리오에서 일관되게 드러나는 가치관:
1. 사용자 경험 중심 — VOC 유형화, Fallback UI, 애니메이션의 UX적 목적
2. 운영 관점 — "구현 난이도보다 사용 흐름·문의·운영 비용을 먼저 본다"
3. 문제 해결 기반 기술 선택 — "기술 선택은 취향이 아닌 문제 해결을 위한 판단"
4. 끝까지 책임지는 오너십 — 1인 개발, 운영, VOC 대응

### 꼬리 질문 대비

#### "좋은 코드란 무엇이라고 생각하시나요?"
읽기 쉬운 코드입니다. 6개월 후에 제가 봐도, 새 팀원이 봐도 의도를 파악할 수 있는 코드. 변수명이 동작을 설명하고, 하나의 함수가 하나의 책임만 갖는 코드입니다. 성능 최적화보다 가독성이 우선이고, 병목이 확인된 지점에서만 최적화합니다.

#### "기술 트렌드를 어떻게 따라가시나요?"
주간 단위로 This Week in React와 GitHub Trending을 확인하고, 관심 기술은 토이 프로젝트에서 먼저 써봅니다. 다만 프로덕션 도입은 최소 3개월 커뮤니티 반응을 지켜본 뒤, 현재 프로젝트에 실질적 이점이 있을 때만 검토합니다.

#### "팀에서 기술적 의견 충돌이 있으면 어떻게 해결하시나요?"
상대방 의견을 끝까지 듣고 양쪽 트레이드오프를 문서화합니다. Tool-Manager 레거시 개선 때 전면 재작성 vs 점진적 마이그레이션 의견이 갈렸는데, 각 방식의 리스크와 소요 시간을 수치로 비교한 문서를 만들어 팀이 합의했습니다. 감정이 아닌 데이터로 대화하는 것이 중요합니다.

## 19. 2년차 개발자로서 본인의 강점과 보완할 점은 무엇인가요?

> 면접관의 의도: 자기 객관화 능력과 성장 의지를 동시에 확인하려는 질문이다. 강점이 업무 성과로 연결되는지, 약점을 회피하지 않고 개선 방향을 갖고 있는지 본다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "약점은 딱히 없는 것 같습니다" — 자기 객관화가 안 된다는 인상
- "완벽주의가 단점입니다" — 전형적인 회피형 답변으로 진정성이 떨어진다
- 강점을 나열만 하고 구체적 사례가 없는 경우 — "다양한 기술을 잘합니다"는 공허하다

### 합격 답변 예시

강점은 "프로덕트 전체를 보는 시야"입니다. 양치킹에서 기획-개발-배포-운영 전 과정을 경험했고, 회사에서도 VOC 수집부터 우선순위 설정, 기능 개선까지 주도적으로 해왔습니다. 보완할 점은 "대규모 코드베이스에서의 아키텍처 경험"입니다. 2인~4인 규모의 팀에서 일해왔기 때문에 대규모 팀에서의 아키텍처 설계, 코드 리뷰 문화에 대한 경험을 더 쌓고 싶습니다.

### 가산점 포인트

보완할 점을 말할 때 "못한다"가 아니라 "경험을 쌓고 싶다"로 표현하면 성장 의지를 보여준다.

### 상세 해설

**강점 소재:**
- 프로덕트 오너 관점 (양치킹 1인 개발·운영)
- 사용자 중심 사고 (VOC → 데이터 → 개선, 수치로 증명)
- 다양한 기술 스택 (React Native, Next.js, Prisma, AWS, VS Code Extension)
- 테스트 문화 도입 (E2E + 단위 테스트)

**보완점 소재 (솔직하되 긍정적으로):**
- 대규모 팀 경험
- 깊은 수준의 성능 최적화
- 백엔드/인프라의 더 깊은 이해

### 꼬리 질문 대비

#### "소규모 팀 경험이 대규모 팀에서 오히려 방해가 되지 않을까요?"
소규모 팀에서는 빠른 판단과 실행력이 길러졌습니다. 대규모 팀에서는 '합의'와 '문서화'가 더 중요하다는 걸 알고 있고, 양치킹에서도 팀이 커지면서 PR 템플릿과 작업 단위를 작게 나누는 방식을 도입한 경험이 있습니다.

#### "VOC를 수치로 바꿨다는 게 구체적으로 어떤 사례인가요?"
특정 화면의 퍼널 데이터를 분석해서 이탈 지점을 특정하고, UI 흐름 개선 후 이탈률 변화를 추적했습니다. 단순히 '불편하다'에서 끝나지 않고 수치로 전후를 비교해서 의사결정 근거로 활용한 것이 핵심입니다.

## 20. 앞으로의 커리어 목표와 성장 방향을 말씀해주세요.

> 면접관의 의도: 지원자가 이 회사에서 얼마나 오래, 어떤 방향으로 성장할지를 가늠하려는 질문이다. 회사의 성장 방향과 겹치는지 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "3년 안에 CTO가 되고 싶습니다" — 비현실적이고 조직 이해 부족
- "아직 구체적으로 생각해본 적 없습니다" — 커리어에 대한 주도성이 없다는 인상
- "창업이 목표입니다" — 회사 입장에서 장기 투자 가치가 낮다고 느낄 수 있다

### 합격 답변 예시

단기적으로는 더 큰 규모의 팀에서 체계적인 프로세스(설계→리뷰→테스트→배포) 안에서 동료들과 함께 "납득되는 결과물"을 만들어보고 싶습니다. 중장기적으로는 비즈니스와 기술 사이의 접점을 이해하는 시니어 개발자가 되고 싶습니다.

### 가산점 포인트

지원하는 회사에서 이 목표를 어떻게 달성할 수 있는지와 연결하면 지원 동기가 강화된다.

### 상세 해설

**좋은 답변의 구조:**
1. 단기 목표 (1-2년): 구체적이고 현실적
2. 중장기 목표 (3-5년): 방향성 제시
3. 지원 회사와의 연결

포트폴리오에서 끌어올 수 있는 근거:
- "혼자 이해하는 코드보다 함께 납득되는 결과물의 가치"
- "기술보다 팀 합의와 맥락 이해가 결과 품질을 좌우한다"

### 꼬리 질문 대비

#### "비즈니스와 기술의 접점이라고 하셨는데, 구체적으로 어떤 상황인가요?"
새 기능 개발 시 단순히 구현하는 게 아니라 '이 기능이 사용자 리텐션에 어떤 영향을 줄까', '이 기술적 선택이 향후 확장에 어떤 제약을 만들까'를 함께 고민하는 것입니다. 양치킹에서 결제 시스템을 붙일 때도 쿠폰 정책과 전환율을 함께 고려해서 설계했습니다.

#### "우리 회사에서 구체적으로 어떤 기여를 할 수 있을 것 같나요?"
프로덕트 전체를 경험한 시야로 기능 개발 시 사용자 관점 검증을 자연스럽게 녹여낼 수 있고, 다양한 스택 경험으로 새 기술 도입이나 마이그레이션에 빠르게 적응할 수 있습니다. 단기적으로는 팀 프로세스를 흡수하며 기여하고, 점차 프로덕트 개선 아이디어를 기술적으로 구현하는 역할을 하고 싶습니다.

## 21. 아트봉봉 스쿨의 네트워크 상태 감지 시스템을 구체적으로 설명해주세요.

> 면접관의 의도: 네트워크 상태 관리라는 현실적 문제를 어떤 계층 구조로 설계했는지, 각 계층의 한계를 인식하고 보완한 경험이 있는지 확인한다.

### 답변 타이밍: 3분

### 이렇게 말하면 감점

- "navigator.onLine만 사용했습니다" — 이 API의 한계를 모르는 것처럼 들린다
- 왜 여러 계층이 필요한지 설명하지 않고 구현만 나열하는 패턴
- "라이브러리를 사용했습니다" — 직접 설계한 경험을 물어보는 질문에서 외부 의존만 언급하면 역량이 안 보인다

### 합격 답변 예시

4계층으로 네트워크 상태를 감지합니다. 첫째, `usePingCheck` 훅이 60초마다 `/api/v3/alive` 엔드포인트에 ping을 보내 서버 도달 여부를 확인합니다. 둘째, `useNavigatorOnline`이 `navigator.onLine`과 online/offline 이벤트를 감시합니다. 셋째, `useNetworkEffectiveType`이 `navigator.connection.effectiveType`으로 네트워크 속도를 감지하고, slow-2g면 토스트 경고를 표시합니다. 넷째, `useContinuousNavigatorOffline`이 1.5초간 지속적으로 오프라인인 경우에만 DISCONNECTED로 판정해 순간적인 끊김에 의한 오탐을 방지합니다. 이 네 가지를 조합한 `useNetworkStatus` 훅이 최종 `disconnectedStatus`를 `PENDING | CONNECTED | DISCONNECTED`로 산출합니다.

### 가산점 포인트

"navigator.onLine만 쓰면 되지 않나요?"라는 꼬리 질문에 대비해, 왜 4계층이 필요한지 설명할 수 있어야 한다. `navigator.onLine`은 물리적 연결만 감지하고 실제 서버 도달 여부는 모르기 때문에 ping이 필요하다.

### 상세 해설

**실제 코드 구조:**

```typescript
// useNetworkStatus.ts - 4계층 조합
export default function useNetworkStatus() {
  const { pingResult, checkPing } = usePingCheck(60000);       // 1) 서버 ping
  const navigatorOnline = useNavigatorOnline();                  // 2) navigator.onLine
  const { networkEffectiveType } = useNetworkEffectiveType();    // 3) 연결 속도
  const continuousOffline = useContinuousNavigatorOffline(navigatorOnline); // 4) 지속 오프라인

  const disconnectedStatus = useMemo(() => {
    if (pingResult === "PENDING") return "PENDING";
    if (continuousOffline) return "DISCONNECTED";
    return "CONNECTED";
  }, [pingResult, continuousOffline]);
}
```

**디바운싱 (1.5초):**
```typescript
const OFFLINE_VERIFY_TIME = 1500;
// 순간적 끊김은 무시, 1.5초 이상 지속 시에만 DISCONNECTED
```

**UI 연동** (`NetworkConnectionStatus.tsx`):
- `DISCONNECTED`일 때만 전체 화면 오버레이 표시
- slow-2g 감지 시 토스트 경고

### 꼬리 질문 대비

#### "왜 60초 간격으로 핑을 보내나요?"
서버 부하와 배터리 소모를 고려한 값입니다. 너무 짧으면 네트워크 대역폭과 배터리를 낭비하고, 실시간성은 navigator.onLine 이벤트로 보완합니다. 더 실시간이 필요하면 WebSocket을 고려할 수 있지만 이 서비스에서는 오버엔지니어링이라 판단했습니다.

#### "navigator.onLine이 false positive를 내는 구체적인 상황은?"
공용 Wi-Fi의 캡티브 포털(로그인 페이지)이 대표적입니다. Wi-Fi 자체는 연결돼 있어서 navigator.onLine은 true지만 인터넷은 안 됩니다. VPN 끊김 시에도 물리적 네트워크는 살아 있지만 서버에 도달 못하는 상황이 발생합니다.

#### "PENDING 상태는 왜 필요한가요?"
앱이 처음 로드될 때 핑 체크가 아직 완료되지 않은 시점이 있습니다. CONNECTED로 가정하면 잘못된 정보를 보여줄 수 있고, DISCONNECTED로 가정하면 불필요한 오버레이가 뜹니다. PENDING 동안 스켈레톤을 보여주면 자연스럽습니다.

## 22. API Client 추상화 패턴을 설명해주세요.

> 면접관의 의도: 공통 로직을 어떻게 추상화하고 확장 가능한 구조를 설계하는지 확인한다. 인증, 에러 처리, 모니터링 등 횡단 관심사를 어떻게 다뤘는지가 핵심이다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "axios를 그냥 사용했습니다" — 추상화나 설계 의도가 안 드러난다
- 에러 처리를 언급하지 않는 경우 — API Client에서 에러 처리는 가장 중요한 관심사
- "모든 API에 같은 코드를 복사했습니다" — DRY 원칙에 대한 이해 부족

### 합격 답변 예시

아트봉봉 스쿨에서는 `ApiClient` 추상 클래스를 만들고 `SchoolApiClient`가 이를 상속하는 패턴으로 API 계층을 설계했습니다. 요청 인터셉터에서 토큰 자동 주입, 응답 인터셉터에서 에러 변환과 Sentry 브레드크럼 추적을 공통으로 처리합니다. API 메서드는 제네릭 기반의 `createSchoolApiFunction<Request, Response>`로 타입 안전하게 정의하고, React Query의 `queryKey`를 API pathname + request JSON으로 자동 생성합니다.

### 가산점 포인트

"Axios를 썼다"가 아니라 "왜 추상화했는가"(공통 관심사 분리, 타입 안전성, 패키지 재사용)를 설명하면 아키텍처 이해도를 보여줄 수 있다.

### 상세 해설

**스쿨 모노레포 - `@school/api-client` 패키지:**

1. **추상 클래스** (`ApiClient.ts`):
   - 요청 인터셉터: public API 목록 외 요청에 자동으로 Bearer 토큰 주입
   - 응답 인터셉터: status >= 250은 에러로 처리, `ApiError` 커스텀 에러로 변환
   - FormData 자동 변환: File/Blob 감지 시 multipart 전환

2. **타입 안전한 API 메서드 정의:**
   ```typescript
   public getSchools = this.createSchoolApiFunction<GetSchoolsReq, GetSchoolsRes>(
     "/v1/school/schools", "GET", true
   );
   // 경로 파라미터 지원
   public getInquiry = this.createSchoolApiFunctionWithPathParameters<
     { inquiryId: number }, unknown, GetInquiryRes
   >(({ inquiryId }) => `/v1/teacher/questions/${inquiryId}`, ["inquiryId"], "GET", true);
   ```

3. **React Query 연동** (`useSchoolApi`):
   ```typescript
   queryKey: [
     schoolApiClient.getPathname(api),     // API 경로
     JSON.stringify(request),               // 요청 파라미터
   ]
   ```

**bb-user-web - Singleton 패턴:**
- `BaseApiClient`에서 토큰 주입 + 401 자동 갱신
- 20개+ 도메인별 클라이언트 (MBTIApiClient, OrderApiClient 등)
- `static getInstance()` 싱글톤 패턴

### 꼬리 질문 대비

#### "왜 250 이상을 에러로 판단했나요?"
백엔드와 협의한 자체 규격입니다. 200~249를 성공, 250 이상을 비즈니스 에러로 구분했습니다. '이미 수강 신청된 과목' 같은 경우 HTTP 자체는 실패가 아니지만 클라이언트에서는 에러로 처리해야 하는 상황을 250번대로 구분했습니다.

#### "싱글톤 패턴을 사용한 이유는?"
토큰 갱신 로직 때문입니다. 여러 클라이언트가 동시에 401을 받으면 refresh 요청이 중복됩니다. 싱글톤으로 하나의 BaseApiClient를 공유하면 refresh 로직을 한 곳에서 관리하면서 동시 갱신을 방지할 수 있습니다.

#### "FormData 자동 변환은 어떻게?"
body가 FormData `instanceof`인지 판단합니다. FormData면 Content-Type 헤더를 제거해 브라우저가 boundary를 자동 생성하게 하고, JSON.stringify도 건너뜁니다.

## 23. React Query 활용 전략을 설명해주세요.

> 면접관의 의도: 서버 상태 관리 라이브러리를 단순히 사용하는 수준인지, 옵션의 의미를 이해하고 프로젝트에 맞게 튜닝한 수준인지 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "기본 설정 그대로 사용했습니다" — 프로젝트에 맞는 튜닝을 하지 않았다는 인상
- staleTime과 cacheTime의 차이를 모르는 경우 — React Query 핵심 개념 이해 부족
- SSR에서의 캐시 격리를 언급하지 않는 경우 — Next.js와 함께 쓸 때 가장 중요한 포인트

### 합격 답변 예시

서버 상태와 클라이언트 상태를 명확히 분리하는 것을 원칙으로 합니다. 아트봉봉 스쿨에서는 `useSchoolApi` 커스텀 훅으로 API 호출을 표준화했는데, `staleTime: 1000`으로 1초 내 중복 요청을 방지하고, 재시도는 지수 백오프(`2000 * 2^attempt`, 최대 60초)로 설정했습니다. QueryClient는 브라우저에서 싱글톤으로 관리하고, SSR에서는 매번 새로 생성해 요청 간 데이터 오염을 방지합니다.

### 가산점 포인트

`staleTime`, 재시도 전략, SSR vs 클라이언트 싱글톤 구분 같은 구체적 설정값을 말하면 실제 사용 경험이 드러난다.

### 상세 해설

**QueryClient 설정:**
```typescript
const makeQueryClient = () => new QueryClient({
  defaultOptions: {
    queries: {
      refetchOnWindowFocus: false,
      retry: 3,
      retryDelay: (attemptIndex) => Math.min(2000 * 2 ** attemptIndex, 60000),
    },
  },
});

// SSR에서는 매번 새로 생성, 브라우저에서는 싱글톤
function getQueryClient() {
  if (isServer) return makeQueryClient();
  if (!browserQueryClient) browserQueryClient = makeQueryClient();
  return browserQueryClient;
}
```

**커스텀 훅 패턴** (`useSchoolApi`):
- `apiAndRequestParameter`가 falsy면 쿼리 비활성화 (조건부 fetch)
- `addApiLog(api, request, response)`로 디버그 로깅 자동화
- `queryKey` 자동 생성으로 캐시 키 관리 부담 제거

### 꼬리 질문 대비

#### "staleTime을 1초로 설정한 이유는?"
목록→상세→뒤로가기 시 1초 이내라면 캐시된 데이터를 보여주는 게 더 좋은 UX입니다. 너무 길면 데이터 정합성 문제가 생기므로 1초가 적절한 균형이었습니다.

#### "SSR에서 QueryClient를 매번 만드는 게 성능에 문제 없나요?"
QueryClient 생성 자체는 가벼운 연산이라 문제없습니다. 오히려 재사용하면 사용자 A의 API 응답이 사용자 B에게 노출될 수 있어 보안 사고입니다. Next.js 공식 문서에서도 서버에서는 매번 새로 생성하라고 안내합니다.

#### "addApiLog는 어디에 기록되나요?"
개발 환경에서는 콘솔에, 프로덕션에서는 Sentry breadcrumb으로 기록합니다. 에러 발생 시 직전 API 호출 이력을 Sentry에서 바로 확인할 수 있어 재현이 어려운 버그 디버깅에 유용했습니다.

## 24. 토스페이먼츠 결제 연동 경험을 설명해주세요.

> 면접관의 의도: 외부 SDK 연동 경험과 결제라는 민감한 도메인에서의 예외 처리, 비즈니스 로직 통합 능력을 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "토스페이먼츠 문서 그대로 따라 했습니다" — 커스터마이징이나 예외 처리 경험이 안 보인다
- 0원 결제 케이스를 고려하지 못한 경우 — 실무에서 흔한 엣지 케이스를 놓친 것
- 결제 데이터 유실 가능성을 언급하지 않는 경우 — 결제 도메인에서 데이터 유실은 가장 민감한 이슈

### 합격 답변 예시

Art-bonbon B2C에서 토스페이먼츠 위젯 SDK를 연동해 결제 기능을 구현했습니다. `@tosspayments/payment-widget-sdk`의 `loadPaymentWidget`으로 위젯을 초기화하고, `renderPaymentMethods`로 결제 수단 UI를 렌더링합니다. 0원 주문은 결제를 건너뛰고 바로 성공 처리하는 분기를 두었고, Bigin 이벤트 트래킹도 결제 페이지에 통합해 전환 분석이 가능하도록 했습니다. 학생 선택은 SessionStorage에 저장해 결제 흐름 중 상태가 유지되도록 했습니다.

### 가산점 포인트

0원 결제 분기처리, 분석 도구(Bigin) 통합, 결제 흐름 중 상태 관리 등 실무적 디테일을 언급하면 좋다.

### 상세 해설

**결제 플로우:**
1. 학생 선택 → SessionStorage에 저장
2. 패키지/일정 선택 → 주문 생성 API
3. 결제 페이지 (`/payment/[orderId]`)
   - 0원 → 결제 건너뛰기 (즉시 성공)
   - 유료 → 토스 위젯 렌더링
4. 성공 페이지 → 주문 확인 + SessionStorage 정리

**Bigin 분석 연동:**
- `window._bigin.orderFormPage` 객체에 상품 정보, 결제 수단, 학생 정보(id, name, gender, age) 매핑
- 결제 요소 바인딩으로 자동 이벤트 추적

### 꼬리 질문 대비

#### "왜 SessionStorage를 사용했나요?"
결제 과정은 하나의 세션 안에서 완결되어야 하므로 탭을 닫으면 자동 정리되는 SessionStorage가 적합합니다. localStorage를 쓰면 이전 결제 시도 데이터가 남아 혼란을 줄 수 있고, 서버 저장은 불필요한 API 호출과 정리 로직이 추가됩니다.

#### "결제 실패 시 재시도는 어떻게 처리하나요?"
토스 SDK가 failUrl로 리다이렉트하면 에러 코드를 파싱해 안내합니다. 잔액 부족이면 다른 결제 수단 안내, 네트워크 에러면 재시도 버튼 제공. SessionStorage에 데이터가 보존돼 있어 처음부터 다시 입력할 필요가 없습니다.

#### "ANONYMOUS 대신 실제 고객 키를 사용하면?"
고객 키를 쓰면 토스에서 결제 수단을 기억해 간편 결제가 가능합니다. 저희는 비회원 결제와 첫 결제 진입 장벽을 낮추는 게 우선이라 ANONYMOUS를 사용했지만, 리텐션이 높아지면 고객 키 연동 도입을 계획했습니다.

## 25. dnd-kit을 활용한 드래그앤드롭 구현을 구체적으로 설명해주세요.

> 면접관의 의도: 복잡한 UI 인터랙션 라이브러리를 실무에서 어떻게 설계하고, 엣지케이스를 어떻게 처리했는지 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "dnd-kit 문서 보고 그대로 따라 했습니다"만 말하고 커스터마이징을 설명하지 못하는 경우
- 센서 설정 값의 근거를 설명하지 못하는 경우
- default 인덱스 보정 같은 엣지케이스를 언급하지 못하는 경우

### 합격 답변 예시

Tool-Manager에서 브러시, 색상, 배경지, 오버레이 등 5개 영역에 dnd-kit을 적용했습니다. `DndContext` + `SortableContext`로 정렬 가능한 그리드를 구성하고, `MouseSensor`(distance: 10px)와 `TouchSensor`(delay: 250ms, tolerance: 5px)로 마우스와 터치를 모두 지원합니다. 특히 드래그 중 default 인덱스가 깨지지 않도록 하는 로직이 중요했는데, 예를 들어 기본 배경지를 드래그하면 `state.config.paper.default`를 이동 방향에 따라 +1 또는 -1로 보정합니다.

### 가산점 포인트

default 인덱스 보정 로직은 실제로 겪어봐야 아는 문제다. "드래그앤드롭을 구현했다"에서 끝나지 않고 이런 엣지 케이스를 설명하면 깊이가 보인다.

### 상세 해설

**실제 코드 기반 구현:**

```typescript
// 색상 DnD (ColorDragAndDrop/index.tsx)
const sensors = useSensors(
  useSensor(MouseSensor, { activationConstraint: { distance: 10 } }),
  useSensor(TouchSensor, { activationConstraint: { delay: 250, tolerance: 5 } }),
);

// 드래그 종료 시 Zustand 액션 호출
const handleDragEnd = (event: DragEndEvent) => {
  const oldIndex = colorLists.findIndex(item => item === active.id);
  const newIndex = colorLists.findIndex(item => item === over.id);
  moveColor(oldIndex, newIndex);
};
```

**Default 인덱스 보정** (`paperActionSlice`):
```typescript
movePaper: (fromIndex, toIndex) => {
  set((state) => {
    const defaultIdx = state.config.paper.default;
    // 기본 아이템 자체를 이동
    if (defaultIdx === fromIndex) state.config.paper.default = toIndex;
    // 기본 아이템 위로 이동 → default +1
    else if (defaultIdx < fromIndex && defaultIdx >= toIndex) state.config.paper.default += 1;
    // 기본 아이템 아래로 이동 → default -1
    else if (defaultIdx > fromIndex && defaultIdx <= toIndex) state.config.paper.default -= 1;
  });
}
```

### 꼬리 질문 대비

#### "closestCenter 대신 다른 충돌 감지 알고리즘은?"
dnd-kit은 closestCorners, rectIntersection, pointerWithin도 제공합니다. 저희는 정사각형 그리드여서 중심점 거리 기반인 closestCenter가 가장 직관적이었고, 실제 테스트에서도 의도한 위치에 정확히 드롭되는 경험이 가장 좋았습니다.

#### "DragOverlay를 왜 별도로 사용했나요?"
DragOverlay 없이 기본 동작만 쓰면 드래그 중인 아이템이 원래 위치에서 사라지면서 레이아웃이 흔들립니다. DragOverlay를 쓰면 원래 위치에 플레이스홀더가 유지되고 드래그 요소는 포탈로 최상위에 렌더링되어 z-index 문제도 없습니다.

#### "react-beautiful-dnd 대신 dnd-kit을 선택한 이유?"
react-beautiful-dnd는 메인테이너가 유지보수를 중단한 상태이고 그리드 레이아웃 지원이 약합니다. dnd-kit은 센서 시스템이 플러그인 방식이라 터치/마우스를 독립적으로 튜닝할 수 있고, 접근성(aria)도 기본 내장되어 있습니다.

## 26. MBTI 검사 기능의 end-to-end 구현을 설명해주세요.

> 면접관의 의도: 인터랙티브 콘텐츠 구현 시 사용자 경험 설계와 서버 측 보안/비즈니스 로직 처리 능력을 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- 셔플을 왜 했는지 공정성 관점의 근거를 설명하지 못하는 경우
- 330ms 스피너를 "로딩이 느려서 넣었다"라고 잘못 설명하는 경우
- 서버 측 1회 제한이나 보안 처리를 전혀 언급하지 않는 경우

### 합격 답변 예시

Art-bonbon B2C에서 20문항 이미지 기반 MBTI 검사를 구현했습니다. 문항과 선택지를 매번 셔플해서 공정성을 확보하고, 선택 간 330ms 로딩 스피너로 자연스러운 전환감을 줬습니다. 결과 계산은 E/I, S/N, T/F, J/P 각 축에서 3개 이상 선택된 쪽을 채택하는 방식입니다. 결과 페이지에서 쿠폰 자동 발급까지 한 흐름으로 연결되며, 서버에서는 학부모당 1회 참여 제한과 쿠폰 유효기간(28일)을 체크합니다.

### 가산점 포인트

"MBTI를 만들었다"보다 "셔플링으로 공정성, 로딩으로 전환감, 1회 제한으로 어뷰징 방지" 같은 설계 판단을 설명하면 좋다.

### 상세 해설

**실제 코드 기반:**

1. **결과 계산** (`MBTIQuestion.tsx`):
   ```typescript
   const countMBTI = (selected) => {
     const types = ["E","I","S","N","T","F","J","P"];
     const counts = types.map(t => selected.filter(s => s.MBTIType === t).length);
     return types.reduce((acc, type, i) => counts[i] >= 3 ? acc + type : acc, "");
   };
   ```

2. **쿠폰 종류:**
   - 체험 8,800원 구입 쿠폰 (체험 수업 전용)
   - 25% 할인 쿠폰 (정규/영어/웹툰 수업)
   - 유효기간: 28일

3. **서버 검증** (`MBTIService.ts`):
   - 동일 학부모 중복 참여 체크
   - Zod로 20개 응답 배열 검증
   - MySQL Prisma로 결과 저장

### 꼬리 질문 대비

#### "동점이 나올 수 있지 않나요?"
각 축당 5문항이라 정확히 반반(2.5:2.5)은 불가능합니다. 5문항이므로 항상 한쪽이 최소 3개가 되어 과반이 보장됩니다. 의도적으로 홀수 문항을 설계해 별도 동점 처리 없이 확정적 결과를 산출합니다.

#### "safeJwtVerify는 일반 verify와 뭐가 다른가요?"
일반 verify는 실패 시 예외를 던지지만, safeJwtVerify는 try-catch로 감싸서 null을 반환합니다. 호출하는 쪽에서 if 분기로 처리할 수 있어 코드 흐름이 명확하고, withHandler의 전역 에러 핸들링과 역할이 겹치지 않습니다.

#### "330ms라는 수치는 어떻게 정했나요?"
Nielsen의 인지 연구에서 100ms 이하는 즉각적, 1초 이상은 지루함으로 분류합니다. 그 사이에서 '처리 중'이라는 인지를 줄 수 있는 적정값으로, 팀 내 테스트를 거쳐 결정했습니다.

## 27. Recoil과 Zustand를 각각 다른 프로젝트에서 사용했는데, 어떤 기준으로 선택하나요?

> 면접관의 의도: 상태 관리 라이브러리 선택의 기술적 근거와 프로젝트 맥락에 따른 판단력을 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "Recoil은 Meta가 관리 안 해서 나쁘고 Zustand가 좋습니다"처럼 인기도로만 판단하는 경우
- 슬라이스 패턴이 뭔지 설명하지 못하고 "Zustand 써봤습니다"로 끝나는 경우
- 기존 Recoil을 유지한 이유를 설명하지 못해 일관성 없는 선택으로 보이는 경우

### 합격 답변 예시

bb-user-web에서는 Recoil을 사용했고, 아트봉봉 스쿨과 Tool-Manager에서는 Zustand를 사용했습니다. Recoil은 bb-user-web이 기존에 사용하던 것을 유지한 것이고, 새 프로젝트에서는 Zustand를 선택했습니다. Zustand가 더 적합했던 이유는 React 외부에서도 상태에 접근할 수 있고(`getState()`), 슬라이스 패턴으로 대규모 상태를 모듈화하기 쉬우며, 보일러플레이트가 최소라서 팀 학습 비용이 낮기 때문입니다. Recoil은 atom 기반으로 세분화된 구독에 강점이 있지만, 복잡한 상태 조합이 필요한 Tool-Manager나 심리검사 스토어에는 Zustand의 단일 스토어 + 슬라이스가 더 적합했습니다.

### 가산점 포인트

"둘 다 써봤기 때문에 비교할 수 있다"는 것 자체가 강점이다. 프로젝트 특성에 맞는 선택 기준을 제시하면 좋다.

### 상세 해설

**bb-user-web Recoil atoms:**
```typescript
const loadingState = atom({ key: "loadingState", default: false });
const userInfoState = atom<IUserInfo>({ key: "userInfo", default: { profileUrl: "" } });
const globalDialogState = atom<IGlobalDialogState>({ key: "globalDialogState", default: { show: false } });
```

**Zustand vs Recoil 비교:**
| 기준 | Zustand | Recoil |
|------|---------|--------|
| 상태 단위 | 스토어 (슬라이스) | Atom |
| React 외부 접근 | `getState()` 가능 | 불가 |
| 미들웨어 | Immer, persist, devtools | 없음 |
| 보일러플레이트 | 최소 | 중간 |
| 복잡한 상태 조합 | 슬라이스로 자연스럽게 | selector로 파생 |

### 꼬리 질문 대비

#### "Zustand의 단점은 뭐라고 생각하시나요?"
슬라이스가 많아지면 StateCreator 제네릭이 미들웨어마다 중첩되어 타입 에러 디버깅이 까다롭습니다. 또한 Recoil의 selector처럼 자동 의존성 추적이 없어서 파생 상태는 직접 useMemo를 작성해야 합니다.

#### "Jotai는 검토하지 않으셨나요?"
검토했습니다. Jotai는 Recoil과 비슷한 atomic 모델이면서 번들이 더 작습니다. 다만 저희는 도메인별 슬라이스 묶음과 React 외부 접근이 핵심이라 Zustand가 더 적합했습니다. Jotai는 폼 같은 세밀한 리렌더링 최적화에 더 어울린다고 판단했습니다.

#### "Recoil을 유지한 건 어떤 판단이었나요?"
bb-user-web의 Recoil은 atom 3개로 간단한 전역 상태만 관리합니다. 작동하는 코드를 굳이 바꿀 이유가 없었고, QA 비용을 고려하면 마이그레이션 ROI가 낮았습니다. 기술 선택은 항상 팀 컨텍스트 안에서 판단해야 합니다.

## 28. 인증/세션 관리는 어떻게 구현했나요?

> 면접관의 의도: 프로젝트 요구사항에 따라 인증 전략을 다르게 설계할 수 있는 유연성과 보안 이해도를 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- JWT와 세션 기반 인증의 차이를 설명하지 못하고 "다 비슷합니다"라고 답하는 경우
- bcrypt와 AES의 용도 차이를 모르고 "둘 다 암호화입니다"로 뭉뚱그리는 경우
- 401 인터셉터에서 refresh 실패 시 처리를 설명하지 못하는 경우

### 합격 답변 예시

프로젝트마다 요구에 맞게 다르게 구현했습니다. Ecole-Admin에서는 `jose` 라이브러리로 JWT(HS256)를 직접 발급하고, Next.js 미들웨어에서 access token(1시간) 만료 시 refresh token(1일)으로 자동 갱신하는 구조를 만들었습니다. 아트봉봉 스쿨에서는 `iron-session`으로 서버 사이드 세션을 관리하고, Server Action에서 `getSessionTokens()`로 토큰을 조회합니다. bb-user-web에서는 Axios 응답 인터셉터에서 401 감지 시 자동으로 `/auth/token/refresh`를 호출해 토큰을 갱신하고 실패한 요청을 재시도합니다.

### 가산점 포인트

프로젝트별로 다른 인증 패턴을 설명하면 "상황에 맞는 기술 선택"을 할 줄 안다는 것을 보여준다.

### 상세 해설

**Ecole-Admin 미들웨어 인증:**
```typescript
// middleware.ts
try {
  await jwtVerify(accessToken, secretKey);
} catch (error) {
  if (error instanceof JWTExpired && refreshToken) {
    return refresh(req); // 자동 갱신
  }
  return NextResponse.redirect("/signin");
}
```

**아트봉봉 스쿨 세션 관리:**
- `useClientUserSession` 훅: `useTransition`으로 토큰 갱신 중 UI 블록 방지
- Server Action 기반 인증: `getSessionTokens()` → 쿠키에서 토큰 조회

**bb-user-web 자동 갱신:**
- Axios 인터셉터에서 401 → `refresh()` → 쿠키 갱신 → 원래 요청 재시도
- Sentry에 비-401 에러 자동 리포팅

### 꼬리 질문 대비

#### "bcrypt와 AES의 차이가 뭔가요?"
bcrypt는 단방향 해시로 비밀번호처럼 원본 복원이 불필요한 데이터에 사용합니다. AES는 양방향 암호화로 이메일/전화번호처럼 복호화해서 읽어야 하는 데이터에 사용합니다. Ecole-Admin에서는 비밀번호는 bcrypt, 개인정보는 AES로 구분 적용했습니다.

#### "토큰을 쿠키에 저장하면 CSRF 위험이 있지 않나요?"
httpOnly, secure, sameSite=lax 옵션을 모두 적용했습니다. httpOnly로 JavaScript 접근 차단, secure로 HTTPS에서만 전송, sameSite=lax로 외부 사이트 POST에 쿠키 미포함. bb-user-web에서 js-cookie를 쓴 건 기존 백엔드가 Authorization 헤더를 요구하는 구조라 클라이언트에서 토큰을 읽어야 했기 때문입니다.

#### "jose를 선택한 이유는?"
Edge Runtime 호환성 때문입니다. jsonwebtoken은 Node.js crypto에 의존해 Edge에서 동작하지 않지만, jose는 Web Crypto API를 사용해서 Next.js middleware(Edge Runtime)에서도 토큰 검증이 가능합니다.

## 29. Sentry 등 에러 모니터링은 어떻게 활용하나요?

> 면접관의 의도: 에러 추적 체계를 단순 설치가 아닌 아키텍처 레벨에서 설계할 수 있는지, 실무에서 모니터링을 어떻게 활용하는지 확인한다.

### 답변 타이밍: 2분

### 이렇게 말하면 감점

- "Sentry 설치하고 DSN 넣으면 끝입니다" — 커스터마이징 경험이 없어 보인다
- breadcrumb이 뭔지 모르고 에러 로그만 보내는 수준
- 터널링의 필요성을 설명하지 못하는 경우

### 합격 답변 예시

아트봉봉 스쿨과 bb-user-web 모두 `@sentry/nextjs`를 사용합니다. API Client의 응답 인터셉터에서 에러 발생 시 자동으로 `Sentry.captureException`과 `addBreadcrumb`를 호출해 요청 경로, 에러 코드, 페이로드를 함께 추적합니다. 커스텀 로거 패키지(`@school/logger`)도 Sentry를 출력 채널로 연동해서, 로깅과 에러 추적을 통합적으로 관리합니다.

### 가산점 포인트

"Sentry를 설치했다"가 아니라 "API 인터셉터에서 자동 추적", "커스텀 로거와 연동"처럼 시스템적으로 통합했다는 점을 강조하면 좋다.

### 상세 해설

**커스텀 로거** (`@school/logger`):
```typescript
// 플러그인 기반 아키텍처
const logger = createLogger(
  ["info", "warn", "error"],
  [consoleChannel, sentryChannel] // 출력 채널을 플러그인으로 주입
);
```

**API Client Sentry 통합:**
- 요청마다 breadcrumb 추가 (경로, 메서드, 파라미터)
- 비정상 응답(status >= 250)을 `ApiError`로 변환 후 capture
- 순환 참조 처리, 브라우저/Node.js 환경 분기

### 꼬리 질문 대비

#### "withHandler는 구체적으로 어떻게 구현했나요?"
고차 함수 패턴입니다. API 핸들러를 인자로 받아 try-catch로 감싼 새 함수를 반환합니다. catch에서 captureException을 호출하고 500 응답을 일관되게 반환합니다. 각 핸들러마다 try-catch를 반복하지 않아도 되고, 에러 보고 형식이 통일됩니다.

#### "Sentry 성능 모니터링도 활용했나요?"
tracesSampleRate을 클라이언트 0.1, 서버 0.2로 설정해 성능 트레이싱을 수집했습니다. Performance 탭에서 p95 응답 시간을 모니터링하고, 특정 엔드포인트가 급격히 느려지면 알림을 받도록 설정했습니다.

#### "로거를 패키지로 분리한 이유는?"
모노레포 내 여러 프로젝트에서 동일한 로깅 체계를 쓰기 위해서입니다. 나중에 DataDog이나 LogRocket 채널을 추가해도 소비하는 쪽 코드 변경이 필요 없습니다. OCP(개방-폐쇄 원칙)를 로깅에 적용한 셈입니다.

## 30. 프로젝트 전체를 통틀어 가장 기술적으로 도전적이었던 경험은 무엇인가요?

> 면접관의 의도: 기술적 난이도가 높은 프로젝트에서 문제를 어떻게 해결하고, 그 과정에서 무엇을 배웠는지 전체적인 성장 스토리를 듣고자 한다.

### 답변 타이밍: 3분

### 이렇게 말하면 감점

- 기술 나열만 하고 왜 도전적이었는지, 어떤 문제를 어떻게 해결했는지 서사가 없는 경우
- 협업 경험을 전혀 언급하지 않고 혼자 다 한 것처럼 말하는 경우
- 교훈이 "열심히 하면 됩니다" 수준으로 추상적인 경우

### 합격 답변 예시

아트봉봉 스쿨의 색채 심리검사(CCT) 개발이 가장 도전적이었습니다. 단순 기능 구현이 아니라 여러 기술적 과제가 동시에 있었기 때문입니다. 첫째, 7개 심리검사를 Zustand 멀티슬라이스로 통합하면서도 각 검사의 독립성을 유지해야 했습니다. 둘째, 불안정한 학교 네트워크에서 검사 데이터가 유실되지 않도록 IndexedDB persistence를 설계해야 했습니다. 셋째, 학생의 몰입감을 위해 3D 카드 플립, 타이프라이터 스토리, 레이더 차트 등 다양한 인터랙션을 Framer Motion + Chart.js로 구현해야 했습니다. 넷째, 기획·디자인팀과 긴밀히 소통하며 "성취감을 주는 검사 경험"이라는 추상적 요구를 기술적으로 구체화해야 했습니다.

### 가산점 포인트

여러 기술적 과제가 "동시에" 있었다는 점, 그리고 기술뿐 아니라 기획/디자인과의 협업까지 포함된다는 점이 좋다. 면접관이 원하는 답은 "어려웠지만 해결했다"는 스토리다.

### 상세 해설

**왜 도전적이었는가:**

1. **상태 설계의 복잡성**: 7개 검사 × 각각의 단계/응답 데이터 × IndexedDB 동기화
2. **환경 제약**: 학교 Wi-Fi 불안정 + iframe 로딩 지연 + 토큰 만료 가능성
3. **UX 요구**: "검사"가 아니라 "경험"이 되어야 한다는 기획 방향
4. **기술 스택 조합**: Zustand + Immer + idb-keyval + Framer Motion + Chart.js + Typewriter

**해결 과정:**
- 슬라이스 패턴으로 검사별 독립성 확보 + `getData(scenario)` 통일 인터페이스
- `idbSet`/`idbGet`으로 상태 변경마다 자동 영속화
- 4계층 네트워크 감지로 "원인 모르는 멈춤" 방지
- Framer Motion variant 기반 공통 애니메이션 컴포넌트로 재사용 기반 마련

### 꼬리 질문 대비

#### "7개 슬라이스 간 의존성은 어떻게 관리하셨나요?"
슬라이스 간 직접 import는 금지하고 `get()`으로 전체 스토어에 접근하는 방식을 사용했습니다. ResultSlice의 calculateResult에서 ResponseSlice의 answers가 필요하면 `get().answers`로 접근합니다. 슬라이스 파일은 독립적으로 유지되면서 런타임에는 같은 스토어를 공유해 순환 의존성이 원천 차단됩니다.

#### "기획/디자인 협업에서 기술적 제약으로 조율한 사례가 있나요?"
기획에서 시나리오 전환 시 3D 카메라 무빙을 원했는데, Three.js를 도입하면 번들 200KB+ 증가에 저사양 기기 프레임 드롭이 심했습니다. Framer Motion의 perspective + rotateY를 조합한 의사 3D 효과를 제안해 프로토타입으로 비교 시연했고, 90%의 시각적 효과를 10%의 비용으로 달성하면서 기획 측도 만족했습니다.

#### "이 경험에서 배운 가장 큰 교훈은?"
프론트엔드 개발자의 역할이 화면을 만드는 게 아니라 제품 생명주기 전체에 책임을 갖는 것이라는 점입니다. 기획 의도를 이해해야 더 나은 기술적 제안을 할 수 있고, 사용 환경을 고려해야 진짜 쓸 수 있는 제품이 됩니다.

이 경험이 준 교훈: "단일 기능이 아닌 제품 생명주기 전체를 책임지는 경험을 통해 시야가 확장된다."
