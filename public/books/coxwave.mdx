> Claude Artifacts 스타일 2-패널 채팅 UI 과제 (React 19 + Vite + TypeScript + SSE) 기반 기술 면접 질문입니다.

## Part 1. 프로젝트 개요 및 설계 판단

### Q1. 이 과제의 전체 아키텍처를 설명해주세요. 왜 이런 구조를 선택했나요?

#### 모범 답변

이 프로젝트는 Claude Artifacts 스타일의 2-패널 채팅 UI로, 왼쪽 채팅 + 오른쪽 코드/마크다운 프리뷰 구조입니다. React 19 + Vite + TypeScript 기반이며 SSE로 AI 응답을 실시간 스트리밍합니다.

설계의 핵심 축은 세 가지입니다:

**1. 단일 페이지 스코프의 상태 관리:** `chatSessionReducer`에서 11개 상태를 18개 액션 타입으로 관리합니다. `detail`, `messages`, `isStreaming`, `previewSections`, `chatWidthPercent` 등이 모두 하나의 reducer에 통합되어 있고, 이 상태들은 `ChatSessionPage` 한 페이지에만 귀속됩니다.

```typescript
export type ChatSessionState = {
  detail: ChatSessionDetail | null;
  messages: Message[];
  errorMessage: string;
  draft: string;
  isLoading: boolean;
  isSubmitting: boolean;
  isStreaming: boolean;
  title: string;
  previewSections: PreviewSection[];
  answerStyle: AnswerStyle;
  chatWidthPercent: number;
};
```

**2. 점진적 로딩:** Vite의 `manualChunks`로 `vendor-react`(React 코어)와 `vendor-markdown`(react-markdown + shiki ~200KB)을 분리했습니다. 홈 페이지에서 마크다운 렌더링이 불필요하므로 ChatSessionPage 진입 시에만 로드합니다.

**3. 방어적 스트리밍:** 서버가 의도적으로 ~10% 확률로 corrupt JSON을 보내는 환경에서도 `parseSseEventData`가 개별 이벤트 파싱 실패를 null로 처리하여 스트림 전체가 끊기지 않습니다.

이 구조를 선택한 이유는, 과제 핵심이 "실시간 스트리밍 + 에러 복원력 + 반응형 UI"였기 때문입니다. 글로벌 상태 라이브러리 없이 페이지 단위 reducer로 충분했고, SSE는 서버→클라이언트 단방향이므로 WebSocket보다 단순하면서도 요구사항에 정확히 맞았습니다.

#### 해설

면접관이 보는 것은 "기술 나열"이 아니라 **"왜 이 조합이 이 문제에 적합한가"**입니다. 상태 구조의 구체적 필드명, 번들 분리의 수치적 근거, 에러 처리의 실제 함수명까지 말할 수 있으면 "직접 만든 사람"이라는 확신을 줍니다.

### Q2. 상태 관리를 Redux나 Zustand 대신 useReducer로 한 이유는 무엇인가요?

#### 모범 답변

이 앱의 상태 11개(detail, messages, errorMessage, draft, isLoading, isSubmitting, isStreaming, title, previewSections, answerStyle, chatWidthPercent)는 전부 `ChatSessionPage` 한 페이지에 귀속됩니다. 다른 페이지(Home, History)와 공유할 상태가 없습니다.

useReducer를 선택한 구체적 이유 세 가지입니다:

**첫째, 액션 기반 명시적 상태 전이.** SSE 스트리밍 중 `TEXT_DELTA`, `ARTIFACT_DELTA`, `STREAM_DONE`, `STREAM_ERROR` 같은 이벤트가 비동기로 도착합니다. 각 이벤트가 상태에 미치는 영향을 reducer의 case문으로 명시하면 상태 전이가 예측 가능하고 디버깅이 쉽습니다. 예를 들어 `ARTIFACT_DELTA`는 기존 섹션이 있으면 content를 append하고, 없으면 새 섹션을 생성합니다:

```typescript
case "ARTIFACT_DELTA": {
  const existingSection = state.previewSections.find(
    (s) => s.id === action.streamPreviewSectionId,
  );
  if (!existingSection) {
    return {
      ...state,
      previewSections: [...state.previewSections, {
        id: action.streamPreviewSectionId,
        content: action.content,
        language: action.language,
        isStreaming: true,
      }],
    };
  }
  return {
    ...state,
    previewSections: state.previewSections.map((s) =>
      s.id === action.streamPreviewSectionId
        ? { ...s, content: `${s.content}${action.content}`, language: action.language ?? s.language, isStreaming: true }
        : s,
    ),
  };
}
```

**둘째, 외부 의존성 제거.** Redux는 store, action creator, middleware가 필요하고, Zustand도 store 설정이 필요합니다. 이 과제처럼 페이지 스코프 상태만 관리할 때는 React 내장 훅으로 충분하고, 번들 크기도 줄어듭니다.

**셋째, 순수 함수이므로 테스트가 간단.** `chatSessionReducer(state, action)` 형태로 입출력만 검증하면 됩니다.

트레이드오프는 페이지 간 상태 공유가 필요할 때입니다. 사이드바 세션 목록 갱신은 `session-list.ts`의 PubSub 이벤트(`emitSessionsChanged` / `subscribeSessionsChanged`)로 해결했습니다. window의 CustomEvent를 활용한 6줄짜리 가벼운 이벤트 시스템입니다.

#### 해설

"왜 안 썼는지"를 설명할 때 반드시 **트레이드오프와 그 해결책**을 함께 말합니다. PubSub까지 언급하면 "한계를 인지하고 보완한 사람"이라는 인상을 줍니다.

### Q3. 2-패널 레이아웃의 반응형 전략을 설명해주세요. 브레이크포인트는 어떻게 결정했나요?

#### 모범 답변

데스크톱(783px 이상)은 좌우 분할, 모바일(783px 미만)은 탭 기반 전환입니다.

783px이라는 비표준 값을 선택한 이유는 iPad 가로(1024px)와 세로(768px) 사이의 전환점입니다. 768px에서 2-패널을 유지하면 각 패널이 너무 좁고, 1024px부터 분할하면 태블릿 가로에서 불필요하게 탭 전환을 강제합니다.

구현에는 `useBreakpoint` 커스텀 훅을 만들었습니다:

```typescript
export function useBreakpoint(key: BreakpointKey) {
  const [matches, setMatches] = useState(false);
  useEffect(() => {
    const mql = window.matchMedia(BREAKPOINTS[key]);
    setMatches(mql.matches);
    const handler = (e: MediaQueryListEvent) => setMatches(e.matches);
    mql.addEventListener("change", handler);
    return () => mql.removeEventListener("change", handler);
  }, [key]);
  return matches;
}
```

`resize` 이벤트 대신 `matchMedia`의 `change` 이벤트를 사용합니다. 패널 드래그 리사이즈 시 resize 이벤트가 초당 60회 이상 발생하지만, matchMedia는 브레이크포인트 임계값을 넘을 때만 1-2회 트리거됩니다.

모바일 탭 전환에서는 프리뷰에 새 콘텐츠가 있을 때 탭 옆에 점(dot) 인디케이터를 표시합니다:

```typescript
{hasPreview && mobileTab !== "preview" ? (
  <span className="ml-1.5 inline-block size-1.5 rounded-full bg-zinc-900" />
) : null}
```

#### 해설

"표준 값을 썼다"보다 **"왜 이 값이 이 UI에 적합한지"**를 설명하는 것이 강합니다. matchMedia vs resize 성능 차이, 모바일 UX 배려(dot indicator)까지 말하면 깊이가 있어 보입니다.

## Part 2. SSE 스트리밍과 실시간 처리

### Q4. SSE(Server-Sent Events)를 WebSocket 대신 선택한 이유는 무엇인가요?

#### 모범 답변

이 채팅의 실시간 통신 패턴은 명확합니다. 사용자 메시지는 `POST /api/sessions/:id/messages`로 보내고, AI 응답만 스트리밍으로 받습니다. 완전한 단방향입니다.

SSE를 선택한 구체적 이유 네 가지입니다:

**1. HTTP 호환:** 별도 프로토콜 업그레이드 없이 표준 HTTP 위에서 동작합니다. 프록시, CDN, 로드밸런서를 그대로 통과합니다.

**2. 자동 재연결:** 브라우저 EventSource가 연결 끊김 시 자동 재시도합니다. WebSocket은 재연결 로직을 직접 구현해야 합니다.

**3. 디버깅 용이:** DevTools EventStream 탭에서 Named Events(text_delta, artifact_delta, done)를 바로 확인할 수 있습니다.

**4. 구현 간결:** 클라이언트는 `new EventSource(url)` 한 줄이면 됩니다:

```typescript
export function createChatStream(sessionId: string, style?: string): EventSource {
  const params = style ? `?style=${encodeURIComponent(style)}` : "";
  return new EventSource(`${BASE}/sessions/${sessionId}/stream${params}`);
}
```

서버(Hono.js)에서도 `SSEStreamingApi`의 `writeSSE`로 이벤트를 보내기만 하면 됩니다.

WebSocket의 양방향 채널은 이 과제에서 사용할 일이 없으므로 오버스펙입니다. 만약 타이핑 인디케이터나 실시간 협업 기능이 있었다면 WebSocket을 고려했을 것입니다.

#### 해설

핵심 판단 기준은 "양방향이 필요한가?"입니다. 구체적으로 **어떤 경우에 WebSocket을 선택했을지**까지 말하면 기술적 판단의 범위가 넓다는 인상을 줍니다.

### Q5. SSE 이벤트 프로토콜을 설명해주세요. text_delta와 artifact_delta를 분리한 이유는?

#### 모범 답변

서버가 보내는 SSE Named Events는 세 가지입니다:

```
text_delta      → { content: string }                        // 채팅 텍스트
artifact_delta  → { content: string, language?: string }     // 코드/문서 프리뷰
done            → { title?: string }                         // 스트림 완료
```

분리한 이유는 **관심사 분리**입니다. 서버의 `stream.ts`에서 OpenRouter 응답을 줄 단위로 파싱하며, 코드 펜스(```` ``` ````) 안팎을 구분합니다:

```typescript
async function processLine(line: string): Promise<void> {
  const fenceMatch = line.match(/^```(\w*)\s*$/);
  if (fenceMatch && !inCodeBlock) {
    inCodeBlock = true;
    codeBlockLang = fenceMatch[1] || "";
    return;
  }
  if (inCodeBlock && line.trimEnd() === "```") {
    inCodeBlock = false;
    return;
  }
  const content = line + "\n";
  if (inCodeBlock) {
    await writeSSE(stream, { type: "artifact_delta", content, ...(codeBlockLang && { language: codeBlockLang }) });
  } else if (content.trim()) {
    await writeSSE(stream, { type: "text_delta", content });
  }
}
```

서버가 파싱 책임을 지므로, 클라이언트(`useStreamChat`)는 이벤트 타입만 보고 적절한 패널에 append합니다. `text_delta`는 messages 배열에, `artifact_delta`는 previewSections 배열에 dispatch할 뿐입니다.

만약 하나의 이벤트로 합쳤다면, 클라이언트에서 매 이벤트마다 코드 펜스 파싱을 해야 하고, 스트리밍 중간에 불완전한 펜스를 처리하는 복잡한 상태 머신이 필요했을 것입니다.

#### 해설

서버의 실제 파싱 코드까지 보여주면 "프론트-백 모두 이해한 사람"이라는 인상을 줍니다. 분리하지 않았을 때의 문제점(클라이언트 펜스 파싱 복잡도)을 구체적으로 말하세요.

### Q6. 서버가 의도적으로 보내는 malformed JSON을 어떻게 처리했나요?

#### 모범 답변

과제의 Hard 테스트 중 하나가 "서버가 ~10% 확률로 corrupt JSON을 보내는 상황에서 스트림이 정상 동작해야 한다"입니다.

`libs/api.ts`의 `parseSseEventData` 함수가 핵심입니다:

```typescript
function isRecord(value: unknown): value is Record<string, unknown> {
  return typeof value === "object" && value !== null;
}

export function parseSseEventData<T extends Record<string, unknown>>(
  data: string,
): T | null {
  try {
    const parsed: unknown = JSON.parse(data);
    return isRecord(parsed) ? (parsed as T) : null;
  } catch {
    return null;
  }
}
```

두 가지 방어를 합니다:
1. `JSON.parse` 실패 → catch에서 null 반환 (corrupt JSON)
2. 파싱 성공이지만 객체가 아닌 경우(예: `"string"`, `42`) → `isRecord` 체크로 null 반환

호출하는 쪽(`useStreamChat`)에서는 optional chaining으로 null을 건너뜁니다:

```typescript
stream.addEventListener("text_delta", (event) => {
  const payload = parseSseEventData<TextDeltaData>(event.data);
  if (!payload?.content) return; // 잘못된 이벤트는 무시
  dispatch({ type: "TEXT_DELTA", content: payload.content, streamMessageId });
});
```

핵심은 **개별 이벤트의 실패가 전체 스트림을 중단시키지 않는 것**입니다. 10개 이벤트 중 1개가 깨져도 나머지 9개는 정상 처리됩니다. throw하지 않고 null로 처리하여 EventSource의 이벤트 루프가 계속 돌아갑니다.

#### 해설

"graceful degradation" 패턴입니다. 면접에서는 "왜 에러를 무시하는 것이 여기서는 올바른가"를 설명해야 합니다. 하나의 이벤트 실패로 전체 대화를 날리는 것이 더 나쁜 UX이기 때문입니다. `isRecord` 타입 가드까지 설명하면 TypeScript 활용 수준도 보여줍니다.

### Q7. useStreamChat 훅의 내부 구조를 설명해주세요. SSE 라이프사이클은 어떻게 관리하나요?

#### 모범 답변

`useStreamChat`은 SSE 스트리밍의 전체 라이프사이클을 캡슐화하는 커스텀 훅입니다. 세 개의 ref로 상태를 추적합니다:

```typescript
const streamRef = useRef<EventSource | null>(null);         // 현재 EventSource
const streamMessageIdRef = useRef<string | null>(null);      // 스트리밍 중인 assistant 메시지 ID
const streamPreviewSectionIdRef = useRef<string | null>(null); // 스트리밍 중인 프리뷰 섹션 ID
```

ref를 사용하는 이유는 이벤트 리스너 내부에서 최신 값을 참조해야 하기 때문입니다. useState를 쓰면 클로저 문제로 stale value를 읽게 됩니다.

라이프사이클은 5단계입니다:

**1단계 - 시작:** `startStream(answerStyle)` 호출 → 기존 스트림 정리 → `createChatStream(sessionId, answerStyle)`로 EventSource 생성 → `STREAM_START` dispatch

**2단계 - 수신:** Named event listeners 3개 등록 (text_delta, artifact_delta, done) + error listener

**3단계 - 처리:** 각 이벤트를 `parseSseEventData`로 파싱 → null이면 skip → 유효하면 reducer에 dispatch

**4단계 - 완료:** `done` 이벤트 수신 시:
```typescript
stream.addEventListener("done", (event) => {
  const payload = parseSseEventData<DoneData>(event.data);
  dispatch({
    type: "STREAM_DONE",
    title: payload?.title && hasUsableSessionTitle(payload.title) ? payload.title : undefined,
    streamPreviewSectionId: streamPreviewSectionIdRef.current,
    streamMessageId: streamMessageIdRef.current,
  });
  closeStream();
  emitSessionsChanged();  // 사이드바 갱신 이벤트
  syncSessionAfterStream(); // 서버와 상태 동기화
});
```

**5단계 - 에러:** 네트워크 에러 시 `STREAM_ERROR` dispatch + 스트림 정리

특히 `done` 이후 `syncSessionAfterStream`으로 서버 상태와 동기화하는 부분이 중요합니다. 스트리밍 중에는 클라이언트가 임시 ID(`streaming-assistant-${Date.now()}`)를 사용하는데, 동기화를 통해 서버가 부여한 실제 ID로 교체합니다.

#### 해설

이 훅을 분리한 이유는 "재사용"이 아니라 **관심사 분리**입니다. ChatSessionPage(~200줄)가 UI 렌더링에만 집중하고, 스트리밍 로직(~130줄)은 훅이 캡슐화합니다. ref를 쓴 이유(클로저 문제)까지 설명하면 React 깊이를 보여줍니다.

## Part 3. React 19 기능 활용

### Q8. useOptimistic를 어떻게 활용했나요? 일반적인 optimistic update와 뭐가 다른가요?

#### 모범 답변

`ChatSessionPage`에서 메시지 전송 시 서버 응답을 기다리지 않고 즉시 화면에 표시하기 위해 `useOptimistic`를 사용합니다:

```typescript
const [optimisticMessages, addOptimisticMessage] = useOptimistic<
  Message[],
  Message
>(messages, (currentMessages, newMessage) => [
  ...currentMessages,
  newMessage,
]);
```

메시지 전송 흐름은:

```typescript
const handleSubmit = useCallback(async () => {
  const optimisticMessage: Message = {
    id: `optimistic-${Date.now()}`,
    role: "user",
    content: trimmed,
    createdAt: new Date().toISOString(),
  };

  dispatch({ type: "SUBMIT_START" });
  startTransition(() => {
    addOptimisticMessage(optimisticMessage);  // 즉시 화면에 표시
  });
  dispatch({ type: "SET_DRAFT", draft: "" });

  const { id: realId } = await sendMessage(sessionId, trimmed);
  dispatch({ type: "SUBMIT_SUCCESS", message: { id: realId, ... } });
  // 서버 응답 후 실제 ID로 교체
}, [...]);
```

기존 방식(setState로 임시 추가 → 서버 응답 후 교체 → 에러 시 수동 롤백)과의 차이점:

1. **자동 상태 관리:** React가 낙관적 상태와 실제 상태(reducer의 messages)를 분리 추적합니다. 트랜지션이 완료되면 낙관적 값은 자동으로 실제 값으로 대체됩니다.
2. **자동 롤백:** 에러 시 React가 이전 상태로 복원합니다. 별도 try-catch 롤백 로직이 불필요합니다.
3. **startTransition 통합:** 낙관적 업데이트를 비긴급 전환으로 마킹하여, 다른 긴급 업데이트(타이핑 등)를 블로킹하지 않습니다.

#### 해설

"그냥 setState로 해도 되지 않나?"라는 후속 질문에 대비하세요. 핵심 차별점은 **자동 롤백 + 트랜지션 통합**입니다. 실제 코드에서 `startTransition`으로 감싸는 부분까지 보여주면 React 19의 concurrency 모델을 이해하고 있다는 것을 증명합니다.

### Q9. useTransition을 어디서, 왜 사용했나요? flushSync와의 차이는?

#### 모범 답변

`ChatSessionPage`의 `handleSubmit`에서 `startTransition`을 사용합니다:

```typescript
import { startTransition } from "react";

startTransition(() => {
  addOptimisticMessage(optimisticMessage);
});
```

SSE 스트리밍 중 매 이벤트마다 상태를 업데이트하면 메인 스레드가 블로킹될 수 있습니다. `startTransition`으로 감싸면:

1. **배치 처리:** React가 여러 업데이트를 모아서 한 번에 렌더링합니다
2. **우선순위 분리:** 사용자 입력(타이핑, 스크롤)이 스트리밍 업데이트보다 높은 우선순위를 가집니다
3. **인터럽트 가능:** 렌더링 도중 더 긴급한 업데이트가 오면 현재 렌더링을 중단하고 긴급 업데이트를 먼저 처리합니다

`flushSync`와의 차이: flushSync는 **동기적으로 즉시** DOM을 업데이트합니다. 스트리밍 중 초당 수십 번 flushSync를 호출하면 매번 동기 렌더가 발생하여 UI가 버벅거립니다. 실제로 개발 과정에서 AI(Claude Code)가 flushSync를 유지하려 했지만, 직접 브라우저에서 테스트해보니 useTransition이 체감 성능이 확실히 좋아서 변경을 강행했습니다.

flushSync는 "이 업데이트 결과가 다음 줄에서 반드시 DOM에 반영되어야 할 때"(예: focus 이동, 스크롤 위치 계산)에만 적합합니다. 스트리밍처럼 연속적이고 비긴급한 업데이트에는 startTransition이 맞습니다.

#### 해설

Concurrent React의 핵심 개념을 검증하는 질문입니다. **동기(flushSync) vs 비동기 배치(startTransition)**의 차이를 체감 기반으로 설명할 수 있으면 강합니다. "AI를 오버룰한 경험"은 독립적 기술 판단력을 보여주는 좋은 포인트입니다.

### Q10. React.lazy와 Suspense를 어떻게 활용했나요?

#### 모범 답변

3개 페이지를 `React.lazy`로 분리하고, Vite의 `manualChunks`와 결합했습니다:

```typescript
// vite.config.ts
manualChunks: {
  "vendor-react": ["react", "react-dom", "react-router"],
  "vendor-markdown": ["react-markdown", "remark-gfm", "rehype-pretty-code", "shiki"],
}
```

특히 `vendor-markdown` 청크(~200KB)가 핵심입니다. react-markdown + shiki(구문 강조)는 ChatSessionPage의 Preview 컴포넌트에서만 사용됩니다. 홈 페이지에서는 전혀 불필요하므로, React.lazy로 ChatSessionPage를 동적 import하면 이 청크가 자연스럽게 지연 로드됩니다.

결과: 홈 페이지 초기 로드 ~500KB → ~280KB로 약 44% 감소.

Suspense fallback으로 로딩 스피너를 보여주면서 사용자 체감 로딩 시간도 개선했습니다. 테스트 환경에서는 React.lazy의 비동기 로딩을 처리하기 위해 `renderUI`를 async로 전환하고 waitFor로 fallback 소멸을 대기합니다:

```typescript
export async function renderUI(initialEntries: string[] = ["/"]) {
  const result = render(
    <MemoryRouter initialEntries={initialEntries}>
      <Suspense fallback={<div data-testid="lazy-fallback" />}>
        <Routes />
      </Suspense>
    </MemoryRouter>,
  );
  await waitFor(() => {
    expect(screen.queryByTestId("lazy-fallback")).not.toBeInTheDocument();
  });
  return result;
}
```

#### 해설

코드 스플리팅의 핵심은 **"왜 이것을 분리하면 효과가 큰가"**입니다. 200KB라는 수치, 홈에서 불필요하다는 판단, 테스트 환경 대응까지 한 흐름으로 설명하세요.

### Q11. @tanstack/react-virtual로 메시지 리스트를 가상화한 이유와 구현 방식을 설명해주세요.

#### 모범 답변

채팅 메시지가 수백 개 이상 쌓이면 DOM 노드 수가 비례 증가하여 스크롤 성능과 메모리에 문제가 생깁니다. `@tanstack/react-virtual`의 `useVirtualizer`를 사용해 뷰포트에 보이는 메시지만 렌더링합니다.

구현에서 주의한 세 가지:

**1. 동적 높이:** 메시지마다 길이가 달라 높이가 제각각입니다. `estimateSize`로 초기 추정치를 제공하고, 실제 렌더링 후 `measureElement`로 정확한 높이를 측정하여 가상화 계산을 보정합니다.

**2. 스크롤 위치 관리:** 새 메시지 도착 시 자동으로 하단 스크롤하되, 사용자가 위로 스크롤한 상태에서는 현재 위치를 유지합니다.

**3. 테스트 환경 대응:** jsdom에서는 `getBoundingClientRect`가 항상 0을 반환하여 가상화가 동작하지 않습니다. `setup.ts`에서 모든 아이템을 렌더링하는 mock을 적용했습니다:

```typescript
vi.mock("@tanstack/react-virtual", () => ({
  useVirtualizer: (options: { count: number }) => ({
    getTotalSize: () => options.count * 80,
    getVirtualItems: () =>
      Array.from({ length: options.count }, (_, i) => ({
        index: i, start: i * 80, size: 80, key: i,
      })),
    measureElement: () => {},
    scrollToIndex: vi.fn(),
  }),
}));
```

이 mock의 트레이드오프는 실제 가상화 동작(스크롤 위치, 부분 렌더링)을 테스트하지 못한다는 것입니다. 이런 부분은 E2E 테스트(Playwright 등)로 보완해야 합니다.

#### 해설

가상화는 "왜 필요한가"와 "어떤 어려움이 있었는가"를 함께 설명해야 합니다. jsdom mock의 **한계까지 인지하고 보완 방안을 제시**하면 테스트 전략의 성숙도를 보여줍니다.

### Q12. React.memo를 어떤 컴포넌트에 적용했고, 어떤 기준으로 결정했나요?

#### 모범 답변

다음 컴포넌트에 `React.memo`를 적용했습니다:

- **Chat:** 프리뷰 상태(previewSections) 변경 시 불필요한 리렌더링 방지
- **Preview:** 채팅 메시지 추가(TEXT_DELTA) 시 프리뷰 내용이 안 바뀌었으면 리렌더링 불필요 (`export default memo(Preview)`)
- **ChatMessage:** 한번 렌더링되면 내용이 변하지 않음 (스트리밍 메시지 제외)
- **MessageInput:** 스트리밍 중에도 입력 필드가 리렌더링되면 안 됨

판단 기준은 **"부모가 자주 리렌더링되는데, 이 자식의 props는 변하지 않는가?"**입니다.

ChatSessionPage는 SSE 스트리밍 중 매 이벤트마다(text_delta, artifact_delta) reducer dispatch가 발생하여 리렌더링됩니다. 이때:

- `TEXT_DELTA` → messages만 변경 → Preview의 props(sections)는 안 변함 → memo 효과
- `ARTIFACT_DELTA` → previewSections만 변경 → Chat의 props(messages)는 안 변함 → memo 효과
- `SET_DRAFT` → draft만 변경 → Chat, Preview 모두 안 변함 → 둘 다 memo 효과

무분별한 memo 적용은 오히려 얕은 비교(shallow comparison) 비용이 발생합니다. props가 매번 바뀌는 컴포넌트에는 적용하지 않았습니다.

또한 `useCallback`으로 이벤트 핸들러 참조를 안정화했습니다:
```typescript
const handleDraftChange = useCallback(
  (value: string) => dispatch({ type: "SET_DRAFT", draft: value }),
  [],
);
```

이렇게 하지 않으면 매 렌더링마다 새 함수 참조가 생성되어 memo가 무력화됩니다.

#### 해설

memo의 핵심은 **"언제 쓰면 안 되는지"**를 아는 것입니다. SSE 스트리밍이라는 맥락(부모가 초당 여러 번 리렌더링)과 연결하고, useCallback과의 조합까지 설명하면 실전 경험을 보여줍니다.

### Q13. Vite의 manualChunks 설정을 route-based code splitting 대신 사용한 이유는?

#### 모범 답변

Vite의 기본 code splitting은 라우트 단위로 청크를 나눕니다. 하지만 이 경우 react-markdown, shiki가 ChatSessionPage 청크에 포함되어 해당 페이지 진입 시 한 번에 로드됩니다.

`manualChunks`로 두 그룹을 명시적으로 분리했습니다:

```typescript
manualChunks: {
  "vendor-react": ["react", "react-dom", "react-router"],
  "vendor-markdown": ["react-markdown", "remark-gfm", "rehype-pretty-code", "shiki"],
}
```

이렇게 하면 세 가지 이점이 있습니다:

1. **vendor-react는 모든 페이지에서 공유** → 한번 캐시되면 페이지 이동 시 재다운로드 불필요
2. **vendor-markdown은 ChatSessionPage에서만 로드** → 홈 페이지 초기 로드 44% 감소 (500KB → 280KB)
3. **라이브러리 업데이트 시 해당 청크만 캐시 무효화** → react 업데이트 시 markdown 청크는 그대로

트레이드오프는 의존성 추가/변경 시 수동으로 manualChunks 설정을 관리해야 한다는 것입니다. 하지만 이 프로젝트에서는 의존성이 "React 코어"와 "마크다운 렌더링"으로 명확히 나뉘므로 관리 부담이 크지 않습니다.

#### 해설

번들 최적화는 **구체적 수치(500KB → 280KB)**와 **실제 사용자 시나리오(홈 진입 시)**로 설명해야 합니다. "이론적으로 좋다"는 설득력이 없습니다.

## Part 5. 에러 핸들링

### Q14. API 에러 정규화(normalizeApiError)를 설명해주세요. 왜 필요한가요?

#### 모범 답변

서버가 보내는 에러 응답 형식이 일관되지 않습니다. 과제의 Hard 테스트에서 두 가지 다른 에러 포맷을 검증합니다:

- 400 Bad Request: `{ error: "Invalid request" }` (error 필드)
- 500 Internal Error: `{ message: "Server failed" }` (message 필드)
- 네트워크 에러: Error 객체
- 기타: 문자열, 빈 객체 등

`normalizeApiError`는 이 모든 케이스를 하나의 문자열로 통합합니다:

```typescript
function isRecord(value: unknown): value is Record<string, unknown> {
  return typeof value === "object" && value !== null;
}

export function normalizeApiError(
  error: unknown,
  fallback = DEFAULT_API_ERROR_MESSAGE,
): string {
  if (typeof error === "string" && error.trim()) return error;
  if (error instanceof Error && error.message.trim()) return error.message;
  if (isRecord(error)) {
    if (typeof error.error === "string" && error.error.trim()) return error.error;
    if (typeof error.message === "string" && error.message.trim()) return error.message;
  }
  return fallback;
}
```

주목할 점은 `isRecord` 타입 가드입니다. `as any` 대신 `unknown`을 좁혀가는 방식으로 타입 안전성을 확보했습니다. 또한 각 단계에서 `.trim()` 체크를 해서 빈 문자열도 거릅니다.

이 함수 덕분에 UI에서는 항상 `string` 타입의 에러 메시지만 다루면 됩니다:

```typescript
dispatch({
  type: "SUBMIT_ERROR",
  errorMessage: normalizeApiError(error, "메시지를 전송하지 못했습니다."),
});
```

에러 포맷에 대한 분기 로직이 컴포넌트에 침투하지 않습니다.

추가로, `remotes.ts`의 `throwServerError` 함수에서 서버 에러 응답 body를 그대로 throw합니다:

```typescript
async function throwServerError(res: Response): Promise<never> {
  let body: unknown;
  try { body = await res.json(); }
  catch { throw new Error(`HTTP ${res.status}: ${res.statusText}`); }
  throw body; // { error: "..." } 또는 { message: "..." } 그대로 throw
}
```

이렇게 throw된 객체가 `normalizeApiError`를 거치면 적절한 문자열로 변환됩니다.

#### 해설

**"서버가 보내는 에러 포맷을 신뢰하지 말라"**는 방어적 프로그래밍 원칙입니다. `throwServerError` → `normalizeApiError`로 이어지는 전체 파이프라인을 설명하면 에러 처리의 완결성을 보여줍니다.

### Q15. 세션 타이틀이 null이거나 잘못된 경우의 처리 전략은?

#### 모범 답변

서버에서 세션 타이틀이 null이거나 코드 펜스(```)로 시작하는 부적합한 값이 올 수 있습니다. `session-display.ts`에 폴백 로직을 순수 함수로 분리했습니다:

```typescript
function trimTitle(title: string | null | undefined) {
  return title?.trim() ?? "";
}

export function hasUsableSessionTitle(title: string | null | undefined) {
  const trimmedTitle = trimTitle(title);
  return Boolean(trimmedTitle) && !trimmedTitle.startsWith("```");
}

export function getDisplaySessionTitle(detail: ChatSessionDetail) {
  if (hasUsableSessionTitle(detail.title)) {
    return trimTitle(detail.title);
  }
  return detail.messages.find((message) => message.role === "user")?.content ?? "새 대화";
}
```

폴백 체인은:
1. title이 유효하고 코드 펜스로 시작하지 않으면 → 그대로 표시
2. title이 null/빈/코드 펜스로 시작 → 첫 번째 사용자 메시지 content를 표시
3. 사용자 메시지도 없으면 → "새 대화"

사이드바에서는 별도 함수(`getDisplaySessionLabel`)를 사용합니다:
```typescript
export function getDisplaySessionLabel(session: ChatSession) {
  return hasUsableSessionTitle(session.title)
    ? trimTitle(session.title)
    : session.lastMessagePreview;
}
```

`hasUsableSessionTitle`을 스트리밍 완료(done 이벤트) 시에도 재사용합니다:
```typescript
title: payload?.title && hasUsableSessionTitle(payload.title) ? payload.title : undefined,
```

#### 해설

엣지 케이스 처리의 핵심은 **로직을 순수 함수로 분리**하여 여러 곳에서 일관되게 적용하는 것입니다. `hasUsableSessionTitle`이 reducer, 스트리밍 훅, 사이드바에서 모두 재사용되는 구조를 보여주면 좋습니다.

### Q16. 104개 테스트의 구성과 전략을 설명해주세요.

테스트는 두 레벨로 구성했습니다:

**1. 통합 테스트 (13개):** 과제에서 제공한 스펙 기반

- Hard 3개: malformed JSON 복원, 에러 포맷 정규화(400/500), null 타이틀 폴백

**2. 컴포넌트 테스트 (91개):** 14개 파일
- Preview: 복사 기능, 섹션 렌더링, 빈 상태
- MessageInput: Enter 전송, Shift+Enter 줄바꿈, 빈 입력 방지
- StyleSelector: 스타일 선택/변경
- ChatMessage: 역할별(user/assistant) 렌더링
- 기타 컴포넌트별 독립 동작 검증

테스트 환경에서의 특수 처리 세 가지:

**matchMedia polyfill:** `useBreakpoint`가 의존하는 `matchMedia`를 직접 구현했습니다. 특이한 점은 `vi.fn()` 대신 일반 함수를 사용한 것입니다:
```typescript
Object.defineProperty(window, "matchMedia", {
  writable: true,
  value: (query: string) => ({
    matches: query === "(min-width: 783px)", // 태블릿+ 시뮬레이션
    ...
  }),
});
```
`vi.fn()`을 쓰면 `afterEach`의 `vi.restoreAllMocks()`가 첫 테스트 이후 mock을 undefined로 되돌려 후속 테스트가 깨집니다.

**react-virtual mock:** jsdom의 레이아웃 계산 불가 → 모든 아이템을 렌더링하는 mock

**renderUI 비동기화:** React.lazy + Suspense fallback 소멸을 waitFor로 대기

#### 해설

"테스트를 많이 작성했다"보다 **"테스트 환경의 한계를 어떻게 극복했는가"**가 포인트입니다. `vi.fn()` vs 일반 함수 차이(`restoreAllMocks` 이슈)처럼 디테일한 부분을 설명하면 실전 경험을 보여줍니다.

### Q17. jsdom에서 matchMedia와 react-virtual을 mock한 이유와 방법은?

#### 모범 답변

jsdom은 순수 DOM 시뮬레이터로 레이아웃 엔진이 없습니다. 두 가지 핵심 API가 부재합니다:

**1. matchMedia 부재:**

```typescript
Object.defineProperty(window, "matchMedia", {
  writable: true,
  value: (query: string) => ({
    matches: query === "(min-width: 783px)",  // 태블릿+ 모드 시뮬레이션
    media: query,
    onchange: null,
    addListener: () => {},
    removeListener: () => {},
    addEventListener: () => {},
    removeEventListener: () => {},
    dispatchEvent: () => false,
  }),
});
```

`matches: query === "(min-width: 783px)"`로 설정한 이유는 테스트가 태블릿+(783px+)을 가정하기 때문입니다. 이 설정에서 `isTablet`이 true가 되어 데스크톱 레이아웃(채팅+프리뷰 동시 표시)이 렌더링됩니다. Easy/Hard 테스트가 프리뷰 영역의 존재를 검증하므로, 모바일 모드에서는 테스트가 실패합니다.

**2. react-virtual 레이아웃 계산 불가:**

jsdom에서 `getBoundingClientRect`는 항상 `{width: 0, height: 0, ...}`을 반환합니다. react-virtual은 요소 높이를 기반으로 어떤 아이템을 렌더링할지 결정하므로, 높이가 0이면 아무것도 렌더링하지 않습니다.

mock은 `getVirtualItems`가 **모든 아이템**을 반환하게 합니다:

```typescript
getVirtualItems: () =>
  Array.from({ length: options.count }, (_, i) => ({
    index: i, start: i * 80, size: 80, key: i,
  })),
```

트레이드오프: 이 mock은 가상화의 "부분 렌더링" 자체를 테스트하지 못합니다. 스크롤 위치에 따른 렌더링 범위, overscan, 동적 높이 재측정 등은 E2E 테스트에서 검증해야 합니다.

#### 해설

Mock의 **한계를 명시적으로 인지하고 보완 방안을 제시**하는 것이 중요합니다. "Mock했으니 OK"가 아니라 "Mock으로 못 잡는 것이 뭐고, 어떻게 보완하는가"까지 말하면 성숙한 테스트 전략을 보여줍니다.

const STYLE_INSTRUCTIONS: Record<string, string> = {

### Q19. 프리뷰 영역의 복사 기능은 어떻게 구현했나요?

#### 모범 답변

프리뷰 영역에는 두 가지 복사가 있습니다:

**1. 전체 복사:** 헤더의 "전체복사" 버튼 → 모든 섹션 내용을 합쳐서 클립보드에 복사
**2. 섹션별 복사:** 각 섹션의 hover 시 나타나는 복사 버튼 → 해당 섹션만 복사

`usePreviewCopy` 커스텀 훅으로 로직을 분리했습니다:

```typescript
export function usePreviewCopy() {
  const [headerFeedback, setHeaderFeedback] = useState<CopyFeedback>("idle");
  const [sectionFeedback, setSectionFeedback] = useState<{id: string; state: CopyFeedback} | null>(null);

  return {
    headerFeedback,
    sectionFeedback,

    copySection(id: string, text: string) { ... },
  };
}
```

핵심 디테일은 **코드 블록 복사 시 펜스 마커 제거**입니다:

```typescript
function getCopyablePreviewSectionContent(section: ResolvedPreviewSection) {
  if (section.variant !== "code") return section.content;
  const trimmedContent = section.content.trim();
  const fencedCodeMatch = trimmedContent.match(/^```([\w-]*)\n([\s\S]*?)(?:\n```)?$/);
  if (!fencedCodeMatch) return trimmedContent;
  return fencedCodeMatch[2].trimEnd();  // 순수 코드만 반환
}
```

사용자가 ````typescript`와 닫는 ` ``` `까지 복사하면 에디터에 붙여넣을 때 불편합니다. 이 UX 판단은 AI가 아니라 제가 직접 내린 결정입니다.

피드백은 1.5초 후 자동으로 사라집니다:

const COPY_FEEDBACK_DURATION = 1500;
headerTimeoutRef.current = window.setTimeout(() => {
  setHeaderFeedback("idle");

```

timeout ref를 관리하여 컴포넌트 언마운트 시 cleanup하고, 빠르게 연속 클릭해도 이전 timeout을 cancel합니다.

#### 해설

복사 기능 자체보다 **펜스 마커 스트리핑이라는 UX 결정**이 면접 포인트입니다. 기술적 구현(정규식 파싱)과 UX 판단(사용자가 뭘 원하는가)을 함께 설명하세요.

### Q20. 패널 리사이저(드래그 핸들)의 구현과 제약 조건은?

#### 모범 답변

데스크톱에서 채팅/프리뷰 패널 사이의 드래그 핸들로 너비를 조절합니다. Pointer Events API를 사용한 구현:

```typescript
function handleResizePointerDown(e: React.PointerEvent) {
  e.preventDefault();
  isDraggingRef.current = true;
  (e.currentTarget as HTMLElement).setPointerCapture(e.pointerId);
}

function handleResizePointerMove(e: React.PointerEvent) {
  if (!isDraggingRef.current || !containerRef.current) return;
  const rect = containerRef.current.getBoundingClientRect();
  const percent = ((e.clientX - rect.left) / rect.width) * 100;
  dispatch({
    type: "SET_CHAT_WIDTH",
    chatWidthPercent: Math.min(MAX_CHAT_WIDTH_PERCENT, Math.max(MIN_CHAT_WIDTH_PERCENT, percent)),
  });
}
```

`setPointerCapture`로 포인터를 캡처하여, 마우스가 드래그 바 밖으로 나가도 계속 추적합니다. 이 없으면 빠르게 드래그할 때 마우스가 바에서 벗어나면 추적이 끊깁니다.

제약 조건:
- **최소 25%, 최대 80%**: 한쪽 패널이 너무 좁아지는 것 방지
- **기본값 57%**: 채팅에 약간 더 많은 공간 할당
- **모바일에서 숨김**: `isTablet` 분기로 탭 전환 모드에서는 리사이저 자체가 렌더링되지 않음

성능 고려:
- 드래그 중 매 프레임 `SET_CHAT_WIDTH` dispatch → `chatWidthPercent`만 변경
- Chat과 Preview는 `React.memo`로 감싸져 있어, `chatWidthPercent` 변경은 CSS `width` 스타일만 업데이트하고 내부 리렌더링은 발생하지 않음
- `matchMedia` 사용으로 드래그 이벤트가 브레이크포인트 재계산을 트리거하지 않음

#### 해설

드래그 리사이저는 구현보다 **성능이 면접 포인트**입니다. `setPointerCapture`, memo와의 조합, matchMedia로 불필요한 계산을 차단하는 구조까지 설명하면 깊이를 보여줍니다.

두 AI 도구를 목적별로 분리해 활용했습니다:

**Claude Code (Anthropic) — 구현 도구:**

- 반응형 브레이크포인트 결정

핵심은 **AI의 제안을 무비판적으로 수용하지 않았다**는 것입니다. 구체적 사례 세 가지:

**1. framer-motion 제거:** AI가 프로젝트에 포함시킨 framer-motion(5.5MB)을 실제로는 어디서도 사용하지 않았습니다. 의존성 분석 후 직접 제거했습니다.

**2. Redux/Zustand 거절:** AI가 글로벌 상태 관리를 제안했지만, 모든 상태가 ChatSessionPage에 귀속된다는 분석 후 useReducer를 고수했습니다.

**3. flushSync → useTransition 변경 강행:** AI가 flushSync를 유지하려 했지만, 직접 브라우저에서 스트리밍 중 UI 반응성을 비교 테스트한 결과 useTransition이 확실히 나았습니다.

50개 이상의 작업 세션을 `docs/ai-work-log/`에 문서화했습니다. 각 세션에서 AI가 뭘 했고, 내가 뭘 결정했고, 왜 그렇게 판단했는지를 기록했습니다.

#### 해설

AI 활용 질문의 핵심은 **"AI를 도구로 쓰되, 판단은 내가 한다"**입니다. 오버룰한 구체적 사례 3개를 준비하고, 각각 "왜 AI가 틀렸는지"를 기술적으로 설명할 수 있어야 합니다.

### Q24. Hono.js를 선택한 이유는? Express와의 차이점은?

#### 모범 답변

Hono.js는 과제 boilerplate에 포함된 프레임워크이지만, Express 대비 이 과제에 적합한 이유가 있습니다:

1. **Web Standards 기반:** Request/Response API를 사용하여 Cloudflare Workers 등 엣지 런타임 호환. Express는 Node.js 전용.
2. **TypeScript 네이티브:** 추가 `@types` 패키지 없이 완전한 타입 추론. Express는 `@types/express` 필요.
3. **SSE 내장 지원:** `SSEStreamingApi`의 `writeSSE`로 Named Events를 자연스럽게 전송. Express에서는 raw response를 직접 다뤄야 함.
4. **경량:** 번들 크기가 Express의 약 1/10.

서버에서 제가 추가한 주요 작업 세 가지:

**max_tokens 설정:** OpenRouter 기본값 65535가 무료 티어 한도를 초과하여 402 에러 발생. 4096 기본값 + 환경 변수 오버라이드 구현.

**style 파라미터 처리:** 쿼리 파라미터 `?style=casual`을 받아 `STYLE_INSTRUCTIONS` 맵에서 시스템 프롬프트에 추가.

**아티팩트 추출:** 코드 펜스 파싱으로 text_delta/artifact_delta 분리. 줄 단위 상태 머신(`inCodeBlock` flag)으로 구현.

#### 해설

boilerplate에 있었으므로 "선택 이유"보다 **"어떤 부분을 직접 추가했는가"**가 중요합니다. max_tokens 402 에러 해결은 실제 운영 이슈 대응 경험으로 좋은 포인트입니다.

### Q25. OpenRouter API의 max_tokens 문제를 어떻게 발견하고 해결했나요?

#### 모범 답변

**발견 과정:**
1. 로컬 개발에서는 `hasOpenRouterKey()`가 false → 시나리오 기반 fallback으로 동작 → 문제 없음
2. `.env`에 OPENROUTER_API_KEY를 설정하고 실제 AI 테스트 → 402 Payment Required 에러
3. OpenRouter 문서 확인 → 무료 모델의 기본 max_tokens가 65535이지만 무료 티어 할당량 초과

**해결:**
```typescript
const DEFAULT_OPENROUTER_MAX_TOKENS = 4096;

function getOpenRouterMaxTokens(): number {
  const rawValue = process.env.OPENROUTER_MAX_TOKENS;
  const parsedValue = Number.parseInt(rawValue || "", 10);
  if (Number.isInteger(parsedValue) && parsedValue > 0) {
    return parsedValue;
  }
  return DEFAULT_OPENROUTER_MAX_TOKENS;
}
```

4096을 기본값으로 선택한 이유는 대부분의 채팅 응답이 이 범위 안에 들어오면서도 무료 티어 할당량을 초과하지 않는 균형점이기 때문입니다.

환경 변수 오버라이드를 둔 이유는 유료 티어에서는 더 큰 값을 설정할 수 있게 하기 위해서입니다. 하드코딩하면 환경마다 코드를 수정해야 합니다.

입력값 검증도 방어적으로 했습니다: `Number.parseInt`가 NaN이나 음수를 반환할 수 있으므로 `Number.isInteger(parsedValue) && parsedValue > 0`으로 체크합니다.

#### 해설

이 질문은 **"문제 발견 → 원인 분석 → 해결 → 확장성 고려"** 구조로 답하면 됩니다. 로컬에서 발견 못한 이유(fallback 모드), 4096이라는 수치의 근거, 환경 변수 설계까지 설명하면 완결성이 높아집니다.

### Q26. 이 프로젝트에서 가장 어려웠던 기술적 챌린지는 무엇이었나요?

#### 모범 답변

가장 어려웠던 것은 **SSE 스트리밍 중 상태 관리의 일관성 유지**였습니다.

구체적으로 세 가지 동시성 문제가 있었습니다:

**1. 이벤트 순서 보장:** text_delta와 artifact_delta가 교차로 오면 각각 다른 상태 필드(messages vs previewSections)를 업데이트해야 합니다. 하나의 이벤트가 다른 이벤트의 상태를 덮어쓰면 안 됩니다.

**2. 스트리밍 중 상태 참조:** EventSource 리스너 내부에서 최신 상태를 참조해야 하지만, 클로저는 리스너 등록 시점의 상태를 캡처합니다. 이 문제를 ref(`streamMessageIdRef`, `streamPreviewSectionIdRef`)로 해결했습니다.

**3. STREAM_DONE의 복잡한 분기:** done 이벤트 처리 시:
- streamPreviewSectionId가 있으면 → 해당 섹션의 isStreaming을 false로
- 없지만 streamMessageId가 있으면 → 메시지 content에서 artifact를 추출하여 프리뷰 생성
- title이 있고 유효하면(코드 펜스로 시작하지 않으면) → 세션 타이틀 업데이트

이 모든 분기가 하나의 reducer case에서 명시적으로 처리됩니다:

```typescript
case "STREAM_DONE": {
  let nextState = { ...state, isStreaming: false };
  if (action.title) nextState.title = action.title;
  if (action.streamPreviewSectionId) {
    nextState.previewSections = state.previewSections.flatMap((s) => {
      if (s.id !== action.streamPreviewSectionId) return [s];
      const nextSection = { ...s, isStreaming: false };
      return nextSection.content.trim() ? [nextSection] : [];
    });
  } else if (action.streamMessageId) {
    // artifact가 서버에서 분리되지 않은 경우 클라이언트에서 추출
    const streamMessage = state.messages.find((m) => m.id === action.streamMessageId);
    if (streamMessage) {
      const previewArtifact = extractPreviewArtifact(streamMessage);
      if (previewArtifact?.content.trim()) {
        nextState.previewSections = [...state.previewSections, { ... }];
      }
    }
  }
  return nextState;
}
```

reducer로 상태 전이를 명시화한 것이 이 복잡성을 관리 가능하게 만든 핵심입니다. 각 case가 하나의 상태 전이를 명확하게 정의하므로, 어떤 순서로 이벤트가 와도 예측 가능합니다.

#### 해설

**"무엇이 어려웠고, 왜 어려웠으며, 어떻게 해결했는가"** 구조로 답하세요. STREAM_DONE의 복잡한 분기를 실제 코드와 함께 보여주면 깊이를 증명합니다.

### Q27. 이 프로젝트를 프로덕션에 배포한다면 추가로 필요한 것은?

#### 모범 답변

우선순위 순으로 네 카테고리입니다:

**1순위 - 안정성 (사용자 경험 보호):**
- 스트림 취소 버튼 ("답변 중지") — 현재 한번 시작된 스트림은 done까지 기다려야 함
- SSE 재연결 로직 — EventSource의 자동 재연결이 있지만, 서버가 새 스트림 ID를 부여하므로 상태 복원이 필요
- React Error Boundary — 컴포넌트 레벨 크래시 시 전체 앱이 아닌 해당 영역만 fallback UI 표시

**2순위 - 보안:**
- API 인증/인가 — 현재 인증 없이 모든 세션에 접근 가능
- Rate limiting — 무제한 메시지 전송 방지
- 마크다운 렌더링 시 XSS sanitize — react-markdown은 기본적으로 안전하지만, rehype 플러그인 체인에서 raw HTML이 통과할 수 있음

**3순위 - 성능:**
- 세션 데이터 캐싱 — 현재 `cache: "no-store"`로 매번 서버 요청
- 데이터베이스 연동 — 현재 인메모리 저장으로 서버 재시작 시 전체 손실

**4순위 - 사용성:**
- 메시지 재생성 / 대화 검색 / 다크 모드 / 키보드 단축키

가장 먼저 하겠다고 말할 것은 **스트림 취소 + Error Boundary**입니다. 기능 추가보다 "장애 시 사용자 경험을 어떻게 보호하는가"가 프로덕션의 핵심이기 때문입니다.

#### 해설

기능 나열보다 **"가장 우선순위가 높은 것이 무엇이고 왜인가"**를 말하는 것이 강합니다. 안정성 → 보안 → 성능 → 사용성 순서로 답하면 운영 감각을 보여줍니다.

### Q28. PubSub 패턴으로 사이드바 업데이트를 처리한 이유는?

#### 모범 답변

ChatSessionPage에서 메시지를 보내면 사이드바 세션 목록도 갱신되어야 합니다. 두 컴포넌트는 서로 다른 라우트/레이아웃 트리에 있어서 props 전달이 불가능합니다.

세 가지 대안을 비교했습니다:

| 방법 | 장점 | 단점 |
|------|------|------|
| Redux/Zustand | 확실한 상태 공유 | 이 하나의 이벤트를 위해 도입하기엔 과도 |
| Context API | 외부 라이브러리 불필요 | Provider 하위 전체가 리렌더링 |
| **PubSub (선택)** | 6줄 구현, 느슨한 결합 | 디버깅 시 이벤트 추적 어려움 |

실제 구현은 window의 CustomEvent를 활용한 6줄입니다:

```typescript
const SESSIONS_CHANGED_EVENT = "chat:sessions-changed";

export function emitSessionsChanged() {
  if (typeof window === "undefined") return;
  window.dispatchEvent(new Event(SESSIONS_CHANGED_EVENT));
}

export function subscribeSessionsChanged(listener: () => void) {
  if (typeof window === "undefined") return () => {};
  window.addEventListener(SESSIONS_CHANGED_EVENT, listener);
  return () => window.removeEventListener(SESSIONS_CHANGED_EVENT, listener);
}
```

사이드바에서 `subscribeSessionsChanged`로 구독하고, ChatSessionPage에서 메시지 전송/스트림 완료 시 `emitSessionsChanged()`를 호출합니다.

이 패턴은 **"전달할 데이터가 없고, 알림만 보내면 되는 경우"**에 적합합니다. 사이드바는 이벤트를 받으면 자체적으로 `getSessions()`를 호출하여 최신 목록을 가져옵니다.

#### 해설

**세 가지 대안을 비교하고 판단 기준을 제시**하는 것이 핵심입니다. "데이터가 아니라 알림만 보내면 된다"는 판단이 PubSub 선택의 근거입니다.

### Q29. 이 코드베이스에서 기술 부채라고 생각하는 부분은?

#### 모범 답변

솔직하게 세 가지를 인정합니다:

**1. 인메모리 세션 저장소:** 서버 재시작 시 모든 데이터 소실. 과제 boilerplate가 이렇게 설계되어 있지만, 프로덕션에서는 DB 연동이 필수입니다.

**2. E2E 테스트 부재:** jsdom 기반 단위/통합 테스트만 있어서 실제 브라우저 동작을 검증하지 못합니다. 가상화 스크롤, 드래그 리사이즈, 모바일 탭 전환 같은 인터랙션은 Playwright E2E로만 확실히 잡을 수 있습니다.

**3. 아티팩트 추출의 다중 폴백:** `session-display.ts`의 `extractPreviewArtifact`가 `message.artifact` → 표준 펜스 → 혼합 콘텐츠 → 비정상 펜스 순으로 4단계 폴백을 합니다. 서버가 항상 artifact를 분리해서 보내면 클라이언트 폴백이 불필요하지만, 과제 서버의 시나리오 모드가 artifact 없이 content에 코드를 섞어 보내는 경우가 있어서 이 복잡도가 필요했습니다.

```typescript
export function extractPreviewArtifact(message: Message): PreviewArtifact | null {
  if (message.role !== "assistant") return null;
  if (message.artifact?.content.trim()) return message.artifact;
  return (
    extractArtifactFromStandardFence(message.content) ??
    extractArtifactFromMixedContent(message.content) ??
    extractArtifactFromMalformedFence(message.content)
  );
}
```

하지만 2일 시간 제약 안에서 이 세 가지는 **의도적인 트레이드오프**였습니다. 우선순위는 "기능 완성 + 104개 테스트 통과 + 코드 품질"이었고, 이 기준에서는 올바른 판단이었다고 생각합니다.

#### 해설

기술 부채 질문은 **자기 인식과 우선순위 판단력**을 봅니다. "없다"는 최악이고, 나열만 해도 부족합니다. **"알고 있었지만 의도적으로 후순위로 뒀고, 그 이유는..."**이 가장 좋은 구조입니다.

### Q30. 같은 과제를 다시 한다면 다르게 할 부분은?

#### 모범 답변

세 가지를 다르게 하겠습니다:

**1. 처음부터 useReducer로 시작:** 처음에는 useState 여러 개(isLoading, isStreaming, messages, previewSections...)로 시작했다가 Day 2에서 useReducer로 리팩토링했습니다. 11개 상태가 하나의 흐름(세션 로드 → 메시지 전송 → 스트리밍 → 완료)에 귀속된다는 것을 처음부터 인지했다면, 리팩토링 비용(상태 업데이트 로직 전체 이동)을 줄였을 것입니다.

**2. Playwright E2E 테스트 최소 1개:** 핵심 플로우(세션 생성 → 메시지 전송 → 스트리밍 → 프리뷰 표시) 한 개만이라도 E2E로 작성했을 것입니다. jsdom mock의 한계(가상화, 드래그 리사이즈, matchMedia)를 보완하는 데 결정적 도움이 됩니다. 2일 안에 가능했을지는 모르지만, 시간이 있었다면 최우선.

**3. AI 코드 수용 기준 강화:** 초반에 AI가 생성한 Container.tsx(불필요한 추상화)나 framer-motion(미사용 의존성)을 빠르게 제거하지 못했습니다. "AI가 생성한 코드도 PR 리뷰하듯 본다"는 원칙을 처음부터 엄격히 적용했을 것입니다.

핵심은 **"더 많이 하겠다"가 아니라 "더 일찍 올바른 판단을 하겠다"**입니다.

#### 해설

**성장 가능성**을 보는 질문입니다. 구체적 사례 + 개선 방향 + 그로부터 배운 원칙의 구조가 가장 좋습니다. "모든 것을 완벽하게"는 비현실적이므로, 임팩트가 큰 1-3가지에 집중하세요.
