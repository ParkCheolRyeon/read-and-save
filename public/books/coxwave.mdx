# Coxwave 과제 면접 질문 리스트

> Claude Artifacts 스타일 2-패널 채팅 UI 과제 (React 19 + Vite + TypeScript + SSE) 기반 기술 면접 질문입니다.
> 이틀 안에 104개 테스트 통과, 보너스 기능 전체 구현, AI 활용 문서화까지 완료한 과제를 심층 검증합니다.

## Part 1. 프로젝트 개요 및 설계 판단

### Q1. 이 과제의 전체 아키텍처를 설명해주세요. 왜 이런 구조를 선택했나요?

#### 합격 답변 예시

이 프로젝트는 Claude Artifacts 스타일의 2-패널 채팅 UI입니다. 왼쪽은 채팅 영역, 오른쪽은 코드/마크다운 프리뷰 영역으로 구성했습니다. React 19 + Vite + TypeScript를 기반으로, SSE(Server-Sent Events)를 통해 AI 응답을 실시간 스트리밍합니다.

핵심 설계 원칙은 세 가지였습니다:
1. **단일 페이지 스코프의 상태 관리**: 채팅 세션의 모든 상태를 `useReducer` 하나로 통합해 예측 가능하게 관리
2. **점진적 로딩**: React.lazy + Vite manual chunks로 마크다운 렌더링 라이브러리(~200KB)를 홈에서 분리
3. **방어적 스트리밍**: 서버가 의도적으로 보내는 malformed JSON에도 스트림이 끊기지 않는 구조

이 구조를 선택한 이유는 과제의 핵심 요구사항이 "실시간 스트리밍 + 에러 복원력 + 반응형 UI"였기 때문입니다. 글로벌 상태 관리 라이브러리 없이도 페이지 단위 reducer로 충분했고, SSE는 단방향 스트리밍에 WebSocket보다 단순하면서도 충분한 프로토콜이었습니다.

#### 상세 해설

면접관이 보고 싶은 것은 "기술 나열"이 아니라 **"왜 이 조합이 이 문제에 적합한가"**입니다.

- useReducer를 선택한 이유 → 16개 액션 타입, 11개 상태값이 하나의 세션에 귀속 → 글로벌 상태 불필요
- SSE를 선택한 이유 → 서버→클라이언트 단방향 → WebSocket의 양방향은 과도
- Vite를 선택한 이유 → 과제 boilerplate에 포함 + HMR 속도 + manual chunks 지원

### Q2. 상태 관리를 Redux나 Zustand 대신 useReducer로 한 이유는 무엇인가요?

#### 합격 답변 예시

이 앱에서 관리해야 할 상태는 세션 상세 정보, 메시지 목록, 스트리밍 상태, UI 상태(로딩/에러/프리뷰 너비 등) 총 11개입니다. 이 상태들은 모두 `ChatSessionPage` 한 페이지에 귀속되고, 다른 페이지와 공유할 필요가 없었습니다.

useReducer를 선택한 구체적 이유는:
1. **액션 기반 상태 전이**: 16개 액션 타입으로 모든 상태 변화를 명시적으로 관리할 수 있어 디버깅이 쉬움
2. **외부 의존성 제거**: Redux/Zustand 없이도 충분하므로 번들 크기 절감
3. **테스트 용이성**: reducer 함수를 순수 함수로 분리하면 단위 테스트가 간단

트레이드오프로는, 여러 페이지에서 세션 데이터를 공유해야 하는 경우(예: 사이드바 실시간 업데이트) 추가 작업이 필요합니다. 이 부분은 PubSub 이벤트 시스템(`session-list.ts`)으로 해결했습니다.

#### 상세 해설

"왜 안 썼는지"를 설명할 때는 반드시 **트레이드오프**를 함께 말해야 합니다. 단순히 "필요 없어서"라고 하면 깊이가 없어 보입니다. PubSub으로 해결한 부분까지 언급하면 설계 판단의 완성도가 높아집니다.

### Q3. 2-패널 레이아웃의 반응형 전략을 설명해주세요. 브레이크포인트는 어떻게 결정했나요?

#### 합격 답변 예시

데스크톱(1024px+)에서는 좌우 분할, 모바일(783px 미만)에서는 탭 기반 전환으로 구현했습니다.

783px이라는 비표준 브레이크포인트를 선택한 이유는 iPad 가로 모드(1024px)와 세로 모드(768px) 사이의 전환점을 고려했기 때문입니다. 768px에서 2-패널을 유지하면 각 패널이 너무 좁아지고, 1024px부터 분할하면 태블릿 가로 모드에서 불필요하게 탭 전환을 강제합니다.

구현에서는 `matchMedia` API를 사용했습니다. `resize` 이벤트 대신 `matchMedia`를 선택한 이유는 패널 드래그 리사이즈 시 resize 이벤트가 초당 60회 이상 발생하는데, matchMedia는 브레이크포인트 임계값을 넘을 때만 1-2회 트리거되기 때문입니다.

#### 상세 해설

브레이크포인트 결정 과정을 설명할 때 "표준 값을 썼다"보다 **"왜 이 값이 이 UI에 적합한지"**를 말하는 것이 훨씬 강합니다. matchMedia vs resize 비교는 성능 감각을 보여주는 좋은 포인트입니다.

## Part 2. SSE 스트리밍과 실시간 처리

### Q4. SSE(Server-Sent Events)를 WebSocket 대신 선택한 이유는 무엇인가요?

#### 합격 답변 예시

이 채팅 앱에서 실시간 통신은 서버→클라이언트 단방향입니다. 사용자 메시지는 REST API로 보내고, AI 응답만 스트리밍으로 받습니다. WebSocket은 양방향 통신이 필요한 경우에 적합하지만, 여기서는 오버스펙입니다.

SSE의 장점은:
1. **HTTP 호환**: 별도 프로토콜 업그레이드 없이 표준 HTTP 위에서 동작
2. **자동 재연결**: 브라우저가 연결 끊김 시 자동 재시도 (WebSocket은 수동 구현 필요)
3. **디버깅 용이**: 브라우저 DevTools의 EventStream 탭에서 바로 확인 가능
4. **서버 구현 간단**: Hono.js에서 ReadableStream으로 바로 전송

트레이드오프는 서버→클라이언트만 가능하다는 것인데, 이 앱의 요구사항과 정확히 일치합니다.

#### 상세 해설

SSE vs WebSocket 비교는 프론트엔드 면접에서 자주 나오는 질문입니다. 핵심은 "양방향이 필요한가?"라는 판단 기준을 명확히 제시하는 것입니다.

---

### Q5. SSE 이벤트 프로토콜을 설명해주세요. text_delta와 artifact_delta를 분리한 이유는?

#### 합격 답변 예시

서버에서 보내는 SSE Named Events는 세 가지입니다:

```
text_delta      → { content: string }              // 채팅 텍스트
artifact_delta  → { content: string, language?: string }  // 코드/문서 프리뷰
done            → { title?: string }               // 스트림 완료
```

text_delta와 artifact_delta를 분리한 이유는 2-패널 UI 때문입니다. 채팅 영역의 텍스트와 프리뷰 영역의 코드는 독립적으로 렌더링되어야 합니다. 만약 하나의 이벤트로 합치면 클라이언트에서 매번 파싱해서 분리해야 하고, 코드 펜스 안/밖을 판단하는 로직이 클라이언트에 들어가야 합니다.

서버에서 이미 코드 펜스를 파싱하여 artifact_delta로 분리해 보내므로, 클라이언트는 이벤트 타입에 따라 적절한 패널에 append하기만 하면 됩니다.

#### 상세 해설

이 질문의 핵심은 **관심사 분리**입니다. 서버가 파싱 책임을 지고, 클라이언트는 렌더링에만 집중하는 구조가 왜 좋은지를 설명하면 됩니다.

### Q6. 서버가 의도적으로 보내는 malformed JSON을 어떻게 처리했나요?

#### 합격 답변 예시

이 과제의 Hard 테스트 중 하나가 "서버가 ~10% 확률로 corrupt JSON을 보내는 상황에서도 스트림이 정상 동작해야 한다"입니다.

해결 방법은 `parseSseEventData()` 함수에서 JSON.parse를 try/catch로 감싸고, 파싱 실패 시 null을 반환하는 것입니다:

```typescript
function parseSseEventData<T>(data: string): T | null {
  try {
    return JSON.parse(data) as T;
  } catch {
    return null;
  }
}
```

호출하는 쪽에서는 null 체크 후 해당 이벤트를 건너뜁니다:

```typescript
const payload = parseSseEventData<TextDeltaData>(event.data);
if (!payload?.content) return; // 잘못된 이벤트는 무시, 스트림 계속 처리
```

핵심은 **개별 이벤트의 실패가 전체 스트림을 중단시키지 않는 것**입니다. 에러를 throw하지 않고 null로 처리해서 다음 이벤트를 계속 수신합니다.

#### 상세 해설

이 패턴은 "graceful degradation"의 전형적인 예입니다. 면접에서는 "왜 에러를 무시하는 것이 여기서는 올바른 전략인가"를 설명해야 합니다 — 10개 중 9개가 정상이면 1개의 실패로 나머지를 버리는 것이 더 나쁜 UX이기 때문입니다.

### Q7. useStreamChat 훅의 내부 구조를 설명해주세요. SSE 라이프사이클은 어떻게 관리하나요?

#### 합격 답변 예시

`useStreamChat`은 약 130줄의 커스텀 훅으로, SSE 스트리밍의 전체 라이프사이클을 관리합니다:

1. **시작**: `createChatStream(sessionId, style)` 호출 → EventSource 생성
2. **수신**: Named event listeners(`text_delta`, `artifact_delta`, `done`) 등록
3. **처리**: 각 이벤트를 파싱하여 dispatch로 reducer에 전달
4. **완료**: `done` 이벤트 수신 시 EventSource 닫고 상태 정리
5. **에러**: 네트워크 에러 시 STREAM_ERROR 액션 dispatch

이 훅을 분리한 이유는 ChatSessionPage의 복잡도를 낮추기 위해서입니다. 페이지 컴포넌트는 UI 렌더링에만 집중하고, 스트리밍 로직은 훅이 캡슐화합니다.

#### 상세 해설

커스텀 훅 설계에서 중요한 것은 **왜 이 로직을 분리했는가**입니다. "재사용을 위해"가 아니라 "관심사 분리와 테스트 용이성"이 더 설득력 있는 이유입니다.

## Part 3. React 19 기능 활용

### Q8. useOptimistic를 어떻게 활용했나요? 일반적인 optimistic update와 뭐가 다른가요?

#### 합격 답변 예시

React 19의 `useOptimistic`를 사용해 사용자가 메시지를 전송하면 서버 응답을 기다리지 않고 즉시 메시지를 화면에 표시합니다.

기존 방식(setState로 임시 데이터 추가 → 서버 응답 후 교체)과의 차이점은:
1. **React가 낙관적 상태와 실제 상태를 자동으로 관리**: 트랜지션이 완료되면 낙관적 값은 자동으로 사라지고 실제 값으로 대체
2. **별도의 롤백 로직 불필요**: 에러 시 React가 자동으로 이전 상태로 복원
3. **useTransition과 통합**: pending 상태와 낙관적 UI를 하나의 흐름으로 관리

구체적으로, 메시지 전송 시 useOptimistic으로 사용자 메시지를 먼저 보여주고, SSE 스트리밍이 시작되면 실제 서버 데이터로 상태가 전환됩니다.

#### 상세 해설

useOptimistic는 React 19의 핵심 신규 훅 중 하나입니다. "그냥 setState로 해도 되는 거 아닌가?"라는 후속 질문에 대비해서, **자동 롤백과 트랜지션 통합**이라는 차별점을 명확히 준비해야 합니다.

### Q9. useTransition을 어디서, 왜 사용했나요? flushSync와의 차이는?

#### 합격 답변 예시

SSE 스트리밍 중 매 이벤트마다 상태를 업데이트하면 메인 스레드가 블로킹될 수 있습니다. `useTransition`을 사용해 스트리밍 업데이트를 비긴급(non-urgent) 전환으로 마킹했습니다.

처음에는 `flushSync`를 고려했지만, flushSync는 동기적으로 DOM을 업데이트하여 스트리밍 중 UI가 버벅거리는 문제가 있었습니다. useTransition으로 전환하니:
1. React가 여러 업데이트를 배치 처리
2. 사용자 입력(타이핑 등)이 스트리밍 업데이트보다 우선순위를 가짐
3. 스크롤 등 인터랙션이 부드러워짐

실제로 AI가 처음에 flushSync를 유지하려 했지만, 직접 테스트해보니 useTransition이 확실히 체감 성능이 좋아서 변경했습니다.

#### 상세 해설

이 질문은 **Concurrent React에 대한 이해도**를 검증합니다. flushSync(동기) vs useTransition(비동기 배치)의 차이를 체감 기반으로 설명할 수 있으면 강합니다. "AI를 오버룰한 경험"은 독립적 기술 판단력을 보여주는 좋은 포인트입니다.

### Q10. React.lazy와 Suspense를 어떻게 활용했나요?

#### 합격 답변 예시

3개 페이지(ChatHomePage, ChatSessionPage, ChatHistoryPage)를 `React.lazy`로 분리했습니다. 특히 ChatSessionPage가 의존하는 마크다운 렌더링 라이브러리(react-markdown + shiki, ~200KB)가 홈 페이지에서는 불필요하므로, Vite의 `manualChunks` 설정과 함께 `vendor-markdown` 청크로 분리했습니다.

```typescript
// vite.config.ts
manualChunks: {
  'vendor-react': ['react', 'react-dom', 'react-router'],
  'vendor-markdown': ['react-markdown', 'remark-gfm', 'rehype-pretty-code', 'shiki'],
}
```

결과적으로 홈 페이지 초기 로드가 ~500KB에서 ~280KB로 줄었습니다. Suspense fallback으로 로딩 스피너를 보여주면서 사용자 체감 로딩 시간도 개선했습니다.

#### 상세 해설

코드 스플리팅의 핵심은 **"무엇을 분리했는가"보다 "왜 이것을 분리하면 효과가 큰가"**입니다. 마크다운 라이브러리가 200KB라는 구체적 수치와, 홈 페이지에서는 불필요하다는 판단 근거를 함께 제시해야 합니다.

## Part 4. 성능 최적화

### Q11. @tanstack/react-virtual로 메시지 리스트를 가상화한 이유와 구현 방식을 설명해주세요.

#### 합격 답변 예시

채팅 메시지가 수백 개 이상 쌓이면 DOM 노드 수가 그에 비례하여 증가하고, 스크롤 성능과 메모리 사용량에 문제가 생깁니다. `@tanstack/react-virtual`을 사용해 뷰포트에 보이는 메시지만 실제 DOM에 렌더링합니다.

구현에서 주의한 점은:
1. **동적 높이**: 메시지마다 길이가 다르므로 `estimateSize` 함수로 초기 높이를 추정하고, 실제 렌더링 후 `measureElement`로 정확한 높이를 측정
2. **스크롤 위치 유지**: 새 메시지 도착 시 자동 스크롤, 사용자가 위로 스크롤한 상태에서는 유지
3. **테스트 환경**: jsdom에서는 레이아웃 계산이 불가능하므로 `setup.ts`에서 모든 아이템을 반환하는 mock을 적용

#### 상세 해설

가상화는 "왜 필요한가"(DOM 노드 수 → 성능)와 "어떤 어려움이 있었는가"(동적 높이, 테스트 환경)를 함께 설명해야 합니다. 특히 jsdom mock 부분은 테스트 전략과도 연결되는 좋은 포인트입니다.

### Q12. React.memo를 어떤 컴포넌트에 적용했고, 어떤 기준으로 결정했나요?

#### 합격 답변 예시

다음 컴포넌트에 `React.memo`를 적용했습니다:
- **Chat**: 프리뷰 상태 변경 시 불필요한 리렌더링 방지
- **Preview**: 채팅 메시지 추가 시 불필요한 리렌더링 방지
- **ChatMessage**: 개별 메시지는 한번 렌더링되면 내용이 변하지 않음
- **MessageInput**: 스트리밍 중에도 입력 필드가 리렌더링되면 안 됨
- **StyleSelector**: 스타일 변경 외에는 리렌더링 불필요

기준은 **"부모가 자주 리렌더링되는데, 이 자식의 props는 변하지 않는가?"**입니다. 예를 들어 ChatSessionPage는 스트리밍 중 매 이벤트마다 리렌더링되지만, MessageInput의 props는 변하지 않으므로 memo가 효과적입니다.

무분별한 memo 적용은 오히려 비교 비용을 증가시키므로, props가 자주 변하는 컴포넌트에는 적용하지 않았습니다.

#### 상세 해설

memo의 핵심은 **"언제 쓰면 안 되는지"**를 아는 것입니다. "모든 컴포넌트에 memo를 걸었다"는 답변은 오히려 감점입니다. SSE 스트리밍이라는 맥락에서 부모가 초당 여러 번 리렌더링되는 상황을 연결하면 설득력이 높아집니다.

### Q13. Vite의 manualChunks 설정을 route-based code splitting 대신 사용한 이유는?

#### 합격 답변 예시

## Part 5. 에러 핸들링

### Q14. API 에러 정규화(normalizeApiError)를 설명해주세요. 왜 필요한가요?

#### 합격 답변 예시

서버가 보내는 에러 응답 형식이 일관되지 않습니다:
- 400 Bad Request: `{ error: "Invalid request" }`
- 500 Internal Error: `{ message: "Server failed" }`
- 네트워크 에러: `Error` 객체
- 기타: 문자열, 알 수 없는 객체

`normalizeApiError`는 이 모든 케이스를 하나의 문자열로 정규화합니다:

```typescript
function normalizeApiError(error: unknown, fallback?: string): string {
  if (typeof error === 'string') return error;
  if (error instanceof Error) return error.message;
  if (typeof error === 'object' && error !== null) {
    if ('error' in error) return String((error as any).error);
    if ('message' in error) return String((error as any).message);
  }
  return fallback ?? '알 수 없는 오류가 발생했습니다.';
}
```

이렇게 하면 UI에서는 항상 `string` 타입의 에러 메시지만 다루면 되고, 에러 포맷에 대한 분기 로직이 컴포넌트에 침투하지 않습니다.

#### 상세 해설

이 패턴은 Hard 테스트의 핵심 요구사항입니다. **"서버가 보내는 에러 포맷을 신뢰하지 말라"**는 방어적 프로그래밍 원칙을 보여주는 좋은 예입니다. 실제 프로덕션에서도 외부 API의 에러 포맷은 문서와 다른 경우가 많습니다.

### Q17. jsdom에서 matchMedia와 react-virtual을 mock한 이유와 방법은?

#### 합격 답변 예시

jsdom은 순수 DOM 시뮬레이터이므로 두 가지 핵심 기능이 없습니다:

**1. matchMedia 부재:**
```typescript
// setup.ts
Object.defineProperty(window, 'matchMedia', {
  value: (query: string) => ({
    matches: false,
    media: query,
    addEventListener: vi.fn(),
    removeEventListener: vi.fn(),
    // ...
  }),
});
```
우리 앱의 `useBreakpoint` 훅이 `matchMedia`에 의존하므로, 이 polyfill 없이는 모든 컴포넌트 테스트가 실패합니다.

**2. react-virtual 레이아웃 계산 불가:**
jsdom에서는 요소의 `getBoundingClientRect`가 항상 0을 반환합니다. react-virtual은 실제 높이를 기반으로 가상화하므로, 테스트에서는 모든 아이템을 렌더링하는 mock으로 교체했습니다.

이런 mock의 트레이드오프는 실제 브라우저 동작과 차이가 있다는 점입니다. 따라서 가상화 관련 버그(예: 스크롤 위치 오차)는 E2E 테스트에서 잡아야 합니다.

#### 상세 해설

Mock 사용 시 반드시 **트레이드오프**를 언급해야 합니다. "Mock했으니 문제 해결"이 아니라 "Mock으로 커버하지 못하는 영역이 무엇이고, 어떻게 보완하는가"까지 말하면 테스트 전략의 성숙도를 보여줍니다.

구체적으로 AI 코드를 수정한 사례:
- Container.tsx 삭제 → AI가 만든 재사용 컴포넌트인데 실제로는 각 페이지마다 max-width가 달라서 오히려 방해
- `.catch(() => {})` 제거 → 내부적으로 try-catch가 이미 있어서 불필요한 방어 코드
- 전체 코드 품질 리뷰 세션 진행 → 50개 이상 작업 세션의 결과물을 일괄 검토

#### 상세 해설

"AI가 만들었으니 신뢰한다"는 최악의 답변입니다. **삭제하거나 수정한 구체적 사례**가 있어야 "도구를 주체적으로 활용한다"는 인상을 줍니다.

## Part 9. 서버 사이드와 API

### Q24. Hono.js를 선택한 이유는? Express와의 차이점은?

#### 합격 답변 예시

Hono.js는 과제 boilerplate에 포함된 서버 프레임워크입니다. Express와 비교했을 때의 장점은:

1. **경량**: 번들 크기가 Express의 1/10 수준
2. **TypeScript 네이티브**: 타입 추론이 뛰어나서 추가 @types 패키지 불필요
3. **Web Standards 기반**: Request/Response API 사용 → Cloudflare Workers 등 엣지 런타임 호환
4. **SSE 지원**: ReadableStream 기반으로 SSE 구현이 자연스러움

서버에서 제가 추가한 주요 작업은:
- `max_tokens` 설정 (OpenRouter 무료 티어의 402 에러 방지)
- `style` 쿼리 파라미터 처리 (답변 스타일 시스템 프롬프트 주입)
- 아티팩트 추출 로직 (코드 펜스 파싱 → artifact_delta 이벤트 분리)

#### 상세 해설

Hono는 boilerplate에 있었으므로 "선택 이유"보다 **"어떤 부분을 직접 수정/추가했는가"**가 더 중요합니다. max_tokens 설정이 실제 운영 이슈(402 에러)를 해결한 사례는 좋은 포인트입니다.

### Q25. OpenRouter API의 max_tokens 문제를 어떻게 발견하고 해결했나요?

#### 합격 답변 예시

OpenRouter는 기본 max_tokens가 65535인데, 무료 티어에서는 이 값이 허용 한도를 초과하여 402 Payment Required 에러가 발생했습니다.

발견 과정:
1. 로컬에서는 시나리오 기반 fallback으로 동작하여 문제 없음
2. OpenRouter API 키를 설정하고 실제 AI 응답을 테스트할 때 402 에러 발생
3. OpenRouter 문서 확인 → 무료 모델의 토큰 한도 확인

해결 방법:
```typescript
const DEFAULT_OPENROUTER_MAX_TOKENS = 4096;

function getOpenRouterMaxTokens(): number {
  const rawValue = process.env.OPENROUTER_MAX_TOKENS;
  const parsedValue = Number.parseInt(rawValue || "", 10);
  return Number.isInteger(parsedValue) && parsedValue > 0
    ? parsedValue
    : DEFAULT_OPENROUTER_MAX_TOKENS;
}
```

환경 변수로 오버라이드 가능하게 만들어서, 유료 티어에서는 더 큰 값을 설정할 수 있습니다.

#### 상세 해설

이 질문은 **"디버깅 과정"**을 설명하는 좋은 예입니다. "문제 발견 → 원인 분석 → 해결 → 확장성 고려"의 구조로 답하면 문제 해결 능력을 보여줄 수 있습니다.

## Part 10. 심화 질문

### Q26. 이 프로젝트에서 가장 어려웠던 기술적 챌린지는 무엇이었나요?

#### 합격 답변 예시

가장 어려웠던 것은 **SSE 스트리밍 중 상태 관리의 일관성 유지**였습니다.

SSE 이벤트는 비동기적으로 도착하고, 각 이벤트마다 상태를 업데이트해야 합니다. 이때 여러 문제가 동시에 발생합니다:
1. 이벤트 순서 보장: text_delta와 artifact_delta가 교차로 오면?
2. 상태 경합: 사용자가 스트리밍 중에 새 메시지를 보내면?
3. 에러 복원: 중간에 malformed JSON이 오면 상태가 꼬이지 않는가?

이를 해결하기 위해 모든 상태 변화를 reducer의 명시적 액션으로 통합했습니다. 16개 액션 타입 각각이 하나의 상태 전이를 명확하게 정의하므로, 어떤 순서로 이벤트가 와도 상태가 일관성을 유지합니다.

#### 상세 해설

"어려웠다"만 말하면 안 됩니다. **무엇이 어려웠고, 왜 어려웠으며, 어떻게 해결했는가**의 구조로 답해야 합니다. reducer로 상태 전이를 명시화한 것이 해결의 핵심이라는 점을 강조하세요.

### Q27. 이 프로젝트를 프로덕션에 배포한다면 추가로 필요한 것은?

#### 합격 답변 예시

현재 과제에서 다루지 않은 프로덕션 요구사항은:

**안정성:**
- 스트림 취소 기능 ("답변 중지" 버튼)
- SSE 재연결 로직 (네트워크 불안정 시)
- 에러 바운더리 (컴포넌트 레벨 크래시 복구)

**보안:**
- API 인증/인가 (현재 없음)
- Rate limiting
- XSS 방지 (마크다운 렌더링 시 sanitize)

**성능:**
- 세션 데이터 캐싱 (현재 `cache: "no-store"`)
- 이미지/미디어 최적화
- CDN 배포

**사용성:**
- 메시지 재생성
- 대화 검색
- 다크 모드
- 키보드 단축키 확장

프로덕션의 핵심은 "기능 추가"보다 **"장애 시 사용자 경험을 어떻게 보호하는가"**입니다.

#### 상세 해설

이 질문은 **"과제와 프로덕션의 차이를 이해하는가"**를 봅니다. 기능 나열보다 "가장 우선순위가 높은 것이 무엇이고 왜인가"를 말하는 것이 더 강합니다. 보안과 안정성을 먼저 언급하면 좋은 인상을 줍니다.

### Q28. PubSub 패턴으로 사이드바 업데이트를 처리한 이유는?

#### 합격 답변 예시

채팅 세션 페이지에서 새 메시지를 보내면 사이드바의 세션 목록도 업데이트되어야 합니다. 하지만 ChatSessionPage와 Sidebar는 서로 다른 컴포넌트 트리에 있어서 직접적인 props 전달이 불가능합니다.

해결 방법으로 세 가지를 고려했습니다:
1. **글로벌 상태(Redux/Zustand)**: 과도한 설정, 이 하나의 이벤트를 위해 도입하기엔 무거움
2. **Context API**: 가능하지만 불필요한 리렌더링 발생
3. **PubSub 이벤트**: 가장 가볍고, 구독/해제가 명확

`session-list.ts`에 간단한 이벤트 시스템을 구현했습니다:
- `emitSessionUpdate()`: 세션 변경 시 이벤트 발행
- `onSessionUpdate(callback)`: 사이드바에서 구독하여 목록 갱신

이 패턴은 두 컴포넌트 간 느슨한 결합을 유지하면서도 실시간 동기화를 달성합니다.

#### 상세 해설

PubSub은 "왜 다른 방법을 쓰지 않았는가"를 비교 형식으로 설명할 때 강합니다. **세 가지 대안을 비교하고 판단 기준을 제시**하는 것이 핵심입니다.

### Q29. 이 코드베이스에서 기술 부채라고 생각하는 부분은?

#### 합격 답변 예시

솔직하게 말하면 세 가지가 있습니다:

1. **인메모리 세션 저장소**: 서버 재시작 시 모든 데이터 소실. 과제 특성상 의도된 것이지만, 프로덕션에서는 DB 연동 필수
2. **타입 안전성 일부 미비**: `normalizeApiError`에서 `as any` 타입 단언 사용. 런타임 안전하지만 TypeScript 순수성 측면에서 개선 가능
3. **E2E 테스트 부재**: jsdom 기반 테스트만 있어서, 실제 브라우저 동작(스크롤, 드래그, 레이아웃)은 검증되지 않음

하지만 2일이라는 시간 제약 안에서 이 부분들은 **의도적인 트레이드오프**였습니다. 우선순위는 "기능 완성 + 104개 테스트 통과 + 코드 품질"이었고, 이 기준에서는 올바른 판단이었다고 생각합니다.

#### 상세 해설

기술 부채 질문은 **자기 인식과 우선순위 판단력**을 봅니다. "없다"는 최악의 답변이고, 부채를 나열만 하는 것도 부족합니다. **"알고 있었지만 의도적으로 후순위로 뒀고, 그 이유는..."**이 가장 좋은 구조입니다.

### Q30. 같은 과제를 다시 한다면 다르게 할 부분은?

#### 합격 답변 예시

세 가지를 다르게 하겠습니다:

1. **처음부터 useReducer로 시작**: 처음에는 useState 여러 개로 시작했다가 Day 2에서 useReducer로 리팩토링했습니다. 상태가 복잡해질 것을 예측했다면 처음부터 reducer로 시작하는 것이 리팩토링 비용을 줄였을 것입니다.

2. **E2E 테스트 일부라도 추가**: Playwright로 핵심 플로우(세션 생성 → 스트리밍 → 프리뷰 표시) 한 개만이라도 작성했을 것입니다. jsdom mock의 한계를 보완하는 데 큰 도움이 됩니다.

3. **AI에 대한 의존도 조절**: 초반에 AI가 생성한 Container.tsx 같은 불필요한 추상화를 빠르게 제거하지 못했습니다. AI 코드를 수용하기 전에 "이것이 정말 필요한가?"를 더 빨리 질문했을 것입니다.

핵심은 **"더 많이 하겠다"가 아니라 "더 일찍 올바른 판단을 하겠다"**입니다.

#### 상세 해설

이 질문은 **성장 가능성**을 봅니다. 구체적인 사례 + 개선 방향 + 그로부터 배운 원칙의 구조가 가장 좋습니다. "모든 것을 완벽하게 하겠다"는 비현실적이므로, 가장 임팩트가 큰 1-3가지에 집중하세요.
