# 챕터 5. React 렌더링 모델에서 상태 구조 설계까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **React는 화면을 직접 조작하는 도구가 아니라, 상태를 기반으로 UI를 다시 계산하는 시스템이라는 관점으로 이해해야 한다.**

### 전체 흐름 맵

```
선언형 모델 (UI = f(state))
  → 렌더 / 커밋 (계산과 반영의 분리)
    → Virtual DOM / Fiber (변경 감지와 우선순위 스케줄링)
      → 재조정 (무엇을 유지하고 무엇을 교체할지)
        → state ownership (누가 상태를 소유하나)
          → Context (props drilling 없이 값 전달)
            → Zustand / Redux (구독 제어와 부분 구독)
              → client state vs server state (역할 분리)
                → React Query (서버 상태 전용 도구)
                  → useReducer (복잡한 상태 전환)
                    → concurrent features (렌더링 우선순위)
                      → 커스텀 훅 (로직 재사용의 기본 단위)
                        ├─ controlled vs uncontrolled (입력 제어 전략)
                        └─ 에러 바운더리 (실패 격리)
                      → 복잡한 UI의 상태 설계
```

## 43. React를 왜 선언형이라고 부르나

> **핵심 요약** — React는 "어떻게 바꿔라"가 아니라 "이 상태일 때 이런 UI가 나와야 한다"고 선언하는 모델이다. `UI = f(state)` — 상태가 바뀌면 UI를 다시 계산한다.

### 명령형 vs 선언형

```js
// 명령형: DOM을 직접 단계별로 조작
const p = document.getElementById("count-display");
const btn = document.getElementById("increment-btn");
let count = 0;

btn.addEventListener("click", () => {
  count++;
  p.textContent = `현재 값: ${count}`; // 직접 DOM을 찾아서 바꾼다
});
```

```tsx
// 선언형: 상태에 따라 어떤 UI가 나와야 하는지 기술
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>현재 값: {count}</p>
      <button onClick={() => setCount((prev) => prev + 1)}>증가</button>
    </div>
  );
}
// "p 태그를 찾아서 텍스트를 바꿔라"가 아니라
// "count가 이 값일 때 이런 UI다"를 선언한다
```

### UI = f(state)

이 공식이 React의 핵심 사고 모델이다.

| | 명령형 (jQuery, Vanilla) | 선언형 (React) |
|---|---|---|
| **관점** | "이 DOM 요소를 이렇게 바꿔라" | "이 상태일 때 UI는 이렇다" |
| **상태-UI 관계** | 개발자가 직접 동기화 | React가 자동 계산 |
| **버그 원인** | 상태와 DOM이 어긋남 | 상태 자체의 문제 |
| **디버깅** | DOM을 추적 | 상태를 추적 |

### 실무 에피소드 — 이런 상황에서 만난다

> jQuery로 만든 레거시 탭 UI에서 "탭은 B인데 내용은 A" 같은 버그가 반복됐다. 탭 상태와 DOM 업데이트가 여러 곳에 흩어져 있어서 동기화가 깨진 것. React로 마이그레이션하면서 `activeTab` state 하나로 탭 UI와 콘텐츠를 선언하니 상태-UI 불일치 버그가 사라짐.

### 자주 하는 오해

- 선언형이면 내부적으로 아무 일도 안 한다 → **틀림.** React는 Virtual DOM 비교, DOM 조작을 모두 수행한다. 개발자가 직접 안 할 뿐.
- React는 DOM을 안 만진다 → **틀림.** 계산된 결과를 바탕으로 최소한의 DOM 변경을 수행한다.
- 선언형이면 성능이 더 좋다 → **항상 그런 건 아니다.** 선언형의 장점은 성능보다 **예측 가능성과 유지보수성**이다.

### 면접 압축 답변

React는 DOM을 직접 조작하는 게 아니라, **현재 상태에서 어떤 UI가 나와야 하는지를 선언**하는 모델이에요. 공식으로 표현하면 `UI = f(state)`인데, 명령형 방식처럼 DOM을 직접 찾아서 바꾸는 게 아니라 **상태와 UI의 대응 관계를 기술**하는 거죠. 상태가 바뀌면 React가 알아서 UI를 다시 계산하고, 이전 결과와 비교해서 **실제로 달라진 DOM만 최소한으로 반영**합니다. 이 덕분에 상태와 화면이 어긋나는 버그가 크게 줄어들고, 디버깅할 때도 DOM이 아니라 상태만 추적하면 됩니다.

> React가 선언형이라는 건 알았다. **상태가 바뀌었을 때 React는 내부에서 어떤 단계를 거쳐 UI를 바꾸는가?**

## 44. 렌더와 커밋, 그리고 Fiber 아키텍처

> **핵심 요약** — React는 렌더 단계(UI 계산)와 커밋 단계(DOM 반영)를 분리한다. Fiber 아키텍처 덕분에 렌더링을 잘게 쪼개고 중단/재개할 수 있어서 concurrent features가 가능하다.

### 렌더 단계 vs 커밋 단계

```
상태 변경
    ↓
┌──────────────────────────────────────────────────┐
│ [렌더 단계] — 순수 계산, 부수효과 금지            │
│  컴포넌트 함수 실행 → 새 JSX 생성 → diffing       │
│  ※ React 18에서는 중단/재개 가능 — 시간 분할(time-slicing)   │
└──────────────────────────────────────────────────┘
    ↓
┌──────────────────────────────────────────────────┐
│ [커밋 단계] — 실제 DOM 변경                       │
│  변경된 DOM 노드만 업데이트                        │
│    → useLayoutEffect (DOM 변경 직후, 동기)        │
│      → 브라우저 페인트                             │
│        → useEffect (페인트 이후, 비동기)           │
└──────────────────────────────────────────────────┘
```

> 시간 분할(time-slicing)이란 렌더링 작업을 작은 조각으로 나눠서, 각 조각 사이에 브라우저가 사용자 입력을 처리할 기회를 주는 방식이다.

### Virtual DOM과 Fiber

> DOM(Document Object Model)은 브라우저가 HTML을 해석해서 만드는 화면 요소의 트리 구조다. JavaScript로 이 트리를 직접 조작하면 화면이 바뀐다.

Virtual DOM은 UI 구조를 **메모리 안의 JS 객체 트리**로 표현한 것이다.

```jsx
// 이 JSX는 내부적으로 JS 객체 트리가 된다
<div>
  <h1>Hello</h1>
  <p>World</p>
</div>

// → { type: 'div', props: { children: [
//     { type: 'h1', props: { children: 'Hello' } },
//     { type: 'p', props: { children: 'World' } }
//   ]}}
```

React 16부터는 이 트리를 **Fiber**라는 구조로 관리한다.

| | Stack Reconciler (React 15 이전) | Fiber Reconciler (React 16+) |
|---|---|---|
| **순회 방식** | 재귀 (한 번 시작하면 끝까지) | linked list (한 노드씩 처리) |
| **중단/재개** | 불가능 | 가능 (time-slicing) |
| **우선순위** | 없음 (모든 업데이트 동등) | 있음 (urgent vs non-urgent) |
| **concurrent features** | 불가능 | 가능 |

각 Fiber 노드가 가지는 정보:

```
Fiber 노드
├── type: 컴포넌트 함수 또는 DOM 태그 ('div', MyComponent)
├── props: 전달된 속성
├── child / sibling / return: 자식, 형제, 부모 포인터
├── stateNode: 실제 DOM 노드 또는 컴포넌트 인스턴스
├── memoizedState: 훅 상태 체인 (linked list)
└── effectTag: 커밋 시 수행할 DOM 조작 (삽입/갱신/삭제)
```

Fiber 덕분에 한 노드 처리 후 **브라우저에 제어권을 돌려줄 수 있다.** 사용자 입력(urgent)이 들어오면, 무거운 리스트 렌더링(non-urgent)을 잠시 멈추고 입력을 먼저 처리한 뒤 나머지를 이어간다.

### 재조정 — 무엇을 유지하고 무엇을 교체하나

React는 두 가지 휴리스틱으로 O(n) diffing을 수행한다:

> O(n)이란 노드 수에 비례하는 시간이 걸린다는 뜻으로, 트리 전체를 딱 한 번만 순회하면 된다는 의미다.

1. **타입이 다르면 전체 교체**: `<div>` → `<span>`이면 하위 트리를 버리고 새로 만든다 (state 초기화)
2. **같은 타입의 리스트 항목은 key로 구분**: 어떤 항목이 같은 존재인지 `key`로 판단

```tsx
// 타입이 바뀌면 — 전체 교체 (state도 초기화)
{showInput ? <input /> : <textarea />}

// 같은 타입이 유지되면 — state 보존, props만 업데이트
{showInput ? <input type="text" /> : <input type="password" />}
// 둘 다 <input>이므로 React는 같은 요소로 보고 속성만 바꾼다
```

> 참고: 리스트에서 key가 정체성을 어떻게 결정하는지, index key의 문제와 key 리셋 패턴은 **챕터 2 섹션 20**에서 상세히 다뤘다.

### 실무 에피소드 — 이런 상황에서 만난다

> 모달에서 "신규 생성"과 "수정" 모드를 같은 `<Form />` 컴포넌트로 전환했는데, 수정 모드에서 신규로 바꿔도 이전 입력값이 남아있는 버그. 원인은 같은 타입(`Form`)이라 React가 state를 보존한 것. `<Form key={mode} />`로 key를 바꿔서 모드 전환 시 state 초기화.

### 자주 하는 오해

- 리렌더가 일어나면 무조건 DOM도 다시 그린다 → **틀림.** 렌더 결과가 이전과 같으면 DOM 변경 없음.
- Virtual DOM이 있으니 무조건 빠르다 → **틀림.** 비교 비용이 있다. 핵심은 속도보다 **선언형 UI를 가능하게 하는 추상화**.
- Fiber는 그냥 Virtual DOM의 다른 이름이다 → **다르다.** Fiber는 중단/재개와 우선순위 스케줄링을 위한 구조.

### 면접 압축 답변

React는 **렌더 단계(UI 계산)**와 **커밋 단계(DOM 반영)**를 분리해요. 렌더 단계에서 컴포넌트 함수를 실행해 새 Virtual DOM을 만들고, 이전 결과와 **O(n) diffing**으로 비교한 뒤, 커밋 단계에서 실제로 변경된 DOM만 반영합니다. 여기서 핵심이 **Fiber 아키텍처**인데, 각 노드를 하나의 작업 단위로 쪼개서 렌더링을 **중단하고 재개**할 수 있게 해줘요. 덕분에 **time-slicing**과 우선순위 스케줄링이 가능합니다. 재조정 규칙은 간단한데, **타입이 다르면 하위 트리를 통째로 교체**하고, 같은 타입의 리스트 항목은 **key로 정체성을 추적**합니다.

> 이제 React의 렌더링 모델을 봤다. **상태는 어떤 위치에 둬야 리렌더 범위와 일관성을 안정적으로 관리할 수 있을까?**

## 45. state ownership과 상태 끌어올리기

> **핵심 요약** — "이 상태를 누가 소유하나?"가 React 구조 설계의 핵심 질문이다. 공유가 필요하면 가장 가까운 공통 부모로 끌어올리되, 불필요하게 높이 올리면 리렌더 범위가 넓어진다.

### 핵심 개념

```tsx
// Bad: 같은 의미의 상태를 각각 관리 — 쉽게 어긋남
function Sidebar() {
  const [selectedId, setSelectedId] = useState(null);
}
function Detail() {
  const [selectedId, setSelectedId] = useState(null);
}

// Good: 소유자를 하나로 정하고 나머지는 props로 받는다
function Parent() {
  const [selectedId, setSelectedId] = useState(null);
  return (
    <>
      <Sidebar selectedId={selectedId} onSelect={setSelectedId} />
      <Detail selectedId={selectedId} />
    </>
  );
}
```

### 상태 위치를 정하는 판단 기준

| 질문 | 로컬에 둘 때 | 끌어올릴 때 | 전역/외부 도구 |
|---|---|---|---|
| 누가 읽나? | 이 컴포넌트만 | 형제/부모도 읽음 | 어디서든 읽음 |
| 누가 바꾸나? | 이 컴포넌트만 | 형제가 바꿔야 함 | 여러 곳에서 바꿈 |
| 서버 데이터인가? | 아님 | 아님 | → React Query 등 |
| 변경 빈도 | - | 높으면 리렌더 범위 주의 | selector로 부분 구독 |

### 상태 위치에 따른 리렌더 범위

```
❌ state를 너무 높이 둔 경우:
    App [state: filter] ← state 변경 시
    ├── Header        ← 리렌더 (불필요)
    ├── Sidebar       ← 리렌더 (불필요)
    └── ProductList   ← 리렌더 (필요)
        └── Product   ← 리렌더 (필요)

✅ state를 적절한 위치에 둔 경우:
    App
    ├── Header
    ├── Sidebar
    └── ProductList [state: filter] ← state 변경 시
        └── Product   ← 리렌더 (필요)
    → Header, Sidebar는 리렌더되지 않음!
```

### 과도한 끌어올리기의 문제

```tsx
// 너무 높이 올린 케이스
function App() {
  const [searchQuery, setSearchQuery] = useState("");
  const [selectedTab, setSelectedTab] = useState("home");
  const [modalOpen, setModalOpen] = useState(false);
  const [cartItems, setCartItems] = useState([]);
  // ... 10개 이상의 state

  // searchQuery가 바뀔 때마다 App 전체(Header, Main, Footer, Modal)가 리렌더
  return (
    <>
      <Header query={searchQuery} onSearch={setSearchQuery} />
      <TabBar tab={selectedTab} onSelect={setSelectedTab} />
      <Main tab={selectedTab} cart={cartItems} />
      <Modal open={modalOpen} />
      <Footer />
    </>
  );
}
```

해결: **state를 사용하는 가장 가까운 곳으로 내린다.**

```tsx
function App() {
  return (
    <>
      <SearchHeader />    {/* searchQuery는 여기 안에 */}
      <TabSection />       {/* selectedTab은 여기 안에 */}
      <CartProvider>       {/* cartItems는 필요한 범위만 */}
        <Main />
      </CartProvider>
      <Footer />
    </>
  );
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 이커머스 상품 페이지에서 "최근 본 상품" 사이드바와 "상품 상세" 메인 영역이 같은 `currentProductId`를 쓰는데, 각각 로컬 state로 관리하니 뒤로가기 시 사이드바만 이전 상품을 보여주는 버그. 공통 부모인 `ProductPage`로 state를 끌어올려서 해결.

### 자주 하는 오해

- 공유되면 무조건 전역 상태다 → **틀림.** 공통 부모에 두는 것만으로 충분한 경우가 많다.
- 가장 상위에 두면 관리가 쉽다 → **리렌더 범위가 불필요하게 넓어진다.**
- 로컬 상태는 작고 단순한 경우에만 쓴다 → **오히려 많은 UI 상태는 로컬이 더 적합하다.**

### 면접 압축 답변

**State ownership**은 어떤 상태를 어느 컴포넌트가 책임질지 정하는 문제예요. 형제 컴포넌트가 같은 상태를 각각 관리하면 쉽게 어긋나기 때문에, 공유가 필요하면 **가장 가까운 공통 부모**가 소유하고 props로 내려주는 게 안전합니다. 다만 무조건 높이 올리면 되는 게 아니라, 너무 높이 두면 **리렌더 범위가 불필요하게 넓어져요**. 그래서 **"누가 읽고 누가 변경하는가"**를 기준으로 가장 가까운 적절한 위치에 두는 판단이 중요합니다.

> 상태 끌어올리기를 봤다. **너무 깊은 트리에서 props를 계속 내려줘야 하면?** Context로 넘어간다.

## 46. Context는 언제 유용하고, 언제 과용되기 쉬운가

> **핵심 요약** — Context는 props drilling을 줄여주는 값 전달 도구다. 단, Context 값이 바뀌면 모든 구독자가 리렌더되므로 변경 빈도가 낮은 값에 적합하고, 관심사별로 분리해야 한다.

### 핵심 개념

```tsx
const ThemeContext = createContext<"light" | "dark">("light");

function App() {
  const [theme, setTheme] = useState<"light" | "dark">("light");
  return (
    <ThemeContext.Provider value={theme}>
      <Header />
      <Main />
    </ThemeContext.Provider>
  );
}

function DeepChild() {
  const theme = useContext(ThemeContext); // props drilling 없이 접근
  return <div className={theme}>...</div>;
}
```

### Context의 리렌더 특성 — 가장 중요한 포인트

Context 값이 바뀌면 `useContext`로 구독하는 **모든 컴포넌트가 리렌더**된다. `React.memo`도 막지 못한다.

```tsx
// Bad: 관련 없는 상태를 한 Context에 묶음
const AppContext = createContext({
  theme: "light",
  user: null,
  locale: "ko",
});
// theme만 바뀌어도 user만 쓰는 컴포넌트도 리렌더!

// Good: 관심사별로 Context를 분리
const ThemeContext = createContext("light");
const UserContext = createContext(null);
const LocaleContext = createContext("ko");
```

### Context + useMemo — value 참조 안정화

Provider의 `value`에 매번 새 객체를 넘기면 불필요한 리렌더가 생긴다.

```tsx
// Bad: 매 렌더마다 새 객체 → 모든 구독자 리렌더
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("light");
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// Good: useMemo로 value 안정화
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("light");
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}
```

### Context가 적합한 것 vs 부적합한 것

| 적합 | 부적합 |
|---|---|
| 테마, 로케일, 사용자 정보 | 빈번히 변하는 폼 입력값 |
| 인증 상태 | 드래그 좌표 |
| 기능 플래그 | 검색 결과 리스트 |
| 변경 빈도 낮음, 구독자 많음 | 변경 빈도 높음, 부분 구독 필요 |

### 실무 에피소드 — 이런 상황에서 만난다

> 다국어 지원을 위해 `{ locale, translations, setLocale }`을 하나의 Context에 넣었더니, 번역 텍스트 하나를 바꿀 때마다 앱 전체가 리렌더됐다. `LocaleContext`(locale + setLocale)와 `TranslationsContext`(번역 데이터)를 분리하고, translations는 locale이 바뀔 때만 갱신되게 구조를 바꿔서 해결.

### 자주 하는 오해

- Context만 쓰면 상태관리 문제는 끝난다 → **틀림.** Context는 "값 전달"이지 "상태관리"가 아니다.
- Context를 쓰면 성능 걱정은 없다 → **값 변경 시 모든 구독자가 리렌더된다.**
- 모든 공유 상태를 Context에 넣어야 한다 → **빈번히 변경되는 상태는 외부 도구가 더 적합하다.**

### 면접 압축 답변

**Context**는 **props drilling을 줄여주는 값 전달 도구**인데, 가장 중요한 특성은 값이 바뀌면 `useContext`로 구독하는 **모든 컴포넌트가 리렌더**된다는 거예요. 그래서 테마, 로케일, 인증처럼 **변경 빈도가 낮은 공통 값**에 잘 맞고, 관련 없는 상태를 하나의 Context에 묶으면 불필요한 리렌더가 생기니까 **관심사별로 분리**해야 합니다. Provider의 value에 매번 새 객체를 넘기면 매 렌더마다 구독자가 리렌더되니 `useMemo`로 참조를 안정화해야 하고, 빈번히 바뀌는 상태나 **부분 구독**이 필요한 경우는 Zustand 같은 외부 도구가 더 적합합니다.

> Context의 한계를 봤다. **그렇다면 Zustand/Redux 같은 외부 도구는 무엇이 다른가?**

## 47. Context와 Zustand/Redux는 어떤 문제를 다르게 푸나

> **핵심 요약** — Context는 "값 전달", Zustand/Redux는 "상태 저장 + 구독 제어". 가장 큰 차이는 **selector로 부분 구독**이 가능한지 여부다.

### 핵심 차이

| | Context | Zustand | Redux (Toolkit) |
|---|---|---|---|
| **본질** | 값 전달 메커니즘 | 경량 상태 저장소 | 상태 + 액션 흐름 관리 |
| **부분 구독** | 불가 (전체 리렌더) | selector로 가능 | selector로 가능 |
| **미들웨어** | 없음 | persist, devtools 등 | 풍부한 미들웨어 생태계 |
| **보일러플레이트** | 최소 | 최소 | 중간 (Toolkit으로 개선) |
| **DevTools** | React DevTools | Zustand devtools | Redux DevTools (시간 여행) |
| **적합한 상태** | 테마, 로케일, 인증 | 일반적인 클라이언트 상태 | 엄격한 액션 추적이 필요할 때 |

```tsx
// Context: theme이 바뀌면 user만 쓰는 컴포넌트도 리렌더
const { user } = useContext(AppContext);

// Zustand: selector로 user만 구독 → theme이 바뀌어도 리렌더 안 됨
const user = useStore((state) => state.user);
```

### Zustand의 selector — 부분 구독의 핵심

```tsx
import { create } from "zustand";

const useStore = create((set) => ({
  user: null,
  theme: "light",
  setUser: (user) => set({ user }),
  setTheme: (theme) => set({ theme }),
}));

// 이 컴포넌트는 user가 바뀔 때만 리렌더
function UserAvatar() {
  const user = useStore((state) => state.user);
  return <img src={user?.avatar} />;
}

// 이 컴포넌트는 theme이 바뀔 때만 리렌더
function ThemeToggle() {
  const theme = useStore((state) => state.theme);
  const setTheme = useStore((state) => state.setTheme);
  return <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>{theme}</button>;
}
```

### Context vs Zustand — 리렌더 범위 비교

```
Context (value 변경 시):          Zustand (store 변경 시):
    Provider [value 변경]              Store { a: 1, b: 2 }
    ├── ChildA (useContext) 🔄         ├── ChildA (useStore(s=>s.a)) 🔄
    ├── ChildB (useContext) 🔄         ├── ChildB (useStore(s=>s.b)) ─ (b 안 바뀜)
    └── ChildC (useContext) 🔄         └── ChildC (useStore(s=>s.a)) 🔄
    → value의 일부만 바뀌어도               → selector로 구독한 값이
      모든 구독자가 리렌더                    실제로 바뀐 컴포넌트만 리렌더
```

### 상태관리 도구 선택 가이드

```
이 상태를 어떤 도구로 관리할까?
│
├── 서버에서 가져오는 데이터? → React Query / SWR
├── 변경 빈도가 낮고 어디서든 접근? → Context
├── 부분 구독이 필요하거나 변경이 잦음? → Zustand
├── 엄격한 액션 흐름과 시간 여행 디버깅? → Redux Toolkit
└── 이 컴포넌트 안에서만 쓰임? → useState / useReducer
```

### 실무 에피소드 — 이런 상황에서 만난다

> 에디터 앱에서 캔버스 상태(선택된 요소, 줌 레벨, 도구 모드)를 Context로 관리했는데, 마우스 이동 시 줌 레벨만 읽는 컴포넌트까지 전부 리렌더되어 60fps를 못 맞췄다. Zustand로 전환하고 각 컴포넌트가 필요한 slice만 selector로 구독하니 리렌더가 줄어서 성능 회복.

### 자주 하는 오해

- Zustand/Redux는 앱이 커야만 쓴다 → **부분 구독이 필요하면 작은 앱에서도 유용하다.**
- Context로도 되는데 외부 도구가 불필요하다 → **구독 제어에서 차이가 크다.**
- 외부 상태관리 도구를 쓰면 설계 고민이 사라진다 → **어떤 상태를 넣을지는 여전히 설계 문제.**

### 면접 압축 답변

Context는 **값 전달** 메커니즘이고, Zustand/Redux는 **상태 저장 + 구독 제어**에 초점이 있어요. 가장 큰 차이는 **selector로 부분 구독**이 가능한지인데, Context 값이 바뀌면 모든 구독자가 리렌더되는 반면 Zustand는 selector로 특정 slice만 구독해서 **불필요한 리렌더를 막을 수 있어요**. 실무에서 선택 기준을 정리하면, 서버 데이터는 **React Query**, 변경 빈도 낮은 공통값은 **Context**, 부분 구독이 필요하거나 변경이 잦으면 **Zustand**, 엄격한 액션 추적과 시간 여행 디버깅이 필요하면 **Redux Toolkit**이 적합합니다.

> 클라이언트 상태 구조를 봤다. **서버에서 받아오는 데이터도 같은 방식으로 다뤄야 할까?**

## 48. client state와 server state는 왜 구분해야 하나

> **핵심 요약** — client state는 브라우저 안에서 완결되는 UI 상태, server state는 서버와 동기화가 필요한 데이터다. 둘을 구분해야 어떤 도구로 관리할지 판단이 쉬워진다.

### 핵심 비교

| | client state | server state |
|---|---|---|
| **예시** | 모달 열림, 탭 선택, 드래그 좌표 | 유저 프로필, 게시글 목록, 검색 결과 |
| **소유자** | 브라우저 (클라이언트) | 서버 (원본은 DB) |
| **동기화** | 불필요 | 서버와 동기화 필요 |
| **캐싱** | 불필요 | 필요 (같은 데이터 중복 요청 방지) |
| **stale 문제** | 없음 | 있음 (서버가 바뀌었을 수 있음) |
| **적합한 도구** | useState, useReducer, Zustand | React Query, SWR |

### useState로 server state를 다루면 생기는 문제

```tsx
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [userId]);

  // 문제점:
  // 1. 매 컴포넌트마다 loading/error/data 3개 state 반복
  // 2. 캐싱 없음 — 같은 userId를 다시 열면 또 fetch
  // 3. stale 관리 없음 — 서버가 바뀌어도 모름
  // 4. 중복 요청 — 여러 컴포넌트가 같은 데이터 요청
  // 5. race condition 대응 — 직접 구현해야 함
}
```

### React Query로 바꾸면

```tsx
function UserProfile({ userId }) {
  const { data: user, isLoading, error } = useQuery({
    queryKey: ["user", userId],
    queryFn: () => fetch(`/api/users/${userId}`).then((r) => r.json()),
    staleTime: 5 * 60 * 1000,  // 5분간 fresh
  });
  // 캐싱, 중복 요청 제거, stale 관리, background refetch가 자동
}
```

### React Query 핵심 개념

| 개념 | 설명 |
|---|---|
| **staleTime** | 데이터가 fresh로 간주되는 시간 (기본 0) — 이 시간 동안은 refetch 안 함 |
| **gcTime** (구 cacheTime) | 비활성 데이터가 메모리에 유지되는 시간 (기본 5분) |
| **background refetch** | 컴포넌트 마운트, 윈도우 포커스 시 자동 재요청 |
| **invalidation** | mutation 후 관련 쿼리를 무효화해서 최신 데이터 fetch |
| **optimistic update** | 서버 응답 전에 UI를 먼저 업데이트하고, 실패 시 롤백 |

```tsx
// mutation 후 관련 데이터 무효화
const mutation = useMutation({
  mutationFn: updateUser,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ["user"] });
  },
});

// optimistic update 패턴
const mutation = useMutation({
  mutationFn: toggleLike,
  onMutate: async (postId) => {
    await queryClient.cancelQueries({ queryKey: ["post", postId] });
    const previous = queryClient.getQueryData(["post", postId]);
    queryClient.setQueryData(["post", postId], (old) => ({
      ...old, liked: !old.liked,
    }));
    return { previous };
  },
  onError: (err, postId, context) => {
    queryClient.setQueryData(["post", postId], context.previous); // 롤백
  },
});
```

### React Query 캐시 라이프사이클

```
시간 →  ────────────────────────────────────────────→

fetch 시작    응답 도착     staleTime 경과     refetch 트리거
   │             │              │                  │
   ▼             ▼              ▼                  ▼
[fetching] → [fresh ✅] → [stale ⚠️] ──────→ [fetching] → [fresh ✅]
                                    │
                                    └─ 컴포넌트가 이 데이터를 다시 쓸 때
                                       자동으로 백그라운드 refetch

언마운트 후:
[stale] → gcTime 경과 → [삭제 🗑️]
          (기본 5분)
```

### 실무 에피소드 — 이런 상황에서 만난다

> 게시판에서 글 작성 후 목록 페이지로 돌아왔는데 방금 쓴 글이 안 보이는 버그. 원인은 목록 데이터가 이전 캐시를 보여주고 있었기 때문. mutation `onSuccess`에서 `queryClient.invalidateQueries(["posts"])`를 호출해서 목록을 무효화하니 해결.

### 자주 하는 오해

- 서버에서 온 값도 다 같은 state다 → **캐싱, stale, 재요청이라는 추가 차원이 있다.**
- React Query가 있으면 Zustand가 필요 없다 → **역할이 다르다.** 서버 데이터는 React Query, UI 상태는 Zustand/useState.
- useEffect + fetch면 충분하다 → **규모가 커질수록 반복 코드와 누락이 많아진다.**

### 면접 압축 답변

**Client state**는 모달 열림, 탭 선택, 드래그 좌표처럼 **브라우저 안에서 완결되는 UI 상태**이고, **server state**는 서버에서 가져오는 데이터라서 **캐싱, stale 관리, background refetch, mutation 후 invalidation**이라는 추가 차원이 있어요. 둘을 구분해야 어떤 도구를 쓸지 명확해지는데, client state는 useState나 Zustand로, server state는 **React Query**로 관리하는 게 자연스럽습니다. React Query는 server state 패턴을 자동화해서 **staleTime/gcTime**으로 캐시 전략을 세우고, **optimistic update**로 체감 성능까지 제어할 수 있어요.

> 상태의 큰 축을 나눴다. 이제 복잡한 상태 전환을 다루는 **useReducer**, 렌더링 우선순위를 제어하는 **concurrent features**, 그리고 구조 설계의 나머지 축들로 넘어간다.

## 49. useReducer는 언제 useState 대신 쓰나

> **핵심 요약** — 다음 상태가 "이전 상태 + 액션"으로 결정되고, 여러 값이 함께 변하거나 전환 조건이 복잡할 때 useReducer가 useState보다 구조적이다.

### useState vs useReducer 판단 기준

| useState가 적합할 때 | useReducer가 적합할 때 |
|---|---|
| 독립적인 단일 값 (모달 열림, 입력값) | 여러 값이 함께 변하는 복합 상태 |
| 전환 로직이 단순 (토글, 증가) | 상태 전환에 조건 분기가 많을 때 |
| 상태 간 의존성이 없을 때 | 다음 상태가 이전 상태 + 액션 종류로 결정 |

### 실무 예시: 폼 + 검증 상태

```tsx
type FormState = {
  values: { name: string; email: string };
  errors: { name?: string; email?: string };
  isSubmitting: boolean;
};

type FormAction =
  | { type: "SET_FIELD"; field: string; value: string }
  | { type: "SUBMIT_START" }
  | { type: "SUBMIT_SUCCESS" }
  | { type: "SUBMIT_FAILURE"; errors: Record<string, string> };

function formReducer(state: FormState, action: FormAction): FormState {
  switch (action.type) {
    case "SET_FIELD":
      return {
        ...state,
        values: { ...state.values, [action.field]: action.value },
        errors: { ...state.errors, [action.field]: undefined }, // 입력 시 에러 클리어
      };
    case "SUBMIT_START":
      return { ...state, isSubmitting: true };
    case "SUBMIT_SUCCESS":
      return { ...state, isSubmitting: false };
    case "SUBMIT_FAILURE":
      return { ...state, isSubmitting: false, errors: action.errors };
  }
}
```

`SET_FIELD`에서 값 변경과 에러 클리어를 동시에 처리한다. useState 여러 개로 하면 `setValues`, `setErrors`를 매번 함께 호출해야 하고, 빠뜨리기 쉽다.

### 실무 에피소드 — 이런 상황에서 만난다

> 회원가입 폼에서 `useState` 5개(이름, 이메일, 비밀번호, 에러, 제출 중)를 관리하다가 필드 변경 시 에러를 클리어하는 로직을 빠뜨려서 "이미 수정했는데 에러가 안 사라져요" 버그. useReducer로 바꾸면서 SET_FIELD 액션에 에러 클리어를 함께 넣으니 누락 불가.

### 자주 하는 오해

- useReducer는 Redux의 축소판이다 → **비슷한 패턴이지만 미들웨어, 전역 스토어 개념이 없다.**
- 복잡한 앱에서만 쓴다 → **상태 2~3개가 함께 바뀌는 경우에도 유용하다.**

### 면접 압축 답변

**useReducer**는 다음 상태가 **이전 상태 + 액션**의 조합으로 결정될 때, 상태 전환 로직을 한 곳에 모아 관리하는 훅이에요. 여러 상태 값이 함께 변하거나 전환 조건에 분기가 많을 때 특히 유용한데, 리듀서 함수에 전환 규칙을 모아두면 컴포넌트는 `dispatch`만 호출하면 되니까 **로직과 UI가 깔끔하게 분리**됩니다. Immer와 함께 쓰면 중첩된 상태 업데이트도 간결하게 작성할 수 있어요.

## 50. React 18 concurrent features

> **핵심 요약** — React 18은 렌더링에 우선순위를 부여해서, 급한 업데이트(입력)가 덜 급한 렌더링(리스트)을 기다리지 않게 한다. `useTransition`, `useDeferredValue`, `Suspense`가 핵심.

### 세 가지 핵심 도구

| 도구 | 역할 | 제어 지점 |
|---|---|---|
| `useTransition` | 상태 업데이트를 non-urgent로 표시 | 업데이트 발생 쪽 |
| `useDeferredValue` | 값의 반영을 지연 | 값 소비 쪽 |
| `Suspense` | 비동기 데이터 로딩의 선언적 처리 | UI 표시 쪽 |

### useTransition — 입력 응답성 유지

```tsx
function SearchPage() {
  const [query, setQuery] = useState("");
  const [filterQuery, setFilterQuery] = useState("");
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    setQuery(e.target.value);          // urgent — 입력 즉시 반영
    startTransition(() => {
      setFilterQuery(e.target.value);  // non-urgent — 리스트 필터링은 나중에
    });
  };

  return (
    <div>
      <input value={query} onChange={handleChange} />
      {isPending && <span>검색 중...</span>}
      <HeavyList filterQuery={filterQuery} />
    </div>
  );
}
```

### useDeferredValue — 비싼 렌더링 지연

```tsx
function SearchResults({ query }) {
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      <ExpensiveList query={deferredQuery} />
    </div>
  );
}
```

| | `useTransition` | `useDeferredValue` |
|---|---|---|
| **제어 대상** | 상태 업데이트를 감쌈 | 값을 지연시킴 |
| **사용 위치** | 업데이트를 발생시키는 쪽 | 값을 소비하는 쪽 |
| **`isPending`** | 있음 | 없음 (`query !== deferredQuery`로 판단) |

### Suspense — 선언적 로딩 처리

```tsx
function App() {
  return (
    <Suspense fallback={<Skeleton />}>
      <UserProfile />
    </Suspense>
  );
}
```

Suspense는 하위 컴포넌트가 데이터를 기다리는 동안 fallback을 보여준다. Next.js나 React Query의 `suspense: true` 옵션과 함께 쓰면 선언적으로 로딩 상태를 처리할 수 있다.

### 실무 에피소드 — 이런 상황에서 만난다

> 10만 행 테이블에서 필터 입력 시 매 키 입력마다 테이블 전체가 리렌더되면서 입력이 끊기는 현상. `useTransition`으로 필터 업데이트를 non-urgent로 표시하니, 입력은 즉시 반영되고 테이블은 여유가 생길 때 업데이트. `isPending`으로 "필터링 중..." 표시까지 추가.

### 면접 압축 답변

React 18의 **concurrent features**는 렌더링에 **우선순위**를 부여해서 급한 업데이트가 덜 급한 렌더링을 기다리지 않게 해줘요. `useTransition`은 상태 업데이트를 **non-urgent**로 표시해서 입력 응답성을 유지하고, `useDeferredValue`는 값의 반영을 지연시켜 비싼 렌더링을 뒤로 미룹니다. **Suspense**는 비동기 데이터의 로딩 상태를 선언적으로 처리하고요. 핵심은 렌더 자체가 빨라지는 게 아니라 **우선순위 제어로 체감 응답성이 좋아지는 것**이에요.

## 51. 커스텀 훅 — 로직 재사용의 기본 단위

> **핵심 요약** — 커스텀 훅은 재사용 가능한 상태+효과 묶음을 컴포넌트에서 추출하는 도구다. 여러 곳에서 반복되거나, 컴포넌트가 UI보다 로직이 더 길어질 때 분리한다.

```tsx
// 재사용 가능한 디바운스 훅
function useDebounce<T>(value: T, delay: number): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id);
  }, [value, delay]);
  return debounced;
}
```

분리 기준: **여러 곳에서 반복** 또는 **컴포넌트가 UI보다 로직이 더 길어질 때**. 한 곳 전용이라도 읽기와 책임 분리에 가치가 있을 수 있다.

### 면접 압축 답변

**커스텀 훅**은 상태와 효과를 묶어서 **재사용 가능한 단위로 추출**하는 함수예요. 여러 곳에서 같은 로직이 반복되거나 컴포넌트가 UI보다 로직이 길어질 때 분리하는데, `use` 접두사로 시작하며 훅 규칙(조건문 안에서 호출 금지 등)을 따릅니다. 한 곳에서만 쓰더라도 **읽기 쉬움과 책임 분리** 측면에서 충분히 가치가 있어요.

### 51-2. controlled vs uncontrolled — 입력 제어 전략

> **핵심 요약** — controlled는 React state가 입력 값을 소유하고, uncontrolled는 DOM이 값을 들고 있다. 즉시 검증이 필요하면 controlled, 리렌더를 최소화해야 하면 uncontrolled이 유리하다.

| | controlled | uncontrolled |
|---|---|---|
| **값의 주인** | React state | DOM |
| **값 접근** | state 변수 | ref로 읽기 |
| **즉시 검증/포맷** | 가능 | 어려움 |
| **리렌더** | 매 입력마다 | 최소화 |
| **대표 라이브러리** | 직접 useState | react-hook-form |

```tsx
// controlled: React state가 값을 관리
const [value, setValue] = useState("");
<input value={value} onChange={(e) => setValue(e.target.value)} />

// uncontrolled: DOM이 값을 들고 있음
const ref = useRef<HTMLInputElement>(null);
<input ref={ref} defaultValue="" />
// 제출 시: ref.current?.value
```

실무에서 `react-hook-form`은 uncontrolled 기반으로 리렌더를 최소화하면서 Zod 연동까지 지원한다.

### 면접 압축 답변

**Controlled**는 React state가 입력 값을 소유하고, **uncontrolled**는 DOM이 값을 들고 있어서 ref로 읽는 방식이에요. 즉시 검증이나 포맷팅이 필요하면 controlled가 적합하고, 리렌더를 최소화해야 하면 uncontrolled이 유리합니다. 실무에서는 `react-hook-form`이 **uncontrolled 기반으로 성능을 챙기면서** Zod 스키마 검증까지 지원해서, 폼이 복잡할수록 많이 선호돼요.

### 51-3. 에러 바운더리 — 실패 격리

> **핵심 요약** — 에러 바운더리는 렌더링 중 발생하는 에러를 잡아 하위 트리의 실패를 격리하는 구조다. 앱 최상단, 라우트 단위, 위젯 단위로 중첩 배치한다.

| 잡는다 | 못 잡는다 |
|---|---|
| 렌더링 중 throw | 이벤트 핸들러 에러 |
| 하위 컴포넌트 트리 | useEffect 안 에러 |
| class 컴포넌트의 생명주기 메서드 | 비동기 코드 (setTimeout, fetch) |
| | 에러 바운더리 자체의 에러 |

> **React 19 변경사항**: React 19에서는 에러 리포팅이 개선되었다. 이전에는 uncaught error가 `console.error`로만 출력되었지만, React 19부터는 `window.reportError`로 보고되어 `window.onerror`/`window.addEventListener('error')`로 통합 관리할 수 있다. 단, Error Boundary가 useEffect 내부 에러를 잡는 동작이 추가된 것은 아니다. useEffect 에러는 여전히 Error Boundary로 잡히지 않으며, 별도의 try/catch 처리가 필요하다.

```tsx
import { ErrorBoundary } from "react-error-boundary";

function App() {
  return (
    <ErrorBoundary fallback={<p>앱 전체 오류</p>}>
      <Header />
      <ErrorBoundary fallback={<p>콘텐츠 오류</p>}>
        <MainContent />
      </ErrorBoundary>
      <Footer />
    </ErrorBoundary>
  );
}
```

배치 전략: **앱 최상단**(마지막 안전망) + **라우트 단위**(페이지 격리) + **위젯 단위**(차트 깨져도 나머지 동작).

### 실무 에피소드 — 이런 상황에서 만난다

> 추천 위젯에서 API 에러가 나면 전체 대시보드가 흰 화면이 되는 장애. 추천 위젯을 ErrorBoundary로 감싸고 "추천을 불러올 수 없습니다" fallback을 보여주도록 변경. 나머지 대시보드(차트, 테이블)는 정상 동작.

### 면접 압축 답변

**에러 바운더리**는 렌더링 중 발생하는 에러를 잡아서 **하위 트리만 fallback으로 교체**하고 나머지는 정상 동작하게 하는 구조예요. 단, **렌더링 중 throw만 잡고** 이벤트 핸들러나 비동기 코드(setTimeout, fetch) 에러는 못 잡는다는 제한이 있어요. 배치 전략은 **앱 최상단**(마지막 안전망), **라우트 단위**(페이지 격리), **위젯 단위**(부분 실패 허용)로 중첩 배치하는 게 일반적이고, `react-error-boundary` 라이브러리를 쓰면 함수형 컴포넌트에서도 편하게 사용할 수 있습니다.

## 52. 복잡한 UI에서는 상태를 어떻게 나눠야 하나

> **핵심 요약** — 상태를 영속/일시/파생/서버로 분류하고 역할별로 관리한다. 파생 상태를 `useState + useEffect`로 동기화하는 건 흔한 안티패턴이다.

### 상태 역할별 분류

| 종류 | 설명 | 관리 위치 |
|---|---|---|
| **영속 상태** | 저장 대상 (DB 반영) | 서버 상태 도구 (React Query) |
| **일시 상태** | hover, drag, focus, 모달 열림 | 로컬 useState |
| **파생 상태** | 기존 상태에서 계산 가능한 값 | `useMemo` 또는 일반 변수 |
| **서버 상태** | 외부에서 동기화 필요 | React Query / SWR |

### 파생 상태 안티패턴

```tsx
// Bad: 파생 상태를 useState + useEffect로 동기화
const [filteredItems, setFilteredItems] = useState([]);
useEffect(() => {
  setFilteredItems(items.filter((i) => i.category === selected));
}, [items, selected]);
// 불필요한 state + effect + 한 틱 지연 + 동기화 버그 가능

// Good: useMemo 또는 렌더 중 직접 계산
const filteredItems = useMemo(
  () => items.filter((i) => i.category === selected),
  [items, selected],
);
// state가 아니라 계산값 — 동기화 문제 없음
```

### 상태 정규화 — 복잡한 데이터 구조

```tsx
// Bad: 중첩 배열 — 특정 아이템 업데이트가 번거로움
const [boards, setBoards] = useState([
  { id: "b1", columns: [
    { id: "c1", cards: [{ id: "k1", title: "..." }] }
  ]}
]);

// Good: 정규화 — id로 접근, 업데이트 쉬움
const [boards, setBoards] = useState({ b1: { id: "b1", columnIds: ["c1"] } });
const [columns, setColumns] = useState({ c1: { id: "c1", cardIds: ["k1"] } });
const [cards, setCards] = useState({ k1: { id: "k1", title: "..." } });
// 특정 카드만 업데이트: setCards(prev => ({ ...prev, k1: { ...prev.k1, title: "new" } }))
```

### 실무 에피소드 — 이런 상황에서 만난다

> 칸반 보드에서 카드를 드래그할 때마다 전체 보드 state를 깊은 복사해서 업데이트하니 렌더링이 느려짐. 보드/컬럼/카드를 정규화된 구조로 분리하니 카드 하나만 업데이트할 때 다른 컬럼은 참조가 유지되어 `React.memo`가 효과를 발휘.

### 자주 하는 오해

- 상태를 한 객체에 모으면 관리가 쉽다 → **규모가 커지면 반대.**
- 파생값도 전부 state로 둬야 한다 → **계산으로 충분. useState + useEffect 동기화는 안티패턴.**
- 드래그 좌표도 전역이 낫다 → **일시 상태는 가장 좁은 범위가 적절.**

### 면접 압축 답변

복잡한 UI에서는 상태를 **영속(DB 반영), 일시(hover/drag), 파생(계산 가능), 서버(동기화 필요)**로 나눠서 역할별로 관리하는 게 핵심이에요. 특히 **파생 상태를 useState + useEffect로 동기화하는 건 대표적인 안티패턴**이고, `useMemo`나 일반 변수로 충분합니다. 저장 대상은 React Query 같은 서버 상태 도구, 순간 상태는 로컬 useState, 계산 가능한 값은 렌더 중 직접 계산하면 되고요. 중첩된 데이터 구조는 **정규화**하면 특정 아이템만 업데이트할 때 나머지 참조가 유지되어 `React.memo` 같은 메모이제이션이 제대로 효과를 발휘합니다.

```
  → 에러 바운더리가 실패를 격리한다
```

# 챕터 5 핵심 요약 — 상태가 UI를 결정하는 구조, 그리고 그 상태를 어디에 둘 것인가

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **React는 상태로부터 UI를 어떻게 계산하고, 그 상태는 어떤 기준으로 나누고 어디에 두어야 하는가?**

---

### 1단계: DOM을 직접 바꾸지 않는다 — 선언형 모델

React는 **"이 상태일 때 UI는 이렇다"고 선언**하는 모델이다. `UI = f(state)` — 상태가 바뀌면 React가 UI를 다시 계산한다. 디버깅의 대상이 DOM이 아니라 상태가 된다.

### 2단계: 계산과 반영의 분리 — 렌더와 커밋, Fiber

React는 **렌더 단계(UI 계산, 순수)**와 **커밋 단계(DOM 반영)**를 분리한다. Fiber 아키텍처는 각 노드를 작업 단위로 만들어 **렌더링을 중단/재개하고 우선순위를 제어**할 수 있게 한다. 재조정은 **타입이 다르면 전체 교체, 같으면 key로 정체성 추적**이라는 두 규칙으로 O(n) diffing을 수행한다.

### 3단계: 이 상태를 누가 소유하나 — state ownership

**"누가 읽고 누가 바꾸는가"**가 판단 기준이다. 형제가 같은 상태를 각각 관리하면 어긋나기 쉬우므로, **가장 가까운 공통 부모**로 끌어올리되, 너무 높이 올리면 리렌더 범위가 넓어진다.

### 4단계: props를 계속 내려주는 고통 — Context

Context는 **props drilling을 줄여주는 값 전달 도구**지만, 값이 바뀌면 **모든 구독자가 리렌더**된다. **변경 빈도가 낮은 값**(테마, 로케일, 인증)에 적합하다.

### 5단계: 부분 구독이 필요할 때 — Zustand/Redux

Context와의 가장 큰 차이는 **selector로 부분 구독**이 가능한지 여부다. **서버 데이터는 React Query, 변경 빈도 낮은 공통값은 Context, 부분 구독이 필요하면 Zustand**라는 선택 가이드가 생긴다.

### 6단계: 서버에서 온 데이터는 다르다 — client state vs server state

**client state**는 브라우저 안에서 완결되지만, **server state**는 캐싱, stale 관리, background refetch, mutation 후 invalidation이라는 추가 차원이 있다. React Query는 이 패턴을 자동화한다.

### 7단계: 여러 값이 함께 바뀌는 복잡한 전환 — useReducer

다음 상태가 **"이전 상태 + 액션"으로 결정**되고, 여러 값이 함께 변하거나 전환 조건이 복잡할 때 useReducer가 구조적이다.

### 8단계: 급한 업데이트가 느린 렌더링을 기다리지 않게 — concurrent features

`useTransition`은 상태 업데이트를 non-urgent로 표시해 **입력 응답성을 유지**하고, `useDeferredValue`는 비싼 렌더링을 지연시킨다. **우선순위 제어로 체감 응답성이 좋아지는 것**이다.

### 9단계: 로직 재사용의 기본 단위 — 커스텀 훅

커스텀 훅은 **재사용 가능한 상태+효과 묶음**을 컴포넌트에서 추출하는 도구다. 여러 곳에서 반복되거나 로직이 UI보다 길어질 때 분리한다.

### 9-2단계: 입력 제어 전략 — controlled vs uncontrolled

controlled는 **React state가 폼 값을 소유**하고, uncontrolled는 **DOM이 값을 들고 있어 ref로 읽는다**. 즉시 검증이 필요하면 controlled, 리렌더 최소화가 필요하면 uncontrolled이 유리하다.

### 9-3단계: 실패 격리 — 에러 바운더리

에러 바운더리는 **렌더링 에러를 격리해서 부분 실패만 보여주는 구조**다. 앱 최상단, 라우트 단위, 위젯 단위로 중첩 배치한다.

### 10단계: 복잡한 UI에서 상태를 나누는 기준 — 역할별 분류와 정규화

상태를 **영속(DB 반영), 일시(hover/drag), 파생(계산 가능), 서버(동기화 필요)**로 분류한다. 파생 상태를 `useState + useEffect`로 동기화하는 건 안티패턴이며, `useMemo`나 일반 변수로 충분하다.

---

### 한 줄로 이어 읽기

**선언형 모델**(`UI = f(state)`)이 React의 사고방식을 정하고 → **렌더/커밋 분리**와 **Fiber**가 계산과 반영을 나누고 → **state ownership**이 "누가 상태를 소유하나"를 결정하고 → **Context**가 props drilling을 줄이되 부분 구독은 못 하므로 → **Zustand/Redux**가 selector 기반 구독을 제공하고 → **server state**는 React Query로 캐싱/동기화하고 → 복잡한 전환은 **useReducer**로, 우선순위는 **concurrent features**로, 실패는 **에러 바운더리**로 격리하고 → 최종적으로 상태를 **역할별로 분류하고 정규화**해서 설계한다.
