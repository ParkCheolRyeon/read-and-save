# 챕터 5. React 렌더링 모델에서 상태 구조 설계까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **React는 화면을 직접 조작하는 도구가 아니라, 상태를 기반으로 UI를 다시 계산하는 시스템이라는 관점으로 이해해야 한다.**

---

## 50. React를 왜 선언형이라고 부르나

React를 처음 배울 때 가장 자주 듣는 표현 중 하나가 “React는 선언형이다”이다.  
그런데 이걸 그냥 “코드가 예쁘다” 정도로 이해하면 별 도움이 안 된다.

### 핵심 개념

선언형이라는 건  
**어떻게 하나하나 조작할지를 직접 적는 대신, 지금 상태에서 어떤 UI가 나와야 하는지를 기술하는 방식**이다.

즉 React에서는 보통 이렇게 생각한다.

- 현재 상태가 무엇인가
- 그 상태라면 어떤 UI가 보여야 하는가
- 상태가 바뀌면 UI는 어떻게 다시 계산되어야 하는가

### 명령형과 비교해서 보기

명령형 방식은 보통 이런 느낌이다.

- DOM 요소를 찾고
- 텍스트를 바꾸고
- 클래스를 붙이고
- display를 바꾸고
- 이벤트마다 수동으로 상태를 맞춘다

반면 React는:

- 상태를 바꾸고
- 그 상태에 맞는 JSX를 다시 계산한다

### 왜 이게 중요한가

이 관점을 이해해야 아래 질문들이 이어진다.

- 왜 state가 중요한가
- 왜 render가 다시 돈다고 표현하는가
- 왜 직접 DOM 조작이 React와 자주 충돌하는가
- 왜 UI 버그를 “상태와 UI 불일치”로 설명할 수 있는가

### 자주 하는 오해

- 선언형이면 내부적으로 아무 일도 안 한다 → 틀림
- 선언형은 그냥 더 추상적이라 편하다는 뜻이다 → 절반만 맞다
- React는 DOM을 안 만진다 → 틀림. 직접 만지는 대신 계산된 결과를 바탕으로 관리한다

### 예시

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>현재 값: {count}</p>
      <button onClick={() => setCount((prev) => prev + 1)}>증가</button>
    </div>
  );
}
```

여기서 개발자는  
“버튼 클릭 시 p 태그 텍스트를 직접 바꾼다”라고 쓰지 않는다.  
그 대신 `count` 상태에 따라 어떤 UI가 보여야 하는지를 선언한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 React가 선언형이라는 건 알았다.  
그다음엔 이런 질문이 자연스럽다.

**그렇다면 상태가 바뀌었을 때 React는 내부에서 어떤 단계를 거쳐 UI를 바꾸는가?**

여기서 render와 commit으로 넘어간다.

### 면접 압축 답변

React를 선언형이라고 부르는 이유는, DOM을 단계별로 직접 조작하는 대신 현재 상태에서 어떤 UI가 나와야 하는지를 기술하기 때문입니다. 즉 개발자는 조작 절차보다 상태와 결과 UI의 대응 관계를 작성하고, React가 그 결과를 실제 화면에 반영합니다.

## 51. 렌더와 커밋은 왜 구분해야 하나

React를 쓰다 보면 “리렌더”라는 말을 정말 자주 쓴다.  
그런데 많은 사람이 렌더와 화면 변경을 같은 말처럼 섞어서 쓴다.  
이걸 분리해서 이해해야 React 성능과 동작을 더 정확하게 설명할 수 있다.

### 핵심 개념

React는 대체로 아래 두 단계를 거친다.

#### 렌더 단계

- 컴포넌트를 실행해서 다음 UI 결과를 계산한다
- 어떤 변경이 필요한지 비교할 재료를 만든다
- 아직 실제 DOM을 바꾸는 단계는 아니다

#### 커밋 단계

- 렌더 결과를 바탕으로 실제 DOM 변경을 반영한다
- ref 갱신, layout effect 실행 같은 작업이 이어질 수 있다

### 왜 이게 중요한가

이 구분이 중요한 이유는:

- 리렌더가 일어나도 실제 DOM 변경은 최소화될 수 있고
- 렌더 횟수만 보고 성능을 섣불리 판단하면 안 되며
- `useLayoutEffect`와 `useEffect` 차이도 이 흐름 위에서 설명해야 하기 때문이다

### 자주 하는 오해

- 렌더되면 무조건 DOM도 다시 그린다 → 틀림
- 함수가 다시 실행되면 성능 문제다 → 항상 그렇진 않다
- 커밋은 렌더와 별 차이 없다 → 실제 DOM 반영 시점이라는 점에서 중요하게 다르다

### 실무 연결

React 성능을 볼 때는 단순히 “렌더가 많이 일어났나”보다 아래를 같이 봐야 한다.

- 렌더 계산이 비싼가
- 실제 커밋 비용이 큰가
- layout thrashing이 생기나
- 무거운 자식 컴포넌트가 함께 계산되나

### 여기서 다음 개념으로 어떻게 넘어가나

이제 렌더와 커밋을 나눴다.  
그다음엔 자연스럽게 이런 질문이 나온다.

**그럼 React는 렌더 단계에서 무엇을 비교하고, 무엇을 바꾸려는 걸까?**

여기서 Virtual DOM으로 넘어간다.

### 면접 압축 답변

렌더는 다음 UI를 계산하는 단계이고, 커밋은 그 계산 결과를 실제 DOM에 반영하는 단계입니다. 그래서 컴포넌트 함수가 다시 실행되었다고 해서 곧바로 DOM이 모두 다시 바뀌는 것은 아니며, 이 구분이 있어야 성능과 훅 실행 시점을 더 정확히 설명할 수 있습니다.

## 52. Virtual DOM은 정확히 무엇을 뜻하나

Virtual DOM은 React 면접에서 정말 많이 나오지만,  
설명이 추상적으로 흐르기 쉬운 개념이다.

### 핵심 개념

Virtual DOM은 브라우저 DOM 그 자체가 아니라,  
**현재 UI 구조를 메모리 안에서 표현한 자바스크립트 객체 트리 같은 추상 표현**으로 이해하면 된다.

즉 React는:

1. 상태를 바탕으로 새로운 UI 구조를 계산하고
2. 이전 표현과 비교한 뒤
3. 실제 DOM에 필요한 변경만 반영하려고 한다

### 왜 이게 중요한가

Virtual DOM을 이해하면 아래 흐름이 보인다.

- 왜 React가 선언형으로 동작할 수 있는가
- 왜 렌더와 커밋이 나뉘는가
- 왜 key가 중요해지는가
- 왜 재조정(reconciliation)이라는 말이 나오는가

### 자주 하는 오해

- Virtual DOM이 있으니 무조건 빠르다 → 틀림
- React는 실제 DOM을 전혀 안 쓴다 → 틀림
- Virtual DOM은 그냥 DOM 복사본이다 → 너무 단순화된 설명이다

### 더 정확한 감각

Virtual DOM의 핵심은 “무조건 빠름”이 아니라,
**상태 기반 UI 계산을 더 다루기 쉽게 만드는 추상화 계층**이라는 점이다.

즉 성능도 중요하지만,
React가 일관된 방식으로 UI를 계산하고 비교할 수 있게 해주는 구조라는 점이 더 본질적이다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 React가 새 UI 표현과 이전 표현을 비교한다는 걸 알았다.  
그다음엔 그 비교 과정이 어떤 원리로 일어나는지, 즉 재조정으로 넘어가야 한다.

### 면접 압축 답변

Virtual DOM은 실제 브라우저 DOM이 아니라, 현재 UI 구조를 메모리 안에서 표현한 추상적인 트리입니다. React는 상태 변경 시 새로운 UI 표현을 계산하고, 이전 표현과 비교해 필요한 실제 DOM 변경만 커밋하려고 합니다.

## 53. 재조정(reconciliation)은 왜 key와 연결되나

Virtual DOM을 이해했다면, 다음은 재조정이다.  
React는 새 UI 표현과 이전 표현을 비교하면서 “무엇을 유지하고 무엇을 바꿀지” 판단해야 한다.

### 핵심 개념

재조정은 React가 이전 렌더 결과와 다음 렌더 결과를 비교하면서  
어떤 노드를 유지하고, 어떤 노드를 교체하고, 어떤 하위 트리를 다시 계산할지 정하는 과정이다.

### 왜 이게 중요한가

이 과정을 이해해야 아래 질문들이 연결된다.

- 리스트에서 key가 왜 중요한가
- 같은 컴포넌트인데 왜 state가 유지되거나 초기화되나
- 조건부 렌더링에서 왜 어떤 경우엔 input 값이 남고 어떤 경우엔 사라지나

### 자주 하는 오해

- 재조정은 무조건 깊은 비교를 다 한다 → 틀림
- 구조가 조금이라도 다르면 전체가 갈아엎어진다 → 단순화가 심하다
- key는 성능 최적화용 옵션이다 → 정체성 유지와 직결된다

### 실무 연결

재조정은 단순 성능 문제가 아니다.  
특히 폼, 리스트, 애니메이션, DnD에서는  
**어떤 요소가 같은 존재인지**를 React가 제대로 이해하는 게 중요하다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 정체성 유지가 왜 중요한지 봤다.  
다음엔 그 정체성을 React에 알려주는 대표 장치인 `key`로 넘어간다.

### 면접 압축 답변

재조정은 React가 이전 렌더 결과와 다음 렌더 결과를 비교해 어떤 UI를 유지하고 어떤 부분만 바꿀지 결정하는 과정입니다. 이때 요소의 정체성을 구분하는 기준이 중요해지고, 그 대표적인 장치가 key입니다.

## 54. key는 왜 단순 경고 제거용이 아닌가

리스트 렌더링을 배우면 `key` 경고를 먼저 만나게 된다.  
그래서 초기에 `key`를 단순히 경고 없애는 값 정도로 오해하기 쉽다.  
하지만 실제로는 React의 정체성 관리에 매우 중요한 값이다.

### 핵심 개념

`key`는 리스트 안 각 요소의 **identity**, 즉 정체성을 React에 알려주는 값이다.

React는 `key`를 기준으로:

- 기존 요소를 유지할지
- 새 요소로 볼지
- 상태를 이어갈지
- 다시 만들지를 판단한다

### 왜 이게 중요한가

`key`가 불안정하면 아래 문제가 생길 수 있다.

- input 상태가 엉뚱한 항목으로 이동
- 애니메이션 시작/종료가 이상해짐
- 리스트 재정렬 시 UI가 섞임
- 불필요한 재생성과 재마운트 발생

### 자주 하는 오해

- `key`는 unique 하기만 하면 된다 → 가능한 한 안정적이어야 한다
- `index`를 쓰면 항상 안 된다 → 정적 리스트에선 괜찮을 수도 있다
- `key`는 성능만의 문제다 → 상태 보존과 정체성 문제이기도 하다

### 예시

```tsx
items.map((item) => <TodoItem key={item.id} item={item} />);
```

이렇게 안정적인 id를 쓰면 React가 항목의 정체성을 더 정확히 유지할 수 있다.

반면 아래는 위험할 수 있다.

```tsx
items.map((item, index) => <TodoItem key={index} item={item} />);
```

정렬, 삽입, 삭제가 자주 일어나면 index 기반 key는 상태 꼬임을 일으킬 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 identity 감각이 생겼다.  
그다음엔 자연스럽게 이런 질문이 나온다.

**그렇다면 상태는 어떤 위치에 둬야 정체성과 리렌더 범위를 더 안정적으로 관리할 수 있을까?**

즉 state ownership으로 넘어간다.

### 면접 압축 답변

`key`는 리스트에서 각 요소의 정체성을 React에 알려주는 값입니다. 불안정한 key를 쓰면 상태가 잘못 유지되거나 input 값이 엉뚱한 항목에 붙을 수 있어서, 단순 경고 제거용이 아니라 identity 설계의 일부로 봐야 합니다.

## 55. state ownership은 왜 구조 설계에서 중요하나

React에서 상태를 다루는 핵심 질문은 단순히  
“state를 쓸까 말까”가 아니다.

더 중요한 건:

> **이 상태를 누가 소유해야 하는가?**

이다.

### 핵심 개념

state ownership은 어떤 상태를 어떤 컴포넌트가 책임지고 관리할지를 정하는 문제다.

이걸 잘 정해야:

- props 흐름이 선명해지고
- 리렌더 범위가 예측 가능해지고
- 상태 동기화 버그가 줄어든다

### 왜 이게 중요한가

상태 위치를 잘못 잡으면 이런 문제가 생긴다.

- 여러 컴포넌트가 같은 의미의 상태를 중복 보유
- 값이 서로 어긋남
- 불필요하게 전역 상태가 커짐
- props drilling이 심해짐
- 리렌더 범위가 과도하게 넓어짐

### 자주 하는 오해

- 공유되면 무조건 전역 상태다 → 틀림
- 가장 상위에 두면 관리가 쉽다 → 무조건은 아니다
- 로컬 상태는 작고 단순한 경우에만 쓴다 → 오히려 많은 UI 상태는 로컬이 더 적합하다

### 상태 위치를 정할 때 보는 기준

- 이 상태를 누가 읽는가
- 이 상태를 누가 바꾸는가
- 여러 곳에서 동시에 일관되게 써야 하는가
- 서버에서 내려오는 데이터인가
- UI 일시 상태인가, 도메인 상태인가

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태 소유의 기준을 봤다.  
그다음엔 가장 흔한 설계 선택 중 하나인 **상태 끌어올리기**로 넘어갈 수 있다.

### 면접 압축 답변

state ownership은 어떤 상태를 어느 컴포넌트가 책임질지 정하는 문제입니다. 이걸 잘 정해야 props 흐름, 리렌더 범위, 상태 일관성이 안정되며, 무조건 상위나 전역에 두기보다 누가 읽고 누가 변경하는지 기준으로 위치를 정하는 게 중요합니다.

---

## 56. 상태 끌어올리기는 언제 필요하고, 언제 과한가

React를 배우면 “state lifting”을 자주 듣는다.  
이건 여러 컴포넌트가 같은 상태를 공유해야 할 때 매우 중요한 패턴이다.

### 핵심 개념

상태 끌어올리기는 공통으로 필요한 상태를  
가장 가까운 공통 부모 컴포넌트로 올리는 방식이다.

### 왜 이게 중요한가

형제 컴포넌트끼리 같은 값을 각각 따로 관리하면  
상태가 쉽게 어긋난다.

예를 들어:

- 검색 필터 UI와 결과 리스트
- 탭 선택 상태와 상세 뷰
- 폼 입력값과 요약 패널

이런 경우엔 공통 부모가 상태를 들고 있는 게 더 자연스럽다.

### 자주 하는 오해

- 공유되면 무조건 끌어올리는 게 정답이다 → 너무 높이 올리면 오히려 관리가 힘들다
- 상태를 상위로 올릴수록 재사용성이 높다 → 리렌더 범위와 복잡도가 같이 커질 수 있다
- 한 번 올린 상태는 다시 내려오면 안 된다 → 상황에 따라 재구성할 수 있다

### 실무 감각

좋은 기준은 이렇다.

- 정말 여러 자식이 공유해야 하는가
- 가장 가까운 공통 부모가 어디인가
- 상위로 올렸을 때 리렌더 비용이 커지지 않는가
- 로컬 상태로 두는 편이 더 단순하지 않은가

### 예시

```tsx
function Parent() {
  const [selectedId, setSelectedId] = useState<string | null>(null);

  return (
    <>
      <Sidebar selectedId={selectedId} onSelect={setSelectedId} />
      <Detail selectedId={selectedId} />
    </>
  );
}
```

이 구조는 형제 컴포넌트가 같은 선택 상태를 공유할 때 자연스럽다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태를 공통 부모로 올리는 패턴을 봤다.  
그다음엔 이런 질문이 나온다.

**그런데 너무 깊은 트리에서 props를 계속 내려줘야 하면 어떻게 하지?**

여기서 Context로 넘어간다.

### 면접 압축 답변

상태 끌어올리기는 여러 컴포넌트가 같은 상태를 공유해야 할 때 가장 가까운 공통 부모가 그 상태를 관리하도록 만드는 방식입니다. 다만 무조건 위로 올리면 리렌더 범위와 복잡도가 커질 수 있어서, 필요한 범위까지만 올리는 게 중요합니다.

---

## 57. Context는 언제 유용하고, 언제 과용되기 쉬운가

상태를 끌어올리다 보면 어느 순간 props 전달이 너무 길어진다.  
그럴 때 Context가 자연스럽게 등장한다.

### 핵심 개념

Context는 깊은 트리 아래로 값을 전달할 때  
중간 props 전달을 줄여주는 메커니즘이다.

즉 Context는 기본적으로 **값 전달 문제**를 해결하려는 도구에 가깝다.

### 왜 이게 중요한가

많은 사람이 Context를 곧바로 “전역 상태관리”라고 부르는데,  
정확히는 전역 상태 그 자체보다  
**전역적으로 접근 가능한 값 전달 구조**에 가깝다.

### Context가 잘 맞는 경우

- 현재 사용자 정보
- 테마
- 로케일
- 디자인 시스템 설정
- 비교적 낮은 빈도로 바뀌는 공통 값

### 자주 하는 오해

- Context만 쓰면 상태관리 문제는 끝난다 → 틀림
- 공유 상태는 다 Context가 정답이다 → 틀림
- Context를 쓰면 성능 걱정은 없어도 된다 → 구독 범위에 따라 리렌더 영향이 생길 수 있다

### 실무 연결

Context가 과용되면:

- 어디서 값이 바뀌는지 추적이 어려워지고
- Provider 중첩이 복잡해지고
- 여러 상태가 한곳에 뭉쳐 구조가 무거워질 수 있다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 Context의 역할을 봤다.  
그다음엔 자연스럽게 이런 질문이 나온다.

**그렇다면 Zustand나 Redux 같은 외부 상태관리 도구는 Context와 무엇이 다른가?**

### 면접 압축 답변

Context는 깊은 트리로 값을 전달할 때 props drilling을 줄여주는 도구입니다. 테마나 사용자 정보처럼 공통으로 참조해야 하는 값엔 잘 맞지만, 복잡한 상태 업데이트와 세밀한 구독 제어까지 모두 해결해주지는 않기 때문에 전역 상태관리 도구와는 역할이 다릅니다.

---

## 58. Context와 Zustand/Redux는 어떤 문제를 다르게 푸나

Context를 보면 자연스럽게 “그럼 왜 Zustand나 Redux가 필요한가?”라는 질문이 나온다.  
이 둘은 겹치는 부분도 있지만, 해결하려는 초점이 다르다.

### 핵심 개념

- Context는 **값 전달**
- Zustand/Redux는 **상태 저장, 업데이트 흐름, 구독 구조 관리**

에 더 초점이 있다.

### 왜 이게 중요한가

복잡한 편집기, DnD, 필터 패널, 설정 패널 같은 UI에서는  
단순 전달보다 아래가 중요해진다.

- 상태를 어디에 저장할 것인가
- 어떤 액션으로 바꿀 것인가
- 어떤 컴포넌트가 어떤 조각만 구독할 것인가
- 디버깅과 추적을 어떻게 할 것인가

### 자주 하는 오해

- Zustand/Redux는 앱이 커야만 쓴다 → 틀림
- Context로도 되는데 굳이 외부 상태관리 툴이 필요 없다 → 업데이트 흐름과 구독 제어에서 차이가 난다
- 외부 상태관리 도구를 쓰면 설계 고민이 사라진다 → 틀림

### 실무 연결

보통 이런 식으로 구분하면 편하다.

- 사용자/테마/로케일 → Context 가능
- 복잡한 인터랙션 편집 상태 → Zustand 가능
- 엄격한 액션 흐름과 추적 필요 → Redux도 가능

### 여기서 다음 개념으로 어떻게 넘어가나

이제 클라이언트 상태 구조를 봤다.  
그다음엔 아주 중요한 분기가 나온다.

**그런데 서버에서 받아오는 데이터도 같은 방식으로 다뤄야 할까?**

여기서 server state와 client state로 넘어간다.

### 면접 압축 답변

Context는 주로 값 전달을 단순화하는 데 강하고, Zustand나 Redux는 상태 저장과 업데이트 흐름, 구독 구조를 더 체계적으로 관리하는 데 강합니다. 그래서 둘은 대체라기보다 문제 축이 조금 다른 도구로 보는 게 더 정확합니다.

---

## 59. client state와 server state는 왜 구분해야 하나

이건 프론트엔드 아키텍처에서 아주 중요한 질문이다.  
예전에는 상태를 전부 “그냥 state”로 뭉뚱그려 생각하기 쉬웠지만,  
실무에서는 client state와 server state를 나눠 보는 게 훨씬 선명하다.

### 핵심 개념

#### client state

브라우저 안에서 사용자가 조작하며 유지하는 UI 중심 상태

예:

- 모달 열림 여부
- 탭 선택 상태
- 입력 중인 값
- 드래그 중 좌표
- 로컬 필터 토글

#### server state

서버에서 가져오고, 다시 동기화가 필요하며, 캐싱/재요청/오래됨(stale) 문제가 있는 상태

예:

- 사용자 프로필 데이터
- 게시글 목록
- 검색 결과
- 알림 목록

### 왜 이게 중요한가

server state를 단순 `useState`로만 다루면 아래 문제가 생긴다.

- 로딩 상태 반복 작성
- 에러 상태 반복 작성
- 재요청 타이밍 관리 어려움
- 캐싱 전략 부재
- stale 데이터 관리 어려움

### 자주 하는 오해

- 서버에서 온 값도 일단 state면 다 똑같다 → 틀림
- React state만 잘 쓰면 서버 데이터도 충분히 관리 가능하다 → 규모가 커질수록 불편해진다
- React Query 같은 라이브러리는 선택 사항일 뿐이다 → 팀 규모와 데이터 복잡도에 따라 큰 차이를 만든다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태를 큰 축에서 나눴다.  
그다음엔 자연스럽게 **React Query 같은 서버 상태 라이브러리는 왜 필요한가**로 넘어간다.

### 면접 압축 답변

client state는 UI 안에서 사용자가 직접 조작하는 상태이고, server state는 서버와 동기화되며 캐싱, 재요청, stale 관리가 필요한 상태입니다. 이 둘을 구분해야 어떤 상태를 로컬로 둘지, 어떤 상태를 별도 서버 상태 관리 도구로 다룰지 판단이 쉬워집니다.

---

## 60. React Query 같은 서버 상태 도구는 왜 필요하나

server state를 별도로 구분했다면,  
왜 전용 도구가 필요한지도 자연스럽게 보인다.

### 핵심 개념

React Query 같은 도구는  
서버 데이터를 단순 저장하는 게 아니라 아래 문제를 함께 다룬다.

- 캐싱
- 재요청
- stale 관리
- background refetch
- loading / error 상태 표준화
- mutation 이후 invalidation

### 왜 이게 중요한가

프론트엔드 앱이 커지면  
“데이터를 한 번 가져왔다”보다  
“언제 다시 가져와야 하나”, “이 데이터는 아직 신선한가”, “수정 후 무엇을 다시 무효화할 것인가”가 더 중요해진다.

### 자주 하는 오해

- fetch는 어차피 `useEffect`에서 하면 된다 → 반복 구조가 금방 늘어난다
- 서버 데이터도 Zustand/Redux에 넣으면 끝이다 → 캐시 무효화와 stale 전략까지 직접 설계해야 한다
- React Query는 너무 무겁다 → 문제를 푸는 범위를 봐야 한다

### 실무 연결

대표적으로 아래에서 강하다.

- 목록/상세 데이터 동기화
- 무한 스크롤
- 필터 조합 검색
- mutation 후 리스트 갱신
- 재방문 시 캐시 활용

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태 구조가 꽤 정리됐다.  
그다음엔 복잡해지는 로직을 UI에서 분리하는 기준, 즉 커스텀 훅으로 넘어갈 수 있다.

### 면접 압축 답변

React Query 같은 서버 상태 도구는 데이터를 단순 저장하는 것을 넘어서 캐싱, 재요청, stale 관리, mutation 이후 invalidation 같은 문제를 함께 다룹니다. 그래서 서버 상태를 로컬 UI 상태와 같은 방식으로 다루는 것보다 더 자연스러운 경우가 많습니다.

---

## 61. 커스텀 훅은 언제 분리하는 게 좋은가

React를 쓰다 보면 컴포넌트 안에 로직이 점점 쌓인다.  
그러면 흔히 “이걸 커스텀 훅으로 빼야 하나?”라는 고민이 나온다.

### 핵심 개념

커스텀 훅은 단순 코드 분리 도구가 아니라,  
**재사용 가능한 상태/효과/행동의 묶음**을 추출하는 방식이다.

### 언제 분리하면 좋은가

- 여러 컴포넌트에서 같은 로직이 반복될 때
- UI보다 상태 전환/비즈니스 규칙이 더 길어질 때
- 컴포넌트가 읽기 어려워질 때
- 테스트나 책임 분리가 필요할 때

### 왜 이게 중요한가

좋은 커스텀 훅 분리는:

- 컴포넌트를 더 선언적으로 만들고
- 로직 재사용성을 높이고
- 상태 전환 흐름을 더 선명하게 만든다

### 자주 하는 오해

- 길어지면 무조건 훅으로 뺀다 → 너무 쪼개면 오히려 추적이 어려워질 수 있다
- 훅으로 분리하면 성능도 좋아진다 → 본질은 책임 분리다
- 훅은 재사용될 때만 의미 있다 → 한 곳 전용이라도 읽기와 책임 분리에 가치가 있을 수 있다

### 실무 연결

예:

- 검색 조건 관리 훅
- 무한 스크롤 훅
- 드래그 상태 훅
- 폼 검증 훅
- API mutation 훅

### 여기서 다음 개념으로 어떻게 넘어가나

이제 로직 분리 기준을 봤다.  
그다음엔 폼 같은 입력 중심 UI에서 자주 나오는 **controlled / uncontrolled**로 넘어갈 수 있다.

### 면접 압축 답변

커스텀 훅은 상태와 효과, 이벤트 로직을 재사용 가능한 단위로 분리하는 방식입니다. 단순히 코드 길이가 아니라 책임이 뚜렷해지고, 여러 컴포넌트에서 같은 상태 전환 규칙이나 비즈니스 로직이 반복될 때 특히 분리 가치가 큽니다.

---

## 62. controlled와 uncontrolled는 왜 폼 설계에서 중요하나

폼은 React에서 아주 대표적인 상태 설계 문제다.  
여기서 controlled와 uncontrolled를 구분해야 이유 없이 복잡한 폼을 만들지 않게 된다.

### 핵심 개념

#### controlled

입력값을 React state가 직접 관리하는 방식

#### uncontrolled

DOM 자체가 값을 들고 있고, 필요할 때 ref 등으로 읽는 방식

### 왜 이게 중요한가

controlled는:

- 값 추적이 명확하고
- 즉시 검증, 포맷팅, 조건부 UI 제어가 쉽다

uncontrolled는:

- 단순 폼에서 코드량이 줄고
- 모든 입력을 매 타이핑마다 state로 동기화하지 않아도 된다

### 자주 하는 오해

- React 폼은 무조건 controlled가 정답이다 → 틀림
- uncontrolled는 구식 방식이다 → 틀림
- controlled가 항상 더 안전하다 → 요구사항에 따라 다르다

### 실무 연결

- 즉시 검증, 마스킹, 조건부 에러 표시 → controlled가 잘 맞음
- 단순 제출형 폼, 라이브러리 기반 폼 처리 → uncontrolled도 잘 맞을 수 있음

### 예시

```tsx
function ControlledInput() {
  const [value, setValue] = useState("");

  return <input value={value} onChange={(e) => setValue(e.target.value)} />;
}
```

### 여기서 다음 개념으로 어떻게 넘어가나

폼 입력 방식을 봤다면,  
다음엔 복잡한 UI에서 렌더 비용과 상태 구조를 어떻게 잡을지로 넘어갈 수 있다.

### 면접 압축 답변

controlled는 입력값을 React state가 직접 관리하는 방식이고, uncontrolled는 DOM이 값을 들고 있다가 필요할 때 읽는 방식입니다. 즉시 검증이나 값 기반 UI 제어가 중요하면 controlled가 잘 맞고, 단순 제출형 폼은 uncontrolled도 충분히 실용적일 수 있습니다.

---

## 63. 에러 바운더리는 왜 React 구조 설계에서 중요하나

대부분의 학습자는 happy path 중심으로 React를 이해한다.  
하지만 실제 서비스에서는 실패를 어디서 어떻게 격리할지도 중요하다.  
여기서 에러 바운더리가 나온다.

### 핵심 개념

에러 바운더리는 하위 트리에서 발생한 렌더링 오류를 잡아  
전체 앱이 다 같이 무너지는 걸 막고, fallback UI를 보여줄 수 있게 해준다.

### 왜 이게 중요한가

복잡한 화면에서 일부 위젯이 깨졌다고 해서  
전체 화면이 하얗게 죽는 경험은 운영 측면에서 좋지 않다.

에러 바운더리를 적절히 두면:

- 장애 범위를 좁히고
- 복구 UI를 제공하고
- 로그 포인트를 명확히 할 수 있다

### 자주 하는 오해

- 에러 바운더리가 모든 에러를 잡는다 → 이벤트 핸들러나 비동기 에러는 별도 처리 맥락이 필요할 수 있다
- 앱 최상단 하나면 충분하다 → 화면 구성에 따라 세분화가 더 낫다
- 에러 바운더리는 예외 처리용 장식이다 → 구조적 장애 격리 도구다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 React 구조 설계에서 실패를 다루는 축도 봤다.  
그다음엔 리스트, 차트, DnD처럼 복잡한 UI에서 상태를 어떻게 나눠야 하는지로 넘어갈 수 있다.

### 면접 압축 답변

에러 바운더리는 하위 UI 트리의 렌더링 오류를 격리해서 전체 앱이 함께 무너지는 것을 막아주는 구조입니다. 따라서 단순 예외 처리 장식이 아니라, 장애 범위를 좁히고 fallback UI를 제공하기 위한 설계 도구로 보는 게 맞습니다.

## 64. 복잡한 UI에서는 상태를 어떻게 나눠야 하나

단순 폼이나 버튼은 상태 구조가 비교적 간단하다.  
하지만 편집기, DnD, 차트, 애니메이션 많은 UI에선 상태를 한 덩어리로 두면 금방 꼬인다.

### 핵심 개념

복잡한 UI에서는 상태를 역할별로 나눠 보는 게 좋다.

예를 들면:

- 영속 상태: 저장 대상이 되는 상태
- 일시 상태: hover, drag, focus 같은 순간 상태
- 파생 상태: 기존 상태로부터 계산 가능한 값
- 서버 상태: 외부에서 동기화되는 값

### 왜 이게 중요한가

이 구분이 없으면:

- 저장할 필요 없는 값까지 전역에 들어가고
- 리렌더 범위가 넓어지고
- 상태 의미가 흐려지고
- 디버깅이 어려워진다

### 자주 하는 오해

- 상태를 한 객체에 모으면 관리가 쉽다 → 규모가 커지면 반대가 되기 쉽다
- 파생값도 전부 state로 둬야 한다 → 계산으로 충분할 수 있다
- 드래그 중 좌표도 전역으로 두는 게 낫다 → 일시 상태는 더 좁은 범위가 적절할 수 있다

### 실무 연결

예를 들어 DnD 편집기에서는:

- 현재 마우스 위치
- 드래그 중 임시 위치
- 실제 저장될 노드 위치
- 선택 상태
- 패널 열림 상태

를 전부 같은 급의 상태로 다루면 구조가 금방 무거워진다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 상태 구조를 나누는 감각이 생겼다.  
그다음엔 React의 선언형 렌더 모델이 Next.js 같은 프레임워크에서 어떻게 확장되는지로 넘어갈 수 있다.

### 면접 압축 답변

복잡한 UI에서는 상태를 영속 상태, 일시 상태, 파생 상태, 서버 상태처럼 역할별로 나눠 보는 것이 중요합니다. 이렇게 해야 저장 대상과 일시 인터랙션 상태가 뒤섞이지 않고, 리렌더 범위와 책임도 더 선명하게 관리할 수 있습니다.

## 65. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- React는 선언형 UI 시스템이다
- 상태가 바뀌면 UI를 다시 계산한다
- 렌더와 커밋은 서로 다른 단계다
- Virtual DOM은 상태 기반 UI 계산을 가능하게 하는 추상화다
- 재조정은 이전 결과와 다음 결과를 비교하는 과정이다
- key는 리스트와 조건부 렌더링에서 정체성을 유지하게 해준다
- state ownership은 구조 설계의 핵심이다
- 상태 끌어올리기, Context, 외부 상태관리 도구는 각각 다른 범위의 문제를 푼다
- client state와 server state는 구분해야 한다
- 커스텀 훅은 상태와 로직을 더 선명한 단위로 분리한다
- 복잡한 UI일수록 상태를 역할별로 나누는 감각이 필요하다

즉,  
이번 챕터는 단순히 React API를 배우는 챕터가 아니라,  
**“React는 상태와 UI의 관계를 어떻게 계산하고, 그 위에 어떤 구조를 세워야 하는가”**를 배우는 챕터다.

# 사용법

이 챕터는 아래 순서로 보면 좋다.

1. 선언형, render, commit, Virtual DOM으로 React의 계산 모델을 먼저 잡는다.
2. key와 reconciliation으로 정체성 문제를 이해한다.
3. state ownership, lifting state up, Context로 상태 위치 감각을 세운다.
4. client state / server state / 외부 상태관리 도구로 상태 종류를 분리한다.
5. 마지막에 커스텀 훅과 복잡한 UI 상태 분리까지 연결한다.

이렇게 보면 React가 단순 컴포넌트 라이브러리가 아니라, 상태 기반 UI 계산 모델이라는 감각이 훨씬 선명해진다.
