# 챕터 5. React 렌더링 모델에서 상태 구조 설계까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **React는 화면을 직접 조작하는 도구가 아니라, 상태를 기반으로 UI를 다시 계산하는 시스템이라는 관점으로 이해해야 한다.**

---

## 50. React를 왜 선언형이라고 부르나

React를 처음 배울 때 가장 자주 듣는 표현 중 하나가 "React는 선언형이다"이다.
그런데 이걸 그냥 "코드가 예쁘다" 정도로 이해하면 별 도움이 안 된다.

### 핵심 개념

선언형이라는 건
**어떻게 하나하나 조작할지를 직접 적는 대신, 지금 상태에서 어떤 UI가 나와야 하는지를 기술하는 방식**이다.

### 명령형과 비교해서 보기

```js
// 명령형: DOM을 직접 단계별로 조작
const p = document.getElementById("count-display");
const btn = document.getElementById("increment-btn");
let count = 0;

btn.addEventListener("click", () => {
  count++;
  p.textContent = `현재 값: ${count}`; // 직접 DOM을 찾아서 바꾼다
});
```

```tsx
// 선언형: 상태에 따라 어떤 UI가 나와야 하는지 기술
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>현재 값: {count}</p>
      <button onClick={() => setCount((prev) => prev + 1)}>증가</button>
    </div>
  );
}
// 개발자는 "p 태그를 찾아서 텍스트를 바꿔라"라고 쓰지 않는다.
// count 상태에 따라 어떤 UI가 보여야 하는지를 선언한다.
```

이 관점을 이해해야 아래 질문들이 이어진다.

- 왜 state가 중요한가
- 왜 render가 다시 돈다고 표현하는가
- 왜 직접 DOM 조작이 React와 자주 충돌하는가
- 왜 UI 버그를 "상태와 UI 불일치"로 설명할 수 있는가

### 자주 하는 오해

- 선언형이면 내부적으로 아무 일도 안 한다 → 틀림. React는 내부적으로 Virtual DOM 비교, DOM 조작을 모두 수행한다. 개발자가 직접 안 할 뿐이다.
- React는 DOM을 안 만진다 → 틀림. 직접 만지는 대신 계산된 결과를 바탕으로 최소한의 DOM 변경을 수행한다.

### 면접 압축 답변

- **10초**: React는 DOM을 직접 조작하는 대신, 현재 상태에서 어떤 UI가 나와야 하는지를 기술하는 선언형 모델입니다.
- **30초**: 명령형은 DOM 요소를 찾고 속성을 직접 바꾸지만, React는 상태와 UI의 대응 관계를 선언하고 상태가 바뀌면 UI를 다시 계산합니다. 개발자는 조작 절차가 아닌 결과를 기술하고, React가 그 결과를 실제 화면에 반영합니다.

> 이제 React가 선언형이라는 건 알았다. **그렇다면 상태가 바뀌었을 때 React는 내부에서 어떤 단계를 거쳐 UI를 바꾸는가?** 여기서 render와 commit으로 넘어간다.

## 51. 렌더와 커밋은 왜 구분해야 하나

React를 쓰다 보면 "리렌더"라는 말을 정말 자주 쓴다.
그런데 많은 사람이 렌더와 화면 변경을 같은 말처럼 섞어서 쓴다.
이걸 분리해서 이해해야 React 성능과 동작을 더 정확하게 설명할 수 있다.

### 핵심 개념

React는 대체로 아래 두 단계를 거친다.

#### 렌더 단계 (Render Phase)

- 컴포넌트를 실행해서 다음 UI 결과를 계산한다
- 어떤 변경이 필요한지 비교할 재료를 만든다
- 아직 실제 DOM을 바꾸는 단계는 아니다
- **순수해야 한다** — 사이드 이펙트 없이 같은 입력이면 같은 출력

#### 커밋 단계 (Commit Phase)

- 렌더 결과를 바탕으로 실제 DOM 변경을 반영한다
- DOM 변경 직후 `useLayoutEffect`가 **동기적으로** 실행된다 (브라우저가 화면을 그리기 전)
- 브라우저가 화면을 그린 후 `useEffect`가 **비동기적으로** 실행된다

```
상태 변경
  → 렌더 단계: 컴포넌트 함수 실행, JSX 계산
  → 커밋 단계: 실제 DOM 반영
    → useLayoutEffect 실행 (동기, 페인트 전)
    → 브라우저 페인트
    → useEffect 실행 (비동기, 페인트 후)
```

### 왜 이게 중요한가

- 리렌더가 일어나도 실제 DOM 변경은 최소화될 수 있다
- 렌더 횟수만 보고 성능을 섣불리 판단하면 안 된다
- `useLayoutEffect`와 `useEffect` 차이가 이 흐름 위에서만 설명된다

### 자주 하는 오해

- 렌더되면 무조건 DOM도 다시 그린다 → 틀림. 렌더 결과가 이전과 같으면 DOM 변경이 없을 수 있다.
- 함수가 다시 실행되면 성능 문제다 → 항상 그렇진 않다. 렌더 계산 자체는 대부분 빠르다.
- `useLayoutEffect`와 `useEffect`는 같은 타이밍이다 → 틀림. `useLayoutEffect`는 DOM 변경 직후 동기 실행, `useEffect`는 페인트 이후 비동기 실행이다.

### 면접 압축 답변

- **10초**: 렌더는 다음 UI를 계산하는 단계, 커밋은 실제 DOM에 반영하는 단계입니다. 리렌더가 곧 DOM 변경은 아닙니다.
- **30초**: 렌더 단계에서 컴포넌트 함수를 실행해 JSX를 계산하고, 커밋 단계에서 변경된 부분만 실제 DOM에 반영합니다. DOM 반영 직후 `useLayoutEffect`가 동기 실행되고, 브라우저 페인트 이후 `useEffect`가 비동기 실행됩니다.

> 이제 렌더와 커밋을 나눴다. **그럼 React는 렌더 단계에서 무엇을 비교하고, 무엇을 바꾸려는 걸까?** 여기서 Virtual DOM으로 넘어간다.

## 52. Virtual DOM은 정확히 무엇을 뜻하나

Virtual DOM은 React 면접에서 정말 많이 나오지만,
설명이 추상적으로 흐르기 쉬운 개념이다.

### 핵심 개념

Virtual DOM은 브라우저 DOM 그 자체가 아니라,
**현재 UI 구조를 메모리 안에서 표현한 자바스크립트 객체 트리**다.

```jsx
// 이 JSX는
<div>
  <h1>Hello</h1>
  <p>World</p>
</div>

// 내부적으로 이런 객체 트리가 된다 (React.createElement의 결과)
{
  type: 'div',
  props: {
    children: [
      { type: 'h1', props: { children: 'Hello' } },
      { type: 'p', props: { children: 'World' } }
    ]
  }
}
```

React는:

1. 상태를 바탕으로 새로운 Virtual DOM 트리를 계산하고
2. 이전 트리와 비교한 뒤 (diffing)
3. 실제 DOM에 필요한 변경만 반영한다 (patching)

### Fiber 아키텍처

React 16부터는 단순 객체 트리가 아니라 **Fiber**라는 내부 구조를 사용한다. Fiber는 각 컴포넌트/요소를 linked list 형태의 노드로 표현하며, 렌더링 작업을 **잘게 쪼개서 중단/재개**할 수 있게 해준다. 이게 React의 concurrent features(Suspense, useTransition 등)를 가능하게 하는 기반이다.

각 Fiber 노드는 하나의 컴포넌트 또는 DOM 요소에 대응하며, 대략 아래와 같은 정보를 갖는다:

- **type**: 컴포넌트 함수 또는 DOM 태그 이름 (`'div'`, `MyComponent`)
- **props**: 해당 노드에 전달된 속성
- **child / sibling / return**: 첫 번째 자식, 다음 형제, 부모를 가리키는 포인터 — 이 세 포인터로 트리를 linked list처럼 순회한다
- **stateNode**: 클래스 컴포넌트의 인스턴스 또는 실제 DOM 노드
- **memoizedState**: 해당 노드의 훅 상태 체인
- **effectTag / effect list**: 커밋 단계에서 수행할 DOM 조작 종류 (삽입, 갱신, 삭제)

이전의 스택 기반 재조정(Stack Reconciler)은 트리를 재귀로 순회했기 때문에 중간에 멈출 수 없었다. Fiber는 각 노드를 **하나의 작업 단위(unit of work)**로 만들어서, 한 노드 처리 후 브라우저에 제어권을 돌려줄 수 있다. 이를 **time-slicing**이라고 한다.

이 구조 덕분에 **우선순위 스케줄링**이 가능해진다. 예를 들어 사용자 입력(urgent)이 들어오면, 진행 중이던 무거운 리스트 렌더링(non-urgent)을 잠시 멈추고 입력을 먼저 처리한 뒤, 나머지 렌더링을 이어서 할 수 있다.

### 자주 하는 오해

- Virtual DOM이 있으니 무조건 빠르다 → 틀림. Virtual DOM은 추가 비교 비용이 있다. 핵심은 속도보다 **선언형 UI를 가능하게 하는 추상화**다.
- React는 실제 DOM을 전혀 안 쓴다 → 틀림. 최종적으로 실제 DOM을 조작한다.
- Virtual DOM은 그냥 DOM 복사본이다 → JS 객체 트리이지 DOM 복사본이 아니다.

### 면접 압축 답변

- **10초**: Virtual DOM은 UI 구조를 메모리 안의 JS 객체 트리로 표현한 것입니다. React는 이전 트리와 새 트리를 비교해 필요한 DOM 변경만 반영합니다.
- **30초**: React는 상태 변경 시 새 Virtual DOM 트리를 계산하고, 이전 트리와 diff해서 최소한의 실제 DOM 조작만 수행합니다. React 16부터는 Fiber 아키텍처를 통해 렌더링을 잘게 쪼개고 중단/재개할 수 있어서 concurrent features의 기반이 됩니다.

> 이제 React가 새 UI 표현과 이전 표현을 비교한다는 걸 알았다. 그 비교 과정이 어떤 원리로 일어나는지, 즉 재조정으로 넘어간다.

## 53. 재조정(reconciliation)은 왜 key와 연결되나

Virtual DOM을 이해했다면, 다음은 재조정이다.
React는 새 UI 표현과 이전 표현을 비교하면서 "무엇을 유지하고 무엇을 바꿀지" 판단해야 한다.

### 핵심 개념

재조정은 React가 이전 렌더 결과와 다음 렌더 결과를 비교하면서
어떤 노드를 유지하고, 어떤 노드를 교체하고, 어떤 하위 트리를 다시 계산할지 정하는 과정이다.

React는 두 가지 핵심 휴리스틱을 사용한다.

1. **타입이 다르면 전체 교체**: 같은 위치에서 `<div>` → `<span>`으로 바뀌면 기존 트리를 버리고 새로 만든다.
2. **같은 타입의 형제 요소는 `key`로 구분**: 리스트에서 어떤 항목이 같은 존재인지 `key`로 판단한다.

```tsx
// 타입이 바뀌면 — 전체 교체 (state도 초기화)
{showInput ? <input /> : <textarea />}

// 같은 타입이 유지되면 — state 보존
{showInput ? <input type="text" /> : <input type="password" />}
// 둘 다 <input>이므로 React는 같은 요소로 보고 props만 바꾼다
```

### 자주 하는 오해

- 재조정은 무조건 깊은 비교를 다 한다 → 틀림. 같은 레벨에서만 비교하며, 타입이 다르면 하위 트리는 비교 없이 교체한다.
- key는 성능 최적화용 옵션이다 → 정체성 유지와 직결된다. key가 바뀌면 React는 완전히 새로운 요소로 취급한다.

### key로 state를 리셋하는 패턴

key가 바뀌면 React는 새 요소로 인식해서 state를 초기화한다. 이걸 의도적으로 활용할 수 있다.

```tsx
// selectedUserId가 바뀌면 EditForm의 state가 완전히 리셋된다
<EditForm key={selectedUserId} userId={selectedUserId} />
```

### 면접 압축 답변

- **10초**: 재조정은 이전 렌더와 다음 렌더를 비교해 유지/교체를 결정하는 과정입니다. 타입이 다르면 전체 교체, 같은 타입의 리스트 항목은 key로 구분합니다.
- **30초**: React는 같은 위치에서 타입이 바뀌면 하위 트리를 통째로 교체하고, 같은 타입이면 props만 업데이트합니다. 리스트에서는 key로 각 항목의 정체성을 추적하며, key가 바뀌면 새 요소로 인식해서 state를 초기화합니다.

> 이제 정체성 유지가 왜 중요한지 봤다. 다음엔 그 정체성을 React에 알려주는 대표 장치인 `key`를 더 깊이 본다.

## 54. key는 왜 단순 경고 제거용이 아닌가

리스트 렌더링을 배우면 `key` 경고를 먼저 만나게 된다.
그래서 초기에 `key`를 단순히 경고 없애는 값 정도로 오해하기 쉽다.

### 핵심 개념

`key`는 리스트 안 각 요소의 **identity**, 즉 정체성을 React에 알려주는 값이다.

```tsx
// 안정적인 id를 key로 — React가 정체성을 정확히 추적
items.map((item) => <TodoItem key={item.id} item={item} />);

// index를 key로 — 정렬/삽입/삭제 시 상태가 꼬일 수 있다
items.map((item, index) => <TodoItem key={index} item={item} />);
```

### index key의 실제 문제

```tsx
// 각 TodoItem에 체크박스가 있다고 가정
// items = ["A", "B", "C"] → 맨 앞에 "X"를 추가
// key=index면: A의 state가 X에 붙고, B의 state가 A에 붙는다
// key=item.id면: 각 항목의 state가 정확히 유지된다
```

### 자주 하는 오해

- `key`는 unique하기만 하면 된다 → 가능한 한 **안정적**이어야 한다. 매 렌더마다 바뀌는 key는 의미가 없다.
- `index`를 쓰면 항상 안 된다 → 정적 리스트(정렬/삭제 없는)에선 괜찮을 수 있다.
- `key`는 성능만의 문제다 → 상태 보존과 정체성 문제이기도 하다.

### 면접 압축 답변

- **10초**: `key`는 리스트에서 각 요소의 정체성을 React에 알려주는 값입니다. 불안정한 key는 상태 꼬임을 일으킵니다.
- **30초**: key는 단순 경고 제거용이 아니라 React의 재조정에서 어떤 요소가 같은 존재인지 판단하는 기준입니다. index를 key로 쓰면 정렬/삽입 시 상태가 엉뚱한 항목에 붙을 수 있고, key를 의도적으로 바꾸면 컴포넌트 state를 리셋할 수도 있습니다.

> 이제 identity 감각이 생겼다. **그렇다면 상태는 어떤 위치에 둬야 정체성과 리렌더 범위를 더 안정적으로 관리할 수 있을까?** 즉 state ownership으로 넘어간다.

## 55. state ownership은 왜 구조 설계에서 중요하나

React에서 상태를 다루는 핵심 질문은 단순히 "state를 쓸까 말까"가 아니다.

더 중요한 건:

> **이 상태를 누가 소유해야 하는가?**

### 핵심 개념

state ownership은 어떤 상태를 어떤 컴포넌트가 책임지고 관리할지를 정하는 문제다.

```tsx
// Bad: 두 컴포넌트가 같은 의미의 상태를 각각 관리
function Sidebar() {
  const [selectedId, setSelectedId] = useState(null); // 여기도 있고
  // ...
}
function Detail() {
  const [selectedId, setSelectedId] = useState(null); // 여기도 있다
  // 두 값이 쉽게 어긋난다
}

// Good: 소유자를 하나로 정하고 나머지는 props로 받는다
function Parent() {
  const [selectedId, setSelectedId] = useState(null);
  return (
    <>
      <Sidebar selectedId={selectedId} onSelect={setSelectedId} />
      <Detail selectedId={selectedId} />
    </>
  );
}
```

### 상태 위치를 정할 때 보는 기준

- 이 상태를 누가 읽는가
- 이 상태를 누가 바꾸는가
- 여러 곳에서 동시에 일관되게 써야 하는가
- 서버에서 내려오는 데이터인가
- UI 일시 상태인가, 도메인 상태인가

### 자주 하는 오해

- 공유되면 무조건 전역 상태다 → 틀림. 공통 부모에 두는 것만으로 충분한 경우가 많다.
- 가장 상위에 두면 관리가 쉽다 → 리렌더 범위가 불필요하게 넓어질 수 있다.
- 로컬 상태는 작고 단순한 경우에만 쓴다 → 오히려 많은 UI 상태는 로컬이 더 적합하다.

### 면접 압축 답변

- **10초**: state ownership은 어떤 상태를 어느 컴포넌트가 책임질지 정하는 문제입니다.
- **30초**: 이걸 잘 정해야 props 흐름, 리렌더 범위, 상태 일관성이 안정됩니다. 무조건 상위나 전역에 두기보다, 누가 읽고 누가 변경하는지 기준으로 가장 가까운 적절한 위치에 두는 게 중요합니다.

> 이제 상태 소유의 기준을 봤다. 다음은 가장 흔한 설계 선택 중 하나인 **상태 끌어올리기**다.

## 56. 상태 끌어올리기는 언제 필요하고, 언제 과한가

여러 컴포넌트가 같은 상태를 공유해야 할 때 매우 중요한 패턴이다.

### 핵심 개념

상태 끌어올리기는 공통으로 필요한 상태를
가장 가까운 공통 부모 컴포넌트로 올리는 방식이다.

```tsx
function Parent() {
  const [selectedId, setSelectedId] = useState<string | null>(null);

  return (
    <>
      <Sidebar selectedId={selectedId} onSelect={setSelectedId} />
      <Detail selectedId={selectedId} />
    </>
  );
}
```

형제 컴포넌트끼리 같은 값을 각각 따로 관리하면 상태가 쉽게 어긋난다. 검색 필터 UI와 결과 리스트, 탭 선택 상태와 상세 뷰 같은 경우에 공통 부모가 상태를 들고 있는 게 더 자연스럽다.

### 자주 하는 오해

- 공유되면 무조건 끌어올리는 게 정답이다 → 너무 높이 올리면 오히려 관리가 힘들다
- 상태를 상위로 올릴수록 재사용성이 높다 → 리렌더 범위와 복잡도가 같이 커질 수 있다

### 면접 압축 답변

- **10초**: 상태 끌어올리기는 공유가 필요한 상태를 가장 가까운 공통 부모에 두는 패턴입니다.
- **30초**: 형제 컴포넌트가 같은 상태를 각각 관리하면 어긋나기 쉬우므로, 공통 부모가 소유하고 props로 내려주는 게 더 안전합니다. 다만 무조건 위로 올리면 리렌더 범위가 커지므로, 필요한 범위까지만 올리는 판단이 중요합니다.

> 이제 상태를 공통 부모로 올리는 패턴을 봤다. **그런데 너무 깊은 트리에서 props를 계속 내려줘야 하면 어떻게 하지?** 여기서 Context로 넘어간다.

## 57. Context는 언제 유용하고, 언제 과용되기 쉬운가

상태를 끌어올리다 보면 어느 순간 props 전달이 너무 길어진다. 그럴 때 Context가 자연스럽게 등장한다.

### 핵심 개념

Context는 깊은 트리 아래로 값을 전달할 때 중간 props 전달을 줄여주는 메커니즘이다.

```tsx
const ThemeContext = createContext<"light" | "dark">("light");

function App() {
  const [theme, setTheme] = useState<"light" | "dark">("light");
  return (
    <ThemeContext.Provider value={theme}>
      <Header />
      <Main />
      <Footer />
    </ThemeContext.Provider>
  );
}

function DeepChild() {
  const theme = useContext(ThemeContext); // props drilling 없이 접근
  return <div className={theme}>...</div>;
}
```

많은 사람이 Context를 곧바로 "전역 상태관리"라고 부르는데, 정확히는 **전역적으로 접근 가능한 값 전달 구조**에 가깝다.

### Context의 중요한 리렌더 특성

Context 값이 바뀌면, 해당 Context를 `useContext`로 구독하는 **모든 컴포넌트가 리렌더**된다. `React.memo`도 이걸 막지 못한다. 그래서 서로 관련 없는 상태를 하나의 Context에 묶으면 불필요한 리렌더가 생긴다.

```tsx
// Bad: 관련 없는 상태를 한 Context에 묶음
const AppContext = createContext({ theme: "light", user: null, locale: "ko" });
// theme만 바뀌어도 user만 쓰는 컴포넌트도 리렌더

// Good: 관심사별로 Context를 분리
const ThemeContext = createContext("light");
const UserContext = createContext(null);
const LocaleContext = createContext("ko");
```

### 자주 하는 오해

- Context만 쓰면 상태관리 문제는 끝난다 → 틀림
- Context를 쓰면 성능 걱정은 없어도 된다 → Context 값 변경 시 모든 구독자가 리렌더된다.
- 공유 상태는 다 Context가 정답이다 → 빈번히 변경되는 상태는 Context보다 외부 상태관리 도구가 더 적합할 수 있다.

### 면접 압축 답변

- **10초**: Context는 깊은 트리로 값을 전달할 때 props drilling을 줄여주는 도구입니다.
- **30초**: 테마, 사용자 정보, 로케일처럼 변경 빈도가 낮은 공통 값에 잘 맞습니다. 단, Context 값이 바뀌면 모든 구독자가 리렌더되고 `React.memo`로도 막을 수 없으므로, 빈번히 바뀌는 상태나 관련 없는 상태를 한 Context에 묶으면 성능 문제가 생깁니다.

> 이제 Context의 역할을 봤다. **그렇다면 Zustand나 Redux 같은 외부 상태관리 도구는 Context와 무엇이 다른가?**

---

## 58. Context와 Zustand/Redux는 어떤 문제를 다르게 푸나

Context를 보면 자연스럽게 "그럼 왜 Zustand나 Redux가 필요한가?"라는 질문이 나온다.

### 핵심 개념

- Context는 **값 전달**에 초점
- Zustand/Redux는 **상태 저장, 업데이트 흐름, 구독 구조 관리**에 초점

가장 큰 실질적 차이는 **부분 구독(selector)**이다.

```tsx
// Context: theme이 바뀌면 user만 쓰는 컴포넌트도 리렌더
const { user } = useContext(AppContext);

// Zustand: selector로 user만 구독 — theme이 바뀌어도 리렌더 안 됨
const user = useStore((state) => state.user);
```

### 자주 하는 오해

- Zustand/Redux는 앱이 커야만 쓴다 → 틀림. 부분 구독이 필요하면 작은 앱에서도 유용하다.
- Context로도 되는데 굳이 외부 상태관리 툴이 필요 없다 → 구독 제어와 업데이트 흐름에서 차이가 크다.
- 외부 상태관리 도구를 쓰면 설계 고민이 사라진다 → 틀림. 어떤 상태를 넣을지는 여전히 설계 문제다.

### 실무 감각

| 상황 | 도구 |
|------|------|
| 사용자/테마/로케일 | Context 적합 |
| 복잡한 인터랙션 편집 상태 | Zustand 적합 |
| 엄격한 액션 흐름과 추적 필요 | Redux 적합 |

### 면접 압축 답변

- **10초**: Context는 값 전달, Zustand/Redux는 상태 저장과 구독 구조 관리에 초점이 다릅니다.
- **30초**: Context 값이 바뀌면 모든 구독자가 리렌더되지만, Zustand는 selector로 필요한 조각만 구독해서 불필요한 리렌더를 막습니다. 그래서 둘은 대체 관계라기보다 문제 축이 다른 도구입니다.

> 이제 클라이언트 상태 구조를 봤다. **그런데 서버에서 받아오는 데이터도 같은 방식으로 다뤄야 할까?** 여기서 server state와 client state로 넘어간다.

## 59. client state와 server state는 왜 구분해야 하나

예전에는 상태를 전부 "그냥 state"로 뭉뚱그려 생각하기 쉬웠지만,
실무에서는 client state와 server state를 나눠 보는 게 훨씬 선명하다.

### 핵심 개념

**client state**: 브라우저 안에서 사용자가 조작하며 유지하는 UI 중심 상태 (모달 열림, 탭 선택, 입력 중인 값, 드래그 좌표)

**server state**: 서버에서 가져오고, 다시 동기화가 필요하며, 캐싱/재요청/stale 문제가 있는 상태 (사용자 프로필, 게시글 목록, 검색 결과)

### server state를 useState로만 다루면 생기는 문제

```tsx
// 이렇게 하면 금방 한계에 부딪힌다
function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then(setUser)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [userId]);

  // 매 컴포넌트마다 loading/error/data 3개 state 반복
  // 캐싱 없음, stale 관리 없음, 재요청 전략 없음
  // 여러 컴포넌트가 같은 데이터를 요청하면 중복 fetch
}
```

### 자주 하는 오해

- 서버에서 온 값도 일단 state면 다 똑같다 → 틀림. 캐싱, stale, 재요청이라는 추가 차원이 있다.
- React state만 잘 쓰면 서버 데이터도 충분히 관리 가능하다 → 규모가 커질수록 반복 코드와 누락이 많아진다.

### 면접 압축 답변

- **10초**: client state는 UI 조작 상태, server state는 서버 동기화가 필요한 데이터입니다.
- **30초**: client state는 모달/탭/입력처럼 브라우저 안에서 완결되고, server state는 캐싱/재요청/stale 관리가 필요합니다. 이 둘을 구분해야 어떤 상태를 로컬로 둘지, 어떤 상태를 별도 도구로 다룰지 판단이 쉬워집니다.

> 이제 상태를 큰 축에서 나눴다. 자연스럽게 **React Query 같은 서버 상태 라이브러리는 왜 필요한가**로 넘어간다.

---

## 60. React Query 같은 서버 상태 도구는 왜 필요하나

server state를 별도로 구분했다면, 왜 전용 도구가 필요한지도 자연스럽게 보인다.

### 핵심 개념

React Query 같은 도구는 서버 데이터를 단순 저장하는 게 아니라:

- 캐싱 / 재요청 / stale 관리
- background refetch
- loading / error 상태 표준화
- mutation 이후 invalidation

을 함께 다룬다.

```tsx
// React Query를 쓰면
function UserProfile({ userId }: { userId: string }) {
  const { data: user, isLoading, error } = useQuery({
    queryKey: ["user", userId],
    queryFn: () => fetch(`/api/users/${userId}`).then((r) => r.json()),
    staleTime: 5 * 60 * 1000, // 5분간 fresh
  });
  // 캐싱, 중복 요청 제거, stale 관리, loading/error가 자동
}

// mutation 후 관련 데이터 무효화
const mutation = useMutation({
  mutationFn: updateUser,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ["user"] });
  },
});
```

### 자주 하는 오해

- fetch는 어차피 `useEffect`에서 하면 된다 → 반복 구조가 금방 늘어난다
- 서버 데이터도 Zustand/Redux에 넣으면 끝이다 → 캐시 무효화와 stale 전략까지 직접 설계해야 한다

### 면접 압축 답변

- **10초**: React Query는 서버 데이터의 캐싱, 재요청, stale 관리를 자동화하는 도구입니다.
- **30초**: useState + useEffect로 매번 loading/error/data를 반복하는 대신, 선언적으로 서버 데이터를 관리합니다. 캐싱, background refetch, mutation 후 invalidation까지 다루므로, 서버 상태를 로컬 state와 같은 방식으로 다루는 것보다 훨씬 자연스럽습니다.

> 이제 상태 구조가 꽤 정리됐다. 다음엔 복잡한 상태 전환을 다루는 useReducer로 넘어간다.

## 61. useReducer는 언제 useState 대신 쓰나

useState가 가장 기본적인 상태 관리 도구라면,
useReducer는 **상태 전환 로직이 복잡해질 때** 쓰는 도구다.

### 핵심 개념

useReducer는 현재 상태와 액션을 받아 다음 상태를 반환하는 **리듀서 함수**를 기반으로 동작한다.
다음 상태가 이전 상태 + 액션의 조합으로 결정될 때 useState보다 구조가 깔끔해진다.

### useState vs useReducer 판단 기준

| useState가 적합할 때 | useReducer가 적합할 때 |
|---------------------|----------------------|
| 독립적인 단일 값 (모달 열림, 입력값) | 여러 값이 함께 변하는 복합 상태 |
| 전환 로직이 단순 (토글, 증가) | 상태 전환에 조건 분기가 많을 때 |
| 상태 간 의존성이 없을 때 | 다음 상태가 이전 상태 + 액션 종류에 따라 달라질 때 |

### 코드 예시: 폼 + 검증 상태

```tsx
type FormState = {
  values: { name: string; email: string };
  errors: { name?: string; email?: string };
  isSubmitting: boolean;
};

type FormAction =
  | { type: "SET_FIELD"; field: string; value: string }
  | { type: "SET_ERROR"; field: string; error: string }
  | { type: "SUBMIT_START" }
  | { type: "SUBMIT_SUCCESS" }
  | { type: "SUBMIT_FAILURE"; errors: Record<string, string> };

function formReducer(state: FormState, action: FormAction): FormState {
  switch (action.type) {
    case "SET_FIELD":
      return {
        ...state,
        values: { ...state.values, [action.field]: action.value },
        errors: { ...state.errors, [action.field]: undefined },
      };
    case "SET_ERROR":
      return {
        ...state,
        errors: { ...state.errors, [action.field]: action.error },
      };
    case "SUBMIT_START":
      return { ...state, isSubmitting: true };
    case "SUBMIT_SUCCESS":
      return { ...state, isSubmitting: false };
    case "SUBMIT_FAILURE":
      return { ...state, isSubmitting: false, errors: action.errors };
  }
}

function SignupForm() {
  const [state, dispatch] = useReducer(formReducer, {
    values: { name: "", email: "" },
    errors: {},
    isSubmitting: false,
  });

  return (
    <form>
      <input
        value={state.values.name}
        onChange={(e) =>
          dispatch({ type: "SET_FIELD", field: "name", value: e.target.value })
        }
      />
      {state.errors.name && <span>{state.errors.name}</span>}
      {/* ... */}
    </form>
  );
}
```

이 예시에서 `SET_FIELD`는 값 변경과 동시에 해당 필드의 에러를 초기화한다.
useState 여러 개로 이걸 하면 `setValues`, `setErrors`를 매번 함께 호출해야 하고, 빠뜨리기 쉽다.

### 자주 하는 오해

- useReducer는 Redux의 축소판이다 → 비슷한 패턴이지만 완전히 다른 도구다. 미들웨어, 전역 스토어 개념이 없다.
- 복잡한 앱에서만 쓴다 → 상태 2~3개가 함께 바뀌는 경우에도 충분히 유용하다.
- useReducer를 쓰면 성능이 좋아진다 → 본질은 상태 전환 로직의 응집도다.

### 면접 압축 답변

- **10초**: useReducer는 다음 상태가 이전 상태와 액션의 조합으로 결정될 때, useState보다 상태 전환 로직을 명확하게 관리할 수 있는 훅입니다.
- **30초**: 여러 상태 값이 함께 변하거나 전환 조건이 복잡할 때 useReducer를 씁니다. 리듀서 함수에 전환 로직을 모아두면 컴포넌트는 dispatch만 호출하면 되고, 상태 전환 규칙을 한 곳에서 관리할 수 있습니다. 단순한 독립 값이면 useState가 더 적합합니다.

> 이제 복잡한 상태 전환을 다루는 도구를 봤다. 다음엔 React 18이 도입한 **concurrent features가 왜 중요한지**로 넘어간다.

## 62. React 18 concurrent features는 왜 중요한가

React 18부터 도입된 concurrent features는 렌더링 자체의 우선순위를 다루는 새로운 축이다.

### 핵심 개념

기존 React 렌더링은 한 번 시작하면 끝까지 완료해야 했다.
Concurrent features는 **렌더링을 중단하고, 더 급한 업데이트를 먼저 처리**할 수 있게 해준다.

핵심 도구 세 가지:

- **Suspense**: 비동기 데이터가 준비될 때까지 fallback UI를 보여준다
- **useTransition**: 상태 업데이트를 non-urgent로 표시해 UI 응답성을 유지한다
- **useDeferredValue**: 값의 반영을 지연시켜 비싼 렌더링을 뒤로 미룬다

### useTransition 예시: 검색 입력 + 무거운 리스트

```tsx
function SearchPage() {
  const [query, setQuery] = useState("");
  const [isPending, startTransition] = useTransition();

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value); // 입력은 즉시 반영 (urgent)

    startTransition(() => {
      // 리스트 필터링은 non-urgent — 입력이 먼저
      setFilterQuery(value);
    });
  };

  return (
    <div>
      <input value={query} onChange={handleChange} />
      {isPending && <span>검색 중...</span>}
      <HeavyList filterQuery={filterQuery} />
    </div>
  );
}
```

`startTransition` 안의 상태 업데이트는 urgent 업데이트(타이핑)에 양보한다.
사용자 입력이 끊기지 않으면서 리스트 렌더링은 여유가 생길 때 수행된다.

### useDeferredValue 예시: 비싼 컴포넌트 지연 렌더링

```tsx
function SearchResults({ query }: { query: string }) {
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      <ExpensiveList query={deferredQuery} />
    </div>
  );
}
```

`useDeferredValue`는 값 자체를 지연시킨다. 새 값이 들어오면 이전 값으로 먼저 렌더하고, 여유가 생기면 새 값으로 다시 렌더한다. `useTransition`이 상태 업데이트를 감싸는 것이라면, `useDeferredValue`는 소비 쪽에서 값을 지연시키는 것이다.

### Suspense와 데이터 패칭

```tsx
// React 18+ Suspense (프레임워크 또는 use() 훅과 함께)
function App() {
  return (
    <Suspense fallback={<Skeleton />}>
      <UserProfile />
    </Suspense>
  );
}
```

Suspense는 하위 컴포넌트가 데이터를 기다리는 동안 fallback을 보여준다.
Next.js 같은 프레임워크나 React의 `use()` 훅과 함께 쓰면 선언적으로 로딩 상태를 처리할 수 있다.

### 자주 하는 오해

- concurrent features를 쓰면 무조건 빨라진다 → 틀림. 렌더 자체가 빨라지는 게 아니라 우선순위 제어로 체감 응답성이 좋아지는 것이다.
- useTransition과 useDeferredValue는 같은 것이다 → useTransition은 업데이트를 감싸고, useDeferredValue는 값을 지연시킨다. 제어 지점이 다르다.
- Suspense는 React Query 없이 단독으로 데이터를 불러올 수 있다 → Suspense 자체는 대기/표시 메커니즘이고, 데이터 패칭은 별도 통합이 필요하다.

### 면접 압축 답변

- **10초**: React 18의 concurrent features는 렌더링에 우선순위를 부여해서, 급한 업데이트가 덜 급한 렌더링을 기다리지 않게 합니다.
- **30초**: useTransition으로 무거운 상태 업데이트를 non-urgent로 표시하면 사용자 입력 응답성이 유지되고, useDeferredValue로 비싼 렌더링을 지연시킬 수 있습니다. Suspense는 비동기 데이터의 로딩 상태를 선언적으로 처리합니다. 이 세 가지가 합쳐져 체감 성능을 크게 개선합니다.

> 이제 렌더링 우선순위까지 봤다. 다음엔 복잡해지는 로직을 UI에서 분리하는 기준, 즉 커스텀 훅으로 넘어간다.

## 63. 커스텀 훅은 언제 분리하는 게 좋은가

React를 쓰다 보면 컴포넌트 안에 로직이 점점 쌓인다. 흔히 "이걸 커스텀 훅으로 빼야 하나?"라는 고민이 나온다.

### 핵심 개념

커스텀 훅은 단순 코드 분리 도구가 아니라, **재사용 가능한 상태/효과/행동의 묶음**을 추출하는 방식이다.

```tsx
// 디바운스 훅 — 여러 컴포넌트에서 재사용
function useDebounce<T>(value: T, delay: number): T {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id);
  }, [value, delay]);

  return debounced;
}

// 토글 훅 — 상태 전환 로직을 캡슐화
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = useCallback(() => setValue((v) => !v), []);
  return [value, toggle] as const;
}
```

### 언제 분리하면 좋은가

- 여러 컴포넌트에서 같은 로직이 반복될 때
- UI보다 상태 전환/비즈니스 규칙이 더 길어질 때
- 컴포넌트가 읽기 어려워질 때
- 테스트나 책임 분리가 필요할 때

### 자주 하는 오해

- 길어지면 무조건 훅으로 뺀다 → 너무 쪼개면 오히려 추적이 어려워질 수 있다
- 훅으로 분리하면 성능도 좋아진다 → 본질은 책임 분리다. 성능과는 직접 관련 없다.
- 훅은 재사용될 때만 의미 있다 → 한 곳 전용이라도 읽기와 책임 분리에 가치가 있을 수 있다

### 면접 압축 답변

- **10초**: 커스텀 훅은 재사용 가능한 상태와 효과의 묶음을 추출하는 방식입니다.
- **30초**: 여러 컴포넌트에서 같은 상태 전환이나 비즈니스 로직이 반복될 때 분리하면 좋습니다. 단순히 코드 길이가 아니라 책임이 뚜렷해지는지가 기준이며, 한 곳 전용이라도 컴포넌트를 선언적으로 만드는 데 가치가 있을 수 있습니다.

> 이제 로직 분리 기준을 봤다. 다음엔 폼 같은 입력 중심 UI에서 자주 나오는 **controlled / uncontrolled**로 넘어간다.

## 64. controlled와 uncontrolled는 왜 폼 설계에서 중요하나

폼은 React에서 아주 대표적인 상태 설계 문제다.

### 핵심 개념

**controlled**: 입력값을 React state가 직접 관리하는 방식

```tsx
function ControlledInput() {
  const [value, setValue] = useState("");

  return <input value={value} onChange={(e) => setValue(e.target.value)} />;
  // React state가 진짜 값 → 즉시 검증, 포맷팅 가능
}
```

**uncontrolled**: DOM 자체가 값을 들고 있고, 필요할 때 ref 등으로 읽는 방식

```tsx
function UncontrolledInput() {
  const inputRef = useRef<HTMLInputElement>(null);

  const handleSubmit = () => {
    console.log(inputRef.current?.value); // 제출 시점에만 값을 읽음
  };

  return (
    <form onSubmit={handleSubmit}>
      <input ref={inputRef} defaultValue="" />
      <button type="submit">제출</button>
    </form>
  );
}
```

### 자주 하는 오해

- React 폼은 무조건 controlled가 정답이다 → 틀림. 단순 제출형 폼에선 uncontrolled이 더 간단할 수 있다.
- uncontrolled는 구식 방식이다 → 틀림. `react-hook-form` 같은 현대적 라이브러리도 uncontrolled 기반이다.

### 면접 압축 답변

- **10초**: controlled는 React state가 값을 관리, uncontrolled는 DOM이 값을 들고 있다가 필요 시 읽는 방식입니다.
- **30초**: 즉시 검증이나 값 기반 UI 제어가 중요하면 controlled가 잘 맞고, 단순 제출형 폼은 uncontrolled도 충분히 실용적입니다. `react-hook-form`처럼 uncontrolled 기반으로 성능을 최적화하는 라이브러리도 있습니다.

> 폼 입력 방식을 봤다면, 다음엔 렌더링 중 에러를 어떻게 격리할지, 즉 에러 바운더리로 넘어간다.

---

## 65. 에러 바운더리는 왜 React 구조 설계에서 중요하나

대부분의 학습자는 happy path 중심으로 React를 이해한다.
하지만 실제 서비스에서는 실패를 어디서 어떻게 격리할지도 중요하다.

### 핵심 개념

에러 바운더리는 하위 트리에서 발생한 **렌더링 중 에러**를 잡아 전체 앱이 다 같이 무너지는 걸 막고, fallback UI를 보여줄 수 있게 해준다.

### 에러 바운더리가 잡는 것과 못 잡는 것

| 잡는다 | 못 잡는다 |
|--------|-----------|
| 렌더링 중 throw | 이벤트 핸들러 안의 에러 |
| useEffect/useLayoutEffect 안의 에러 | 비동기 코드 (setTimeout, fetch) |
| 생명주기 메서드 안의 에러 | 서버 사이드 렌더링 에러 |
| 하위 컴포넌트 트리 | 에러 바운더리 자체의 에러 |

### 구현

에러 바운더리는 클래스 컴포넌트에서만 구현 가능하다. 직접 구현하면 아래와 같다:

```tsx
class ErrorBoundary extends React.Component<
  { children: React.ReactNode; fallback: React.ReactNode },
  { hasError: boolean }
> {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    console.error(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) return this.props.fallback;
    return this.props.children;
  }
}
```

`getDerivedStateFromError`로 에러 발생 시 상태를 바꾸고, `componentDidCatch`로 에러 정보를 로깅한다. 실무에서는 이걸 직접 만들기보다 `react-error-boundary`를 더 많이 쓴다.

```tsx
// react-error-boundary 사용
import { ErrorBoundary } from "react-error-boundary";

function App() {
  return (
    <ErrorBoundary fallback={<p>앱 전체 오류</p>}>
      <Header />
      <ErrorBoundary fallback={<p>콘텐츠 영역 오류</p>}>
        <MainContent />
      </ErrorBoundary>
      <Footer />
    </ErrorBoundary>
  );
}
```

이벤트 핸들러의 에러는 에러 바운더리가 못 잡으므로 별도로 처리해야 한다.

```tsx
function SubmitButton() {
  const handleClick = async () => {
    try {
      await submitData();
    } catch (e) {
      // 이벤트 핸들러 에러는 try/catch로 직접 처리
      showErrorToast(e);
    }
  };
  return <button onClick={handleClick}>제출</button>;
}
```

### 배치 전략

- **앱 최상단**: 전체 앱 crash 방지 (마지막 안전망)
- **라우트/페이지 단위**: 한 페이지 오류가 다른 페이지에 영향 안 줌
- **위젯/패널 단위**: 차트나 추천 위젯이 깨져도 나머지는 동작

### 자주 하는 오해

- 에러 바운더리가 모든 에러를 잡는다 → 이벤트 핸들러, 비동기 에러는 못 잡는다
- 앱 최상단 하나면 충분하다 → 화면 구성에 따라 세분화가 더 낫다

### 면접 압축 답변

- **10초**: 에러 바운더리는 하위 트리의 렌더링 에러를 격리해 전체 앱 crash를 막는 구조입니다.
- **30초**: 렌더링 중 에러와 useEffect 에러는 잡지만, 이벤트 핸들러와 비동기 에러는 별도 처리가 필요합니다. 앱/라우트/위젯 레벨로 세분화해서 배치하면 장애 범위를 좁히고 복구 UI를 제공할 수 있습니다.

> 이제 React 구조 설계에서 실패를 다루는 축도 봤다. 마지막으로 복잡한 UI에서 상태를 어떻게 나눠야 하는지로 넘어간다.

## 66. 복잡한 UI에서는 상태를 어떻게 나눠야 하나

단순 폼이나 버튼은 상태 구조가 비교적 간단하다.
하지만 편집기, DnD, 차트, 애니메이션 많은 UI에선 상태를 한 덩어리로 두면 금방 꼬인다.

### 핵심 개념

복잡한 UI에서는 상태를 역할별로 나눠 보는 게 좋다.

| 종류 | 설명 | 관리 위치 |
|------|------|-----------|
| 영속 상태 | 저장 대상 (DB에 반영) | 서버 상태 도구 (React Query 등) |
| 일시 상태 | hover, drag, focus | 로컬 state |
| 파생 상태 | 기존 상태로부터 계산 | `useMemo` 또는 일반 변수 |
| 서버 상태 | 외부에서 동기화 | 서버 상태 도구 |

### 파생 상태의 흔한 안티패턴

```tsx
// Bad: 파생 상태를 useState + useEffect로 동기화
const [filteredItems, setFilteredItems] = useState([]);
useEffect(() => {
  setFilteredItems(items.filter((i) => i.category === selected));
}, [items, selected]);
// 불필요한 state + effect + 한 틱 지연

// Good: useMemo 또는 렌더 중 직접 계산
const filteredItems = useMemo(
  () => items.filter((i) => i.category === selected),
  [items, selected]
);
// state가 아니라 계산값 — 동기화 문제 없음
```

### 자주 하는 오해

- 상태를 한 객체에 모으면 관리가 쉽다 → 규모가 커지면 반대가 되기 쉽다
- 파생값도 전부 state로 둬야 한다 → 계산으로 충분할 수 있다. `useState` + `useEffect`로 동기화하면 오히려 버그 원인이 된다.
- 드래그 중 좌표도 전역으로 두는 게 낫다 → 일시 상태는 더 좁은 범위가 적절하다

### 면접 압축 답변

- **10초**: 복잡한 UI에서는 상태를 영속/일시/파생/서버로 나눠 역할별로 관리합니다.
- **30초**: 저장 대상은 서버 상태 도구로, hover/drag 같은 순간 상태는 로컬로, 기존 상태에서 계산 가능한 값은 `useMemo`나 일반 변수로 처리합니다. 파생 상태를 `useState` + `useEffect`로 동기화하는 건 흔한 안티패턴입니다.

---

## 67. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- React는 선언형 UI 시스템이다
- 상태가 바뀌면 UI를 다시 계산한다 (렌더 → 커밋)
- Fiber 기반 Virtual DOM으로 변경을 최소화한다
- 재조정은 타입과 key로 정체성을 판단한다
- state ownership이 구조 설계의 핵심이다
- Context는 값 전달, Zustand/Redux는 구독 제어 — 축이 다르다
- client state와 server state는 구분해야 한다
- useReducer는 복잡한 상태 전환을 한 곳에 모아 관리한다
- React 18 concurrent features로 렌더링 우선순위를 제어한다
- 에러 바운더리로 실패를 격리한다
- 복잡한 UI일수록 상태를 역할별로 나누는 감각이 필요하다

이번 챕터는 단순히 React API를 배우는 챕터가 아니라,
**"React는 상태와 UI의 관계를 어떻게 계산하고, 그 위에 어떤 구조를 세워야 하는가"**를 배우는 챕터다.
