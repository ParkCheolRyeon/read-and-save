## 1. `this` 결정 규칙과 화살표 함수

> **핵심 요약** — `this`는 함수가 **선언된 위치**가 아니라 **호출된 방식**에 따라 결정된다. 네 가지 규칙(`new` > 명시적 > 암시적 > 기본)으로 우선순위가 정해지며, 화살표 함수는 자체 `this` 바인딩이 없어 선언 시점의 외부 `this`를 그대로 사용한다.

많은 사람이 `this`를 직관적으로 이해하려다 혼동하는 이유는,
같은 함수라도 호출 방식이 달라지면 `this`가 완전히 바뀌기 때문이다.

### `this`의 네 가지 결정 규칙

`this`는 네 가지 규칙으로 결정되며, 우선순위가 있다.

1. **`new` 바인딩** — `new Foo()`로 호출하면 `this`는 새로 생성된 객체
2. **명시적 바인딩** — `call`, `apply`, `bind`로 직접 지정
3. **암시적 바인딩** — `obj.method()`처럼 점 앞의 객체
4. **기본 바인딩** — 위 어디에도 해당하지 않으면 전역 객체 (strict mode에서는 `undefined`)

```js
function greet() {
  console.log(this.name);
}

const user = { name: "Kim", greet };

// 암시적 바인딩 — 점 앞의 객체가 this
user.greet(); // "Kim"

// 기본 바인딩 — 단독 호출이므로 전역 (strict mode면 undefined)
const fn = user.greet;
fn(); // undefined (strict) 또는 전역의 name

// 명시적 바인딩 — call로 this 지정
greet.call({ name: "Lee" }); // "Lee"

// new 바인딩 — 새 객체가 this
function Person(name) {
  this.name = name;
}
const p = new Person("Park");
console.log(p.name); // "Park"
```

### 화살표 함수 vs 일반 함수

화살표 함수는 단순히 짧은 문법이 아니라, **`this` 바인딩 방식 자체가 다르다.**

| 구분 | 일반 함수 | 화살표 함수 |
|------|-----------|-------------|
| `this` | 호출 방식에 따라 결정 | 선언 시점의 외부 스코프 `this`를 그대로 사용 (렉시컬 `this`) |
| `arguments` | 자체 `arguments` 객체 있음 | 없음 (외부 스코프의 `arguments` 참조) |
| `new` | 생성자로 사용 가능 | 사용 불가 (`prototype`이 없음) |
| `prototype` | 있음 | 없음 |

```js
const obj = {
  name: "Kim",
  // 일반 함수 — this는 호출 방식에 따라 결정
  regular() {
    console.log(this.name); // "Kim"
    setTimeout(function () {
      console.log(this.name); // undefined — 콜백의 this는 전역
    }, 100);
  },
  // 화살표 함수 — 선언 시점의 this를 캡처
  arrow() {
    console.log(this.name); // "Kim"
    setTimeout(() => {
      console.log(this.name); // "Kim" — 외부 this를 그대로 사용
    }, 100);
  },
};
```

### 자주 하는 오해

- `this`는 함수가 정의된 객체를 가리킨다 → 틀림. 호출 방식에 따라 달라진다.
- `strict mode`에서도 기본 바인딩이 전역을 가리킨다 → 틀림. `undefined`가 된다.
- `bind`는 반환된 함수의 `this`를 바꿀 수 있다 → 틀림. `bind`된 함수는 다시 `bind`해도 최초 바인딩이 유지된다.
- 화살표 함수는 `this`가 없다 → 틀림. 자체 `this` 바인딩이 없을 뿐, 외부 스코프의 `this`를 사용한다.
- 화살표 함수에 `call`/`bind`를 쓰면 `this`를 바꿀 수 있다 → 틀림. 무시된다.
- 항상 화살표 함수가 낫다 → 틀림. 객체 메서드나 생성자 함수에서는 일반 함수가 필요하다.

### 실무 에피소드 — 이런 상황에서 만난다

> 클래스형 컴포넌트에서 이벤트 핸들러 메서드를 `onClick={this.handleClick}`으로 전달했더니 `this`가 `undefined`가 돼서 `setState`가 터졌다. 원인은 메서드를 콜백으로 넘기면서 암시적 바인딩이 풀린 것이었다. `handleClick = () => {}` 형태로 클래스 필드에서 화살표 함수로 선언하면 렉시컬 `this`가 인스턴스를 캡처해서 해결됐다. 함수형 컴포넌트 + 훅 환경에서는 `this` 문제 자체가 사라지지만, 라이브러리 내부나 vanilla JS 코드에서는 여전히 마주치는 문제다.

### 면접 압축 답변

- **10초**: `this`는 호출 방식에 따라 결정됩니다. `new` > 명시적(`call`/`bind`) > 암시적(`obj.method`) > 기본(전역/`undefined`) 순이며, 화살표 함수는 자체 `this`가 없어 선언 시점의 외부 `this`를 사용합니다.
- **30초**: `this`는 선언 위치가 아니라 호출 시점에 결정됩니다. `new`로 호출하면 새 객체, `call`/`apply`/`bind`면 지정한 객체, 메서드 호출이면 점 앞 객체, 단독 호출이면 전역 또는 `undefined`입니다. 화살표 함수는 렉시컬 스코프의 `this`를 캡처하므로 `setTimeout` 콜백이나 이벤트 핸들러에서 `this` 유실을 막습니다. 다만 `new`로 호출할 수 없고 `prototype`도 없어서 생성자나 메서드 정의에는 일반 함수가 적합합니다.

> `this`와 화살표 함수를 이해했으면, 다음은 JavaScript의 객체 상속 모델인 프로토타입 체인을 봐야 한다. `class` 문법이 내부적으로 어떻게 동작하는지를 알면 `this`와 연결된다.

## 2. 프로토타입 체인과 class 문법의 내부 동작

> **핵심 요약** — JavaScript의 `class`는 프로토타입 기반 상속의 문법적 설탕이다. 메서드는 `prototype` 객체에 저장되고, 프로퍼티 검색은 `[[Prototype]]` 체인을 따라 올라간다.

### 핵심 개념

- 모든 객체는 `[[Prototype]]`이라는 내부 슬롯을 가지며, 이를 통해 다른 객체와 연결된다.
- 프로퍼티를 찾을 때 자신에게 없으면 `[[Prototype]]` 체인을 따라 올라간다.
- `class`의 메서드는 `prototype` 객체에 저장된다.

```js
class Animal {
  constructor(name) {
    this.name = name; // 인스턴스 프로퍼티
  }
  speak() { // Animal.prototype.speak에 저장됨
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  bark() { // Dog.prototype.bark에 저장됨
    return `${this.name} barks`;
  }
}

const dog = new Dog("Rex");

// 프로토타입 체인 확인
console.log(dog.hasOwnProperty("name"));   // true — 인스턴스 자체에 있음
console.log(dog.hasOwnProperty("bark"));   // false — Dog.prototype에 있음
console.log(dog.hasOwnProperty("speak"));  // false — Animal.prototype에 있음

// 체인: dog → Dog.prototype → Animal.prototype → Object.prototype → null

// class는 내부적으로 이것과 같다
function AnimalOld(name) {
  this.name = name;
}
AnimalOld.prototype.speak = function () {
  return `${this.name} makes a sound`;
};
```

### 자주 하는 오해

- JavaScript의 `class`는 Java의 클래스와 같다 → 틀림. 내부적으로 프로토타입 체인이다.
- `extends`는 프로퍼티를 복사한다 → 틀림. 프로토타입 체인을 연결할 뿐이다.
- `instanceof`는 타입을 확인한다 → 정확히는 프로토타입 체인에 해당 생성자의 `prototype`이 있는지 확인한다.

### 실무 에피소드 — 이런 상황에서 만난다

> 서드파티 라이브러리에서 제공하는 클래스를 `extends`로 확장해서 커스텀 에러 타입을 만들었다. 그런데 `catch` 블록에서 `instanceof CustomError`가 `false`를 반환해서 분기가 작동하지 않았다. 원인은 트랜스파일러(Babel)가 클래스 상속을 처리하면서 프로토타입 체인을 제대로 연결하지 못한 것이었다. `Object.setPrototypeOf(this, new.target.prototype)`을 생성자에 추가해서 체인을 수동으로 복구했다. 프로토타입 체인이 문법적 설탕 아래에서 실제로 동작한다는 걸 체감한 순간이었다.

### 면접 압축 답변

- **10초**: JavaScript의 `class`는 프로토타입 기반 상속의 문법적 설탕이며, 메서드는 `prototype` 객체에 저장되고 프로토타입 체인을 따라 검색됩니다.
- **30초**: JavaScript는 클래스 기반이 아니라 프로토타입 기반 상속입니다. `class` 문법을 쓰더라도 내부적으로는 `prototype` 체인으로 동작합니다. 프로퍼티를 찾을 때 자기 자신에 없으면 `[[Prototype]]` 링크를 따라 올라가고, `extends`는 이 체인을 연결해 주는 것이지 복사가 아닙니다.

> 언어의 비교 연산도 JavaScript에서 자주 혼동되는 주제다. `==`와 `===`의 차이를 명확히 잡고 가자.

## 3. `==` vs `===`

> **핵심 요약** — `===`는 타입 변환 없이 값과 타입을 모두 비교하고, `==`는 암시적 타입 변환을 수행한 뒤 비교한다. 실무에서는 예측 가능성을 위해 `===`를 기본으로 사용한다.

### 핵심 개념

- `===` (일치 연산자): 타입 변환 없이 값과 타입을 모두 비교한다.
- `==` (동등 연산자): 타입이 다르면 **암시적 타입 변환**을 수행한 뒤 비교한다.

```js
// === 는 타입까지 비교
1 === 1     // true
1 === "1"   // false — 타입이 다름

// == 는 타입 변환 후 비교
1 == "1"    // true — "1"이 숫자 1로 변환
0 == false  // true — false가 0으로 변환
"" == false // true — 둘 다 0으로 변환
null == undefined // true — 특수 규칙

// 실무에서 주의할 위험한 케이스
0 == ""     // true
0 == "0"    // true
"" == "0"   // false — 둘 다 문자열이라 변환 없이 비교
```

### 실무 원칙

- 기본적으로 항상 `===`를 사용한다.
- `== null`은 `null`과 `undefined`를 동시에 체크할 때 관용적으로 허용하는 경우도 있다.

```js
// null / undefined 동시 체크 — 이 경우만 == 허용하기도 함
if (value == null) {
  // value가 null 또는 undefined
}

// 위와 동일하지만 더 명시적인 방식
if (value === null || value === undefined) {
  // ...
}
```

### 자주 하는 오해

- `==`는 느슨하게 비교해서 편하다 → 위험하다. `0 == ""`이 `true`가 되는 등 의도하지 않은 결과를 만든다.
- `===`만 쓰면 모든 비교 문제가 해결된다 → 객체 비교에서는 `===`도 참조만 비교하므로, 깊은 비교가 필요할 때는 별도 로직이 필요하다.

### 실무 에피소드 — 이런 상황에서 만난다

> API 응답에서 `status` 필드가 숫자 `0`으로 내려왔는데, `if (!response.status)`로 체크하자 "상태 없음"으로 처리돼 버렸다. `0`은 falsy이기 때문이다. `if (response.status === undefined)`로 바꿔야 했다. 암시적 변환이 버그를 만드는 가장 흔한 패턴이고, 이런 미묘한 차이를 알고 있으면 디버깅 시간이 줄어든다.

### 면접 압축 답변

- **10초**: `===`는 타입 변환 없이 값과 타입을 모두 비교하고, `==`는 암시적 타입 변환 후 비교합니다. 실무에서는 `===`를 기본으로 씁니다.
- **30초**: `==`는 비교 전에 암시적 타입 변환이 일어나서 `0 == ""`이 `true`가 되는 등 예측하기 어려운 결과가 나옵니다. `===`는 타입 변환 없이 비교하므로 버그를 줄일 수 있습니다. 예외적으로 `value == null`로 `null`과 `undefined`를 동시에 체크하는 관용 패턴이 있지만, 팀 컨벤션에 따릅니다.

> 비교 연산을 잡았으면, 다음은 DOM에서의 이벤트 흐름이다. 이벤트가 어떻게 전파되는지를 모르면 React의 이벤트 위임 방식도 이해하기 어렵다.

## 4. 이벤트 버블링 / 캡처링 / 이벤트 위임

> **핵심 요약** — DOM 이벤트는 캡처링 → 타깃 → 버블링 순으로 전파된다. 이벤트 위임은 이 버블링을 활용해 부모 요소에 핸들러 하나를 등록하는 패턴이며, React도 내부적으로 이 방식을 사용한다.

### 핵심 개념

DOM 이벤트는 세 단계로 전파된다.

1. **캡처링 단계** — 최상위(`window`)에서 타깃 요소까지 내려옴
2. **타깃 단계** — 이벤트가 실제로 발생한 요소에 도달
3. **버블링 단계** — 타깃에서 다시 최상위로 올라감

```js
// 버블링 (기본) — 자식 클릭 시 부모 핸들러도 실행됨
document.getElementById("parent").addEventListener("click", () => {
  console.log("parent clicked");
});
document.getElementById("child").addEventListener("click", () => {
  console.log("child clicked");
});
// child 클릭 시: "child clicked" → "parent clicked"

// 캡처링 — 세 번째 인자를 true로 설정
document.getElementById("parent").addEventListener(
  "click",
  () => console.log("parent capture"),
  true // 캡처링 단계에서 실행
);
// child 클릭 시: "parent capture" → "child clicked" → "parent clicked"

// 이벤트 전파 중단
element.addEventListener("click", (e) => {
  e.stopPropagation(); // 캡처링/버블링 양방향 전파를 중단
});
```

### 이벤트 위임

이벤트 위임은 **버블링을 이용해 부모 요소 하나에만 핸들러를 등록**하는 패턴이다.
자식 요소가 많거나 동적으로 추가되는 경우 매우 유용하다.

```js
// 이벤트 위임 — 리스트 아이템이 100개여도 핸들러는 하나
document.getElementById("list").addEventListener("click", (e) => {
  const item = e.target.closest("li");
  if (!item) return;
  console.log("clicked item:", item.dataset.id);
});
```

### React에서의 이벤트 위임

React는 내부적으로 이벤트 위임을 사용한다. React 17부터는 `document`가 아닌 **루트 DOM 컨테이너**에 이벤트를 위임한다. 이 덕분에 여러 React 앱이 한 페이지에 공존할 수 있다.

```jsx
// React에서는 합성 이벤트(SyntheticEvent)를 사용
function List({ items }) {
  // 이 핸들러는 React가 루트에 위임한 핸들러를 통해 호출됨
  const handleClick = (e) => {
    console.log("clicked:", e.target.textContent);
  };

  return (
    <ul onClick={handleClick}>
      {items.map((item) => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  );
}
```

### 자주 하는 오해

- 모든 이벤트가 버블링된다 → 틀림. `focus`, `blur`, `mouseenter`, `mouseleave` 등은 버블링되지 않는다. 버블링되는 대안(`focusin`, `focusout`, `mouseover`, `mouseout`)을 써야 한다.
- `e.preventDefault()`가 전파를 막는다 → 틀림. 기본 동작(링크 이동, 폼 제출 등)만 막는다. 전파를 막으려면 `e.stopPropagation()`이 필요하다.

### 실무 에피소드 — 이런 상황에서 만난다

> 테이블의 각 행에 클릭 핸들러를 달았는데, 행이 500개가 넘으니 스크롤 성능이 저하되었다. `<tbody>`에 이벤트 위임으로 핸들러 하나만 등록하고 `e.target.closest('tr')`로 클릭된 행을 판별하자, 핸들러 등록 수가 500개에서 1개로 줄었고 메모리 사용량이 눈에 띄게 감소했다. 행이 동적으로 추가/삭제되는 경우에도 별도 핸들러 관리가 필요 없어져서 코드도 단순해졌다.

### 면접 압축 답변

- **10초**: DOM 이벤트는 캡처링 → 타깃 → 버블링 순으로 전파됩니다. 이벤트 위임은 버블링을 이용해 부모에 핸들러를 하나만 등록하는 패턴입니다.
- **30초**: DOM 이벤트는 최상위에서 타깃까지 내려오는 캡처링, 타깃에 도달하는 타깃 단계, 다시 올라가는 버블링 세 단계로 전파됩니다. 이벤트 위임은 이 버블링을 활용해 부모 요소에 핸들러 하나를 등록하고 `e.target`으로 실제 클릭 요소를 판별하는 패턴입니다. 동적으로 생성되는 요소에도 별도 핸들러 등록이 불필요하고, React도 내부적으로 이 방식을 사용합니다.

> 마지막으로 JavaScript의 내장 컬렉션 비교를 정리하면 JS 언어 파트 보강이 거의 마무리된다.

## 5. `Map` vs `Object`, `Set` vs `Array`

> **핵심 요약** — `Map`은 모든 타입을 키로 쓸 수 있고 삽입 순서가 보장되며, `Set`은 중복을 허용하지 않고 `has` 검색이 O(1)이다. `Object`/`Array`로 충분한 경우가 많지만, 키 타입이 다양하거나 대량 존재 확인이 필요하면 `Map`/`Set`이 적합하다.

### 핵심 개념

**Map vs Object**

| 구분 | Object | Map |
|------|--------|-----|
| 키 타입 | 문자열/심볼만 | 모든 타입 (객체, 함수 등) |
| 순서 보장 | ES2015+ 정수 키 먼저, 나머지 삽입 순 | 삽입 순서 보장 |
| 크기 확인 | `Object.keys(obj).length` | `map.size` |
| 순회 | `for...in`, `Object.entries()` | `for...of`, `map.forEach()` |
| 성능 | 소규모 데이터에 적합 | 빈번한 추가/삭제에 유리 |

```js
// Map — 키로 객체 사용 가능
const map = new Map();
const userA = { id: 1 };
map.set(userA, "admin");
console.log(map.get(userA)); // "admin"
console.log(map.size);       // 1

// Object — 키가 항상 문자열로 변환됨
const obj = {};
obj[userA] = "admin";
console.log(Object.keys(obj)); // ["[object Object]"]
```

**Set vs Array**

| 구분 | Array | Set |
|------|-------|-----|
| 중복 | 허용 | 불허 |
| 검색 | `indexOf`/`includes` — O(n) | `has` — O(1) |
| 순서 | 인덱스 기반 | 삽입 순서 |

```js
// Set — 중복 제거
const arr = [1, 2, 2, 3, 3, 3];
const unique = [...new Set(arr)]; // [1, 2, 3]

// Set — 빠른 존재 확인
const allowedIds = new Set([101, 202, 303]);
if (allowedIds.has(userId)) {
  // O(1) 조회
}
```

### 자주 하는 오해

- `Map`은 항상 `Object`보다 빠르다 → 틀림. 소규모 데이터에서는 `Object`가 엔진 최적화를 더 잘 받는다. `Map`은 키가 다양하거나 빈번한 추가/삭제가 있을 때 유리하다.
- `Set`에 객체를 넣으면 내용이 같은 객체가 중복 제거된다 → 틀림. `Set`은 참조 동일성으로 비교하므로 `{ id: 1 }`과 `{ id: 1 }`은 서로 다른 값으로 취급된다.

### 실무 에피소드 — 이런 상황에서 만난다

> 관리자 페이지에서 "선택된 항목"을 배열로 관리하고 `includes`로 체크했는데, 항목이 수천 개로 늘어나자 체크박스 토글 시 체감 지연이 발생했다. 선택 상태를 `Set`으로 바꾸고 `has`/`add`/`delete`를 사용하자, 조회가 O(1)이 되어 지연이 사라졌다. 작은 리스트에서는 차이가 없지만, 대량 데이터에서는 자료구조 선택이 UX에 직접 영향을 준다.

### 면접 압축 답변

- **10초**: `Map`은 모든 타입을 키로 쓸 수 있고 삽입 순서가 보장되며, `Set`은 중복을 허용하지 않고 `has`가 O(1)입니다.
- **30초**: `Object`는 키가 문자열/심볼로 제한되고 `Map`은 모든 값을 키로 사용할 수 있어서 DOM 노드나 객체를 키로 매핑할 때 유용합니다. `Set`은 중복을 자동으로 배제하고 `has` 검색이 O(1)이라 대량 데이터에서 존재 여부 확인에 적합합니다. 실무에서는 배열 중복 제거나 ID 기반 빠른 조회에 `Set`을 자주 씁니다.

## 6. 접근성(a11y) 기초

> **핵심 요약** — 웹 접근성의 기본은 시맨틱 HTML을 올바르게 사용하는 것이다. HTML만으로 부족할 때 ARIA 속성을 보완적으로 쓰고, 모달 같은 인터랙티브 UI에서는 포커스 트랩으로 키보드 접근성을 확보해야 한다.

웹 접근성은 단순한 선택이 아니라, 법적 준수 요건이자 더 넓은 사용자 도달 범위를 확보하는 수단이다. 면접에서도 "접근성을 어떻게 챙기는가"는 시니어 레벨에서 자주 등장하는 질문이다.

### 핵심 개념

**시맨틱 HTML**

시맨틱 태그를 사용하면 스크린 리더가 페이지 구조를 정확히 파악할 수 있다.

| 잘못된 사용 | 올바른 사용 | 이유 |
|------------|-----------|------|
| `<div onclick>` | `<button>` | 키보드 이벤트, 포커스, role이 자동 부여 |
| `<div class="nav">` | `<nav>` | 스크린 리더가 내비게이션 영역으로 인식 |
| `<div class="main">` | `<main>` | 페이지의 주요 콘텐츠 영역을 명시 |
| `<div class="article">` | `<article>` | 독립적으로 배포/재사용 가능한 콘텐츠 |
| `<div class="section">` | `<section>` | 관련 콘텐츠의 논리적 그룹 |

**ARIA 속성**

HTML만으로 의미를 전달할 수 없을 때 ARIA(Accessible Rich Internet Applications) 속성을 사용한다.

- `role` — 요소의 역할을 명시 (`role="dialog"`, `role="alert"`)
- `aria-label` — 시각적 텍스트가 없는 요소에 이름 부여 (`<button aria-label="닫기">×</button>`)
- `aria-hidden="true"` — 스크린 리더에서 숨김 (장식용 아이콘 등)
- `aria-live="polite"` — 동적으로 변경되는 콘텐츠를 스크린 리더에 알림

**키보드 네비게이션**

- `tabIndex={0}` — 자연스러운 탭 순서에 포함
- `tabIndex={-1}` — 프로그래밍적으로만 포커스 가능 (탭 순서에서 제외)
- 모달 내부에서 **포커스 트랩(focus trap)** 을 구현해야 탭 키가 모달 밖으로 나가지 않는다

```jsx
import { useEffect, useRef, useCallback } from "react";

function AccessibleModal({ isOpen, onClose, children }) {
  const modalRef = useRef(null);
  const previousFocusRef = useRef(null);

  // 모달 열릴 때 이전 포커스 저장 + 모달에 포커스
  useEffect(() => {
    if (isOpen) {
      previousFocusRef.current = document.activeElement;
      modalRef.current?.focus();
    }
    return () => {
      // 모달 닫힐 때 이전 포커스 복원
      previousFocusRef.current?.focus();
    };
  }, [isOpen]);

  // 포커스 트랩 — Tab 키가 모달 내부에서만 순환
  const handleKeyDown = useCallback(
    (e) => {
      if (e.key === "Escape") {
        onClose();
        return;
      }
      if (e.key !== "Tab") return;

      const focusableElements = modalRef.current?.querySelectorAll(
        'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
      );
      if (!focusableElements || focusableElements.length === 0) return;

      const first = focusableElements[0];
      const last = focusableElements[focusableElements.length - 1];

      if (e.shiftKey) {
        // Shift+Tab — 첫 번째 요소에서 마지막으로
        if (document.activeElement === first) {
          e.preventDefault();
          last.focus();
        }
      } else {
        // Tab — 마지막 요소에서 첫 번째로
        if (document.activeElement === last) {
          e.preventDefault();
          first.focus();
        }
      }
    },
    [onClose]
  );

  if (!isOpen) return null;

  return (
    <div
      className="modal-overlay"
      role="dialog"
      aria-modal="true"
      aria-label="모달 다이얼로그"
      ref={modalRef}
      tabIndex={-1}
      onKeyDown={handleKeyDown}
    >
      <div className="modal-content">
        {children}
        <button onClick={onClose} aria-label="모달 닫기">
          닫기
        </button>
      </div>
    </div>
  );
}
```

### 자주 하는 오해

- ARIA를 많이 쓸수록 접근성이 좋다 → 틀림. 시맨틱 HTML로 충분하면 ARIA는 불필요하다. 과도한 ARIA는 오히려 혼란을 준다 ("No ARIA is better than bad ARIA").
- `tabIndex={1}` 이상의 양수 값을 쓰면 포커스 순서를 제어할 수 있다 → 기술적으로 가능하지만 관리가 어렵고 예측 불가능한 탭 순서를 만든다. 0 또는 -1만 사용하라.
- 접근성은 시각장애인만을 위한 것이다 → 틀림. 키보드만 사용하는 사용자, 운동 장애 사용자, 고령 사용자 등 다양한 대상을 포함한다.

### 실무 에피소드 — 이런 상황에서 만난다

> 디자인 시스템의 모달 컴포넌트를 만들면서, 처음에는 Escape 키와 포커스 트랩을 빠트렸다. QA 과정에서 키보드만 사용하는 테스터가 모달이 열린 상태에서 Tab 키를 누르자 포커스가 모달 뒤의 페이지로 빠져나가는 것을 발견했다. 포커스 트랩을 추가하고 모달이 닫힐 때 이전 포커스를 복원하는 로직을 넣은 뒤에야, WCAG 2.1 AA 기준을 충족할 수 있었다. 접근성은 "나중에 할 일"이 아니라 컴포넌트 설계의 기본 요구사항이다.

### 면접 압축 답변

- **10초**: 시맨틱 HTML을 기본으로 사용하고, 부족한 부분은 ARIA 속성으로 보완합니다. 모달에서는 포커스 트랩을 구현해 키보드 접근성을 확보합니다.
- **30초**: 접근성의 기본은 `<button>`, `<nav>`, `<main>` 같은 시맨틱 태그를 올바르게 사용하는 것입니다. HTML만으로 의미 전달이 어려울 때 `role`, `aria-label`, `aria-live` 같은 ARIA 속성을 보완적으로 씁니다. 모달 같은 인터랙티브 UI에서는 포커스 트랩으로 키보드가 모달 밖으로 나가지 않게 하고, Escape로 닫을 수 있어야 합니다. 이는 법적 준수(WCAG) 요건이기도 하며, 면접에서 접근성을 챙기는 개발자인지를 확인하는 빈출 질문입니다.

## 7. TypeScript 실전 디테일 — 옵셔널, 오버로드, enum 대안

> **핵심 요약** — `age?`는 키 자체가 없어도 되지만 `age: number | undefined`는 키가 반드시 있어야 한다. 함수 오버로드는 입력 타입별로 반환 타입을 정확히 좁힐 때 쓰고, 상수 집합은 `enum`보다 union literal이나 `as const` 객체가 프론트엔드에서 더 유리하다.

이 섹션은 세 가지 실전 디테일을 한 번에 다룬다. 각각이 면접에서 독립적으로 나오기도 하지만, 모두 **"타입을 얼마나 정밀하게 설계하는가"**라는 같은 축에 놓여 있다.

### 7-1. 옵셔널 프로퍼티 `?`와 `undefined` 허용의 차이

이 둘은 비슷해 보이지만 의미가 다르다.

```ts
// 옵셔널 프로퍼티 — 키 자체가 없어도 됨
// 주의: 아래 구분은 tsconfig의 exactOptionalPropertyTypes: true일 때 명확해진다.
// 기본 설정에서는 age?: number도 undefined 할당이 가능하다.
interface UserA {
  name: string;
  age?: number; // { name: "Kim" } — OK (키 자체 생략 가능)
}

// undefined 허용 — 키는 반드시 있어야 하고 값이 undefined일 수 있음
interface UserB {
  name: string;
  age: number | undefined; // { name: "Kim" } — Error! age 키 필요
}

const a: UserA = { name: "Kim" };             // OK
const b: UserB = { name: "Kim" };             // Error
const c: UserB = { name: "Kim", age: undefined }; // OK
```

**실무에서 왜 구분이 중요한가**

- API 응답에서 **필드가 아예 안 오는 것**과 **`null`/`undefined`로 오는 것**은 다르다.
- `Partial<T>`은 모든 키를 옵셔널로 만들어서 "키 없음"을 허용하지만, 명시적으로 `undefined`를 보내야 하는 PATCH 요청 등에서는 `| undefined`가 더 정확하다.

```ts
// PATCH 요청 — 명시적으로 undefined를 보내면 "이 필드를 지워라"
interface PatchUser {
  name?: string;
  age?: number | undefined; // 명시적 삭제 의도 표현 가능
}
```

### 7-2. 함수 오버로드 vs 유니온 파라미터

```ts
// 유니온 파라미터 — 간단하지만 반환 타입이 넓어짐
function format(value: string | number): string {
  if (typeof value === "string") return value.trim();
  return value.toFixed(2);
}
// format("hello")의 반환 타입도 string, format(42)도 string — 이 경우는 괜찮음

// 오버로드 — 입력에 따라 반환 타입을 정확히 좁힐 수 있음
function parse(input: string): string[];
function parse(input: number): number[];
function parse(input: string | number): string[] | number[] {
  if (typeof input === "string") return input.split(",");
  return [input];
}

const a = parse("a,b,c"); // string[] — 정확히 좁혀짐
const b = parse(42);      // number[] — 정확히 좁혀짐
```

**언제 오버로드를 쓰는가**

- **입력 타입에 따라 반환 타입이 달라질 때** 오버로드가 유리하다.
- 반환 타입이 동일하면 유니온 파라미터가 더 간결하다.
- 오버로드가 3개 이상 늘어나면 제네릭으로 대체할 수 있는지 검토한다.

### 7-3. `enum` vs union literal vs `const` 객체 패턴 비교

```ts
// 1. enum — 런타임에 객체가 생성됨
enum StatusEnum {
  PENDING = "PENDING",
  APPROVED = "APPROVED",
  REJECTED = "REJECTED",
}
// 컴파일 후: var StatusEnum; (function(StatusEnum){...})(StatusEnum || ...)

// 2. union literal — 런타임 코드 없음 (타입만)
type StatusUnion = "PENDING" | "APPROVED" | "REJECTED";

// 3. const 객체 + as const — 런타임 객체 + 타입 안전
const STATUS = {
  PENDING: "PENDING",
  APPROVED: "APPROVED",
  REJECTED: "REJECTED",
} as const;
type StatusConst = (typeof STATUS)[keyof typeof STATUS];
// "PENDING" | "APPROVED" | "REJECTED"
```

| 구분 | `enum` | union literal | `const` 객체 |
|------|--------|---------------|--------------|
| 런타임 존재 | O (객체 생성) | X (타입만) | O (일반 객체) |
| Tree-shaking | 어려움 | 해당 없음 | 쉬움 |
| 값 → 타입 매핑 | 자동 | 수동 | `as const` + `typeof` |
| 순회 가능 | `Object.values(StatusEnum)` | 불가 | `Object.values(STATUS)` |

**실무 추천**

- **번들 크기를 신경 쓰는 프론트엔드**: union literal 또는 `const` 객체를 선호
- **값 순회가 필요하면**: `const` 객체
- **`enum`을 써야 한다면**: `const enum`으로 인라인 가능하지만, isolatedModules 환경에서는 제한이 있음

### 자주 하는 오해

- 옵셔널(`?`)과 `| undefined`는 같다 → `exactOptionalPropertyTypes: true` 설정에서는 다르다. 옵셔널은 키 생략을 허용하고, `| undefined`는 키가 있되 값이 `undefined`인 경우를 허용한다.
- 오버로드가 항상 유니온보다 낫다 → 틀림. 반환 타입이 같으면 유니온이 더 간결하고 유지보수가 쉽다.
- `enum`은 TS의 모범 사례다 → 프론트엔드에서는 번들 크기와 tree-shaking 때문에 `as const` 객체나 union literal이 선호되는 추세다.

### 실무 에피소드 — 이런 상황에서 만난다

> API 설계 시 `status` 필드의 가능한 값을 `enum`으로 정의했더니, 프론트엔드 번들에 `enum` 런타임 코드가 포함되면서 tree-shaking이 안 되는 문제가 발생했다. `as const` 객체로 바꾸자 번들에서 사용하지 않는 상태값이 제거됐고, `Object.values(STATUS)`로 드롭다운 옵션도 자동 생성할 수 있었다. PATCH API에서는 `age?: number | undefined` 패턴을 써서 "이 필드를 명시적으로 삭제한다"는 의도를 타입으로 표현했다.

### 면접 압축 답변

- **10초**: `age?`는 키 생략 허용, `age: number | undefined`는 키 필수 + 값만 `undefined` 허용입니다. 오버로드는 입력별 반환 타입을 좁힐 때, `as const` 객체는 `enum` 대신 번들 친화적 상수 패턴으로 씁니다.
- **30초**: 옵셔널 프로퍼티(`?`)는 키 자체를 생략할 수 있지만, `| undefined`는 키가 반드시 있되 값만 `undefined`를 허용합니다. 함수 오버로드는 입력-출력 타입 조합을 명시해서 호출부의 타입 추론을 정확하게 만들고, 반환 타입이 같으면 유니온이 더 간결합니다. 상수 집합은 `enum` 대신 `as const` 객체를 쓰면 런타임 값 순회와 타입 안전을 동시에 얻으면서 tree-shaking도 가능합니다.

## 8. `useEffect` vs `useLayoutEffect`

> **핵심 요약** — `useEffect`는 브라우저가 paint한 후 비동기로 실행되어 대부분의 사이드 이펙트에 적합하고, `useLayoutEffect`는 paint 전 동기로 실행되어 DOM 측정 후 즉시 반영이 필요한 경우에 쓴다. SSR 환경에서는 `useIsomorphicLayoutEffect` 패턴으로 경고를 피한다.

### 핵심 개념

둘 다 사이드 이펙트를 실행하지만 **실행 시점**이 다르다.

| 구분 | `useEffect` | `useLayoutEffect` |
|------|-------------|-------------------|
| 실행 시점 | 브라우저가 **paint한 후** | DOM 변경 후 브라우저가 **paint하기 전** |
| 동기/비동기 | 비동기적 (paint를 막지 않음) | 동기적 (paint를 막음) |
| 주 용도 | 데이터 fetching, 구독, 로깅 | DOM 측정, 레이아웃 기반 조정 |

```jsx
import { useEffect, useLayoutEffect, useRef, useState } from "react";

function Tooltip({ text, targetRect }) {
  const ref = useRef(null);
  const [position, setPosition] = useState({ top: 0, left: 0 });

  // useLayoutEffect — paint 전에 위치를 계산해야 깜빡임이 없다
  useLayoutEffect(() => {
    if (!ref.current || !targetRect) return;
    const { height } = ref.current.getBoundingClientRect();
    setPosition({
      top: targetRect.top - height - 8,
      left: targetRect.left,
    });
  }, [targetRect]);

  return (
    <div ref={ref} style={{ position: "fixed", ...position }}>
      {text}
    </div>
  );
}
```

### 보충: `useIsomorphicLayoutEffect` 패턴

SSR 환경에서 `useLayoutEffect`를 사용하면 React가 경고를 발생시킨다. 서버에는 DOM이 없기 때문이다. 이를 해결하는 실무 패턴이 `useIsomorphicLayoutEffect`다.

```ts
import { useEffect, useLayoutEffect } from "react";

const useIsomorphicLayoutEffect =
  typeof window !== "undefined" ? useLayoutEffect : useEffect;

export default useIsomorphicLayoutEffect;
```

- 브라우저 환경(`typeof window !== 'undefined'`)에서는 `useLayoutEffect`를 사용해 paint 전 동기 실행
- 서버 환경에서는 `useEffect`로 폴백해 경고를 방지
- 많은 라이브러리(framer-motion, radix-ui 등)가 내부적으로 이 패턴을 사용한다

```tsx
// 사용 예 — SSR 안전한 레이아웃 측정
function ResponsiveComponent() {
  const ref = useRef<HTMLDivElement>(null);
  const [width, setWidth] = useState(0);

  useIsomorphicLayoutEffect(() => {
    if (ref.current) {
      setWidth(ref.current.getBoundingClientRect().width);
    }
  }, []);

  return <div ref={ref}>Width: {width}px</div>;
}
```

### 자주 하는 오해

- `useLayoutEffect`를 쓰면 성능이 나빠진다 → 부분적으로 맞지만, DOM 측정 후 즉시 반영해야 하는 경우에는 `useEffect`가 오히려 깜빡임을 유발한다.
- SSR에서 `useLayoutEffect`를 쓰면 안 된다 → 경고가 발생한다. SSR 환경에서는 조건부로 `useEffect`를 사용하거나, 서버에서는 건너뛰는 처리가 필요하다.

### 실무 에피소드 — 이런 상황에서 만난다

> 드롭다운 메뉴의 위치를 버튼 아래에 맞추는 로직을 `useEffect`로 구현했더니, 처음 렌더 시 드롭다운이 잠깐 왼쪽 상단에 나타났다가 올바른 위치로 이동하는 깜빡임이 있었다. `useLayoutEffect`로 바꾸자 paint 전에 위치가 계산되어 깜빡임이 사라졌다. 이후 Next.js로 SSR을 적용하면서 `useIsomorphicLayoutEffect`로 교체해 서버 경고도 해결했다.

### 면접 압축 답변

- **10초**: `useEffect`는 paint 후 비동기 실행, `useLayoutEffect`는 paint 전 동기 실행입니다. DOM 측정 후 위치를 바로 반영할 때 `useLayoutEffect`를 씁니다.
- **30초**: `useEffect`는 브라우저가 화면을 그린 후 비동기로 실행되어 대부분의 사이드 이펙트에 적합합니다. `useLayoutEffect`는 DOM 변경 후 paint 전에 동기로 실행되어서, 요소 크기를 측정하고 위치를 즉시 조정하는 등 깜빡임 없는 레이아웃 계산에 씁니다. 다만 동기 실행이라 무거운 로직을 넣으면 paint가 지연되므로 꼭 필요한 경우에만 사용합니다.

## 9. 디자인 패턴 — HOC, Render Props, Compound Components

> **핵심 요약** — HOC는 컴포넌트를 감싸 횡단 관심사를 처리하고, Render Props는 렌더링을 위임하며, Compound Components는 암시적 상태 공유로 선언형 API를 만든다. React 16.8 이후 커스텀 훅이 HOC/Render Props 대부분을 대체했지만, Compound Components는 여전히 유효하다.

React에서 컴포넌트 로직을 재사용하는 패턴은 시대에 따라 변해왔다. 세 가지 클래식 패턴의 원리를 이해하면, 왜 현대 React가 훅 중심으로 전환됐는지도 자연스럽게 이해된다.

### 핵심 개념

**HOC (Higher-Order Component)**

컴포넌트를 인자로 받아 **새로운 컴포넌트를 반환**하는 함수다. 로깅, 인증, 데이터 로딩 같은 **횡단 관심사(cross-cutting concern)** 에 적합하다.

```jsx
// HOC — 인증 확인 로직을 감싸기
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const { user, isLoading } = useAuth();

    if (isLoading) return <Spinner />;
    if (!user) return <Navigate to="/login" />;

    return <WrappedComponent {...props} user={user} />;
  };
}

// 사용
const ProtectedDashboard = withAuth(Dashboard);
// <ProtectedDashboard /> — 인증된 사용자만 Dashboard를 볼 수 있음
```

**Render Props**

컴포넌트가 **함수를 prop(또는 children)으로 받아** 렌더링을 위임하는 패턴이다. HOC보다 유연한 렌더링 제어가 가능하다.

```jsx
// Render Props — 마우스 위치를 추적하는 컴포넌트
function MouseTracker({ children }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener("mousemove", handleMove);
    return () => window.removeEventListener("mousemove", handleMove);
  }, []);

  return children(position);
}

// 사용 — 소비자가 렌더링 방식을 결정
function App() {
  return (
    <MouseTracker>
      {({ x, y }) => (
        <div>
          마우스 위치: {x}, {y}
        </div>
      )}
    </MouseTracker>
  );
}
```

**Compound Components**

여러 컴포넌트가 **암시적으로 상태를 공유**하며 함께 동작하는 패턴이다. HTML의 `<select>` + `<option>` 관계처럼 자연스러운 API를 만들 수 있다.

```jsx
// Compound Components — Accordion 예제
const AccordionContext = createContext(null);

function Accordion({ children }) {
  const [openIndex, setOpenIndex] = useState(null);
  const toggle = (index) =>
    setOpenIndex((prev) => (prev === index ? null : index));

  return (
    <AccordionContext.Provider value={{ openIndex, toggle }}>
      <div className="accordion">{children}</div>
    </AccordionContext.Provider>
  );
}

function AccordionItem({ index, title, children }) {
  const { openIndex, toggle } = useContext(AccordionContext);
  const isOpen = openIndex === index;

  return (
    <div>
      <button onClick={() => toggle(index)} aria-expanded={isOpen}>
        {title}
      </button>
      {isOpen && <div className="accordion-content">{children}</div>}
    </div>
  );
}

// 사용 — 직관적인 선언형 API
function FAQ() {
  return (
    <Accordion>
      <AccordionItem index={0} title="React란?">
        UI 라이브러리입니다.
      </AccordionItem>
      <AccordionItem index={1} title="Virtual DOM이란?">
        실제 DOM의 가벼운 복사본입니다.
      </AccordionItem>
    </Accordion>
  );
}
```

### 언제 무엇을 쓰는가

| 패턴 | 적합한 상황 | 단점 |
|------|-----------|------|
| HOC | 인증, 로깅 등 횡단 관심사 | props 충돌 가능, wrapper hell |
| Render Props | 소비자가 렌더링을 유연하게 제어할 때 | JSX 깊어짐, 가독성 저하 |
| Compound Components | 관련 UI 조각이 암시적 상태를 공유할 때 | Context 의존, 구조가 복잡해질 수 있음 |

### 현대적 대안: 훅

React 16.8 이후 **커스텀 훅**이 HOC와 Render Props의 대부분의 사용 사례를 대체했다. 로직 재사용이 컴포넌트 트리 구조에 영향을 주지 않기 때문이다.

```jsx
// 위의 MouseTracker를 훅으로 대체
function useMousePosition() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener("mousemove", handleMove);
    return () => window.removeEventListener("mousemove", handleMove);
  }, []);

  return position;
}

// 사용 — wrapper 없이 깔끔
function App() {
  const { x, y } = useMousePosition();
  return <div>마우스 위치: {x}, {y}</div>;
}
```

다만 Compound Components는 훅으로 대체되지 않는다. 선언형 API를 만드는 패턴이므로 여전히 유효하다.

### 자주 하는 오해

- HOC와 Render Props는 더 이상 쓸 일이 없다 → 틀림. Compound Components는 여전히 유효하고, 라이브러리 내부에서도 HOC 패턴을 쓴다 (e.g., `React.memo`, `React.forwardRef`).
- 훅이 모든 것을 대체한다 → 대부분의 로직 재사용은 대체하지만, 컴포넌트 구조적 패턴(Compound Components)은 훅과 병행한다.

### 실무 에피소드 — 이런 상황에서 만난다

> 사내 디자인 시스템에서 `<Select>` 컴포넌트를 만들 때, 처음에는 `options` prop으로 배열을 넘기는 방식으로 구현했다. 그런데 옵션에 아이콘, 뱃지, 구분선 같은 커스텀 렌더링이 필요해지면서 prop이 점점 복잡해졌다. Compound Components 패턴으로 `<Select>`, `<Select.Option>`, `<Select.Group>`으로 분리하자, 소비자가 렌더링을 자유롭게 조합하면서도 상태 공유는 내부 Context로 자동 처리됐다. API가 직관적이어서 팀원들의 사용 실수도 줄었다.

### 면접 압축 답변

- **10초**: HOC는 컴포넌트를 감싸 횡단 관심사를 처리하고, Render Props는 렌더링을 위임하며, Compound Components는 암시적 상태 공유로 선언형 API를 만듭니다. 현재는 커스텀 훅이 HOC/Render Props 대부분을 대체합니다.
- **30초**: HOC는 인증이나 로깅 같은 공통 로직을 컴포넌트에 주입할 때, Render Props는 소비자가 렌더링 방식을 유연하게 결정해야 할 때, Compound Components는 Select+Option처럼 관련 컴포넌트가 상태를 암시적으로 공유할 때 씁니다. React 16.8 이후 커스텀 훅이 HOC와 Render Props의 로직 재사용 역할을 대부분 가져갔지만, Compound Components는 선언형 API 설계 패턴이므로 여전히 유효합니다.

## 10. CSS 아키텍처 — Modules, CSS-in-JS, Tailwind

> **핵심 요약** — CSS Modules는 빌드 타임 스코프 격리로 런타임 비용이 없고, CSS-in-JS는 동적 스타일링이 자유롭지만 런타임 비용이 있으며, Tailwind는 유틸리티 클래스 조합으로 디자인 일관성과 빠른 개발이 가능하다. 프로젝트 상황에 따라 선택이 달라지며, 최근에는 SSR/RSC 호환성을 고려해 빌드 타임 솔루션을 선호하는 추세다.

스타일링 방식 선택은 프로젝트 규모, 팀 구조, 성능 요구에 따라 달라진다.

### 핵심 개념

**CSS Modules**

파일 단위로 클래스명을 자동 해시하여 **스코프를 격리**한다. 빌드 타임에 처리되므로 런타임 오버헤드가 없다.

```css
/* Button.module.css */
.button {
  background: #3b82f6;
  color: white;
  padding: 8px 16px;
  border: none;
  border-radius: 4px;
  cursor: pointer;
}

.button:hover {
  background: #2563eb;
}

.primary {
  background: #3b82f6;
}

.secondary {
  background: #6b7280;
}
```

```jsx
// Button.jsx — CSS Modules 사용
import styles from "./Button.module.css";

function Button({ variant = "primary", children, ...props }) {
  return (
    <button
      className={`${styles.button} ${styles[variant]}`}
      {...props}
    >
      {children}
    </button>
  );
}

// 렌더 결과: <button class="Button_button_x7ks2 Button_primary_a3f1k">
// 클래스명이 자동 해시되어 다른 컴포넌트와 충돌하지 않음
```

**CSS-in-JS (styled-components, emotion)**

JavaScript 안에서 스타일을 작성하여 **동적 스타일링**이 쉽다. 다만 런타임에 스타일을 생성하므로 성능 비용이 있다.

```jsx
// styled-components 예시
import styled from "styled-components";

const Button = styled.button`
  background: ${(props) => (props.primary ? "#3b82f6" : "#6b7280")};
  color: white;
  padding: 8px 16px;
  border: none;
  border-radius: 4px;
  cursor: pointer;

  &:hover {
    opacity: 0.9;
  }
`;

// 사용: <Button primary>확인</Button>
```

**Tailwind CSS**

유틸리티 클래스를 조합하는 방식이다. 디자인 시스템이 클래스명에 내장되어 있어 일관성을 유지하기 쉽다.

```jsx
// Tailwind 예시
function Button({ primary, children, ...props }) {
  return (
    <button
      className={`px-4 py-2 rounded text-white cursor-pointer ${
        primary ? "bg-blue-500 hover:bg-blue-600" : "bg-gray-500 hover:bg-gray-600"
      }`}
      {...props}
    >
      {children}
    </button>
  );
}
```

### CSS Specificity (우선순위)

스타일이 충돌할 때 어떤 규칙이 이기는지를 결정하는 기준이다.

```
!important > inline style > ID (#) > class (.) / attribute / pseudo-class > element (div, p)
```

- `!important`는 거의 모든 것을 이기지만, 디버깅을 어렵게 만드므로 **최후의 수단**으로만 사용한다.
- 같은 specificity라면 나중에 선언된 규칙이 이긴다.

### 비교

| 구분 | CSS Modules | CSS-in-JS | Tailwind |
|------|-------------|-----------|----------|
| 스코프 격리 | 자동 (해시) | 자동 (고유 클래스) | 유틸리티라 충돌 적음 |
| 런타임 비용 | 없음 | 있음 (스타일 생성) | 없음 |
| 동적 스타일 | 클래스 토글 | prop 기반 자유자재 | 클래스 조건부 조합 |
| 번들 크기 | 사용한 만큼 | JS 번들에 포함 | purge로 최소화 |
| 러닝 커브 | 낮음 | 중간 | 유틸리티 클래스 암기 필요 |
| SSR 호환 | 우수 | 설정 필요 | 우수 |

### 언제 무엇을 쓰는가

- **CSS Modules**: SSR 프로젝트, 런타임 비용을 없애고 싶을 때, Next.js 기본 지원
- **CSS-in-JS**: 동적 테마, prop 기반 스타일 변화가 많을 때 (단, RSC 환경에서 제약이 있으므로 주의)
- **Tailwind**: 빠른 프로토타이핑, 디자인 시스템 일관성, 팀 규모가 클 때

### 자주 하는 오해

- Tailwind는 인라인 스타일과 같다 → 틀림. Tailwind는 미리 정의된 디자인 토큰 기반의 유틸리티 클래스이고, 미디어 쿼리, hover, focus 등을 지원한다.
- CSS-in-JS는 항상 느리다 → 부분적으로 맞지만, 정적 스타일은 빌드 타임에 추출하는 zero-runtime 라이브러리(vanilla-extract, Panda CSS)도 있다.
- CSS Modules는 동적 스타일이 불가능하다 → 틀림. CSS 변수와 클래스 조건부 적용으로 충분히 동적 스타일을 구현할 수 있다.

### 실무 에피소드 — 이런 상황에서 만난다

> 프로젝트 초기에 styled-components로 시작했는데, Next.js App Router + 서버 컴포넌트로 마이그레이션하면서 문제가 생겼다. 서버 컴포넌트에서는 런타임 CSS-in-JS가 동작하지 않기 때문이다. 클라이언트 컴포넌트에만 styled-components를 쓰고 나머지는 CSS Modules로 전환하면서, 점진적으로 빌드 타임 솔루션으로 옮겼다. 스타일링 도구 선택이 아키텍처 전환 비용에 직접 영향을 준다는 걸 체감한 경험이었다.

### 면접 압축 답변

- **10초**: CSS Modules는 빌드 타임 스코프 격리로 런타임 비용이 없고, CSS-in-JS는 동적 스타일링이 자유롭지만 런타임 비용이 있으며, Tailwind는 유틸리티 클래스 조합으로 빠른 개발이 가능합니다.
- **30초**: CSS Modules는 파일 단위로 클래스명을 해시해 스코프를 격리하고 런타임 비용이 없어 SSR에 적합합니다. CSS-in-JS(styled-components 등)는 prop 기반 동적 스타일이 편하지만 런타임에 스타일을 생성하는 비용이 있습니다. Tailwind는 유틸리티 클래스를 조합하는 방식으로 디자인 일관성을 유지하며, purge를 통해 번들 크기를 최소화합니다. Specificity는 inline > ID > class > element 순이며, `!important`는 최후의 수단입니다. 프로젝트 상황에 따라 선택이 달라지며, 최근에는 SSR과 RSC 호환성을 고려해 빌드 타임 솔루션을 선호하는 추세입니다.

## 11. CS 기초 — 프로세스/스레드, 복잡도, 해시 테이블, 스택/큐

> **핵심 요약** — 프로세스는 독립된 메모리 공간을 가진 실행 단위이고, 스레드는 프로세스 내에서 힙을 공유하는 실행 흐름이다. 알고리즘 효율은 Big-O로 표현하며, 해시 테이블은 평균 O(1) 조회를, 스택/큐는 각각 콜 스택과 태스크 큐의 기반 구조다. 프론트엔드에서도 메인 스레드 블로킹, 자료구조 선택, 이벤트 루프 이해에 직접 연결된다.

### 11-1. 프로세스와 스레드

- **프로세스**: 운영체제에서 실행 중인 프로그램의 인스턴스. 독립된 메모리 공간(코드, 데이터, 힙, 스택)을 가진다.
- **스레드**: 프로세스 내의 실행 흐름 단위. 같은 프로세스의 스레드끼리는 코드, 데이터, 힙 메모리를 공유하고, 각 스레드는 자신만의 스택과 레지스터를 가진다.

```
프로세스 A               프로세스 B
┌──────────────┐       ┌──────────────┐
│ 코드 │ 데이터  │       │ 코드 │ 데이터  │
│ 힙            │       │ 힙            │
│ ┌────┐┌────┐ │       │ ┌────┐       │
│ │스레드││스레드│ │       │ │스레드│       │
│ │스택 ││스택 │ │       │ │스택 │       │
│ └────┘└────┘ │       │ └────┘       │
└──────────────┘       └──────────────┘
  메모리 독립              메모리 독립
  스레드끼리 힙 공유
```

**프론트엔드와의 연결**

- **브라우저 탭**은 기본적으로 별도 프로세스다 (Chrome의 멀티 프로세스 아키텍처).
- **메인 스레드 하나**에서 JavaScript 실행, DOM 조작, 레이아웃, 페인트가 모두 처리된다.
- 메인 스레드가 오래 점유되면 UI가 멈추는 이유가 여기에 있다.
- **Web Worker**는 별도 스레드에서 JS를 실행할 수 있지만, DOM에 접근할 수 없다.

### 11-2. 시간복잡도와 공간복잡도

알고리즘의 효율성을 분석하는 도구다. **Big-O 표기법**으로 입력 크기 n에 대한 증가율을 표현한다.

| 표기 | 이름 | 예시 |
|------|------|------|
| O(1) | 상수 | 배열 인덱스 접근, `Map.get()` |
| O(log n) | 로그 | 이진 탐색 |
| O(n) | 선형 | 배열 순회, `Array.includes()` |
| O(n log n) | 선형 로그 | 정렬 (merge sort, quick sort 평균) |
| O(n^2) | 이차 | 이중 중첩 루프 |

**프론트엔드에서 왜 중요한가**

```js
// O(n^2) — 대량 데이터에서 성능 문제
const hasDuplicate = (arr) => {
  for (let i = 0; i < arr.length; i++) {
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) return true;
    }
  }
  return false;
};

// O(n) — Set을 쓰면 한 번만 순회
const hasDuplicateOptimized = (arr) => {
  const seen = new Set();
  for (const item of arr) {
    if (seen.has(item)) return true; // O(1) 조회
    seen.add(item);
  }
  return false;
};
```

- 리스트 필터링, 검색, 중복 제거 등에서 O(n^2)를 O(n)으로 줄이면 대량 데이터에서 체감 성능이 크게 달라진다.
- `Array.includes()`는 O(n)이고 `Set.has()`는 O(1)이라는 차이를 알면 컬렉션 선택이 달라진다.

### 11-3. 해시 테이블

해시 테이블은 **키를 해시 함수로 변환해 인덱스를 구하고, 해당 위치에 값을 저장**하는 자료구조다. 평균적으로 삽입, 삭제, 조회가 **O(1)**이다.

- JavaScript의 `Object`와 `Map`은 내부적으로 해시 테이블(또는 유사 구조)로 구현되어 있다.
- **해시 충돌**: 서로 다른 키가 같은 인덱스를 가리키면 체이닝 등으로 해결한다.

```js
// JavaScript에서 해시 테이블의 실전 활용
// 두 배열에서 공통 요소 찾기 — O(n) 풀이
function intersection(a, b) {
  const setA = new Set(a);        // O(n) 구축
  return b.filter((x) => setA.has(x)); // O(m) 순회, has는 O(1)
}

// 빈도수 세기
function countFrequency(arr) {
  const freq = new Map();
  for (const item of arr) {
    freq.set(item, (freq.get(item) || 0) + 1);
  }
  return freq;
}
```

### 11-4. 스택과 큐

| 구분 | 스택 (Stack) | 큐 (Queue) |
|------|-------------|------------|
| 원칙 | LIFO (Last In, First Out) | FIFO (First In, First Out) |
| 삽입 | push (맨 위) | enqueue (뒤) |
| 제거 | pop (맨 위) | dequeue (앞) |
| JS 예시 | 콜 스택, `Array.push/pop` | 태스크 큐, `Array.push/shift` (단, `shift`는 O(n)) |

**프론트엔드와의 연결**

```js
// 콜 스택 — 함수 호출이 쌓이고 끝나면 빠지는 LIFO 구조
function a() { b(); }
function b() { c(); }
function c() { console.log("done"); }
a();
// 스택: [a] → [a, b] → [a, b, c] → [a, b] → [a] → []

// 태스크 큐 — 비동기 콜백이 순서대로 대기하는 FIFO 구조
setTimeout(() => console.log("first"), 0);
setTimeout(() => console.log("second"), 0);
// 큐: [first콜백] → [first콜백, second콜백]
// 콜 스택이 비면 순서대로 꺼냄

// 실무 활용 — Undo 기능은 스택
const undoStack = [];
function doAction(action) {
  undoStack.push(action);
  action.execute();
}
function undo() {
  const last = undoStack.pop();
  if (last) last.reverse();
}
```

### 자주 하는 오해

- 프론트엔드 개발자에게 CS 기초는 불필요하다 → 틀림. 메인 스레드 블로킹, 이벤트 루프, 자료구조 선택이 모두 CS 기초 위에 놓여 있다.
- JavaScript는 싱글 스레드이므로 동시성 문제가 없다 → 틀림. 비동기 콜백 순서에 의한 Race Condition은 발생할 수 있다 (챕터 3 참조).
- `Array.shift()`는 O(1)이다 → 틀림. 배열의 모든 요소를 한 칸씩 앞으로 이동해야 하므로 O(n)이다.

### 실무 에피소드 — 이런 상황에서 만난다

> 대시보드에서 1만 개의 로그 항목을 필터링하는 기능을 구현했는데, 필터 조건마다 이중 루프로 중복을 체크하자 입력할 때마다 200ms 이상의 지연이 발생했다. `Set`을 이용한 O(n) 풀이로 바꾸자 지연이 5ms 이하로 줄었다. 기술 면접에서 "프론트엔드에서 시간복잡도가 실제로 중요한 경험이 있냐"는 질문에 이 사례를 연결하면 설득력이 생긴다.

### 면접 압축 답변

- **10초**: 프로세스는 독립 메모리, 스레드는 힙 공유. Big-O는 알고리즘 효율 지표. 해시 테이블은 O(1) 조회. 스택은 LIFO(콜 스택), 큐는 FIFO(태스크 큐)입니다.
- **30초**: 프로세스는 OS에서 독립된 메모리를 할당받는 실행 단위이고, 스레드는 프로세스 안에서 힙을 공유하며 독립 스택으로 실행됩니다. 브라우저의 메인 스레드 하나에서 JS 실행과 DOM 조작이 모두 일어나므로 무거운 연산은 UI를 막습니다. Big-O는 입력 크기에 따른 증가율을 표현하며, `Set.has()`가 O(1)인 것과 `Array.includes()`가 O(n)인 차이가 대량 데이터에서 UX에 직접 영향을 줍니다. 해시 테이블은 Object/Map의 내부 원리이고, 스택과 큐는 각각 콜 스택과 이벤트 루프의 태스크 큐에 대응합니다.

# 챕터 12 핵심 요약 — 200문항의 공백을 채우는 네 개의 축

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **챕터 1~11에서 놓친 퍼즐 조각들은 무엇이고, 그것들은 어떻게 기존 지식의 빈틈을 메우는가?**

이 질문에 답하려면, 각 영역에서 빠진 핵심을 하나씩 되짚어야 한다.

---

### 1단계: 함수가 호출되는 방식이 곧 문맥이다 — this와 프로토타입

챕터 1에서 클로저와 실행 컨텍스트를 배웠지만, 정작 **함수가 "누구의 것"인지 결정하는 규칙**은 다루지 못했다. `this`는 선언 위치가 아니라 호출 방식으로 결정되며, `new` > 명시적 > 암시적 > 기본이라는 우선순위가 있다. 화살표 함수는 자체 `this`가 없어 선언 시점의 외부 스코프를 캡처한다. 그리고 `class` 문법 아래에서 메서드는 `prototype` 객체에 저장되고, 프로퍼티 검색은 `[[Prototype]]` 체인을 따라 올라간다. **`this`의 동적 바인딩과 프로토타입의 위임 모델이 JavaScript 객체 시스템의 두 기둥이다.**

### 2단계: 비교의 함정과 이벤트의 흐름 — == vs ===, 이벤트 전파

`this`와 프로토타입으로 언어의 객체 모델을 잡았으면, 다음은 값을 비교하고 사건을 전달하는 방식이다. `===`는 타입 변환 없이 비교하고, `==`는 암시적 변환을 거쳐 `0 == ""`이 `true`가 되는 함정을 만든다. DOM 이벤트는 캡처링 → 타깃 → 버블링 세 단계로 전파되며, **이 버블링을 이용해 부모에 핸들러 하나만 등록하는 것이 이벤트 위임**이다. React도 루트 컨테이너에 이벤트를 위임하는 방식으로 동작한다.

### 3단계: 올바른 자료구조와 모든 사용자를 위한 설계 — Map/Set, 접근성

`Object`는 키가 문자열로 제한되지만 `Map`은 모든 타입을 키로 쓸 수 있고, `Array.includes()`는 O(n)이지만 `Set.has()`는 O(1)이다. **대량 데이터에서 자료구조 선택이 UX에 직접 영향을 준다.** 그리고 접근성은 시맨틱 HTML을 기본으로, ARIA를 보완적으로, 모달에서는 포커스 트랩으로 키보드 접근성을 확보하는 것이 핵심이다. "나중에 할 일"이 아니라 컴포넌트 설계의 기본 요구사항이다.

---

여기까지가 JavaScript의 언어 모델과 DOM의 빈틈이었다. 다음은 TypeScript의 타입 설계 정밀도로 축이 이동한다.

---

### 4단계: 타입의 정밀도가 곧 설계의 정밀도다 — TS 실전 디테일

`age?`는 키 자체가 없어도 되지만 `age: number | undefined`는 키가 반드시 있어야 한다. 함수 오버로드는 입력 타입별로 반환 타입을 정확히 좁힐 때 쓰고, 상수 집합은 `enum`보다 union literal이나 `as const` 객체가 번들 친화적이다. **"타입을 얼마나 정밀하게 설계하는가"가 PATCH 요청의 삭제 의도 표현부터 tree-shaking까지 실무 전반에 영향을 준다.**

### 5단계: 실행 시점의 차이와 로직 재사용의 진화 — useLayoutEffect, 디자인 패턴

`useEffect`는 브라우저가 paint한 후 비동기로 실행되고, `useLayoutEffect`는 paint 전 동기로 실행되어 DOM 측정 후 깜빡임 없는 반영에 쓴다. SSR 환경에서는 `useIsomorphicLayoutEffect` 패턴으로 경고를 피한다. 로직 재사용 패턴은 HOC(감싸기) → Render Props(위임) → Compound Components(암시적 공유)로 진화했고, **커스텀 훅이 대부분을 대체했지만, 선언형 API를 만드는 Compound Components는 여전히 유효하다.**

### 6단계: 스타일링 도구 선택이 아키텍처 전환 비용을 좌우한다 — CSS 아키텍처

CSS Modules는 빌드 타임 스코프 격리로 런타임 비용이 없고, CSS-in-JS는 동적 스타일링이 자유롭지만 런타임 비용이 있으며, Tailwind는 유틸리티 클래스 조합으로 빠른 개발이 가능하다. **SSR과 RSC 호환성을 고려하면 빌드 타임 솔루션이 유리한 추세이며, 스타일링 도구 선택이 아키텍처 마이그레이션 비용에 직접 영향을 준다.**

### 7단계: CS 기초는 프론트엔드의 토양이다 — 프로세스/스레드, 복잡도, 자료구조

프로세스는 독립된 메모리 공간을 가진 실행 단위이고, 스레드는 힙을 공유하는 실행 흐름이다. 브라우저 메인 스레드 하나에서 JS 실행과 DOM 조작이 모두 일어나므로, 무거운 연산은 UI를 멈춘다. Big-O는 알고리즘 효율의 지표이고, **해시 테이블은 Object/Map의 내부 원리이며, 스택은 콜 스택, 큐는 태스크 큐에 직접 대응한다.** 프론트엔드 면접에서 "CS 기초가 왜 필요하냐"는 질문의 답이 여기에 있다.

---

### 한 줄로 이어 읽기

**`this`와 프로토타입**으로 JavaScript 객체 모델의 빈틈을 채우고 → **비교 연산과 이벤트 전파**로 값 판단과 DOM 사건 흐름을 잡고 → **Map/Set과 접근성**으로 자료구조 선택과 모든 사용자를 위한 설계를 익히고 → **옵셔널/오버로드/const 객체**로 TypeScript 타입 설계의 정밀도를 높이고 → **useLayoutEffect와 디자인 패턴**으로 실행 시점과 로직 재사용의 현대적 흐름을 정리하고 → **CSS 아키텍처**로 스타일링 전략이 아키텍처에 미치는 영향을 이해하고 → **프로세스/스레드, 복잡도, 해시 테이블, 스택/큐**로 프론트엔드 아래 깔린 CS 토양을 확인한다.

---

### 1단계: 함수가 호출되는 방식이 곧 문맥이다 — this와 프로토타입

챕터 1에서 클로저와 실행 컨텍스트를 배웠지만, 정작 **함수가 "누구의 것"인지 결정하는 규칙**은 다루지 못했다. `this`는 선언 위치가 아니라 호출 방식으로 결정되며, `new` > 명시적 > 암시적 > 기본이라는 우선순위가 있다. 화살표 함수는 자체 `this`가 없어 선언 시점의 외부 스코프를 캡처한다. 그리고 `class` 문법 아래에서 메서드는 `prototype` 객체에 저장되고, 프로퍼티 검색은 `[[Prototype]]` 체인을 따라 올라간다. **`this`의 동적 바인딩과 프로토타입의 위임 모델이 JavaScript 객체 시스템의 두 기둥이다.**

### 2단계: 비교의 함정과 이벤트의 흐름 — == vs ===, 이벤트 전파

`this`와 프로토타입으로 언어의 객체 모델을 잡았으면, 다음은 값을 비교하고 사건을 전달하는 방식이다. `===`는 타입 변환 없이 비교하고, `==`는 암시적 변환을 거쳐 `0 == ""`이 `true`가 되는 함정을 만든다. DOM 이벤트는 캡처링 → 타깃 → 버블링 세 단계로 전파되며, **이 버블링을 이용해 부모에 핸들러 하나만 등록하는 것이 이벤트 위임**이다. React도 루트 컨테이너에 이벤트를 위임하는 방식으로 동작한다.

### 3단계: 올바른 자료구조와 모든 사용자를 위한 설계 — Map/Set, 접근성

`Object`는 키가 문자열로 제한되지만 `Map`은 모든 타입을 키로 쓸 수 있고, `Array.includes()`는 O(n)이지만 `Set.has()`는 O(1)이다. **대량 데이터에서 자료구조 선택이 UX에 직접 영향을 준다.** 그리고 접근성은 시맨틱 HTML을 기본으로, ARIA를 보완적으로, 모달에서는 포커스 트랩으로 키보드 접근성을 확보하는 것이 핵심이다. "나중에 할 일"이 아니라 컴포넌트 설계의 기본 요구사항이다.

---

여기까지가 JavaScript의 언어 모델과 DOM의 빈틈이었다. 다음은 TypeScript의 타입 설계 정밀도로 축이 이동한다.

---

### 4단계: 타입의 정밀도가 곧 설계의 정밀도다 — TS 실전 디테일

`age?`는 키 자체가 없어도 되지만 `age: number | undefined`는 키가 반드시 있어야 한다. 함수 오버로드는 입력 타입별로 반환 타입을 정확히 좁힐 때 쓰고, 상수 집합은 `enum`보다 union literal이나 `as const` 객체가 번들 친화적이다. **"타입을 얼마나 정밀하게 설계하는가"가 PATCH 요청의 삭제 의도 표현부터 tree-shaking까지 실무 전반에 영향을 준다.**

### 5단계: 실행 시점의 차이와 로직 재사용의 진화 — useLayoutEffect, 디자인 패턴

`useEffect`는 브라우저가 paint한 후 비동기로 실행되고, `useLayoutEffect`는 paint 전 동기로 실행되어 DOM 측정 후 깜빡임 없는 반영에 쓴다. SSR 환경에서는 `useIsomorphicLayoutEffect` 패턴으로 경고를 피한다. 로직 재사용 패턴은 HOC(감싸기) → Render Props(위임) → Compound Components(암시적 공유)로 진화했고, **커스텀 훅이 대부분을 대체했지만, 선언형 API를 만드는 Compound Components는 여전히 유효하다.**

### 6단계: 스타일링 도구 선택이 아키텍처 전환 비용을 좌우한다 — CSS 아키텍처

CSS Modules는 빌드 타임 스코프 격리로 런타임 비용이 없고, CSS-in-JS는 동적 스타일링이 자유롭지만 런타임 비용이 있으며, Tailwind는 유틸리티 클래스 조합으로 빠른 개발이 가능하다. **SSR과 RSC 호환성을 고려하면 빌드 타임 솔루션이 유리한 추세이며, 스타일링 도구 선택이 아키텍처 마이그레이션 비용에 직접 영향을 준다.**

### 7단계: CS 기초는 프론트엔드의 토양이다 — 프로세스/스레드, 복잡도, 자료구조

프로세스는 독립된 메모리 공간을 가진 실행 단위이고, 스레드는 힙을 공유하는 실행 흐름이다. 브라우저 메인 스레드 하나에서 JS 실행과 DOM 조작이 모두 일어나므로, 무거운 연산은 UI를 멈춘다. Big-O는 알고리즘 효율의 지표이고, **해시 테이블은 Object/Map의 내부 원리이며, 스택은 콜 스택, 큐는 태스크 큐에 직접 대응한다.** 프론트엔드 면접에서 "CS 기초가 왜 필요하냐"는 질문의 답이 여기에 있다.

### 한 줄로 이어 읽기

**`this`와 프로토타입**으로 JavaScript 객체 모델의 빈틈을 채우고 → **비교 연산과 이벤트 전파**로 값 판단과 DOM 사건 흐름을 잡고 → **Map/Set과 접근성**으로 자료구조 선택과 모든 사용자를 위한 설계를 익히고 → **옵셔널/오버로드/const 객체**로 TypeScript 타입 설계의 정밀도를 높이고 → **useLayoutEffect와 디자인 패턴**으로 실행 시점과 로직 재사용의 현대적 흐름을 정리하고 → **CSS 아키텍처**로 스타일링 전략이 아키텍처에 미치는 영향을 이해하고 → **프로세스/스레드, 복잡도, 해시 테이블, 스택/큐**로 프론트엔드 아래 깔린 CS 토양을 확인한다.
