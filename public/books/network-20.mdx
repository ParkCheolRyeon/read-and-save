# 06. 네트워크 20문항

> HTTP, TLS, CORS, 캐시, 실패 대응 UX까지 네트워크 관점을 묻는 20문항입니다.

## 총평

이 섹션은 프론트엔드에서 진짜 중요하다. 왜냐하면 사용자는 결국 네트워크를 타고 들어오는 서비스를 쓰기 때문이다. 좋은 답변은 아래 3층이 같이 있어야 한다.

- HTTP/TLS/CORS 같은 **프로토콜 이해**
- 브라우저가 실제로 어떻게 행동하는지에 대한 **클라이언트 이해**
- 실패했을 때 어떻게 UX를 설계할지에 대한 **서비스 감각**

### 전체 흐름 맵

```
HTTP vs HTTPS (보안 기본)
  → TLS 핸드셰이크 (신뢰 수립)
    → HTTP 메서드 (요청 의도)
      → 멱등성 (재시도 안전성)
        → 상태 코드 (응답 의미)
          → 쿠키/세션/토큰 (인증 구조)
            → Access/Refresh Token (토큰 전략)
              → CORS (브라우저 보안 정책)
                → Preflight (사전 협상)
                  → 캐시 헤더 (재사용 전략)
                    → DNS & 요청 흐름 (전체 경로)
                      → CDN (전달 최적화)
                        → WebSocket vs Polling (실시간)
                          → 네트워크 불안정 대응 (UX)
                            → 온라인/오프라인 한계 (감지)
                              → timeout/retry/backoff (복구)
                                → 중복 요청 방지 (안전)
                                  → 파일 업로드 UX (실무)
                                    → 장애 원인 분리 (진단)
                                      → 사용자 관점 (서비스 철학)
```

## 1. HTTP와 HTTPS의 차이를 설명해보세요.

### 합격 답변 예시

HTTP는 데이터를 평문으로 주고받는 프로토콜이고, HTTPS는 HTTP 위에 TLS를 얹어 암호화와 무결성 보호, 서버 인증을 제공하는 방식입니다. 즉 HTTPS는 단순히 "보안 버전 HTTP"가 아니라, 중간에서 내용을 읽거나 변조하거나 서버를 사칭하는 위험을 줄이기 위한 구조입니다.

### 가산점 포인트

HTTPS의 세 축을 정확히 말하면 좋다: **암호화**(기밀성), **무결성**(변조 방지), **서버 인증**(신원 확인). "암호화된 HTTP"라고만 말하면 절반짜리다. 또한 브라우저 기능 일부(Service Worker, Secure Cookie, 최신 API)는 HTTPS 환경에서만 동작한다는 점도 알면 좋다.

### 상세 해설

#### 왜 암호화만으로는 부족한가

```
HTTPS의 세 축:
  1. 암호화 (기밀성) — 중간에서 내용을 읽지 못하게
  2. 무결성 — 데이터가 중간에서 변조되지 않았음을 보장 (HMAC)
  3. 서버 인증 — 통신 상대가 진짜 그 서버인지 인증서로 확인

세 축이 모두 있어야 "중간자 공격(MITM)"을 막을 수 있다.
```

```ts
// 브라우저에서 HTTPS 강제 — HSTS 헤더
// 서버가 이 헤더를 보내면 브라우저는 이후 HTTP 요청을 자동으로 HTTPS로 전환
Strict-Transport-Security: max-age=31536000; includeSubDomains

// 프론트엔드에서 HTTPS 여부 확인
if (window.location.protocol !== 'https:') {
  // Service Worker 등록 불가, Secure Cookie 불가
  console.warn('HTTPS가 아닙니다');
}
```

#### 자주 하는 오해

- "HTTP/2는 자동으로 암호화된다" → **스펙상 아님.** 하지만 브라우저들이 TLS 없는 HTTP/2를 거부하므로 사실상 HTTPS 필수.
- "HTTPS는 느리다" → **과거 이야기.** TLS 1.3은 1-RTT 핸드셰이크로 오버헤드가 크게 줄었다.
- "내부 서비스는 HTTP여도 된다" → **위험.** 내부 네트워크도 완전히 안전하지 않으며, 보안 감사에서 지적 대상이 된다.

#### 실무 에피소드

> mixed content 경고로 이미지가 안 뜨는 이슈가 있었는데, CDN 일부 URL이 `http://`로 하드코딩되어 있었다. HTTPS 전환은 서버만 바꾸면 끝이 아니라, **모든 리소스 URL도 확인**해야 한다.

> HTTPS의 보안 구조를 알았으니, 다음에는 **그 보안을 실제로 어떻게 수립하는지** — TLS 핸드셰이크를 봐야 한다.

## 2. TLS 핸드셰이크가 대략 어떻게 이뤄지는지 설명해보세요.

### 합격 답변 예시

클라이언트가 지원 가능한 암호 스위트와 랜덤 값을 서버에 보내고, 서버는 자신의 인증서와 선택한 암호 방식, 랜덤 값을 응답합니다. 클라이언트는 서버 인증서를 검증한 뒤, 키 교환 과정을 통해 세션 키를 합의하고, 이후 실제 데이터 통신은 주로 대칭키 기반으로 빠르게 암호화해 진행합니다.

### 가산점 포인트

**왜 비대칭 + 대칭을 같이 쓰는지** 설명하면 좋다. 비대칭 암호는 신뢰 수립과 키 교환에 강하지만 느리고, 대칭 암호는 빠르지만 키를 안전하게 공유하는 방법이 필요하다. 그래서 처음에 비대칭으로 세션 키를 합의하고, 이후에는 대칭 암호로 빠르게 통신한다.

### 상세 해설

```
TLS 핸드셰이크 흐름:

Client                          Server
  │                                │
  │──── Client Hello ────────────→│  지원 가능한 암호 스위트, 랜덤 값
  │                                │
  │←── Server Hello + 인증서 ─────│  선택한 암호, 인증서, 랜덤 값
  │                                │
  │  (인증서 검증: CA 체인 확인)      │
  │                                │
  │──── 키 교환 ─────────────────→│  세션 키 합의
  │                                │
  │←→ 이후 대칭키로 암호화 통신 ←→│  빠른 데이터 전송
```

#### TLS 1.2 vs TLS 1.3

| | TLS 1.2 | TLS 1.3 |
|---|---|---|
| **핸드셰이크** | 2 RTT | 1 RTT (0-RTT 재연결 가능) |
| **암호 스위트** | 많음 (일부 취약) | 안전한 것만 허용 |
| **성능** | 보통 | 더 빠름 |

#### 자주 하는 오해

- "TLS 핸드셰이크는 매 요청마다 일어난다" → **아님.** 한 번 세션이 수립되면 Keep-Alive로 재사용하고, TLS 1.3에서는 0-RTT 재연결도 가능하다.

> TLS로 보안 통신이 수립되었으니, 다음에는 **그 위에서 실제로 어떤 요청을 보내는지** — HTTP 메서드를 봐야 한다.

## 3. HTTP 메서드 GET, POST, PUT, PATCH, DELETE의 차이를 설명해보세요.

| 메서드 | 주 용도 | 멱등성 | 비고 |
| --- | --- | --- | --- |
| `GET` | 조회 | 멱등 | 캐시 친화적 |
| `POST` | 생성 / 처리 | 비멱등 | 서버가 새 작업을 만듦 |
| `PUT` | 전체 교체 | 멱등 | 리소스 전체를 덮어씀 |
| `PATCH` | 부분 수정 | 설계 의존 | 일부 필드만 변경 |
| `DELETE` | 삭제 | 멱등 | 이미 없어도 결과는 같음 |

### 합격 답변 예시

GET은 조회, POST는 생성이나 서버 측 처리 요청, PUT은 리소스 전체 교체, PATCH는 부분 수정, DELETE는 삭제를 의미합니다. 다만 실무에서는 단순 암기보다 **"이 메서드가 어떤 의도를 표현하느냐"**가 중요합니다.

### 가산점 포인트

**PUT vs PATCH의 차이**를 정확히 구분하면 좋다. PUT은 "이 리소스의 전체 상태를 이것으로 교체하라", PATCH는 "이 리소스의 일부 필드만 바꿔라"이다. PUT은 보내지 않은 필드를 null로 만들 수 있고, PATCH는 보낸 필드만 변경한다.

### 상세 해설

```ts
// PUT: 전체 교체 — 보내지 않은 필드가 null이 될 수 있음
PUT /api/users/1
{ "name": "홍길동", "email": "hong@test.com" }
// → bio 필드가 있었다면 null로 초기화될 수 있음

// PATCH: 부분 수정 — 보낸 필드만 변경
PATCH /api/users/1
{ "name": "홍길동" }
// → email, bio 등 다른 필드는 그대로 유지

// GET: 캐시 가능, URL에 파라미터
GET /api/users?page=1&limit=20
// → 브라우저가 캐시 가능, 북마크 가능

// POST: 본문에 데이터, 캐시 안 됨
POST /api/users
{ "name": "새 사용자" }
// → 매번 서버가 새 작업을 수행
```

#### 자주 하는 오해

- "GET에는 body를 못 보낸다" → **스펙상 가능하지만 실무에서 피해야.** 많은 서버/프록시가 GET body를 무시하거나 제거한다.
- "DELETE는 항상 데이터를 지운다" → **의미적(semantic)으로 삭제.** soft delete로 플래그만 바꿀 수도 있다.

> HTTP 메서드를 알았으니, 다음에는 **재시도 안전성과 직결되는 개념** — 멱등성을 봐야 한다.

## 4. 멱등성이 무엇인지 설명하고, 왜 중요한지 말해보세요.

### 합격 답변 예시

같은 요청을 여러 번 보내도 결과 상태가 같다면 멱등적이라고 합니다. 이 개념이 중요한 이유는 네트워크 재시도, 중복 클릭, 장애 복구 상황에서 요청이 여러 번 발생해도 시스템이 예상 밖으로 망가지지 않게 만들 수 있기 때문입니다.

### 가산점 포인트

**"멱등성은 서버만의 문제가 아니라 프론트엔드도 함께 설계해야"**라고 말하면 좋다. 프론트에서 버튼 disable + debounce를 하고, 서버에서 Idempotency-Key를 적용하면 이중 방어가 된다.

### 상세 해설

#### 왜 프론트엔드에게 중요한가

```
사용자가 "결제" 버튼을 빠르게 2번 누름:
  ❌ 멱등성 없음: 결제가 2번 발생 → 2배 청구
  ✅ 멱등성 있음: 같은 결제 요청이므로 1번만 처리

네트워크 타임아웃 후 자동 재시도:
  ❌ 멱등성 없음: POST /orders가 2번 실행 → 주문 2개 생성
  ✅ 멱등성 있음: 같은 Idempotency-Key이므로 1건만 생성
```

```ts
// 프론트엔드에서 중복 요청 방지
const [isSubmitting, setIsSubmitting] = useState(false);

async function handlePayment() {
  if (isSubmitting) return;  // 중복 클릭 방지
  setIsSubmitting(true);

  try {
    await fetch('/api/payment', {
      method: 'POST',
      headers: {
        'Idempotency-Key': crypto.randomUUID(),  // 서버측 멱등성 보장
      },
      body: JSON.stringify(paymentData),
    });
  } finally {
    setIsSubmitting(false);
  }
}
```

#### 자주 하는 오해

- "GET은 멱등적이니 아무 부작용도 없다" → **조회 자체에 부작용이 있을 수 있음.** 예: 조회수 카운터 증가, 로그 기록. 하지만 "리소스 상태" 관점에서는 멱등적이다.

> 멱등성을 알았으니, 다음에는 **서버 응답의 의미** — HTTP 상태 코드를 봐야 한다.

## 5. HTTP 상태코드 200, 201, 204, 400, 401, 403, 404, 409, 500의 의미를 설명해보세요.

| 코드 | 의미 | 프론트엔드 처리 |
| --- | --- | --- |
| `200` | 성공 | 데이터 표시 |
| `201` | 리소스 생성 성공 | 생성 완료 UI |
| `204` | 성공, 본문 없음 | UI 업데이트만 |
| `400` | 잘못된 요청 | 입력 오류 메시지 |
| `401` | 인증 필요/실패 | 로그인 페이지로 이동 |
| `403` | 권한 없음 | 접근 불가 메시지 |
| `404` | 리소스 없음 | 404 페이지 |
| `409` | 충돌 | 충돌 해결 UI |
| `500` | 서버 오류 | 일반 에러 + 재시도 |

### 합격 답변 예시

200은 일반적인 성공, 201은 리소스 생성 성공, 204는 성공했지만 응답 본문이 없는 경우입니다. 400은 잘못된 요청, 401은 인증 필요/실패, 403은 인증은 됐지만 권한 없음, 404는 리소스 없음, 409는 충돌, 500은 서버 내부 오류입니다.

### 가산점 포인트

**401 vs 403 구분**이 핵심이다. 401은 "누구인지 모르겠다" (로그인 안 됨, 토큰 만료), 403은 "누구인지는 아는데 권한이 없다" (일반 사용자가 관리자 페이지 접근). 프론트엔드에서 401이면 로그인 리다이렉트, 403이면 접근 불가 메시지를 보여줘야 한다.

### 상세 해설

```ts
// 프론트엔드에서 상태 코드별 처리
async function fetchWithErrorHandling(url: string) {
  const res = await fetch(url);

  switch (res.status) {
    case 200:
    case 201:
      return res.json();
    case 204:
      return null;  // 본문 없음
    case 400:
      const error = await res.json();
      throw new ValidationError(error.message);  // 입력 오류 표시
    case 401:
      redirectToLogin();  // 토큰 만료 → 로그인 유도
      throw new AuthError('인증이 필요합니다');
    case 403:
      throw new ForbiddenError('접근 권한이 없습니다');
    case 404:
      throw new NotFoundError('리소스를 찾을 수 없습니다');
    case 409:
      throw new ConflictError('이미 존재하는 리소스입니다');
    case 500:
      throw new ServerError('서버 오류가 발생했습니다');
    default:
      throw new Error(`Unexpected status: ${res.status}`);
  }
}
```

#### 자주 하는 오해

- "에러는 다 400이나 500으로 보내면 된다" → **상태 코드를 세분화해야 프론트에서 적절한 UX를 제공 가능.** 401이면 로그인 유도, 409면 충돌 해결 UI 등.

> 상태 코드를 알았으니, 다음에는 **인증 구조의 기본** — 쿠키, 세션, 토큰을 봐야 한다.

## 6. 쿠키, 세션, 토큰의 차이를 설명해보세요.

### 합격 답변 예시

쿠키는 브라우저에 저장되는 작은 데이터 조각이고, 세션은 서버가 사용자 상태를 관리하는 방식이며, 토큰은 클라이언트가 들고 다니며 인증 정보를 증명하는 값입니다. 즉 쿠키는 **저장/전달 수단**이고, 세션과 토큰은 **인증/상태 관리 전략**입니다.

### 가산점 포인트

**"쿠키 = 세션"이 아니라는 점을 분리**해야 한다. 세션 ID를 쿠키에 저장할 수도 있고, JWT 토큰도 쿠키에 저장할 수 있다. 쿠키는 운반 수단이지 인증 방식 자체가 아니다.

### 상세 해설

```
쿠키 (저장/전달 수단):
  ┌─────────────────┐
  │ 브라우저 쿠키 저장소 │
  │ session_id=abc   │  ← 세션 ID를 쿠키에 저장
  │ token=eyJhbG...  │  ← JWT를 쿠키에 저장할 수도 있음
  └─────────────────┘
  → 매 요청 시 자동 전송 (Same-Origin)

세션 (서버 중심):
  클라이언트: 쿠키에 세션 ID 보관
  서버: 세션 ID → 사용자 정보 매핑 (메모리/DB)
  → 서버가 상태를 관리 (stateful)

토큰 (클라이언트 중심):
  클라이언트: 토큰 자체에 인증 정보 포함 (JWT)
  서버: 토큰을 검증만 하면 됨 (서명 확인)
  → 서버가 상태를 안 가져도 됨 (stateless)
```

#### 자주 하는 오해

- "JWT를 localStorage에 저장하면 안전하다" → **XSS에 취약.** HttpOnly 쿠키에 저장하는 것이 더 안전하다.
- "세션은 오래된 방식이다" → **아님.** 세션 기반과 토큰 기반은 각각 장단이 있고, 요구사항에 따라 선택한다.

> 쿠키/세션/토큰을 알았으니, 다음에는 **토큰 기반 인증의 구체적인 전략** — Access/Refresh Token을 봐야 한다.

## 7. Access Token과 Refresh Token 구조를 설명해보세요.

### 합격 답변 예시

Access Token은 실제 API 요청 인증에 사용하는 짧은 수명의 토큰이고, Refresh Token은 Access Token이 만료됐을 때 새 Access Token을 재발급받기 위한 더 긴 수명의 토큰입니다. 이렇게 나누는 이유는 **탈취 위험을 줄이면서도 사용자가 자주 다시 로그인하지 않도록** 하기 위해서입니다.

### 가산점 포인트

프론트엔드에서의 **silent refresh** 패턴을 알면 좋다. Access Token 만료 시 자동으로 Refresh Token으로 새 Access Token을 발급받아 사용자가 만료를 느끼지 못하게 하는 것. 실패하면 로그인 페이지로 리다이렉트.

### 상세 해설

```ts
// Silent Refresh 패턴
async function fetchWithAuth(url: string) {
  let res = await fetch(url, {
    headers: { Authorization: `Bearer ${accessToken}` },
  });

  if (res.status === 401) {
    // Access Token 만료 → Refresh Token으로 재발급 시도
    const refreshRes = await fetch('/api/auth/refresh', {
      method: 'POST',
      credentials: 'include',  // Refresh Token은 HttpOnly 쿠키
    });

    if (refreshRes.ok) {
      const { accessToken: newToken } = await refreshRes.json();
      accessToken = newToken;
      // 원래 요청 재시도
      res = await fetch(url, {
        headers: { Authorization: `Bearer ${newToken}` },
      });
    } else {
      // Refresh도 실패 → 로그인 필요
      redirectToLogin();
    }
  }

  return res;
}
```

```
Access Token 만료 흐름:
  요청 → 401 → Refresh 요청 → 새 Access Token → 원래 요청 재시도
                   ↓ 실패 시
              로그인 페이지 리다이렉트
```

#### 자주 하는 오해

- "Refresh Token이 있으면 Access Token을 길게 해도 된다" → **목적이 다름.** Access Token은 짧게 유지해 탈취 시 피해를 최소화하는 것이 설계 의도.
- "무한 refresh를 허용해도 된다" → **위험.** Refresh Token도 만료 기한과 회전(rotation) 정책이 필요.

> 토큰 전략을 알았으니, 다음에는 **브라우저의 보안 정책** — CORS를 봐야 한다.

## 8. CORS가 왜 발생하는지 설명해보세요.

### 합격 답변 예시

CORS는 브라우저의 same-origin policy 때문에 발생합니다. 브라우저는 기본적으로 다른 출처의 리소스 접근을 제한하고, 서버가 명시적으로 허용할 때만 cross-origin 요청 결과를 읽을 수 있게 합니다. 즉 **CORS는 서버가 갑자기 까다로운 게 아니라, 브라우저 보안 정책을 서버가 헤더로 조정하는 구조**입니다.

### 가산점 포인트

**CORS는 브라우저 전용 이슈**라는 점이 핵심이다. 서버-서버 통신(Node.js에서 fetch 등)에서는 CORS가 없다. same-origin policy는 **브라우저의 보안 모델**이기 때문이다.

### 상세 해설

```
Same-Origin Policy:
  프론트: https://myapp.com
  API:   https://api.myapp.com  ← 도메인이 다르므로 다른 출처

  브라우저: "다른 출처 요청이네, 서버가 허용하는지 확인할게"
  서버 응답 헤더: Access-Control-Allow-Origin: https://myapp.com
  브라우저: "서버가 허용했으니 결과를 프론트에 전달"

  서버가 헤더를 안 보내면?
  브라우저: "서버가 허용 안 했으니 결과를 차단" → CORS 에러
```

```ts
// Next.js API Route에서 CORS 헤더 설정
export async function GET(request: Request) {
  return new Response(JSON.stringify({ data: 'hello' }), {
    headers: {
      'Access-Control-Allow-Origin': 'https://myapp.com',
      'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE',
      'Access-Control-Allow-Headers': 'Content-Type, Authorization',
    },
  });
}
```

#### 자주 하는 오해

- "CORS는 백엔드가 막는 것이다" → **아님.** 서버는 정상 응답을 보냈지만, **브라우저가** 허용 헤더가 없으면 결과를 차단한다.
- "CORS 에러가 나면 서버에 문제가 있다" → **서버 응답 자체는 정상.** CORS 헤더만 추가하면 해결.
- "프록시를 쓰면 CORS가 해결된다" → **브라우저 → 프록시(같은 출처) → API.** 브라우저 입장에서 같은 출처로 요청하므로 CORS가 안 걸린다.

> CORS를 알았으니, 다음에는 **CORS의 구체적인 메커니즘** — preflight 요청을 봐야 한다.

## 9. 브라우저가 preflight 요청을 보내는 경우를 설명해보세요.

### 합격 답변 예시

브라우저는 **단순 요청(simple request)이 아닌 경우** 먼저 OPTIONS 요청으로 서버에 허용 여부를 묻습니다. 이 OPTIONS 요청이 preflight입니다. 예를 들어 메서드가 GET/POST/HEAD 외이거나, 커스텀 헤더를 쓰거나, Content-Type이 `application/json`인 경우 preflight가 발생합니다.

### 가산점 포인트

네트워크 탭에서 **요청이 두 번 가는 것처럼 보이는 이유**가 preflight다. OPTIONS + 실제 요청이 쌍으로 나간다. `Access-Control-Max-Age` 헤더로 preflight 결과를 캐시하면 불필요한 OPTIONS 요청을 줄일 수 있다.

### 상세 해설

```
Simple Request (preflight 없음):
  - GET, POST, HEAD 중 하나
  - Content-Type이 text/plain, multipart/form-data, application/x-www-form-urlencoded 중 하나
  - 커스텀 헤더 없음

Non-Simple Request (preflight 발생):
  - PUT, PATCH, DELETE 메서드
  - Authorization 헤더
  - Content-Type: application/json  ← 가장 흔한 원인
  - 커스텀 헤더 (X-Request-Id 등)
```

```
Preflight 흐름:
  1. 브라우저 → OPTIONS /api/users (허용 여부 확인)
     Origin: https://myapp.com
     Access-Control-Request-Method: PUT
     Access-Control-Request-Headers: Content-Type, Authorization

  2. 서버 → 200 OK
     Access-Control-Allow-Origin: https://myapp.com
     Access-Control-Allow-Methods: GET, POST, PUT
     Access-Control-Allow-Headers: Content-Type, Authorization
     Access-Control-Max-Age: 86400  ← 24시간 동안 preflight 캐시

  3. 브라우저 → PUT /api/users (실제 요청)
```

#### 실무 에피소드

> API 요청이 두 번 가는 것 같다는 버그 리포트를 받았는데, DevTools 네트워크 탭을 보니 OPTIONS + PUT이 쌍으로 나가고 있었다. preflight를 모르면 "요청이 중복된다"고 오해할 수 있다.

> preflight를 알았으니, 다음에는 **네트워크 요청의 재사용** — 캐시 헤더를 봐야 한다.

## 10. 캐시 헤더(Cache-Control, ETag, Last-Modified)의 역할을 설명해보세요.

### 합격 답변 예시

Cache-Control은 캐시 정책 자체를 정하는 헤더이고, ETag는 리소스 버전을 식별해 조건부 요청에 활용하며, Last-Modified는 마지막 수정 시점을 기준으로 재검증할 때 사용됩니다. 즉 캐시는 "다시 안 받는 것"이 아니라, **안 바뀌었으면 재사용하고 바뀌었는지 효율적으로 확인하는 전략**입니다.

### 가산점 포인트

`Cache-Control: no-cache`와 `no-store`의 차이를 알면 좋다. `no-cache`는 "캐시해도 되지만 사용 전에 서버에 확인하라", `no-store`는 "아예 캐시하지 마라"다.

### 상세 해설

```
Cache-Control 동작 흐름:

max-age=3600 (1시간):
  0~60분: 캐시 히트 → 서버 요청 없이 즉시 반환
  60분 후: 캐시 만료 → 서버에 재검증 요청

재검증 (ETag 방식):
  클라이언트: GET /style.css
              If-None-Match: "abc123"  ← 이전에 받은 ETag
  서버:
    ETag가 같으면 → 304 Not Modified (본문 없이 "그대로 써")
    ETag가 다르면 → 200 OK + 새 리소스

재검증 (Last-Modified 방식):
  클라이언트: GET /style.css
              If-Modified-Since: Wed, 01 Jan 2025 00:00:00 GMT
  서버:
    수정 안 됐으면 → 304 Not Modified
    수정 됐으면 → 200 OK + 새 리소스
```

```ts
// Next.js에서 정적 자산 캐시 전략
// _next/static/ 파일: 해시가 파일명에 포함 → 변경 시 URL 변경
// → 오래 캐시해도 안전 (immutable)
Cache-Control: public, max-age=31536000, immutable

// API 응답: 자주 바뀔 수 있으므로 짧게 캐시
Cache-Control: max-age=60, stale-while-revalidate=300
```

#### 자주 하는 오해

- "`no-cache` = 캐시 안 함" → **아님!** `no-cache`는 "캐시하되 사용 전 서버 확인". 캐시를 아예 안 하려면 `no-store`.
- "ETag와 Last-Modified 중 하나만 쓰면 된다" → **ETag가 더 정확.** Last-Modified는 초 단위라 1초 내 변경을 감지 못할 수 있다.

> 캐시 전략을 알았으니, 다음에는 **요청이 서버에 도달하는 전체 경로** — DNS부터 응답까지를 봐야 한다.

## 11. DNS가 무엇이고, 사용자가 URL을 입력했을 때 어떤 흐름으로 서버에 도달하는지 설명해보세요.

### 합격 답변 예시

DNS는 사람이 읽는 도메인 이름을 IP 주소로 변환하는 시스템입니다. 사용자가 URL을 입력하면 브라우저와 OS의 DNS 캐시를 먼저 확인하고, 없으면 DNS 서버에 조회해 IP를 얻습니다. 그 다음 TCP 연결 → TLS 핸드셰이크(HTTPS) → HTTP 요청 → 응답 수신 순서로 진행됩니다.

### 가산점 포인트

`dns-prefetch`와 `preconnect`를 활용하면 DNS 조회와 TCP/TLS 연결을 미리 수행해 LCP를 개선할 수 있다. 특히 외부 도메인(CDN, API 서버, 폰트 서버)에 유용하다.

### 상세 해설

```
URL 입력부터 화면 표시까지:

1. DNS 조회: myapp.com → 203.0.113.1
   (캐시: 브라우저 → OS → 라우터 → ISP DNS → 루트 DNS)

2. TCP 3-Way Handshake (1 RTT)
   SYN → SYN-ACK → ACK

3. TLS Handshake (1-2 RTT, HTTPS)
   ClientHello → ServerHello → 키 교환 → 완료

4. HTTP 요청 전송
   GET / HTTP/1.1
   Host: myapp.com

5. 서버 처리 + 응답

6. 브라우저 렌더링
   HTML 파싱 → DOM → CSSOM → Render Tree → Layout → Paint
```

```html
<!-- DNS 조회 + 연결을 미리 수행 -->
<link rel="dns-prefetch" href="https://cdn.example.com" />
<link rel="preconnect" href="https://fonts.googleapis.com" crossorigin />
```

> DNS 흐름을 알았으니, 다음에는 **전달 경로를 단축하는 방법** — CDN을 봐야 한다.

## 12. CDN이 왜 필요한지 설명해보세요.

### 합격 답변 예시

CDN은 사용자와 가까운 엣지 서버에 정적 리소스를 배치해 지연 시간을 줄이고 원본 서버 부하를 낮추기 위해 필요합니다. 이미지, JS, CSS, 영상 세그먼트 같은 자산은 CDN 효과가 매우 큽니다. 즉 CDN은 **속도 향상 도구이면서 동시에 확장성과 안정성 도구**이기도 합니다.

### 가산점 포인트

CDN은 **캐시 + 지리적 분산**이 핵심이다. 원본 서버가 서울에 있을 때 미국 사용자는 RTT가 200ms 이상이지만, CDN 엣지(미국)가 있으면 20ms로 줄어든다. 또 원본 서버가 다운되어도 CDN 캐시가 남아있으면 정적 자산은 계속 서빙 가능하다.

### 상세 해설

```
CDN 없이:
  미국 사용자 ─(200ms RTT)──→ 서울 서버
  매 요청마다 왕복

CDN 있으면:
  미국 사용자 ─(20ms RTT)──→ CDN 엣지 (미국)
                              ↓ cache miss 시
                         서울 원본 서버
  → 첫 요청만 원본까지, 이후 엣지에서 캐시 서빙
```

| CDN 장점 | 설명 |
|----------|------|
| 지연 감소 | 사용자 가까이에서 응답 |
| 서버 부하 감소 | 원본 요청 빈도 줄어듦 |
| 가용성 향상 | 원본 다운 시에도 캐시된 자산 서빙 |
| DDoS 방어 | 엣지에서 트래픽 분산/흡수 |

> CDN을 알았으니, 다음에는 **실시간 통신 전략** — WebSocket vs Polling을 봐야 한다.

## 13. 웹소켓과 HTTP polling의 차이를 설명해보세요.

### 합격 답변 예시

HTTP polling은 일정 주기로 반복 요청을 보내 상태 변화를 확인하는 방식이고, 웹소켓은 연결을 계속 유지하면서 양방향으로 메시지를 주고받는 방식입니다. polling은 구현이 단순하지만 불필요한 요청이 많아질 수 있고, 웹소켓은 실시간성이 뛰어나지만 연결 관리와 운영 복잡도가 높습니다.

### 가산점 포인트

**"기술 우열이 아니라 요구사항 적합성"**이 선택 기준이다. 알림 피드처럼 서버→클라이언트 단방향이면 SSE가 더 적합하고, 채팅처럼 양방향이면 WebSocket, 상태 확인 정도면 polling으로 충분하다.

### 상세 해설

```
HTTP Polling:
  클라이언트: "새 메시지 있어?" → 없음
  클라이언트: "새 메시지 있어?" → 없음  (불필요한 요청)
  클라이언트: "새 메시지 있어?" → 있음!
  → 간격이 1초면 최대 1초 지연, 10초면 최대 10초 지연

WebSocket:
  [연결 수립 — 한 번만]
  서버: "새 메시지 왔어!" → 즉시 전달 (0ms 지연)
  클라이언트: "읽었어!" → 즉시 전달
  → 양방향, 실시간, 하지만 연결 유지 비용 있음
```

```ts
// Polling 예시
useEffect(() => {
  const interval = setInterval(async () => {
    const res = await fetch('/api/notifications');
```

## 14. 네트워크가 불안정한 환경에서 프론트엔드가 해야 할 역할은 무엇이라고 생각하나요?

### 합격 답변 예시

프론트엔드는 단순히 에러를 보여주는 역할이 아니라, **사용자의 의도를 최대한 보존하는 역할**을 해야 한다고 생각합니다. 재시도, timeout, fallback UI, 임시 저장, 중복 제출 방지, 부분 성공/실패 구분, 상태 가시화가 중요합니다.

### 가산점 포인트

사용자는 네트워크 문제와 앱 문제를 구분하지 않는다. **"안 된다"로만 느끼기 때문에**, 프론트엔드는 실패를 "복구 가능한 경험"으로 바꾸는 쪽까지 설계해야 한다.

### 상세 해설

```ts
// 임시 저장으로 입력 데이터 보호
function useAutoSave(key: string, value: string) {
  useEffect(() => {
    const timeout = setTimeout(() => {
      localStorage.setItem(`draft:${key}`, value);
    }, 1000);  // 1초 debounce로 임시 저장
    return () => clearTimeout(timeout);
  }, [key, value]);

  // 초기값으로 임시 저장 데이터 복원
  return localStorage.getItem(`draft:${key}`) || '';
}

// 재시도 + exponential backoff
async function fetchWithRetry(url: string, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const res = await fetch(url);
      if (res.ok) return res;
    } catch (err) {
      if (i === maxRetries - 1) throw err;
      await new Promise(r => setTimeout(r, 1000 * Math.pow(2, i)));
      // 1초 → 2초 → 4초 간격으로 재시도
    }
  }
}
```

#### 실무 에피소드

> 학교 Wi-Fi에서 과제 제출 중 네트워크가 끊겨 작성 내용이 사라지는 불만이 있었다. localStorage 임시 저장 + retry 로직을 추가하니, 네트워크 복구 후 자동 제출되어 데이터 유실이 사라졌다.

> 불안정한 환경 대응을 알았으니, 다음에는 **온라인/오프라인 감지의 한계**를 봐야 한다.

## 15. 온라인/오프라인 상태 감지만으로 충분하지 않은 이유를 설명해보세요.

### 합격 답변 예시

브라우저의 `navigator.onLine`은 네트워크 연결의 아주 거친 상태만 알려줄 뿐, 실제 서비스 접근 가능 여부를 보장하지 않습니다. Wi-Fi는 연결돼 있어도 DNS 장애, 서버 장애, captive portal이 있을 수 있습니다.

### 가산점 포인트

**`navigator.onLine`은 힌트일 뿐 source of truth는 아니다.** 실제 서비스에서는 heartbeat 요청이나 health check API, 최근 실패 횟수 같은 추가 신호를 함께 봐야 한다.

### 상세 해설

```ts
// ❌ navigator.onLine만 믿기
if (navigator.onLine) {
  // "온라인"이라고 표시 — 하지만 서버가 죽어있을 수 있음
}

// ✅ 실제 서비스 접근 가능 여부 확인
async function checkServiceAvailability() {
  try {
    const res = await fetch('/api/health', {
      signal: AbortSignal.timeout(3000),  // 3초 timeout
    });
    return res.ok;
  } catch {
    return false;  // 네트워크 에러 또는 timeout
  }
}

// 두 신호를 함께 사용
const isOnline = navigator.onLine && await checkServiceAvailability();
```

> 감지의 한계를 알았으니, 다음에는 **실패에서 복구하는 구체적 전략** — timeout, retry, backoff를 봐야 한다.

## 16. timeout, retry, backoff 전략은 각각 왜 필요한가요?

### 합격 답변 예시

timeout은 무한 대기를 막기 위해, retry는 일시적 실패를 자동 복구하기 위해, backoff는 재시도를 너무 공격적으로 반복해서 서버를 악화시키지 않기 위해 필요합니다. 셋은 따로 쓰는 게 아니라 **함께 설계**해야 합니다.

### 가산점 포인트

**exponential backoff + jitter**가 가장 현실적인 전략이다. 재시도 간격을 지수적으로 늘리되, 모든 클라이언트가 같은 타이밍에 재시도하는 "thundering herd" 문제를 방지하기 위해 랜덤 지터를 추가한다.

### 상세 해설

```ts
// timeout + retry + exponential backoff + jitter
async function resilientFetch(url: string, options?: RequestInit) {
  const maxRetries = 3;
  const baseDelay = 1000;  // 1초

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const controller = new AbortController();
      const timeoutId = setTimeout(() => controller.abort(), 5000);  // 5초 timeout

      const res = await fetch(url, {
        ...options,
        signal: controller.signal,
      });
      clearTimeout(timeoutId);

      if (res.ok) return res;
      if (res.status < 500) throw new Error(`Client error: ${res.status}`);
      // 5xx는 재시도 가능
    } catch (err) {
      if (attempt === maxRetries) throw err;
      // exponential backoff + jitter
      const delay = baseDelay * Math.pow(2, attempt) + Math.random() * 1000;
      await new Promise(r => setTimeout(r, delay));
      // 1차: ~1-2초, 2차: ~2-3초, 3차: ~4-5초
    }
  }
}
```

```
timeout 없이 retry만 있으면:
  요청 → (30초 대기) → timeout 없음 → 사용자가 30초 기다림
  → UX 최악

retry 없이 timeout만 있으면:
  요청 → 5초 timeout → 실패 → 끝
  → 일시적 네트워크 문제에 약함

backoff 없이 retry만 있으면:
  실패 → 즉시 재시도 → 실패 → 즉시 재시도 → 실패
  → 서버에 트래픽 폭탄 (상황 악화)
```

#### 자주 하는 오해

- "retry를 많이 하면 좋다" → **서버 부하를 더 키울 수 있음.** 최대 횟수를 정하고, backoff로 간격을 늘려야 한다.

> retry 전략을 알았으니, 다음에는 **중복 요청 자체를 방지하는 방법**을 봐야 한다.

## 17. API 요청이 중복 전송될 때 어떤 문제가 생기고, 어떻게 방지할 수 있나요?

### 합격 답변 예시

중복 요청은 중복 생성, 중복 결제, 상태 꼬임 같은 문제를 만들 수 있습니다. 프론트에서는 버튼 disable, loading flag, debounce, abort로 막고, 서버에서는 Idempotency-Key나 중복 처리 방지 로직으로 보완해야 합니다. 즉 **프론트와 백엔드가 함께 막아야** 합니다.

### 가산점 포인트

프론트만으로는 완벽히 막을 수 없다 — 사용자가 DevTools에서 요청을 복사해 보낼 수도 있고, 네트워크 재시도가 자동으로 발생할 수도 있다. **서버측 Idempotency-Key가 마지막 방어선**이다.

### 상세 해설

```ts
// 프론트엔드 방어: 요청 중 중복 방지
function useSubmit() {
  const [isPending, setIsPending] = useState(false);

  const submit = useCallback(async (data: FormData) => {
    if (isPending) return;  // 이미 요청 중이면 무시
    setIsPending(true);
    try {
      await fetch('/api/orders', {
        method: 'POST',
        headers: {
          'Idempotency-Key': crypto.randomUUID(),  // 서버측 방어
        },
        body: JSON.stringify(data),
      });
    } finally {
      setIsPending(false);
    }
  }, [isPending]);

  return { submit, isPending };
}

// UI에서 버튼 disable
<button onClick={submit} disabled={isPending}>
  {isPending ? '처리 중...' : '결제하기'}
</button>
```

> 중복 방지를 알았으니, 다음에는 **실제 업로드 시나리오에서의 네트워크 실패 대응**을 봐야 한다.

## 18. 파일 업로드 기능에서 네트워크 실패를 고려한다면 어떤 UX를 설계하겠습니까?

### 합격 답변 예시

업로드 진행률, 취소, 재시도, 실패 파일 목록, 부분 성공/부분 실패 구분을 먼저 설계하겠습니다. 대용량 업로드라면 **"전송 실패인지, 서버 처리 실패인지"를 분리**해서 보여줘야 합니다.

### 가산점 포인트

**진행률 표시 + 개별 파일 상태 관리**가 핵심이다. "업로드 중..." 하나로 퉁치면 사용자는 어떤 파일이 성공하고 어떤 파일이 실패했는지 알 수 없다.

### 상세 해설

```tsx
// 파일별 상태 관리
type FileUploadState = {
  file: File;
  status: 'pending' | 'uploading' | 'success' | 'error';
  progress: number;
  error?: string;
};

// 업로드 진행률 표시
async function uploadFile(file: File, onProgress: (pct: number) => void) {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest();
    xhr.upload.addEventListener('progress', (e) => {
      if (e.lengthComputable) onProgress(Math.round(e.loaded / e.total * 100));
    });
    xhr.addEventListener('load', () => resolve(xhr.response));
    xhr.addEventListener('error', () => reject(new Error('업로드 실패')));
    xhr.open('POST', '/api/upload');
    xhr.send(file);
  });
}
```

```
좋은 업로드 UX:
  📄 report.csv    ████████████ 100% ✅ 완료
  📄 data.xlsx     ████████░░░░  67% ⏳ 업로드 중
  📄 invalid.txt   ──────────── ❌ 형식 오류
  📄 large.zip     ████░░░░░░░░  33% ⚠️ 네트워크 실패 [재시도]

나쁜 업로드 UX:
  "업로드 실패" (어떤 파일? 왜? 다시 해도 되나?)
```

> 업로드 UX를 알았으니, 다음에는 **복잡한 장애 상황에서의 원인 분리** 능력을 봐야 한다.

## 19. iframe 로딩 지연이나 세션 만료가 겹치는 상황에서 어떤 원인 분리를 시도하겠습니까?

### 합격 답변 예시

먼저 **네트워크 문제, 인증 문제, iframe 콘텐츠 자체 문제**를 분리해서 봅니다. 세션 검증 API와 토큰 상태를 확인하고, iframe src 요청의 응답 상태, 쿠키 전달 여부, CORS 문제, 콘솔 에러를 함께 확인합니다.

### 가산점 포인트

UI에서도 이 셋을 **하나의 "로딩 실패"로 묶지 않아야** 한다. "세션 만료" vs "외부 콘텐츠 응답 없음" vs "네트워크 지연"을 구분해서 보여주는 것이 원인 분리의 완성이다.

### 상세 해설

```
원인 분리 순서:

1. 세션 유효한가?
   → /api/auth/me 호출 → 401이면 세션 만료

2. iframe 요청이 가는가?
   → Network 탭에서 iframe src URL 확인
   → 요청이 안 나가면 브라우저 차단 (CSP, X-Frame-Options)

3. 응답 상태는?
   → 200이면 콘텐츠 로딩 문제
   → 403이면 iframe 차단 (X-Frame-Options: DENY)
   → timeout이면 네트워크 문제

4. 쿠키/인증이 전달되는가?
   → cross-origin iframe에서는 SameSite 쿠키가 안 갈 수 있음
```

> 원인 분리를 알았으니, 마지막으로 **서비스 전체 관점에서 네트워크 문제의 의미**를 정리해보자.

## 20. 실제 서비스에서 "사용자는 네트워크 문제와 웹앱 문제를 구분하지 못한다"는 말이 왜 중요한지 설명해보세요.

### 합격 답변 예시

사용자 입장에서는 내부 원인이 무엇이든 결국 **"안 된다"**로 느껴지기 때문입니다. 그래서 개발자는 내부적으로는 원인을 분리하더라도, 외부적으로는 **사용자가 이해할 수 있는 메시지와 복구 경로**를 제공해야 합니다.

### 가산점 포인트

**"프론트엔드 개발자는 오류를 단순히 표시하는 사람이 아니라, 사용자가 실패를 견딜 수 있게 만드는 사람이다"**라고 말하면 좋다.

### 상세 해설

```
기술적 원인:           사용자가 보는 것:
  DNS 실패              "페이지가 안 뜬다"
  API 500               "뭔가 에러 났다"
  CORS 에러             "화면이 빈 채로 멈췄다"
  토큰 만료             "로그인이 풀렸다"
  CDN 장애              "이미지가 안 나온다"

→ 사용자에게는 전부 "이 앱 고장 났나?"

좋은 에러 메시지:
  ❌ "Error: CORS policy blocked"
  ✅ "데이터를 불러올 수 없습니다. 다시 시도해주세요."

  ❌ "401 Unauthorized"
  ✅ "세션이 만료되었습니다. 다시 로그인해주세요."
```

#### 실무 에피소드

> 고객 문의: "앱이 고장 났어요." 로그를 보니 CDN 장애로 JS 번들이 안 내려온 것이었다. 사용자는 CDN이 뭔지 모른다. 사용자에게는 "현재 일시적 문제가 있어 잠시 후 다시 시도해주세요"라는 메시지와 자동 새로고침이 필요했다.

## 보완내용

### 왜 보완이 필요한가

기존 네트워크 20문항은 HTTP, TLS, 인증, CORS, 캐시, DNS, CDN, retry, timeout, 네트워크 불안정 UX까지 폭넓고 실무적이다. 다만 프론트엔드 면접에서 변별력이 더 커지는 지점은 브라우저 보안 정책, 최신 HTTP 진화, 쿠키 보안 속성, 캐시 계층 차이처럼 조금 더 구체적인 디테일이다.

### 추가하면 좋은 질문

#### 1. HTTP/1.0, HTTP/1.1, HTTP/2, HTTP/3의 차이를 설명해보세요.

##### 합격 답변 예시

HTTP/1.0은 요청마다 TCP 연결을 새로 여는 구조, HTTP/1.1은 Keep-Alive로 연결 재사용, HTTP/2는 멀티플렉싱으로 동시 전송, HTTP/3는 QUIC(UDP 기반)로 TCP HOL Blocking까지 제거한 구조입니다.

##### 상세 해설

| 버전 | 연결 방식 | 핵심 개선 | 남은 한계 |
| --- | --- | --- | --- |
| HTTP/1.0 | 요청마다 새 TCP | 단순 구조 | 연결 비용 큼 |
| HTTP/1.1 | Keep-Alive 연결 재사용 | 연결 비용 절감 | HOL Blocking |
| HTTP/2 | 멀티플렉싱 | 동시 전송, 헤더 압축 | TCP 패킷 손실 시 전체 지연 |
| HTTP/3 | QUIC (UDP 기반) | TCP HOL Blocking 제거 | UDP 차단 가능 환경 |

- 프론트엔드 연결: HTTP/2 이후 번들 분할 전략이 바뀜 (여러 파일 동시 전송 가능)

#### 2. Keep-Alive와 connection reuse가 왜 중요한가요?

##### 합격 답변 예시

TCP + TLS 핸드셰이크 비용이 크기 때문에, 한 번 맺은 연결을 재사용하면 성능이 크게 향상됩니다. 웹 페이지 하나에 수십 개 리소스 요청이 나가므로 연결 재사용은 필수입니다.

##### 상세 해설

- 연결을 매번 새로 여는 비용: TCP 1 RTT + TLS 1-2 RTT
- Keep-Alive: 한 번 맺은 연결을 유지하며 여러 요청 순차 전송
- HTTP/2 이상: 하나의 연결에서 동시에 여러 요청 처리 (멀티플렉싱)

#### 3. `SameSite`, `HttpOnly`, `Secure` 쿠키 속성을 설명해보세요.

##### 합격 답변 예시

`SameSite`는 CSRF 방어, `HttpOnly`는 XSS를 통한 쿠키 탈취 방지, `Secure`는 HTTPS 전용 전송입니다. 인증 쿠키는 보통 `HttpOnly; Secure; SameSite=Lax` 조합으로 설정합니다.

##### 상세 해설

| 속성 | 역할 | 방어 대상 |
| --- | --- | --- |
| `SameSite` | cross-site 요청 시 쿠키 전송 조건 제한 | CSRF |
| `HttpOnly` | JavaScript에서 쿠키 접근 차단 | XSS로 쿠키 탈취 |
| `Secure` | HTTPS에서만 쿠키 전송 | 평문 전송 도청 |

#### 4. `Idempotency-Key`가 왜 필요한지 설명해보세요.

##### 합격 답변 예시

POST 요청의 멱등성을 애플리케이션 수준에서 보장하기 위해 필요합니다. 네트워크 재시도, 중복 클릭 시 서버가 같은 키를 감지해 한 번만 처리합니다.

##### 상세 해설

- 클라이언트가 UUID를 생성해 `Idempotency-Key` 헤더에 담아 전송
- 서버는 키를 저장하고, 같은 키 재요청 시 이전 응답 반환
- 결제, 주문 같은 크리티컬한 작업에서 특히 중요

#### 5. SSE와 WebSocket의 차이를 설명해보세요.

##### 합격 답변 예시

SSE는 서버→클라이언트 단방향 실시간 스트림, WebSocket은 양방향 통신입니다. SSE는 HTTP 위에서 동작하며 자동 재연결을 기본 지원하고, WebSocket은 별도 프로토콜로 업그레이드됩니다.

##### 상세 해설

| | SSE | WebSocket |
| --- | --- | --- |
| 방향 | 서버→클라이언트 | 양방향 |
| 프로토콜 | HTTP | WS |
| 재연결 | 자동 | 직접 구현 |
| 적합 | 알림, 피드 | 채팅, 게임 |

#### 6. 브라우저 캐시와 서비스 워커 캐시의 차이를 설명해보세요.

##### 합격 답변 예시

브라우저 캐시는 서버 헤더(Cache-Control)에 따라 자동 동작하고, 서비스 워커 캐시는 JavaScript로 직접 제어합니다. 서비스 워커가 먼저 요청을 가로채고, 그 뒤에 브라우저 HTTP 캐시를 거치는 계층 구조입니다.

##### 상세 해설

| | 브라우저 캐시 | 서비스 워커 캐시 |
| --- | --- | --- |
| 제어 방식 | 서버 응답 헤더 | JavaScript 코드 |
| 오프라인 | 제한적 | 가능 |
| 전략 유연성 | 고정 정책 | 자유 |
| 순서 | SW 캐시 뒤에 위치 | 네트워크 요청 전에 확인 |

### 이 섹션 보완 핵심

네트워크 섹션은 이미 강하다. 여기에 HTTP 진화, 연결 재사용, 쿠키 보안 속성, Idempotency-Key, SSE, 서비스 워커 캐시까지 더하면 브라우저 네트워크 감각이 있는 답변자를 더 잘 가를 수 있다.
