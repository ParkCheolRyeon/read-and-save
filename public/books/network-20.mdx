# 06. 네트워크 20문항

> HTTP(HyperText Transfer Protocol), TLS(Transport Layer Security), CORS(Cross-Origin Resource Sharing), 캐시, 실패 대응 UX까지 네트워크 관점을 묻는 20문항입니다.

## 6. 네트워크 지식 질문 20문항

## 총평

이 섹션은 프론트엔드에서 진짜 중요하다. 왜냐하면 사용자는 결국 네트워크를 타고 들어오는 서비스를 쓰기 때문이다. 좋은 답변은 아래 3층이 같이 있어야 한다.

- HTTP/TLS/CORS 같은 프로토콜 이해
- 브라우저가 실제로 어떻게 행동하는지에 대한 클라이언트 이해
- 실패했을 때 어떻게 UX를 설계할지에 대한 서비스 감각

## 1. HTTP와 HTTPS의 차이를 설명해보세요.

### 합격 답변 예시

HTTP는 데이터를 평문으로 주고받는 프로토콜이고, HTTPS는 HTTP 위에 TLS를 얹어 암호화와 무결성 보호, 서버 인증을 제공하는 방식입니다. 즉 HTTPS는 단순히 “보안 버전 HTTP”가 아니라, 중간에서 내용을 읽거나 변조하거나 서버를 사칭하는 위험을 줄이기 위한 구조입니다. 그래서 로그인이나 결제뿐 아니라 일반 웹서비스도 기본적으로 HTTPS를 사용하는 것이 맞습니다.

### 상세 해설

이 질문에서 꼭 들어가야 하는 3가지는 이거다.

- 암호화
- 무결성
- 서버 인증

많은 지원자가 “HTTPS는 암호화된 HTTP입니다”에서 끝낸다. 그건 절반짜리다.

#### 왜 중요한가?

- 중간자 공격 방지
- 쿠키 탈취 방지
- 콘텐츠 변조 방지
- 신뢰 가능한 서버 확인
- 프론트엔드 연결

브라우저 기능 일부는 HTTPS 환경에서만 제대로 동작한다.

- 예:
- service worker
- secure cookie
- 많은 최신 브라우저 API(Application Programming Interface)

## 2. TLS 핸드셰이크가 대략 어떻게 이뤄지는지 설명해보세요.

### 합격 답변 예시

클라이언트가 지원 가능한 암호 스위트와 랜덤 값을 서버에 보내고, 서버는 자신의 인증서와 선택한 암호 방식, 랜덤 값을 응답합니다. 클라이언트는 서버 인증서를 검증한 뒤, 키 교환 과정을 통해 세션 키를 합의하고, 이후 실제 데이터 통신은 주로 대칭키 기반으로 빠르게 암호화해 진행합니다. 즉 핵심 흐름은 서버 신원 확인, 키 합의, 이후 안전한 통신입니다.

### 상세 해설

면접에서는 TLS 1.2, 1.3 세부 차이를 끝없이 말할 필요는 없다. 핵심 구조만 정확히 잡으면 된다.

- 흐름 요약
- Client Hello
- Server Hello + 인증서
- 인증서 검증
- 키 교환
- 세션 키 확립
- 이후 대칭 암호화 통신

#### 왜 비대칭 + 대칭을 같이 쓰나?

비대칭은 신뢰성과 키 교환에 좋지만 느리고, 대칭은 빠르기 때문이다.

- 즉,
- 처음 신뢰를 만들 때는 비대칭
- 실제 데이터 통신은 대칭

## 3. HTTP 메서드 GET, POST, PUT, PATCH, DELETE의 차이를 설명해보세요.

| 메서드 | 주 용도 | 멱등성 기본 감각 | 비고 |
| --- | --- | --- | --- |
| `GET` | 조회 | 보통 멱등적 | 캐시 친화적 |
| `POST` | 생성 / 처리 요청 | 보통 멱등적이지 않음 | 서버가 새 작업을 만들기 쉬움 |
| `PUT` | 전체 교체 | 보통 멱등적 | 리소스 전체 상태를 덮어쓰는 의미가 강함 |
| `PATCH` | 부분 수정 | 설계에 따라 다름 | 일부 필드만 바꾸는 데 자연스러움 |
| `DELETE` | 삭제 | 보통 멱등적으로 설계 | 이미 삭제된 상태라도 최종 결과는 같게 두는 편 |

### 합격 답변 예시

GET은 조회, POST는 생성이나 서버 측 처리 요청, PUT은 리소스 전체 교체 성격, PATCH는 부분 수정, DELETE는 삭제를 의미합니다. 다만 실무에서는 단순 암기보다 “이 메서드가 어떤 의도를 표현하느냐”가 중요합니다. 예를 들어 PUT은 전체 상태를 덮어쓴다는 의미가 강하고, PATCH는 일부 필드만 바꾸는 데 더 자연스럽습니다.

### 상세 해설

좋은 답변은 메서드 이름 나열에서 끝나지 않는다.

- GET
- 조회
- 보통 멱등적
- 캐시 친화적
- POST
- 생성
- 처리 요청
- 보통 멱등적이지 않음
- PUT
- 전체 교체 의미
- 보통 멱등적
- PATCH
- 부분 수정
- 멱등성은 설계에 따라 달라질 수 있음
- DELETE
- 삭제
- 보통 멱등적으로 설계

## 4. 멱등성이 무엇인지 설명하고, 왜 중요한지 말해보세요.

### 합격 답변 예시

같은 요청을 여러 번 보내도 결과 상태가 같다면 멱등적이라고 합니다. 예를 들어 같은 리소스를 같은 값으로 PUT 하는 요청은 여러 번 보내도 최종 상태가 같기 때문에 멱등적이라고 볼 수 있습니다. 이 개념이 중요한 이유는 네트워크 재시도, 중복 클릭, 장애 복구 상황에서 요청이 여러 번 발생해도 시스템이 예상 밖으로 망가지지 않게 만들 수 있기 때문입니다.

### 상세 해설

이 질문은 네트워크 실패 대응과 아주 밀접하다.

#### 왜 중요한가?

- retry 안전성
- 중복 요청 방지
- 장애 후 복구
- API 설계 일관성

#### 프론트엔드 예시

사용자가 결제 버튼을 두 번 눌렀을 때 서버가 멱등성 없이 설계돼 있으면 중복 결제가 날 수 있다. 그래서 프론트도 막아야 하지만, 서버 설계도 멱등성을 고려해야 한다.

## 5. HTTP 상태코드 200, 201, 204, 400, 401, 403, 404, 409, 500의 의미를 설명해보세요.

| 상태 코드 | 의미 | 면접에서 같이 말하면 좋은 포인트 |
| --- | --- | --- |
| `200` | 일반적인 성공 | 가장 흔한 성공 응답 |
| `201` | 리소스 생성 성공 | 생성 완료 응답에 적합 |
| `204` | 성공했지만 본문 없음 | 삭제/갱신 후 본문이 필요 없을 때 자주 사용 |
| `400` | 잘못된 요청 | 유효성 문제, 잘못된 파라미터 |
| `401` | 인증 필요 / 인증 실패 | 로그인 안 됨, 토큰 만료와 연결 |
| `403` | 권한 없음 | 인증은 됐지만 접근 권한이 없음 |
| `404` | 리소스를 찾을 수 없음 | URL 또는 대상 리소스 부재 |
| `409` | 충돌 | 중복 생성, 버전 충돌, 상태 충돌 |
| `500` | 서버 내부 오류 | 서버 쪽 예외 또는 처리 실패 |

### 합격 답변 예시

200은 일반적인 성공, 201은 리소스 생성 성공, 204는 성공했지만 응답 본문이 없는 경우입니다. 400은 잘못된 요청, 401은 인증이 필요하거나 인증 실패, 403은 인증은 되었지만 권한이 없는 경우, 404는 리소스를 찾을 수 없는 경우, 409는 충돌, 500은 서버 내부 오류입니다. 면접에서는 단순 암기보다 401과 403의 차이, 201과 204를 언제 쓰는지까지 설명할 수 있으면 좋습니다.

### 상세 해설

- 자주 헷갈리는 것
- 401: 인증 문제
- 403: 권한 문제
- 즉
- 로그인 안 됨 / 토큰 만료 → 401
- 로그인은 됐지만 접근 권한 없음 → 403
- 409

실무에서 꽤 중요하다.

- 예:
- 중복 생성
- 버전 충돌
- 이미 존재하는 자원
- 좋은 답변은

“에러는 다 400/500으로 몰지 않는다”는 감각까지 보여준다.

## 6. 쿠키, 세션, 토큰의 차이를 설명해보세요.

### 합격 답변 예시

쿠키는 브라우저에 저장되는 작은 데이터 조각이고, 세션은 서버가 사용자 상태를 관리하는 방식이며, 토큰은 클라이언트가 들고 다니며 인증 정보를 증명하는 값입니다. 즉 쿠키는 저장 수단에 가깝고, 세션과 토큰은 인증/상태 관리 전략에 더 가깝습니다. 실무에서는 세션 ID를 쿠키에 저장할 수도 있고, 토큰도 쿠키나 메모리 등 다양한 방식으로 보관할 수 있습니다.

### 상세 해설

이 질문에서 중요한 건 “쿠키=세션”이 아니라는 걸 분리하는 것이다.

- 쿠키
- 저장 매체/전달 매체
- 자동 전송 가능
- 세션
- 서버 중심 상태 관리
- 보통 쿠키에 세션 식별자 저장
- 토큰
- 클라이언트가 인증 증거로 제시
- stateless하게 설계 가능

즉 서로 경쟁 개념이 아니라 축이 다르다.

## 7. Access Token과 Refresh Token 구조를 설명해보세요.

### 합격 답변 예시

Access Token은 실제 API 요청 인증에 사용하는 짧은 수명의 토큰이고, Refresh Token은 Access Token이 만료됐을 때 새 Access Token을 재발급받기 위한 더 긴 수명의 토큰입니다. 이렇게 나누는 이유는 탈취 위험을 줄이면서도 사용자가 자주 다시 로그인하지 않도록 하기 위해서입니다. 보통 Access Token은 짧고 가볍게, Refresh Token은 더 엄격하게 보호하며 재발급 정책과 폐기 전략을 함께 설계합니다.

### 상세 해설

#### 왜 둘로 나누나?

하나만 길게 쓰면 편하긴 하지만, 유출 시 위험이 너무 크다.

#### 핵심

- Access Token: 짧음, 자주 사용
- Refresh Token: 길음, 재발급용
- 보안/UX 균형

#### 프론트엔드 포인트

- 만료 처리 UX
- silent refresh
- refresh 실패 시 로그인 유도
- 무한 재시도 방지

## 8. CORS가 왜 발생하는지 설명해보세요.

### 합격 답변 예시

CORS는 브라우저의 same-origin policy 때문에 발생합니다. 브라우저는 기본적으로 다른 출처의 리소스 접근을 제한하고, 서버가 명시적으로 허용할 때만 cross-origin 요청 결과를 읽을 수 있게 합니다. 즉 CORS는 서버가 갑자기 까다로운 게 아니라, 브라우저 보안 정책을 서버가 헤더로 조정하는 구조라고 보는 게 맞습니다.

### 상세 해설

이 질문의 핵심은 CORS는 브라우저 이슈라는 점이다. 서버-서버 통신엔 보통 CORS가 없다. 왜냐하면 same-origin policy는 브라우저 보안 모델이기 때문이다.

- 자주 하는 오해
- “백엔드가 막았다”
- “브라우저 버그다”

아니다. 브라우저가 사용자를 보호하려고 기본 차단하고, 서버가 허용하는 것이다.

## 9. 브라우저가 preflight 요청을 보내는 경우를 설명해보세요.

### 합격 답변 예시

브라우저는 단순 요청이 아닌 경우, 예를 들어 메서드가 GET/POST/HEAD 외이거나, 커스텀 헤더를 쓰거나, 특정 단순 Content-Type이 아닌 경우 먼저 OPTIONS 요청으로 허용 여부를 확인할 수 있습니다. 이 OPTIONS 요청이 preflight입니다. 즉 실제 요청 전에 “이 요청을 보내도 되는지”를 묻는 사전 협상입니다.

### 상세 해설

- preflight가 생기는 대표 상황
- PUT, PATCH, DELETE
- Authorization 헤더
- 커스텀 헤더
- Content-Type이 `application/json`이면 simple request 조건에서 벗어남

#### 실무 연결

네트워크 탭 보면 요청이 두 번 가는 것처럼 보여서 당황하는 경우가 많다. 그때 preflight를 이해하고 있으면 원인을 바로 설명할 수 있다.

## 10. 캐시 헤더(Cache-Control, ETag, Last-Modified)의 역할을 설명해보세요.

### 합격 답변 예시

Cache-Control은 캐시 정책 자체를 정하는 헤더이고, ETag는 리소스 버전을 식별해 조건부 요청에 활용하며, Last-Modified는 마지막 수정 시점을 기준으로 재검증할 때 사용됩니다. 즉 캐시는 단순히 “다시 안 받는 것”이 아니라, 안 바뀌었으면 재사용하고 바뀌었는지 효율적으로 확인하는 것까지 포함한 전략입니다.

### 상세 해설

- Cache-Control
- max-age
- no-store
- no-cache
- public/private
- ETag
- 리소스 fingerprint
- If-None-Match와 함께 사용
- 바뀌지 않으면 304 Not Modified
- Last-Modified
- 마지막 수정 시각
- If-Modified-Since와 함께 사용

#### 프론트엔드 포인트

번들, 이미지, API 응답의 캐시 전략이 UX와 서버 비용 둘 다 바꾼다.

## 11. DNS(Domain Name System)가 무엇이고, 사용자가 URL을 입력했을 때 어떤 흐름으로 서버에 도달하는지 설명해보세요.

### 합격 답변 예시

DNS는 사람이 읽는 도메인 이름을 IP 주소로 변환하는 시스템입니다. 사용자가 URL을 입력하면 브라우저와 OS, DNS 캐시를 먼저 확인하고, 필요하면 DNS 조회로 IP를 얻습니다. 그 다음 TCP(Transmission Control Protocol) 연결을 맺고, HTTPS라면 TLS 핸드셰이크를 거친 뒤, HTTP 요청을 보내고 응답을 받게 됩니다.

### 상세 해설

이 질문은 흐름 순서가 중요하다.

- URL 입력
- DNS 조회
- IP 획득
- TCP 연결
- TLS 핸드셰이크 (HTTPS라면)
- HTTP 요청
- 응답
- 좋은 답변
- 네트워크 계층을 모두 길게 읊기보다,

사용자 요청이 실제로 서버에 도달하기까지의 큰 흐름을 정확히 잡는 게 좋다.

## 12. CDN(Content Delivery Network)이 왜 필요한지 설명해보세요.

### 합격 답변 예시

CDN은 사용자와 가까운 엣지 서버에 정적 리소스를 배치해 지연 시간을 줄이고 원본 서버 부하를 낮추기 위해 필요합니다. 이미지, JS, CSS, 영상 세그먼트 같은 자산은 CDN 효과가 매우 큽니다. 즉 CDN은 단순 속도 향상 도구이면서 동시에 확장성과 안정성 도구이기도 합니다.

### 상세 해설

- CDN 장점
- 지리적으로 가까운 곳에서 응답
- 원본 서버 부하 감소
- 캐시 효율 향상
- 장애 완충 역할
- 프론트엔드 연결
- 이미지 로딩
- JS 번들 전달
- HLS/DASH 세그먼트 전송
- 다운로드 파일 분산

## 13. 웹소켓과 HTTP polling의 차이를 설명해보세요.

### 합격 답변 예시

HTTP polling은 일정 주기로 반복 요청을 보내 상태 변화를 확인하는 방식이고, 웹소켓은 연결을 계속 유지하면서 양방향으로 메시지를 주고받는 방식입니다. polling은 구현이 단순하지만 불필요한 요청이 많아질 수 있고, 웹소켓은 실시간성이 뛰어나지만 연결 관리와 운영 복잡도가 더 높습니다. 그래서 채팅, 실시간 알림 같은 경우 웹소켓이 유리하고, 단순 상태 확인은 polling으로도 충분할 수 있습니다.

### 상세 해설

이 질문의 핵심은 기술 우열이 아니라 요구사항 적합성이다.

- polling이 맞는 경우
- 단순 주기 확인
- 실시간성 아주 높지 않음
- 운영 단순성 우선
- 웹소켓이 맞는 경우
- 실시간성 중요
- 서버 푸시 필요
- 양방향 이벤트 많음

## 14. 네트워크가 불안정한 환경에서 프론트엔드가 해야 할 역할은 무엇이라고 생각하나요?

### 합격 답변 예시

프론트엔드는 단순히 에러를 보여주는 역할이 아니라, 사용자의 의도를 최대한 보존하는 역할을 해야 한다고 생각합니다. 그래서 재시도, timeout, fallback UI, 임시 저장, 중복 제출 방지, 부분 성공/실패 구분, 상태 가시화가 중요합니다. 사용자는 네트워크 문제와 앱 문제를 구분하지 않기 때문에, 프론트엔드는 실패를 “복구 가능한 경험”으로 바꾸는 쪽까지 설계해야 합니다.

### 상세 해설

이 질문은 기술보다 UX 철학을 본다.

#### 좋은 답변 포인트

- 실패를 숨기지 않는다
- 원인을 가능한 이해 가능한 언어로 보여준다
- 사용자가 다시 시도할 수 있게 한다
- 입력 손실을 줄인다
- 아주 좋은 답변

실제 불안정한 환경 경험과 연결해서 말하면 강하다.

## 15. 온라인/오프라인 상태 감지만으로 충분하지 않은 이유를 설명해보세요.

### 합격 답변 예시

브라우저의 online/offline 신호는 네트워크 연결의 아주 거친 상태만 알려줄 뿐이라, 실제 서비스 접근 가능 여부를 완전히 보장하지 않습니다. 예를 들어 Wi-Fi는 연결돼 있어도 DNS 장애, 서버 장애, captive portal, 특정 도메인 차단이 있을 수 있습니다. 그래서 실제 서비스에서는 단순 navigator.onLine만 믿지 않고, heartbeat 요청이나 헬스체크 API, 최근 실패 횟수 같은 추가 신호를 함께 보는 편이 좋습니다.

### 상세 해설

#### 왜 부족한가?

- 인터넷은 연결됐지만 서버는 죽었을 수 있음
- 로컬 네트워크만 연결돼 있을 수 있음
- 특정 API만 실패할 수 있음

#### 좋은 답변 포인트

“online/offline은 힌트일 뿐 source of truth는 아니다.”

## 16. timeout, retry, backoff 전략은 각각 왜 필요한가요?

### 합격 답변 예시

timeout은 무한 대기를 막기 위해 필요하고, retry는 일시적 실패를 자동 복구하기 위해 필요하며, backoff는 재시도를 너무 공격적으로 반복해서 서버와 네트워크를 더 악화시키지 않기 위해 필요합니다. 즉 셋은 따로 쓰는 개념이 아니라 함께 설계해야 하는 경우가 많습니다. 실무에서는 timeout 없이 retry만 두면 사용자가 너무 오래 기다릴 수 있고, backoff 없이 retry만 두면 장애 상황에서 트래픽 폭탄이 될 수 있습니다.

### 상세 해설

- timeout
- 대기 상한선
- UX 보호
- retry
- 일시적 실패 복구
- backoff
- 재시도 간격 증가
- 서버 보호
- 네트워크 혼잡 완화

#### 프론트엔드 포인트

retry는 “많이 하면 좋다”가 아니다. 언제 멈출지까지 포함해야 한다.

## 17. API 요청이 중복 전송될 때 어떤 문제가 생기고, 어떻게 방지할 수 있나요?

### 합격 답변 예시

중복 요청은 중복 생성, 중복 결제, 상태 꼬임, 재고 차감 오류 같은 문제를 만들 수 있습니다. 프론트엔드에서는 버튼 중복 클릭 방지, 요청 중 disable, debounce/throttle, request id 관리 같은 방법으로 막을 수 있고, 서버 측에서는 idempotency key나 중복 처리 방지 로직으로 보완해야 합니다. 즉 이 문제는 프론트와 백엔드가 함께 막아야 합니다.

### 상세 해설

좋은 답변은 클라이언트만 말하지 않는다.

- 프론트에서 막는 법
- 버튼 disable
- loading flag
- debounce
- optimistic lock
- abort
- 서버에서 막는 법
- idempotency key
- unique constraint
- duplicate detection

## 18. 파일 업로드 기능에서 네트워크 실패를 고려한다면 어떤 UX를 설계하겠습니까?

### 합격 답변 예시

업로드 진행률, 취소, 재시도, 실패 파일 목록, 부분 성공/부분 실패 구분을 먼저 설계하겠습니다. 그리고 대용량 업로드라면 “파일 전송이 실패한 것인지”, “전송은 됐지만 서버 처리/검증이 실패한 것인지”를 분리해서 보여줘야 합니다. 사용자는 단순히 ‘업로드 안 됨’이 아니라, 지금 어느 단계에서 막혔는지 알아야 다음 행동을 선택할 수 있기 때문입니다.

### 상세 해설

#### 핵심 포인트

업로드는 보통 두 단계다.

- 전송
- 서버 처리/검증

이 둘을 안 나누면 UX가 매우 나빠진다.

- 좋은 UX
- 37/100개 업로드 완료
- 5개 형식 오류
- 2개 네트워크 실패, 재시도 가능

이런 식으로 보여줘야 한다.

## 19. iframe 로딩 지연이나 세션 만료가 겹치는 상황에서 어떤 원인 분리를 시도하겠습니까?

### 합격 답변 예시

먼저 네트워크 문제, 인증 문제, iframe 콘텐츠 자체 문제를 분리해서 보겠습니다. 세션 검증 API와 토큰 상태를 확인하고, iframe src 요청의 응답 상태, 쿠키 전달 여부, CORS 문제, 콘솔 에러를 함께 확인합니다. UI에서도 이 셋을 하나의 “로딩 실패”로 묶지 않고, 세션 만료인지, 외부 콘텐츠 응답 없음인지, 네트워크 지연인지 구분해서 보여주는 게 중요합니다.

### 상세 해설

이 질문은 장애 분석 사고를 본다.

#### 분리 순서 예시

세션 유효한가? iframe 요청 자체가 가는가? 응답 상태는 어떤가? 브라우저에서 막히는가? 네트워크가 불안정한가? 좋은 답변은 원인을 나눠서 보는 습관이 드러난다.

## 20. 실제 서비스에서 “사용자는 네트워크 문제와 웹앱 문제를 구분하지 못한다”는 말이 왜 중요한지 설명해보세요.

### 합격 답변 예시

사용자 입장에서는 내부 원인이 무엇이든 결국 “안 된다”로 느껴지기 때문입니다. 그래서 개발자는 내부적으로는 원인을 분리하더라도, 외부적으로는 사용자가 이해할 수 있는 메시지와 복구 경로를 제공해야 합니다. 즉 기술적으로는 네트워크 오류, 서버 오류, 인증 오류를 구분하더라도 UX는 “지금 무엇을 해야 하는지”를 알려줘야 하고, 이게 서비스 신뢰를 지키는 데 중요합니다.

### 상세 해설

이 질문은 사실 네트워크 질문이면서도 서비스 철학 질문이다. 좋은 답변은 보통 이렇게 끝난다. 그래서 프론트엔드 개발자는 오류를 단순히 표시하는 사람이 아니라, 사용자가 실패를 견딜 수 있게 만드는 사람이라고 생각합니다.

## 보완내용

### 왜 보완이 필요한가

기존 네트워크 20문항은 HTTP, TLS, 인증, CORS, 캐시, DNS, CDN, retry, timeout, 네트워크 불안정 UX까지 폭넓고 실무적이다. 다만 프론트엔드 면접에서 변별력이 더 커지는 지점은 브라우저 보안 정책, 최신 HTTP 진화, 쿠키 보안 속성, 캐시 계층 차이처럼 조금 더 구체적인 디테일이다.

아래 질문을 보강하면 “HTTP 상식” 수준의 답변과 실제 브라우저 네트워크 감각이 있는 답변을 더 잘 구분할 수 있다.

### 추가하면 좋은 질문

#### 1. HTTP/1.0, HTTP/1.1, HTTP/2, HTTP/3의 차이를 설명해보세요.

이 질문은 단순 상식이 아니라 실제 성능과 연결된다. 멀티플렉싱, 헤더 압축, 전송 계층 차이를 짧게라도 설명할 수 있으면 좋다.

##### 합격 답변 예시

HTTP/1.0은 요청마다 TCP 연결을 새로 열고 닫는 구조여서 매 요청에 연결 비용이 들었습니다. HTTP/1.1은 Keep-Alive를 기본으로 지원해 하나의 연결을 재사용할 수 있게 됐지만, 한 연결 안에서 응답 순서가 보장돼야 하므로 Head-of-Line Blocking 문제가 남아 있었습니다. HTTP/2는 하나의 TCP 연결 위에서 멀티플렉싱으로 여러 요청·응답을 동시에 처리하고, 헤더 압축(HPACK)으로 반복 전송 비용도 줄였습니다. HTTP/3는 전송 계층을 TCP에서 QUIC(UDP 기반)로 바꿔 TCP 수준의 Head-of-Line Blocking까지 제거하고, 연결 수립 시 TLS 핸드셰이크를 함께 처리해 RTT(Round Trip Time)를 더 줄인 구조입니다.

##### 상세 해설

면접에서 HTTP 버전 차이를 물을 때 핵심은 "왜 바뀌었는가"다. 각 버전은 이전 버전의 병목을 해결하기 위해 나왔다.

| 버전 | 연결 방식 | 핵심 개선 | 남은 한계 |
| --- | --- | --- | --- |
| HTTP/1.0 | 요청마다 새 TCP 연결 | 단순 구조 | 연결 비용이 큼 |
| HTTP/1.1 | Keep-Alive로 연결 재사용 | 연결 비용 절감 | 응답 순서 대기(HOL Blocking) |
| HTTP/2 | 하나의 TCP 위 멀티플렉싱 | 동시 전송, 헤더 압축, 서버 푸시 | TCP 자체의 패킷 손실 시 전체 지연 |
| HTTP/3 | QUIC(UDP 기반) | TCP HOL Blocking 제거, 빠른 연결 수립 | 아직 일부 환경에서 UDP 차단 가능 |

- 프론트엔드 연결
  - 번들 분할 전략이 HTTP/2 이후에는 달라질 수 있다 (여러 파일을 동시에 받을 수 있으므로 하나로 합칠 필요가 줄어듦)
  - HTTP/3 환경에서는 모바일처럼 네트워크 전환이 잦은 환경에서 연결 끊김이 줄어든다

#### 2. Keep-Alive와 connection reuse가 왜 중요한가요?

여러 요청이 반복되는 웹앱에서 매번 새 연결을 여는 비용이 크다는 감각을 보기 좋다. 네트워크 성능 최적화를 연결 수준까지 생각하는지 확인할 수 있다.

##### 합격 답변 예시

TCP 연결을 맺을 때마다 3-way handshake가 필요하고, HTTPS라면 TLS 핸드셰이크까지 추가로 들어갑니다. 이 비용은 요청 하나에는 작아 보여도, 웹 페이지 하나를 그리는 데 수십 개의 리소스 요청이 나가는 상황에서는 누적 지연이 커집니다. Keep-Alive는 한 번 맺은 TCP 연결을 여러 요청에 걸쳐 재사용하게 해주는 메커니즘이고, HTTP/1.1부터는 기본 동작입니다. 즉 connection reuse는 단순한 최적화가 아니라, 현대 웹 성능의 기본 전제 중 하나입니다.

##### 상세 해설

- 연결을 매번 새로 여는 비용
  - TCP 3-way handshake: 1 RTT
  - TLS 핸드셰이크: 추가 1~2 RTT
  - 요청 10개면 최대 10배 지연 가능
- Keep-Alive 효과
  - 한 번 맺은 연결을 유지하며 여러 요청을 순차적으로 보냄
  - handshake 비용을 첫 요청에서만 부담
- HTTP/2 이상에서의 발전
  - 멀티플렉싱으로 하나의 연결에서 동시에 여러 요청 처리
  - connection reuse의 효과가 더 극대화됨
- 프론트엔드 연결
  - 이미지, API, 폰트 등 여러 리소스를 빠르게 받으려면 연결 재사용이 필수
  - 도메인 샤딩(여러 도메인에 리소스 분산)은 HTTP/1.1 시대의 우회책이었고, HTTP/2에서는 오히려 불필요하거나 해로울 수 있다

#### 3. `SameSite`, `HttpOnly`, `Secure` 쿠키 속성을 설명해보세요.

기존 쿠키/세션/토큰 질문을 보안 관점으로 더 깊게 확장하는 질문이다. CSRF, XSS 완화와 연결해서 설명할 수 있으면 좋다.

##### 합격 답변 예시

`SameSite`는 쿠키가 cross-site 요청에 포함될 조건을 제한하는 속성으로, CSRF 공격을 완화하는 데 도움이 됩니다. `Strict`이면 같은 사이트 요청에만, `Lax`이면 top-level 네비게이션까지만, `None`이면 모든 cross-site 요청에도 전송되지만 반드시 `Secure`와 함께 써야 합니다. `HttpOnly`는 JavaScript에서 `document.cookie`로 접근할 수 없게 만들어 XSS로 인한 쿠키 탈취를 어렵게 합니다. `Secure`는 HTTPS 연결에서만 쿠키를 전송하게 해 평문 전송 중 탈취를 막습니다. 즉 세 속성은 각각 다른 공격 벡터를 줄이는 역할이고, 실무에서는 보통 함께 설정합니다.

##### 상세 해설

| 속성 | 역할 | 방어 대상 |
| --- | --- | --- |
| `SameSite` | cross-site 요청 시 쿠키 전송 조건 제한 | CSRF |
| `HttpOnly` | JavaScript에서 쿠키 접근 차단 | XSS를 통한 쿠키 탈취 |
| `Secure` | HTTPS에서만 쿠키 전송 | 평문 전송 중 도청 |

- SameSite 값 정리
  - `Strict`: 같은 사이트 요청에서만 전송. 가장 엄격하지만, 외부 링크 클릭 시 로그인이 풀린 것처럼 보일 수 있음
  - `Lax`: top-level 네비게이션(링크 클릭, GET 이동)에는 전송. 대부분의 브라우저 기본값
  - `None`: 모든 cross-site 요청에 전송. 반드시 `Secure` 필수. 서드파티 쿠키 필요 시 사용
- 프론트엔드 연결
  - 인증 쿠키는 `HttpOnly; Secure; SameSite=Lax` 조합이 가장 흔함
  - `SameSite=None`을 쓰는 경우는 iframe 안에서 인증이 필요하거나, 서드파티 연동 시

#### 4. `Idempotency-Key`가 왜 필요한지 설명해보세요.

기존 멱등성 질문의 실무 확장판이다. 중복 결제, 중복 생성, 재시도 안전성과 연결해서 설명하기 좋다.

##### 합격 답변 예시

`Idempotency-Key`는 클라이언트가 요청에 고유 키를 붙여서, 같은 키로 여러 번 요청이 와도 서버가 한 번만 처리하도록 보장하는 메커니즘입니다. POST처럼 원래 멱등적이지 않은 메서드에서 네트워크 재시도, 중복 클릭, 타임아웃 후 재전송이 발생했을 때 중복 결제나 중복 생성을 막을 수 있습니다. 즉 멱등성을 프로토콜 수준이 아니라 애플리케이션 수준에서 보장하기 위한 실무적 장치입니다.

##### 상세 해설

- 왜 필요한가
  - POST 요청은 기본적으로 멱등적이지 않음
  - 네트워크 타임아웃 후 클라이언트가 재시도하면 서버는 첫 요청이 성공했는지 알 수 없음
  - 결제, 주문, 포인트 차감 같은 경우 중복 처리가 치명적
- 동작 원리
  - 클라이언트가 UUID 등 고유 키를 `Idempotency-Key` 헤더에 담아 전송
  - 서버는 이 키를 저장하고, 같은 키가 다시 오면 이전 응답을 그대로 반환
  - 보통 일정 시간(예: 24시간) 후 키를 만료
- 프론트엔드 연결
  - 결제 버튼 클릭 시 프론트에서 UUID를 생성해 요청에 포함
  - retry 로직에서 같은 키를 유지해야 의미가 있음
  - 버튼 disable과 함께 쓰면 이중 방어가 됨

#### 5. SSE(Server-Sent Events)와 WebSocket의 차이를 설명해보세요.

기존 polling vs WebSocket 질문을 보강하는 용도다. 단방향 실시간 스트림과 양방향 연결의 선택 기준을 알고 있는지 확인하기 좋다.

##### 합격 답변 예시

SSE는 서버에서 클라이언트로의 단방향 실시간 스트림이고, WebSocket은 양방향 통신입니다. SSE는 일반 HTTP 위에서 동작하며, 자동 재연결과 이벤트 ID를 기본 지원하고, 브라우저 `EventSource` API로 간단히 사용할 수 있습니다. 반면 WebSocket은 별도 프로토콜로 업그레이드되며, 클라이언트도 서버에 자유롭게 메시지를 보낼 수 있어 채팅이나 게임처럼 양방향 이벤트가 빈번한 경우에 적합합니다. 즉 "서버가 일방적으로 알려주면 되는지, 양쪽이 주고받아야 하는지"가 선택 기준입니다.

##### 상세 해설

| 구분 | SSE | WebSocket |
| --- | --- | --- |
| 방향 | 서버 → 클라이언트 (단방향) | 양방향 |
| 프로토콜 | HTTP 위에서 동작 | WS 프로토콜로 업그레이드 |
| 재연결 | 브라우저가 자동 재연결 지원 | 직접 구현 필요 |
| 데이터 형식 | 텍스트(UTF-8) | 텍스트 + 바이너리 |
| 브라우저 API | `EventSource` | `WebSocket` |
| 적합한 경우 | 알림, 피드 업데이트, 주가 스트림 | 채팅, 게임, 협업 편집 |

- SSE가 유리한 경우
  - 서버에서 클라이언트로만 데이터를 보내면 되는 경우
  - HTTP 인프라(로드밸런서, 프록시)를 그대로 쓸 수 있어 운영이 단순
  - 자동 재연결, Last-Event-ID 기반 이어받기를 기본 지원
- WebSocket이 유리한 경우
  - 클라이언트도 서버에 실시간으로 메시지를 보내야 하는 경우
  - 바이너리 데이터 전송이 필요한 경우
  - 메시지 빈도가 매우 높고 양방향인 경우

#### 6. 브라우저 캐시와 서비스 워커 캐시의 차이를 설명해보세요.

프론트엔드 면접에서 네트워크, 성능, 오프라인 지원을 한 번에 연결할 수 있는 질문이다. 캐시 계층이 다르다는 점을 설명할 수 있어야 한다.

##### 합격 답변 예시

브라우저 캐시(HTTP 캐시)는 서버가 보내는 Cache-Control, ETag 같은 헤더에 따라 자동으로 동작하는 캐시이고, 서비스 워커 캐시는 개발자가 JavaScript로 직접 제어하는 프로그래밍 가능한 캐시입니다. 브라우저 캐시는 설정이 간단하고 별도 코드가 필요 없지만, 세밀한 전략 변경이 어렵습니다. 서비스 워커 캐시는 요청을 가로채서 캐시 우선, 네트워크 우선, stale-while-revalidate 같은 전략을 자유롭게 구현할 수 있고, 오프라인 지원까지 가능합니다. 즉 둘은 경쟁이 아니라 계층이 다르며, 요청이 서비스 워커를 먼저 거치고 그 뒤에 브라우저 HTTP 캐시를 거치는 구조입니다.

##### 상세 해설

| 구분 | 브라우저 캐시 (HTTP 캐시) | 서비스 워커 캐시 |
| --- | --- | --- |
| 제어 방식 | 서버 응답 헤더 (Cache-Control 등) | JavaScript 코드로 직접 제어 |
| 오프라인 지원 | 제한적 (캐시된 것만) | 가능 (fallback 페이지 등 구현 가능) |
| 전략 유연성 | 헤더 기반 고정 정책 | cache-first, network-first, stale-while-revalidate 등 자유 |
| 캐시 무효화 | 헤더 변경, URL 변경 | 코드로 직접 삭제/갱신 가능 |
| 요청 순서 | 서비스 워커 뒤에 위치 | 네트워크 요청 전에 먼저 확인 |

- 캐시 계층 순서
  - 요청 발생 → 서비스 워커가 먼저 가로챔 → 서비스 워커 캐시 확인 → 필요하면 네트워크로 전달 → 브라우저 HTTP 캐시 확인 → 서버 요청
- 프론트엔드 연결
  - PWA에서 오프라인 경험을 만들려면 서비스 워커 캐시가 필수
  - 정적 자산(JS, CSS, 이미지)은 브라우저 캐시로도 충분한 경우가 많음
  - API 응답을 캐싱하거나 오프라인 fallback을 보여주려면 서비스 워커 캐시가 필요
  - 두 캐시를 함께 이해해야 "왜 새 배포가 반영이 안 되는지" 같은 문제를 빠르게 진단할 수 있다

### 이 섹션 보완 핵심

네트워크 섹션은 이미 강하다. 여기에 HTTP 진화, 연결 재사용, 쿠키 보안 속성, `Idempotency-Key`, SSE, 서비스 워커 캐시까지 더하면 브라우저 네트워크 감각이 있는 답변자를 더 잘 가를 수 있다.
