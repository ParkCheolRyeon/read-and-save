# 챕터 3. 비동기 실행 모델에서 race condition까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **비동기 문법을 외우는 것이 아니라, 브라우저가 일을 어떤 순서로 처리하는지 이해해서 실무 버그를 설명할 수 있어야 한다.**

---

## 21. 동기와 비동기의 차이는 무엇인가

자바스크립트 면접에서 비동기 질문은 거의 반드시 나온다.
그런데 여기서 많은 사람이 “오래 걸리면 비동기” 정도로 설명해서 흐려진다.

### 핵심 개념

- 동기는 현재 작업이 끝날 때까지 다음 흐름으로 넘어가지 않는 방식이다.
- 비동기는 작업 완료를 기다리는 동안에도 다른 흐름을 계속 진행할 수 있게 설계된 방식이다.
- 핵심은 속도 차이가 아니라 **제어 흐름 차이**다.

### 왜 이게 중요한가

이걸 정확히 잡아야 다음 질문들이 자연스럽게 이어진다.

- Promise는 왜 필요한가
- `async/await`는 무엇을 감추는가
- 이벤트 루프는 어떤 역할을 하는가
- React에서 비동기 요청 버그는 왜 생기는가

### 자주 하는 오해

- 비동기는 무조건 빠르다 → 틀림
- 오래 걸리는 작업은 다 비동기다 → 틀림
- `async/await`를 쓰면 동기 코드가 된다 → 틀림. 문법이 동기처럼 보일 뿐이다

### 예시

```ts
console.log(“start”);

setTimeout(() => {
  console.log(“timeout”);
}, 0);

console.log(“end”);
```

이 코드는 시간이 오래 걸려서 비동기인 게 아니다.
타이머 콜백이 현재 실행 흐름이 끝난 뒤 나중에 처리되도록 예약되기 때문에 비동기다.

### 실무 연결

프론트엔드에서 사용자 입력, 네트워크 요청, 타이머, 애니메이션 이벤트, 파일 업로드는 대부분 비동기 흐름과 연결된다.
그래서 비동기를 이해하지 못하면 “가끔 늦게 뜨는 버그”, “옛날 데이터가 덮어쓰는 버그”, “로딩 상태가 꼬이는 버그”를 설명하기 어렵다.

### 여기서 다음 개념으로 어떻게 넘어가나

동기/비동기를 알았다면, 다음엔 자주 같이 헷갈리는 블로킹/논블로킹으로 넘어가야 한다.

### 면접 압축 답변

동기는 현재 작업이 끝날 때까지 다음 흐름으로 넘어가지 않는 방식이고, 비동기는 작업 완료를 기다리는 동안 다른 흐름을 계속 진행할 수 있게 한 방식입니다. 핵심은 속도가 아니라 제어 흐름 차이입니다.

## 22. 블로킹과 논블로킹은 왜 동기/비동기와 별개인가

동기/비동기와 블로킹/논블로킹은 같이 나오지만 같은 말은 아니다.

### 핵심 개념

- 블로킹은 호출한 쪽이 결과를 받을 때까지 제어권을 돌려받지 못하는가에 대한 이야기다.
- 논블로킹은 호출 후 바로 제어권을 돌려받을 수 있는가에 대한 이야기다.

즉:

- 동기/비동기는 흐름 설계
- 블로킹/논블로킹은 제어권 반환 시점

에 더 가깝다.

### 왜 이게 중요한가

이걸 구분해야 브라우저와 네트워크, 파일 처리, 사용자 입력, 긴 계산을 정확히 설명할 수 있다.

### 자주 하는 오해

- 동기면 무조건 블로킹이고 비동기면 무조건 논블로킹이다 → 단순화가 심하다
- async 함수는 자동으로 논블로킹이다 → 문법만 보고 그렇게 말하면 부정확하다
- 네트워크 요청은 비동기니까 UI가 절대 안 멈춘다 → 틀림. 메인 스레드를 다른 코드가 막고 있으면 UI는 충분히 멈출 수 있다

### 실무 연결

브라우저에서 가장 중요한 건 메인 스레드를 막느냐다.
긴 계산, 무거운 JSON 파싱, 큰 리스트 렌더링, 비싼 애니메이션 계산이 있으면 사용자는 버튼도 못 누르고 스크롤도 버벅이는 체감을 하게 된다.

즉 브라우저 관점에서 블로킹은 단순 CS 개념이 아니라 사용자 경험 문제다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 실행 흐름과 제어권 반환을 구분했다.
그 다음엔 브라우저와 JS 엔진이 실제로 이걸 어떻게 처리하는지, 즉 이벤트 루프로 넘어가야 한다.

### 면접 압축 답변

동기/비동기는 작업 흐름의 설계 방식이고, 블로킹/논블로킹은 호출 후 제어권이 언제 돌아오는지에 대한 개념입니다. 둘은 관련 있지만 완전히 같은 축은 아닙니다.

## 23. 이벤트 루프는 왜 자바스크립트 비동기 이해의 중심인가

자바스크립트는 보통 단일 스레드 실행 모델로 설명된다.
그런데 실제 브라우저에서는 타이머도 되고, 네트워크 요청도 되고, 클릭 이벤트도 처리되고, Promise도 순서 있게 동작한다.

이걸 연결해주는 개념이 이벤트 루프다.

### 핵심 개념

이벤트 루프를 이해할 때는 아래 4개를 같이 봐야 한다.

- Call Stack
- Web APIs 또는 브라우저 외부 처리 영역
- Task Queue
- Microtask Queue

### 동작 흐름

- 현재 실행 중인 코드가 Call Stack에서 실행된다.
- setTimeout, DOM 이벤트, 네트워크 콜백 같은 작업은 브라우저가 따로 관리한다.
- 준비가 끝난 콜백은 Queue로 들어간다.
- Call Stack이 비면 이벤트 루프가 Queue에서 작업을 꺼내 Stack에 넣는다.
- 이때 Microtask가 Task보다 우선 처리된다.

### 왜 이게 중요한가

이걸 이해하면 아래 질문이 한 번에 연결된다.

- 왜 setTimeout(..., 0)이 바로 실행되지 않나
- 왜 Promise then이 타이머보다 먼저 실행되나
- 왜 무한 마이크로태스크가 UI를 굳게 만들 수 있나
- React 업데이트 타이밍을 왜 “즉시 값 변경”으로 이해하면 안 되나

### 예시

```ts
console.log(“1”);

setTimeout(() => {
  console.log(“2”);
}, 0);

Promise.resolve().then(() => {
  console.log(“3”);
});

console.log(“4”);
```

실행 결과는 보통 아래와 같다.

```
1
4
3
2
```

이유는:

- 1, 4는 현재 Call Stack에서 즉시 실행
- Promise then은 Microtask Queue
- setTimeout 콜백은 Task Queue
- Stack이 비면 Microtask가 먼저 비워지고, 그 다음 Task가 실행되기 때문이다

### 자주 하는 오해

- setTimeout(..., 0)은 즉시 실행된다 → 틀림
- Promise는 그냥 더 빠른 타이머다 → 틀림
- 이벤트 루프는 JS 엔진 내부에서만 전부 처리한다 → 브라우저와의 협력이 중요하다

### 실무 연결

이벤트 루프 감각이 있으면:

- 로딩 표시가 왜 늦게 바뀌는지
- Promise 체인이 왜 예상과 다르게 먼저 실행되는지
- 무거운 동기 코드가 왜 비동기 콜백도 못 돌게 만드는지

설명할 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 실행 순서의 기반을 봤다.
그 다음엔 브라우저 비동기를 더 다루기 쉽게 만든 추상화인 Promise로 넘어가야 한다.

### 면접 압축 답변

이벤트 루프는 Call Stack, 브라우저 비동기 처리, Task Queue, Microtask Queue를 조율해 자바스크립트 비동기 실행 순서를 결정하는 구조입니다. 특히 Promise의 then 같은 Microtask가 일반 Task보다 먼저 처리된다는 점이 중요합니다.

## 24. Promise는 왜 필요한가

이벤트 루프를 알았다면 이제 다음 질문은 자연스럽다.

“그럼 비동기 작업을 그냥 콜백으로만 다루면 안 되나?”

초기 자바스크립트는 콜백 중심이었다.
그런데 콜백만으로 비동기를 관리하면 읽기 어렵고, 에러 흐름이 분산되고, 여러 비동기 조합이 복잡해진다.
Promise는 이 문제를 줄이기 위해 등장한 추상화다.

### 핵심 개념

Promise는 미래에 완료될 값을 나타내는 객체다.
상태는 크게 아래처럼 생각하면 된다.

- pending
- fulfilled
- rejected

### 왜 이게 중요한가

Promise가 중요한 이유는 단순히 문법이 예뻐서가 아니다.

- 성공/실패 흐름을 구조화할 수 있고
- 체이닝이 가능하며
- 에러 전파 방식이 일관되고
- 여러 비동기를 조합하기 쉬워진다

### 콜백과 비교해서 이해하기

콜백 기반 코드는 중첩이 깊어지기 쉽다.

```ts
getUser(userId, (user) => {
  getPosts(user.id, (posts) => {
    getComments(posts[0].id, (comments) => {
      console.log(comments);
    });
  });
});
```

Promise를 쓰면 흐름을 더 평평하게 정리할 수 있다.

```ts
getUser(userId)
  .then((user) => getPosts(user.id))
  .then((posts) => getComments(posts[0].id))
  .then((comments) => {
    console.log(comments);
  })
  .catch((error) => {
    console.error(error);
  });
```

### 자주 하는 오해

- Promise를 만들면 즉시 실행되지 않는다 → 틀림. 보통 executor는 생성 시점에 바로 실행된다
- Promise는 비동기를 만든다 → Promise는 비동기 결과를 다루는 인터페이스이지, 비동기 작업 자체를 “자동 생성”하는 마법은 아니다
- then을 여러 번 붙이면 무조건 순차 실행이다 → 어떤 Promise를 반환하는지에 따라 흐름이 달라진다

### 실무 연결

Promise를 제대로 이해하면:

- API 요청 체인
- 순차/병렬 요청
- 에러 전파
- 재시도 로직
- 로딩 상태 관리

를 훨씬 더 정확하게 설계할 수 있다.

### 여기서 다음 개념으로 어떻게 넘어가나

Promise를 알았다면, 이제 많은 사람이 더 자주 쓰는 문법인 async/await가 Promise와 어떤 관계인지를 봐야 한다.

### 면접 압축 답변

Promise는 미래에 완료될 값을 나타내는 객체로, 비동기 작업의 성공과 실패 흐름을 구조적으로 다루기 위한 추상화입니다. 콜백보다 체이닝과 에러 전파가 일관돼서 복잡한 비동기 흐름을 다루기 쉬워집니다.

## 25. async/await는 Promise와 어떤 관계인가

많은 사람이 async/await를 Promise와 별개의 기술처럼 느낀다.
하지만 실제로는 Promise 위에서 더 읽기 쉬운 문법을 제공하는 계층이다.

### 핵심 개념

- async 함수는 항상 Promise를 반환한다
- await는 Promise가 해결될 때까지 해당 함수 내부 흐름을 일시 중단한 것처럼 보이게 만든다
- 하지만 이것이 전체 자바스크립트 실행을 멈추는 것은 아니다

### 왜 이게 중요한가

이걸 이해해야:

- await를 쓸 수 있는 위치
- 병렬 요청과 순차 요청 차이
- try/catch로 Promise 에러를 잡는 방식

을 정확히 설명할 수 있다.

### 예시

```ts
async function fetchData() {
  const response = await fetch("/api/data");
  const data = await response.json();
  return data;
}
```

겉보기엔 동기 코드처럼 읽히지만, 내부적으로는 Promise 흐름 위에서 동작한다.

### 자주 하는 오해

- await를 쓰면 비동기가 동기로 바뀐다 → 틀림
- await는 어디서나 쓸 수 있다 → 기본적으로 async 함수 안 또는 일부 특별한 환경에서만 가능
- await를 많이 쓰면 무조건 코드가 더 좋다 → 병렬로 처리할 수 있는 작업까지 순차로 묶으면 오히려 느려질 수 있다

### 병렬과 순차를 구분해야 하는 이유

```ts
const user = await fetchUser();
const posts = await fetchPosts();
```

이건 순차 실행처럼 보인다.
두 요청이 서로 독립적이면 병렬이 더 적절할 수 있다.

```ts
const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
```

### 실무 연결

프론트엔드에서 async/await를 많이 쓰지만,
무조건 await를 줄줄이 쓰면:

- 독립 요청이 불필요하게 순차 실행되고
- 로딩 시간이 길어지고
- 사용자 체감이 나빠질 수 있다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 자연스럽게 다음 질문이 나온다.

그렇다면 여러 Promise를 한꺼번에 다룰 때는 어떤 조합 도구를 써야 하나?

### 면접 압축 답변

async/await는 Promise를 더 읽기 쉽게 다루는 문법입니다. async 함수는 Promise를 반환하고, await는 해당 함수 내부에서 Promise 결과를 기다리는 것처럼 보이게 하지만 전체 자바스크립트 실행을 멈추는 것은 아닙니다.

## 26. Promise.all, allSettled, race, any는 어떻게 구분하나

비동기를 실무에서 다루다 보면 “여러 요청을 어떻게 조합할 것인가”가 중요해진다.
이때 Promise 조합 함수를 목적에 맞게 구분해야 한다.

### 핵심 개념

**Promise.all**

- 모든 Promise가 성공해야 성공
- 하나라도 실패하면 전체 실패
- 모두 성공해야 의미가 있는 작업에 적합

**Promise.allSettled**

- 전부 끝날 때까지 기다림
- 성공/실패를 각각 결과로 받음
- 부분 성공/부분 실패를 같이 보고 싶을 때 적합

**Promise.race**

- 가장 먼저 끝난 Promise의 결과로 결정
- 성공이든 실패든 가장 먼저 끝난 쪽이 이김
- timeout 경쟁 등에 활용 가능

**Promise.any**

- 가장 먼저 성공한 Promise로 결정
- 모두 실패하면 실패
- 여러 후보 중 하나만 성공하면 되는 경우에 적합

### 왜 이게 중요한가

이걸 구분 못 하면:

- 하나만 실패해도 전체가 깨질지
- 부분 성공을 살릴지
- timeout 처리에 무엇을 써야 할지

판단하기 어렵다.

### 예시

```ts
const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
```

둘 다 있어야 화면을 완성할 수 있다면 all이 맞다.

```ts
const results = await Promise.allSettled([
  fetchBanner(),
  fetchNotice(),
  fetchRecommendation(),
]);
```

배너 하나 실패해도 공지와 추천은 보여주고 싶다면 allSettled가 더 어울린다.

### 자주 하는 오해

- Promise.all이 항상 제일 좋은 기본값이다 → 틀림
- race는 가장 먼저 성공한 걸 고른다 → 정확히는 가장 먼저 끝난 것이다
- allSettled는 실패를 숨긴다 → 틀림. 실패를 포함해서 결과를 돌려준다

### 실무 연결

실제 서비스에서는:

- 핵심 데이터는 all
- 부가 위젯은 allSettled
- timeout 제어는 race

처럼 역할을 나누는 경우가 많다.

### 여기서 다음 개념으로 어떻게 넘어가나

여기까지는 문법 조합이다.
그런데 실제 서비스에서는 여러 요청이 겹칠 때 응답 순서 때문에 버그가 생길 수 있다.
그게 race condition이다.

### 면접 압축 답변

Promise.all은 모두 성공해야 성공하고, allSettled는 전부 끝날 때까지 기다린 뒤 각 결과를 모두 돌려줍니다. race는 가장 먼저 끝난 결과로 결정되고, any는 가장 먼저 성공한 결과를 사용합니다. 따라서 요구사항에 따라 조합 방식을 다르게 선택해야 합니다.

## 27. Race Condition은 프론트엔드에서 어떻게 생기나

비동기 문법을 아는 것과, 비동기 버그를 설명할 수 있는 것은 다르다.
실무 면접에서 강한 답변은 바로 여기서 나온다.

### 핵심 개념

Race Condition은 여러 작업이 같은 상태나 자원에 접근할 때, 실행 순서에 따라 결과가 달라지는 문제다.

### 대표적인 프론트엔드 예시

검색 입력창을 생각해보자.

- 사용자가 a를 입력한다
- 요청 A가 나간다
- 곧바로 ab를 입력한다
- 요청 B가 나간다
- 요청 B가 먼저 도착한다
- 화면은 ab 결과를 보여준다
- 그런데 요청 A가 나중에 도착해서 화면을 다시 덮어쓴다

이러면 사용자는 마지막 입력이 ab였는데 화면엔 a 결과가 보일 수 있다.

### 왜 이게 중요한가

이 문제를 모르면:

- “가끔 옛날 데이터가 보여요”
- “탭 전환하다 보면 값이 바뀌어요”
- “로딩이 끝났는데 다시 이전 상태가 나와요”

같은 버그를 설명할 수 없다.

### 자주 하는 오해

- race condition은 멀티스레드에서만 생긴다 → 틀림. 프론트엔드 비동기 흐름에서도 충분히 생긴다
- 요청은 나중에 보낸 게 항상 나중에 도착한다 → 틀림. 네트워크는 그렇게 보장하지 않는다
- 최신 state만 보면 자동 해결된다 → 조건에 따라 여전히 오래된 응답이 화면을 덮을 수 있다

### 실무 연결

대표적인 발생 지점:

- 자동완성 검색
- 탭/필터 전환
- 무한 스크롤 페이지 전환
- 버튼 중복 클릭
- optimistic update 이후 rollback 타이밍 꼬임

### 여기서 다음 개념으로 어떻게 넘어가나

Race Condition을 알았다면 다음 질문은 당연하다.

그럼 이전 요청을 취소하거나, 최신 요청만 반영하려면 어떻게 해야 하나?

여기서 AbortController와 최신 요청 보장 전략이 등장한다.

### 면접 압축 답변

Race Condition은 여러 비동기 작업의 완료 순서가 보장되지 않아 결과가 엉뚱하게 덮어써지는 문제입니다. 프론트엔드에서는 검색 자동완성이나 탭 전환처럼 요청이 연속으로 발생하는 UI에서 자주 생기며, 이전 요청 취소나 최신 요청만 반영하는 정책이 필요합니다.

---

## 28. AbortController는 왜 필요한가

Race Condition을 이해했다면, 이제 이전 요청을 어떻게 다룰지 생각해야 한다.
그중 가장 대표적인 도구가 AbortController다.

### 핵심 개념

AbortController는 진행 중인 비동기 작업, 특히 fetch 요청 같은 작업을 취소 신호로 제어할 수 있게 해준다.

### 예시

```ts
const controller = new AbortController();

fetch(“/api/search?q=abc”, {
  signal: controller.signal,
})
  .then((res) => res.json())
  .then((data) => {
    console.log(data);
  })
  .catch((error) => {
    if (error.name === “AbortError”) {
      console.log(“요청 취소됨”);
    }
  });

controller.abort();
```

### 왜 이게 중요한가

AbortController가 중요한 이유는 단순히 네트워크 낭비를 줄이는 것만이 아니다.

- 오래된 요청이 의미 없이 끝까지 가지 않게 하고
- 최신 요청 중심으로 UI를 유지하고
- 언마운트 후 setState 같은 위험한 흐름을 줄이고
- 불필요한 에러 처리 분기를 줄이는 데 도움이 된다

### 자주 하는 오해

- abort하면 서버 작업도 무조건 완전히 중단된다 → 클라이언트 요청 취소와 서버 내부 처리 중단은 별개일 수 있다
- AbortController만 쓰면 race condition이 전부 해결된다 → 최신 요청 식별이나 상태 반영 정책도 같이 필요할 수 있다
- 취소된 요청도 일반 에러와 똑같이 처리하면 된다 → 보통은 별도로 구분해서 처리하는 게 낫다

### React와 연결해서 보기

컴포넌트가 언마운트되거나 검색 조건이 바뀔 때, 이전 요청을 취소하는 패턴은 useEffect cleanup과 잘 연결된다.

```ts
useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/search?q=${keyword}`, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => setResult(data))
    .catch((error) => {
      if (error.name !== “AbortError”) {
        console.error(error);
      }
    });

  return () => {
    controller.abort();
  };
}, [keyword]);
```

### 여기서 다음 개념으로 어떻게 넘어가나

AbortController는 “취소”에 초점이 있다.
그런데 네트워크는 취소만 필요한 게 아니라 timeout, retry, backoff 같은 실패 복구 전략도 필요하다.

### 면접 압축 답변

AbortController는 진행 중인 요청에 취소 신호를 전달해 이전 요청을 중단하거나, 컴포넌트 언마운트 후 불필요한 처리 흐름을 줄이는 데 유용합니다. 특히 검색 자동완성처럼 요청이 연속으로 발생하는 UI에서 race condition 완화에 도움이 됩니다.

---

## 29. timeout, retry, backoff는 왜 같이 생각해야 하나

실제 네트워크 환경은 항상 안정적이지 않다.
응답이 느릴 수도 있고, 일시적으로 실패할 수도 있고, 서버가 순간적으로 과부하일 수도 있다.

이때 중요한 건 “실패하면 그냥 에러”가 아니라 어떤 실패를 어떻게 복구할 것인가다.

### 핵심 개념

**timeout**

- 일정 시간 이상 응답이 없으면 기다리기를 중단하는 기준이다.

**retry**

- 실패한 요청을 다시 시도하는 전략이다.

**backoff**

- 재시도 간격을 점점 늘리는 방식이다.
- 대표적으로 exponential backoff가 많이 언급된다.

### 왜 이게 중요한가

이걸 구분해야:

- 사용자는 계속 무한 로딩만 보지 않게 되고
- 일시적인 장애는 자동 복구할 수 있고
- 서버에 과도한 재시도를 몰아주지 않게 된다

### 자주 하는 오해

- 실패하면 무조건 재시도하면 된다 → 틀림. 멱등하지 않은 요청은 위험할 수 있다
- timeout이 길수록 사용자 경험이 좋다 → 틀림. 너무 길면 체감상 앱이 멈춘 것처럼 느껴질 수 있다
- retry는 프론트에서만 고민하면 된다 → 서버의 멱등성, rate limiting, 에러 정책과 같이 봐야 한다

### 실무 연결

예를 들어:

- GET 요청은 재시도에 더 적합한 편이다
- 결제 같은 POST 요청은 멱등성 키 없이 함부로 재시도하면 위험할 수 있다
- 업로드는 실패 원인에 따라 재시도 정책을 더 신중하게 설계해야 한다

### 간단한 timeout 경쟁 예시

```ts
function timeoutPromise(ms: number) {
  return new Promise((_, reject) => {
    setTimeout(() => reject(new Error("timeout")), ms);
  });
}

async function fetchWithTimeout(url: string) {
  return Promise.race([fetch(url), timeoutPromise(5000)]);
}
```

### 여기서 다음 개념으로 어떻게 넘어가나

이제 요청 실패 복구 전략까지 봤다.
그 다음엔 연속 입력이나 중복 클릭처럼 요청 자체를 덜 만들고 최신 것만 유지하는 전략으로 넘어가야 한다.

### 면접 압축 답변

timeout은 오래 기다리지 않기 위한 기준이고, retry는 실패한 요청을 다시 시도하는 전략이며, backoff는 재시도 간격을 점점 늘려 과부하를 줄이는 방식입니다. 다만 모든 요청을 무조건 재시도하면 안 되고, 멱등성과 사용자 경험을 함께 고려해야 합니다.

## 30. 중복 요청 방지와 최신 요청만 반영하는 전략은 어떻게 설계하나

이제 비동기 처리의 마지막 실무 감각으로 넘어간다.
좋은 프론트엔드는 실패를 잘 처리하는 것뿐 아니라, 애초에 불필요한 요청을 덜 만들고, 의미 있는 응답만 반영한다.

### 핵심 개념

중복 요청 방지 전략은 보통 아래처럼 나눠볼 수 있다.

- 입력 빈도 줄이기: debounce, throttle
- 이전 요청 취소: AbortController
- 최신 요청만 반영: request id 비교, 시퀀스 번호 비교
- 같은 키의 요청 재사용: 캐시, dedupe
- 중복 클릭 방지: 버튼 disable, pending 상태 관리

### 왜 이게 중요한가

실제 서비스에서 문제는 “요청이 실패했느냐”만이 아니라,
너무 많이 나가거나, 순서가 꼬이거나, UI가 여러 번 반응하는 데서 자주 생긴다.

### 최신 요청만 반영하는 예시

```ts
let requestId = 0;

async function search(keyword: string) {
  const currentId = ++requestId;
  const response = await fetch(`/api/search?q=${keyword}`);
  const data = await response.json();

  if (currentId !== requestId) return;

  renderResult(data);
}
```

이 방식은 취소하지 못한 요청이 남아 있어도, 가장 마지막 요청만 화면에 반영하게 만들 수 있다.

### 자주 하는 오해

- AbortController만 쓰면 충분하다 → 상황에 따라 최신 요청만 반영하는 체크도 필요하다
- debounce만 걸면 race condition이 사라진다 → 요청 횟수는 줄지만 순서 문제는 남을 수 있다
- 서버가 알아서 막아줄 것이다 → 프론트에서도 충분히 방어해야 사용자 경험이 좋아진다

### 실무 연결

대표적인 사용처:

- 검색 자동완성
- 필터 조합 변경
- 무한 스크롤
- CSV 업로드 중 중복 제출 방지
- 저장 버튼 연타 방지

### 여기서 다음 개념으로 어떻게 넘어가나

이제 비동기 흐름의 큰 축을 봤다.
다음 챕터에서는 자연스럽게 타입 시스템과 런타임 검증으로 넘어간다.

### 다음 흐름

**TypeScript를 왜 쓰는가 → any/unknown/never → narrowing → generic → utility types → 런타임 검증으로서의 Zod**

즉 이번 챕터가 “실행 순서와 비동기 제어”였다면, 다음 챕터는 “정적 계약과 런타임 경계”가 된다.

### 면접 압축 답변

중복 요청 방지는 debounce, throttle, 버튼 disable, 캐시 재사용 같은 방식으로 할 수 있고, race condition 대응은 AbortController나 최신 요청 ID 비교로 할 수 있습니다. 핵심은 요청을 덜 만들고, 만들어진 응답 중 의미 있는 것만 반영하는 것입니다.

## 31. 디바운스와 스로틀은 왜 비동기 제어와 같이 봐야 하나

많은 사람이 디바운스와 스로틀을 단순 유틸 함수로 외운다.
그런데 실무에서는 이 둘도 비동기 요청 전략의 일부다.

### 핵심 개념

**디바운스**

- 연속 입력이 끝난 뒤 일정 시간이 지나면 한 번 실행한다.

**스로틀**

- 일정 시간 동안 최대 한 번만 실행되게 제한한다.

### 왜 이게 중요한가

이 둘은 단순 성능 유틸이 아니라:

- 검색 요청 수를 줄이고
- 서버 부하를 낮추고
- 사용자 입력 도중 의미 없는 요청을 줄이는

흐름 제어 전략이다.

### 자주 하는 오해

- debounce를 쓰면 race condition도 끝난다 → 틀림. 요청 횟수는 줄지만 순서 문제는 별개다
- throttle은 무조건 스크롤에만 쓴다 → 틀림. 상황에 따라 다양한 이벤트에 적용 가능하다
- 둘 중 하나만 외우면 된다 → 틀림. 사용자 경험 목적이 다르다

### 실무 연결

- 검색창 자동완성 → debounce
- 스크롤/리사이즈 이벤트 → throttle
- 입력과 네트워크 요청을 함께 다룰 때는 debounce + AbortController 조합도 자주 나온다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 요청 빈도 자체를 조절하는 방법까지 봤다.
다음 챕터에서는 “요청/응답 데이터의 형태를 어떻게 믿고 다룰 것인가”라는 질문으로 넘어간다.

### 면접 압축 답변

디바운스는 연속 입력이 멈춘 뒤 한 번 실행하게 하고, 스로틀은 일정 시간마다 최대 한 번만 실행되게 합니다. 프론트엔드에서는 이 둘이 단순 유틸이 아니라 요청 빈도 제어와 사용자 경험 설계의 일부입니다.

## 32. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래가 한 덩어리로 보여야 한다.

- 비동기는 속도 문제가 아니라 흐름 제어 문제다
- 블로킹은 사용자 경험과 직접 연결된다
- 이벤트 루프는 실행 순서를 설명하는 기반이다
- Promise와 async/await는 비동기 흐름을 구조적으로 다루기 위한 도구다
- 여러 요청이 겹치면 race condition이 생긴다
- AbortController, request id 비교, debounce, retry/backoff는 그에 대한 실무 대응이다

즉,
이번 챕터는 단순 문법 챕터가 아니라
**”실행 순서가 꼬일 때 어떤 버그가 생기고, 어떻게 통제할 것인가”**를 배우는 챕터다.

## 이 챕터를 읽은 뒤 자연스럽게 이어질 다음 질문들

1. TypeScript를 왜 쓰는가?
2. any, unknown, never는 어떤 철학 차이가 있나?
3. narrowing은 왜 런타임 조건과 연결되나?
4. generic은 왜 재사용성과 타입 안전성을 동시에 주나?
5. TypeScript가 있어도 왜 Zod 같은 런타임 검증이 필요하나?

## 사용법

이 챕터는 아래 순서로 보면 좋다.

1. 동기/비동기, 블로킹/논블로킹으로 개념 축을 먼저 잡는다.
2. 이벤트 루프로 실제 실행 순서를 이해한다.
3. Promise와 async/await로 문법 계층을 정리한다.
4. race condition, AbortController, retry/backoff로 실무 문제를 연결한다.
5. 마지막에 면접 압축 답변만 따로 말로 꺼내 본다.

이렇게 보면 단순 문법 암기가 아니라, 비동기 흐름 전체가 하나의 구조로 잡힌다.
