# 챕터 3. 비동기 실행 모델에서 race condition까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **비동기 문법을 외우는 것이 아니라, 브라우저가 일을 어떤 순서로 처리하는지 이해해서 실무 버그를 설명할 수 있어야 한다.**

---

## 21. 동기와 비동기의 차이는 무엇인가

자바스크립트 면접에서 비동기 질문은 거의 반드시 나온다.
그런데 여기서 많은 사람이 “오래 걸리면 비동기” 정도로 설명해서 흐려진다.

### 핵심 개념

- 동기는 현재 작업이 끝날 때까지 다음 흐름으로 넘어가지 않는 방식이다.
- 비동기는 작업 완료를 기다리는 동안에도 다른 흐름을 계속 진행할 수 있게 설계된 방식이다.
- 핵심은 속도 차이가 아니라 **제어 흐름 차이**다.

### 왜 이게 중요한가

이걸 정확히 잡아야 다음 질문들이 자연스럽게 이어진다.

- Promise는 왜 필요한가
- `async/await`는 무엇을 감추는가
- 이벤트 루프는 어떤 역할을 하는가
- React에서 비동기 요청 버그는 왜 생기는가

### 자주 하는 오해

- 비동기는 무조건 빠르다 → 틀림
- 오래 걸리는 작업은 다 비동기다 → 틀림
- `async/await`를 쓰면 동기 코드가 된다 → 틀림. 문법이 동기처럼 보일 뿐이다

### 예시

```ts
console.log(“start”);

setTimeout(() => {
  console.log(“timeout”);
}, 0);

console.log(“end”);
```

출력 순서는 `start` → `end` → `timeout`이다.
이 코드는 시간이 오래 걸려서 비동기인 게 아니다.
타이머 콜백이 현재 실행 흐름이 끝난 뒤 나중에 처리되도록 예약되기 때문에 비동기다.

### 실무 연결

프론트엔드에서 사용자 입력, 네트워크 요청, 타이머, 애니메이션 이벤트, 파일 업로드는 대부분 비동기 흐름과 연결된다.
그래서 비동기를 이해하지 못하면 “가끔 늦게 뜨는 버그”, “옛날 데이터가 덮어쓰는 버그”, “로딩 상태가 꼬이는 버그”를 설명하기 어렵다.

> 동기/비동기를 알았다면, 다음엔 자주 같이 헷갈리는 블로킹/논블로킹과의 차이를 짚어야 한다.

### 면접 압축 답변

동기는 현재 작업이 끝날 때까지 다음 흐름으로 넘어가지 않는 방식이고, 비동기는 작업 완료를 기다리는 동안 다른 흐름을 계속 진행할 수 있게 한 방식입니다. 핵심은 속도가 아니라 제어 흐름 차이입니다.

## 22. 블로킹과 논블로킹은 왜 동기/비동기와 별개인가

동기/비동기와 블로킹/논블로킹은 같이 나오지만 같은 말은 아니다.

### 핵심 개념

- 블로킹은 호출한 쪽이 결과를 받을 때까지 제어권을 돌려받지 못하는가에 대한 이야기다.
- 논블로킹은 호출 후 바로 제어권을 돌려받을 수 있는가에 대한 이야기다.

즉:

- 동기/비동기는 흐름 설계
- 블로킹/논블로킹은 제어권 반환 시점

에 더 가깝다.

### 왜 이게 중요한가

이걸 구분해야 브라우저와 네트워크, 파일 처리, 사용자 입력, 긴 계산을 정확히 설명할 수 있다.

### 예시: 메인 스레드를 막는 동기 블로킹

```ts
// 동기 + 블로킹: 5초 동안 메인 스레드가 완전히 멈춘다
function heavySync() {
  const start = Date.now();
  while (Date.now() - start < 5000) {
    // 아무것도 하지 않지만 CPU를 점유한다
  }
  console.log("done");
}

heavySync();
// 이 5초 동안 클릭, 스크롤, 애니메이션 전부 멈춘다
```

```ts
// 비동기 + 논블로킹: 제어권을 즉시 돌려받는다
fetch("/api/data").then((res) => res.json());
console.log("fetch 호출 직후 바로 실행됨");
```

두 코드를 비교하면 핵심이 보인다. 첫 번째는 작업이 끝날 때까지 제어권을 돌려주지 않고(블로킹), 두 번째는 호출 즉시 제어권을 돌려준다(논블로킹).

### 자주 하는 오해

- 동기면 무조건 블로킹이고 비동기면 무조건 논블로킹이다 → 단순화가 심하다
- async 함수는 자동으로 논블로킹이다 → 문법만 보고 그렇게 말하면 부정확하다. async 함수 안에서 무거운 동기 연산을 하면 첫 번째 await 전까지는 블로킹이다
- 네트워크 요청은 비동기니까 UI가 절대 안 멈춘다 → 틀림. 메인 스레드를 다른 코드가 막고 있으면 UI는 충분히 멈출 수 있다

### 실무 연결

브라우저에서 가장 중요한 건 메인 스레드를 막느냐다.
긴 계산, 무거운 JSON 파싱, 큰 리스트 렌더링, 비싼 애니메이션 계산이 있으면 사용자는 버튼도 못 누르고 스크롤도 버벅이는 체감을 하게 된다.

즉 브라우저 관점에서 블로킹은 단순 CS 개념이 아니라 사용자 경험 문제다.

> 이제 실행 흐름과 제어권 반환을 구분했다. 그 다음엔 브라우저와 JS 엔진이 실제로 이걸 어떻게 처리하는지, 즉 이벤트 루프로 넘어간다.

### 면접 압축 답변

동기/비동기는 작업 흐름의 설계 방식이고, 블로킹/논블로킹은 호출 후 제어권이 언제 돌아오는지에 대한 개념입니다. 둘은 관련 있지만 완전히 같은 축은 아닙니다.

## 23. 이벤트 루프는 왜 자바스크립트 비동기 이해의 중심인가

자바스크립트는 보통 단일 스레드 실행 모델로 설명된다.
그런데 실제 브라우저에서는 타이머도 되고, 네트워크 요청도 되고, 클릭 이벤트도 처리되고, Promise도 순서 있게 동작한다.

이걸 연결해주는 개념이 이벤트 루프다.

### 핵심 개념

이벤트 루프를 이해할 때는 아래 4개를 같이 봐야 한다.

- **Call Stack** — 현재 실행 중인 함수가 쌓이는 곳
- **Web APIs** — setTimeout, fetch, DOM 이벤트 등 브라우저가 별도로 처리하는 영역
- **Task Queue (Macrotask Queue)** — setTimeout, setInterval, I/O 콜백 등이 대기하는 큐
- **Microtask Queue** — Promise.then, queueMicrotask, MutationObserver 콜백이 대기하는 큐

### 동작 흐름

1. 현재 실행 중인 코드가 Call Stack에서 실행된다.
2. setTimeout, DOM 이벤트, 네트워크 콜백 같은 작업은 브라우저가 따로 관리한다.
3. 준비가 끝난 콜백은 Queue로 들어간다.
4. Call Stack이 비면 이벤트 루프가 **Microtask Queue를 전부 비운 뒤**, Task Queue에서 하나를 꺼내 Stack에 넣는다.
5. 이 사이클이 반복된다.

핵심 규칙: **매 Task 사이에 Microtask Queue 전체가 비워진다.** 그래서 Promise.then이 setTimeout보다 먼저 실행된다.

### 왜 이게 중요한가

이걸 이해하면 아래 질문이 한 번에 연결된다.

- 왜 setTimeout(..., 0)이 바로 실행되지 않나
- 왜 Promise then이 타이머보다 먼저 실행되나
- 왜 무한 마이크로태스크가 UI를 굳게 만들 수 있나
- React 업데이트 타이밍을 왜 “즉시 값 변경”으로 이해하면 안 되나

### 예시

```ts
console.log(“1”);

setTimeout(() => {
  console.log(“2”);
}, 0);

Promise.resolve().then(() => {
  console.log(“3”);
});

console.log(“4”);
```

실행 결과:

```
1
4
3
2
```

이유:

- 1, 4는 현재 Call Stack에서 즉시 실행
- Promise then은 Microtask Queue에 등록
- setTimeout 콜백은 Task Queue에 등록
- Stack이 비면 Microtask가 먼저 **전부** 비워지고, 그 다음 Task가 실행되기 때문이다

### 심화 예시: Microtask가 Microtask를 만들면

```ts
Promise.resolve().then(() => {
  console.log("micro 1");
  Promise.resolve().then(() => {
    console.log("micro 2");
  });
});

setTimeout(() => {
  console.log("macro");
}, 0);
```

출력: `micro 1` → `micro 2` → `macro`

Microtask 안에서 새 Microtask를 등록하면 그것도 같은 사이클에서 처리된다. 이게 무한히 반복되면 Task Queue는 영원히 실행되지 않고, UI도 멈춘다.

### 자주 하는 오해

- setTimeout(..., 0)은 즉시 실행된다 → 틀림. 최소 지연 시간이 있고, 무엇보다 Task Queue를 거치기 때문에 현재 스택과 모든 Microtask가 끝난 뒤에야 실행된다
- Promise는 그냥 더 빠른 타이머다 → 틀림. Promise.then은 Microtask이고 setTimeout은 Task다. 같은 큐가 아니다
- 이벤트 루프는 JS 엔진 내부에서만 전부 처리한다 → 브라우저(또는 Node.js 런타임)와의 협력이 핵심이다

### 실무 연결

이벤트 루프 감각이 있으면:

- 로딩 표시가 왜 늦게 바뀌는지
- Promise 체인이 왜 예상과 다르게 먼저 실행되는지
- 무거운 동기 코드가 왜 비동기 콜백도 못 돌게 만드는지

설명할 수 있다.

> 이제 실행 순서의 기반을 봤다. 그 다음엔 브라우저 비동기를 더 다루기 쉽게 만든 추상화인 Promise로 넘어간다.

### 면접 압축 답변

이벤트 루프는 Call Stack, 브라우저 비동기 처리, Task Queue, Microtask Queue를 조율해 자바스크립트 비동기 실행 순서를 결정하는 구조입니다. 핵심 규칙은 매 Task 사이에 Microtask Queue 전체가 비워진다는 것이며, 이 때문에 Promise.then이 setTimeout보다 먼저 실행됩니다.

## 24. Promise는 왜 필요한가

이벤트 루프를 알았다면 이제 다음 질문은 자연스럽다.

“그럼 비동기 작업을 그냥 콜백으로만 다루면 안 되나?”

초기 자바스크립트는 콜백 중심이었다.
그런데 콜백만으로 비동기를 관리하면 읽기 어렵고, 에러 흐름이 분산되고, 여러 비동기 조합이 복잡해진다.
Promise는 이 문제를 줄이기 위해 등장한 추상화다.

### 핵심 개념

Promise는 미래에 완료될 값을 나타내는 객체다.
상태는 세 가지이며, 한 번 결정되면 바뀌지 않는다(settled).

- **pending** — 아직 결과가 나오지 않음
- **fulfilled** — 성공적으로 완료됨
- **rejected** — 실패함

### 왜 이게 중요한가

Promise가 중요한 이유는 단순히 문법이 예뻐서가 아니다.

- 성공/실패 흐름을 구조화할 수 있고
- 체이닝이 가능하며
- 에러 전파 방식이 일관되고
- 여러 비동기를 조합하기 쉬워진다

### 콜백과 비교해서 이해하기

콜백 기반 코드는 중첩이 깊어지기 쉽다.

```ts
getUser(userId, (user) => {
  getPosts(user.id, (posts) => {
    getComments(posts[0].id, (comments) => {
      console.log(comments);
    });
  });
});
```

Promise를 쓰면 흐름을 더 평평하게 정리할 수 있다.

```ts
getUser(userId)
  .then((user) => getPosts(user.id))
  .then((posts) => getComments(posts[0].id))
  .then((comments) => {
    console.log(comments);
  })
  .catch((error) => {
    console.error(error);
  })
  .finally(() => {
    // 성공이든 실패든 항상 실행된다
    // 로딩 상태 해제 같은 정리 작업에 유용하다
    setLoading(false);
  });
```

### 자주 하는 오해

- Promise를 만들면 즉시 실행되지 않는다 → 틀림. executor 함수(`new Promise((resolve, reject) => { ... })` 안의 코드)는 생성 시점에 **동기적으로** 즉시 실행된다
- Promise는 비동기를 만든다 → Promise는 비동기 결과를 다루는 인터페이스이지, 비동기 작업 자체를 “자동 생성”하는 마법은 아니다
- then을 여러 번 붙이면 무조건 순차 실행이다 → 같은 Promise에 then을 여러 개 붙이면 독립적으로 실행된다. 체이닝(`.then().then()`)과 분기(`.then(); .then()`)는 다르다

### 실무 연결

Promise를 제대로 이해하면:

- API 요청 체인
- 순차/병렬 요청
- 에러 전파
- 재시도 로직
- 로딩 상태 관리

를 훨씬 더 정확하게 설계할 수 있다.

> Promise를 알았다면, 이제 많은 사람이 더 자주 쓰는 문법인 async/await가 Promise와 어떤 관계인지를 봐야 한다.

### 면접 압축 답변

Promise는 미래에 완료될 값을 나타내는 객체로, 비동기 작업의 성공과 실패 흐름을 구조적으로 다루기 위한 추상화입니다. 상태는 pending, fulfilled, rejected 세 가지이며 한 번 결정되면 변하지 않습니다. 콜백보다 체이닝과 에러 전파가 일관돼서 복잡한 비동기 흐름을 다루기 쉬워집니다.

## 25. async/await는 Promise와 어떤 관계인가

많은 사람이 async/await를 Promise와 별개의 기술처럼 느낀다.
하지만 실제로는 Promise 위에서 더 읽기 쉬운 문법을 제공하는 **syntactic sugar**다.

### 핵심 개념

- async 함수는 항상 Promise를 반환한다
- await는 Promise가 해결될 때까지 해당 함수 내부 흐름을 일시 중단한 것처럼 보이게 만든다
- 하지만 이것이 전체 자바스크립트 실행을 멈추는 것은 아니다. await 지점에서 함수가 **양보(yield)**하고, 나머지 코드는 이벤트 루프를 통해 계속 돌아간다

### 왜 이게 중요한가

이걸 이해해야:

- await를 쓸 수 있는 위치
- 병렬 요청과 순차 요청 차이
- try/catch로 Promise 에러를 잡는 방식

을 정확히 설명할 수 있다.

### 예시

```ts
async function fetchData() {
  const response = await fetch("/api/data");
  const data = await response.json();
  return data;
}

// 위 코드는 아래와 동일하게 동작한다
function fetchData() {
  return fetch("/api/data")
    .then((response) => response.json());
}
```

겉보기엔 동기 코드처럼 읽히지만, 내부적으로는 Promise 흐름 위에서 동작한다.

### 에러 처리

```ts
async function fetchData() {
  try {
    const response = await fetch("/api/data");
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error("요청 실패:", error);
    throw error; // 상위로 전파하려면 다시 throw
  }
}
```

try/catch로 Promise rejection을 잡을 수 있다는 점이 async/await의 큰 장점이다.

### 자주 하는 오해

- await를 쓰면 비동기가 동기로 바뀐다 → 틀림. 해당 함수 내부의 실행 흐름만 멈추는 것처럼 보일 뿐, 이벤트 루프는 계속 돌고 있다
- await는 어디서나 쓸 수 있다 → 기본적으로 async 함수 안에서만 가능하다. ES 모듈의 최상위 레벨(top-level await)에서도 가능하지만, 일반 스크립트나 CommonJS에서는 안 된다
- await를 많이 쓰면 무조건 코드가 더 좋다 → 병렬로 처리할 수 있는 작업까지 순차로 묶으면 오히려 느려질 수 있다

### 병렬과 순차를 구분해야 하는 이유

```ts
// 순차 실행: fetchUser가 끝나야 fetchPosts가 시작된다
const user = await fetchUser();
const posts = await fetchPosts();
// 총 시간 = fetchUser 시간 + fetchPosts 시간
```

두 요청이 서로 독립적이면 병렬이 더 적절하다.

```ts
// 병렬 실행: 두 요청이 동시에 시작된다
const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
// 총 시간 = 둘 중 더 오래 걸리는 시간
```

### 실무 연결

프론트엔드에서 async/await를 많이 쓰지만,
무조건 await를 줄줄이 쓰면:

- 독립 요청이 불필요하게 순차 실행되고
- 로딩 시간이 길어지고
- 사용자 체감이 나빠질 수 있다

> 그렇다면 여러 Promise를 한꺼번에 다룰 때는 어떤 조합 도구를 써야 하나?

### 면접 압축 답변

async/await는 Promise를 더 읽기 쉽게 다루는 문법적 설탕입니다. async 함수는 항상 Promise를 반환하고, await는 해당 함수 내부에서 Promise 결과를 기다리는 것처럼 보이게 하지만, 실제로는 그 지점에서 실행을 양보하고 이벤트 루프가 계속 동작합니다.

---

## 26. Promise.all, allSettled, race, any는 어떻게 구분하나

비동기를 실무에서 다루다 보면 “여러 요청을 어떻게 조합할 것인가”가 중요해진다.
이때 Promise 조합 함수를 목적에 맞게 구분해야 한다.

### 핵심 개념

**Promise.all**

- 모든 Promise가 성공해야 성공
- 하나라도 실패하면 **즉시** 전체 실패 (나머지 Promise는 취소되지 않고 무시된다)
- 모두 성공해야 의미가 있는 작업에 적합

**Promise.allSettled**

- 전부 끝날 때까지 기다림
- 성공/실패를 각각 `{ status, value }` 또는 `{ status, reason }` 형태의 결과로 받음
- 부분 성공/부분 실패를 같이 보고 싶을 때 적합

**Promise.race**

- 가장 먼저 끝난 Promise의 결과로 결정
- 성공이든 실패든 가장 먼저 끝난 쪽이 이김
- timeout 경쟁 등에 활용 가능

**Promise.any**

- 가장 먼저 성공한 Promise로 결정
- 모두 실패하면 AggregateError로 실패
- 여러 후보 중 하나만 성공하면 되는 경우에 적합

### 왜 이게 중요한가

이걸 구분 못 하면:

- 하나만 실패해도 전체가 깨질지
- 부분 성공을 살릴지
- timeout 처리에 무엇을 써야 할지

판단하기 어렵다.

### 예시

```ts
// all: 둘 다 있어야 화면을 완성할 수 있을 때
const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
```

```ts
// allSettled: 배너 하나 실패해도 공지와 추천은 보여주고 싶을 때
const results = await Promise.allSettled([
  fetchBanner(),
  fetchNotice(),
  fetchRecommendation(),
]);

results.forEach((result) => {
  if (result.status === "fulfilled") {
    render(result.value);
  } else {
    console.warn("부분 실패:", result.reason);
  }
});
```

```ts
// race: timeout 경쟁 패턴
const result = await Promise.race([
  fetch("/api/data"),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error("timeout")), 5000)
  ),
]);
```

### 자주 하는 오해

- Promise.all이 항상 제일 좋은 기본값이다 → 틀림. 부분 실패를 허용해야 하는 UI에서는 allSettled가 더 적절하다
- race는 가장 먼저 성공한 걸 고른다 → 정확히는 가장 먼저 **settled(성공 또는 실패)**된 것이다. 실패가 먼저 오면 전체가 실패한다
- allSettled는 실패를 숨긴다 → 틀림. 실패를 포함해서 결과를 돌려준다. 숨기는 게 아니라 판단을 개발자에게 맡긴다

### 실무 연결

실제 서비스에서는:

- 핵심 데이터는 all
- 부가 위젯은 allSettled
- timeout 제어는 race

처럼 역할을 나누는 경우가 많다.

> 여기까지는 문법 조합이다. 그런데 실제 서비스에서는 여러 요청이 겹칠 때 응답 순서 때문에 버그가 생길 수 있다. 그게 race condition이다.

### 면접 압축 답변

Promise.all은 모두 성공해야 성공하고, allSettled는 전부 끝날 때까지 기다린 뒤 각 결과를 모두 돌려줍니다. race는 가장 먼저 끝난 결과로 결정되고, any는 가장 먼저 성공한 결과를 사용합니다. 따라서 요구사항에 따라 조합 방식을 다르게 선택해야 합니다.

## 27. Race Condition은 프론트엔드에서 어떻게 생기나

비동기 문법을 아는 것과, 비동기 버그를 설명할 수 있는 것은 다르다.
실무 면접에서 강한 답변은 바로 여기서 나온다.

### 핵심 개념

Race Condition은 여러 작업이 같은 상태나 자원에 접근할 때, 실행 순서에 따라 결과가 달라지는 문제다.

### 대표적인 프론트엔드 예시

검색 입력창을 생각해보자.

- 사용자가 a를 입력한다
- 요청 A가 나간다
- 곧바로 ab를 입력한다
- 요청 B가 나간다
- 요청 B가 먼저 도착한다
- 화면은 ab 결과를 보여준다
- 그런데 요청 A가 **나중에** 도착해서 화면을 다시 덮어쓴다

이러면 사용자는 마지막 입력이 `ab`였는데 화면엔 `a` 결과가 보일 수 있다.

```ts
// 이 코드에서 race condition이 발생한다
function SearchBox() {
  const [keyword, setKeyword] = useState("");
  const [results, setResults] = useState([]);

  useEffect(() => {
    fetch(`/api/search?q=${keyword}`)
      .then((res) => res.json())
      .then((data) => setResults(data));
    // keyword가 빠르게 바뀌면, 느린 응답이 빠른 응답을 덮어쓸 수 있다
  }, [keyword]);

  return <input onChange={(e) => setKeyword(e.target.value)} />;
}
```

### 왜 이게 중요한가

이 문제를 모르면:

- “가끔 옛날 데이터가 보여요”
- “탭 전환하다 보면 값이 바뀌어요”
- “로딩이 끝났는데 다시 이전 상태가 나와요”

같은 버그를 설명할 수 없다.

### 자주 하는 오해

- race condition은 멀티스레드에서만 생긴다 → 틀림. 프론트엔드 비동기 흐름에서도 충분히 생긴다. 자바스크립트는 싱글 스레드지만 비동기 응답의 **도착 순서**는 보장되지 않기 때문이다
- 요청은 나중에 보낸 게 항상 나중에 도착한다 → 틀림. 네트워크는 그렇게 보장하지 않는다
- 최신 state만 보면 자동 해결된다 → 조건에 따라 여전히 오래된 응답이 화면을 덮을 수 있다

### 실무 연결

대표적인 발생 지점:

- 자동완성 검색
- 탭/필터 전환
- 무한 스크롤 페이지 전환
- 버튼 중복 클릭
- optimistic update 이후 rollback 타이밍 꼬임

> Race Condition을 알았다면 다음 질문은 당연하다. 이전 요청을 취소하거나, 최신 요청만 반영하려면 어떻게 해야 하나?

### 면접 압축 답변

Race Condition은 여러 비동기 작업의 완료 순서가 보장되지 않아 결과가 엉뚱하게 덮어써지는 문제입니다. 프론트엔드에서는 검색 자동완성이나 탭 전환처럼 요청이 연속으로 발생하는 UI에서 자주 생기며, 이전 요청 취소나 최신 요청만 반영하는 정책이 필요합니다.

---

## 28. AbortController는 왜 필요한가

Race Condition을 이해했다면, 이제 이전 요청을 어떻게 다룰지 생각해야 한다.
그중 가장 대표적인 도구가 AbortController다.

### 핵심 개념

AbortController는 진행 중인 비동기 작업, 특히 fetch 요청 같은 작업을 취소 신호로 제어할 수 있게 해준다.

- `AbortController` 인스턴스를 만든다
- `controller.signal`을 fetch의 옵션으로 전달한다
- `controller.abort()`를 호출하면 해당 요청이 취소된다
- 취소된 요청은 `AbortError`라는 이름의 에러를 던진다

### 예시

```ts
const controller = new AbortController();

fetch(“/api/search?q=abc”, {
  signal: controller.signal,
})
  .then((res) => res.json())
  .then((data) => {
    console.log(data);
  })
  .catch((error) => {
    if (error.name === “AbortError”) {
      console.log(“요청 취소됨”);
    } else {
      console.error(“실제 에러:”, error);
    }
  });

// 필요한 시점에 취소
controller.abort();
```

### 왜 이게 중요한가

AbortController가 중요한 이유는 단순히 네트워크 낭비를 줄이는 것만이 아니다.

- 오래된 요청이 의미 없이 끝까지 가지 않게 하고
- 최신 요청 중심으로 UI를 유지하고
- 언마운트 후 setState 같은 위험한 흐름을 줄이고
- 불필요한 에러 처리 분기를 줄이는 데 도움이 된다

### 자주 하는 오해

- abort하면 서버 작업도 무조건 완전히 중단된다 → 클라이언트 요청 취소와 서버 내부 처리 중단은 별개다. abort()는 브라우저가 응답을 기다리는 것을 중단하는 것이지, 서버에 "작업 중단" 신호를 보내는 건 아니다
- AbortController만 쓰면 race condition이 전부 해결된다 → 최신 요청 식별이나 상태 반영 정책도 같이 필요할 수 있다
- 취소된 요청도 일반 에러와 똑같이 처리하면 된다 → `error.name === "AbortError"`로 구분해서 사용자에게 에러 메시지를 보여주지 않는 게 보통이다

### React useEffect와 연결해서 보기

컴포넌트가 언마운트되거나 의존성이 바뀔 때, useEffect의 cleanup 함수가 실행된다.
이 cleanup에서 이전 요청을 취소하면 race condition을 방어할 수 있다.

```ts
useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/search?q=${keyword}`, { signal: controller.signal })
    .then((res) => res.json())
    .then((data) => setResult(data))
    .catch((error) => {
      if (error.name !== “AbortError”) {
        console.error(error);
      }
    });

  // keyword가 바뀌거나 컴포넌트가 언마운트되면 이전 요청을 취소한다
  return () => {
    controller.abort();
  };
}, [keyword]);
```

여기서 cleanup이 실행되는 타이밍을 정확히 알아야 한다. useEffect의 cleanup은 **컴포넌트가 언마운트될 때뿐 아니라, 의존성이 바뀌어서 effect가 다시 실행되기 직전에도** 호출된다. 그래서 keyword가 바뀔 때마다 이전 fetch가 취소되는 것이다.

> AbortController는 “취소”에 초점이 있다. 그런데 네트워크는 취소만 필요한 게 아니라 timeout, retry, backoff 같은 실패 복구 전략도 필요하다.

### 면접 압축 답변

AbortController는 진행 중인 요청에 취소 신호를 전달해 이전 요청을 중단하거나, 컴포넌트 언마운트 후 불필요한 처리 흐름을 줄이는 데 유용합니다. useEffect cleanup과 결합하면 의존성이 바뀔 때마다 이전 요청을 취소해 race condition을 방어할 수 있습니다.

---

## 29. timeout, retry, backoff는 왜 같이 생각해야 하나

실제 네트워크 환경은 항상 안정적이지 않다.
응답이 느릴 수도 있고, 일시적으로 실패할 수도 있고, 서버가 순간적으로 과부하일 수도 있다.

이때 중요한 건 “실패하면 그냥 에러”가 아니라 어떤 실패를 어떻게 복구할 것인가다.

### 핵심 개념

**timeout**

- 일정 시간 이상 응답이 없으면 기다리기를 중단하는 기준이다.
- 사용자가 무한 로딩을 경험하지 않게 하는 첫 번째 방어선이다.

**retry**

- 실패한 요청을 다시 시도하는 전략이다.
- 일시적인 네트워크 문제나 서버 과부하에 대응할 수 있다.

**backoff**

- 재시도 간격을 점점 늘리는 방식이다.
- 대표적으로 exponential backoff가 많이 언급된다.
- 서버가 과부하일 때 즉시 재시도를 몰아주면 상황이 악화될 수 있어서, 간격을 점점 넓힌다.

### 왜 이게 중요한가

이걸 구분해야:

- 사용자는 계속 무한 로딩만 보지 않게 되고
- 일시적인 장애는 자동 복구할 수 있고
- 서버에 과도한 재시도를 몰아주지 않게 된다

### 자주 하는 오해

- 실패하면 무조건 재시도하면 된다 → 틀림. 멱등하지 않은 요청(결제, 주문 생성 등)은 재시도하면 중복 처리될 위험이 있다
- timeout이 길수록 사용자 경험이 좋다 → 틀림. 너무 길면 체감상 앱이 멈춘 것처럼 느껴질 수 있다
- retry는 프론트에서만 고민하면 된다 → 서버의 멱등성, rate limiting, 에러 정책과 같이 봐야 한다

### 실무 연결

예를 들어:

- GET 요청은 재시도에 더 적합한 편이다 (같은 요청을 여러 번 보내도 결과가 같다)
- 결제 같은 POST 요청은 멱등성 키 없이 함부로 재시도하면 위험할 수 있다
- 업로드는 실패 원인에 따라 재시도 정책을 더 신중하게 설계해야 한다
- HTTP 상태 코드에 따라 재시도 여부를 나눌 수도 있다: 503(일시 과부하)은 재시도, 400(잘못된 요청)은 재시도해도 의미 없음

### timeout 경쟁 예시

```ts
function timeoutPromise(ms: number) {
  return new Promise((_, reject) => {
    setTimeout(() => reject(new Error("timeout")), ms);
  });
}

async function fetchWithTimeout(url: string, ms = 5000) {
  return Promise.race([fetch(url), timeoutPromise(ms)]);
}
```

### exponential backoff 예시

```ts
async function fetchWithRetry(
  url: string,
  maxRetries = 3,
  baseDelay = 1000
): Promise<Response> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch(url);
      if (!response.ok) throw new Error(`HTTP ${response.status}`);
      return response;
    } catch (error) {
      if (attempt === maxRetries) throw error;

      // 1초, 2초, 4초... 로 간격이 늘어난다
      const delay = baseDelay * Math.pow(2, attempt);
      // jitter를 추가해 여러 클라이언트가 동시에 재시도하는 것을 분산
      const jitter = delay * Math.random() * 0.5;
      await new Promise((resolve) => setTimeout(resolve, delay + jitter));
    }
  }
  throw new Error("unreachable");
}
```

jitter를 추가하는 이유: 서버 장애 시 여러 클라이언트가 정확히 같은 간격으로 재시도하면 동시에 요청이 몰린다. 랜덤 지연을 더해서 이를 분산시킨다.

> 이제 요청 실패 복구 전략까지 봤다. 다음엔 연속 입력이나 중복 클릭처럼 요청 자체를 덜 만들고 최신 것만 유지하는 전략으로 넘어간다.

### 면접 압축 답변

timeout은 오래 기다리지 않기 위한 기준이고, retry는 실패한 요청을 다시 시도하는 전략이며, backoff는 재시도 간격을 점점 늘려 과부하를 줄이는 방식입니다. 다만 모든 요청을 무조건 재시도하면 안 되고, 멱등성과 사용자 경험을 함께 고려해야 합니다.

## 30. 중복 요청 방지, 최신 요청 반영, 그리고 디바운스/스로틀

좋은 프론트엔드는 실패를 잘 처리하는 것뿐 아니라, 애초에 불필요한 요청을 덜 만들고, 의미 있는 응답만 반영한다.
이 섹션에서는 요청 빈도 제어부터 최신 응답 반영까지 한꺼번에 본다.

### 핵심 전략 정리

| 전략 | 목적 | 대표 도구 |
|------|------|-----------|
| 입력 빈도 줄이기 | 불필요한 요청 자체를 줄임 | debounce, throttle |
| 이전 요청 취소 | 의미 없는 진행 중 요청을 중단 | AbortController |
| 최신 요청만 반영 | 늦게 온 응답이 화면을 덮지 않게 함 | request id 비교, 시퀀스 번호 |
| 같은 키 요청 재사용 | 동일 요청 중복을 없앰 | 캐시, dedupe |
| 중복 클릭 방지 | 사용자 실수로 여러 번 실행되는 것을 막음 | 버튼 disable, pending 상태 |

### 디바운스와 스로틀

**디바운스(debounce)**

연속 입력이 멈춘 뒤 일정 시간이 지나면 **한 번** 실행한다.

```ts
function debounce<T extends (...args: any[]) => void>(fn: T, delay: number) {
  let timer: ReturnType<typeof setTimeout>;
  return (...args: Parameters<T>) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}

// 사용 예: 입력이 300ms 동안 멈추면 검색 실행
const debouncedSearch = debounce((keyword: string) => {
  fetch(`/api/search?q=${keyword}`);
}, 300);

input.addEventListener(“input”, (e) => {
  debouncedSearch(e.target.value);
});
```

**스로틀(throttle)**

일정 시간 동안 **최대 한 번만** 실행되게 제한한다.

```ts
function throttle<T extends (...args: any[]) => void>(fn: T, limit: number) {
  let throttled = false;
  return (...args: Parameters<T>) => {
    if (throttled) return;
    throttled = true;
    fn(...args);
    setTimeout(() => {
      throttled = false;
    }, limit);
  };
}

// 사용 예: 스크롤 이벤트를 200ms마다 최대 한 번만 처리
window.addEventListener(“scroll”, throttle(() => {
  checkScrollPosition();
}, 200));
```

**언제 뭘 쓰나:**

- 검색창 자동완성 → debounce (입력이 끝난 뒤 요청)
- 스크롤/리사이즈 이벤트 → throttle (일정 간격으로 계속 반응)
- 버튼 연타 방지 → throttle 또는 disable 상태 관리

### 최신 요청만 반영하는 패턴

```ts
let requestId = 0;

async function search(keyword: string) {
  const currentId = ++requestId;
  const response = await fetch(`/api/search?q=${keyword}`);
  const data = await response.json();

  // 이 사이에 새 요청이 시작됐으면 이 응답은 버린다
  if (currentId !== requestId) return;

  renderResult(data);
}
```

이 방식은 AbortController로 취소하지 못한 요청이 남아 있어도, 가장 마지막 요청만 화면에 반영하게 만든다.

### 실전 조합: debounce + AbortController + 최신 요청 체크

```ts
function useSearch(keyword: string) {
  const [results, setResults] = useState([]);
  const requestIdRef = useRef(0);

  useEffect(() => {
    const controller = new AbortController();
    const currentId = ++requestIdRef.current;

    const timer = setTimeout(() => {
      fetch(`/api/search?q=${keyword}`, { signal: controller.signal })
        .then((res) => res.json())
        .then((data) => {
          if (currentId === requestIdRef.current) {
            setResults(data);
          }
        })
        .catch((error) => {
          if (error.name !== “AbortError”) console.error(error);
        });
    }, 300); // debounce 300ms

    return () => {
      clearTimeout(timer);
      controller.abort();
    };
  }, [keyword]);

  return results;
}
```

이 패턴은 세 가지 방어를 동시에 한다:
1. debounce로 요청 빈도를 줄이고
2. AbortController로 이전 요청을 취소하고
3. request id로 혹시 취소가 안 된 응답이 와도 무시한다

### 자주 하는 오해

- debounce만 걸면 race condition이 사라진다 → 요청 횟수는 줄지만 순서 문제는 남을 수 있다
- AbortController만 쓰면 충분하다 → 취소 타이밍에 따라 응답이 이미 도착했을 수 있다
- 서버가 알아서 막아줄 것이다 → 프론트에서도 충분히 방어해야 사용자 경험이 좋아진다

### 실무 연결

대표적인 사용처:

- 검색 자동완성 → debounce + AbortController
- 필터 조합 변경 → debounce + 최신 요청 체크
- 무한 스크롤 → throttle + 페이지 번호 체크
- CSV 업로드 중 중복 제출 방지 → 버튼 disable + pending 상태
- 저장 버튼 연타 방지 → throttle 또는 optimistic lock

> 이제 비동기 흐름의 큰 축을 봤다. 다음 챕터에서는 “요청/응답 데이터의 형태를 어떻게 믿고 다룰 것인가”라는 질문으로 넘어간다.

### 면접 압축 답변

중복 요청 방지는 debounce, throttle, 버튼 disable, 캐시 재사용으로 요청 자체를 줄이고, race condition 대응은 AbortController로 이전 요청을 취소하거나 request id 비교로 최신 응답만 반영합니다. 실무에서는 이 전략들을 조합해서 쓰는 것이 일반적입니다.

## 31. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래가 한 덩어리로 보여야 한다.

- 비동기는 속도 문제가 아니라 흐름 제어 문제다
- 블로킹은 사용자 경험과 직접 연결된다
- 이벤트 루프는 실행 순서를 설명하는 기반이다
- Promise와 async/await는 비동기 흐름을 구조적으로 다루기 위한 도구다
- 여러 요청이 겹치면 race condition이 생긴다
- AbortController, request id 비교, debounce, retry/backoff는 그에 대한 실무 대응이다
