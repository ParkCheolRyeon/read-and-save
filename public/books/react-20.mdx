# 03. react20문항

> React 렌더링 모델, 상태 구조, 성능 최적화 감각을 점검하는 20문항입니다.

### 전체 흐름 맵

```
선언형 모델 (UI = f(state))
  → 렌더 / 커밋 (계산과 반영의 분리)
    → Virtual DOM (변경 감지 추상화)
      → 리렌더링 조건 (state, props, context)
        → 부모→자식 리렌더 (기본 동작)
          → useState snapshot (값의 시점 고정)
            → useEffect / useLayoutEffect (외부 동기화)
              → useMemo / useCallback (참조 안정화)
                → React.memo (리렌더 스킵)
                  → Controlled / Uncontrolled (입력 제어)
                    → key와 identity (state 보존/리셋)
                      → 상태 끌어올리기 vs 전역 (소유 범위)
                        → Context vs 외부 Store (전달 vs 관리)
                          → 커스텀 훅 (로직 재사용)
                            → 에러 바운더리 (실패 격리)
                              → 서버 상태 vs 로컬 상태 (역할 분리)
                                → 복잡한 폼 (상태 설계)
                                  → 애니메이션 UI (구조 분리)
                                    → DnD 상태 (도메인 vs 인터랙션)
                                      → 마이그레이션 (재설계 전략)
```

## 1. React가 선언형이라고 하는 이유를 설명해보세요.

### 합격 답변 예시

React가 선언형이라고 하는 이유는, 개발자가 DOM을 한 줄씩 직접 조작하는 대신 현재 상태에서 UI가 어떤 모습이어야 하는지 선언하면 React가 그 결과를 계산하고 실제 DOM 반영을 맡기기 때문입니다. 즉 "어떻게 바꿀지"보다 "무엇이 보여야 하는지"에 집중하게 해줍니다.

### 가산점 포인트

**"선언형이 가능하려면 render가 순수해야 한다"**까지 말하면 좋다. 같은 props/state에서 같은 JSX를 반환해야(순수성) React가 렌더 결과를 비교하고 최적화할 수 있다. render 안에서 부수효과를 일으키면 선언형 모델이 깨진다.

### 상세 해설

#### 명령형 vs 선언형 비교

```ts
// 명령형: DOM을 직접 단계별로 조작
const p = document.getElementById('count-display');
const btn = document.getElementById('increment-btn');
let count = 0;
btn.addEventListener('click', () => {
  count++;
  p.textContent = `현재 값: ${count}`;  // DOM을 직접 찾아서 바꿈
});
```

```tsx
// 선언형: 상태에 따라 UI가 어떤 모습이어야 하는지 기술
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>현재 값: {count}</p>
      <button onClick={() => setCount(c => c + 1)}>증가</button>
    </div>
  );
  // "p를 찾아서 텍스트를 바꿔라"가 아니라
  // "count가 이 값일 때 이런 UI다"를 선언
}
```

#### UI = f(state)

이 공식이 React의 핵심 사고 모델이다. 상태가 바뀌면 React가 알아서 UI를 다시 계산하고, 이전 결과와 비교해서 **달라진 DOM만 최소한으로 반영**한다.

#### 자주 하는 오해

- "선언형이면 내부적으로 아무 일도 안 한다" → **틀림.** React는 Virtual DOM 비교, DOM 조작을 모두 수행한다. 개발자가 직접 안 할 뿐.
- "선언형이면 성능이 더 좋다" → **항상 그런 건 아니다.** 선언형의 장점은 성능보다 **예측 가능성과 유지보수성**이다.

#### 실무 에피소드

> jQuery로 만든 레거시 탭 UI에서 "탭은 B인데 내용은 A" 같은 버그가 반복됐다. 탭 상태와 DOM 업데이트가 여러 곳에 흩어져 있어서 동기화가 깨진 것. React로 마이그레이션하면서 `activeTab` state 하나로 탭과 콘텐츠를 선언하니 상태-UI 불일치 버그가 사라졌다.

> React가 선언형이라는 건 알았다. 다음에는 **상태가 바뀌었을 때 React가 내부에서 어떤 단계를 거쳐 UI를 바꾸는지** — 렌더와 커밋을 봐야 한다.

## 2. React의 렌더링 과정과 커밋 과정을 설명해보세요.

### 합격 답변 예시

React 업데이트는 크게 렌더(render)와 커밋(commit)으로 나눕니다. 렌더 단계에서는 React가 컴포넌트 함수를 호출해서 다음 UI를 계산하고, 이전 계산과 비교합니다. 커밋 단계에서는 실제로 필요한 최소 DOM 변경을 적용하고, Effect가 실행됩니다.

### 가산점 포인트

**"렌더가 일어났다고 항상 DOM 변경이 생기는 건 아니다"**라고 말하면 좋다. 렌더 결과가 이전과 같으면 커밋 단계에서 DOM을 건드리지 않는다. 렌더 = 계산, 커밋 = 반영.

### 상세 해설

```
상태 변경
    ↓
┌──────────────────────────────────────────┐
│ [렌더 단계] — 순수 계산, 부수효과 금지     │
│  컴포넌트 함수 실행 → 새 JSX → diffing     │
│  ※ 중단/재개 가능 (Fiber)                  │
└──────────────────────────────────────────┘
    ↓
┌──────────────────────────────────────────┐
│ [커밋 단계] — 실제 DOM 변경               │
│  변경된 DOM 노드만 업데이트                 │
│    → useLayoutEffect (DOM 직후, 동기)     │
│      → 브라우저 페인트                     │
│        → useEffect (페인트 이후)           │
└──────────────────────────────────────────┘
```

#### 자주 하는 오해

- "render = DOM 조작" → **틀림.** render는 "다음 UI를 계산하는 단계"이고, DOM 조작은 commit에서 일어난다.
- "렌더가 일어나면 무조건 느려진다" → **아님.** 렌더는 JS 함수 호출일 뿐이고, DOM 변경이 없으면 커밋도 가볍다. 문제는 **느린 렌더**다.

> 렌더와 커밋을 알았으니, 다음에는 **렌더 결과를 어떻게 비교하는지** — Virtual DOM을 봐야 한다.

## 3. Virtual DOM은 무엇이고, 왜 필요한가요?

### 합격 답변 예시

Virtual DOM은 React가 렌더 단계에서 계산하는 다음 UI의 메모리상 표현입니다. React는 이전 렌더 결과와 다음 렌더 결과를 비교해서 실제 DOM에는 필요한 최소 변경만 반영합니다. 핵심은 DOM 조작 절차 대신 **UI를 선언하고, 변경분만 커밋**할 수 있게 해주는 추상화입니다.

### 가산점 포인트

Virtual DOM을 **"무조건 빠르게 만드는 마법"**으로 말하면 안 된다. 본질은 성능보다 **추상화와 일관성**이다. React가 렌더 모델을 유지하면서 필요한 최소 변경만 반영하는 전략이다.

### 상세 해설

```tsx
// 이 JSX는 내부적으로 JS 객체 트리가 된다
<div>
  <h1>Hello</h1>
  <p>World</p>
</div>

// → { type: 'div', props: { children: [
//     { type: 'h1', props: { children: 'Hello' } },
//     { type: 'p', props: { children: 'World' } }
//   ]}}
```

React는 이 객체 트리(Virtual DOM)를 이전 것과 비교(diffing)해서, 실제로 바뀐 부분만 real DOM에 반영한다. 이 덕분에 개발자는 **전체 UI를 매번 선언**해도, React가 필요한 최소 변경만 찾아서 적용한다.

#### 자주 하는 오해

- "Virtual DOM은 DOM보다 빠르다" → **항상 그런 건 아님.** 비교 비용이 있다. 핵심은 **개발자 편의 + 일관된 업데이트 전략**이다.

> Virtual DOM을 알았으니, 다음에는 **언제 렌더가 다시 일어나는지** — 리렌더링 조건을 봐야 한다.

---

## 4. 리렌더링이 발생하는 조건을 설명해보세요.

### 합격 답변 예시

React 컴포넌트는 초기 렌더 후, 주로 state가 바뀌거나, 부모가 다시 렌더되거나, 사용하는 context 값이 바뀌면 다시 렌더됩니다. 외부 스토어를 `useSyncExternalStore`로 구독하는 경우에는 스냅샷이 바뀌었을 때도 리렌더됩니다. 다만 **리렌더가 일어났다고 항상 DOM 변경이 생기는 것은 아닙니다**.

### 가산점 포인트

리렌더링 조건을 **트리거 관점**으로 정확히 나열할 수 있으면 좋다: (1) setState 호출, (2) 부모 리렌더, (3) context 값 변경, (4) 외부 스토어 snapshot 변경.

### 상세 해설

```tsx
// 리렌더링 원인별 예시

// 1. state 변경 → 자신이 리렌더
const [count, setCount] = useState(0);
setCount(1);  // → 이 컴포넌트 리렌더

// 2. 부모 리렌더 → 자식도 리렌더 (기본 동작)
function Parent() {
  const [x, setX] = useState(0);
  return <Child />;  // Parent가 리렌더되면 Child도 리렌더
}

// 3. context 변경 → 해당 context를 쓰는 모든 컴포넌트 리렌더
const theme = useContext(ThemeContext);
// ThemeContext.Provider의 value가 바뀌면 리렌더

// 4. 외부 스토어 → snapshot 변경 시 리렌더
const count = useStore(state => state.count);
// count가 바뀌면 리렌더
```

#### 자주 하는 오해

- "props가 바뀌면 리렌더된다" → **정확하지 않음.** props가 바뀌려면 부모가 리렌더되어야 하고, 부모가 리렌더되면 props와 무관하게 자식도 기본적으로 리렌더된다.
- "setState를 호출하면 즉시 렌더된다" → **아님.** React는 업데이트를 batching해서 한 번에 처리할 수 있다.

> 리렌더링 조건을 알았으니, 다음에는 **부모-자식 리렌더의 기본 동작과 최적화**를 봐야 한다.

## 5. 부모가 리렌더링되면 자식은 항상 리렌더링되나요?

### 합격 답변 예시

기본적으로는 그렇습니다. React는 부모가 다시 렌더되면 자식도 재귀적으로 다시 렌더합니다. 하지만 자식이 `React.memo`로 감싸져 있고 props가 이전과 같다면 리렌더를 건너뛸 수 있습니다. 즉 **기본값은 "부모 리렌더 → 자식도 리렌더"이고, 최적화가 들어가면 예외가 생깁니다**.

### 가산점 포인트

memo가 효과적이려면 **props가 안정적이어야** 한다. 부모가 매 렌더마다 `{ onClick: () => {} }` 같은 새 함수/객체를 만들면 memo의 얕은 비교가 항상 "다르다"고 판단해서 의미가 없다. 그래서 `useCallback`/`useMemo`와 함께 써야 할 때가 많다.

### 상세 해설

```tsx
// ❌ memo가 무력화되는 경우
function Parent() {
  const [count, setCount] = useState(0);
  // 매 렌더마다 새 함수 생성 → Child의 memo가 무력화
  return <Child onClick={() => console.log(count)} />;
}
const Child = React.memo(function Child({ onClick }: { onClick: () => void }) {
  return <button onClick={onClick}>Click</button>;
});

// ✅ memo가 효과적인 경우
function Parent() {
  const [count, setCount] = useState(0);
  // useCallback으로 함수 참조 안정화
  const handleClick = useCallback(() => console.log(count), [count]);
  return <Child onClick={handleClick} />;
}
```

> 부모-자식 리렌더를 알았으니, 다음에는 **state가 렌더 시점에 어떻게 동작하는지** — useState의 snapshot 특성을 봐야 한다.

## 6. useState의 비동기적 업데이트 특성을 설명해보세요.

### 합격 답변 예시

setState를 호출한다고 현재 렌더 안에서 바로 state 변수가 바뀌는 것은 아닙니다. React는 현재 렌더의 state를 "snapshot"처럼 취급하고, setState는 **다음 렌더를 예약**합니다. 같은 핸들러에서 `setCount(count + 1)`을 여러 번 호출하면 현재 snapshot의 같은 count를 기준으로 계산되므로, 누적하려면 함수형 업데이트 `setCount(c => c + 1)`를 써야 합니다.

### 가산점 포인트

**"비동기"라는 말보다 "즉시 현재 변수 값을 바꾸지 않고 다음 렌더를 예약한다"**라고 설명하면 더 정확하다. 또 React 18+에서는 이벤트 핸들러뿐 아니라 setTimeout, Promise 안의 업데이트도 자동 batching된다.

### 상세 해설

```ts
// ❌ 기대한 대로 안 됨: 같은 snapshot의 count를 두 번 읽음
setCount(count + 1);  // count = 0이면, 0 + 1 = 1
setCount(count + 1);  // 여전히 count = 0이므로, 0 + 1 = 1
// 결과: count = 1 (2가 아님)

// ✅ 함수형 업데이트: 이전 결과를 기준으로 누적
setCount(c => c + 1);  // 0 → 1
setCount(c => c + 1);  // 1 → 2
// 결과: count = 2
```

```tsx
// Stale closure 예시: setTimeout 안에서 이전 렌더의 값을 봄
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    setTimeout(() => {
      console.log(count);  // 항상 클릭 시점의 count (0, 1, 2...)
      // 최신 값이 아님 — 이 클로저가 캡처한 렌더 시점의 값
    }, 3000);
  }

  return <button onClick={handleClick}>{count}</button>;
}
```

#### 자주 하는 오해

- "setState 후 console.log 하면 새 값이 보인다" → **아님.** 현재 렌더의 state 변수는 바뀌지 않는다. 새 값은 다음 렌더에서 보인다.
- "React 18에서 batching이 바뀌었다" → **맞음.** React 18 이전에는 이벤트 핸들러에서만 batching되었지만, 18부터는 setTimeout, Promise 등 모든 곳에서 자동 batching된다.

> useState의 snapshot을 알았으니, 다음에는 **외부 시스템과의 동기화** — useEffect와 useLayoutEffect를 봐야 한다.

## 7. useEffect와 useLayoutEffect의 차이를 설명해보세요.

### 합격 답변 예시

useEffect는 컴포넌트를 외부 시스템과 동기화하기 위한 훅이고, 보통 브라우저가 화면을 그린 뒤 실행됩니다. 반면 useLayoutEffect는 브라우저가 repaint하기 전에 실행되므로, 레이아웃 측정이나 DOM 읽기/쓰기처럼 **화면 깜빡임 없이 처리해야 하는 경우**에 씁니다. 기본값은 useEffect이고, 정말 레이아웃 측정 타이밍이 필요할 때만 useLayoutEffect를 씁니다.

### 가산점 포인트

useEffect의 본질은 **"외부 시스템과의 동기화"**이다. 소켓 연결, 구독 등록, 브라우저 이벤트 리스너, 서드파티 라이브러리 초기화가 전형적인 사용처다. props/state에서 계산 가능한 값을 Effect로 다시 state에 넣는 패턴은 안티패턴이다 (→ Q. "You Might Not Need an Effect" 보완 문항 참조).

### 상세 해설

```tsx
// useEffect: 외부 시스템 동기화 (페인트 후 실행)
useEffect(() => {
  const ws = new WebSocket('wss://api.example.com');
  ws.onmessage = (e) => setData(JSON.parse(e.data));
  return () => ws.close();  // cleanup: 구독 해제
}, []);

// useLayoutEffect: DOM 측정 + 반영 (페인트 전 실행)
useLayoutEffect(() => {
  const rect = tooltipRef.current.getBoundingClientRect();
  // 측정 후 위치를 조정해서 깜빡임 방지
  setTooltipPosition({ top: rect.bottom, left: rect.left });
}, [targetRef]);
```

```
실행 순서:
  렌더 → 커밋(DOM 변경) → useLayoutEffect → 브라우저 페인트 → useEffect
                            ↑ 페인트 전              ↑ 페인트 후
```

#### 자주 하는 오해

- "useEffect에서 데이터를 fetch하면 안 된다" → **괜찮다.** 외부 데이터와 동기화하는 것은 Effect의 정당한 용도. 다만 React Query 같은 라이브러리가 캐시/에러/로딩을 더 잘 관리해준다.
- "모든 Effect를 useLayoutEffect로 바꾸면 더 안전하다" → **오히려 성능 저하.** useLayoutEffect는 페인트를 막으므로 꼭 필요한 경우만 써야 한다.

> useEffect를 알았으니, 다음에는 **성능 최적화 도구** — useMemo와 useCallback을 봐야 한다.

## 8. useMemo와 useCallback은 언제 써야 하고, 남용하면 왜 안 되나요?

### 합격 답변 예시

useMemo는 비싼 계산 결과나 참조 동일성이 중요한 값을 캐싱할 때, useCallback은 함수 참조를 안정화할 때 씁니다. 하지만 둘 다 dependency 관리와 메모리 유지 비용이 있고, **실제 병목이 확인된 경우에만 쓰는 것**이 좋습니다.

### 가산점 포인트

React Compiler가 수동 메모이제이션의 필요성을 줄일 수 있다는 점을 알면 좋다. 또 useMemo/useCallback은 **"이 값/함수가 매 렌더마다 새로 만들어져도 되는가?"**를 기준으로 판단한다.

### 상세 해설

```tsx
// useMemo: 비싼 계산 캐싱
const sortedItems = useMemo(() => {
  return items.sort((a, b) => a.price - b.price);  // O(n log n) — 매 렌더마다 할 필요 없음
}, [items]);

// useCallback: memo된 자식에게 안정적인 함수 전달
const handleDelete = useCallback((id: string) => {
  setItems(prev => prev.filter(item => item.id !== id));
}, []);

// 둘이 함께 쓰이는 패턴
const MemoizedList = React.memo(function List({ items, onDelete }: Props) {
  return items.map(item => (
    <Item key={item.id} item={item} onDelete={onDelete} />
  ));
});

function Parent() {
  const [items, setItems] = useState(initialItems);
  const sorted = useMemo(() => sortByPrice(items), [items]);
  const handleDelete = useCallback((id: string) => {
    setItems(prev => prev.filter(i => i.id !== id));
  }, []);

  return <MemoizedList items={sorted} onDelete={handleDelete} />;
}
```

#### 남용하면 왜 안 좋나?

- dependency 배열 관리가 복잡해진다
- stale closure 위험이 커진다
- 코드가 읽기 어려워진다
- 실제 이득이 없는 곳에도 기계적으로 써서 유지보수가 나빠진다

#### 자주 하는 오해

- "모든 함수에 useCallback을 써야 한다" → **대부분 불필요.** memo된 자식에게 넘기거나, Effect dependency로 쓰는 경우에만 의미 있다.

> memoization을 알았으니, 다음에는 **컴포넌트 수준의 리렌더 최적화** — React.memo를 봐야 한다.

## 9. React.memo는 어떤 경우에 효과가 있고, 어떤 경우엔 효과가 거의 없나요?

### 합격 답변 예시

React.memo는 부모가 자주 리렌더되더라도 자식의 props가 같다면 렌더를 건너뛰게 해줍니다. **비용이 큰 자식 컴포넌트**에서 특히 효과가 있습니다. 하지만 props가 매번 새 객체/함수로 만들어지면 비교 결과가 달라져서 효과가 없어집니다.

### 가산점 포인트

memo는 **"자주 리렌더되고, props가 안정적이며, 렌더 비용이 있는 컴포넌트"**에 써야 효과적이다. 세 조건 중 하나라도 빠지면 효과가 미미하거나 오히려 비교 비용만 추가된다.

### 상세 해설

```tsx
// ✅ memo가 효과적인 경우: 렌더 비용이 큰 리스트
const ExpensiveChart = React.memo(function Chart({ data }: { data: Point[] }) {
  // 차트 렌더링은 비용이 큼
  return <canvas>{/* 복잡한 렌더링 */}</canvas>;
});

// ❌ memo가 무의미한 경우: 렌더 비용이 거의 없는 단순 컴포넌트
const Label = React.memo(function Label({ text }: { text: string }) {
  return <span>{text}</span>;  // 비교 비용이 렌더 비용보다 클 수 있음
});
```

#### 자주 하는 오해

- "모든 컴포넌트에 memo를 붙이면 좋다" → **과잉 최적화.** 비교 비용이 있고, 간단한 컴포넌트는 리렌더가 더 가벼울 수 있다.

> React.memo를 알았으니, 다음에는 **입력 제어 방식** — Controlled vs Uncontrolled를 봐야 한다.

## 10. Controlled Component와 Uncontrolled Component의 차이를 설명해보세요.

### 합격 답변 예시

Controlled Component는 입력값의 source of truth가 React state에 있고, `value`와 `onChange`로 제어합니다. Uncontrolled Component는 DOM이 값을 자체적으로 가지고, React는 `defaultValue`나 `ref`로 초기값/읽기만 관여합니다.

### 가산점 포인트

controlled와 uncontrolled를 **생명주기 중간에 왔다 갔다 하면 안 된다**. React는 이 전환을 감지하고 경고를 보낸다. 또 React는 이 패턴을 input에만 한정하지 않고, **"부모가 제어하는가 / 내부가 제어하는가"**의 일반적 설계 패턴으로도 설명한다.

### 상세 해설

```tsx
// Controlled: React state가 source of truth
function ControlledInput() {
  const [value, setValue] = useState('');
  return (
    <input
      value={value}                              // React가 값을 제어
      onChange={(e) => setValue(e.target.value)}   // 변경도 React가 관리
    />
  );
}

// Uncontrolled: DOM이 값을 가짐
function UncontrolledInput() {
  const inputRef = useRef<HTMLInputElement>(null);
  function handleSubmit() {
    console.log(inputRef.current?.value);  // 필요할 때만 DOM에서 읽기
  }
  return <input ref={inputRef} defaultValue="초기값" />;
}
```

| | Controlled | Uncontrolled |
|---|---|---|
| **source of truth** | React state | DOM |
| **실시간 검증** | 가능 | 어려움 |
| **다른 state와 동기화** | 쉬움 | 어려움 |
| **구현 복잡도** | 높음 | 낮음 |
| **사용처** | 검증/동기화 필요한 폼 | 단순 제출 중심 폼 |

> Controlled/Uncontrolled를 알았으니, 다음에는 **React가 컴포넌트 identity를 추적하는 방법** — key를 봐야 한다.

## 11. key를 왜 써야 하는지, index를 key로 쓰면 어떤 문제가 생기는지 설명해보세요.

### 합격 답변 예시

key는 React가 리스트 항목의 **정체성(identity)**을 추적해서, 어떤 항목이 유지/생성/이동되었는지 판단하게 해줍니다. index를 key로 쓰면 항목 삽입/삭제/재정렬 시 데이터와 state가 엇갈려서 **입력값이 잘못된 항목에 붙거나 선택 상태가 이동하는 문제**가 생길 수 있습니다.

### 가산점 포인트

key는 단순 경고 제거용이 아니라 **state 보존/리셋 기준**과 직접 연결된다. key를 바꾸면 같은 컴포넌트여도 state가 초기화된다. 이를 활용해 form reset 패턴도 구현할 수 있다: `<EditForm key={selectedId} />`.

### 상세 해설

```tsx
// ❌ index key: 삽입 시 state가 꼬임
// 항목 B를 맨 앞에 삽입하면:
// 이전: [A(key=0), C(key=1)]
// 다음: [B(key=0), A(key=1), C(key=2)]
// → React: key=0의 내용이 A→B로 바뀐 걸로 인식 → A의 state가 B에 붙음

// ✅ id key: 항목 identity가 유지됨
{items.map(item => (
  <TodoItem key={item.id} item={item} />
  // item.id가 안정적이므로 삽입/삭제해도 각 항목의 state가 정확히 유지
))}

// key를 활용한 form 초기화 패턴
<EditForm key={selectedUserId} userId={selectedUserId} />
// selectedUserId가 바뀌면 EditForm의 state가 완전히 초기화됨
```

> key를 알았으니, 다음에는 **상태의 소유 범위** — 끌어올리기 vs 전역을 봐야 한다.

## 12. 상태 끌어올리기와 전역 상태 사용의 경계를 어떻게 판단하나요?

### 합격 답변 예시

기본 원칙은 **가장 가까운 공통 부모**로 상태를 올리는 것입니다. 공유 범위가 너무 넓어지거나, 여러 페이지에서 같은 상태에 접근해야 하거나, 도메인 규칙을 한 곳에서 관리해야 할 때 전역화를 검토합니다.

### 가산점 포인트

**"먼저 로컬하게 두고, 실제 공유 범위와 변경 패턴이 커졌을 때 전역화를 검토하는 것"**이 좋은 순서다. 처음부터 전역에 두면 상태 추적이 어려워지고, 불필요한 리렌더 범위가 넓어진다.

### 상세 해설

```tsx
// 1단계: 가장 가까운 공통 부모로 끌어올리기
function ProductPage() {
  const [selectedColor, setSelectedColor] = useState('red');
  return (
    <>
      <ColorPicker color={selectedColor} onChange={setSelectedColor} />
      <ProductImage color={selectedColor} />
    </>
  );
  // 두 컴포넌트가 같은 color를 공유 → 부모에서 관리
}

// 2단계: 전역이 필요한 경우의 신호
// - 여러 페이지에서 접근 (장바구니, 인증 상태)
// - props drilling이 3단계 이상 깊어짐
// - 도메인 규칙을 한 곳에서 관리하고 싶음
// - undo/redo, devtools가 필요
```

> 상태 소유를 알았으니, 다음에는 **전달 메커니즘 vs 상태관리 솔루션** — Context와 외부 라이브러리를 봐야 한다.

## 13. Context API와 Zustand/Redux 같은 외부 상태관리 라이브러리의 차이를 설명해보세요.

### 합격 답변 예시

Context는 본질적으로 **값을 트리 아래로 전달하는 메커니즘**이고, Redux/Zustand는 그 위에 스토어 모델, 구독 방식, 업데이트 패턴, 도구 생태계를 얹은 **상태관리 솔루션**입니다. Context만으로 작은 범위 공유는 충분하지만, 구독 최적화나 복잡한 도메인 상태 관리가 필요하면 외부 라이브러리가 더 적합합니다.

### 가산점 포인트

Context의 핵심 한계: **값이 바뀌면 해당 Context를 구독하는 모든 컴포넌트가 리렌더**된다. selector 기반 부분 구독이 안 된다. Zustand/Redux는 `useStore(state => state.count)`처럼 **필요한 값만 구독**할 수 있어서 리렌더 범위가 좁다.

### 상세 해설

```tsx
// Context: 값이 바뀌면 모든 소비자 리렌더
const AppContext = createContext({ user: null, theme: 'light', cart: [] });
// user만 바뀌어도 theme만 읽는 컴포넌트도 리렌더됨

// Zustand: 필요한 값만 구독
const useStore = create((set) => ({
  user: null,
  theme: 'light',
  cart: [],
  setTheme: (t) => set({ theme: t }),
}));

// theme만 구독 → user가 바뀌어도 이 컴포넌트는 리렌더 안 됨
const theme = useStore(state => state.theme);
```

| | Context | Zustand/Redux |
|---|---|---|
| **역할** | 값 전달 | 상태 관리 + 전달 |
| **구독 최적화** | 없음 (전체 리렌더) | selector 기반 부분 구독 |
| **미들웨어** | 없음 | 있음 |
| **DevTools** | 없음 | 있음 |

> Context와 외부 라이브러리를 알았으니, 다음에는 **로직 재사용 단위** — 커스텀 훅을 봐야 한다.

## 14. 커스텀 훅을 만들 때 어떤 기준으로 분리하나요?

### 합격 답변 예시

커스텀 훅은 UI를 재사용하려고 만드는 게 아니라, **상태와 effect를 포함한 로직을 재사용**하려고 분리합니다. 여러 컴포넌트에서 반복되는 구독, 폼 처리, debounce, viewport 감지 같은 로직이 있고, 그 로직이 React 훅의 조합으로 표현된다면 커스텀 훅으로 빼는 편이 좋습니다.

### 가산점 포인트

**"반환 API를 단순하게 만든다"**까지 말하면 좋다. 좋은 커스텀 훅은 내부 복잡도를 숨기고, 사용하는 쪽에 간결한 인터페이스를 제공한다.

### 상세 해설

```tsx
// 좋은 커스텀 훅: 여러 곳에서 반복되는 로직을 추출
function useDebounce<T>(value: T, delay: number): T {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debounced;
}

// 사용: 간결한 인터페이스
function SearchPage() {
  const [query, setQuery] = useState('');
  const debouncedQuery = useDebounce(query, 300);

  useEffect(() => {
    if (debouncedQuery) searchAPI(debouncedQuery);
  }, [debouncedQuery]);
}
```

#### 분리 기준

- 둘 이상의 컴포넌트에서 같은 로직이 반복된다
- 로직이 state/effect/ref 조합으로 이뤄져 있다
- UI 마크업과 로직을 분리하면 읽기 쉬워진다
- 반환값 인터페이스를 작고 예측 가능하게 만들 수 있다

> 커스텀 훅을 알았으니, 다음에는 **실패 격리** — 에러 바운더리를 봐야 한다.

## 15. 에러 바운더리는 무엇이고, 어디까지 잡을 수 있나요?

### 합격 답변 예시

에러 바운더리는 하위 트리에서 렌더링 중 발생한 오류를 잡아서 fallback UI를 보여주는 React 컴포넌트입니다. 다만 **이벤트 핸들러, 비동기 콜백, 자기 자신 내부 오류는 잡지 못합니다**.

### 가산점 포인트

에러 바운더리의 범위를 **과대평가하면 안 된다**. 이벤트 핸들러 안의 에러는 일반 try/catch로 잡아야 하고, 비동기 에러도 별도 처리가 필요하다. 에러 바운더리는 **"렌더 트리 복구용 안전장치"**이다.

### 상세 해설

```tsx
// 에러 바운더리 사용
<ErrorBoundary fallback={<p>문제가 발생했습니다</p>}>
  <Suspense fallback={<Skeleton />}>
    <UserProfile />
  </Suspense>
</ErrorBoundary>

// 잡을 수 있는 것: 렌더 중 에러
function UserProfile() {
  const data = use(fetchUser());  // 렌더 중 에러 → 에러 바운더리가 잡음
  return <p>{data.name}</p>;
}

// 잡을 수 없는 것: 이벤트 핸들러 에러
function Button() {
  function handleClick() {
    throw new Error('click error');  // 에러 바운더리가 못 잡음
    // → try/catch로 직접 처리해야 함
  }
  return <button onClick={handleClick}>Click</button>;
}
```

> 에러 바운더리를 알았으니, 다음에는 **서버 상태와 로컬 상태의 역할 분리**를 봐야 한다.

## 16. React Query 같은 서버 상태 라이브러리와 로컬 상태의 차이를 설명해보세요.

### 합격 답변 예시

서버 상태는 **서버에 원본이 있고**, 클라이언트는 가져와 캐시하고 동기화하는 값입니다. 로컬 상태는 UI 열림/닫힘, 입력 중 값처럼 **클라이언트가 직접 소유**하는 상태입니다. TanStack Query는 서버 상태의 캐싱, refetch, stale 관리를, Redux/Zustand는 클라이언트 도메인 상태를 관리합니다.

### 가산점 포인트

서버 상태의 핵심 특성: **원본 소유자가 서버**, 비동기 fetch 필요, stale/fresh 개념 있음, background sync가 중요함. 이것을 useState로 관리하면 로딩/에러/캐시/refetch를 전부 수동으로 구현해야 한다.

### 상세 해설

```tsx
// 서버 상태: TanStack Query가 관리
const { data, isLoading, error } = useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  staleTime: 5 * 60 * 1000,  // 5분간 fresh
});
// 캐시, 재시도, 백그라운드 갱신, 에러 처리를 자동으로 관리

// 로컬 상태: useState로 관리
const [isModalOpen, setIsModalOpen] = useState(false);
const [selectedTab, setSelectedTab] = useState('overview');
// UI에만 필요한 값 — 서버에 원본 없음
```

| | 서버 상태 | 로컬 상태 |
|---|---|---|
| **원본** | 서버 | 클라이언트 |
| **동기화** | fetch + cache + refetch | 즉시 |
| **stale 개념** | 있음 | 없음 |
| **도구** | TanStack Query, SWR | useState, Zustand |
| **예시** | 사용자 목록, 상품 데이터 | 모달 열림, 탭 선택 |

> 서버/로컬 상태를 알았으니, 다음에는 **복잡한 입력 상태 설계** — 폼 상태를 봐야 한다.

## 17. React에서 폼 상태가 복잡해질 때 어떤 식으로 설계하나요?

### 합격 답변 예시

폼이 복잡해질수록 먼저 **상태 구조를 단순하게 정리**하고 중복 상태를 줄이는 게 중요합니다. 필드가 많고 상태 전이가 복잡하면 `useReducer`로 이벤트 중심 전이를 정리합니다. 모든 걸 controlled로 몰아넣기보다, 실시간 동기화가 필요한 필드만 controlled로 두는 것도 실용적입니다.

### 가산점 포인트

React 공식 문서의 **state structure 원칙**을 적용하면 좋다: redundant state 제거, duplicate state 제거, 계산 가능한 값은 별도 state로 두지 않기. 예를 들어 `isValid`를 별도 state로 두지 않고 현재 값에서 계산하면 동기화 버그를 방지할 수 있다.

### 상세 해설

```tsx
// useReducer로 복잡한 폼 상태 관리
type FormState = {
  values: { name: string; email: string; bio: string };
  errors: Record<string, string>;
  touched: Record<string, boolean>;
  isSubmitting: boolean;
};

type FormAction =
  | { type: 'CHANGE'; field: string; value: string }
  | { type: 'BLUR'; field: string }
  | { type: 'SUBMIT_START' }
  | { type: 'SUBMIT_SUCCESS' }
  | { type: 'SUBMIT_ERROR'; errors: Record<string, string> };

function formReducer(state: FormState, action: FormAction): FormState {
  switch (action.type) {
    case 'CHANGE':
      return {
        ...state,
        values: { ...state.values, [action.field]: action.value },
        errors: { ...state.errors, [action.field]: '' },  // 입력 시 에러 초기화
      };
    case 'BLUR':
      return {
        ...state,
        touched: { ...state.touched, [action.field]: true },
      };
    // ...
  }
}
```

> 폼 상태를 알았으니, 다음에는 **시각적으로 복잡한 UI** — 애니메이션 구조를 봐야 한다.

## 18. 애니메이션이 많은 UI에서 React 컴포넌트 구조를 어떻게 설계해야 유지보수가 쉬워지나요?

### 합격 답변 예시

애니메이션이 많은 UI일수록 **상태와 표현을 분리**하는 게 중요합니다. 애니메이션의 source of truth가 되는 상태는 좁은 범위에 두고, 실제 애니메이션 수행은 leaf 컴포넌트에서 맡깁니다. 레이아웃 측정이 필요한 경우에만 `useLayoutEffect`와 `ref`를 쓰고, 오버레이는 Portal로 분리합니다.

### 가산점 포인트

애니메이션 UI에서 가장 흔한 문제는 **작은 시각 효과까지 상위 공통 state에 몰아넣어 전체 트리를 자주 흔드는 것**이다. 도메인 상태와 시각 효과 상태를 분리해야 한다.

### 상세 해설

```tsx
// 도메인 상태: 페이지 수준
const [activeTab, setActiveTab] = useState('overview');

// 시각 효과 상태: leaf 컴포넌트 수준
function AnimatedTab({ isActive, children }: Props) {
  // 애니메이션 상태는 이 컴포넌트 안에서만 관리
  const [isAnimating, setIsAnimating] = useState(false);

  useEffect(() => {
    setIsAnimating(true);
    const timer = setTimeout(() => setIsAnimating(false), 300);
    return () => clearTimeout(timer);
  }, [isActive]);

  return (
    <div className={isAnimating ? 'tab-enter' : 'tab-idle'}>
      {children}
    </div>
  );
}
```

#### 구조 설계 원칙

- 도메인 상태와 시각 효과 상태를 구분한다
- 애니메이션 상태는 가능하면 leaf 가까이에 둔다
- DOM 측정이 필요할 때만 useLayoutEffect
- overlay/modal/tooltip은 Portal로 분리
- 공용 transition 규칙은 래퍼 컴포넌트나 훅으로 모은다

> 애니메이션 UI를 알았으니, 다음에는 **상호작용이 복잡한 UI** — DnD 상태 구조를 봐야 한다.

## 19. DnD UI 같은 상호작용이 많은 화면에서 React 상태 구조를 어떻게 잡나요?

### 합격 답변 예시

DnD에서는 **지속되어야 하는 도메인 상태**와 **드래그 중에만 필요한 일시 상태**를 분리합니다. 실제 아이템 순서는 도메인 상태로, 현재 드래그 중인 item id, hover target, preview 좌표는 transient state로 둡니다. 복잡한 전이는 reducer로 관리합니다.

### 가산점 포인트

DnD에서 흔한 문제는 **"영구 상태와 임시 상태가 섞이는 것"**이다. 드래그 중 좌표까지 도메인 상태에 넣으면 업데이트 폭이 너무 크고, 반대로 순서 변경 로직까지 임시 상태에 두면 drop 이후 동기화가 꼬인다.

### 상세 해설

```tsx
// 도메인 상태: 실제 아이템 순서 (영구)
const [items, dispatch] = useReducer(dndReducer, initialItems);

// 인터랙션 상태: 드래그 중에만 필요 (일시)
const [dragState, setDragState] = useState<{
  draggingId: string | null;
  hoverId: string | null;
}>({ draggingId: null, hoverId: null });

// reducer로 명확한 전이 관리
function dndReducer(state: Item[], action: DndAction) {
  switch (action.type) {
    case 'REORDER':
      return reorderItems(state, action.fromIndex, action.toIndex);
    case 'MOVE_TO_GROUP':
      return moveToGroup(state, action.itemId, action.groupId);
    default:
      return state;
  }
}
```

> DnD 상태를 알았으니, 마지막으로 **기존 코드베이스를 React로 전환하는 전략**을 봐야 한다.

## 20. Vue 레거시를 React로 마이그레이션한다면, 컴포넌트 구조와 상태 흐름을 어떤 순서로 재설계하겠습니까?

### 합격 답변 예시

"템플릿을 JSX로 번역"하는 것보다 **"데이터 흐름을 React식으로 다시 설계"**하는 게 먼저입니다. 순서는: (1) 화면/도메인 단위 경계 파악, (2) 상태를 로컬/공유/서버로 분리, (3) 중복 상태 제거, (4) 가장 가까운 공통 부모로 끌어올리기, (5) 복잡한 전이는 reducer로 정리, (6) 반복 로직은 커스텀 훅으로 추출.

### 가산점 포인트

**"Vue의 반응성 모델(ref/reactive)과 React의 snapshot 모델은 근본적으로 다르다"**는 점을 인식해야 한다. Vue는 mutable reactive, React는 immutable snapshot. 1:1 번역하면 안 되고 **사고 모델 자체를 전환**해야 한다.

### 상세 해설

```
마이그레이션 순서:

1. 현재 화면을 도메인 단위로 나눈다
2. 각 화면의 상태를 "로컬 / 공유 / 서버"로 분리한다
3. 계산 가능한 값은 derived state로 남기지 않는다
4. state를 가장 가까운 owner로 이동시킨다
5. 복잡한 흐름은 reducer/store로 정리한다
6. 반복되는 side effect/구독 로직은 커스텀 훅으로 뺀다
7. 마지막에 세부 컴포넌트와 스타일 계층을 다듬는다
```

#### 실무 에피소드

> Vue 2에서 React로 마이그레이션할 때, Vuex store를 그대로 Redux로 옮기려다 실패했다. Vue의 mutation은 React의 immutable 업데이트와 다르고, computed property는 useMemo와 1:1이 아니었다. 결국 상태 구조를 처음부터 React 방식으로 재설계한 뒤에야 마이그레이션이 진행됐다.

## 보완내용

### 왜 보완이 필요한가

기존 20문항은 렌더/커밋, 상태 구조, memoization, 서버 상태와 로컬 상태 경계까지 잘 잡혀 있어서 실무 면접용으로 매우 좋다. 다만 React를 진짜로 이해한 사람은 `state snapshot`, `effect`의 본질, `key` 기반 identity처럼 렌더 모델의 핵심 개념에서 차이가 난다.

### 추가하면 좋은 질문

#### 1. React에서 state를 왜 "snapshot"이라고 하나요? stale closure와 어떻게 연결되나요?

##### 합격 답변 예시

각 렌더에서 state 값이 그 렌더 시점에 고정된 상수처럼 동작하기 때문입니다. setState를 호출해도 현재 렌더 안에서 state 변수가 바뀌지 않고, 다음 렌더에서 새 값이 적용됩니다. 이 때문에 비동기 콜백이 이전 렌더의 값을 참조하는 stale closure 문제가 발생합니다.

##### 상세 해설

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    setTimeout(() => {
      console.log(count);  // 항상 클릭 시점의 값 (stale closure)
    }, 3000);
  }

  return <button onClick={handleClick}>{count}</button>;
}
```

stale closure 해결: 의존성 배열을 정확히 채우거나, 함수형 업데이트 `setCount(c => c + 1)`을 사용.

#### 2. 왜 "You Might Not Need an Effect"라고 하나요?

##### 합격 답변 예시

props/state에서 계산 가능한 값을 useEffect로 다시 state에 넣는 패턴이 불필요하기 때문입니다. 렌더 중에 계산 가능하면 Effect 없이 바로 계산하는 것이 맞습니다.

##### 상세 해설

```tsx
// ❌ 불필요한 Effect
const [fullName, setFullName] = useState('');
useEffect(() => {
  setFullName(firstName + ' ' + lastName);
}, [firstName, lastName]);

// ✅ 렌더 중에 바로 계산
const fullName = firstName + ' ' + lastName;
```

Effect가 진짜 필요한 경우: 서버 fetch, WebSocket 구독, 브라우저 이벤트 리스너, 서드파티 초기화.

#### 3. state가 보존되는 기준은 무엇인가요?

##### 합격 답변 예시

React는 state를 **UI 트리에서의 위치와 key**를 기준으로 보존합니다. 같은 위치에 같은 타입이면 유지, 위치/타입/key가 바뀌면 초기화됩니다.

##### 상세 해설

```tsx
// key 변경으로 form 초기화
<EditForm key={selectedId} contactId={selectedId} />

// 조건부 렌더링에서 state 보존
{isA ? <Counter person="A" /> : <Counter person="B" />}
// → 같은 위치, 같은 타입 → state 유지 (주의!)

{isA ? <Counter key="A" /> : <Counter key="B" />}
// → key가 다름 → state 초기화 (의도적)
```

#### 4. Context는 상태관리 도구인가요, 전달 메커니즘인가요?

##### 합격 답변 예시

Context는 **값 전달 메커니즘**이지 상태관리 도구가 아닙니다. 상태를 만들고 갱신하는 건 useState/useReducer이고, Context는 그 값을 트리 깊은 곳까지 전달하는 역할입니다. Redux/Zustand는 상태 갱신 + 구독 최적화 + 도구까지 포함한 솔루션입니다.

##### 상세 해설

Context의 한계: 값이 바뀌면 모든 소비자가 리렌더, selector 기반 부분 구독 안 됨, 미들웨어/DevTools 없음.

#### 5. derived state를 언제 피해야 하나요?

##### 합격 답변 예시

기존 state/props에서 계산 가능한 값을 별도 state로 중복 저장하면 동기화 버그가 생기기 쉽습니다. 계산이 비싸면 useMemo로 캐싱하되, state를 이중으로 두지 않는 것이 안전합니다.

##### 상세 해설

```tsx
// ❌ redundant state
const [items, setItems] = useState([]);
const [totalPrice, setTotalPrice] = useState(0);  // items에서 계산 가능!

// ✅ 렌더 중 계산
const totalPrice = items.reduce((sum, i) => sum + i.price, 0);
```

#### 6. useRef는 왜 필요한가요?

##### 합격 답변 예시

useRef는 **리렌더를 일으키지 않으면서 렌더 사이에 값을 유지**하는 escape hatch입니다. DOM 참조, 타이머 ID, 이전 값 추적에 사용합니다. state는 변경 시 리렌더, ref는 변경해도 리렌더 안 됨.

##### 상세 해설

```tsx
function StopWatch() {
  const [elapsed, setElapsed] = useState(0);
  const intervalRef = useRef<number | null>(null);  // 화면에 안 보임 → ref

  function handleStart() {
    intervalRef.current = setInterval(() => setElapsed(e => e + 1), 1000);
  }
  function handleStop() {
    clearInterval(intervalRef.current!);
  }
  return <p>{elapsed}초</p>;
}
```

핵심: "화면에 보여야 하는 값"은 state, "렌더와 무관한 값"은 ref.

### 이 섹션 보완 핵심

React 섹션은 이미 훌륭하다. 여기에 snapshot, effect 최소화, identity, Context의 역할, derived state, useRef까지 더하면 훅 사용 경험자와 React 모델 이해자를 더 명확하게 구분할 수 있다.
