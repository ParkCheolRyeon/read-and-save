# 03. react20문항

> React 렌더링 모델, 상태 구조, 성능 최적화 감각을 점검하는 20문항입니다.

## 1. React가 선언형이라고 하는 이유를 설명해보세요.

### 합격 답변 예시

React가 선언형이라고 하는 이유는, 개발자가 DOM(Document Object Model)을 한 줄씩 직접 조작하는 대신 현재 상태에서 UI(User Interface)가 어떤 모습이어야 하는지 선언하면 React가 그 결과를 계산하고 실제 DOM 반영을 맡기기 때문입니다. 즉 “어떻게 바꿀지”보다 “무엇이 보여야 하는지”에 집중하게 해줍니다. 상태가 바뀌면 컴포넌트는 같은 입력에 대해 같은 JSX를 반환해야 하고, React는 그 결과를 바탕으로 필요한 최소 DOM 변경만 커밋합니다.

### 가산점 포인트

“선언형이 가능하려면 render가 순수해야 한다”까지 말하면 좋다.

### 상세 해설

React는 컴포넌트를 입력(props, state, context) → 출력(JSX(JavaScript XML)) 으로 보는 모델에 가깝다. 공식 문서도 컴포넌트와 훅은 같은 입력에 대해 같은 출력을 내는 순수성(idempotency)을 지켜야 React가 예측 가능하게 렌더하고 최적화할 수 있다고 설명한다. 렌더 단계에서는 다음 UI를 계산하고, 커밋 단계에서 최소 DOM 변경만 반영한다.

#### 비유

선언형 React는 식당에서 “이 요리를 주세요”라고 주문하는 방식에 가깝다. 주방에서 팬을 몇 번 흔들고 불을 몇 초 켤지는 내가 직접 지시하지 않는다. 나는 결과만 선언하고, 내부 조리는 React가 맡는다. React 공식 문서도 렌더와 커밋을 요리/서빙 비유로 설명한다.

## 2. React의 렌더링 과정과 커밋 과정을 설명해보세요.

### 합격 답변 예시

React 업데이트는 크게 렌더(render) 와 커밋(commit) 으로 나눌 수 있습니다. 렌더 단계에서는 React가 컴포넌트 함수를 호출해서 다음 UI가 어떤 모습이어야 하는지 계산하고, 이전 계산과 비교할 준비를 합니다. 커밋 단계에서는 실제로 필요한 최소 DOM 변경을 적용하고, 그 이후 브라우저가 화면을 그리며, 일반적인 Effect는 커밋 이후 실행됩니다.

### 가산점 포인트

“렌더가 일어났다고 항상 DOM 변경이 생기는 건 아니다”라고 말하면 좋다.

### 상세 해설

공식 문서 기준 흐름은 이렇다. Trigger a render 초기 렌더이거나 state update 등으로 다시 렌더가 요청된다. Render React가 컴포넌트들을 호출해 “다음 UI 계산”을 만든다. Commit 필요한 DOM 변경만 실제로 반영한다. 그 다음 브라우저가 paint하고, Effects가 실행된다. 핵심은 렌더 = 계산, 커밋 = 반영이라는 점이다. 많은 지원자가 render를 곧 DOM 조작으로 생각하는데, 그건 틀리다. 렌더는 “무엇을 보여줄지 계산하는 단계”이고, 실제 DOM 변경은 커밋에서 일어난다. React docs도 렌더 후 이전 계산과 비교해 최소 변경만 커밋한다고 설명한다.

## 3. Virtual DOM은 무엇이고, 왜 필요한가요?

### 합격 답변 예시

보통 Virtual DOM이라고 부르는 것은 React가 렌더 단계에서 계산하는 다음 UI의 메모리상 표현이라고 볼 수 있습니다. React는 이전 렌더 결과와 다음 렌더 결과를 비교해서 실제 DOM에는 필요한 최소 변경만 반영합니다. 핵심은 “DOM을 직접 여기저기 수정하는 명령형 방식” 대신, 컴포넌트 출력 전체를 계산한 뒤 변경분만 커밋할 수 있게 해주는 추상화라는 점입니다.

### 상세 해설

공식 문서가 요즘 “Virtual DOM”이라는 용어를 중심에 놓지는 않지만, 렌더와 커밋 문서가 설명하는 구조가 바로 그 개념의 핵심이다. React는 렌더 단계에서 다음 UI를 계산하고, 이전 계산과 비교한 뒤, 실제 DOM에는 최소 변경만 적용한다. 이 방식 덕분에 개발자는 DOM 조작 절차 대신 UI 선언에 집중할 수 있다. 중요한 건 Virtual DOM을 “무조건 빠르게 만드는 마법”으로 말하면 안 된다는 점이다. 본질은 성능 그 자체보다 추상화와 일관성이다. 필요하다면 DOM 업데이트를 더 적게 할 수 있고, React가 렌더 모델을 유지하면서 최적화도 할 수 있다.

## 4. 리렌더링이 발생하는 조건을 설명해보세요.

### 합격 답변 예시

React 컴포넌트는 초기 렌더 때 한 번 실행되고, 이후에는 주로 state가 바뀌거나, 부모가 다시 렌더되거나, 사용하는 context 값이 바뀌면 다시 렌더됩니다. 외부 스토어를 useSyncExternalStore 같은 방식으로 구독하는 경우에는 스냅샷 값이 바뀌었을 때도 리렌더됩니다. 다만 리렌더가 일어났다고 해서 항상 실제 DOM 변경이 생기는 것은 아니고, 커밋 단계에서 변경이 없으면 DOM은 그대로일 수 있습니다.

### 상세 해설

React 공식 문서는 state update가 새로운 렌더를 큐잉한다고 설명하고, 기본적으로 부모가 렌더되면 자식도 재귀적으로 렌더된다고 설명한다. 또 memo 문서는 부모가 렌더되면 자식도 보통 렌더되지만, 메모이제이션으로 같은 props면 건너뛸 수 있다고 말한다. useSyncExternalStore는 스토어 snapshot이 바뀌었을 때만 리렌더하도록 계약을 제공한다.

## 5. 부모가 리렌더링되면 자식은 항상 리렌더링되나요?

### 합격 답변 예시

기본적으로는 그렇습니다. React는 부모가 다시 렌더되면 자식도 재귀적으로 다시 렌더합니다. 하지만 자식이 React.memo로 감싸져 있고 props가 이전과 같다면 그 자식은 리렌더를 건너뛸 수 있습니다. 즉 기본값은 “부모 리렌더 → 자식도 리렌더”이고, 최적화가 들어가면 예외가 생깁니다.

### 상세 해설

공식 useMemo 문서는 “컴포넌트가 리렌더되면 자식들도 기본적으로 재귀적으로 리렌더된다”고 아주 명확하게 말한다. 그리고 memo 문서는 props가 같다면 부모 리렌더에도 자식이 다시 렌더되지 않을 수 있다고 설명한다. 면접에서 중요한 건 여기서 “항상 아니다”라고만 끝내지 않는 것이다. 왜 자식이 다시 렌더되었는지를 말할 수 있어야 한다.

- 부모가 렌더되면 기본적으로 자식도 렌더
- 단, memo가 있고 props가 같으면 스킵 가능
- context를 쓰면 context 변경 시 다시 렌더
- props가 객체/함수라 매번 새 참조면 memo가 무력화될 수 있음

## 6. useState의 비동기적 업데이트 특성을 설명해보세요.

### 합격 답변 예시

setState를 호출한다고 현재 렌더 안에서 바로 state 변수가 바뀌는 것은 아닙니다. React는 현재 렌더의 state를 “snapshot”처럼 취급하고, setState는 다음 렌더를 예약합니다. 그래서 같은 이벤트 핸들러 안에서 setCount(count + 1)를 여러 번 호출하면 현재 snapshot 기준으로 계산될 수 있고, 여러 번 누적하려면 함수형 업데이트를 써야 합니다. React는 여러 state update를 batching해서 처리하기도 합니다.

### 가산점 포인트

“비동기”라는 말보다 “즉시 현재 변수 값을 바꾸지 않고 다음 렌더를 예약한다”라고 설명하면 더 정확하다.

### 상세 해설

React는 state를 “평범한 mutable 변수”처럼 다루지 않는다. 공식 문서도 state는 snapshot처럼 동작하고, setState는 다음 렌더를 일으킬 뿐 현재 핸들러 안의 state 값을 바로 바꾸지 않는다고 설명한다. 또 여러 업데이트는 batching될 수 있다.

```ts
setCount(count + 1);
setCount(count + 1);
```

이 코드는 현재 렌더의 같은 count를 두 번 읽기 때문에 기대한 누적이 안 될 수 있다. 반면

```ts
setCount((c) => c + 1);
setCount((c) => c + 1);
```

이렇게 함수형 업데이트를 쓰면 큐에 쌓인 이전 결과를 기준으로 안전하게 누적할 수 있다. 공식 문서도 같은 state를 연속 갱신할 때 updater function을 권장한다.

## 7. useEffect와 useLayoutEffect의 차이를 설명해보세요.

### 합격 답변 예시

useEffect는 컴포넌트를 외부 시스템과 동기화하기 위한 훅이고, 보통 브라우저가 화면을 그린 뒤 실행됩니다. 다만 상호작용으로 발생한 업데이트에서는 paint 이전에 실행될 수도 있습니다. 반면 useLayoutEffect는 브라우저가 repaint하기 전에 실행되기 때문에, 레이아웃 측정이나 DOM 읽기/쓰기처럼 화면 깜빡임 없이 처리해야 하는 경우에 씁니다. 그래서 기본값은 useEffect이고, 정말 레이아웃 측정 타이밍이 필요할 때만 useLayoutEffect를 쓰는 편이 좋습니다. React 공식 문서도 useLayoutEffect는 성능에 부담이 될 수 있어 가능하면 useEffect를 선호하라고 안내합니다.

### 상세 해설

useEffect는 본질적으로 React 바깥과의 동기화용이다. 예를 들어 소켓 연결, 구독 등록, DOM 외부 라이브러리 초기화 같은 경우다. 공식 문서도 Effect는 “외부 시스템”과 동기화할 때 쓴다고 설명한다. 또한 상호작용이 아닌 일반적인 경우에는 브라우저가 paint를 마친 뒤 실행되는 쪽이 일반적이지만, 상호작용 때문에 발생한 Effect는 paint 이전에 실행될 수도 있다고 안내한다. useLayoutEffect는 repaint 전에 실행되므로, tooltip 위치 측정처럼 “먼저 DOM을 읽고 그 결과를 반영한 뒤 paint되어야” 하는 경우에 맞다. 공식 문서도 useLayoutEffect의 목적을 “layout information for rendering”으로 설명한다. 다만 paint를 막을 수 있어 남용하면 성능에 좋지 않다.

## 8. useMemo와 useCallback은 언제 써야 하고, 남용하면 왜 안 되나요?

### 합격 답변 예시

useMemo는 비싼 계산 결과나 참조 동일성이 중요한 값을 캐싱할 때 쓰고, useCallback은 함수 자체를 캐싱해서 memoized child나 dependency 안정화가 필요할 때 씁니다. 하지만 둘 다 공짜가 아니고, dependency 관리와 메모리 유지 비용이 있습니다. 실제로 느린 렌더를 줄이는 데 기여하지 않는다면 오히려 코드 복잡도만 올리고 stale dependency 문제를 만들 수 있어서, 성능 병목이 확인되었거나 참조 안정성이 실제로 필요한 경우에만 쓰는 게 좋습니다. 공식 문서도 메모이제이션은 특정 목적이 있을 때 사용하라고 설명합니다.

### 가산점 포인트

현재 공식 문서는 React Compiler가 수동 메모이제이션 필요성을 줄일 수 있다고도 설명하므로, 그 점을 알고 있으면 더 좋다.

### 상세 해설

useMemo는 계산 결과를 캐싱한다. 대표적인 목적은 두 가지다.

- 계산 자체가 비쌀 때
- 자식에게 넘기는 값의 참조를 안정화해서 memo가 의미 있게 작동하도록 할 때

useCallback은 함수 참조를 안정화한다. 주로 아래와 같은 경우에 사용한다.

- memo된 자식에게 callback을 prop으로 전달할 때
- effect dependency로 들어가는 함수를 안정화해야 할 때

React 공식 docs와 eslint exhaustive-deps 문서는 의존성 누락이 stale closure를 만들 수 있다고 설명한다.

#### 남용하면 왜 안 좋나?

- dependency 배열 관리가 복잡해진다
- stale closure 위험이 커진다
- 코드가 읽기 어려워진다
- 실제 이득이 없는 곳에도 기계적으로 써서 오히려 유지보수가 나빠진다

## 9. React.memo는 어떤 경우에 효과가 있고, 어떤 경우엔 효과가 거의 없나요?

### 합격 답변 예시

React.memo는 부모가 자주 리렌더되더라도 자식의 props가 같다면 자식 렌더를 건너뛰게 해줄 수 있어서, 비용이 큰 자식 컴포넌트에서 특히 효과가 있습니다. 하지만 props가 매번 새 객체, 새 배열, 새 함수로 만들어지면 비교 결과가 달라져서 효과가 거의 없어질 수 있습니다. 또 자식 렌더 비용이 작은데 비교 비용만 늘어나면 오히려 의미가 약합니다. 그래서 memo는 “자주 리렌더되고, props가 안정적이며, 실제 렌더 비용이 있는 컴포넌트”에 써야 효과적입니다.

### 상세 해설

공식 문서는 memo를 “부모가 리렌더돼도 props가 같으면 자식 리렌더를 건너뛰게 하는 최적화”로 설명한다. 하지만 useMemo 문서 예시처럼 자식에게 넘기는 배열이 매번 새로 생성되면 memo가 제대로 이득을 못 본다. 그래서 종종 memo와 useMemo/useCallback이 함께 쓰인다. 즉 memo 자체가 마법이 아니라, props 안정성과 함께 봐야 한다. 그리고 React 공식 문서도 React Compiler가 이런 수동 최적화 필요성을 일부 줄일 수 있다고 안내한다.

## 10. Controlled Component와 Uncontrolled Component의 차이를 설명해보세요.

### 합격 답변 예시

Controlled Component는 입력값의 실제 source of truth가 React state에 있는 경우이고, value와 onChange로 값을 제어합니다. Uncontrolled Component는 DOM이 값을 자체적으로 들고 있고, React는 defaultValue나 ref를 통해 초기값이나 읽기만 관여하는 방식입니다. 제어가 필요한 검증, 동기화, 다른 상태와의 결합이 중요하면 controlled가 유리하고, 단순 폼이나 브라우저 기본 동작을 더 살리고 싶으면 uncontrolled가 더 단순할 수 있습니다. React 공식 문서도 input, select, textarea가 controlled와 uncontrolled 사이를 중간에 바꿀 수 없고, controlled라면 일관되게 value와 onChange를 써야 한다고 설명합니다.

### 상세 해설

React 공식 <input> 문서는 value/checked를 주면 controlled input이 되고, 이 경우 onChange로 값을 갱신해야 한다고 설명한다. 반대로 defaultValue/defaultChecked는 uncontrolled 초기값이다. select도 동일한 원리다. 그리고 controlled와 uncontrolled를 생명주기 중간에 왔다 갔다 하면 안 된다고 명시한다. 또 React는 controlled/uncontrolled를 폼에만 한정하지 않고, 더 일반적으로 “부모 props가 제어하는가 / 내부 state가 제어하는가”의 관점으로도 설명한다.

## 11. key를 왜 써야 하는지, index를 key로 쓰면 어떤 문제가 생기는지 설명해보세요.

### 합격 답변 예시

key는 React가 리스트 항목의 정체성(identity) 을 추적해서, 어떤 항목이 유지되고 어떤 항목이 새로 생기거나 이동했는지를 판단하게 해줍니다. 이게 중요한 이유는 key가 단순 경고 제거용이 아니라, state 보존/리셋 기준과 연결되기 때문입니다. index를 key로 쓰면 항목 삽입, 삭제, 재정렬 시 실제 데이터와 컴포넌트 state가 엇갈릴 수 있어서 입력값이 다른 항목으로 붙거나, 선택 상태가 이상하게 이동하는 문제가 생길 수 있습니다. React 공식 문서도 key는 항목 순서를 추적하는 데 필요하고, state 보존/리셋과 직접 연결된다고 설명합니다.

### 상세 해설

React는 state를 단순히 “컴포넌트 이름”에 붙여두지 않고, UI 트리에서의 위치와 key를 기준으로 연결한다. 그래서 key가 바뀌면 같은 컴포넌트 타입이라도 다른 인스턴스로 보고 state를 리셋할 수 있다. 공식 문서는 key를 바꿔 form state를 강제로 초기화하는 예시도 보여준다.

#### 왜 index가 위험한가?

예를 들어 리스트 두 번째 항목 앞에 새 항목이 삽입되면 뒤 항목들의 index가 모두 바뀐다. 그러면 React 입장에선 “같은 데이터가 이동한 것”이 아니라 “각 자리의 항목이 다른 것”처럼 볼 수 있어 state가 잘못 매칭될 수 있다. 공식 리스트 렌더링 문서도 key가 항목의 안정적인 정체성을 표현해야 한다고 강조한다.

## 12. 상태 끌어올리기와 전역 상태 사용의 경계를 어떻게 판단하나요?

### 합격 답변 예시

기본 원칙은 먼저 가장 가까운 공통 부모로 상태를 올리는 것입니다. 두 컴포넌트가 같은 값을 공유해야 하면 그 값의 source of truth를 공통 부모에 두는 게 가장 단순합니다. 그런데 공유 범위가 너무 넓어져서 여러 페이지나 먼 트리까지 계속 props를 전달해야 하거나, 도메인 단위로 공통 접근이 필요해지면 그때 Context나 외부 상태관리로 올릴지를 판단합니다. 즉 먼저 로컬하게 두고, 실제 공유 범위와 변경 패턴이 커졌을 때 전역화를 검토하는 편이 좋습니다. React 문서도 먼저 closest common parent로 state를 끌어올리는 패턴을 기본으로 설명합니다.

### 상세 해설

React는 state sharing의 기본 해법으로 lifting state up을 제시한다. 두 컴포넌트가 함께 바뀌어야 한다면 상태를 공통 부모로 올리고 props와 이벤트 핸들러로 제어하라는 것이다. 전역 상태는 언제 필요한가? 내 기준은 이렇다.

- 여러 멀리 떨어진 하위 트리가 자주 읽는다
- props drilling이 구조적으로 과해진다
- 페이지 전환 후에도 유지돼야 한다
- 도메인 규칙을 한 곳에서 관리하는 게 낫다
- undo/redo, devtools, middleware 같은 추가 기능이 필요하다

즉 공유 범위와 변경 복잡도가 경계를 결정한다. React의 managing state 문서도 앱이 커질수록 state 구조와 공유 방식을 더 의도적으로 설계해야 한다고 설명한다.

## 13. Context API와 Zustand/Redux 같은 외부 상태관리 라이브러리의 차이를 설명해보세요.

### 합격 답변 예시

Context API는 본질적으로 값을 트리 아래로 전달하는 메커니즘에 가깝고, Redux나 Zustand는 그 위에 스토어 모델, 구독 방식, 업데이트 패턴, 도구 생태계를 얹은 상태관리 솔루션에 가깝습니다. Redux는 예측 가능한 데이터 흐름, 중앙 store, middleware, devtools 같은 강한 구조가 장점이고, Zustand는 비교적 가볍고 hook 기반 API로 빠르게 도입하기 쉬운 편입니다. 반면 Context만으로도 작은 범위 공유는 충분하지만, 구독 최적화나 복잡한 상태 규칙, 디버깅 도구, 대규모 도메인 분리가 필요해지면 외부 라이브러리가 더 적합할 수 있습니다. Redux FAQ도 Context와 Redux는 목적이 겹치는 부분이 있지만 차이가 있고, React-Redux는 내부적으로 Context를 사용한다고 설명합니다. Zustand 공식 문서도 자신을 작고 빠르고 확장 가능한 state management solution으로 소개합니다.

### 상세 해설

Context는 “깊은 트리로 값을 전달하는 문제”를 해결해준다. 하지만 그것만으로는 아래와 같은 문제를 전부 해결해주지 않는다.

- 상태 갱신 규칙
- 구독 단위
- 미들웨어
- 타임트래블/디버깅
- 도메인 분리

Redux FAQ도 두 도구가 모두 prop drilling 완화에는 쓰이지만, Redux는 상태 관리 전반에 대한 더 강한 구조와 툴링을 제공한다고 설명한다. Zustand는 Redux보다 보일러플레이트가 적고 hook 기반으로 쓰기 쉬운 편이라, 중간 규모 앱에서 가볍게 선택되는 경우가 많다. 반면 Redux는 크고 복잡한 상태 흐름을 명시적으로 관리하고 싶을 때 여전히 강하다.

## 14. 커스텀 훅을 만들 때 어떤 기준으로 분리하나요?

### 합격 답변 예시

커스텀 훅은 UI를 재사용하려고 만드는 게 아니라, 상태와 effect를 포함한 로직을 재사용하려고 분리합니다. 즉 여러 컴포넌트에서 반복되는 구독, 폼 처리, 서버 호출, debounce, viewport 감지 같은 로직이 있고, 그 로직이 React 훅의 조합으로 표현된다면 커스텀 훅으로 빼는 편이 좋습니다. 반대로 한 컴포넌트에서만 쓰이고 문맥이 강하게 묶인 로직이라면 너무 일찍 분리하지 않는 편이 가독성에 더 좋습니다. React 공식 문서도 커스텀 훅은 애플리케이션에 필요한 reusable logic을 공유하는 수단이라고 설명합니다.

### 가산점 포인트

“훅을 분리할 때 반환 API를 단순하게 만든다”까지 말하면 좋다.

### 상세 해설

좋은 분리 기준은 보통 이렇다.

- 둘 이상의 컴포넌트에서 같은 로직이 반복된다
- 로직이 state/effect/ref 조합으로 이뤄져 있다
- UI 마크업과 로직을 분리하면 읽기 쉬워진다
- 반환값 인터페이스를 작고 예측 가능하게 만들 수 있다

React 문서는 custom hook의 목적을 “logic sharing”으로 설명한다. 훅은 여전히 Hooks 규칙을 따라야 하므로, 그냥 일반 함수를 분리하는 것과는 다르다.

## 15. 에러 바운더리는 무엇이고, 어디까지 잡을 수 있나요?

### 합격 답변 예시

에러 바운더리는 하위 트리에서 렌더링 중 발생한 오류를 잡아서 fallback UI를 보여주고, 필요하면 로깅까지 할 수 있는 React 컴포넌트입니다. 공식적으로는 class component에서 getDerivedStateFromError나 componentDidCatch로 구현합니다. 다만 에러 바운더리는 이벤트 핸들러 내부 오류, 비동기 콜백 오류, 자기 자신 내부 오류까지 다 잡아주는 만능 try/catch는 아닙니다. React 공식 문서와 레거시 에러 바운더리 문서도 descendant render error를 다루는 용도이며 event handler/async는 범위 밖이라고 설명합니다.

### 상세 해설

에러 바운더리는 “화면 트리 복구용 안전장치”에 가깝다. 하위 트리 렌더 중 예외가 나면 전체 앱을 깨뜨리지 않고 특정 영역만 fallback으로 대체할 수 있다. componentDidCatch는 로깅 용도로도 자주 쓰인다. 하지만 범위를 과대평가하면 안 된다. 레거시 공식 문서는 에러 바운더리가 이벤트 핸들러 내부 에러나 비동기 코드 에러를 잡지 않는다고 명시한다. 그런 경우는 일반 try/catch나 별도 에러 처리 전략이 필요하다.

## 16. React Query 같은 서버 상태 라이브러리와 로컬 상태의 차이를 설명해보세요.

### 합격 답변 예시

서버 상태는 클라이언트가 소유한 값이라기보다 서버에 원본이 있고, 클라이언트는 그것을 가져와 캐시하고 동기화하는 값입니다. 반면 로컬 상태는 UI 열림/닫힘, 입력 중 값, 탭 선택, 드래그 중 좌표처럼 클라이언트가 직접 소유하고 즉시 제어하는 상태입니다. TanStack Query는 자신을 서버 상태/비동기 상태 관리 도구로 설명하고, query key 기반 캐싱, refetch, stale 관리, background update 같은 문제를 해결합니다. 반대로 Redux나 Zustand 같은 client-state 도구는 UI나 클라이언트 도메인 상태를 관리하는 데 더 가깝습니다.

### 상세 해설

서버 상태의 특징은 보통 이렇다.

- 원본 소유자가 서버다
- 비동기 fetch가 필요하다
- stale/fresh 개념이 있다
- 재시도, refetch, background sync가 중요하다
- 캐시 정책이 핵심이다

TanStack Query 공식 문서는 query를 비동기 데이터 소스에 대한 선언적 의존성이라고 설명하고, 자신을 서버 상태 유틸리티로 소개한다. 또 기존 client-state 라이브러리와는 역할이 다르다고 명시한다. 반대로 로컬 상태는 UI 로직에 밀접하다. 예를 들어 modal open 여부, hover state, form dirty 여부 등은 서버 상태 라이브러리로 다루기보다 컴포넌트 로컬이나 client-state store가 더 맞다.

## 17. React에서 폼 상태가 복잡해질 때 어떤 식으로 설계하나요?

### 합격 답변 예시

폼이 복잡해질수록 먼저 상태 구조를 단순하게 정리하고, 중복 상태를 줄이는 게 중요합니다. 필드가 많고 상태 전이가 복잡하면 useReducer로 이벤트 중심 전이를 정리하고, 입력값·에러·dirty·touched 같은 개념을 명확히 분리하는 편이 좋습니다. 또 모든 걸 무조건 controlled로 몰아넣기보다, 실시간 동기화가 필요한 필드만 controlled로 두고, 단순 초기값/제출 중심이면 uncontrolled나 브라우저 기본 <form> 흐름을 일부 활용하는 것도 실용적입니다. React 문서도 state structure를 신중히 고르고, 복잡한 상태 로직은 reducer로 옮기라고 설명합니다.

### 상세 해설

React 공식 문서는 좋은 state structure를 위해 redundant state와 duplicate state를 피하라고 강조한다. 폼에서 흔한 실수는 이미 계산 가능한 값을 또 state로 들고 있는 것이다. 예를 들면 isValid를 매 키 입력마다 별도 state로 들기보다, 현재 값과 에러 목록에서 계산할 수 있으면 계산하는 편이 낫다. 복잡한 폼에서 reducer가 유리한 이유는 입력 이벤트, validation 이벤트, submit 이벤트를 액션 단위로 정리할 수 있기 때문이다. React 공식 문서도 state logic이 복잡해지면 reducer로 추출하는 패턴을 제시한다.

## 18. 애니메이션이 많은 UI에서 React 컴포넌트 구조를 어떻게 설계해야 유지보수가 쉬워지나요?

### 합격 답변 예시

애니메이션이 많은 UI일수록 상태와 표현을 분리하는 게 중요합니다. 애니메이션의 source of truth가 되는 상태는 가능한 좁은 범위에 두고, 실제 애니메이션 수행은 leaf 컴포넌트나 전용 프레젠테이션 계층에서 맡기는 편이 유지보수에 유리합니다. 레이아웃 측정이 필요한 경우에만 useLayoutEffect와 ref를 쓰고, 그렇지 않으면 렌더를 순수하게 유지하는 것이 중요합니다. 오버레이 성격의 애니메이션은 Portal로 구조를 분리하는 것도 유용합니다. React 문서도 render purity, layout measurement 용도의 useLayoutEffect, DOM escape 용도의 portal을 각각 설명합니다.

### 상세 해설

애니메이션 UI에서 구조가 망가지는 가장 흔한 원인은, 작은 시각 효과까지 상위 공통 state에 몰아넣어 전체 트리를 자주 흔드는 것이다. React 관점에서는 render가 순수해야 하고, 애니메이션처럼 DOM 측정이나 imperative 제어가 필요한 부분만 escape hatch(ref, layout effect, portal)를 써야 한다. 내가 면접에서 듣고 싶은 포인트는 대략 이렇다.

- 도메인 상태와 시각 효과 상태를 구분한다
- 애니메이션 상태는 가능하면 leaf 가까이에 둔다
- 측정이 필요할 때만 useLayoutEffect
- overlay/modal/tooltip은 Portal로 분리
- 공용 transition 규칙은 래퍼 컴포넌트나 훅으로 모은다

이렇게 답하면 실무 감각이 있다.

## 19. DnD UI 같은 상호작용이 많은 화면에서 React 상태 구조를 어떻게 잡나요?

### 합격 답변 예시

DnD 화면에서는 보통 지속되어야 하는 도메인 상태와 드래그 중에만 필요한 일시 상태를 분리합니다. 예를 들어 실제 아이템 순서나 그룹 배치는 도메인 상태로 두고, 현재 드래그 중인 item id, hover target, preview 좌표 같은 값은 transient interaction state로 둡니다. 또 리스트가 복잡할수록 배열 전체를 계속 깊게 바꾸기보다 id 기반 구조와 reducer를 써서 전이 규칙을 명확히 관리하는 편이 좋습니다. React 문서도 state structure를 신중히 정하고, 복잡한 업데이트 로직은 reducer로 분리하라고 설명합니다.

### 상세 해설

DnD에서 흔한 문제는 “영구 상태와 임시 상태가 섞이는 것”이다. 예를 들면 드래그 중 좌표까지 글로벌 도메인 상태에 넣으면 업데이트 폭이 너무 커지고, 반대로 실제 순서 변경 로직까지 임시 상태에 두면 drop 이후 동기화가 꼬이기 쉽다. React의 state structure 원칙은 이런 중복과 얽힘을 줄이는 데 도움이 된다. 또 업데이트 규칙이 복잡하면 useReducer가 좋다. dragStart, dragOver, drop, cancel 같은 액션을 명시적으로 분리하면 디버깅이 쉬워지고, 복잡한 배열/객체 업데이트도 더 관리하기 쉬워진다. React 문서도 상태 로직이 복잡할수록 reducer 패턴이 유리하다고 설명한다.

## 20. Vue 레거시를 React로 마이그레이션한다면, 컴포넌트 구조와 상태 흐름을 어떤 순서로 재설계하겠습니까?

### 합격 답변 예시

마이그레이션에서는 먼저 UI를 그대로 옮기려 하기보다, 현재 화면의 source of truth와 상태 흐름을 먼저 재정의하는 편이 좋습니다. 순서는 보통 1) 화면/도메인 단위 경계 파악, 2) 어떤 상태가 로컬인지 공유인지 서버 상태인지 분리, 3) 중복 상태 제거, 4) 가장 가까운 공통 부모로 끌어올릴 상태와 전역화할 상태를 결정, 5) 복잡한 전이는 reducer나 store로 정리, 6) 반복 로직은 custom hook으로 추출하는 순서로 봅니다. 즉 “템플릿을 JSX로 번역”하는 것보다 “데이터 흐름을 React식으로 다시 설계”하는 게 먼저입니다. React 문서도 앱이 커질수록 state structure, lifting state up, reducer, context 같은 축으로 의도적으로 재설계해야 한다고 설명합니다.

### 상세 해설

좋은 마이그레이션 순서는 보통 이렇다.

- 현재 화면을 도메인 단위로 나눈다
- 각 화면의 상태를 “로컬 / 공유 / 서버”로 분리한다
- 계산 가능한 값은 derived state로 남기지 않는다
- state를 가장 가까운 owner로 이동시킨다
- 복잡한 흐름은 reducer/store로 정리한다
- 반복되는 side effect/구독 로직은 custom hook으로 뺀다
- 마지막에야 세부 컴포넌트와 스타일 계층을 다듬는다

React의 managing state 문서가 말하는 핵심도 결국 비슷하다. 상태 구조를 잘 잡고, 공유는 끌어올리고, 멀리 전달해야 하면 context를 검토하고, 복잡한 상태 전이는 reducer로 옮기라는 흐름이다.

## 5. 면접관 기준으로 이 React 20문항을 더 강하게 운영하는 방법

질문이 좋아도 운영이 약하면 변별력이 떨어진다. 내가 실제 면접관이라면 이렇게 꼬리질문을 붙인다.

### 방식 1) 렌더 모델 중심으로 엮기

예를 들어 이런 흐름으로 간다.

- React가 선언형이라는 게 무슨 뜻이죠?
- 그럼 render와 commit은 어떻게 다르죠?
- re-render가 나도 DOM 업데이트가 항상 생기나요?
- state snapshot이라는 말은 왜 나오죠?
- 그래서 stale closure는 어떻게 생기죠?

이렇게 묶으면 진짜 이해도를 볼 수 있다. React 공식 문서는 이 개념들을 하나의 흐름으로 연결해 설명한다.

### 방식 2) key/state identity를 깊게 판다

- key가 왜 identity와 연결되죠?
- index key가 왜 위험하죠?
- form reset에 key를 어떻게 쓰죠?
- 같은 컴포넌트인데 왜 state가 초기화되죠?

이 질문 흐름은 React를 “컴포넌트 이름”이 아니라 “트리 위치와 identity”로 이해하는지 잘 드러낸다.

### 방식 3) effect와 state 구조를 같이 본다

- 왜 effect가 필요한가요?
- effect 없이 계산 가능한 건 뭐가 있죠?
- 복잡한 폼에서 reducer로 가는 기준은?
- Context로 충분한가요, store가 필요한가요?

이렇게 연결하면 React를 “훅 모음집”이 아니라 “상태 설계 모델”로 이해하는지 판단하기 좋다.

## 6. 최종 평가

이 React 20문항은 좋다. 상당히 좋다. 특히 3년차 프론트엔드 면접 기준으로는 단순히 “React 써봤다”와 “React를 이해하고 구조를 설계할 수 있다”를 꽤 잘 가른다. 핵심은 결국 이 5개다.

- 렌더와 커밋을 구분할 수 있는가
- state가 snapshot이고 identity에 묶인다는 걸 이해하는가
- effect를 외부 시스템 동기화로 이해하는가
- memoization을 성능 도구로 적절히 판단하는가
- 로컬/전역/서버 상태 경계를 실무적으로 설계할 수 있는가
- 원하면 다음 단계로 이어서

이 React 20문항을 실제 면접에서 1분~1분 30초 분량으로 말할 수 있는 구어체 답변 스크립트 버전으로 바로 정리해주겠다.

## 보완내용

### 왜 보완이 필요한가

기존 20문항은 렌더/커밋, 상태 구조, memoization, 서버 상태와 로컬 상태 경계까지 잘 잡혀 있어서 실무 면접용으로 매우 좋다. 다만 React를 진짜로 이해한 사람은 결국 `state snapshot`, `effect`의 본질, `key` 기반 identity처럼 렌더 모델의 핵심 개념에서 차이가 난다.

훅을 써본 경험과 React 모델을 이해한 수준을 더 분명하게 구분하려면, 아래 질문처럼 “왜 그렇게 설계되었는가”를 묻는 문항을 추가하는 편이 좋다.

### 추가하면 좋은 질문

#### 1. React에서 state를 왜 “snapshot”이라고 하나요? stale closure와 어떻게 연결되나요?

이 질문은 `useState`, 이벤트 핸들러, 비동기 콜백, 의존성 배열 문제를 한 번에 묶어 설명하게 만든다. React를 값 저장소가 아니라 렌더 시점의 스냅샷 모델로 이해하는지 보기 좋다.

##### 합격 답변 예시

React에서 state를 snapshot이라고 부르는 이유는, 각 렌더에서 state 값이 그 렌더 시점에 고정된 상수처럼 동작하기 때문입니다. setState를 호출해도 현재 렌더 안에서 state 변수가 바뀌지 않고, 다음 렌더에서 새 값이 적용됩니다. 이 때문에 이벤트 핸들러나 setTimeout 콜백 안에서 참조한 state는 그 렌더 시점의 값을 캡처한 클로저가 됩니다. 만약 이 동작을 모르고 비동기 콜백 안에서 “최신 값”을 기대하면, 이전 렌더의 값을 읽게 되는 stale closure 문제가 발생합니다.

##### 상세 해설

React 공식 문서는 “state as a snapshot” 개념을 명확히 설명한다. 핵심은 아래와 같다.

- 렌더링이란 React가 컴포넌트 함수를 호출하는 것이고, 그 함수가 반환하는 JSX는 그 시점의 state를 기준으로 만든 UI 스냅샷이다.
- setState는 현재 변수를 바꾸는 게 아니라 다음 렌더를 예약하는 것이다.
- 이벤트 핸들러 안의 state는 그 렌더 시점에 고정된다.

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    // 여기서 count는 여전히 0 (현재 렌더의 snapshot)
    setTimeout(() => {
      // 3초 후에도 이 클로저가 캡처한 count는 0
      console.log(count); // 0
    }, 3000);
  }

  return <button onClick={handleClick}>{count}</button>;
}
```

stale closure가 문제가 되는 대표적인 경우는 useEffect 의존성 배열에서 값을 빠뜨렸을 때다. effect 안의 콜백이 이전 렌더의 state를 캡처한 채 남아 있으면, 최신 값이 아닌 오래된 값을 참조하게 된다. 해결 방법은 의존성 배열을 정확히 채우거나, 함수형 업데이트 `setCount(c => c + 1)`를 사용하는 것이다.

#### 2. 왜 “You Might Not Need an Effect”라고 하나요?

이 질문은 계산 가능한 값을 `effect`로 다시 `state`에 넣는 패턴, 불필요한 동기화, 이중 source of truth를 잘 걸러낸다. React를 외부 시스템 동기화 도구로 이해하는지 확인하기 좋다.

##### 합격 답변 예시

React에서 Effect는 본질적으로 외부 시스템(서버, 소켓, 브라우저 API 등)과 동기화하기 위한 도구입니다. 그런데 많은 경우 props나 state에서 계산 가능한 값을 useEffect로 다시 state에 넣는 불필요한 패턴을 쓰게 됩니다. 이러면 렌더가 두 번 일어나고 source of truth가 이중화되어 버그가 생기기 쉽습니다. 그래서 공식 문서도 “렌더 중에 계산 가능한 값은 Effect 없이 바로 계산하라”고 강조합니다.

##### 상세 해설

React 공식 문서 “You Might Not Need an Effect”에서 제시하는 대표적인 안티패턴은 다음과 같다.

**안티패턴: Effect로 파생 값을 동기화하기**

```tsx
// 나쁜 예: fullName을 effect로 동기화
function Form({ firstName, lastName }) {
  const [fullName, setFullName] = useState('');

  useEffect(() => {
    setFullName(firstName + ' ' + lastName);
  }, [firstName, lastName]);

  return <p>{fullName}</p>;
}
```

```tsx
// 좋은 예: 렌더 중에 바로 계산
function Form({ firstName, lastName }) {
  const fullName = firstName + ' ' + lastName;

  return <p>{fullName}</p>;
}
```

Effect가 진짜 필요한 경우는 외부 시스템과의 동기화다.

- 서버에서 데이터를 fetch할 때
- WebSocket 구독을 설정/해제할 때
- 브라우저 이벤트 리스너를 등록할 때
- 서드파티 라이브러리 위젯을 초기화할 때

반면 아래와 같은 경우에는 Effect가 필요 없다.

- props나 state에서 계산 가능한 값: 렌더 중에 바로 계산
- 사용자 이벤트에 대한 반응: 이벤트 핸들러에서 직접 처리
- 다른 state 변경에 따른 state 업데이트: 대부분 파생 값으로 해결 가능

핵심 원칙은 “렌더링 중에 계산할 수 있으면 Effect를 쓰지 말라”는 것이다. 비싼 계산이라면 useMemo로 캐싱하면 된다.

#### 3. state가 보존되는 기준은 무엇인가요? 같은 컴포넌트인데 왜 어떤 경우엔 초기화되나요?

`key`, 조건부 렌더링, form reset, 탭 전환, 컴포넌트 identity를 한 번에 확인할 수 있다. React에서 state가 “컴포넌트 이름”이 아니라 “트리 위치와 정체성”에 묶여 있다는 설명이 나와야 한다.

##### 합격 답변 예시

React는 state를 컴포넌트 함수 이름이 아니라 UI 트리에서의 위치와 key를 기준으로 보존합니다. 같은 위치에 같은 타입의 컴포넌트가 렌더되면 state가 유지되고, 위치가 바뀌거나 타입이 달라지거나 key가 바뀌면 state가 초기화됩니다. 그래서 같은 컴포넌트라도 key를 바꾸면 새 인스턴스로 취급되어 state가 리셋되고, 이를 활용해 form 초기화 같은 패턴을 구현할 수 있습니다.

##### 상세 해설

React 공식 문서 “Preserving and Resetting State”에서 설명하는 핵심 규칙은 다음과 같다.

**state가 보존되는 조건:**

- 같은 위치에 같은 타입의 컴포넌트가 렌더될 때
- key가 동일할 때

**state가 초기화되는 조건:**

- 같은 위치에 다른 타입의 컴포넌트가 렌더될 때
- key가 변경될 때
- 컴포넌트가 트리에서 제거되었다가 다시 추가될 때

```tsx
// key를 활용한 form 초기화 패턴
function App() {
  const [selectedId, setSelectedId] = useState(1);

  return (
    <>
      <ContactList onSelect={setSelectedId} />
      {/* key가 바뀌면 EditForm의 state가 완전히 초기화된다 */}
      <EditForm key={selectedId} contactId={selectedId} />
    </>
  );
}
```

조건부 렌더링에서도 같은 원리가 적용된다.

```tsx
// 케이스 A: state가 보존됨 (같은 위치, 같은 타입)
{isPlayerA ? <Counter person=”A” /> : <Counter person=”B” />}

// 케이스 B: state가 초기화됨 (key가 다름)
{isPlayerA
  ? <Counter key=”A” person=”A” />
  : <Counter key=”B” person=”B” />}
```

케이스 A에서는 두 Counter가 같은 위치에 같은 타입이므로 React가 같은 인스턴스로 보고 state를 유지한다. 케이스 B에서는 key가 다르므로 별개의 인스턴스로 취급하여 state를 초기화한다.

#### 4. Context는 상태관리 도구인가요, 전달 메커니즘인가요?

Context를 Redux나 Zustand의 대체재처럼만 이해하는 사람을 걸러낼 수 있다. prop drilling 완화와 구독 최적화, 도메인 상태 관리의 차이를 구분해서 말할 수 있어야 한다.

##### 합격 답변 예시

Context는 본질적으로 값 전달 메커니즘이지, 그 자체로 상태관리 도구는 아닙니다. Context는 트리 깊은 곳까지 값을 prop drilling 없이 전달하는 역할을 하고, 실제 상태를 만들고 갱신하는 건 useState나 useReducer 같은 다른 훅이 담당합니다. Redux나 Zustand 같은 외부 라이브러리는 상태 갱신 규칙, 구독 최적화, 미들웨어, 디버깅 도구까지 포함하는 상태관리 솔루션입니다. Context만으로 작은 범위 공유는 충분하지만, 구독 단위 최적화나 복잡한 도메인 상태 관리가 필요해지면 외부 라이브러리가 더 적합합니다.

##### 상세 해설

React 공식 문서에서 Context는 “passing data deeply” 문제의 해결책으로 소개된다. 핵심 차이를 정리하면 이렇다.

**Context가 해결하는 문제:**

- 깊은 트리로 값을 전달할 때 중간 컴포넌트를 거치지 않아도 됨
- 테마, 로케일, 인증 정보 같은 “넓은 범위에서 읽히는 값” 전달에 적합

**Context가 해결하지 않는 문제:**

- 구독 단위 최적화: Context 값이 바뀌면 해당 Context를 구독하는 모든 컴포넌트가 리렌더된다
- 복잡한 상태 갱신 로직: 미들웨어, 액션 로그, undo/redo 같은 기능이 없다
- 성능 최적화: selector 기반 부분 구독이 기본적으로 지원되지 않는다

```tsx
// Context는 전달 메커니즘 + useState/useReducer가 상태 관리
const ThemeContext = createContext('light');

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');

  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}
```

Redux나 Zustand 같은 도구는 내부적으로 Context를 사용하거나 자체 구독 시스템을 두면서, 상태 갱신 패턴과 최적화까지 포함한다. 그래서 Context는 “전달 파이프”이고, 상태관리 라이브러리는 “전달 + 갱신 + 최적화 + 도구”를 묶은 솔루션이라고 구분하는 게 정확하다.

#### 5. derived state를 언제 피해야 하나요?

폼, 필터, 정렬, 집계 값처럼 계산 가능한 값을 중복 저장하면 버그가 늘어난다. 이 질문은 상태 구조 설계를 얼마나 의식적으로 하는지 보기 좋다.

##### 합격 답변 예시

derived state, 즉 기존 state나 props에서 계산 가능한 값을 별도의 state로 중복 저장하는 것은 대부분 피해야 합니다. 두 곳에 같은 정보를 두면 하나만 갱신하고 다른 하나를 빠뜨리는 동기화 버그가 생기기 쉽습니다. React 공식 문서도 “Don't mirror props in state”와 “Avoid redundant state” 원칙을 강조합니다. 계산이 비싸더라도 useMemo로 캐싱하면 되고, 별도 state로 두지 않는 편이 source of truth를 하나로 유지하는 데 유리합니다.

##### 상세 해설

React 공식 문서 “Choosing the State Structure”에서 제시하는 원칙 중 핵심은 redundant state를 피하라는 것이다.

**대표적인 안티패턴:**

```tsx
// 나쁜 예: items에서 계산 가능한 값을 별도 state로 관리
function ShoppingCart() {
  const [items, setItems] = useState([]);
  const [totalPrice, setTotalPrice] = useState(0);    // 중복!
  const [itemCount, setItemCount] = useState(0);       // 중복!

  // items가 바뀔 때마다 total과 count도 맞춰야 하는데,
  // 빠뜨리면 동기화 버그가 생긴다
}
```

```tsx
// 좋은 예: 렌더 중에 계산
function ShoppingCart() {
  const [items, setItems] = useState([]);

  // source of truth는 items 하나뿐
  const totalPrice = items.reduce((sum, item) => sum + item.price, 0);
  const itemCount = items.length;
}
```

**derived state를 피해야 하는 대표 사례:**

- 필터링된 리스트: 원본 리스트 + 필터 조건에서 계산
- 정렬된 리스트: 원본 리스트 + 정렬 기준에서 계산
- 폼 유효성: 입력값에서 계산
- 합계/집계 값: 원본 배열에서 계산
- 선택된 항목 객체: id만 state로 두고, 실제 객체는 목록에서 find

계산 비용이 걱정된다면 useMemo를 쓰면 된다. 그래도 state를 이중으로 두는 것보다 훨씬 안전하다.

#### 6. React에서 `useRef`는 왜 필요한가요? state와는 어떤 역할 차이가 있나요?

DOM 참조, 렌더 사이 값 유지, mutable escape hatch 용도를 제대로 설명할 수 있으면 좋다. 특히 “리렌더링을 일으키지 않는 값 저장”과 “화면에 영향을 주는 상태”를 구분할 수 있어야 한다.

##### 합격 답변 예시

useRef는 렌더 사이에 값을 유지하면서도 변경 시 리렌더를 일으키지 않는 “탈출구(escape hatch)”입니다. 대표적으로 DOM 노드 참조, 타이머 ID 보관, 이전 값 추적 같은 용도로 쓰입니다. state와의 핵심 차이는 state는 변경 시 리렌더를 일으켜 화면에 반영되지만, ref는 변경해도 리렌더가 일어나지 않고 화면에 직접 영향을 주지 않는다는 점입니다. 그래서 “화면에 보여야 하는 값”은 state, “렌더와 무관하게 유지해야 하는 값”은 ref가 적합합니다.

##### 상세 해설

React 공식 문서는 ref를 “렌더링에 사용하지 않는 값을 기억하는 탈출구”로 설명한다. state와 ref의 차이를 정리하면 이렇다.

| 특성 | useState | useRef |
|------|----------|--------|
| 변경 시 리렌더 | O | X |
| 변경 타이밍 | 다음 렌더에 반영 | 즉시 변경(mutable) |
| 렌더 중 읽기/쓰기 | 안전 | 렌더 중 읽기/쓰기 지양 |
| 주 용도 | 화면에 반영되는 값 | DOM 참조, 렌더 외 값 보관 |

```tsx
function StopWatch() {
  const [elapsed, setElapsed] = useState(0);
  const intervalRef = useRef(null);  // 타이머 ID는 화면에 안 보임

  function handleStart() {
    intervalRef.current = setInterval(() => {
      setElapsed(e => e + 1);
    }, 1000);
  }

  function handleStop() {
    clearInterval(intervalRef.current);
    intervalRef.current = null;
  }

  return (
    <>
      <p>{elapsed}초</p>
      <button onClick={handleStart}>시작</button>
      <button onClick={handleStop}>정지</button>
    </>
  );
}
```

이 예시에서 `elapsed`는 화면에 표시되므로 state가 적합하고, `intervalRef`는 타이머 ID를 보관할 뿐 화면에 표시하지 않으므로 ref가 적합하다. 만약 타이머 ID를 state로 관리하면 불필요한 리렌더가 발생한다.

**useRef의 주요 사용 사례:**

- DOM 노드에 접근 (포커스, 스크롤, 크기 측정)
- setInterval/setTimeout의 ID 보관
- 이전 렌더의 값 추적
- 렌더와 무관한 외부 라이브러리 인스턴스 보관

핵심 규칙은 “ref.current를 렌더링 중에 읽거나 쓰지 말라”는 것이다. ref는 React의 선언형 모델 바깥에서 값을 다루는 탈출구이므로, 렌더 로직 안에서는 state를 쓰고, 이벤트 핸들러나 Effect 안에서 ref를 다루는 편이 안전하다.

### 이 섹션 보완 핵심

React 섹션은 이미 훌륭하다. 여기에 snapshot, effect 최소화, identity, Context의 역할, derived state, `useRef`까지 더하면 훅 사용 경험자와 React 모델 이해자를 더 명확하게 구분할 수 있다.
