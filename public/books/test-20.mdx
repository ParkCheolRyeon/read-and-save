# 08. 테스트 20문항

> 단위 테스트부터 E2E까지 비용 대비 효과를 따지는 테스트 20문항입니다.

## 총평

이 섹션은 "테스트를 쓰냐 안 쓰냐"보다 **어디에 어떤 테스트를 써야 하는지 판단할 수 있느냐**를 본다. 좋은 답변은 늘 아래를 포함한다.

- 테스트 목적
- 비용 대비 효과
- 유지보수성
- 실제 운영 안정성과의 연결

즉 테스트를 많이 썼다는 말보다, **어떤 층에서 어떤 리스크를 막으려 했는지**가 중요하다.

### 전체 흐름 맵

```
테스트의 목적 (왜 쓰는가)
  → 테스트 계층 (단위/통합/E2E)
    → 계층별 선택 기준 (ROI)
      → 도구 선택 (Vitest/Playwright)
        → 테스트 기법 (스냅샷, 사용자 관점, mock)
          → 테스트 안정성 (flaky, 비동기)
            → 실무 시나리오 (DnD, CSV, 네트워크 오류)
              → CI 통합 (실행 시간, 전략)
                → 운영 방어선 (테스트 너머의 안전장치)
```

---

## 1. 테스트를 왜 작성해야 한다고 생각하나요?

### 합격 답변 예시

테스트는 버그를 잡기 위한 도구이기도 하지만, 더 중요한 가치는 변경에 대한 자신감을 주는 데 있다고 생각합니다. 리팩터링, 기능 추가, 협업 과정에서 기존 동작이 깨지지 않았다는 신뢰를 주기 때문에 장기적으로 개발 속도를 높입니다. 즉 테스트는 개발 속도를 늦추는 절차가 아니라, 큰 변경을 가능하게 만드는 안전장치입니다.

### 가산점 포인트

테스트의 가장 큰 가치는 **"버그 발견"이 아니라 "변경 내성(change confidence)"**이라고 말하면 좋다. 테스트가 있으면 "이거 고쳐도 다른 데 안 깨질까?"라는 두려움이 줄어들기 때문에, 리팩터링과 기능 추가 속도가 올라간다. 테스트가 없는 코드베이스에서는 아무도 감히 코드를 건드리지 못해서 레거시가 된다.

### 상세 해설

#### 왜 "버그 잡기"를 넘어서야 하는가

많은 사람이 테스트를 "버그 잡기 도구"로만 설명하는데, 그건 테스트 가치의 절반만 본 것이다.

```
테스트가 없는 코드:
  "이 함수 수정해야 하는데... 다른 데서 쓰고 있으면 어떡하지?"
  → 수정을 미룸 → 기술 부채 증가 → 레거시화

테스트가 있는 코드:
  "이 함수 수정하면 테스트가 잡아줄 거야"
  → 자신 있게 수정 → 코드 품질 유지 → 지속 가능한 개발
```

#### 테스트의 4가지 가치

1. **회귀 방지** — 기존 기능이 새 변경으로 깨지지 않았음을 확인
2. **변경 내성** — 리팩터링을 두려움 없이 할 수 있는 자신감
3. **문서 역할** — 테스트 코드가 "이 함수는 이렇게 동작해야 한다"는 명세서
4. **협업 안전장치** — 내가 모르는 코드를 다른 사람이 바꿔도 테스트가 검증

```ts
// 테스트가 곧 문서다 — 이 함수의 동작을 테스트가 설명함
describe('formatPrice', () => {
  test('원 단위로 포맷한다', () => {
    expect(formatPrice(1000)).toBe('1,000원');
  });
  test('0원은 "무료"로 표시한다', () => {
    expect(formatPrice(0)).toBe('무료');
  });
  test('음수는 에러를 던진다', () => {
    expect(() => formatPrice(-1)).toThrow();
  });
});
```

#### 자주 하는 오해

- "테스트를 쓰면 개발이 느려진다" → **단기적으로는 맞지만 장기적으로는 반대.** 테스트 없이 수동 확인하는 시간, 버그 수정 시간, "이거 건드려도 되나" 고민 시간을 합치면 테스트 작성 시간보다 훨씬 크다.
- "완벽한 테스트를 작성해야 의미가 있다" → **틀림.** 핵심 경로 몇 개만 테스트해도 변경 내성이 크게 올라간다. 완벽한 테스트보다 **있는 테스트가 낫다**.
- "QA팀이 있으면 개발자 테스트는 필요 없다" → **역할이 다름.** 개발자 테스트는 빠른 피드백 루프, QA는 사용자 관점의 탐색적 테스트에 강하다.

#### 실무 에피소드

> 결제 모듈을 리팩터링해야 했는데 테스트가 없어서 아무도 건드리지 못했다. 3개월 동안 "나중에 하자"만 반복하다가, 결국 테스트를 먼저 작성하고 나서야 리팩터링을 시작할 수 있었다. 테스트를 쓰는 데 이틀, 리팩터링은 하루면 끝났다. 테스트가 없어서 3개월을 미룬 셈이었다.

> 테스트의 목적을 알았으니, 다음에는 **테스트의 종류와 각각의 역할** — 단위, 통합, E2E의 차이를 봐야 한다.

## 2. 단위 테스트, 통합 테스트, E2E 테스트의 차이를 설명해보세요.

| 테스트 종류 | 보는 범위 | 장점 | 주의점 |
| --- | --- | --- | --- |
| 단위 테스트 | 작은 로직 단위 | 빠르고 원인 파악이 쉽다 | 실제 통합 문제는 놓칠 수 있다 |
| 통합 테스트 | 여러 모듈/컴포넌트 조합 | 실제 연결 지점을 검증한다 | 경계가 넓어질수록 세팅이 늘어난다 |
| E2E 테스트 | 사용자 관점 전체 흐름 | 현실성이 가장 높다 | 느리고 유지 비용이 크다 |

### 합격 답변 예시

단위 테스트는 작은 로직 단위를 검증하고, 통합 테스트는 여러 모듈이나 컴포넌트가 함께 동작하는 흐름을 검증합니다. E2E 테스트는 실제 사용자 관점에서 전체 시스템이 의도대로 동작하는지 확인합니다. 단위 테스트는 빠르고 원인 파악이 쉽고, E2E는 가장 현실적이지만 느리고 유지 비용이 큽니다.

### 가산점 포인트

**"세 종류가 경쟁하는 게 아니라 서로 역할이 다르다"**고 말하면 좋다. 단위 테스트가 잘 잡는 것(로직 버그)과 E2E가 잘 잡는 것(통합 문제)은 다르다. 프론트엔드에서는 Kent C. Dodds의 **테스트 트로피** 개념처럼 통합 테스트 비중을 높이는 게 비용 대비 효과가 좋다.

### 상세 해설

#### 왜 세 종류가 모두 필요한가

각 계층이 잡는 문제의 종류가 다르기 때문이다:

```ts
// 단위 테스트가 잡는 것: 로직 자체의 정확성
function calcTax(price: number, rate: number): number {
  return Math.round(price * rate);
}
test('세금 계산이 정확하다', () => {
  expect(calcTax(10000, 0.1)).toBe(1000);
});

// 통합 테스트가 잡는 것: 컴포넌트 간 연결
test('상품을 장바구니에 담으면 총액이 업데이트된다', async () => {
  render(<ProductPage />);
  await userEvent.click(screen.getByRole('button', { name: '담기' }));
  expect(screen.getByText('총액: 11,000원')).toBeInTheDocument();
});

// E2E가 잡는 것: 실제 사용자 흐름 전체
test('로그인 → 상품 선택 → 결제 완료', async ({ page }) => {
  await page.goto('/login');
  await page.fill('[name=email]', 'user@test.com');
  await page.click('button:text("로그인")');
  await page.click('text=상품 A');
  await page.click('button:text("결제")');
  await expect(page.locator('text=결제 완료')).toBeVisible();
});
```

#### 실행 속도와 신뢰도의 트레이드오프

```
            느림 / 높은 현실성
              ┌────────┐
              │  E2E   │  ← 실제 브라우저, 실제 서버
              ├────────┤
              │  통합   │  ← 컴포넌트 + 상태 + mock API
              ├────────┤
              │  단위   │  ← 순수 함수, 격리된 로직
              └────────┘
            빠름 / 높은 격리성
```

#### 자주 하는 오해

- "단위 테스트만 많으면 된다" → **통합 문제를 못 잡음.** 각 함수가 맞아도 합치면 깨질 수 있다.
- "E2E만 있으면 다 커버된다" → **느리고 원인 파악이 어렵다.** E2E가 실패하면 "어디서" 깨졌는지 찾기 힘들다.
- "프론트엔드는 E2E만 하면 된다" → **비용 대비 비효율.** 대부분의 UI 검증은 통합 테스트(Testing Library)로 충분하다.

> 테스트 계층을 알았으니, 다음에는 **어떤 로직에 어떤 계층을 적용할지** — 선택 기준을 봐야 한다.

## 3. 어떤 로직은 단위 테스트로, 어떤 로직은 E2E로 검증해야 한다고 보나요?

### 합격 답변 예시

순수 함수, 파서, validation, 상태 전이 규칙 같은 것은 단위 테스트로 검증하는 게 효율적입니다. 반면 로그인, 결제, 업로드, 검색-필터-결과 반영처럼 여러 계층이 연결된 핵심 사용자 흐름은 E2E가 더 적절합니다. 즉 기준은 기술 종류가 아니라, 무엇을 증명하고 싶은지와 실패 비용이 얼마나 큰지입니다.

### 가산점 포인트

**"테스트의 격리 수준 = 증명하고 싶은 것의 범위"**라고 말하면 좋다. "이 함수가 맞나"는 단위, "이 화면이 잘 동작하나"는 통합, "이 흐름이 끝까지 가나"는 E2E. 기준은 기술이 아니라 **증명 범위**다.

### 상세 해설

#### 판단 기준 매트릭스

| 특성 | 단위 테스트 | 통합 테스트 | E2E |
|------|-----------|-----------|-----|
| **순수 로직** (계산, 변환, 검증) | ✅ 최적 | △ 가능 | ✗ 과도 |
| **컴포넌트 동작** (클릭, 입력, 표시) | △ 제한적 | ✅ 최적 | △ 가능 |
| **사용자 흐름** (로그인→결제) | ✗ 불가 | △ mock 의존 | ✅ 최적 |
| **실패 비용이 큰 경로** | △ 부분만 | △ 부분만 | ✅ 필수 |

```ts
// 단위가 맞는 것: 순수 로직
function validateEmail(email: string): boolean {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);
}
test('유효한 이메일을 통과시킨다', () => {
  expect(validateEmail('a@b.com')).toBe(true);
  expect(validateEmail('invalid')).toBe(false);
});

// 통합이 맞는 것: 컴포넌트 + 상태
test('이메일이 유효하지 않으면 에러 메시지를 표시한다', async () => {
  render(<LoginForm />);
  await userEvent.type(screen.getByLabelText('이메일'), 'invalid');
  await userEvent.click(screen.getByRole('button', { name: '로그인' }));
  expect(screen.getByText('유효한 이메일을 입력하세요')).toBeInTheDocument();
});
```

#### 자주 하는 오해

- "모든 컴포넌트를 단위 테스트해야 한다" → **과잉.** 단순 표시 컴포넌트는 통합 테스트에서 자연스럽게 검증된다.
- "E2E는 QA가 하는 것" → **개발자도 핵심 smoke test는 작성해야.** 배포 전 자동 검증으로 쓸 수 있다.

> 계층 선택 기준을 알았으니, 다음에는 **프론트엔드에서 테스트 ROI가 높은 구체적인 영역**을 봐야 한다.

## 4. 프론트엔드에서 테스트 ROI가 높은 영역은 어디라고 생각하나요?

### 합격 답변 예시

비즈니스 규칙이 담긴 로직, 자주 바뀌는 폼 상태, 인증/결제/업로드 같은 핵심 흐름, 재현이 어려운 회귀 버그 영역이 ROI가 높다고 생각합니다. 반대로 구조가 단순하고 자주 바뀌는 정적 마크업까지 지나치게 세밀하게 테스트하면 유지 비용이 더 커질 수 있습니다. 즉 테스트는 많은 곳에 쓰는 것보다, 실패 비용이 큰 곳에 우선 배치하는 게 중요합니다.

### 가산점 포인트

ROI의 기준은 **"이 테스트가 없으면 얼마나 위험한가"**다. 정적 마크업 테스트가 실패해도 사용자가 못 느끼지만, 결제 로직 테스트가 실패하면 매출 손실이다. **실패 비용이 큰 곳에 테스트를 집중**하는 것이 현실적이다.

### 상세 해설

#### ROI 판단 프레임워크

```
테스트 ROI = 실패 시 피해 / 테스트 작성+유지 비용

ROI 높음:
  결제 로직 (실패 = 매출 손실)     → 비용 큼 / 작성 보통 = 높은 ROI
  인증 흐름 (실패 = 보안 사고)     → 비용 큼 / 작성 보통 = 높은 ROI
  데이터 변환 (실패 = 잘못된 표시) → 비용 중간 / 작성 쉬움 = 높은 ROI

ROI 낮음:
  정적 텍스트 (실패 = 오타)        → 비용 작음 / 작성 쉬움 = 낮은 ROI
  CSS 스타일 (실패 = 약간 어색)    → 비용 작음 / 유지 비용 큼 = 낮은 ROI
```

#### 프론트엔드 ROI 맵

| 영역 | ROI | 이유 |
|------|-----|------|
| 결제/인증/업로드 | ★★★ | 실패 시 직접적 비즈니스 손실 |
| 폼 검증/상태 전이 | ★★★ | 자주 바뀌고, 엣지 케이스 많음 |
| 데이터 변환 유틸 | ★★☆ | 순수 함수라 테스트 쉽고, 여러 곳에서 사용 |
| 라우팅/네비게이션 | ★★☆ | 잘못되면 사용자가 페이지를 못 찾음 |
| 정적 텍스트/마크업 | ★☆☆ | 실패해도 기능에 영향 적음 |
| 애니메이션/트랜지션 | ★☆☆ | 테스트 유지 비용이 높고 깨지기 쉬움 |

#### 자주 하는 오해

- "커버리지 100%를 목표로 해야 한다" → **숫자 게임.** assertion 없이 호출만 해도 커버리지는 올라간다. 핵심은 커버리지가 아니라 **"중요한 것을 검증하고 있는가"**다.
- "모든 컴포넌트에 테스트를 써야 한다" → **ROI가 낮은 곳에 시간을 쓰면 오히려 비효율.** 정적 마크업보다 비즈니스 로직에 투자하는 게 맞다.

> ROI를 알았으니, 다음에는 **도구 선택의 기준** — Vitest와 Playwright를 왜 선택하는지를 봐야 한다.

---

## 5. Vitest를 선택한 이유를 설명해보세요.

### 합격 답변 예시

Vite 기반 프로젝트와 통합이 좋고 실행 속도가 빠르며 설정이 비교적 가벼워서 선택할 수 있습니다. 특히 프론트엔드에서는 테스트 피드백 루프가 짧은 게 중요한데, Vitest는 이 부분에서 DX가 좋습니다. 즉 선택 이유는 단순 유행이 아니라, 프로젝트 빌드 체인과 잘 맞고 테스트 실행 부담이 적다는 점입니다.

### 가산점 포인트

"많이 써서"가 아니라 **"빌드 체인과의 호환성과 피드백 속도"**로 이유를 말하면 좋다. Vitest는 Vite의 ESM 변환을 그대로 사용하므로 Jest처럼 별도의 babel/ts-jest 설정이 필요 없고, HMR 기반 watch 모드가 빠르다.

### 상세 해설

#### 왜 Jest 대신 Vitest인가

| | Jest | Vitest |
|---|---|---|
| **변환 방식** | babel / ts-jest | Vite의 ESM 변환 재사용 |
| **설정** | 별도 설정 필요 (특히 ESM) | `vite.config.ts` 재사용 |
| **실행 속도** | 보통 | 빠름 (ESBuild 기반) |
| **Watch 모드** | 파일 변경 감지 | HMR 기반 (더 빠름) |
| **호환성** | Jest API 표준 | Jest 호환 API + 추가 기능 |

```ts
// vitest.config.ts — Vite 설정을 그대로 확장
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',          // 브라우저 환경 시뮬레이션
    setupFiles: ['./src/test/setup.ts'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html'],
    },
  },
});
```

#### 자주 하는 오해

- "Vitest는 Vite 프로젝트에서만 쓸 수 있다" → **아님.** Vite를 안 쓰는 프로젝트에서도 독립적으로 사용 가능하다.
- "Jest에서 Vitest로 마이그레이션이 어렵다" → **대부분 쉬움.** API가 거의 호환되므로 import만 바꾸면 되는 경우가 많다.

> 단위/통합 테스트 도구를 알았으니, 다음에는 **E2E 도구** — Playwright의 선택 이유를 봐야 한다.

---

## 6. Playwright를 선택한 이유를 설명해보세요.

### 합격 답변 예시

실제 브라우저 기반으로 E2E를 검증하기 좋고, auto-waiting, trace, screenshot, video, network mocking, 병렬 실행 같은 기능이 강해서 선택할 수 있습니다. 특히 flaky를 줄이는 기본기가 좋고, 디버깅 경험이 좋아서 운영 품질을 보는 테스트에 유리합니다. 즉 단순히 브라우저를 띄울 수 있어서가 아니라, 실패한 테스트를 다시 분석하고 안정화하기 좋은 도구라는 점이 큽니다.

### 가산점 포인트

Cypress와의 차이를 물을 수 있다. Playwright의 핵심 차별점은 **멀티 브라우저 지원**(Chromium, Firefox, WebKit), **병렬 실행**, **네트워크 수준 mocking**, 그리고 **Trace Viewer**다. Cypress는 단일 탭/단일 도메인 제약이 있지만 Playwright는 없다.

### 상세 해설

#### Playwright vs Cypress 비교

| | Playwright | Cypress |
|---|---|---|
| **브라우저** | Chromium, Firefox, WebKit | Chromium 기반 (Electron) |
| **병렬 실행** | 내장 | 유료 (Dashboard) |
| **멀티 탭/도메인** | 지원 | 제한적 |
| **디버깅** | Trace Viewer, Video | Time Travel |
| **네트워크 mocking** | route API | intercept API |
| **실행 환경** | Node.js (외부 제어) | 브라우저 안에서 실행 |

```ts
// Playwright 테스트 예시 — auto-waiting이 flaky를 줄임
import { test, expect } from '@playwright/test';

test('검색하면 결과가 표시된다', async ({ page }) => {
  await page.goto('/search');
  await page.fill('[name=query]', 'React');
  await page.click('button:text("검색")');

  // auto-waiting: 요소가 나타날 때까지 자동 대기
  await expect(page.locator('.result-item')).toHaveCount(10);
  await expect(page.locator('.result-item').first()).toContainText('React');
});

test('네트워크 실패 시 에러 UI를 보여준다', async ({ page }) => {
  // 네트워크 수준에서 응답을 제어
  await page.route('/api/search*', (route) =>
    route.fulfill({ status: 500 })
  );
  await page.goto('/search');
  await page.fill('[name=query]', 'test');
  await page.click('button:text("검색")');
  await expect(page.locator('text=검색에 실패했습니다')).toBeVisible();
});
```

#### 자주 하는 오해

- "E2E 도구는 다 비슷하다" → **실행 모델이 다름.** Cypress는 브라우저 안에서 테스트가 실행되고, Playwright는 외부에서 브라우저를 제어한다. 이 차이가 멀티 탭, 멀티 도메인, 병렬 실행 가능 여부를 결정한다.

> E2E 도구를 알았으니, 다음에는 **구체적인 테스트 기법** — 스냅샷 테스트의 장단점을 봐야 한다.

---

## 7. 스냅샷 테스트의 장단점을 설명해보세요.

### 합격 답변 예시

스냅샷 테스트는 넓은 출력 변화를 빠르게 감지할 수 있다는 장점이 있습니다. 하지만 의도 없는 작은 변화도 많이 잡아서 노이즈가 커질 수 있고, 리뷰 없이 스냅샷을 그냥 갱신하는 습관이 생기면 신뢰도가 떨어집니다. 그래서 스냅샷은 핵심 동작을 검증하는 메인 테스트라기보다 보조 도구로 쓰는 것이 좋다고 생각합니다.

### 가산점 포인트

스냅샷의 가장 큰 위험은 **"update snapshot" 자동화 습관**이다. CI에서 스냅샷 실패가 뜨면 내용을 확인하지 않고 `pnpm test -u`로 갱신해버리는 팀이 많은데, 이러면 스냅샷 테스트의 의미가 사라진다.

### 상세 해설

#### 스냅샷이 유용한 경우 vs 위험한 경우

```ts
// ✅ 유용한 스냅샷: 구조화된 데이터의 변경 감지
test('API 응답 변환 결과가 유지된다', () => {
  const result = transformApiResponse(mockData);
  expect(result).toMatchSnapshot();
  // 변환 로직이 바뀌면 즉시 감지됨
});

// ❌ 위험한 스냅샷: 거대한 컴포넌트 트리
test('ProductPage가 렌더링된다', () => {
  const { container } = render(<ProductPage />);
  expect(container).toMatchSnapshot();
  // 1000줄짜리 스냅샷 → 뭐가 바뀌었는지 아무도 안 봄
  // → "그냥 업데이트" → 테스트 무의미화
});
```

#### 자주 하는 오해

- "스냅샷 테스트 = 시각적 테스트" → **틀림.** 스냅샷은 텍스트 비교이고, 시각적 회귀 테스트(Visual Regression)는 스크린샷 이미지 비교다. 완전히 다른 것.
- "스냅샷을 많이 쓸수록 안전하다" → **오히려 위험.** 스냅샷이 너무 많으면 "update all" 습관이 생겨서 변경을 감지하는 의미가 사라진다.

> 스냅샷의 한계를 알았으니, 다음에는 **더 좋은 테스트 관점** — 사용자 관점 vs 구현 관점 테스트를 봐야 한다.

---

## 8. 사용자 관점 테스트와 구현 관점 테스트의 차이를 설명해보세요.

### 합격 답변 예시

사용자 관점 테스트는 화면에 무엇이 보이고, 어떤 행동이 가능하며, 결과가 어떻게 변하는지를 검증합니다. 반면 구현 관점 테스트는 내부 state, 함수 호출, DOM 구조 세부처럼 구현 방식에 더 의존합니다. 프론트엔드에서는 사용자 관점 테스트가 리팩터링 내성이 더 높고, 유지보수도 더 유리한 경우가 많습니다.

### 가산점 포인트

Testing Library의 철학 — **"테스트가 소프트웨어를 사용하는 방식과 닮을수록 더 많은 신뢰를 줄 수 있다"** (Kent C. Dodds) — 를 인용하면 좋다. `getByRole`, `getByLabelText` 같은 쿼리가 사용자 관점을 강제하는 설계다.

### 상세 해설

#### 같은 기능, 다른 테스트 관점

```tsx
// ❌ 구현 관점: 내부 state에 의존
test('isOpen state가 true가 된다', () => {
  const { result } = renderHook(() => useModal());
  act(() => result.current.open());
  expect(result.current.isOpen).toBe(true);
  // 문제: isOpen 이름을 바꾸면 테스트가 깨짐
});

// ✅ 사용자 관점: 화면에 보이는 것 검증
test('열기 버튼을 누르면 모달이 표시된다', async () => {
  render(<ModalPage />);
  await userEvent.click(screen.getByRole('button', { name: '열기' }));
  expect(screen.getByRole('dialog')).toBeInTheDocument();
  // isOpen을 visible로 바꿔도, 모달이 보이기만 하면 테스트는 통과
});
```

#### 왜 사용자 관점이 리팩터링에 강한가

- 구현 관점: `state.isOpen === true` → state 이름 변경 시 깨짐
- 사용자 관점: `dialog가 화면에 있나` → 내부 구현이 바뀌어도 결과가 같으면 통과

#### 자주 하는 오해

- "getByTestId를 많이 쓰면 사용자 관점이다" → **아님.** `data-testid`는 사용자가 보는 것이 아니다. `getByRole`, `getByLabelText`가 더 사용자에 가깝다.

> 사용자 관점을 알았으니, 다음에는 **나쁜 테스트가 리팩터링을 방해하는 구조적 이유**를 봐야 한다.

---

## 9. 테스트 코드가 리팩터링을 방해하는 상황은 왜 생기나요?

### 합격 답변 예시

테스트가 동작 계약이 아니라 내부 구현 세부에 너무 강하게 묶였기 때문입니다. 예를 들어 특정 state 변수 이름, DOM 구조, 내부 함수 호출 횟수에 의존하면 리팩터링 때 동작은 같아도 테스트가 많이 깨집니다. 즉 테스트가 제품 계약을 검증해야 하는데, 구현 방식을 고정해버리면 오히려 개발 속도를 방해하게 됩니다.

### 가산점 포인트

**"깨지기 쉬운 테스트(brittle test)"**의 원인을 구조적으로 설명할 수 있으면 좋다. 핵심은 **테스트가 "무엇을 하는가(WHAT)"가 아니라 "어떻게 하는가(HOW)"를 검증**하면 리팩터링을 방해한다는 것.

### 상세 해설

#### 나쁜 테스트의 패턴

```ts
// ❌ 구현에 묶인 테스트 — 리팩터링을 방해
test('fetchUsers가 정확히 1번 호출된다', () => {
  const spy = vi.spyOn(api, 'fetchUsers');
  render(<UserList />);
  expect(spy).toHaveBeenCalledTimes(1);
  // 문제: React Query로 바꾸면 fetchUsers 호출 방식이 달라져서 깨짐
});

test('div.user-card 안에 span.name이 있다', () => {
  const { container } = render(<UserCard name="홍길동" />);
  expect(container.querySelector('div.user-card span.name')).toHaveTextContent('홍길동');
  // 문제: CSS class나 태그를 바꾸면 깨짐
});

// ✅ 동작 계약에 묶인 테스트 — 리팩터링에 강함
test('사용자 목록이 화면에 표시된다', async () => {
  render(<UserList />);
  expect(await screen.findByText('홍길동')).toBeInTheDocument();
  // API 호출 방식이 바뀌어도, 화면에 이름이 나오면 통과
});
```

#### 실무 에피소드

> 컴포넌트 내부를 class에서 hooks로 마이그레이션했는데, 기능은 동일한데 테스트가 50개 깨졌다. 테스트가 `instance().state`를 직접 접근하고 있어서였다. 사용자 관점으로 다시 쓴 뒤에는 내부 구현을 바꿔도 테스트가 안 깨졌다.

> 테스트가 리팩터링을 방해하는 이유를 알았으니, 다음에는 **테스트 자체의 안정성 문제** — flaky test를 봐야 한다.

---

## 10. flaky test가 무엇이고, 어떻게 줄일 수 있나요?

### 합격 답변 예시

flaky test는 같은 코드인데 실행할 때마다 통과와 실패가 들쑥날쑥한 테스트입니다. 원인은 타이밍 의존, 네트워크 불안정, 테스트 간 공유 상태, 애니메이션, 비동기 처리 누락 등이 많습니다. 줄이려면 deterministic한 fixture를 쓰고, 적절한 wait 전략을 사용하며, 테스트를 서로 격리하고, 외부 의존성을 통제해야 합니다.

### 가산점 포인트

**"retry로 가립니다"는 약한 답변**이다. retry는 증상을 숨기는 것이지 원인을 해결하는 것이 아니다. flaky의 근본 원인을 찾아 제거하는 것이 답이다.

### 상세 해설

#### flaky의 주요 원인과 해결

```ts
// ❌ flaky 원인 1: 고정 시간 대기
test('데이터가 로드된다', async () => {
  render(<DataPage />);
  await new Promise(r => setTimeout(r, 2000));  // 2초면 될까? 안 될 수도
  expect(screen.getByText('데이터')).toBeInTheDocument();
});

// ✅ 해결: 조건 기반 대기
test('데이터가 로드된다', async () => {
  render(<DataPage />);
  expect(await screen.findByText('데이터')).toBeInTheDocument();
  // findBy는 요소가 나타날 때까지 자동 대기 (기본 1초)
});

// ❌ flaky 원인 2: 테스트 간 상태 공유
let globalCount = 0;
test('첫 번째', () => { globalCount++; expect(globalCount).toBe(1); });
test('두 번째', () => { expect(globalCount).toBe(0); }); // 실행 순서에 따라 실패

// ✅ 해결: 테스트마다 상태 초기화
beforeEach(() => { globalCount = 0; });
```

#### flaky test가 팀에 미치는 영향

```
flaky test → CI에서 가끔 실패 → "또 flaky네, 다시 돌려" → retry 습관
→ 진짜 버그도 "flaky겠지" 하고 넘김 → 프로덕션 버그
```

flaky test가 쌓이면 **테스트 전체에 대한 신뢰가 무너진다.** 이것이 flaky를 적극적으로 고쳐야 하는 이유다.

#### 자주 하는 오해

- "E2E는 원래 flaky하다" → **대부분은 테스트 코드의 문제.** auto-waiting, deterministic fixture, 네트워크 mocking을 제대로 하면 E2E도 안정적으로 만들 수 있다.

> flaky를 이해했으니, 다음에는 **비동기 UI 테스트에서 실수하기 쉬운 구체적인 패턴**을 봐야 한다.

---

## 11. 비동기 UI 테스트에서 가장 자주 실수하는 부분은 무엇인가요?

### 합격 답변 예시

화면이 아직 준비되지 않았는데 동기적으로 바로 단정해버리는 실수가 가장 많다고 생각합니다. 즉 실제 UI는 loading → success/error 흐름을 거치는데, 테스트가 그 중간 상태를 무시하고 바로 결과를 기대해 실패하거나 flaky해지는 경우가 많습니다. 그래서 비동기 UI 테스트는 '지금 당장 있는지'보다 '나타날 때까지 기다린 뒤 검증하는지'가 중요합니다.

### 가산점 포인트

Testing Library의 **`findBy` vs `getBy` vs `queryBy`** 차이를 정확히 알면 좋다. `getBy`는 즉시 없으면 에러, `findBy`는 나타날 때까지 대기, `queryBy`는 없어도 null 반환. 비동기 UI에서는 **`findBy`가 기본**이다.

### 상세 해설

#### 비동기 UI의 시간 흐름

```
사용자 클릭 → loading 스피너 → API 응답 → 결과 표시
                ↑ 여기서 바로 결과를 기대하면 실패
```

```ts
// ❌ 잘못된 비동기 테스트: loading 중인데 바로 검증
test('데이터가 표시된다', () => {
  render(<UserProfile userId={1} />);
  // getByText는 즉시 찾으므로, API 응답 전에 실패
  expect(screen.getByText('홍길동')).toBeInTheDocument();
});

// ✅ 올바른 비동기 테스트: 나타날 때까지 대기
test('데이터가 표시된다', async () => {
  render(<UserProfile userId={1} />);
  // findByText는 요소가 나타날 때까지 자동 대기
  expect(await screen.findByText('홍길동')).toBeInTheDocument();
});

// ✅ loading → success 전체 흐름 검증
test('로딩 후 데이터가 표시된다', async () => {
  render(<UserProfile userId={1} />);
  expect(screen.getByText('로딩 중...')).toBeInTheDocument();      // 즉시 확인
  expect(await screen.findByText('홍길동')).toBeInTheDocument();    // 대기 후 확인
  expect(screen.queryByText('로딩 중...')).not.toBeInTheDocument(); // 사라졌는지 확인
});
```

#### 자주 하는 오해

- "`await`만 붙이면 비동기 처리가 된다" → **`getBy`에 await를 붙여도 소용없다.** `getBy`는 동기 함수이므로, 비동기 대기가 필요하면 반드시 `findBy`를 써야 한다.

> 비동기 테스트 패턴을 알았으니, 다음에는 **mock의 적절한 사용 범위**를 봐야 한다.

## 12. mock을 어디까지 써야 한다고 생각하나요?

### 합격 답변 예시

mock은 테스트 목적을 흔들지 않는 선까지 써야 한다고 생각합니다. 외부 네트워크, 시간, 랜덤성, 브라우저 API처럼 테스트를 불안정하게 만드는 요소는 mock이 유용하지만, 내부 구현을 전부 mock해버리면 실제 동작을 거의 검증하지 못하게 됩니다. 즉 mock의 기준은 편의성이 아니라, 무엇을 증명하려는 테스트인지에 따라 정해야 합니다.

### 가산점 포인트

mock의 기준은 **"테스트 경계 바깥의 불확실성을 제거하는 것"**이다. 내가 증명하려는 것(비즈니스 로직)은 실제로 실행하고, 내가 통제할 수 없는 것(네트워크, 시간)만 mock한다.

### 상세 해설

#### mock의 경계 판단

```
┌─────────────────────────────────────────────┐
│         내가 증명하려는 것 (실제 실행)         │
│  ┌─────────────────────────────────────────┐│
│  │  컴포넌트 렌더링                          ││
│  │  상태 전이                                ││
│  │  비즈니스 로직                            ││
│  └─────────────────────────────────────────┘│
├─────────────────────────────────────────────┤
│         외부 의존성 (mock 대상)              │
│  네트워크 요청, 시간(Date.now), 랜덤,         │
│  localStorage, 브라우저 API                   │
└─────────────────────────────────────────────┘
```

```ts
// ✅ 적절한 mock: 외부 API 응답을 제어
server.use(
  http.get('/api/users', () => HttpResponse.json([{ id: 1, name: '홍길동' }]))
);
test('사용자 목록을 표시한다', async () => {
  render(<UserList />);
  expect(await screen.findByText('홍길동')).toBeInTheDocument();
});

// ❌ 과도한 mock: 내부 훅까지 mock
vi.mock('./useUsers', () => ({
  useUsers: () => ({ data: [{ id: 1, name: '홍길동' }], isLoading: false }),
}));
// 문제: useUsers 구현이 바뀌면 mock도 바뀌어야 함 → 실제 동작 검증 안 됨
```

#### 자주 하는 오해

- "mock을 많이 쓸수록 테스트가 안정적이다" → **가짜 안정성.** mock이 많으면 "테스트는 통과하지만 프로덕션에서 깨지는" 상황이 생긴다.

> mock의 경계를 알았으니, 다음에는 **API mocking과 실제 서버 연동의 균형**을 봐야 한다.

## 13. API mocking과 실제 서버 연동 테스트의 균형은 어떻게 잡나요?

### 합격 답변 예시

대부분의 UI와 상태 전이 테스트는 API mocking으로 빠르고 안정적으로 돌리고, 핵심 사용자 플로우 몇 개는 실제 서버나 스테이징 환경과 연동해 검증하는 식으로 균형을 잡습니다. mock만 쓰면 통합 문제를 놓칠 수 있고, 실제 서버만 쓰면 테스트가 느리고 불안정해질 수 있습니다. 그래서 빠른 피드백과 실제 신뢰성 사이에서 테스트 계층을 나누는 것이 중요합니다.

### 가산점 포인트

**"mock의 가장 큰 위험은 mock이 실제와 달라도 테스트가 통과한다는 것"**이라고 말하면 좋다. 이 문제를 해결하는 것이 Contract Testing이다 (보완 문항 참고).

### 상세 해설

#### 계층별 전략

```
단위/통합 테스트 (Vitest + MSW)
  → API mocking으로 빠르게 실행
  → 대부분의 UI 동작 검증
  → PR마다 실행

E2E smoke test (Playwright + 스테이징 서버)
  → 핵심 3~5개 흐름만 실제 서버 연동
  → 로그인 → 결제 같은 critical path
  → 배포 전 실행

E2E 전체 회귀 (Playwright + 스테이징 서버)
  → 전체 시나리오 실제 연동
  → nightly 또는 주 단위 실행
```

#### 실무 에피소드

> mock으로만 테스트하다가, 백엔드에서 응답 필드 이름을 `user_name`에서 `userName`으로 바꿨는데 프론트 테스트는 mock을 안 바꿔서 계속 통과했다. 프로덕션에서 이름이 "undefined"로 표시되고 나서야 발견. 이후 핵심 흐름은 스테이징 서버와 연동 테스트를 추가했다.

> API mocking의 균형을 알았으니, 다음에는 **상태관리 로직과 렌더링 테스트를 어떻게 분리하는지** 봐야 한다.

## 14. 상태관리 로직 테스트와 컴포넌트 렌더링 테스트는 어떻게 나눠서 생각하나요?

### 합격 답변 예시

상태 전이 규칙, reducer, selector, store helper처럼 순수한 로직은 별도 로직 테스트로 검증하고, 컴포넌트는 그 상태를 바탕으로 사용자에게 무엇이 보이는지 검증하는 편이 좋습니다. 이렇게 나누면 실패 원인을 더 빨리 찾을 수 있고, UI 리팩터링과 로직 변경을 분리해서 다룰 수 있습니다.

### 가산점 포인트

이 분리는 **관심사 분리(Separation of Concerns)**의 테스트 버전이다. "로직이 틀린 건가, 화면 표시가 틀린 건가"를 빠르게 구분할 수 있어야 디버깅이 빠르다.

### 상세 해설

```ts
// 로직 테스트: 상태 전이 규칙 (순수 함수)
describe('cartReducer', () => {
  test('상품을 추가하면 수량이 1 늘어난다', () => {
    const state = { items: [{ id: 1, qty: 1 }] };
    const next = cartReducer(state, { type: 'ADD', id: 1 });
    expect(next.items[0].qty).toBe(2);
  });

  test('수량이 0이면 항목을 제거한다', () => {
    const state = { items: [{ id: 1, qty: 1 }] };
    const next = cartReducer(state, { type: 'REMOVE', id: 1 });
    expect(next.items).toHaveLength(0);
  });
});

// 렌더링 테스트: 사용자에게 보이는 결과
test('장바구니에 상품이 있으면 총액이 표시된다', () => {
  render(<CartSummary items={[{ id: 1, name: '상품A', price: 10000, qty: 2 }]} />);
  expect(screen.getByText('총액: 20,000원')).toBeInTheDocument();
});
```

#### 왜 분리가 유리한가

- 로직 테스트 실패 → "계산이 틀렸다" (원인 명확)
- 렌더링 테스트 실패 → "화면 표시가 틀렸다" (원인 명확)
- 둘 다 한 테스트에 → "뭐가 틀린 건지 모르겠다" (디버깅 어려움)

> 테스트 분리를 알았으니, 다음에는 **복잡한 UI 테스트 시나리오** — 드래그 앤 드롭을 봐야 한다.

## 15. 드래그 앤 드롭 UI는 어떻게 테스트하겠습니까?

### 합격 답변 예시

드래그 앤 드롭은 상호작용이 복잡하기 때문에, 순서 변경이나 이동 규칙 같은 핵심 로직은 별도 단위 테스트로 검증하고, 실제 드래그 이벤트 흐름은 브라우저 기반 통합 테스트나 E2E로 확인하는 방식이 좋습니다. 즉 좌표와 이벤트 시퀀스를 전부 단위 테스트로 해결하려 하기보다, 로직과 상호작용 검증을 나누는 게 현실적입니다.

### 가산점 포인트

DnD 테스트의 핵심은 **"로직과 인터랙션을 분리"**하는 것이다. 순서 변경 알고리즘은 단위 테스트, 실제 마우스/터치 이벤트는 E2E로 나누면 각각 안정적으로 검증할 수 있다.

### 상세 해설

```ts
// 단위 테스트: 순서 변경 로직 (순수 함수)
test('항목을 위치 0에서 2로 이동한다', () => {
  const items = ['A', 'B', 'C', 'D'];
  const result = reorderItems(items, 0, 2);
  expect(result).toEqual(['B', 'C', 'A', 'D']);
});

// E2E: 실제 드래그 인터랙션 (Playwright)
test('항목을 드래그하여 순서를 바꿀 수 있다', async ({ page }) => {
  await page.goto('/kanban');
  const source = page.locator('[data-testid="task-1"]');
  const target = page.locator('[data-testid="task-3"]');
  await source.dragTo(target);
  // 순서가 바뀌었는지 확인
  const items = await page.locator('.task-item').allTextContents();
  expect(items[2]).toContain('Task 1');
});
```

> DnD 테스트를 알았으니, 다음에는 **파일 업로드 같은 현실적인 시나리오**의 테스트를 봐야 한다.

## 16. CSV 업로드 같은 기능은 어떤 테스트 시나리오를 우선 작성하겠습니까?

### 합격 답변 예시

정상 업로드, 필수 컬럼 누락, 잘못된 형식, 일부 행만 실패하는 경우, 대용량 처리 지연, 중복 데이터, 네트워크 실패 후 재시도, 업로드 성공 후 결과 반영을 우선 검증하겠습니다. CSV 업로드는 단순히 파일을 올리는 것보다, 검증과 오류 표시, 부분 실패 처리 경험이 핵심이기 때문입니다. 즉 **성공 케이스 하나보다 실패 케이스 설계가 훨씬 중요합니다**.

### 가산점 포인트

CSV 업로드처럼 **"실패 모드가 다양한 기능"**은 테스트 시나리오 설계 능력을 보여주기 좋다. "정상 동작만 테스트했다"는 약하고, **"부분 실패, 형식 오류, 네트워크 단절까지 커버했다"**가 강하다.

### 상세 해설

```ts
// 시나리오별 테스트 예시
describe('CSV 업로드', () => {
  test('정상 CSV를 업로드하면 성공 메시지를 표시한다', async () => {
    const file = new File(['name,email\n홍길동,a@b.com'], 'users.csv', { type: 'text/csv' });
    render(<CsvUploader />);
    await userEvent.upload(screen.getByLabelText('파일 선택'), file);
    await userEvent.click(screen.getByRole('button', { name: '업로드' }));
    expect(await screen.findByText('1건 업로드 완료')).toBeInTheDocument();
  });

  test('필수 컬럼이 없으면 에러를 표시한다', async () => {
    const file = new File(['wrong_column\nvalue'], 'bad.csv', { type: 'text/csv' });
    render(<CsvUploader />);
    await userEvent.upload(screen.getByLabelText('파일 선택'), file);
    await userEvent.click(screen.getByRole('button', { name: '업로드' }));
    expect(await screen.findByText('필수 컬럼 누락: name, email')).toBeInTheDocument();
  });

  test('일부 행만 실패하면 부분 결과를 표시한다', async () => {
    // MSW에서 부분 실패 응답 설정
    server.use(
      http.post('/api/upload', () => HttpResponse.json({
        success: 8, failed: 2, errors: [{ row: 3, reason: '이메일 형식 오류' }]
      }))
    );
    // ...업로드 후 "8건 성공, 2건 실패" 메시지 확인
  });
});
```

> CSV 업로드 시나리오를 알았으니, 다음에는 **네트워크 오류와 재시도 흐름** 테스트를 봐야 한다.

## 17. 세션 만료, 네트워크 오류, 재시도 흐름은 어떻게 검증하겠습니까?

### 합격 답변 예시

401이나 403 응답, timeout, 일시적 5xx, offline/online 전환, retry 후 성공과 retry 후 최종 실패를 각각 시나리오로 나눠 검증하겠습니다. 중요한 건 단순히 에러가 발생했는지만 보는 게 아니라, 사용자에게 어떤 메시지가 보이고 어떤 복구 경로가 제공되는지까지 함께 확인하는 것입니다. 즉 네트워크 실패 테스트는 기술 검증이면서 UX 검증이기도 합니다.

### 가산점 포인트

**"에러가 뜬다"가 아니라 "사용자가 어떻게 복구하느냐"를 검증**하는 것이 핵심이다. "다시 시도" 버튼이 보이는지, 로그인 페이지로 리다이렉트되는지, 오프라인 배너가 표시되는지까지 확인해야 한다.

### 상세 해설

```ts
// 세션 만료: 401 응답 시 로그인 페이지로 이동
test('401 응답 시 로그인 페이지로 리다이렉트한다', async () => {
  server.use(http.get('/api/me', () => new HttpResponse(null, { status: 401 })));
  render(<DashboardPage />);
  await waitFor(() => {
    expect(window.location.pathname).toBe('/login');
  });
});

// 네트워크 오류: 재시도 UI
test('API 실패 시 재시도 버튼을 표시한다', async () => {
  server.use(http.get('/api/data', () => HttpResponse.error()));
  render(<DataPage />);
  expect(await screen.findByText('데이터를 불러올 수 없습니다')).toBeInTheDocument();

  // 재시도 성공
  server.use(http.get('/api/data', () => HttpResponse.json({ value: 42 })));
  await userEvent.click(screen.getByRole('button', { name: '다시 시도' }));
  expect(await screen.findByText('42')).toBeInTheDocument();
});
```

> 네트워크 오류 테스트를 알았으니, 다음에는 **CI에서 E2E를 효율적으로 실행하는 전략**을 봐야 한다.

## 18. CI에 E2E 테스트를 넣을 때 실행 시간 문제를 어떻게 다루나요?

### 합격 답변 예시

핵심 smoke flow만 CI blocking 단계에 넣고, 전체 회귀는 nightly나 별도 병렬 job으로 분리하는 방식이 현실적입니다. 또 변경 영향 기반 실행, test sharding, 브라우저 수 조절, fixture 최적화로 실행 시간을 줄일 수 있습니다. 즉 모든 E2E를 모든 커밋마다 다 돌리는 것보다, 중요한 흐름과 비용의 균형을 맞추는 게 중요합니다.

### 가산점 포인트

Playwright의 **sharding** 기능으로 테스트를 여러 머신에 분산하면 실행 시간을 크게 줄일 수 있다. `--shard=1/3`, `--shard=2/3` 등으로 CI에서 병렬 실행이 가능하다.

### 상세 해설

#### 전략 시각화

```yaml
# GitHub Actions에서 E2E 전략 분리
name: CI
on: [pull_request]
jobs:
  # 빠른 검증: 모든 PR
  smoke-e2e:
    runs-on: ubuntu-latest
    steps:
      - run: pnpm playwright test --grep @smoke  # 핵심 5개만
    # 실행 시간: ~2분

  # 전체 회귀: nightly
  full-e2e:
    if: github.event_name == 'schedule'  # cron으로 매일 밤 실행
    strategy:
      matrix:
        shard: [1, 2, 3]  # 3개 머신에 분산
    runs-on: ubuntu-latest
    steps:
      - run: pnpm playwright test --shard=${{ matrix.shard }}/3
    # 전체 50분 → 분산 후 ~17분
```

#### 실행 시간 줄이는 기법들

| 기법 | 효과 | 예시 |
|------|------|------|
| **Smoke 분리** | PR당 실행 시간 감소 | `--grep @smoke` |
| **Sharding** | 병렬 분산 | `--shard=1/3` |
| **브라우저 제한** | 불필요한 브라우저 제거 | Chromium만 CI에서 실행 |
| **영향 기반 실행** | 변경 없는 영역 스킵 | paths-filter로 관련 테스트만 |

> CI 전략을 알았으니, 다음에는 **테스트 너머의 운영 방어 장치**를 봐야 한다.

## 19. 운영 안정화를 위해 테스트 외에 어떤 방어 장치가 더 필요하다고 보나요?

### 합격 답변 예시

테스트 외에도 로그, 에러 모니터링, feature flag, 점진 배포, fallback UI, rollback, 헬스체크, 알림 체계가 필요합니다. 테스트는 출시 전 품질 방어선이고, 이런 장치들은 출시 후 운영 방어선입니다. 즉 안정성은 테스트만으로 완성되지 않고, 운영 중 관측과 복구 체계까지 있어야 완성됩니다.

### 가산점 포인트

**"테스트는 알려진 시나리오만 검증한다"**는 한계를 인식해야 한다. 프로덕션에서는 테스트에 없는 상황이 반드시 발생하므로, 모니터링과 복구 체계가 두 번째 방어선이 된다.

### 상세 해설

#### 방어 장치의 계층

```
출시 전 방어선 (Prevention):
  ├─ 정적 분석 (lint, typecheck)
  ├─ 단위/통합 테스트
  ├─ E2E 테스트
  └─ 코드 리뷰

출시 후 방어선 (Detection & Recovery):
  ├─ 에러 모니터링 (Sentry)
  ├─ 성능 모니터링 (Web Vitals)
  ├─ Feature Flag (점진적 롤아웃)
  ├─ Canary 배포 (일부 트래픽만)
  ├─ Rollback (빠른 복구)
  ├─ Health Check (자동 감지)
  ├─ Fallback UI (에러 바운더리)
  └─ Alert (즉시 대응)
```

#### 자주 하는 오해

- "테스트가 충분하면 모니터링은 필요 없다" → **틀림.** 테스트는 "알려진 시나리오"만 검증한다. 프로덕션의 다양한 환경, 데이터, 동시성 문제는 테스트에서 잡기 어렵다.
- "feature flag는 복잡도만 늘린다" → **위험 관리 도구.** 새 기능을 1%에게만 보여주고 문제가 없으면 100%로 올리는 것이 훨씬 안전하다.

> 운영 방어 장치를 알았으니, 마지막으로 **실제 경험을 기반으로 테스트 도입의 체감 변화**를 정리해보자.

## 20. 네가 도입했던 Vitest/Playwright 경험을 기준으로, 테스트 도입 전후의 차이를 설명해보세요.

### 합격 답변 예시

도입 전에는 변경 후 수동 확인 비중이 높아서, 기능을 건드릴 때마다 불안감이 컸고 릴리즈 직전 반복 확인 비용도 컸습니다. 도입 후에는 핵심 흐름에 대해 회귀 확인이 빨라졌고, 특히 리팩터링이나 버그 수정 후 재검증 시간이 줄어들었습니다. 즉 테스트 도입의 가장 큰 변화는 버그 개수 자체보다, **변경을 두려워하지 않게 된 점**이라고 생각합니다.

### 가산점 포인트

**"무엇을 썼다"보다 "무엇이 달라졌는가"**를 말해야 한다. 도구 이름이 아니라 **개발 프로세스와 팀 문화의 변화**를 구체적으로 말하면 강한 답변이 된다.

### 상세 해설

#### 체감 변화의 구체적 지표

| | 도입 전 | 도입 후 |
|---|---|---|
| **리팩터링** | "건드리면 어디가 깨질지 몰라서 안 함" | "테스트가 잡아주니 자신 있게 고침" |
| **배포 전 확인** | 30분~1시간 수동 확인 | CI에서 5분 자동 확인 |
| **회귀 버그** | 릴리즈 후 발견 → hotfix | PR 단계에서 발견 → 머지 전 수정 |
| **신규 팀원** | "이 코드 뭐 하는 거예요?" | 테스트 코드가 동작 명세서 역할 |

#### 실무 에피소드

> Playwright로 결제 흐름 E2E를 작성한 뒤, 카드사 연동 모듈을 업데이트할 때 "이거 건드려도 되나?"에서 "E2E 돌려보고 결정하자"로 대화가 바뀌었다. 실제로 업데이트 후 결제 3단계에서 실패하는 것을 E2E가 잡아줘서, 프로덕션 배포 전에 수정할 수 있었다.

## 보완내용

### 왜 보완이 필요한가

기존 테스트 20문항은 ROI, 단위/통합/E2E 경계, mock, flaky, CI 시간, 운영 방어 장치까지 포함하고 있어서 꽤 강하다. 다만 최근 프론트엔드 테스트는 단순 단위/E2E를 넘어서 계약 테스트, 시각 회귀, 접근성 자동화처럼 품질 전략의 폭이 더 넓어졌다.

아래 질문을 추가하면 테스트 도구를 써본 사람과 테스트 전략을 설계할 줄 아는 사람을 더 잘 나눌 수 있다.

### 추가하면 좋은 질문

#### 1. 테스트 피라미드와 테스트 트로피 개념을 어떻게 생각하나요?

단순 암기보다 테스트 전략 철학을 보기 좋은 질문이다. unit test 중심이 좋은지, integration 중심이 더 현실적인지 판단 기준을 들을 수 있다.

##### 합격 답변 예시

테스트 피라미드는 단위 테스트를 가장 많이, E2E를 가장 적게 쌓는 전통적인 모델입니다. 반면 테스트 트로피는 통합 테스트를 가장 두텁게 가져가는 모델로, Kent C. Dodds가 프론트엔드 맥락에서 제안했습니다. 프론트엔드는 컴포넌트, 훅, 상태, API 호출이 밀접하게 엮여 있어서 단위 테스트만으로는 실제 동작 신뢰도가 낮을 수 있습니다. 그래서 저는 통합 테스트 비중을 높게 가져가되, 순수 로직은 단위 테스트, 핵심 흐름은 E2E로 보완하는 트로피 형태가 프론트엔드에 더 현실적이라고 생각합니다.

##### 상세 해설

두 모델의 차이는 "어디에 가장 많은 테스트를 집중하느냐"다.

- 테스트 피라미드(Martin Fowler)
  - 단위 테스트가 가장 많고 빠르다
  - 위로 갈수록 느리고 비싸다
  - 백엔드, 마이크로서비스에서 유효하다
- 테스트 트로피(Kent C. Dodds)
  - 통합 테스트가 가장 두텁다
  - Static → Unit → Integration → E2E 순서
  - 프론트엔드에서 비용 대비 효과가 좋다

```typescript
// 피라미드 관점: 순수 로직은 단위 테스트
function calcDiscount(price: number, rate: number): number {
  if (rate < 0 || rate > 1) throw new Error('Invalid rate');
  return Math.round(price * (1 - rate));
}

// 단위 테스트
test('할인 계산이 정확하다', () => {
  expect(calcDiscount(10000, 0.1)).toBe(9000);
  expect(() => calcDiscount(10000, -1)).toThrow();
});
```

```tsx
// 트로피 관점: 컴포넌트 통합 테스트
test('쿠폰을 적용하면 할인 가격이 표시된다', async () => {
  render(<CheckoutPage />);
  await userEvent.type(screen.getByLabelText('쿠폰 코드'), 'SAVE10');
  await userEvent.click(screen.getByRole('button', { name: '적용' }));
  expect(await screen.findByText('할인 적용: -1,000원')).toBeInTheDocument();
});
```

핵심은 어느 모델이 맞느냐가 아니라, 프로젝트 특성에 맞게 비중을 조절할 줄 아느냐다.

#### 2. Contract Testing이 왜 필요하다고 보나요?

프론트와 백엔드 API 계약이 어긋나는 문제를 얼마나 실무적으로 이해하는지 확인할 수 있다. mock만으로는 놓치기 쉬운 위험을 설명하기 좋다.

##### 합격 답변 예시

프론트엔드가 mock 기반으로 테스트를 통과해도, 실제 백엔드 응답 형식이 바뀌면 운영에서 깨질 수 있습니다. Contract Testing은 프론트와 백엔드가 합의한 API 스펙을 별도 계약으로 정의하고, 양쪽이 각각 그 계약을 지키는지 독립적으로 검증하는 방식입니다. 이렇게 하면 통합 환경 없이도 API 변경으로 인한 연동 실패를 사전에 잡을 수 있습니다. 즉 Contract Testing은 mock의 신뢰 한계를 보완하는 장치입니다.

##### 상세 해설

mock 기반 테스트의 가장 큰 위험은 "mock이 실제와 달라도 테스트가 통과한다"는 점이다.

- Contract Testing이 해결하는 문제
  - 백엔드가 필드 이름을 바꿨는데 프론트가 모른다
  - 응답 구조가 바뀌었는데 mock은 그대로다
  - 프론트가 없는 필드를 기대하고 있다
- 대표 도구
  - Pact: Consumer-Driven Contract Testing 대표 도구
  - OpenAPI/Swagger 기반 자동 검증

```typescript
// Pact Consumer 테스트 예시 (프론트엔드 측)
import { PactV3 } from '@pact-foundation/pact';

const provider = new PactV3({
  consumer: 'FrontendApp',
  provider: 'UserService',
});

test('사용자 프로필을 정상 조회한다', async () => {
  await provider
    .given('사용자 ID 1이 존재한다')
    .uponReceiving('프로필 조회 요청')
    .withRequest({ method: 'GET', path: '/api/users/1' })
    .willRespondWith({
      status: 200,
      body: {
        id: 1,
        name: '홍길동',
        email: 'hong@example.com',
      },
    })
    .executeTest(async (mockServer) => {
      const res = await fetch(`${mockServer.url}/api/users/1`);
      const data = await res.json();
      expect(data.name).toBe('홍길동');
    });
});
```

Contract Testing은 E2E를 대체하는 것이 아니라, mock과 실제 서버 사이의 틈을 메우는 역할을 한다.

#### 3. Visual Regression Test는 언제 ROI가 높다고 보나요?

디자인 시스템, 마케팅 페이지, 어드민처럼 UI 변경 영향 범위가 넓은 제품에서 특히 유용하다. 기능 correctness와 시각 회귀를 분리해서 보는지 확인할 수 있다.

##### 합격 답변 예시

디자인 시스템 컴포넌트, 마케팅 랜딩 페이지, 어드민 대시보드처럼 UI 변경이 여러 곳에 영향을 미치는 경우에 ROI가 높다고 생각합니다. 기능 테스트는 동작이 맞는지를 보지만, 레이아웃이 밀렸는지, 색상이 바뀌었는지, 간격이 어긋났는지는 잡아주지 못합니다. Visual Regression Test는 스크린샷 비교를 통해 이런 시각적 변화를 자동으로 감지해줍니다. 다만 동적 콘텐츠가 많거나 자주 바뀌는 화면은 false positive가 잦아서 유지 비용이 높을 수 있습니다.

##### 상세 해설

Visual Regression Test는 "눈으로 봐야 아는 문제"를 자동화하는 것이다.

- ROI가 높은 경우
  - 디자인 시스템: 컴포넌트 하나가 수십 곳에 쓰인다
  - 마케팅 페이지: 레이아웃 깨짐이 매출에 직결된다
  - 다국어 지원: 텍스트 길이 변화로 레이아웃이 밀릴 수 있다
- ROI가 낮은 경우
  - 매일 바뀌는 대시보드
  - 동적 데이터가 많은 목록 페이지
- 대표 도구
  - Chromatic (Storybook 연동)
  - Percy
  - Playwright 내장 스크린샷 비교

```typescript
// Playwright Visual Regression Test 예시
import { test, expect } from '@playwright/test';

test('버튼 컴포넌트가 시각적으로 변하지 않았다', async ({ page }) => {
  await page.goto('/storybook/button--primary');
  await expect(page.locator('.btn-primary')).toHaveScreenshot(
    'button-primary.png',
    { maxDiffPixelRatio: 0.01 }
  );
});

test('결제 페이지 레이아웃이 유지된다', async ({ page }) => {
  await page.goto('/checkout');
  // 동적 데이터를 고정해서 스냅샷 안정성 확보
  await page.route('/api/cart', (route) =>
    route.fulfill({
      json: { items: [{ name: '테스트 상품', price: 10000 }] },
    })
  );
  await expect(page).toHaveScreenshot('checkout-layout.png');
});
```

핵심은 기능 correctness 테스트와 시각 회귀 테스트를 분리해서 관리하는 것이다. 둘을 섞으면 실패 원인 파악이 어려워진다.

#### 4. 접근성, a11y(Accessibility) 테스트는 자동화와 수동 검증을 어떻게 나누나요?

품질을 기능 correctness만으로 보지 않는 사람인지 드러나는 질문이다. 자동 검사 도구와 실제 키보드/스크린리더 확인의 경계를 설명할 수 있으면 좋다.

##### 합격 답변 예시

자동화 도구로는 axe-core, eslint-plugin-jsx-a11y 같은 것으로 ARIA 속성 누락, 색상 대비, alt 텍스트 빈값, 잘못된 역할 같은 기계적 규칙 위반을 잡을 수 있습니다. 하지만 실제 키보드 탐색 순서가 자연스러운지, 스크린리더가 읽는 흐름이 의미가 통하는지, 포커스 이동이 사용자 기대와 맞는지는 수동 확인이 필요합니다. 즉 자동화는 기본 품질 바닥을 깔아주고, 수동 검증은 실제 사용 경험 품질을 확인하는 역할입니다.

##### 상세 해설

a11y 테스트는 자동화만으로 전체 접근성 문제의 약 30~40% 정도만 잡을 수 있다는 점을 알아야 한다.

- 자동화가 잡을 수 있는 것
  - ARIA 속성 누락/잘못된 사용
  - 색상 대비 부족 (WCAG AA 기준)
  - img alt 텍스트 빈값
  - 중복된 id
  - label 연결 누락
- 수동 검증이 필요한 것
  - 키보드 탐색 순서와 흐름
  - 스크린리더 읽기 순서의 의미
  - 모달/드롭다운 포커스 트랩
  - 동적 콘텐츠 변경 알림 (live region)
  - 복잡한 위젯의 키보드 상호작용

```typescript
// axe-core를 활용한 자동 접근성 테스트
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('메인 페이지가 접근성 기본 규칙을 충족한다', async ({ page }) => {
  await page.goto('/');
  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa'])
    .analyze();
  expect(results.violations).toEqual([]);
});
```

```typescript
// Vitest + Testing Library에서 접근성 검증
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

test('모달이 열리면 포커스가 모달 내부로 이동한다', async () => {
  render(<ModalTrigger />);
  await userEvent.click(screen.getByRole('button', { name: '열기' }));

  const modal = screen.getByRole('dialog');
  expect(modal).toBeInTheDocument();
  // 포커스가 모달 내부 첫 번째 요소로 이동했는지 확인
  expect(modal.contains(document.activeElement)).toBe(true);
});

test('ESC 키로 모달을 닫을 수 있다', async () => {
  render(<ModalTrigger />);
  await userEvent.click(screen.getByRole('button', { name: '열기' }));
  await userEvent.keyboard('{Escape}');
  expect(screen.queryByRole('dialog')).not.toBeInTheDocument();
});
```

좋은 답변은 "axe 돌리면 됩니다"에 그치지 않고, 자동화의 한계와 수동 검증이 필요한 구체적인 시나리오를 함께 말해야 한다.

#### 5. MSW(Mock Service Worker) 같은 API mocking 도구를 왜 쓰나요?

기존 API mocking 질문을 더 구체화해준다. 네트워크 계층을 실제에 가깝게 흉내 내면서도 테스트 안정성을 높이는 방법을 설명할 수 있다.

##### 합격 답변 예시

MSW는 Service Worker를 이용해 네트워크 수준에서 요청을 가로채기 때문에, 애플리케이션 코드를 전혀 수정하지 않고도 API 응답을 제어할 수 있습니다. jest.mock이나 함수 레벨 mocking은 구현에 의존하게 되지만, MSW는 실제 fetch/axios 호출이 그대로 실행되고 네트워크 계층에서만 응답을 바꾸므로 테스트 신뢰도가 더 높습니다. 또한 브라우저 환경(Storybook, 개발 서버)과 Node 환경(Vitest, Jest) 모두에서 같은 핸들러를 재사용할 수 있어서 mock 관리 비용이 줄어듭니다.

##### 상세 해설

MSW의 핵심 가치는 "테스트 코드가 아니라 네트워크 계층에서 mocking한다"는 점이다.

- 기존 방식의 문제
  - jest.mock으로 모듈을 직접 교체 → 구현 의존
  - axios adapter 교체 → 라이브러리 변경 시 깨짐
  - 전역 fetch를 덮어쓰기 → 부작용 위험
- MSW의 장점
  - 네트워크 수준 가로채기 → 코드 무수정
  - 브라우저/Node 환경 동일 핸들러
  - 에러, 지연, 부분 실패 시뮬레이션 용이
  - Storybook, 개발 서버에서도 활용 가능

```typescript
// MSW 핸들러 정의
import { http, HttpResponse, delay } from 'msw';

export const handlers = [
  // 정상 응답
  http.get('/api/users', () => {
    return HttpResponse.json([
      { id: 1, name: '홍길동' },
      { id: 2, name: '김영희' },
    ]);
  }),

  // 에러 응답
  http.get('/api/users/:id', ({ params }) => {
    if (params.id === '999') {
      return new HttpResponse(null, { status: 404 });
    }
    return HttpResponse.json({ id: params.id, name: '홍길동' });
  }),

  // 네트워크 지연 시뮬레이션
  http.post('/api/orders', async () => {
    await delay(2000);
    return HttpResponse.json({ orderId: 'ORD-001' });
  }),
];
```

```typescript
// Vitest에서 MSW 사용
import { setupServer } from 'msw/node';
import { handlers } from './handlers';

const server = setupServer(...handlers);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

test('사용자 목록을 불러와 표시한다', async () => {
  render(<UserList />);
  expect(await screen.findByText('홍길동')).toBeInTheDocument();
  expect(screen.getByText('김영희')).toBeInTheDocument();
});

test('API 실패 시 에러 메시지를 표시한다', async () => {
  // 특정 테스트에서만 핸들러 오버라이드
  server.use(
    http.get('/api/users', () => {
      return new HttpResponse(null, { status: 500 });
    })
  );
  render(<UserList />);
  expect(await screen.findByText('목록을 불러올 수 없습니다')).toBeInTheDocument();
});
```

핵심은 MSW를 "mock 도구"로만 보지 않고, 테스트와 개발 환경 전반에서 네트워크 계약을 일관되게 관리하는 도구로 이해하는 것이다.

#### 6. Coverage 수치가 높다고 좋은 테스트라고 볼 수 없는 이유는 무엇인가요?

이 질문은 테스트를 숫자 게임으로 보는 사람을 걸러낸다. "무엇을 실행했는가"와 "무엇을 의미 있게 검증했는가"의 차이를 설명할 수 있어야 한다.

##### 합격 답변 예시

Coverage는 코드가 실행되었는지만 측정하지, 그 실행 결과를 의미 있게 검증했는지는 알려주지 않습니다. 예를 들어 assertion 없이 함수를 호출만 해도 커버리지는 올라갑니다. 또한 100% 커버리지라 해도 경계값, 에러 경로, 비동기 타이밍 같은 실제 장애 원인을 놓칠 수 있습니다. 그래서 커버리지는 테스트가 닿지 않는 사각지대를 찾는 보조 지표로는 유용하지만, 테스트 품질의 기준이 되어서는 안 됩니다.

##### 상세 해설

커버리지의 한계를 정확히 아는 것이 중요하다.

- 커버리지가 높아도 놓치는 것
  - assertion 없는 테스트 → 실행만 하고 검증 안 함
  - 정상 경로만 테스트 → 에러/경계값 빠짐
  - 비동기 완료 전 검증 → flaky하지만 커버리지는 잡힘
  - 구현 세부에 묶인 테스트 → 숫자는 높지만 리팩터링에 약함
- 커버리지가 유용한 경우
  - 아예 테스트가 닿지 않는 코드를 발견할 때
  - PR 리뷰에서 새 코드의 테스트 유무를 확인할 때
  - 팀 최소 기준선을 설정할 때 (예: 70% 이상)

```typescript
// 커버리지는 100%지만 나쁜 테스트
function divide(a: number, b: number): number {
  if (b === 0) throw new Error('Division by zero');
  return a / b;
}

// assertion 없이 호출만 → 커버리지는 올라감
test('divide를 호출한다', () => {
  divide(10, 2);
  divide(10, 0).catch(() => {}); // 에러를 무시
  // expect가 없다 → 아무것도 검증하지 않았다
});
```

```typescript
// 커버리지는 같지만 좋은 테스트
test('정상 나눗셈 결과를 반환한다', () => {
  expect(divide(10, 2)).toBe(5);
  expect(divide(-10, 2)).toBe(-5);
  expect(divide(0, 5)).toBe(0);
});

test('0으로 나누면 에러가 발생한다', () => {
  expect(() => divide(10, 0)).toThrow('Division by zero');
});
```

커버리지 수치를 목표로 삼으면 "수치를 올리기 위한 테스트"를 쓰게 된다. 좋은 팀은 커버리지를 최소 기준선으로만 쓰고, 테스트의 진짜 가치는 변경 내성과 장애 예방으로 판단한다.

### 이 섹션 보완 핵심

테스트 섹션은 이미 좋다. 여기에 전략 철학, 계약 테스트, 시각 회귀, 접근성, API mocking 도구, coverage 한계까지 더하면 "테스트 도구 사용자"와 "테스트 전략 설계자"를 더 잘 구분할 수 있다.
