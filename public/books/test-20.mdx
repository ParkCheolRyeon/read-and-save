# 08. 테스트 20문항

> 단위 테스트부터 E2E까지 비용 대비 효과를 따지는 테스트 20문항입니다.

## 8. 테스트 방식에 대한 질문 20문항

## 총평

이 섹션은 “테스트를 쓰냐 안 쓰냐”보다 어디에 어떤 테스트를 써야 하는지 판단할 수 있느냐를 본다. 좋은 답변은 늘 아래를 포함한다.

- 테스트 목적
- 비용 대비 효과
- 유지보수성
- 실제 운영 안정성과의 연결
- 즉 테스트를 많이 썼다는 말보다,

어떤 층에서 어떤 리스크를 막으려 했는지가 중요하다.

## 1. 테스트를 왜 작성해야 한다고 생각하나요?

### 합격 답변 예시

테스트는 버그를 잡기 위한 도구이기도 하지만, 더 중요한 가치는 변경에 대한 자신감을 주는 데 있다고 생각합니다. 리팩터링, 기능 추가, 협업 과정에서 기존 동작이 깨지지 않았다는 신뢰를 주기 때문에 장기적으로 개발 속도를 높입니다. 즉 테스트는 개발 속도를 늦추는 절차가 아니라, 큰 변경을 가능하게 만드는 안전장치입니다.

### 상세 해설

많은 사람이 테스트를 “버그 잡기” 정도로만 말한다. 그것도 맞지만, 더 중요한 건 변경 내성이다.

- 테스트의 진짜 가치
- 리팩터링 가능
- 회귀 방지
- 협업 안전장치
- 문서 역할 일부 수행

## 2. 단위 테스트, 통합 테스트, E2E 테스트의 차이를 설명해보세요.

| 테스트 종류 | 보는 범위 | 장점 | 주의점 |
| --- | --- | --- | --- |
| 단위 테스트 | 작은 로직 단위 | 빠르고 원인 파악이 쉽다 | 실제 통합 문제는 놓칠 수 있다 |
| 통합 테스트 | 여러 모듈/컴포넌트 조합 | 실제 연결 지점을 검증한다 | 경계가 넓어질수록 세팅이 늘어난다 |
| E2E 테스트 | 사용자 관점 전체 흐름 | 현실성이 가장 높다 | 느리고 유지 비용이 크다 |

### 합격 답변 예시

단위 테스트는 작은 로직 단위를 검증하고, 통합 테스트는 여러 모듈이나 컴포넌트가 함께 동작하는 흐름을 검증합니다. E2E 테스트는 실제 사용자 관점에서 전체 시스템이 의도대로 동작하는지 확인합니다. 단위 테스트는 빠르고 원인 파악이 쉽고, E2E는 가장 현실적이지만 느리고 유지 비용이 큽니다.

### 상세 해설

- 단위 테스트
- 빠름
- 작음
- 원인 명확
- 통합 테스트
- 실제 조합 검증
- UI + 상태 + API(Application Programming Interface) mocking 조합
- E2E

#### 실제 사용자 흐름

- 브라우저 환경
- 느리지만 신뢰도 높음
- 좋은 답변은

“세 종류가 경쟁하는 게 아니라 서로 역할이 다르다”고 말해야 한다.

## 3. 어떤 로직은 단위 테스트로, 어떤 로직은 E2E로 검증해야 한다고 보나요?

### 합격 답변 예시

순수 함수, 파서, validation, 상태 전이 규칙 같은 것은 단위 테스트로 검증하는 게 효율적입니다. 반면 로그인, 결제, 업로드, 검색-필터-결과 반영처럼 여러 계층이 연결된 핵심 사용자 흐름은 E2E가 더 적절합니다. 즉 기준은 기술 종류가 아니라, 무엇을 증명하고 싶은지와 실패 비용이 얼마나 큰지입니다.

### 상세 해설

이 질문은 ROI(Return On Investment) 감각을 본다.

- 단위가 맞는 것
- 독립 로직
- 복잡한 계산
- reducer
- parser
- formatter
- E2E가 맞는 것
- 진짜 사용자 플로우
- 페이지 전환
- 인증
- 업로드
- 결제

## 4. 프론트엔드에서 테스트 ROI가 높은 영역은 어디라고 생각하나요?

### 합격 답변 예시

비즈니스 규칙이 담긴 로직, 자주 바뀌는 폼 상태, 인증/결제/업로드 같은 핵심 흐름, 재현이 어려운 회귀 버그 영역이 ROI가 높다고 생각합니다. 반대로 구조가 단순하고 자주 바뀌는 정적 마크업까지 지나치게 세밀하게 테스트하면 유지 비용이 더 커질 수 있습니다. 즉 테스트는 많은 곳에 쓰는 것보다, 실패 비용이 큰 곳에 우선 배치하는 게 중요합니다.

### 상세 해설

좋은 답변은 “모든 걸 다 테스트해야 한다”가 아니다.

- ROI 높은 영역
- 실수하면 큰일 나는 곳
- 자주 바뀌는 곳
- 사람이 수동 확인하기 힘든 곳
- ROI 낮은 영역
- 정적인 문구
- 단순 스타일
- 구현에 너무 가까운 세부

## 5. Vitest를 선택한 이유를 설명해보세요.

### 합격 답변 예시

Vite 기반 프로젝트와 통합이 좋고 실행 속도가 빠르며 설정이 비교적 가벼워서 선택할 수 있습니다. 특히 프론트엔드에서는 테스트 피드백 루프가 짧은 게 중요한데, Vitest는 이 부분에서 DX가 좋습니다. 즉 선택 이유는 단순 유행이 아니라, 프로젝트 빌드 체인과 잘 맞고 테스트 실행 부담이 적다는 점입니다.

### 상세 해설

이 질문은 툴 선호보다 선택 이유를 본다.

#### 좋은 답변 포인트

- 빠름
- Vite 친화적
- 설정 단순
- 개발 경험 좋음
- 약한 답변
- “많이 써서요”

## 6. Playwright를 선택한 이유를 설명해보세요.

### 합격 답변 예시

실제 브라우저 기반으로 E2E를 검증하기 좋고, auto-waiting, trace, screenshot, video, network mocking, 병렬 실행 같은 기능이 강해서 선택할 수 있습니다. 특히 flaky를 줄이는 기본기가 좋고, 디버깅 경험이 좋아서 운영 품질을 보는 테스트에 유리합니다. 즉 단순히 브라우저를 띄울 수 있어서가 아니라, 실패한 테스트를 다시 분석하고 안정화하기 좋은 도구라는 점이 큽니다.

### 상세 해설

좋은 답변은 “Cypress 대신 Playwright를 왜?” 같은 꼬리 질문을 대비할 수 있어야 한다.

- 강점
- 멀티 브라우저
- auto waiting
- 디버깅 강함
- 병렬 처리
- trace viewer

## 7. 스냅샷 테스트의 장단점을 설명해보세요.

### 합격 답변 예시

스냅샷 테스트는 넓은 출력 변화를 빠르게 감지할 수 있다는 장점이 있습니다. 하지만 의도 없는 작은 변화도 많이 잡아서 노이즈가 커질 수 있고, 리뷰 없이 스냅샷을 그냥 갱신하는 습관이 생기면 신뢰도가 떨어집니다. 그래서 스냅샷은 핵심 동작을 검증하는 메인 테스트라기보다 보조 도구로 쓰는 것이 좋다고 생각합니다.

### 상세 해설

이 질문은 스냅샷을 맹신하는지 보는 질문이다.

#### 장점

- 빠른 회귀 감지
- 넓은 커버리지

#### 단점

- 노이즈
- 의미 없는 업데이트
- 실제 사용자 가치와 멀 수 있음

## 8. 사용자 관점 테스트와 구현 관점 테스트의 차이를 설명해보세요.

### 합격 답변 예시

사용자 관점 테스트는 화면에 무엇이 보이고, 어떤 행동이 가능하며, 결과가 어떻게 변하는지를 검증합니다. 반면 구현 관점 테스트는 내부 state, 함수 호출, DOM 구조 세부처럼 구현 방식에 더 의존합니다. 프론트엔드에서는 사용자 관점 테스트가 리팩터링 내성이 더 높고, 유지보수도 더 유리한 경우가 많습니다.

### 상세 해설

좋은 테스트는 사용자가 보는 계약을 검증해야 한다.

- 예:
- 버튼을 누르면 모달이 열린다
- 에러 메시지가 보인다
- 결과 목록이 바뀐다
- 반대로
- 내부 state가 true가 된다
- 특정 메서드가 1번 호출된다

이런 건 구현에 더 묶인다.

## 9. 테스트 코드가 리팩터링을 방해하는 상황은 왜 생기나요?

### 합격 답변 예시

테스트가 동작 계약이 아니라 내부 구현 세부에 너무 강하게 묶였기 때문입니다. 예를 들어 특정 state 변수 이름, DOM 구조, 내부 함수 호출 횟수에 의존하면 리팩터링 때 동작은 같아도 테스트가 많이 깨집니다. 즉 테스트가 제품 계약을 검증해야 하는데, 구현 방식을 고정해버리면 오히려 개발 속도를 방해하게 됩니다.

### 상세 해설

이 질문의 핵심은 나쁜 테스트도 존재한다는 걸 아는지다.

- 테스트가 방해가 되는 이유
- 구현 의존
- 과도한 mocking
- brittle selector
- 너무 세세한 구조 검증

## 10. flaky test가 무엇이고, 어떻게 줄일 수 있나요?

### 합격 답변 예시

flaky test는 같은 코드인데 실행할 때마다 통과와 실패가 들쑥날쑥한 테스트입니다. 원인은 타이밍 의존, 네트워크 불안정, 테스트 간 공유 상태, 애니메이션, 비동기 처리 누락 등이 많습니다. 줄이려면 deterministic한 fixture를 쓰고, 적절한 wait 전략을 사용하며, 테스트를 서로 격리하고, 외부 의존성을 통제해야 합니다.

### 상세 해설

- 흔한 원인
- sleep 기반 대기
- 실제 네트워크 의존
- 공유 DB 상태
- 애니메이션 중 클릭
- 비동기 완료 전 assertion
- 좋은 답변

“retry로 가립니다”는 약하다. retry는 임시 방편일 수 있지만 근본 해결이 아니다.

## 11. 비동기 UI 테스트에서 가장 자주 실수하는 부분은 무엇인가요?

### 합격 답변 예시

화면이 아직 준비되지 않았는데 동기적으로 바로 단정해버리는 실수가 가장 많다고 생각합니다. 즉 실제 UI는 loading → success/error 흐름을 거치는데, 테스트가 그 중간 상태를 무시하고 바로 결과를 기대해 실패하거나 flaky해지는 경우가 많습니다. 그래서 비동기 UI 테스트는 ‘지금 당장 있는지’보다 ‘나타날 때까지 기다린 뒤 검증하는지’가 중요합니다.

### 상세 해설

이 질문은 실무적이다.

- 흔한 실수
- await 누락
- 화면 업데이트 전에 assertion
- timer/network 완료 전 검증

#### 좋은 답변 포인트

비동기 UI는 코드가 아니라 사용자 경험의 시간 흐름으로 생각해야 한다.

## 12. mock을 어디까지 써야 한다고 생각하나요?

### 합격 답변 예시

mock은 테스트 목적을 흔들지 않는 선까지 써야 한다고 생각합니다. 외부 네트워크, 시간, 랜덤성, 브라우저 API처럼 테스트를 불안정하게 만드는 요소는 mock이 유용하지만, 내부 구현을 전부 mock해버리면 실제 동작을 거의 검증하지 못하게 됩니다. 즉 mock의 기준은 편의성이 아니라, 무엇을 증명하려는 테스트인지에 따라 정해야 합니다.

### 상세 해설

- mock이 필요한 곳
- 외부 API
- 시간
- 랜덤
- 파일 시스템
- 브라우저 환경 일부
- 너무 과한 mock의 문제
- 실제 동작 안 검증
- 구현 의존성 증가
- 가짜 성공

## 13. API mocking과 실제 서버 연동 테스트의 균형은 어떻게 잡나요?

### 합격 답변 예시

대부분의 UI와 상태 전이 테스트는 API mocking으로 빠르고 안정적으로 돌리고, 핵심 사용자 플로우 몇 개는 실제 서버나 스테이징 환경과 연동해 검증하는 식으로 균형을 잡습니다. mock만 쓰면 통합 문제를 놓칠 수 있고, 실제 서버만 쓰면 테스트가 느리고 불안정해질 수 있습니다. 그래서 빠른 피드백과 실제 신뢰성 사이에서 테스트 계층을 나누는 것이 중요합니다.

### 상세 해설

이 질문도 정답형이 아니다.

- 균형점
- 대부분은 mock
- 핵심 플로우는 real integration
- 장애 포인트가 큰 부분만 실제 연동

## 14. 상태관리 로직 테스트와 컴포넌트 렌더링 테스트는 어떻게 나눠서 생각하나요?

### 합격 답변 예시

상태 전이 규칙, reducer, selector, store helper처럼 순수한 로직은 별도 로직 테스트로 검증하고, 컴포넌트는 그 상태를 바탕으로 사용자에게 무엇이 보이는지 검증하는 편이 좋습니다. 이렇게 나누면 실패 원인을 더 빨리 찾을 수 있고, UI 리팩터링과 로직 변경을 분리해서 다룰 수 있습니다. 즉 상태관리와 렌더링은 연결돼 있지만, 테스트 관점에서는 역할을 나누는 게 유지보수에 유리합니다.

### 상세 해설

이 질문은 관심사 분리 감각을 본다.

- 로직 테스트
- 입력 → 상태 변화
- 순수성
- 규칙 검증
- 렌더링 테스트
- 화면 표시
- 사용자 이벤트
- 메시지/버튼/에러 상태

## 15. 드래그 앤 드롭 UI는 어떻게 테스트하겠습니까?

### 합격 답변 예시

드래그 앤 드롭은 상호작용이 복잡하기 때문에, 순서 변경이나 이동 규칙 같은 핵심 로직은 별도 단위 테스트로 검증하고, 실제 드래그 이벤트 흐름은 브라우저 기반 통합 테스트나 E2E로 확인하는 방식이 좋습니다. 즉 좌표와 이벤트 시퀀스를 전부 단위 테스트로 해결하려 하기보다, 로직과 상호작용 검증을 나누는 게 현실적입니다. 이렇게 해야 테스트 비용 대비 신뢰도를 높일 수 있습니다.

### 상세 해설

DnD는 테스트가 어려운 영역이다.

- 좋은 전략
- reorder algorithm → 단위 테스트
- 실제 drag interaction → E2E
- 약한 전략
- 전부 snapshot이나 전부 단위 테스트로 해결하려는 것

## 16. CSV 업로드 같은 기능은 어떤 테스트 시나리오를 우선 작성하겠습니까?

### 합격 답변 예시

정상 업로드, 필수 컬럼 누락, 잘못된 형식, 일부 행만 실패하는 경우, 대용량 처리 지연, 중복 데이터, 네트워크 실패 후 재시도, 업로드 성공 후 결과 반영을 우선 검증하겠습니다. CSV 업로드는 단순히 파일을 올리는 것보다, 검증과 오류 표시, 부분 실패 처리 경험이 핵심이기 때문입니다. 즉 성공 케이스 하나보다 실패 케이스 설계가 훨씬 중요합니다.

### 상세 해설

이 질문은 네 실제 경험과도 잘 맞는다.

- 꼭 들어가야 할 것
- 파일 형식 오류
- 필수 컬럼 누락
- 부분 실패
- 네트워크 실패
- 재시도
- 성공 후 반영

## 17. 세션 만료, 네트워크 오류, 재시도 흐름은 어떻게 검증하겠습니까?

### 합격 답변 예시

401이나 403 응답, timeout, 일시적 5xx, offline/online 전환, retry 후 성공과 retry 후 최종 실패를 각각 시나리오로 나눠 검증하겠습니다. 중요한 건 단순히 에러가 발생했는지만 보는 게 아니라, 사용자에게 어떤 메시지가 보이고 어떤 복구 경로가 제공되는지까지 함께 확인하는 것입니다. 즉 네트워크 실패 테스트는 기술 검증이면서 UX 검증이기도 합니다.

### 상세 해설

- “에러가 뜬다”가 아니라

“사용자가 어떻게 복구하느냐”를 포함한다.

## 18. CI(Continuous Integration)에 E2E 테스트를 넣을 때 실행 시간 문제를 어떻게 다루나요?

### 합격 답변 예시

### 상세 해설

- 핵심만 blocking
- 전체는 nightly
- 병렬 처리
- 변경 영향 기반 실행

## 19. 운영 안정화를 위해 테스트 외에 어떤 방어 장치가 더 필요하다고 보나요?

- feature flag

## 보완내용

### 왜 보완이 필요한가

기존 테스트 20문항은 ROI, 단위/통합/E2E 경계, mock, flaky, CI 시간, 운영 방어 장치까지 포함하고 있어서 꽤 강하다. 다만 최근 프론트엔드 테스트는 단순 단위/E2E를 넘어서 계약 테스트, 시각 회귀, 접근성 자동화처럼 품질 전략의 폭이 더 넓어졌다.

아래 질문을 추가하면 테스트 도구를 써본 사람과 테스트 전략을 설계할 줄 아는 사람을 더 잘 나눌 수 있다.

### 추가하면 좋은 질문

#### 1. 테스트 피라미드와 테스트 트로피 개념을 어떻게 생각하나요?

단순 암기보다 테스트 전략 철학을 보기 좋은 질문이다. unit test 중심이 좋은지, integration 중심이 더 현실적인지 판단 기준을 들을 수 있다.

##### 합격 답변 예시

테스트 피라미드는 단위 테스트를 가장 많이, E2E를 가장 적게 쌓는 전통적인 모델입니다. 반면 테스트 트로피는 통합 테스트를 가장 두텁게 가져가는 모델로, Kent C. Dodds가 프론트엔드 맥락에서 제안했습니다. 프론트엔드는 컴포넌트, 훅, 상태, API 호출이 밀접하게 엮여 있어서 단위 테스트만으로는 실제 동작 신뢰도가 낮을 수 있습니다. 그래서 저는 통합 테스트 비중을 높게 가져가되, 순수 로직은 단위 테스트, 핵심 흐름은 E2E로 보완하는 트로피 형태가 프론트엔드에 더 현실적이라고 생각합니다.

##### 상세 해설

두 모델의 차이는 “어디에 가장 많은 테스트를 집중하느냐”다.

- 테스트 피라미드(Martin Fowler)
  - 단위 테스트가 가장 많고 빠르다
  - 위로 갈수록 느리고 비싸다
  - 백엔드, 마이크로서비스에서 유효하다
- 테스트 트로피(Kent C. Dodds)
  - 통합 테스트가 가장 두텁다
  - Static → Unit → Integration → E2E 순서
  - 프론트엔드에서 비용 대비 효과가 좋다

```typescript
// 피라미드 관점: 순수 로직은 단위 테스트
function calcDiscount(price: number, rate: number): number {
  if (rate < 0 || rate > 1) throw new Error('Invalid rate');
  return Math.round(price * (1 - rate));
}

// 단위 테스트
test('할인 계산이 정확하다', () => {
  expect(calcDiscount(10000, 0.1)).toBe(9000);
  expect(() => calcDiscount(10000, -1)).toThrow();
});
```

```tsx
// 트로피 관점: 컴포넌트 통합 테스트
test('쿠폰을 적용하면 할인 가격이 표시된다', async () => {
  render(<CheckoutPage />);
  await userEvent.type(screen.getByLabelText('쿠폰 코드'), 'SAVE10');
  await userEvent.click(screen.getByRole('button', { name: '적용' }));
  expect(await screen.findByText('할인 적용: -1,000원')).toBeInTheDocument();
});
```

핵심은 어느 모델이 맞느냐가 아니라, 프로젝트 특성에 맞게 비중을 조절할 줄 아느냐다.

#### 2. Contract Testing이 왜 필요하다고 보나요?

프론트와 백엔드 API 계약이 어긋나는 문제를 얼마나 실무적으로 이해하는지 확인할 수 있다. mock만으로는 놓치기 쉬운 위험을 설명하기 좋다.

##### 합격 답변 예시

프론트엔드가 mock 기반으로 테스트를 통과해도, 실제 백엔드 응답 형식이 바뀌면 운영에서 깨질 수 있습니다. Contract Testing은 프론트와 백엔드가 합의한 API 스펙을 별도 계약으로 정의하고, 양쪽이 각각 그 계약을 지키는지 독립적으로 검증하는 방식입니다. 이렇게 하면 통합 환경 없이도 API 변경으로 인한 연동 실패를 사전에 잡을 수 있습니다. 즉 Contract Testing은 mock의 신뢰 한계를 보완하는 장치입니다.

##### 상세 해설

mock 기반 테스트의 가장 큰 위험은 “mock이 실제와 달라도 테스트가 통과한다”는 점이다.

- Contract Testing이 해결하는 문제
  - 백엔드가 필드 이름을 바꿨는데 프론트가 모른다
  - 응답 구조가 바뀌었는데 mock은 그대로다
  - 프론트가 없는 필드를 기대하고 있다
- 대표 도구
  - Pact: Consumer-Driven Contract Testing 대표 도구
  - OpenAPI/Swagger 기반 자동 검증

```typescript
// Pact Consumer 테스트 예시 (프론트엔드 측)
import { PactV3 } from '@pact-foundation/pact';

const provider = new PactV3({
  consumer: 'FrontendApp',
  provider: 'UserService',
});

test('사용자 프로필을 정상 조회한다', async () => {
  await provider
    .given('사용자 ID 1이 존재한다')
    .uponReceiving('프로필 조회 요청')
    .withRequest({ method: 'GET', path: '/api/users/1' })
    .willRespondWith({
      status: 200,
      body: {
        id: 1,
        name: '홍길동',
        email: 'hong@example.com',
      },
    })
    .executeTest(async (mockServer) => {
      const res = await fetch(`${mockServer.url}/api/users/1`);
      const data = await res.json();
      expect(data.name).toBe('홍길동');
    });
});
```

Contract Testing은 E2E를 대체하는 것이 아니라, mock과 실제 서버 사이의 틈을 메우는 역할을 한다.

#### 3. Visual Regression Test는 언제 ROI가 높다고 보나요?

디자인 시스템, 마케팅 페이지, 어드민처럼 UI 변경 영향 범위가 넓은 제품에서 특히 유용하다. 기능 correctness와 시각 회귀를 분리해서 보는지 확인할 수 있다.

##### 합격 답변 예시

디자인 시스템 컴포넌트, 마케팅 랜딩 페이지, 어드민 대시보드처럼 UI 변경이 여러 곳에 영향을 미치는 경우에 ROI가 높다고 생각합니다. 기능 테스트는 동작이 맞는지를 보지만, 레이아웃이 밀렸는지, 색상이 바뀌었는지, 간격이 어긋났는지는 잡아주지 못합니다. Visual Regression Test는 스크린샷 비교를 통해 이런 시각적 변화를 자동으로 감지해줍니다. 다만 동적 콘텐츠가 많거나 자주 바뀌는 화면은 false positive가 잦아서 유지 비용이 높을 수 있습니다.

##### 상세 해설

Visual Regression Test는 “눈으로 봐야 아는 문제”를 자동화하는 것이다.

- ROI가 높은 경우
  - 디자인 시스템: 컴포넌트 하나가 수십 곳에 쓰인다
  - 마케팅 페이지: 레이아웃 깨짐이 매출에 직결된다
  - 다국어 지원: 텍스트 길이 변화로 레이아웃이 밀릴 수 있다
- ROI가 낮은 경우
  - 매일 바뀌는 대시보드
  - 동적 데이터가 많은 목록 페이지
- 대표 도구
  - Chromatic (Storybook 연동)
  - Percy
  - Playwright 내장 스크린샷 비교

```typescript
// Playwright Visual Regression Test 예시
import { test, expect } from '@playwright/test';

test('버튼 컴포넌트가 시각적으로 변하지 않았다', async ({ page }) => {
  await page.goto('/storybook/button--primary');
  await expect(page.locator('.btn-primary')).toHaveScreenshot(
    'button-primary.png',
    { maxDiffPixelRatio: 0.01 }
  );
});

test('결제 페이지 레이아웃이 유지된다', async ({ page }) => {
  await page.goto('/checkout');
  // 동적 데이터를 고정해서 스냅샷 안정성 확보
  await page.route('/api/cart', (route) =>
    route.fulfill({
      json: { items: [{ name: '테스트 상품', price: 10000 }] },
    })
  );
  await expect(page).toHaveScreenshot('checkout-layout.png');
});
```

핵심은 기능 correctness 테스트와 시각 회귀 테스트를 분리해서 관리하는 것이다. 둘을 섞으면 실패 원인 파악이 어려워진다.

#### 4. 접근성, a11y(Accessibility) 테스트는 자동화와 수동 검증을 어떻게 나누나요?

품질을 기능 correctness만으로 보지 않는 사람인지 드러나는 질문이다. 자동 검사 도구와 실제 키보드/스크린리더 확인의 경계를 설명할 수 있으면 좋다.

##### 합격 답변 예시

자동화 도구로는 axe-core, eslint-plugin-jsx-a11y 같은 것으로 ARIA 속성 누락, 색상 대비, alt 텍스트 빈값, 잘못된 역할 같은 기계적 규칙 위반을 잡을 수 있습니다. 하지만 실제 키보드 탐색 순서가 자연스러운지, 스크린리더가 읽는 흐름이 의미가 통하는지, 포커스 이동이 사용자 기대와 맞는지는 수동 확인이 필요합니다. 즉 자동화는 기본 품질 바닥을 깔아주고, 수동 검증은 실제 사용 경험 품질을 확인하는 역할입니다.

##### 상세 해설

a11y 테스트는 자동화만으로 전체 접근성 문제의 약 30~40% 정도만 잡을 수 있다는 점을 알아야 한다.

- 자동화가 잡을 수 있는 것
  - ARIA 속성 누락/잘못된 사용
  - 색상 대비 부족 (WCAG AA 기준)
  - img alt 텍스트 빈값
  - 중복된 id
  - label 연결 누락
- 수동 검증이 필요한 것
  - 키보드 탐색 순서와 흐름
  - 스크린리더 읽기 순서의 의미
  - 모달/드롭다운 포커스 트랩
  - 동적 콘텐츠 변경 알림 (live region)
  - 복잡한 위젯의 키보드 상호작용

```typescript
// axe-core를 활용한 자동 접근성 테스트
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';
```

#### 5. MSW(Mock Service Worker) 같은 API mocking 도구를 왜 쓰나요?

기존 API mocking 질문을 더 구체화해준다. 네트워크 계층을 실제에 가깝게 흉내 내면서도 테스트 안정성을 높이는 방법을 설명할 수 있다.

##### 합격 답변 예시

MSW는 Service Worker를 이용해 네트워크 수준에서 요청을 가로채기 때문에, 애플리케이션 코드를 전혀 수정하지 않고도 API 응답을 제어할 수 있습니다. jest.mock이나 함수 레벨 mocking은 구현에 의존하게 되지만, MSW는 실제 fetch/axios 호출이 그대로 실행되고 네트워크 계층에서만 응답을 바꾸므로 테스트 신뢰도가 더 높습니다. 또한 브라우저 환경(Storybook, 개발 서버)과 Node 환경(Vitest, Jest) 모두에서 같은 핸들러를 재사용할 수 있어서 mock 관리 비용이 줄어듭니다.

##### 상세 해설

MSW의 핵심 가치는 “테스트 코드가 아니라 네트워크 계층에서 mocking한다”는 점이다.

- 기존 방식의 문제
  - jest.mock으로 모듈을 직접 교체 → 구현 의존
  - axios adapter 교체 → 라이브러리 변경 시 깨짐
  - 전역 fetch를 덮어쓰기 → 부작용 위험
- MSW의 장점
  - 네트워크 수준 가로채기 → 코드 무수정
  - 브라우저/Node 환경 동일 핸들러
  - 에러, 지연, 부분 실패 시뮬레이션 용이
  - Storybook, 개발 서버에서도 활용 가능

```typescript
// MSW 핸들러 정의
import { http, HttpResponse, delay } from 'msw';

export const handlers = [
  // 정상 응답
  http.get('/api/users', () => {
    return HttpResponse.json([
      { id: 1, name: '홍길동' },
      { id: 2, name: '김영희' },
    ]);
  }),

  // 에러 응답
  http.get('/api/users/:id', ({ params }) => {
    if (params.id === '999') {
      return new HttpResponse(null, { status: 404 });
    }
    return HttpResponse.json({ id: params.id, name: '홍길동' });
  }),

  // 네트워크 지연 시뮬레이션
  http.post('/api/orders', async () => {
    await delay(2000);
    return HttpResponse.json({ orderId: 'ORD-001' });
  }),
];
```

```typescript
// Vitest에서 MSW 사용
import { setupServer } from 'msw/node';
import { handlers } from './handlers';

const server = setupServer(...handlers);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

test('사용자 목록을 불러와 표시한다', async () => {
  render(<UserList />);
  expect(await screen.findByText('홍길동')).toBeInTheDocument();
  expect(screen.getByText('김영희')).toBeInTheDocument();
});

test('API 실패 시 에러 메시지를 표시한다', async () => {
  // 특정 테스트에서만 핸들러 오버라이드
  server.use(
    http.get('/api/users', () => {
      return new HttpResponse(null, { status: 500 });
    })
  );
  render(<UserList />);
  expect(await screen.findByText('목록을 불러올 수 없습니다')).toBeInTheDocument();
});
```

핵심은 MSW를 “mock 도구”로만 보지 않고, 테스트와 개발 환경 전반에서 네트워크 계약을 일관되게 관리하는 도구로 이해하는 것이다.

#### 6. Coverage 수치가 높다고 좋은 테스트라고 볼 수 없는 이유는 무엇인가요?

이 질문은 테스트를 숫자 게임으로 보는 사람을 걸러낸다. “무엇을 실행했는가”와 “무엇을 의미 있게 검증했는가”의 차이를 설명할 수 있어야 한다.

##### 합격 답변 예시

Coverage는 코드가 실행되었는지만 측정하지, 그 실행 결과를 의미 있게 검증했는지는 알려주지 않습니다. 예를 들어 assertion 없이 함수를 호출만 해도 커버리지는 올라갑니다. 또한 100% 커버리지라 해도 경계값, 에러 경로, 비동기 타이밍 같은 실제 장애 원인을 놓칠 수 있습니다. 그래서 커버리지는 테스트가 닿지 않는 사각지대를 찾는 보조 지표로는 유용하지만, 테스트 품질의 기준이 되어서는 안 됩니다.

##### 상세 해설

커버리지의 한계를 정확히 아는 것이 중요하다.

- 커버리지가 높아도 놓치는 것
  - assertion 없는 테스트 → 실행만 하고 검증 안 함
  - 정상 경로만 테스트 → 에러/경계값 빠짐
  - 비동기 완료 전 검증 → flaky하지만 커버리지는 잡힘
  - 구현 세부에 묶인 테스트 → 숫자는 높지만 리팩터링에 약함
- 커버리지가 유용한 경우
  - 아예 테스트가 닿지 않는 코드를 발견할 때
  - PR 리뷰에서 새 코드의 테스트 유무를 확인할 때
  - 팀 최소 기준선을 설정할 때 (예: 70% 이상)

```typescript
// 커버리지는 100%지만 나쁜 테스트
function divide(a: number, b: number): number {
  if (b === 0) throw new Error('Division by zero');
  return a / b;
}

// assertion 없이 호출만 → 커버리지는 올라감
test('divide를 호출한다', () => {
  divide(10, 2);
  divide(10, 0).catch(() => {}); // 에러를 무시
  // expect가 없다 → 아무것도 검증하지 않았다
});
```

```typescript
// 커버리지는 같지만 좋은 테스트
test('정상 나눗셈 결과를 반환한다', () => {
  expect(divide(10, 2)).toBe(5);
  expect(divide(-10, 2)).toBe(-5);
  expect(divide(0, 5)).toBe(0);
});

test('0으로 나누면 에러가 발생한다', () => {
  expect(() => divide(10, 0)).toThrow('Division by zero');
});
```

커버리지 수치를 목표로 삼으면 “수치를 올리기 위한 테스트”를 쓰게 된다. 좋은 팀은 커버리지를 최소 기준선으로만 쓰고, 테스트의 진짜 가치는 변경 내성과 장애 예방으로 판단한다.

### 이 섹션 보완 핵심

테스트 섹션은 이미 좋다. 여기에 전략 철학, 계약 테스트, 시각 회귀, 접근성, API mocking 도구, coverage 한계까지 더하면 “테스트 도구 사용자”와 “테스트 전략 설계자”를 더 잘 구분할 수 있다.
