# 04. nextjs20문항

> App Router, 서버·클라이언트 경계, 캐싱과 배포 관점을 다루는 20문항입니다.

## 1. Next.js를 React 단독 구성보다 선호하는 이유를 설명해보세요.

### 합격 답변 예시

Next.js를 React 단독 구성보다 선호하는 가장 큰 이유는, React 위에 라우팅, 서버 렌더링, 데이터 페칭 구조, 이미지/메타데이터 최적화, Route Handlers, 배포 친화적 빌드 구성 같은 웹 애플리케이션에 필요한 기능을 프레임워크 수준에서 제공하기 때문입니다. 공식 문서도 Next.js를 full-stack web applications를 위한 React framework로 설명하고, bundler/compiler 같은 하위 도구도 자동 설정해준다고 말합니다. 팀 생산성 관점에서는 “기능이 많다”보다 기본 구조를 통일한다는 점이 더 큽니다. 프로젝트마다 라우터, SSR(Server-Side Rendering) 설정, 이미지 최적화, SEO(Search Engine Optimization), API(Application Programming Interface) layer를 따로 조립하지 않아도 되니 의사결정 비용이 줄고, App Router 기준으로는 layouts, metadata, route handlers, proxy, error/loading 같은 패턴이 파일 규칙으로 강제돼서 팀 내 일관성을 만들기 좋습니다.

### 상세 해설

React만으로도 앱은 만들 수 있다. 하지만 React는 기본적으로 UI 라이브러리이고, Next.js는 그 위에 웹앱을 실제로 운영하기 위한 기본 인프라를 얹는다. 최신 docs 첫 화면이 App Router 기준으로 layouts, navigation, Server/Client Components, cache, route handlers, metadata, image optimization, proxy, deploying까지 한 흐름으로 제공하는 이유가 바로 그거다. 면접에서는 여기서 한 걸음 더 들어가야 한다.

- “왜 우리 팀에서 Next.js가 더 나았는가” 를 말해야 한다. 예를 들면:
- 검색 유입이 있어 SEO가 중요했다
- 로그인 대시보드와 공개 콘텐츠가 같이 있었다
- 이미지와 OG(Open Graph) metadata 최적화가 중요했다
- BFF(Backend For Frontend)/API layer를 같이 두고 싶었다
- 서버/클라이언트 경계를 프레임워크 레벨에서 관리하고 싶었다

이런 식으로 제품 요구사항과 연결하면 강하다.

## 2. Pages Router와 App Router의 차이를 설명해보세요.

### 합격 답변 예시

Pages Router는 기존의 파일 시스템 라우터로, pages 디렉토리 기반이고 getServerSideProps, getStaticProps, API Routes 같은 모델을 중심으로 동작합니다. 반면 App Router는 app 디렉토리 기반으로 Server Components, nested layouts, loading/error/not-found 같은 special files, Route Handlers, Metadata API를 중심으로 설계된 최신 모델입니다. 공식 docs도 Pages Router는 계속 지원되지만, 최신 기능은 App Router 쪽을 기준으로 설명하고 있습니다. 또 마이그레이션 관점에서 App Router는 단순히 폴더만 바뀌는 게 아니라 mental model이 바뀝니다. pages/_app, pages/_document, pages/_error, pages/api/*, getServerSideProps/getStaticProps/getStaticPaths 같은 개념이 각각 app/layout, error.js, route.js, 직접적인 fetch, generateStaticParams 등으로 재구성됩니다.

### 상세 해설

핵심 차이는 아래다.

- Pages Router
- 페이지 중심, 페이지별 데이터 페칭 함수 중심, API Routes 중심
- App Router

세그먼트/레이아웃 중심, Server Components 중심, special files 중심, 더 세밀한 streaming/caching 중심 면접에서 좋은 답변은 “App Router가 더 최신이다”에서 끝나지 않는다. 왜 더 나은가 / 언제 Pages Router가 아직 현실적인가까지 가야 한다.

- 신규 앱: App Router 우선
- 기존 대형 레거시: app과 pages 공존시키며 점진 이전 가능

공식 가이드도 app과 pages를 동시에 둘 수 있고, page-by-page migration을 권장한다.

## 3. SSR, SSG(Static Site Generation), ISR(Incremental Static Regeneration), CSR(Client-Side Rendering)의 차이와 각각의 적절한 사용 사례를 설명해보세요.

### 합격 답변 예시

SSR은 요청마다 HTML을 서버에서 생성하는 방식이고, SSG는 빌드 시점에 HTML을 생성해 재사용하는 방식입니다. ISR은 정적 생성의 장점을 유지하면서 일정 주기나 on-demand 방식으로 정적 결과를 갱신하는 전략이고, CSR은 초기엔 최소 HTML과 JS를 내려보내고 실제 렌더를 클라이언트가 수행하는 방식입니다. Next.js 공식 문서도 SSR은 request time, SSG는 build time, ISR은 static page를 runtime에 갱신, CSR은 클라이언트 JS 중심 렌더링으로 설명합니다. 사용 사례는 보통 이렇게 봅니다. 로그인별 개인화가 강하고 최신성이 중요하면 SSR, 공개 콘텐츠나 문서처럼 빠른 응답과 CDN(Content Delivery Network) 캐시가 중요하면 SSG, 자주 바뀌지만 매 요청 SSR까진 과한 목록/카탈로그는 ISR, SEO가 크게 중요하지 않고 인터랙션 중심인 내부 도구 일부 화면은 CSR이 맞습니다. 다만 App Router에서는 이걸 단순히 페이지 단위 이분법으로 보기보다, route prerender + fetch/cache/revalidate + client boundary 조합으로 생각하는 편이 더 최신에 가깝습니다.

### 상세 해설

Pages Router 기준 공식 정의는 비교적 명확하다.

- SSR: 요청 때마다 HTML 생성
- SSG: 빌드 시 HTML 생성, CDN 캐시 가능
- ISR: 정적 페이지를 런타임에 갱신
- CSR: 브라우저가 JS를 받아 DOM을 구성, 초기 표시 지연 가능

하지만 최신 App Router에서는 이 분류만 외우면 부족하다. 공식 docs는 캐싱을 route shell, data cache, revalidate, dynamic APIs와 함께 설명하고, Cache Components로 정적/동적을 한 라우트 안에서 섞을 수 있다고 말한다. 즉 지금 면접에서는 “SSG냐 SSR이냐”만 답하면 약하고, 어떤 부분은 prerender하고 어떤 부분은 request-time으로 미루는지까지 말해야 강하다.

## 4. Next.js에서 서버 컴포넌트와 클라이언트 컴포넌트의 차이를 설명해보세요.

### 합격 답변 예시

App Router에서 기본은 Server Component입니다. 공식 문서는 layout과 page가 기본적으로 Server Component라고 설명하고, 이 컴포넌트들은 서버에서 데이터에 가깝게 접근하고, 비밀값을 숨기고, 클라이언트로 보내는 JavaScript 양을 줄이며, 결과를 캐시하거나 스트리밍할 수 있다고 말합니다. 반면 Client Component는 state, event handler, useEffect, 브라우저 API, custom hook이 필요할 때 사용합니다. 정리하면 Server Component는 데이터 접근/비밀 보호/초기 렌더 최적화에 강하고, Client Component는 인터랙션과 브라우저 기능에 강합니다. 그래서 실무에서는 페이지의 큰 껍데기와 데이터 fetch는 서버 쪽에 두고, 버튼/입력/애니메이션 같은 상호작용 조각만 client boundary로 내려보내는 식으로 설계하는 게 일반적입니다.

### 상세 해설

공식 문서는 언제 Client를 쓰고 언제 Server를 쓰는지를 아주 분명히 적어둔다.

- Client가 필요한 경우:
- state, event handlers
- lifecycle/useEffect
- browser APIs
- custom hooks
- Server가 유리한 경우:
- DB/API에 가까운 fetch
- API key, token, secret 보호
- JS bundle 축소
- FCP 개선과 streaming

면접에서는 여기서 “둘 중 하나만 쓰는 게 아니라 조합해서 쓴다”까지 말해야 좋다. 공식 docs도 서버가 데이터를 fetch해서 client component에 props로 넘기고, client는 상호작용만 담당하는 예시를 계속 보여준다.

## 5. use client가 붙는 순간 어떤 제약과 비용이 생기나요?

### 합격 답변 예시

'use client'는 해당 파일이 서버-클라이언트 경계의 entry point가 되도록 선언하는 directive입니다. 이 파일에서 export한 컴포넌트들은 클라이언트 쪽에서 렌더될 수 있고, state/event/browser API를 쓸 수 있습니다. 대신 공식 docs 기준으로 Server Component에서 Client Component로 넘기는 props는 serializable해야 하고, 그 지점부터 클라이언트 번들에 포함될 수 있는 비용이 생깁니다. 즉 비용은 두 가지입니다. 첫째, JS bundle이 늘 수 있습니다. 둘째, 서버에서만 안전하게 두고 싶던 로직을 잘못 경계 안으로 끌고 오면 서버 전용 코드 분리가 어려워집니다. 그래서 use client는 “편하니까 위에 붙이는 문장”이 아니라, 정말 인터랙션이 필요한 leaf에만 좁게 두는 것이 좋습니다. 공식 docs도 모든 Client Component 파일마다 붙일 필요는 없고, boundary entry file에만 두면 된다고 설명합니다.

### 상세 해설

많이 하는 실수가 page.tsx나 큰 layout에 통째로 use client를 붙이는 것이다. 그러면 그 경계 아래가 필요 이상으로 클라이언트 세계로 밀려서, 서버에서 할 수 있었던 일들을 잃고 JS 전송량도 늘어난다. 공식 문서가 “Reduce JS bundle size”를 계속 강조하는 이유가 여기 있다. 면접에서 가산점 포인트는 이것이다. use client의 진짜 비용은 문법 비용이 아니라 boundary 비용이다.

- 즉,
- props 직렬화 제약
- client JS 증가
- 서버 전용 비밀/무거운 로직을 boundary 위로 올려야 하는 설계 압박

이걸 말하면 강하다.

## 6. getServerSideProps, getStaticProps를 써본 경험이 있다면 각각 언제 썼는지 설명해보세요.

### 합격 답변 예시

Pages Router에서는 getServerSideProps는 요청 시점마다 최신 데이터가 필요할 때, getStaticProps는 빌드 시점 생성과 CDN 캐시가 유리할 때 사용했습니다. 공식 docs도 getServerSideProps는 request time rendering용이고, getStaticProps는 build time prerender와 CDN-friendly한 사용처를 설명합니다. 다만 App Router에서는 이 함수를 더 이상 쓰지 않고, 공식 마이그레이션 가이드 기준으로 fetch와 async Server Components, generateStaticParams 같은 새 API로 옮겨갑니다. 그래서 면접에서는 “예전엔 Pages Router에서 이렇게 썼고, 지금 신규 프로젝트라면 App Router에선 fetch의 cache/no-store/revalidate로 같은 의도를 표현하겠다”고 연결해서 답하면 좋습니다.

### 상세 해설

- getStaticProps가 맞는 대표 사례:
- 블로그 글
- 상품 상세처럼 공개 페이지
- CMS 기반 문서
- 자주 안 바뀌고 SEO가 중요한 페이지
- getServerSideProps가 맞는 대표 사례:
- 사용자별 맞춤 대시보드
- 요청 시 쿠키/헤더 기반 분기
- 항상 최신성이 중요하고 공개 캐시가 어려운 데이터

그리고 지금 면접에서는 꼭 덧붙여야 한다. **“App Router에선 mental model이 바뀌었다”**고. 공식 가이드가 직접 getServerSideProps/getStaticProps는 app에서 더 단순한 fetch 기반 API로 대체되었다고 적고 있다.

## 7. Next.js에서 SEO를 고려한 페이지 설계를 어떻게 하나요?

### 합격 답변 예시

최신 App Router에서는 next/head보다 Metadata API를 기본 축으로 잡는 게 맞습니다. metadata 객체나 generateMetadata를 사용해 title, description, OG/Twitter metadata를 정의하고, 필요하면 route segment별로 동적으로 생성합니다. 공식 docs도 Metadata API와 generateMetadata를 SEO와 web shareability 개선용으로 설명합니다. 또 파일 기반 metadata도 중요합니다. opengraph-image, twitter-image, robots, sitemap, favicon/icon/apple-icon 같은 special file convention을 쓰면 SEO와 공유 경험을 더 일관되게 설계할 수 있습니다. 공식 문서는 metadata files와 OG image special files를 별도로 안내합니다.

### 상세 해설

SEO 설계에서 내가 보는 핵심은 세 가지다.

- 초기 HTML에 의미 있는 콘텐츠가 있어야 한다
- route별 metadata가 도메인 모델과 같이 움직여야 한다
- 공유용 OG/Twitter, robots, sitemap을 따로 챙겨야 한다

최신 docs에서 흥미로운 포인트는 streaming metadata다. generateMetadata가 늦게 끝나더라도 초기 UI를 먼저 보낼 수 있고, JS를 실행하는 봇은 나중에 붙는 metadata도 해석할 수 있다. 다만 HTML-limited bots에 대해서는 metadata가 head에서 blocking 동작을 유지한다. 즉 SEO도 단순 title 몇 개가 아니라 bot 특성과 렌더 타이밍까지 포함한 설계가 된다.

## 8. 이미지 최적화는 어떻게 이뤄지고, next/image 사용 시 주의할 점은 무엇인가요?

### 합격 답변 예시

공식 문서 기준 next/image는 사이즈 최적화, CLS 방지, native lazy loading, blur placeholder, remote image on-demand resize 같은 최적화를 제공합니다. 즉 단순 <img> wrapper가 아니라 이미지 전달 전략 자체를 프레임워크가 관리해주는 컴포넌트입니다. 주의할 점은 몇 가지가 있습니다. 외부 이미지는 remotePatterns로 허용 범위를 엄격하게 제한해야 하고, fill을 쓰면 부모에 positioning과 레이아웃 공간이 있어야 하며, responsive 이미지라면 sizes를 제대로 써야 브라우저가 과도하게 큰 이미지를 받지 않습니다. 공식 docs도 fill의 부모 position 요구사항과, sizes를 빠뜨리면 100vw로 가정돼 불필요하게 큰 이미지가 다운로드될 수 있다고 설명합니다.

### 상세 해설

실무에서 가장 자주 놓치는 건 세 가지다.

- remotePatterns 안 잡고 외부 도메인만 대충 허용
- fill 썼는데 부모에 position: relative 안 줌
- 반응형인데 sizes 안 써서 과한 이미지 다운로드

추가로, 인증 헤더가 필요한 원본 이미지는 기본 이미지 최적화 loader가 헤더를 전달하지 않기 때문에 주의해야 한다. 공식 docs는 이런 경우 unoptimized를 고려하라고 한다. 또 SVG는 기본적으로 최적화 대상이 아니고, dangerouslyAllowSVG를 쓰면 CSP와 content disposition을 함께 고려하라고 안내한다.

## 9. 라우팅 구조를 설계할 때 URL과 UI 상태를 어떻게 나누나요?

### 합격 답변 예시

기본 원칙은 공유·북마크·새로고침·직접 진입 시 복원되어야 하는 상태는 URL, 일시적이고 화면 내부 표현에 가까운 상태는 로컬 UI state로 둡니다. 예를 들어 검색어, 필터, 정렬, 페이지네이션, 현재 탭처럼 화면 의미를 바꾸는 상태는 URL에 두는 편이 좋고, 모달 open 여부나 hover, temporary selection 같은 것은 로컬 상태에 두는 편이 낫습니다. Next.js App Router도 dynamic segments, searchParams, useSearchParams를 통해 URL 기반 상태를 공식적으로 지원합니다. 또 App Router에서는 위치에 따라 읽는 방식도 중요합니다. 공식 docs는 layout은 navigation 시 재렌더되지 않으므로 search params를 직접 다루면 stale해질 수 있다고 설명하고, 최신 query는 Page의 searchParams prop이나 Client Component의 useSearchParams로 다루라고 권장합니다.

### 상세 해설

면접에서 이 질문은 사실 “라우트 설계”보다 상태 모델링 질문에 가깝다.

- URL에 두면 좋은 것:
- 검색
- 필터
- 정렬
- 페이지 번호
- 선택된 리소스 id
- 공유 가능한 뷰 상태
- URL에 두지 않는 게 좋은 것:
- 애니메이션 진행률
- hover
- 드래그 중 좌표
- 일시적 open/close
- 입력 중 임시값 일부

그리고 Next.js 특성상 layout은 query/pathname 최신값 source가 아니다라는 점을 꼭 말하면 좋다. 이건 App Router를 써본 사람만 잘 말하는 포인트다.

## 10. 검색/필터 상태를 URL 쿼리와 동기화하는 장점과 단점을 설명해보세요.

### 합격 답변 예시

장점은 명확합니다. URL에 동기화하면 공유, 북마크, 새로고침 복원, 뒤로가기/앞으로가기, 서버 측 데이터 페칭과 직접 연결이 쉬워집니다. 공식 docs도 next/form과 useSearchParams, Page searchParams prop을 통해 검색 폼을 URL search params와 자연스럽게 연결하는 패턴을 보여줍니다. 단점도 있습니다. 쿼리가 너무 자주 변하면 history가 지저분해질 수 있고, 모든 UI 상태가 URL에 올라가면 직렬화와 파싱 비용이 커지고 의미도 흐려집니다. 또 공식 에러 문서처럼 useSearchParams()를 잘못 쓰면, 특히 Suspense boundary 없이 사용할 때 페이지 전체가 client-side rendering으로 deopt될 수 있어 주의가 필요합니다.

### 상세 해설

최신 docs 기준으로 검색 폼은 next/form이 꽤 실용적이다. 문자열 action이면 GET처럼 동작해서 form data를 URL search params로 붙이고, client-side navigation과 prefetching까지 제공한다. 그래서 검색/필터 화면에 잘 맞는다. 다만 URL 상태는 “영속 상태”에 가까워서 너무 자주 바뀌는 입력 중간 상태까지 모두 넣는 건 안 좋다.

- 내 기준은:
- 제출/적용된 검색 조건 → URL
- 입력창에서 아직 확정 안 된 draft 값 → 로컬 state

이렇게 분리하는 답변이 가장 실무적이다.

## 11. Server Action은 무엇이고, API Route 대비 어떤 장단점이 있나요?

### 합격 답변 예시

최신 docs 기준으로 넓게는 Server Function, 그중 action/mutation 맥락에서 쓰이면 Server Action이라고 부릅니다. use server로 선언되고, form의 action이나 button의 formAction, 혹은 transition 맥락에서 호출될 수 있습니다. 공식 docs는 action이 호출되면 updated UI와 new data를 한 번의 server roundtrip으로 반환할 수 있고, Next.js 캐싱 아키텍처와 통합된다고 설명합니다. 반면 Route Handlers는 route.ts에서 Web Request/Response API 기반으로 HTTP 메서드를 직접 처리하는 방식입니다. 그래서 Server Action은 UI 가까운 mutation에 좋고, Route Handlers는 공개 API, 웹훅, 업로드, 외부 클라이언트가 호출하는 엔드포인트, 세밀한 HTTP 제어에 더 적합합니다. 다만 중요한 주의점으로, 공식 data security 문서는 exported Server Action이 public HTTP endpoint처럼 취급되어야 하며 인증/인가 체크가 필요하다고 분명히 말합니다.

### 상세 해설

장단점을 정리하면 이렇다.

- Server Action 장점
- 폼과 자연스럽게 연결
- 캐시 무효화/redirect/revalidate와 궁합 좋음
- UI mutation 코드가 서버 로직과 가깝다
- Server Action 단점
- 외부 공개 API 스타일 설계에는 덜 자연스러움
- 보안상 내부 함수처럼 생각하면 위험
- 큰 업로드/복잡한 HTTP 프로토콜 처리엔 Route Handler가 더 명확한 경우가 많음
- Route Handler 장점
- GET/POST/PUT/PATCH/DELETE 등 메서드 단위 제어
- Request/Response, cookies, headers 제어 쉬움
- BFF/API layer에 적합

## 12. BFF 패턴을 Next.js로 구현한다면 어떤 식으로 구성하겠습니까?

### 합격 답변 예시

최신 docs는 아예 Next.js가 Backend for Frontend 패턴을 지원한다고 설명합니다. 공식 가이드에 따르면 Route Handlers로 public endpoint를 만들고, 그 안에서 데이터 소스 접근이나 side effect를 수행할 수 있습니다. 그래서 저는 BFF를 구현할 때 app/api/.../route.ts를 entry point로 두고, 그 아래에 data access layer / service layer를 분리해서 외부 API 조합, 인증 토큰 처리, 응답 정규화, 프론트용 DTO(Data Transfer Object) shaping을 맡기겠습니다. 다만 공식 docs도 Next.js backend capabilities는 full backend replacement가 아니라 API layer라고 선을 긋습니다. 그래서 장기 배치 작업, 복잡한 비동기 워커, 강한 도메인 트랜잭션이 많은 시스템이라면 BFF는 Next.js에 두되, 코어 백엔드는 별도 서비스로 나누는 답변이 더 현실적입니다.

### 상세 해설

좋은 BFF 답변은 “API Route 하나 만들겠습니다”에서 끝나면 약하다. 면접관이 듣고 싶은 건 보통 이 구조다.

- Route Handlers: 외부 공개 인터페이스
- Auth/session 읽기
- Upstream API 합성
- 응답 스키마 정규화
- 캐시/revalidate 정책
- 에러 매핑
- 프론트가 소비하기 좋은 DTO 반환
- 즉 BFF의 핵심은 “서버 코드가 있는 것”이 아니라

프론트 요구사항 중심으로 백엔드 응답을 재구성하는 것이다.

## 13. 인증/인가를 Next.js 프로젝트에서 어떻게 처리해봤는지 설명해보세요.

### 합격 답변 예시

공식 auth 가이드처럼 저는 인증을 authentication / session management / authorization으로 나눠서 설명하는 편입니다. 로그인 시 세션을 만들고, 서버 쪽에서는 cookies() 같은 API로 세션 쿠키를 읽고, 데이터 접근 시점에 권한 검사를 수행합니다. 공식 docs도 이 세 가지를 명확히 구분하고, cookies()가 Server Components와 Server Functions, Route Handlers에서 쿠키를 읽고 쓰는 API라고 설명합니다. 또 최신 docs 기준으로 Proxy는 초기 리다이렉트나 optimistic check에는 유용하지만, 주요 보안 검사는 데이터 소스에 최대한 가깝게 해야 합니다. 공식 auth 문서도 Proxy에서는 매 라우트마다 실행되므로 DB 조회 대신 cookie 기반의 빠른 optimistic check만 권장하고, 진짜 보호는 data access layer 가까이서 하라고 안내합니다.

### 상세 해설

면접에서 강한 답변 구조는 이렇다.

- 로그인 성공 → 세션 생성
- 세션은 보통 cookie 기반 저장
- 서버에서 쿠키 해독/검증
- 페이지 보호는 Proxy로 1차 UX 가드
- 실제 데이터 접근은 서버 함수/데이터 계층에서 다시 권한 검사

즉 Proxy만으로 auth를 끝냈다고 말하면 약하다. 공식 docs가 직접 “Proxy는 your only line of defense가 아니다”라고 말하는 이유가 그거다.

## 14. 세션 만료가 자주 발생하는 서비스라면, Next.js 레벨에서 어떤 사용자 경험 보호 장치를 넣겠습니까?

### 합격 답변 예시

공식 auth 문서는 세션 만료 시간을 연장하는 updateSession() 예시와 refresh token 사용 팁을 제공합니다. 그래서 세션 만료가 잦은 서비스라면 먼저 서버에서 세션 갱신 전략을 두고, 사용자가 다시 접근했을 때 쿠키 만료 시간을 연장하거나 refresh token 기반 재발급을 시도하는 흐름을 두겠습니다. UX 보호 장치로는, Proxy에서 쿠키 기반 optimistic check를 해 만료 사용자를 로그인 페이지로 부드럽게 보내되, 원래 가려던 URL을 유지해 재로그인 후 복귀시키고, mutation 실패 시에는 route segment의 error.js나 명시적 폼 에러 상태로 안내하는 편이 좋습니다. App Router는 error.js로 segment 단위 fallback UI를 만들 수 있고, Proxy는 redirect/rewrite/headers 조작에 적합합니다.

### 상세 해설

이 질문은 인증 지식보다 UX 설계 감각을 본다. 좋은 답변은 보통 이 순서다.

- 만료를 늦춘다 — 세션 업데이트/refresh token
- 조용히 복구 가능하면 복구한다
- 안 되면 로그인 유도
- 원래 의도한 작업과 경로를 최대한 보존한다
- 재로그인 후 이어서 진행하게 한다

여기서 가산점은 “Proxy는 가벼운 체크만, 실제 권한은 서버에서 재검증”을 다시 연결하는 것이다.

## 15. 대용량 CSV 업로드 이후 결과를 바로 화면에 반영해야 할 때 Next.js에서 어떤 흐름으로 구현하겠습니까?

### 합격 답변 예시

이 경우 저는 업로드는 Route Handler, 결과 반영은 Server Action/Revalidation 또는 polling으로 나누겠습니다. 이유는 Route Handler가 custom request handler로서 POST 업로드 같은 HTTP 작업을 다루기에 더 명확하고, 이후 결과 반영은 Next.js의 revalidation과 잘 연결되기 때문입니다. 공식 docs는 Route Handlers가 custom request handlers이며 여러 HTTP method를 지원한다고 설명하고, Server Functions/Actions는 mutation 이후 updated UI와 data를 한 번의 roundtrip으로 돌려줄 수 있다고 설명합니다. 흐름은 보통 이렇게 답하면 강합니다. CSV를 POST /api/upload 같은 Route Handler로 받는다. 서버에서 파싱/검증/비동기 처리 job을 시작하고 job id를 반환한다. 클라이언트는 pending UI를 보여주고 job 상태를 polling하거나 SSE/WebSocket 계층이 있으면 연결한다. 완료 시 revalidatePath 또는 revalidateTag로 관련 목록을 무효화해서 최신 결과를 바로 반영한다. 공식 docs도 revalidatePath를 Route Handler나 Server Action에서 사용할 수 있다고 안내합니다.

### 상세 해설

이 질문은 한마디로 **“무거운 작업과 UI 반영을 어떻게 분리하느냐”**를 본다. 대용량 CSV를 Server Action 하나에 다 몰아넣는 답변은 종종 약하다. 왜냐하면 업로드/장시간 처리/상태 추적은 HTTP handler와 비동기 job 모델이 더 명확하기 때문이다. Route Handlers는 그 entry로 자연스럽다. 반면 처리 완료 후 화면을 다시 갱신하는 순간은 Next.js의 강점이 나온다. revalidatePath, revalidateTag, redirect, 혹은 action 후 UI refresh를 써서 “업로드는 HTTP, 반영은 framework cache invalidation” 으로 나누는 답변이 가장 좋다.

## 16. 모노레포에서 Next.js 앱을 여러 개 운영할 때 공통 패키지는 어떻게 관리하나요?

### 합격 답변 예시

공통 UI, 타입, 유틸, API SDK 같은 것은 workspace 패키지로 분리하고, Next.js 앱에서는 transpilePackages로 로컬 패키지를 자동 transpile/bundle하도록 설정하는 방식이 기본입니다. 공식 docs도 transpilePackages가 local packages like monorepos를 자동으로 transpile/bundle한다고 설명합니다. 앱이 여러 개이고 하나의 도메인 아래 path 단위로 쪼개야 한다면 Multi-Zones도 고려할 수 있습니다. 공식 docs는 Multi-Zones를 여러 Next.js 앱을 한 도메인 아래 경로별로 분리 배포하는 micro-frontends 접근으로 설명합니다. 그래서 “공통 패키지 공유”와 “앱 자체 분리 배포”는 서로 다른 축으로 관리하는 게 좋습니다.

### 상세 해설

이 질문에서 듣고 싶은 포인트는 보통 세 가지다.

- shared package 경계
- 빌드/번들 문제
- 앱 분할 전략

실무적으로는 이렇게 답하면 강하다. packages/ui, packages/config, packages/types, packages/api-client

- 각 앱은 정말 필요한 공용 패키지만 의존
- client 전용 코드와 server 전용 코드를 패키지 경계에서도 섞지 않음
- 필요 시 output: 'standalone' 같은 배포 전략도 같이 고려

## 17. middleware는 어떤 경우에 쓰고, 남용하면 왜 안 되나요?

### 합격 답변 예시

최신 docs 기준으로는 Next.js 16부터 Middleware가 Proxy로 이름이 바뀌었습니다. 기능은 이어지지만, 공식 문서는 이제 Proxy라는 용어를 사용합니다. Proxy는 요청이 완료되기 전에 실행되어 rewrite, redirect, header 수정, 직접 응답 등을 할 수 있습니다. 좋은 사용처는 초기 인증 리다이렉트, 지역/언어 분기, A/B 라우팅, 특정 경로 헤더 조정 같은 가벼운 request-level 분기입니다. 남용하면 안 되는 이유는 공식 auth docs가 말하듯 Proxy는 모든 라우트, 심지어 prefetched route에도 걸릴 수 있어서, 여기서 무거운 DB 조회나 복잡한 비즈니스 로직을 돌리면 성능이 바로 나빠질 수 있기 때문입니다. 그래서 공식 예시도 auth에서는 cookie만 읽는 optimistic check를 권장합니다.

### 상세 해설

면접에서는 이렇게 분리해서 말하면 좋다.

- Proxy에 넣어도 되는 것
- redirect/rewrite
- 간단한 쿠키 확인
- 경량 헤더 조작
- 경로별 빠른 분기
- Proxy에 넣지 않는 게 좋은 것
- 무거운 DB 질의
- 핵심 권한 판정의 전부
- 업로드 처리
- 복잡한 도메인 로직

그리고 최신 docs 기준으로 Proxy는 Node.js runtime 사용 관련 안내도 있으니, 라이브러리 런타임 호환성까지 확인하는 답변이면 더 좋다.

## 18. Next.js에서 캐싱 전략을 어디까지 프레임워크에 맡기고 어디부터 직접 제어하나요?

### 합격 답변 예시

최신 docs는 캐싱을 “개발자가 전부 수동으로 관리하는 것”보다, 프레임워크 기본 휴리스틱 위에 필요한 부분만 직접 제어하는 방식으로 설명합니다. 공식 caching guide도 대부분의 휴리스틱은 API 사용에 따라 자동 결정되고, 기본값만으로도 성능상 괜찮게 동작하도록 설계됐다고 말합니다. 그래서 저는 기본적으로는 framework default를 따르고, 데이터 특성상 명확한 요구가 있을 때만 직접 제어합니다. 예를 들면 서버 fetch에 cache: 'no-store', cache: 'force-cache', next.revalidate, tags, revalidatePath, revalidateTag, use cache, route segment config 같은 API를 사용합니다. 공식 docs도 서버 fetch의 Data Cache semantics, revalidatePath, revalidateTag, use cache, Route Handler의 기본 uncached 동작을 각각 설명합니다.

### 상세 해설

이 질문에서 가장 중요한 건 **“캐싱 단위가 하나가 아니다”**라는 점이다.

- route shell / prerendered HTML
- fetch Data Cache
- tag/path 기반 invalidation
- Route Handler caching
- Cache Components / use cache

또 최신 docs 기준으로 fetch 기본을 단순히 “무조건 cached”라고 말하면 위험하다. 공식 API 문서는 default를 auto no cache로 설명하고, 동시에 동적 API가 없으면 route는 정적으로 prerender될 수 있다고 설명한다. 이 미묘한 차이를 이해하고 있으면 답변이 확 달라진다.

## 19. 배포 후 장애가 났을 때 Next.js 애플리케이션 기준으로 가장 먼저 확인할 포인트는 무엇인가요?

### 합격 답변 예시

Next.js 관점에서 가장 먼저 보는 건 어디 계층에서 깨졌는지 분리하는 것입니다. build/deploy artifact 문제인지, 2) runtime 예외인지, 3) 캐시/재검증 문제인지, 4) 프론트 세그먼트 단위 오류인지부터 나눕니다. 공식 docs에는 production observability용 instrumentation.js, segment 단위 fallback인 error.js, 그리고 ISR/cache 디버깅용 NEXT_PRIVATE_DEBUG_CACHE 같은 포인트가 각각 존재합니다. 실무적으로는 저는 보통 이 순서로 봅니다. 빌드 산출물/배포 구성(output/tracing) 문제, 서버 로그와 instrumentation, 특정 route segment의 error.js fallback 여부, 캐시 미스/재검증 이상 여부, 마지막으로 Proxy/Route Handler/외부 API 장애 전파 여부를 확인합니다. 공식 docs도 Output File Tracing과 standalone 배포, instrumentation, error handling, ISR cache 디버깅을 각각 안내합니다.

### 상세 해설

좋은 답변은 “로그부터 보겠습니다”에서 끝나지 않는다. Next.js에선 특히 아래를 나눠 보는 게 중요하다.

- build artifact / module trace
- App Router route segment error
- fetch/cache/revalidate 이상
- Proxy 오작동
- Route Handler / upstream API 장애
- 즉 면접에서는
- “장애를 Next.js 구조 기준으로 계층화해서 본다”

이 말이 나오면 좋다.

## 20. 네가 했던 프로젝트 중 하나를 골라, 왜 Next.js가 적절한 선택이었는지 설명해보세요.

### 합격 답변 예시

이 문항은 실제로는 네 프로젝트로 답하는 게 가장 좋다. 다만 구조 예시를 들면, 공개 콘텐츠 + 로그인 대시보드 + 검색/필터 + 이미지/OG 공유 + 간단한 BFF가 같이 있는 서비스라면 Next.js가 매우 적절합니다. 공개 영역은 metadata와 prerendering으로 SEO를 챙기고, 대시보드는 Server Components와 Client Components를 섞어 인터랙션과 데이터 fetch를 분리할 수 있고, 필요한 API는 Route Handlers로 붙일 수 있기 때문입니다. 면접에서는 이렇게 말하면 강합니다. “그 프로젝트는 단순 SPA가 아니라 검색 유입이 있는 공개 화면과 로그인 후 인터랙션이 강한 화면이 공존했고, App Router의 nested layouts, metadata, server/client 분리, route handlers, caching/revalidation이 요구사항과 잘 맞았기 때문에 Next.js를 선택했다.” 이런 식으로 프레임워크 기능을 제품 요구사항과 1:1로 연결하는 답변이 가장 좋습니다.

### 상세 해설

이 질문은 정답이 없다. 대신 왜 React만으로는 부족했고, 왜 Next.js의 특정 기능이 실제 요구사항과 맞았는지를 말해야 한다.

- 좋은 예시 포인트:
- 공개 페이지 SEO 필요 → Metadata / prerendering
- 대시보드 공유 레이아웃 → nested layouts
- 인터랙션은 많지만 전부 CSR로 가면 부담 → server/client boundary
- 업로드/검색 API 필요 → Route Handlers
- 갱신 후 목록 바로 반영 → revalidation

이 구조로 답하면 면접관이 듣고 싶어 하는 “기술 선택의 이유”가 살아난다.

## 보완내용

### 왜 보완이 필요한가

기존 20문항은 App Router, Server Components, Server Actions, BFF, 캐싱, Proxy까지 들어가 있어 상당히 강하다. 다만 최신 Next.js는 기본 동작을 어설프게 암기하면 오답이 나오기 쉬운 프레임워크라서, “실제로 써본 사람만 아는 함정”을 몇 개 더 넣는 편이 좋다.

특히 App Router의 캐싱 기본값, stale `searchParams`, Server Action 보안, Suspense/streaming 이해도는 문서만 대충 읽은 사람과 실제 운영 경험이 있는 사람을 잘 가른다.

### 추가하면 좋은 질문

#### 1. App Router에서 `fetch`의 기본 동작을 설명해보세요. “무조건 캐시된다”고 보면 왜 위험한가요?

이 질문은 최신 Next.js에서 매우 중요하다. route prerender와 Data Cache를 함께 이해하고 있는지, 그리고 동적 API 사용 여부가 동작에 어떤 영향을 주는지 설명할 수 있어야 한다.

##### 합격 답변 예시

최신 Next.js 공식 문서 기준으로 서버 fetch의 기본 동작은 “무조건 캐시”가 아닙니다. fetch의 기본 cache 옵션은 `auto`이며, 동적 API(cookies, headers, searchParams 등)가 사용되면 해당 라우트는 동적으로 전환되어 fetch도 매 요청마다 실행됩니다. 반면 동적 API가 없고 라우트가 정적으로 prerender 가능하면 빌드 시점에 fetch가 실행되고 결과가 Data Cache에 저장됩니다. “무조건 캐시된다”고 가정하면 위험한 이유는, 동적 API 하나만 추가해도 라우트 전체가 동적으로 전환되면서 캐시 동작이 완전히 바뀌기 때문입니다. 그래서 fetch 캐싱을 제어하려면 `cache: 'force-cache'`, `cache: 'no-store'`, `next: { revalidate: N }`, `next: { tags: [...] }` 같은 옵션을 의도에 맞게 명시하는 것이 안전합니다.

##### 상세 해설

핵심은 fetch 캐싱이 단독으로 결정되는 게 아니라 라우트의 정적/동적 판정과 맞물린다는 점이다.

```typescript
// 정적 라우트 — 빌드 시 fetch가 실행되고 결과가 캐시됨
export default async function Page() {
  const data = await fetch('https://api.example.com/posts');
  const posts = await data.json();
  return <PostList posts={posts} />;
}

// 동적 라우트 — cookies() 사용으로 매 요청마다 fetch 실행
import { cookies } from 'next/headers';

export default async function Dashboard() {
  const token = (await cookies()).get('session')?.value;
  const data = await fetch('https://api.example.com/me', {
    headers: { Authorization: `Bearer ${token}` },
  });
  const user = await data.json();
  return <Profile user={user} />;
}

// 명시적 revalidate — 60초마다 백그라운드 갱신
export default async function Catalog() {
  const data = await fetch('https://api.example.com/products', {
    next: { revalidate: 60 },
  });
  const products = await data.json();
  return <ProductGrid products={products} />;
}
```

면접에서 “fetch는 기본 캐시됩니다”라고 단정하면 바로 약해진다. “라우트가 정적이냐 동적이냐에 따라 달라지고, 명시적으로 제어하는 게 안전하다”까지 말해야 강하다.

#### 2. `searchParams`를 layout에서 직접 다루면 왜 안 좋은가요?

App Router를 실제로 써본 사람은 layout이 navigation마다 다시 렌더되지 않을 수 있다는 점을 안다. stale query 문제를 설명할 수 있으면 세그먼트 기반 모델을 제대로 이해하고 있는 것이다.

##### 합격 답변 예시

App Router에서 layout은 네비게이션 시 상태를 유지하며 다시 렌더되지 않도록 설계되어 있습니다. 공식 문서도 layout은 re-render되지 않는다고 설명합니다. 그래서 layout에서 searchParams를 직접 읽으면 URL이 바뀌어도 layout이 다시 실행되지 않아 stale한 값을 보게 됩니다. 올바른 방법은 두 가지입니다. 첫째, Page 컴포넌트의 searchParams prop으로 읽는 것입니다. Page는 URL이 바뀔 때마다 다시 렌더됩니다. 둘째, Client Component에서 useSearchParams() 훅을 사용하는 것입니다. 이 훅은 클라이언트에서 URL 변화를 감지합니다. layout에서 검색 조건이나 필터 상태를 다루고 싶다면, layout 자체가 아니라 layout 안에 배치한 Client Component에서 useSearchParams를 사용해야 합니다.

##### 상세 해설

```typescript
// 잘못된 패턴 — layout에서 searchParams 직접 사용
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
  searchParams, // layout에서는 stale해질 수 있음
}: {
  children: React.ReactNode;
  searchParams: { query?: string };
}) {
  // URL이 바뀌어도 layout은 re-render되지 않아 query가 갱신 안 됨
  return (
    <div>
      <Sidebar currentQuery={searchParams.query} />
      {children}
    </div>
  );
}

// 올바른 패턴 1 — Page에서 searchParams 읽기
// app/dashboard/page.tsx
export default async function DashboardPage({
  searchParams,
}: {
  searchParams: Promise<{ query?: string }>;
}) {
  const { query } = await searchParams;
  const data = await fetchResults(query);
  return <ResultList data={data} />;
}

// 올바른 패턴 2 — Client Component에서 useSearchParams 사용
// app/dashboard/search-filter.tsx
'use client';

import { useSearchParams } from 'next/navigation';

export function SearchFilter() {
  const searchParams = useSearchParams();
  const query = searchParams.get('query') ?? '';
  // URL이 바뀔 때마다 최신 값을 반영
  return <input defaultValue={query} />;
}
```

layout의 “상태 유지” 특성은 장점이자 함정이다. 세그먼트 기반 모델을 이해하면 “layout은 껍데기, page는 내용, query는 page나 client에서”라는 원칙이 자연스럽게 나온다.

#### 3. Server Action을 내부 함수처럼 생각하면 왜 위험한가요?

Server Action은 편하게 보이지만 보안 관점에선 공개 엔드포인트처럼 다뤄야 한다. 인증/인가를 빠뜨리기 쉬운 지점이라 실무 감각을 보기 좋다.

##### 합격 답변 예시

Server Action은 `'use server'`로 선언된 서버 함수이지만, export되는 순간 공개 HTTP 엔드포인트와 동일하게 외부에서 호출 가능합니다. 공식 보안 문서도 exported Server Action은 public HTTP endpoint처럼 취급해야 하며 반드시 인증/인가 체크를 넣어야 한다고 명시합니다. 코드상으로는 일반 함수 호출처럼 보이기 때문에 인증 검사를 빠뜨리기 쉽고, 입력값 검증도 소홀해지기 쉽습니다. 그래서 Server Action 내부에서도 반드시 세션 확인, 권한 검사, 입력값 검증을 수행해야 하며, 민감한 데이터를 closure로 캡처하지 않도록 주의해야 합니다. 면접에서는 “Server Action은 편리하지만 보안 모델은 API Route와 동일하게 적용해야 한다”고 말하면 강합니다.

##### 상세 해설

```typescript
// 위험한 패턴 — 인증 없이 Server Action 노출
'use server';

export async function deletePost(postId: string) {
  // 인증/인가 체크 없이 바로 삭제 — 누구나 호출 가능!
  await db.post.delete({ where: { id: postId } });
}

// 안전한 패턴 — 인증 + 인가 + 입력 검증
'use server';

import { cookies } from 'next/headers';
import { verifySession } from '@/lib/auth';
import { z } from 'zod';

const DeletePostSchema = z.object({
  postId: z.string().uuid(),
});

export async function deletePost(rawInput: { postId: string }) {
  // 1. 인증 확인
  const session = await verifySession(
    (await cookies()).get('session')?.value
  );
  if (!session) throw new Error('Unauthorized');

  // 2. 입력값 검증
  const { postId } = DeletePostSchema.parse(rawInput);

  // 3. 인가 확인 — 본인 글인지
  const post = await db.post.findUnique({ where: { id: postId } });
  if (post?.authorId !== session.userId) {
    throw new Error('Forbidden');
  }

  // 4. 실행
  await db.post.delete({ where: { id: postId } });
  revalidatePath('/posts');
}
```

Server Action을 “서버에서 돌아가니까 안전하다”고 생각하는 순간 보안 구멍이 생긴다. 코드가 서버에서 실행되는 것과 접근이 제한되는 것은 완전히 다른 문제다.

#### 4. Proxy(이전 Middleware)에 어떤 로직까지 넣고, 어떤 로직은 절대 넣지 않나요?

가벼운 optimistic auth check, redirect, header 조작 정도는 적절하지만, 무거운 DB 조회나 핵심 권한 판정까지 넣으면 성능과 운영 복잡도가 바로 커진다. 이 경계를 설명할 수 있어야 한다.

##### 합격 답변 예시

Proxy에 넣어도 되는 로직은 빠르고 가벼운 request-level 분기입니다. 쿠키 존재 여부만 확인하는 optimistic auth check, locale/region 기반 redirect나 rewrite, A/B 테스트 라우팅, 응답 헤더 추가 같은 작업이 적합합니다. 반면 절대 넣지 않아야 하는 것은 무거운 DB 조회, 외부 API 호출 대기, 핵심 권한 판정의 전부, 비즈니스 로직 실행입니다. 공식 문서도 Proxy는 모든 라우트(prefetch 포함)에서 실행되므로 여기서 느린 작업을 하면 전체 응답 시간에 직접 영향을 준다고 설명합니다. 핵심 원칙은 “Proxy는 문지기가 아니라 교통 정리”입니다. 진짜 보안 검증은 데이터 접근 계층에서 해야 합니다.

##### 상세 해설

```typescript
// proxy.ts (Next.js 16+)
import { NextRequest, NextResponse } from 'next/server';

export function middleware(request: NextRequest) {
  // 좋은 예: 쿠키 기반 빠른 리다이렉트
  const session = request.cookies.get('session');
  if (!session && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(
      new URL(`/login?callbackUrl=${request.nextUrl.pathname}`, request.url)
    );
  }

  // 좋은 예: locale 분기
  const locale = request.headers.get('accept-language')?.split(',')[0] ?? 'ko';
  if (request.nextUrl.pathname === '/') {
    return NextResponse.rewrite(new URL(`/${locale}/home`, request.url));
  }

  // 나쁜 예 (절대 하지 말 것):
  // await db.user.findUnique(...)       // DB 조회
  // await fetch('https://auth.api/...')  // 외부 API 대기
  // 복잡한 권한 트리 계산               // 비즈니스 로직

  return NextResponse.next();
}

export const config = {
  matcher: ['/dashboard/:path*', '/'],
};
```

면접에서는 “Proxy에서 쿠키 읽기까지는 OK, DB 질의부터는 NO”라는 경계를 명확히 그으면 된다. 그리고 “진짜 auth는 Server Component나 data access layer에서 다시 검증한다”를 연결하면 완벽하다.

#### 5. Pages Router에서 App Router로 옮길 때 가장 먼저 바뀌는 mental model이 무엇인가요?

이 질문은 단순 폴더 이동이 아니라, 페이지 중심 사고에서 세그먼트/layout 중심 사고와 서버-클라이언트 경계 설계로 바뀐다는 점을 설명하게 만든다.

##### 합격 답변 예시

가장 먼저 바뀌는 mental model은 “페이지 단위 사고”에서 “세그먼트/레이아웃 단위 사고 + 서버-클라이언트 경계 설계”로의 전환입니다. Pages Router에서는 각 페이지가 독립적인 단위이고, 데이터 페칭도 getServerSideProps/getStaticProps라는 페이지 레벨 함수에 집중됩니다. App Router에서는 URL의 각 세그먼트가 layout, loading, error, page라는 special files로 구성되고, 데이터 페칭은 컴포넌트 레벨에서 직접 async/await로 수행합니다. 또한 “모든 컴포넌트가 클라이언트”라는 전제가 “기본은 서버, 필요한 곳만 클라이언트”로 뒤집힙니다. 이 전환을 이해하지 못하면 App Router에서도 Pages Router처럼 코드를 짜게 되고, 서버 컴포넌트의 장점을 전혀 활용하지 못하게 됩니다.

##### 상세 해설

```
Pages Router mental model:
pages/
  _app.tsx          ← 전역 wrapper (하나)
  _document.tsx     ← HTML shell (하나)
  index.tsx         ← 페이지 + getServerSideProps/getStaticProps
  posts/[id].tsx    ← 페이지 + getStaticPaths + getStaticProps
  api/hello.ts      ← API Route

→ 모든 컴포넌트가 클라이언트, 데이터는 페이지 함수에서

App Router mental model:
app/
  layout.tsx        ← 루트 레이아웃 (중첩 가능)
  page.tsx          ← 세그먼트별 페이지
  loading.tsx       ← 세그먼트별 로딩 UI
  error.tsx         ← 세그먼트별 에러 UI
  posts/
    [id]/
      layout.tsx    ← 중첩 레이아웃
      page.tsx      ← Server Component (기본)
  api/hello/
    route.ts        ← Route Handler

→ 기본은 서버 컴포넌트, 인터랙션 필요한 곳만 'use client'
→ 데이터 페칭은 컴포넌트에서 직접 fetch
→ UI 단위가 페이지가 아니라 세그먼트
```

정리하면 세 가지 전환이다.

- 페이지 단위 → 세그먼트/레이아웃 단위
- 전부 클라이언트 → 기본 서버, 필요 시 클라이언트
- 페이지 레벨 데이터 함수 → 컴포넌트 레벨 async fetch

이 세 가지를 말할 수 있으면 “실제로 마이그레이션 해봤다”는 신뢰를 준다.

#### 6. Suspense와 streaming을 Next.js App Router에서 어떻게 이해하고 있나요?

서버 컴포넌트만 말하는 수준을 넘어서, 응답 흐름을 쪼개고 준비된 부분부터 먼저 보내는 모델까지 설명할 수 있으면 중상급 분별력이 높아진다.

##### 합격 답변 예시

App Router에서 Suspense와 streaming은 “서버 응답을 한 번에 보내지 않고, 준비된 부분부터 점진적으로 보내는 모델”입니다. 전통적인 SSR은 모든 데이터를 다 가져온 뒤 전체 HTML을 한 번에 보내지만, streaming SSR은 React의 Suspense 경계를 기준으로 준비된 shell을 먼저 보내고, 느린 부분은 나중에 채워 넣습니다. App Router에서는 loading.tsx가 route segment 단위의 자동 Suspense boundary 역할을 하고, 직접 Suspense 컴포넌트를 배치해서 더 세밀하게 streaming 경계를 제어할 수도 있습니다. 공식 문서도 streaming을 통해 서버에서 UI를 점진적으로 렌더링하고, 준비된 조각부터 클라이언트에 보낼 수 있다고 설명합니다. 실무에서는 페이지의 핵심 레이아웃은 즉시 보여주고, 느린 데이터 영역만 fallback을 보여준 뒤 완료되면 교체하는 패턴이 기본입니다.

##### 상세 해설

```tsx
// app/dashboard/page.tsx
import { Suspense } from 'react';
import { DashboardShell } from '@/components/dashboard-shell';
import { RecentOrders } from '@/components/recent-orders';
import { Analytics } from '@/components/analytics';
import { Recommendations } from '@/components/recommendations';

export default function DashboardPage() {
  return (
    <DashboardShell>
      {/* 빠른 영역 — shell과 함께 즉시 전송 */}
      <h1>대시보드</h1>

      {/* 느린 영역 1 — 주문 데이터 로딩 중 skeleton 표시 */}
      <Suspense fallback={<OrdersSkeleton />}>
        <RecentOrders />
      </Suspense>

      {/* 느린 영역 2 — 분석 데이터 로딩 중 skeleton 표시 */}
      <Suspense fallback={<AnalyticsSkeleton />}>
        <Analytics />
      </Suspense>

      {/* 가장 느린 영역 — ML 추천, 마지막에 도착 */}
      <Suspense fallback={<RecommendationsSkeleton />}>
        <Recommendations />
      </Suspense>
    </DashboardShell>
  );
}

// components/recent-orders.tsx — Server Component
export async function RecentOrders() {
  // 이 fetch가 완료되면 해당 Suspense 경계가 resolve됨
  const orders = await fetch('https://api.example.com/orders', {
    next: { revalidate: 30 },
  }).then((res) => res.json());

  return (
    <ul>
      {orders.map((order: { id: string; name: string }) => (
        <li key={order.id}>{order.name}</li>
      ))}
    </ul>
  );
}
```

streaming 흐름을 정리하면 이렇다.

- 서버가 요청을 받으면 layout + page shell을 먼저 HTML로 보낸다
- 각 Suspense boundary 안의 서버 컴포넌트가 데이터를 가져오는 동안 fallback을 보여준다
- 데이터가 준비되면 해당 조각의 HTML을 추가로 stream한다
- 클라이언트에서 React가 fallback을 실제 컴포넌트로 교체한다

loading.tsx는 route segment 전체에 대한 자동 Suspense boundary이고, 직접 Suspense를 배치하면 한 페이지 안에서도 여러 독립적인 streaming 경계를 만들 수 있다. 이 차이를 설명할 수 있으면 “Suspense를 실제로 활용해봤다”는 신뢰를 준다.

### 이 섹션 보완 핵심

Next.js 섹션은 이미 강하다. 여기에 App Router의 최신 기본 동작, stale params, Server Action 보안, Proxy 경계, 마이그레이션 mental model, Suspense/streaming을 더하면 문서 요약형 답변과 실전 App Router 경험자를 더 잘 구분할 수 있다.
