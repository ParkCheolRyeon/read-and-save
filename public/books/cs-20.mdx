# 05. cs 20문항

> 프로세스, 메모리, 자료구조, 데이터베이스, 캐시까지 프론트엔드와 연결되는 CS 20문항입니다.

## 총평

이 섹션은 프론트엔드 면접에서도 생각보다 중요하다. 이유는 단순하다. 프론트엔드 개발자가 매일 다루는 문제들이 결국 CS 개념과 연결되기 때문이다.

- 이벤트 루프 → 동기/비동기, 블로킹/논블로킹
- React 상태 꼬임 → race condition
- 메모리 누수 → 힙, GC, 참조 유지
- 리스트 성능 → 자료구조, 복잡도
- API 캐싱 → 해시, 인덱싱, 시간/공간 trade-off

그래서 좋은 답변은 "교과서 정의"만 말하면 부족하다. **브라우저/자바스크립트/웹앱 문제와 연결해서 말해야 합격권**이다.

### 전체 흐름 맵

```
프로세스/스레드 (실행 단위)
  → 멀티프로세스/멀티스레드 (병렬 전략)
    → 동기/비동기 (흐름 제어)
      → 블로킹/논블로킹 (제어권)
        → 데드락 (동시성 위험)
          → Race Condition (순서 의존 버그)
            → 메모리 구조 (스택/힙)
              → GC (메모리 회수)
                → 배열/연결 리스트 (기본 자료구조)
                  → 해시 테이블 (빠른 조회)
                    → 스택/큐 (실행 모델)
                      → 트리/그래프 (구조적 사고)
                        → 복잡도 (성능 감각)
                          → 정렬 (알고리즘 기본)
                            → DB 인덱스 (저장소 최적화)
                              → 정규화 (데이터 설계)
                                → 트랜잭션 (데이터 안전성)
                                  → CAP (분산 시스템)
                                    → 캐시 (성능 전략)
                                      → CS가 왜 필요한가 (종합)
```

## 1. 프로세스와 스레드의 차이를 설명해보세요.

### 합격 답변 예시

프로세스는 실행 중인 프로그램의 독립된 단위이고, 각 프로세스는 자신만의 메모리 공간을 가집니다. 스레드는 프로세스 내부에서 실행되는 작업 흐름 단위로, 같은 프로세스 안의 코드 영역, 힙, 데이터 영역을 공유하고 스택만 개별적으로 가집니다. 그래서 프로세스는 서로 격리돼 안정성이 높고, 스레드는 자원 공유가 쉬워서 효율적이지만 동기화 문제가 생기기 쉽습니다.

### 가산점 포인트

**Chrome 브라우저가 탭마다 별도 프로세스를 쓰는 이유**를 말하면 좋다. 한 탭이 크래시해도 다른 탭에 영향이 없도록 프로세스 격리를 선택한 것이다. 대신 메모리 사용량이 커지는 trade-off가 있다. 이것이 프로세스 격리의 실제 사례다.

### 상세 해설

#### 비유

프로세스는 **각각 독립 사무실**이고, 스레드는 **같은 사무실 안의 팀원**이라고 보면 된다. 사무실이 다르면 사고가 나도 번지기 어렵고, 같은 사무실이면 협업은 빠르지만 책상 위 자료를 같이 건드리다 꼬일 수 있다.

#### 왜 이 구분이 중요한가

```
프로세스 (독립 메모리):
  ┌─────────┐  ┌─────────┐
  │ 프로세스 A │  │ 프로세스 B │
  │ 코드      │  │ 코드      │
  │ 데이터    │  │ 데이터    │
  │ 힙       │  │ 힙       │
  │ 스택      │  │ 스택      │
  └─────────┘  └─────────┘
  A가 죽어도 B는 영향 없음

스레드 (메모리 공유):
  ┌─────────────────────┐
  │ 프로세스               │
  │ 코드 (공유)            │
  │ 데이터 (공유)          │
  │ 힙 (공유)             │
  │ ┌──────┐ ┌──────┐   │
  │ │스택 A │ │스택 B │   │
  │ └──────┘ └──────┘   │
  └─────────────────────┘
  A와 B가 힙을 동시에 수정하면 충돌
```

#### 프론트엔드 연결

```ts
// Web Worker = 별도 스레드에서 실행 (메인 스레드와 메모리 공유 안 함)
const worker = new Worker('/heavy-calc.js');
worker.postMessage({ data: largeArray });  // 메시지로 통신
worker.onmessage = (e) => {
  console.log('결과:', e.data);  // 결과를 메시지로 받음
};
// 메인 스레드는 막히지 않으므로 UI가 버벅이지 않는다
```

#### 자주 하는 오해

- "JavaScript는 싱글 스레드이므로 브라우저도 싱글 스레드다" → **틀림.** JS 실행은 단일 스레드지만, 브라우저는 렌더링, 네트워크, GPU 등 여러 스레드가 함께 동작한다.
- "프로세스가 무조건 스레드보다 좋다" → **상황에 따라 다름.** 격리가 필요하면 프로세스, 효율이 필요하면 스레드.

> 프로세스와 스레드의 차이를 알았으니, 다음에는 **이 두 가지를 어떻게 조합하는지** — 멀티프로세스 vs 멀티스레드를 봐야 한다.

---

## 2. 멀티프로세스와 멀티스레드의 장단점을 설명해보세요.

### 합격 답변 예시

멀티프로세스는 프로세스마다 메모리가 분리되어 있어서 안정성과 격리가 좋고, 하나가 죽어도 전체에 영향이 덜합니다. 대신 프로세스 생성 비용이 크고, 프로세스 간 통신도 상대적으로 무겁습니다. 멀티스레드는 같은 프로세스의 자원을 공유하므로 생성과 통신이 더 가볍고 효율적이지만, race condition, deadlock 같은 동기화 문제가 더 쉽게 발생합니다. 즉 멀티프로세스는 안정성, 멀티스레드는 효율성 쪽에 강점이 있습니다.

### 가산점 포인트

**Chrome이 멀티프로세스, Node.js가 싱글 스레드(+ Worker Threads)**인 이유를 말하면 좋다. Chrome은 탭 격리가 중요하므로 멀티프로세스, Node.js는 I/O 중심이므로 이벤트 루프 기반 싱글 스레드가 효율적이다.

### 상세 해설

| | 멀티프로세스 | 멀티스레드 |
|---|---|---|
| **메모리** | 독립 (격리) | 공유 (효율) |
| **안정성** | 하나 죽어도 다른 것 영향 적음 | 하나 죽으면 전체 프로세스 위험 |
| **통신** | IPC (무거움) | 공유 메모리 (가벼움) |
| **생성 비용** | 큼 | 작음 |
| **동기화** | 덜 복잡 | race condition, deadlock 위험 |

#### 실무 연결

Node.js는 JS 실행 자체는 단일 스레드처럼 보이지만, 실제로는 libuv, worker threads, OS 수준 비동기 I/O 같은 구조와 맞물린다. 프론트엔드 개발자도 **"왜 어떤 작업은 메인 스레드를 막는가"**를 이해하려면 이 감각이 필요하다.

```ts
// Node.js Worker Threads 예시
import { Worker, isMainThread, parentPort } from 'worker_threads';

if (isMainThread) {
  const worker = new Worker(__filename);
  worker.on('message', (result) => console.log('결과:', result));
  worker.postMessage({ data: [3, 1, 4, 1, 5] });
} else {
  parentPort?.on('message', (msg) => {
    const sorted = msg.data.sort((a: number, b: number) => a - b);
    parentPort?.postMessage(sorted);
  });
}
```

#### 자주 하는 오해

- "멀티스레드가 항상 멀티프로세스보다 빠르다" → **아님.** 동기화 오버헤드가 크면 오히려 느릴 수 있다.

> 멀티프로세스/멀티스레드를 알았으니, 다음에는 **작업의 흐름 제어 방식** — 동기와 비동기를 봐야 한다.

---

## 3. 동기와 비동기의 차이를 설명해보세요.

### 합격 답변 예시

동기는 요청한 작업의 완료와 다음 작업 진행이 강하게 연결된 방식이고, 비동기는 작업을 요청한 뒤 완료를 기다리는 동안 다른 작업을 진행할 수 있는 방식입니다. 중요한 건 동기와 비동기는 속도의 문제가 아니라 **흐름 제어 방식의 차이**라는 점입니다.

### 가산점 포인트

**"동기 = 느림, 비동기 = 빠름"은 틀린 구분**이다. 동기/비동기는 "완료를 기다리는가 아닌가"이지, 속도와는 다른 축의 개념이다. 비동기여도 네트워크가 느리면 느리고, 동기여도 계산이 빠르면 빠르다.

### 상세 해설

#### 왜 "속도"가 아닌 "흐름 제어"인가

```ts
// 동기: 결과가 나올 때까지 다음 줄로 못 감
const data = readFileSync('/large-file.json');  // 여기서 멈춤
console.log(data);  // 파일 읽기 끝나야 실행

// 비동기: 요청 후 바로 다음으로 넘어감
readFile('/large-file.json', (err, data) => {
  console.log(data);  // 나중에 실행
});
console.log('다음 작업');  // 즉시 실행
```

```
동기 실행 타임라인:
  ─────[파일 읽기 1초]─────[console.log]─────
  → 파일 읽는 동안 아무것도 못 함

비동기 실행 타임라인:
  ─────[파일 읽기 요청]─[다음 작업 즉시]─────
                    └──[파일 완료 시 콜백]──
  → 읽는 동안 다른 작업 가능
```

#### 프론트엔드에서 비동기가 필수인 이유

브라우저에서 메인 스레드가 동기 작업으로 막히면 **UI가 멈춘다**. 사용자가 버튼을 눌러도 반응이 없고, 스크롤도 안 된다. 그래서 네트워크 요청, 타이머, 파일 읽기 등은 반드시 비동기로 처리해야 한다.

```ts
// 겉으로는 동기처럼 읽혀도 실제론 비동기
const response = await fetch('/api/users');
// await는 "이 함수를 일시 중단하고 다른 작업에 양보"하는 것이지,
// 메인 스레드를 블로킹하는 것이 아니다.
```

#### 자주 하는 오해

- "`await`가 있으면 동기 코드다" → **틀림.** `await`는 비동기 작업을 동기처럼 읽히게 해주는 문법이지, 실행 자체가 동기가 되는 건 아니다.

> 동기/비동기를 알았으니, 다음에는 **제어권 관점** — 블로킹과 논블로킹을 봐야 한다.

---

## 4. 블로킹과 논블로킹의 차이를 설명해보세요.

### 합격 답변 예시

블로킹은 어떤 작업을 요청한 쪽이 그 작업이 끝날 때까지 제어권을 돌려받지 못하는 방식이고, 논블로킹은 요청 후 바로 제어권을 돌려받아 다른 작업을 계속할 수 있는 방식입니다. 동기/비동기와 비슷해 보이지만 다른 축의 개념입니다.

### 가산점 포인트

**동기/비동기는 "완료 알림 방식"이고, 블로킹/논블로킹은 "제어권 반환 시점"**이라고 구분하면 좋다. 두 축은 독립적이라 조합이 가능하다: 동기+블로킹, 동기+논블로킹, 비동기+논블로킹 등.

### 상세 해설

#### 두 축의 차이

```
동기/비동기: "완료를 누가 확인하느냐"
  동기: 요청자가 직접 확인 (끝날 때까지 기다림)
  비동기: 완료되면 콜백/이벤트로 알려줌

블로킹/논블로킹: "제어권을 바로 돌려주느냐"
  블로킹: 끝날 때까지 제어권을 안 돌려줌
  논블로킹: 일단 제어권을 바로 돌려줌
```

#### 프론트엔드에서의 블로킹

```ts
// 블로킹: 메인 스레드를 점유하는 무거운 계산
function heavyCalc() {
  let sum = 0;
  for (let i = 0; i < 1_000_000_000; i++) sum += i;
  return sum;
}
heavyCalc();  // 이 동안 UI 완전 멈춤 (블로킹)

// 논블로킹: Web Worker로 분리
const worker = new Worker('/calc-worker.js');
worker.postMessage({ iterations: 1_000_000_000 });
// 메인 스레드는 즉시 반환 (논블로킹) → UI 계속 동작
```

#### 자주 하는 오해

- "비동기 = 논블로킹이다" → **항상 같지는 않다.** 개념적으로 구분되는 축이다. 다만 실무에서 비동기 API는 대부분 논블로킹이기도 하다.

> 블로킹/논블로킹을 알았으니, 다음에는 **동시성 환경의 위험** — 데드락을 봐야 한다.

---

## 5. 데드락의 발생 조건 4가지를 설명해보세요.

### 합격 답변 예시

데드락의 고전적인 발생 조건은 상호 배제, 점유 대기, 비선점, 순환 대기입니다. 즉 어떤 자원을 한 번에 하나만 사용할 수 있고, 이미 자원을 가진 상태로 다른 자원을 기다리고, 자원을 강제로 빼앗을 수 없고, 기다림이 원형으로 이어지면 데드락이 발생할 수 있습니다.

### 가산점 포인트

조건을 말한 뒤 **"그래서 실무에서는 자원 획득 순서를 강제하거나 timeout을 두어 회피할 수 있습니다"**까지 말하면 좋다. 4조건 중 하나라도 깨면 데드락을 방지할 수 있다는 원리다.

### 상세 해설

#### 4조건 시각화

```
A가 리소스 1을 점유 + 리소스 2를 기다림
B가 리소스 2를 점유 + 리소스 1을 기다림
→ A와 B 둘 다 영원히 대기 = 데드락

  A ──[리소스 1]──→ [리소스 2]
                        ↑
  B ──[리소스 2]──→ [리소스 1]
       ↑─────────────────┘  (순환)
```

#### 프론트엔드에서 유사 패턴

```ts
// 완전한 데드락은 아니지만, 비동기 순환 대기와 유사한 패턴
// useEffect A가 state B를 기다리고,
// useEffect B가 state A를 기다리는 무한 루프
useEffect(() => {
  if (!stateB) setStateA(true);  // B가 없으면 A 설정
}, [stateB]);

useEffect(() => {
  if (!stateA) setStateB(true);  // A가 없으면 B 설정
}, [stateA]);
// → 무한 리렌더링 (데드락 유사 상황)
```

#### 자주 하는 오해

- "JavaScript는 싱글 스레드라 데드락이 없다" → **전통적 데드락은 없지만, 비동기 순환 대기나 Promise 체인 꼬임은 가능.** 개념을 이해하면 이런 패턴을 진단할 수 있다.

> 데드락을 알았으니, 다음에는 **프론트엔드에서 더 자주 발생하는 동시성 문제** — Race Condition을 봐야 한다.

---

## 6. Race Condition이 무엇인지, 프론트엔드에서도 비슷한 문제가 생길 수 있는지 설명해보세요.

### 합격 답변 예시

Race Condition은 여러 작업이 같은 상태나 자원에 접근할 때, 실행 순서에 따라 결과가 달라지는 문제입니다. 프론트엔드에서도 자주 발생합니다. 대표적으로 검색 요청을 연속으로 보냈는데, 더 늦게 보낸 최신 요청보다 이전 요청 응답이 나중에 도착해서 화면을 덮어쓰는 경우가 있습니다.

### 가산점 포인트

**AbortController**로 이전 요청을 취소하는 패턴을 알면 좋다. 또는 React Query처럼 요청 식별자를 비교해 최신 응답만 반영하는 방식도 있다.

### 상세 해설

#### 프론트엔드 Race Condition 시각화

```
사용자 입력: "a"  → 요청 A 발생 (200ms 걸림)
사용자 입력: "ab" → 요청 B 발생 (50ms 걸림)

시간 순서:
  t=0   요청 A 전송
  t=10  요청 B 전송
  t=60  요청 B 응답 도착 → 화면에 "ab" 결과 표시 ✅
  t=200 요청 A 응답 도착 → 화면에 "a" 결과로 덮어씀 ❌
```

#### 해결 방법

```ts
// AbortController로 이전 요청 취소
let controller: AbortController | null = null;

async function search(query: string) {
  // 이전 요청이 있으면 취소
  controller?.abort();
  controller = new AbortController();

  try {
    const res = await fetch(`/api/search?q=${query}`, {
      signal: controller.signal,
    });
    const data = await res.json();
    setResults(data);
  } catch (err) {
    if (err instanceof DOMException && err.name === 'AbortError') {
      return; // 취소된 요청은 무시
    }
    throw err;
  }
}
```

#### 실무 에피소드

> 검색 자동완성에서 "가끔 옛날 결과가 보인다"는 버그가 있었다. 원인은 이전 요청의 응답이 늦게 도착해서 최신 결과를 덮어쓴 것. AbortController를 추가하니 이전 요청이 자동 취소되어 문제가 해결됐다.

#### 자주 하는 오해

- "debounce만 하면 race condition이 해결된다" → **아님.** debounce는 요청 빈도를 줄이지만, 이미 보낸 요청의 순서 꼬임은 막지 못한다. abort나 요청 ID 비교가 필요하다.

> Race Condition을 알았으니, 다음에는 **코드가 실행되는 메모리 구조** — 스택과 힙을 봐야 한다.

---

## 7. 메모리 영역(코드, 스택, 힙, 데이터)을 설명해보세요.

### 합격 답변 예시

코드 영역은 실행 명령어가 저장되는 곳이고, 데이터 영역은 전역 변수나 정적 데이터가 저장되는 영역입니다. 스택은 함수 호출 정보와 지역 변수처럼 실행 흐름과 밀접한 값이 쌓이는 공간이고, 힙은 런타임에 동적으로 할당되는 객체들이 저장되는 공간입니다.

### 가산점 포인트

**"JavaScript에서 원시값은 스택, 객체는 힙"**이라는 간단한 모델을 알면 좋다. 그래서 객체를 많이 만들면 힙이 커지고, GC가 바쁘다. 재귀가 깊으면 스택이 넘친다.

### 상세 해설

```
메모리 구조:
┌──────────────────────┐
│  코드 영역 (Text)      │ ← 실행할 명령어 (읽기 전용)
├──────────────────────┤
│  데이터 영역 (Data)    │ ← 전역 변수, static 데이터
├──────────────────────┤
│  힙 (Heap)            │ ← 동적 객체, 배열, 클로저
│  ↓ (위에서 아래로 증가)  │    → GC가 관리
│                       │
│  ↑ (아래에서 위로 증가)  │
│  스택 (Stack)          │ ← 함수 호출 프레임, 지역 변수
├──────────────────────┤
│                       │    → LIFO, 크기 제한 있음
└──────────────────────┘
```

#### JavaScript에서의 메모리

```ts
// 스택에 저장: 원시값
let count = 42;          // number → 스택
let name = 'hello';      // string(짧은) → 스택

// 힙에 저장: 객체
let user = { id: 1 };    // object → 힙 (변수 user는 힙 주소를 참조)
let items = [1, 2, 3];   // array → 힙

// 스택 오버플로우: 재귀가 너무 깊을 때
function infinite() { infinite(); }
infinite();  // RangeError: Maximum call stack size exceeded

// 힙 증가: 객체를 계속 만들고 참조를 유지할 때
const cache = new Map();
for (let i = 0; i < 1_000_000; i++) {
  cache.set(i, { data: new Array(1000) });  // 힙 계속 증가
}
```

#### 자주 하는 오해

- "JavaScript는 메모리를 신경 안 써도 된다" → **GC가 있지만, 참조가 남아있으면 회수 못 한다.** 이벤트 리스너, 클로저, 전역 캐시에서 메모리 누수가 흔하다.

> 메모리 구조를 알았으니, 다음에는 **힙에서 메모리를 회수하는 메커니즘** — GC를 봐야 한다.

## 8. 가비지 컬렉션이 무엇인지, JavaScript에서 어떻게 메모리를 회수하는지 설명해보세요.

### 합격 답변 예시

가비지 컬렉션은 프로그램에서 더 이상 사용할 수 없는 메모리를 자동으로 회수하는 메커니즘입니다. JavaScript 엔진은 보통 루트 객체에서 **도달 가능한지(reachability)**를 기준으로 객체의 생존 여부를 판단합니다.

### 가산점 포인트

**"GC가 알아서 해준다"가 아니라 "참조가 남으면 못 치운다"**가 핵심이다. 프론트엔드에서 흔한 메모리 누수는 이벤트 리스너를 등록하고 해제하지 않는 경우, 클로저가 큰 객체를 계속 붙잡는 경우, DOM 참조가 남아있는 경우다.

### 상세 해설

#### 도달 가능성(Reachability) 원리

```
루트 (window, 전역 변수, 콜 스택)
  ↓ 참조
객체 A → 객체 B → 객체 C
                      ↗
              객체 D ─┘

루트에서 도달 가능: A, B, C, D → 유지
루트에서 도달 불가: → GC 대상 (회수)
```

#### JavaScript에서 흔한 메모리 누수

```ts
// ❌ 이벤트 리스너 누수: 등록만 하고 해제 안 함
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // cleanup이 없으면 컴포넌트 언마운트 후에도 리스너가 남음
}, []);

// ✅ 해제 포함
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);

// ❌ 캐시 누수: Map에 계속 추가만 하고 삭제 안 함
const cache = new Map();
function addToCache(key: string, value: object) {
  cache.set(key, value);  // 삭제 로직 없으면 무한 증가
}

// ✅ WeakMap 사용: 키 객체가 GC되면 자동으로 제거
const weakCache = new WeakMap();
```

#### DevTools로 메모리 누수 찾기

1. Chrome DevTools → Memory 탭
2. Heap Snapshot 찍기
3. 작업 수행 후 다시 Snapshot
4. 비교(Comparison)로 증가한 객체 확인
5. Retainer 경로를 보면 "왜 이 객체가 GC 안 되는지" 알 수 있음

#### 자주 하는 오해

- "GC가 있으니 메모리 관리를 안 해도 된다" → **참조만 끊어야 GC가 회수 가능.** 참조가 남아있으면 GC도 손 못 댄다.
- "WeakRef를 쓰면 모든 메모리 문제가 해결된다" → **용도가 제한적.** WeakRef/WeakMap은 캐시 같은 특수 상황에만 적합하다.

> GC를 알았으니, 다음에는 **데이터를 어떻게 저장하고 접근하는지** — 기본 자료구조를 봐야 한다.

---

## 9. 자료구조 중 배열과 연결 리스트의 차이를 설명해보세요.

### 합격 답변 예시

배열은 인덱스를 통한 임의 접근이 빠르고, 메모리상 연속 구간에 저장됩니다. 연결 리스트는 각 노드가 다음 노드를 가리키는 방식이라 중간 삽입·삭제는 유리할 수 있지만, 원하는 위치까지 순차적으로 따라가야 해서 임의 접근이 느립니다.

### 가산점 포인트

프론트엔드에서는 **대부분 배열로 충분**하다. 하지만 "배열을 매 렌더마다 전체 복사하는 비용" 같은 감각은 중요하다. React에서 불변 업데이트를 위해 `[...arr, newItem]`을 할 때, 배열 크기에 비례하는 복사 비용이 발생한다.

### 상세 해설

| 연산 | 배열 | 연결 리스트 |
|------|------|-----------|
| **인덱스 접근** | O(1) | O(n) |
| **앞에 삽입** | O(n) (밀어야 함) | O(1) |
| **중간 삽입** | O(n) | O(1) (위치 알 때) |
| **검색** | O(n) | O(n) |
| **캐시 친화성** | 높음 (연속 메모리) | 낮음 (포인터 따라감) |

```ts
// 배열: 인덱스 접근 빠름
const items = ['A', 'B', 'C', 'D'];
items[2];  // O(1) — 즉시 'C' 접근

// React에서 배열 불변 업데이트 비용
const newItems = [...items, 'E'];  // 전체 복사 O(n)
const filtered = items.filter(x => x !== 'B');  // 새 배열 생성 O(n)

// 10,000개 아이템 배열을 매 렌더마다 복사하면?
// → 렌더당 10,000번의 복사 → useMemo로 캐시하면 렌더 비용 절약
```

> 배열과 리스트를 알았으니, 다음에는 **빠른 조회를 위한 자료구조** — 해시 테이블을 봐야 한다.

## 10. 해시 테이블이 무엇이고 평균 시간복잡도가 왜 O(1)인지 설명해보세요.

### 합격 답변 예시

해시 테이블은 키를 해시 함수로 변환해 저장 위치를 찾는 자료구조입니다. 평균적으로 O(1)인 이유는 키를 거의 즉시 특정 버킷으로 매핑할 수 있기 때문입니다. 다만 충돌이 많아지면 성능이 나빠질 수 있으므로, 엄밀히는 **평균 O(1)**이라고 보는 게 맞습니다.

### 가산점 포인트

프론트엔드 실무에서 해시 테이블을 쓰는 대표적인 사례: **`id → entity` Map으로 lookup 최적화**. 배열에서 `.find()`로 매번 O(n) 검색하는 대신, Map으로 O(1) 접근이 가능하다.

### 상세 해설

```ts
// ❌ 배열에서 매번 find → O(n)
const users = [{ id: 1, name: '홍길동' }, { id: 2, name: '김영희' }, ...];
const user = users.find(u => u.id === targetId);  // 매번 전체 순회

// ✅ Map으로 lookup → O(1)
const userMap = new Map(users.map(u => [u.id, u]));
const user = userMap.get(targetId);  // 즉시 접근

// React에서 normalized state
// Redux/Zustand에서 엔티티를 id 기반 Map으로 저장
const state = {
  users: {
    byId: { 1: { id: 1, name: '홍길동' }, 2: { id: 2, name: '김영희' } },
    allIds: [1, 2],
  },
};
// state.users.byId[id] → O(1) 접근
```

#### 자주 하는 오해

- "JS 객체는 해시 테이블이다" → **엔진에 따라 다름.** V8은 내부적으로 히든 클래스(hidden class) 최적화를 사용한다. Map은 더 전통적인 해시 테이블 구현에 가깝다.

> 해시 테이블을 알았으니, 다음에는 **실행 모델과 직결되는 자료구조** — 스택과 큐를 봐야 한다.

---

## 11. 스택과 큐를 실제 웹 개발 상황과 연결해서 설명해보세요.

### 합격 답변 예시

스택은 후입선출(LIFO) 구조라 함수 호출 스택, 브라우저의 콜 스택, undo 히스토리와 연결됩니다. 큐는 선입선출(FIFO) 구조라 이벤트 큐, 작업 큐, 토스트 메시지 순차 처리와 연결됩니다. 프론트엔드에서는 특히 **이벤트 루프를 이해할 때 스택과 큐 개념**이 매우 중요합니다.

### 가산점 포인트

JavaScript 이벤트 루프는 **콜 스택(Stack) + 태스크 큐(Queue) + 마이크로태스크 큐**로 동작한다. 이 구조를 설명할 수 있으면 비동기 실행 순서를 정확히 예측할 수 있다.

### 상세 해설

```ts
// 이벤트 루프에서 스택과 큐
console.log('1');            // 콜 스택에서 즉시 실행

setTimeout(() => {
  console.log('2');          // 매크로태스크 큐에 추가
}, 0);

Promise.resolve().then(() => {
  console.log('3');          // 마이크로태스크 큐에 추가
});

console.log('4');            // 콜 스택에서 즉시 실행

// 출력 순서: 1, 4, 3, 2
// 이유:
// 1. 콜 스택: 1, 4 (동기 코드 먼저)
// 2. 마이크로태스크 큐: 3 (Promise)
// 3. 매크로태스크 큐: 2 (setTimeout)
```

```
이벤트 루프 구조:
  ┌──────────────┐
  │  콜 스택      │ ← 현재 실행 중인 함수 (LIFO)
  │  (Stack)      │
  └──────┬───────┘
         ↓ 비었으면
  ┌──────────────┐
  │ 마이크로태스크 │ ← Promise, queueMicrotask (FIFO)
  │  큐 (Queue)   │    → 콜 스택이 빌 때마다 전부 처리
  └──────┬───────┘
         ↓ 비었으면
  ┌──────────────┐
  │ 매크로태스크   │ ← setTimeout, setInterval, I/O (FIFO)
  │  큐 (Queue)   │    → 하나씩 처리
  └──────────────┘
```

> 스택과 큐를 알았으니, 다음에는 **더 복잡한 구조** — 트리와 그래프를 봐야 한다.

---

## 12. 트리와 그래프의 차이를 설명해보세요.

### 합격 답변 예시

트리는 그래프의 한 종류로, 계층 구조를 가지며 사이클이 없습니다. 그래프는 더 일반적인 개념이라 사이클도 있을 수 있고 부모-자식 관계가 꼭 필요하지 않습니다. DOM 구조나 React 컴포넌트 구조는 트리, 페이지 링크 구조나 의존성 구조는 그래프로 이해하기 좋습니다.

### 가산점 포인트

React Fiber 아키텍처도 **트리 구조**다. `child`, `sibling`, `return` 포인터로 트리를 연결하고, 이 트리를 순회하면서 렌더링을 수행한다. DOM 트리, CSSOM 트리, Render 트리도 모두 트리 구조다.

### 상세 해설

```
트리 (DOM):                      그래프 (페이지 링크):
    html                           홈 ↔ 소개
   /    \                          ↕     ↕
  head   body                     블로그 → 글 → 댓글
         /  \                          ↘       ↗
        nav  main                      검색 ──┘
  (사이클 없음, 부모→자식)          (사이클 가능, 양방향 가능)
```

#### 프론트엔드 실무 연결

| 트리 구조 | 그래프 구조 |
|----------|-----------|
| DOM tree | 페이지 내비게이션 구조 |
| React 컴포넌트 트리 | npm 의존성 그래프 |
| 파일 시스템 (폴더 구조) | Git 커밋 히스토리 |
| AST (Abstract Syntax Tree) | 모듈 의존성 그래프 |

> 트리/그래프를 알았으니, 다음에는 **자료구조의 효율성을 측정하는 도구** — 복잡도를 봐야 한다.

## 13. 시간복잡도와 공간복잡도를 설명하고, 실무에서 둘 중 무엇이 더 중요한지 사례와 함께 말해보세요.

### 합격 답변 예시

시간복잡도는 입력 크기에 따라 실행 시간이 어떻게 증가하는지, 공간복잡도는 메모리 사용량이 어떻게 증가하는지를 설명합니다. 실무에서는 둘 중 하나가 항상 더 중요한 게 아니라, **병목이 어디 있는지에 따라 우선순위가 달라집니다**.

### 가산점 포인트

**"시간복잡도가 더 중요합니다"라고 단정하면 약하다.** 좋은 답변은 "실무에서는 데이터 크기, 기기 성능, UX 요구사항에 따라 다릅니다"이다.

### 상세 해설

```ts
// 시간복잡도가 중요한 경우: 검색 자동완성
// 매 키 입력마다 실행 → O(n²)이면 사용자가 느낌
function filterItems(items: Item[], query: string) {
  return items.filter(item => item.name.includes(query));
  // O(n * m) — n: 아이템 수, m: 문자열 길이
}

// 공간복잡도가 중요한 경우: 모바일 이미지 캐시
// 메모리 제한이 있는 환경에서 캐시를 무한히 늘리면 OOM
const imageCache = new Map();  // 무한 증가하면 위험
// → LRU 캐시로 크기 제한 필요
```

| 상황 | 우선 고려 | 이유 |
|------|----------|------|
| 검색 자동완성 | 시간복잡도 | 매 키 입력마다 실행 |
| 모바일 이미지 캐시 | 공간복잡도 | 메모리 제한 |
| 브라우저 탭 오래 열림 | 공간 (메모리 누수) | 힙 계속 증가 |
| 수천 행 테이블 렌더링 | 둘 다 | DOM 수 + 렌더 시간 |

> 복잡도를 알았으니, 다음에는 **복잡도 감각이 필요한 대표 알고리즘** — 정렬을 봐야 한다.

## 14. 정렬 알고리즘 중 퀵 정렬과 병합 정렬의 차이를 설명해보세요.

### 합격 답변 예시

퀵 정렬은 평균적으로 매우 빠르고 실용성이 높지만, pivot 선택이 나쁘면 최악 O(n²)까지 갈 수 있습니다. 병합 정렬은 항상 O(n log n)을 안정적으로 보장하고 stable sort 특성이 있지만, 추가 메모리가 필요합니다.

### 가산점 포인트

JavaScript의 `Array.sort()`는 엔진마다 다르지만, V8은 **TimSort** (병합 정렬 + 삽입 정렬 하이브리드)를 사용한다. stable sort가 보장되며, 실무에서는 대부분 내장 sort를 쓰되, **비교 함수(comparator)를 올바르게 작성하는 것**이 더 중요하다.

### 상세 해설

| | 퀵 정렬 | 병합 정렬 |
|---|---|---|
| **평균** | O(n log n) | O(n log n) |
| **최악** | O(n²) | O(n log n) |
| **안정성** | 불안정 (unstable) | 안정 (stable) |
| **추가 메모리** | O(log n) | O(n) |
| **실용성** | 캐시 친화적, 실제로 빠름 | 안정적, 예측 가능 |

```ts
// JavaScript sort()는 stable (V8 TimSort)
const items = [
  { name: '홍길동', score: 90 },
  { name: '김영희', score: 90 },
  { name: '이철수', score: 85 },
];

items.sort((a, b) => b.score - a.score);
// 홍길동과 김영희의 순서가 유지됨 (stable)

// ⚠️ 비교 함수 없이 sort()하면 문자열 비교
[10, 2, 1].sort();          // [1, 10, 2] ← 틀림!
[10, 2, 1].sort((a, b) => a - b);  // [1, 2, 10] ← 올바름
```

> 정렬을 알았으니, 다음에는 **저장소 수준의 최적화** — DB 인덱스를 봐야 한다.

## 15. 데이터베이스에서 인덱스가 왜 필요한지 설명해보세요.

### 합격 답변 예시

인덱스는 특정 컬럼 값을 빠르게 찾기 위한 보조 자료구조입니다. 인덱스가 없으면 전체 테이블을 훑어야(full scan) 하지만, 인덱스가 있으면 필요한 범위를 훨씬 빨리 찾을 수 있습니다. 다만 **읽기는 빨라지지만 쓰기 비용이 증가**합니다.

### 가산점 포인트

**"인덱스는 공짜가 아니다"**를 알아야 한다. insert/update/delete 때마다 인덱스도 갱신해야 하므로 쓰기 비용이 증가한다. 읽기가 많은 테이블에는 인덱스가 유리하고, 쓰기가 많은 테이블에는 인덱스를 신중히 추가해야 한다.

### 상세 해설

```
인덱스 없음 (Full Scan):
  테이블: [A, B, C, D, E, F, G, H, I, J]
  "F를 찾아라" → A부터 하나씩 확인 → 6번 비교 → O(n)

인덱스 있음 (B-Tree):
  인덱스: [A-C] [D-F] [G-J]
           ↓      ↓
          [D] [E] [F] ← 직접 접근 → O(log n)
```

#### 프론트엔드 연결

```ts
// 프론트엔드에서 "인덱스" 같은 패턴
// ❌ 매번 find → Full Scan과 비슷
const user = users.find(u => u.id === 42);  // O(n)

// ✅ Map으로 "인덱스" 역할
const userIndex = new Map(users.map(u => [u.id, u]));
const user = userIndex.get(42);  // O(1)
```

#### 실무 에피소드

> 검색 API가 느려서 프론트엔드에서 loading spinner를 3초나 보여줘야 했다. 원인을 추적하니 DB에 인덱스가 없어서 100만 행을 full scan하고 있었다. 인덱스 추가 후 응답 시간이 3초 → 50ms로 줄었다. 프론트엔드 개발자도 "API가 왜 느린지" 감각이 있어야 백엔드와 효과적으로 소통할 수 있다.

> DB 인덱스를 알았으니, 다음에는 **데이터 구조 자체의 설계** — 정규화를 봐야 한다.

## 16. 정규화와 비정규화의 개념을 설명해보세요.

### 합격 답변 예시

정규화는 데이터 중복을 줄이고 이상 현상을 막기 위해 데이터를 역할별로 분리하는 과정입니다. 비정규화는 조회 성능이나 단순한 읽기 구조를 위해 일부 중복을 허용하는 전략입니다. 실무에서는 **데이터 정합성과 조회 성능 사이의 trade-off**를 보고 선택합니다.

### 가산점 포인트

프론트엔드 상태 관리에서도 **정규화 개념이 적용**된다. Redux에서 엔티티를 `byId` + `allIds`로 관리하는 것이 정규화, 컴포넌트 state에 관련 데이터를 모두 담는 것이 비정규화와 같은 원리다.

### 상세 해설

```ts
// 정규화된 상태 (Redux 스타일)
const state = {
  users: {
    byId: {
      1: { id: 1, name: '홍길동' },
      2: { id: 2, name: '김영희' },
    },
    allIds: [1, 2],
  },
  posts: {
    byId: {
      10: { id: 10, title: '글 1', authorId: 1 },  // user를 id로만 참조
    },
  },
};

// 비정규화된 상태 (중복 허용)
const state2 = {
  posts: [
    { id: 10, title: '글 1', author: { id: 1, name: '홍길동' } },
    // author 정보가 post마다 중복 → 이름이 바뀌면 모든 post를 수정해야
  ],
};
```

| | 정규화 | 비정규화 |
|---|---|---|
| **중복** | 적음 | 많음 |
| **수정** | 한 곳만 바꾸면 됨 | 여러 곳 바꿔야 함 |
| **조회** | join / lookup 필요 | 바로 읽기 가능 |
| **일관성** | 유지하기 쉬움 | 깨질 위험 |

> 정규화를 알았으니, 다음에는 **데이터 안전성의 기본** — 트랜잭션 ACID를 봐야 한다.

## 17. 트랜잭션의 ACID를 설명해보세요.

### 합격 답변 예시

ACID는 원자성(Atomicity), 일관성(Consistency), 격리성(Isolation), 지속성(Durability)입니다. 원자성은 작업이 전부 성공하거나 전부 실패해야 한다는 의미이고, 일관성은 트랜잭션 전후로 데이터 규칙이 유지되어야 한다는 뜻입니다. 격리성은 동시에 실행되는 트랜잭션이 서로 간섭하지 않도록 하는 것이고, 지속성은 커밋된 결과가 장애가 나도 유지되어야 한다는 의미입니다.

### 가산점 포인트

암기형으로 끝내지 말고 **실무 예시를 붙이면** 좋다. "주문 생성 + 결제 기록 + 재고 차감이 하나로 묶여야 하고, 중간에 실패하면 전부 롤백되어야 한다"는 식으로.

### 상세 해설

```ts
// 프론트엔드에서의 ACID 유사 개념: Optimistic Update + Rollback
const mutation = useMutation({
  mutationFn: createOrder,
  onMutate: async (newOrder) => {
    // 원자성: UI를 먼저 업데이트 (optimistic)
    await queryClient.cancelQueries({ queryKey: ['orders'] });
    const previous = queryClient.getQueryData(['orders']);
    queryClient.setQueryData(['orders'], (old) => [...old, newOrder]);
    return { previous };
  },
  onError: (err, newOrder, context) => {
    // 실패 시 롤백 (원자성 보장)
    queryClient.setQueryData(['orders'], context?.previous);
  },
});
```

#### ACID를 실무와 연결

| 속성 | 의미 | 실무 예시 |
|------|------|----------|
| **원자성** | 전부 성공 or 전부 실패 | 결제 실패 시 주문도 취소 |
| **일관성** | 데이터 규칙 유지 | 재고가 음수가 되면 안 됨 |
| **격리성** | 동시 작업 간섭 방지 | 같은 좌석을 두 명이 예약 못 하게 |
| **지속성** | 커밋 후 영구 저장 | 서버 재시작해도 데이터 유지 |

> ACID를 알았으니, 다음에는 **분산 시스템의 제약** — CAP 이론을 봐야 한다.

## 18. CAP 이론을 들어본 적이 있다면 설명해보세요.

### 합격 답변 예시

CAP 이론은 분산 시스템에서 Consistency(일관성), Availability(가용성), Partition Tolerance(분할 내성) 세 가지를 동시에 완벽하게 만족시키기 어렵다는 설명입니다. 특히 네트워크 분할이 발생하는 상황에서는 일관성과 가용성 사이에서 우선순위를 정해야 합니다.

### 가산점 포인트

**"항상 둘만 선택 가능"처럼 기계적으로 말하면 약하다.** 더 좋은 표현: "분산 환경에서 네트워크 분할은 피할 수 없으므로, 일관성과 가용성 사이의 균형이 필요하다." 프론트엔드에서는 **오프라인 상태에서 캐시된 데이터를 보여줄지(가용성), 최신 데이터가 올 때까지 기다릴지(일관성)**가 같은 trade-off다.

### 상세 해설

```
프론트엔드에서의 CAP 유사 결정:

네트워크 끊김 (Partition)
  ↓
선택 1: 캐시된 데이터라도 보여준다 (Availability 우선)
  → 사용자가 뭔가 볼 수 있지만, 오래된 데이터일 수 있음

선택 2: "오프라인입니다" 메시지만 보여준다 (Consistency 우선)
  → 정확한 상태지만, 사용자가 아무것도 못 함

실무에서는 대부분 A를 선택하고, 데이터 신선도를 표시:
  "마지막 업데이트: 5분 전 (오프라인)"
```

> CAP을 알았으니, 다음에는 **성능 전략의 핵심** — 캐시를 봐야 한다.

---

## 19. 캐시가 왜 성능을 높이는지 CPU/브라우저/네트워크 관점 중 하나로 설명해보세요.

### 합격 답변 예시

네트워크 관점에서 캐시는 느리고 먼 원본 서버 접근을 줄여주기 때문에 성능을 높입니다. 같은 리소스를 매번 원 서버에서 가져오지 않고 브라우저 캐시나 CDN에서 재사용하면 지연 시간이 줄고 서버 부하도 감소합니다.

### 가산점 포인트

**캐시의 핵심 trade-off는 "신선도 vs 속도"**라고 말하면 좋다. 오래 캐시하면 빠르지만 오래된 데이터를 볼 위험이 있고, 캐시를 안 하면 항상 최신이지만 느리다.

### 상세 해설

#### 프론트엔드 캐시 계층

```
속도:    빠름 ←──────────────────────→ 느림
         메모리   디스크    CDN     원본 서버
         캐시     캐시      캐시

React Query   브라우저     CloudFront   /api/users
staleTime     Cache-Control  Edge Cache   DB 쿼리
```

```ts
// 브라우저 캐시: HTTP Cache-Control 헤더
// 정적 자산: 오래 캐시 (내용이 바뀌면 파일명이 바뀜)
Cache-Control: public, max-age=31536000, immutable
// → app.abc123.js — 해시가 바뀌면 새 파일

// API 응답: stale-while-revalidate
Cache-Control: max-age=60, stale-while-revalidate=300
// → 60초간 fresh, 이후 5분간 캐시 보여주면서 백그라운드 갱신
```

#### 자주 하는 오해

- "캐시를 지우면 문제가 해결된다" → **근본 원인이 아닐 수 있음.** 캐시 무효화 전략이 잘못된 것이지, 캐시 자체가 문제는 아니다.

> 캐시를 알았으니, 마지막으로 **CS 지식이 프론트엔드에서 왜 필요한지** 종합 정리를 해보자.

## 20. 프론트엔드 개발자도 CS 지식을 알아야 하는 이유를 본인 경험과 연결해서 설명해보세요.

### 합격 답변 예시

프론트엔드는 단순히 화면만 그리는 역할이 아니라, 비동기 처리, 상태 일관성, 렌더링 비용, 메모리, 네트워크 실패까지 다루는 직무라서 CS 지식이 실무 판단에 직접 연결됩니다. 예를 들어 race condition을 이해해야 오래된 응답이 최신 UI를 덮어쓰는 문제를 막을 수 있고, 자료구조와 복잡도 감각이 있어야 대용량 리스트 성능을 설계할 수 있습니다.

### 가산점 포인트

**"CS를 알아야 개발자처럼 보이니까요"가 아니라, "CS를 알아야 문제를 설명하고, 원인을 구분하고, 구조적으로 해결할 수 있기 때문입니다"**라고 말하면 좋다.

### 상세 해설

#### CS 개념 → 프론트엔드 문제 매핑

| CS 개념 | 프론트엔드 문제 |
|---------|--------------|

## 보완내용

### 왜 보완이 필요한가

기존 CS 20문항은 운영체제, 자료구조, DB, 캐시까지 넓고 안정적으로 구성돼 있다. 다만 프론트엔드 면접 관점에서는 "학교에서 배운 정도"와 "실무 판단에 연결 가능한 정도"를 더 잘 가르기 위해 몇 가지 운영체제·동기화 질문을 보강하는 편이 좋다.

### 추가하면 좋은 질문

#### 1. 뮤텍스와 세마포어의 차이를 설명해보세요.

##### 합격 답변 예시

뮤텍스는 하나의 스레드만 자원에 접근할 수 있도록 보장하는 잠금 장치이고, 세마포어는 동시에 접근 가능한 수를 제한하는 카운터 기반 동기화 도구입니다. 뮤텍스는 열쇠가 하나인 화장실이고, 세마포어는 남은 자리 수를 세는 주차장과 비슷합니다.

##### 상세 해설

- 뮤텍스: 한 번에 하나, 소유권 있음
- 세마포어: 카운터 기반, 소유권 없음
- 프론트엔드 연결: `p-limit` 같은 동시 요청 수 제한 라이브러리가 세마포어 원리

#### 2. starvation과 livelock은 무엇이고 deadlock과 어떻게 다른가요?

##### 합격 답변 예시

starvation은 특정 작업이 자원을 계속 못 받는 상태, livelock은 서로 양보하느라 진전이 없는 상태입니다. deadlock은 아예 멈추고, livelock은 움직이지만 전진 안 되고, starvation은 특정 주체만 소외됩니다.

##### 상세 해설

- 프론트엔드 연결: 마이크로태스크가 계속 쌓이면 매크로태스크(렌더링, setTimeout)가 밀리는 현상이 starvation의 일종

#### 3. 컨텍스트 스위칭 비용이 무엇인지 설명해보세요.

##### 합격 답변 예시

CPU가 현재 작업을 멈추고 다른 작업으로 전환할 때, 상태를 저장하고 복원하는 과정의 오버헤드입니다. 프로세스 간 전환이 스레드 간 전환보다 비쌉니다.

##### 상세 해설

- 직접 비용: 레지스터 저장/복원, 메모리 매핑 변경
- 간접 비용: 캐시 무효화, TLB 플러시
- 프론트엔드 연결: Chrome 탭이 많으면 메모리와 전환 비용이 늘어나는 이유

#### 4. CPU-bound와 I/O-bound 작업의 차이를 설명해보세요.

##### 합격 답변 예시

CPU-bound는 연산 자체가 병목, I/O-bound는 외부 입출력 대기가 병목입니다. 프론트엔드에서 무거운 계산은 Web Worker로, 네트워크 대기는 비동기 패턴으로 다룹니다.

##### 상세 해설

- CPU-bound: 이미지 리사이즈, JSON 대량 파싱 → 알고리즘 개선, 웹 워커
- I/O-bound: API 호출, 파일 읽기 → 비동기 처리, 캐싱, 재시도

#### 5. 재귀와 반복의 trade-off를 설명해보세요.

##### 합격 답변 예시

재귀는 트리 순회 같은 구조적 문제에 직관적이지만 스택 프레임이 쌓여 stack overflow 위험이 있습니다. 반복은 스택 부담이 없지만 코드가 덜 직관적일 수 있습니다.

##### 상세 해설

- 프론트엔드 연결: DOM 트리 재귀 순회, 깊은 중첩 객체 처리에서 깊이가 보장 안 되면 반복 + 명시적 스택으로 전환

#### 6. B-Tree 인덱스와 해시 인덱스의 차이는?

##### 합격 답변 예시

B-Tree는 범위 검색, 정렬, 부분 일치에 활용 가능하고, 해시는 정확한 값 일치에만 매우 빠릅니다. 대부분의 RDBMS는 B-Tree를 기본으로 사용합니다.

##### 상세 해설

- 프론트엔드 연결: 프론트에서 `Map`(해시)으로 id lookup, 정렬된 배열에서 이진 탐색(B-Tree 유사)을 하는 것이 같은 원리

### 이 섹션 보완 핵심

CS 섹션은 지금도 충분히 좋다. 여기에 동기화, 작업 유형 구분, 전환 비용, 재귀 실행 모델, 인덱스 구조를 더하면 "배운 적 있는 사람"과 "실무 판단에 연결 가능한 사람"을 더 잘 나눌 수 있다.
