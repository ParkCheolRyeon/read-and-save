# 05. cs 20문항

> 프로세스, 메모리, 자료구조, 데이터베이스, 캐시까지 프론트엔드와 연결되는 CS 20문항입니다.

## 5. CS 지식 질문 20문항

## 총평

이 섹션은 프론트엔드 면접에서도 생각보다 중요하다. 이유는 단순하다. 프론트엔드 개발자가 매일 다루는 문제들이 결국 CS 개념과 연결되기 때문이다.

- 이벤트 루프 → 동기/비동기, 블로킹/논블로킹
- React 상태 꼬임 → race condition
- 메모리 누수 → 힙, GC(Garbage Collection), 참조 유지
- 리스트 성능 → 자료구조, 복잡도
- API 캐싱 → 해시, 인덱싱, 시간/공간 trade-off

그래서 좋은 답변은 “교과서 정의”만 말하면 부족하다. 브라우저/자바스크립트/웹앱 문제와 연결해서 말해야 합격권이다.

## 1. 프로세스와 스레드의 차이를 설명해보세요.

### 합격 답변 예시

프로세스는 실행 중인 프로그램의 독립된 단위이고, 각 프로세스는 자신만의 메모리 공간을 가집니다. 스레드는 프로세스 내부에서 실행되는 작업 흐름 단위로, 같은 프로세스 안의 코드 영역, 힙, 데이터 영역을 공유하고 스택만 개별적으로 가집니다. 그래서 프로세스는 서로 격리돼 안정성이 높고, 스레드는 자원 공유가 쉬워서 효율적이지만 동기화 문제가 생기기 쉽습니다.

### 상세 해설

이 질문은 거의 모든 CS 면접의 시작점이다. 여기서 중요한 건 “독립 메모리냐, 공유 메모리냐”를 정확히 말하는 것이다.

- 프로세스
  - 운영체제가 관리하는 실행 단위
  - 서로 메모리 공간이 분리됨
  - 하나가 죽어도 다른 프로세스에 직접 영향이 덜 감
- 스레드
  - 프로세스 안의 실행 흐름
  - 자원을 공유함
  - 통신은 빠르지만, 같은 데이터를 만지기 때문에 충돌이 생기기 쉬움

#### 왜 프론트엔드도 알아야 하나?

브라우저도 내부적으로 렌더링, 네트워크, JS 실행, GPU 처리 같은 여러 계층으로 움직인다. 직접 스레드를 다루진 않아도, “공유 자원”과 “경합” 개념을 이해해야 UI 버그를 잘 설명할 수 있다.

#### 비유

프로세스는 각각 독립 사무실이고, 스레드는 같은 사무실 안의 팀원이라고 보면 된다. 사무실이 다르면 사고가 나도 번지기 어렵고, 같은 사무실이면 협업은 빠르지만 책상 위 자료를 같이 건드리다 꼬일 수 있다.

## 2. 멀티프로세스와 멀티스레드의 장단점을 설명해보세요.

### 합격 답변 예시

멀티프로세스는 프로세스마다 메모리가 분리되어 있어서 안정성과 격리가 좋고, 하나가 죽어도 전체에 영향이 덜합니다. 대신 프로세스 생성 비용이 크고, 프로세스 간 통신도 상대적으로 무겁습니다. 멀티스레드는 같은 프로세스의 자원을 공유하므로 생성과 통신이 더 가볍고 효율적이지만, race condition, deadlock 같은 동기화 문제가 더 쉽게 발생합니다. 즉 멀티프로세스는 안정성, 멀티스레드는 효율성 쪽에 강점이 있습니다.

### 상세 해설

면접에서 좋은 답변은 장점 1개, 단점 1개에서 끝나지 않는다. 반드시 **“왜 그런가”**를 설명해야 한다.

- 멀티프로세스 장점
  - 메모리 분리
  - 안정성
  - 보안/격리 유리
- 멀티프로세스 단점
  - 메모리 많이 먹음
  - IPC(Inter-Process Communication) 비용 큼
  - 컨텍스트 스위칭 비용 큼
- 멀티스레드 장점
  - 같은 메모리 공유
  - 통신 빠름
  - 가벼움
- 멀티스레드 단점
  - 동기화 복잡
  - 한 스레드 버그가 전체 프로세스에 영향 줄 수 있음

#### 실무 연결

Node.js는 JS 실행 자체는 단일 스레드처럼 보이지만, 실제로는 libuv, worker threads, OS(Operating System) 수준 비동기 I/O 같은 구조와 맞물린다. 프론트엔드 개발자도 “왜 어떤 작업은 메인 스레드를 막는가”를 이해하려면 이 감각이 필요하다.

## 3. 동기와 비동기의 차이를 설명해보세요.

### 합격 답변 예시

동기는 요청한 작업의 완료와 다음 작업 진행이 강하게 연결된 방식이고, 비동기는 작업을 요청한 뒤 완료를 기다리는 동안 다른 작업을 진행할 수 있는 방식입니다. 중요한 건 동기와 비동기는 속도의 문제가 아니라 흐름 제어 방식의 차이라는 점입니다. 프론트엔드에서는 네트워크 요청, 타이머, 사용자 입력 처리 등 대부분의 외부 작업을 비동기적으로 다뤄야 UI가 멈추지 않습니다.

### 상세 해설

많이 하는 실수는 이거다.

- 동기 = 느림
- 비동기 = 빠름

이건 틀렸다.

#### 핵심

동기/비동기는 기다리는 방식이 아니라 완료와 제어 흐름이 어떻게 연결되는지에 대한 개념이다.

- 예를 들어,
  - 동기 함수는 결과가 나와야 다음 줄로 감
  - 비동기 함수는 결과를 나중에 callback/promise/await로 이어감

#### 프론트엔드 예시

```ts
const data = await fetch("/api");
```

겉으로는 동기처럼 읽혀도 실제론 비동기다. 즉 await가 있다고 해서 “작업 자체”가 동기가 되는 건 아니다.

#### 면접 포인트

좋은 답변은 “작업 순서의 제어권이 어디 있느냐”를 말해야 한다.

## 4. 블로킹과 논블로킹의 차이를 설명해보세요.

### 합격 답변 예시

블로킹은 어떤 작업을 요청한 쪽이 그 작업이 끝날 때까지 제어권을 돌려받지 못하는 방식이고, 논블로킹은 요청 후 바로 제어권을 돌려받아 다른 작업을 계속할 수 있는 방식입니다. 동기/비동기와 비슷해 보이지만 다른 축의 개념입니다. 예를 들어 비동기 API를 호출해도 실제 사용자 입장에서는 로딩 스피너만 계속 돌고 아무 상호작용이 안 되면 체감상 블로킹처럼 느껴질 수 있습니다.

### 상세 해설

이 질문은 동기/비동기와 자주 묶여 나온다.

#### 구분

- 동기/비동기: 완료 시점과 흐름 연결
- 블로킹/논블로킹: 제어권을 바로 돌려주느냐

#### 예시

- 블로킹: 함수 호출 후 끝날 때까지 못 돌아옴
- 논블로킹: 일단 돌아오고 나중에 결과를 처리

#### 프론트엔드 연결

메인 스레드에서 긴 계산을 돌리면 JS는 사실상 블로킹된다. 사용자는 버튼 클릭도 안 되고 스크롤도 버벅이는 경험을 한다. 즉 브라우저에서는 “메인 스레드를 막는가”가 매우 중요하다.

## 5. 데드락의 발생 조건 4가지를 설명해보세요.

### 합격 답변 예시

데드락의 고전적인 발생 조건은 상호 배제, 점유 대기, 비선점, 순환 대기입니다. 즉 어떤 자원을 한 번에 하나만 사용할 수 있고, 이미 자원을 가진 상태로 다른 자원을 기다리고, 자원을 강제로 빼앗을 수 없고, 기다림이 원형으로 이어지면 데드락이 발생할 수 있습니다.

### 상세 해설

이건 암기형처럼 보이지만, 사실 조건이 하나라도 깨지면 데드락을 막을 수 있다는 점이 핵심이다.

- 4조건
  - 상호 배제: 자원은 한 번에 하나만 사용 가능
  - 점유 대기: 이미 가진 채로 다른 자원을 기다림
  - 비선점: 강제로 뺏을 수 없음
  - 순환 대기: A는 B를 기다리고, B는 C를, C는 A를 기다림

#### 면접에서 좋은 답변

조건을 말한 뒤, “그래서 실무에서는 자원 획득 순서를 강제하거나 timeout을 두어 회피할 수 있습니다”까지 말하면 좋다.

## 6. Race Condition이 무엇인지, 프론트엔드에서도 비슷한 문제가 생길 수 있는지 설명해보세요.

### 합격 답변 예시

Race Condition은 여러 작업이 같은 상태나 자원에 접근할 때, 실행 순서에 따라 결과가 달라지는 문제입니다. 프론트엔드에서도 자주 발생합니다. 대표적으로 검색 요청을 연속으로 보냈는데, 더 늦게 보낸 최신 요청보다 이전 요청 응답이 나중에 도착해서 화면을 덮어쓰는 경우가 있습니다. 그래서 AbortController, 요청 식별자 비교, 최신 요청만 반영하는 정책 같은 방어 장치가 필요합니다.

### 상세 해설

프론트엔드 면접에서는 이 질문을 매우 잘 연결해서 답할 수 있어야 한다.

- 대표 사례
- 검색창 자동완성
- 탭 전환 중 API 응답 순서 꼬임
- 버튼 중복 클릭
- optimistic update 후 rollback 타이밍 꼬임

#### 예시 흐름

- 사용자 입력 a
- 요청 A 발생
- 사용자 입력 ab
- 요청 B 발생
- B가 먼저 성공
- A가 늦게 와서 화면을 다시 덮음

이게 race condition이다.

#### 왜 중요한가?

이걸 이해 못하면 “가끔 이상하게 옛날 데이터가 보여요” 같은 버그를 설명하기 어렵다.

## 7. 메모리 영역(코드, 스택, 힙, 데이터)을 설명해보세요.

### 합격 답변 예시

코드 영역은 실행 명령어가 저장되는 곳이고, 데이터 영역은 전역 변수나 정적 데이터가 저장되는 영역입니다. 스택은 함수 호출 정보와 지역 변수처럼 실행 흐름과 밀접한 값이 쌓이는 공간이고, 힙은 런타임에 동적으로 할당되는 객체들이 저장되는 공간입니다. 스택은 빠르지만 크기가 제한적이고, 힙은 유연하지만 참조 관리와 가비지 컬렉션이 중요합니다.

### 상세 해설

이 질문은 운영체제/언어 실행 모델의 기본이다.

- 코드 영역
  - 실행할 명령어
  - 보통 읽기 전용 성격
- 데이터 영역
  - 전역 변수
  - static 데이터
- 스택
  - 함수 호출 프레임
  - 매개변수
  - 지역 변수
  - LIFO(Last In First Out) 구조
- 힙
  - 동적 객체
  - 배열, 객체, 함수 클로저 캡처 등

#### 프론트엔드 연결

- 재귀 너무 깊음 → stack overflow
- 객체 참조 안 끊김 → heap 증가
- 이벤트 리스너/클로저 → 힙 유지

## 8. 가비지 컬렉션이 무엇인지, JavaScript에서 어떻게 메모리를 회수하는지 설명해보세요.

### 합격 답변 예시

가비지 컬렉션은 프로그램에서 더 이상 사용할 수 없는 메모리를 자동으로 회수하는 메커니즘입니다. JavaScript 엔진은 보통 루트 객체에서 도달 가능한지를 기준으로 객체의 생존 여부를 판단합니다. 그래서 변수에서 끊겼다고 생각해도, 클로저, 이벤트 리스너, DOM 참조, 전역 캐시 등에서 참조가 남아 있으면 메모리가 계속 유지될 수 있습니다.

### 상세 해설

이 질문의 핵심은 **“GC가 알아서 해준다”가 아니라 “참조가 남으면 못 치운다”**는 것이다.

- JS에서 흔한 누수
  - window에 리스너 달고 안 지움
  - 큰 배열을 closure가 계속 잡음
  - 캐시에 계속 push만 하고 정리 안 함
  - 제거된 DOM 노드를 참조 중

#### 좋은 답변 포인트

GC 알고리즘 이름을 길게 말하는 것보다, 도달 가능성(reachability) 과 참조 유지를 정확히 말하는 게 중요하다.

## 9. 자료구조 중 배열과 연결 리스트의 차이를 설명해보세요.

### 합격 답변 예시

배열은 인덱스를 통한 임의 접근이 빠르고, 전통적인 배열은 메모리상 연속 구간을 전제로 설명하는 경우가 많습니다. 반면 연결 리스트는 각 노드가 다음 노드를 가리키는 방식이라 중간 삽입·삭제는 유리할 수 있지만, 원하는 위치까지 순차적으로 따라가야 해서 임의 접근이 느립니다. 실무에서는 대부분 배열을 더 많이 쓰지만, 중요한 건 어떤 연산이 많은지에 따라 자료구조를 선택하는 감각입니다.

### 상세 해설

면접에서 중요한 건 “배열이 더 좋다”가 아니다.

- 배열 강점
  - 인덱스 접근 빠름
  - 캐시 친화적
  - JS/브라우저에서 매우 자연스러움
- 배열 약점
  - 중간 삽입/삭제 비용 큼
- 연결 리스트 강점
  - 중간 삽입/삭제 유리
- 연결 리스트 약점
  - 임의 접근 느림
  - 메모리 locality가 약함

#### 프론트엔드 실무

대부분 배열로 충분하다. 하지만 정렬/필터/가상화 설계에서 “배열을 매 렌더마다 전체 복사하는 비용” 같은 관점은 중요하다.

## 10. 해시 테이블이 무엇이고 평균 시간복잡도가 왜 O(1)인지 설명해보세요.

### 합격 답변 예시

해시 테이블은 키를 해시 함수로 변환해 저장 위치를 찾는 자료구조입니다. 평균적으로 O(1)인 이유는 키를 거의 즉시 특정 버킷으로 매핑할 수 있기 때문입니다. 다만 충돌이 많아지면 같은 버킷 안에서 여러 값을 비교해야 해서 최악의 경우 성능이 나빠질 수 있으므로, 엄밀히는 평균 O(1)이라고 보는 게 맞습니다.

### 상세 해설

#### 핵심

- 해시 함수가 키를 위치로 바꿈
- 바로 찾아갈 수 있으니 평균 빠름
- 충돌 관리가 성능의 핵심

#### 프론트엔드 연결

- JS Map, 객체 딕셔너리, 캐시 테이블, id 기반 lookup
- 예를 들어 배열에서 find로 매번 찾는 대신, id → entity map으로 저장하면 lookup 비용을 줄일 수 있다.

## 11. 스택과 큐를 실제 웹 개발 상황과 연결해서 설명해보세요.

### 합격 답변 예시

스택은 후입선출 구조라 함수 호출 스택, 브라우저의 콜 스택, undo 히스토리 일부 모델과 연결해 설명할 수 있습니다. 큐는 FIFO(First In First Out), 즉 선입선출 구조라 이벤트 큐, 작업 큐, 토스트 메시지 순차 처리, 요청 버퍼링 같은 상황과 연결할 수 있습니다. 프론트엔드에서는 특히 이벤트 루프를 이해할 때 스택과 큐 개념이 매우 중요합니다.

### 상세 해설

이 질문은 개념을 브라우저와 엮는 순간 답변이 확 강해진다.

#### 스택 예시

- JS 콜 스택
- 재귀 호출
- 뒤로 가기 히스토리 일부 모델링

#### 큐 예시

- 이벤트 큐
- 작업 순차 처리
- 업로드 대기열
- 알림 메시지 순서

#### 면접 포인트

“자료구조를 알고 있다”보다 브라우저 실행 모델과 연결해서 설명할 수 있다가 중요하다.

## 12. 트리와 그래프의 차이를 설명해보세요.

### 합격 답변 예시

트리는 그래프의 한 종류로 볼 수 있고, 보통 계층 구조를 가지며 사이클이 없습니다. 그래프는 더 일반적인 개념이라 노드와 간선으로 연결되고, 사이클도 있을 수 있으며 부모-자식 관계가 꼭 필요하지 않습니다. DOM 구조나 React 컴포넌트 구조는 트리로 이해하기 좋고, 페이지 링크 구조나 의존성 구조는 그래프로 이해하기 좋습니다.

### 상세 해설

- 트리
  - 계층 구조
  - 루트 존재
  - 보통 사이클 없음
- 그래프
  - 더 일반적
  - 방향/무방향 가능
  - 사이클 가능

#### 프론트엔드 연결

- DOM tree, 컴포넌트 tree, 라우팅 tree
- 번들 dependency graph, 웹사이트 링크 graph

## 13. 시간복잡도와 공간복잡도를 설명하고, 실무에서 둘 중 무엇이 더 중요한지 사례와 함께 말해보세요.

### 합격 답변 예시

시간복잡도는 입력 크기에 따라 실행 시간이 어떻게 증가하는지를 설명하는 개념이고, 공간복잡도는 필요한 메모리 사용량이 얼마나 증가하는지를 설명합니다. 실무에서는 둘 중 하나가 항상 더 중요한 게 아니라, 병목이 어디 있는지에 따라 우선순위가 달라집니다. 예를 들어 대용량 리스트 렌더링에서는 시간복잡도와 렌더 비용이 더 중요할 수 있고, 모바일이나 큰 캐시 구조에서는 메모리 사용량이 더 중요할 수 있습니다.

### 상세 해설

많은 지원자가 “시간복잡도가 더 중요합니다”라고 단정한다. 그건 약하다.

- 좋은 답변: “실무에서는 데이터 크기, 기기 성능, UX 요구사항에 따라 다릅니다.”
- 사례
  - 검색 자동완성: 시간복잡도 중요
  - 이미지 캐시 앱: 공간복잡도 중요
  - 브라우저 탭 오래 열림: 메모리 누수 중요
  - 수천 행 테이블: 시간+공간 둘 다 중요

즉 정답은 측정하고, 실제 병목을 기준으로 판단하는 것이다.

## 14. 정렬 알고리즘 중 퀵 정렬과 병합 정렬의 차이를 설명해보세요.

### 합격 답변 예시

퀵 정렬은 평균적으로 매우 빠르고 실용성이 높지만, pivot 선택이 나쁘면 최악의 경우 O(n²)까지 갈 수 있습니다. 병합 정렬은 항상 O(n log n)을 안정적으로 보장하고 stable sort 특성이 있지만, 보통 추가 메모리가 더 필요합니다. 즉 퀵 정렬은 평균 성능과 실용성, 병합 정렬은 안정성과 예측 가능성이 강점입니다.

### 상세 해설

- 퀵 정렬
  - 분할 정복
  - 평균 빠름
  - 최악 O(n²)
- 병합 정렬
  - 분할 후 병합
  - 항상 O(n log n)
  - stable
  - 추가 메모리 필요

#### 실무 포인트

프론트엔드에서 직접 정렬 알고리즘을 구현할 일은 적지만, 정렬의 비용, 안정성, 데이터량에 대한 감각은 중요하다.

## 15. 데이터베이스에서 인덱스가 왜 필요한지 설명해보세요.

### 합격 답변 예시

인덱스는 특정 컬럼 값을 빠르게 찾기 위한 보조 자료구조입니다. 인덱스가 없으면 조건 검색 시 테이블 전체를 훑어야 할 수 있지만, 인덱스가 있으면 필요한 범위를 훨씬 빨리 찾을 수 있습니다. 다만 읽기 성능은 좋아지지만 쓰기, 수정, 삭제 시에는 인덱스도 같이 갱신해야 하므로 비용이 생깁니다.

### 상세 해설

이 질문의 핵심은 “왜 빠른가”와 “왜 공짜가 아닌가” 둘 다 말해야 한다.

#### 장점

- 조회 빠름
- 정렬/검색 효율 향상

#### 단점

- 저장 공간 사용
- insert/update/delete 비용 증가

#### 프론트엔드 연결

검색/필터 API가 느릴 때 프론트만 의심하면 안 된다. DB 인덱스 설계가 병목일 수 있다는 감각이 중요하다.

## 16. 정규화와 비정규화의 개념을 설명해보세요.

### 합격 답변 예시

정규화는 데이터 중복을 줄이고 이상 현상을 막기 위해 데이터를 역할별로 분리하는 과정입니다. 비정규화는 조회 성능이나 단순한 읽기 구조를 위해 일부 중복을 허용하는 전략입니다. 실무에서는 무조건 정규화만 추구하기보다, 데이터 정합성과 조회 성능 사이의 trade-off를 보고 선택합니다.

### 상세 해설

- 정규화 장점
  - 중복 감소
  - 수정 이상 방지
  - 데이터 일관성 향상
- 정규화 단점
  - join 증가
  - 조회 복잡도 증가 가능
- 비정규화 장점
  - 읽기 최적화
  - 단순한 조회 모델
- 비정규화 단점
  - 중복 증가
  - 정합성 관리 어려움

## 17. 트랜잭션의 ACID(Atomicity, Consistency, Isolation, Durability)를 설명해보세요.

### 합격 답변 예시

ACID는 원자성, 일관성, 격리성, 지속성입니다. 원자성은 작업이 전부 성공하거나 전부 실패해야 한다는 의미이고, 일관성은 트랜잭션 전후로 데이터 규칙이 유지되어야 한다는 뜻입니다. 격리성은 동시에 실행되는 트랜잭션이 서로 잘못 간섭하지 않도록 하는 것이고, 지속성은 커밋된 결과가 장애가 나도 유지되어야 한다는 의미입니다.

### 상세 해설

이 질문은 보통 암기형처럼 답하기 쉽다. 그런데 좋은 답변은 실무 예시가 붙는다.

- 예:
  - 주문 생성 + 결제 기록 + 재고 차감은 하나로 묶여야 한다
  - 중간에 하나 실패하면 전부 롤백돼야 한다

이런 식으로 말하면 훨씬 좋다.

## 18. CAP(Consistency, Availability, Partition tolerance) 이론을 들어본 적이 있다면 설명해보세요.

### 합격 답변 예시

CAP 이론은 분산 시스템에서 Consistency, Availability, Partition Tolerance 세 가지를 동시에 완벽하게 만족시키기 어렵다는 설명입니다. 특히 네트워크 분할이 발생하는 상황에서는 일관성과 가용성 사이에서 우선순위를 정해야 한다는 의미로 많이 사용됩니다. 면접에서는 CAP을 절대 규칙처럼 말하기보다, 분산 시스템의 trade-off를 설명하는 틀로 이해하고 있다고 답하면 좋습니다.

### 상세 해설

여기서 조심할 점은 “항상 둘만 선택 가능”처럼 기계적으로 말하는 것이다.

- 더 좋은 표현은:
  - 분산 환경은 네트워크 분할을 완전히 피할 수 없다
  - 그래서 일관성과 가용성 사이의 균형이 필요하다

이 정도면 충분히 좋다.

## 19. 캐시가 왜 성능을 높이는지 CPU(Central Processing Unit)/브라우저/네트워크 관점 중 하나로 설명해보세요.

### 합격 답변 예시

네트워크 관점에서 캐시는 느리고 먼 원본 서버 접근을 줄여주기 때문에 성능을 높입니다. 같은 리소스를 매번 원 서버에서 가져오지 않고 브라우저 캐시나 CDN에서 재사용하면 지연 시간이 줄고 서버 부하도 감소합니다. 프론트엔드에서는 이미지, JS 번들, CSS, API 응답 캐시가 체감 성능에 큰 영향을 줍니다.

### 상세 해설

이 질문은 하나의 관점만 깊게 잡으면 된다.

- 네트워크 관점 핵심
  - 원본 접근 감소
  - RTT 감소
  - 서버 부하 감소
  - 체감 로딩 개선
- 프론트엔드 연결
  - CDN 캐시
  - 브라우저 캐시
  - React Query 캐시
  - Next.js Data Cache

## 20. 프론트엔드 개발자도 CS 지식을 알아야 하는 이유를 본인 경험과 연결해서 설명해보세요.

### 합격 답변 예시

프론트엔드는 단순히 화면만 그리는 역할이 아니라, 비동기 처리, 상태 일관성, 렌더링 비용, 메모리, 네트워크 실패까지 다루는 직무라서 CS 지식이 실무 판단에 직접 연결됩니다. 예를 들어 race condition을 이해해야 오래된 응답이 최신 UI를 덮어쓰는 문제를 막을 수 있고, 자료구조와 복잡도 감각이 있어야 대용량 리스트나 차트 성능을 설계할 수 있습니다. 저는 실제로 네트워크 불안정 환경 대응이나 상태 구조 설계, 렌더링 최적화 같은 문제를 다루면서, 단순 구현력보다 개념 이해가 유지보수 품질을 더 크게 좌우한다고 느꼈다고 말하면 좋습니다.

### 상세 해설

이 질문은 사실 지원자의 개발 철학을 본다.

여기서 좋은 답변은 “CS를 알아야 개발자처럼 보이니까요”가 아니다.

정답은 “CS를 알아야 문제를 설명하고, 원인을 구분하고, 구조적으로 해결할 수 있기 때문입니다.”이다.

## 보완내용

### 왜 보완이 필요한가

기존 CS 20문항은 운영체제, 자료구조, DB, 캐시까지 넓고 안정적으로 구성돼 있다. 다만 프론트엔드 면접 관점에서는 “학교에서 배운 정도”와 “실무 판단에 연결 가능한 정도”를 더 잘 가르기 위해 몇 가지 운영체제·동기화 질문을 보강하는 편이 좋다.

특히 공유 자원 제어, CPU/I/O 작업 구분, 컨텍스트 스위칭 비용 같은 개념은 브라우저 성능과 비동기 설계에 직접 연결되기 때문에 프론트엔드 개발자에게도 실전성이 높다.

### 추가하면 좋은 질문

#### 1. 뮤텍스와 세마포어의 차이를 설명해보세요.

동기화 기본기 확인용으로 좋다. 직접 OS 수준 코드를 짜지 않더라도 “공유 자원을 어떤 방식으로 제어하는가”를 설명할 수 있으면 시스템 감각이 드러난다.

##### 합격 답변 예시

뮤텍스는 하나의 스레드만 자원에 접근할 수 있도록 보장하는 잠금 장치이고, 세마포어는 동시에 접근 가능한 수를 제한하는 카운터 기반 동기화 도구입니다. 뮤텍스는 잠금을 건 스레드만 해제할 수 있다는 소유권 개념이 있지만, 세마포어는 소유권 없이 카운트만 관리합니다. 쉽게 말하면 뮤텍스는 열쇠가 하나인 화장실이고, 세마포어는 남은 자리 수를 세는 주차장과 비슷합니다.

##### 상세 해설

뮤텍스와 세마포어는 둘 다 공유 자원 접근을 제어하지만 설계 목적이 다르다.

- 뮤텍스
  - 한 번에 하나의 스레드만 진입 가능
  - 잠금을 건 스레드만 해제 가능 (소유권 존재)
  - 주로 임계 영역 보호에 사용
- 세마포어
  - 카운터 값으로 동시 접근 수를 제한
  - 카운트가 1이면 바이너리 세마포어로 뮤텍스와 비슷하게 동작
  - 카운트가 N이면 최대 N개 스레드가 동시에 접근 가능
  - 소유권 개념이 없어서 다른 스레드가 signal을 보낼 수 있음

프론트엔드에서 직접 뮤텍스나 세마포어를 쓸 일은 드물지만, 동시 요청 수를 제한하거나(예: API 요청을 최대 3개까지만 병렬로 보내기) 공유 상태에 대한 접근을 직렬화하는 패턴은 같은 원리에서 나온다. Promise 기반 큐나 p-limit 같은 라이브러리가 세마포어와 동일한 개념이다.

#### 2. starvation과 livelock은 무엇이고 deadlock과 어떻게 다른가요?

기존 deadlock 질문의 좋은 확장판이다. 단순 암기에서 한 단계 더 들어간 동시성 감각을 확인할 수 있다.

##### 합격 답변 예시

starvation은 특정 프로세스나 스레드가 자원을 계속 할당받지 못해 무한히 대기하는 상태이고, livelock은 서로 양보하느라 실제 작업이 전혀 진행되지 않는 상태입니다. deadlock은 아예 멈춰 있는 반면, livelock은 움직이고 있지만 전진이 안 되고, starvation은 특정 주체만 소외되는 것이 핵심 차이입니다.

##### 상세 해설

세 가지 모두 동시성 환경에서 발생하는 진행 불가 문제이지만 원인과 양상이 다르다.

- deadlock
  - 두 개 이상의 작업이 서로가 가진 자원을 기다리며 완전히 멈춤
  - 아무 작업도 실행되지 않음
- starvation
  - 시스템 전체는 동작하지만 특정 작업이 계속 밀려남
  - 우선순위가 낮은 스레드가 높은 스레드에 계속 밀리는 경우가 대표적
  - 해결 방법으로는 공정 스케줄링, aging(대기 시간이 길어지면 우선순위를 높이는 기법) 등이 있음
- livelock
  - 서로 충돌을 감지하고 양보를 반복하지만 결과적으로 아무도 진행 못 함
  - 복도에서 마주친 두 사람이 같은 방향으로 계속 비키는 상황과 비슷함
  - 해결 방법으로는 랜덤 대기, 백오프 전략 등이 있음

프론트엔드에서 starvation은 이벤트 루프에서도 나타날 수 있다. 예를 들어 마이크로태스크가 계속 쌓이면 매크로태스크(렌더링, setTimeout)가 밀려서 화면 업데이트가 안 되는 현상이 starvation의 일종이다.

#### 3. 컨텍스트 스위칭 비용이 무엇인지 설명해보세요.

멀티프로세스/멀티스레드 장단점 질문과 자연스럽게 이어진다. 왜 가벼운 단위가 항상 공짜가 아닌지, 전환 비용이 왜 성능 trade-off가 되는지 설명할 수 있어야 한다.

##### 합격 답변 예시

컨텍스트 스위칭은 CPU가 현재 실행 중인 프로세스나 스레드를 멈추고 다른 것으로 전환할 때, 기존 상태를 저장하고 새로운 상태를 복원하는 과정입니다. 이 과정에서 레지스터 저장/복원, 메모리 매핑 변경, 캐시 무효화 등의 오버헤드가 발생합니다. 프로세스 간 전환이 스레드 간 전환보다 비용이 더 큰 이유는 메모리 공간 자체를 바꿔야 하기 때문입니다.

##### 상세 해설

컨텍스트 스위칭이 발생하면 다음과 같은 비용이 생긴다.

- 직접 비용
  - CPU 레지스터 값 저장 및 복원
  - 프로그램 카운터, 스택 포인터 등 상태 정보 교체
  - 프로세스 전환의 경우 메모리 주소 공간(페이지 테이블) 변경
- 간접 비용
  - CPU 캐시(L1, L2)가 무효화되어 캐시 미스 증가
  - TLB(Translation Lookaside Buffer) 플러시
  - 파이프라인 비움

스레드 간 전환은 같은 프로세스 내에서 일어나므로 메모리 공간 변경이 필요 없어 프로세스 전환보다 가볍다. 그래도 공짜는 아니다.

프론트엔드 관점에서 보면, 브라우저가 여러 탭을 관리할 때 각 탭이 별도 프로세스로 돌아가는 이유(크롬의 멀티프로세스 아키텍처)와, 탭이 많아지면 메모리와 전환 비용이 늘어나는 이유가 바로 이 컨텍스트 스위칭 비용과 관련된다. 또한 JS 이벤트 루프에서 태스크 간 전환도 비용이 있으므로, 작업을 너무 잘게 쪼개는 것이 항상 좋은 건 아니라는 감각이 필요하다.

#### 4. CPU-bound와 I/O-bound 작업의 차이를 설명해보세요.

프론트엔드에서도 매우 중요하다. 어떤 작업은 웹 워커로 넘겨야 하고, 어떤 작업은 네트워크 대기와 재시도 전략이 더 핵심인지 구분하는 기반이 된다.

##### 합격 답변 예시

CPU-bound 작업은 연산 자체가 병목인 작업이고, I/O-bound 작업은 외부 입출력 대기가 병목인 작업입니다. 예를 들어 이미지 처리, 대량 데이터 정렬, 암호화 같은 작업은 CPU-bound이고, API 호출, 파일 읽기, DB 쿼리 같은 작업은 I/O-bound입니다. 프론트엔드에서는 무거운 계산은 웹 워커로 분리하고, 네트워크 요청은 비동기 패턴과 재시도 전략으로 다루는 것이 핵심입니다.

##### 상세 해설

이 구분이 중요한 이유는 병목의 위치에 따라 최적화 전략이 완전히 다르기 때문이다.

- CPU-bound
  - 병목: 연산 처리 속도
  - 예시: 이미지 리사이즈, JSON 대량 파싱, 복잡한 정렬, 암호화/해싱
  - 최적화 방향: 알고리즘 개선, 병렬 처리(웹 워커, worker threads), 연산 분할
- I/O-bound
  - 병목: 외부 자원 응답 대기
  - 예시: API 호출, 파일 시스템 접근, DB 쿼리, 사용자 입력 대기
  - 최적화 방향: 비동기 처리, 캐싱, 병렬 요청, timeout/재시도 전략

프론트엔드에서 메인 스레드가 버벅이는 문제는 대부분 CPU-bound 작업이 메인 스레드를 점유하고 있기 때문이다. 이럴 때 웹 워커로 분리하거나 requestIdleCallback으로 작업을 쪼개는 전략이 필요하다. 반면 API 응답이 느린 문제는 I/O-bound이므로 로딩 상태 처리, 캐시 활용, optimistic update 같은 UX 전략이 더 효과적이다.

#### 5. 재귀와 반복의 trade-off를 설명해보세요. stack overflow는 왜 발생하나요?

렌더링 트리 순회, DFS(Depth First Search)/BFS(Breadth First Search) 사고, 깊은 호출 체인과 연결하기 좋은 질문이다. 추상 자료구조 이해를 실제 실행 모델과 연결해 설명할 수 있다.

##### 합격 답변 예시

재귀는 함수가 자기 자신을 호출하는 방식으로, 트리 순회나 분할 정복 같은 문제를 직관적으로 표현할 수 있지만, 호출마다 스택 프레임이 쌓이므로 깊이가 깊으면 stack overflow가 발생합니다. 반복은 루프로 같은 로직을 구현하는 방식으로, 스택 부담이 없고 성능이 안정적이지만 코드가 덜 직관적일 수 있습니다. stack overflow는 호출 스택의 크기 제한을 초과할 때 발생하며, 종료 조건이 없거나 입력이 너무 큰 재귀에서 주로 나타납니다.

##### 상세 해설

재귀와 반복은 표현력과 실행 비용 사이의 trade-off다.

- 재귀의 장점
  - 트리, 그래프 순회 같은 구조적 문제에서 코드가 간결하고 직관적
  - 분할 정복, 백트래킹 패턴에 자연스러움
- 재귀의 단점
  - 호출마다 스택 프레임(매개변수, 지역 변수, 복귀 주소)이 쌓임
  - 깊이가 깊으면 스택 메모리 한계를 넘어 stack overflow 발생
  - 함수 호출 자체의 오버헤드 존재
- 반복의 장점
  - 스택 프레임 누적 없음
  - 메모리 사용이 예측 가능하고 안정적
- 반복의 단점
  - 트리 순회 같은 문제에서는 별도 스택 자료구조를 직접 관리해야 함
  - 코드가 길어지고 가독성이 떨어질 수 있음

프론트엔드에서 DOM 트리를 재귀로 순회하거나, 깊게 중첩된 객체를 재귀적으로 처리하는 경우가 있다. 이때 데이터가 예상보다 깊으면 stack overflow가 발생할 수 있으므로, 깊이가 보장되지 않는 경우에는 반복 + 명시적 스택 패턴으로 바꾸는 것이 안전하다. 또한 꼬리 재귀 최적화(Tail Call Optimization)는 일부 엔진에서만 지원하므로, 실무에서는 이에 의존하지 않는 편이 좋다.

#### 6. B-Tree 계열 인덱스와 해시 기반 인덱스는 어떤 차이가 있나요?

기존 DB 질문을 조금 더 깊게 보고 싶을 때 좋다. 범위 검색, 정렬, equality lookup의 차이를 이해하는지 확인할 수 있다.

##### 합격 답변 예시

B-Tree 인덱스는 데이터를 정렬된 트리 구조로 저장하므로 등호 검색뿐 아니라 범위 검색, 정렬, 부분 일치에도 활용할 수 있습니다. 해시 인덱스는 해시 함수로 키를 버킷에 매핑하므로 정확한 값 일치(equality lookup)에는 매우 빠르지만, 범위 검색이나 정렬에는 사용할 수 없습니다. 그래서 대부분의 RDBMS는 B-Tree 계열을 기본 인덱스로 사용하고, 해시 인덱스는 특정 용도에서만 선택적으로 사용합니다.

##### 상세 해설

두 인덱스의 차이는 데이터를 찾는 방식의 근본적인 차이에서 나온다.

- B-Tree 계열 인덱스
  - 데이터를 정렬된 상태로 유지하는 균형 트리
  - 등호(=), 범위(>, <, BETWEEN), 정렬(ORDER BY), 부분 일치(LIKE 'abc%')에 모두 활용 가능
  - 삽입/삭제 시 트리 재균형 비용이 있지만 전반적으로 범용성이 높음
  - MySQL InnoDB, PostgreSQL의 기본 인덱스
- 해시 인덱스
  - 해시 함수로 키를 버킷에 직접 매핑
  - 정확한 값 일치(=) 검색에서 평균 O(1)로 매우 빠름
  - 범위 검색, 정렬, 부분 일치 불가
  - 충돌 관리 필요

프론트엔드 관점에서 보면, 검색이나 필터 API가 느릴 때 DB 인덱스 전략을 이해하면 백엔드 팀과 더 효과적으로 소통할 수 있다. 예를 들어 “가격 범위로 필터링하는 API가 느리다”면 해시 인덱스로는 해결이 안 되고 B-Tree 인덱스가 필요하다는 판단을 함께 내릴 수 있다. 또한 프론트엔드에서 id로 엔티티를 찾을 때 Map(해시 기반)을 쓰는 것과, 정렬된 배열에서 이진 탐색을 하는 것의 차이도 같은 원리다.

### 이 섹션 보완 핵심

CS 섹션은 지금도 충분히 좋다. 여기에 동기화, 작업 유형 구분, 전환 비용, 재귀 실행 모델, 인덱스 구조를 더하면 “배운 적 있는 사람”과 “실무 판단에 연결 가능한 사람”을 더 잘 나눌 수 있다.
