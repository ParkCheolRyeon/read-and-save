# 챕터 8. 웹 성능 최적화에서 Core Web Vitals와 체감 성능까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **성능 최적화는 단순히 번들 크기를 줄이는 작업이 아니라, 사용자가 “얼마나 빨리 보이고, 얼마나 빨리 반응하고, 얼마나 덜 흔들리는가”를 측정하고 개선하는 작업이라는 관점으로 이해해야 한다.**

---

## 99. 성능 최적화는 무엇부터 측정해야 하나

성능 최적화 이야기를 시작하면 많은 사람이 바로 코드 스플리팅이나 이미지 압축부터 떠올린다.  
그런데 실제로는 그 순서가 자주 틀린다.

좋은 성능 최적화의 시작은:

> **무엇이 느린지 먼저 측정하는 것**

이다.

### 핵심 개념

성능 최적화는 보통 아래 순서로 가는 게 좋다.

1. 무엇이 느린지 측정한다
2. 느림이 사용자에게 어떤 체감으로 나타나는지 본다
3. 병목이 네트워크인지, 렌더링인지, JS 실행인지 분해한다
4. 가장 영향 큰 지점부터 개선한다
5. 다시 측정해서 실제로 나아졌는지 확인한다

### 왜 이게 중요한가

이걸 놓치면 흔히 이런 일이 생긴다.

- 번들 줄였는데 체감은 별 차이 없음
- useMemo를 잔뜩 넣었는데 실제 병목은 이미지였음
- lazy loading을 넣었는데 오히려 LCP가 나빠짐
- 숫자는 좋아졌는데 UX는 여전히 답답함

### 자주 하는 오해

- 성능 최적화는 일단 번들부터 줄이면 된다 → 틀림
- 렌더 횟수만 줄이면 된다 → 틀림
- Lighthouse 점수만 오르면 성공이다 → 실제 사용자 체감과 차이가 있을 수 있다

### 실무 연결

성능 최적화는 보통 아래 질문으로 시작하면 좋다.

- 첫 화면이 느린가?
- 클릭 후 반응이 늦는가?
- 화면이 흔들리는가?
- 특정 페이지나 특정 기기에서만 느린가?
- 필드 데이터와 로컬 테스트가 같은가?

### 여기서 다음 개념으로 어떻게 넘어가나

이제 측정부터 해야 한다는 걸 알았다.  
그다음엔 현재 웹 성능에서 가장 중요한 공통 기준인 **Core Web Vitals**로 넘어가야 한다.

### 면접 압축 답변

성능 최적화는 해결책부터 적용하는 것이 아니라, 먼저 무엇이 느린지 측정하고 병목을 분해하는 것부터 시작해야 합니다. 네트워크, 렌더링, 자바스크립트 실행 중 어디가 실제 사용자 체감에 가장 큰 영향을 주는지 확인한 뒤 개선해야 효과가 큽니다.

## 100. Core Web Vitals는 왜 지금 웹 성능의 중심 지표인가

성능 이야기를 할 때 현재 가장 자주 나오는 기준이 Core Web Vitals다.  
이건 단순 점수놀이가 아니라, 실제 사용자 경험의 핵심 축을 세 가지로 압축해 보여준다.

### 핵심 개념

Core Web Vitals는 현재 아래 세 지표로 구성된다.

- **LCP**: 얼마나 빨리 주요 콘텐츠가 보이는가
- **INP**: 사용자 상호작용에 얼마나 빨리 반응하는가
- **CLS**: 화면이 얼마나 덜 흔들리는가

즉:

- 로딩
- 반응성
- 시각적 안정성

을 각각 대표한다고 보면 된다.

### 왜 이게 중요한가

이 세 지표가 중요한 이유는  
“페이지가 빨라 보이는가”를 더 구체적으로 분해하게 해주기 때문이다.

예를 들어:

- 페이지는 열렸는데 주요 콘텐츠가 늦게 보이면 LCP 문제
- 눌렀는데 반응이 굼뜨면 INP 문제
- 다 보였는데 버튼이 밀리면 CLS 문제

### 자주 하는 오해

- Core Web Vitals만 좋으면 모든 성능이 완벽하다 → 틀림
- 점수만 맞추면 된다 → 실제 사용자 흐름과 같이 봐야 한다
- FID가 아직도 핵심이다 → 현재는 INP가 핵심 응답성 지표다

### 실무 연결

좋은 성능 답변은 보통 이렇게 한다.

- 초기 표시 문제 → LCP 관점
- 클릭 반응 문제 → INP 관점
- 레이아웃 밀림 문제 → CLS 관점

즉 “느리다”를 구체 지표로 번역할 수 있어야 한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 Core Web Vitals 큰 그림은 봤다.  
그다음엔 각각의 지표를 하나씩 정확히 이해해야 한다.

먼저 **LCP**로 넘어간다.

### 면접 압축 답변

Core Web Vitals는 현재 웹 성능에서 가장 중요한 공통 사용자 경험 지표로, LCP는 주요 콘텐츠 표시 속도, INP는 상호작용 반응성, CLS는 시각적 안정성을 나타냅니다. 그래서 “느리다”는 말을 실제 사용자 경험 기준으로 더 구체적으로 설명할 수 있게 해줍니다.

## 101. LCP는 왜 “첫 화면이 빠른가”를 가장 잘 드러내는 지표인가

많은 사람이 성능을 말할 때 “페이지 로드가 빠르다”라고 뭉뚱그린다.  
하지만 사용자는 전체 문서가 다 끝날 때까지 기다리지 않는다.  
보통은 **주요 콘텐츠가 보였는가**를 기준으로 빠름을 느낀다.

### 핵심 개념

LCP는 페이지에서 가장 큰 주요 콘텐츠 요소가 화면에 표시되는 시점을 본다.

대체로 LCP 후보가 되는 건:

- 큰 이미지
- 큰 텍스트 블록
- 대표 비주얼 요소

같은 것들이다.

### 왜 이게 중요한가

이 지표가 중요한 이유는  
사용자가 “이제 페이지가 뜨기 시작했다”라고 느끼는 순간과 비교적 잘 맞기 때문이다.

즉:

- 스켈레톤만 빨리 떠도 주요 콘텐츠가 늦으면 LCP는 여전히 나쁠 수 있고
- 작은 텍스트만 먼저 보여도 핵심 이미지가 늦으면 체감은 느릴 수 있다

### 자주 하는 오해

- DOMContentLoaded가 빠르면 첫 화면도 빠르다 → 틀림
- spinner가 빨리 보이면 충분하다 → 핵심 콘텐츠가 중요하다
- LCP는 이미지 사이트에만 중요하다 → 큰 텍스트 블록도 될 수 있다

### 실무 연결

LCP가 나쁠 때 흔히 의심해야 할 건:

- 서버 응답 지연
- HTML에서 LCP 요소 발견이 늦음
- 큰 이미지 다운로드 지연
- 렌더 차단 CSS/JS
- LCP 이미지를 잘못 lazy load한 경우

### 여기서 다음 개념으로 어떻게 넘어가나

이제 LCP가 중요한 이유를 봤다.  
그다음엔 실제로 **무엇이 LCP를 늦추는가**를 더 구체적으로 봐야 한다.

### 면접 압축 답변

LCP는 사용자가 주요 콘텐츠를 얼마나 빨리 보게 되는지를 보여주는 지표입니다. 페이지 전체가 다 끝나는 시점보다, 가장 중요한 큰 콘텐츠가 언제 나타나는지가 첫 화면 체감에 더 직접적이기 때문에 핵심 로딩 지표로 쓰입니다.

## 102. LCP를 개선할 때 가장 먼저 봐야 하는 건 무엇인가

LCP가 나쁘다고 해서 무조건 이미지 포맷부터 바꾸는 건 아니다.  
LCP는 보통 여러 지연이 합쳐진 결과다.

### 핵심 개념

LCP를 늦추는 대표 요인은 아래처럼 볼 수 있다.

- 서버 응답이 늦다
- HTML 파싱 후 LCP 요소 발견이 늦다
- LCP 리소스 요청 시작이 늦다
- LCP 리소스 다운로드가 늦다
- 렌더 차단 리소스 때문에 표시가 늦다

### 왜 이게 중요한가

LCP는 “화면에 큰 요소가 늦게 나타났다”라는 결과일 뿐이라,  
원인은 하나가 아닐 수 있다.

즉 개선도 아래처럼 분기해야 한다.

- 서버가 느리면 TTFB 쪽을 본다
- 리소스 발견이 늦으면 HTML 구조를 본다
- 이미지 로드가 늦으면 preload/fetchpriority/크기 최적화를 본다
- 렌더 차단이 크면 CSS/JS를 본다

### 자주 하는 오해

- LCP는 이미지만 줄이면 해결된다 → 틀림
- skeleton 넣으면 LCP도 좋아진다 → LCP 요소 자체가 빨라져야 한다
- lazy loading은 무조건 성능에 좋다 → LCP 이미지엔 오히려 안 좋을 수 있다

### 실무 연결

특히 중요한 원칙 하나는 이거다.

- **LCP 이미지는 lazy loading하면 안 된다**

왜냐하면 중요한 리소스를 의도적으로 늦게 찾게 만들 수 있기 때문이다.

### 예시

```html
<!-- 좋지 않은 예 -->
<img src="/hero.jpg" loading="lazy" alt="hero" />
```

```html
<!-- 더 나은 방향 -->
<img src="/hero.jpg" fetchpriority="high" alt="hero" />
```

### 여기서 다음 개념으로 어떻게 넘어가나

이제 로딩 성능의 핵심 축 하나를 봤다.  
그다음엔 사용자가 눌렀을 때 얼마나 빨리 반응하는지를 보는 **INP**로 넘어가야 한다.

### 면접 압축 답변

LCP를 개선할 때는 큰 이미지 파일만 줄이는 게 아니라, 서버 응답, LCP 리소스 발견 시점, 다운로드 우선순위, 렌더 차단 리소스까지 함께 봐야 합니다. 특히 LCP 이미지에 lazy loading을 걸면 중요한 리소스 로딩이 늦어질 수 있어 주의해야 합니다.

## 103. INP는 왜 “클릭했는데 답답한 느낌”을 더 잘 설명하나

예전엔 응답성 얘기에서 FID를 많이 들었지만,  
현재는 INP가 더 핵심 지표다.

### 핵심 개념

INP는 사용자의 상호작용에 대해 페이지가 시각적으로 반응하기까지의 지연을 본다.

즉 단순히 “첫 입력이 빨랐는가”가 아니라,  
전체 상호작용 경험에서 **클릭, 탭, 입력 후 화면이 실제로 반응한 시점**에 더 주목한다.

### 왜 이게 중요한가

사용자는 보통 이렇게 느낀다.

- 눌렀는데 아무 일도 안 일어남
- 입력했는데 UI가 한 박자 늦음
- 드롭다운이 눌렀는데 늦게 열림
- 필터 누르면 잠깐 얼어붙음

이건 로딩 문제가 아니라 **응답성 문제**다.  
INP는 바로 이 축을 더 잘 잡아낸다.

### 자주 하는 오해

- 로딩만 빠르면 UX도 좋다 → 틀림
- 첫 입력만 빠르면 충분하다 → 실제 서비스는 여러 상호작용이 있다
- INP는 프레임워크 쓰면 자동으로 좋아진다 → JS 실행량과 메인 스레드 점유가 중요하다

### 실무 연결

INP가 나쁠 때 흔한 원인은:

- 긴 JavaScript 작업
- 무거운 렌더 계산
- 입력 직후 비싼 state 업데이트
- 너무 많은 동기 처리
- 큰 리스트 재계산

### 여기서 다음 개념으로 어떻게 넘어가나

이제 응답성 축을 봤다.  
그다음엔 화면이 흔들리는 경험을 나타내는 **CLS**로 넘어가야 한다.

### 면접 압축 답변

INP는 사용자가 클릭하거나 입력한 뒤 화면이 실제로 반응하기까지의 지연을 보는 지표입니다. 그래서 단순 초기 로딩 속도와 별개로, 눌렀는데 늦게 반응하는 답답한 UX를 설명하는 데 더 적합합니다.

---

## 104. CLS는 왜 “보이긴 하는데 불편한 페이지”를 설명하나

어떤 페이지는 빨리 뜨고, 버튼도 눌린다.  
그런데 읽고 누르려는 순간 요소가 밀리면 체감은 나빠진다.  
이걸 설명하는 대표 지표가 CLS다.

### 핵심 개념

CLS는 예기치 않은 레이아웃 이동이 얼마나 발생했는지를 나타낸다.

즉 사용자가 의도하지 않았는데:

- 버튼이 아래로 밀리거나
- 이미지 로딩 후 본문이 튀거나
- 광고 슬롯이 갑자기 공간을 차지하거나
- 폰트 교체로 줄바꿈이 바뀌는 현상
  을 측정한다.

### 왜 이게 중요한가

CLS가 큰 페이지는 단순히 “못생겨 보이는” 수준이 아니라,
실제 사용 실수를 만들 수 있다.

예:

- 누르려던 버튼 대신 다른 버튼 클릭
- 읽던 문단 위치를 잃음
- 입력 중인 폼 위치가 흔들림

### 자주 하는 오해

- 조금 움직이는 건 상관없다 → 사용 중 움직이면 큰 불편이 된다
- 이미지가 늦게 뜨는 것만 CLS다 → 광고, 폰트, 동적 UI 삽입도 포함될 수 있다
- skeleton이 있으면 CLS는 자동 해결된다 → 공간 예약이 핵심이다

### 실무 연결

CLS를 줄일 때 대표적으로 보는 것:

- 이미지/비디오에 width/height 또는 aspect-ratio 명시
- 광고/배너 슬롯 공간 미리 확보
- 늦게 삽입되는 UI의 공간 전략
- 폰트 로딩 전략 검토

### 여기서 다음 개념으로 어떻게 넘어가나

이제 Core Web Vitals 세 축을 다 봤다.  
그다음엔 이걸 **어떻게 측정하는지**, 즉 필드 데이터와 실험실 데이터로 넘어가야 한다.

### 면접 압축 답변

CLS는 사용자가 예상하지 않은 레이아웃 이동이 얼마나 발생하는지를 나타내는 지표입니다. 페이지가 빨리 떠도 버튼이나 텍스트 위치가 밀리면 실제 사용성이 크게 나빠질 수 있기 때문에, 시각적 안정성을 따로 관리해야 합니다.

---

## 105. 필드 데이터와 실험실 데이터는 왜 둘 다 봐야 하나

성능 측정 도구를 보다 보면  
어떤 곳은 실제 사용자 데이터처럼 보이고, 어떤 곳은 로컬 테스트처럼 보인다.  
이 둘을 구분해야 성능 분석이 덜 헷갈린다.

### 핵심 개념

#### 필드 데이터

실제 사용자 환경에서 수집된 데이터

#### 실험실 데이터

통제된 환경에서 테스트한 데이터

### 왜 이게 중요한가

둘은 각자 장단점이 다르다.

필드 데이터는:

- 실제 사용자 환경을 반영하고
- 기기, 네트워크, 지역 차이를 포함한다

실험실 데이터는:

- 재현이 쉽고
- 디버깅이 편하며
- 변경 전후 비교가 쉽다

### 자주 하는 오해

- Lighthouse만 보면 충분하다 → 실제 사용자 환경과 다를 수 있다
- 실제 데이터만 보면 된다 → 디버깅과 재현이 어렵다
- 둘 중 하나가 더 진짜다 → 목적이 다르다

### 실무 연결

좋은 분석 흐름은 보통 이렇다.

1. 필드 데이터에서 실제 문제를 본다
2. 로컬/DevTools/Lighthouse에서 재현한다
3. 원인을 찾아 수정한다
4. 다시 필드 데이터로 개선 여부를 확인한다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 측정 체계를 봤다.  
그다음엔 실제로 프론트엔드가 가장 자주 만지는 성능 축, 즉 **번들 크기와 코드 스플리팅**으로 넘어간다.

### 면접 압축 답변

필드 데이터는 실제 사용자 환경을 반영하고, 실험실 데이터는 재현과 디버깅에 유리합니다. 그래서 성능 분석은 보통 실제 사용자 데이터를 보고 문제를 찾은 뒤, 로컬 실험 환경에서 원인을 재현하고 수정하는 방식으로 함께 보는 것이 좋습니다.

---

## 106. 번들 크기는 왜 여전히 중요한가

요즘은 네트워크도 빨라지고 브라우저도 좋아졌지만,  
번들 크기는 여전히 중요하다.  
이유는 단순 다운로드 때문만이 아니다.

### 핵심 개념

큰 번들은 아래 비용을 함께 만든다.

- 다운로드 비용
- 파싱 비용
- 실행 비용
- hydration 비용 증가
- 메인 스레드 점유 증가

### 왜 이게 중요한가

특히 모바일/저사양 기기에선  
“다운로드는 빨랐는데 실행이 느린” 경우가 많다.

즉 번들 문제는:

- 로딩 문제
- INP 문제
- 초기 상호작용 지연
  으로 이어질 수 있다.

### 자주 하는 오해

- 네트워크만 빠르면 번들 크기는 상관없다 → 파싱/실행 비용도 크다
- gzip/brotli가 있으니 문제 없다 → 압축 후 전송과 실행 비용은 다르다
- 프레임워크가 알아서 최적화해준다 → 앱 코드와 서드파티 의존성 영향이 크다

### 실무 연결

번들 크기가 커지는 흔한 원인:

- 큰 유틸 라이브러리 전체 import
- 필요 없는 polyfill
- 무거운 차트/에디터 라이브러리
- 페이지 전역에 불필요한 클라이언트 코드 포함
- `'use client'` 경계 과도 확장

### 여기서 다음 개념으로 어떻게 넘어가나

이제 번들 자체의 비용을 봤다.  
그다음엔 이걸 한 번에 다 보내지 않도록 나누는 **코드 스플리팅**으로 넘어가야 한다.

### 면접 압축 답변

번들 크기는 다운로드뿐 아니라 파싱, 실행, hydration 비용에도 영향을 주기 때문에 여전히 중요합니다. 특히 저사양 기기나 무거운 인터랙션 화면에서는 번들 증가가 곧 초기 반응성과 INP 저하로 이어질 수 있습니다.

---

## 107. 코드 스플리팅은 왜 “덜 보내기” 이상의 의미가 있나

코드 스플리팅을 단순히 “파일 쪼개기”로 이해하면 반만 맞다.  
핵심은 **현재 꼭 필요한 코드만 먼저 보내고, 나머지는 늦추는 것**이다.

### 핵심 개념

코드 스플리팅은 자바스크립트를 의미 단위로 나눠서  
초기 로드에 꼭 필요하지 않은 코드를 뒤로 미루는 전략이다.

### 왜 이게 중요한가

이걸 통해 줄일 수 있는 건:

- 초기 다운로드량
- 초기 파싱/실행량
- 초기 hydration 부담
  이다.

즉 단순 전송량 감소를 넘어서,
**초기 상호작용 가능 시점**에도 영향을 준다.

### 자주 하는 오해

- 페이지를 나누면 자동으로 충분하다 → 컴포넌트/기능 단위 분리가 더 필요할 수 있다
- 동적 import만 쓰면 끝이다 → 실제로 언제 로드되는지 봐야 한다
- 작게 쪼갤수록 무조건 좋다 → 요청 수와 복잡도도 같이 본다

### 예시

```tsx
import dynamic from "next/dynamic";

const HeavyChart = dynamic(() => import("./HeavyChart"), {
  ssr: false,
  loading: () => <p>차트 불러오는 중...</p>,
});
```

이런 방식은 무거운 차트 컴포넌트를  
초기 핵심 UI와 분리할 때 유용할 수 있다.

### 실무 연결

코드 스플리팅이 잘 맞는 대상:

- 차트
- WYSIWYG 에디터
- 지도
- 관리자 전용 기능
- rarely used modal
- 대형 드래그 앤 드롭 모듈

### 여기서 다음 개념으로 어떻게 넘어가나

이제 JS 코드 양을 나누는 전략을 봤다.  
그다음엔 실제 초기 화면에서 가장 큰 용량을 차지하는 경우가 많은 **이미지 최적화**로 넘어가야 한다.

### 면접 압축 답변

코드 스플리팅은 자바스크립트를 단순히 쪼개는 것이 아니라, 초기 화면에 꼭 필요하지 않은 코드를 뒤로 미뤄 첫 렌더와 초기 상호작용 부담을 줄이는 전략입니다. 그래서 전송량뿐 아니라 파싱과 실행, hydration 비용까지 함께 줄일 수 있습니다.

---

## 108. 이미지 최적화는 왜 LCP와 가장 자주 연결되나

대부분의 실제 서비스에서 첫 화면의 가장 큰 요소는 이미지인 경우가 많다.  
그래서 이미지 최적화는 성능 얘기에서 거의 빠지지 않는다.

### 핵심 개념

이미지 최적화는 단순히 파일 용량만 줄이는 게 아니다.

- 적절한 크기 제공
- 적절한 포맷 사용
- 필요한 해상도만 전달
- 우선순위 조정
- 레이아웃 공간 예약
  이 함께 가야 한다.

### 왜 이게 중요한가

이미지는:

- 다운로드 비용이 크고
- LCP 후보가 되기 쉽고
- 크기 미지정 시 CLS까지 만들 수 있다

즉 한 자산이:

- 로딩
- 시각 안정성
  두 문제를 동시에 만들 수 있다.

### 자주 하는 오해

- 화질만 조금 줄이면 끝이다 → 크기, 우선순위, 반응형 제공이 더 중요할 수 있다
- lazy loading은 이미지엔 무조건 좋다 → LCP 이미지엔 부적절할 수 있다
- width/height는 예전 방식이다 → CLS 방지에 여전히 중요하다

### 실무 연결

이미지 최적화에서 자주 보는 것:

- 반응형 이미지
- modern format
- 적절한 압축
- width/height 또는 aspect-ratio 지정
- hero 이미지 우선순위 설정

### 여기서 다음 개념으로 어떻게 넘어가나

이제 이미지 최적화의 이유를 봤다.  
그다음엔 이를 실제로 늦게 불러오는 전략인 **lazy loading**을 언제 써야 하고 언제 피해야 하는지로 넘어가야 한다.

### 면접 압축 답변

이미지 최적화는 파일 용량을 줄이는 것만이 아니라, 적절한 크기와 포맷, 우선순위, 레이아웃 공간 예약을 함께 설계하는 것입니다. 특히 이미지는 LCP 후보가 되기 쉽고 CLS까지 유발할 수 있어서 성능 최적화의 핵심 자산입니다.

---

## 109. Lazy loading은 언제 유용하고, 언제 오히려 해가 되나

lazy loading은 매우 유용한 도구지만,  
무조건 붙이는 습관은 오히려 성능을 해칠 수 있다.

### 핵심 개념

lazy loading은  
현재 화면에 꼭 필요하지 않은 리소스를 나중에 로드하게 하는 전략이다.

즉 “지금 안 보여도 되는 것”을 늦추는 데 적합하다.

### 왜 이게 중요한가

이 전략은:

- 초기 다운로드량 감소
- 초기 렌더 부담 감소
- 스크롤 아래 리소스 지연 로딩
  에 도움을 줄 수 있다.

하지만 반대로,  
초기 화면의 핵심 요소에 lazy loading을 걸면:

- 요청 발견이 늦어지고
- LCP가 악화될 수 있다

### 자주 하는 오해

- lazy loading은 무조건 성능 최적화다 → 틀림
- 모든 이미지에 걸면 좋다 → 틀림
- 코드도 이미지도 다 늦출수록 좋다 → 사용자 목적과 첫 화면 중요도를 봐야 한다

### 실무 연결

보통 잘 맞는 경우:

- fold 아래 이미지
- 늦게 열리는 모달 내부 컴포넌트
- 후기 탭, 차트 탭, 관리자용 패널
- 긴 페이지 하단 섹션

반대로 조심할 경우:

- hero image
- 첫 화면 CTA와 직결된 핵심 콘텐츠
- 초기 상호작용에 꼭 필요한 코드

### 여기서 다음 개념으로 어떻게 넘어가나

이제 리소스를 늦추는 전략을 봤다.  
그다음엔 반대로 **초기에 로딩을 막는 요소**, 즉 렌더 차단 리소스로 넘어가야 한다.

### 면접 압축 답변

Lazy loading은 현재 화면에 꼭 필요하지 않은 리소스를 나중에 불러오는 전략이라 초기 로드 최적화에 유용합니다. 하지만 첫 화면 핵심 리소스, 특히 LCP 이미지에 적용하면 오히려 요청 시작이 늦어져 성능이 나빠질 수 있습니다.

---

## 110. 렌더 차단 리소스는 왜 첫 화면을 늦추나

사용자는 HTML을 받았다고 바로 화면을 다 보는 게 아니다.  
브라우저는 그 뒤에도 CSS와 JS의 영향을 받는다.

### 핵심 개념

렌더 차단 리소스는  
브라우저가 중요한 화면 표시를 시작하기 전에 기다리게 만드는 자원이다.

대표적으로:

- 중요한 CSS
- 동기 스크립트
  같은 것들이 영향을 줄 수 있다.

### 왜 이게 중요한가

이걸 이해해야:

- HTML은 빨리 왔는데 화면이 늦게 보이는 이유
- CSS 크기와 순서가 왜 중요한지
- 스크립트 로딩 전략이 왜 첫 화면 체감과 연결되는지
  설명할 수 있다.

### 자주 하는 오해

- HTML만 빨리 보내면 된다 → CSS/JS가 화면 표시를 늦출 수 있다
- 스크립트는 다 끝에 두면 된다 → 실행 시점과 의존성을 같이 봐야 한다
- CSS는 어차피 작아서 문제 없다 → 크리티컬 CSS 여부가 중요할 수 있다

### 실무 연결

대표 전략:

- 중요한 CSS 우선
- 덜 중요한 JS 지연
- 서드파티 스크립트 최소화
- 초기 화면에 불필요한 스타일/스크립트 분리

### 여기서 다음 개념으로 어떻게 넘어가나

이제 첫 화면을 늦추는 정적 자원 문제를 봤다.  
그다음엔 JS가 실제 실행될 때 메인 스레드를 얼마나 오래 점유하는지로 넘어가야 한다.

### 면접 압축 답변

렌더 차단 리소스는 브라우저가 실제 화면 표시를 시작하기 전에 기다리게 만드는 자원입니다. 그래서 HTML 응답이 빨라도 CSS와 동기 스크립트가 무거우면 첫 화면 표시와 LCP가 늦어질 수 있습니다.

---

## 111. 메인 스레드 점유가 왜 INP와 연결되나

성능 얘기에서 자주 놓치는 지점이 있다.  
다운로드가 끝났다고 상호작용이 곧바로 빨라지는 건 아니다.

### 핵심 개념

브라우저 메인 스레드가 긴 작업에 묶이면:

- 클릭 처리
- 입력 반응
- 렌더 업데이트
- 페인트
  가 지연될 수 있다.

이건 곧 INP와 연결된다.

### 왜 이게 중요한가

사용자는 “다운로드 끝났냐”보다  
“눌렀을 때 반응하냐”를 더 민감하게 느낀다.

즉 메인 스레드를 오래 점유하는 작업은:

- 로딩 이후에도 UX를 나쁘게 만들 수 있다

### 흔한 원인

- 큰 JSON 파싱
- 무거운 계산
- 대형 리스트 렌더링
- 차트 초기화
- 드래그 중 과도한 state 업데이트
- 많은 컴포넌트 동시 재계산

### 자주 하는 오해

- 네트워크만 빠르면 반응성도 좋다 → 틀림
- React 쓰면 UI 멈춤은 알아서 해결된다 → 메인 스레드 점유는 여전히 중요하다
- 렌더가 많으면 무조건 문제다 → 무엇이 얼마나 오래 막느냐가 더 중요하다

### 실무 연결

INP를 개선하려면:

- 긴 작업 쪼개기
- 덜 중요한 연산 뒤로 미루기
- 렌더 범위 줄이기
- 무거운 라이브러리 초기화 시점 조정
  을 같이 고려해야 한다.

### 여기서 다음 개념으로 어떻게 넘어가나

이제 메인 스레드와 반응성 연결을 봤다.  
그다음엔 렌더링 비용이 큰 대표 사례인 **긴 리스트와 무거운 UI**로 넘어가야 한다.

### 면접 압축 답변

메인 스레드가 긴 작업에 점유되면 클릭과 입력 처리, 화면 업데이트가 지연되어 INP가 나빠질 수 있습니다. 그래서 네트워크 최적화와 별개로, 긴 자바스크립트 작업과 과도한 렌더 계산을 줄이는 것이 응답성 개선에 중요합니다.

---

## 112. 긴 리스트와 무거운 UI는 왜 별도 전략이 필요한가

실제 서비스에서 성능 병목은 자주 특정 컴포넌트 패턴에 집중된다.  
그 대표가 긴 리스트, 차트, 복잡한 편집기, DnD UI다.

### 핵심 개념

이런 화면들은 단순 페이지보다 아래 비용이 크다.

- 렌더할 항목 수가 많다
- 계산량이 많다
- 이벤트가 빈번하다
- 상태 업데이트가 촘촘하다

### 왜 이게 중요한가

이런 UI를 일반 화면처럼 다루면:

- 초기 렌더가 무겁고
- 스크롤이 끊기고
- 클릭 반응이 느리고
- 작은 state 변화에도 전체가 다시 계산될 수 있다

### 실무 연결

대표 전략:

- virtualization
- 메모이제이션의 신중한 사용
- 파생 계산 최소화
- 상태 범위 축소
- 일시 상태와 저장 상태 분리
- 비싼 라이브러리 lazy load

### 자주 하는 오해

- 리스트는 map만 잘 쓰면 된다 → 길이가 길면 별도 전략이 필요할 수 있다
- memo만 걸면 끝이다 → 데이터 구조와 렌더 범위가 더 중요할 수 있다
- 성능 문제는 라이브러리 탓이다 → 상태 설계와 업데이트 범위 문제일 때가 많다

### 여기서 다음 개념으로 어떻게 넘어가나

이제 수치 최적화 쪽을 많이 봤다.  
그다음엔 사용자가 느끼는 성능, 즉 **체감 성능과 fallback UI**로 넘어가야 한다.

### 면접 압축 답변

긴 리스트나 차트, DnD 같은 무거운 UI는 렌더 수보다 항목 수와 계산량, 업데이트 범위가 문제인 경우가 많습니다. 그래서 virtualization, 상태 범위 축소, 무거운 모듈 지연 로딩 같은 별도 전략이 필요합니다.

---

## 113. 체감 성능과 fallback UI는 왜 숫자 최적화와 따로 볼 수 없나

성능 지표를 좋아지게 만드는 것과, 사용자가 덜 답답하게 느끼게 만드는 것은 겹치면서도 완전히 같진 않다.  
그래서 **체감 성능**이라는 관점을 같이 봐야 한다.

### 핵심 개념

체감 성능은 사용자가:

- 지금 뭔가 진행 중인지
- 언제 결과가 나올지
- 현재 조작이 먹었는지
  를 얼마나 빨리 이해하느냐와 연결된다.

### 왜 이게 중요한가

같은 2초 대기라도:

- 아무 변화가 없으면 멈춘 것처럼 느껴지고
- 스켈레톤, progress, optimistic UI, 버튼 상태 변화가 있으면 덜 답답하게 느낄 수 있다

즉 UX는 숫자만으로 끝나지 않는다.

### 자주 하는 오해

- spinner만 있으면 체감 성능도 좋아진다 → 지나치게 오래 도는 spinner는 오히려 불안감을 준다
- skeleton이 있으면 LCP도 좋아진다 → 지표와 체감은 다를 수 있다
- optimistic UI는 성능 최적화가 아니다 → 사용자 체감엔 큰 영향을 준다

### 실무 연결

대표 전략:

- skeleton
- progressive disclosure
- optimistic update
- 버튼 pending 상태
- 부분 로딩
- 실패 시 즉시 복구 가능 UI

### 여기서 다음 개념으로 어떻게 넘어가나

이제 성능 최적화의 마지막 축까지 봤다.  
그다음엔 이 챕터 전체를 하나의 설계 감각으로 정리해야 한다.

### 면접 압축 답변

체감 성능은 사용자가 현재 작업이 진행 중이고 조작이 반영됐다고 느끼는 속도와 연결됩니다. 그래서 숫자 최적화와 별개로 skeleton, pending 상태, optimistic UI 같은 피드백 설계가 함께 필요합니다.

---

## 114. 이 챕터를 읽고 나면 무엇이 연결되어야 하나

이 챕터를 다 읽고 나면 아래 흐름이 한 덩어리로 보여야 한다.

- 성능 최적화는 측정부터 시작한다
- Core Web Vitals는 로딩, 반응성, 안정성을 대표한다
- LCP는 주요 콘텐츠 표시 속도다
- INP는 상호작용 반응성이다
- CLS는 시각적 안정성이다
- 필드 데이터와 실험실 데이터는 함께 봐야 한다
- 번들 크기는 다운로드뿐 아니라 파싱/실행 비용과 연결된다
- 코드 스플리팅은 초기 필요 코드만 먼저 보내기 위한 전략이다
- 이미지 최적화는 LCP와 CLS에 동시에 영향을 준다
- lazy loading은 유용하지만 핵심 리소스엔 독이 될 수 있다
- 렌더 차단 리소스와 메인 스레드 점유는 첫 화면과 응답성을 늦춘다
- 긴 리스트와 무거운 UI는 별도 전략이 필요하다
- 체감 성능은 숫자 최적화와 함께 설계해야 한다

즉,  
이번 챕터는 단순 팁 모음이 아니라,  
**“무엇을 측정하고, 어떤 병목을 어떤 사용자 경험 문제로 해석하며, 어떤 전략으로 개선할 것인가”**를 배우는 챕터다.

# 사용법

이 챕터는 아래 순서로 보면 좋다.

1. 측정부터 해야 한다는 원칙을 먼저 잡는다.
2. Core Web Vitals로 느림을 사용자 경험 기준으로 분해한다.
3. LCP, INP, CLS를 각각 병목 유형과 연결한다.
4. 번들, 코드 스플리팅, 이미지, lazy loading으로 구현 전략을 본다.
5. 마지막에 체감 성능과 fallback UI까지 묶어 UX 관점으로 정리한다.

이렇게 보면 성능 최적화가 단순 숫자 개선이 아니라, 실제 사용자가 느끼는 빠름과 반응성, 안정성을 설계하는 작업이라는 감각이 더 선명해진다.
