# 챕터 8. 웹 성능 최적화에서 Core Web Vitals와 체감 성능까지

이 챕터의 목표는 아래 한 줄로 요약된다.

> **성능 최적화는 단순히 번들 크기를 줄이는 작업이 아니라, 사용자가 "얼마나 빨리 보이고, 얼마나 빨리 반응하고, 얼마나 덜 흔들리는가"를 측정하고 개선하는 작업이다.**

### 전체 흐름 맵

```
측정부터 시작한다 (무엇이 느린가?)
  → Core Web Vitals (LCP / INP / CLS)
    → 필드 데이터 + 실험실 데이터 (**RUM(Real User Monitoring, 실제 사용자 모니터링 — 실제 방문자 브라우저에서 수집하는 성능 데이터)** + Lighthouse)
      → 브라우저 렌더링 파이프라인 (DOM → Layout → Paint → Composite)
        → 번들 크기 + 코드 스플리팅 (JS 최적화)
          → 이미지 최적화 + Lazy Loading (리소스 최적화)
            → 렌더 차단 리소스 + 메인 스레드 점유 (실행 최적화)
              → 긴 리스트 / Observer API (무거운 UI 전략)
                → 체감 성능 + fallback UI (UX 설계)
```

## 67. 측정부터 시작한다 — Core Web Vitals

> **핵심 요약** — 성능 최적화는 해결책부터가 아니라 측정부터. Core Web Vitals는 LCP(로딩), INP(반응성), CLS(안정성)로 "느리다"를 구체 지표로 분해한다. 필드 데이터와 실험실 데이터를 함께 봐야 한다.

### 최적화의 올바른 순서

1. **무엇이 느린지 측정한다** (Lighthouse, DevTools, web-vitals)
2. 느림이 사용자에게 어떤 체감으로 나타나는지 본다
3. 병목이 네트워크인지, 렌더링인지, JS 실행인지 분해한다
4. 가장 영향 큰 지점부터 개선한다
5. 다시 측정해서 실제로 나아졌는지 확인한다

### Core Web Vitals 3대 지표

| 지표 | 측정 대상 | 기준 (Good) | "느리다"의 번역 |
|---|---|---|---|
| **LCP** | 주요 콘텐츠 표시 속도 | 2.5초 이하 | "페이지가 안 떠요" |
| **INP** | 상호작용 반응성 | 200ms 이하 | "눌렀는데 반응이 없어요" |
| **CLS** | 시각적 안정성 | 0.1 이하 | "버튼이 밀려서 다른 거 눌렀어요" |

### Core Web Vitals — 페이지 로드 타임라인

```
페이지 로드 시간 →
│
│  [요청]─[TTFB]─[첫 번째 바이트]─[HTML 파싱]─[리소스 로딩]─[렌더링]
│                                                │
│         ┌── LCP (Largest Contentful Paint) ────┘
│         │   "주요 콘텐츠가 보이는 시점" (2.5초 이내 = 좋음)
│         │
│  ──────────── 페이지 사용 중 ─────────────────────────→
│         │         │              │
│         │    ┌── INP ──┐    ┌── INP ──┐
│         │    │클릭/입력 │    │클릭/입력 │
│         │    │→ 반응    │    │→ 반응    │
│         │    └─────────┘    └─────────┘
│         │    "상호작용 반응성" (200ms 이내 = 좋음)
│         │
│    ─── CLS 누적 ──────────────────────────────→
│         레이아웃이 갑자기 밀리는 정도 (0.1 이내 = 좋음)
```

### 필드 데이터 vs 실험실 데이터

| | 필드 데이터 (RUM) | 실험실 데이터 (Lab) |
|---|---|---|
| **수집 환경** | 실제 사용자 브라우저 | 통제된 테스트 환경 |
| **대표 도구** | CrUX, PageSpeed Insights, web-vitals | Lighthouse, DevTools, WebPageTest |
| **장점** | 실제 기기/네트워크 반영 | 재현 쉬움, 디버깅 편함 |
| **단점** | 디버깅 어려움 | 실제 환경과 다를 수 있음 |

```tsx
// web-vitals 라이브러리로 필드 데이터 수집
import { onLCP, onINP, onCLS } from "web-vitals";

onLCP((metric) => sendToAnalytics({ name: "LCP", value: metric.value }));
onINP((metric) => sendToAnalytics({ name: "INP", value: metric.value }));
onCLS((metric) => sendToAnalytics({ name: "CLS", value: metric.value }));
```

### 실무 에피소드 — 이런 상황에서 만난다

> useMemo를 모든 컴포넌트에 넣었는데 체감 속도가 안 변함. Lighthouse로 분석하니 실제 병목은 히어로 이미지(2MB PNG)의 LCP 지연이었다. 이미지를 WebP로 바꾸고 `fetchpriority="high"`를 설정하니 LCP가 4초 → 1.8초로 개선. **측정 없이 감으로 최적화하면 엉뚱한 곳에 시간을 쓴다.**

### 자주 하는 오해

- 번들부터 줄이면 된다 → **측정 없이 시작하면 병목을 놓친다.**
- Lighthouse 점수만 오르면 성공이다 → **실제 사용자 체감(필드 데이터)과 다를 수 있다.**
- FID가 아직 핵심이다 → **2024년 3월부터 INP가 FID를 대체했다.**

### 면접 압축 답변

- **10초**: Core Web Vitals는 LCP(로딩), INP(반응성), CLS(안정성)로 사용자 경험을 측정하며, 성능 최적화는 측정부터 시작해야 합니다.
- **30초**: 성능 최적화는 측정 → 분해 → 개선 → 재측정 순서로 합니다. Core Web Vitals의 LCP(2.5초 이하), INP(200ms 이하), CLS(0.1 이하)가 핵심 기준이며, 필드 데이터(CrUX, web-vitals)로 실제 문제를 발견하고 실험실 데이터(Lighthouse, DevTools)로 디버깅합니다.

## 68. LCP — 주요 콘텐츠 표시 속도

> **핵심 요약** — LCP는 뷰포트에서 가장 큰 콘텐츠가 렌더링 완료되는 시점. 서버 응답(TTFB), 리소스 발견 시점, 다운로드 우선순위, 렌더 차단이 복합 원인. LCP 이미지에 lazy loading을 걸면 오히려 악화.

### LCP를 늦추는 원인과 개선

| 원인 | 개선 방향 |
|---|---|
| 서버 응답 지연 (TTFB) | 서버 최적화, CDN, 캐시 |
| LCP 리소스 발견이 늦음 | HTML 구조 개선, `<link rel="preload">` |
| LCP 리소스 다운로드 지연 | 이미지 최적화, `fetchpriority="high"` |
| 렌더 차단 CSS/JS | Critical CSS, `async`/`defer` |
| LCP 이미지를 잘못 lazy load | lazy 제거, priority 설정 |

```html
<!-- 나쁜 예: LCP 이미지에 lazy loading -->
<img src="/hero.jpg" loading="lazy" alt="hero" />

<!-- 좋은 예: LCP 이미지에 높은 우선순위 -->
<img src="/hero.jpg" fetchpriority="high" alt="hero" />
<link rel="preload" as="image" href="/hero.jpg" />
```

```tsx
// Next.js에서 LCP 이미지
<Image src="/hero.jpg" alt="hero" width={1200} height={600} priority />
```

### 실무 에피소드 — 이런 상황에서 만난다

> 랜딩 페이지 LCP가 4초인데 원인을 못 찾겠다는 의뢰. DevTools Performance 패널에서 LCP 마커를 확인하니 히어로 이미지가 CSS `background-image`로 들어가 있어서 브라우저가 CSS 파싱 후에야 이미지 요청을 시작함. `<img>` 태그로 바꾸고 `preload` 힌트를 추가하니 LCP 2.1초로 개선.

### 면접 압축 답변

- **10초**: LCP는 뷰포트에서 가장 큰 콘텐츠의 렌더링 완료 시점이며, TTFB, 리소스 우선순위, 렌더 차단이 복합 원인입니다.
- **30초**: LCP는 사용자가 "페이지가 떴다"고 느끼는 순간과 가장 잘 맞는 지표입니다. 서버 응답, 리소스 발견 시점, 다운로드 우선순위, 렌더 차단을 함께 봐야 하며, LCP 이미지에 lazy loading을 걸면 오히려 악화됩니다. `fetchpriority="high"`나 `preload`로 우선 로딩하는 것이 핵심입니다.

## 69. INP — 상호작용 반응성

> **핵심 요약** — INP는 페이지 방문 동안 관측된 모든 상호작용 중 가장 긴 지연 시간을 측정한다(50개 이상일 때 상위 이상치 일부 제외). CWV(Core Web Vitals) 평가 시에는 방문자들의 75번째 백분위수(75th percentile) 기준으로 '좋음/개선 필요/나쁨'을 판정한다. FID가 첫 입력만 봤다면 INP는 전체를 본다. Long Task(50ms+)와 메인 스레드 점유가 핵심 원인.

### FID vs INP

| | FID (폐기) | INP (현재) |
|---|---|---|
| 측정 범위 | 첫 번째 입력만 | 모든 상호작용 |
| 측정 구간 | 입력 지연만 | 입력 지연 + 처리 + 렌더 지연 전체 |

### INP가 나쁠 때 흔한 원인과 개선

| 원인 | 개선 |
|---|---|
| 긴 JS 작업 (Long Task) | `setTimeout`/`scheduler.yield()`로 쪼개기 |

> 50ms가 기준인 이유: 60fps 기준 한 프레임이 약 16ms이고, 사용자 입력 처리까지 고려하면 50ms가 '버벅임'을 느끼는 임계점이다.

| 무거운 렌더 계산 | `startTransition`으로 urgent/non-urgent 분리 |

> startTransition은 React 18이 도입한 '급한 업데이트(사용자 입력)와 덜 급한 업데이트(화면 전환)를 분리'하는 메커니즘이다.

| 큰 리스트 재계산 | virtualization, 상태 범위 축소 |
| 무거운 JSON 파싱 | Web Worker로 오프로드 |

```tsx
// 나쁜 예: 클릭 핸들러에서 무거운 동기 작업
function handleClick() {
  const filtered = hugeList.filter(complexFilter).sort(complexSort);
  setItems(filtered);
}

// 개선: startTransition으로 분리
function handleClick() {
  setIsLoading(true);
  startTransition(() => {
    setItems(hugeList.filter(complexFilter).sort(complexSort));
    setIsLoading(false);
  });
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 필터 드롭다운을 누르면 0.5초 동안 UI가 얼어붙는 현상. DevTools Performance에서 Long Task(800ms)가 보임. 원인은 5000개 상품을 동기적으로 필터+정렬+그룹핑. `startTransition`으로 필터 업데이트를 non-urgent로 분리하고, 정렬/그룹핑은 Web Worker로 옮기니 INP 150ms로 개선.

### 면접 압축 답변

- **10초**: INP는 모든 상호작용 중 가장 느린 응답 시간을 측정하며, Long Task와 메인 스레드 점유가 핵심 원인입니다.
- **30초**: INP는 FID와 달리 페이지 전체 수명 동안의 모든 상호작용을 보고 가장 느린 쪽을 대표값으로 잡습니다. 긴 JS 작업을 쪼개거나 `startTransition`으로 우선순위를 분리하고, 무거운 계산은 Web Worker로 오프로드하는 것이 효과적입니다.

## 70. CLS — 시각적 안정성

> **핵심 요약** — CLS는 예기치 않은 레이아웃 이동의 누적 점수. 이미지 크기 미지정, 동적 콘텐츠 삽입, 폰트 교체가 주요 원인. 공간을 미리 예약하는 것이 핵심.

### CLS를 유발하는 원인과 해결

| 원인 | 해결 |
|---|---|
| 이미지 크기 미지정 | `width`/`height` 또는 `aspect-ratio` 명시 |
| 광고/배너 삽입 | `min-height`로 공간 예약 |
| 웹폰트 교체 | `font-display: swap` 또는 `optional` |
| 동적 UI 삽입 | 삽입 위치에 공간 미리 확보 |

```html
<!-- 나쁜 예 -->
<img src="/banner.jpg" alt="banner" />

<!-- 좋은 예: 공간 예약 -->
<img src="/banner.jpg" alt="banner" width="800" height="400" />

<!-- 반응형 대응 -->
<style>.banner { width: 100%; aspect-ratio: 2/1; object-fit: cover; }</style>
```

### 실무 에피소드 — 이런 상황에서 만난다

> "확인" 버튼을 눌렀는데 "삭제" 버튼이 눌리는 민원. 원인은 버튼 위에 늦게 로드되는 광고 배너가 삽입되면서 버튼이 아래로 밀린 것. 광고 슬롯에 `min-height: 250px`로 공간을 미리 확보하니 CLS 0.35 → 0.04로 개선.

### 면접 압축 답변

- **10초**: CLS는 예기치 않은 레이아웃 이동의 누적 점수이며, 이미지 크기 명시와 공간 예약이 핵심 해결책입니다.
- **30초**: CLS는 이미지에 `width`/`height`를 명시하고, 광고 슬롯의 공간을 미리 확보하며, 폰트 로딩 전략을 설정하는 것이 대표적인 개선 방법입니다. 사용자 입력 후 500ms 이내의 이동은 제외되며, skeleton도 실제 콘텐츠와 크기가 일치해야 의미가 있습니다.

## 71. 브라우저 렌더링 파이프라인 — Reflow, Repaint, Composite

> **핵심 요약** — 브라우저는 DOM → CSSOM → 렌더 트리 → Layout(Reflow) → Paint → Composite 순으로 화면을 그린다. Reflow는 비싸고, `transform`/`opacity` 애니메이션은 Composite만 발생해서 저렴하다.

### 파이프라인

```
HTML → DOM 트리
CSS → **CSSOM(CSS Object Model, CSS 객체 모델 — 브라우저가 CSS를 파싱하여 만드는 트리 구조)** 트리
DOM + CSSOM → 렌더 트리
  → Layout (Reflow): 위치/크기 계산 — 비쌈!
    → Paint: 픽셀 채우기
      → Composite: GPU 레이어 합성 — 가장 저렴
```

### Reflow vs Repaint vs Composite

| | Reflow | Repaint | Composite Only |
|---|---|---|---|
| **트리거** | width, height, margin, DOM 추가/삭제 | color, background, visibility | transform, opacity |
| **비용** | 높음 (하위 요소 연쇄 재계산) | 중간 | 낮음 (GPU) |
| **관계** | Reflow → Repaint → Composite | Repaint → Composite | Composite만 |

```css
/* 나쁜 예: top/left → 매 프레임 reflow */
.box { transition: top 0.3s, left 0.3s; }

/* 좋은 예: transform → composite만 */
.box { transition: transform 0.3s; }
/* transform: translateY(100px) 사용 */
```

```js
// 나쁜 예: DOM 읽기/쓰기 번갈아 → 강제 동기 레이아웃
elements.forEach((el) => {
  const h = el.offsetHeight;        // 읽기 → 강제 layout
  el.style.height = h + 10 + "px";  // 쓰기 → layout 무효화
});

// 좋은 예: 읽기 일괄 → 쓰기 일괄
const heights = elements.map((el) => el.offsetHeight);
elements.forEach((el, i) => { el.style.height = heights[i] + 10 + "px"; });
```

### 실무 에피소드 — 이런 상황에서 만난다

> 카드 호버 애니메이션이 저사양 모바일에서 끊김. `margin-top`으로 구현되어 매 프레임 reflow 발생. `transform: translateY()`로 바꾸니 composite만 발생해서 60fps 유지.

### 면접 압축 답변

- **10초**: 브라우저는 Layout → Paint → Composite 순으로 화면을 그리며, `transform`/`opacity` 애니메이션은 composite만 발생해 reflow를 피합니다.
- **30초**: Reflow는 기하학적 속성 변경 시 레이아웃을 다시 계산하므로 비용이 크고, `transform`과 `opacity`는 composite 단계에서만 처리되어 reflow를 피합니다. DOM 읽기/쓰기를 번갈아 하면 강제 동기 레이아웃이 발생하므로, 읽기를 먼저 일괄 처리하고 쓰기를 나중에 일괄 처리하는 패턴이 좋습니다.

## 72. 번들 크기와 코드 스플리팅

> **핵심 요약** — 번들 크기는 다운로드뿐 아니라 파싱/컴파일/실행 비용에 영향. 저사양 기기에서 체감 차이가 크다. 코드 스플리팅으로 초기 필요 코드만 먼저 보내고 나머지는 늦춘다.

### 번들이 큰 게 왜 문제인가

같은 1MB JS를 고성능 데스크톱에서 파싱/실행하면 ~100ms, 저사양 모바일에서는 ~1000ms+. **gzip은 전송 크기만 줄이지 실행 크기는 그대로.**

> gzip과 brotli는 네트워크 전송 시 파일을 압축하는 알고리즘이다. 브라우저가 받은 후 자동으로 압축을 해제한다.

```bash
# 번들 분석
npx webpack-bundle-analyzer stats.json  # webpack
ANALYZE=true next build                   # Next.js
npx vite-bundle-visualizer                # Vite
```

### 코드 스플리팅 — 현재 필요한 코드만 먼저

| 분리 단위 | 방법 | 예시 |
|---|---|---|
| 라우트 | 프레임워크 자동 | Next.js 페이지별 번들 |
| 컴포넌트 | `React.lazy` + Suspense / `next/dynamic` | 차트, 에디터 |
| 기능 | 이벤트 기반 `import()` | PDF 내보내기, 결제 모듈 |

```tsx
// React.lazy + Suspense
const HeavyChart = lazy(() => import("./HeavyChart"));
<Suspense fallback={<p>차트 로딩...</p>}><HeavyChart /></Suspense>

// Next.js dynamic (SSR 제외 옵션)
const HeavyChart = dynamic(() => import("./HeavyChart"), { ssr: false });

// 이벤트 기반 (클릭 시에만 로드)
async function handleExport() {
  const { exportToPDF } = await import("./pdfExporter");
  exportToPDF(data);
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 번들 분석을 돌려보니 `moment.js`가 300KB를 차지. `date-fns`로 교체하고 필요한 함수만 import하니 번들이 280KB 감소. 추가로 차트 라이브러리를 `dynamic(() => import(...), { ssr: false })`로 분리하니 초기 JS가 40% 줄어듦.

### 면접 압축 답변

- **10초**: 번들 크기는 다운로드뿐 아니라 파싱/실행 비용에 영향을 주며, 코드 스플리팅으로 초기 필요 코드만 먼저 보냅니다.
- **30초**: 번들 크기는 특히 저사양 모바일에서 파싱/실행 비용이 크고, gzip은 전송만 줄입니다. 번들 분석 도구로 병목을 찾고, 라우트/컴포넌트/기능 단위로 코드 스플리팅하며, 무거운 라이브러리는 가벼운 대안으로 교체하거나 동적 import로 지연 로딩합니다.

## 73. 이미지 최적화와 Lazy Loading

> **핵심 요약** — 이미지는 LCP 후보이자 CLS 원인. 적절한 크기/포맷/우선순위/공간 예약을 함께 설계한다. Lazy loading은 fold 아래에만 — LCP 이미지에 걸면 악화.

### 이미지 최적화 체크리스트

| 항목 | 방법 |
|---|---|
| 포맷 | WebP/**AVIF(AV1 Image File Format, AV1 이미지 형식)** 사용 |
| 크기 | `srcset`/`sizes`로 기기별 분기 |
| 우선순위 | LCP 이미지에 `fetchpriority="high"` |
| 공간 예약 | `width`/`height` 또는 `aspect-ratio` |
| Lazy loading | fold **아래**에만 `loading="lazy"` |

```html
<!-- 반응형 + modern format -->
<picture>
  <source srcset="/hero.avif" type="image/avif" />
  <source srcset="/hero.webp" type="image/webp" />
  <img src="/hero.jpg" alt="hero" width="1200" height="600" fetchpriority="high" />
</picture>

<!-- fold 아래 이미지만 lazy -->
<img src="/review.jpg" loading="lazy" alt="review" width="400" height="300" />
```

```tsx
// Next.js Image
<Image src="/hero.jpg" alt="hero" width={1200} height={600} priority placeholder="blur" />
```

### Lazy Loading — 언제 유용하고 언제 해로운가

| 유용 | 해로움 |
|---|---|
| fold 아래 이미지 | LCP 이미지 (hero image) |
| 늦게 열리는 모달 내부 | 첫 화면 **CTA(Call to Action, 행동 유도 요소 — 클릭을 유도하는 버튼이나 이미지)** 이미지 |
| 긴 페이지 하단 | 초기 상호작용에 필요한 코드 |

### 실무 에피소드 — 이런 상황에서 만난다

> 모든 이미지에 `loading="lazy"`를 일괄 적용했는데 오히려 LCP가 나빠짐. LCP 후보인 히어로 이미지까지 lazy가 걸려서 뷰포트에 가까워질 때까지 요청이 안 나간 것. 히어로 이미지만 `fetchpriority="high"` + lazy 제거, 나머지는 lazy 유지로 개선.

### 면접 압축 답변

- **10초**: 이미지 최적화는 크기/포맷/우선순위/공간 예약을 함께 설계하며, lazy loading은 LCP 이미지를 제외하고 fold 아래에만 적용합니다.
- **30초**: 이미지는 LCP 후보이자 CLS 원인이므로 `srcset`/`sizes`로 반응형 제공, WebP/AVIF로 포맷 최적화, `width`/`height`로 공간 예약을 합니다. LCP 이미지에는 `fetchpriority="high"`를, fold 아래에만 `loading="lazy"`를 적용합니다. 전체 일괄 lazy는 LCP를 악화시킵니다.

## 74. 렌더 차단 리소스와 메인 스레드 점유

> **핵심 요약** — CSS는 렌더를 차단하고 동기 스크립트는 파싱을 차단한다. `defer`/`async`와 Critical CSS로 첫 화면을 앞당기고, Long Task는 쪼개거나 Web Worker로 오프로드한다.

### 스크립트 로딩 전략

| 속성 | 다운로드 | 실행 시점 | 순서 보장 | 파싱 차단 |
|---|---|---|---|---|
| (없음) | 즉시 | 즉시 | O | O |
| `async` | 병렬 | 다운로드 완료 시 | X | 실행 중만 |
| `defer` | 병렬 | DOMContentLoaded 전 | O | X |

> DOMContentLoaded는 HTML 파싱이 끝났을 때 발생하는 브라우저 이벤트다(이미지/스타일시트 로딩 완료와는 별개).

### script 로딩 전략 비교

```
<script> (기본):
  HTML ──[파싱]──┤정지├──[JS 다운로드]──[JS 실행]──┤재개├──[파싱]──
                 ↑ HTML 파싱이 멈춤!

<script async>:
  HTML ──[파싱]──────────[파싱]──┤정지├──[JS 실행]──┤재개├──[파싱]──
              [JS 다운로드 병렬]──┘
              ↑ 다운로드 후 즉시 실행 (순서 보장 안 됨)

<script defer>:
  HTML ──[파싱]──────────────────────[파싱 완료]──[JS 실행]──
              [JS 다운로드 병렬]──────────────────┘
              ↑ 파싱 끝난 후 순서대로 실행
```

### Critical CSS

```html
<head>
  <!-- 첫 화면 필수 스타일만 인라인 -->
  <style>.hero { display: flex; min-height: 80vh; }</style>
  <!-- 나머지는 비동기 로드 -->
  <link rel="preload" href="/styles/full.css" as="style"
        onload="this.onload=null;this.rel='stylesheet'" />
</head>
```

### 메인 스레드 점유 — INP와의 연결

메인 스레드가 50ms+ Long Task에 묶이면 클릭/입력 처리가 지연 → INP 악화.

```tsx
// Web Worker로 무거운 계산 오프로드
const worker = new Worker(new URL("./worker.ts", import.meta.url));
worker.postMessage({ rawData });
worker.onmessage = (e) => setData(e.data);

// scheduler.yield()로 메인 스레드에 양보
async function processInChunks(items) {
  for (let i = 0; i < items.length; i += 100) {
    processChunk(items.slice(i, i + 100));
    if ("scheduler" in globalThis) await scheduler.yield();
    else await new Promise((r) => setTimeout(r, 0));
  }
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> HTML은 200ms에 도착하는데 화면이 2초 동안 비어있음. 원인은 `<head>`에 800KB CSS와 동기 `<script>` 3개가 있어서 파싱이 멈춤. Critical CSS만 인라인하고 나머지는 `defer`로 바꾸니 첫 화면이 800ms에 표시.

### 면접 압축 답변

- **10초**: CSS는 렌더를 차단하고 동기 스크립트는 파싱을 차단합니다. `defer`/Critical CSS로 첫 화면을 앞당기고, Long Task는 Web Worker로 오프로드합니다.
- **30초**: 렌더 차단 리소스(CSS, 동기 스크립트)와 메인 스레드 Long Task는 각각 첫 화면과 응답성을 악화시킵니다. `defer`는 파싱과 병렬로 다운로드하고 파싱 완료 후 순서대로 실행하며, Critical CSS를 인라인하고 나머지를 비동기 로드합니다. 50ms+ Long Task는 `scheduler.yield()`나 Web Worker로 쪼개야 INP가 개선됩니다.

## 75. 긴 리스트, Observer API, 무거운 UI 전략

> **핵심 요약** — 긴 리스트는 virtualization으로 보이는 영역만 렌더. IntersectionObserver/ResizeObserver는 scroll/resize 이벤트보다 성능이 좋다. 무거운 UI는 상태 범위 축소 + lazy load가 핵심.

### Virtualization — 보이는 영역만 렌더

```tsx
// 10,000개 중 화면에 보이는 ~15개만 DOM에 존재
import { FixedSizeList } from "react-window";

<FixedSizeList height={600} itemCount={items.length} itemSize={40} width="100%">
  {({ index, style }) => <div style={style}>{items[index].name}</div>}
</FixedSizeList>
```

### IntersectionObserver vs ResizeObserver

| | IntersectionObserver | ResizeObserver |
|---|---|---|
| **감지 대상** | 뷰포트 진입/이탈 | 요소 크기 변화 |
| **용도** | lazy loading, 무한 스크롤, 노출 분석 | 반응형 차트, 동적 레이아웃 |
| **장점** | scroll 이벤트 대체, 강제 reflow 없음 | resize 이벤트 대체, 개별 요소 감지 |

```tsx
// IntersectionObserver로 무한 스크롤
useEffect(() => {
  const observer = new IntersectionObserver(([entry]) => {
    if (entry.isIntersecting) loadMore();
  });
  if (sentinelRef.current) observer.observe(sentinelRef.current);
  return () => observer.disconnect();
}, [loadMore]);
```

### scroll/resize 이벤트보다 나은 이유

| | scroll/resize 이벤트 | Observer API |
|---|---|---|
| 실행 빈도 | 초당 수십~수백 회 | 필요할 때만 |
| 강제 reflow | `getBoundingClientRect()` 호출 시 발생 | 없음 |
| 코드 복잡도 | throttle + 수동 계산 | 선언적 |

### 실무 에피소드 — 이런 상황에서 만난다

> 상품 목록 500개를 전부 렌더하니 초기 로딩 3초, 스크롤 시 버벅임. `react-window`로 virtualization 적용하니 DOM 노드가 500개 → 20개로 줄어들고 초기 로딩 0.3초, 스크롤 60fps. 무한 스크롤은 IntersectionObserver로 하단 센티널 감지.

### 면접 압축 답변

- **10초**: 긴 리스트는 virtualization, 뷰포트 감지는 IntersectionObserver, 크기 감지는 ResizeObserver가 scroll/resize 이벤트보다 성능이 좋습니다.
- **30초**: 수백 개 이상의 리스트는 `react-window` 같은 라이브러리로 보이는 영역만 렌더하고, IntersectionObserver로 무한 스크롤과 lazy loading을, ResizeObserver로 반응형 컴포넌트를 구현합니다. 이 API들은 scroll/resize 이벤트와 달리 강제 reflow를 발생시키지 않아 성능이 좋습니다.

## 76. 체감 성능과 Fallback UI

> **핵심 요약** — 같은 대기 시간이라도 skeleton, optimistic UI, pending 상태가 있으면 덜 답답하다. 숫자 최적화와 UX 피드백을 함께 설계해야 한다. 300ms 이내 응답이면 로딩 표시를 생략하는 것도 방법.

### 체감 성능 전략

| 전략 | 효과 | 주의 |
|---|---|---|
| **Skeleton UI** | 레이아웃 힌트로 대기감 감소 | 실제 콘텐츠 크기와 일치해야 CLS 방지 |
| **Optimistic Update** | 서버 응답 전 UI 먼저 반영 | 실패 시 롤백 로직 필수 |
| **Pending 상태** | 클릭 즉시 피드백 | 300ms 이내면 생략이 더 자연스러울 수 있음 |
| **Progressive disclosure** | 중요한 콘텐츠부터 단계적 표시 | Suspense boundary 설계 필요 |

```tsx
// Optimistic Update: 서버 응답 전 UI 먼저 반영
async function handleLike() {
  setIsLiked(true);        // 즉시 UI 반영
  setCount((c) => c + 1);
  try {
    await api.likePost(postId);
  } catch {
    setIsLiked(false);     // 실패 시 롤백
    setCount((c) => c - 1);
  }
}
```

### 실무 에피소드 — 이런 상황에서 만난다

> 좋아요 버튼을 누르면 서버 응답까지 1초 동안 아무 반응이 없어서 여러 번 누르는 사용자가 많았다. Optimistic update로 즉시 하트 색상과 카운트를 바꾸고, 서버 실패 시에만 롤백하도록 변경. 체감상 "즉각 반응"이 되면서 중복 클릭도 줄어듦.

### 자주 하는 오해

- spinner만 있으면 체감이 좋다 → **오래 도는 spinner는 오히려 불안감을 준다.**
- skeleton이 있으면 LCP도 좋아진다 → **체감은 개선하지만 LCP 지표는 실제 콘텐츠가 나타나야 개선.**
- 숫자 최적화만 하면 UX도 좋아진다 → **피드백 없는 빠른 응답보다 피드백 있는 적당한 응답이 더 좋게 느껴질 수 있다.**

### 면접 압축 답변

- **10초**: 체감 성능은 skeleton, optimistic UI, pending 상태 같은 즉각적 피드백으로 개선하며, 숫자 최적화와 함께 설계해야 합니다.
- **30초**: 같은 대기 시간이라도 skeleton이나 optimistic update가 있으면 덜 답답합니다. 300ms 이내 응답이면 로딩 표시를 생략하고, 그 이상이면 skeleton이나 pending 상태를 보여주는 게 자연스럽습니다. 숫자(LCP, INP)를 줄이는 것과 피드백(skeleton, optimistic)을 주는 것을 함께 설계해야 진짜 UX가 좋아집니다.

### 1단계: 느림을 숫자로 분해한다 — Core Web Vitals

성능 최적화의 첫 번째 원칙은 **"해결책이 아니라 측정부터"**다. Core Web Vitals는 사용자의 "느리다"를 LCP(로딩), INP(반응성), CLS(안정성)이라는 세 축으로 분해한다.

### 2단계: 각 지표의 병목을 추적한다 — LCP, INP, CLS

LCP는 뷰포트에서 가장 큰 콘텐츠의 렌더링 완료 시점이고, **TTFB, 리소스 발견 시점, 다운로드 우선순위, 렌더 차단이 복합 원인**이다. INP는 모든 상호작용 중 가장 느린 응답의 대표값으로, **Long Task와 메인 스레드 점유가 핵심 원인**이다. CLS는 예기치 않은 레이아웃 이동의 누적 점수로, **공간을 미리 예약하는 것**이 해결의 핵심이다.

### 3단계: 화면이 그려지는 경로를 이해한다 — 렌더링 파이프라인

브라우저는 DOM → CSSOM → 렌더 트리 → Layout(Reflow) → Paint → Composite 순서로 화면을 그린다. **Reflow는 비싸고, `transform`/`opacity` 애니메이션은 Composite만 발생해서 저렴하다.**

### 4단계: JS 전송량과 실행 비용을 줄인다 — 번들 크기와 코드 스플리팅

**gzip은 전송 크기만 줄이지 실행 크기는 그대로다.** 라우트/컴포넌트/기능 단위로 코드를 나눠 초기 필요 코드만 먼저 보내는 것이 핵심이다.

### 5단계: 리소스 자체를 최적화한다 — 이미지와 Lazy Loading

**적절한 크기/포맷/우선순위/공간 예약을 함께 설계**해야 한다. LCP 이미지에 lazy loading을 걸면 오히려 악화된다. LCP 이미지에는 `fetchpriority="high"`를 설정한다.

---

여기서부터 관점이 "리소스 전달"에서 "실행 환경"으로 전환된다.

---

### 6단계: 첫 화면을 가로막는 요소를 치운다 — 렌더 차단과 메인 스레드

**`defer`로 파싱과 병렬 다운로드하고, Critical CSS만 인라인해서 첫 화면을 앞당긴다.** Long Task(50ms+)는 `scheduler.yield()`나 Web Worker로 쪼개야 INP가 개선된다.

### 7단계: 대량 DOM과 무거운 UI를 전략적으로 다룬다 — Virtualization과 Observer API

긴 리스트는 virtualization으로 **화면에 보이는 영역만 렌더**한다. IntersectionObserver와 ResizeObserver는 강제 reflow를 발생시키지 않아 성능이 좋다.

### 8단계: 숫자와 UX 피드백을 함께 설계한다 — 체감 성능과 Fallback UI

**숫자(LCP, INP)를 줄이는 것과 피드백(skeleton, optimistic)을 주는 것을 함께 설계해야 진짜 UX가 좋아진다.**

### 한 줄로 이어 읽기

**측정**이 느림을 숫자로 분해하고 → **Core Web Vitals(LCP/INP/CLS)**가 각각 로딩·반응성·안정성의 병목을 가리키고 → **렌더링 파이프라인**이 비용 차이를 드러내고 → **번들 크기와 코드 스플리팅**이 JS 전송/실행 비용을 줄이고 → **이미지 최적화**가 리소스를 경량화하고 → **렌더 차단 제거와 메인 스레드 해방**이 첫 화면과 응답성을 확보하고 → **Virtualization과 Observer API**가 대량 DOM을 처리하고 → **체감 성능과 Fallback UI**가 숫자 최적화와 UX 피드백을 하나로 엮는다.
