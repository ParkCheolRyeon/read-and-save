# 09. 웹 성능 20문항

> Core Web Vitals, 렌더링 비용, 로딩 전략, 체감 성능을 묻는 20문항입니다.

## 총평

이 섹션은 숫자 몇 개 외우는 문제가 아니다. 면접관이 듣고 싶은 건 결국 이거다.

- 병목을 어디서 찾는가
- 무엇을 먼저 줄이는가
- 측정과 체감 성능을 어떻게 구분하는가
- 네트워크와 렌더링, UX를 함께 보느냐

즉 성능 최적화 답변은 **"빨라지게 했습니다"**가 아니라 **"왜 느렸고, 무엇을 줄였고, 사용자가 무엇을 덜 기다리게 됐는지"**를 말해야 한다.

### 전체 흐름 맵

```
성능 지표 측정 (무엇을 보는가)
  → Core Web Vitals (LCP, CLS, INP)
    → LCP 개선 (첫 화면 속도)
      → CLS 개선 (시각 안정성)
        → 번들 크기 (전송량 줄이기)
          → 코드 스플리팅 (필요한 것만 보내기)
            → 이미지 최적화 (미디어 전송)
              → 비디오 최적화 (스트리밍)
                → 렌더 차단 리소스 (브라우저 병목)
                  → lazy loading (지연 로딩)
                    → React 리렌더링 (클라이언트 렌더링)
                      → 긴 리스트 (DOM 수 관리)
                        → 캐싱 (반복 요청 제거)
                          → 체감 성능 (느린 환경 대응)
                            → fallback UI (장애 = 성능)
                              → 차트/시각화 (계산 비용)
                                → 애니메이션 (GPU 활용)
                                  → Lighthouse vs 현실 (측정의 한계)
                                    → 실전 경험 (종합 적용)
```

---

## 1. 웹 성능 최적화에서 가장 먼저 측정해야 할 지표는 무엇이라고 생각하나요?

### 합격 답변 예시

정답이 하나로 고정되진 않지만, 저는 사용자 체감 기준으로는 Core Web Vitals와 화면별 핵심 시나리오를 먼저 봅니다. 즉 LCP, INP, CLS 같은 공통 지표를 보되, 실제로 우리 사용자 불만이 어디서 발생하는지도 함께 봐야 합니다. 성능 최적화는 점수 맞추기가 아니라, 사용자가 느리다고 느끼는 순간을 줄이는 작업이라고 생각합니다.

### 가산점 포인트

**"지표를 먼저 재고, 그 다음 개선한다"**는 순서를 말하면 좋다. 감으로 최적화하면 효과 없는 곳에 시간을 쓰게 된다. **측정 → 병목 식별 → 개선 → 재측정** 사이클이 성능 최적화의 기본이다.

### 상세 해설

#### 왜 "측정 먼저"가 중요한가

```
❌ 감 기반 최적화:
  "이미지가 무거울 것 같으니 최적화하자"
  → 실제 병목은 서드파티 스크립트 → 효과 없음

✅ 측정 기반 최적화:
  Lighthouse → "LCP 4.2초, 원인: 렌더 차단 CSS 2.1초"
  → CSS 인라인 + 비핵심 CSS 지연 → LCP 1.8초로 개선
```

#### 측정 도구별 역할

| 도구 | 특성 | 용도 |
|------|------|------|
| **Lighthouse** | 실험실 측정, 재현 가능 | 개발 중 빠른 피드백 |
| **Chrome DevTools Performance** | 실시간 프로파일링 | Long Task, 렌더링 병목 분석 |
| **CrUX (Chrome UX Report)** | 실사용자 데이터 | 배포 후 실제 체감 확인 |
| **web-vitals 라이브러리** | 코드에서 수집 | 자체 RUM 구축 |

```ts
// web-vitals로 Core Web Vitals 수집
import { onLCP, onINP, onCLS } from 'web-vitals';

onLCP((metric) => {
  console.log('LCP:', metric.value, 'ms');
  // analytics에 전송
  sendToAnalytics({ name: 'LCP', value: metric.value });
});

onINP((metric) => console.log('INP:', metric.value, 'ms'));
onCLS((metric) => console.log('CLS:', metric.value));
```

#### 자주 하는 오해

- "Lighthouse 100점이면 성능이 좋은 거다" → **실험실 점수일 뿐.** 실사용자의 기기, 네트워크, 데이터 양은 반영 안 됨.
- "성능 최적화는 개발 마지막에 하면 된다" → **늦을수록 비용이 큼.** 아키텍처 결정(SSR vs CSR, 번들 구조)이 성능에 가장 큰 영향을 미친다.

#### 실무 에피소드

> Lighthouse 점수는 95점인데 실사용자 불만이 많았다. CrUX 데이터를 보니 p75 LCP가 4초였다. 원인은 실사용자의 이미지 데이터 양이 Lighthouse 테스트보다 훨씬 많았기 때문. Lab과 RUM의 차이를 체감한 계기였다.

> 성능 측정의 원칙을 알았으니, 다음에는 **가장 중요한 지표 세트** — Core Web Vitals를 봐야 한다.

## 2. Core Web Vitals를 설명해보세요.

| 지표 | 보는 축 | 짧은 해석 | 좋음 기준 |
| --- | --- | --- | --- |
| `LCP` | 로딩 | 주요 콘텐츠가 언제 보였는가 | ≤ 2.5초 |
| `CLS` | 시각 안정성 | 화면이 얼마나 흔들렸는가 | ≤ 0.1 |
| `INP` | 반응성 | 입력 후 다음 페인트까지 얼마나 걸렸는가 | ≤ 200ms |

### 합격 답변 예시

Core Web Vitals는 사용자 경험을 대표하는 핵심 웹 성능 지표입니다. LCP는 주요 콘텐츠가 얼마나 빨리 보이는지, CLS는 레이아웃이 얼마나 안정적인지, INP는 사용자 상호작용에 얼마나 빠르게 반응하는지를 봅니다. 즉 로딩, 시각 안정성, 반응성의 세 축을 측정한다고 이해하면 됩니다.

### 가산점 포인트

**INP가 FID를 대체**했다는 점을 알면 좋다. FID는 첫 입력의 지연만 봤지만, INP는 페이지 전체 생명주기에서 가장 느린 상호작용을 측정한다. 2024년부터 공식 교체되었다.

### 상세 해설

#### 왜 이 세 지표인가

사용자가 웹에서 느끼는 불만은 크게 세 가지로 분류된다:

```
"안 뜬다" → LCP (로딩)
  사용자가 페이지를 열었는데 주요 콘텐츠가 늦게 나타남

"흔들린다" → CLS (시각 안정성)
  읽고 있는데 갑자기 레이아웃이 밀려서 잘못된 버튼을 누름

"버벅인다" → INP (반응성)
  버튼을 눌렀는데 반응이 늦음, 입력했는데 커서가 안 따라옴
```

#### 각 지표의 측정 방식

```ts
// 브라우저에서 Core Web Vitals 직접 확인 (DevTools Console)
// 또는 web-vitals 라이브러리로 수집
import { onLCP, onCLS, onINP } from 'web-vitals';

// LCP: viewport에서 가장 큰 콘텐츠 요소의 렌더 시점
onLCP((metric) => {
  console.log(`LCP: ${metric.value}ms`);
  console.log('LCP 요소:', metric.entries[0]?.element);
  // → <img src="hero.jpg"> 같은 요소가 잡힘
});

// CLS: 예상치 못한 레이아웃 이동의 누적 점수
onCLS((metric) => {
  console.log(`CLS: ${metric.value}`);
  // 0에 가까울수록 좋음, 0.1 이하가 목표
});

// INP: 가장 느린 상호작용의 입력→다음 페인트 지연
onINP((metric) => {
  console.log(`INP: ${metric.value}ms`);
  // 200ms 이하가 좋음
});
```

#### 자주 하는 오해

- "FID가 Core Web Vitals이다" → **과거 이야기.** 2024년부터 INP로 공식 교체됨. INP는 첫 입력만이 아니라 모든 상호작용을 본다.
- "세 지표만 좋으면 성능이 좋다" → **기본 기준일 뿐.** 제품 특성에 따라 TTFB, FCP, TBT 등 추가 지표도 중요할 수 있다.

> Core Web Vitals의 전체 그림을 알았으니, 다음에는 **각 지표를 구체적으로 어떻게 개선하는지** — LCP부터 봐야 한다.

## 3. LCP를 개선하는 방법을 설명해보세요.

### 합격 답변 예시

LCP를 개선하려면 사용자가 가장 먼저 보는 큰 콘텐츠가 빨리 나오도록 해야 합니다. 대표적으로 큰 이미지나 비디오 최적화, 서버 응답 시간 단축, 렌더 차단 리소스 축소, CDN 활용, preload나 preconnect 같은 우선순위 제어가 중요합니다. 즉 첫 화면에서 가장 중요한 요소가 무엇인지 먼저 찾고, 그 요소가 늦어지는 원인을 줄이는 방식으로 접근해야 합니다.

### 가산점 포인트

LCP 개선의 첫 단계는 **"LCP 요소가 뭔지 찾는 것"**이다. DevTools에서 `performance.getEntriesByType('largest-contentful-paint')`로 어떤 요소가 LCP인지 확인하고, 그 요소의 로딩 경로를 추적해야 한다.

### 상세 해설

#### LCP가 느려지는 4가지 원인과 해결

```
1. 서버 응답이 느림 (TTFB)
   → CDN 사용, 서버 사이드 캐시, Edge 렌더링

2. 렌더 차단 리소스
   → Critical CSS 인라인, 비핵심 CSS defer, JS async/defer

3. LCP 리소스 로딩이 느림
   → 이미지 최적화, preload, fetchpriority="high"

4. 클라이언트 렌더링 지연
   → SSR/SSG 활용, hydration 최적화
```

```html
<!-- LCP 이미지를 우선 로드 -->
<link rel="preload" as="image" href="/hero.webp" fetchpriority="high" />

<!-- 비핵심 CSS는 지연 로드 -->
<link rel="preload" as="style" href="/non-critical.css" onload="this.rel='stylesheet'" />

<!-- LCP 이미지에 fetchpriority 지정 -->
<img src="/hero.webp" alt="Hero" fetchpriority="high" width="1200" height="600" />
```

```tsx
// Next.js에서 LCP 이미지 최적화
import Image from 'next/image';

// priority 속성으로 LCP 이미지를 preload
<Image src="/hero.webp" alt="Hero" width={1200} height={600} priority />
```

#### 자주 하는 오해

- "이미지를 압축하면 LCP가 좋아진다" → **부분만 맞음.** 이미지 크기도 중요하지만, TTFB가 느리면 이미지 최적화 효과가 제한적이다. 전체 경로를 봐야 한다.
- "lazy loading을 모든 이미지에 적용하면 좋다" → **LCP 이미지에는 절대 안 됨.** LCP 이미지를 lazy load하면 오히려 LCP가 나빠진다.

#### 실무 에피소드

> LCP가 3.5초였는데 이미지 최적화를 해도 개선이 안 됐다. DevTools로 보니 TTFB가 2초였다 — 서버 응답 자체가 느린 거였다. CDN + Edge 캐시를 적용하니 TTFB가 200ms로 줄고, LCP가 1.8초로 개선됐다.

> LCP를 알았으니, 다음에는 **시각 안정성 지표** — CLS 원인과 해결을 봐야 한다.

## 4. CLS가 발생하는 대표 원인과 해결 방법을 설명해보세요.

### 합격 답변 예시

CLS는 이미지, 광고, iframe처럼 늦게 들어오는 요소의 공간이 미리 확보되지 않았을 때 자주 발생합니다. 또 웹폰트가 늦게 적용되면서 텍스트 크기가 바뀌거나, 상단 배너가 나중에 삽입될 때도 흔들림이 생길 수 있습니다. 해결하려면 width/height나 aspect-ratio를 명시해 공간을 확보하고, 동적으로 삽입되는 요소를 위한 자리도 미리 마련해야 합니다.

### 가산점 포인트

CLS의 핵심은 **"브라우저가 크기를 모르는 요소"**다. 이미지에 width/height가 없으면 브라우저는 로드 전까지 크기를 모르고, 로드 후 레이아웃을 다시 계산(reflow)한다. 이 reflow가 CLS를 만든다.

### 상세 해설

#### 왜 브라우저가 레이아웃을 "흔드는가"

```
1. 이미지 로드 전: 브라우저는 크기 0으로 레이아웃 계산
   ┌──────────────────────┐
   │  텍스트 내용...        │
   │  [이미지: 크기 0]      │  ← 공간 0
   │  다음 텍스트...        │
   └──────────────────────┘

2. 이미지 로드 후: 갑자기 크기가 생기면서 아래 요소가 밀림
   ┌──────────────────────┐
   │  텍스트 내용...        │
   │  ┌─────────────────┐ │
   │  │   이미지 400px    │ │  ← 갑자기 400px 공간 생성
   │  └─────────────────┘ │
   │  다음 텍스트...        │  ← 이만큼 밀림 = CLS
   └──────────────────────┘
```

#### 해결 방법

```html
<!-- ❌ CLS 유발: 크기 없는 이미지 -->
<img src="/photo.jpg" alt="사진" />

<!-- ✅ CLS 방지: 크기 명시 -->
<img src="/photo.jpg" alt="사진" width="800" height="600" />

<!-- ✅ CSS aspect-ratio로 반응형 공간 확보 -->
<style>
  .img-container {
    aspect-ratio: 16 / 9;
    width: 100%;
  }
</style>
```

```tsx
// Next.js Image 컴포넌트는 자동으로 CLS를 방지
<Image src="/photo.jpg" alt="사진" width={800} height={600} />
// → width/height 기반으로 placeholder 공간을 미리 확보
```

#### 대표 원인 체크리스트

| 원인 | 해결 |
|------|------|
| 크기 없는 이미지 | width/height 또는 aspect-ratio 명시 |
| 동적 광고/배너 | 삽입 전 공간 예약 |
| 웹폰트 교체 | `font-display: swap` + `size-adjust` |
| 동적 콘텐츠 삽입 | 기존 콘텐츠 위에 삽입하지 않기 |
| iframe 자리 미확보 | 고정 크기 또는 aspect-ratio |

#### 자주 하는 오해

- "CLS는 이미지에서만 발생한다" → **아님.** 폰트 교체, 동적 광고, 늦게 뜨는 쿠키 배너 등 다양한 원인이 있다.
- "display: none으로 숨겼다가 보여주면 CLS가 안 생긴다" → **상황에 따라 다름.** `display: none` → `block` 전환 시 주변 요소가 밀리면 CLS가 발생한다.

> CLS를 알았으니, 다음에는 **전송량 자체를 줄이는 방법** — JS 번들 크기 최적화를 봐야 한다.

## 5. JavaScript 번들 크기를 줄이는 방법을 설명해보세요.

### 합격 답변 예시

불필요한 라이브러리를 줄이고, 코드 스플리팅과 dynamic import를 활용하고, 트리 쉐이킹이 잘 되도록 구조를 잡는 것이 기본입니다. 또 locale 데이터나 큰 유틸 패키지처럼 숨어 있는 무거운 의존성을 줄이고, 클라이언트에서만 필요한 코드만 번들에 포함되도록 경계를 잘 나누는 것이 중요합니다. 즉 번들 크기 최적화는 코드를 줄이는 게 아니라, **사용자에게 지금 필요한 것만 보내는 전략**에 가깝습니다.

### 가산점 포인트

**"압축하면 된다" 수준이 아니라 "전송 자체를 줄인다"**고 말해야 한다. gzip은 전송 시 압축일 뿐이고, 번들 크기 최적화는 **아예 보내지 않아도 되는 코드를 제거**하는 것이다. `@next/bundle-analyzer`로 번들 구성을 시각화하면 숨은 거대 의존성을 찾기 쉽다.

### 상세 해설

#### 번들 분석 → 병목 찾기 → 제거/분리

```bash
# Next.js 번들 분석
ANALYZE=true pnpm build
# → .next/analyze/ 에서 번들 구성 시각화
```

```
번들 분석 결과 예시:
  lodash (전체)     72KB  → lodash-es + 개별 import로 8KB
  moment            67KB  → date-fns 또는 dayjs로 3KB
  chart.js          60KB  → dynamic import로 분리
  아이콘 라이브러리    45KB  → 사용하는 아이콘만 import
```

#### 대표 최적화 기법

```ts
// ❌ 전체 라이브러리 import
import _ from 'lodash';
const sorted = _.sortBy(items, 'name');

// ✅ 필요한 함수만 import (tree shaking 가능)
import sortBy from 'lodash-es/sortBy';
const sorted = sortBy(items, 'name');

// ✅ dynamic import로 무거운 컴포넌트 분리
const Chart = dynamic(() => import('./Chart'), {
  loading: () => <Skeleton />,
  ssr: false,  // 서버에서는 불필요
});
```

#### 자주 하는 오해

- "트리 쉐이킹이 자동으로 다 해준다" → **CommonJS 모듈은 트리 쉐이킹 안 됨.** ESM(`import/export`)을 써야 번들러가 사용하지 않는 코드를 제거할 수 있다.
- "번들 크기는 빌드 후에야 확인할 수 있다" → **번들 분석 도구를 쓰면 개발 중에도 확인 가능.** `@next/bundle-analyzer`, `webpack-bundle-analyzer` 등.

> 번들 크기를 알았으니, 다음에는 **번들을 나누는 전략** — 코드 스플리팅을 봐야 한다.

## 6. 코드 스플리팅은 무엇이고, 어떤 기준으로 분리하나요?

### 합격 답변 예시

코드 스플리팅은 애플리케이션 코드를 한 번에 다 보내지 않고, 라우트나 기능 단위로 나눠 필요한 시점에 로드하는 전략입니다. 기준은 초기 진입에 꼭 필요한지, 특정 화면이나 모달, 차트, 에디터처럼 나중에만 필요한지입니다. 너무 잘게 쪼개면 요청 수와 관리 복잡도가 늘 수 있으니, 사용자 흐름 단위로 나누는 편이 보통 좋습니다.

### 가산점 포인트

Next.js App Router에서는 **라우트 기반 코드 스플리팅이 자동**이다. 각 `page.tsx`가 별도 청크로 분리된다. 추가로 `dynamic()`이나 `React.lazy()`로 컴포넌트 단위 분리도 가능하다.

### 상세 해설

#### 분리 기준 판단

```
"이 코드가 첫 화면에 꼭 필요한가?"
  → YES: 메인 번들에 포함
  → NO: 별도 청크로 분리 (lazy load)

분리하기 좋은 대상:
  ✅ 라우트별 페이지 (자동)
  ✅ 모달/다이얼로그 (열 때만 필요)
  ✅ 차트/에디터 (무거운 라이브러리)
  ✅ 관리자 전용 기능 (일반 사용자 불필요)

분리하면 안 되는 대상:
  ❌ 항상 같이 필요한 유틸 (분리하면 추가 요청만 증가)
  ❌ 너무 작은 컴포넌트 (청크 오버헤드가 더 큼)
```

```tsx
// Next.js에서 무거운 컴포넌트 dynamic import
import dynamic from 'next/dynamic';

const RichEditor = dynamic(() => import('@/components/RichEditor'), {
  loading: () => <div className="h-64 animate-pulse bg-zinc-100" />,
  ssr: false,
});

// 모달 안에서만 필요한 차트
const AnalyticsChart = dynamic(() => import('@/components/AnalyticsChart'), {
  ssr: false,
});
```

#### 자주 하는 오해

- "코드 스플리팅을 많이 할수록 좋다" → **과도하면 역효과.** 청크가 너무 많으면 HTTP 요청 수가 늘어나고, 각 청크의 로드 타이밍이 예측 불가능해진다.

> 코드 스플리팅을 알았으니, 다음에는 **가장 큰 전송 대상** — 이미지 최적화를 봐야 한다.

## 7. 이미지 최적화에서 고려할 요소들을 설명해보세요.

### 합격 답변 예시

이미지 최적화에서는 포맷, 크기, 반응형 전달, lazy loading, 캐시, CDN, CLS 방지가 핵심입니다. 즉 단순히 압축만 하는 게 아니라, 화면 크기에 맞는 이미지를 보내고, 늦게 로드해도 레이아웃이 흔들리지 않게 하고, 재사용 가능한 자산은 캐시하는 것까지 포함합니다.

### 가산점 포인트

**"적절한 크기의 적절한 포맷을 적절한 시점에 보낸다"**가 이미지 최적화의 한 줄 요약이다. next/image는 이 세 가지를 자동화한다.

### 상세 해설

#### 이미지 최적화 체크리스트

```tsx
// Next.js Image — 자동 최적화
<Image
  src="/product.jpg"
  alt="상품"
  width={600}
  height={400}
  // 자동으로:
  // 1. WebP/AVIF 변환
  // 2. srcset 생성 (반응형)
  // 3. lazy loading (priority 없으면)
  // 4. width/height로 CLS 방지
  // 5. 적절한 크기로 리사이즈
/>

// LCP 이미지는 priority 추가
<Image src="/hero.webp" alt="Hero" width={1200} height={600} priority />
```

| 체크 항목 | 설명 |
|----------|------|
| **포맷** | WebP/AVIF (JPEG 대비 30~50% 작음) |
| **크기** | 표시 크기에 맞는 이미지 (2x 레티나 고려) |
| **반응형** | srcset/sizes로 화면에 맞는 이미지 전달 |
| **lazy loading** | below-the-fold 이미지는 지연 로드 |
| **CLS 방지** | width/height 또는 aspect-ratio 명시 |
| **캐시** | CDN + immutable 캐시 헤더 |
| **placeholder** | blur 또는 LQIP로 로딩 중 공간 채움 |

#### 자주 하는 오해

- "이미지를 작게 만들면 끝이다" → **화면 크기에 맞게 보내야.** 모바일에서 2000px 이미지를 보내면 낭비.
- "모든 이미지에 lazy loading을 적용하면 좋다" → **LCP 이미지는 제외.** hero 이미지를 lazy load하면 LCP가 나빠진다.

> 이미지를 알았으니, 다음에는 **더 복잡한 미디어** — 비디오 최적화를 봐야 한다.

## 8. 비디오 최적화는 이미지 최적화와 무엇이 다른가요?

### 합격 답변 예시

비디오는 이미지보다 훨씬 크고, 연속 재생과 seek, 버퍼링, 화질 적응 같은 문제가 있어서 최적화 관점이 더 복잡합니다. 이미지는 한 장을 얼마나 효율적으로 보여줄지가 핵심이라면, 비디오는 **지속적으로 재생 가능한 흐름을 만드는 것**이 핵심입니다.

### 가산점 포인트

비디오는 **"전송 최적화 + 재생 경험 최적화"가 함께 필요**하다. 이미지는 한 번 다운로드하면 끝이지만, 비디오는 재생 내내 네트워크와 상호작용한다.

### 상세 해설

#### 이미지 vs 비디오 최적화 차이

| | 이미지 | 비디오 |
|---|---|---|
| **크기** | KB~수 MB | 수십 MB~수 GB |
| **전달 방식** | 한 번 다운로드 | 스트리밍 (세그먼트 단위) |
| **네트워크 적응** | 불필요 | adaptive bitrate 필요 |
| **사용자 행동** | 보기만 함 | seek, 일시정지, 화질 변경 |
| **CLS** | width/height로 해결 | poster + 고정 크기 컨테이너 |

```html
<!-- 비디오 최적화 기본 -->
<video
  poster="/thumbnail.webp"
  preload="metadata"
  width="1280"
  height="720"
>
  <source src="/video.webm" type="video/webm" />
  <source src="/video.mp4" type="video/mp4" />
</video>
```

#### 자주 하는 오해

- "mp4를 그냥 올리면 된다" → **단일 파일은 adaptive bitrate 불가.** 느린 네트워크에서 버퍼링이 끊임없이 발생한다. DASH/HLS로 세그먼트를 나눠야 환경에 따라 화질을 조절할 수 있다.

> 비디오를 알았으니, 다음에는 **스트리밍 방식의 구체적인 이점**을 봐야 한다.

## 9. 단일 mp4 대신 MPEG-DASH 같은 스트리밍 방식을 쓰는 이유를 설명해보세요.

### 합격 답변 예시

MPEG-DASH 같은 스트리밍 방식은 영상을 작은 세그먼트로 나누고, 네트워크 상태에 따라 적절한 비트레이트를 선택해 재생할 수 있기 때문에 단일 mp4보다 훨씬 유연합니다. 그래서 초기 재생이 빠르고, 중간 버퍼링이 줄고, 느린 환경에서도 완전히 끊기기보다 낮은 화질로라도 이어서 볼 수 있습니다.

### 가산점 포인트

**adaptive bitrate streaming (ABR)**의 핵심은 "네트워크가 좋으면 고화질, 나쁘면 저화질로 자동 전환"이다. 단일 mp4에서는 이게 불가능하다. 느린 환경에서 버퍼링으로 완전히 멈추는 것보다, 화질을 낮춰서라도 재생을 이어가는 게 UX에 훨씬 좋다.

### 상세 해설

```
단일 mp4:
  다운로드 ████████████████████ 100% 완료 → 재생 시작
  → 느린 네트워크에서 영상 시작까지 30초+ 대기

MPEG-DASH:
  세그먼트 1 (2초) ██ → 즉시 재생 시작
  세그먼트 2 (2초) ██ → 재생 중 다운로드
  네트워크 느려짐 → 자동으로 저화질 세그먼트로 전환
  네트워크 복구 → 다시 고화질로 올림
```

#### 실무 에피소드

> 교육 플랫폼에서 100MB mp4를 그대로 서빙했더니, 학교 Wi-Fi 환경에서 재생 시작까지 20초가 걸렸다. DASH로 전환하니 2초 만에 재생이 시작되고, 네트워크가 느려져도 화질만 낮아질 뿐 끊기지 않았다.

> 미디어 최적화를 알았으니, 다음에는 **브라우저 렌더링 경로에서의 병목** — 렌더 차단 리소스를 봐야 한다.

## 10. 렌더 차단 리소스가 무엇인지 설명해보세요.

### 합격 답변 예시

렌더 차단 리소스는 브라우저가 초기 화면을 그리기 전에 반드시 처리해야 해서 화면 표시를 늦추는 리소스입니다. 대표적으로 동기 스크립트와 중요한 CSS가 있습니다. 성능 최적화에서는 첫 화면에 꼭 필요한 것만 먼저 남겨두고, 나머지는 지연 로딩하거나 우선순위를 낮추는 것이 중요합니다.

### 가산점 포인트

**CSS는 렌더 차단, 동기 JS는 파싱 차단**이라는 구분을 정확히 하면 좋다. CSS가 로드되기 전에는 Render Tree를 만들 수 없고(렌더 차단), 동기 `<script>`를 만나면 HTML 파싱 자체가 멈춘다(파싱 차단).

### 상세 해설

#### 브라우저 렌더링 경로에서의 차단

```
HTML 파싱 시작
  ↓
<link rel="stylesheet" href="style.css">  ← CSS 렌더 차단
  ↓ (CSS 다운로드 + 파싱 완료까지 대기)
<script src="app.js"></script>              ← JS 파싱 차단
  ↓ (JS 다운로드 + 실행 완료까지 HTML 파싱 멈춤)
나머지 HTML 파싱 계속...
  ↓
Render Tree → Layout → Paint → 화면 표시
```

#### 해결 방법

```html
<!-- ❌ 렌더/파싱 차단 -->
<link rel="stylesheet" href="/all-styles.css" />
<script src="/app.js"></script>

<!-- ✅ 차단 최소화 -->
<!-- Critical CSS 인라인 -->
<style>
  .hero { font-size: 2rem; color: #333; }
  /* 첫 화면에 필요한 최소 스타일만 */
</style>

<!-- 비핵심 CSS 지연 로드 -->
<link rel="preload" as="style" href="/non-critical.css"
      onload="this.rel='stylesheet'" />

<!-- JS는 defer로 파싱 차단 방지 -->
<script src="/app.js" defer></script>
```

#### 자주 하는 오해

- "`async`와 `defer`는 같다" → **다름.** `async`는 다운로드 완료 시 즉시 실행(실행 순서 보장 안 됨), `defer`는 HTML 파싱 완료 후 순서대로 실행. 대부분의 경우 `defer`가 더 안전하다.

> 렌더 차단을 알았으니, 다음에는 **지연 로딩의 적용 범위**를 봐야 한다.

## 11. lazy loading은 어디까지 적용하는 게 좋은가요?

### 합격 답변 예시

첫 화면에 꼭 필요하지 않은 이미지, 컴포넌트, 모달, 차트, 에디터, 비디오 등에 적용하는 것이 좋습니다. 반대로 첫 화면 핵심 콘텐츠까지 lazy로 미루면 오히려 LCP와 체감 성능이 나빠질 수 있습니다. 즉 lazy loading은 많이 쓰는 게 중요한 게 아니라, **늦게 와도 되는 것만 늦게 가져오는 것**이 중요합니다.

### 가산점 포인트

lazy loading의 기준은 **"사용자가 지금 이것을 보고 있는가"**다. viewport 안에 있는 것은 즉시 로드, viewport 밖에 있는 것은 lazy. 이미지는 `loading="lazy"`, 컴포넌트는 `React.lazy()` 또는 `dynamic()`.

### 상세 해설

```tsx
// 이미지: native lazy loading
<img src="/photo.jpg" alt="사진" loading="lazy" width="400" height="300" />

// 컴포넌트: dynamic import
const HeavyChart = dynamic(() => import('./Chart'), {
  ssr: false,
  loading: () => <Skeleton className="h-64" />,
});

// Intersection Observer로 viewport 진입 시 로드
function LazySection({ children }: { children: React.ReactNode }) {
  const [visible, setVisible] = useState(false);
  const ref = useRef<HTMLDivElement>(null);

  useEffect(() => {
    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) setVisible(true);
    });
    if (ref.current) observer.observe(ref.current);
    return () => observer.disconnect();
  }, []);

  return <div ref={ref}>{visible ? children : <Skeleton />}</div>;
}
```

#### 적용 vs 제외 판단

| 대상 | lazy? | 이유 |
|------|-------|------|
| hero 이미지 | ❌ | LCP 요소, 즉시 필요 |
| below-the-fold 이미지 | ✅ | 스크롤해야 보임 |
| 모달 내부 컴포넌트 | ✅ | 열 때만 필요 |
| 차트/에디터 | ✅ | 무거운 라이브러리 |
| 네비게이션 메뉴 | ❌ | 항상 필요 |

> lazy loading을 알았으니, 다음에는 **클라이언트 렌더링 최적화** — React 리렌더링 관리를 봐야 한다.

## 12. React에서 불필요한 리렌더링을 어떻게 찾아내고 줄이나요?

### 합격 답변 예시

먼저 React DevTools Profiler나 렌더 로그로 어떤 컴포넌트가 왜 자주 렌더되는지 확인합니다. 그 다음 상태 소유 범위를 줄이고, memoization이 실제로 의미 있는 곳에만 React.memo, useMemo, useCallback을 적용하고, selector 기반 구독이나 상태 분리를 통해 영향 범위를 좁힙니다. 핵심은 **memo를 많이 붙이는 게 아니라, 리렌더링 원인을 먼저 찾는 것**입니다.

### 가산점 포인트

React DevTools의 **"Highlight updates when components render"** 옵션을 켜면 리렌더링되는 컴포넌트가 시각적으로 표시된다. 또 Profiler의 **"Why did this render?"** 기능으로 원인을 정확히 파악할 수 있다.

### 상세 해설

#### 리렌더링 원인 진단 → 해결

```tsx
// 원인 1: 부모 리렌더 → 자식도 리렌더
// 해결: React.memo로 props가 같으면 스킵
const ExpensiveList = React.memo(function ExpensiveList({ items }: Props) {
  return items.map(item => <Item key={item.id} {...item} />);
});

// 원인 2: 매번 새 객체/함수 참조를 props로 전달
// ❌ 매 렌더마다 새 함수 생성
<Button onClick={() => handleClick(id)} />

// ✅ useCallback으로 참조 안정화
const handleClick = useCallback(() => onClick(id), [id]);
<Button onClick={handleClick} />

// 원인 3: Context가 바뀌면 모든 소비자가 리렌더
// 해결: Context 값을 분리하거나, 외부 상태 관리 도구의 selector 사용
const count = useStore((state) => state.count); // Zustand selector
// count가 안 바뀌면 리렌더 안 됨
```

#### 자주 하는 오해

- "모든 컴포넌트에 React.memo를 붙이면 좋다" → **오히려 역효과 가능.** memo는 얕은 비교(shallow comparison) 비용이 있고, props가 항상 바뀌는 컴포넌트에서는 비교만 하고 어차피 리렌더된다.
- "리렌더링은 무조건 나쁘다" → **아님.** React는 리렌더링을 전제로 설계되었다. 리렌더가 문제가 되는 건 **느린 리렌더**일 때만이다.

> 리렌더링을 알았으니, 다음에는 **DOM 수가 많을 때의 성능 문제** — 긴 리스트를 봐야 한다.

## 13. 리스트가 긴 화면에서 성능 저하를 줄이는 방법을 설명해보세요.

### 합격 답변 예시

가장 대표적인 방법은 가상화로, 화면에 보이는 항목만 렌더하고 나머지는 필요할 때만 그리는 방식입니다. 또 pagination이나 infinite scroll, 아이템 단위 memoization, 이미지 lazy loading, 무거운 셀 구조 단순화도 함께 사용할 수 있습니다. 핵심은 **수천 개를 한 번에 모두 DOM에 올리지 않는 것**입니다.

### 가산점 포인트

가상화의 핵심 원리는 **"viewport에 보이는 아이템만 실제 DOM에 존재"**하는 것이다. 10,000개 아이템이 있어도 화면에 보이는 20개만 DOM에 있으므로 렌더링 비용이 일정하게 유지된다.

### 상세 해설

```tsx
// react-virtuoso로 가상화 리스트
import { Virtuoso } from 'react-virtuoso';

function LongList({ items }: { items: Item[] }) {
  return (
    <Virtuoso
      data={items}                    // 10,000개의 데이터
      itemContent={(index, item) => ( // 보이는 것만 렌더
        <div className="p-4 border-b">
          <h3>{item.title}</h3>
          <p>{item.description}</p>
        </div>
      )}
      style={{ height: '100vh' }}
    />
    // DOM에는 ~20개만 존재, 스크롤하면 재활용
  );
}
```

#### 가상화 없이 vs 가상화 적용

```
10,000개 아이템:
  가상화 없음: DOM 노드 10,000개 → 렌더 3초, 메모리 500MB
  가상화 적용: DOM 노드 ~20개 → 렌더 50ms, 메모리 50MB
```

> 리스트를 알았으니, 다음에는 **반복 요청을 줄이는 전략** — 캐싱을 봐야 한다.

## 14. 캐싱 전략이 성능에 미치는 영향을 설명해보세요.

### 합격 답변 예시

캐싱은 같은 데이터를 다시 계산하거나 다시 요청하지 않게 해서 네트워크 지연과 서버 부하, 렌더 지연을 줄입니다. 브라우저 캐시, CDN 캐시, API 응답 캐시, 클라이언트 데이터 캐시 모두 체감 성능에 영향을 줍니다. 다만 캐시는 항상 **stale 문제와 invalidation 문제**를 같이 가져오기 때문에, 단순히 많이 캐시하는 것보다 어떤 데이터를 얼마나 신뢰할 수 있는지까지 설계해야 합니다.

### 가산점 포인트

**"캐시에서 가장 어려운 문제는 invalidation"**이라고 말하면 좋다. 캐시된 데이터가 오래되었는지 아닌지 판단하는 것이 핵심이다. `stale-while-revalidate`는 "일단 캐시를 보여주고, 백그라운드에서 최신 데이터를 가져온다"는 전략으로 속도와 최신성의 균형을 맞춘다.

### 상세 해설

#### 캐시 계층별 역할

```
사용자 요청
  ↓
브라우저 메모리 캐시 (0ms) — 같은 세션 내 반복 요청
  ↓ miss
브라우저 디스크 캐시 (~10ms) — Cache-Control 헤더 기반
  ↓ miss
CDN 엣지 캐시 (~50ms) — CloudFront, Vercel Edge
  ↓ miss
서버 캐시 (~100ms) — Redis, 서버 메모리
  ↓ miss
원본 서버 (~500ms+) — DB 쿼리, 계산
```

```ts
// React Query에서 stale-while-revalidate
const { data } = useQuery({
  queryKey: ['users'],
  queryFn: fetchUsers,
  staleTime: 5 * 60 * 1000,     // 5분간 fresh
  gcTime: 30 * 60 * 1000,       // 30분간 캐시 유지
  // 5분 이내 재요청 → 캐시 반환 (네트워크 요청 없음)
  // 5분 후 재요청 → 캐시 반환 + 백그라운드에서 갱신
});
```

#### 자주 하는 오해

- "캐시를 오래 하면 할수록 좋다" → **stale 위험.** 사용자 데이터처럼 자주 바뀌는 것을 오래 캐시하면 옛날 데이터를 보게 된다.
- "캐시 문제가 생기면 `?v=123` 쿼리 파라미터로 해결" → **임시방편.** 빌드 해시 기반 파일명(`app.abc123.js`)이 더 정확하고 CDN 캐시와도 잘 작동한다.

> 캐싱을 알았으니, 다음에는 **느린 환경에서의 체감 성능** — 불안정한 네트워크 대응을 봐야 한다.

## 15. 모바일 환경, 특히 학교 Wi-Fi처럼 불안정한 환경에서 체감 성능을 높이는 방법은 무엇인가요?

### 합격 답변 예시

절대 속도를 높이는 것도 중요하지만, 그보다 **사용자가 흐름을 잃지 않게 만드는 설계**가 더 중요합니다. 재시도, timeout, backoff, fallback UI, 임시 저장, optimistic UI, 이미지·비디오 적응형 로딩을 통해 느린 환경에서도 작업을 이어갈 수 있게 해야 합니다. 즉 불안정한 환경에서는 성능 최적화와 장애 대응 UX가 사실상 같은 문제입니다.

### 가산점 포인트

**optimistic UI**는 서버 응답을 기다리지 않고 먼저 UI를 업데이트하는 패턴이다. 좋아요 버튼을 누르면 즉시 하트가 채워지고, 서버 요청은 백그라운드에서 처리된다. 느린 네트워크에서 체감 속도를 크게 높인다.

### 상세 해설

```ts
// Optimistic UI 예시
const likeMutation = useMutation({
  mutationFn: (postId: string) => api.likePost(postId),
  onMutate: async (postId) => {
    // 서버 응답 전에 먼저 UI 업데이트
    await queryClient.cancelQueries({ queryKey: ['post', postId] });
    const previous = queryClient.getQueryData(['post', postId]);
    queryClient.setQueryData(['post', postId], (old: Post) => ({
      ...old,
      liked: true,
      likeCount: old.likeCount + 1,
    }));
    return { previous };
  },
  onError: (err, postId, context) => {
    // 실패 시 롤백
    queryClient.setQueryData(['post', postId], context?.previous);
  },
});
```

#### 느린 환경 대응 전략

| 전략 | 효과 |
|------|------|
| **Optimistic UI** | 즉각적인 피드백으로 체감 속도 향상 |
| **Skeleton UI** | 로딩 중에도 레이아웃 안정성 유지 |
| **Retry + Backoff** | 일시적 실패 자동 복구 |
| **임시 저장** | 입력 데이터 유실 방지 |
| **적응형 미디어** | 느린 환경에서 저화질 이미지/비디오 |
| **Service Worker 캐시** | 오프라인에서도 기본 UI 표시 |

#### 실무 에피소드

> 학교 Wi-Fi 환경에서 과제 제출 중 네트워크가 끊겨 작성 중이던 내용이 모두 사라지는 불만이 있었다. `localStorage`에 임시 저장 + retry 로직을 추가하니 네트워크 복구 후 자동 제출되어 불만이 크게 줄었다.

> 체감 성능을 알았으니, 다음에는 **장애 상황에서의 체감 성능** — fallback UI를 봐야 한다.

## 16. 네트워크 오류 상황에서 fallback UI가 왜 성능 최적화의 일부라고 볼 수 있나요?

### 합격 답변 예시

사용자 체감 성능은 단순히 응답 시간이 아니라, 기다리는 동안 무엇을 이해할 수 있는지도 포함하기 때문입니다. 네트워크 오류 상황에서 아무 반응 없는 화면보다, 현재 상태와 다음 행동을 알려주는 fallback UI가 있으면 사용자는 덜 답답하게 느낍니다. 즉 **fallback UI는 장애 대응이면서 동시에 체감 성능 최적화**이기도 합니다.

### 가산점 포인트

**"성능은 숫자만이 아니다"**라는 관점이 핵심이다. 3초 동안 빈 화면을 보는 것과, 3초 동안 skeleton + 진행 표시를 보는 것은 같은 3초지만 체감이 완전히 다르다.

### 상세 해설

```tsx
// React Error Boundary + fallback
function ErrorFallback({ error, resetErrorBoundary }: FallbackProps) {
  return (
    <div className="p-8 text-center">
      <p className="text-zinc-600">데이터를 불러올 수 없습니다</p>
      <p className="mt-2 text-sm text-zinc-400">{error.message}</p>
      <button
        onClick={resetErrorBoundary}
        className="mt-4 rounded bg-zinc-900 px-4 py-2 text-white"
      >
        다시 시도
      </button>
    </div>
  );
}

// 사용
<ErrorBoundary FallbackComponent={ErrorFallback}>
  <Suspense fallback={<Skeleton />}>
    <DataSection />
  </Suspense>
</ErrorBoundary>
```

#### 체감 성능 비교

```
❌ 에러 시 빈 화면:
  사용자: "고장 났나? 새로고침해야 하나? 다른 페이지로 가야 하나?"

✅ 에러 시 fallback UI:
  사용자: "아, 네트워크 문제구나. 다시 시도 버튼이 있네."
  → 같은 에러 상황이지만 스트레스가 훨씬 적음
```

> fallback을 알았으니, 다음에는 **무거운 UI 요소** — 차트/시각화의 성능 관리를 봐야 한다.

## 17. Chart.js 같은 시각화가 많은 화면에서 성능 이슈를 어떻게 관리하겠습니까?

### 합격 답변 예시

데이터 가공 비용을 줄이기 위해 memoization을 하고, 보이지 않는 차트는 lazy mount하며, resize나 hover 이벤트를 과도하게 처리하지 않도록 제어합니다. 또 데이터 포인트 수를 줄이거나 sampling을 적용하고, 차트가 많다면 한 번에 모두 그리지 않고 우선순위를 두는 것도 중요합니다.

### 가산점 포인트

차트 성능의 핵심은 **"렌더 횟수 × 데이터 양"**이다. 데이터가 1만 포인트인 차트가 부모 리렌더마다 다시 그려지면 프레임이 뚝뚝 끊긴다. `React.memo` + `useMemo`로 데이터가 실제로 바뀔 때만 다시 그리게 해야 한다.

### 상세 해설

```tsx
// 차트 성능 최적화
const ChartSection = React.memo(function ChartSection({ data }: Props) {
  // 데이터 변환을 memoize — 데이터가 안 바뀌면 재계산 안 함
  const chartData = useMemo(() => {
    // 1만 포인트 → 100 포인트로 sampling
    return downsample(data, 100);
  }, [data]);

  return <Line data={chartData} options={chartOptions} />;
});

// viewport에 보일 때만 차트 마운트
function LazyChart({ data }: Props) {
  const [visible, setVisible] = useState(false);
  const ref = useRef<HTMLDivElement>(null);

  useEffect(() => {
    const observer = new IntersectionObserver(([e]) => {
      if (e.isIntersecting) setVisible(true);
    });
    if (ref.current) observer.observe(ref.current);
    return () => observer.disconnect();
  }, []);

  return (
    <div ref={ref} className="h-64">
      {visible ? <ChartSection data={data} /> : <Skeleton className="h-64" />}
    </div>
  );
}
```

> 차트를 알았으니, 다음에는 **애니메이션이 많은 화면에서의 렌더링 원칙**을 봐야 한다.

## 18. 애니메이션이 많은 화면에서 성능을 해치지 않으려면 어떤 CSS/렌더링 원칙을 지켜야 하나요?

### 합격 답변 예시

가능하면 transform과 opacity 중심으로 애니메이션하고, layout을 자주 바꾸는 속성은 피하는 것이 좋습니다. 또 레이아웃 값을 읽고 바로 스타일을 다시 쓰는 패턴을 줄여 강제 동기 레이아웃을 피해야 합니다. 즉 애니메이션이 많은 화면에서는 예쁜 효과보다, **layout thrashing을 피하고 영향 범위를 좁히는 것**이 먼저입니다.

### 가산점 포인트

**"compositor-only 속성"**을 활용하면 GPU에서 처리되어 메인 스레드를 막지 않는다. `transform`과 `opacity`가 대표적이다. `width`, `height`, `top`, `left` 등은 layout을 다시 계산해야 하므로 비싸다.

### 상세 해설

#### 브라우저 렌더링 비용 순서

```
Layout (Reflow) — 가장 비쌈
  width, height, margin, padding, top, left, font-size...
  → 주변 요소까지 다시 계산

Paint (Repaint) — 중간
  color, background, box-shadow, border-radius...
  → 해당 요소만 다시 그림

Composite — 가장 저렴
  transform, opacity
  → GPU에서 처리, 메인 스레드 안 막음
```

```css
/* ❌ layout 애니메이션: 매 프레임마다 reflow */
.box {
  transition: left 0.3s, top 0.3s;
}

/* ✅ composite 애니메이션: GPU 처리, 60fps 유지 */
.box {
  transition: transform 0.3s;
  will-change: transform;  /* GPU 레이어 힌트 */
}
```

```ts
// ❌ Layout Thrashing: 읽기-쓰기 반복
elements.forEach(el => {
  const height = el.offsetHeight;     // 읽기 → layout 계산
  el.style.height = height + 10 + 'px'; // 쓰기 → layout 무효화
  // 다음 루프에서 다시 읽기 → 또 layout 계산 (강제 동기 레이아웃)
});

// ✅ 읽기 모아서 → 쓰기 모아서
const heights = elements.map(el => el.offsetHeight); // 읽기 한번에
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px'; // 쓰기 한번에
});
```

#### 자주 하는 오해

- "`will-change`를 모든 요소에 붙이면 좋다" → **오히려 메모리 낭비.** GPU 레이어가 많아지면 메모리 소비가 늘어난다. 실제로 애니메이션하는 요소에만 적용해야 한다.

> 애니메이션을 알았으니, 다음에는 **측정의 한계** — Lighthouse와 현실의 차이를 봐야 한다.

## 19. Lighthouse 점수와 실제 사용자 체감 성능이 다를 수 있는 이유는 무엇인가요?

### 합격 답변 예시

Lighthouse는 통제된 환경에서 측정한 실험실 지표이고, 실제 사용자는 기기 성능, 네트워크 품질, 캐시 상태, 로그인 상태, 데이터 양이 다 다르기 때문입니다. 그래서 점수는 참고 기준으로 유용하지만, 실제 사용자 데이터와 함께 봐야 진짜 체감 성능을 이해할 수 있습니다.

### 가산점 포인트

Lighthouse의 **throttling은 시뮬레이션**이다. 실제 모바일 기기의 CPU, GPU, 메모리 제약을 완벽히 재현하지 못한다. CrUX(Chrome User Experience Report)나 web-vitals 라이브러리로 **실사용자 데이터(RUM)**를 수집해야 진짜 체감을 볼 수 있다.

### 상세 해설

| | Lighthouse (Lab) | 실사용자 (RUM) |
|---|---|---|
| **환경** | 통제된 네트워크/CPU | 다양한 기기/네트워크 |
| **데이터** | 테스트용 최소 데이터 | 실제 사용자 데이터 (수천 건) |
| **캐시** | cold start (캐시 없음) | warm visit 가능 |
| **재현성** | 높음 | 낮음 (편차 큼) |
| **용도** | 문제 탐지, A/B 비교 | 실제 체감 확인 |

```ts
// RUM 데이터 수집 예시
import { onLCP, onINP, onCLS } from 'web-vitals';

function sendMetric(metric: { name: string; value: number }) {
  // 실사용자 데이터를 analytics로 전송
  navigator.sendBeacon('/api/metrics', JSON.stringify({
    name: metric.name,
    value: metric.value,
    url: window.location.pathname,
    device: navigator.userAgent,
    connection: (navigator as any).connection?.effectiveType,
  }));
}

onLCP(sendMetric);
onINP(sendMetric);
onCLS(sendMetric);
```

#### 실무 에피소드

> Lighthouse 95점인데 실사용자 p75 LCP가 4초였다. 원인: 실사용자의 API 응답에 포함된 데이터가 테스트보다 10배 많았고, 이미지도 훨씬 많았다. Lab 테스트는 "빈 게시판"이었지만 실사용자는 "이미지 100개 게시판"이었던 셈.

> Lighthouse의 한계를 알았으니, 마지막으로 **실전 경험을 기반으로 성능 최적화를 종합 정리**해보자.

## 20. 네가 수행한 영상 로딩 최적화나 흐름 단절 완화 작업을 성능 관점에서 설명해보세요.

### 합격 답변 예시

이 질문은 경험과 연결해 답하면 매우 강합니다. 예를 들어 큰 mp4를 그대로 전달하던 구조를 적응형 스트리밍으로 바꾸면 초기 재생 지연과 중간 버퍼링을 줄일 수 있고, fallback UI와 재시도 설계를 함께 넣으면 느린 환경에서도 사용자가 흐름을 잃지 않게 만들 수 있습니다. 즉 성능 최적화는 단순히 초 단위 개선이 아니라, **사용자가 중간에 이탈하지 않게 만드는 경험 설계**까지 포함한다고 답하면 좋습니다.

### 가산점 포인트

**"숫자만 말하기보다, 무엇이 느렸고 → 어떤 구조를 바꿨고 → 사용자가 무엇을 덜 잃게 됐는지"**를 말해야 한다. "LCP를 3초에서 1.5초로 줄였습니다"보다 "영상 시작 대기 시간이 20초에서 2초로 줄어, 교육 영상 이탈률이 40%에서 15%로 떨어졌습니다"가 훨씬 강하다.

### 상세 해설

#### 성능 최적화 스토리텔링 구조

```
1. 문제 정의: "무엇이 느렸는가"
   → 100MB mp4를 그대로 서빙, 학교 Wi-Fi에서 재생 시작까지 20초

2. 원인 분석: "왜 느렸는가"
   → 단일 파일 전체 다운로드 필요, adaptive bitrate 불가, CDN 미적용

3. 해결 방법: "어떤 구조를 바꿨는가"
   → DASH 세그먼트 전환, CDN 적용, fallback UI + retry

4. 결과: "사용자가 무엇을 덜 잃게 됐는가"
   → 재생 시작 2초, 버퍼링 80% 감소, 이탈률 40% → 15%
```

#### 자주 하는 오해

- "성능 개선은 기술적 숫자만 말하면 된다" → **비즈니스 임팩트를 연결해야.** "TTFB 200ms 단축"보다 "결제 전환율 5% 향상"이 면접관에게 더 와닿는다.

## 보완내용

### 왜 보완이 필요한가

기존 성능 20문항은 Core Web Vitals, 번들, 코드 스플리팅, 이미지/비디오, 리스트, 캐시, 체감 성능까지 잘 들어가 있어서 매우 좋다. 다만 성능 면접에서 한 단계 더 올라가려면 "측정 기반으로 병목을 찾는 능력"과 "브라우저 렌더링 경로를 구조적으로 설명하는 능력"이 추가로 필요하다.

특히 INP, Critical Rendering Path, long task, RUM처럼 실제 운영과 연결되는 질문을 더하면 체크리스트 암기형과 분석형 답변을 더 잘 나눌 수 있다.

### 추가하면 좋은 질문

#### 1. INP가 왜 중요해졌고, 어떤 상황에서 나빠지나요?

기존 Core Web Vitals 질문을 더 깊게 파는 질문이다. 긴 task, 무거운 이벤트 핸들러, 메인 스레드 점유와 연결해서 설명할 수 있으면 좋다.

##### 합격 답변 예시

INP는 FID를 대체한 반응성 지표로, 페이지 전체 생명주기에서 가장 느렸던 상호작용을 기준으로 측정합니다. FID는 첫 입력의 지연만 봤기 때문에 이후 인터랙션에서 버벅거려도 잡지 못했지만, INP는 클릭, 키 입력, 탭 등 모든 상호작용 중 최악의 응답 지연을 반영합니다. INP가 나빠지는 대표 상황은 메인 스레드를 오래 점유하는 Long Task가 있을 때, 이벤트 핸들러 안에서 무거운 계산이나 동기 DOM 조작이 일어날 때, 그리고 서드파티 스크립트가 메인 스레드를 막고 있을 때입니다.

##### 상세 해설

- INP가 중요해진 이유
  - FID는 첫 입력만 측정 → 페이지 후반부 버벅거림을 놓침
  - INP는 전체 상호작용 중 최악을 잡음 → 실사용자 체감에 더 가까움
  - 2024년부터 Core Web Vitals에서 FID를 공식 대체
- INP가 나빠지는 상황
  - 긴 이벤트 핸들러 (데이터 가공, DOM 대량 조작)
  - Long Task로 메인 스레드 점유
  - 무거운 서드파티 스크립트
  - hydration 중 입력 처리 지연
- 개선 방향
  - 핸들러를 최소화하고 무거운 작업은 분리
  - `requestAnimationFrame`, `requestIdleCallback` 활용
  - 입력 후 시각 피드백을 먼저 주고 처리를 나중에

#### 2. Critical Rendering Path를 설명해보세요.

HTML 파싱, DOM/CSSOM, Render Tree, Layout/Paint 흐름을 구조적으로 설명하게 만드는 문항이다.

##### 합격 답변 예시

Critical Rendering Path는 브라우저가 HTML을 받아서 화면에 픽셀을 그리기까지 거치는 핵심 경로입니다. HTML을 파싱해 DOM을 만들고, CSS를 파싱해 CSSOM을 만든 뒤, 둘을 합쳐 Render Tree를 구성합니다. 이후 Layout → Paint → Composite로 화면에 그립니다. 이 과정 중 어디서 차단이 일어나느냐에 따라 FCP와 LCP가 크게 달라집니다.

##### 상세 해설

- 핵심 흐름
  - HTML 파싱 → DOM 생성
  - CSS 파싱 → CSSOM 생성
  - DOM + CSSOM → Render Tree
  - Layout → Paint → Composite
- 차단 요소
  - CSS: 렌더 차단 (CSSOM 완성까지 Render Tree 못 만듦)
  - 동기 JS: 파싱 차단 (DOM 생성을 멈춤)
- 최적화 전략
  - critical CSS 인라인
  - 비핵심 CSS는 media query 분리 또는 preload
  - JS는 async/defer 활용
  - TTFB 단축으로 전체 시작 시점 앞당기기

#### 3. 웹폰트 최적화는 어떻게 하겠습니까?

CLS와 체감 성능에 직접 영향을 주는 주제다.

##### 합격 답변 예시

웹폰트 최적화에서 가장 중요한 것은 폰트가 늦게 로드되면서 생기는 레이아웃 흔들림(CLS)과 텍스트 안 보이는 시간(FOIT/FOUT)을 줄이는 것입니다. `font-display: swap`이나 `optional`을 설정하고, 서브셋 폰트로 크기를 줄이고, woff2 포맷을 사용하고, `<link rel="preload">`로 일찍 요청하면 됩니다.

##### 상세 해설

- 문제 유형
  - FOIT: 폰트 로드까지 텍스트가 안 보임
  - FOUT: 기본 폰트에서 웹폰트로 교체되며 깜빡임
  - CLS: 폰트 교체 시 글자 크기 변화로 레이아웃 이동
- 핵심 최적화
  - `font-display: swap` 또는 `optional`
  - woff2 포맷 (압축 효율 최고)
  - 서브셋팅 (한글은 특히 효과가 큼 — 전체 파일 수 MB)
  - `<link rel="preload" as="font" crossorigin>`
  - 시스템 폰트 fallback 크기 맞추기 (`size-adjust`)

#### 4. Long Task를 어떻게 찾고 줄이겠습니까?

실행 시점 병목을 보는 사람인지 확인하는 질문이다.

##### 합격 답변 예시

Long Task는 메인 스레드를 50ms 이상 차단하는 작업입니다. Chrome DevTools Performance 탭에서 찾거나 `PerformanceObserver`로 수집합니다. 줄이려면 큰 작업을 청크로 쪼개고, Web Worker로 오프로드하고, 서드파티 스크립트를 지연 로드합니다.

##### 상세 해설

- Long Task 정의: 메인 스레드를 50ms 이상 차단하는 단일 작업
- 찾는 방법
  - DevTools Performance 탭 (빨간 삼각형 마커)
  - `PerformanceObserver({ type: 'longtask' })`
  - Lighthouse "Avoid long main-thread tasks" 항목
- 줄이는 방법
  - 작업 분할 (time slicing, `scheduler.yield()`)
  - Web Worker로 오프로드
  - 서드파티 스크립트 지연 로드
  - hydration 최적화

#### 5. RUM(실사용자 모니터링)과 실험실 측정(Lab)의 차이를 설명해보세요.

Lighthouse 질문을 더 운영적으로 확장하는 질문이다.

##### 합격 답변 예시

Lab은 Lighthouse처럼 통제된 환경에서 측정하고, RUM은 실제 사용자의 브라우저에서 수집합니다. Lab으로 문제를 찾고, RUM으로 실제 효과를 확인하는 방식으로 둘을 함께 써야 합니다.

##### 상세 해설

- Lab: 재현 가능, 비교 용이, 디버깅 편리 / 실사용자 환경 미반영
- RUM: 실제 체감 반영, p75/p90 분포 확인 / 노이즈 큼, 원인 분석 어려움
- 실무: 개발 단계에서 Lab, 배포 후 RUM으로 확인

#### 6. Service Worker를 성능 최적화에 어떻게 활용할 수 있나요?

오프라인 대응뿐 아니라 반복 로딩 최적화에 좋은 질문이다.

##### 합격 답변 예시

Service Worker는 브라우저와 네트워크 사이에서 요청을 가로채 캐시된 응답을 먼저 돌려줄 수 있어 반복 방문 시 로딩 속도를 크게 개선합니다. Cache First, Network First, Stale-While-Revalidate 등 전략별로 적용하면 됩니다.

##### 상세 해설

- 대표 캐시 전략
  - Cache First: 정적 자산에 적합
  - Network First: 자주 바뀌는 API에 적합
  - Stale-While-Revalidate: 빠른 응답 + 최신성 균형
- 성능 개선 포인트
  - 반복 방문 시 네트워크 요청 자체를 줄임
  - precache로 핵심 자산을 설치 시점에 미리 저장
  - 오프라인에서도 기본 UI와 최근 데이터 표시 가능
- 주의사항
  - 캐시 무효화 전략이 없으면 오래된 자산이 계속 보임
  - 버전 관리와 캐시 키 설계가 중요

### 이 섹션 보완 핵심

성능 섹션은 이미 강하지만, INP, 렌더링 경로, 웹폰트, long task, RUM, 서비스 워커까지 더하면 최적화 체크리스트 답변자와 측정-분석-개선형 답변자를 더 선명하게 구분할 수 있다.
