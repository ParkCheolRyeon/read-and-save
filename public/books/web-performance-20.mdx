# 09. 웹 성능 20문항

> Core Web Vitals, 렌더링 비용, 로딩 전략, 체감 성능을 묻는 20문항입니다.

## 9. 웹 성능 최적화 질문 20문항

## 총평

이 섹션은 숫자 몇 개 외우는 문제가 아니다. 면접관이 듣고 싶은 건 결국 이거다.

- 병목을 어디서 찾는가
- 무엇을 먼저 줄이는가
- 측정과 체감 성능을 어떻게 구분하는가
- 네트워크와 렌더링, UX를 함께 보느냐
- 즉 성능 최적화 답변은
- **“빨라지게 했습니다”**가 아니라

“왜 느렸고, 무엇을 줄였고, 사용자가 무엇을 덜 기다리게 됐는지”를 말해야 한다.

## 1. 웹 성능 최적화에서 가장 먼저 측정해야 할 지표는 무엇이라고 생각하나요?

### 합격 답변 예시

정답이 하나로 고정되진 않지만, 저는 사용자 체감 기준으로는 Core Web Vitals와 화면별 핵심 시나리오를 먼저 봅니다. 즉 LCP(Largest Contentful Paint), INP(Interaction to Next Paint), CLS(Cumulative Layout Shift) 같은 공통 지표를 보되, 실제로 우리 사용자 불만이 어디서 발생하는지도 함께 봐야 합니다. 성능 최적화는 점수 맞추기가 아니라, 사용자가 느리다고 느끼는 순간을 줄이는 작업이라고 생각합니다.

### 상세 해설

이 질문에서 “LCP입니다”만 말하면 약하다. 좋은 답변은 지표와 사용자 경험을 같이 본다고 말해야 한다.

#### 핵심

- 공통 지표 필요
- 하지만 제품별 핵심 시나리오도 중요
- 실제 사용자가 느끼는 병목이 우선

## 2. Core Web Vitals를 설명해보세요.

| 지표 | 보는 축 | 짧은 해석 |
| --- | --- | --- |
| `LCP` | 로딩 | 주요 콘텐츠가 언제 보였는가 |
| `CLS` | 시각 안정성 | 화면이 얼마나 흔들렸는가 |
| `INP` | 반응성 | 입력 후 다음 페인트까지 얼마나 걸렸는가 |

### 합격 답변 예시

Core Web Vitals는 사용자 경험을 대표하는 핵심 웹 성능 지표입니다. LCP는 주요 콘텐츠가 얼마나 빨리 보이는지, CLS는 레이아웃이 얼마나 안정적인지, INP는 사용자 상호작용에 얼마나 빠르게 반응하는지를 봅니다. 즉 로딩, 시각 안정성, 반응성의 세 축을 측정한다고 이해하면 됩니다.

### 상세 해설

- LCP
- Largest Contentful Paint
- 가장 중요한 콘텐츠가 언제 보였는가
- CLS
- Cumulative Layout Shift
- 화면이 얼마나 흔들렸는가
- INP
- Interaction to Next Paint
- 입력 후 반응이 얼마나 빨랐는가

좋은 답변은 각 지표가 왜 중요한지까지 말해야 한다.

## 3. LCP를 개선하는 방법을 설명해보세요.

### 합격 답변 예시

LCP를 개선하려면 사용자가 가장 먼저 보는 큰 콘텐츠가 빨리 나오도록 해야 합니다. 대표적으로 큰 이미지나 비디오 최적화, 서버 응답 시간 단축, 렌더 차단 리소스 축소, CDN(Content Delivery Network) 활용, preload나 preconnect 같은 우선순위 제어가 중요합니다. 즉 첫 화면에서 가장 중요한 요소가 무엇인지 먼저 찾고, 그 요소가 늦어지는 원인을 줄이는 방식으로 접근해야 합니다.

### 상세 해설

- LCP는 대개
- hero 이미지
- 큰 제목 블록
- 대표 배너
- 비디오 썸네일

같은 것에서 잡힌다.

#### 개선 포인트

- 이미지 사이즈/포맷 최적화
- 캐시
- CSS/JS 차단 감소
- HTML에 핵심 콘텐츠 포함
- 서버 응답 개선

#### 핵심

“LCP를 개선한다”는 건 가장 중요한 것을 먼저 보여주는 것이다.

## 4. CLS가 발생하는 대표 원인과 해결 방법을 설명해보세요.

### 합격 답변 예시

CLS는 이미지, 광고, iframe처럼 늦게 들어오는 요소의 공간이 미리 확보되지 않았을 때 자주 발생합니다. 또 웹폰트가 늦게 적용되면서 텍스트 크기가 바뀌거나, 상단 배너가 나중에 삽입될 때도 흔들림이 생길 수 있습니다. 해결하려면 width/height나 aspect-ratio를 명시해 공간을 확보하고, 동적으로 삽입되는 요소를 위한 자리도 미리 마련해야 합니다.

### 상세 해설

- 대표 원인
- 크기 없는 이미지
- 동적 광고
- 늦게 뜨는 배너
- 폰트 교체
- iframe 자리 미확보
- 핵심 해결
- 공간 예약
- 안정된 레이아웃
- 늦게 바뀌는 요소 최소화

## 5. JavaScript 번들 크기를 줄이는 방법을 설명해보세요.

### 합격 답변 예시

불필요한 라이브러리를 줄이고, 코드 스플리팅과 dynamic import를 활용하고, 트리 쉐이킹이 잘 되도록 구조를 잡는 것이 기본입니다. 또 locale 데이터나 큰 유틸 패키지처럼 숨어 있는 무거운 의존성을 줄이고, 클라이언트에서만 필요한 코드만 번들에 포함되도록 경계를 잘 나누는 것이 중요합니다. 즉 번들 크기 최적화는 코드를 줄이는 게 아니라, 사용자에게 지금 필요한 것만 보내는 전략에 가깝습니다.

### 상세 해설

- 대표 방법
- 큰 라이브러리 교체
- route-based splitting
- feature-based lazy loading
- tree shaking
- 중복 의존성 제거
- client boundary 최소화

#### 좋은 답변 포인트

“압축하면 된다” 수준이 아니라 전송 자체를 줄인다고 말해야 한다.

## 6. 코드 스플리팅은 무엇이고, 어떤 기준으로 분리하나요?

### 합격 답변 예시

코드 스플리팅은 애플리케이션 코드를 한 번에 다 보내지 않고, 라우트나 기능 단위로 나눠 필요한 시점에 로드하는 전략입니다. 기준은 초기 진입에 꼭 필요한지, 특정 화면이나 모달, 차트, 에디터처럼 나중에만 필요한지입니다. 너무 잘게 쪼개면 요청 수와 관리 복잡도가 늘 수 있으니, 사용자 흐름 단위로 나누는 편이 보통 좋습니다.

### 상세 해설

#### 좋은 분리 기준

- 라우트 단위
- 무거운 기능 단위
- rarely used UI
- admin only 영역
- 나쁜 분리
- 너무 잘게 쪼개서 네트워크 조각만 많아짐
- 실제로 항상 같이 필요한데 인위적으로 분리

## 7. 이미지 최적화에서 고려할 요소들을 설명해보세요.

### 합격 답변 예시

이미지 최적화에서는 포맷, 크기, 반응형 전달, lazy loading, 캐시, CDN, CLS 방지가 핵심입니다. 즉 단순히 압축만 하는 게 아니라, 화면 크기에 맞는 이미지를 보내고, 늦게 로드해도 레이아웃이 흔들리지 않게 하고, 재사용 가능한 자산은 캐시하는 것까지 포함합니다. 결국 이미지 최적화는 전송량과 표시 안정성을 함께 다루는 문제입니다.

### 상세 해설

#### 핵심 체크리스트

- webp/avif
- 적절한 dimensions
- srcset / sizes
- lazy load
- placeholder
- CLS 방지
- CDN 캐시

## 8. 비디오 최적화는 이미지 최적화와 무엇이 다른가요?

### 합격 답변 예시

비디오는 이미지보다 훨씬 크고, 연속 재생과 seek, 버퍼링, 화질 적응 같은 문제가 있어서 최적화 관점이 더 복잡합니다. 이미지는 한 장을 얼마나 효율적으로 보여줄지가 핵심이라면, 비디오는 지속적으로 재생 가능한 흐름을 만드는 것이 핵심입니다. 그래서 포맷과 압축뿐 아니라 인코딩 프로파일, 세그먼트 전략, adaptive streaming까지 고려해야 합니다.

### 상세 해설

- 이미지 vs 비디오 차이
- 이미지:
- 한 번 보여주면 끝
- 비디오:
- 계속 재생
- 구간 이동
- 네트워크 변화 대응 필요

즉 비디오는 전송 최적화 + 재생 경험 최적화가 함께 필요하다.

## 9. 단일 mp4 대신 MPEG-DASH 같은 스트리밍 방식을 쓰는 이유를 설명해보세요.

### 합격 답변 예시

MPEG-DASH 같은 스트리밍 방식은 영상을 작은 세그먼트로 나누고, 네트워크 상태에 따라 적절한 비트레이트를 선택해 재생할 수 있기 때문에 단일 mp4보다 훨씬 유연합니다. 그래서 초기 재생이 빠르고, 중간 버퍼링이 줄고, 느린 환경에서도 완전히 끊기기보다 낮은 화질로라도 이어서 볼 수 있습니다. 즉 스트리밍은 단순 포맷 선택이 아니라, 불안정한 네트워크에서 재생 경험을 지키기 위한 전략입니다.

### 상세 해설

이 질문은 네 프로젝트 경험과 아주 잘 맞는다.

#### 장점

- adaptive bitrate
- 빠른 시작
- seek 유리
- 네트워크 적응성
- 버퍼링 감소

## 10. 렌더 차단 리소스가 무엇인지 설명해보세요.

### 합격 답변 예시

렌더 차단 리소스는 브라우저가 초기 화면을 그리기 전에 반드시 처리해야 해서 화면 표시를 늦추는 리소스입니다. 대표적으로 동기 스크립트와 중요한 CSS가 있습니다. 성능 최적화에서는 첫 화면에 꼭 필요한 것만 먼저 남겨두고, 나머지는 지연 로딩하거나 우선순위를 낮추는 것이 중요합니다.

### 상세 해설

- 대표 차단 리소스
- blocking CSS
- sync script
- 큰 폰트/스타일 의존

#### 핵심

사용자가 아무것도 못 보는 시간이 길어지면 체감 성능이 급격히 나빠진다.

## 11. lazy loading은 어디까지 적용하는 게 좋은가요?

### 합격 답변 예시

첫 화면에 꼭 필요하지 않은 이미지, 컴포넌트, 모달, 차트, 에디터, 비디오 등에 적용하는 것이 좋습니다. 반대로 첫 화면 핵심 콘텐츠까지 lazy로 미루면 오히려 LCP와 체감 성능이 나빠질 수 있습니다. 즉 lazy loading은 많이 쓰는 게 중요한 게 아니라, 늦게 와도 되는 것만 늦게 가져오는 것이 중요합니다.

### 상세 해설

좋은 답변은 lazy loading을 만능처럼 말하지 않는다.

- 적용하면 좋은 대상
- below-the-fold 이미지
- 관리자 기능
- 차트/에디터
- rarely opened modal
- 피해야 할 대상
- hero image
- 초기 CTA
- 첫 화면 핵심 텍스트/미디어

## 12. React에서 불필요한 리렌더링을 어떻게 찾아내고 줄이나요?

### 합격 답변 예시

먼저 React DevTools Profiler나 렌더 로그로 어떤 컴포넌트가 왜 자주 렌더되는지 확인합니다. 그 다음 상태 소유 범위를 줄이고, memoization이 실제로 의미 있는 곳에만 React.memo, useMemo, useCallback을 적용하고, selector 기반 구독이나 상태 분리를 통해 영향 범위를 좁힙니다. 핵심은 memo를 많이 붙이는 게 아니라, 리렌더링 원인을 먼저 찾는 것입니다.

### 상세 해설

- 원인 찾기
- 부모 리렌더
- props 새 참조
- context 변화
- store 전체 구독
- state 위치가 너무 위
- 줄이는 법
- state localize
- stable props
- memo strategically
- virtualization
- selector

## 13. 리스트가 긴 화면에서 성능 저하를 줄이는 방법을 설명해보세요.

### 합격 답변 예시

가장 대표적인 방법은 가상화로, 화면에 보이는 항목만 렌더하고 나머지는 필요할 때만 그리는 방식입니다. 또 pagination이나 infinite scroll, 아이템 단위 memoization, 이미지 lazy loading, 무거운 셀 구조 단순화도 함께 사용할 수 있습니다. 핵심은 수천 개를 한 번에 모두 DOM에 올리지 않는 것입니다.

### 상세 해설

- 중요한 방법
- virtualization
- pagination
- incremental render
- item memoization

#### 실무 포인트

리스트 문제는 데이터 양 문제이면서 동시에 DOM 수 문제다.

## 14. 캐싱 전략이 성능에 미치는 영향을 설명해보세요.

### 합격 답변 예시

캐싱은 같은 데이터를 다시 계산하거나 다시 요청하지 않게 해서 네트워크 지연과 서버 부하, 렌더 지연을 줄입니다. 브라우저 캐시, CDN 캐시, API 응답 캐시, 클라이언트 데이터 캐시 모두 체감 성능에 영향을 줍니다. 다만 캐시는 항상 stale 문제와 invalidation 문제를 같이 가져오기 때문에, 단순히 많이 캐시하는 것보다 어떤 데이터를 얼마나 신뢰할 수 있는지까지 설계해야 합니다.

### 상세 해설

좋은 답변은 “캐시 = 빠름”에서 끝나지 않는다.

- 반드시 같이 말해야 할 것
- stale risk
- invalidation
- consistency trade-off

## 15. 모바일 환경, 특히 학교 Wi-Fi처럼 불안정한 환경에서 체감 성능을 높이는 방법은 무엇인가요?

### 합격 답변 예시

## 16. 네트워크 오류 상황에서 fallback UI가 왜 성능 최적화의 일부라고 볼 수 있나요?

### 합격 답변 예시

사용자 체감 성능은 단순히 응답 시간이 아니라, 기다리는 동안 무엇을 이해할 수 있는지도 포함하기 때문입니다. 네트워크 오류 상황에서 아무 반응 없는 화면보다, 현재 상태와 다음 행동을 알려주는 fallback UI가 있으면 사용자는 덜 답답하게 느낍니다. 즉 fallback UI는 장애 대응이면서 동시에 체감 성능 최적화이기도 합니다.

### 상세 해설

성능은 숫자만이 아니다. 기다림을 견딜 수 있게 만드는 것도 성능 설계다.

- 실제 데이터 크기
- 로그인/권한 상태 차이

## 20. 네가 수행한 영상 로딩 최적화나 흐름 단절 완화 작업을 성능 관점에서 설명해보세요.

### 합격 답변 예시

이 질문은 네 경험과 연결해 답하면 매우 강합니다. 예를 들어 큰 mp4를 그대로 전달하던 구조를 적응형 스트리밍으로 바꾸면 초기 재생 지연과 중간 버퍼링을 줄일 수 있고, fallback UI와 재시도 설계를 함께 넣으면 느린 환경에서도 사용자가 흐름을 잃지 않게 만들 수 있습니다. 즉 성능 최적화는 단순히 초 단위 개선이 아니라, 사용자가 중간에 이탈하지 않게 만드는 경험 설계까지 포함한다고 답하면 좋습니다.

### 상세 해설

- 숫자만 말하기보다,
- 무엇이 느렸고
- 어떤 구조를 바꿨고
- 사용자가 무엇을 덜 잃게 됐는지

를 말해야 한다.

## 보완내용

### 왜 보완이 필요한가

기존 성능 20문항은 Core Web Vitals, 번들, 코드 스플리팅, 이미지/비디오, 리스트, 캐시, 체감 성능까지 잘 들어가 있어서 매우 좋다. 다만 성능 면접에서 한 단계 더 올라가려면 “측정 기반으로 병목을 찾는 능력”과 “브라우저 렌더링 경로를 구조적으로 설명하는 능력”이 추가로 필요하다.

특히 INP, Critical Rendering Path, long task, RUM(Real User Monitoring)처럼 실제 운영과 연결되는 질문을 더하면 체크리스트 암기형과 분석형 답변을 더 잘 나눌 수 있다.

### 추가하면 좋은 질문

#### 1. INP가 왜 중요해졌고, 어떤 상황에서 나빠지나요?

기존 Core Web Vitals 질문을 더 깊게 파는 질문이다. 긴 task, 무거운 이벤트 핸들러, 메인 스레드 점유와 연결해서 설명할 수 있으면 좋다.

##### 합격 답변 예시

INP는 FID(First Input Delay)를 대체한 반응성 지표로, 페이지 전체 생명주기에서 가장 느렸던 상호작용을 기준으로 측정합니다. FID는 첫 입력의 지연만 봤기 때문에 이후 인터랙션에서 버벅거려도 잡지 못했지만, INP는 클릭, 키 입력, 탭 등 모든 상호작용 중 최악의 응답 지연을 반영합니다. INP가 나빠지는 대표 상황은 메인 스레드를 오래 점유하는 Long Task가 있을 때, 이벤트 핸들러 안에서 무거운 계산이나 동기 DOM 조작이 일어날 때, 그리고 서드파티 스크립트가 메인 스레드를 막고 있을 때입니다. 즉 INP를 개선하려면 이벤트 핸들러를 가볍게 만들고, 무거운 작업은 비동기로 분리하거나 requestIdleCallback, Web Worker 등을 활용해야 합니다.

##### 상세 해설

- INP가 중요해진 이유
  - FID는 첫 입력만 측정 → 페이지 후반부 버벅거림을 놓침
  - INP는 전체 상호작용 중 최악을 잡음 → 실사용자 체감에 더 가까움
  - 2024년부터 Core Web Vitals에서 FID를 공식 대체
- INP가 나빠지는 상황
  - 긴 이벤트 핸들러 (데이터 가공, DOM 대량 조작)
  - Long Task로 메인 스레드 점유
  - 무거운 서드파티 스크립트
  - hydration 중 입력 처리 지연
- 개선 방향
  - 핸들러를 최소화하고 무거운 작업은 분리
  - `requestAnimationFrame`, `requestIdleCallback` 활용
  - 입력 후 시각 피드백을 먼저 주고 처리를 나중에

#### 2. Critical Rendering Path를 설명해보세요.

HTML 파싱, DOM/CSSOM, Render Tree, Layout/Paint 흐름을 구조적으로 설명하게 만드는 문항이다. 렌더 차단 리소스 질문을 더 큰 틀로 묶어준다.

##### 합격 답변 예시

Critical Rendering Path는 브라우저가 HTML을 받아서 화면에 픽셀을 그리기까지 거치는 핵심 경로입니다. HTML을 파싱해 DOM을 만들고, CSS를 파싱해 CSSOM을 만든 뒤, 둘을 합쳐 Render Tree를 구성합니다. 이후 각 요소의 위치와 크기를 계산하는 Layout 단계를 거쳐, 최종적으로 Paint와 Composite로 화면에 그립니다. 성능 최적화에서 이 경로가 중요한 이유는, 이 과정 중 어디서 차단이 일어나느냐에 따라 FCP(First Contentful Paint)와 LCP가 크게 달라지기 때문입니다. CSS는 렌더 차단 리소스이고, 동기 스크립트는 파싱 차단 리소스입니다. 그래서 critical CSS를 인라인하고, 비핵심 CSS와 JS는 async나 defer로 늦추는 것이 기본 전략입니다.

##### 상세 해설

- 핵심 흐름
  - HTML 파싱 → DOM 생성
  - CSS 파싱 → CSSOM 생성
  - DOM + CSSOM → Render Tree
  - Layout → Paint → Composite
- 차단 요소
  - CSS: 렌더 차단 (CSSOM 완성까지 Render Tree 못 만듦)
  - 동기 JS: 파싱 차단 (DOM 생성을 멈춤)
- 최적화 전략
  - critical CSS 인라인
  - 비핵심 CSS는 media query 분리 또는 preload
  - JS는 async/defer 활용
  - TTFB(Time To First Byte) 단축으로 전체 시작 시점 앞당기기

#### 3. 웹폰트 최적화는 어떻게 하겠습니까?

의외로 많이 빠지는 주제지만 CLS와 체감 성능에 직접 영향을 준다. `font-display`, preload, 서브셋팅 같은 포인트를 설명할 수 있으면 좋다.

##### 합격 답변 예시

웹폰트 최적화에서 가장 중요한 것은 폰트가 늦게 로드되면서 생기는 레이아웃 흔들림(CLS)과 텍스트 안 보이는 시간(FOIT/FOUT)을 줄이는 것입니다. 먼저 `font-display: swap`이나 `optional`을 설정해 폰트가 늦어도 텍스트는 먼저 보이게 합니다. 다음으로 사용하는 글자만 포함하는 서브셋 폰트를 만들어 파일 크기를 줄이고, woff2 포맷을 사용합니다. 또 `<link rel="preload">`로 폰트를 일찍 요청하면 렌더링 시점에 폰트가 이미 준비되어 있을 확률이 높아집니다. 즉 웹폰트 최적화는 안 보이는 시간을 줄이고, 바뀌는 순간을 줄이고, 전송량을 줄이는 세 방향으로 접근해야 합니다.

##### 상세 해설

- 문제 유형
  - FOIT (Flash of Invisible Text): 폰트 로드까지 텍스트가 안 보임
  - FOUT (Flash of Unstyled Text): 기본 폰트에서 웹폰트로 교체되며 깜빡임
  - CLS: 폰트 교체 시 글자 크기 변화로 레이아웃 이동
- 핵심 최적화
  - `font-display: swap` 또는 `optional`
  - woff2 포맷 사용 (압축 효율 최고)
  - 서브셋팅 (한글은 특히 효과가 큼)
  - `<link rel="preload" as="font" crossorigin>`
  - 시스템 폰트 fallback 크기 맞추기 (`size-adjust`)
- 실무 포인트
  - 한글 폰트는 전체 파일이 수 MB → 서브셋 필수
  - 폰트 CDN(Google Fonts 등)도 preconnect하면 효과적

#### 4. Long Task를 어떻게 찾고 줄이겠습니까?

프론트엔드 성능 문제에서 매우 실무적인 질문이다. 번들 크기만이 아니라 실제 실행 시점 병목을 보는 사람인지 확인할 수 있다.

##### 합격 답변 예시

Long Task는 메인 스레드를 50ms 이상 차단하는 작업을 말합니다. 찾는 방법으로는 Chrome DevTools Performance 탭에서 빨간색으로 표시되는 Long Task를 확인하거나, `PerformanceObserver`로 `longtask` 엔트리를 수집할 수 있습니다. 줄이는 방법은 크게 세 가지입니다. 첫째, 큰 작업을 `requestIdleCallback`이나 `setTimeout`으로 청크 단위로 쪼갭니다. 둘째, 무거운 계산은 Web Worker로 메인 스레드 밖에서 처리합니다. 셋째, 불필요한 서드파티 스크립트를 제거하거나 지연 로드합니다. 즉 Long Task를 줄이는 건 번들 크기를 줄이는 것과 다른 축의 최적화로, 실행 시점의 병목을 관리하는 것입니다.

##### 상세 해설

- Long Task 정의
  - 메인 스레드를 50ms 이상 차단하는 단일 작업
  - 이 동안 사용자 입력, 애니메이션, 렌더링이 모두 멈춤
- 찾는 방법
  - DevTools Performance 탭 (빨간 삼각형 마커)
  - `PerformanceObserver({ type: 'longtask' })`
  - Lighthouse "Avoid long main-thread tasks" 항목
- 줄이는 방법
  - 작업 분할 (time slicing, `scheduler.yield()`)
  - Web Worker로 오프로드
  - 서드파티 스크립트 지연 로드
  - SSR(Server-Side Rendering) 또는 CSR(Client-Side Rendering) hydration 최적화
  - 무거운 초기화를 idle 시점으로 미루기

#### 5. RUM(실사용자 모니터링)과 실험실 측정(Lab)의 차이를 설명해보세요.

기존 Lighthouse 질문을 더 운영적으로 확장하는 질문이다. 점수와 실제 사용자 경험의 차이를 데이터를 기준으로 설명할 수 있어야 한다.

##### 합격 답변 예시

실험실 측정(Lab)은 Lighthouse처럼 통제된 환경에서 일관된 조건으로 측정하는 방식이고, RUM은 실제 사용자의 브라우저에서 수집한 성능 데이터입니다. Lab은 재현 가능하고 디버깅에 유리하지만, 실제 사용자의 기기 성능, 네트워크, 캐시 상태, 데이터 양을 반영하지 못합니다. RUM은 실제 사용자가 겪는 성능을 그대로 보여주지만, 편차가 크고 원인 분석이 어려울 수 있습니다. 그래서 좋은 성능 운영은 Lab으로 문제를 찾고 개선 효과를 검증하되, RUM으로 실제 사용자에게 효과가 있었는지를 확인하는 방식으로 둘을 함께 씁니다.

##### 상세 해설

- Lab 측정
  - Lighthouse, WebPageTest, DevTools 등
  - 장점: 재현 가능, 비교 용이, 디버깅 편리
  - 단점: 실사용자 환경 미반영, 캐시/로그인 상태 다름
- RUM (Real User Monitoring)
  - CrUX, web-vitals 라이브러리, 자체 수집 등
  - 장점: 실제 사용자 체감 반영, p75/p90 분포 확인 가능
  - 단점: 노이즈 큼, 원인 분석 어려움
- 실무 활용
  - 개발 단계: Lab으로 빠르게 피드백
  - 배포 후: RUM으로 실제 영향 확인
  - 둘을 같이 봐야 점수와 체감의 괴리를 메울 수 있음

#### 6. Service Worker를 성능 최적화에 어떻게 활용할 수 있나요?

오프라인 대응뿐 아니라 반복 로딩 최적화, 캐시 전략, 네트워크 우회 설계를 연결하기 좋은 질문이다.

##### 합격 답변 예시

Service Worker는 브라우저와 네트워크 사이에서 요청을 가로채 캐시된 응답을 먼저 돌려줄 수 있기 때문에, 반복 방문 시 로딩 속도를 크게 개선할 수 있습니다. 대표적인 캐시 전략으로 Cache First는 캐시에 있으면 바로 반환하고, Network First는 최신 데이터를 먼저 시도한 뒤 실패하면 캐시를 쓰고, Stale-While-Revalidate는 캐시를 먼저 보여주면서 백그라운드에서 갱신합니다. 또 오프라인 fallback 페이지를 제공하거나, 정적 자산을 precache해서 첫 로딩 이후에는 네트워크 없이도 앱이 동작하게 만들 수 있습니다. 즉 Service Worker는 오프라인 대응만이 아니라, 네트워크 불안정 환경에서 체감 성능을 지키는 캐시 레이어로 활용할 수 있습니다.

##### 상세 해설

- 대표 캐시 전략
  - Cache First: 정적 자산(이미지, 폰트, CSS/JS)에 적합
  - Network First: 자주 바뀌는 API 응답에 적합
  - Stale-While-Revalidate: 빠른 응답 + 최신성 균형
- 성능 개선 포인트
  - 반복 방문 시 네트워크 요청 자체를 줄임
  - precache로 핵심 자산을 설치 시점에 미리 저장
  - 오프라인에서도 기본 UI와 최근 데이터 표시 가능
- 주의사항
  - 캐시 무효화 전략이 없으면 오래된 자산이 계속 보임
  - 버전 관리와 캐시 키 설계가 중요
  - 디버깅이 어려울 수 있으므로 DevTools Application 탭 활용

### 이 섹션 보완 핵심

성능 섹션은 이미 강하지만, INP, 렌더링 경로, 웹폰트, long task, RUM, 서비스 워커까지 더하면 최적화 체크리스트 답변자와 측정-분석-개선형 답변자를 더 선명하게 구분할 수 있다.
