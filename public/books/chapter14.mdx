# Part 1. JavaScript / TypeScript 기본기

## 1. stale closure가 왜 생기고, React에서는 왜 더 자주 문제 되나요?

### 모범 답변

stale closure는 함수가 생성될 당시의 렉시컬 환경을 기억하기 때문에 발생합니다.  
자바스크립트 자체의 클로저 특성인데, React에서는 렌더마다 state와 props가 새로운 스냅샷처럼 취급되기 때문에 더 자주 체감됩니다.

예를 들어 `setTimeout`, `setInterval`, 비동기 요청 콜백, 이벤트 리스너 안에서 과거 렌더 시점의 값을 계속 참조할 수 있습니다.  
그래서 화면에서는 이미 최신 상태인데, 콜백 내부에서는 이전 값을 보고 있는 문제가 생깁니다.

저는 이 문제를 해결할 때 항상 먼저 "이 값이 최신이어야 하는가, 이전 값 기반 업데이트면 충분한가"를 구분합니다.  
이전 값 기반이면 함수형 업데이트를 사용하고, 최신 값을 렌더와 무관하게 참조해야 하면 `useRef`를 사용합니다.  
그리고 effect 내부 의존성이 빠져서 closure가 낡는 경우에는 dependency를 다시 설계합니다.

### 꼬리 질문

#### `useRef`는 왜 stale closure 문제를 줄이는 데 도움이 되나요?

- `ref.current`는 렌더 간에 같은 객체를 유지하므로, 콜백이 오래 유지돼도 최신 값을 참조할 수 있습니다.
- state처럼 값 자체가 렌더 스냅샷으로 복제되는 개념이 아니라, 가변 컨테이너를 참조합니다.
- 단, `ref`는 UI 갱신을 일으키지 않으므로 화면 반영이 목적일 때는 state를 써야 합니다.

#### dependency array를 비우면 왜 위험할 수 있나요?

- 첫 렌더 시점 값만 캡처한 effect가 계속 유지되기 때문에 외부 값이 바뀌어도 effect 내부는 모를 수 있습니다.
- 특히 이벤트 등록, interval, subscription에서 오래된 값을 계속 들고 있는 문제가 잘 생깁니다.
- "한 번만 실행"이 목적이 아니라 "한 번만 등록"인지 구분해서 써야 합니다.

#### `useCallback`을 쓰면 stale closure가 자동으로 해결되나요?

- 아닙니다. `useCallback`은 함수 참조를 재사용하는 도구이지, 최신 값을 자동 보장하는 도구는 아닙니다.
- dependency가 잘못되면 오히려 오래된 값을 더 오래 유지할 수 있습니다.
- 핵심은 함수 메모이제이션이 아니라, 어떤 값을 언제 캡처하는지 이해하는 것입니다.

## 2. React에서 왜 불변성이 중요하다고 보나요?

### 모범 답변

불변성이 중요한 이유는 React가 상태 변화와 props 변화를 판단할 때 기본적으로 **참조 동일성**을 많이 활용하기 때문입니다.  
객체나 배열을 직접 mutate하면 내부 데이터는 바뀌어도 참조가 그대로일 수 있어서, 변경 감지가 불안정해지고 최적화 포인트가 무너질 수 있습니다.

예를 들어 리스트 항목을 직접 수정하면 memoization이나 selector 기반 구독 최적화가 깨질 수 있습니다.  
반면 변경이 있는 경로만 새 객체로 만들고 나머지는 재사용하면, 어떤 단위가 바뀌었는지 경계가 명확해집니다.

저는 불변성을 단순 스타일 규칙이 아니라, **예측 가능한 상태 관리와 최적화의 전제조건**으로 봅니다.  
특히 복잡한 UI에서는 "어디가 바뀌었는지"를 추적할 수 있어야 디버깅이 쉬워지고, 리렌더 범위도 통제하기 쉬워집니다.

### 꼬리 질문

#### 깊은 복사는 항상 좋은가요?

- 아닙니다. 깊은 복사는 비용이 크고 불필요한 참조 변경을 많이 만들어 리렌더를 늘릴 수 있습니다.
- 실무에서는 변경이 발생한 경로만 얕게 복사하는 구조가 보통 더 효율적입니다.
- 그래서 상태 구조를 평평하게 설계하는 것이 중요합니다.

#### Immer를 쓰는 이유는 무엇인가요?

- 불변 업데이트를 직접 펼치면 코드가 길어지고 실수가 늘어나기 때문에 생산성을 높이기 위해 씁니다.
- draft를 mutate하는 형태로 쓰지만 내부적으로는 불변 구조를 만들어줍니다.
- 다만 모든 상태에 무조건 적용하기보다, 복잡한 중첩 업데이트가 많은 곳에 선택적으로 쓰는 편이 낫습니다.

#### 불변성과 렌더링 최적화는 어떻게 연결되나요?

- 참조가 바뀐 지점을 기준으로 memo, selector, shallow compare가 동작하기 때문입니다.
- 불변성이 깨지면 "실제로 바뀌었는지"를 안전하게 판단하기 어렵습니다.
- 결국 불변성은 최적화 자체보다, 최적화가 가능하게 만드는 기반입니다.

## 3. 비동기 실행 순서와 race condition을 실무 기준으로 설명해보세요.

### 모범 답변

자바스크립트는 단일 스레드에서 동작하지만, 비동기 작업 완료 시점은 이벤트 루프가 조정합니다.  
중요한 건 `async/await`가 동기 실행으로 바꿔주는 게 아니라, Promise 흐름을 읽기 쉽게 만든다는 점입니다.

실무에서 더 중요한 건 race condition입니다.  
예를 들어 검색창에서 `a`, `ab`, `abc` 순으로 요청을 보냈는데 응답은 `ab`가 제일 늦게 돌아오면, 오래된 응답이 최신 화면을 덮어쓸 수 있습니다.

저는 이런 문제를 해결할 때 3단계로 봅니다.  
첫째, 이전 요청을 `AbortController`로 취소할 수 있는가.  
둘째, 취소가 어렵다면 request id를 두고 **가장 최신 요청만 반영**할 수 있는가.  
셋째, optimistic UI나 캐시까지 있으면 **서버 상태와 로컬 상태의 우선순위**를 어떻게 둘지 정합니다.

### 꼬리 질문

#### Promise와 `setTimeout` 중 무엇이 먼저 실행되나요?

- 일반적으로 현재 call stack이 비워진 뒤에는 microtask인 Promise 콜백이 먼저 실행됩니다.
- `setTimeout`은 macrotask 큐에서 다음 턴에 실행됩니다.
- 그래서 `Promise.then`이 보통 `setTimeout(..., 0)`보다 먼저 실행됩니다.

#### `AbortController`는 언제 특히 유용한가요?

- 검색 자동완성, 필터 변경, 페이지 이탈처럼 이전 요청이 의미를 잃는 경우에 유용합니다.
- 네트워크 비용과 불필요한 상태 갱신을 줄일 수 있습니다.
- 다만 서버가 이미 처리 중인 작업 자체를 완전히 무효화하는 건 별도 보장이 필요합니다.

#### retry는 어떤 요청에나 적용해도 되나요?

- 아닙니다. 멱등성(idempotent)이란 같은 요청을 몇 번 보내도 결과가 동일한 성질이다. GET, PUT, DELETE는 멱등이고 POST는 아니다. 멱등성이 없는 요청에 무분별하게 retry를 걸면 중복 처리 문제가 생길 수 있습니다.
- 네트워크 불안정이나 일시적 5xx에는 유효하지만, 4xx나 검증 실패에는 보통 맞지 않습니다.
- 요청 성격별로 retry 정책을 분리해야 합니다.

## 4. `any`, `unknown`, `never`를 어떻게 구분하나요?

### 모범 답변

`any`는 타입 검사를 포기하는 선택이라서, 정말 이행 단계가 아니면 최대한 피하는 편입니다.  
한 번 `any`가 들어오면 타입 안정성이 연쇄적으로 무너질 수 있기 때문입니다.

`unknown`은 외부 입력을 안전하게 다룰 때 유용합니다.
일단 모른다고 선언하고, 실제 사용 전에 narrowing이나 런타임 검증을 거치게 강제합니다.
Narrowing(타입 좁히기)이란 넓은 타입(예: `unknown`)을 런타임 검사(`typeof`, `instanceof`, 판별 유니온 등)를 통해 좁은 타입으로 정밀하게 좁혀나가는 과정이다.

`never`는 절대 도달하지 않는 상태를 표현할 때 씁니다.  
예를 들어 판별 유니온 분기에서 모든 케이스를 처리했는지 확인하는 exhaustive check에 자주 씁니다.

제 기준으로는 **외부 경계는 `unknown`**, **내부 도메인 타입은 명확하게 모델링**, **빠진 분기 검증은 `never`**가 가장 실무적입니다.

### 꼬리 질문

#### `unknown`이 `any`보다 좋은 이유는 무엇인가요?

- `unknown`은 사용 전에 타입을 좁히게 강제하므로, 무분별한 접근을 막아줍니다.
- 반면 `any`는 모든 연산을 허용해서 런타임 오류 가능성을 타입 시스템이 막지 못합니다.
- 외부 입력이 많을수록 `unknown`이 훨씬 안전합니다.

#### `never`를 실무에서 어디에 쓰나요?

- 대표적으로 switch 문에서 모든 분기를 처리했는지 확인할 때 씁니다.
- 예상 불가능한 상태에 도달했을 때 컴파일 단계에서 누락을 잡는 데 유용합니다.
- 복잡한 UI 상태 모델이나 reducer 분기에서 특히 효과적입니다.

#### `any`를 완전히 금지해야 하나요?

- 완전 금지보다는, 사용 이유와 범위를 매우 제한하는 것이 현실적입니다.
- 마이그레이션 중 임시 브리지 구간에서는 쓸 수 있지만, 빠르게 제거 대상이어야 합니다.
- 팀 차원에서는 lint 규칙과 코드리뷰 기준이 중요합니다.

## 5. 제네릭과 판별 유니온을 UI 상태 모델링에 어떻게 활용하나요?

### 모범 답변

제네릭은 "구조는 같지만 내부 타입만 달라지는 경우"에 재사용성을 높여줍니다.  
예를 들어 API 응답 래퍼, 테이블 컬럼 정의, 폼 필드 설정처럼 공통 구조가 반복될 때 유용합니다.

판별 유니온은 "상태가 여러 단계로 나뉘고, 각 단계마다 가능한 필드가 다른 경우"에 강합니다.  
예를 들어 `idle | loading | success | error`처럼 상태별로 접근 가능한 속성을 타입으로 제한할 수 있습니다.

저는 UI 상태를 다룰 때 boolean 여러 개로 흩어놓기보다, 가능하면 판별 유니온으로 모델링하는 편입니다.  
그렇게 하면 불가능한 조합을 타입 수준에서 막을 수 있고, 렌더 분기와 에러 처리도 더 명확해집니다.

### 꼬리 질문

#### boolean 여러 개보다 판별 유니온이 나은 이유는 무엇인가요?

- `isLoading`과 `isError`가 동시에 `true`가 되는 식의 모순 상태를 줄일 수 있습니다.
- 상태 전이 가능한 경우의 수를 더 명확하게 제한할 수 있습니다.
- 렌더 분기와 타입 narrowing이 자연스럽게 연결됩니다.

#### 제네릭을 과하게 쓰면 어떤 문제가 생기나요?

- 추상화가 과해지면 사용처에서 타입 추론이 어려워지고 가독성이 떨어질 수 있습니다.
- 특히 3중, 4중 제네릭은 팀 전체 생산성을 해칠 수 있습니다.
- 저는 "반복되는 구조가 명확할 때만" 제네릭을 올리는 편입니다.

#### 상태 머신까지 가야 하나요?

- 모든 UI에 필요한 건 아닙니다.
- 다만 상태 전이가 복잡하고, 액션/가드/사이드이펙트가 많다면 상태 머신이 큰 도움이 됩니다.
- 워크플로우 에디터나 멀티스텝 폼 같은 경우는 검토 가치가 큽니다.

## 6. TypeScript만으로 충분하지 않고 런타임 검증이 필요한 이유는 무엇인가요?

### 모범 답변

TypeScript는 컴파일 타임 도구이기 때문에, 실제 런타임에 들어오는 외부 데이터의 진위를 보장해주지는 못합니다.  
특히 API 응답, URL 파라미터, 로컬스토리지, 서드파티 **SDK(Software Development Kit, 소프트웨어 개발 키트)** 입력값은 타입 선언과 실제 값이 다를 수 있습니다.

그래서 저는 **내부 타입 안정성**과 **외부 입력 검증**을 분리해서 생각합니다.  
외부 입력은 `unknown`으로 받고, Zod 같은 런타임 스키마 검증을 통해 신뢰 가능한 도메인 타입으로 변환하는 흐름이 안전합니다.

이 방식의 장점은 에러를 더 앞단에서 잡을 수 있고, 잘못된 데이터가 앱 깊숙이 퍼지는 걸 막는 데 있습니다.  
특히 AI 응답이나 유저 생성 데이터처럼 스키마 일관성이 약한 경우에 효과가 큽니다.

### 꼬리 질문

#### Zod 같은 라이브러리를 어디에 적용하나요?

- API 응답 파싱, 폼 검증, URL query parsing, 환경변수 검증에 주로 씁니다.
- 특히 앱 초기 설정값이나 외부 연동 구간에서 효과가 큽니다.
- 모든 곳에 적용하기보다 경계 지점에 집중하는 편이 좋습니다.

#### 백엔드가 타입을 보장하면 프론트 검증은 필요 없나요?

- 여전히 필요합니다. 네트워크, 버전 차이, 누락된 필드, 예외 응답은 언제든 발생할 수 있습니다.
- 프론트는 사용자에게 가장 가까운 방어선이라, 최소한의 안정성 검증이 있어야 합니다.
- 특히 운영 중에는 계약 위반 데이터를 완전히 믿고 들어오면 장애 전파가 빨라집니다.

#### 런타임 검증 비용이 큰 것 아닌가요?

- 비용은 있지만, 오류 전파와 디버깅 비용보다 훨씬 작은 경우가 많습니다.
- 모든 응답 전체를 매번 깊게 검증하기보다, 핵심 경계와 중요한 도메인에 집중하는 전략이 현실적입니다.
- 성능과 안정성의 균형을 잡는 것이 중요합니다.

# Part 2. React 렌더링 / 상태 구조 / 복잡한 UI

## 7. React를 왜 선언형이라고 부르나요?

### 모범 답변

React를 선언형이라고 부르는 이유는 "어떻게 DOM을 직접 바꿀지"보다 "현재 상태에서 UI가 무엇이어야 하는지"를 기술하기 때문입니다.  
개발자는 원하는 UI 상태를 선언하고, 실제 DOM 업데이트 계산은 React가 담당합니다.

명령형 방식에서는 DOM 탐색, 클래스 토글, 이벤트별 수동 동기화 코드가 늘어나기 쉽습니다.  
반면 React에서는 state나 props가 바뀌면, 그 결과 UI가 어떻게 보여야 하는지를 중심으로 코드를 작성합니다.

저는 선언형의 장점이 단순 코드량 감소가 아니라, **상태와 UI의 대응 관계를 읽기 쉽게 만드는 것**이라고 봅니다.  
특히 인터랙션이 많은 제품에서 수동 DOM 조작을 줄일수록 예측 가능성이 올라갑니다.

### 꼬리 질문

#### 선언형이면 DOM 제어를 전혀 안 해도 되나요?

- 아닙니다. 포커스 이동, 스크롤 제어, canvas, 측정 같은 영역은 여전히 명령형 제어가 필요합니다.
- 다만 그 범위를 좁히고, 상태 중심 UI를 기본으로 두는 것이 React의 강점입니다.
- 명령형 제어는 예외 처리로 다루는 게 좋습니다.

#### React의 선언형이 성능상 불리한가요?

- 무조건 불리하다고 보진 않습니다.
- 잘못 설계한 상태 구조는 오히려 비효율적일 수 있지만, React의 diff와 메모이제이션 전략을 활용하면 충분히 효율적으로 운영할 수 있습니다.
- 중요한 건 선언형 자체보다 상태 구조와 렌더 범위 통제입니다.

#### 선언형과 추상화는 어떤 관계인가요?

- 선언형은 의도를 더 높은 수준에서 표현하게 만들어 추상화와 잘 맞습니다.
- 다만 추상화가 과하면 실제 상태 흐름이 숨겨져 디버깅이 어려워질 수 있습니다.
- 그래서 UI 추상화는 항상 사용처의 명확성을 기준으로 잡는 편이 좋습니다.

## 8. React의 render phase와 commit phase를 어떻게 이해하고 있나요?

### 모범 답변

React는 크게 render phase와 commit phase로 나눠서 이해하고 있습니다.  
render phase에서는 어떤 UI가 필요할지 계산하고, commit phase에서는 실제 DOM 반영과 effect 실행이 일어납니다.

render phase는 중단되거나 다시 계산될 수 있습니다.  
그래서 render 중에는 순수해야 하고, 외부 상태를 바꾸거나 부수효과를 일으키면 안 됩니다.

commit phase는 실제 변경이 확정된 뒤이기 때문에 DOM 접근이나 effect 실행이 가능합니다.  
이 차이를 이해하고 있으면 "왜 render 내부에서 side effect를 두면 안 되는지", "`useEffect`와 `useLayoutEffect`가 왜 commit 이후에 동작하는지"를 자연스럽게 설명할 수 있습니다.

### 꼬리 질문

#### render phase가 여러 번 실행될 수 있다는 건 무슨 의미인가요?

- React가 최종 반영 전에 여러 후보 UI를 계산할 수 있다는 뜻입니다.
- 특히 Strict Mode나 concurrent 환경에서는 더 자주 체감됩니다.
- 그래서 render는 결과 계산만 해야 하고, 외부 세계를 건드리면 안 됩니다.

#### commit phase에서는 어떤 일이 일어나나요?

- 실제 DOM 반영, ref 연결, layout effect 실행, 이후 passive effect 실행이 이어집니다.
- 즉 사용자 눈에 보이는 변경은 commit을 기준으로 생각하는 편이 정확합니다.
- 측정이나 DOM 의존 로직은 이 시점을 기준으로 다뤄야 합니다.

#### Strict Mode에서 effect가 두 번 도는 걸 어떻게 보나요?

- 개발 환경에서 순수성 검증과 잠재적 버그 탐지를 돕기 위한 동작으로 이해합니다.
- "React가 이상하다"가 아니라 effect가 idempotent한지 점검할 기회라고 봅니다.
- 클린업 로직이 정확한지 확인하는 데도 도움이 됩니다.

## 9. state ownership은 어떻게 잡나요?

### 모범 답변

state ownership은 "이 상태를 누가 가장 잘 알고, 누가 변경 책임을 가져야 하는가"를 기준으로 잡습니다.  
단순히 props drilling이 싫다고 전역으로 올리면 오히려 변경 경계가 흐려지고 리렌더 범위가 커질 수 있습니다.

저는 상태를 크게 세 가지로 나눕니다.

1. 특정 컴포넌트 안에서만 의미가 있는 **로컬 UI 상태**
2. 여러 하위 컴포넌트가 공유하지만 특정 기능 범위 안에 머무는 **도메인 상태**
3. 서버에서 가져오고 캐시, 동기화, 재요청 정책이 필요한 **서버 상태**

핵심은 "가까운 곳에 둘 수 있으면 최대한 가깝게 두고, 공유 필요가 명확할 때만 올린다"입니다.  
복잡한 UI일수록 상태를 많이 올리는 것보다, **책임과 범위가 분명한 상태 설계**가 더 중요하다고 생각합니다.

### 꼬리 질문

#### props drilling이 무조건 나쁜가요?

- 아닙니다. 데이터 흐름이 명확하고 깊이가 얕다면 가장 단순하고 읽기 쉬운 방식일 수 있습니다.
- 너무 이른 전역화보다 명시적 전달이 더 나은 경우도 많습니다.
- 문제는 깊이 자체보다, 변경 책임이 불명확해지는 지점입니다.

#### 상태를 너무 낮게 두면 어떤 문제가 생기나요?

- 서로 다른 하위 컴포넌트 간 동기화가 어려워지고, 부모가 전체 흐름을 통제하기 어려워질 수 있습니다.
- 공통 정책을 적용할 때 중복 로직이 늘 수 있습니다.
- 그래서 "공유 필요성"과 "변경 빈도"를 같이 봐야 합니다.

#### 서버 상태와 클라이언트 상태를 왜 구분하나요?

- 서버 상태는 비동기, 캐시, 재검증, 동기화 같은 문제가 포함되기 때문입니다.
- 일반 state 관리 방식으로 다루면 stale data나 중복 요청 문제가 쉽게 생깁니다.
- 그래서 React Query 같은 전용 도구를 별도로 쓰는 것이 합리적입니다.

## 10. Context, Zustand, React Query를 어떻게 구분해서 사용하나요?

### 모범 답변

저는 세 도구를 같은 "전역 상태"로 보지 않고 역할이 다른 도구로 구분합니다.

`Context`는 주로 테마, locale, 권한 정보처럼 **비교적 드물게 바뀌고 트리 전반에서 읽어야 하는 값**에 적합합니다.  
상태 관리 자체보다 전달 메커니즘에 가깝다고 봅니다.

`Zustand`는 로컬 state보다 넓고, 서버 상태보다는 UI 쪽에 가까운 **클라이언트 도메인 상태**에 잘 맞습니다.  
예를 들어 에디터 선택 상태, 패널 열림 상태, 일시적인 작업 맥락 같은 데 유용합니다.

`React Query`는 서버 상태 전용으로 봅니다.  
fetch, cache, stale time, refetch, retry, invalidation 같은 concerns가 붙는 순간 일반 전역 상태보다 서버 상태 관리 도구가 적합합니다.

### 꼬리 질문

#### Context로 모든 상태를 관리하면 안 되나요?

- 가능은 하지만, 변경이 잦은 상태까지 넣으면 불필요한 리렌더와 구조 복잡도가 커질 수 있습니다.
- Context는 전달 도구에 가깝기 때문에, 자주 변하는 상태의 정교한 구독 최적화에는 불리합니다.
- 범위를 좁혀 쓰는 편이 안정적입니다.

#### Zustand의 장점은 무엇인가요?

- 보일러플레이트가 적고, selector 기반 구독으로 필요한 부분만 읽기 쉬운 점이 좋습니다.
- 복잡한 UI 도메인 상태를 가볍게 다루기에 적합합니다.
- 다만 무분별한 전역 store 확장은 경계 불분명 문제를 만들 수 있습니다.

#### React Query를 전역 상태처럼 쓰면 안 되나요?

- 서버 데이터 캐시 용도로는 좋지만, 순수 UI state까지 다 담는 건 역할이 맞지 않습니다.
- refetch와 cache lifecycle이 UI 의도와 충돌할 수 있습니다.
- 서버 상태와 클라이언트 상태의 관심사를 분리하는 것이 더 유지보수하기 쉽습니다.

## 11. `useEffect`와 `useLayoutEffect`를 어떻게 구분하나요?

### 모범 답변

두 훅 모두 commit 이후에 실행되지만, 시점과 목적이 다릅니다.  
`useEffect`는 브라우저 페인트 이후 비동기적으로 실행되기 때문에, 데이터 동기화나 이벤트 등록처럼 레이아웃 차단이 필요 없는 작업에 적합합니다.

반면 `useLayoutEffect`는 DOM이 반영된 직후, 브라우저가 화면을 그리기 전에 동기적으로 실행됩니다.  
그래서 측정 후 바로 스타일 보정이 필요한 경우, 깜빡임을 막아야 하는 경우에 적합합니다.

다만 `useLayoutEffect`는 렌더 경로를 막기 때문에 무분별하게 쓰지 않는 편입니다.  
제 기준으로는 **화면 깜빡임이나 측정 정확도가 핵심일 때만 `useLayoutEffect`**, 나머지는 기본적으로 `useEffect`를 사용합니다.

### 꼬리 질문

#### `useLayoutEffect`를 남용하면 왜 문제인가요?

- 브라우저 페인트 전에 동기적으로 실행되므로, 작업이 무거우면 화면 표시가 지연될 수 있습니다.
- 즉 체감 성능을 떨어뜨릴 수 있습니다.
- 필요한 경우에만 좁게 쓰는 것이 좋습니다.

#### 데이터 fetch를 `useLayoutEffect`에 두면 안 되나요?

- fetch 자체는 레이아웃 측정과 무관하므로 `useEffect`가 더 적절합니다.
- `useLayoutEffect`는 화면 표시 전에 꼭 처리해야 하는 DOM 관련 작업에 가깝습니다.
- 네트워크 요청은 보통 그 성격이 아닙니다.

#### SSR 환경에서 주의할 점이 있나요?

- 서버에는 실제 DOM이 없어서 `useLayoutEffect` 관련 경고나 의미 없는 실행 문제가 생길 수 있습니다.
- Next.js에서는 클라이언트 전용 구간인지 명확히 분리하는 편이 좋습니다.
- DOM 접근 훅은 브라우저 환경을 전제로 둬야 합니다.

## 12. 워크플로우 에디터 상태 구조를 어떻게 설계할 것인가요?

### 모범 답변

워크플로우 에디터는 일반 폼보다 상태가 훨씬 복잡하기 때문에, 저는 먼저 상태를 **영속 상태와 파생 상태**로 나눕니다.

영속 상태에는 노드, 엣지, 노드별 설정값, 저장 가능한 메타데이터를 둡니다.  
파생 상태에는 현재 선택 노드, hover 상태, 드래그 중 좌표, 연결 프리뷰, 임시 validation 결과처럼 매 순간 바뀌는 UI 상태를 둡니다.

그리고 가능하면 노드/엣지는 정규화해서 관리합니다.  
예를 들어 `nodesById`, `edgesById`, `nodeOrder`처럼 나누면 부분 업데이트와 추적이 쉬워집니다.

추가로 undo/redo, autosave, collaborative editing 가능성까지 고려하면 "무엇이 저장 대상인지" 경계가 명확해야 합니다.  
저는 이런 에디터에서는 처음부터 **서버 저장 모델과 화면 인터랙션 모델을 섞지 않는 것**이 가장 중요하다고 봅니다.

실제 구현에서는 react-flow 같은 라이브러리를 검토할 수 있습니다.
react-flow는 노드/엣지 렌더링, 줌/팬, 연결선 드래그를 기본 제공하기 때문에 빠르게 프로토타이핑하기 좋습니다.
하지만 커스텀 노드 UI가 많거나, 상태 관리 방식을 팀 기준에 맞춰야 하거나, 성능 제어가 필요한 경우에는 라이브러리 위에 자체 상태 레이어를 얹거나, 부분적으로 직접 구현하는 판단도 필요합니다.
저는 "라이브러리로 시작하되, 상태 모델은 직접 설계한다"는 기준으로 접근하는 편입니다.

### 꼬리 질문

#### react-flow 같은 라이브러리를 쓸지, 직접 만들지 어떻게 판단하나요?

- 빠른 검증이 우선이면 라이브러리로 시작하는 편이 합리적입니다.
- 하지만 커스텀 인터랙션이 많거나 상태 구조를 완전히 제어해야 하면 라이브러리 위에 자체 레이어를 쌓거나 부분 직접 구현을 검토합니다.
- 핵심은 "렌더링은 라이브러리에 맡기되, 데이터 모델과 상태 흐름은 직접 설계한다"는 경계입니다.

#### 왜 정규화가 중요한가요?

- 일부 노드만 바뀌는 상황에서 전체 트리를 다시 순회하지 않아도 되기 때문입니다.
- 참조 안정성과 부분 업데이트가 쉬워집니다.
- 대형 그래프에서 성능과 디버깅 모두 유리합니다.

#### undo/redo는 어떻게 설계하나요?

- 전체 상태를 무조건 통째로 쌓기보다, 액션 기반 또는 변경 diff 기반으로 관리하는 방식을 먼저 검토합니다.
- 저장 비용과 복원 단위를 고려해야 합니다.
- 영속 상태와 일시적 UI 상태를 분리해두면 히스토리 관리가 훨씬 쉬워집니다.

#### DnD 중 상태는 어디에 두는 게 좋나요?

- 드래그 좌표나 hover 같은 초고빈도 UI 상태는 가능한 한 로컬 혹은 별도 store로 분리하는 편이 낫습니다.
- 저장 대상 도메인 상태와 같은 계층에 두면 리렌더 범위가 커질 수 있습니다.
- 상호작용 빈도가 높은 상태일수록 분리 전략이 중요합니다.

# Part 3. Next.js / 서버 경계 / 인증 / 네트워크

## 13. SSR, SSG, ISR, CSR를 어떻게 선택하나요?

### 모범 답변

저는 렌더링 전략을 기술 선호가 아니라 **데이터 특성, SEO, 초기 응답 속도, 사용자 개인화 정도**로 판단합니다.

정적이고 자주 안 바뀌는 콘텐츠는 SSG가 유리합니다.  
주기적 재생성이 가능하면 ISR이 운영 효율과 성능의 균형이 좋습니다.

요청마다 최신 데이터가 중요하거나 인증된 사용자별 결과가 달라지면 SSR이 적합할 수 있습니다.  
반면 로그인 이후의 대시보드처럼 SEO 중요도가 낮고 인터랙션 중심이면 CSR도 충분히 합리적입니다.

Next.js에서는 이걸 페이지 단위가 아니라, 이제는 **서버/클라이언트 경계와 데이터 fetch 위치**까지 같이 봐야 한다고 생각합니다.  
결국 질문은 "이 데이터를 언제, 어디서, 누구 기준으로 그릴 것인가"입니다.

### 꼬리 질문

#### SSR이 항상 더 좋은 사용자 경험을 주나요?

- 아닙니다. 초기 HTML은 빠를 수 있어도 서버 응답 지연이 크면 전체 체감이 나빠질 수 있습니다.
- 상호작용 중심 화면은 CSR이 더 단순하고 유리할 수 있습니다.
- 데이터 특성에 맞는 선택이 중요합니다.

#### ISR은 어떤 상황에 특히 잘 맞나요?

- 콘텐츠가 자주 바뀌지만 매 요청마다 실시간일 필요는 없는 경우에 적합합니다.
- 마케팅 페이지, 문서, 공개 리스트형 화면에서 유용합니다.
- 빌드 부담과 최신성 요구 사이 균형을 잡기 좋습니다.

#### 하나의 앱에서 전략을 혼합해도 되나요?

- 오히려 그렇게 하는 것이 일반적입니다.
- 공개 페이지, 인증 페이지, 실시간 페이지는 요구사항이 서로 다릅니다.
- 화면 특성별로 전략을 다르게 가져가는 것이 자연스럽습니다.

## 14. Server Component와 Client Component의 경계를 어떻게 잡나요?

### 모범 답변

기본적으로 저는 **상호작용이 없는 UI와 데이터 fetch는 서버 쪽**, **이벤트 처리와 브라우저 API가 필요한 부분은 클라이언트 쪽**으로 시작합니다.

Server Component의 장점은 번들 크기를 줄이고, 서버 자원에 가까운 곳에서 데이터 fetch를 하며, 클라이언트에 불필요한 코드 전송을 줄일 수 있다는 점입니다.  
반면 버튼 클릭, 입력 제어, 애니메이션, 브라우저 API 접근은 Client Component가 필요합니다.

중요한 건 경계를 너무 크게 잡지 않는 것입니다.  
상위 전체를 클라이언트로 만들면 Server Component 이점을 잃고, 반대로 너무 잘게 쪼개면 구조 복잡도가 올라갑니다.

저는 대체로 **페이지 뼈대와 정적 영역은 서버**, **작은 인터랙션 섬들만 클라이언트**로 두는 방식을 선호합니다.

### 꼬리 질문

#### 서버 컴포넌트 안에서 상태 훅을 못 쓰는 이유는 무엇인가요?

- 서버 컴포넌트는 브라우저 상호작용 컨텍스트가 아니라, 서버에서 렌더 결과를 만드는 역할이기 때문입니다.
- 사용자 이벤트 루프와 연결된 로컬 상태 개념이 맞지 않습니다.
- 그래서 인터랙션이 필요한 부분은 클라이언트 경계로 나눠야 합니다.

#### 클라이언트 컴포넌트를 너무 넓게 잡으면 어떤 문제가 생기나요?

- 번들 크기가 커지고, 서버 컴포넌트의 데이터/성능 이점을 잃습니다.
- 렌더 비용과 hydration 비용도 함께 증가할 수 있습니다.
- "필요한 최소 인터랙션 단위"로 클라이언트화하는 것이 좋습니다.

#### 경계를 너무 잘게 나누면 어떤 문제가 생기나요?

- 컴포넌트 간 데이터 전달과 이해 비용이 늘 수 있습니다.
- 협업 시 구조를 파악하기 어려워질 수 있습니다.
- 성능만 보지 말고 유지보수성까지 같이 판단해야 합니다.

## 15. Server Action과 API Route/Route Handler는 어떻게 구분하나요?

### 모범 답변

저는 둘을 대체 관계라기보다, 같은 mutation 문제를 푸는 다른 인터페이스로 봅니다.

Server Action은 Next.js 안에서 폼 액션이나 서버 로직 호출을 더 자연스럽게 연결할 수 있는 장점이 있습니다.  
특히 같은 앱 안에서 간단한 mutation을 처리할 때 보일러플레이트를 줄일 수 있습니다.

반면 API Route나 Route Handler는 외부 클라이언트와도 계약을 공유하거나, REST 엔드포인트 성격이 분명하거나, 캐시/인증/미들웨어 제어가 더 명시적이어야 할 때 적합합니다.

제 판단 기준은 이렇습니다.  
**이 앱 내부 UI에서만 쓰이고, 액션 중심 흐름이 자연스러우면 Server Action**,  
**명시적 HTTP 인터페이스가 중요하면 Route Handler**입니다.

### 꼬리 질문

#### Server Action이 무조건 더 최신 방식이라 항상 좋은가요?

- 아닙니다. 도구 성격이 다를 뿐이고, 외부 소비자나 명시적 API 계약이 필요하면 Route Handler가 더 적절할 수 있습니다.
- 팀의 디버깅/관찰 가능성 요구도 중요합니다.
- 유행보다 경계의 성격을 먼저 봐야 합니다.

#### API Route의 장점은 무엇인가요?

- HTTP 계층이 명시적이라 테스트, 관찰, 외부 연동, 도구 호환성이 좋습니다.
- 클라이언트가 앱 외부로 늘어날 가능성이 있으면 유리합니다.
- 계약 중심 설계가 필요한 팀에 잘 맞습니다.

#### Server Action을 쓸 때 주의할 점은 무엇인가요?

- 네트워크 경계가 덜 드러나서 호출 비용이나 실패 처리 감각이 흐려질 수 있습니다.
- 권한 체크, 검증, 에러 처리 위치를 명확히 해야 합니다.
- "함수처럼 보여도 서버 호출"이라는 점을 팀이 이해하고 있어야 합니다.

## 16. URL 상태를 왜 중요하게 보나요?

### 모범 답변

저는 URL을 단순 라우팅 정보가 아니라, **공유 가능하고 복원 가능한 상태 저장소**로 봅니다.  
특히 필터, 정렬, 검색어, 탭, 페이지네이션 같은 상태는 URL에 올릴 가치가 큽니다.

이렇게 하면 새로고침이나 공유, 뒤로가기/앞으로가기에서 상태를 자연스럽게 복원할 수 있습니다.  
반대로 전부 로컬 state에만 두면 사용자가 현재 화면 맥락을 잃기 쉽습니다.

물론 모든 상태를 URL에 넣지는 않습니다.  
모달 열림, hover, 일시적 드래그 좌표처럼 휘발성 UI 상태는 URL보다 로컬이 맞습니다.

결국 기준은 **사용자가 다시 들어왔을 때 유지되길 원하는가**, **공유 가치가 있는가**, **브라우저 히스토리와 연결될 의미가 있는가**입니다.

### 꼬리 질문

#### URL에 너무 많은 상태를 넣으면 문제 아닌가요?

- 맞습니다. 휘발성 상태나 민감 정보까지 넣으면 오히려 복잡도와 보안 위험이 올라갑니다.
- 공유/복원 가치가 있는 상태만 선별해서 넣는 것이 중요합니다.
- URL은 전체 상태 저장소가 아니라, 사용자 맥락 저장소에 가깝습니다.

#### 검색어와 필터는 왜 URL이 잘 맞나요?

- 북마크, 공유, 새로고침 복원이 모두 자연스럽기 때문입니다.
- 사용자는 같은 조건의 결과 화면을 다시 보고 싶어하는 경우가 많습니다.
- 브라우저 히스토리와도 잘 맞습니다.

#### URL 상태와 전역 상태는 어떻게 함께 쓰나요?

- URL은 외부 표현 계층, 전역 상태는 내부 동작 계층으로 나눠 생각합니다.
- URL 변경 시 전역 상태를 동기화하거나, 반대로 내부 상태를 URL로 직렬화할 수 있습니다.
- 단일 진실 공급원을 어디로 둘지 먼저 정해야 혼선이 줄어듭니다.

## 17. 쿠키, 세션, 토큰, 그리고 세션 만료 UX를 어떻게 이해하고 있나요?

### 모범 답변

인증 저장 방식은 기술 취향보다 **보안 요구사항과 운영 편의성**으로 선택해야 한다고 생각합니다.  
일반적으로 httpOnly cookie 기반 세션은 XSS 방어 측면에서 유리하고, 브라우저가 자동으로 보내준다는 장점이 있습니다.

토큰 기반 방식은 모바일, 외부 클라이언트, 마이크로서비스 환경에서는 유연할 수 있지만, 저장 위치와 갱신 전략을 잘못 잡으면 보안 리스크가 커집니다.

프론트엔드 입장에서는 저장 방식보다 **만료 UX**가 중요합니다.  
세션이 만료됐을 때 갑자기 데이터 저장이 실패하거나, 사용자가 작성 중인 내용을 잃어버리면 경험이 나빠집니다.

그래서 저는 만료를 감지하면 재인증을 유도하되, 가능하면 작성 중인 입력 상태는 보존하고, 저장 실패 이유를 명확히 보여주는 흐름이 좋다고 봅니다.

### 꼬리 질문

#### localStorage에 토큰을 저장하면 왜 위험하다고 하나요?

- XSS가 발생하면 스크립트가 토큰에 접근할 수 있기 때문입니다.
- 물론 httpOnly cookie도 만능은 아니고 CSRF 대응이 필요합니다.
- 저장 방식은 위협 모델을 기준으로 판단해야 합니다.

#### 세션 만료를 프론트에서 어떻게 감지하나요?

- 보통 401/419 같은 인증 실패 응답, 혹은 갱신 토큰 실패를 통해 감지합니다.
- 중요한 건 감지 이후의 UX로, 사용자 작업 손실을 최소화해야 합니다.
- 조용한 재시도와 명시적 재로그인 요구를 상황별로 나눕니다.

#### 세션 만료 중 폼 작성 내용은 어떻게 보호하나요?

- 가능하면 로컬 임시 저장이나 메모리 상태 유지로 작성값을 보존합니다.
- 재로그인 후 복원 가능한 흐름이 사용자 경험에 매우 중요합니다.
- 인증 실패가 발생해도 입력 상태까지 함께 날아가면 불만이 커집니다.

## 18. SSE, WebSocket, Polling 중 무엇을 고를 것인가요?

### 모범 답변

저는 먼저 요구사항을 **방향성, 빈도, 실시간성, 인프라 복잡도**로 나눠 봅니다.

서버에서 클라이언트로 단방향 스트리밍이 필요하고, 예를 들어 AI 응답 토큰 스트리밍처럼 서버 push가 핵심이면 SSE가 단순하고 적합할 수 있습니다.  
반면 양방향 통신이 필요하고, 협업 편집이나 실시간 상태 동기화가 중요하면 WebSocket이 더 잘 맞습니다.

Polling은 가장 단순하지만, 업데이트 빈도가 낮거나 실시간성이 덜 중요할 때 적합합니다.  
구현이 쉽고 운영 가시성도 좋지만, 불필요한 요청이 많아질 수 있습니다.

AI 챗 UI처럼 서버에서 생성 중인 응답을 순차적으로 보여주는 경우에는, 저는 먼저 SSE를 검토할 가능성이 높습니다.

### 꼬리 질문

#### SSE의 장점은 무엇인가요?

- 단방향 스트리밍에 단순하고 HTTP 기반이라 도입 장벽이 비교적 낮습니다.
- AI 응답 토큰 스트리밍 같은 패턴에 잘 맞습니다.
- 다만 양방향 통신에는 한계가 있습니다.

#### WebSocket은 언제 꼭 필요하다고 보나요?

- 클라이언트와 서버가 상호작용을 자주 주고받고, 상태 동기화가 짧은 주기로 필요한 경우입니다.
- 실시간 협업, presence, 멀티 유저 동시 편집 같은 시나리오가 대표적입니다.
- 운영 복잡도는 SSE보다 높아질 수 있습니다.

#### Polling이 여전히 유효한가요?

- 충분히 유효합니다. 요구사항 대비 가장 단순한 해결책일 수 있습니다.
- 실시간성이 낮고, 장애 대응이나 관찰 가능성이 중요하면 장점이 있습니다.
- 과한 기술 선택보다 요구사항에 맞는 단순함이 더 좋을 때가 많습니다.

# Part 4. AI 제품 프론트 / 성능 / 테스트 / 운영

## 19. 스트리밍 챗 UI를 어떻게 설계할 것인가요?

### 모범 답변

스트리밍 챗 UI에서는 단순히 메시지를 배열로 렌더하는 것보다, **메시지 생명주기**를 먼저 정의하는 것이 중요합니다.  
예를 들어 사용자 메시지, assistant placeholder, streaming 중간 상태, 완료 상태, 실패 상태를 구분합니다.

저는 보통 사용자 전송 직후 optimistic하게 사용자 메시지를 먼저 넣고, assistant 쪽은 빈 placeholder를 만든 뒤 스트리밍 토큰을 누적합니다.  
그리고 완료 시 최종 메시지로 확정하고, 실패 시 retry 가능한 상태를 남깁니다.

추가로 메시지 id, 요청 id, 중복 전송 방지, 취소 버튼, 자동 스크롤 정책도 중요합니다.  
긴 응답에서는 무조건 맨 아래로 끌고 가기보다, 사용자가 위를 읽는 중인지도 고려해야 합니다.

결국 챗 UI는 리스트 렌더링 문제가 아니라, **비동기 상태 머신과 UX 설계 문제**라고 생각합니다.

구체적으로 SSE 기반 스트리밍을 연결한다면 이런 흐름이 됩니다.

```tsx
// 메시지 상태 타입
type ChatMessage =
  | { role: "user"; content: string }
  | { role: "assistant"; status: "streaming"; content: string }
  | { role: "assistant"; status: "done"; content: string }
  | { role: "assistant"; status: "error"; content: string; error: string };

// SSE 연결 + 토큰 누적
async function streamChat(prompt: string, onToken: (token: string) => void) {
  const response = await fetch("/api/chat", {
    method: "POST",
    body: JSON.stringify({ prompt }),
    headers: { "Content-Type": "application/json" },
  });

  const reader = response.body!.getReader();
  const decoder = new TextDecoder();

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    onToken(decoder.decode(value));
  }
}

// 컴포넌트에서 사용
function ChatInput({ onSend }) {
  const handleSubmit = async (text: string) => {
    // 1. 사용자 메시지 즉시 추가 (optimistic)
    addMessage({ role: "user", content: text });
    // 2. assistant placeholder 추가
    const assistantId = addMessage({ role: "assistant", status: "streaming", content: "" });

    try {
      await streamChat(text, (token) => {
        // 3. 토큰 누적
        appendToMessage(assistantId, token);
      });
      // 4. 완료 상태로 전환
      finalizeMessage(assistantId, "done");
    } catch (err) {
      // 5. 실패 상태 + retry 가능
      finalizeMessage(assistantId, "error", err.message);
    }
  };
}
```

### 스트리밍 챗 UI 상태 머신

```
          사용자 전송
              │
              ▼
┌──────────────────┐
│   user_sent      │  사용자 메시지 표시
└────────┬─────────┘
         │ 서버 응답 시작
         ▼
┌──────────────────┐
│ assistant_       │  스트리밍 중... (타이핑 애니메이션)
│ streaming        │  청크마다 UI 업데이트
└──┬───────────┬───┘
   │           │
성공│           │에러
   ▼           ▼
┌────────┐  ┌────────┐
│  done  │  │ error  │  재시도 버튼 표시
└────────┘  └───┬────┘
                │ 재시도
                └──→ assistant_streaming으로 복귀
```

### 꼬리 질문

#### 스트리밍 중 실패하면 어떻게 처리하나요?

- assistant 메시지를 완전히 제거하기보다, "생성 중 실패" 상태를 남기고 재시도 UX를 주는 편이 좋습니다.
- 사용자가 이미 본 문맥을 잃지 않게 해야 합니다.
- 에러 메시지도 기술적 문구보다 사용자 관점으로 바꾸는 것이 좋습니다.

#### 자동 스크롤은 항상 아래로 보내야 하나요?

- 아닙니다. 사용자가 이전 대화를 읽는 중이면 무조건 아래로 끌고 가면 방해가 됩니다.
- "현재 하단 근처에 있을 때만 자동 스크롤" 같은 정책이 실무적입니다.
- UI는 편의와 통제권 균형이 중요합니다.

#### 메시지 상태를 어떻게 모델링하나요?

- `pending`, `streaming`, `done`, `error` 같은 판별 유니온 형태가 다루기 좋습니다.
- 메시지 배열과 별도 요청 상태를 분리하면 더 안정적입니다.
- 토큰 누적 중 상태와 최종 확정 상태를 구분하는 것이 중요합니다.

## 20. optimistic UI는 언제 쓰고, 실패 복구는 어떻게 보나요?

### 모범 답변

optimistic UI는 사용자가 기대하는 결과가 비교적 명확하고, 실패 확률이 낮으며, 실패 시 복구 전략을 설계할 수 있을 때 효과적입니다.  
예를 들어 좋아요, 즐겨찾기, 정렬 변경, 임시 드래그 저장 같은 경우에 잘 맞습니다.

하지만 금융 처리, 권한 변경, 결제처럼 실패 비용이 큰 작업은 더 신중해야 합니다.  
optimistic update 자체보다 중요한 건 실패했을 때 **화면을 어떻게 되돌리고, 사용자에게 무엇을 설명할지**입니다.

저는 optimistic UI를 도입할 때 항상 세 가지를 함께 봅니다.  
첫째, rollback 가능한가.  
둘째, 서버 응답이 늦거나 순서가 뒤집혀도 일관성을 유지할 수 있는가.  
셋째, 사용자가 중간 상태를 이해할 수 있는가.

### 꼬리 질문

#### rollback은 어떻게 구현하나요?

- 변경 전 상태를 잠깐 저장해두고 실패 시 복원하는 방식이 기본입니다.
- 다만 상태 규모가 크면 전체 복원보다 패치 단위 복원이 더 나을 수 있습니다.
- 서버 응답 순서 문제까지 같이 고려해야 합니다.

#### optimistic UI가 오히려 혼란을 줄 수도 있나요?

- 그렇습니다. 성공처럼 보였는데 나중에 실패하면 신뢰를 떨어뜨릴 수 있습니다.
- 그래서 중간 상태 표시와 실패 안내가 중요합니다.
- "빠르게 보이기"보다 "일관되게 이해되기"가 더 중요합니다.

#### 모든 mutation에 optimistic UI를 쓰면 안 되는 이유는 무엇인가요?

- 실패 비용과 데이터 무결성 영향이 작업마다 다르기 때문입니다.
- 사용자가 결과를 확신해도 되는 작업과 아닌 작업을 구분해야 합니다.
- UI 반응 속도보다 정확성이 우선인 도메인도 많습니다.

## 21. 성능을 어떻게 측정하고 개선하나요?

### 모범 답변

저는 성능 최적화를 시작할 때 먼저 "느리다"를 구체적 문제로 바꿉니다.  
초기 로딩이 느린지, 입력 반응이 느린지, 스크롤이 끊기는지, 특정 액션 후 지연이 큰지부터 나눕니다.

측정은 브라우저 DevTools, React Profiler, network waterfall, bundle 분석으로 시작합니다.  
그리고 렌더 문제인지, JS 실행 비용인지, 네트워크인지, 이미지/리소스인지 구분합니다.

React 쪽에서는 상태 범위 축소, 메모이제이션, 리스트 가상화, 불필요한 effect 제거를 봅니다.  
네트워크 쪽에서는 캐시, prefetch, 요청 병렬화/직렬화 전략, 중복 fetch 제거를 봅니다.

중요한 건 최적화를 많이 하는 게 아니라, **병목을 계측하고 가장 비싼 구간부터 줄이는 것**이라고 생각합니다.

### 꼬리 질문

#### `useMemo`, `useCallback`을 많이 쓰면 성능이 좋아지나요?

- 무조건 그렇지는 않습니다.
- 메모이제이션 자체도 비용이 있고, 필요 없는 곳에 쓰면 코드만 복잡해질 수 있습니다.
- 병목 지점을 확인한 뒤 선택적으로 적용하는 게 맞습니다.

#### React Profiler로 무엇을 보나요?

- 어떤 컴포넌트가 자주 렌더되는지, 왜 렌더되는지, 시간이 많이 드는 구간이 어디인지 봅니다.
- 특히 부모 상태 변경이 하위 전체를 자주 흔드는지 확인할 때 유용합니다.
- 렌더 횟수보다 렌더 비용과 원인을 같이 봐야 합니다.

#### 성능 최적화의 우선순위는 어떻게 정하나요?

- 체감 영향이 큰 사용자 경로부터 봅니다.
- 로그인, 검색, 에디터, 저장처럼 핵심 흐름의 병목을 먼저 줄이는 편입니다.
- 숫자상 작은 개선보다 핵심 UX를 지키는 개선이 더 가치 있습니다.

## 22. 긴 리스트, 그래프, 에디터 같은 무거운 화면에서 성능을 어떻게 다루나요?

### 모범 답변

이런 화면은 보통 "한 번 렌더가 무거운 문제"보다 **변화가 자주 일어나는 상태를 전체가 함께 받는 구조**가 병목인 경우가 많습니다.  
그래서 저는 먼저 상태 범위를 줄이고, 변경 빈도가 높은 상태를 분리합니다.

리스트는 가상화, 아이템 단위 memo, key 안정성, 비싼 계산의 사전 처리 여부를 봅니다.  
그래프나 에디터는 전체 재계산을 피하고, 변경된 부분만 업데이트할 수 있는 데이터 구조를 우선 봅니다.

또한 hover, drag, selection 같은 초고빈도 UI 상태는 전역 저장보다 로컬 처리나 별도 store 분리가 유리할 수 있습니다.  
특히 워크플로우 에디터에서는 "모든 노드가 같은 store 전체를 구독하는 구조"를 피하려고 합니다.

### 꼬리 질문

#### 가상화는 언제 유효한가요?

- 화면에 동시에 보이는 항목 수보다 전체 항목 수가 훨씬 많을 때 특히 효과가 큽니다.
- DOM 수를 줄여 스크롤 성능과 초기 렌더 비용을 낮출 수 있습니다.
- 다만 항목 높이 계산이나 접근성 이슈는 함께 고려해야 합니다.

#### key를 잘못 주면 왜 문제인가요?

- React가 항목 identity를 잘못 판단해 재사용/교체를 엉뚱하게 할 수 있습니다.
- 상태 보존과 렌더 최적화가 깨질 수 있습니다.
- 특히 정렬/삽입/삭제가 잦은 리스트에서 index key는 주의가 필요합니다.

#### 드래그 중 성능 저하는 어떻게 줄이나요?

- 매 프레임 전체 state를 갱신하지 않도록 구조를 나누는 것이 중요합니다.
- 가능하면 transform 기반 렌더와 로컬 상태를 활용합니다.
- 드래그 중 계산량을 줄이고, 종료 시점에만 영속 상태를 확정하는 방법도 좋습니다.

## 23. 테스트 전략을 어떻게 가져가나요?

### 모범 답변

저는 테스트를 많이 쓰는 것보다 **어떤 실패를 어디서 가장 싸게 잡을지** 기준으로 설계합니다.  
보통 단위 테스트, 통합 테스트, E2E 테스트의 역할을 분리합니다.

단위 테스트는 순수 함수, 포맷터, 유틸, 상태 전이 로직처럼 빠르고 명확한 영역에 적합합니다.  
통합 테스트는 컴포넌트 간 상호작용, 폼 제출, 로딩/에러/성공 흐름처럼 사용자 관점 행동을 검증할 때 유효합니다.  
E2E는 핵심 사용자 여정, 예를 들어 로그인, 생성, 저장, 결제 같은 가장 중요한 흐름을 보장하는 데 씁니다.

제 기준으로는 **단위 테스트로 로직 신뢰성**, **통합 테스트로 UI 흐름**, **E2E로 최종 계약**을 확보하는 구조가 가장 현실적입니다.

### 꼬리 질문

#### E2E를 많이 쓰면 더 안전하지 않나요?

- 안전할 수 있지만 실행 속도, flaky 이슈, 유지보수 비용이 커집니다.
- 모든 흐름을 E2E로 덮기보다 핵심 시나리오에 집중하는 편이 효율적입니다.
- 테스트 피라미드나 트로피 개념으로 균형을 잡는 게 좋습니다.

#### 통합 테스트가 중요한 이유는 무엇인가요?

- 사용자가 실제로 겪는 상호작용 단위를 비교적 빠르고 안정적으로 검증할 수 있기 때문입니다.
- 단위 테스트만으로는 놓치기 쉬운 컴포넌트 간 연결 문제를 잘 잡습니다.
- 프론트엔드에서는 실효성이 매우 높은 층이라고 생각합니다.

#### 스냅샷 테스트는 어떻게 보나요?

- 구조가 안정적이고 변동 의도가 명확한 출력에는 유용할 수 있습니다.
- 하지만 의미 없는 diff가 많아지면 검증 가치가 떨어질 수 있습니다.
- 저는 핵심 계약 검증보다 보조 도구로 쓰는 편입니다.

## 24. flaky test와 CI 신뢰성을 어떻게 다루나요?

### 모범 답변

flaky test는 단순히 귀찮은 문제가 아니라, 테스트 시스템 전체 신뢰도를 떨어뜨리는 문제라고 봅니다.  
한두 번 실패해도 그냥 다시 돌리는 문화가 쌓이면, 진짜 실패도 무시되기 쉽습니다.

원인은 보통 타이밍 의존, 네트워크/외부 환경 의존, 랜덤 데이터, 공유 상태 오염에 많습니다.  
그래서 저는 flaky가 나오면 먼저 "진짜 비동기 안정화가 안 된 건지", "테스트가 구현 세부사항에 너무 묶인 건지"부터 봅니다.

CI에서는 테스트를 계층화하는 편이 좋습니다.  
예를 들어 PR 단계에서는 빠른 unit/integration, main이나 nightly에서는 더 넓은 E2E 세트를 돌릴 수 있습니다.  
핵심은 속도와 신뢰성을 동시에 관리하는 것입니다.

### 꼬리 질문

#### `waitFor`를 많이 쓰면 해결되나요?

- 임시로 통과할 수는 있지만, 원인을 숨길 수도 있습니다.
- 명확한 상태 변화나 화면 조건을 기다리는 것이 중요합니다.
- 무작정 timeout을 늘리는 건 좋은 해결책이 아닙니다.

#### 테스트 격리는 왜 중요한가요?

- 이전 테스트가 남긴 상태가 다음 테스트 결과에 영향을 주면 재현성이 떨어집니다.
- 데이터, 쿠키, 스토리지, mock, DOM cleanup이 모두 대상입니다.
- 각 테스트가 독립적으로 통과해야 신뢰할 수 있습니다.

#### CI 속도와 품질은 어떻게 균형 잡나요?

- 모든 테스트를 모든 시점에 돌리기보다 목적별 파이프라인으로 나누는 편이 좋습니다.
- 빠른 피드백이 필요한 구간과, 넓은 보장이 필요한 구간을 분리합니다.
- 실패 비용이 큰 경로일수록 더 강하게 보호합니다.

## 25. 배포, 롤백, 로그, 메트릭을 프론트엔드 관점에서 어떻게 보나요?

### 모범 답변

프론트엔드도 배포 이후를 고려한 설계가 중요하다고 생각합니다.  
기능 구현만 끝나는 게 아니라, 배포 후 장애를 빨리 감지하고 되돌릴 수 있어야 운영 가능한 제품이 됩니다.

롤백 전략은 단순 git revert보다, 기능 플래그나 점진 배포와 같이 위험 범위를 줄이는 방식이 더 실무적일 수 있습니다.  
특히 실험성 기능이나 AI 기능은 품질 변동성이 커서 단계적 노출이 유리합니다.

로그와 메트릭은 사용자 행동보다 **실패 지점**을 먼저 잡는 데 큰 의미가 있습니다.  
예를 들어 특정 액션 실패율, 응답 지연, JS 에러 발생 위치, 페이지별 이탈 지점을 보면 어디가 문제인지 빨리 좁힐 수 있습니다.

저는 프론트 운영의 핵심을 "빠르게 감지하고, 안전하게 줄이고, 빨리 회복하는 능력"으로 봅니다.

### 꼬리 질문

#### 기능 플래그를 왜 중요하게 보나요?

- 코드 배포와 기능 노출을 분리할 수 있기 때문입니다.
- 문제가 생기면 전체 롤백 없이 기능만 끌 수도 있습니다.
- 실험, 점진 배포, 긴급 대응에 모두 유용합니다.

#### 프론트 로그는 무엇을 남겨야 하나요?

- 사용자 여정 전체보다, 실패 맥락과 재현에 필요한 최소 정보가 우선입니다.
- 요청 id, 화면 상태, 액션 종류, 에러 경로 같은 정보가 유용합니다.
- 민감 정보는 남기지 않도록 주의해야 합니다.

#### 메트릭과 로그의 차이는 무엇인가요?

- 로그는 개별 사건의 상세 기록에 가깝고, 메트릭은 집계된 추세를 보기 좋습니다.
- 장애 탐지는 메트릭, 원인 분석은 로그가 강한 경우가 많습니다.
- 둘을 함께 설계해야 운영 가시성이 좋아집니다.

# Part 5. 아키텍처 / 제품 감각 / AI 도구 / 프로젝트 설명

## 26. 프로젝트가 빠르게 늘어날 때 프론트엔드 아키텍처를 어떻게 설계하나요?

### 모범 답변

프로젝트가 빠르게 늘어나는 환경에서는 처음부터 완벽한 구조보다, **변화 방향을 견디는 경계 설계**가 중요하다고 생각합니다.  
저는 보통 공용 레이어와 도메인 레이어를 분리하는 것부터 시작합니다.

예를 들어 버튼, 모달, 입력 같은 공용 UI와, 워크플로우 편집, 챗 세션, 프로젝트 관리처럼 도메인 기능을 섞지 않도록 합니다.  
그리고 기능 단위로 폴더를 나누되, 너무 이른 공용화는 피합니다.

공통 코드는 여러 기능에서 반복 사용되면서 의미가 안정됐을 때 올리는 편이 낫습니다.  
너무 초기에 공용 레이어로 빼면, 오히려 각 도메인의 차이를 가리는 잘못된 추상화가 생길 수 있습니다.

결국 아키텍처는 멋있는 구조보다, **기능 증가 시 수정 범위를 예측 가능하게 만드는 것**이 핵심이라고 생각합니다.

### 꼬리 질문

#### 기능 중심 구조와 타입/컴포넌트 중심 구조 중 무엇을 선호하나요?

- 저는 대체로 기능 중심 구조를 선호합니다.
- 화면과 도메인 맥락이 가까워져서 변경 이유를 추적하기 쉽기 때문입니다.
- 다만 공용 레이어는 별도로 두어 균형을 맞춥니다.

#### 너무 이른 추상화가 왜 문제인가요?

- 아직 차이를 모르는 상태에서 공통점을 먼저 만들면, 나중에 예외 처리가 늘어나기 쉽습니다.
- 결과적으로 더 복잡하고 읽기 어려운 구조가 될 수 있습니다.
- 반복이 의미 있게 쌓인 뒤 추상화하는 편이 안전합니다.

#### 모노레포가 꼭 필요한가요?

- 팀 규모와 공유 자산 범위에 따라 다릅니다.
- 디자인 시스템, 공용 타입, 여러 앱 간 공유가 많다면 장점이 큽니다.
- 하지만 도입 비용과 운영 복잡도도 같이 봐야 합니다.

## 27. 디자인 시스템과 Product Engineer 관점을 어떻게 이해하고 있나요?

### 모범 답변

디자인 시스템은 단순히 공용 컴포넌트 모음이 아니라, **제품 전반의 일관성과 개발 속도를 동시에 높이는 계약층**이라고 생각합니다.  
버튼, 폼, 타이포그래피, 간격 토큰 같은 시각 요소뿐 아니라, 상태 표현 규칙과 접근성 기준도 포함됩니다.

다만 디자인 시스템을 지나치게 폐쇄적으로 만들면 제품 속도가 오히려 느려질 수 있습니다.  
그래서 저는 핵심 공통성은 강하게 유지하되, 실험 영역에서는 확장 가능한 슬롯이나 variant 구조를 두는 편이 좋다고 봅니다.

Product Engineer 관점에서는 구현 자체보다 **사용자 문제 해결과 제품 지표**까지 함께 봐야 합니다.  
즉 "이 컴포넌트를 예쁘게 만들었다"보다, "사용자가 더 빠르게 이해하고 덜 실패하게 만들었는가"가 중요합니다.

### 꼬리 질문

#### 디자인 시스템이 있으면 개발이 느려질 수도 있나요?

- 그럴 수 있습니다. 너무 엄격하거나 예외 대응이 어려우면 오히려 우회 구현이 많아집니다.
- 시스템은 통제와 속도의 균형이 필요합니다.
- 제품 변화 속도를 감당할 수 있어야 진짜 시스템입니다.

#### Product Engineer를 어떻게 정의하나요?

- 기능 구현에만 머물지 않고, 문제 정의와 사용자 가치까지 함께 보는 엔지니어라고 생각합니다.
- 기술 선택이 제품 임팩트와 연결되어야 합니다.
- UX 감도와 엔지니어링 판단이 함께 필요한 역할입니다.

#### 디자인 시스템과 UX 실험은 충돌하지 않나요?

- 충돌할 수 있지만, 시스템이 실험을 막기보다 안전한 범위 안에서 가능하게 해야 합니다.
- 안정적인 베이스 위에 실험 슬롯을 두는 방식이 현실적입니다.
- 시스템은 정답집이 아니라, 변화 비용을 줄이는 기반이어야 합니다.

## 28. Claude Code 같은 AI 코딩 도구를 어떻게 활용했나요?

### 모범 답변

저는 AI 코딩 도구를 "정답 생성기"보다 **초안 생성기이자 사고 가속기**로 보는 편입니다.  
반복적인 보일러플레이트, 테스트 초안, 문서화, 타입 초안, 비교안 정리 같은 작업에서 생산성이 높다고 느꼈습니다.

다만 핵심 로직이나 상태 경계, 성능 민감 구간은 그대로 믿지 않고 항상 검증합니다.  
특히 프론트엔드는 UI가 얼핏 동작해 보여도 상태 흐름이나 접근성, 에러 처리, 장기 유지보수 관점에서 문제가 숨어 있는 경우가 많습니다.

그래서 저는 AI가 제안한 코드를 볼 때도 "왜 이 구조를 택했는가", "실패 케이스가 빠진 건 없는가", "우리 프로젝트 규칙과 맞는가"를 직접 확인합니다.  
결국 AI 도구는 속도를 올려주지만, **설계 책임까지 대신하진 않는다**고 생각합니다.

### 꼬리 질문

#### AI가 만든 코드에서 가장 먼저 보는 부분은 무엇인가요?

- 상태 경계, 에러 처리, 의존성 관리, 성능 민감 구간부터 봅니다.
- 겉으로 동작하는지보다, 장기적으로 유지 가능한지가 더 중요합니다.
- 특히 불필요한 추상화와 숨은 사이드이펙트를 주의합니다.

#### AI 도구를 쓰면 코드 품질이 떨어질 수도 있나요?

- 검증 없이 붙이면 충분히 그럴 수 있습니다.
- 중복 패턴, 과한 추상화, 잘못된 최신 API 사용, 문맥 미스매치가 자주 나옵니다.
- 그래서 생성보다 검증 프로세스가 더 중요합니다.

#### 팀에서 AI 도구 활용 기준은 어떻게 잡는 게 좋나요?

- 어떤 작업까지 허용하는지, 검증 책임은 누구에게 있는지, 보안/민감 정보 기준은 무엇인지 명확해야 합니다.
- 코드리뷰 기준도 함께 정해야 합니다.
- 도구 도입보다 사용 원칙이 먼저라고 봅니다.

## 29. 기술 선택의 근거를 어떻게 설명하나요?

### 모범 답변

저는 기술 선택을 설명할 때 "좋아서 썼다"가 아니라, **문제와 제약을 먼저 말한 뒤 선택 이유를 붙이는 방식**을 씁니다.

예를 들어 "상태가 복잡해서 Zustand를 썼다"보다,  
"드래그와 선택 상태가 잦게 변하고, 여러 UI 조각이 읽어야 했지만 서버 상태는 아니어서 selector 기반 구독이 가능한 클라이언트 상태 도구가 필요했다"처럼 설명하는 편입니다.

그리고 항상 대안과 트레이드오프를 함께 말하려고 합니다.  
왜 Context가 아니었는지, 왜 React Query가 아니었는지, 왜 local state만으로는 부족했는지까지 설명해야 판단 근거가 살아납니다.

면접에서는 정답보다 **의사결정의 논리와 일관성**이 더 중요하다고 생각합니다.

### 꼬리 질문

#### 기술 선택에서 가장 먼저 말해야 하는 것은 무엇인가요?

- 기술 이름보다 문제 정의와 제약 조건입니다.
- 무엇이 어려웠고, 무엇을 우선했는지 먼저 말해야 선택 이유가 살아납니다.
- 도구는 그다음입니다.

#### 트레이드오프를 꼭 말해야 하나요?

- 네. 장점만 말하면 도구를 이해한 게 아니라 홍보한 것처럼 들릴 수 있습니다.
- 단점과 보완책까지 말해야 판단력이 드러납니다.
- 실무자는 보통 완벽한 도구가 없다는 걸 알고 있습니다.

#### "팀이 원해서 썼다"는 답변은 약한가요?

- 그대로 끝나면 약할 수 있습니다.
- 하지만 팀 기준을 이해하고, 그 안에서 어떤 장단점을 봤는지 설명하면 충분히 괜찮은 답변이 됩니다.
- 중요한 건 수동적 선택이 아니라 이해 기반 선택처럼 들리는 것입니다.

## 30. **MVP(Minimum Viable Product, 최소 기능 제품 — 핵심 가설을 검증할 수 있는 최소한의 기능을 갖춘 제품)**에서 production-grade로 넘어갈 때 무엇을 바꾸나요?

### 모범 답변

MVP 단계에서는 속도가 중요하기 때문에, 구조가 다소 거칠더라도 핵심 가설을 빨리 검증하는 것이 우선일 수 있습니다.  
하지만 production 단계로 갈수록 요구사항이 바뀝니다. 속도뿐 아니라 안정성, 관찰 가능성, 테스트 가능성, 확장성을 함께 봐야 합니다.

그래서 저는 MVP에서 production으로 갈 때 보통 네 가지를 점검합니다.

1. **상태와 책임 경계 재정리**
2. **에러 처리와 실패 복구 흐름 보강**
3. **테스트 및 배포 안정성 확보**
4. **로그/메트릭/기능 플래그 등 운영 장치 추가**

예를 들어 MVP에서는 한 컴포넌트에 몰아넣었던 로직도, production에선 도메인 훅, 프레젠테이션 컴포넌트, 서버 상태 계층으로 나눌 수 있습니다.  
핵심은 "처음부터 완벽하게"가 아니라, **검증 이후에 부채를 언제 어떤 기준으로 갚을지 아는 것**이라고 생각합니다.

### 꼬리 질문

#### MVP 코드가 꼭 나쁜 코드는 아닌가요?

- 그렇습니다. 목적이 다를 뿐입니다.
- 검증 단계에서 빠른 실험을 가능하게 했다면 충분히 가치가 있습니다.
- 다만 그 구조를 운영 단계까지 그대로 끌고 가면 문제가 커질 수 있습니다.

#### production-grade를 어떻게 판단하나요?

- 장애 대응 가능성, 테스트 가능성, 관찰 가능성, 변경 용이성 기준으로 봅니다.
- 기능이 동작하는지만으로는 부족합니다.
- 운영 중 실패를 감당할 수 있어야 production-grade에 가깝습니다.

#### 언제 리팩터링해야 하나요?

- 기능 변화가 반복되며 같은 고통이 누적될 때가 신호라고 봅니다.
- "예쁘게 만들고 싶어서"보다, 변경 비용과 장애 위험이 커졌을 때 우선순위가 올라갑니다.
- 리팩터링도 문제 기반으로 설명할 수 있어야 합니다.

## 보너스. 접근성(a11y)을 워크플로우 에디터나 챗 UI에서 어떻게 고려하나요?

### 모범 답변

워크플로우 에디터나 챗 UI처럼 인터랙션이 많은 화면에서는 접근성이 특히 중요합니다.
키보드만으로 노드 탐색, 연결, 삭제가 가능해야 하고, 스크린 리더가 현재 상태를 알 수 있어야 합니다.

챗 UI에서는 새 메시지가 추가될 때 `aria-live="polite"`로 스크린 리더에 알리고, 입력 필드에 `aria-label`을 명확히 두며, 포커스가 자연스럽게 이동해야 합니다.

에디터에서는 노드 선택/이동을 키보드로 할 수 있게 `tabIndex`와 `onKeyDown` 핸들러를 설계하고, 현재 선택된 노드 정보를 `aria-selected`나 `aria-describedby`로 전달합니다.

접근성은 "추가 기능"이 아니라 **제품 품질의 기본 축**이라고 생각합니다.

### 꼬리 질문

#### 시맨틱 HTML이 왜 중요하나요?

- `<button>`, `<nav>`, `<main>`, `<dialog>` 같은 시맨틱 태그는 브라우저와 보조 기술이 역할을 자동으로 이해합니다.
- `<div onClick>`으로 버튼을 만들면 키보드 포커스, Enter/Space 처리, 스크린 리더 인식을 전부 직접 구현해야 합니다.
- 시맨틱 태그를 쓰면 접근성 기본값이 자동으로 확보됩니다.

#### 포커스 트랩은 언제 필요하나요?

- 모달이나 다이얼로그가 열렸을 때, Tab 키가 모달 밖으로 빠져나가면 안 되는 경우에 필요합니다.
- 워크플로우 에디터에서 설정 패널이 열렸을 때도 포커스가 패널 안에 머물러야 합니다.
- `focus-trap-react` 같은 라이브러리를 쓰거나, `inert` 속성으로 외부를 비활성화할 수 있습니다.

#### aria-live는 어떤 상황에 쓰나요?

- 화면 변화가 사용자 액션 없이 동적으로 발생할 때(새 메시지 도착, 토스트 알림, 검증 에러 표시) 사용합니다.
- `polite`는 현재 읽는 내용을 끝낸 후 알리고, `assertive`는 즉시 알립니다.
- 챗 UI에서 assistant 응답이 완료됐을 때 `aria-live="polite"` 영역에 메시지를 넣으면 스크린 리더가 자동으로 읽어줍니다.

# 챕터 14 핵심 요약 — 이론에서 답변으로, 개념에서 설계 판단으로

이 챕터 전체를 관통하는 하나의 질문이 있다.

> **챕터 1~13에서 쌓아온 이론적 뼈대를, 실제 면접에서 "왜 그런 판단을 했는가"라는 질문에 어떻게 연결하는가?**

이 질문에 답하려면, 암기가 아니라 답변 구조를 훈련해야 한다. 문제 정의 → 판단 기준 → 선택지 비교 → 트레이드오프 → 운영 관점, 이 순서를 반복하는 것이 핵심이다.

---

### 1단계: 언어의 뿌리를 면접 언어로 바꾸기 — JS/TS 기본기

stale closure는 챕터 1에서 배운 클로저와 React의 렌더 스냅샷이 결합할 때 발생한다. 불변성은 챕터 2의 참조 동일성과 연결되어 최적화의 전제조건이 된다. 비동기 실행 순서와 race condition은 챕터 3의 이벤트 루프 위에 놓여 있다. **이론을 아는 것과, "저는 이 문제를 dependency 정리 → 함수형 업데이트 → ref 순서로 고려합니다"라고 말하는 것은 완전히 다른 능력이다.** TypeScript에서는 `any`/`unknown`/`never`의 구분, 제네릭과 판별 유니온의 UI 상태 모델링, 런타임 검증이 필요한 이유까지 — 타입 시스템이 설계 도구임을 설명할 수 있어야 한다.

### 2단계: React를 "왜"로 설명하기 — 렌더링, 상태 구조, 복잡한 UI

React를 선언형이라 부르는 이유, render phase와 commit phase의 구분, state ownership을 잡는 기준, Context/Zustand/React Query를 역할로 나누는 판단. **핵심은 "가까운 곳에 둘 수 있으면 최대한 가깝게 두고, 공유 필요가 명확할 때만 올린다"는 원칙이다.** `useEffect`와 `useLayoutEffect`의 시점 차이, 워크플로우 에디터의 상태 구조를 영속/파생으로 분리하는 설계 — 이 모든 것이 "어떻게"가 아니라 "왜 그 판단을 했는가"로 답변돼야 한다.

### 3단계: 서버 경계와 네트워크를 설계 관점에서 말하기 — Next.js, 인증, 통신

SSR/SSG/ISR/CSR은 기술 선호가 아니라 데이터 특성, SEO, 초기 응답 속도, 개인화 정도로 판단한다. Server Component와 Client Component의 경계는 "페이지 뼈대는 서버, 인터랙션 섬은 클라이언트"로 잡는다. **URL은 공유 가능하고 복원 가능한 상태 저장소이며, 세션 만료 UX에서는 입력 상태 보존이 기술보다 중요하다.** SSE, WebSocket, Polling은 방향성, 빈도, 실시간성, 인프라 복잡도로 구분한다.

### 4단계: 제품을 만드는 엔지니어의 시선 — AI 제품, 성능, 테스트, 운영

스트리밍 챗 UI는 리스트 렌더링 문제가 아니라 비동기 상태 머신과 UX 설계 문제다. optimistic UI는 실패 비용이 낮고 rollback이 가능한 작업에만 쓴다. 성능 최적화는 "느리다"를 구체적 병목으로 바꾸는 것에서 시작하고, **가장 비싼 구간부터 줄이는 것이 핵심이다.** 테스트는 단위로 로직 신뢰성, 통합으로 UI 흐름, E2E로 최종 계약을 확보한다. 배포 이후에는 기능 플래그로 노출을 제어하고, 로그와 메트릭으로 실패를 빨리 잡는다.

### 5단계: 구조와 판단력을 보여주기 — 아키텍처, 기술 선택, 성장

프로젝트가 빠르게 늘어날 때 처음부터 완벽한 구조보다 변화 방향을 견디는 경계 설계가 중요하다. 디자인 시스템은 공용 컴포넌트 모음이 아니라 제품 전반의 일관성과 속도를 높이는 계약층이다. AI 코딩 도구는 초안 생성기이자 사고 가속기지만 설계 책임까지 대신하지 않는다. **기술 선택은 "좋아서 썼다"가 아니라 문제와 제약을 먼저 말한 뒤 선택 이유를 붙이는 방식으로 설명해야 한다.** MVP에서 production-grade로 갈 때는 상태 경계 재정리, 에러 처리 보강, 테스트 확보, 운영 장치 추가 네 가지를 점검한다.

---

### 한 줄로 이어 읽기

**stale closure와 불변성**으로 JS/TS 이론을 면접 언어로 바꾸고 → **선언형 패러다임과 state ownership**으로 React의 "왜"를 설명하고 → **서버/클라이언트 경계와 URL 상태**로 Next.js 설계 판단을 보여주고 → **스트리밍 UI, optimistic update, 성능 계측, 테스트 전략**으로 제품 엔지니어의 시선을 드러내고 → **아키텍처 경계, 기술 선택 근거, MVP에서 production으로의 전환**으로 설계 판단력과 성장 가능성을 증명한다.

### 전체 시리즈를 되돌아보며

챕터 1에서 스코프와 클로저로 "함수가 값을 어떻게 기억하는가"를 물었다. 챕터 2~3에서 원시값/참조값, 이벤트 루프로 "변경을 어떻게 감지하고, 비동기를 어떻게 다루는가"로 확장했다. 챕터 4~5에서 TypeScript와 React 내부 구조로 "타입과 상태가 설계를 어떻게 단단하게 만드는가"를 배웠다. 챕터 6~11에서 Next.js, 네트워크, 성능, 테스트, 인프라, 면접 스토리텔링으로 "실무에서 이것들이 어떻게 결합되는가"를 훑었다. 챕터 12~13에서 남은 이론적 공백을 모두 채웠다. 그리고 이 마지막 챕터에서, 그 모든 것을 "면접에서 자기 언어로 설명하는 능력"으로 전환했다.

기술 면접의 본질은 암기가 아니다. 하나의 개념이 다른 개념과 어떻게 연결되는지 — 이벤트 루프가 왜 싱글 스레드와 연결되는지, 불변성이 왜 최적화의 전제조건인지, 타입 시스템이 왜 런타임과 별개의 문제인지 — 이 연결고리를 자기 경험 위에서 설명할 수 있을 때, 어떤 질문이 와도 흔들리지 않는다. **개념을 아는 것에서 끝나지 않고, 그것이 자신의 코드에서 어떻게 작동했는지를 말할 수 있을 때 비로소 설득력이 생긴다.**
